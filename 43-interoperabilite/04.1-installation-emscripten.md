# 43.4.1 — Installation Emscripten

## Module 15 : Interopérabilité · Niveau Expert

---

## Introduction

Emscripten est distribué sous la forme d'un SDK autonome (`emsdk`) qui inclut tout le nécessaire : le compilateur Clang configuré pour le backend Wasm, l'outil Binaryen pour l'optimisation du code WebAssembly, Node.js pour exécuter les outils de build, et les bibliothèques d'émulation (libc, libcxx, SDL2, OpenGL). L'installation ne modifie pas le compilateur système — `emsdk` gère sa propre copie isolée de la toolchain.

Cette section couvre l'installation sur Ubuntu, la vérification de l'environnement, l'intégration avec CMake, et la configuration pour les environnements Docker et CI/CD.

---

## 43.4.1.1 — Installation du SDK sur Ubuntu

### Prérequis système

Emscripten nécessite quelques outils de base déjà présents sur un poste de développement Ubuntu configuré selon le chapitre 2 :

```bash
# Prérequis (normalement déjà installés)
sudo apt update  
sudo apt install -y git python3 cmake ninja-build  
```

Python 3 est requis car certains outils internes d'Emscripten sont des scripts Python. Git est nécessaire pour cloner le dépôt `emsdk`.

### Installation via emsdk

`emsdk` est le gestionnaire de versions officiel d'Emscripten. Il permet d'installer, d'activer et de basculer entre plusieurs versions de la toolchain :

```bash
# Cloner le dépôt emsdk
cd ~/tools    # ou tout autre répertoire de votre choix  
git clone https://github.com/emscripten-core/emsdk.git  
cd emsdk  

# Installer la dernière version stable
./emsdk install latest

# Activer la version installée (configure les variables d'environnement)
./emsdk activate latest

# Charger les variables d'environnement dans le shell courant
source ./emsdk_env.sh
```

La commande `source ./emsdk_env.sh` ajoute les outils Emscripten (`emcc`, `em++`, `emcmake`, `emmake`, `emar`) au `PATH` du shell courant. Cette commande doit être exécutée dans chaque nouveau terminal, ou ajoutée au fichier d'initialisation du shell.

### Rendre l'activation permanente

Pour éviter de relancer `source emsdk_env.sh` à chaque session :

```bash
# Option 1 : Ajouter au .bashrc (ou .zshrc)
echo 'source ~/tools/emsdk/emsdk_env.sh 2>/dev/null' >> ~/.bashrc

# Option 2 : Ajouter uniquement le PATH (plus propre, moins verbeux)
echo 'export PATH="$HOME/tools/emsdk:$HOME/tools/emsdk/upstream/emscripten:$PATH"' >> ~/.bashrc
```

La redirection `2>/dev/null` dans l'option 1 supprime les messages informatifs qu'`emsdk_env.sh` affiche à chaque chargement.

### Vérification de l'installation

```bash
# Vérifier les outils disponibles
emcc --version
# emcc (Emscripten gcc/clang-like replacement + linker emulating GNU ld) 3.x.x
# (la version exacte dépend de la date d'installation)

em++ --version
# Même sortie — em++ est l'équivalent de g++ pour Emscripten

# Vérifier que le backend Wasm fonctionne
echo '#include <cstdio>  
int main() { std::printf("Hello from Wasm!\\n"); return 0; }' > /tmp/test.cpp  

emcc -std=c++20 -O2 -o /tmp/test.html /tmp/test.cpp

# Vérifier les fichiers produits
ls -la /tmp/test.html /tmp/test.js /tmp/test.wasm
```

Si les trois fichiers sont générés sans erreur, l'installation est fonctionnelle.

### Tester dans un navigateur

Le fichier `.html` généré ne peut pas être ouvert directement avec `file://` dans le navigateur — le chargement du `.wasm` échoue en raison des restrictions CORS sur le protocole `file://`. Il faut un serveur HTTP local :

```bash
# Serveur HTTP minimal (Python 3)
cd /tmp  
python3 -m http.server 8080  

# Ouvrir dans le navigateur
# http://localhost:8080/test.html
```

La page affiche une console Emscripten avec le message "Hello from Wasm!".

### Tester avec Node.js (sans navigateur)

Emscripten inclut Node.js dans le SDK. On peut exécuter le `.js` directement :

```bash
node /tmp/test.js
# Hello from Wasm!
```

Cette méthode est utile pour les tests automatisés en CI où aucun navigateur n'est disponible.

## 43.4.1.2 — Gestion des versions

### Installer une version spécifique

Pour la reproductibilité des builds, il est recommandé d'épingler une version précise plutôt que d'utiliser `latest` :

```bash
# Lister les versions disponibles
./emsdk list

# Installer une version précise
./emsdk install 3.1.74

# Activer cette version
./emsdk activate 3.1.74
source ./emsdk_env.sh

# Vérifier
emcc --version
```

### Basculer entre versions

```bash
# Activer une version différente (déjà installée)
./emsdk activate 3.1.70
source ./emsdk_env.sh

# Revenir à la dernière
./emsdk activate latest
source ./emsdk_env.sh
```

### Documenter la version dans le projet

Pour garantir la reproductibilité, consigner la version Emscripten requise dans le projet :

```bash
# Fichier .emscripten-version à la racine du projet
echo "3.1.74" > .emscripten-version
```

Ce fichier n'est pas lu automatiquement par Emscripten (contrairement à `.node-version` ou `.python-version`), mais il sert de documentation pour les contributeurs et les scripts CI.

## 43.4.1.3 — Les outils de la toolchain

### Correspondance avec les outils natifs

Emscripten fournit des remplacements drop-in pour les outils de compilation classiques :

| Outil natif | Équivalent Emscripten | Rôle |
|---|---|---|
| `gcc` / `clang` | `emcc` | Compilateur C |
| `g++` / `clang++` | `em++` | Compilateur C++ |
| `ar` | `emar` | Archiveur (bibliothèques statiques `.a`) |
| `ranlib` | `emranlib` | Index d'archive |
| `nm` | `emnm` | Symboles des fichiers objet |
| `cmake` | `emcmake cmake` | Wrapper CMake avec toolchain Emscripten |
| `make` | `emmake make` | Wrapper Make avec environnement Emscripten |
| `pkg-config` | `emconfigure pkg-config` | Configuration des packages |

Les wrappers `emcmake`, `emmake` et `emconfigure` ne font que configurer les variables d'environnement (`CC`, `CXX`, `AR`, etc.) pour pointer vers les outils Emscripten, puis exécuter la commande sous-jacente. Ils sont un raccourci pour éviter de définir manuellement ces variables.

### emcc vs em++

`emcc` compile du C, `em++` compile du C++. La distinction est identique à celle entre `gcc` et `g++` — `em++` lie automatiquement la bibliothèque standard C++. En pratique, on peut utiliser `emcc` pour tout (il détecte le langage par l'extension du fichier), mais `em++` est plus explicite pour les projets C++.

## 43.4.1.4 — Intégration avec CMake

### La toolchain file Emscripten

Emscripten fournit une **toolchain file** CMake qui configure automatiquement le compilateur, les flags et les chemins pour la cross-compilation vers Wasm. C'est la méthode recommandée pour les projets CMake.

```bash
# Méthode 1 : emcmake (la plus simple)
emcmake cmake -B build -G Ninja  
cmake --build build  
```

`emcmake cmake` est un wrapper qui passe automatiquement `-DCMAKE_TOOLCHAIN_FILE=<chemin_vers_emsdk>/upstream/emscripten/cmake/Modules/Platform/Emscripten.cmake` à CMake.

```bash
# Méthode 2 : toolchain file explicite (plus de contrôle)
cmake -B build -G Ninja \
    -DCMAKE_TOOLCHAIN_FILE=$EMSDK/upstream/emscripten/cmake/Modules/Platform/Emscripten.cmake

cmake --build build
```

La méthode 2 est préférable dans les scripts CI et les CMake Presets, car elle ne dépend pas du wrapper `emcmake` et rend la toolchain explicite.

### CMakeLists.txt compatible Emscripten

Un projet CMake bien structuré compile souvent vers Wasm sans modification du `CMakeLists.txt`. La toolchain file configure les variables nécessaires (`CMAKE_SYSTEM_NAME = Emscripten`, `CMAKE_C_COMPILER = emcc`, etc.).

Cependant, certains ajustements sont souvent nécessaires :

```cmake
cmake_minimum_required(VERSION 3.24)  
project(mylib LANGUAGES CXX)  

set(CMAKE_CXX_STANDARD 20)  
set(CMAKE_CXX_STANDARD_REQUIRED ON)  

add_executable(myapp src/main.cpp src/engine.cpp)

# Options spécifiques à Emscripten
if(EMSCRIPTEN)
    # Générer un .html avec la console Emscripten
    set_target_properties(myapp PROPERTIES SUFFIX ".html")

    # Options de linkage Emscripten (-s flags)
    target_link_options(myapp PRIVATE
        -sWASM=1
        -sALLOW_MEMORY_GROWTH=1
        -sEXPORTED_RUNTIME_METHODS=ccall,cwrap
    )
endif()
```

La variable `EMSCRIPTEN` est définie automatiquement par la toolchain file. Le guard `if(EMSCRIPTEN)` permet de maintenir un CMakeLists.txt unique qui compile vers le natif **et** vers Wasm selon la toolchain utilisée.

### CMake Presets pour Emscripten

Les CMake Presets (section 27.6) permettent d'encoder la configuration Emscripten de manière portable :

```json
{
    "version": 6,
    "configurePresets": [
        {
            "name": "native-release",
            "generator": "Ninja",
            "binaryDir": "${sourceDir}/build/native",
            "cacheVariables": {
                "CMAKE_BUILD_TYPE": "Release"
            }
        },
        {
            "name": "wasm-release",
            "generator": "Ninja",
            "binaryDir": "${sourceDir}/build/wasm",
            "toolchainFile": "$env{EMSDK}/upstream/emscripten/cmake/Modules/Platform/Emscripten.cmake",
            "cacheVariables": {
                "CMAKE_BUILD_TYPE": "Release"
            }
        }
    ]
}
```

```bash
# Compilation native
cmake --preset native-release  
cmake --build build/native  

# Compilation Wasm
cmake --preset wasm-release  
cmake --build build/wasm  
```

Ce setup permet aux développeurs de basculer entre la compilation native (pour le développement et le débogage) et la compilation Wasm (pour le déploiement web) avec un simple changement de preset.

## 43.4.1.5 — Configuration Docker pour la CI/CD

### Image Docker officielle

Emscripten fournit des images Docker officielles prêtes à l'emploi :

```dockerfile
FROM emscripten/emsdk:3.1.74

WORKDIR /app  
COPY . .  

RUN emcmake cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Release \
    && cmake --build build
```

L'image `emscripten/emsdk` inclut la toolchain complète, Node.js, Python et CMake. Elle est volumineuse (environ 2 Go) mais évite toute configuration manuelle.

### Multi-stage pour un déploiement léger

```dockerfile
# ── Stage 1 : Build ──────────────────────────────────────────
FROM emscripten/emsdk:3.1.74 AS builder

WORKDIR /app  
COPY . .  

RUN emcmake cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Release \
    && cmake --build build

# ── Stage 2 : Serveur web minimal ────────────────────────────
FROM nginx:alpine

COPY --from=builder /app/build/*.html /usr/share/nginx/html/  
COPY --from=builder /app/build/*.js   /usr/share/nginx/html/  
COPY --from=builder /app/build/*.wasm /usr/share/nginx/html/  
COPY --from=builder /app/build/*.data /usr/share/nginx/html/ 2>/dev/null || true  
```

Le stage de build contient la toolchain Emscripten complète. Le stage de production ne contient que Nginx et les fichiers web générés — quelques mégaoctets au total.

### Pipeline GitHub Actions

```yaml
name: Build WebAssembly

on: [push, pull_request]

jobs:
  build-wasm:
    runs-on: ubuntu-24.04
    container:
      image: emscripten/emsdk:3.1.74

    steps:
      - uses: actions/checkout@v4

      - name: Configure
        run: emcmake cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Release

      - name: Build
        run: cmake --build build

      - name: Test (Node.js)
        run: node build/myapp.js

      - name: Upload artifacts
        uses: actions/upload-artifact@v4
        with:
          name: wasm-build
          path: |
            build/*.html
            build/*.js
            build/*.wasm
```

L'utilisation du conteneur Docker Emscripten dans le job CI garantit la reproductibilité exacte du build, indépendamment de l'environnement du runner.

## 43.4.1.6 — Options de compilation essentielles

### Flags de `emcc` / `em++`

Les flags standard de Clang fonctionnent avec `emcc` :

```bash
# Standard C++
emcc -std=c++20 ...

# Optimisation
emcc -O0 ...   # Pas d'optimisation (développement)  
emcc -O2 ...   # Optimisation recommandée (production)  
emcc -O3 ...   # Optimisation agressive  
emcc -Os ...   # Optimisation pour la taille (web = réseau)  
emcc -Oz ...   # Optimisation maximale pour la taille  

# Warnings
emcc -Wall -Wextra ...

# Debug (génère source maps pour le débogage navigateur)
emcc -g ...  
emcc -gsource-map ...  
```

Pour le web, `-Os` ou `-Oz` sont souvent préférés à `-O3` : la taille du `.wasm` impacte directement le temps de téléchargement et de parsing. Un module 500 Ko se charge quasi instantanément ; un module 10 Mo crée une attente perceptible.

### Flags spécifiques Emscripten (-s)

Les flags `-s` (ou `-sNOM=VALEUR` dans la syntaxe moderne) configurent le comportement du runtime Emscripten. Les plus importants :

```bash
# Mémoire
-sALLOW_MEMORY_GROWTH=1    # Le heap peut grandir au-delà de la taille initiale
-sINITIAL_MEMORY=67108864   # Taille initiale du heap (64 Mo)
-sMAXIMUM_MEMORY=536870912   # Taille maximale du heap (512 Mo)
-sSTACK_SIZE=1048576          # Taille de la pile (1 Mo)

# Exports
-sEXPORTED_FUNCTIONS=_main,_myFunction   # Fonctions C à exporter
-sEXPORTED_RUNTIME_METHODS=ccall,cwrap   # Méthodes du runtime JS à exposer

# Modules ES6
-sEXPORT_ES6=1               # Exporter comme module ES6
-sMODULARIZE=1               # Encapsuler dans une factory function

# Threads
-pthread                      # Activer le support des threads (Web Workers)
-sPTHREAD_POOL_SIZE=4        # Pré-créer 4 Web Workers

# Système de fichiers
-sFORCE_FILESYSTEM=1          # Inclure le FS même si non détecté
--preload-file assets@/assets  # Pré-charger un répertoire dans le FS virtuel

# WebGL
-sUSE_WEBGL2=1                # Activer WebGL 2.0 (OpenGL ES 3.0)
-sFULL_ES3=1                  # Émulation complète OpenGL ES 3.0

# Bindings
--bind                         # Activer Embind (classes C++ → JavaScript)
-lembind                       # Équivalent (syntaxe alternative)
```

### Le flag `-sALLOW_MEMORY_GROWTH`

Ce flag mérite une attention particulière. Par défaut, Emscripten alloue un bloc de mémoire linéaire de taille fixe (16 Mo par défaut) pour le heap Wasm. Si le programme dépasse cette taille, il crashe avec une erreur `out of memory`. `-sALLOW_MEMORY_GROWTH=1` permet au heap de grandir dynamiquement, comme un programme natif.

L'inconvénient historique de ce flag était un léger overhead de performance (les accès mémoire ne pouvaient plus être optimisés comme des offsets fixes). En 2026, cet overhead est négligeable dans la majorité des cas grâce aux optimisations des moteurs Wasm modernes. **Activer `-sALLOW_MEMORY_GROWTH=1` systématiquement** est la recommandation pour tout projet non trivial.

## 43.4.1.7 — Diagnostic des problèmes courants

### `emcc: command not found`

L'environnement Emscripten n'est pas activé dans le shell courant.

```bash
source ~/tools/emsdk/emsdk_env.sh
```

### `wasm streaming compile failed` (dans le navigateur)

Le serveur web ne sert pas le fichier `.wasm` avec le bon type MIME. Le navigateur attend `application/wasm`.

```bash
# Vérifier le Content-Type
curl -I http://localhost:8080/myapp.wasm | grep Content-Type
# Doit être : Content-Type: application/wasm
```

Avec le serveur Python :

```bash
# Python 3.x sert .wasm avec le bon MIME par défaut
python3 -m http.server 8080
```

Si le type MIME est incorrect (certains serveurs anciens), ajouter la configuration manuellement. Pour Nginx :

```nginx
types {
    application/wasm wasm;
}
```

### `SharedArrayBuffer is not defined` (threads)

Les threads Wasm (pthreads) nécessitent `SharedArrayBuffer`, qui exige les headers HTTP COOP/COEP :

```
Cross-Origin-Opener-Policy: same-origin  
Cross-Origin-Embedder-Policy: require-corp  
```

Sans ces headers, le navigateur désactive `SharedArrayBuffer` pour des raisons de sécurité (mitigation Spectre). Si le projet n'utilise pas les threads, ce problème ne se pose pas.

### `out of memory`

Le programme dépasse la taille du heap Wasm alloué.

```bash
# Activer la croissance mémoire
emcc -sALLOW_MEMORY_GROWTH=1 ...

# Ou augmenter la taille initiale (ici 256 Mo)
emcc -sINITIAL_MEMORY=268435456 ...
```

### `undefined symbol: _Z...` (symbole manglé)

Une fonction exportée utilise le mangling C++. Ajouter `extern "C"` ou utiliser Embind (`--bind`).

### Temps de compilation longs

Emscripten invoque Clang et Binaryen, qui sont plus lents que la compilation native (pas d'avantage de ccache pour le wasm par défaut). Pour accélérer :

```bash
# Utiliser -O0 pendant le développement
emcc -O0 ...

# Activer le cache d'Emscripten (activé par défaut)
echo $EM_CACHE   # Vérifier le chemin du cache

# Compilation parallèle avec Ninja (via CMake)
cmake --build build -- -j$(nproc)
```

Le cache Emscripten (`$EM_CACHE`) stocke les bibliothèques système pré-compilées (libc, libcxx, etc.). La première compilation est lente car ces bibliothèques sont construites ; les suivantes sont significativement plus rapides.

## 43.4.1.8 — Vérification de l'environnement complet

Un script de vérification rapide pour valider que tout est en place :

```bash
#!/bin/bash
# verify_emscripten.sh — Vérification de l'environnement Emscripten

set -e

echo "=== Vérification de l'environnement Emscripten ==="

echo -n "emcc: "  
emcc --version | head -1  

echo -n "em++: "  
em++ --version | head -1  

echo -n "node: "  
node --version  

echo -n "cmake: "  
cmake --version | head -1  

echo -n "ninja: "  
ninja --version  

echo ""  
echo "=== Test de compilation ==="  
TMPDIR=$(mktemp -d)  
cat > "$TMPDIR/test.cpp" << 'EOF'  
#include <cstdio>
#include <vector>
#include <numeric>

int main() {
    std::vector<int> v = {1, 2, 3, 4, 5};
    int sum = std::accumulate(v.begin(), v.end(), 0);
    std::printf("Sum: %d\n", sum);
    return 0;
}
EOF

emcc -std=c++20 -O2 -o "$TMPDIR/test.js" "$TMPDIR/test.cpp"  
RESULT=$(node "$TMPDIR/test.js")  

if [ "$RESULT" = "Sum: 15" ]; then
    echo "Compilation et exécution : OK"
else
    echo "ERREUR : résultat inattendu : $RESULT"
    exit 1
fi

echo ""  
echo "=== Test CMake ==="  
cat > "$TMPDIR/CMakeLists.txt" << 'EOF'  
cmake_minimum_required(VERSION 3.20)  
project(test_wasm LANGUAGES CXX)  
set(CMAKE_CXX_STANDARD 20)  
add_executable(test_wasm test.cpp)  
EOF  

emcmake cmake -B "$TMPDIR/build" -S "$TMPDIR" -G Ninja > /dev/null 2>&1  
cmake --build "$TMPDIR/build" > /dev/null 2>&1  
RESULT2=$(node "$TMPDIR/build/test_wasm.js")  

if [ "$RESULT2" = "Sum: 15" ]; then
    echo "CMake + Emscripten : OK"
else
    echo "ERREUR CMake : résultat inattendu : $RESULT2"
    exit 1
fi

rm -rf "$TMPDIR"  
echo ""  
echo "=== Environnement Emscripten validé ==="  
```

```bash
chmod +x verify_emscripten.sh
./verify_emscripten.sh
```

Si les deux tests passent, l'environnement est prêt pour la compilation de projets C++ vers WebAssembly.

## Résumé

| Étape | Commande / Action |
|---|---|
| Cloner emsdk | `git clone https://github.com/emscripten-core/emsdk.git` |
| Installer | `./emsdk install latest` (ou version épinglée) |
| Activer | `./emsdk activate latest && source ./emsdk_env.sh` |
| Vérifier | `emcc --version` |
| Compiler (CLI) | `emcc -std=c++20 -O2 -o app.html main.cpp` |
| Compiler (CMake) | `emcmake cmake -B build -G Ninja && cmake --build build` |
| Tester (Node) | `node app.js` |
| Tester (navigateur) | `python3 -m http.server 8080` + `localhost:8080/app.html` |
| CI/CD (Docker) | `FROM emscripten/emsdk:3.1.74` |

L'installation d'Emscripten est simple — un clone Git et trois commandes. La complexité réside dans la configuration des options de compilation (mémoire, exports, threads) et l'intégration dans le workflow de build existant. Les CMake Presets permettent de maintenir une configuration portable qui compile vers le natif et vers Wasm depuis le même projet.
