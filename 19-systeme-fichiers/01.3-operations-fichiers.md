# 19.1.3 — Opérations sur fichiers et répertoires

## Module 7 — Programmation Système sur Linux *(Niveau Avancé)*

---

## Introduction

Les sections précédentes ont couvert le parcours d'arborescences (19.1.1) et la manipulation de chemins (19.1.2). Cette section se concentre sur les **opérations concrètes** que `std::filesystem` permet d'effectuer sur les fichiers et répertoires : création, copie, déplacement, suppression, gestion des liens symboliques et des fichiers temporaires.

Ces opérations constituent le quotidien de la programmation système : créer une arborescence de build, copier des artefacts de compilation, nettoyer des fichiers temporaires, déplacer des logs archivés, mettre en place des liens symboliques pour le déploiement. Chaque opération interagit avec le noyau Linux via des appels système, et chacune peut échouer de multiples façons. La robustesse du code dépend directement de la rigueur avec laquelle ces erreurs sont anticipées et traitées.

Rappel : `std::filesystem` gère l'**arborescence et les métadonnées**, mais pas le **contenu** des fichiers. La lecture et l'écriture de données restent du domaine de `<fstream>` ou des appels POSIX (section 19.2).

## Création de répertoires

### `create_directory()` : un seul niveau

`fs::create_directory()` crée un unique répertoire. Le répertoire parent doit déjà exister, sinon l'opération échoue :

```cpp
#include <filesystem>
#include <print>

namespace fs = std::filesystem;

int main() {
    // Crée /tmp/mon_app — /tmp doit exister
    bool created = fs::create_directory("/tmp/mon_app");
    std::println("Créé : {}", created);

    // Si le répertoire existe déjà : retourne false, pas d'erreur
    bool again = fs::create_directory("/tmp/mon_app");
    std::println("Déjà existant, créé : {}", again);  // false

    // Si le parent n'existe pas : lance une exception
    try {
        fs::create_directory("/tmp/absent/sous_repertoire");
    } catch (const fs::filesystem_error& e) {
        std::println("Erreur : {}", e.what());
    }
}
```

La valeur de retour `bool` indique si le répertoire a effectivement été créé (`true`) ou s'il existait déjà (`false`). Ce n'est **pas** un indicateur d'erreur — les erreurs sont signalées par exception ou par `std::error_code`.

### `create_directories()` : arborescence complète

`fs::create_directories()` est l'équivalent de `mkdir -p` : elle crée tous les répertoires intermédiaires nécessaires :

```cpp
#include <filesystem>
#include <print>

namespace fs = std::filesystem;

int main() {
    // Crée toute la hiérarchie d'un coup
    bool created = fs::create_directories("/tmp/mon_app/build/release/x86_64");
    std::println("Créé : {}", created);

    // Idempotent : si tout existe déjà, retourne false sans erreur
    bool again = fs::create_directories("/tmp/mon_app/build/release/x86_64");
    std::println("Déjà existant : {}", again);  // false
}
```

`create_directories()` est idempotente : l'appeler plusieurs fois avec le même chemin ne produit pas d'erreur. C'est le choix par défaut pour la création de répertoires dans du code de production, car il ne nécessite pas de vérifier l'existence préalable de chaque niveau.

### Permissions par défaut

Les répertoires créés héritent des permissions par défaut déterminées par le `umask` du processus. Sur la plupart des systèmes Linux, le umask est `022`, ce qui donne des répertoires en `755` (`rwxr-xr-x`).

La surcharge à deux arguments de `create_directory()` accepte un **chemin existant** dont les permissions sont copiées (pas un `fs::perms` directement) :

```cpp
namespace fs = std::filesystem;

// Copie les permissions d'un répertoire existant
fs::create_directory("/tmp/nouveau", "/tmp/existant");
```

Cependant, le `umask` est appliqué par le noyau *après* la demande, ce qui peut masquer certains bits. Pour un contrôle précis des permissions, il est préférable de créer le répertoire puis d'appeler `fs::permissions()` explicitement (section 19.4) :

```cpp
namespace fs = std::filesystem;

fs::create_directory("/tmp/secret");  
fs::permissions("/tmp/secret", fs::perms::owner_all, fs::perm_options::replace);  
// Résultat : rwx------ (700)
```

## Copie de fichiers

### `copy_file()` : copie d'un fichier unique

```cpp
#include <filesystem>
#include <print>

namespace fs = std::filesystem;

int main() {
    fs::path source = "/etc/hostname";
    fs::path dest = "/tmp/hostname_backup";

    // Copie simple — échoue si la destination existe déjà
    bool copied = fs::copy_file(source, dest);
    std::println("Copié : {}", copied);

    // Deuxième appel : échoue car dest existe
    std::error_code ec;
    fs::copy_file(source, dest, ec);
    if (ec) {
        std::println("Erreur attendue : {}", ec.message());
    }
}
```

### `copy_options` : contrôle fin du comportement

Le troisième paramètre de `copy_file()` est un bitmask `fs::copy_options` qui détermine le comportement en cas de conflit :

```cpp
#include <filesystem>
#include <print>

namespace fs = std::filesystem;

int main() {
    fs::path source = "/etc/hostname";
    fs::path dest = "/tmp/hostname_backup";

    // Écraser si la destination existe
    fs::copy_file(source, dest, fs::copy_options::overwrite_existing);

    // Écraser uniquement si la source est plus récente
    fs::copy_file(source, dest, fs::copy_options::update_existing);

    // Ne rien faire si la destination existe (pas d'erreur)
    bool copied = fs::copy_file(source, dest, fs::copy_options::skip_existing);
    std::println("Copié (skip) : {}", copied);  // false si dest existait
}
```

Les trois options de conflit s'excluent mutuellement :

| Option | Comportement si dest existe |
|---|---|
| *(aucune, défaut)* | Erreur |
| `overwrite_existing` | Écrase inconditionnellement |
| `update_existing` | Écrase si la source est plus récente |
| `skip_existing` | Ne fait rien, retourne `false` |

### `copy()` : copie générique (fichiers et répertoires)

`fs::copy()` est la fonction générique qui gère à la fois les fichiers et les répertoires. Son comportement dépend du type de la source et des options fournies :

```cpp
#include <filesystem>
#include <print>

namespace fs = std::filesystem;

int main() {
    // Copie d'un fichier unique (équivalent à copy_file)
    fs::copy("/etc/hostname", "/tmp/hostname_copy");

    // Copie d'un répertoire (contenu de premier niveau uniquement)
    fs::create_directories("/tmp/backup");
    fs::copy("/etc/nginx", "/tmp/backup/nginx");

    // Copie récursive d'un répertoire entier
    fs::copy("/home/user/projet/src", "/tmp/backup/src",
        fs::copy_options::recursive);

    // Copie récursive avec écrasement
    fs::copy("/home/user/projet/src", "/tmp/backup/src",
        fs::copy_options::recursive | fs::copy_options::overwrite_existing);
}
```

### Options supplémentaires de `copy()`

Au-delà des options de conflit, `fs::copy_options` contrôle le traitement des liens symboliques et le filtrage par type :

```cpp
namespace fs = std::filesystem;

// Copier les symlinks tels quels (au lieu de copier les fichiers cibles)
fs::copy("src_dir", "dst_dir",
    fs::copy_options::recursive | fs::copy_options::copy_symlinks);

// Ne copier que la structure de répertoires, pas les fichiers
fs::copy("src_dir", "dst_dir",
    fs::copy_options::recursive | fs::copy_options::directories_only);

// Créer des symlinks dans la destination au lieu de copier
fs::copy("src_dir", "dst_dir",
    fs::copy_options::recursive | fs::copy_options::create_symlinks);

// Créer des hard links au lieu de copier
fs::copy("src_dir", "dst_dir",
    fs::copy_options::recursive | fs::copy_options::create_hard_links);
```

L'option `create_hard_links` est particulièrement intéressante pour les systèmes de build : elle permet de "copier" une arborescence quasi instantanément en créant des hard links vers les fichiers originaux. Les données ne sont pas dupliquées sur le disque, et chaque lien partage le même inode. La modification de l'un n'affecte pas l'autre uniquement si le programme ouvre le fichier en mode troncation et réécriture (ce qui crée un nouvel inode), ce qui est le comportement normal des compilateurs.

## Déplacement et renommage

`fs::rename()` assure à la fois le renommage et le déplacement de fichiers et répertoires :

```cpp
#include <filesystem>
#include <print>

namespace fs = std::filesystem;

int main() {
    // Renommer un fichier dans le même répertoire
    fs::rename("/tmp/ancien.txt", "/tmp/nouveau.txt");

    // Déplacer un fichier vers un autre répertoire
    fs::rename("/tmp/nouveau.txt", "/var/archive/nouveau.txt");

    // Renommer un répertoire
    fs::rename("/tmp/build_old", "/tmp/build_backup");
}
```

### Comportement et contraintes

`fs::rename()` encapsule l'appel système `rename(2)`, dont le comportement sur Linux mérite attention :

**Atomicité** — Le renommage est **atomique** au sein d'un même système de fichiers. À aucun moment le fichier n'est dans un état intermédiaire : soit l'ancien nom est valide, soit le nouveau, jamais ni l'un ni l'autre. C'est une propriété fondamentale pour l'écriture sûre de fichiers (voir le pattern d'écriture atomique plus loin dans cette section).

**Cross-filesystem** — Si la source et la destination sont sur des systèmes de fichiers différents (par exemple `/tmp` sur tmpfs et `/home` sur ext4), `rename()` échoue avec `EXDEV`. Contrairement à ce qu'on pourrait attendre, `fs::rename()` ne tente pas automatiquement un copy+delete. Si le déplacement cross-filesystem est nécessaire, il faut le gérer explicitement :

```cpp
#include <filesystem>
#include <print>

namespace fs = std::filesystem;

void move_cross_filesystem(const fs::path& source, const fs::path& dest) {
    std::error_code ec;
    fs::rename(source, dest, ec);

    if (ec) {
        // Probable cross-filesystem — fallback sur copy + remove
        fs::copy(source, dest, fs::copy_options::recursive
                              | fs::copy_options::overwrite_existing);
        fs::remove_all(source);
        std::println("Déplacé (via copie) : {} → {}", source.string(), dest.string());
    } else {
        std::println("Déplacé (atomique)  : {} → {}", source.string(), dest.string());
    }
}
```

**Écrasement silencieux** — Si la destination existe et est un fichier, `rename()` l'écrase **sans avertissement**. Si la destination est un répertoire non vide, l'opération échoue. Ce comportement est hérité de POSIX et ne peut pas être contrôlé via des options (contrairement à `copy_file()`).

## Suppression

### `remove()` : suppression unitaire

```cpp
#include <filesystem>
#include <print>

namespace fs = std::filesystem;

int main() {
    // Supprimer un fichier
    bool removed = fs::remove("/tmp/fichier_temp.txt");
    std::println("Supprimé : {}", removed);  // true si le fichier existait

    // Supprimer un répertoire vide
    fs::remove("/tmp/repertoire_vide");

    // Supprimer un répertoire NON vide : échoue
    std::error_code ec;
    fs::remove("/tmp/repertoire_avec_contenu", ec);
    if (ec) {
        std::println("Erreur : {}", ec.message());
        // "Directory not empty"
    }

    // Si le chemin n'existe pas : retourne false, pas d'erreur
    bool absent = fs::remove("/tmp/inexistant");
    std::println("Inexistant, supprimé : {}", absent);  // false
}
```

### `remove_all()` : suppression récursive

```cpp
#include <filesystem>
#include <print>
#include <cstdint>

namespace fs = std::filesystem;

int main() {
    // Supprime le répertoire et tout son contenu
    std::uintmax_t count = fs::remove_all("/tmp/mon_app/build");
    std::println("{} éléments supprimés", count);

    // Si le chemin n'existe pas : retourne 0
    auto zero = fs::remove_all("/tmp/inexistant");
    std::println("Inexistant : {} éléments", zero);  // 0
}
```

`remove_all()` retourne le nombre total d'éléments supprimés (fichiers + répertoires). C'est l'équivalent de `rm -rf`.

### Précautions avec `remove_all()`

`remove_all()` est une opération **irréversible** et potentiellement destructrice. Quelques précautions essentielles :

```cpp
namespace fs = std::filesystem;

// ❌ DANGEREUX — un chemin vide ou mal construit peut avoir des conséquences graves
void cleanup(const fs::path& dir) {
    fs::remove_all(dir);  // Et si dir == "/" ou "/home" par erreur ?
}

// ✅ Vérifications défensives avant suppression récursive
void safe_cleanup(const fs::path& dir) {
    // Vérifier que le chemin n'est pas vide
    if (dir.empty()) {
        std::println("Erreur : chemin vide, suppression annulée");
        return;
    }

    // Vérifier que ce n'est pas un chemin sensible
    auto canonical = fs::weakly_canonical(dir);
    if (canonical == "/" || canonical == "/home" || canonical == "/etc") {
        std::println("Erreur : tentative de suppression d'un répertoire système");
        return;
    }

    // Vérifier que le répertoire est bien sous notre arborescence de travail
    auto expected_parent = fs::weakly_canonical("/tmp/mon_app");
    auto relative = canonical.lexically_relative(expected_parent);
    if (relative.empty() || relative.string().starts_with("..")) {
        std::println("Erreur : {} n'est pas sous {}", 
            canonical.string(), expected_parent.string());
        return;
    }

    auto count = fs::remove_all(dir);
    std::println("{} éléments supprimés dans {}", count, dir.string());
}
```

Ce pattern de vérification n'est pas de la paranoïa : dans un pipeline CI/CD ou un outil CLI, les chemins proviennent souvent de variables d'environnement, de fichiers de configuration ou d'arguments utilisateur. Une variable non définie qui produit un chemin vide ou `/` passé à `remove_all()` est un scénario de production réel.

## Liens symboliques et hard links

### Création de liens symboliques

```cpp
#include <filesystem>
#include <print>

namespace fs = std::filesystem;

int main() {
    // Lien symbolique vers un fichier
    fs::create_symlink("/usr/bin/python3.12", "/tmp/python");

    // Lien symbolique vers un répertoire
    fs::create_directory_symlink("/var/log", "/tmp/logs");

    // Lire la cible d'un symlink
    fs::path cible = fs::read_symlink("/tmp/python");
    std::println("python -> {}", cible.string());
}
```

Sur Linux, `create_symlink()` et `create_directory_symlink()` produisent le même résultat (un lien symbolique via `symlink(2)`). La distinction existe pour la portabilité Windows, où les symlinks vers fichiers et répertoires sont techniquement différents. Par clarté sémantique, il est tout de même recommandé d'utiliser `create_directory_symlink()` pour les liens vers des répertoires.

### Création de hard links

```cpp
#include <filesystem>
#include <print>

namespace fs = std::filesystem;

int main() {
    // Créer un hard link
    fs::create_hard_link("/tmp/original.txt", "/tmp/hardlink.txt");

    // Les deux chemins pointent vers le même inode
    std::println("Équivalents : {}", fs::equivalent("/tmp/original.txt", 
                                                      "/tmp/hardlink.txt"));
    // true

    // Le compteur de liens est incrémenté
    std::println("Hard links : {}", fs::hard_link_count("/tmp/original.txt"));
    // 2
}
```

Les hard links ne peuvent pas traverser les frontières de systèmes de fichiers et ne peuvent pas pointer vers des répertoires (restriction Linux). Ils sont utiles pour les systèmes de cache et de déduplication.

## Modification des métadonnées

### Taille de fichier

```cpp
#include <filesystem>
#include <print>

namespace fs = std::filesystem;

int main() {
    auto taille = fs::file_size("/usr/bin/bash");
    std::println("bash : {} octets ({:.1f} Ko)", taille,
        static_cast<double>(taille) / 1024);

    // Tronquer un fichier à une taille donnée
    fs::resize_file("/tmp/test.log", 1024);  // Tronque ou étend à 1 Ko
    fs::resize_file("/tmp/test.log", 0);     // Vide le fichier
}
```

`fs::resize_file()` fonctionne comme `truncate(2)` : si la nouvelle taille est inférieure, le fichier est tronqué ; si elle est supérieure, le fichier est étendu avec des octets nuls. C'est utile pour la rotation de logs ou la pré-allocation de fichiers.

### Date de dernière modification

```cpp
#include <filesystem>
#include <print>
#include <chrono>

namespace fs = std::filesystem;

int main() {
    fs::path p = "/tmp/test.txt";

    // Lire la date de dernière modification
    auto ftime = fs::last_write_time(p);

    // Modifier la date (touch)
    auto now = fs::file_time_type::clock::now();
    fs::last_write_time(p, now);

    // Copier la date d'un fichier vers un autre
    auto source_time = fs::last_write_time("/etc/hostname");
    fs::last_write_time(p, source_time);
}
```

### Conversion de `file_time_type` vers `system_clock`

La conversion entre `fs::file_time_type` et `std::chrono::system_clock` a été simplifiée en **C++20** grâce à l'introduction de `std::chrono::file_clock` :

```cpp
#include <filesystem>
#include <print>
#include <chrono>

namespace fs = std::filesystem;

int main() {
    auto ftime = fs::last_write_time("/etc/hostname");

    // C++20 : conversion via clock_cast
    auto sys_time = std::chrono::clock_cast<std::chrono::system_clock>(ftime);

    // Affichage avec std::print (C++23 supporte les chrono time_points)
    std::println("Dernière modification : {}", sys_time);
}
```

## Pattern d'écriture atomique (write-then-rename)

L'un des patterns les plus importants en programmation système est l'**écriture atomique**. Le problème : si un programme écrit directement dans un fichier et qu'un crash ou une coupure de courant survient en cours d'écriture, le fichier se retrouve dans un état corrompu — ni l'ancienne version ni la nouvelle ne sont intactes.

La solution repose sur la propriété d'atomicité de `rename()` sur Linux :

```cpp
#include <filesystem>
#include <fstream>
#include <print>

namespace fs = std::filesystem;

void atomic_write(const fs::path& target, const std::string& content) {
    // 1. Écrire dans un fichier temporaire dans le MÊME répertoire
    //    (nécessaire pour que rename() soit atomique)
    fs::path tmp = target;
    tmp += ".tmp";

    {
        std::ofstream out(tmp, std::ios::binary | std::ios::trunc);
        if (!out) {
            throw std::runtime_error("Impossible de créer " + tmp.string());
        }
        out.write(content.data(), static_cast<std::streamsize>(content.size()));
        out.flush();

        // 2. Forcer l'écriture sur le disque (pas seulement dans le buffer OS)
        // std::ofstream ne fournit pas fsync — on peut utiliser sync() 
        // ou passer par les API POSIX pour un fsync ciblé (section 19.2)
    }
    // Le destructeur de ofstream ferme le fichier (RAII)

    // 3. Renommage atomique : remplace la cible d'un seul coup
    fs::rename(tmp, target);
}

int main() {
    fs::path config = "/tmp/mon_app/config.yaml";
    fs::create_directories(config.parent_path());

    atomic_write(config, "server:\n  port: 8080\n  host: localhost\n");
    std::println("Configuration écrite de manière atomique");
}
```

Ce pattern garantit qu'à tout instant, le fichier cible est soit l'ancienne version complète, soit la nouvelle version complète, jamais un état intermédiaire. Il est utilisé par de nombreux outils systèmes : `systemd`, `apt`, les gestionnaires de configuration, les bases de données embarquées.

Les points clés du pattern :

- Le fichier temporaire doit être dans le **même système de fichiers** que la cible (sinon `rename()` échoue avec `EXDEV`).
- Le `flush()` force le transfert des données du buffer C++ vers le noyau.
- Pour une garantie de durabilité complète (survie à une coupure de courant), un `fsync()` POSIX sur le descripteur est nécessaire avant le `rename()`. Ce point sera approfondi en section 19.2.
- Le `rename()` écrase atomiquement le fichier cible s'il existe.

## Gestion des fichiers temporaires

`std::filesystem` ne fournit pas de fonction dédiée à la création de fichiers temporaires avec un nom unique (contrairement à `mkstemp()` en POSIX). Voici les approches courantes :

### Avec `temp_directory_path()` et un identifiant unique

```cpp
#include <filesystem>
#include <fstream>
#include <print>
#include <format>
#include <chrono>
#include <unistd.h>

namespace fs = std::filesystem;

auto make_temp_path(const std::string& prefix, const std::string& suffix = "")
    -> fs::path
{
    auto now = std::chrono::steady_clock::now().time_since_epoch().count();
    auto pid = getpid();
    auto name = std::format("{}_{}_{}{}",prefix, pid, now, suffix);
    return fs::temp_directory_path() / name;
}

int main() {
    auto tmp = make_temp_path("mon_app", ".dat");
    std::println("Fichier temporaire : {}", tmp.string());

    // Créer et utiliser le fichier
    {
        std::ofstream out(tmp);
        out << "données temporaires\n";
    }

    // Nettoyage
    fs::remove(tmp);
}
```

### Répertoire temporaire RAII

Un pattern plus robuste encapsule la création et la suppression d'un répertoire temporaire dans une classe RAII :

```cpp
#include <filesystem>
#include <print>
#include <format>
#include <unistd.h>
#include <chrono>

namespace fs = std::filesystem;

class TempDirectory {  
public:  
    explicit TempDirectory(const std::string& prefix = "tmp") {
        auto now = std::chrono::steady_clock::now().time_since_epoch().count();
        auto name = std::format("{}_{}_{}",prefix, getpid(), now);
        path_ = fs::temp_directory_path() / name;
        fs::create_directories(path_);
    }

    ~TempDirectory() {
        std::error_code ec;
        fs::remove_all(path_, ec);
        // Ignorer les erreurs de nettoyage dans le destructeur
    }

    // Non copiable, déplaçable
    TempDirectory(const TempDirectory&) = delete;
    TempDirectory& operator=(const TempDirectory&) = delete;
    TempDirectory(TempDirectory&& other) noexcept : path_(std::move(other.path_)) {
        other.path_.clear();
    }
    TempDirectory& operator=(TempDirectory&& other) noexcept {
        if (this != &other) {
            cleanup();
            path_ = std::move(other.path_);
            other.path_.clear();
        }
        return *this;
    }

    [[nodiscard]] const fs::path& path() const noexcept { return path_; }

    // Accès rapide pour construire des chemins enfants
    [[nodiscard]] fs::path operator/(const fs::path& child) const {
        return path_ / child;
    }

private:
    void cleanup() noexcept {
        if (!path_.empty()) {
            std::error_code ec;
            fs::remove_all(path_, ec);
        }
    }

    fs::path path_;
};

int main() {
    {
        TempDirectory tmp("build_cache");
        std::println("Répertoire temporaire : {}", tmp.path().string());

        // Utiliser le répertoire
        fs::create_directory(tmp / "objects");
        fs::create_directory(tmp / "deps");

        // ... travail ...

    }  // Nettoyage automatique ici — RAII

    std::println("Répertoire temporaire supprimé");
}
```

Ce pattern est un exemple direct du principe RAII (section 6.3) : la ressource (le répertoire temporaire) est acquise à la construction et libérée à la destruction, même en cas d'exception. La classe est non copiable mais déplaçable, ce qui évite les doubles suppressions tout en permettant le transfert de propriété.

## Espace disque

`fs::space()` retourne les informations d'espace disque pour le système de fichiers contenant le chemin donné :

```cpp
#include <filesystem>
#include <print>

namespace fs = std::filesystem;

void print_space_info(const fs::path& p) {
    std::error_code ec;
    auto info = fs::space(p, ec);

    if (ec) {
        std::println("Erreur pour {} : {}", p.string(), ec.message());
        return;
    }

    constexpr auto Go = 1024.0 * 1024.0 * 1024.0;
    std::println("Filesystem contenant {} :", p.string());
    std::println("  Capacité   : {:.2f} Go", info.capacity / Go);
    std::println("  Libre      : {:.2f} Go", info.free / Go);
    std::println("  Disponible : {:.2f} Go", info.available / Go);
    std::println("  Utilisation : {:.1f}%",
        100.0 * (1.0 - static_cast<double>(info.available)
                      / static_cast<double>(info.capacity)));
}

int main() {
    print_space_info("/");
    print_space_info("/home");
    print_space_info("/tmp");
}
```

La distinction entre `free` et `available` reflète la réserve de blocs ext4 : `free` inclut les blocs réservés au super-utilisateur (typiquement 5 % du disque), tandis que `available` indique l'espace réellement utilisable par un utilisateur non-root. Pour vérifier si une opération d'écriture risque de manquer d'espace, c'est `available` qu'il faut consulter.

> 💡 **Note** — Cette section couvre les opérations "destructrices" du filesystem : suppression, écrasement, déplacement. En production, chaque opération irréversible mérite une vérification défensive. Le pattern le plus fiable est de toujours valider les chemins avant d'agir : vérifier qu'ils ne sont pas vides, qu'ils pointent vers l'arborescence attendue, et que le type de l'entrée (fichier, répertoire, symlink) correspond à ce qu'on attend. La combinaison de `weakly_canonical()` pour normaliser, de `lexically_relative()` pour valider l'appartenance à un sous-arbre, et de `error_code` pour gérer les échecs sans exception forme un trio défensif robuste que vous retrouverez dans tout code système de qualité production.
