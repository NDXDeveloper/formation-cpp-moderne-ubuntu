# 19.1.1 — Parcours de répertoires

## Module 7 — Programmation Système sur Linux *(Niveau Avancé)*

---

## Introduction

Parcourir une arborescence de fichiers est l'une des opérations les plus fréquentes en programmation système : lister les fichiers d'un répertoire de logs, rechercher des artefacts de build, inventorier des fichiers de configuration, nettoyer des fichiers temporaires. Avant C++17, cette tâche nécessitait l'API POSIX `opendir`/`readdir`/`closedir` ou des bibliothèques tierces. `std::filesystem` offre désormais deux itérateurs standard pour cela : `directory_iterator` pour un parcours à plat (un seul niveau) et `recursive_directory_iterator` pour un parcours récursif de l'arborescence complète.

---

## `directory_iterator` : parcours à un seul niveau

`fs::directory_iterator` parcourt les entrées directes d'un répertoire donné, sans descendre dans les sous-répertoires. Il se comporte comme un itérateur d'entrée (input iterator) et s'utilise naturellement dans un range-based `for` :

```cpp
#include <filesystem>
#include <print>

namespace fs = std::filesystem;

int main() {
    fs::path dir = "/etc/nginx";

    for (const fs::directory_entry& entry : fs::directory_iterator(dir)) {
        std::println("{}", entry.path().string());
    }
}
```

Sortie possible (l'ordre n'est **pas garanti**) :

```
/etc/nginx/nginx.conf
/etc/nginx/mime.types
/etc/nginx/conf.d
/etc/nginx/sites-available
/etc/nginx/sites-enabled
/etc/nginx/modules-enabled
```

Un point essentiel : l'ordre de parcours est **défini par l'implémentation** et dépend du système de fichiers sous-jacent. Sur ext4, c'est généralement l'ordre des entrées dans l'inode du répertoire, qui ne correspond ni à l'ordre alphabétique ni à l'ordre chronologique. Si vous avez besoin d'un ordre particulier, il faut collecter les entrées dans un conteneur puis les trier explicitement.

### Tri des résultats

```cpp
#include <filesystem>
#include <print>
#include <vector>
#include <algorithm>

namespace fs = std::filesystem;

int main() {
    fs::path dir = "/var/log";
    std::vector<fs::directory_entry> entries;

    for (const auto& entry : fs::directory_iterator(dir)) {
        entries.push_back(entry);
    }

    // Tri alphabétique par nom de fichier
    std::ranges::sort(entries, [](const auto& a, const auto& b) {
        return a.path().filename() < b.path().filename();
    });

    for (const auto& entry : entries) {
        std::println("{}", entry.path().filename().string());
    }
}
```

## `directory_entry` : le cache intelligent

Chaque élément produit par l'itérateur est un objet `fs::directory_entry`. Ce n'est pas un simple chemin : c'est un objet qui **met en cache** les résultats des appels système. C'est un détail d'implémentation qui a un impact direct sur la performance.

Lorsque le système d'exploitation le supporte (ce qui est le cas de Linux avec `getdents64`), l'itérateur récupère certaines métadonnées au moment du parcours lui-même, sans appel `stat()` supplémentaire. Le `directory_entry` stocke ces informations et les expose via ses méthodes :

```cpp
#include <filesystem>
#include <print>

namespace fs = std::filesystem;

int main() {
    fs::path dir = "/var/log";

    for (const auto& entry : fs::directory_iterator(dir)) {
        // Ces appels exploitent le cache du directory_entry
        // Pas d'appel stat() supplémentaire pour le type
        if (entry.is_regular_file()) {
            std::println("{} — {} octets",
                entry.path().filename().string(),
                entry.file_size());
        } else if (entry.is_directory()) {
            std::println("{}/", entry.path().filename().string());
        } else if (entry.is_symlink()) {
            std::println("{} -> (lien)", entry.path().filename().string());
        }
    }
}
```

### Comparaison : `entry.is_regular_file()` vs `fs::is_regular_file(entry.path())`

```cpp
// ✅ Utilise le cache du directory_entry — rapide
entry.is_regular_file();

// ❌ Appelle stat() sur le chemin — appel système supplémentaire
fs::is_regular_file(entry.path());
```

La différence semble mineure, mais lors du parcours d'un répertoire contenant des milliers d'entrées (un répertoire de cache, un arbre de build, `/proc`), les appels `stat()` redondants s'accumulent. La règle est simple : quand vous disposez d'un `directory_entry`, utilisez toujours ses méthodes plutôt que les fonctions libres sur `entry.path()`.

Les méthodes disponibles sur `directory_entry` sont : `exists()`, `is_regular_file()`, `is_directory()`, `is_symlink()`, `is_block_file()`, `is_character_file()`, `is_fifo()`, `is_socket()`, `is_other()`, `file_size()`, `hard_link_count()`, `last_write_time()` et `status()` / `symlink_status()`.

### Rafraîchir le cache

Si le filesystem a pu être modifié entre le moment du parcours et l'utilisation du `directory_entry`, la méthode `refresh()` force un nouvel appel système pour mettre à jour le cache :

```cpp
entry.refresh();  // Force un stat() pour actualiser les métadonnées
```

En pratique, cela est rarement nécessaire dans une boucle de parcours synchrone. C'est surtout utile dans les scénarios où un `directory_entry` est stocké et réutilisé ultérieurement.

## `recursive_directory_iterator` : parcours en profondeur

`fs::recursive_directory_iterator` parcourt l'intégralité d'une arborescence en descendant automatiquement dans chaque sous-répertoire :

```cpp
#include <filesystem>
#include <print>

namespace fs = std::filesystem;

int main() {
    fs::path racine = "/home/user/projet";

    auto it = fs::recursive_directory_iterator(racine);
    for (const auto& entry : it) {
        // depth() est une méthode de l'itérateur, pas du directory_entry
        std::string indent(it.depth() * 2, ' ');
        std::println("{}{}", indent, entry.path().filename().string());
    }
}
```

Sortie possible :

```
src
  main.cpp
  utils.cpp
  utils.h
include
  projet
    config.h
build
  CMakeCache.txt
  Makefile
CMakeLists.txt  
README.md  
```

### Contrôle de la profondeur

`recursive_directory_iterator` offre des méthodes pour contrôler la descente dans l'arborescence :

```cpp
#include <filesystem>
#include <print>

namespace fs = std::filesystem;

int main() {
    fs::path racine = "/home/user/projet";

    auto it = fs::recursive_directory_iterator(racine);
    for (auto& entry : it) {
        std::println("[profondeur {}] {}", it.depth(), entry.path().string());

        // Ignorer le contenu du répertoire "build"
        if (entry.is_directory() && entry.path().filename() == "build") {
            it.disable_recursion_pending();
        }
    }
}
```

`disable_recursion_pending()` empêche l'itérateur de descendre dans le répertoire courant lors de la prochaine incrémentation. L'appel ne prend effet que pour le répertoire en cours de traitement et se réinitialise automatiquement à l'itération suivante.

Il existe aussi `pop()`, qui remonte d'un niveau dans l'arborescence, quittant le sous-répertoire en cours pour reprendre le parcours au niveau parent :

```cpp
// Remonter d'un niveau si la profondeur dépasse 3
if (it.depth() > 3) {
    it.pop();
}
```

## `directory_options` : contrôle du comportement

Le constructeur des deux itérateurs accepte un paramètre optionnel `fs::directory_options` qui contrôle le comportement face aux situations particulières :

```cpp
namespace fs = std::filesystem;

// Suivre les liens symboliques vers des répertoires (par défaut, ils sont ignorés
// dans le parcours récursif pour éviter les boucles infinies)
auto it1 = fs::recursive_directory_iterator(
    "/home/user",
    fs::directory_options::follow_directory_symlink
);

// Ignorer silencieusement les répertoires inaccessibles (permission denied)
// au lieu de lancer une exception
auto it2 = fs::recursive_directory_iterator(
    "/var",
    fs::directory_options::skip_permission_denied
);

// Les deux options combinées
auto it3 = fs::recursive_directory_iterator(
    "/home",
    fs::directory_options::follow_directory_symlink
    | fs::directory_options::skip_permission_denied
);
```

### `skip_permission_denied`

Cette option est particulièrement importante en pratique. Sans elle, un parcours de `/var` ou `/home` échoue dès qu'un sous-répertoire protégé est rencontré : l'itérateur lance une `fs::filesystem_error`. Avec `skip_permission_denied`, les répertoires inaccessibles sont silencieusement ignorés.

Pour un contrôle plus fin (par exemple, journaliser les répertoires ignorés), vous pouvez utiliser la surcharge avec `std::error_code` lors de l'incrémentation :

```cpp
#include <filesystem>
#include <print>
#include <system_error>

namespace fs = std::filesystem;

int main() {
    std::error_code ec;
    auto it = fs::recursive_directory_iterator("/var", ec);

    if (ec) {
        std::println("Erreur à l'ouverture : {}", ec.message());
        return 1;
    }

    for (; it != fs::recursive_directory_iterator(); it.increment(ec)) {
        if (ec) {
            std::println("Erreur sur {} : {}", it->path().string(), ec.message());
            ec.clear();
            continue;
        }
        std::println("{}", it->path().string());
    }
}
```

Cette approche remplace le range-based `for` par une boucle manuelle avec `increment(ec)`. Elle est plus verbeuse mais offre un contrôle total : vous pouvez journaliser les erreurs, les compter, ou décider au cas par cas de continuer ou d'arrêter.

### `follow_directory_symlink`

Par défaut, `recursive_directory_iterator` ne suit **pas** les liens symboliques pointant vers des répertoires. C'est un choix de sécurité délibéré : suivre les symlinks peut créer des boucles infinies si un lien pointe vers un ancêtre dans l'arborescence. Si vous activez cette option, assurez-vous que l'arborescence ne contient pas de cycles, ou implémentez une détection manuelle (par exemple, en maintenant un `std::unordered_set` des inodes déjà visités).

## Patterns courants

### Filtrer par extension

```cpp
#include <filesystem>
#include <print>
#include <vector>

namespace fs = std::filesystem;

auto find_by_extension(const fs::path& dir, const std::string& ext)
    -> std::vector<fs::path>
{
    std::vector<fs::path> results;

    for (const auto& entry : fs::recursive_directory_iterator(
            dir, fs::directory_options::skip_permission_denied))
    {
        if (entry.is_regular_file() && entry.path().extension() == ext) {
            results.push_back(entry.path());
        }
    }

    return results;
}

int main() {
    auto cpp_files = find_by_extension("/home/user/projet", ".cpp");

    std::println("Fichiers .cpp trouvés : {}", cpp_files.size());
    for (const auto& f : cpp_files) {
        std::println("  {}", f.string());
    }
}
```

### Calculer la taille totale d'un répertoire

```cpp
#include <filesystem>
#include <print>
#include <cstdint>

namespace fs = std::filesystem;

auto directory_size(const fs::path& dir) -> std::uintmax_t {
    std::uintmax_t total = 0;

    for (const auto& entry : fs::recursive_directory_iterator(
            dir, fs::directory_options::skip_permission_denied))
    {
        if (entry.is_regular_file() && !entry.is_symlink()) {
            total += entry.file_size();
        }
    }

    return total;
}

int main() {
    auto taille = directory_size("/home/user/projet");

    if (taille < 1024) {
        std::println("Taille totale : {} octets", taille);
    } else if (taille < 1024 * 1024) {
        std::println("Taille totale : {:.1f} Ko",
            static_cast<double>(taille) / 1024);
    } else {
        std::println("Taille totale : {:.1f} Mo",
            static_cast<double>(taille) / (1024 * 1024));
    }
}
```

Notez le test `!entry.is_symlink()` : sans cette vérification, un lien symbolique pointant vers un fichier régulier serait compté (car `is_regular_file()` suit les liens). Selon votre intention, vous pouvez inclure ou exclure les cibles des symlinks.

### Trouver les fichiers récemment modifiés

```cpp
#include <filesystem>
#include <print>
#include <chrono>
#include <vector>

namespace fs = std::filesystem;

auto find_recent_files(const fs::path& dir, std::chrono::hours max_age)
    -> std::vector<fs::directory_entry>
{
    std::vector<fs::directory_entry> results;
    auto now = fs::file_time_type::clock::now();

    for (const auto& entry : fs::recursive_directory_iterator(
            dir, fs::directory_options::skip_permission_denied))
    {
        if (entry.is_regular_file()) {
            auto age = now - entry.last_write_time();
            if (age <= max_age) {
                results.push_back(entry);
            }
        }
    }

    return results;
}

int main() {
    using namespace std::chrono_literals;

    // Fichiers modifiés dans les dernières 24 heures
    auto recents = find_recent_files("/home/user/projet", 24h);

    std::println("Fichiers modifiés dans les dernières 24h :");
    for (const auto& entry : recents) {
        std::println("  {} ({} octets)",
            entry.path().filename().string(),
            entry.file_size());
    }
}
```

### Exclure des répertoires du parcours récursif

Un besoin très fréquent dans les projets de développement : exclure les répertoires de build, les dépendances vendored et les répertoires de contrôle de version :

```cpp
#include <filesystem>
#include <print>
#include <unordered_set>

namespace fs = std::filesystem;

int main() {
    const std::unordered_set<std::string> exclusions = {
        "build", ".git", "node_modules", "third_party", "__pycache__"
    };

    fs::path projet = "/home/user/projet";
    auto it = fs::recursive_directory_iterator(
        projet, fs::directory_options::skip_permission_denied);

    for (auto& entry : it) {
        if (entry.is_directory()
            && exclusions.contains(entry.path().filename().string()))
        {
            it.disable_recursion_pending();
            continue;
        }

        if (entry.is_regular_file()) {
            std::println("{}", entry.path().lexically_relative(projet).string());
        }
    }
}
```

## Parcours et Ranges C++20

Les itérateurs de répertoire sont compatibles avec les algorithmes standard, mais ils ne modélisent qu'un **input_iterator** (parcours en une seule passe). Cela signifie qu'ils fonctionnent avec les range-based `for` et avec `std::ranges::for_each`, mais pas directement avec des algorithmes qui nécessitent un forward iterator ou supérieur (comme `std::ranges::sort`).

Pour utiliser les Ranges C++20 avec filtrage, la technique consiste à combiner l'itérateur avec des vues :

```cpp
#include <filesystem>
#include <print>
#include <ranges>

namespace fs = std::filesystem;

int main() {
    fs::path dir = "/home/user/projet/src";

    // Filtrer les fichiers .h avec une vue ranges
    auto headers = fs::directory_iterator(dir)
        | std::views::filter([](const auto& e) {
            return e.is_regular_file() && e.path().extension() == ".h";
        })
        | std::views::transform([](const auto& e) {
            return e.path().filename().string();
        });

    for (const auto& name : headers) {
        std::println("{}", name);
    }
}
```

Cette combinaison est élégante et efficace : le filtrage et la transformation sont appliqués de manière lazy, sans conteneur intermédiaire.

## Répertoire vide et itérateur de fin

L'itérateur de fin pour les deux types d'itérateurs est simplement un itérateur construit par défaut. Un répertoire vide produit un itérateur de début égal à l'itérateur de fin :

```cpp
namespace fs = std::filesystem;

fs::path dir = "/tmp/repertoire_vide";  
fs::create_directories(dir);  

auto it = fs::directory_iterator(dir);  
auto end = fs::directory_iterator();  // Itérateur de fin  

if (it == end) {
    std::println("Le répertoire est vide");
}

// Équivalent plus direct
if (fs::is_empty(dir)) {
    std::println("Le répertoire est vide");
}
```

## Résumé des itérateurs

| Caractéristique | `directory_iterator` | `recursive_directory_iterator` |
|---|---|---|
| **Profondeur** | 1 niveau | Arborescence complète |
| **Type d'itérateur** | Input iterator | Input iterator |
| **`depth()`** | Non disponible | Oui |
| **`disable_recursion_pending()`** | Non disponible | Oui |
| **`pop()`** | Non disponible | Oui |
| **`directory_options`** | `skip_permission_denied` | `skip_permission_denied`, `follow_directory_symlink` |
| **Ordre de parcours** | Non garanti | Non garanti (profondeur d'abord) |
| **Éléments `.` et `..`** | Exclus automatiquement | Exclus automatiquement |

Les entrées spéciales `.` (répertoire courant) et `..` (répertoire parent) ne sont **jamais** incluses dans le parcours. C'est un avantage notable par rapport à l'API POSIX `readdir`, où ces entrées doivent être filtrées manuellement.

---

> 💡 **Note** — Le parcours de répertoires est souvent l'un des premiers points où un développeur C++ mesure l'écart entre un prototype fonctionnel et un outil de production robuste. Un `for (const auto& entry : fs::recursive_directory_iterator(dir))` fonctionne parfaitement sur votre machine de développement, mais échoue en production face à un lien symbolique cyclique, un répertoire sans permission de lecture, ou un fichier supprimé entre le moment de l'itération et celui de l'accès. Les options `skip_permission_denied`, la boucle manuelle avec `increment(ec)` et les vérifications `is_symlink()` transforment un prototype fragile en code de production fiable.
