# 16.6.1 — Syntaxe `requires`

## Chapitre 16 : Templates et Métaprogrammation · Section 16.6 : Concepts (C++20)

---

## Introduction

Le mot-clé `requires` est le cœur syntaxique des Concepts C++20. Il intervient dans deux rôles distincts mais complémentaires :

- la **clause `requires`** — une condition booléenne attachée à un template, qui détermine si la surcharge ou la spécialisation est active ;
- l'**expression `requires`** — un bloc structuré qui teste la validité d'un ensemble d'expressions et de types pour un type donné.

Ces deux formes peuvent être combinées, y compris dans une même déclaration. Cette sous-section les examine en détail, en couvrant les quatre types d'exigences disponibles dans une expression `requires`, la composition de contraintes, et les pièges syntaxiques à éviter.

---

## Clause `requires` : activer ou désactiver un template

La clause `requires` est une condition booléenne placée dans la déclaration d'un template. Si elle vaut `false` pour un type donné, le template est écarté — comme le ferait `enable_if` avec SFINAE, mais avec un diagnostic clair.

### Position de la clause

La clause peut apparaître à deux endroits dans la déclaration d'une fonction template :

**Après la liste de paramètres template** (forme recommandée) :

```cpp
template <typename T>
    requires std::integral<T>
T doubler(T valeur) {
    return valeur * 2;
}
```

**Après la signature de la fonction** (trailing requires clause) :

```cpp
template <typename T>  
T doubler(T valeur) requires std::integral<T> {  
    return valeur * 2;
}
```

Les deux formes sont strictement équivalentes. La première est généralement préférée pour les fonctions libres car la contrainte est visible avant la signature. La seconde est utile dans les déclarations de méthodes de classes, où elle évite de surcharger le début de la déclaration :

```cpp
template <typename T>  
class Container {  
public:  
    void sort() requires std::totally_ordered<T>;
    
    void print() const requires std::formattable<T, char>;
};
```

Ici, les méthodes `sort` et `print` n'existent que si le type `T` satisfait respectivement `std::totally_ordered` et `std::formattable`. Tenter d'appeler `sort()` sur un `Container<Point>` où `Point` n'est pas ordonnançable produit un message d'erreur direct.

### La clause accepte toute expression booléenne constante

La clause `requires` n'est pas limitée aux Concepts. Elle accepte n'importe quelle expression `constexpr bool` :

```cpp
template <typename T>
    requires (sizeof(T) <= 8)
void traiter_petit(T val) {
    std::print("Type petit ({} octets)\n", sizeof(T));
}

template <typename T>
    requires std::is_trivially_copyable_v<T>
void copie_rapide(const T& src, T& dst) {
    std::memcpy(&dst, &src, sizeof(T));
}
```

Les traits de type de `<type_traits>` sont directement utilisables dans les clauses `requires`, ce qui simplifie la migration depuis le code SFINAE existant.

### Attention aux parenthèses

Lorsque la contrainte est une expression composée (pas un simple concept ou trait), des **parenthèses** sont nécessaires :

```cpp
// ERREUR de syntaxe : le compilateur parse mal l'expression
template <typename T>
    requires sizeof(T) <= 8      // Ambigu pour le parser
void f(T);

// CORRECT : parenthèses obligatoires autour de l'expression
template <typename T>
    requires (sizeof(T) <= 8)
void f(T);
```

La règle est simple : si l'expression après `requires` n'est pas un concept nommé, un trait `_v`, ou une combinaison de concepts avec `&&` / `||`, encadrez-la de parenthèses.

## Expression `requires` : tester la validité d'opérations

L'expression `requires` est un bloc entre accolades qui décrit un ensemble d'**exigences** (*requirements*) qu'un type doit satisfaire. Elle retourne `true` si toutes les exigences sont valides, `false` sinon. Contrairement à SFINAE, les exigences sont **explicites et déclaratives**.

### Syntaxe générale

```cpp
requires (liste_de_paramètres) {
    exigence_1;
    exigence_2;
    // ...
}
```

La liste de paramètres entre parenthèses déclare des « variables fictives » utilisées pour écrire les expressions à tester. Ces variables ne sont jamais instanciées ; elles servent uniquement à vérifier que les expressions seraient valides :

```cpp
template <typename T>  
concept Addable = requires(T a, T b) {  
    a + b;    // L'expression a + b doit être valide pour T
};
```

### Expression `requires` dans une clause `requires`

La combinaison la plus fréquente est d'utiliser une expression `requires` directement dans une clause `requires`. Cela produit la construction `requires requires`, parfois source de confusion :

```cpp
template <typename T>
    requires requires(T a, T b) { a + b; }
//  ^^^^^^^^ ^^^^^^^^
//  clause   expression
T additionner(T a, T b) {
    return a + b;
}
```

Le premier `requires` est la **clause** (condition sur le template). Le second `requires` introduit l'**expression** (bloc de test). C'est syntaxiquement correct mais peu lisible. En pratique, on extrait l'expression dans un **concept nommé** pour éviter cette répétition :

```cpp
template <typename T>  
concept Addable = requires(T a, T b) { a + b; };  

template <Addable T>  
T additionner(T a, T b) {  
    return a + b;
}
```

Cette forme est plus claire et le concept `Addable` est réutilisable.

## Les quatre types d'exigences

Une expression `requires` peut contenir quatre types d'exigences, chacune testant un aspect différent du type.

### 1. Simple requirements (exigences simples)

Une exigence simple vérifie qu'une **expression est valide** — qu'elle compile. Elle ne vérifie pas le type du résultat ni ne l'évalue à l'exécution :

```cpp
template <typename T>  
concept Incrementable = requires(T a) {  
    a++;        // L'expression a++ doit compiler
    ++a;        // L'expression ++a doit compiler
    a + 1;      // L'expression a + 1 doit compiler
};
```

Si `T = int`, toutes ces expressions sont valides → le concept est satisfait. Si `T = std::string`, `a + 1` n'est pas valide → le concept est rejeté.

On peut tester des appels de méthodes :

```cpp
template <typename T>  
concept HasSize = requires(T t) {  
    t.size();   // T doit avoir une méthode size() appelable
};
```

Et des expressions impliquant plusieurs types :

```cpp
template <typename T, typename U>  
concept AddableWith = requires(T a, U b) {  
    a + b;      // T + U doit être valide
    b + a;      // U + T doit aussi être valide
};
```

### 2. Type requirements (exigences de type)

Une exigence de type vérifie qu'un **nom de type est valide**. Elle utilise le mot-clé `typename` suivi du type à vérifier :

```cpp
template <typename T>  
concept HasValueType = requires {  
    typename T::value_type;       // T doit posséder un type membre value_type
};

template <typename T>  
concept HasIterator = requires {  
    typename T::iterator;          // T doit posséder un type membre iterator
    typename T::const_iterator;    // et un type membre const_iterator
};
```

```cpp
static_assert(HasValueType<std::vector<int>>);   // true — value_type = int  
static_assert(!HasValueType<int>);                // false — int n'a pas de value_type  
```

L'exigence de type est l'équivalent propre de ce qu'on faisait avec `std::void_t<typename T::value_type>` en SFINAE (section 16.4).

On peut aussi vérifier qu'une instanciation de template est valide :

```cpp
template <typename T>  
concept Hashable = requires {  
    typename std::hash<T>;         // std::hash<T> doit être un type valide
};
```

### 3. Compound requirements (exigences composées)

Une exigence composée teste simultanément **trois aspects** d'une expression :

1. L'expression est valide (compile).
2. Le résultat satisfait un concept donné (optionnel).
3. L'expression ne lance pas d'exception (optionnel).

La syntaxe utilise des accolades autour de l'expression, suivies d'un type de retour optionnel et/ou de `noexcept` :

```cpp
{ expression } -> Concept;          // Valide + type contraint
{ expression } noexcept;            // Valide + ne lance pas
{ expression } noexcept -> Concept; // Valide + ne lance pas + type contraint
```

Le `-> Concept` applique le concept au **type du résultat** de l'expression. Le type est automatiquement passé comme premier argument du concept.

```cpp
template <typename T>  
concept Comparable = requires(T a, T b) {  
    { a == b } -> std::convertible_to<bool>;  // a == b doit retourner qqch convertible en bool
    { a != b } -> std::convertible_to<bool>;  // idem pour !=
    { a < b }  -> std::convertible_to<bool>;  // idem pour <
};
```

Ici, `{ a == b } -> std::convertible_to<bool>` signifie : « l'expression `a == b` doit être valide, **et** le type de son résultat doit satisfaire `std::convertible_to<bool>` ». C'est plus précis qu'une exigence simple qui vérifierait seulement que `a == b` compile (un `operator==` retournant `void` passerait l'exigence simple, mais serait rejeté ici).

Un exemple avec `noexcept` :

```cpp
template <typename T>  
concept NothrowMovable = requires(T a) {  
    { T(std::move(a)) } noexcept;  // La construction par mouvement ne doit pas lancer
};
```

Et un exemple combinant contrainte de type et `noexcept` :

```cpp
template <typename T>  
concept SafeHash = requires(T val) {  
    { std::hash<T>{}(val) } noexcept -> std::same_as<std::size_t>;
};
```

Ce concept `SafeHash` exige que le hash de `T` soit à la fois non-throwing et qu'il retourne exactement `std::size_t`.

### 4. Nested requirements (exigences imbriquées)

Une exigence imbriquée évalue une **expression booléenne constante** à l'intérieur du bloc `requires`. Elle utilise un `requires` interne (sans accolades) :

```cpp
template <typename T>  
concept SmallTrivial = requires {  
    requires sizeof(T) <= 16;                       // Taille <= 16 octets
    requires std::is_trivially_copyable_v<T>;       // Trivialement copiable
};
```

Les exigences imbriquées sont utiles pour combiner des tests structurels (exigences simples/composées) avec des conditions numériques ou des traits :

```cpp
template <typename T>  
concept FixedSizeContainer = requires(T t) {  
    t.size();                                       // Doit avoir .size()
    t.begin();                                      // Doit avoir .begin()
    t.end();                                        // Doit avoir .end()
    typename T::value_type;                         // Doit avoir value_type
    requires sizeof(T) <= 256;                      // Taille en mémoire bornée
};
```

La différence entre une exigence imbriquée et une exigence simple est subtile mais importante. L'exigence simple `sizeof(T) <= 256;` vérifierait seulement que l'expression est **syntaxiquement valide** (elle l'est toujours pour tout type complet). L'exigence imbriquée `requires sizeof(T) <= 256;` vérifie que l'expression **s'évalue à `true`**. Sans le mot-clé `requires` interne, la condition de taille serait ignorée.

## Composer des contraintes

Les clauses `requires` supportent les opérateurs logiques `&&` (conjonction) et `||` (disjonction) pour combiner des concepts et des expressions :

### Conjonction (`&&`)

Toutes les contraintes doivent être satisfaites :

```cpp
template <typename T>
    requires std::integral<T> && std::signed_integral<T>
T valeur_absolue(T val) {
    return (val < 0) ? -val : val;
}
```

### Disjonction (`||`)

Au moins une contrainte doit être satisfaite :

```cpp
template <typename T>
    requires std::integral<T> || std::floating_point<T>
T clamper(T val, T min, T max) {
    if (val < min) return min;
    if (val > max) return max;
    return val;
}
```

### Composition dans les concepts

Les concepts eux-mêmes se composent naturellement :

```cpp
template <typename T>  
concept Printable = requires(T val) {  
    { std::print("{}", val) };
};

template <typename T>  
concept Numeric = std::integral<T> || std::floating_point<T>;  

template <typename T>  
concept PrintableNumeric = Printable<T> && Numeric<T>;  
```

Cette composabilité est l'un des avantages majeurs des Concepts sur SFINAE. Chaque concept élémentaire est simple, testable indépendamment, et la composition est lisible comme du langage naturel.

## Subsomption et ordonnancement des contraintes

Quand plusieurs surcharges contraintes sont applicables, le compilateur utilise la **subsomption** pour choisir la plus spécifique. Une contrainte `A` subsume une contrainte `B` si `A` implique logiquement `B`.

La subsomption ne fonctionne que lorsque les concepts sont **syntaxiquement identiques** dans les expressions de contrainte. C'est pourquoi il est important de réutiliser les mêmes concepts nommés :

```cpp
template <typename T>  
concept Animal = requires(T a) {  
    a.respirer();
    a.manger();
};

template <typename T>  
concept Oiseau = Animal<T> && requires(T a) {  
    a.voler();
};
```

`Oiseau` subsume `Animal` : tout `Oiseau` est un `Animal`, mais pas l'inverse. Le compilateur le sait parce que la définition de `Oiseau` inclut syntaxiquement `Animal<T>`.

```cpp
template <Animal T>  
void decrire(T a) {  
    std::print("Un animal\n");
}

template <Oiseau T>  
void decrire(T a) {  
    std::print("Un oiseau\n");
}
```

```cpp
struct Chien {
    void respirer() {}
    void manger() {}
};

struct Aigle {
    void respirer() {}
    void manger() {}
    void voler() {}
};

decrire(Chien{});   // "Un animal" — satisfait Animal, pas Oiseau  
decrire(Aigle{});   // "Un oiseau" — satisfait les deux, Oiseau est plus spécifique  
```

Pour `Aigle`, les deux surcharges sont applicables. Grâce à la subsomption, le compilateur préfère automatiquement la surcharge `Oiseau` (plus contrainte). Aucune ambiguïté.

### Piège : duplication au lieu de réutilisation

Si les contraintes sont dupliquées plutôt que composées via un concept nommé, la subsomption **ne fonctionne pas** :

```cpp
// MAUVAIS : contraintes dupliquées, pas de subsomption
template <typename T>
    requires requires(T a) { a.respirer(); a.manger(); }
void decrire(T a);

template <typename T>
    requires requires(T a) { a.respirer(); a.manger(); a.voler(); }
void decrire(T a);

// ERREUR : ambiguïté — le compilateur ne voit pas que la seconde subsume la première
```

Le compilateur ne peut établir la subsomption qu'en comparant des **contraintes atomiques identiques**. Deux expressions `requires` syntaxiquement similaires mais écrites indépendamment sont considérées comme des contraintes distinctes. Toujours passer par des concepts nommés pour permettre la subsomption.

## Contraindre les méthodes individuellement

Les clauses `requires` permettent de rendre certaines méthodes d'un template de classe conditionnellement disponibles, sans spécialiser la classe entière :

```cpp
template <typename T>  
class SmartContainer {  
public:  
    void push(const T& val) {
        data_.push_back(val);
    }

    // sort() n'existe que si T est ordonnançable
    void sort() requires std::totally_ordered<T> {
        std::sort(data_.begin(), data_.end());
    }

    // print_all() n'existe que si T est formatable
    void print_all() const requires std::formattable<T, char> {
        for (const auto& elem : data_) {
            std::print("{} ", elem);
        }
        std::print("\n");
    }

    // sum() n'existe que si T supporte l'addition et la construction depuis 0
    auto sum() const requires requires(T a, T b) {
        { a + b } -> std::convertible_to<T>;
        { T{0} };
    } {
        T total{0};
        for (const auto& elem : data_) {
            total = total + elem;
        }
        return total;
    }

    std::size_t size() const { return data_.size(); }

private:
    std::vector<T> data_;
};
```

```cpp
SmartContainer<int> ci;  
ci.push(3); ci.push(1); ci.push(2);  
ci.sort();                 // OK : int est totally_ordered  
ci.print_all();            // OK : int est formatable → "1 2 3"  
auto s = ci.sum();         // OK : int supporte + et {0} → 6  

struct Point { int x, y; };  
SmartContainer<Point> cp;  
cp.push(Point{1, 2});  
cp.size();                 // OK : size() est toujours disponible  
// cp.sort();              // ERREUR : Point ne satisfait pas totally_ordered
// cp.print_all();         // ERREUR : Point ne satisfait pas formattable
```

Ce pattern est le remplaçant moderne de l'instanciation paresseuse implicite (section 16.2). Les contraintes sont **visibles dans la déclaration** de chaque méthode, documentant explicitement les exigences.

## `requires` et `constexpr if` : quand utiliser lequel ?

Les deux mécanismes permettent d'adapter le comportement selon les propriétés d'un type, mais à des niveaux différents :

| Aspect | `requires` (clause/concept) | `if constexpr` |
|---|---|---|
| **Niveau d'action** | Signature — active/désactive la surcharge | Corps — sélectionne une branche |
| **Visibilité** | L'interface documente la contrainte | La contrainte est cachée dans le corps |
| **Participation à la surcharge** | Oui — le compilateur sélectionne la meilleure | Non — une seule fonction existe |
| **Messages d'erreur** | Le concept violé est nommé | Erreur dans la branche compilée |
| **Usage principal** | Interfaces publiques, API | Détails d'implémentation internes |

En règle générale, `requires` est préférable pour les **interfaces publiques** : l'utilisateur voit les contraintes dans la signature et obtient des messages d'erreur clairs. `if constexpr` est adapté aux **détails d'implémentation** internes, quand une seule fonction doit gérer plusieurs chemins de code sans exposer cette complexité.

Les deux se combinent naturellement :

```cpp
template <typename T>
    requires Numeric<T>  // Contrainte publique : T doit être numérique
std::string formater(T val) {
    if constexpr (std::integral<T>) {
        return std::format("{}", val);         // Entiers : pas de décimales
    } else {
        return std::format("{:.2f}", val);     // Flottants : 2 décimales
    }
}
```

## Résumé des formes syntaxiques

Voici un récapitulatif de toutes les formes de `requires` rencontrées dans cette sous-section :

```cpp
// 1. Clause requires avec un concept nommé
template <typename T>
    requires std::integral<T>
void f(T);

// 2. Clause requires avec une expression booléenne
template <typename T>
    requires (sizeof(T) <= 8)
void f(T);

// 3. Clause requires avec une expression requires
template <typename T>
    requires requires(T a) { a.size(); }
void f(T);

// 4. Trailing requires clause (sur une méthode)
void sort() requires std::totally_ordered<T>;

// 5. Expression requires dans une définition de concept
template <typename T>  
concept C = requires(T a, T b) {  
    a + b;                                  // Simple requirement
    typename T::value_type;                 // Type requirement
    { a * b } -> std::same_as<T>;          // Compound requirement
    requires sizeof(T) <= 32;               // Nested requirement
};

// 6. Composition de contraintes
template <typename T>
    requires std::integral<T> && (sizeof(T) >= 4)
void f(T);

// 7. Concept en position de paramètre template
template <std::integral T>  
void f(T);  

// 8. auto contraint
void f(std::integral auto val);
```

Les formes 7 et 8 sont des raccourcis syntaxiques pour la forme 1. Toutes participent au même mécanisme de contraintes et de subsomption.

## Bonnes pratiques

**Nommez vos concepts plutôt que d'utiliser `requires requires`.** La forme `requires requires(T a) { ... }` est rarement justifiée en code de production. Extrayez la contrainte dans un concept nommé : c'est plus lisible, réutilisable et compatible avec la subsomption.

**Utilisez les exigences composées quand le type de retour compte.** Une exigence simple `a + b;` vérifie seulement que l'expression compile. Si le type du résultat est important, utilisez `{ a + b } -> std::convertible_to<T>;`.

**Distinguez exigence simple et exigence imbriquée.** `sizeof(T) <= 16;` dans un bloc `requires` est une exigence simple — elle vérifie que l'expression est syntaxiquement valide (toujours vrai). `requires sizeof(T) <= 16;` est une exigence imbriquée — elle vérifie que l'expression s'évalue à `true`. Confondre les deux est un bug silencieux.

**Parenthésez les expressions complexes.** Après le mot-clé `requires` en position de clause, les expressions composées avec `sizeof`, `alignof` ou d'autres opérateurs nécessitent des parenthèses pour être parsées correctement.

**Composez par concepts nommés pour la subsomption.** Définissez des concepts hiérarchiques (`Animal` → `Oiseau`) en réutilisant le concept parent dans la définition du concept enfant. C'est la seule façon de garantir que le compilateur détecte la relation de subsomption.

**Préférez les concepts standard quand ils existent.** Avant de créer un concept personnalisé, vérifiez si `<concepts>`, `<ranges>` ou `<iterator>` ne fournissent pas déjà ce dont vous avez besoin. Les concepts standard sont testés, documentés et reconnus par la subsomption du compilateur. La sous-section 16.6.2 en dresse l'inventaire.
