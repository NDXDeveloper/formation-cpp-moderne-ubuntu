# 16.6.3 — Création de concepts personnalisés

## Chapitre 16 : Templates et Métaprogrammation · Section 16.6 : Concepts (C++20)

---

## Introduction

Les deux sous-sections précédentes ont posé les fondations : la syntaxe `requires` (16.6.1) et le vocabulaire des concepts standard (16.6.2). Mais les concepts standard décrivent des propriétés **génériques** du langage — types numériques, itérateurs, comparabilité. Votre code, lui, opère dans un **domaine métier** spécifique : sérialisation, réseau, calcul scientifique, moteur de jeu, outils DevOps.

Les concepts personnalisés comblent cet écart. Ils permettent de nommer et d'encoder les exigences propres à votre architecture, de sorte que vos templates deviennent auto-documentés, que vos messages d'erreur parlent le langage de votre domaine, et que vos contraintes soient réutilisables et testables indépendamment.

Cette sous-section couvre la conception, l'implémentation et l'organisation de concepts personnalisés — depuis les patterns de composition les plus simples jusqu'aux hiérarchies élaborées intégrées dans une architecture de projet.

---

## Anatomie d'un concept personnalisé

Un concept personnalisé suit toujours le même schéma :

```cpp
template <typename T>  
concept NomDuConcept = /* contrainte booléenne */;  
```

La contrainte peut être :

- un concept standard existant ;
- un trait de type (`std::is_..._v<T>`) ;
- une expression `requires` ;
- une combinaison des trois avec `&&` et `||`.

Commençons par les formes les plus simples et progressons vers les plus élaborées.

---

## Composer à partir de concepts standard

La forme la plus sûre et la plus lisible est de **combiner des concepts existants**. Vous bénéficiez de la subsomption, des messages d'erreur déjà bons, et d'une sémantique bien définie :

```cpp
// Un type numérique : entier ou flottant
template <typename T>  
concept Numeric = std::integral<T> || std::floating_point<T>;  

// Un type numérique signé
template <typename T>  
concept SignedNumeric = (Numeric<T> && std::signed_integral<T>)  
                                   || std::floating_point<T>;

// Un conteneur triable : range + éléments ordonnés
template <typename R>  
concept SortableRange = std::ranges::random_access_range<R>  
                     && std::totally_ordered<std::ranges::range_value_t<R>>;
```

```cpp
static_assert(Numeric<int>);  
static_assert(Numeric<double>);  
static_assert(!Numeric<std::string>);  

static_assert(SortableRange<std::vector<int>>);  
static_assert(!SortableRange<std::list<int>>);      // list n'est pas random_access  
static_assert(!SortableRange<std::vector<std::complex<double>>>); // complex n'est pas ordered  
```

### Attention à la précédence `&&` / `||`

Les opérateurs `&&` et `||` dans les contraintes suivent les mêmes règles de précédence que dans le C++ ordinaire (`&&` lie plus fort que `||`). Utilisez des parenthèses pour lever toute ambiguïté :

```cpp
// Ambigu pour le lecteur humain, même si le compilateur parse correctement
template <typename T>  
concept Ambiguous = std::integral<T> || std::floating_point<T> && std::same_as<T, float>;  
// Signifie : integral<T> || (floating_point<T> && same_as<T, float>)

// Intention claire avec parenthèses
template <typename T>  
concept Clear = (std::integral<T> || std::floating_point<T>) && (sizeof(T) <= 8);  
```

## Concepts basés sur des expressions `requires`

Quand aucun concept standard ne capture votre besoin, une expression `requires` teste directement la validité d'opérations sur le type.

### Détecter une interface (duck typing formel)

```cpp
template <typename T>  
concept Serializable = requires(const T& obj) {  
    { obj.serialize() } -> std::convertible_to<std::string>;
};
```

Ce concept exige que `T` possède une méthode `serialize() const` retournant quelque chose de convertible en `std::string`. Peu importe que `T` hérite d'une interface, implémente un trait, ou soit un type ad hoc — si la méthode existe avec la bonne signature, le concept est satisfait.

```cpp
struct Config {
    std::string name;
    int version;
    std::string serialize() const {
        return std::format("{}:v{}", name, version);
    }
};

struct RawData {
    void* ptr;
    std::size_t len;
    // Pas de méthode serialize()
};

static_assert(Serializable<Config>);  
static_assert(!Serializable<RawData>);  
```

### Exiger plusieurs opérations

Un concept peut regrouper un ensemble complet d'exigences formant un « protocole » :

```cpp
template <typename T>  
concept Loggable = requires(const T& obj, std::ostream& os) {  
    { obj.to_string() }  -> std::convertible_to<std::string>;  // Convertible en texte
    { obj.log_level() }  -> std::same_as<int>;                 // Niveau de log
    { os << obj }         -> std::same_as<std::ostream&>;      // Streamable
};
```

### Exiger des types membres

```cpp
template <typename T>  
concept StlContainer = requires(T c) {  
    typename T::value_type;
    typename T::iterator;
    typename T::const_iterator;
    typename T::size_type;
    { c.begin() }  -> std::same_as<typename T::iterator>;
    { c.end() }    -> std::same_as<typename T::iterator>;
    { c.size() }   -> std::convertible_to<typename T::size_type>;
    { c.empty() }  -> std::convertible_to<bool>;
};
```

```cpp
static_assert(StlContainer<std::vector<int>>);  
static_assert(StlContainer<std::list<double>>);  
static_assert(StlContainer<std::map<std::string, int>>);  
static_assert(!StlContainer<int>);  
static_assert(!StlContainer<int[10]>);  // C-array n'a pas de types membres  
```

### Exiger des expressions statiques

Les exigences imbriquées (`requires` interne) permettent de tester des propriétés statiques :

```cpp
template <typename T>  
concept SmallObject = requires {  
    requires sizeof(T) <= 64;
    requires std::is_trivially_copyable_v<T>;
};

template <typename T>  
concept PowerOfTwoSize = requires {  
    requires (sizeof(T) & (sizeof(T) - 1)) == 0;  // Taille puissance de 2
};
```

## Concepts multi-paramètres

Les concepts ne sont pas limités à un seul paramètre de type. Les concepts multi-paramètres expriment des **relations entre types** :

```cpp
template <typename From, typename To>  
concept Encodable = requires(const From& src) {  
    { encode(src) } -> std::convertible_to<To>;
};

template <typename T, typename Container>  
concept InsertableInto = requires(Container& c, T val) {  
    c.push_back(val);
} || requires(Container& c, T val) {
    c.insert(val);
};
```

```cpp
template <typename T, typename Container>
    requires InsertableInto<T, Container>
void ajouter(Container& c, const T& val) {
    if constexpr (requires { c.push_back(val); }) {
        c.push_back(val);
    } else {
        c.insert(val);
    }
}
```

```cpp
std::vector<int> v;  
ajouter(v, 42);           // Utilise push_back  

std::set<int> s;  
ajouter(s, 42);           // Utilise insert  
```

Le concept `InsertableInto` utilise `||` pour accepter les deux familles de conteneurs. La distinction est ensuite faite dans le corps avec `if constexpr` et une expression `requires` inline.

## Construire des hiérarchies de concepts

Comme les concepts standard (section 16.6.2), vos concepts personnalisés gagnent à être organisés en **hiérarchies** de subsomption. Chaque concept raffine le précédent en ajoutant des exigences :

```cpp
// Niveau 1 : un type qui peut être identifié par un nom
template <typename T>  
concept Named = requires(const T& obj) {  
    { obj.name() } -> std::convertible_to<std::string_view>;
};

// Niveau 2 : un Named avec une représentation textuelle complète
template <typename T>  
concept Describable = Named<T> && requires(const T& obj) {  
    { obj.description() } -> std::convertible_to<std::string>;
};

// Niveau 3 : un Describable sérialisable en JSON
template <typename T>  
concept JsonExportable = Describable<T> && requires(const T& obj) {  
    { obj.to_json() } -> std::convertible_to<std::string>;
};
```

La hiérarchie `Named` → `Describable` → `JsonExportable` est reconnue par la subsomption. Le compilateur préfère automatiquement la surcharge la plus contrainte :

```cpp
template <Named T>  
void afficher(const T& obj) {  
    std::print("Nom : {}\n", obj.name());
}

template <Describable T>  
void afficher(const T& obj) {  
    std::print("{}: {}\n", obj.name(), obj.description());
}

template <JsonExportable T>  
void afficher(const T& obj) {  
    std::print("{}\n", obj.to_json());
}
```

Un type satisfaisant `JsonExportable` déclenche la troisième surcharge. Un type `Describable` mais pas `JsonExportable` déclenche la deuxième. Un type seulement `Named` déclenche la première. Aucune ambiguïté, aucun `enable_if`, aucun tag dispatching.

### Pourquoi la subsomption exige la composition syntaxique

Pour que le compilateur reconnaisse que `JsonExportable` subsume `Describable`, la définition de `JsonExportable` doit **inclure syntaxiquement** `Describable<T>` dans son expression. C'est le cas ici grâce au `&&`.

Si les deux concepts étaient définis indépendamment avec des expressions `requires` distinctes mais équivalentes, le compilateur ne détecterait pas la relation — il ne fait pas de raisonnement sémantique, seulement syntaxique. C'est pourquoi la composition par concepts nommés est fondamentale.

## Concepts pour les politiques (Policy-based design)

Les concepts sont particulièrement adaptés au *policy-based design*, un pattern classique en C++ où une classe template est paramétrée par des « politiques » — des types qui fournissent un comportement configurable :

```cpp
// Concept pour une politique de logging
template <typename P>  
concept LogPolicy = requires(P policy, std::string_view message, int level) {  
    { policy.write(message, level) };
    { P::min_level } -> std::convertible_to<int>;
};

// Concept pour une politique d'allocation
template <typename P, typename T>  
concept AllocPolicy = requires(P alloc, std::size_t n) {  
    { alloc.allocate(n) }   -> std::same_as<T*>;
    { alloc.deallocate(std::declval<T*>(), n) };
};
```

```cpp
// Politique de logging vers la console
struct ConsoleLog {
    static constexpr int min_level = 0;
    void write(std::string_view msg, int level) const {
        if (level >= min_level) {
            std::print("[{}] {}\n", level, msg);
        }
    }
};

// Politique de logging silencieuse
struct NullLog {
    static constexpr int min_level = 999;
    void write(std::string_view, int) const {}
};

static_assert(LogPolicy<ConsoleLog>);  
static_assert(LogPolicy<NullLog>);  

// Classe paramétrée par une politique
template <LogPolicy Logger>  
class Service {  
public:  
    void execute() {
        logger_.write("Service démarré", 1);
        // ...
        logger_.write("Service terminé", 1);
    }
private:
    Logger logger_;
};
```

```cpp
Service<ConsoleLog> s1;  
s1.execute();  // Affiche les logs  

Service<NullLog> s2;  
s2.execute();  // Silencieux  
```

Sans les Concepts, le développeur découvrirait l'incompatibilité d'une politique incorrecte au fond d'une instanciation. Avec le concept `LogPolicy`, l'erreur est immédiate et nommée.

## Pattern : concept + type trait personnalisé

Pour les propriétés qu'une expression `requires` ne peut pas capturer directement (informations numériques, flags de configuration, associations de types), la combinaison **trait de type + concept** est idiomatique :

```cpp
// Trait : associer une taille de buffer par défaut à chaque type
template <typename T>  
struct buffer_traits;  // Non spécialisé → erreur si utilisé sans spécialisation  

template <>  
struct buffer_traits<float> {  
    static constexpr std::size_t default_size = 1024;
    static constexpr bool gpu_compatible = true;
};

template <>  
struct buffer_traits<double> {  
    static constexpr std::size_t default_size = 512;
    static constexpr bool gpu_compatible = true;
};

// Concept : un type est Bufferable s'il a des traits définis
template <typename T>  
concept Bufferable = requires {  
    { buffer_traits<T>::default_size } -> std::convertible_to<std::size_t>;
    { buffer_traits<T>::gpu_compatible } -> std::convertible_to<bool>;
};
```

```cpp
static_assert(Bufferable<float>);  
static_assert(Bufferable<double>);  
// static_assert(Bufferable<int>);  // Échec : buffer_traits<int> non spécialisé
```

Ce pattern est extensible : pour rendre un nouveau type `Bufferable`, il suffit de spécialiser `buffer_traits` pour ce type, sans modifier le concept ni le code qui l'utilise. C'est le même mécanisme qu'utilise la STL avec `std::hash`, `std::numeric_limits`, etc.

## Tester les concepts avec `static_assert`

Chaque concept personnalisé devrait être accompagné de **tests de compilation** vérifiant les cas positifs et négatifs. Ces tests servent à la fois de validation et de documentation :

```cpp
// ===== Tests pour le concept Serializable =====

// Types qui DOIVENT satisfaire le concept
struct GoodSerializer {
    std::string serialize() const { return ""; }
};
static_assert(Serializable<GoodSerializer>,
    "GoodSerializer doit être Serializable");

// Types qui NE DOIVENT PAS satisfaire le concept
struct NoSerialize {};  
static_assert(!Serializable<NoSerialize>,  
    "Un type sans serialize() ne doit pas être Serializable");

struct BadReturn {
    int serialize() const { return 0; }  // Retourne int, pas convertible en string
};
static_assert(!Serializable<BadReturn>,
    "serialize() doit retourner un type convertible en std::string");

struct NonConst {
    std::string serialize() { return ""; }  // Pas const
};
static_assert(!Serializable<NonConst>,
    "serialize() doit être const");
```

Ces `static_assert` ne produisent aucun code à l'exécution. Ils sont évalués à la compilation et documentent précisément le contrat du concept. Si une refactorisation casse un concept, les tests échouent immédiatement lors de la compilation — sans même exécuter la suite de tests.

### Organiser les tests

Placez les `static_assert` dans un fichier dédié ou dans un bloc `namespace tests` dans le header du concept :

```cpp
// concepts/serializable.h
#pragma once

template <typename T>  
concept Serializable = requires(const T& obj) {  
    { obj.serialize() } -> std::convertible_to<std::string>;
};

// Tests compile-time
namespace concepts_tests {  
namespace {  
    struct ValidSerializer {
        std::string serialize() const { return {}; }
    };
    static_assert(Serializable<ValidSerializer>);
    static_assert(!Serializable<int>);
    static_assert(!Serializable<std::string>);
}  // namespace anonyme
}  // namespace concepts_tests
```

Le namespace anonyme empêche les types de test de polluer le scope global. Les tests sont compilés avec le header mais ne produisent aucun symbole.

## Organisation dans un projet

### Structure de fichiers

Les concepts personnalisés méritent un espace dédié dans l'arborescence du projet :

```
include/
  monprojet/
    concepts/
      serializable.h
      loggable.h
      bufferable.h
      numeric.h
      policies.h
    core/
      ...
    net/
      ...
```

Chaque header de concept est autonome et inclut ses dépendances (`<concepts>`, `<string>`, etc.).

### Conventions de nommage

Les conventions dominantes dans l'écosystème C++ pour les concepts sont :

- **PascalCase** : `Serializable`, `SortableRange`, `LogPolicy`. C'est le style de la STL et le plus répandu.
- Préfixe par domaine si nécessaire : `net::Connectable`, `db::Queryable`.
- Un concept décrit une **capacité** ou une **propriété** : préférer des adjectifs (`Serializable`, `Copyable`, `Hashable`) ou des noms de rôle (`Iterator`, `Container`, `Allocator`).

### Documenter les concepts

Un concept bien documenté inclut :

- une description en une phrase de ce qu'il représente ;
- la liste des exigences syntaxiques (les expressions `requires`) ;
- les exigences sémantiques (invariants, complexité) que le compilateur ne vérifie pas ;
- des exemples de types qui satisfont et ne satisfont pas le concept.

```cpp
/// @brief Un type dont les instances peuvent être sérialisées en texte.
///
/// Exigences syntaxiques :
/// - `obj.serialize()` est valide pour `const T& obj`
/// - Le résultat est convertible en `std::string`
///
/// Exigences sémantiques :
/// - serialize() est une fonction pure (sans effet de bord)
/// - Deux objets égaux produisent la même sérialisation
///
/// Satisfait par : Config, UserProfile, LogEntry
/// Non satisfait par : int, RawData, std::mutex
template <typename T>  
concept Serializable = requires(const T& obj) {  
    { obj.serialize() } -> std::convertible_to<std::string>;
};
```

## Patterns avancés

### Concept récursif pour les structures imbriquées

Un concept peut être utilisé récursivement pour contraindre des structures arborescentes :

```cpp
template <typename T>  
concept JsonValue = std::integral<T>  
                 || std::floating_point<T>
                 || std::same_as<T, std::string>
                 || std::same_as<T, bool>
                 || std::same_as<T, std::nullptr_t>;

template <typename T>  
concept JsonSerializable = JsonValue<T> || requires(const T& obj) {  
    { obj.to_json() } -> std::convertible_to<std::string>;
};
```

### Concept conditionné par un autre concept

On peut paramétrer un concept par un autre concept grâce à des templates de templates, mais la syntaxe devient complexe. Une approche plus pragmatique utilise un concept multi-paramètre :

```cpp
template <typename Container, typename Pred>  
concept FilterableBy = std::ranges::input_range<Container>  
    && std::predicate<Pred, std::ranges::range_value_t<Container>>;

template <typename Container, typename Pred>
    requires FilterableBy<Container, Pred>
auto filtrer(const Container& c, Pred pred) {
    using T = std::ranges::range_value_t<Container>;
    std::vector<T> result;
    for (const auto& elem : c) {
        if (pred(elem)) {
            result.push_back(elem);
        }
    }
    return result;
}
```

```cpp
std::vector<int> nums{1, 2, 3, 4, 5, 6, 7, 8};  
auto pairs = filtrer(nums, [](int n) { return n % 2 == 0; });  
// pairs = {2, 4, 6, 8}
```

### Concept avec contrainte sur les types associés

Quand un concept porte sur un conteneur ou un protocole avec des types associés, la combinaison de type requirements et de compound requirements est puissante :

```cpp
template <typename Q>  
concept MessageQueue = requires(Q queue) {  
    typename Q::message_type;
    typename Q::error_type;

    { queue.push(std::declval<typename Q::message_type>()) }
        -> std::same_as<bool>;

    { queue.pop() }                               // std::expected est C++23
        -> std::same_as<std::expected<typename Q::message_type,
                                       typename Q::error_type>>;

    { queue.size() } -> std::convertible_to<std::size_t>;
    { queue.empty() } -> std::same_as<bool>;

    requires std::movable<typename Q::message_type>;
};
```

Ce concept `MessageQueue` encode un contrat complet : types associés, signatures de méthodes, types de retour, et même des contraintes sur les types associés eux-mêmes (`message_type` doit être `movable`).

## Erreurs courantes

### Concept trop large

Un concept qui n'exige presque rien n'apporte pas de valeur :

```cpp
// Trop vague — quasiment tout type passe
template <typename T>  
concept Usable = requires(T val) {  
    val;  // Toute expression est "valide" si T est un type complet
};
```

Un bon concept doit rejeter les types qui ne fonctionneraient pas avec le template qu'il protège.

### Concept trop étroit

À l'inverse, un concept qui encode chaque détail d'une implémentation spécifique empêche la réutilisation :

```cpp
// Trop spécifique — couplé à une implémentation concrète
template <typename T>  
concept MyExactFormat = requires(const T& obj) {  
    { obj.to_json_v2() } -> std::same_as<std::string>;
    { obj.schema_version() } -> std::same_as<int>;
    { obj.checksum() } -> std::same_as<uint32_t>;
    requires sizeof(T) == 128;
};
```

Demandez-vous : « si j'écris un nouveau type conforme à l'esprit de ce concept, quelles exigences sont *essentielles* et lesquelles sont des détails d'implémentation ? »

### Oublier `const` dans les paramètres fictifs

```cpp
// Bug subtil : exige une méthode non-const
template <typename T>  
concept Printable = requires(T obj) {       // T obj, pas const T& obj  
    { obj.to_string() } -> std::convertible_to<std::string>;
};

// Correct : exige une méthode const
template <typename T>  
concept Printable = requires(const T& obj) {  
    { obj.to_string() } -> std::convertible_to<std::string>;
};
```

Si la méthode est censée être appelable sur un objet constant (ce qui est presque toujours le cas pour les opérations de lecture), le paramètre fictif doit être `const T&`.

## Bonnes pratiques — synthèse

**Composez d'abord à partir de concepts standard.** Chaque concept personnalisé devrait, autant que possible, inclure ou affiner des concepts standard. Cela maximise la subsomption et l'interopérabilité.

**Un concept = un rôle.** Chaque concept devrait capturer une responsabilité unique et nommable. Si la description nécessite le mot « et » entre deux idées indépendantes, c'est probablement deux concepts à composer.

**Testez les deux polarités.** Chaque concept doit être accompagné de `static_assert` positifs (types conformes) **et** négatifs (types non conformes, cas limites).

**Documentez les exigences sémantiques.** Le compilateur vérifie la syntaxe, pas la sémantique. Les invariants, la complexité, l'absence d'effets de bord — tout ce que le compilateur ne peut pas vérifier doit être documenté et testé unitairement.

**Nommez selon la capacité, pas selon l'implémentation.** `Serializable` est meilleur que `HasSerializeMethod`. Le concept décrit ce que le type *peut faire*, pas comment il le fait.

**Évitez `requires requires` en code de production.** Extrayez toujours la contrainte dans un concept nommé. Le code gagne en lisibilité, en réutilisabilité et en compatibilité avec la subsomption.

**Visez la granularité juste.** Ni trop large (tout passe) ni trop étroit (un seul type passe). Un bon concept rejette les types qui causeraient des erreurs à l'instanciation et accepte tout type qui fonctionnerait correctement.

---

## En résumé

Les concepts personnalisés sont les briques avec lesquelles vous construisez le vocabulaire générique de votre projet. Ils se créent en composant des concepts standard, en écrivant des expressions `requires` ciblées, et en les organisant en hiérarchies de subsomption. Chaque concept encode un contrat vérifiable à la compilation, documenté par des `static_assert` et des commentaires sur les invariants sémantiques.

Combinés avec les concepts standard (16.6.2) et la syntaxe `requires` (16.6.1), les concepts personnalisés complètent la boîte à outils des Concepts C++20. Ils représentent l'aboutissement de l'évolution de la programmation générique en C++ : du code invisible dans le corps des templates (C++98), en passant par SFINAE (C++11), jusqu'à des contraintes nommées, lisibles et composables.
