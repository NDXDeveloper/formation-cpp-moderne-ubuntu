# 16.6.2 — Concepts standard de la STL

## Chapitre 16 : Templates et Métaprogrammation · Section 16.6 : Concepts (C++20)

---

## Introduction

La sous-section précédente a couvert la syntaxe `requires` — le *comment* des Concepts. Cette sous-section couvre le *quoi* : les concepts que la bibliothèque standard fournit prêts à l'emploi.

C++20 introduit une collection riche de concepts prédéfinis, répartis principalement dans les headers `<concepts>`, `<iterator>`, `<ranges>` et `<type_traits>`. Ces concepts forment un **vocabulaire commun** pour la programmation générique. Ils encodent les exigences que la communauté C++ a identifiées et affinées au fil de deux décennies de pratique avec les templates et la STL.

Utiliser les concepts standard plutôt que de réinventer les vôtres offre trois avantages immédiats : ils sont **testés** sur toutes les implémentations majeures, ils sont **reconnus par la subsomption** (le compilateur connaît leurs relations hiérarchiques), et ils forment un **langage commun** que tout développeur C++ moderne peut lire.

---

## Vue d'ensemble par familles

Les concepts standard se regroupent en grandes familles thématiques. Le tableau ci-dessous donne une cartographie avant de plonger dans chaque famille :

| Famille | Header principal | Rôle |
|---|---|---|
| Core language | `<concepts>` | Relations entre types, propriétés fondamentales |
| Arithmetic | `<concepts>` | Classification des types numériques |
| Object | `<concepts>` | Capacités de copie, déplacement, construction |
| Comparison | `<concepts>` | Opérateurs de comparaison |
| Callable | `<concepts>` | Fonctions, prédicats, relations |
| Iterator | `<iterator>` | Catégories d'itérateurs |
| Range | `<ranges>` | Catégories de plages |

## Famille Core Language (`<concepts>`)

Ces concepts expriment les relations fondamentales entre types et les propriétés du système de types.

### `std::same_as<T, U>`

Vérifie que `T` et `U` sont exactement le même type, sans conversion :

```cpp
static_assert(std::same_as<int, int>);  
static_assert(!std::same_as<int, long>);       // Types différents  
static_assert(!std::same_as<int, const int>);  // const change le type  
```

`same_as` est symétrique : `same_as<T, U>` est équivalent à `same_as<U, T>`. C'est garanti par la spécification.

Usage typique dans les compound requirements :

```cpp
template <typename T>  
concept SquareRoot = requires(T val) {  
    { val.sqrt() } -> std::same_as<T>;  // sqrt() retourne exactement T
};
```

### `std::derived_from<Derived, Base>`

Vérifie que `Derived` hérite publiquement de `Base` :

```cpp
struct Animal {};  
struct Chien : public Animal {};  
struct Robot {};  

static_assert(std::derived_from<Chien, Animal>);  
static_assert(!std::derived_from<Robot, Animal>);  
static_assert(std::derived_from<Animal, Animal>);  // Un type dérive de lui-même  
```

### `std::convertible_to<From, To>`

Vérifie que `From` est implicitement **et** explicitement convertible en `To` :

```cpp
static_assert(std::convertible_to<int, double>);        // Promotion numérique  
static_assert(std::convertible_to<double, int>);         // Conversion avec troncation  
static_assert(!std::convertible_to<std::string, int>);   // Pas de conversion  
```

C'est le concept le plus fréquemment utilisé dans les compound requirements pour contraindre le type de retour d'une expression :

```cpp
template <typename T>  
concept BoolTestable = requires(T val) {  
    { !val } -> std::convertible_to<bool>;
};
```

### `std::common_reference_with<T, U>` et `std::common_with<T, U>`

`common_with` vérifie que `T` et `U` partagent un **type commun** vers lequel les deux peuvent être convertis. `common_reference_with` est la variante pour les références. Ces concepts sont surtout utilisés en interne par les concepts d'itérateurs et de ranges :

```cpp
static_assert(std::common_with<int, double>);  // Type commun : double
```

### `std::assignable_from<LHS, RHS>`

Vérifie que `RHS` peut être assigné à `LHS` via l'opérateur `=` :

```cpp
static_assert(std::assignable_from<int&, int>);  
static_assert(std::assignable_from<std::string&, const char*>);  
```

Notez que `LHS` est une référence : l'assignation nécessite un lvalue à gauche.

### `std::swappable` et `std::swappable_with<T, U>`

Vérifie que des valeurs du type peuvent être échangées via `std::swap` ou un swap trouvé par ADL :

```cpp
static_assert(std::swappable<int>);  
static_assert(std::swappable<std::string>);  
```

## Famille Arithmetic (`<concepts>`)

Ces concepts classifient les types numériques fondamentaux du langage.

### `std::integral<T>`

Satisfait par tous les types entiers : `bool`, `char`, `short`, `int`, `long`, `long long`, et leurs variantes signées/non signées, ainsi que `char8_t`, `char16_t`, `char32_t` et `wchar_t` :

```cpp
static_assert(std::integral<int>);  
static_assert(std::integral<unsigned long>);  
static_assert(std::integral<bool>);  
static_assert(std::integral<char>);  
static_assert(!std::integral<double>);  
static_assert(!std::integral<std::string>);  
```

### `std::signed_integral<T>` et `std::unsigned_integral<T>`

Sous-ensembles de `std::integral` :

```cpp
static_assert(std::signed_integral<int>);  
static_assert(!std::signed_integral<unsigned int>);  
static_assert(std::unsigned_integral<std::size_t>);  
```

La hiérarchie de subsomption est correctement établie : `std::signed_integral` subsume `std::integral`. Le compilateur sait qu'une surcharge contrainte par `signed_integral` est plus spécifique qu'une surcharge contrainte par `integral`.

### `std::floating_point<T>`

Satisfait par `float`, `double` et `long double` :

```cpp
static_assert(std::floating_point<double>);  
static_assert(std::floating_point<float>);  
static_assert(!std::floating_point<int>);  
```

### Absence de concept `arithmetic`

La STL ne fournit pas de concept `std::arithmetic` regroupant entiers et flottants. Si vous en avez besoin, c'est un cas typique de concept personnalisé (section 16.6.3) :

```cpp
template <typename T>  
concept Arithmetic = std::integral<T> || std::floating_point<T>;  
```

## Famille Object (`<concepts>`)

Ces concepts décrivent les capacités fondamentales d'un type en termes de construction, copie, déplacement et destruction. Ils forment une hiérarchie progressive :

### Concepts de construction et destruction

```cpp
std::constructible_from<T, Args...>   // T est constructible depuis Args...  
std::default_initializable<T>         // T() ou T{} est valide  
std::destructible<T>                  // Le destructeur est accessible et non-throwing  
```

```cpp
static_assert(std::default_initializable<int>);  
static_assert(std::default_initializable<std::string>);  
static_assert(std::constructible_from<std::string, const char*>);  
static_assert(std::constructible_from<std::vector<int>, std::size_t, int>);  
```

### Concepts de déplacement et copie

```cpp
std::movable<T>     // Déplaçable : move constructible + move assignable + swappable  
std::copyable<T>    // Copiable : movable + copy constructible + copy assignable  
```

```cpp
static_assert(std::movable<std::unique_ptr<int>>);  
static_assert(!std::copyable<std::unique_ptr<int>>);  // unique_ptr n'est pas copiable  
static_assert(std::copyable<std::string>);  
static_assert(std::copyable<int>);  
```

### `std::semiregular<T>`

Un type `semiregular` est `copyable` et `default_initializable`. Il se comporte « presque comme un type natif » — on peut le créer par défaut, le copier, le déplacer :

```cpp
static_assert(std::semiregular<int>);  
static_assert(std::semiregular<std::string>);  
static_assert(!std::semiregular<std::unique_ptr<int>>);  // Pas copiable  
```

### `std::regular<T>`

Un type `regular` est `semiregular` **et** `equality_comparable` (supporte `==` et `!=`). C'est le concept qui capture la notion de « type bien comporté », comparable à un type natif :

```cpp
static_assert(std::regular<int>);  
static_assert(std::regular<std::string>);  
```

La hiérarchie complète de subsomption est :

```
destructible
    ← constructible_from
        ← default_initializable
    ← movable
        ← copyable
            ← semiregular
                ← regular
```

Chaque concept subsume ceux qui le précèdent. Une surcharge contrainte par `regular` est automatiquement préférée à une surcharge contrainte par `copyable` pour un type qui satisfait les deux.

## Famille Comparison (`<concepts>`)

Ces concepts formalisent les relations de comparaison.

### `std::equality_comparable<T>` et `std::equality_comparable_with<T, U>`

Vérifie que `T` supporte `==` et `!=`, et que le résultat est convertible en `bool` :

```cpp
static_assert(std::equality_comparable<int>);  
static_assert(std::equality_comparable<std::string>);  

// Comparaison entre types différents
static_assert(std::equality_comparable_with<int, double>);  
static_assert(std::equality_comparable_with<std::string, const char*>);  
```

### `std::totally_ordered<T>` et `std::totally_ordered_with<T, U>`

Vérifie que `T` supporte les six opérateurs de comparaison (`==`, `!=`, `<`, `>`, `<=`, `>=`) avec une sémantique d'ordre total :

```cpp
static_assert(std::totally_ordered<int>);  
static_assert(std::totally_ordered<std::string>);  
```

`totally_ordered` subsume `equality_comparable` : si un type a un ordre total, il est forcément comparable en égalité.

### `std::three_way_comparable<T>`

Vérifie que `T` supporte l'opérateur spaceship `<=>` (section 8.5) :

```cpp
static_assert(std::three_way_comparable<int>);  
static_assert(std::three_way_comparable<std::string>);  
```

## Famille Callable (`<concepts>`)

Ces concepts contraignent les types appelables : fonctions, lambdas, foncteurs, pointeurs de fonctions.

### `std::invocable<F, Args...>`

Vérifie que `F` peut être appelé avec les arguments `Args...` via `std::invoke` :

```cpp
static_assert(std::invocable<decltype([](int){}), int>);

auto carre = [](double x) { return x * x; };  
static_assert(std::invocable<decltype(carre), double>);  
static_assert(!std::invocable<decltype(carre), std::string>);  
```

### `std::regular_invocable<F, Args...>`

Identique à `invocable` en termes de vérification syntaxique, mais exprime une **exigence sémantique** supplémentaire : l'appel doit être sans effet de bord observable et doit être equality-preserving (appeler deux fois avec les mêmes arguments donne le même résultat). Le compilateur ne peut pas vérifier cette propriété ; le concept sert de documentation.

### `std::predicate<F, Args...>`

Un prédicat est un `regular_invocable` dont le résultat est convertible en `bool` :

```cpp
auto est_pair = [](int n) { return n % 2 == 0; };

static_assert(std::predicate<decltype(est_pair), int>);
```

C'est le concept utilisé par les algorithmes de la STL qui prennent un prédicat : `std::find_if`, `std::remove_if`, `std::count_if`, etc.

### `std::relation<R, T, U>`

Un concept pour les relations binaires — un callable qui prend deux arguments et retourne `bool` :

```cpp
auto moins_que = [](int a, int b) { return a < b; };

static_assert(std::relation<decltype(moins_que), int, int>);
```

### `std::equivalence_relation<R, T, U>` et `std::strict_weak_order<R, T, U>`

Sous-ensembles sémantiques de `relation`. `strict_weak_order` est le concept attendu par `std::sort` et les conteneurs ordonnés. Comme `regular_invocable`, la sémantique est documentaire — le compilateur vérifie la syntaxe, le développeur garantit la sémantique.

## Famille Iterator (`<iterator>`)

Les concepts d'itérateurs formalisent les catégories qui existaient depuis C++98 sous forme de tags (`std::input_iterator_tag`, etc.), mais en les rendant vérifiables à la compilation.

### Hiérarchie des concepts d'itérateurs

```
std::input_or_output_iterator
    ← std::input_iterator
        ← std::forward_iterator
            ← std::bidirectional_iterator
                ← std::random_access_iterator
                    ← std::contiguous_iterator
    ← std::output_iterator
```

Chaque niveau ajoute des capacités :

```cpp
// input_iterator : lecture séquentielle (une seule passe)
static_assert(std::input_iterator<std::istream_iterator<int>>);

// forward_iterator : lecture multi-passe
static_assert(std::forward_iterator<std::forward_list<int>::iterator>);

// bidirectional_iterator : avance et recule
static_assert(std::bidirectional_iterator<std::list<int>::iterator>);

// random_access_iterator : accès par index, arithmétique
static_assert(std::random_access_iterator<std::deque<int>::iterator>);

// contiguous_iterator : mémoire contiguë garantie
static_assert(std::contiguous_iterator<std::vector<int>::iterator>);  
static_assert(std::contiguous_iterator<int*>);  
```

### `std::sentinel_for<S, I>`

Le concept de sentinelle généralise la notion de « fin d'itération ». Un sentinel pour un itérateur `I` est un type `S` tel que `I` et `S` sont comparables par `==` :

```cpp
// Les itérateurs classiques utilisent le même type comme sentinelle
static_assert(std::sentinel_for<
    std::vector<int>::iterator,
    std::vector<int>::iterator
>);
```

Les sentinelles sont au cœur de la bibliothèque Ranges (section 12.5), où itérateur et sentinelle peuvent avoir des types différents.

### Utilisation dans du code générique

Les concepts d'itérateurs permettent d'écrire des algorithmes avec des contraintes explicites :

```cpp
template <std::forward_iterator It, std::sentinel_for<It> Sent>  
std::size_t compter(It debut, Sent fin) {  
    std::size_t n = 0;
    while (debut != fin) {
        ++debut;
        ++n;
    }
    return n;
}
```

Le compilateur vérifie à l'appel que les itérateurs fournis satisfont les concepts requis. Passer un `std::input_iterator` (qui ne garantit pas le multi-passe) à un algorithme exigeant `forward_iterator` produit un message d'erreur nommant exactement le concept violé.

## Famille Range (`<ranges>`)

Les concepts de ranges (section 12.5) opèrent à un niveau plus haut : ils caractérisent des conteneurs et des vues plutôt que des itérateurs individuels.

### `std::ranges::range<R>`

Le concept le plus fondamental : `R` possède `begin()` et `end()` :

```cpp
static_assert(std::ranges::range<std::vector<int>>);  
static_assert(std::ranges::range<std::string>);  
static_assert(std::ranges::range<int[10]>);        // C-array  
static_assert(!std::ranges::range<int>);             // int n'est pas un range  
```

### Sous-concepts de range

```cpp
std::ranges::input_range<R>            // Itérateur input  
std::ranges::forward_range<R>          // Itérateur forward  
std::ranges::bidirectional_range<R>    // Itérateur bidirectionnel  
std::ranges::random_access_range<R>    // Itérateur random-access  
std::ranges::contiguous_range<R>       // Mémoire contiguë  
std::ranges::sized_range<R>            // .size() en O(1)  
```

```cpp
static_assert(std::ranges::contiguous_range<std::vector<int>>);  
static_assert(std::ranges::bidirectional_range<std::list<int>>);  
static_assert(std::ranges::random_access_range<std::deque<int>>);  
static_assert(std::ranges::sized_range<std::vector<int>>);  
```

### `std::ranges::view<V>`

Un `view` est un range léger (déplaçable en O(1)) qui ne possède pas ses éléments :

```cpp
static_assert(std::ranges::view<std::ranges::ref_view<std::vector<int>>>);  
static_assert(std::ranges::view<std::string_view>);  
static_assert(!std::ranges::view<std::vector<int>>);  // vector possède ses éléments  
```

### Exemple : une fonction contrainte par range

```cpp
template <std::ranges::input_range R>
    requires std::totally_ordered<std::ranges::range_value_t<R>>
auto trouver_min_max(R&& range) {
    auto it = std::ranges::begin(range);
    auto sent = std::ranges::end(range);

    if (it == sent) {
        throw std::runtime_error("Range vide");
    }

    auto min_val = *it;
    auto max_val = *it;

    while (++it != sent) {
        if (*it < min_val) min_val = *it;
        if (*it > max_val) max_val = *it;
    }

    return std::pair{min_val, max_val};
}
```

```cpp
std::vector v{3, 1, 4, 1, 5, 9};  
auto [min, max] = trouver_min_max(v);  
std::print("Min: {}, Max: {}\n", min, max);  // Min: 1, Max: 9  

auto [a, b] = trouver_min_max(std::array{7, 2, 8});  // Fonctionne aussi
```

## Concepts utilitaires additionnels

Quelques concepts ne rentrent pas dans les familles précédentes mais sont fréquemment utiles.

### `std::constructible_from<T, Args...>`

Déjà mentionné dans la famille Object, mais particulièrement utile en combinaison avec d'autres contraintes pour les fonctions factory :

```cpp
template <typename T, typename... Args>
    requires std::constructible_from<T, Args...>
T creer(Args&&... args) {
    return T(std::forward<Args>(args)...);
}
```

### `std::move_constructible<T>` et `std::copy_constructible<T>`

Concepts atomiques testant une seule capacité, utiles quand `movable` ou `copyable` est trop exigeant :

```cpp
template <std::move_constructible T>  
void stocker(T&& val) {  
    // On sait que T peut être déplacé, mais pas nécessairement copié
    auto stored = std::move(val);
}
```

## Choisir le bon concept

Avec autant de concepts disponibles, un guide de sélection aide à choisir le plus approprié :

**Vous contraignez un type numérique ?**
→ `std::integral`, `std::signed_integral`, `std::unsigned_integral` ou `std::floating_point`.

**Vous contraignez un type « bien comporté » (copiable, comparable) ?**
→ `std::regular` est le gold standard. Descendre vers `std::semiregular`, `std::copyable` ou `std::movable` si les exigences sont moindres.

**Vous contraignez un paramètre callable (lambda, foncteur) ?**
→ `std::invocable<F, Args...>` pour un callable générique, `std::predicate<F, Args...>` s'il retourne `bool`.

**Vous contraignez un itérateur ?**
→ Choisir le concept d'itérateur le **moins exigeant** qui suffit : `input_iterator` si une seule passe suffit, `forward_iterator` pour le multi-passe, `random_access_iterator` pour l'accès indexé.

**Vous contraignez un conteneur ou une vue ?**
→ `std::ranges::range` au minimum, puis affiner avec `input_range`, `forward_range`, `sized_range`, etc.

**Vous contraignez un type de retour ?**
→ `std::same_as<T>` pour un type exact, `std::convertible_to<T>` pour accepter les conversions.

**Le concept standard ne couvre pas votre besoin ?**
→ Créez un concept personnalisé (section 16.6.3), idéalement en composant des concepts standard existants.

---

## Bonnes pratiques

**Utilisez le concept le moins contraignant qui suffit.** Contraindre par `std::random_access_iterator` quand `std::forward_iterator` suffit exclut inutilement des types valides (`std::list`, `std::forward_list`). Appliquer le même principe que pour les paramètres de fonctions : demander le minimum, offrir le maximum.

**Exploitez la subsomption plutôt que la duplication.** Les concepts standard sont conçus pour former des hiérarchies de subsomption. Utilisez-les directement plutôt que de recréer des contraintes équivalentes qui ne seraient pas reconnues par le compilateur.

**Consultez la documentation des algorithmes.** La STL documente les concepts requis pour chaque algorithme. Par exemple, `std::sort` requiert `std::random_access_iterator` et `std::sortable` — utiliser les mêmes concepts dans votre code assure la cohérence.

**Distinguez contraintes syntaxiques et sémantiques.** Des concepts comme `std::regular_invocable` et `std::strict_weak_order` incluent des exigences sémantiques que le compilateur ne peut pas vérifier. Ces exigences sont la responsabilité du développeur et doivent être testées via des tests unitaires.

**Testez systématiquement avec `static_assert`.** Vérifiez que vos types satisfont les concepts attendus :

```cpp
struct MonType { /* ... */ };

static_assert(std::regular<MonType>,
    "MonType doit être regular pour être utilisable dans les conteneurs standard");
```

Un `static_assert` avec un message explicite sert à la fois de test et de documentation.
