# 9.2.1 Comptage de références

## Le principe

Le comptage de références (*reference counting*) est le mécanisme interne qui permet à `std::shared_ptr` de savoir **quand libérer la ressource**. L'idée est simple : chaque `shared_ptr` qui pointe vers une même ressource partage un compteur commun. Ce compteur reflète le nombre de co-propriétaires actifs.

- Quand un `shared_ptr` est **copié**, le compteur est incrémenté.
- Quand un `shared_ptr` est **détruit** ou **réaffecté**, le compteur est décrémenté.
- Quand le compteur **atteint zéro**, la ressource est libérée.

C'est un contrat automatique : tant qu'au moins un propriétaire existe, la ressource vit. Dès que le dernier propriétaire disparaît, la ressource meurt.

---

## Visualiser le compteur avec use_count()

La méthode `use_count()` retourne le nombre actuel de `shared_ptr` qui partagent la même ressource. Elle est utile pour comprendre le mécanisme, même si elle est rarement utilisée en production.

```cpp
#include <memory>

auto a = std::make_shared<std::string>("Bonjour");  
std::print("Après création :      use_count = {}\n", a.use_count());  // 1  

{
    auto b = a;  // Copie — incrémentation
    std::print("Après copie (b) :     use_count = {}\n", a.use_count());  // 2

    auto c = a;  // Nouvelle copie
    std::print("Après copie (c) :     use_count = {}\n", a.use_count());  // 3

}   // b et c détruits — deux décrémentations

std::print("Après destruction b,c: use_count = {}\n", a.use_count());  // 1

a.reset();  // a abandonne la ressource — dernière décrémentation
// use_count atteint 0 → la string "Bonjour" est libérée
```

Chaque ligne raconte une histoire précise : la ressource gagne et perd des co-propriétaires, et sa durée de vie suit exactement ces fluctuations.

## Le bloc de contrôle (*control block*)

Le compteur de références ne flotte pas dans le vide — il doit être stocké quelque part, accessible par tous les `shared_ptr` qui partagent la même ressource. C'est le rôle du **bloc de contrôle**, une structure allouée sur le tas qui accompagne chaque ressource partagée.

### Contenu du bloc de contrôle

Le bloc de contrôle contient quatre éléments :

| Champ | Rôle |
|---|---|
| **strong count** | Nombre de `shared_ptr` actifs. Quand il atteint 0, l'objet pointé est détruit. |
| **weak count** | Nombre de `weak_ptr` actifs. Quand strong count ET weak count atteignent 0, le bloc de contrôle lui-même est libéré. |
| **deleter** | La logique de destruction de la ressource (par défaut `delete`, mais personnalisable). |
| **allocateur** | L'allocateur utilisé pour le bloc de contrôle lui-même. |

Un point subtil : le bloc de contrôle **survit à l'objet pointé** si des `weak_ptr` existent encore. L'objet est détruit quand strong count atteint 0, mais le bloc de contrôle reste en vie tant que des `weak_ptr` ont besoin de le consulter pour savoir que la ressource a expiré. Le bloc n'est libéré que quand les deux compteurs sont à zéro.

```
Phase 1 : Ressource vivante (strong_count > 0)
─────────────────────────────────────────────
  Control Block          Objet T
  ┌────────────────┐    ┌──────────┐
  │ strong_count: 2│    │          │
  │ weak_count:  1 │    │ (vivant) │
  │ deleter        │    │          │
  └────────────────┘    └──────────┘
        ✅ Actif              ✅ Actif


Phase 2 : Ressource détruite, weak_ptr encore actifs
─────────────────────────────────────────────────────
  Control Block          Objet T
  ┌────────────────┐    ┌──────────┐
  │ strong_count: 0│    │          │
  │ weak_count:  1 │    │ (détruit)│
  │ deleter        │    │          │
  └────────────────┘    └──────────┘
        ✅ Actif              ❌ Libéré


Phase 3 : Plus aucune référence
────────────────────────────────
  Control Block          Objet T
  ┌────────────────┐    ┌──────────┐
  │ strong_count: 0│    │          │
  │ weak_count:  0 │    │ (détruit)│
  │ deleter        │    │          │
  └────────────────┘    └──────────┘
        ❌ Libéré             ❌ Libéré
```

### Quand un bloc de contrôle est-il créé ?

Un nouveau bloc de contrôle est créé dans exactement trois situations :

1. **Appel à `std::make_shared<T>(...)`** — c'est la voie recommandée. L'objet et le bloc de contrôle sont alloués en un seul bloc mémoire contigu (voir plus bas).

2. **Construction d'un `shared_ptr` à partir d'un pointeur brut** — `std::shared_ptr<T>(new T(...))`. Un bloc de contrôle séparé est alloué.

3. **Construction d'un `shared_ptr` à partir d'un `unique_ptr`** — le `unique_ptr` est consommé (déplacé) et un nouveau bloc de contrôle est créé.

**La copie d'un `shared_ptr` ne crée jamais de nouveau bloc de contrôle.** Elle incrémente le compteur du bloc existant. C'est tout l'intérêt du mécanisme.

## Les opérations qui modifient le compteur

Comprendre quelles opérations incrémentent ou décrémentent le compteur est essentiel pour raisonner sur la durée de vie des ressources.

### Opérations qui incrémentent (strong count +1)

```cpp
auto a = std::make_shared<int>(42);   // Création : strong_count = 1

auto b = a;                           // Copie : strong_count = 2

auto c(a);                            // Copie (constructeur) : strong_count = 3

std::shared_ptr<int> d;  
d = a;                                // Affectation par copie : strong_count = 4  

auto locked = weak.lock();            // Promotion weak → shared (si non expiré) : +1
```

### Opérations qui décrémentent (strong count -1)

```cpp
b.reset();       // b abandonne la ressource : strong_count -1

c = nullptr;     // Équivalent à reset() : strong_count -1

d = autre_ptr;   // d change de ressource : -1 sur l'ancienne, +1 sur la nouvelle

// Sortie de scope (destruction) : strong_count -1
{
    auto temp = a;   // +1
}                    // temp détruit → -1
```

### Opérations qui ne modifient PAS le compteur

```cpp
auto e = std::move(a);   // Déplacement : le compteur ne change pas
                          // a devient nul, e prend sa place
                          // C'est un transfert, pas un partage

int* raw = e.get();       // Pointeur brut : aucun impact sur le compteur

int& ref = *e;            // Référence : aucun impact

std::weak_ptr<int> w = e; // weak_ptr : incrémente le weak_count, pas le strong_count
```

Le déplacement (`std::move`) est une optimisation importante : il transfère la propriété sans toucher au compteur atomique. C'est pourquoi il faut **préférer `std::move` à la copie** quand le `shared_ptr` source n'est plus nécessaire.

```cpp
void consommer(std::shared_ptr<Config> config);

auto config = std::make_shared<Config>();

// ⚠️ Copie → incrémentation atomique + décrémentation atomique
consommer(config);

// ✅ Move → aucune opération atomique, juste une copie de pointeurs
consommer(std::move(config));
// config est maintenant nul
```

## Atomicité du compteur : thread-safety et coût

Les compteurs de références d'un `shared_ptr` sont **atomiques** : les incrémentations et décrémentations utilisent des instructions CPU spéciales qui garantissent la cohérence même quand plusieurs threads manipulent des `shared_ptr` vers la même ressource simultanément.

### Ce qui est thread-safe

Le **compteur** est thread-safe. Vous pouvez copier, déplacer et détruire des `shared_ptr` vers la même ressource depuis différents threads sans risque de corruption du compteur :

```cpp
auto data = std::make_shared<std::vector<int>>(1000, 0);

// Thread 1 : crée une copie locale
std::thread t1([data]() {         // Copie du shared_ptr → +1 atomique
    // utilise data...
});                               // Destruction de la copie → -1 atomique

// Thread 2 : crée une autre copie locale  
std::thread t2([data]() {         // Copie du shared_ptr → +1 atomique
    // utilise data...
});                               // Destruction de la copie → -1 atomique
```

### Ce qui n'est PAS thread-safe

L'**objet pointé** n'est pas protégé par le mécanisme de `shared_ptr`. Si plusieurs threads accèdent à l'objet et qu'au moins un le modifie, il faut une synchronisation externe (mutex, atomic…) :

```cpp
auto compteur = std::make_shared<int>(0);

// ❌ DATA RACE — l'objet pointé n'est pas protégé
std::thread t1([compteur]() { (*compteur)++; });  
std::thread t2([compteur]() { (*compteur)++; });  

// ✅ Protection explicite nécessaire
auto mutex = std::make_shared<std::mutex>();  
std::thread t3([compteur, mutex]() {  
    std::lock_guard lock(*mutex);
    (*compteur)++;
});
```

De même, un **même `shared_ptr`** (pas une copie, le même objet) ne doit pas être lu et modifié simultanément depuis plusieurs threads :

```cpp
std::shared_ptr<int> global = std::make_shared<int>(42);

// ❌ DATA RACE — modification et lecture du même shared_ptr
std::thread t1([&]() { global = std::make_shared<int>(100); });  // Écriture  
std::thread t2([&]() { auto local = global; });                   // Lecture  

// ✅ Chaque thread utilise sa propre copie
auto copie1 = global;  
auto copie2 = global;  
std::thread t3([copie1]() { /* ... */ });  // Copie indépendante  
std::thread t4([copie2]() { /* ... */ });  // Copie indépendante  
```

> Pour un traitement approfondi de la concurrence, voir [chapitre 21](/21-threads-concurrence/README.md).

### Coût des opérations atomiques

Sur x86_64, une incrémentation/décrémentation atomique coûte typiquement **5 à 40 fois plus** qu'une opération non-atomique simple, selon la contention entre les cœurs CPU. Ce coût est négligeable pour la plupart des applications, mais peut devenir significatif dans les scénarios suivants :

- Boucles serrées qui copient des `shared_ptr` des millions de fois par seconde.
- Structures de données où chaque accès à un nœud passe par une copie de `shared_ptr`.
- Code haute performance où chaque nanoseconde compte (trading, moteurs de jeu, traitement temps réel).

Dans ces cas, les stratégies d'atténuation sont les suivantes, par ordre de préférence :

- **Utiliser `unique_ptr`** si la possession partagée n'est pas réellement nécessaire — c'est la solution la plus radicale et la plus efficace.
- **Passer par référence** (`const shared_ptr<T>&`) plutôt que par copie quand la fonction n'a pas besoin de prolonger la durée de vie.
- **Utiliser `std::move`** plutôt que la copie quand le `shared_ptr` source n'est plus nécessaire.
- **Passer un pointeur brut ou une référence** (`T&`, `T*`) quand la fonction n'a pas besoin de participer à la gestion de propriété — c'est la recommandation des C++ Core Guidelines.

```cpp
// ⚠️ Chaque appel copie le shared_ptr → opération atomique
void traiter_copie(std::shared_ptr<Data> d) { /* ... */ }

// ✅ Référence vers le shared_ptr → aucune opération atomique
void traiter_ref_shared(const std::shared_ptr<Data>& d) { /* ... */ }

// ✅✅ Encore mieux : la fonction n'a pas besoin de savoir que c'est un shared_ptr
void traiter_ref(const Data& d) { /* ... */ }

auto data = std::make_shared<Data>();  
traiter_ref(*data);  // Aucun surcoût lié au shared_ptr  
```

## use_count() en pratique : utile pour le debug, pas pour la logique

`use_count()` retourne le nombre actuel de propriétaires. C'est un outil de diagnostic précieux, mais il ne devrait **jamais** être utilisé pour piloter la logique du programme.

### Pourquoi ne pas se fier à use_count()

En environnement multi-thread, la valeur retournée par `use_count()` peut être **obsolète** au moment où vous l'utilisez : un autre thread peut avoir copié ou détruit un `shared_ptr` entre l'appel à `use_count()` et votre prise de décision.

```cpp
auto data = std::make_shared<Data>();

// ❌ Anti-pattern — race condition potentielle
if (data.use_count() == 1) {
    // "Je suis le seul propriétaire, je peux modifier librement"
    data->modifier();  // 💥 Un autre thread a pu copier data entre-temps
}
```

### Usage légitime : debug et tests

```cpp
auto ressource = std::make_shared<Ressource>();  
auto copie = ressource;  

// Assertion dans un test unitaire
assert(ressource.use_count() == 2);

// Log de diagnostic
std::print("[debug] Ressource partagée entre {} propriétaires\n",
           ressource.use_count());
```

### unique() — supprimé en C++20

La méthode `unique()` retournait `true` si `use_count() == 1`. Elle a été **supprimée en C++20** précisément parce qu'elle encourageait des raisonnements non thread-safe. Si votre code l'utilise, c'est un signal de modernisation.

```cpp
// ❌ Code pré-C++20 — supprimé
if (data.unique()) { /* ... */ }

// ✅ Si vous avez réellement besoin de cette information (debug uniquement)
if (data.use_count() == 1) { /* ... */ }
```

## Cycle de vie complet : exemple annoté

Voici un scénario complet qui trace chaque modification du compteur :

```cpp
#include <memory>

struct Sensor {
    std::string nom;
    Sensor(std::string n) : nom(std::move(n)) {
        std::print("[+] Sensor '{}' créé\n", nom);
    }
    ~Sensor() {
        std::print("[-] Sensor '{}' détruit\n", nom);
    }
};

int main() {
    std::print("=== Étape 1 : Création ===\n");
    auto s1 = std::make_shared<Sensor>("Température");
    std::print("use_count = {}\n\n", s1.use_count());
    // [+] Sensor 'Température' créé
    // use_count = 1

    std::print("=== Étape 2 : Copie ===\n");
    auto s2 = s1;
    std::print("use_count = {}\n\n", s1.use_count());
    // use_count = 2

    std::print("=== Étape 3 : Scope interne ===\n");
    {
        auto s3 = s1;
        std::print("use_count (dans le scope) = {}\n", s1.use_count());
        // use_count (dans le scope) = 3
    }
    std::print("use_count (après le scope) = {}\n\n", s1.use_count());
    // use_count (après le scope) = 2

    std::print("=== Étape 4 : Move ===\n");
    auto s4 = std::move(s2);
    std::print("s2 est nul ? {}\n", (s2 == nullptr));
    std::print("use_count = {}\n\n", s1.use_count());
    // s2 est nul ? true
    // use_count = 2  (inchangé — move ne touche pas le compteur)

    std::print("=== Étape 5 : Reset de s1 ===\n");
    s1.reset();
    std::print("s1 est nul ? {}\n", (s1 == nullptr));
    std::print("use_count via s4 = {}\n\n", s4.use_count());
    // s1 est nul ? true
    // use_count via s4 = 1

    std::print("=== Étape 6 : Fin de main ===\n");
    // s4 est détruit → use_count passe de 1 à 0
    // [-] Sensor 'Température' détruit
}
```

Ce programme illustre chaque nuance : la création initiale, l'effet de la copie, le scope comme mécanisme de durée de vie, la neutralité du `std::move` sur le compteur, et la destruction au moment exact où le dernier propriétaire disparaît.

## Résumé

| Concept | Détail |
|---|---|
| **Mécanisme** | Compteur partagé via un bloc de contrôle alloué sur le tas |
| **Incrémentation** | Copie d'un `shared_ptr`, `lock()` d'un `weak_ptr` |
| **Décrémentation** | Destruction, `reset()`, réaffectation |
| **Neutre** | `std::move`, `get()`, déréférencement, création de `weak_ptr` |
| **Libération de l'objet** | Quand strong count atteint 0 |
| **Libération du bloc de contrôle** | Quand strong count ET weak count atteignent 0 |
| **Thread-safety** | Le compteur est atomique. L'objet pointé ne l'est pas. |
| **Coût** | 5-40× plus cher qu'un incrément non-atomique. Négligeable sauf hot path. |
| **use_count()** | Debug et tests uniquement. Jamais pour la logique métier. |
