# 3.3.2 — `reinterpret_cast` : Réinterprétation mémoire

> **Chapitre 3 — Types, Variables et Opérateurs** · Section 3.3 · Sous-section 2 sur 4  
> Prérequis : [3.3.1 — `static_cast` : Conversions sûres](/03-types-variables-operateurs/03.1-static-cast.md)

---

## Introduction

Si `static_cast` est le cast « poli » du C++ — celui qui respecte le système de types et refuse les conversions incohérentes — alors `reinterpret_cast` est son opposé radical. Il dit au compilateur : **« je sais ce que je fais, prends ces bits et interprète-les comme un autre type, sans poser de questions »**.

`reinterpret_cast` ne transforme pas la valeur, ne la convertit pas, ne l'adapte pas. Il prend la représentation binaire brute d'un objet et la réinterprète comme si elle appartenait à un type complètement différent. C'est l'opération la plus proche de la manipulation directe de la mémoire que le C++ offre dans son système de casts, et c'est aussi la plus dangereuse.

Son existence se justifie par les besoins de la **programmation système** : interaction avec le matériel, parsing de protocoles réseau, interfaçage avec des API C bas niveau, implémentation d'allocateurs mémoire. Dans du code applicatif ordinaire, sa présence devrait être exceptionnelle et toujours accompagnée d'un commentaire expliquant pourquoi aucune alternative plus sûre n'est possible.

---

## Syntaxe

```cpp
reinterpret_cast<TypeCible>(expression)
```

Comme tous les casts nommés C++, la syntaxe est volontairement verbeuse. Quand vous voyez `reinterpret_cast` dans du code, c'est un **signal d'alerte** qui appelle une inspection attentive.

---

## Ce que `reinterpret_cast` fait réellement

Contrairement à une idée reçue, `reinterpret_cast` ne copie pas les bits d'un objet vers un autre. Il change uniquement la manière dont le **compilateur interprète** un pointeur ou une référence. La représentation mémoire sous-jacente n'est pas modifiée.

Plus précisément, `reinterpret_cast` effectue une **correspondance d'adresse** (*mapping*) : il prend l'adresse contenue dans un pointeur et la traite comme si elle pointait vers un objet d'un type différent. Le standard C++ ne garantit pas grand-chose sur le résultat, à l'exception d'un aller-retour : si vous convertissez un `T*` en `U*` puis de nouveau en `T*`, le pointeur final est garanti d'être identique à l'original.

---

## Cas d'usage légitimes

### Inspection des octets d'un objet

Le cas d'usage le plus courant et le plus sûr de `reinterpret_cast` est la conversion d'un pointeur vers `char*` (ou `unsigned char*`, ou `std::byte*`) pour inspecter la représentation mémoire brute d'un objet :

```cpp
#include <cstdint>
#include <print>

uint32_t value = 0xDEADBEEF;  
auto* bytes = reinterpret_cast<unsigned char*>(&value);  

for (std::size_t i = 0; i < sizeof(value); ++i) {
    std::print("{:02X} ", bytes[i]);
}
// Sur une machine little-endian : EF BE AD DE
```

Ce cas est **explicitement autorisé** par le standard : tout objet peut être examiné comme une séquence de `char`, `unsigned char` ou `std::byte`. C'est la base de la sérialisation binaire, du logging hexadécimal et de l'inspection mémoire dans les débogueurs.

### Interaction avec les API matérielles et système

Les registres matériels sont souvent mappés en mémoire à des adresses physiques connues. `reinterpret_cast` permet de convertir une adresse entière en pointeur :

```cpp
// Programmation embarquée : accès à un registre matériel
// (cet exemple est illustratif — les adresses varient selon le matériel)
volatile uint32_t* gpio_register =
    reinterpret_cast<volatile uint32_t*>(0x4002'0000);

*gpio_register |= (1 << 5); // Activer le bit 5 du registre GPIO
```

La conversion `uintptr_t` → pointeur est l'un des rares cas où `reinterpret_cast` est incontournable. Notez l'utilisation de `volatile` : elle empêche le compilateur d'optimiser (supprimer ou réordonner) les accès au registre, car chaque lecture et écriture a un effet matériel observable.

### Conversion de pointeurs pour l'interopérabilité C

Certaines API C utilisent des types de pointeurs qui ne correspondent pas exactement aux types C++ de votre code. `reinterpret_cast` est parfois nécessaire pour franchir cette barrière :

```cpp
// L'API réseau POSIX utilise struct sockaddr* comme type générique
struct sockaddr_in addr;  
addr.sin_family = AF_INET;  
addr.sin_port = htons(8080);  

// bind() attend un struct sockaddr*, pas un struct sockaddr_in*
int result = bind(sockfd,
    reinterpret_cast<struct sockaddr*>(&addr),
    sizeof(addr));
```

Ce pattern est omniprésent dans la programmation réseau POSIX, où `sockaddr` est une « classe de base » simulée en C via des structures partageant un préfixe commun. Le `reinterpret_cast` est ici un mal nécessaire dicté par la conception de l'API.

### Conversion pointeur ↔ entier

`reinterpret_cast` permet de convertir un pointeur vers un entier suffisamment grand (`uintptr_t`) et inversement :

```cpp
#include <cstdint>

int x = 42;  
uintptr_t addr = reinterpret_cast<uintptr_t>(&x);  
std::print("Adresse de x : 0x{:X}\n", addr);  

int* ptr = reinterpret_cast<int*>(addr);  
std::print("Valeur via ptr : {}\n", *ptr); // 42  
```

`uintptr_t` (défini dans `<cstdint>`) est un type entier non signé garanti de pouvoir contenir un pointeur. Cette conversion est utile pour le hachage de pointeurs, le logging d'adresses ou le passage de pointeurs via des API C qui n'acceptent que des entiers (certains mécanismes de callback legacy).

> ⚠️ La conversion dans les deux sens est définie uniquement si le type entier est `uintptr_t` ou `intptr_t`. Utiliser un `int` ou un `long` ordinaire peut tronquer l'adresse sur les plateformes 64 bits.

---

## La règle d'aliasing strict (*strict aliasing rule*)

`reinterpret_cast` est indissociable de la **règle d'aliasing strict**, l'une des règles les plus mal comprises du C++. Cette règle stipule que l'on ne peut pas accéder à un objet via un pointeur d'un type incompatible :

```cpp
float f = 3.14f;  
int* ip = reinterpret_cast<int*>(&f);  
int bits = *ip; // ❌ Comportement indéfini : violation de strict aliasing  
```

Ce code semble « fonctionner » sur la plupart des compilateurs en mode debug, mais il est formellement un **comportement indéfini**. Le compilateur est en droit de supposer qu'un `int*` et un `float*` ne pointent jamais vers la même adresse mémoire, et il peut optimiser le code en conséquence — par exemple, en retirant la lecture via `ip` parce qu'il « sait » qu'aucun `int` n'a été modifié.

### Types autorisés pour l'aliasing

Le standard définit une liste restreinte de types via lesquels un objet peut être accédé sans violer l'aliasing :

- Le **type réel** de l'objet (ou une version qualifiée avec `const`/`volatile`).
- Un type `char`, `unsigned char` ou `std::byte` — c'est pourquoi l'inspection octet par octet fonctionne toujours.
- Un type **compatible** au sens du standard (types signés/non signés correspondants, etc.).

En dehors de ces cas, accéder à un objet via un pointeur d'un type différent est un comportement indéfini, même si la taille et l'alignement correspondent.

### L'alternative sûre : `std::memcpy`

Si vous avez besoin de réinterpréter les bits d'une valeur comme un autre type (par exemple, lire les bits IEEE 754 d'un `float` comme un `uint32_t`), la méthode **sûre et portable** est `std::memcpy` :

```cpp
#include <cstring>
#include <cstdint>

float f = 3.14f;  
uint32_t bits;  
std::memcpy(&bits, &f, sizeof(bits)); // ✅ Défini, portable, optimisé  

std::print("Bits de 3.14f : 0x{:08X}\n", bits); // 0x4048F5C3
```

`std::memcpy` copie les octets bruts sans violer l'aliasing. Les compilateurs modernes reconnaissent ce pattern et le compilent généralement en une seule instruction `mov` — il n'y a aucune pénalité de performance par rapport au `reinterpret_cast` illégal.

Depuis C++20, `std::bit_cast` offre une alternative encore plus élégante :

```cpp
#include <bit>

float f = 3.14f;  
auto bits = std::bit_cast<uint32_t>(f); // ✅ C++20 — type-safe et constexpr  

std::print("Bits de 3.14f : 0x{:08X}\n", bits); // 0x4048F5C3
```

`std::bit_cast` est la méthode recommandée en C++ moderne pour la réinterprétation de bits entre types de même taille. Il est `constexpr`, type-safe (il refuse les conversions entre types de tailles différentes), et il ne viole pas l'aliasing. C'est le **remplacement officiel** du pattern `reinterpret_cast` + déréférencement pour la réinterprétation de valeurs.

---

## Ce que `reinterpret_cast` ne peut pas faire

Malgré sa permissivité, `reinterpret_cast` a ses limites :

**Il ne peut pas retirer `const` ou `volatile`.** C'est le domaine exclusif de `const_cast` :

```cpp
const int x = 42;  
int* p = reinterpret_cast<int*>(&x); // ❌ Erreur : ne retire pas const  
```

**Il ne peut pas convertir entre pointeurs et types non-pointeurs** (à l'exception des entiers `uintptr_t`/`intptr_t`) :

```cpp
float f = 3.14f;  
int n = reinterpret_cast<int>(f); // ❌ Erreur : float n'est pas un pointeur  
// Utilisez static_cast<int>(f) pour une conversion numérique
// Utilisez std::bit_cast<int>(f) pour une réinterprétation de bits
```

**Il ne peut pas effectuer de conversion entre pointeur de fonction et pointeur d'objet** de manière portable. La norme autorise cette conversion comme une extension conditionnelle, mais elle n'est pas garantie sur toutes les plateformes.

---

## `reinterpret_cast` et l'optimisation du compilateur

Un aspect souvent sous-estimé de `reinterpret_cast` est son interaction avec les **optimisations du compilateur**. Les compilateurs modernes (GCC, Clang) activent par défaut l'optimisation basée sur l'aliasing strict (option `-fstrict-aliasing`, incluse dans `-O2` et au-delà).

Cela signifie que le compilateur **suppose** que les pointeurs de types incompatibles ne partagent pas la même mémoire. S'il détecte deux pointeurs de types différents, il peut les traiter comme indépendants et réordonner ou éliminer des accès mémoire :

```cpp
void process(int* ip, float* fp) {
    *ip = 42;
    *fp = 3.14f;
    std::print("{}\n", *ip);
    // Le compilateur peut supposer que *fp n'a pas modifié *ip
    // et optimiser en imprimant directement 42 sans relire la mémoire
}
```

Si `ip` et `fp` pointent en réalité vers la même adresse (via un `reinterpret_cast`), l'optimiseur produira un résultat incorrect sans aucun avertissement. C'est pourquoi la violation de l'aliasing strict est si pernicieuse : le bug peut apparaître uniquement en mode optimisé, pas en mode debug.

> 💡 L'option `-fno-strict-aliasing` désactive cette optimisation et rend les violations d'aliasing « sûres » en pratique. Le noyau Linux compile avec cette option, car son code C legacy repose massivement sur le type punning via pointeurs. En C++ moderne, préférez corriger le code plutôt que désactiver l'optimisation.

---

## Quand `reinterpret_cast` est-il justifié ?

En résumé, `reinterpret_cast` est justifié dans un nombre limité de scénarios :

| Scénario | Exemple | Justification |
|---|---|---|
| Inspection octet par octet | `reinterpret_cast<unsigned char*>(&obj)` | Explicitement autorisé par le standard |
| Adresse physique → pointeur | `reinterpret_cast<volatile uint32_t*>(0x4002'0000)` | Programmation embarquée/matérielle |
| API C avec types opaques | `reinterpret_cast<struct sockaddr*>(&addr_in)` | Imposé par la conception de l'API |
| Pointeur ↔ `uintptr_t` | `reinterpret_cast<uintptr_t>(ptr)` | Hachage, logging, callbacks legacy |
| Aller-retour de type | `T*` → `U*` → `T*` | Stockage temporaire dans un type générique |

Pour la **réinterprétation de valeurs** (lire les bits d'un `float` comme un `int`), utilisez `std::bit_cast` (C++20) ou `std::memcpy` — jamais `reinterpret_cast` suivi d'un déréférencement.

## Bonnes pratiques

**Considérez `reinterpret_cast` comme un signal d'alerte.** Chaque occurrence devrait être accompagnée d'un commentaire expliquant pourquoi elle est nécessaire et pourquoi `static_cast` ou `std::bit_cast` ne suffisent pas.

**Préférez toujours `std::bit_cast` pour la réinterprétation de bits entre valeurs.** C'est type-safe, `constexpr`, et ne viole pas l'aliasing. Si vous ciblez un standard antérieur à C++20, utilisez `std::memcpy`.

**Limitez la portée du code utilisant `reinterpret_cast`.** Encapsulez les conversions dangereuses dans des fonctions utilitaires bien nommées et testées, plutôt que de les disperser dans le code applicatif :

```cpp
// Fonction utilitaire encapsulant le reinterpret_cast
inline const struct sockaddr* to_sockaddr(const struct sockaddr_in* addr) {
    return reinterpret_cast<const struct sockaddr*>(addr);
}

// Utilisation propre dans le code applicatif
bind(sockfd, to_sockaddr(&addr), sizeof(addr));
```

**Ne désactivez pas `-fstrict-aliasing` pour « corriger » un bug.** Si votre code produit des résultats différents en `-O0` et `-O2`, c'est probablement une violation d'aliasing. Corrigez le code en utilisant `std::memcpy` ou `std::bit_cast` plutôt que de contourner l'optimiseur.

**En revue de code, questionnez systématiquement chaque `reinterpret_cast`.** Demandez : « est-ce que `static_cast` ne suffirait pas ? », « est-ce qu'on viole l'aliasing ? », « est-ce qu'un `std::bit_cast` ferait l'affaire ? ». Dans la majorité des cas, une alternative plus sûre existe.

---

## En résumé

`reinterpret_cast` est l'outil de dernier recours du système de casts C++. Il ne convertit rien — il réinterprète des bits bruts, en contournant le système de types. Son usage est légitime dans la programmation système, l'embarqué et l'interopérabilité avec des API C bas niveau, mais il est dangereux dans le code applicatif à cause de la règle d'aliasing strict. Le C++ moderne offre des alternatives plus sûres (`std::bit_cast`, `std::memcpy`) qui couvrent la plupart des cas où `reinterpret_cast` était autrefois la seule option. Quand vous le croisez dans du code, traitez-le comme un panneau « attention danger » et vérifiez qu'aucune alternative n'est possible.
