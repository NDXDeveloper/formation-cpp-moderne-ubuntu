# 3.1.1 — `auto` : Déduction automatique du type

> **Chapitre 3 — Types, Variables et Opérateurs** · Section 3.1 · Sous-section 1 sur 3  
> Prérequis : [3.1 — Le typage statique fort et l'inférence de type](/03-types-variables-operateurs/01-typage-statique-inference.md)

---

## Introduction

Le mot-clé `auto` est probablement la fonctionnalité C++11 qui a le plus transformé la manière d'écrire du C++ au quotidien. Son principe est simple : au lieu d'écrire explicitement le type d'une variable, on demande au compilateur de le **déduire** à partir de l'expression d'initialisation.

```cpp
auto x = 42;          // int  
auto pi = 3.14;       // double  
auto name = "hello"s; // std::string (avec le suffixe littéral s)  
```

Le type n'est pas « absent » — il est déduit, vérifié et fixé à la compilation exactement comme si vous l'aviez écrit vous-même. `auto` est un raccourci d'écriture, pas un passage vers le typage dynamique.

Cette sous-section couvre les règles de déduction de `auto`, ses interactions avec les qualificateurs et les références, les pièges courants, et les contextes dans lesquels `auto` peut (et ne peut pas) être utilisé.

---

## Syntaxe de base

La forme la plus simple est la déclaration d'une variable locale avec un initialiseur :

```cpp
auto count = 0;              // int  
auto ratio = 1.5;            // double  
auto flag = true;            // bool  
auto letter = 'A';           // char  
auto message = "bonjour";    // const char* (attention, pas std::string)  
```

La règle fondamentale est la suivante : **`auto` déduit le type exactement comme le ferait un template de fonction pour son paramètre.** Comprendre cette analogie permet de prédire le comportement de `auto` dans tous les cas de figure.

Considérons une fonction template :

```cpp
template <typename T>  
void f(T param);  
```

Lorsqu'on appelle `f(expr)`, le compilateur déduit `T` à partir du type de `expr`. La déduction de `auto` suit exactement les mêmes règles : dans `auto var = expr;`, le compilateur déduit le type de `var` comme il déduirait `T` dans l'appel `f(expr)`.

---

## Règles de déduction

### Déduction de base : les qualificateurs de haut niveau sont retirés

Lorsque `auto` est utilisé seul (sans `&` ni `&&`), les qualificateurs **de haut niveau** (*top-level*) sont retirés. Cela signifie que `const` et `volatile` appliqués directement à l'objet source ne sont pas propagés :

```cpp
const int cx = 10;  
auto a = cx;          // a est int, pas const int  
                      // (la copie est indépendante de l'original)

volatile int vx = 20;  
auto b = vx;          // b est int, pas volatile int  
```

Ce comportement est logique : `a` est une **copie** de `cx`. Modifier `a` ne modifie pas `cx`, donc il n'y a aucune raison d'imposer la constance sur la copie. C'est exactement le même raisonnement qu'un passage de paramètre par valeur dans un template.

### Déduction avec référence : `auto&`

Si vous ajoutez `&` après `auto`, vous déclarez une **référence**. Dans ce cas, les qualificateurs de l'objet référencé sont **préservés** :

```cpp
const int cx = 10;

auto& ref = cx;       // ref est const int& (pas int&)
                       // Le const est préservé, car ref pointe vers cx
```

Le compilateur ne peut pas retirer `const` ici — vous référencez directement `cx`, et permettre une modification via `ref` violerait la promesse de constance de `cx`. Tenter de forcer un `auto&` non-const sur un objet `const` est une erreur de compilation :

```cpp
// auto& ref2 = cx;  // ref2 serait const int&, pas int&
ref = 99;             // ❌ Erreur : assignment of read-only reference
```

### Déduction avec référence universelle : `auto&&`

La syntaxe `auto&&` ne signifie pas « référence rvalue ». C'est une **référence universelle** (*forwarding reference*) qui se comporte différemment selon la catégorie de valeur de l'expression :

```cpp
int x = 10;  
const int cx = 20;  

auto&& r1 = x;       // x est une lvalue → r1 est int&  
auto&& r2 = cx;      // cx est une lvalue const → r2 est const int&  
auto&& r3 = 42;      // 42 est une rvalue → r3 est int&&  
```

Les références universelles sont un sujet avancé lié à la sémantique de mouvement (chapitre 10) et au *perfect forwarding*. Pour l'instant, retenez que `auto&&` est la forme la plus « flexible » : elle se lie à tout type d'expression sans copie.

### Récapitulatif des règles

| Déclaration | Expression source | Type déduit |
|---|---|---|
| `auto a = cx;` | `const int cx` | `int` (copie, const retiré) |
| `auto& b = cx;` | `const int cx` | `const int&` (const préservé) |
| `auto&& c = cx;` | `const int cx` (lvalue) | `const int&` |
| `auto&& d = 42;` | `42` (rvalue) | `int&&` |
| `auto e = &cx;` | `const int*` | `const int*` (pointeur copié) |

## `auto` et les initialiseurs par accolades

C'est le piège le plus célèbre de `auto` et il mérite une attention particulière. En C++11, `auto` avec des accolades déduisait un `std::initializer_list` :

```cpp
// C++11/C++14
auto x = {1, 2, 3};   // std::initializer_list<int>  — souvent inattendu
```

C++17 a clarifié et restreint ce comportement. Les règles actuelles (C++17 et au-delà) sont les suivantes :

```cpp
auto a = {1, 2, 3};   // std::initializer_list<int> — copie avec =  
auto b{42};            // int — initialisation directe avec un seul élément  
// auto c{1, 2, 3};    // ❌ Erreur en C++17+ : initialisation directe
                       //    avec plusieurs éléments n'est plus autorisée
```

La distinction importante est entre l'**initialisation par copie** (`auto x = {...}`) et l'**initialisation directe** (`auto x{...}`). Avec l'initialisation directe et un seul élément, `auto` déduit le type de cet élément. Avec l'initialisation par copie et des accolades, `auto` déduit un `std::initializer_list`.

**Recommandation pratique** : pour les types simples, préférez l'initialisation avec `=` qui ne surprend personne :

```cpp
auto count = 0;        // int — clair et sans ambiguïté  
auto ratio = 1.5;      // double — idem  
```

Réservez l'initialisation par accolades aux cas où vous souhaitez explicitement bénéficier de la protection contre les conversions restrictives, en écrivant alors le type explicitement :

```cpp
int count{0};          // Protection contre narrowing, type explicite
```

## Contextes d'utilisation de `auto`

### Variables locales

C'est le cas d'usage le plus fréquent et le plus naturel :

```cpp
auto config = load_configuration("/etc/app.conf");  
auto connection = database.connect(credentials);  
auto tokens = tokenize(input_line);  
```

Dans chacun de ces cas, le type est déterminé par la valeur de retour de la fonction appelée. Le code est concis et s'adapte automatiquement si le type de retour de la fonction évolue.

### Boucles range-based `for`

`auto` est quasi systématiquement utilisé dans les boucles range-based. Trois formes sont courantes :

```cpp
std::vector<std::string> names = {"Alice", "Bob", "Charlie"};

// Copie de chaque élément (rarement souhaité pour des objets lourds)
for (auto name : names) { /* ... */ }

// Référence constante (lecture seule — forme la plus courante)
for (const auto& name : names) { /* ... */ }

// Référence mutable (modification des éléments)
for (auto& name : names) { /* ... */ }
```

La forme `const auto&` est le choix par défaut recommandé pour itérer sur un conteneur sans le modifier. Elle évite les copies inutiles tout en exprimant clairement l'intention de lecture seule.

### Type de retour de fonctions

Depuis C++14, `auto` peut être utilisé comme type de retour d'une fonction. Le compilateur déduit le type à partir des instructions `return` :

```cpp
auto add(int a, int b) {
    return a + b; // Le type de retour est déduit comme int
}
```

Cette forme est utile pour les petites fonctions utilitaires et les lambdas, mais elle peut nuire à la lisibilité des API publiques. Pour une fonction faisant partie d'une interface (déclarée dans un header), un type de retour explicite est généralement préférable, car il documente le contrat de la fonction sans obliger le lecteur à inspecter le corps :

```cpp
// Dans un header public : type de retour explicite
int add(int a, int b);
```

### Trailing return type

C++11 a introduit la syntaxe de type de retour « en queue » (*trailing return type*), souvent combinée avec `auto` :

```cpp
auto add(int a, int b) -> int {
    return a + b;
}
```

Cette syntaxe peut sembler redondante pour un cas aussi simple, mais elle devient indispensable dans les templates où le type de retour dépend des paramètres :

```cpp
template <typename T, typename U>  
auto multiply(T a, U b) -> decltype(a * b) {  
    return a * b;
}
```

Ici, `decltype(a * b)` ne pourrait pas être écrit avant la liste des paramètres, car `a` et `b` ne seraient pas encore déclarés. Le trailing return type résout ce problème de portée. Depuis C++14, la déduction automatique du type de retour rend souvent cette syntaxe inutile pour les fonctions non-template, mais elle reste courante dans le code générique.

### Paramètres de lambdas (C++14) et de fonctions (C++20)

Depuis C++14, les paramètres de lambdas peuvent utiliser `auto`, créant un lambda **générique** (équivalent à un template) :

```cpp
auto square = [](auto x) { return x * x; };

std::print("{}\n", square(5));     // 25 (int)  
std::print("{}\n", square(2.5));   // 6.25 (double)  
```

C++20 a étendu cette possibilité aux fonctions ordinaires avec les **paramètres `auto` abrégés** (*abbreviated function templates*) :

```cpp
// C++20 : équivalent à un template<typename T> void print_value(T value)
void print_value(auto value) {
    std::print("{}\n", value);
}
```

Cette syntaxe est un raccourci pour un template de fonction. Chaque paramètre `auto` introduit un paramètre de template implicite distinct.

## Pièges courants

### Piège n°1 : les littéraux chaîne ne sont pas des `std::string`

```cpp
auto greeting = "hello"; // const char*, pas std::string !
```

Les littéraux chaîne entre guillemets doubles sont de type `const char[]`, qui se dégrade en `const char*`. Pour obtenir un `std::string`, utilisez le suffixe littéral `s` défini dans `<string>` :

```cpp
using namespace std::string_literals;  
auto greeting = "hello"s; // std::string  
```

### Piège n°2 : les expressions avec des proxy objects

Certaines classes de la bibliothèque standard retournent des objets **proxy** dont le type n'est pas celui auquel on s'attend. Le cas le plus connu est `std::vector<bool>` :

```cpp
std::vector<bool> flags = {true, false, true};  
auto flag = flags[0]; // ⚠️ Pas bool ! C'est un std::vector<bool>::reference  
```

`std::vector<bool>` est une spécialisation qui stocke les booléens sous forme de bits individuels. L'opérateur `[]` ne peut pas retourner une référence vers un bit unique, il retourne donc un objet proxy. Avec `auto`, vous capturez ce proxy au lieu d'un `bool`. Si le vecteur est ensuite détruit ou réalloué, le proxy devient invalide — c'est un dangling reference déguisé.

La solution est d'utiliser un type explicite ou un `static_cast` :

```cpp
bool flag = flags[0];                  // Conversion explicite vers bool  
auto flag2 = static_cast<bool>(flags[0]); // Idem avec auto  
```

Ce piège s'applique également à d'autres proxy objects, comme les expressions de la bibliothèque Eigen (algèbre linéaire) ou certains retours de `std::bitset`.

### Piège n°3 : `auto` avec des entiers non signés

```cpp
std::vector<int> v = {10, 20, 30};  
auto size = v.size(); // std::vector<int>::size_type, alias de std::size_t  
                      // C'est un entier NON SIGNÉ
```

Si vous comparez ensuite `size` avec un entier signé, le compilateur émet un avertissement (comparaison signed/unsigned). Ce n'est pas un bug de `auto` — le type était déjà non signé — mais `auto` le rend moins visible. Soyez attentif aux avertissements du compilateur, en particulier avec `-Wall -Wextra`.

### Piège n°4 : oublier que `auto` copie par défaut

```cpp
std::map<std::string, int> scores = {{"Alice", 95}, {"Bob", 87}};

for (auto pair : scores) {
    // pair est une COPIE de std::pair<const std::string, int>
    // Chaque itération copie la clé (std::string) — coûteux !
}

for (const auto& pair : scores) {
    // Référence constante — aucune copie, aucune allocation
}
```

`auto` sans `&` crée toujours une copie. Pour des types légers comme `int` ou `double`, c'est négligeable. Pour des objets contenant des allocations dynamiques (`std::string`, `std::vector`, `std::map`…), la copie a un coût réel. La forme `const auto&` devrait être votre réflexe par défaut dans les boucles range-based.

## Structured bindings et `auto` (C++17)

C++17 a étendu `auto` avec les **structured bindings**, qui permettent de décomposer un objet composite en nommant directement ses éléments :

```cpp
std::map<std::string, int> scores = {{"Alice", 95}, {"Bob", 87}};

for (const auto& [name, score] : scores) {
    std::print("{} : {}\n", name, score);
}
```

Ici, `name` est lié à la clé (`const std::string&`) et `score` à la valeur (`const int&`). Les structured bindings fonctionnent avec les paires, les tuples, les tableaux C, et toute structure dont les membres sont publics. Ils seront couverts en détail dans la section 12.1.

## Limites de `auto`

`auto` ne peut pas être utilisé partout. Voici les contextes où il n'est pas autorisé ou n'est pas approprié :

**Membres de classe non-statiques.** On ne peut pas déclarer un attribut de classe avec `auto`, car le compilateur a besoin de connaître la taille de la classe au moment de sa définition, et un membre `auto` sans initialiseur ne fournit pas cette information :

```cpp
class Widget {
    auto value = 42; // ❌ Erreur : auto non autorisé pour les membres non-statiques
    static inline auto max_size = 1024; // ✅ OK pour les membres statiques inline
};
```

**Paramètres de fonctions (avant C++20).** Avant C++20, seuls les lambdas pouvaient avoir des paramètres `auto`. Depuis C++20, les fonctions ordinaires le peuvent aussi (abbreviated function templates), comme vu plus haut.

**Déclarations sans initialiseur.** `auto` a besoin d'une expression pour déduire le type. Sans initialiseur, la déduction est impossible :

```cpp
auto x; // ❌ Erreur : impossible de déduire le type sans initialiseur
```

## Bonnes pratiques

**Utilisez `auto` lorsque le type est évident d'après le contexte.** C'est le cas avec les appels à `new`, `make_unique`, `make_shared`, les casts explicites, et les fonctions dont le nom révèle le type de retour :

```cpp
auto ptr = std::make_unique<Connection>(host, port); // Évidemment unique_ptr<Connection>  
auto count = static_cast<int>(raw_value);             // Évidemment int  
auto it = container.begin();                          // Évidemment un itérateur  
```

**Préférez un type explicite lorsque le type n'est pas déductible à la lecture.** Si le lecteur doit naviguer vers la définition d'une fonction pour comprendre ce que contient une variable, le type explicite est un meilleur choix :

```cpp
// Moins clair — quel type retourne process() ?
auto result = process(data);

// Plus clair — l'intention et le type sont visibles
ParseResult result = process(data);
```

**Utilisez `const auto&` comme forme par défaut dans les boucles range-based** pour éviter les copies involontaires et exprimer l'intention de lecture seule.

**Qualifiez `auto` avec `const` lorsque la variable ne doit pas être modifiée**, même s'il s'agit d'une copie :

```cpp
const auto max_retries = config.get_max_retries(); // Intention claire : valeur fixe
```

---

## En résumé

`auto` est un outil de **concision** qui préserve toute la rigueur du typage statique. Il ne supprime pas les types — il délègue leur écriture au compilateur. Utilisé judicieusement, il rend le code plus lisible, plus maintenable et plus résistant aux changements de types. Utilisé sans discernement, il peut masquer des intentions et introduire des surprises.

La règle d'or est pragmatique : **le code doit rester lisible par un humain qui ne connaît pas le contexte par cœur.** Si `auto` aide à atteindre cet objectif, utilisez-le. Sinon, écrivez le type.
