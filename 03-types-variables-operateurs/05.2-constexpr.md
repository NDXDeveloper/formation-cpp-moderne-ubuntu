# 3.5.2 — `constexpr` : Calcul à la compilation

> **Chapitre 3 — Types, Variables et Opérateurs** · Section 3.5 · Sous-section 2 sur 3  
> Prérequis : [3.5.1 — `const` : Immutabilité à l'exécution](/03-types-variables-operateurs/05.1-const.md)

---

## Introduction

`const` garantit qu'une valeur ne sera pas modifiée après son initialisation, mais il ne garantit pas que cette valeur est connue à la compilation. Or, dans de nombreuses situations, le C++ exige une valeur **connue au moment de la compilation** : taille d'un tableau C, paramètre de template, branche d'un `if constexpr`, argument d'un `static_assert`. C'est le domaine de `constexpr`.

Le mot-clé `constexpr` (introduit en C++11, considérablement étendu par la suite) indique au compilateur qu'une variable ou une fonction **peut** être évaluée à la compilation. « Peut » est le mot important : `constexpr` est un mécanisme **opportuniste**. Si tous les ingrédients sont disponibles à la compilation (arguments constants, opérations autorisées), le calcul a lieu au compile-time. Sinon, le code s'exécute normalement au runtime. Cette dualité fait de `constexpr` un outil remarquablement flexible.

---

## Variables `constexpr`

### Déclaration et garanties

Une variable `constexpr` doit être initialisée avec une **expression constante** — une expression dont la valeur peut être entièrement déterminée à la compilation :

```cpp
constexpr int max_size = 1024;          // ✅ Littéral entier  
constexpr double pi = 3.14159265358979; // ✅ Littéral flottant  
constexpr int doubled = max_size * 2;   // ✅ Opération sur des constexpr  
```

Une variable `constexpr` est **implicitement `const`** — il est inutile (et redondant) d'écrire `constexpr const` :

```cpp
constexpr const int x = 42; // Fonctionne mais le const est redondant  
constexpr int y = 42;       // Identique et plus concis  
```

### Différence concrète avec `const`

La distinction entre `const` et `constexpr` se manifeste quand on tente d'utiliser la variable dans un contexte qui exige une constante de compilation :

```cpp
const int runtime_val = get_user_input(); // Valeur inconnue à la compilation  
constexpr int compile_val = 42;           // Valeur garantie à la compilation  

int arr1[compile_val];  // ✅ constexpr → utilisable comme taille de tableau
// int arr2[runtime_val]; // ❌ const mais pas constexpr → taille non constante

static_assert(compile_val == 42);  // ✅ Vérification à la compilation
// static_assert(runtime_val > 0); // ❌ Impossible : valeur inconnue à la compilation

template <int N>  
struct Buffer { char data[N]; };  

Buffer<compile_val> buf;  // ✅ Paramètre de template
// Buffer<runtime_val> bad; // ❌ Paramètre de template non constant
```

La règle est simple : **`constexpr` est obligatoire quand la valeur doit être disponible à la compilation**. `const` suffit quand on veut uniquement empêcher la modification.

### Cas particulier : `const` avec un littéral

Un cas subtil : une variable `const` initialisée avec un **littéral entier** est traitée comme une expression constante par le compilateur, même sans `constexpr` :

```cpp
const int size = 100;  
int arr[size]; // ✅ Fonctionne — le compilateur sait que size vaut 100  
```

Ce comportement existe pour des raisons de compatibilité historique. Cependant, il ne fonctionne pas dans tous les contextes (notamment les templates dans certaines situations), et il ne s'applique pas aux types flottants. La recommandation est d'utiliser `constexpr` explicitement quand vous avez besoin d'une constante de compilation — l'intention est plus claire et les garanties sont plus fortes.

## Fonctions `constexpr`

La véritable puissance de `constexpr` se révèle avec les fonctions. Une fonction `constexpr` est une fonction que le compilateur **peut** évaluer à la compilation si elle est appelée avec des arguments constants :

```cpp
constexpr int factorial(int n) {
    int result = 1;
    for (int i = 2; i <= n; ++i) {
        result *= i;
    }
    return result;
}

// Appel avec un argument constant → évalué à la compilation
constexpr int fact10 = factorial(10); // fact10 vaut 3'628'800 — calculé au compile-time

// Appel avec un argument runtime → évalué à l'exécution
int n = get_user_input();  
int fact_n = factorial(n); // Calculé au runtime — parfaitement valide  
```

La même fonction sert dans les deux contextes. C'est ce qui distingue `constexpr` de `consteval` (sous-section suivante) : `constexpr` est **dual**, `consteval` est exclusivement compile-time.

### Comment savoir si un appel est évalué au compile-time ?

Le compilateur évalue une fonction `constexpr` au compile-time **uniquement** si le résultat est utilisé dans un contexte qui l'exige : initialisation d'une variable `constexpr`, paramètre de template, taille de tableau, `static_assert`, ou branche de `if constexpr`.

Dans les autres cas, le compilateur **peut** évaluer au compile-time (optimisation), mais il n'y est pas obligé :

```cpp
constexpr int a = factorial(5);  // Obligatoirement compile-time (variable constexpr)  
int b = factorial(5);            // Probablement compile-time (optimisation), pas garanti  
const int c = factorial(5);      // Probablement compile-time, pas garanti non plus  
```

Pour **forcer** l'évaluation à la compilation sans déclarer une variable `constexpr`, vous pouvez utiliser `static_assert` comme vérification ou assigner le résultat à une variable `constexpr` :

```cpp
constexpr auto result = factorial(12); // Garanti compile-time  
static_assert(factorial(5) == 120);    // Garanti compile-time  
```

## Évolution des règles : de C++11 à C++23

Les restrictions sur les fonctions `constexpr` ont été drastiquement réduites au fil des standards. Comprendre cette évolution est utile pour savoir ce qui est permis selon le standard que vous ciblez.

### C++11 : le minimum vital

En C++11, une fonction `constexpr` ne pouvait contenir qu'**une seule instruction `return`**. Pas de boucle, pas de variable locale, pas de `if/else` :

```cpp
// C++11 : seule forme autorisée
constexpr int factorial_11(int n) {
    return n <= 1 ? 1 : n * factorial_11(n - 1); // Récursion + ternaire obligatoire
}
```

Cette limitation rendait l'écriture de fonctions `constexpr` non triviales très artificielle — tout devait être exprimé en récursion et en opérateur ternaire.

### C++14 : les fonctions deviennent réellement utilisables

C++14 a levé la plupart des restrictions. Les fonctions `constexpr` peuvent désormais contenir des variables locales, des boucles, des `if/else`, des instructions multiples :

```cpp
// C++14 : boucles et variables locales autorisées
constexpr int factorial_14(int n) {
    int result = 1;
    for (int i = 2; i <= n; ++i) {
        result *= i;
    }
    return result;
}
```

C'est à partir de C++14 que `constexpr` est devenu un outil de programmation courant, pas seulement un gadget pour les experts en métaprogrammation.

### C++17 : `if constexpr`

C++17 a introduit `if constexpr`, une construction qui évalue une condition à la compilation et **élimine** la branche non prise du code compilé. C'est un outil de métaprogrammation puissant qui remplace de nombreux usages de SFINAE et de spécialisation de templates :

```cpp
template <typename T>  
std::string to_string(const T& value) {  
    if constexpr (std::is_arithmetic_v<T>) {
        return std::to_string(value);
    } else if constexpr (std::is_same_v<T, std::string>) {
        return value;
    } else {
        return "unknown";
    }
}
```

La branche non sélectionnée n'est même **pas compilée** — elle peut contenir du code invalide pour le type `T` courant sans provoquer d'erreur. C'est fondamentalement différent d'un `if` ordinaire, qui compile toutes les branches même si la condition est connue à la compilation.

### C++20 : allocation dynamique et conteneurs `constexpr`

C++20 a franchi un cap majeur en autorisant l'**allocation dynamique** (`new`/`delete`) dans les fonctions `constexpr`, à condition que toute mémoire allouée soit libérée avant la fin de l'évaluation. Cela a permis de rendre `std::vector` et `std::string` utilisables en contexte `constexpr` :

```cpp
constexpr int sum_of_squares(int n) {
    std::vector<int> squares; // Allocation dynamique à la compilation !
    for (int i = 1; i <= n; ++i) {
        squares.push_back(i * i);
    }
    int sum = 0;
    for (int s : squares) {
        sum += s;
    }
    return sum; // Le vecteur est détruit ici — mémoire libérée avant la fin
}

constexpr int result = sum_of_squares(10); // 385 — calculé à la compilation
```

La contrainte est que la mémoire dynamique ne peut pas « fuiter » hors de l'évaluation `constexpr` : un `std::vector` construit à la compilation ne peut pas être stocké dans une variable `constexpr` globale, car sa mémoire interne est allouée dynamiquement. Le résultat de la fonction doit être un type à taille fixe.

C++20 a aussi introduit `constexpr` pour les destructeurs virtuels et un grand nombre d'algorithmes de la STL (`std::sort`, `std::find`, etc.).

### C++23 : expansion continue

C++23 poursuit l'extension de `constexpr` à la bibliothèque standard. De plus en plus de fonctions de `<algorithm>`, `<numeric>`, `<memory>` et d'autres en-têtes sont marquées `constexpr`, rendant possible l'évaluation à la compilation de code de plus en plus complexe.

## `constexpr` sur les constructeurs et les classes

Les classes peuvent avoir des constructeurs `constexpr`, ce qui permet de créer des instances `constexpr` :

```cpp
class Point {
    double x_, y_;

public:
    constexpr Point(double x, double y) : x_(x), y_(y) {}
    constexpr double x() const { return x_; }
    constexpr double y() const { return y_; }
    constexpr double distance_to_origin() const {
        return x_ * x_ + y_ * y_; // Pas de std::sqrt — pas constexpr dans tous les contextes
    }
};

constexpr Point origin(0.0, 0.0);  
constexpr Point p(3.0, 4.0);  
constexpr double dist_sq = p.distance_to_origin(); // 25.0 — calculé à la compilation  

static_assert(dist_sq == 25.0); // ✅ Vérification compile-time
```

Pour qu'une classe soit utilisable en contexte `constexpr`, ses constructeurs, destructeurs et les méthodes appelées doivent tous être `constexpr`. Les types de la bibliothèque standard comme `std::pair`, `std::tuple`, `std::array` et `std::optional` sont `constexpr`-ready.

---

## `constinit` : l'initialisation sans la constance (C++20)

C++20 a introduit un troisième qualificateur, `constinit`, qui résout un problème différent de `const` et `constexpr`. Il garantit qu'une variable à durée de vie statique est **initialisée à la compilation**, sans pour autant la rendre `const` :

```cpp
constinit int request_counter = 0; // Initialisé à la compilation, mais mutable

void handle_request() {
    ++request_counter; // ✅ Modification autorisée — pas const
}
```

Le problème que `constinit` résout est le **static initialization order fiasco** : quand des variables globales dans des fichiers différents dépendent les unes des autres, l'ordre d'initialisation n'est pas garanti par le standard, ce qui peut provoquer des lectures de variables non encore initialisées. `constinit` élimine ce risque en garantissant que l'initialisation a lieu **avant** l'exécution du programme.

### `constinit` vs `constexpr` vs `const`

| Caractéristique | `const` | `constexpr` | `constinit` |
|---|---|---|---|
| Immutabilité | Oui | Oui | **Non** |
| Initialisation compile-time | Pas garanti | Oui | Oui |
| Applicable aux variables locales | Oui | Oui | **Non** (statiques uniquement) |
| Résout le static init order fiasco | Non | Oui (mais rend const) | **Oui** (sans rendre const) |

`constinit` est le bon choix quand vous avez une variable globale ou `static` qui doit être initialisée à la compilation pour des raisons de sécurité d'initialisation, mais qui sera modifiée par la suite.

## Cas d'usage de `constexpr`

### Constantes mathématiques et physiques

```cpp
constexpr double pi = 3.14159265358979323846;  
constexpr double e  = 2.71828182845904523536;  
constexpr double speed_of_light = 299'792'458.0; // m/s  
constexpr double planck = 6.62607015e-34;         // J·s  
```

> 💡 Depuis C++20, la bibliothèque standard fournit ces constantes dans `<numbers>` : `std::numbers::pi`, `std::numbers::e`, etc. Préférez-les aux définitions manuelles.

### Tables de lookup précalculées

```cpp
constexpr auto generate_squares() {
    std::array<int, 100> table{};
    for (int i = 0; i < 100; ++i) {
        table[i] = i * i;
    }
    return table;
}

constexpr auto squares = generate_squares(); // Table calculée à la compilation

// Accès en O(1) au runtime — aucun calcul
int sq = squares[42]; // 1764
```

La table `squares` est intégrée directement dans le segment de données du binaire, comme si elle avait été écrite manuellement. Le programme ne passe aucun cycle CPU à la calculer au démarrage.

### Dimensions et tailles de buffers

```cpp
constexpr std::size_t cache_line_size = 64;  
constexpr std::size_t page_size = 4096;  
constexpr std::size_t max_packet_size = 1500; // MTU Ethernet  

alignas(cache_line_size) char buffer[page_size];
```

### Validation au compile-time avec `static_assert`

```cpp
constexpr int version_major = 3;  
constexpr int version_minor = 2;  

static_assert(version_major >= 3, "Ce code nécessite la version 3+");  
static_assert(version_minor >= 0, "Version mineure invalide");  
```

### Fonctions utilitaires pures

Toute fonction pure (sans effet de bord, dont le résultat ne dépend que de ses arguments) est candidate à `constexpr` :

```cpp
constexpr int clamp(int value, int min, int max) {
    if (value < min) return min;
    if (value > max) return max;
    return value;
}

constexpr bool is_power_of_two(unsigned int n) {
    return n > 0 && (n & (n - 1)) == 0;
}

constexpr uint32_t rgb(uint8_t r, uint8_t g, uint8_t b) {
    return (static_cast<uint32_t>(r) << 16)
         | (static_cast<uint32_t>(g) << 8)
         | static_cast<uint32_t>(b);
}

constexpr auto red = rgb(255, 0, 0); // 0xFF0000 — calculé à la compilation
```

## Limites des fonctions `constexpr`

Bien que les restrictions aient été largement levées, certaines opérations restent interdites dans les fonctions `constexpr` :

- **Assembleur inline** (`asm`) — incompatible avec l'évaluation par le compilateur.
- **Allocation dynamique persistante** (C++20) — la mémoire allouée avec `new` doit être libérée avant la fin de l'évaluation constexpr. Un `std::vector` construit à la compilation ne peut pas « survivre » comme variable `constexpr` globale.
- **`reinterpret_cast`** — les réinterprétations de bits ne sont pas autorisées dans un contexte `constexpr` (utilisez `std::bit_cast` à la place).
- **Appels système et I/O** — `std::print`, lecture de fichiers, accès réseau : toute interaction avec l'environnement extérieur est impossible à la compilation.
- **Comportement indéfini** — le compilateur **détecte** les comportements indéfinis (dépassement d'entier signé, déréférencement de pointeur nul, etc.) dans les évaluations `constexpr` et les signale comme des erreurs de compilation. C'est un avantage inattendu : `constexpr` est un outil de détection de bugs.

Ce dernier point mérite d'être souligné. Si une fonction `constexpr` provoque un comportement indéfini avec des arguments donnés, le compilateur refuse l'évaluation compile-time et produit une erreur explicite :

```cpp
constexpr int divide(int a, int b) {
    return a / b;
}

constexpr int ok = divide(10, 2);   // ✅ → 5
// constexpr int ub = divide(10, 0); // ❌ Erreur de compilation : division par zéro détectée
```

Le même code avec des arguments runtime provoquerait un crash silencieux. `constexpr` transforme le comportement indéfini en erreur de compilation — un filet de sécurité précieux.

## Bonnes pratiques

**Marquez `constexpr` toute fonction pure qui peut l'être.** Si une fonction ne dépend que de ses arguments et ne produit pas d'effets de bord, elle est candidate. Le coût d'ajout de `constexpr` est nul (aucun changement de comportement au runtime), et le bénéfice est une évaluation potentiellement compile-time.

**Utilisez `constexpr` plutôt que `const` pour les constantes dont la valeur est connue à la compilation.** L'intention est plus claire et les garanties sont plus fortes :

```cpp
// Préférez ceci :
constexpr int max_retries = 5;

// À ceci :
const int max_retries = 5;
```

**Utilisez `constexpr` pour les tables de lookup et les données précalculées.** C'est l'un des gains de performance les plus faciles à obtenir : le calcul est déplacé du runtime au compile-time sans aucune perte de lisibilité.

**Utilisez `static_assert` pour valider vos hypothèses sur les valeurs `constexpr`.** C'est une forme de test unitaire à la compilation — gratuit et exécuté à chaque build.

**N'abusez pas de `constexpr` sur des fonctions complexes.** Si une fonction `constexpr` de 200 lignes n'est jamais appelée dans un contexte compile-time, le `constexpr` n'apporte rien et peut ralentir la compilation (le compilateur doit vérifier qu'elle respecte les contraintes `constexpr`). Réservez `constexpr` aux fonctions qui seront effectivement utilisées à la compilation.

**Préférez `constexpr` à la métaprogrammation par templates.** Avant C++11, les calculs à la compilation nécessitaient des techniques obscures de template metaprogramming. `constexpr` offre la même capacité avec un code lisible, débuggable et maintenable. Un `constexpr factorial(10)` est infiniment plus clair qu'un `Factorial<10>::value`.

---

## En résumé

`constexpr` est le pont entre l'exécution et la compilation. Il permet d'écrire du code avec la lisibilité d'un programme normal et les performances d'un calcul effectué avant même le démarrage du programme. Sa dualité — compile-time quand c'est possible, runtime quand c'est nécessaire — en fait un outil d'une flexibilité remarquable. Les standards successifs ont levé la quasi-totalité des restrictions initiales, faisant de `constexpr` un citoyen de première classe du C++ moderne. Combiné avec `static_assert` pour la validation, `std::array` pour les tables de données et `if constexpr` pour la métaprogrammation, il constitue l'un des piliers de l'écriture de code C++ performant et expressif.
