# 3.3.1 — `static_cast` : Conversions sûres

> **Chapitre 3 — Types, Variables et Opérateurs** · Section 3.3 · Sous-section 1 sur 4  
> Prérequis : [3.3 — Conversion de types : Cast implicite vs explicite](/03-types-variables-operateurs/03-conversion-types.md)

---

## Introduction

`static_cast` est le cast que vous utiliserez le plus souvent — et dans la grande majorité des cas, le seul dont vous aurez besoin. C'est l'équivalent C++ d'une conversion que le compilateur **sait vérifier à la compilation** : il s'assure que la conversion demandée a un sens du point de vue du système de types, et il refuse celles qui sont manifestement incohérentes.

Le qualificatif « statique » fait référence au fait que toutes les vérifications sont effectuées **au moment de la compilation** (*statically*), sans aucun coût à l'exécution. Le binaire produit est identique à celui d'un cast C-style dans les cas où `static_cast` est applicable — la différence est que le compilateur vous protège contre les erreurs au moment de l'écriture du code.

---

## Syntaxe

```cpp
static_cast<TypeCible>(expression)
```

Le type cible est spécifié entre chevrons (`<>`), et l'expression à convertir entre parenthèses. La syntaxe est volontairement verbeuse par rapport au cast C-style — c'est un choix de conception qui rend les conversions visibles et faciles à rechercher dans le code.

---

## Conversions numériques

C'est le cas d'usage le plus fréquent de `static_cast` : convertir explicitement entre types arithmétiques lorsque la conversion implicite serait ambiguë ou produirait un avertissement.

### Flottant vers entier

```cpp
double pi = 3.14159;  
int n = static_cast<int>(pi); // 3 — la partie fractionnaire est tronquée (pas arrondie)  
```

Sans le cast, l'affectation `int n = pi;` compile mais déclenche un avertissement de conversion narrowing. Le `static_cast` exprime clairement que la troncation est intentionnelle. Le lecteur du code sait que le développeur n'a pas oublié la partie décimale — il a choisi de la supprimer.

### Entier vers flottant

```cpp
int numerator = 7;  
int denominator = 2;  

// Sans cast : division entière → 3 (pas 3.5)
double bad = numerator / denominator;

// Avec cast : division flottante → 3.5
double good = static_cast<double>(numerator) / denominator;
```

Ce piège de la division entière est l'une des erreurs les plus fréquentes chez les débutants en C++. Il suffit de convertir **un** des deux opérandes en `double` pour que le compilateur applique les conversions arithmétiques usuelles et effectue une division flottante. Le `static_cast` rend cette intention explicite.

### Entre types entiers de tailles différentes

```cpp
int64_t large = 42;  
int32_t small = static_cast<int32_t>(large); // Le programmeur assume que la valeur tient dans 32 bits  
```

Ici, le `static_cast` documente une hypothèse : le développeur affirme que la valeur de `large` est dans la plage d'un `int32_t`. Si cette hypothèse est fausse, la valeur sera tronquée silencieusement — `static_cast` n'ajoute pas de vérification à l'exécution. Si vous avez besoin d'une vérification runtime, utilisez une assertion ou un test explicite :

```cpp
int64_t large = get_value();  
assert(large >= INT32_MIN && large <= INT32_MAX); // Vérification explicite  
int32_t small = static_cast<int32_t>(large);  
```

### Signé vers non signé et inversement

```cpp
int signed_val = -1;  
unsigned int unsigned_val = static_cast<unsigned int>(signed_val);  
// unsigned_val vaut 4'294'967'295 (wrap-around modulo 2^32)
```

La conversion est bien définie (arithmétique modulo 2^N pour les non signés), mais le résultat peut être surprenant. Le `static_cast` rend la conversion visible — un `grep` sur `static_cast<unsigned` dans la base de code permet de retrouver tous les endroits où ce type de conversion est effectué.

## Conversions dans les hiérarchies de classes

`static_cast` est utilisé pour les conversions de pointeurs et de références au sein d'une hiérarchie d'héritage, dans les deux directions.

### Upcast : dérivée vers base

La conversion d'un pointeur vers une classe dérivée en pointeur vers sa classe de base est **implicite** — `static_cast` n'est pas nécessaire :

```cpp
class Animal { public: virtual ~Animal() = default; };  
class Dog : public Animal {};  

Dog rex;  
Animal* a = &rex; // Upcast implicite — pas besoin de cast  
```

Vous pouvez écrire `static_cast<Animal*>(&rex)` si vous souhaitez être explicite, mais la communauté considère généralement que l'upcast implicite est suffisamment sûr et lisible.

### Downcast : base vers dérivée

La conversion inverse — d'un pointeur vers la base en pointeur vers la dérivée — n'est **pas** implicite, car elle peut échouer si l'objet réel n'est pas du type attendu. `static_cast` permet de l'effectuer :

```cpp
Animal* animal = new Dog();  
Dog* dog = static_cast<Dog*>(animal); // Downcast — compile et fonctionne  
```

Ce code est correct **si et seulement si** `animal` pointe réellement vers un `Dog`. Si `animal` pointait vers un `Cat`, le `static_cast` compilerait quand même mais produirait un **comportement indéfini** à l'utilisation du pointeur résultant.

C'est la limitation fondamentale de `static_cast` pour les downcasts : **aucune vérification à l'exécution**. Le compilateur vous fait confiance. Si vous n'êtes pas absolument certain du type réel de l'objet, utilisez `dynamic_cast` (section 3.3.4) qui effectue une vérification runtime.

**Quand utiliser `static_cast` pour un downcast ?** Lorsque le type est garanti par la logique du programme et que le coût de `dynamic_cast` (RTTI) est jugé inacceptable. Par exemple, dans un visitor pattern où le type est déjà connu grâce au dispatch :

```cpp
void visit(Animal& animal, AnimalType type) {
    switch (type) {
        case AnimalType::Dog: {
            // Le type est garanti par le switch — static_cast est sûr ici
            auto& dog = static_cast<Dog&>(animal);
            dog.fetch();
            break;
        }
        case AnimalType::Cat: {
            auto& cat = static_cast<Cat&>(animal);
            cat.purr();
            break;
        }
    }
}
```

## Conversions vers et depuis `void*`

`void*` est le type pointeur générique du C — il peut pointer vers n'importe quel type d'objet, mais on ne peut pas le déréférencer directement. `static_cast` permet de convertir un `void*` vers un type pointeur concret :

```cpp
void* raw = malloc(sizeof(int) * 10);  
int* data = static_cast<int*>(raw); // Conversion void* → int*  
```

Cette conversion est sûre tant que le `void*` pointe réellement vers de la mémoire contenant (ou destinée à contenir) des `int`. En C, cette conversion est implicite ; en C++, elle nécessite un cast explicite — une différence de philosophie entre les deux langages.

Le cas d'usage le plus fréquent est l'interaction avec des **API C** (comme `malloc`, `pthread_create`, etc.) qui utilisent `void*` comme type générique de passage de données :

```cpp
// Callback C avec contexte utilisateur
void* context = static_cast<void*>(&my_config);  
pthread_create(&thread, nullptr, thread_func, context);  

// Dans le thread
void* thread_func(void* arg) {
    auto* config = static_cast<Config*>(arg);
    // Utiliser config...
    return nullptr;
}
```

> 💡 En C++ moderne, on préfère `std::thread` à `pthread_create`, ce qui élimine le besoin de `void*`. Mais de nombreuses bibliothèques C exposent encore cette interface.

## Conversions avec les `enum`

`static_cast` est le mécanisme standard pour convertir entre énumérations et types entiers.

### `enum` non-scoped (hérité du C)

Les énumérations non-scoped se convertissent implicitement vers les entiers, mais la conversion inverse nécessite un cast :

```cpp
enum Color { Red, Green, Blue };

int n = Red;                        // Implicite : enum → int (OK)  
Color c = static_cast<Color>(1);    // Explicite : int → enum (nécessite un cast)  
```

### `enum class` (C++11)

Les énumérations scoped (`enum class`) ne se convertissent **jamais** implicitement — dans aucune direction. `static_cast` est nécessaire pour les deux sens :

```cpp
enum class Direction : uint8_t { North = 0, East = 1, South = 2, West = 3 };

// Direction d = 1;                  // ❌ Erreur : pas de conversion implicite
Direction d = static_cast<Direction>(1);  // ✅ Explicite : int → enum class

// int n = d;                        // ❌ Erreur : pas de conversion implicite
int n = static_cast<int>(d);             // ✅ Explicite : enum class → int
```

Cette rigueur est l'un des avantages de `enum class` par rapport aux `enum` hérités : elle empêche les confusions accidentelles entre des valeurs d'énumération et des entiers arbitraires.

Un cas d'usage courant est la conversion d'un `enum class` vers son type sous-jacent pour le sérialiser, l'afficher ou l'utiliser comme indice de tableau :

```cpp
enum class LogLevel : int { Debug = 0, Info = 1, Warning = 2, Error = 3 };

const char* names[] = {"DEBUG", "INFO", "WARNING", "ERROR"};

void log(LogLevel level, const std::string& msg) {
    auto index = static_cast<int>(level);
    std::print("[{}] {}\n", names[index], msg);
}
```

> 💡 C++23 introduit `std::to_underlying(e)` (dans `<utility>`) qui est un raccourci type-safe pour `static_cast<std::underlying_type_t<E>>(e)`. Préférez-le lorsque votre compilateur le supporte.

## Ce que `static_cast` refuse

La force de `static_cast` réside autant dans ce qu'il autorise que dans ce qu'il **interdit**. Voici des conversions qu'un cast C-style accepterait mais que `static_cast` refuse :

### Conversion entre pointeurs non liés

```cpp
int x = 42;  
double* dp = static_cast<double*>(&x); // ❌ Erreur de compilation  
// Un int* et un double* n'ont aucune relation de type
```

Un cast C-style `(double*)&x` compilerait sans broncher, en effectuant un `reinterpret_cast` silencieux. `static_cast` refuse cette conversion parce qu'elle n'a aucun sens dans le système de types — et c'est exactement ce qu'on attend d'un cast « sûr ».

### Retrait de `const`

```cpp
const int x = 42;  
int* p = static_cast<int*>(&x); // ❌ Erreur de compilation  
// static_cast ne peut pas retirer const
```

Retirer `const` nécessite `const_cast` — un cast dédié qui rend l'opération visible et traçable. `static_cast` ne permet pas de contourner accidentellement la const-correctness.

### Downcast entre hiérarchies non liées

```cpp
class A {};  
class B {}; // Aucune relation avec A  

A a;  
B* bp = static_cast<B*>(&a); // ❌ Erreur : A et B n'ont aucun lien d'héritage  
```

`static_cast` vérifie que la conversion a un sens dans la hiérarchie de classes. Si les types ne sont pas liés par héritage, la conversion est refusée.

## `static_cast` vs cast C-style : comparaison

Pour illustrer concrètement les avantages de `static_cast`, comparons les deux approches sur une série de scénarios :

| Scénario | Cast C-style | `static_cast` | Différence |
|---|---|---|---|
| `double` → `int` | `(int)pi` | `static_cast<int>(pi)` | Équivalents, mais `static_cast` est plus visible |
| `int*` → `double*` | `(double*)p` — **compile** | `static_cast<double*>(p)` — **refuse** | `static_cast` détecte l'erreur |
| Retrait de `const` | `(int*)cp` — **compile** | `static_cast<int*>(cp)` — **refuse** | `static_cast` protège `const` |
| Downcast `Base*` → `Derived*` | `(Derived*)bp` | `static_cast<Derived*>(bp)` | Équivalents (pas de vérification runtime dans les deux cas) |
| Types non liés `A*` → `B*` | `(B*)ap` — **compile** | `static_cast<B*>(ap)` — **refuse** | `static_cast` détecte l'erreur |

Le cast C-style compile dans **tous** les cas — y compris ceux qui sont des erreurs. `static_cast` compile uniquement quand la conversion a un sens, et refuse celles qui ne correspondent à aucune relation de type connue. C'est cette sélectivité qui en fait l'outil par défaut du C++ moderne.

## Bonnes pratiques

**Utilisez `static_cast` pour toute conversion numérique explicite.** Dès que le compilateur émet un avertissement de conversion narrowing ou que vous souhaitez rendre une conversion intentionnelle visible, `static_cast` est le bon choix :

```cpp
auto index = static_cast<int>(v.size()); // Intention documentée
```

**Utilisez `static_cast` pour les downcasts uniquement si le type est garanti.** Si vous avez le moindre doute sur le type réel de l'objet, préférez `dynamic_cast`. Le coût de la vérification runtime est négligeable comparé au coût d'un comportement indéfini.

**Ne remplacez pas une conversion implicite sûre par un `static_cast`.** Si la conversion est une promotion sans perte (par exemple `int` → `double`), le cast explicite n'apporte rien et alourdit le code :

```cpp
int x = 42;  
double d = x;                      // ✅ Promotion implicite — parfaitement sûre  
double d2 = static_cast<double>(x); // Inutilement verbeux dans ce cas  
```

**Préférez `std::to_underlying` à `static_cast<int>` pour les `enum class`** (C++23) — c'est plus lisible et plus robuste si le type sous-jacent de l'enum change.

**Commentez les downcasts `static_cast` dans les hiérarchies de classes.** Un commentaire court expliquant pourquoi le type est garanti aide les futurs lecteurs à comprendre que ce n'est pas un oubli de `dynamic_cast` :

```cpp
// Le type est garanti par le dispatch du visitor — voir ligne 42
auto& dog = static_cast<Dog&>(animal);
```

---

## En résumé

`static_cast` est le couteau suisse des conversions explicites en C++. Il couvre la grande majorité des besoins — conversions numériques, conversions de pointeurs dans les hiérarchies de classes, conversions `void*`, conversions d'énumérations — tout en refusant les opérations dangereuses comme la réinterprétation mémoire ou le retrait de `const`. Sa syntaxe verbeuse est un atout, pas un défaut : elle rend les conversions visibles, recherchables et intentionnelles. Quand vous vous demandez quel cast utiliser, commencez toujours par `static_cast`. Si le compilateur le refuse, demandez-vous si la conversion est vraiment nécessaire avant de passer à un cast plus permissif.
