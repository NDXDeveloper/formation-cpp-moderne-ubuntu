# 3.2.2 — Flottants : `float`, `double`, `long double`

> **Chapitre 3 — Types, Variables et Opérateurs** · Section 3.2 · Sous-section 2 sur 3  
> Prérequis : [3.2.1 — Entiers](/03-types-variables-operateurs/02.1-entiers.md)

---

## Introduction

Les types à virgule flottante permettent de représenter des nombres réels — ou plus exactement, une **approximation** de nombres réels. C'est cette nuance qui fait toute la difficulté de l'arithmétique flottante : contrairement aux entiers, où `3 + 4` donne toujours exactement `7`, les opérations sur les flottants introduisent des erreurs d'arrondi qui peuvent s'accumuler de manière inattendue.

Le C++ propose trois types à virgule flottante : `float`, `double` et `long double`. Ils diffèrent par leur taille en mémoire, leur précision et leur plage de valeurs, mais partagent le même modèle de représentation sous-jacent — le standard IEEE 754, adopté par la quasi-totalité des processeurs modernes.

Cette sous-section explore la représentation interne des flottants, les caractéristiques de chaque type, les pièges classiques de l'arithmétique flottante et les bonnes pratiques pour écrire du code numérique fiable.

---

## Le standard IEEE 754

Depuis les années 1980, la grande majorité des processeurs implémentent le standard IEEE 754 pour l'arithmétique à virgule flottante. Ce standard définit comment un nombre réel est encodé en binaire en le décomposant en trois composantes :

- **Le bit de signe** (1 bit) : 0 pour positif, 1 pour négatif.
- **L'exposant** (biaisé) : détermine l'ordre de grandeur du nombre.
- **La mantisse** (ou *significand*, ou *fraction*) : détermine la précision du nombre.

La valeur représentée est : **(−1)^signe × 1.mantisse × 2^(exposant − biais)**

Le « 1. » devant la mantisse est implicite (il n'est pas stocké) — c'est le *leading bit* caché. Ce détail permet de gagner un bit de précision sans coût de stockage.

### Structure des trois types

| Type | Taille totale | Signe | Exposant | Mantisse | Précision décimale |
|---|---|---|---|---|---|
| `float` | 32 bits (4 octets) | 1 bit | 8 bits | 23 bits | ~7 chiffres |
| `double` | 64 bits (8 octets) | 1 bit | 11 bits | 52 bits | ~15-16 chiffres |
| `long double` | *variable* | 1 bit | *variable* | *variable* | *variable* |

`float` et `double` sont parfaitement standardisés par IEEE 754 et ont la même taille sur toutes les plateformes modernes. `long double`, en revanche, varie considérablement :

- **Linux x86_64 (GCC/Clang)** : 80 bits de précision étendue (format x87), stockés sur 16 octets avec du padding. La mantisse est de 64 bits, offrant environ 18-19 chiffres significatifs.
- **Windows x86_64 (MSVC)** : `long double` est identique à `double` (64 bits). Aucune précision supplémentaire.
- **ARM / Apple Silicon** : `long double` est généralement identique à `double` (64 bits), ou parfois IEEE 754 *quadruple precision* (128 bits) selon le compilateur.

Cette variabilité fait de `long double` un choix non portable. Nous y reviendrons dans les recommandations.

## Plages de valeurs

Les plages approximatives des types flottants sont les suivantes :

| Type | Valeur minimale positive | Valeur maximale | Epsilon machine |
|---|---|---|---|
| `float` | ~1.18 × 10⁻³⁸ | ~3.40 × 10³⁸ | ~1.19 × 10⁻⁷ |
| `double` | ~2.23 × 10⁻³⁰⁸ | ~1.80 × 10³⁰⁸ | ~2.22 × 10⁻¹⁶ |

L'**epsilon machine** (`std::numeric_limits<T>::epsilon()`) est la plus petite valeur telle que `1.0 + epsilon != 1.0`. C'est la mesure directe de la précision du type autour de la valeur 1. Pour `float`, cet epsilon est d'environ 10⁻⁷ — ce qui signifie que deux valeurs qui diffèrent de moins d'un dix-millionième sont indistinguables.

Ces limites et bien d'autres sont accessibles programmatiquement via l'en-tête `<limits>` :

```cpp
#include <limits>

std::print("float  — max : {}, epsilon : {}\n",
    std::numeric_limits<float>::max(),
    std::numeric_limits<float>::epsilon());

std::print("double — max : {}, epsilon : {}\n",
    std::numeric_limits<double>::max(),
    std::numeric_limits<double>::epsilon());

std::print("double — digits10 : {}\n",
    std::numeric_limits<double>::digits10); // 15 — chiffres décimaux garantis
```

La propriété `digits10` indique le nombre de chiffres décimaux significatifs que le type peut stocker sans perte. Pour `float`, c'est 6. Pour `double`, c'est 15.

## Littéraux flottants

Par défaut, un littéral à virgule flottante est de type `double` :

```cpp
auto a = 3.14;      // double  
auto b = 1.0;       // double (le .0 suffit à en faire un flottant)  
auto c = 2.5e10;    // double (notation scientifique : 2.5 × 10¹⁰)  
auto d = 1e-3;      // double (0.001)  
```

Les suffixes permettent de forcer un type spécifique :

| Suffixe | Type | Exemple |
|---|---|---|
| *(aucun)* | `double` | `3.14` |
| `f` ou `F` | `float` | `3.14f` |
| `l` ou `L` | `long double` | `3.14L` |

Le suffixe `f` est important lorsque vous travaillez avec des `float` : sans lui, le littéral est un `double`, et l'affectation à un `float` implique une conversion avec perte potentielle de précision :

```cpp
float x = 3.14;   // ⚠️ Le littéral 3.14 (double) est converti en float
                   //    avec perte potentielle au dernier bit

float y = 3.14f;  // ✅ Le littéral est directement un float — aucune conversion
```

### Littéraux hexadécimaux flottants (C++17)

C++17 a introduit les littéraux flottants en hexadécimal, utiles pour exprimer une valeur binaire exacte sans ambiguïté d'arrondi :

```cpp
double exact = 0x1.921fb54442d18p+1; // Représentation exacte de π en double
```

La syntaxe est `0x` + mantisse hexadécimale + `p` + exposant en puissance de 2. Ce format est surtout utilisé dans le code numérique de haute précision et dans les tests unitaires qui vérifient des valeurs flottantes exactes.

## Valeurs spéciales

IEEE 754 définit plusieurs valeurs spéciales que C++ expose :

### L'infini

Les opérations qui dépassent la plage représentable produisent l'infini positif ou négatif :

```cpp
double inf = 1.0 / 0.0;   // +infinity  
double neg_inf = -1.0 / 0.0; // -infinity  

std::print("{}\n", inf);          // inf  
std::print("{}\n", std::isinf(inf)); // true  
```

L'infini se propage dans les calculs de manière cohérente : `inf + 1` reste `inf`, `inf * -1` donne `-inf`.

### NaN (Not a Number)

Les opérations mathématiquement indéfinies produisent `NaN` :

```cpp
double nan = 0.0 / 0.0;           // NaN  
double nan2 = std::sqrt(-1.0);    // NaN  

std::print("{}\n", nan);                 // nan  
std::print("{}\n", std::isnan(nan));     // true  
```

`NaN` a une propriété unique et contre-intuitive : **il n'est égal à rien, pas même à lui-même** :

```cpp
double nan = std::numeric_limits<double>::quiet_NaN();

if (nan == nan) {
    std::print("égal\n");
} else {
    std::print("pas égal\n"); // ← C'est cette branche qui s'exécute
}
```

C'est pourquoi tester `x == x` qui retourne `false` est un indicateur de `NaN`, mais la fonction `std::isnan(x)` est la méthode recommandée. Le piège est qu'un `NaN` qui s'introduit dans une chaîne de calculs contamine silencieusement tous les résultats en aval.

### Zéro positif et zéro négatif

IEEE 754 distingue `+0.0` et `-0.0`. Ils sont considérés comme **égaux** par l'opérateur `==`, mais se comportent différemment dans certaines opérations :

```cpp
double pos_zero = +0.0;  
double neg_zero = -0.0;  

std::print("{}\n", pos_zero == neg_zero);   // true  
std::print("{}\n", 1.0 / pos_zero);         // inf  
std::print("{}\n", 1.0 / neg_zero);         // -inf  
```

### Les nombres dénormalisés (subnormals)

Entre zéro et la plus petite valeur normale positive, IEEE 754 définit des nombres **dénormalisés** (*subnormals*) qui sacrifient de la précision pour représenter des valeurs très proches de zéro. Ils évitent un « trou » brutal entre zéro et le premier nombre représentable, mais les opérations sur ces nombres sont souvent beaucoup plus lentes sur certains processeurs — un point à considérer dans le code haute performance.

## Le piège fondamental : les erreurs d'arrondi

La plupart des nombres décimaux courants n'ont **pas** de représentation binaire exacte en virgule flottante. Le nombre `0.1` en est l'exemple canonique :

```cpp
double x = 0.1;  
std::print("{:.20f}\n", x); // 0.10000000000000000555...  
```

La valeur stockée n'est pas exactement 0.1 — elle en est l'approximation la plus proche représentable en 64 bits. Cette erreur est minuscule (de l'ordre de 10⁻¹⁷), mais elle s'accumule :

```cpp
double sum = 0.0;  
for (int i = 0; i < 10; ++i) {  
    sum += 0.1;
}
std::print("{:.20f}\n", sum); // 0.99999999999999988898...
                               // Pas exactement 1.0
```

Ce comportement n'est pas un bug — c'est une conséquence inévitable de la représentation binaire à précision finie. Mais il a des implications directes sur la manière de comparer des flottants.

### Ne jamais comparer des flottants avec `==`

La comparaison directe d'égalité entre flottants est presque toujours une erreur :

```cpp
double a = 0.1 + 0.2;  
double b = 0.3;  

if (a == b) {
    std::print("égaux\n");
} else {
    std::print("différents\n"); // ← 0.1 + 0.2 != 0.3 en IEEE 754
}
```

### Comparaison avec tolérance (epsilon)

La technique standard consiste à comparer avec une **tolérance** (*epsilon*) — une marge d'erreur acceptable :

```cpp
#include <cmath>

bool nearly_equal(double a, double b, double epsilon = 1e-9) {
    return std::abs(a - b) < epsilon;
}
```

Cette approche fonctionne pour les valeurs proches de 1, mais elle est inadaptée pour les très grands ou très petits nombres, où l'epsilon absolu n'a pas de sens. Une version plus robuste utilise un **epsilon relatif** :

```cpp
bool nearly_equal_relative(double a, double b, double rel_epsilon = 1e-9) {
    double diff = std::abs(a - b);
    double largest = std::max(std::abs(a), std::abs(b));
    return diff <= largest * rel_epsilon;
}
```

Pour les cas simples (résultats attendus proches de 1 en valeur absolue), l'epsilon absolu suffit. Pour du code numérique robuste, la version relative est préférable. Les frameworks de test comme Google Test fournissent des matchers spécialisés (`EXPECT_NEAR`, `EXPECT_DOUBLE_EQ`) qui encapsulent cette logique.

## Perte de précision par absorption

Un autre piège classique survient lorsqu'on additionne un très grand nombre et un très petit nombre :

```cpp
float big   = 1'000'000.0f;  
float small = 0.001f;  

float result = big + small;  
std::print("{:.6f}\n", result); // 1000000.000000 — le 0.001 a été absorbé  
```

Le `float` dispose d'environ 7 chiffres significatifs. Le nombre `1'000'000.001` en nécessite 10 — il dépasse la précision de `float`, et la partie fractionnaire est perdue. Ce phénomène s'appelle l'**absorption** (*absorption*). Un phénomène cousin, l'**annulation catastrophique** (*catastrophic cancellation*), survient lors de la soustraction de deux nombres presque égaux, où les chiffres significatifs communs s'annulent et amplifient l'erreur relative.

Avec un `double` (15-16 chiffres significatifs), le problème est repoussé mais pas éliminé :

```cpp
double big   = 1e16;  
double small = 1.0;  

double result = big + small;  
std::print("{:.1f}\n", result); // 10000000000000000.0 — le 1.0 est absorbé  
```

La leçon est que la précision d'un flottant est **relative** à la magnitude du nombre, pas absolue.

## `float` vs `double` : lequel choisir ?

| Critère | `float` (32 bits) | `double` (64 bits) |
|---|---|---|
| Précision | ~7 chiffres | ~15-16 chiffres |
| Taille mémoire | 4 octets | 8 octets |
| Performance CPU | Comparable à `double` sur x86_64 | Type par défaut du FPU x86_64 |
| Performance SIMD | 2× plus d'éléments par registre | Référence |
| Usage typique | Graphique, audio, GPU, données massives | Calcul scientifique, finance, usage général |

Sur les processeurs x86_64 modernes, les opérations `float` et `double` prennent le même nombre de cycles en mode scalaire (une valeur à la fois). L'avantage de `float` se manifeste avec la **vectorisation SIMD** : un registre AVX de 256 bits peut traiter 8 `float` simultanément contre 4 `double`. Pour le traitement de données massives (images, audio, machine learning), cette différence est significative.

**Recommandation générale** : utilisez `double` par défaut. C'est le type naturel de l'arithmétique flottante en C++ (les littéraux sans suffixe sont `double`, les fonctions de `<cmath>` retournent `double`). Réservez `float` aux contextes où la mémoire ou la bande passante sont critiques et où la précision réduite est acceptable.

---

## Et `long double` ?

`long double` est le type flottant le moins portable du langage. Sa taille et sa précision varient d'une plateforme à l'autre (80 bits sur x86 Linux, 64 bits sur Windows, parfois 128 bits sur ARM). Son utilisation a aussi un coût : sur x86_64, les opérations `long double` passent par l'unité x87 legacy, plus lente que les unités SSE/AVX utilisées pour `float` et `double`.

En pratique, `long double` est utile dans un seul cas : les algorithmes numériques qui ont besoin de précision intermédiaire supérieure pour éviter l'accumulation d'erreurs dans un résultat final en `double` (technique dite de *double-double* ou de calcul compensé). En dehors de ce cas de figure, préférez `double`.

Si vous avez besoin de précision arbitraire, tournez-vous vers des bibliothèques spécialisées comme GMP (GNU Multiple Precision Arithmetic Library) ou Boost.Multiprecision, qui ne sont pas limitées par les contraintes matérielles.

## Fonctions mathématiques : `<cmath>`

L'en-tête `<cmath>` fournit les fonctions mathématiques standard, surchargées pour `float`, `double` et `long double` :

```cpp
#include <cmath>

double x = 2.0;

std::sqrt(x);    // Racine carrée  
std::pow(x, 3);  // Puissance : x³  
std::log(x);     // Logarithme naturel  
std::log10(x);   // Logarithme base 10  
std::exp(x);     // Exponentielle  
std::abs(x);     // Valeur absolue (⚠️ std::abs, pas abs du C)  
std::ceil(x);    // Arrondi supérieur  
std::floor(x);   // Arrondi inférieur  
std::round(x);   // Arrondi au plus proche  

std::isnan(x);   // Teste si NaN  
std::isinf(x);   // Teste si infini  
std::isfinite(x);// Teste si fini (ni NaN, ni infini)  
```

> ⚠️ Utilisez toujours `std::abs` (de `<cmath>`) pour les flottants, pas `abs` (de `<cstdlib>`), qui tronque vers un entier. C'est une erreur subtile qui compile sans avertissement.

---

## En résumé

Les types à virgule flottante sont des outils puissants mais exigeants. Leur représentation binaire finie implique des erreurs d'arrondi systématiques que le programmeur doit anticiper. Les règles essentielles à retenir sont les suivantes : ne jamais comparer des flottants avec `==`, utiliser `double` par défaut, réserver `float` aux cas où la mémoire ou la vectorisation priment, éviter `long double` sauf besoin de précision intermédiaire spécifique, et toujours vérifier la présence de `NaN` et d'infinis dans les résultats de calculs susceptibles de produire des valeurs dégénérées.
