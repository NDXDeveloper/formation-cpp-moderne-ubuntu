# 3.3.4 — `dynamic_cast` : Cast polymorphique

> **Chapitre 3 — Types, Variables et Opérateurs** · Section 3.3 · Sous-section 4 sur 4  
> Prérequis : [3.3.1 — `static_cast`](/03-types-variables-operateurs/03.1-static-cast.md), notions de base sur l'héritage (aperçu — couvert en détail au chapitre 7)

---

## Introduction

Les trois casts que nous avons étudiés jusqu'ici — `static_cast`, `reinterpret_cast`, `const_cast` — ont un point commun : toutes leurs vérifications sont effectuées **à la compilation**. Le compilateur accepte ou refuse le cast selon les règles du système de types, puis génère du code sans aucune vérification supplémentaire à l'exécution.

`dynamic_cast` est fondamentalement différent. C'est le seul cast du C++ qui effectue une vérification **à l'exécution** (*runtime*). Il interroge le type réel de l'objet au moment où le programme tourne, et il répond honnêtement : « oui, cet objet est bien du type que tu demandes » ou « non, ce n'est pas le cas ». Cette capacité en fait l'outil de choix pour les **downcasts sûrs** dans les hiérarchies polymorphiques — les conversions d'un pointeur ou d'une référence vers la classe de base en pointeur ou référence vers une classe dérivée.

Cette sous-section nécessite une compréhension élémentaire de l'héritage et des fonctions virtuelles, qui seront couverts en profondeur au chapitre 7. Nous introduirons les concepts nécessaires au fil de la présentation.

---

## Syntaxe

```cpp
dynamic_cast<TypeCible>(expression)
```

`TypeCible` doit être un pointeur ou une référence vers une classe. L'expression doit être un pointeur ou une référence vers une classe **polymorphique** — c'est-à-dire une classe qui possède au moins une fonction virtuelle (y compris un destructeur virtuel).

---

## Prérequis : classes polymorphiques et vtable

Pour comprendre `dynamic_cast`, il faut comprendre ce qui rend une classe « polymorphique ». En C++, une classe est polymorphique dès qu'elle déclare au moins une fonction `virtual` :

```cpp
class Animal {  
public:  
    virtual ~Animal() = default; // Destructeur virtuel → classe polymorphique
    virtual std::string speak() const = 0;
};

class Dog : public Animal {  
public:  
    std::string speak() const override { return "Woof!"; }
    void fetch() const { std::print("Fetching!\n"); }
};

class Cat : public Animal {  
public:  
    std::string speak() const override { return "Meow!"; }
    void purr() const { std::print("Purrrr...\n"); }
};
```

Quand une classe est polymorphique, le compilateur ajoute un pointeur caché dans chaque instance — le **vptr** (*virtual table pointer*) — qui pointe vers une table de fonctions virtuelles (**vtable**). Cette vtable contient, entre autres, des informations sur le type réel de l'objet. C'est cette information que `dynamic_cast` consulte à l'exécution pour déterminer si la conversion est valide.

Sans fonction virtuelle, pas de vtable, pas d'information de type à l'exécution, et `dynamic_cast` ne peut pas fonctionner — le compilateur émet une erreur.

---

## Downcast avec des pointeurs

Le cas d'usage principal de `dynamic_cast` est le **downcast** — convertir un pointeur vers la classe de base en pointeur vers une classe dérivée, avec vérification du type réel :

```cpp
Animal* animal = get_some_animal(); // Peut être un Dog, un Cat, ou autre chose

Dog* dog = dynamic_cast<Dog*>(animal);

if (dog != nullptr) {
    // La conversion a réussi — animal est bien un Dog
    dog->fetch();
} else {
    // La conversion a échoué — animal n'est pas un Dog
    std::print("Pas un chien.\n");
}
```

Quand `dynamic_cast` est appliqué à un **pointeur** et que la conversion échoue, il retourne **`nullptr`**. C'est un mécanisme de signalement d'échec propre et non intrusif — le programme continue à s'exécuter et le développeur peut tester le résultat.

### Comparaison avec `static_cast`

La même opération avec `static_cast` compilerait sans erreur mais ne vérifierait rien :

```cpp
Animal* animal = new Cat();  
Dog* dog = static_cast<Dog*>(animal); // Compile — aucune vérification runtime  
dog->fetch(); // ❌ Comportement indéfini : animal est un Cat, pas un Dog  
```

`static_cast` fait confiance au programmeur. `dynamic_cast` fait confiance aux données. Dans un contexte où le type réel de l'objet n'est pas garanti par la logique du programme, `dynamic_cast` est le seul choix sûr.

---

## Downcast avec des références

`dynamic_cast` fonctionne aussi avec des références, mais le mécanisme de signalement d'échec est différent. Une référence ne peut pas être « nulle » — il n'y a pas d'équivalent de `nullptr` pour les références. En cas d'échec, `dynamic_cast` **lève une exception** `std::bad_cast` :

```cpp
#include <typeinfo> // Pour std::bad_cast

Animal& animal_ref = get_animal_ref();

try {
    Dog& dog_ref = dynamic_cast<Dog&>(animal_ref);
    dog_ref.fetch(); // Exécuté uniquement si animal_ref est un Dog
} catch (const std::bad_cast& e) {
    std::print("Conversion échouée : {}\n", e.what());
}
```

Le choix entre la forme pointeur et la forme référence dépend de la sémantique souhaitée :

- **Forme pointeur** (`dynamic_cast<Dog*>`) : « l'objet est-il un Dog ? Si oui, je veux travailler avec. Si non, j'ai un plan B. » C'est un test conditionnel.
- **Forme référence** (`dynamic_cast<Dog&>`) : « l'objet devrait être un Dog. Si ce n'est pas le cas, c'est une erreur exceptionnelle. » C'est une assertion de type à l'exécution.

---

## Upcast avec `dynamic_cast`

`dynamic_cast` peut aussi effectuer un **upcast** (dérivée vers base), mais c'est redondant : l'upcast est toujours valide et il est déjà implicite en C++. Écrire `dynamic_cast<Animal*>(dog_ptr)` est fonctionnellement identique à une conversion implicite, avec le surcoût inutile d'une vérification runtime :

```cpp
Dog* dog = new Dog();  
Animal* animal = dynamic_cast<Animal*>(dog); // Fonctionne, mais inutile  
Animal* animal2 = dog;                       // Identique, sans surcoût  
```

N'utilisez pas `dynamic_cast` pour les upcasts — la conversion implicite suffit.

---

## Cross-cast : conversions latérales

Un cas moins connu mais puissant de `dynamic_cast` est le **cross-cast** — la conversion entre deux branches d'une hiérarchie d'héritage multiple. C'est un scénario que ni `static_cast` ni aucun autre cast ne peut gérer de manière sûre :

```cpp
class Drawable {  
public:  
    virtual ~Drawable() = default;
    virtual void draw() const = 0;
};

class Clickable {  
public:  
    virtual ~Clickable() = default;
    virtual void on_click() = 0;
};

class Button : public Drawable, public Clickable {  
public:  
    void draw() const override { std::print("Drawing button\n"); }
    void on_click() override { std::print("Button clicked\n"); }
};

// Quelque part dans le code, on a un Drawable*
Drawable* widget = new Button();

// Cross-cast : Drawable* → Clickable* (branches différentes de la hiérarchie)
Clickable* clickable = dynamic_cast<Clickable*>(widget);

if (clickable) {
    clickable->on_click(); // "Button clicked"
}
```

`static_cast<Clickable*>(widget)` serait une erreur de compilation, car `Drawable` et `Clickable` n'ont aucune relation d'héritage directe. Seul `dynamic_cast` peut traverser la hiérarchie latéralement en consultant le type réel de l'objet à l'exécution.

---

## Le mécanisme RTTI

`dynamic_cast` repose sur le mécanisme de **RTTI** (*Runtime Type Information*), le système d'information de type à l'exécution du C++. Le RTTI inclut deux composantes principales :

- **`dynamic_cast`** : vérification de type et conversion sûre.
- **`typeid`** : inspection du type réel d'un objet polymorphique.

`typeid` retourne une référence vers un objet `std::type_info` qui identifie le type réel :

```cpp
#include <typeinfo>

Animal* animal = new Dog();

std::print("Type statique : {}\n", typeid(Animal*).name());  
std::print("Type dynamique : {}\n", typeid(*animal).name());  
// Le type dynamique sera "Dog" (le format exact dépend du compilateur)
```

> ⚠️ Les chaînes retournées par `typeid(...).name()` ne sont pas standardisées — elles varient entre compilateurs. Sur GCC, elles sont « manglées » (par exemple `3Dog`) et nécessitent le démangling via `abi::__cxa_demangle` ou l'outil `c++filt` pour être lisibles. Ne les utilisez pas pour des comparaisons programmatiques fiables ; préférez `typeid(a) == typeid(b)`.

---

## Coût en performance

`dynamic_cast` est le seul cast C++ qui a un **coût runtime non négligeable**. Ce coût provient de la consultation de la vtable et de la traversée de la hiérarchie de classes pour vérifier la validité de la conversion.

### Caractéristiques du coût

- La complexité dépend de la **profondeur** et de la **largeur** de la hiérarchie de classes. Une hiérarchie simple à deux niveaux est vérifiée très rapidement. Une hiérarchie profonde avec héritage multiple et virtuel est plus coûteuse.
- Sur les implémentations courantes (libstdc++, libc++), `dynamic_cast` parcourt la table RTTI en effectuant des comparaisons de pointeurs. Pour des hiérarchies simples, le coût est de l'ordre de quelques dizaines de nanosecondes.
- Le RTTI doit être activé à la compilation. L'option `-fno-rtti` (GCC/Clang) le désactive, ce qui rend `dynamic_cast` et `typeid` inutilisables mais réduit légèrement la taille des binaires.

### Quand le coût est-il un problème ?

Dans la grande majorité du code applicatif, le coût de `dynamic_cast` est invisible — il est noyé dans le bruit des accès mémoire, des I/O et de la logique métier. Le coût devient significatif uniquement dans des boucles très serrées qui effectuent des millions de casts par seconde, ce qui est un signe que l'architecture du code pourrait bénéficier d'une refonte.

Si `dynamic_cast` apparaît dans un hot path identifié par le profiler, les alternatives incluent le pattern Visitor (qui déplace le dispatch dans le système de vtable), `std::variant` avec `std::visit` (polymorphisme statique, couvert en section 12.2) ou un enum de type accompagné de `static_cast` (quand le type est garanti par un champ discriminant).

---

## `dynamic_cast` et `nullptr`

Un détail important : `dynamic_cast` appliqué à un pointeur `nullptr` retourne **toujours `nullptr`**, quel que soit le type cible. C'est un comportement défini et cohérent :

```cpp
Animal* null_animal = nullptr;  
Dog* dog = dynamic_cast<Dog*>(null_animal); // dog == nullptr (pas d'exception)  
```

Ce comportement simplifie le code dans les cas où le pointeur peut être nul — il n'est pas nécessaire de tester `nullptr` avant le cast.

---

## Anti-patterns avec `dynamic_cast`

### Cascades de `dynamic_cast`

Le pattern le plus courant — et le plus problématique — est la cascade de `dynamic_cast` dans une série de `if-else` :

```cpp
void process(Animal* animal) {
    if (auto* dog = dynamic_cast<Dog*>(animal)) {
        dog->fetch();
    } else if (auto* cat = dynamic_cast<Cat*>(animal)) {
        cat->purr();
    } else if (auto* bird = dynamic_cast<Bird*>(animal)) {
        bird->fly();
    } else {
        std::print("Animal inconnu\n");
    }
}
```

Ce code fonctionne, mais il présente plusieurs problèmes :

- **Violation du principe ouvert/fermé.** Ajouter un nouveau type d'animal oblige à modifier cette fonction. Si d'autres fonctions contiennent des cascades similaires, chacune doit être mise à jour.
- **Coût cumulé.** Chaque `dynamic_cast` a un coût runtime. Avec N types, le pire cas effectue N vérifications.
- **Fragilité.** Si un développeur oublie d'ajouter un cas lors de l'ajout d'un nouveau type, le code compile sans erreur mais le nouveau type tombe dans la branche `else`.

La solution idiomatique en C++ est d'utiliser le **polymorphisme virtuel** — exactement ce pour quoi les fonctions virtuelles ont été conçues :

```cpp
class Animal {  
public:  
    virtual ~Animal() = default;
    virtual void perform_action() const = 0; // Chaque animal sait quoi faire
};

class Dog : public Animal {  
public:  
    void perform_action() const override { fetch(); }
    void fetch() const { std::print("Fetching!\n"); }
};

class Cat : public Animal {  
public:  
    void perform_action() const override { purr(); }
    void purr() const { std::print("Purrrr...\n"); }
};

// Plus besoin de dynamic_cast
void process(Animal* animal) {
    animal->perform_action(); // Le dispatch est automatique via la vtable
}
```

### Utiliser `dynamic_cast` pour contourner un design rigide

Si vous utilisez fréquemment `dynamic_cast` pour accéder à des méthodes spécifiques d'une classe dérivée, c'est souvent le signe que l'interface de base est trop étroite ou que la hiérarchie de classes ne modélise pas correctement le domaine. Le reflexe devrait être de repenser l'interface avant de multiplier les casts.

---

## Quand `dynamic_cast` est-il réellement approprié ?

Malgré les mises en garde, `dynamic_cast` reste un outil légitime dans certains contextes précis.

**Plugins et extensions dynamiques.** Quand des modules sont chargés dynamiquement (via `dlopen`/`dlsym` sur Linux), le type exact des objets n'est pas connu à la compilation. `dynamic_cast` permet de vérifier si un objet implémente une interface spécifique :

```cpp
// Module chargé dynamiquement
auto* plugin = load_plugin("audio_plugin.so");  
Animal* obj = plugin->create();  

// Vérifier si le plugin supporte une interface optionnelle
if (auto* streamable = dynamic_cast<Streamable*>(obj)) {
    streamable->stream_to(output);
}
```

**Interfaces optionnelles.** Quand une hiérarchie de classes est stable et qu'une opération ne s'applique qu'à certaines classes dérivées sans justifier l'ajout d'une méthode virtuelle dans la base :

```cpp
// Tous les widgets ne sont pas redimensionnables
if (auto* resizable = dynamic_cast<Resizable*>(widget)) {
    resizable->resize(new_width, new_height);
}
```

**Débogage et logging.** Identifier le type exact d'un objet à des fins de diagnostic, sans modifier l'interface de la classe :

```cpp
void debug_log(const Animal& animal) {
    if (dynamic_cast<const Dog*>(&animal)) {
        std::print("[DEBUG] Type réel : Dog\n");
    } else if (dynamic_cast<const Cat*>(&animal)) {
        std::print("[DEBUG] Type réel : Cat\n");
    }
}
```

**Frameworks de test.** Les frameworks de mocking et de test utilisent parfois `dynamic_cast` pour vérifier que les objets sont du bon type dans les assertions.

---

## Alternatives modernes

Le C++ moderne offre des alternatives à `dynamic_cast` qui évitent le coût runtime et les cascades de casts :

**`std::variant` et `std::visit` (C++17)** — polymorphisme fermé sans héritage. Si l'ensemble des types possibles est connu à la compilation, `std::variant` est plus sûr, plus performant et vérifié à la compilation :

```cpp
using Animal = std::variant<Dog, Cat, Bird>;

void process(const Animal& animal) {
    std::visit([](const auto& a) { a.perform_action(); }, animal);
}
```

Le compilateur garantit que tous les cas sont couverts — un oubli provoque une erreur de compilation, pas un bug silencieux à l'exécution. `std::variant` sera couvert en détail dans la section 12.2.

**Le pattern Visitor** — dispatch double sans `dynamic_cast`. Il déplace la résolution de type dans le système de vtable, éliminant le besoin de RTTI. Ce pattern sera couvert au chapitre 44.

**`enum` discriminant + `static_cast`** — quand un champ de type explicite est déjà présent dans la hiérarchie (fréquent dans les parsers, les protocoles réseau, les AST de compilateurs). Le `dynamic_cast` est alors inutile car le type est garanti par le discriminant.

---

## En résumé

`dynamic_cast` est le seul cast C++ qui vérifie le type réel d'un objet à l'exécution. Il retourne `nullptr` en cas d'échec sur un pointeur, lève `std::bad_cast` en cas d'échec sur une référence, et il est le seul mécanisme capable d'effectuer des cross-casts dans les hiérarchies à héritage multiple. Son coût runtime, bien que réel, est rarement un problème en pratique.

Son véritable danger n'est pas la performance mais l'**abus architectural** : les cascades de `dynamic_cast` signalent presque toujours un design qui gagnerait à utiliser le polymorphisme virtuel, `std::variant` ou le pattern Visitor. Utilisé avec discernement — pour les interfaces optionnelles, les plugins, le débogage — il est un outil précieux. Utilisé comme substitut au polymorphisme, il est un anti-pattern.
