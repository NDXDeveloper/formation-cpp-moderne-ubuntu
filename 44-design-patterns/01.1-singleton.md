# 44.1.1 — Singleton Thread-Safe

## Le pattern le plus controversé, implémenté correctement

---

## Définition et intention

Le Singleton garantit qu'une classe ne possède **qu'une seule instance** dans tout le programme et fournit un **point d'accès global** à cette instance. C'est l'un des patterns les plus simples à comprendre, les plus faciles à mal implémenter, et les plus dangereux à surutiliser.

En C++ moderne, la question n'est plus *"comment implémenter un Singleton thread-safe ?"* — le langage a résolu ce problème depuis C++11. La vraie question est : *"ai-je réellement besoin d'un Singleton ?"*

---

## L'évolution historique : du dangereux au trivial

### Le Singleton naïf (pré-C++11) — à ne jamais reproduire

Avant C++11, l'implémentation classique reposait sur un pointeur statique et une vérification manuelle :

```cpp
// ⚠️ CODE HISTORIQUE — NE PAS UTILISER
class Logger {
    static Logger* instance_;
    Logger() = default;
public:
    static Logger* getInstance() {
        if (instance_ == nullptr) {          // Race condition !
            instance_ = new Logger();        // Fuite mémoire potentielle !
        }
        return instance_;
    }
    void log(const std::string& msg);
};

Logger* Logger::instance_ = nullptr;
```

Ce code souffre de **trois problèmes majeurs** :

1. **Race condition** : deux threads peuvent simultanément voir `instance_` à `nullptr` et créer deux instances. Le programme se retrouve avec deux "singletons" — ce qui viole la garantie fondamentale du pattern.
2. **Fuite mémoire** : l'instance allouée avec `new` n'est jamais `delete`. Sur un système moderne avec un processus de longue durée (serveur, démon), cette fuite persiste jusqu'à la terminaison.
3. **Ordre de destruction indéfini** : si d'autres objets statiques dépendent du Singleton, l'ordre de destruction entre unités de compilation est non déterministe (*static initialization order fiasco*).

### Le Double-Checked Locking (DCLP) — piège subtil

Pour corriger la race condition, une technique répandue dans les années 2000 ajoutait un mutex avec double vérification :

```cpp
// ⚠️ CODE HISTORIQUE — SUBTILMENT CASSÉ AVANT C++11
class Logger {
    static Logger* instance_;
    static std::mutex mtx_;
    Logger() = default;
public:
    static Logger* getInstance() {
        if (instance_ == nullptr) {               // 1ère vérification (sans lock)
            std::lock_guard<std::mutex> lock(mtx_);
            if (instance_ == nullptr) {           // 2ème vérification (avec lock)
                instance_ = new Logger();
            }
        }
        return instance_;
    }
};
```

Cette approche semble correcte mais ne l'est pas sans garanties de modèle mémoire. Avant C++11, le standard ne définissait pas de modèle mémoire formel pour le multithreading. L'instruction `instance_ = new Logger()` implique trois opérations (allocation mémoire, construction, assignation du pointeur) que le compilateur ou le CPU pouvaient **réordonner**. Un thread pouvait lire un pointeur non-nul vers un objet pas encore construit.

Depuis C++11, le DCLP peut être corrigé avec `std::atomic`, mais c'est inutilement complexe. Le langage offre une solution bien plus élégante.

---

## Le Meyers' Singleton : la solution moderne

Scott Meyers a proposé une implémentation qui exploite une garantie du standard C++11 (§6.7) : **l'initialisation d'une variable `static` locale est thread-safe**. Le runtime garantit qu'un seul thread exécute l'initialisation, et que tous les autres threads qui tentent d'accéder à la variable pendant cette initialisation sont bloqués jusqu'à sa complétion.

```cpp
class Logger {  
public:  
    // Point d'accès unique — thread-safe garanti depuis C++11
    static Logger& instance() {
        static Logger inst;
        return inst;
    }

    // Suppression de la copie et du déplacement
    Logger(const Logger&)            = delete;
    Logger& operator=(const Logger&) = delete;
    Logger(Logger&&)                 = delete;
    Logger& operator=(Logger&&)      = delete;

    void log(std::string_view message) {
        // Implémentation...
        std::print("[LOG] {}\n", message);
    }

private:
    Logger() {
        // Initialisation coûteuse (ouverture de fichier, etc.)
    }

    ~Logger() {
        // Nettoyage (fermeture de fichier, flush, etc.)
    }
};
```

### Pourquoi cette implémentation est correcte

Quatre propriétés sont garanties :

**Thread-safety de l'initialisation.** Le standard C++11 (§6.7/4) stipule que si le contrôle entre dans la déclaration d'une variable `static` locale de manière concurrente, l'exécution concurrente doit attendre la fin de l'initialisation. GCC et Clang implémentent cette garantie via un *guard variable* et un mécanisme de type futex sur Linux.

**Initialisation paresseuse (lazy).** L'instance n'est créée qu'au premier appel à `instance()`. Si le Singleton n'est jamais utilisé dans une exécution donnée, il n'est jamais construit — aucune allocation inutile.

**Destruction déterministe.** Les variables `static` locales sont détruites dans l'**ordre inverse** de leur construction (LIFO), lors de la terminaison du programme (appel à `std::atexit`). C'est plus prévisible que la destruction d'objets statiques de portée fichier, dont l'ordre entre unités de compilation est indéfini.

**Pas de fuite mémoire.** L'instance est sur la stack statique, pas sur le heap. Aucun `new`, aucun `delete`, aucune possibilité de fuite.

### Pourquoi retourner une référence et non un pointeur

La signature `static Logger& instance()` retourne une **référence**, pas un pointeur. C'est un choix délibéré :

- Une référence ne peut pas être `nullptr` — le code client n'a pas besoin de vérifier la validité.  
- La sémantique est claire : le Singleton possède l'instance, le client l'emprunte.  
- Un pointeur inviterait à faire `delete` sur le résultat, ce qui serait catastrophique sur une variable statique.

## Rendre les opérations du Singleton thread-safe

La garantie du C++11 ne porte que sur l'**initialisation**. Les **opérations** sur l'instance doivent être protégées indépendamment si elles sont appelées depuis plusieurs threads :

```cpp
class Logger {  
public:  
    static Logger& instance() {
        static Logger inst;
        return inst;
    }

    void log(std::string_view message) {
        std::lock_guard lock(mtx_);       // Protection des opérations
        buffer_.emplace_back(message);
        if (buffer_.size() >= flush_threshold_) {
            flush_locked();
        }
    }

    void flush() {
        std::lock_guard lock(mtx_);
        flush_locked();
    }

    Logger(const Logger&)            = delete;
    Logger& operator=(const Logger&) = delete;
    Logger(Logger&&)                 = delete;
    Logger& operator=(Logger&&)      = delete;

private:
    Logger() : flush_threshold_(100) {}
    ~Logger() { flush_locked(); }

    void flush_locked() {
        for (const auto& msg : buffer_) {
            std::print(stderr, "[LOG] {}\n", msg);
        }
        buffer_.clear();
    }

    std::mutex                mtx_;
    std::vector<std::string>  buffer_;
    std::size_t               flush_threshold_;
};
```

Le pattern interne `flush_locked()` (méthode privée appelée avec le mutex déjà acquis) évite la récursion de lock et centralise la logique de flush.

## Cas d'usage légitimes

Le Singleton est justifié lorsque les **trois conditions** suivantes sont réunies simultanément :

### 1. Unicité imposée par le domaine

L'objet représente une ressource qui est **intrinsèquement unique** dans le système : un logger principal, un pool de connexions à une base de données, un gestionnaire de configuration globale, un générateur de nombres aléatoires partagé. Si avoir deux instances provoquerait un bug logique ou une corruption de ressource, le Singleton a sa place.

### 2. Accès depuis de nombreux points du programme

Si l'objet n'est utilisé que dans un module restreint, le passer en paramètre est préférable. Le Singleton se justifie lorsque des couches très éloignées dans la pile d'appels ont besoin du même objet, et que le propager en paramètre à travers toute la chaîne serait excessivement invasif.

### 3. Initialisation paresseuse ou coûteuse

L'objet a un coût de construction significatif (ouverture de fichiers, allocation de pools, connexion réseau) et ne doit être créé que s'il est effectivement utilisé.

Quelques exemples concrets dans un contexte système/DevOps :

```cpp
// Logger applicatif — cas d'usage classique et légitime
auto& logger = Logger::instance();  
logger.log("Service démarré");  

// Pool de connexions — ressource intrinsèquement partagée
auto& pool = ConnectionPool::instance();  
auto conn = pool.acquire();  

// Configuration globale chargée une seule fois
const auto& config = AppConfig::instance();  
auto port = config.get<int>("server.port");  
```

## Quand ne pas utiliser le Singleton

### Variable globale déguisée

Si le Singleton n'a pas de contrainte d'unicité réelle et sert simplement à éviter de passer un objet en paramètre, c'est une **variable globale déguisée**. Le code devient difficile à tester (impossible de substituer un mock) et crée un couplage implicite entre des modules qui semblent indépendants.

```cpp
// ❌ Mauvais usage : pas de contrainte d'unicité réelle
class MathUtils {  
public:  
    static MathUtils& instance();
    double compute(double x);
};

// ✅ Alternative : fonction libre ou classe instanciable
double compute(double x);
```

### Obstacle au testing

Le Singleton est l'ennemi naturel du test unitaire. Un test ne peut pas remplacer l'instance par un mock ou un stub. Si la testabilité est prioritaire (et elle devrait l'être), l'injection de dépendances (section 44.5) est supérieure :

```cpp
// ❌ Couplage fort au Singleton — non testable
class OrderService {
    void process() {
        Logger::instance().log("Processing order");  // Dépendance cachée
    }
};

// ✅ Injection de dépendance — testable
class OrderService {
    std::function<void(std::string_view)> log_;
public:
    explicit OrderService(std::function<void(std::string_view)> logger)
        : log_(std::move(logger)) {}

    void process() {
        log_("Processing order");  // Dépendance explicite et substituable
    }
};
```

### Ordre de destruction problématique

Si un Singleton A dépend d'un Singleton B pendant sa destruction, il faut que B soit détruit **après** A. Les `static` locales sont détruites en ordre LIFO de construction, donc si A est construit avant B et que A::~A() appelle B::instance(), B est déjà détruit. C'est le *dead reference problem*.

```cpp
// ⚠️ Danger : B peut être détruit avant A
class A {  
public:  
    static A& instance() { static A a; return a; }
    ~A() {
        B::instance().cleanup();  // B est-il encore vivant ?
    }
};
```

La solution (Nifty Counter ou Phoenix Singleton) est complexe et fragile. Si vos singletons ont des dépendances croisées à la destruction, c'est un signal fort que le design doit être repensé.

## Alternatives au Singleton

### Injection par constructeur

La méthode la plus propre : l'objet est créé une seule fois dans `main()` (ou un point d'initialisation similaire) et injecté explicitement dans les composants qui en ont besoin.

```cpp
int main() {
    auto logger = std::make_shared<Logger>("/var/log/app.log");
    auto config = load_config("config.yaml");

    auto service = OrderService(logger, config);
    service.run();
}
```

L'unicité est assurée par construction (un seul `make_shared`), la testabilité est totale (on passe un mock en test), et les dépendances sont explicites.

### `std::optional` paresseux au niveau module

Pour un objet à initialisation paresseuse mais limité à un module (une unité de compilation), un `std::optional` dans un namespace anonyme offre un contrôle fin sans les inconvénients du Singleton global :

```cpp
// Dans un .cpp — visibilité limitée à cette unité de compilation
namespace {
    std::optional<HeavyResource> resource;
    std::once_flag init_flag;
}

HeavyResource& get_resource() {
    std::call_once(init_flag, [] {
        resource.emplace(/* params */);
    });
    return *resource;
}
```

Cette approche évite le point d'accès global tout en conservant l'initialisation paresseuse et la thread-safety (via `std::call_once`).

### Contexte applicatif (Application Context)

Plutôt que de disperser les singletons, regrouper les services partagés dans un objet contexte unique, créé au démarrage et propagé à travers l'application :

```cpp
struct AppContext {
    Logger&          logger;
    ConnectionPool&  db_pool;
    const AppConfig& config;
};

class OrderService {  
public:  
    explicit OrderService(const AppContext& ctx) : ctx_(ctx) {}

    void process() {
        ctx_.logger.log("Processing order");
        auto conn = ctx_.db_pool.acquire();
        // ...
    }

private:
    const AppContext& ctx_;
};
```

Un seul objet à propager au lieu de N singletons éparpillés. Les dépendances sont visibles, le testing est simple (on construit un `AppContext` de test), et l'ordre de destruction est contrôlé.

## Singleton et templates : le Singleton générique

Pour éviter de répéter le boilerplate dans chaque classe Singleton, on peut factoriser le mécanisme via CRTP (cf. section 44.3) :

```cpp
template<typename Derived>  
class Singleton {  
public:  
    static Derived& instance() {
        static Derived inst{};
        return inst;
    }

    Singleton(const Singleton&)            = delete;
    Singleton& operator=(const Singleton&) = delete;
    Singleton(Singleton&&)                 = delete;
    Singleton& operator=(Singleton&&)      = delete;

protected:
    Singleton()  = default;
    ~Singleton() = default;
};
```

Utilisation :

```cpp
class Logger : public Singleton<Logger> {
    friend class Singleton<Logger>;  // Accès au constructeur privé
private:
    Logger() { /* initialisation */ }
public:
    void log(std::string_view msg) {
        std::print(stderr, "[LOG] {}\n", msg);
    }
};

// Usage
Logger::instance().log("Hello");
```

Cette approche est élégante mais à utiliser avec discernement : elle facilite la création de singletons, ce qui peut encourager leur prolifération. La facilité d'implémentation ne doit pas faire oublier la question préalable : *"cette classe a-t-elle réellement besoin d'être un Singleton ?"*

## Vérification à la compilation avec `static_assert`

Pour renforcer les invariants du Singleton, on peut ajouter des vérifications à la compilation dans la classe de base CRTP :

```cpp
template<typename Derived>  
class Singleton {  
public:  
    static Derived& instance() {
        static_assert(!std::is_copy_constructible_v<Derived>,
                      "Un Singleton ne doit pas être copiable");
        static_assert(!std::is_move_constructible_v<Derived>,
                      "Un Singleton ne doit pas être déplaçable");
        static Derived inst{};
        return inst;
    }

protected:
    Singleton()  = default;
    ~Singleton() = default;

    Singleton(const Singleton&)            = delete;
    Singleton& operator=(const Singleton&) = delete;
    Singleton(Singleton&&)                 = delete;
    Singleton& operator=(Singleton&&)      = delete;
};
```

Si un développeur oublie de supprimer les opérations de copie dans la classe dérivée, le `static_assert` déclenche une erreur explicite à la compilation plutôt qu'un comportement inattendu à l'exécution.

## Résumé des implémentations

| Approche | Thread-safe | Lazy | Destruction propre | Recommandée |
|---|---|---|---|---|
| Pointeur statique + `new` | ❌ | ✅ | ❌ (fuite) | ❌ Jamais |
| Double-checked locking (pré-C++11) | ⚠️ Fragile | ✅ | ❌ | ❌ Jamais |
| Meyers' Singleton (`static` local) | ✅ (C++11+) | ✅ | ✅ (LIFO) | ✅ **Oui** |
| `std::call_once` + `std::optional` | ✅ | ✅ | Manuelle | ✅ Cas spécifiques |
| CRTP Singleton template | ✅ (C++11+) | ✅ | ✅ (LIFO) | ⚠️ Avec modération |

---

## Points clés à retenir

- Le **Meyers' Singleton** (variable `static` locale) est la seule implémentation à utiliser en C++ moderne. Elle est thread-safe, lazy, sans fuite, et tient en cinq lignes.  
- Toujours **supprimer** les constructeurs de copie/déplacement et les opérateurs d'affectation.  
- La thread-safety de l'initialisation est **gratuite** depuis C++11. Celle des opérations ne l'est pas — protéger les méthodes avec des mutex reste nécessaire.  
- Avant de créer un Singleton, se poser la question : *l'injection de dépendances ou un contexte applicatif ne seraient-ils pas plus appropriés ?* Dans la majorité des cas, la réponse est oui.  
- Les cas légitimes existent : logger, pool de connexions, configuration globale. Dans ces cas, le Meyers' Singleton est la solution propre et idiomatique.
