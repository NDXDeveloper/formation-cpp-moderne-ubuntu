# 44.1.2 — Factory et Abstract Factory

## Déléguer la création d'objets avec élégance

---

## Définition et intention

Le pattern Factory encapsule la logique de création d'objets dans un composant dédié, découplant le code client des types concrets qu'il utilise. Au lieu d'instancier directement un objet avec son constructeur, le client demande à une factory de le créer — sans connaître (ni dépendre de) la classe concrète retournée.

Le Gang of Four distinguait deux variantes :

- **Factory Method** : une méthode (souvent virtuelle) dans une classe de base, redéfinie par les sous-classes pour créer le bon type de produit.  
- **Abstract Factory** : une interface complète qui regroupe la création de plusieurs objets apparentés (une *famille* de produits).

En C++ moderne, ces deux variantes se rejoignent et se simplifient considérablement. La frontière entre elles devient souvent floue : une simple fonction libre retournant un `std::unique_ptr` est techniquement une factory, et les templates permettent de paramétrer la création sans héritage. Cette section couvre le spectre complet, de la factory function la plus simple aux architectures extensibles à base de registry.

---

## Factory Function : le cas le plus courant

La forme la plus simple et la plus fréquente du pattern en C++ moderne est une **fonction libre** qui retourne un smart pointer vers une interface :

```cpp
#include <memory>
#include <string_view>
#include <stdexcept>

// Interface (classe abstraite)
class Compressor {  
public:  
    virtual ~Compressor() = default;
    virtual std::vector<std::byte> compress(std::span<const std::byte> data) = 0;
    virtual std::vector<std::byte> decompress(std::span<const std::byte> data) = 0;
};

// Implémentations concrètes
class GzipCompressor : public Compressor {  
public:  
    std::vector<std::byte> compress(std::span<const std::byte> data) override {
        // Compression gzip...
        return {};
    }
    std::vector<std::byte> decompress(std::span<const std::byte> data) override {
        // Décompression gzip...
        return {};
    }
};

class Lz4Compressor : public Compressor {
    // Même structure...
};

class ZstdCompressor : public Compressor {
    // Même structure...
};

// Factory function — point d'entrée unique
std::unique_ptr<Compressor> create_compressor(std::string_view algorithm) {
    if (algorithm == "gzip") return std::make_unique<GzipCompressor>();
    if (algorithm == "lz4")  return std::make_unique<Lz4Compressor>();
    if (algorithm == "zstd") return std::make_unique<ZstdCompressor>();

    throw std::invalid_argument(
        std::format("Algorithme de compression inconnu : '{}'", algorithm)
    );
}
```

Utilisation côté client :

```cpp
// Le client ne connaît que l'interface Compressor
auto compressor = create_compressor(config.get("compression.algorithm"));  
auto compressed = compressor->compress(raw_data);  
```

### Pourquoi `std::unique_ptr` et non un pointeur brut

Le retour par `std::unique_ptr` est le choix par défaut pour une factory. Il exprime clairement la **sémantique de propriété** : l'appelant devient le propriétaire unique de l'objet créé. Le destructeur sera appelé automatiquement quand le `unique_ptr` sort du scope — aucun risque de fuite, aucun `delete` à écrire.

Si la propriété doit être partagée, l'appelant peut convertir le `unique_ptr` en `shared_ptr` :

```cpp
std::shared_ptr<Compressor> shared = create_compressor("zstd");
```

L'inverse n'est pas possible. C'est pourquoi la factory retourne toujours un `unique_ptr` : c'est le type de propriété le plus restrictif, convertible vers le plus permissif.

### Quand la factory function suffit

Cette forme simple couvre la majorité des besoins en pratique. Elle est appropriée lorsque :

- L'ensemble des types est **relativement stable** (on n'ajoute pas un nouveau compresseur chaque semaine).  
- La logique de sélection est **simple** (un `if`/`else` ou un `switch` sur une valeur de configuration).  
- Le code de la factory et les classes concrètes vivent dans la **même unité de compilation** ou la même bibliothèque.

## Factory avec `std::variant` : le polymorphisme sans vtable

Lorsque l'ensemble des types est **fermé** (connu à la compilation) et que la performance est critique, `std::variant` remplace avantageusement la hiérarchie de classes :

```cpp
#include <variant>
#include <string_view>

// Types valeurs — pas d'héritage, pas de virtual
struct GzipCodec {
    int compression_level = 6;
    // Méthodes de compression/décompression...
};

struct Lz4Codec {
    bool high_compression = false;
    // Méthodes de compression/décompression...
};

struct ZstdCodec {
    int compression_level = 3;
    // Méthodes de compression/décompression...
};

// Le "polymorphisme" est porté par le variant
using Codec = std::variant<GzipCodec, Lz4Codec, ZstdCodec>;

// Factory retournant un variant
Codec create_codec(std::string_view algorithm) {
    if (algorithm == "gzip") return GzipCodec{};
    if (algorithm == "lz4")  return Lz4Codec{};
    if (algorithm == "zstd") return ZstdCodec{};

    throw std::invalid_argument(
        std::format("Codec inconnu : '{}'", algorithm)
    );
}
```

L'utilisation se fait via `std::visit` :

```cpp
auto codec = create_codec("zstd");

// Dispatch statique — pas de vtable, pas d'indirection
std::visit([&](auto& c) {
    auto compressed = c.compress(raw_data);
    // ...
}, codec);
```

### Comparaison avec l'approche par héritage

| Critère | `unique_ptr<Interface>` | `std::variant` |
|---|---|---|
| Allocation | Heap | Stack (inline dans le variant) |
| Dispatch | Dynamique (vtable) | Statique (`std::visit`) |
| Extensibilité | Ouverte (nouveau type = nouvelle classe) | Fermée (recompilation requise) |
| Exhaustivité | Non vérifiée à la compilation | Vérifiée (le compilateur signale les cas manquants) |
| Taille mémoire | Pointeur + allocation | `sizeof` du plus gros type + discriminant |
| Cache-friendliness | Indirection, potentiel cache miss | Données inline, cache-friendly |

La règle est simple : **si vous pouvez lister tous les types à la compilation, préférez `std::variant`**. Si l'ensemble de types est ouvert (plugins, extensions dynamiques), l'héritage avec `unique_ptr` reste la bonne approche.

## Factory avec Registry : extensibilité dynamique

Dans les systèmes extensibles — plugins, formats configurables, protocoles ajoutés à l'exécution — la factory doit pouvoir enregistrer de nouveaux types **sans modifier son code source**. C'est le pattern Factory avec Registry (aussi appelé *self-registering factory*).

Le principe : une `std::unordered_map` associe des clés (chaînes, enums…) à des **fonctions de création**. Chaque type concret s'enregistre auprès du registry, et la factory consulte cette map pour instancier le bon type.

```cpp
#include <functional>
#include <memory>
#include <string>
#include <unordered_map>

class Compressor {  
public:  
    virtual ~Compressor() = default;
    virtual std::vector<std::byte> compress(std::span<const std::byte> data) = 0;
};

// Registry : associe un nom à une fonction de création
class CompressorFactory {  
public:  
    // Type de la fonction de création
    using Creator = std::function<std::unique_ptr<Compressor>()>;

    // Enregistrement d'un nouveau type
    static bool register_type(const std::string& name, Creator creator) {
        return registry().emplace(name, std::move(creator)).second;
    }

    // Création par nom
    static std::unique_ptr<Compressor> create(const std::string& name) {
        auto& reg = registry();
        auto it = reg.find(name);
        if (it == reg.end()) {
            throw std::invalid_argument(
                std::format("Compressor inconnu : '{}'", name)
            );
        }
        return it->second();
    }

    // Liste des types enregistrés (utile pour l'aide CLI, diagnostics)
    static std::vector<std::string> registered_types() {
        std::vector<std::string> names;
        for (const auto& [name, _] : registry()) {
            names.push_back(name);
        }
        return names;
    }

private:
    static std::unordered_map<std::string, Creator>& registry() {
        static std::unordered_map<std::string, Creator> reg;
        return reg;
    }
};
```

### Auto-enregistrement via une variable statique

Chaque type concret s'enregistre automatiquement au démarrage du programme grâce à une variable statique dont la construction déclenche l'enregistrement :

```cpp
class ZstdCompressor : public Compressor {  
public:  
    std::vector<std::byte> compress(std::span<const std::byte> data) override {
        // Implémentation zstd...
        return {};
    }

private:
    // Auto-enregistrement : la variable statique appelle register_type
    // au chargement de l'unité de compilation
    static inline bool registered_ = CompressorFactory::register_type(
        "zstd",
        [] { return std::make_unique<ZstdCompressor>(); }
    );
};
```

Le mot-clé `inline` (C++17) sur la variable statique garantit une initialisation unique même si le header est inclus dans plusieurs unités de compilation. L'enregistrement se produit **avant** l'entrée dans `main()`.

### Utilisation

```cpp
int main() {
    // Tous les compresseurs sont déjà enregistrés
    auto names = CompressorFactory::registered_types();
    std::print("Compresseurs disponibles : ");
    for (const auto& n : names) std::print("{} ", n);
    std::print("\n");

    // Création depuis une configuration ou un argument CLI
    auto compressor = CompressorFactory::create("zstd");
    auto compressed = compressor->compress(raw_data);
}
```

### Pièges du Registry

**Ordre d'initialisation statique.** Le *static initialization order fiasco* peut frapper si le registry lui-même est une variable statique globale. C'est pourquoi nous utilisons une **fonction retournant une référence statique locale** (`registry()`) — la même technique que le Meyers' Singleton (section 44.1.1). La map est créée au premier accès, pas avant.

**Bibliothèques dynamiques.** Si les types concrets vivent dans des `.so` chargés avec `dlopen`, l'auto-enregistrement fonctionne si le linker ne supprime pas les symboles "inutilisés" de la bibliothèque. Le flag `-Wl,--no-as-needed` ou l'attribut `__attribute__((used))` peuvent être nécessaires.

**Thread-safety.** L'enregistrement se produit typiquement avant `main()` (single-threaded). Si des types sont enregistrés dynamiquement après le démarrage (plugins chargés à chaud), un `std::shared_mutex` en lecture/écriture est nécessaire pour protéger la map.

## Factory avec paramètres de construction

Dans les exemples précédents, les créateurs ne prenaient aucun paramètre. En pratique, les objets nécessitent souvent des arguments de construction. Deux approches se distinguent.

### Créateurs avec paramètres communs

Si tous les types concrets partagent le même jeu de paramètres :

```cpp
class CompressorFactory {  
public:  
    using Creator = std::function<std::unique_ptr<Compressor>(int level)>;

    static bool register_type(const std::string& name, Creator creator) {
        return registry().emplace(name, std::move(creator)).second;
    }

    static std::unique_ptr<Compressor> create(const std::string& name, int level) {
        auto& reg = registry();
        auto it = reg.find(name);
        if (it == reg.end()) {
            throw std::invalid_argument(
                std::format("Compressor inconnu : '{}'", name)
            );
        }
        return it->second(level);
    }

    // ...
};

// Enregistrement avec paramètre
static inline bool reg = CompressorFactory::register_type(
    "zstd",
    [](int level) { return std::make_unique<ZstdCompressor>(level); }
);
```

### Créateurs avec configuration hétérogène

Si chaque type a des paramètres différents, une structure de configuration polymorphique ou un `std::any` / `nlohmann::json` (cf. section 24.1) sert d'enveloppe :

```cpp
class CompressorFactory {  
public:  
    using Creator = std::function<
        std::unique_ptr<Compressor>(const nlohmann::json& config)
    >;

    static std::unique_ptr<Compressor> create(
        const std::string& name,
        const nlohmann::json& config
    ) {
        return registry().at(name)(config);
    }

    // ...
};

// Chaque type interprète la config comme il l'entend
static inline bool reg = CompressorFactory::register_type(
    "zstd",
    [](const nlohmann::json& cfg) {
        int level = cfg.value("level", 3);
        int threads = cfg.value("threads", 1);
        return std::make_unique<ZstdCompressor>(level, threads);
    }
);
```

## Abstract Factory : familles d'objets cohérents

L'Abstract Factory intervient lorsqu'un système doit créer des **familles d'objets apparentés** qui doivent être utilisés ensemble. Un exemple classique est un toolkit d'interface graphique (boutons + fenêtres + menus pour un thème donné), mais en contexte système/DevOps, on retrouve ce pattern dans la création de **backends cohérents** : un backend de stockage comprenant un client, un sérialiseur et un logger adaptés à la même technologie.

### Approche classique (héritage)

```cpp
// Famille de produits abstraits
class StorageClient {  
public:  
    virtual ~StorageClient() = default;
    virtual void put(std::string_view key, std::span<const std::byte> data) = 0;
    virtual std::vector<std::byte> get(std::string_view key) = 0;
};

class StorageSerializer {  
public:  
    virtual ~StorageSerializer() = default;
    virtual std::vector<std::byte> serialize(const Metadata& meta) = 0;
};

// Abstract Factory
class StorageBackendFactory {  
public:  
    virtual ~StorageBackendFactory() = default;
    virtual std::unique_ptr<StorageClient>     create_client() = 0;
    virtual std::unique_ptr<StorageSerializer>  create_serializer() = 0;
};

// Factory concrète pour S3
class S3BackendFactory : public StorageBackendFactory {  
public:  
    std::unique_ptr<StorageClient> create_client() override {
        return std::make_unique<S3Client>(/* ... */);
    }
    std::unique_ptr<StorageSerializer> create_serializer() override {
        return std::make_unique<S3Serializer>(/* ... */);
    }
};

// Factory concrète pour le système de fichiers local
class LocalBackendFactory : public StorageBackendFactory {
    // ...
};
```

### Approche moderne (templates + concepts)

Quand les familles de produits sont connues à la compilation, les templates éliminent l'indirection :

```cpp
// Concept définissant ce qu'est un backend de stockage
template<typename T>  
concept StorageBackend = requires {  
    typename T::Client;
    typename T::Serializer;
    requires requires(typename T::Client c, std::string_view k) {
        { c.get(k) } -> std::convertible_to<std::vector<std::byte>>;
    };
};

// Backend S3 — types concrets regroupés dans un trait
struct S3Backend {
    using Client     = S3Client;
    using Serializer = S3Serializer;
};

// Backend local
struct LocalBackend {
    using Client     = LocalClient;
    using Serializer = LocalSerializer;
};

// Code générique contraint par le concept
template<StorageBackend Backend>  
class StorageService {  
    typename Backend::Client     client_;
    typename Backend::Serializer serializer_;

public:
    void store(std::string_view key, const Metadata& meta) {
        auto data = serializer_.serialize(meta);
        client_.put(key, data);
    }
};

// Instanciation — zéro vtable, zéro allocation heap
auto service = StorageService<S3Backend>{};
```

### Quand choisir l'une ou l'autre

L'Abstract Factory classique (héritage) est justifiée quand le **backend est sélectionné à l'exécution** (fichier de configuration, argument CLI, variable d'environnement). C'est le cas typique en DevOps où le déploiement détermine l'infrastructure.

L'Abstract Factory template est préférable quand le choix est **résolu à la compilation** — par exemple dans des bibliothèques paramétrées, des tests compilés avec un backend mock, ou des systèmes embarqués où le backend est fixé par la cible.

Il est également possible de **combiner les deux** : une factory dynamique choisit un backend à l'exécution, puis instancie un template concrétisé pour ce backend. C'est l'approche *type erasure + static dispatch* que nous approfondirons en section 44.4.

## Factory et `enum class` : éviter les magic strings

Plutôt que de sélectionner les types par des chaînes littérales (source d'erreurs silencieuses), un `enum class` offre une sécurité de type à la compilation :

```cpp
enum class CompressionAlgo { Gzip, Lz4, Zstd };

std::unique_ptr<Compressor> create_compressor(CompressionAlgo algo) {
    switch (algo) {
        case CompressionAlgo::Gzip: return std::make_unique<GzipCompressor>();
        case CompressionAlgo::Lz4:  return std::make_unique<Lz4Compressor>();
        case CompressionAlgo::Zstd: return std::make_unique<ZstdCompressor>();
    }
    std::unreachable();  // C++23 — indique au compilateur que ce point
                         // est inatteignable (tous les cas sont couverts)
}
```

Si un développeur ajoute une valeur à l'enum sans mettre à jour le `switch`, le compilateur émet un warning (`-Wswitch`) — voire une erreur avec `-Werror`. Cette vérification d'exhaustivité est un avantage majeur de l'enum sur les chaînes.

Pour les cas où la sélection vient d'une chaîne externe (fichier de config, argument CLI), une fonction de conversion explicite fait le lien :

```cpp
CompressionAlgo parse_algorithm(std::string_view name) {
    if (name == "gzip") return CompressionAlgo::Gzip;
    if (name == "lz4")  return CompressionAlgo::Lz4;
    if (name == "zstd") return CompressionAlgo::Zstd;

    throw std::invalid_argument(
        std::format("Algorithme inconnu : '{}'", name)
    );
}

// Usage
auto algo = parse_algorithm(config.get("compression.algorithm"));  
auto compressor = create_compressor(algo);  
```

La validation de la chaîne est concentrée en un seul point (`parse_algorithm`). Le reste du code manipule un type fort.

## Anti-patterns et pièges courants

### Factory retournant un pointeur brut

```cpp
// ❌ Qui est propriétaire ? Qui fait le delete ?
Compressor* create_compressor(std::string_view algo);
```

Toujours retourner un `std::unique_ptr`. C'est le signal universel en C++ moderne que la propriété est transférée à l'appelant.

### Factory qui en fait trop

Une factory doit **créer** des objets, pas les **configurer**. Si la factory contient de la logique métier, de la validation complexe ou des effets de bord (connexion réseau, ouverture de fichier), elle viole le principe de responsabilité unique. La configuration doit être séparée — le pattern Builder (section 44.1.3) est souvent le bon compagnon.

### Hiérarchie de factories pour trois types

Si le système ne comprend que deux ou trois types concrets et que cette liste est stable, une fonction `if/else` ou un `switch` sur un enum est plus clair qu'une hiérarchie Abstract Factory / Concrete Factory. Le pattern formel se justifie quand la **combinatoire** est réelle (plusieurs familles × plusieurs produits) ou quand l'extensibilité à l'exécution est requise.

### Oublier `std::unreachable()` ou le default

Dans un `switch` sur un enum, omettre le cas `default` est intentionnel (pour que le compilateur puisse vérifier l'exhaustivité). Mais en mode release avec optimisations, si une valeur invalide se glisse dans l'enum (par un cast sauvage), le comportement est indéfini. `std::unreachable()` (C++23) signale explicitement au compilateur et au développeur que ce chemin ne doit jamais être atteint.

## Résumé : choisir la bonne forme de Factory

| Situation | Forme recommandée |
|---|---|
| Ensemble de types fermé, performance critique | `std::variant` + `std::visit` |
| Ensemble de types fermé, polymorphisme runtime acceptable | Factory function + `unique_ptr` |
| Ensemble de types ouvert, extensible à l'exécution | Factory avec Registry |
| Familles de produits cohérents, choix runtime | Abstract Factory (héritage) |
| Familles de produits cohérents, choix compile-time | Abstract Factory (templates + concepts) |
| Sélection par valeur de configuration | `enum class` + `switch` + factory function |

---

## Points clés à retenir

- Une factory retourne **toujours** un `std::unique_ptr` (ou un `std::variant` pour l'approche sans vtable). Jamais un pointeur brut.  
- **`std::variant`** est la factory des hiérarchies fermées : zéro allocation, exhaustivité vérifiée, cache-friendly. C'est le premier choix quand les types sont connus à la compilation.  
- Le **Registry** avec auto-enregistrement est la solution idiomatique pour les systèmes extensibles (plugins, formats configurables). Utiliser une fonction avec `static` local pour le registry lui-même (pas une variable globale).  
- L'**Abstract Factory** formelle se justifie uniquement quand on crée des **familles** d'objets apparentés. Pour un seul type de produit, une factory function suffit.  
- Préférer les **`enum class`** aux chaînes littérales pour la sélection de type. Concentrer le parsing des chaînes externes en un seul point de conversion.  
- Une factory crée des objets — elle ne les configure pas. Séparer création et configuration.
