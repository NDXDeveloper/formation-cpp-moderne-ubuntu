# 44.1.3 — Builder Fluent

## Construire des objets complexes pas à pas

---

## Définition et intention

Le pattern Builder sépare la **construction** d'un objet complexe de sa **représentation**, permettant au même processus de construction de créer des configurations différentes. En termes concrets : quand un objet possède de nombreux paramètres dont certains sont optionnels, avec des valeurs par défaut, des contraintes de validation et potentiellement des dépendances entre paramètres, un constructeur classique devient illisible. Le Builder résout ce problème en offrant une API pas-à-pas, explicite et auto-documentée.

L'adjectif **fluent** désigne le style d'API qui permet le chaînage d'appels : chaque méthode du Builder retourne une référence vers le Builder lui-même, autorisant l'écriture `builder.setX(...).setY(...).build()` en une seule expression.

---

## Le problème : constructeurs illisibles

Considérons la configuration d'un serveur HTTP — un cas réaliste en développement système/DevOps :

```cpp
// ❌ Constructeur "télescopique" — illisible et fragile
auto server = HttpServer(
    "0.0.0.0",       // host — ou est-ce le cert_path ?
    8080,             // port
    16,               // threads — ou max_connections ?
    1024,             // max_connections
    30,               // timeout_seconds
    true,             // enable_tls
    "/etc/certs/...", // cert_path
    "/etc/certs/...", // key_path
    false,            // enable_compression
    65536             // max_body_size
);
```

Ce code souffre de plusieurs problèmes. Les paramètres positionnels sont **ambigus** : rien ne distingue visuellement `16` (threads) de `1024` (connections) ni `true` (TLS) de `false` (compression). L'ajout d'un nouveau paramètre impose de modifier tous les sites d'appel. Et les paramètres optionnels avec valeurs par défaut génèrent une explosion combinatoire de surcharges de constructeurs.

---

## Approche 1 : le Designated Initializers (C++20)

Avant de construire un Builder complet, il faut connaître l'alternative la plus légère : les **designated initializers** de C++20, combinés à une struct de configuration. Pour de nombreux cas, cette approche suffit et rend le Builder superflu.

```cpp
struct HttpServerConfig {
    std::string host              = "0.0.0.0";
    uint16_t    port              = 8080;
    int         thread_count      = std::thread::hardware_concurrency();
    int         max_connections   = 1024;
    int         timeout_seconds   = 30;
    bool        enable_tls        = false;
    std::string cert_path         = "";
    std::string key_path          = "";
    bool        enable_compression = false;
    std::size_t max_body_size     = 65536;
};

class HttpServer {  
public:  
    explicit HttpServer(HttpServerConfig config)
        : config_(std::move(config))
    {
        // Validation dans le constructeur
        if (config_.enable_tls && config_.cert_path.empty()) {
            throw std::invalid_argument("TLS activé mais cert_path vide");
        }
    }

private:
    HttpServerConfig config_;
};
```

Utilisation avec designated initializers :

```cpp
auto server = HttpServer({
    .port              = 9090,
    .thread_count      = 8,
    .enable_tls        = true,
    .cert_path         = "/etc/certs/server.crt",
    .key_path          = "/etc/certs/server.key",
    .enable_compression = true
});
```

Chaque champ est nommé, l'ordre n'importe pas (tant qu'il respecte l'ordre de déclaration dans la struct — contrainte C++20), les valeurs par défaut s'appliquent aux champs omis, et le code est immédiatement lisible.

### Quand les designated initializers suffisent

Cette approche est préférable au Builder lorsque :

- Les paramètres sont **indépendants** les uns des autres (pas de contrainte conditionnelle complexe).  
- La **validation** se résume à quelques vérifications simples dans le constructeur.  
- L'objet est **immutable** après construction (la config ne change plus).  
- Il n'y a pas de logique de construction **séquentielle** (étape A avant étape B).

Le Builder devient nécessaire lorsque ces conditions ne sont pas remplies.

## Approche 2 : le Builder fluent classique

### Structure de base

Le Builder est un objet temporaire qui accumule les paramètres puis construit l'objet final via une méthode `build()` :

```cpp
class HttpServer {  
public:  
    class Builder;  // Déclaration anticipée

    // L'accès au constructeur est privé — seul le Builder peut construire
    void start();
    void stop();

    // Accesseurs const
    std::string_view host() const { return config_.host; }
    uint16_t port() const { return config_.port; }
    // ...

private:
    struct Config {
        std::string host            = "0.0.0.0";
        uint16_t    port            = 8080;
        int         thread_count    = 0;  // 0 = auto-detect
        int         max_connections = 1024;
        int         timeout_seconds = 30;
        bool        enable_tls      = false;
        std::string cert_path;
        std::string key_path;
        bool        enable_compression = false;
        std::size_t max_body_size   = 65536;
    };

    explicit HttpServer(Config config) : config_(std::move(config)) {}
    Config config_;

    friend class Builder;
};
```

### Le Builder lui-même

```cpp
class HttpServer::Builder {  
public:  
    Builder& host(std::string host) {
        config_.host = std::move(host);
        return *this;
    }

    Builder& port(uint16_t port) {
        config_.port = port;
        return *this;
    }

    Builder& threads(int count) {
        config_.thread_count = count;
        return *this;
    }

    Builder& max_connections(int count) {
        config_.max_connections = count;
        return *this;
    }

    Builder& timeout(std::chrono::seconds duration) {
        config_.timeout_seconds = static_cast<int>(duration.count());
        return *this;
    }

    Builder& tls(std::string cert_path, std::string key_path) {
        config_.enable_tls = true;
        config_.cert_path  = std::move(cert_path);
        config_.key_path   = std::move(key_path);
        return *this;
    }

    Builder& compression(bool enable = true) {
        config_.enable_compression = enable;
        return *this;
    }

    Builder& max_body_size(std::size_t size) {
        config_.max_body_size = size;
        return *this;
    }

    // Construction finale avec validation
    HttpServer build() {
        validate();
        return HttpServer(std::move(config_));
    }

private:
    void validate() const {
        if (config_.port == 0) {
            throw std::invalid_argument("Le port ne peut pas être 0");
        }
        if (config_.enable_tls) {
            if (config_.cert_path.empty() || config_.key_path.empty()) {
                throw std::invalid_argument(
                    "TLS activé : cert_path et key_path sont obligatoires"
                );
            }
        }
        if (config_.thread_count < 0) {
            throw std::invalid_argument("thread_count ne peut pas être négatif");
        }
        if (config_.max_body_size == 0) {
            throw std::invalid_argument("max_body_size doit être > 0");
        }
    }

    HttpServer::Config config_;
};
```

### Utilisation

```cpp
auto server = HttpServer::Builder()
    .port(9090)
    .threads(8)
    .tls("/etc/certs/server.crt", "/etc/certs/server.key")
    .compression()
    .max_body_size(1024 * 1024)  // 1 MiB
    .timeout(std::chrono::seconds(60))
    .build();

server.start();
```

Chaque appel est **auto-documenté** par le nom de la méthode. L'ordre des appels n'a pas d'importance (sauf si le design l'impose). Les valeurs par défaut sont héritées de la struct `Config`. La validation est centralisée dans `build()`.

---

## Anatomie du chaînage : pourquoi `return *this`

Le chaînage fonctionne parce que chaque méthode retourne une **référence** vers le Builder :

```cpp
Builder& host(std::string host) {
    config_.host = std::move(host);
    return *this;  // Retourne le Builder lui-même
}
```

Le type de retour `Builder&` (référence lvalue) permet d'enchaîner les appels car chaque appel retourne le même objet. C'est une convention fondamentale du style fluent en C++.

### Pourquoi pas `Builder` par valeur ?

Retourner par valeur (`Builder host(...)`) déclencherait une copie du Builder à chaque appel — coûteux si la config interne contient des `std::string` ou des `std::vector`. En théorie, le compilateur pourrait appliquer la copy elision, mais ce n'est pas garanti dans un contexte de chaînage.

### Pourquoi pas `Builder*` (pointeur) ?

Retourner un pointeur (`Builder* host(...)`) forcerait une syntaxe `->` pour le chaînage (`builder->host(...)->port(...)`) qui est moins lisible et sugère une allocation heap. La référence est le choix idiomatique.

---

## Validation dans `build()` : le moment de vérité

La méthode `build()` est le point de validation unique. C'est un avantage majeur du Builder par rapport au constructeur télescopique : toutes les contraintes sont vérifiées **en un seul endroit**, après que tous les paramètres ont été fournis.

### Validation simple (exceptions)

L'approche vue précédemment lance des exceptions en cas de configuration invalide. C'est le style le plus courant :

```cpp
HttpServer build() {
    validate();                              // Lance si invalide
    return HttpServer(std::move(config_));   // Construit si valide
}
```

### Validation avec `std::expected` (C++23)

Pour les contextes où les exceptions ne sont pas souhaitées (systèmes embarqués, code performance-critique, cf. section 12.8), `std::expected` offre une alternative élégante :

```cpp
std::expected<HttpServer, std::string> build() {
    if (config_.port == 0) {
        return std::unexpected("Le port ne peut pas être 0");
    }
    if (config_.enable_tls && config_.cert_path.empty()) {
        return std::unexpected("TLS activé sans certificat");
    }
    return HttpServer(std::move(config_));
}
```

Utilisation :

```cpp
auto result = HttpServer::Builder()
    .port(9090)
    .tls("/etc/certs/server.crt", "")  // key_path vide — erreur
    .build();

if (!result) {
    std::print(stderr, "Erreur de configuration : {}\n", result.error());
    return 1;
}

auto server = std::move(*result);  
server.start();  
```

Le choix entre exceptions et `std::expected` dépend de la politique d'erreurs du projet (cf. section 17.5). Le Builder s'adapte aux deux approches sans changer de structure.

---

## Builder et move semantics

La méthode `build()` utilise `std::move(config_)` pour **transférer** les données internes vers l'objet construit. C'est un point important : après l'appel à `build()`, le Builder est dans un état *moved-from* et ne doit plus être utilisé.

```cpp
HttpServer build() {
    validate();
    return HttpServer(std::move(config_));  // Transfert sans copie
}
```

Les `std::string` et `std::vector` contenus dans `config_` sont déplacés, pas copiés. Pour un objet avec plusieurs chaînes longues, la différence de performance est significative.

### Builder à usage unique vs réutilisable

Le design ci-dessus produit un Builder **à usage unique** : après `build()`, il est épuisé. C'est le cas le plus courant et le plus sûr.

Si le Builder doit être réutilisable (créer plusieurs objets avec des variations mineures), il faut copier la config au lieu de la déplacer :

```cpp
// Builder réutilisable — copie au lieu de move
HttpServer build() const {
    validate();
    auto config_copy = config_;                   // Copie
    return HttpServer(std::move(config_copy));
}
```

Le `const` sur `build()` signale clairement que le Builder n'est pas modifié. Mais ce design est moins fréquent : en pratique, le Builder est presque toujours un objet temporaire construit inline.

## Variante : Builder avec `std::optional` pour les champs obligatoires

Les valeurs par défaut ne conviennent pas toujours. Certains champs sont **obligatoires** et n'ont pas de défaut raisonnable. `std::optional` permet de distinguer "non renseigné" de "renseigné avec une valeur" :

```cpp
class DatabaseConnection {  
public:  
    class Builder {
    public:
        Builder& host(std::string h) {
            host_ = std::move(h);
            return *this;
        }

        Builder& port(uint16_t p) {
            port_ = p;
            return *this;
        }

        Builder& database(std::string db) {
            database_ = std::move(db);
            return *this;
        }

        Builder& username(std::string user) {
            username_ = std::move(user);
            return *this;
        }

        Builder& password(std::string pwd) {
            password_ = std::move(pwd);
            return *this;
        }

        // Optionnel — valeur par défaut raisonnable
        Builder& pool_size(int size) {
            pool_size_ = size;
            return *this;
        }

        Builder& timeout(std::chrono::seconds t) {
            timeout_ = t;
            return *this;
        }

        DatabaseConnection build() {
            // Vérification des champs obligatoires
            if (!host_)     throw std::invalid_argument("host est obligatoire");
            if (!port_)     throw std::invalid_argument("port est obligatoire");
            if (!database_) throw std::invalid_argument("database est obligatoire");
            if (!username_) throw std::invalid_argument("username est obligatoire");
            if (!password_) throw std::invalid_argument("password est obligatoire");

            return DatabaseConnection(
                std::move(*host_),
                *port_,
                std::move(*database_),
                std::move(*username_),
                std::move(*password_),
                pool_size_.value_or(10),
                timeout_.value_or(std::chrono::seconds(30))
            );
        }

    private:
        // Obligatoires — std::optional (pas de défaut)
        std::optional<std::string> host_;
        std::optional<uint16_t>    port_;
        std::optional<std::string> database_;
        std::optional<std::string> username_;
        std::optional<std::string> password_;

        // Optionnels — std::optional avec value_or() dans build()
        std::optional<int>                    pool_size_;
        std::optional<std::chrono::seconds>   timeout_;
    };
};
```

Le `std::optional` rend explicite la distinction entre champs obligatoires et optionnels. La validation dans `build()` produit des messages d'erreur ciblés indiquant exactement quel champ manque.

## Builder et immutabilité : construire des objets `const`

Un avantage subtil du Builder est qu'il permet de construire des objets **immutables**. L'objet final n'expose aucun setter — toute sa configuration est fixée à la construction :

```cpp
// L'objet final est entièrement const-correct
const auto server = HttpServer::Builder()
    .port(9090)
    .threads(8)
    .build();

// server.set_port(8080);  // ❌ N'existe pas — l'objet est immutable
server.start();             // ✅ Les méthodes const fonctionnent
```

Le Builder est le seul chemin pour configurer l'objet. Une fois construit, l'objet est figé. Cette approche est particulièrement précieuse dans du code concurrent : un objet immutable est intrinsèquement thread-safe.

## Builder depuis une source externe : JSON, YAML, CLI

En contexte DevOps, la configuration provient rarement du code source. Le Builder peut servir d'**adaptateur** entre une source externe (JSON, YAML, arguments CLI) et l'objet final :

```cpp
// Construction depuis un fichier JSON (cf. section 24.1)
HttpServer server_from_json(const nlohmann::json& cfg) {
    auto builder = HttpServer::Builder()
        .host(cfg.value("host", "0.0.0.0"))
        .port(cfg.value("port", 8080))
        .threads(cfg.value("threads", 0))
        .max_connections(cfg.value("max_connections", 1024));

    // Paramètres conditionnels
    if (cfg.contains("tls")) {
        builder.tls(
            cfg["tls"]["cert_path"],
            cfg["tls"]["key_path"]
        );
    }

    if (cfg.value("compression", false)) {
        builder.compression();
    }

    return builder.build();
}
```

```cpp
// Construction depuis des arguments CLI avec CLI11 (cf. section 36.1)
HttpServer server_from_cli(int argc, char* argv[]) {
    auto builder = HttpServer::Builder();

    CLI::App app{"HTTP Server"};

    uint16_t    port = 8080;
    int         threads = 0;
    std::string cert, key;

    app.add_option("-p,--port", port, "Port d'écoute");
    app.add_option("-t,--threads", threads, "Nombre de threads");
    app.add_option("--cert", cert, "Chemin du certificat TLS");
    app.add_option("--key", key, "Chemin de la clé TLS");

    app.parse(argc, argv);

    builder.port(port).threads(threads);

    if (!cert.empty() && !key.empty()) {
        builder.tls(std::move(cert), std::move(key));
    }

    return builder.build();
}
```

Le Builder devient un point de convergence : quelle que soit la source des paramètres (JSON, YAML, CLI, variables d'environnement, tests unitaires), le même processus de construction et de validation s'applique.

## Anti-patterns et pièges courants

### Builder pour un objet simple

Un Builder n'a aucune valeur ajoutée pour une classe à deux ou trois paramètres obligatoires sans logique conditionnelle :

```cpp
// ❌ Over-engineering — un constructeur classique suffit
class Point {  
public:  
    class Builder {
    public:
        Builder& x(double x) { x_ = x; return *this; }
        Builder& y(double y) { y_ = y; return *this; }
        Point build() { return Point(x_, y_); }
    private:
        double x_ = 0, y_ = 0;
    };
    // ...
};

// ✅ Direct et clair
auto p = Point(3.0, 4.0);
```

Règle empirique : le Builder se justifie à partir de **cinq à six paramètres**, ou dès qu'il y a des **paramètres optionnels avec des dépendances conditionnelles**.

### Validation dispersée dans les setters

```cpp
// ❌ Validation à chaque setter — incohérent et incomplet
Builder& port(uint16_t p) {
    if (p == 0) throw std::invalid_argument("Port invalide");
    config_.port = p;
    return *this;
}

Builder& tls(std::string cert, std::string key) {
    // Comment vérifier que port != 443 implique un warning ?
    // On ne connaît pas encore le port à ce stade...
}
```

La validation doit être **centralisée dans `build()`**, quand tous les paramètres sont connus. Les setters ne font que stocker des valeurs. Exception : une validation purement locale (type ou plage d'une valeur isolée) peut figurer dans le setter si elle n'a aucune dépendance avec d'autres champs.

### Builder mutable après `build()`

```cpp
// ❌ Dangereux — le Builder et l'objet partagent-ils des données ?
auto builder = HttpServer::Builder().port(9090);  
auto server1 = builder.build();  
builder.threads(16);               // Le Builder est-il encore valide ?  
auto server2 = builder.build();    // Quel état a config_ après le move ?  
```

Si `build()` utilise `std::move`, le Builder est dans un état *moved-from* après le premier appel. Réutiliser un Builder après `build()` est une erreur logique dans le design à usage unique. Pour un Builder réutilisable, utiliser la copie (cf. section précédente).

## Comparaison des approches de construction

| Approche | Cas d'usage | Avantages | Limites |
|---|---|---|---|
| Constructeur direct | ≤ 3-4 paramètres obligatoires | Simple, pas de boilerplate | Illisible au-delà de 4 paramètres |
| Designated initializers (C++20) | Struct de config, champs indépendants | Nommé, concis, zéro boilerplate | Pas de validation intégrée, ordre de déclaration imposé |
| Builder fluent | Nombreux paramètres, validation complexe, dépendances conditionnelles | Auto-documenté, validation centralisée, immutabilité | Boilerplate du Builder lui-même |
| Factory + Builder | Objet polymorphique avec config complexe | Découplage type + configuration | Complexité accrue, justifiée seulement si les deux besoins coexistent |

---

## Points clés à retenir

- **Designated initializers (C++20)** sont la première option à considérer. Ils couvrent la majorité des cas avec zéro boilerplate. Le Builder ne se justifie que quand la validation, les dépendances conditionnelles ou la construction séquentielle l'imposent.  
- Le chaînage fluent repose sur le retour de `Builder&` (`return *this`). Chaque setter stocke une valeur et rend le contrôle au Builder.  
- Toute la **validation** est centralisée dans `build()`, quand l'ensemble des paramètres est connu. Les setters ne valident pas (ou uniquement des contraintes strictement locales).  
- `build()` utilise **`std::move`** pour transférer les données sans copie. Après `build()`, le Builder ne doit plus être utilisé (design à usage unique).  
- **`std::optional`** distingue les champs obligatoires (non renseigné = erreur) des champs optionnels (non renseigné = valeur par défaut via `value_or()`).  
- **`std::expected`** (C++23) remplace les exceptions dans `build()` pour les contextes qui les évitent.  
- Le Builder est un excellent **adaptateur** entre sources de configuration externes (JSON, YAML, CLI) et objets métier : un seul point de validation quel que soit l'origine des paramètres.
