# 38.3.1 Configuration ccache en CI

## Introduction

La section 2.3 a couvert l'installation et l'utilisation de ccache en développement local. En CI, les principes sont identiques mais l'environnement introduit des contraintes supplémentaires : le cache doit survivre entre les exécutions (alors que le runner est souvent éphémère), les chemins de checkout varient d'une plateforme à l'autre, et plusieurs combinaisons compilateur/standard partagent la même infrastructure. Cette section détaille la configuration complète de ccache pour GitLab CI et GitHub Actions, les variables d'environnement critiques, les stratégies de clé de cache, et les techniques de diagnostic lorsque le taux de hit n'est pas à la hauteur des attentes.

## Variables d'environnement essentielles

La configuration de ccache passe par des variables d'environnement. En CI, certaines variables ont un impact bien plus important qu'en développement local car elles déterminent si le cache sera réellement exploitable d'une exécution à l'autre.

### Configuration recommandée pour la CI

```bash
# ── Emplacement du cache ──
CCACHE_DIR="${CI_PROJECT_DIR}/.ccache"    # GitLab CI  
CCACHE_DIR="${GITHUB_WORKSPACE}/.ccache"  # GitHub Actions  

# ── Taille maximale ──
CCACHE_MAXSIZE="2G"

# ── Vérification du compilateur ──
CCACHE_COMPILERCHECK="content"

# ── Répertoire de base pour les chemins relatifs ──
CCACHE_BASEDIR="${CI_PROJECT_DIR}"        # GitLab CI  
CCACHE_BASEDIR="${GITHUB_WORKSPACE}"      # GitHub Actions  

# ── Tolérance aux macros de temps ──
CCACHE_SLOPPINESS="time_macros,include_file_mtime,include_file_ctime"

# ── Compression du cache ──
CCACHE_COMPRESS="true"  
CCACHE_COMPRESSLEVEL="6"  
```

Examinons chaque variable en détail.

### `CCACHE_DIR` — Emplacement du cache

```yaml
CCACHE_DIR: "${CI_PROJECT_DIR}/.ccache"
```

En CI, le choix de l'emplacement du cache est contraint par le mécanisme de persistance de la plateforme. GitLab CI ne peut cacher que des chemins situés sous `${CI_PROJECT_DIR}` (la racine du checkout). GitHub Actions avec `actions/cache` peut cacher n'importe quel chemin absolu, mais `hendrikmuhs/ccache-action` gère ce détail automatiquement.

Placer le cache sous le répertoire du projet (`${CI_PROJECT_DIR}/.ccache`) est la convention la plus portable et la plus simple. Elle fonctionne sur les deux plateformes et avec tous les mécanismes de persistance.

> ⚠️ **Ne pas utiliser le chemin par défaut `~/.cache/ccache` sur GitLab CI.** Le répertoire home du runner Docker (`/root/` ou `/home/gitlab-runner/`) n'est pas sous `${CI_PROJECT_DIR}` et ne sera donc pas couvert par la directive `cache: paths:`. Le cache est recréé à chaque exécution et le taux de hit reste à 0%.

### `CCACHE_COMPILERCHECK` — Identification du compilateur

```yaml
CCACHE_COMPILERCHECK: "content"
```

Par défaut, ccache identifie le compilateur par son chemin et son timestamp (`mtime`). Ce mécanisme pose un problème systématique en CI conteneurisée : même si l'image Docker contient la même version de GCC, le timestamp du binaire `g++` varie à chaque build de l'image. Résultat : le cache entier est invalidé à chaque mise à jour de l'image de build, même si le compilateur est strictement identique.

La valeur `content` utilise le hash SHA256 du binaire du compilateur. Deux compilateurs identiques (même version, même build) produisent le même hash, indépendamment de leur date d'installation. Le surcoût de ce hashing est négligeable (quelques millisecondes) par rapport au temps économisé en évitant les invalidations injustifiées.

| Valeur | Méthode | Risque en CI | Recommandation |
|--------|---------|-------------|----------------|
| `mtime` (défaut) | Chemin + timestamp | Cache invalidé à chaque rebuild d'image Docker | Non recommandé en CI |
| `content` | Hash du binaire | Aucun risque d'invalidation injustifiée | **Recommandé pour la CI** |
| `string:<version>` | Chaîne définie par l'utilisateur | Risque si la chaîne ne correspond pas au compilateur réel | Utile pour les cross-compilateurs |

### `CCACHE_BASEDIR` — Portabilité des chemins

```yaml
CCACHE_BASEDIR: "${CI_PROJECT_DIR}"
```

ccache inclut par défaut les chemins absolus des fichiers source dans le hash. En CI, le chemin de checkout peut varier entre les exécutions (par exemple `/builds/abc123/0/groupe/projet` vs `/builds/def456/0/groupe/projet` sur GitLab CI avec des runners concurrents). Sans `CCACHE_BASEDIR`, deux checkouts du même code au même commit produisent des hash différents, et le cache est inutilisable.

`CCACHE_BASEDIR` définit un préfixe à soustraire des chemins avant hashing. Les chemins sont normalisés en relatifs par rapport à cette base, ce qui rend le cache portable entre différents répertoires de checkout.

Cette variable est particulièrement importante dans deux cas :

- **Runners GitLab avec builds concurrents** (`concurrent > 1` dans la config du runner) — chaque build concurrent utilise un sous-répertoire différent (`/builds/.../0/`, `/builds/.../1/`).  
- **GitHub Actions** — le chemin de checkout est `/home/runner/work/<repo>/<repo>`, stable entre les exécutions, mais il est prudent de définir `CCACHE_BASEDIR` par habitude.

### `CCACHE_SLOPPINESS` — Tolérance aux variations non significatives

```yaml
CCACHE_SLOPPINESS: "time_macros,include_file_mtime,include_file_ctime"
```

Par défaut, ccache est très conservateur : la moindre variation dans les entrées provoque un cache miss. La variable `CCACHE_SLOPPINESS` relaxe certaines vérifications qui sont source de cache miss injustifiés en CI :

**`time_macros`** — Ignore les macros `__TIME__`, `__DATE__` et `__TIMESTAMP__`. Ces macros intègrent l'heure et la date de compilation dans le fichier objet, ce qui rend chaque build unique. Si votre code (ou une librairie tierce) les utilise, chaque exécution de pipeline provoque un cache miss pour les fichiers concernés. Cette option dit à ccache d'ignorer cette différence. L'impact est que les macros de temps dans les binaires ne seront pas à jour — ce qui est acceptable en CI où la date de build est rarement significative.

**`include_file_mtime`** — Ignore le timestamp des fichiers d'inclusion. En CI, les headers de la librairie standard et des dépendances système peuvent avoir des timestamps différents entre deux exécutions (par exemple après une mise à jour de l'image Docker ou un `apt-get install`). Leur contenu n'a pas changé, mais leur mtime oui. Cette option évite des invalidations injustifiées.

**`include_file_ctime`** — Même logique pour le ctime (date de changement de métadonnées) des fichiers d'inclusion.

D'autres valeurs de `CCACHE_SLOPPINESS` existent (`pch_defines`, `locale`, `modules`), mais les trois ci-dessus couvrent l'immense majorité des causes de cache miss injustifiés en CI.

### `CCACHE_COMPRESS` et `CCACHE_COMPRESSLEVEL` — Compression du cache

```yaml
CCACHE_COMPRESS: "true"  
CCACHE_COMPRESSLEVEL: "6"  
```

La compression réduit la taille du cache sur disque et, surtout, le volume de données transférées lors de la sauvegarde et de la restauration du cache en CI. Un cache non compressé de 2 Go se réduit typiquement à 400-600 Mo une fois compressé, ce qui divise par 3 à 5 le temps de transfert.

Le niveau de compression va de 1 (rapide, compression faible) à 19 (lent, compression maximale, algorithme Zstandard). Le niveau 6 offre un bon compromis : compression significative sans surcoût de CPU perceptible lors de l'écriture dans le cache.

> 💡 Depuis ccache 4.0, la compression est activée par défaut avec Zstandard (zstd). Si votre image Docker contient une version récente de ccache (4.x+), `CCACHE_COMPRESS=true` est redondant mais ne cause aucun problème.

### `CCACHE_MAXSIZE` — Dimensionnement du cache

Le dimensionnement du cache est un compromis entre le taux de hit et le temps de transfert :

| Taille du cache | Volume transféré (compressé) | Temps de transfert | Adapté pour |
|-----------------|-----------------------------|--------------------|-------------|
| 500 Mo | ~100-150 Mo | 5-10s | Petits projets (<50 fichiers) |
| 1 Go | ~200-350 Mo | 8-15s | Projets moyens (50-200 fichiers) |
| 2 Go | ~400-700 Mo | 15-30s | Projets moyens à grands |
| 5 Go | ~1-1.5 Go | 30-60s | Grands projets (500+ fichiers) |

**Commencez à 2 Go** et ajustez en fonction des statistiques. Si `ccache --show-stats` indique de nombreuses évictions (*cleanups performed*) et un cache plein à 100%, augmentez la taille. Si le cache n'est rempli qu'à 30-40% après plusieurs semaines, vous pouvez le réduire pour accélérer les transferts.

## Configuration complète sur GitLab CI

Voici l'intégration complète de ccache dans un pipeline GitLab CI, en reprenant la structure de template établie en section 38.1.1 :

```yaml
# ── Variables globales ──────────────────────────────────────
variables:
  BUILD_DIR: "build"
  # Configuration ccache
  CCACHE_DIR: "${CI_PROJECT_DIR}/.ccache"
  CCACHE_MAXSIZE: "2G"
  CCACHE_COMPILERCHECK: "content"
  CCACHE_BASEDIR: "${CI_PROJECT_DIR}"
  CCACHE_SLOPPINESS: "time_macros,include_file_mtime,include_file_ctime"
  CCACHE_COMPRESS: "true"
  CCACHE_COMPRESSLEVEL: "6"

# ── Template de build avec ccache ───────────────────────────
.cmake_build:
  stage: build
  before_script:
    - ccache --zero-stats
    - ccache --show-config
  script:
    - cmake -B ${BUILD_DIR}
        -G Ninja
        -DCMAKE_BUILD_TYPE=${BUILD_TYPE}
        -DCMAKE_CXX_STANDARD=${CPP_STANDARD}
        -DCMAKE_CXX_COMPILER=${CXX}
        -DCMAKE_C_COMPILER=${CC}
        -DCMAKE_CXX_COMPILER_LAUNCHER=ccache
        -DCMAKE_C_COMPILER_LAUNCHER=ccache
        -DCMAKE_EXPORT_COMPILE_COMMANDS=ON
    - cmake --build ${BUILD_DIR} --parallel $(nproc)
  after_script:
    - echo "=== ccache statistics ==="
    - ccache --show-stats
    - echo "=== cache size ==="
    - du -sh ${CCACHE_DIR} || true
  cache:
    paths:
      - .ccache/
    policy: pull-push
  artifacts:
    paths:
      - ${BUILD_DIR}/bin/
      - ${BUILD_DIR}/lib/
      - ${BUILD_DIR}/tests/
      - ${BUILD_DIR}/CTestTestfile.cmake
    expire_in: 2 hours

# ── Jobs de build concrets ──────────────────────────────────
build-gcc-release:
  extends: .cmake_build
  variables:
    CXX: "g++-15"
    CC: "gcc-15"
    BUILD_TYPE: "Release"
    CPP_STANDARD: "20"
  cache:
    key: "ccache-gcc15-release-${CI_COMMIT_REF_SLUG}"

build-gcc-debug:
  extends: .cmake_build
  variables:
    CXX: "g++-15"
    CC: "gcc-15"
    BUILD_TYPE: "Debug"
    CPP_STANDARD: "20"
  cache:
    key: "ccache-gcc15-debug-${CI_COMMIT_REF_SLUG}"

build-clang-release:
  extends: .cmake_build
  variables:
    CXX: "clang++-20"
    CC: "clang-20"
    BUILD_TYPE: "Release"
    CPP_STANDARD: "20"
  cache:
    key: "ccache-clang20-release-${CI_COMMIT_REF_SLUG}"
```

### Stratégie de clé de cache GitLab CI

La clé de cache mérite une attention particulière. Le pattern `ccache-{compiler}-{build_type}-{branch}` offre un bon équilibre entre isolation et réutilisation :

**Isolation par compilateur et type de build** — Comme expliqué en section 38.3, les caches GCC et Clang sont incompatibles, et les caches Debug et Release le sont également (`-O0 -g` vs `-O2 -DNDEBUG` produisent des fichiers objets différents). L'isolation évite les invalidations croisées.

**Segmentation par branche** — `${CI_COMMIT_REF_SLUG}` isole les caches par branche. Une branche de feature qui modifie massivement les headers ne pollue pas le cache de `main`. Quand la branche est mergée, le cache de `main` est mis à jour progressivement.

**Fallback sur le cache de la branche par défaut.** GitLab CI propose un mécanisme de fallback automatique : si aucun cache n'est trouvé pour la clé exacte de la branche courante, GitLab recherche un cache correspondant sur la branche par défaut (`main`). Cela signifie qu'une nouvelle branche de feature bénéficie immédiatement du cache de `main` sans démarrer à froid. Ce comportement est activé par défaut et ne nécessite aucune configuration supplémentaire.

### Politique de cache pour les jobs non-build

Les jobs qui ne compilent pas (test, lint, package) n'ont pas besoin d'écrire dans le cache ccache. Utiliser `policy: pull` évite un upload inutile :

```yaml
.cmake_test:
  stage: test
  cache:
    key: "ccache-gcc15-release-${CI_COMMIT_REF_SLUG}"
    paths:
      - .ccache/
    policy: pull          # Lecture seule — pas d'upload
```

En pratique, les jobs de test n'ont même pas besoin du cache ccache (ils ne compilent pas). Le cache peut être entièrement désactivé pour ces jobs avec `cache: []`. Le `policy: pull` n'est utile que si le job de test effectue une recompilation partielle (par exemple pour activer les sanitizers).

## Configuration complète sur GitHub Actions

Sur GitHub Actions, l'action `hendrikmuhs/ccache-action` encapsule la logique de cache, mais une configuration manuelle reste possible et offre plus de contrôle.

### Approche recommandée : `ccache-action`

```yaml
jobs:
  build:
    name: "Build — ${{ matrix.compiler }} ${{ matrix.build_type }}"
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      matrix:
        include:
          - compiler: gcc
            cxx: g++-15
            cc: gcc-15
            cache_key: gcc15
          - compiler: clang
            cxx: clang++-20
            cc: clang-20
            cache_key: clang20
        build_type: [Release, Debug]
    env:
      CXX: ${{ matrix.cxx }}
      CC: ${{ matrix.cc }}
      BUILD_TYPE: ${{ matrix.build_type }}
      CCACHE_COMPILERCHECK: content
      CCACHE_BASEDIR: ${{ github.workspace }}
      CCACHE_SLOPPINESS: time_macros,include_file_mtime,include_file_ctime

    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 1

      - name: Install toolchain
        uses: aminya/setup-cpp@v1
        with:
          compiler: ${{ matrix.compiler == 'gcc' && 'gcc-15' || 'llvm-20' }}
          cmake: true
          ninja: true
          ccache: true

      - name: Setup ccache
        uses: hendrikmuhs/ccache-action@v1
        with:
          key: ${{ matrix.cache_key }}-${{ matrix.build_type }}
          max-size: 2G
          verbose: 2

      - name: CMake configure
        run: |
          cmake -B build \
            -G Ninja \
            -DCMAKE_BUILD_TYPE=${{ env.BUILD_TYPE }} \
            -DCMAKE_CXX_STANDARD=20 \
            -DCMAKE_CXX_COMPILER=${{ env.CXX }} \
            -DCMAKE_C_COMPILER=${{ env.CC }} \
            -DCMAKE_CXX_COMPILER_LAUNCHER=ccache \
            -DCMAKE_C_COMPILER_LAUNCHER=ccache \
            -DCMAKE_EXPORT_COMPILE_COMMANDS=ON

      - name: CMake build
        run: cmake --build build --parallel $(nproc)
```

L'action `ccache-action` gère automatiquement la sauvegarde et la restauration du cache via `actions/cache` en interne. Elle ajoute un composant temporel à la clé de cache pour contourner l'immutabilité des clés GitHub : à chaque exécution, une nouvelle clé est créée (par exemple `ccache-gcc15-Release-20260318-143022`), et le `restore-keys` avec le préfixe `ccache-gcc15-Release-` restaure le cache le plus récent.

### Approche manuelle : contrôle complet

Pour les situations où `ccache-action` ne convient pas (politique de sécurité interdisant les actions tierces, besoin de contrôle fin sur le comportement du cache), voici l'équivalent manuel :

```yaml
      - name: Install ccache
        run: sudo apt-get install -y ccache

      - name: Configure ccache
        run: |
          ccache --set-config=max_size=2G
          ccache --set-config=compiler_check=content
          ccache --set-config=base_dir="${{ github.workspace }}"
          ccache --set-config=sloppiness=time_macros,include_file_mtime,include_file_ctime
          ccache --set-config=compress=true
          ccache --set-config=compression_level=6

      - name: Restore ccache
        uses: actions/cache/restore@v4
        with:
          path: ~/.cache/ccache
          key: ccache-${{ matrix.cache_key }}-${{ matrix.build_type }}-${{ github.sha }}
          restore-keys: |
            ccache-${{ matrix.cache_key }}-${{ matrix.build_type }}-

      - name: Zero ccache stats
        run: ccache --zero-stats

      # ... (configure + build)

      - name: Show ccache stats
        if: always()
        run: ccache --show-stats --verbose

      - name: Save ccache
        uses: actions/cache/save@v4
        if: always()
        with:
          path: ~/.cache/ccache
          key: ccache-${{ matrix.cache_key }}-${{ matrix.build_type }}-${{ github.sha }}
```

L'approche manuelle utilise `actions/cache/restore` et `actions/cache/save` séparément (plutôt que l'action combinée `actions/cache`) pour garantir que le cache est sauvegardé même si le build échoue (`if: always()`). C'est important car un build qui échoue à 90% de la compilation a quand même rempli le cache avec des résultats réutilisables — les perdre forcerait le prochain build à repartir de zéro.

**La clé inclut `github.sha`** : chaque commit crée une nouvelle entrée de cache. Le `restore-keys` avec le préfixe (sans le SHA) restaure le cache du commit précédent. Ce pattern garantit que le cache est sauvegardé à chaque exécution (puisque la clé exacte n'existe jamais au moment du restore) tout en bénéficiant du cache le plus récent.

## Diagnostic et monitoring

Un cache mal configuré peut donner l'illusion de fonctionner alors que le taux de hit est catastrophique. Surveiller les statistiques ccache est indispensable pour valider la configuration et détecter les régressions.

### Lire les statistiques ccache

La commande `ccache --show-stats` affiche un résumé détaillé :

```
Cacheable calls:      203
  Hits:               189
    Direct:           185
    Preprocessed:       4
  Misses:              14
Uncacheable calls:      2  
Local storage:  
  Cache size (GB):   0.47 / 2.00 (23.5%)
  Hits:               189
  Misses:              14
```

Les métriques clés à surveiller :

**Taux de hit** : `Hits / (Hits + Misses)`. Sur un build incrémental typique (quelques fichiers modifiés), un taux de 85-98% est attendu. En dessous de 70%, quelque chose ne va pas.

**Direct hits vs Preprocessed hits** : les *direct hits* sont les plus rapides — ccache reconnaît le fichier sans même le préprocesser. Les *preprocessed hits* nécessitent un passage par le préprocesseur avant de trouver le résultat dans le cache. Un ratio élevé de preprocessed hits peut indiquer que les headers changent fréquemment ou que `CCACHE_BASEDIR` n'est pas correctement configuré.

**Uncacheable calls** : compilations que ccache ne peut pas cacher (assembleur en ligne, output vers stdout, etc.). Ce nombre doit rester faible.

**Cache size** : si le cache est constamment plein (100%), des évictions se produisent et le taux de hit peut en souffrir. Augmentez `CCACHE_MAXSIZE`.

### Diagnostic des cache miss

Lorsque le taux de hit est anormalement bas, ccache fournit un outil de diagnostic. La variable `CCACHE_LOGFILE` active un log détaillé de chaque décision de cache :

```yaml
# Activer temporairement pour diagnostiquer
variables:
  CCACHE_LOGFILE: "${CI_PROJECT_DIR}/ccache.log"
  CCACHE_DEBUG: "1"

# En fin de job, inspecter les miss
after_script:
  - grep -c "Result: cache miss" ccache.log || true
  - grep -c "Result: cache hit" ccache.log || true
  - grep "Result: cache miss" ccache.log | head -20
```

Le log indique pour chaque cache miss la raison exacte : source modifiée, flag de compilation différent, compilateur différent, chemin différent, etc. C'est l'outil définitif pour comprendre pourquoi le cache n'est pas efficace.

> ⚠️ **Ne pas laisser `CCACHE_DEBUG=1` activé en permanence.** Le mode debug ralentit significativement ccache (écriture d'un fichier de log pour chaque fichier compilé) et produit des fichiers de log volumineux. Utilisez-le uniquement pour le diagnostic ponctuel.

### Causes courantes de taux de hit bas en CI

| Symptôme | Cause probable | Solution |
|----------|----------------|----------|
| 0% de hits | Cache non restauré ou `CCACHE_DIR` mal configuré | Vérifier le chemin et la politique de cache |
| 0% de hits | `CMAKE_CXX_COMPILER_LAUNCHER` absent | Ajouter la directive CMake |
| 100% de miss au premier build de chaque branche | Pas de fallback sur le cache de `main` | GitLab : automatique. GitHub : vérifier `restore-keys` |
| Cache invalidé après chaque mise à jour d'image Docker | `CCACHE_COMPILERCHECK=mtime` (défaut) | Passer à `content` |
| Cache invalidé malgré des sources identiques | Chemins absolus différents entre runners | Configurer `CCACHE_BASEDIR` |
| Évictions fréquentes, taille à 100% | Cache trop petit | Augmenter `CCACHE_MAXSIZE` |
| Miss sur des fichiers utilisant `__TIME__` | Macros de temps dans le hash | Ajouter `time_macros` à `CCACHE_SLOPPINESS` |
| Beaucoup de preprocessed hits mais peu de direct hits | Headers avec mtimes variables | Ajouter `include_file_mtime` à `CCACHE_SLOPPINESS` |

### Intégrer les statistiques dans le suivi du pipeline

Pour un suivi à long terme, vous pouvez extraire les métriques ccache et les exposer comme métriques du pipeline. Sur GitLab CI, un script `after_script` peut formater les statistiques en une ligne parseable par un outil de monitoring :

```yaml
after_script:
  - |
    STATS=$(ccache --print-stats 2>/dev/null)
    HITS=$(echo "$STATS" | grep "cache_hit_direct\|cache_hit_preprocessed" \
      | awk '{s+=$2} END {print s}')
    MISSES=$(echo "$STATS" | grep "cache_miss" | awk '{print $2}')
    TOTAL=$((HITS + MISSES))
    if [ "$TOTAL" -gt 0 ]; then
      RATE=$(awk "BEGIN {printf \"%.1f\", ($HITS/$TOTAL)*100}")
      echo "CCACHE_HIT_RATE=${RATE}% (${HITS}/${TOTAL})"
    fi
```

Sur GitHub Actions, l'action `ccache-action` avec `verbose: 2` affiche automatiquement les statistiques dans le résumé du job (*job summary*), visible directement dans l'interface.

## Gestion du cache sur les branches de feature

Un pattern courant en CI est le suivant : un développeur crée une branche de feature, pousse quelques commits, ouvre une merge request, et itère. Chaque push déclenche un build. La question est : d'où vient le cache au premier build de la branche ?

### Sur GitLab CI : fallback automatique

GitLab CI gère ce cas nativement. Si aucun cache n'existe pour la clé de la branche courante, le runner cherche un cache correspondant sur la branche par défaut (`main`). Le premier build de la branche de feature bénéficie donc du cache de `main` (cache chaud avec un taux de hit élevé), et les builds suivants sur la même branche utilisent leur propre cache progressivement mis à jour.

Ce comportement est activé par défaut. La seule précaution est de s'assurer que la clé de cache inclut bien `${CI_COMMIT_REF_SLUG}` (le nom de la branche) pour que chaque branche ait son propre espace de cache tout en bénéficiant du fallback.

### Sur GitHub Actions : `restore-keys` explicite

GitHub Actions n'a pas de fallback automatique — le mécanisme doit être configuré manuellement via `restore-keys` :

```yaml
- uses: actions/cache/restore@v4
  with:
    path: ~/.cache/ccache
    key: ccache-gcc15-release-${{ github.ref_name }}-${{ github.sha }}
    restore-keys: |
      ccache-gcc15-release-${{ github.ref_name }}-
      ccache-gcc15-release-main-
      ccache-gcc15-release-
```

Les `restore-keys` sont essayées dans l'ordre. Si aucun cache n'existe pour la branche courante (`feature/xyz`), GitHub cherche un cache dont la clé commence par `ccache-gcc15-release-main-` (le cache de `main`), puis par `ccache-gcc15-release-` (n'importe quelle branche). Ce mécanisme en cascade garantit qu'un cache est presque toujours trouvé, même pour une branche fraîchement créée.

L'action `ccache-action` configure automatiquement des `restore-keys` similaires, ce qui est l'une des raisons pour lesquelles elle est recommandée plutôt que la configuration manuelle.

## ccache et les builds avec sanitizers

Les builds avec sanitizers (ASan, TSan, UBSan) ajoutent des flags de compilation significatifs (`-fsanitize=address`, `-fno-omit-frame-pointer`) qui modifient les fichiers objets produits. Le cache d'un build standard n'est pas réutilisable par un build sanitizer et vice versa.

La solution est une clé de cache dédiée pour chaque variante sanitizer :

```yaml
# GitLab CI
build-asan:
  extends: .cmake_build
  variables:
    CXX: "clang++-20"
    CC: "clang-20"
    BUILD_TYPE: "Debug"
    CMAKE_EXTRA_FLAGS: >-
      -DCMAKE_CXX_FLAGS="-fsanitize=address,undefined -fno-omit-frame-pointer"
      -DCMAKE_EXE_LINKER_FLAGS="-fsanitize=address,undefined"
  cache:
    key: "ccache-clang20-asan-${CI_COMMIT_REF_SLUG}"    # Clé dédiée ASan
```

```yaml
# GitHub Actions
- name: Setup ccache (ASan build)
  uses: hendrikmuhs/ccache-action@v1
  with:
    key: clang20-asan                                    # Clé dédiée ASan
    max-size: 1G
```

Le cache sanitizer peut être plus petit (`1G` au lieu de `2G`) car il ne couvre généralement qu'un seul compilateur et un seul type de build.

## Résumé des variables d'environnement

| Variable | Valeur recommandée CI | Rôle |
|----------|----------------------|------|
| `CCACHE_DIR` | `${CI_PROJECT_DIR}/.ccache` | Emplacement du cache (sous le checkout) |
| `CCACHE_MAXSIZE` | `2G` | Taille maximale du cache |
| `CCACHE_COMPILERCHECK` | `content` | Hash du compilateur au lieu du timestamp |
| `CCACHE_BASEDIR` | `${CI_PROJECT_DIR}` | Portabilité des chemins entre runners |
| `CCACHE_SLOPPINESS` | `time_macros,include_file_mtime,include_file_ctime` | Tolérance aux variations non significatives |
| `CCACHE_COMPRESS` | `true` | Compression du cache (réduction transfert) |
| `CCACHE_COMPRESSLEVEL` | `6` | Niveau de compression Zstandard |
| `CCACHE_LOGFILE` | *(diagnostic uniquement)* | Log détaillé des décisions de cache |
| `CCACHE_DEBUG` | *(diagnostic uniquement)* | Mode debug verbeux |
