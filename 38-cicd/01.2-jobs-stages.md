# 38.1.2 Jobs et stages

## Introduction

La section précédente a posé la structure générale du fichier `.gitlab-ci.yml` et les mécanismes fondamentaux (variables, cache, artifacts, templates, `extends`, `needs`). Cette section entre dans le détail concret de chaque job : que fait-il exactement, pourquoi est-il configuré ainsi, et quels pièges éviter. Nous parcourons les cinq stages d'un pipeline C++ mature — lint, build, test, package, deploy — en fournissant pour chacun des configurations complètes et commentées.

Tous les exemples s'appuient sur le projet type présenté en section 38.1 (CMake + Ninja + Google Test) et sur les templates définis en section 38.1.1.

## Stage 1 : Lint

Le stage `lint` est le premier filtre du pipeline. Son rôle est de rejeter rapidement un code qui ne respecte pas les conventions de l'équipe, avant d'engager les ressources coûteuses de la compilation. Ce stage contient typiquement deux jobs : la vérification du formatage et l'analyse statique.

### Job `format-check` : vérification clang-format

```yaml
format-check:
  stage: lint
  image: silkeh/clang:20       # Image légère avec clang-format
  variables:
    GIT_DEPTH: 1
  script:
    - |
      echo "Vérification du formatage sur les fichiers C++..."
      SOURCES=$(find src include tests -type f \( -name '*.cpp' -o -name '*.hpp' -o -name '*.h' \))
      if [ -z "$SOURCES" ]; then
        echo "Aucun fichier source trouvé."
        exit 0
      fi
      echo "$SOURCES" | xargs clang-format-20 --dry-run --Werror 2>&1
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
    - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH'
  allow_failure: false
```

**Pourquoi une image séparée ?** Ce job n'a pas besoin de CMake, Ninja, ccache ou des librairies de développement. Utiliser une image légère contenant uniquement Clang (et donc clang-format) réduit le temps de pull de l'image et isole le job des changements dans l'image de build principale.

**`--dry-run --Werror`** : le flag `--dry-run` empêche clang-format de modifier les fichiers, et `--Werror` transforme chaque différence de formatage en erreur. Si un seul fichier n'est pas correctement formaté, le job échoue et le développeur voit exactement quelles lignes posent problème dans les logs.

**`allow_failure: false`** est la valeur par défaut, mais l'expliciter ici renforce l'intention : un code mal formaté ne doit pas atteindre le stage de build. Certaines équipes préfèrent temporairement `allow_failure: true` pendant une phase de migration vers un nouveau style, puis le repasser à `false` une fois la base de code uniformisée.

### Job `static-analysis` : clang-tidy

L'analyse statique avec clang-tidy est plus complexe car elle nécessite la compilation database (`compile_commands.json`) pour comprendre les flags de compilation, les chemins d'inclusion et les macros de chaque fichier source :

```yaml
static-analysis:
  stage: lint
  script:
    # Étape 1 : Générer la compilation database sans compiler
    - cmake -B ${BUILD_DIR}
        -G Ninja
        -DCMAKE_BUILD_TYPE=Debug
        -DCMAKE_CXX_STANDARD=${CPP_STANDARD}
        -DCMAKE_CXX_COMPILER=clang++-20
        -DCMAKE_EXPORT_COMPILE_COMMANDS=ON
    # Étape 2 : Exécuter clang-tidy sur tous les fichiers source
    - |
      SOURCES=$(find src include -type f \( -name '*.cpp' -o -name '*.hpp' \))
      echo "$SOURCES" | xargs -P $(nproc) clang-tidy-20 \
        -p ${BUILD_DIR}/compile_commands.json \
        --warnings-as-errors='*' \
        --quiet
  cache:
    key: "clang-tidy-${CI_COMMIT_REF_SLUG}"
    paths:
      - .ccache/
    policy: pull
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
    - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH'
  allow_failure: false
```

**La subtilité CMake** : la commande `cmake -B` configure le projet et génère `compile_commands.json`, mais ne lance pas la compilation. C'est suffisant pour clang-tidy, qui a besoin de savoir comment chaque fichier est compilé mais n'a pas besoin des fichiers objets. Cela rend le job nettement plus rapide qu'une compilation complète.

**`-P $(nproc)`** dans `xargs` exécute clang-tidy en parallèle sur plusieurs fichiers, exploitant tous les cœurs du runner. Sans cette option, l'analyse est séquentielle et peut prendre plusieurs minutes sur un projet de taille moyenne.

**`--warnings-as-errors='*'`** transforme tous les diagnostics clang-tidy en erreurs fatales. Le wildcard `'*'` s'applique à toutes les catégories de checks. En pratique, le fichier `.clang-tidy` à la racine du projet contrôle quels checks sont activés — cette option garantit simplement que les checks activés sont bloquants.

**Cache en `pull` uniquement** : ce job ne compile pas réellement, il ne produit donc rien qui mérite d'être caché. Le mode `pull` lui permet de bénéficier du cache ccache existant (pour l'étape CMake configure) sans en polluer le contenu.

### Variante : analyse différentielle sur les merge requests

Sur un gros projet, exécuter clang-tidy sur l'intégralité du code source à chaque merge request peut être lent. Une variante efficace consiste à n'analyser que les fichiers modifiés :

```yaml
static-analysis-diff:
  stage: lint
  script:
    - cmake -B ${BUILD_DIR}
        -G Ninja
        -DCMAKE_CXX_COMPILER=clang++-20
        -DCMAKE_EXPORT_COMPILE_COMMANDS=ON
    - |
      # Récupérer les fichiers C++ modifiés dans cette MR
      git fetch origin ${CI_MERGE_REQUEST_TARGET_BRANCH_NAME}
      CHANGED=$(git diff --name-only --diff-filter=ACMR \
        origin/${CI_MERGE_REQUEST_TARGET_BRANCH_NAME}...HEAD \
        -- '*.cpp' '*.hpp' '*.h')
      if [ -z "$CHANGED" ]; then
        echo "Aucun fichier C++ modifié, analyse ignorée."
        exit 0
      fi
      echo "Fichiers analysés :"
      echo "$CHANGED"
      echo "$CHANGED" | xargs -P $(nproc) clang-tidy-20 \
        -p ${BUILD_DIR}/compile_commands.json \
        --warnings-as-errors='*'
  variables:
    GIT_DEPTH: 0           # Historique complet nécessaire pour le diff
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
```

Notez que `GIT_DEPTH: 0` surcharge la variable globale pour ce job spécifique — le diff entre la branche source et la branche cible nécessite l'accès à l'historique complet.

## Stage 2 : Build

Le stage de build est le cœur du pipeline. Il compile le projet et produit les binaires qui seront testés dans le stage suivant. La complexité principale réside dans la gestion de plusieurs combinaisons compilateur/type de build/standard.

### Template de build

Le template défini en section 38.1.1 est la base de tous les jobs de build. Le voici enrichi avec des détails supplémentaires :

```yaml
.cmake_build:
  stage: build
  script:
    - cmake -B ${BUILD_DIR}
        -G Ninja
        -DCMAKE_BUILD_TYPE=${BUILD_TYPE}
        -DCMAKE_CXX_STANDARD=${CPP_STANDARD}
        -DCMAKE_CXX_COMPILER=${CXX}
        -DCMAKE_C_COMPILER=${CC}
        -DCMAKE_CXX_COMPILER_LAUNCHER=ccache
        -DCMAKE_EXPORT_COMPILE_COMMANDS=ON
        ${CMAKE_EXTRA_FLAGS}
    - cmake --build ${BUILD_DIR} --parallel $(nproc)
    # Vérifier que les binaires de test ont été produits
    - test -d ${BUILD_DIR}/tests || echo "WARNING - Répertoire tests/ absent du build"
  after_script:
    - ccache --show-stats
  cache:
    paths:
      - .ccache/
    policy: pull-push
  artifacts:
    paths:
      - ${BUILD_DIR}/bin/
      - ${BUILD_DIR}/lib/
      - ${BUILD_DIR}/tests/
      - ${BUILD_DIR}/compile_commands.json
      - ${BUILD_DIR}/CTestTestfile.cmake
      - ${BUILD_DIR}/tests/CTestTestfile.cmake
    expire_in: 2 hours
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
    - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH'
    - if: '$CI_COMMIT_TAG'
```

**`${CMAKE_EXTRA_FLAGS}`** est une variable intentionnellement vide par défaut. Les jobs qui étendent ce template peuvent la définir pour ajouter des options CMake sans réécrire tout le bloc `script`. C'est particulièrement utile pour les builds avec sanitizers (section suivante).

**Les artifacts sont ciblés** : plutôt que de déclarer tout `${BUILD_DIR}/`, seuls les répertoires contenant les binaires finaux, les librairies, les tests et les fichiers CTest sont archivés. Les fichiers objets intermédiaires (`.o`) ne sont pas inclus, ce qui réduit considérablement la taille des artifacts.

**Les fichiers `CTestTestfile.cmake`** sont inclus dans les artifacts car CTest en a besoin pour découvrir les tests dans le stage suivant. Sans ces fichiers, `ctest` ne trouve aucun test à exécuter.

### Jobs de build concrets

```yaml
build-gcc-release:
  extends: .cmake_build
  variables:
    CXX: "g++-15"
    CC: "gcc-15"
    BUILD_TYPE: "Release"
  cache:
    key: "gcc15-rel-${CI_COMMIT_REF_SLUG}"

build-gcc-debug:
  extends: .cmake_build
  variables:
    CXX: "g++-15"
    CC: "gcc-15"
    BUILD_TYPE: "Debug"
  cache:
    key: "gcc15-dbg-${CI_COMMIT_REF_SLUG}"

build-clang-release:
  extends: .cmake_build
  variables:
    CXX: "clang++-20"
    CC: "clang-20"
    BUILD_TYPE: "Release"
  cache:
    key: "clang20-rel-${CI_COMMIT_REF_SLUG}"

build-clang-debug:
  extends: .cmake_build
  variables:
    CXX: "clang++-20"
    CC: "clang-20"
    BUILD_TYPE: "Debug"
  cache:
    key: "clang20-dbg-${CI_COMMIT_REF_SLUG}"
```

Quatre jobs, quatre combinaisons, quatre clés de cache distinctes. Si vos runners disposent de quatre exécuteurs disponibles, ces quatre builds s'exécutent en parallèle. Chaque job hérite de toute la logique du template `.cmake_build` et ne redéfinit que ce qui varie.

### Builds avec sanitizers

Les sanitizers méritent des jobs de build dédiés car ils nécessitent des flags de compilation spécifiques et produisent des binaires distincts qui ne peuvent pas se substituer aux binaires "normaux" :

```yaml
build-asan:
  extends: .cmake_build
  variables:
    CXX: "clang++-20"
    CC: "clang-20"
    BUILD_TYPE: "Debug"
    CMAKE_EXTRA_FLAGS: >-
      -DCMAKE_CXX_FLAGS="-fsanitize=address,undefined
        -fno-omit-frame-pointer
        -fno-optimize-sibling-calls"
      -DCMAKE_EXE_LINKER_FLAGS="-fsanitize=address,undefined"
  cache:
    key: "clang20-asan-${CI_COMMIT_REF_SLUG}"

build-tsan:
  extends: .cmake_build
  variables:
    CXX: "clang++-20"
    CC: "clang-20"
    BUILD_TYPE: "Debug"
    CMAKE_EXTRA_FLAGS: >-
      -DCMAKE_CXX_FLAGS="-fsanitize=thread
        -fno-omit-frame-pointer"
      -DCMAKE_EXE_LINKER_FLAGS="-fsanitize=thread"
  cache:
    key: "clang20-tsan-${CI_COMMIT_REF_SLUG}"
```

**Pourquoi Clang pour les sanitizers ?** Les sanitizers sont développés comme partie intégrante du projet LLVM. Bien que GCC les supporte également, l'implémentation Clang est généralement plus mature, mieux documentée, et produit des messages d'erreur plus lisibles.

**`-fno-omit-frame-pointer`** est essentiel pour que les sanitizers puissent produire des stack traces lisibles lorsqu'ils détectent un problème. Sans ce flag, les adresses de retour ne sont pas systématiquement enregistrées sur la pile et les traces deviennent incomplètes.

**ASan et TSan ne sont pas combinables.** AddressSanitizer et ThreadSanitizer ne peuvent pas être activés simultanément dans un même binaire — leurs instrumentations sont incompatibles. C'est pourquoi ce sont deux jobs de build séparés. En revanche, AddressSanitizer et UndefinedBehaviorSanitizer se combinent très bien, ce que fait le job `build-asan` ci-dessus.

### Réduire le nombre de jobs en merge request

Quatre builds réguliers plus deux builds sanitizers, cela fait six jobs de compilation en parallèle. Sur les merge requests, cela peut être excessif, surtout si les runners sont partagés. Une stratégie courante consiste à réduire la matrice sur les MR et à n'exécuter la matrice complète que sur la branche principale :

```yaml
build-gcc-release:
  extends: .cmake_build
  variables:
    CXX: "g++-15"
    CC: "gcc-15"
    BUILD_TYPE: "Release"
  cache:
    key: "gcc15-rel-${CI_COMMIT_REF_SLUG}"
  rules:
    # Toujours exécuter sur main et sur les tags
    - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH'
    - if: '$CI_COMMIT_TAG'
    # Sur les MR, exécuter aussi
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'

build-clang-debug:
  extends: .cmake_build
  variables:
    CXX: "clang++-20"
    CC: "clang-20"
    BUILD_TYPE: "Debug"
  cache:
    key: "clang20-dbg-${CI_COMMIT_REF_SLUG}"
  rules:
    # Uniquement sur main et tags — pas sur les MR
    - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH'
    - if: '$CI_COMMIT_TAG'

build-asan:
  extends: .cmake_build
  # ...
  rules:
    # Sur les MR : uniquement si le code de test a changé
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
      changes:
        - src/**/*
        - tests/**/*
    - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH'
```

Avec cette configuration, une merge request ne déclenche qu'un build GCC Release et un build ASan (si des fichiers source ont changé), tandis que la branche principale exécute la matrice complète. C'est un compromis pragmatique entre couverture et vitesse de feedback.

## Stage 3 : Test

Le stage de test exécute les suites de tests sur les binaires produits par le stage de build. La directive `needs` relie chaque job de test au job de build dont il dépend.

### Template de test

```yaml
.cmake_test:
  stage: test
  variables:
    CTEST_OUTPUT_ON_FAILURE: "1"
  script:
    - cd ${BUILD_DIR}
    - ctest --parallel $(nproc) --timeout 120 --no-compress-output
        --test-output-size-passed 65536
        --test-output-size-failed 65536
  cache: []               # Pas de cache nécessaire pour les tests
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
    - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH'
    - if: '$CI_COMMIT_TAG'
```

**`CTEST_OUTPUT_ON_FAILURE: "1"`** indique à CTest d'afficher la sortie standard des tests qui échouent. Sans cette variable, CTest n'affiche qu'un résumé PASS/FAIL et le développeur doit fouiller dans les fichiers de log pour comprendre la cause de l'échec.

**`--parallel $(nproc)`** exécute les tests en parallèle. Google Test produit un binaire par suite de tests, et CTest peut lancer plusieurs de ces binaires simultanément. Sur un runner avec 8 cœurs, 8 suites de tests s'exécutent en même temps.

**`--timeout 120`** impose une durée maximale de 120 secondes par test. C'est une protection contre les tests qui bouclent ou deadlockent — un problème fréquent avec du code concurrent.

**`cache: []`** désactive explicitement le cache pour les jobs de test. Ces jobs ne compilent pas et n'ont pas besoin de ccache.

### Jobs de test concrets

```yaml
test-gcc-release:
  extends: .cmake_test
  needs:
    - job: build-gcc-release
      artifacts: true

test-gcc-debug:
  extends: .cmake_test
  needs:
    - job: build-gcc-debug
      artifacts: true

test-clang-release:
  extends: .cmake_test
  needs:
    - job: build-clang-release
      artifacts: true
```

Chaque job de test est minimaliste : il étend le template et déclare son unique dépendance via `needs`. Grâce à `needs`, `test-gcc-release` démarre dès que `build-gcc-release` est terminé, sans attendre les builds Clang.

### Jobs de test avec sanitizers

Les tests avec sanitizers nécessitent une configuration supplémentaire car les sanitizers communiquent leurs résultats via des variables d'environnement et peuvent nécessiter plus de ressources (mémoire, temps) que les tests classiques :

```yaml
test-asan:
  extends: .cmake_test
  needs:
    - job: build-asan
      artifacts: true
  variables:
    # Options AddressSanitizer
    ASAN_OPTIONS: >-
      detect_leaks=1
      detect_stack_use_after_return=1
      check_initialization_order=1
      strict_init_order=1
      halt_on_error=1
    # Options UndefinedBehaviorSanitizer
    UBSAN_OPTIONS: >-
      print_stacktrace=1
      halt_on_error=1
  script:
    - cd ${BUILD_DIR}
    - ctest --parallel $(nproc) --timeout 300
  allow_failure: false

test-tsan:
  extends: .cmake_test
  needs:
    - job: build-tsan
      artifacts: true
  variables:
    TSAN_OPTIONS: >-
      second_deadlock_stack=1
      halt_on_error=1
      history_size=4
  script:
    - cd ${BUILD_DIR}
    - ctest --parallel 2 --timeout 300
  allow_failure: false
```

**`detect_leaks=1`** dans `ASAN_OPTIONS` active LeakSanitizer en plus d'AddressSanitizer. Cela détecte les fuites mémoire en plus des accès invalides (buffer overflow, use-after-free, double free).

**`halt_on_error=1`** arrête le programme au premier problème détecté plutôt que de tenter de continuer. En CI, c'est le comportement souhaité : un seul problème de mémoire suffit à faire échouer le job.

**Timeout augmenté à 300 secondes** : les sanitizers ajoutent une instrumentation significative qui ralentit l'exécution des tests. Un facteur 2x à 10x est courant selon le sanitizer et la nature du code. Le timeout doit être ajusté en conséquence.

**Parallélisme réduit pour TSan** (`--parallel 2`) : ThreadSanitizer instrumente les accès mémoire de chaque thread, ce qui est très gourmand en mémoire. Exécuter trop de tests TSan en parallèle peut saturer la RAM du runner. Deux tests simultanés sont un compromis prudent — ajustez selon les ressources de votre runner.

### Rapports de tests au format JUnit

GitLab CI peut afficher les résultats des tests directement dans l'interface de la merge request si les jobs produisent un rapport au format JUnit XML. Google Test et CTest supportent tous deux ce format :

```yaml
.cmake_test_with_report:
  extends: .cmake_test
  script:
    - cd ${BUILD_DIR}
    - ctest --parallel $(nproc) --timeout 120
        --output-junit ${CI_PROJECT_DIR}/report-${CI_JOB_NAME}.xml
  artifacts:
    when: always             # Uploader même si les tests échouent
    reports:
      junit: report-*.xml
    expire_in: 7 days
```

**`when: always`** est crucial ici. Par défaut, les artifacts ne sont uploadés que si le job réussit. Or, le rapport JUnit est précisément le plus utile quand des tests échouent — il permet à GitLab d'afficher dans la merge request exactement quels tests ont cassé et leur message d'erreur.

**`reports: junit:`** est une directive spéciale de GitLab CI qui parse le fichier XML et l'intègre dans l'interface : onglet "Tests" de la merge request, indication du nombre de tests passés/échoués, et diff avec la branche cible.

Les jobs de test concrets étendent alors ce template enrichi :

```yaml
test-gcc-release:
  extends: .cmake_test_with_report
  needs:
    - job: build-gcc-release
      artifacts: true
```

## Stage 4 : Package

Le stage `package` génère les livrables du projet : paquets Debian, images Docker, archives binaires. Ce stage ne s'exécute généralement que sur des événements spécifiques (tag de version, merge sur la branche principale).

### Packaging DEB

```yaml
package-deb:
  stage: package
  needs:
    - job: build-gcc-release
      artifacts: true
  script:
    - |
      VERSION=${CI_COMMIT_TAG:-0.0.0-dev}
      VERSION=${VERSION#v}            # Supprimer le préfixe 'v' si présent
      PKG_NAME="mon-projet"
      PKG_DIR="pkg-build/${PKG_NAME}_${VERSION}_amd64"

      # Créer la structure du paquet
      mkdir -p ${PKG_DIR}/DEBIAN
      mkdir -p ${PKG_DIR}/usr/local/bin

      # Copier les binaires
      cp ${BUILD_DIR}/bin/* ${PKG_DIR}/usr/local/bin/

      # Générer le fichier control
      cat > ${PKG_DIR}/DEBIAN/control << EOF
      Package: ${PKG_NAME}
      Version: ${VERSION}
      Section: utils
      Priority: optional
      Architecture: amd64
      Maintainer: Équipe Dev <dev@exemple.com>
      Description: Description du projet
       Description longue du projet sur
       plusieurs lignes.
      EOF

      # Construire le paquet
      dpkg-deb --build ${PKG_DIR}
      mv pkg-build/*.deb .
  artifacts:
    paths:
      - "*.deb"
    expire_in: 90 days
  rules:
    - if: '$CI_COMMIT_TAG =~ /^v\d+\.\d+\.\d+$/'
```

**Versioning automatique** : la version est extraite du tag Git (`v1.2.3` → `1.2.3`). Si le job est exécuté sans tag (par exemple pour un test de packaging sur `main`), la version tombe sur `0.0.0-dev`. Ce mécanisme simple garantit que les paquets de release sont correctement versionnés sans intervention manuelle.

**La règle `rules`** limite l'exécution aux tags qui respectent le format semver (`v1.2.3`). Un push de tag `v1.0.0` déclenche le packaging ; un tag `test-deploy` ou une branche quelconque ne le déclenche pas.

### Construction d'image Docker

Pour les projets déployés en conteneurs, le stage package inclut un job de build Docker :

```yaml
docker-build:
  stage: package
  image: docker:27
  services:
    - docker:27-dind
  needs:
    - job: build-gcc-release
      artifacts: true
  variables:
    DOCKER_TLS_CERTDIR: "/certs"
    IMAGE_TAG: "${CI_REGISTRY_IMAGE}:${CI_COMMIT_TAG:-latest}"
  script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
    - docker build
        --build-arg BUILD_DIR=${BUILD_DIR}
        -t ${IMAGE_TAG}
        -f Dockerfile.runtime .
    - docker push ${IMAGE_TAG}
    # Taguer aussi comme 'latest' si c'est un tag de version
    - |
      if [ -n "$CI_COMMIT_TAG" ]; then
        docker tag ${IMAGE_TAG} ${CI_REGISTRY_IMAGE}:latest
        docker push ${CI_REGISTRY_IMAGE}:latest
      fi
  rules:
    - if: '$CI_COMMIT_TAG =~ /^v\d+\.\d+\.\d+$/'
    - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH'
```

Ce job utilise Docker-in-Docker (`docker:27-dind` comme service) pour construire l'image. Le `Dockerfile.runtime` référencé ici est un Dockerfile minimal qui copie les binaires pré-compilés dans une image d'exécution légère — typiquement un multi-stage réduit au stage final puisque la compilation a déjà été faite par le pipeline CI :

```dockerfile
# Dockerfile.runtime — Image d'exécution uniquement
FROM ubuntu:24.04 AS runtime  
RUN apt-get update && apt-get install -y --no-install-recommends \  
    libstdc++6 ca-certificates \
    && rm -rf /var/lib/apt/lists/*

ARG BUILD_DIR=build  
COPY ${BUILD_DIR}/bin/mon-application /usr/local/bin/  
ENTRYPOINT ["mon-application"]  
```

**`CI_REGISTRY_IMAGE`**, **`CI_REGISTRY_USER`** et **`CI_REGISTRY_PASSWORD`** sont des variables prédéfinies par GitLab qui donnent accès au Container Registry intégré du projet. Aucune configuration manuelle de credentials n'est nécessaire.

## Stage 5 : Deploy

Le stage `deploy` publie les artifacts générés. Son contenu varie considérablement selon le mode de distribution du projet.

### Création d'une release GitLab

GitLab CI permet de créer une release associée à un tag, avec des assets téléchargeables (binaires, paquets) :

```yaml
create-release:
  stage: deploy
  image: registry.gitlab.com/gitlab-org/release-cli:latest
  needs:
    - job: package-deb
      artifacts: true
  script:
    - echo "Création de la release ${CI_COMMIT_TAG}"
  release:
    tag_name: $CI_COMMIT_TAG
    name: "Release $CI_COMMIT_TAG"
    description: |
      ## Changements

      Voir le CHANGELOG pour les détails.

      ## Artifacts

      - Paquet Debian (.deb) pour Ubuntu 24.04 amd64
      - Image Docker : `${CI_REGISTRY_IMAGE}:${CI_COMMIT_TAG}`
    assets:
      links:
        - name: "Paquet Debian amd64"
          url: "${CI_PROJECT_URL}/-/jobs/${CI_JOB_ID}/artifacts/file/mon-projet_*.deb"
          filepath: "/binaries/mon-projet.deb"
          link_type: "package"
  rules:
    - if: '$CI_COMMIT_TAG =~ /^v\d+\.\d+\.\d+$/'
```

Le bloc `release:` est une directive native de GitLab CI qui interagit avec l'API Releases. L'image `release-cli` est requise car c'est elle qui contient l'outil `release-cli` utilisé en coulisse par GitLab pour créer la release.

### Déploiement de la documentation

Si le projet génère de la documentation (Doxygen, mdBook, Sphinx), un job de déploiement vers GitLab Pages peut la publier automatiquement :

```yaml
pages:
  stage: deploy
  script:
    - cmake -B ${BUILD_DIR}-docs
        -G Ninja
        -DBUILD_DOCS=ON
    - cmake --build ${BUILD_DIR}-docs --target docs
    - mv ${BUILD_DIR}-docs/docs/html public
  artifacts:
    paths:
      - public
  rules:
    - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH'
```

Le nom du job **doit** être exactement `pages` — c'est une convention GitLab. Le répertoire `public/` est automatiquement déployé sur GitLab Pages lorsqu'il est déclaré comme artifact d'un job nommé `pages`.

## Vue d'ensemble du pipeline complet

Voici le graphe d'exécution complet du pipeline tel qu'il apparaîtrait dans l'interface GitLab, en montrant les dépendances `needs` :

```
                    ┌──────────────────┐
                    │   format-check   │
                    └────────┬─────────┘
                    ┌────────┴─────────┐
                    │ static-analysis  │
                    └────────┬─────────┘
          ┌──────────────────┼──────────────────┐
          ▼                  ▼                  ▼
 ┌─────────────────┐ ┌────────────────┐ ┌────────────┐
 │build-gcc-release│ │build-clang-rel │ │ build-asan │    ← stage: build (parallèle)
 └────────┬────────┘ └───────┬────────┘ └─────┬──────┘
          ▼                  ▼                ▼
 ┌─────────────────┐ ┌────────────────┐ ┌────────────┐
 │test-gcc-release │ │test-clang-rel  │ │ test-asan  │    ← stage: test (via needs)
 └────────┬────────┘ └────────────────┘ └────────────┘
          ▼
 ┌─────────────────┐ ┌────────────────┐
 │   package-deb   │ │  docker-build  │                   ← stage: package (tags)
 └────────┬────────┘ └────────────────┘
          ▼
 ┌─────────────────┐ ┌────────────────┐
 │ create-release  │ │     pages      │                   ← stage: deploy
 └─────────────────┘ └────────────────┘
```

Grâce aux dépendances `needs`, les flux GCC, Clang et ASan sont indépendants. Le test GCC démarre dès que le build GCC est terminé, sans attendre le build Clang. Le stage `package` attend la fin des tests (séquencement par stage) mais ne dépend concrètement que du build GCC Release pour le packaging DEB.
