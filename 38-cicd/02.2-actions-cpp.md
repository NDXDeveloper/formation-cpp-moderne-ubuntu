# 38.2.2 Actions utiles pour C++

## Introduction

L'un des avantages majeurs de GitHub Actions par rapport à d'autres plateformes CI est son Marketplace : un catalogue de milliers d'actions pré-construites qui encapsulent des tâches courantes. Pour un projet C++, ces actions permettent de remplacer des dizaines de lignes de scripts shell par une invocation concise et maintenable. L'installation d'un compilateur, la configuration de ccache, la publication de rapports de test ou la création d'une release deviennent des opérations en quelques lignes de YAML.

Cette section présente les actions les plus utiles pour un projet C++ en 2026, organisées par catégorie. Pour chaque action, nous détaillons la configuration, les paramètres pertinents et les pièges à connaître.

## Actions fondamentales

Ces actions sont utilisées dans pratiquement tous les workflows C++ et constituent le socle sur lequel reposent les jobs de build et de test.

### `actions/checkout` — Récupération du code source

```yaml
- name: Checkout
  uses: actions/checkout@v4
  with:
    submodules: recursive
    fetch-depth: 1
```

C'est invariablement le premier step de tout job. Les paramètres pertinents pour un projet C++ :

| Paramètre | Valeur recommandée | Justification |
|-----------|-------------------|---------------|
| `fetch-depth` | `1` | Shallow clone — réduit le temps de checkout. Passer à `0` si `git describe` est nécessaire pour le versioning |
| `submodules` | `recursive` | Indispensable si le projet intègre des dépendances via sous-modules Git (pattern courant en C++) |
| `lfs` | `true` | Nécessaire uniquement si le dépôt utilise Git LFS pour des assets binaires (données de test, modèles 3D) |

Un point souvent négligé : sur les pull requests, `actions/checkout` récupère par défaut le commit de *merge* (fusion de la branche source dans la cible), pas le HEAD de la branche source. Ce comportement est généralement souhaitable pour le CI car il teste le résultat réel du merge. Si vous avez besoin du HEAD de la branche source (par exemple pour un diff), utilisez `ref: ${{ github.event.pull_request.head.sha }}`.

### `actions/upload-artifact` et `actions/download-artifact` — Transfert de fichiers entre jobs

```yaml
# Job de build : upload
- name: Upload build artifacts
  uses: actions/upload-artifact@v4
  with:
    name: build-${{ matrix.compiler }}-${{ matrix.build_type }}
    path: |
      build/bin/
      build/lib/
      build/tests/
      build/CTestTestfile.cmake
      build/tests/CTestTestfile.cmake
    retention-days: 1
    if-no-files-found: error
    compression-level: 6

# Job de test : download
- name: Download build artifacts
  uses: actions/download-artifact@v4
  with:
    name: build-${{ matrix.compiler }}-${{ matrix.build_type }}
    path: build/
```

Ces deux actions forment le mécanisme de passage d'artifacts entre jobs — l'équivalent de la directive `artifacts:` de GitLab CI.

**`if-no-files-found: error`** fait échouer le step si aucun fichier ne correspond aux chemins spécifiés. Sans cette option, un artifact vide serait uploadé silencieusement, et le job de test échouerait plus tard avec un message d'erreur cryptique ("no tests found"). Mieux vaut échouer tôt avec un message clair.

**`compression-level`** contrôle le niveau de compression (0 à 9, défaut 6). Pour des binaires C++ compilés, le niveau par défaut offre un bon compromis entre taille et vitesse. Les binaires Debug (avec symboles) peuvent être volumineux — la compression réduit significativement le temps de transfert.

**Piège classique : les permissions d'exécution.** L'action `upload-artifact` ne préserve pas les permissions Unix des fichiers. Les binaires de test uploadés perdent leur bit exécutable. Le job de test doit explicitement les restaurer :

```yaml
- name: Restore execute permissions
  run: |
    find build/bin build/tests -type f -executable -o -type f -name '*.sh' \
      | xargs chmod +x 2>/dev/null || true
    # Approche alternative si les noms des binaires sont connus :
    chmod +x build/tests/* build/bin/* 2>/dev/null || true
```

Ce problème n'existe pas dans GitLab CI, où les artifacts conservent leurs métadonnées. C'est l'une des frictions les plus fréquemment rencontrées lors d'une migration de GitLab vers GitHub Actions pour un projet C++.

### `actions/cache` — Cache générique

```yaml
- name: Cache Conan packages
  uses: actions/cache@v4
  with:
    path: ~/.conan2
    key: conan-${{ runner.os }}-${{ hashFiles('conanfile.*') }}
    restore-keys: |
      conan-${{ runner.os }}-
```

L'action `actions/cache` est le mécanisme de cache générique de GitHub Actions. Pour un projet C++, les répertoires à cacher sont typiquement :

| Répertoire | Contenu | Clé recommandée |
|------------|---------|-----------------|
| `~/.conan2` | Packages Conan téléchargés et compilés | `conan-{os}-{hash(conanfile.*)}` |
| `~/.cache/vcpkg` | Packages vcpkg | `vcpkg-{os}-{hash(vcpkg.json)}` |
| `~/.cache/ccache` | Cache de compilation | Préférer `hendrikmuhs/ccache-action` |
| `build/_deps` | Dépendances FetchContent de CMake | `fetchcontent-{os}-{hash(CMakeLists.txt)}` |

**Stratégie de clé et `restore-keys`.** La clé principale inclut un hash des fichiers de dépendances (`conanfile.py`, `vcpkg.json`). Lorsque les dépendances changent, une nouvelle clé est créée et le cache est reconstruit. Les `restore-keys` avec un préfixe moins spécifique servent de fallback : même si les dépendances ont changé, le cache précédent est restauré comme point de départ, et seules les dépendances modifiées sont mises à jour. Sans `restore-keys`, un changement mineur dans `conanfile.py` invaliderait totalement le cache et forcerait un téléchargement complet de toutes les dépendances.

**Immutabilité des clés.** Rappel important : une clé de cache GitHub est immutable. Une fois créée, elle ne peut pas être écrasée. Le cache n'est sauvegardé en fin de job que si la clé exacte n'existait pas au moment du restore. Ce comportement est fondamentalement différent du cache GitLab CI (qui écrase le cache à chaque exécution en mode `pull-push`). Pour les caches qui évoluent progressivement (comme ccache), il faut inclure un élément variable dans la clé (hash des sources, numéro de run) pour qu'un nouveau snapshot soit sauvegardé régulièrement.

## Installation de la toolchain

Sur les runners hébergés GitHub, les versions pré-installées de GCC et Clang ne sont pas toujours celles requises par le projet. Plusieurs actions simplifient l'installation de versions spécifiques.

### `aminya/setup-cpp` — Installation tout-en-un

```yaml
- name: Setup C++ toolchain
  uses: aminya/setup-cpp@v1
  with:
    compiler: gcc-15
    cmake: true
    ninja: true
    ccache: true
    clangtidy: '20'
    clangformat: '20'
    gcovr: true
```

`setup-cpp` est l'action la plus complète pour préparer un environnement de développement C++. Elle peut installer en une seule invocation le compilateur, CMake, Ninja, ccache, les outils Clang, et même des outils de couverture de code. L'action gère les spécificités de chaque OS (Ubuntu, macOS, Windows) de manière transparente.

Les paramètres les plus utilisés pour un projet C++ sur Linux :

| Paramètre | Valeurs courantes | Description |
|-----------|-------------------|-------------|
| `compiler` | `gcc-15`, `llvm-20` | Installe le compilateur avec toutes ses dépendances |
| `cmake` | `true` ou `3.31.0` | Installe CMake (dernière version ou version précise) |
| `ninja` | `true` | Installe Ninja build |
| `ccache` | `true` | Installe ccache |
| `clangtidy` | `'20'` | Installe clang-tidy (version spécifique) |
| `clangformat` | `'20'` | Installe clang-format (version spécifique) |
| `conan` | `true` | Installe Conan 2.x |
| `vcpkg` | `true` | Installe et bootstrap vcpkg |
| `gcovr` | `true` | Installe gcovr pour la couverture de code |

**Avantage principal** : une seule action remplace cinq à dix steps d'installation manuelle (`apt-get install`, `pip install`, téléchargements directs). Le workflow est plus concis et plus résistant aux changements dans les dépôts de paquets.

**Inconvénient** : l'action installe chaque outil séquentiellement, ce qui peut prendre 1 à 3 minutes selon le nombre d'outils demandés. Sur un pipeline optimisé où chaque seconde compte, une image Docker pré-construite avec la toolchain sera toujours plus rapide.

### Installation manuelle ciblée

Quand seul le compilateur doit être installé, une approche manuelle ciblée peut être plus rapide que `setup-cpp` :

```yaml
# GCC via le PPA ubuntu-toolchain-r
- name: Install GCC 15
  run: |
    sudo add-apt-repository ppa:ubuntu-toolchain-r/test -y
    sudo apt-get update
    sudo apt-get install -y g++-15 gcc-15
    sudo update-alternatives --install /usr/bin/g++ g++ /usr/bin/g++-15 150
    sudo update-alternatives --install /usr/bin/gcc gcc /usr/bin/gcc-15 150

# Clang via le script officiel LLVM
- name: Install Clang 20
  run: |
    wget -qO- https://apt.llvm.org/llvm.sh | sudo bash -s -- 20
    sudo update-alternatives --install /usr/bin/clang++ clang++ /usr/bin/clang++-20 200
    sudo update-alternatives --install /usr/bin/clang clang /usr/bin/clang-20 200
```

L'appel à `update-alternatives` est une bonne pratique qui configure le compilateur installé comme choix par défaut. Cela évite de devoir spécifier le chemin complet (`g++-15`) dans chaque commande CMake — `g++` pointe directement vers la version souhaitée.

## Accélération de la compilation

### `hendrikmuhs/ccache-action` — Cache de compilation

```yaml
- name: Setup ccache
  uses: hendrikmuhs/ccache-action@v1
  with:
    key: ${{ matrix.cache_key }}-${{ matrix.build_type }}
    max-size: 2G
    verbose: 2
    append-timestamp: false
```

C'est l'action de référence pour intégrer ccache dans un workflow GitHub Actions. Elle encapsule toute la logique de cache que l'on devrait autrement configurer manuellement : sauvegarde et restauration du répertoire ccache via `actions/cache`, configuration des variables d'environnement, et affichage des statistiques en fin de job.

| Paramètre | Valeur recommandée | Description |
|-----------|-------------------|-------------|
| `key` | `{compiler}-{build_type}` | Clé de cache — séparer par compilateur est impératif |
| `max-size` | `2G` | Taille maximale du cache ccache |
| `verbose` | `2` | Niveau de verbosité des stats (0 à 2) |
| `append-timestamp` | `false` | Désactive l'ajout de timestamp à la clé (contrôle manuel préféré) |

**Pourquoi cette action plutôt que `actions/cache` directement ?** L'action `ccache-action` gère automatiquement plusieurs subtilités : elle configure les variables d'environnement (`CCACHE_DIR`, `CCACHE_MAXSIZE`), elle crée une clé de cache qui inclut un composant temporel pour garantir que le cache est sauvegardé régulièrement (contournant l'immutabilité des clés), et elle affiche les statistiques ccache comme un résumé de job dans l'interface GitHub. Reproduire tout cela manuellement nécessiterait une quinzaine de lignes de YAML.

Après avoir utilisé cette action, il faut s'assurer que CMake utilise ccache comme launcher :

```yaml
- name: CMake configure
  run: |
    cmake -B build \
      -DCMAKE_CXX_COMPILER_LAUNCHER=ccache \
      -DCMAKE_C_COMPILER_LAUNCHER=ccache \
      # ... autres options
```

Sans `-DCMAKE_CXX_COMPILER_LAUNCHER=ccache`, le cache est correctement configuré mais jamais utilisé — le compilateur est invoqué directement sans passer par ccache. C'est une erreur silencieuse qui ne se manifeste que par des temps de build anormalement longs et un taux de cache hit à 0% dans les statistiques.

### `mozilla/sccache-action` — Cache distribué

```yaml
- name: Setup sccache
  uses: mozilla/sccache-action@v0.0.6

- name: CMake configure
  run: |
    cmake -B build \
      -DCMAKE_CXX_COMPILER_LAUNCHER=sccache \
      -DCMAKE_C_COMPILER_LAUNCHER=sccache \
      # ...
```

sccache (Shared Compilation Cache) est l'alternative distribuée à ccache, développée par Mozilla. Son principal avantage est la possibilité de stocker le cache sur un backend distant (S3, GCS, Azure Blob Storage), ce qui le rend partageable entre plusieurs runners et même entre différentes branches ou dépôts.

L'action `mozilla/sccache-action` configure sccache et peut le connecter au cache GitHub Actions par défaut, sans nécessiter de backend cloud externe. La section 38.3 (Accélération CI) couvrira en détail la configuration de sccache avec les backends cloud.

## Analyse statique et qualité de code

### `cpp-linter/cpp-linter-action` — clang-tidy et clang-format intégrés

```yaml
- name: C++ Linter
  uses: cpp-linter/cpp-linter-action@v2
  id: linter
  with:
    style: file                    # Utiliser le .clang-format du projet
    tidy-checks: ''                # Utiliser le .clang-tidy du projet
    lines-changed-only: true       # Analyser uniquement les lignes modifiées
    files-changed-only: true       # Analyser uniquement les fichiers modifiés
    database: build                # Chemin vers compile_commands.json
    version: '20'                  # Version de clang
    thread-comments: true          # Poster des commentaires sur la PR
    step-summary: true             # Résumé dans le job summary

- name: Fail on linter issues
  if: steps.linter.outputs.checks-failed > 0
  run: exit 1
```

Cette action exécute clang-format et clang-tidy, puis poste les résultats directement comme commentaires de review sur la pull request. Chaque problème détecté apparaît comme une annotation sur la ligne de code concernée, exactement comme le ferait un reviewer humain. C'est nettement plus ergonomique que de fouiller dans les logs du job pour trouver les avertissements.

**`lines-changed-only: true`** est un paramètre particulièrement utile pour les projets avec une base de code existante qui n'est pas entièrement conforme. L'analyse se limite aux lignes modifiées dans la PR, ce qui évite d'inonder le développeur de warnings sur du code qu'il n'a pas touché. C'est l'approche dite "boy scout rule" appliquée au CI : on n'exige pas que tout le projet soit parfait, mais on s'assure que chaque modification respecte les standards.

**`thread-comments: true`** nécessite la permission `pull-requests: write` dans le workflow.

### `SonarSource/sonarqube-scan-action` — Analyse approfondie avec SonarQube

```yaml
- name: SonarQube Scan
  uses: SonarSource/sonarqube-scan-action@v5
  env:
    SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
    SONAR_HOST_URL: ${{ vars.SONAR_HOST_URL }}
  with:
    args: >
      -Dsonar.cfamily.compile-commands=build/compile_commands.json
      -Dsonar.cfamily.threads=$(nproc)
```

Pour les équipes qui utilisent SonarQube ou SonarCloud, cette action officielle intègre l'analyse dans le workflow. SonarQube offre une analyse statique plus approfondie que clang-tidy seul, avec détection des code smells, suivi de la dette technique, et métriques de qualité dans le temps. Le paramètre `sonar.cfamily.compile-commands` pointe vers la compilation database générée par CMake — SonarQube l'utilise pour comprendre la configuration du projet.

## Tests et couverture de code

### `mikepenz/action-junit-report` — Rapports de test enrichis

```yaml
- name: Run CTest with JUnit output
  working-directory: build
  run: ctest --parallel $(nproc) --timeout 120
    --output-junit ${{ github.workspace }}/test-report.xml
  env:
    CTEST_OUTPUT_ON_FAILURE: 1

- name: Publish test report
  uses: mikepenz/action-junit-report@v5
  if: always()
  with:
    report_paths: test-report.xml
    include_passed: true
    detailed_summary: true
    check_name: "Test Results — ${{ matrix.compiler }}"
```

Cette action parse les rapports JUnit XML (produits par CTest via `--output-junit` ou directement par Google Test avec `--gtest_output=xml:`) et les affiche comme un check GitHub avec un résumé détaillé : nombre de tests passés, échoués, ignorés, durée totale, et détail de chaque test échoué avec son message d'erreur.

**`if: always()`** est essentiel : sans cette condition, le step de publication ne s'exécute que si tous les steps précédents ont réussi. Or, c'est précisément quand des tests échouent que le rapport est le plus utile. `always()` garantit la publication du rapport quel que soit le résultat des tests.

**`check_name`** personnalise le nom du check affiché dans la PR. Avec une matrice de builds, inclure le compilateur dans le nom permet de distinguer les rapports de chaque combinaison.

### `codecov/codecov-action` — Couverture de code

```yaml
- name: Build with coverage
  run: |
    cmake -B build \
      -DCMAKE_BUILD_TYPE=Debug \
      -DCMAKE_CXX_FLAGS="--coverage -fprofile-arcs -ftest-coverage" \
      -DCMAKE_EXE_LINKER_FLAGS="--coverage"
    cmake --build build --parallel $(nproc)

- name: Run tests
  working-directory: build
  run: ctest --parallel $(nproc)

- name: Generate coverage report
  run: |
    sudo apt-get install -y gcovr
    gcovr --xml-pretty --exclude-unreachable-branches \
      --exclude 'tests/' --exclude 'build/_deps/' \
      -r . --object-directory build \
      -o coverage.xml

- name: Upload to Codecov
  uses: codecov/codecov-action@v5
  with:
    files: coverage.xml
    token: ${{ secrets.CODECOV_TOKEN }}
    flags: unittests
    fail_ci_if_error: true
```

Ce workflow compile avec les flags de couverture GCC (`--coverage`), exécute les tests, génère un rapport XML via gcovr, et l'uploade sur Codecov. L'action poste automatiquement un commentaire sur la PR avec un résumé de la couverture et un diff par rapport à la branche cible.

**`--exclude 'tests/'`** dans gcovr exclut les fichiers de test eux-mêmes de la mesure de couverture. Inclure les fichiers de test gonflerait artificiellement le pourcentage de couverture puisque le code de test est par définition exécuté à 100%. L'exclusion de `build/_deps/` évite de mesurer la couverture des dépendances tierces téléchargées par FetchContent.

**`fail_ci_if_error: true`** fait échouer le job si l'upload vers Codecov échoue. C'est une question de politique d'équipe : certaines préfèrent un échec silencieux pour ne pas bloquer une PR à cause d'un problème d'infrastructure Codecov.

## Packaging et release

### `softprops/action-gh-release` — Création de releases

```yaml
- name: Create GitHub Release
  uses: softprops/action-gh-release@v2
  if: startsWith(github.ref, 'refs/tags/v')
  with:
    files: |
      build/packages/*.deb
      build/packages/*.rpm
      build/bin/mon-application
    generate_release_notes: true
    draft: false
    prerelease: ${{ contains(github.ref_name, 'rc') || contains(github.ref_name, 'beta') }}
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

Cette action crée une release GitHub associée au tag courant et y attache les fichiers spécifiés (paquets DEB/RPM, binaires). Les utilisateurs peuvent ensuite télécharger ces fichiers depuis la page Releases du dépôt.

**`generate_release_notes: true`** demande à GitHub de générer automatiquement les notes de release à partir des PR fusionnées depuis le tag précédent. C'est un gain de temps considérable par rapport à la rédaction manuelle d'un changelog.

**`prerelease`** est conditionné sur le nom du tag : un tag contenant `rc` ou `beta` (comme `v2.0.0-rc1`) est marqué comme pre-release. Les releases stables (`v2.0.0`) sont publiées comme releases normales.

### `docker/build-push-action` — Build et push d'images Docker

```yaml
- name: Login to GitHub Container Registry
  uses: docker/login-action@v3
  with:
    registry: ghcr.io
    username: ${{ github.actor }}
    password: ${{ secrets.GITHUB_TOKEN }}

- name: Build and push Docker image
  uses: docker/build-push-action@v6
  with:
    context: .
    file: Dockerfile.runtime
    push: true
    tags: |
      ghcr.io/${{ github.repository }}:${{ github.ref_name }}
      ghcr.io/${{ github.repository }}:latest
    build-args: |
      BUILD_DIR=build
    cache-from: type=gha
    cache-to: type=gha,mode=max
```

Cette paire d'actions gère l'authentification au registry et le build/push de l'image Docker. Le point remarquable est **`cache-from: type=gha`** et **`cache-to: type=gha,mode=max`** qui utilisent le cache GitHub Actions comme backend pour le cache des couches Docker. Cela signifie que les couches Docker inchangées ne sont pas reconstruites d'un run à l'autre, même sans Docker layer cache local.

Pour un projet C++ avec un multi-stage build (stage de compilation + stage d'exécution), le cache des couches Docker est particulièrement efficace : la couche d'installation des dépendances système (`apt-get install`) n'est reconstruite que lorsque le Dockerfile change, pas à chaque commit.

## Sécurité des actions tierces

Les actions du Marketplace sont du code tiers exécuté dans votre pipeline CI avec accès au code source, aux secrets et au token GitHub. Quelques pratiques de sécurité s'imposent.

**Épingler par hash de commit.** Plutôt que de référencer une action par son tag (`@v4`), vous pouvez utiliser le hash du commit correspondant. Un tag peut être déplacé par le mainteneur (intentionnellement ou suite à une compromission), un hash ne peut pas :

```yaml
# Tag (pratique mais moins sécurisé)
- uses: actions/checkout@v4

# Hash (plus sécurisé, moins lisible)
- uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683  # v4.2.2
```

Pour les projets sensibles, l'épinglage par hash est recommandé par GitHub et par les guides de sécurité de la supply chain. Des outils comme `pin-github-action` automatisent la conversion tag → hash et ajoutent le tag en commentaire pour la lisibilité.

**Auditer les actions avant adoption.** Avant d'intégrer une action communautaire, vérifiez son dépôt source : nombre de stars, activité de maintenance, qualité du code, nombre de contributeurs. Une action maintenue par un seul développeur et rarement mise à jour présente un risque supérieur.

**Préférer les actions officielles.** Quand le choix existe, les actions maintenues par les organisations officielles (`actions/*`, `docker/*`, `github/*`, `codecov/*`) sont plus fiables que les alternatives communautaires.

**Limiter les permissions.** Le bloc `permissions` du workflow restreint les droits du `GITHUB_TOKEN`. Une action malveillante avec un token en lecture seule (`contents: read`) ne peut pas modifier le dépôt ni publier de packages.
