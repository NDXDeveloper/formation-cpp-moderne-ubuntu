# 38.2.1 Structure workflow YAML

## Introduction

Un workflow GitHub Actions est un fichier YAML placé dans le répertoire `.github/workflows/` du dépôt. Contrairement à GitLab CI qui centralise toute la configuration dans un unique `.gitlab-ci.yml`, GitHub Actions encourage la séparation des workflows : un fichier pour le CI (build + test), un autre pour le packaging, un troisième pour la documentation. Cette séparation est particulièrement pertinente pour un projet C++ où les workflows de build quotidien et de release ont des déclencheurs, des durées et des besoins très différents.

Cette section détaille la syntaxe d'un fichier workflow, directive par directive, puis présente un workflow complet et commenté pour un projet C++ construit avec CMake.

## Anatomie d'un fichier workflow

Un workflow pour un projet C++ suit une structure à quatre niveaux :

```yaml
# ── Nom du workflow ─────────────────────────────────────────
name: CI

# ── Événements déclencheurs ─────────────────────────────────
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

# ── Variables d'environnement globales ──────────────────────
env:
  BUILD_DIR: build
  CPP_STANDARD: 20

# ── Permissions ─────────────────────────────────────────────
permissions:
  contents: read

# ── Jobs ────────────────────────────────────────────────────
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: cmake -B build && cmake --build build
```

Examinons chaque bloc en détail.

## Le bloc `name`

```yaml
name: CI C++ — Build & Test
```

Le nom apparaît dans l'onglet Actions du dépôt et dans les badges de statut des pull requests. Pour un projet qui possède plusieurs workflows, des noms descriptifs sont essentiels : `CI C++ — Build & Test`, `Release — Package & Publish`, `Docs — Build & Deploy` permettent d'identifier immédiatement le rôle de chaque workflow.

## Le bloc `on` : événements déclencheurs

Le bloc `on` définit quand le workflow s'exécute. Pour un projet C++, la configuration la plus courante couvre quatre cas d'usage :

```yaml
on:
  # Déclencher sur les push vers main et develop
  push:
    branches: [main, develop]
    paths-ignore:
      - 'docs/**'
      - '**.md'
      - 'LICENSE'

  # Déclencher sur les pull requests ciblant main
  pull_request:
    branches: [main]
    paths:
      - 'src/**'
      - 'include/**'
      - 'tests/**'
      - 'CMakeLists.txt'
      - 'cmake/**'
      - 'conanfile.*'
      - '.github/workflows/**'

  # Build périodique hebdomadaire
  schedule:
    - cron: '0 4 * * 1'

  # Déclenchement manuel avec paramètres
  workflow_dispatch:
    inputs:
      build_type:
        description: 'Type de build (Debug/Release)'
        required: false
        default: 'Release'
        type: choice
        options: [Debug, Release]
```

### Filtrage par chemins : `paths` vs `paths-ignore`

Pour un projet C++, le filtrage par chemins est un levier d'optimisation important. Il évite de déclencher une compilation de plusieurs minutes lorsqu'un commit ne touche que la documentation ou le README.

**`paths-ignore`** (sur `push`) fonctionne en liste noire : le workflow s'exécute sauf si seuls des fichiers correspondant aux patterns ont changé. C'est adapté au push sur `main` où l'on veut compiler par défaut et ne faire exception que pour des modifications clairement non-techniques.

**`paths`** (sur `pull_request`) fonctionne en liste blanche : le workflow ne s'exécute que si au moins un fichier correspondant aux patterns a changé. C'est plus restrictif et adapté aux pull requests où chaque exécution consomme des minutes CI.

Les deux directives sont mutuellement exclusives au sein d'un même événement — on ne peut pas combiner `paths` et `paths-ignore` dans le même bloc `push` ou `pull_request`.

> ⚠️ **Attention au piège des required checks.** Si un workflow est configuré comme "required status check" sur une branche protégée et qu'il est filtré par `paths`, une pull request qui ne modifie que de la documentation ne déclenchera jamais le workflow — et ne pourra donc jamais satisfaire le check requis. GitHub propose le setting "Do not require status checks to pass for pull requests that don't trigger the check" pour gérer ce cas, ou vous pouvez utiliser l'action `dorny/paths-filter` pour un contrôle plus fin à l'intérieur du workflow plutôt qu'au niveau du déclencheur.

### Le déclencheur `workflow_dispatch`

La directive `inputs` de `workflow_dispatch` ajoute un formulaire dans l'interface GitHub lors du déclenchement manuel. Pour un projet C++, exposer le type de build comme paramètre permet de relancer un build Debug à la demande sans modifier le fichier workflow. La valeur est accessible dans les jobs via `${{ github.event.inputs.build_type }}`.

## Le bloc `env` : variables d'environnement

Le bloc `env` au niveau racine définit des variables accessibles par tous les jobs et steps du workflow :

```yaml
env:
  BUILD_DIR: build
  BUILD_TYPE: Release
  CPP_STANDARD: 20
  CMAKE_GENERATOR: Ninja
  CCACHE_MAXSIZE: 2G
  CCACHE_COMPILERCHECK: content
```

Les variables peuvent également être définies à deux niveaux plus spécifiques :

**Au niveau d'un job** — elles surchargent les variables globales pour tous les steps de ce job :

```yaml
jobs:
  build-debug:
    env:
      BUILD_TYPE: Debug
```

**Au niveau d'un step** — elles ne s'appliquent qu'à cette étape :

```yaml
steps:
  - name: Run tests with ASan
    env:
      ASAN_OPTIONS: detect_leaks=1:halt_on_error=1
    run: ctest --parallel $(nproc)
```

La résolution est hiérarchique : step surcharge job, qui surcharge workflow. Ce mécanisme permet de définir des valeurs par défaut globales (Release, C++20) et de les surcharger localement dans les jobs qui en ont besoin (Debug, C++23, sanitizers).

### Expressions et variables contextuelles

GitHub Actions utilise la syntaxe `${{ ... }}` pour les expressions dynamiques. Les contextes les plus utiles pour un projet C++ sont :

```yaml
env:
  # Branche ou tag courant
  REF_NAME: ${{ github.ref_name }}
  # Hash court du commit
  SHORT_SHA: ${{ github.sha }}
  # 'true' si c'est une pull request
  IS_PR: ${{ github.event_name == 'pull_request' }}
  # Paramètre du workflow_dispatch
  MANUAL_BUILD_TYPE: ${{ github.event.inputs.build_type || 'Release' }}
  # OS du runner
  RUNNER_OS: ${{ runner.os }}
  # Nombre de cœurs
  NPROC: ${{ runner.arch == 'ARM64' && '4' || '4' }}
```

Ces expressions sont évaluées avant l'exécution du job. La syntaxe `${{ x || 'default' }}` fournit une valeur par défaut lorsque l'expression est vide ou nulle — utile pour les paramètres optionnels de `workflow_dispatch`.

## Le bloc `permissions`

```yaml
permissions:
  contents: read
  packages: write
  pull-requests: write
```

Le bloc `permissions` contrôle les droits du token `GITHUB_TOKEN` automatiquement fourni au workflow. Par défaut, le token a des permissions larges. La bonne pratique est de suivre le principe du moindre privilège et de ne déclarer que les permissions nécessaires :

- **`contents: read`** — Suffisant pour la majorité des jobs CI (checkout, build, test).  
- **`packages: write`** — Nécessaire pour publier des images Docker sur GitHub Container Registry (ghcr.io).  
- **`pull-requests: write`** — Nécessaire si un step doit poster un commentaire sur la PR (par exemple un résumé des résultats de test ou de couverture de code).

Pour les workflows de release qui doivent créer des tags ou des releases, `contents: write` est requis.

## Structure d'un job

Un job est l'unité d'exécution fondamentale. Voici un job de build C++ complet avec toutes les directives pertinentes :

```yaml
jobs:
  build-gcc:
    name: "Build — GCC 15 Release"
    runs-on: ubuntu-latest
    timeout-minutes: 30
    env:
      CXX: g++-15
      CC: gcc-15
    needs: [lint]
    if: ${{ !cancelled() }}
    steps:
      # ...
```

### `name`

Le nom affiché dans l'interface GitHub Actions. Sans `name`, GitHub utilise la clé du job (`build-gcc`). Un nom explicite comme `Build — GCC 15 Release` est plus lisible dans la vue du workflow, surtout avec les matrix builds où plusieurs jobs partagent la même clé.

### `runs-on`

Spécifie le runner sur lequel le job s'exécute. Les valeurs courantes pour un projet C++ :

| Valeur | Environnement | Usage typique |
|--------|---------------|---------------|
| `ubuntu-latest` | Ubuntu 24.04, 4 vCPUs | Build standard |
| `ubuntu-24.04` | Ubuntu 24.04 (épinglé) | Reproductibilité |
| `macos-latest` | macOS, Apple Silicon | Cross-platform |
| `windows-latest` | Windows Server | Build MSVC |
| `self-hosted` | Votre infrastructure | Performance, accès matériel |

**Épingler la version d'OS** (`ubuntu-24.04` plutôt que `ubuntu-latest`) est recommandé pour les projets sensibles aux changements d'environnement. `ubuntu-latest` est un alias qui évolue au fil du temps — lorsque GitHub le bascule vers Ubuntu 26.04, les versions pré-installées de GCC et Clang changeront, ce qui peut casser le build.

### `timeout-minutes`

```yaml
timeout-minutes: 30
```

Durée maximale du job en minutes (par défaut 360, soit 6 heures). Pour un job de build C++, 30 minutes est un timeout raisonnable qui protège contre les builds qui bouclent ou un deadlock dans le processus de compilation sans être trop restrictif. Ajustez selon la taille de votre projet.

### `needs`

```yaml
needs: [lint]
```

Déclare les dépendances du job. `build-gcc` ne démarrera qu'après la réussite du job `lint`. Sans `needs`, tous les jobs s'exécutent en parallèle — ce qui signifierait que le build démarre avant la fin du lint, gaspillant des minutes CI si le code est mal formaté.

Contrairement à GitLab CI où les stages imposent un ordre séquentiel global, `needs` dans GitHub Actions crée un graphe de dépendances explicite. C'est plus flexible mais aussi plus verbeux : chaque relation doit être déclarée manuellement.

### `if` : exécution conditionnelle

```yaml
if: ${{ !cancelled() }}
```

La directive `if` au niveau du job contrôle si le job doit s'exécuter. Les conditions les plus utiles pour un projet C++ :

```yaml
# Exécuter sauf si le workflow a été annulé
if: ${{ !cancelled() }}

# Exécuter uniquement sur la branche main
if: ${{ github.ref == 'refs/heads/main' }}

# Exécuter uniquement sur un tag de version
if: ${{ startsWith(github.ref, 'refs/tags/v') }}

# Exécuter uniquement sur les pull requests
if: ${{ github.event_name == 'pull_request' }}
```

La condition `!cancelled()` est une subtilité importante. Par défaut, quand un job dont on dépend via `needs` échoue, les jobs dépendants sont ignorés (status "skipped"). Avec `!cancelled()`, le job s'exécute tant que le workflow n'a pas été explicitement annulé — même si une dépendance a échoué. C'est utile pour les jobs de notification ou de nettoyage qui doivent s'exécuter dans tous les cas.

## Structure d'un step

Les steps sont les briques élémentaires au sein d'un job. Il en existe deux types : les **actions** (`uses:`) et les **commandes** (`run:`).

### Steps d'action (`uses`)

```yaml
steps:
  - name: Checkout du code
    uses: actions/checkout@v4
    with:
      submodules: recursive
      fetch-depth: 1
```

La syntaxe `uses: owner/repo@ref` référence une action publiée. Le bloc `with:` passe des paramètres à l'action. Chaque action définit ses propres paramètres — il faut consulter sa documentation pour connaître les options disponibles.

**`fetch-depth: 1`** dans `actions/checkout` effectue un shallow clone, équivalent à `GIT_DEPTH: 1` dans GitLab CI. Pour un projet C++ avec un historique volumineux, cela réduit le temps de checkout de plusieurs secondes.

### Steps de commande (`run`)

```yaml
steps:
  - name: CMake configure
    run: |
      cmake -B ${{ env.BUILD_DIR }} \
        -G Ninja \
        -DCMAKE_BUILD_TYPE=${{ env.BUILD_TYPE }} \
        -DCMAKE_CXX_STANDARD=${{ env.CPP_STANDARD }} \
        -DCMAKE_CXX_COMPILER=${{ env.CXX }} \
        -DCMAKE_CXX_COMPILER_LAUNCHER=ccache \
        -DCMAKE_EXPORT_COMPILE_COMMANDS=ON

  - name: CMake build
    run: cmake --build ${{ env.BUILD_DIR }} --parallel $(nproc)
```

Le pipe `|` permet d'écrire des commandes multi-lignes. Le backslash `\` poursuit une commande sur la ligne suivante. Les deux sont nécessaires pour les longues lignes de commande CMake.

Par défaut, les commandes `run` sont exécutées avec `bash -e` sur les runners Linux — le flag `-e` signifie que le step échoue dès qu'une commande retourne un code de sortie non-nul. Ce comportement est souhaitable en CI.

### Steps conditionnels

Un step individuel peut être conditionné avec `if:` :

```yaml
- name: Install GCC 15
  if: ${{ matrix.compiler == 'gcc' }}
  run: |
    sudo add-apt-repository ppa:ubuntu-toolchain-r/test -y
    sudo apt-get update
    sudo apt-get install -y g++-15

- name: Install Clang 20
  if: ${{ matrix.compiler == 'clang' }}
  run: |
    wget https://apt.llvm.org/llvm.sh
    chmod +x llvm.sh
    sudo ./llvm.sh 20
```

Cela permet de mutualiser un même job (avec une matrice) tout en adaptant l'installation du compilateur au paramètre de la matrice. Les steps dont la condition `if` est fausse sont ignorés sans erreur.

### Sorties entre steps

Les steps d'un même job partagent le même système de fichiers, mais il est parfois nécessaire de passer des valeurs dynamiques d'un step à un autre. GitHub Actions utilise les fichiers d'environnement pour cela :

```yaml
steps:
  - name: Déterminer la version
    id: version
    run: |
      if [[ "${{ github.ref_type }}" == "tag" ]]; then
        VERSION="${{ github.ref_name }}"
        VERSION="${VERSION#v}"
      else
        VERSION="0.0.0-dev+$(git rev-parse --short HEAD)"
      fi
      echo "version=${VERSION}" >> "$GITHUB_OUTPUT"

  - name: CMake configure avec version
    run: |
      cmake -B build \
        -DPROJECT_VERSION=${{ steps.version.outputs.version }}
```

Le step `version` écrit une paire clé-valeur dans `$GITHUB_OUTPUT`. Les steps suivants y accèdent via `${{ steps.<id>.outputs.<key> }}`. Ce mécanisme est l'équivalent des variables dynamiques dans GitLab CI et est particulièrement utile pour le versioning automatique des binaires C++.

## La matrice dans la structure du job

La directive `strategy.matrix` transforme un seul job en N jobs paramétrés. Voici une matrice complète pour un projet C++ :

```yaml
jobs:
  build:
    name: "Build — ${{ matrix.compiler }} ${{ matrix.build_type }}"
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      matrix:
        compiler: [gcc, clang]
        build_type: [Debug, Release]
        cpp_standard: ['20']
        include:
          # Associer chaque nom de compilateur aux binaires réels
          - compiler: gcc
            cxx: g++-15
            cc: gcc-15
            cache_key: gcc15
          - compiler: clang
            cxx: clang++-20
            cc: clang-20
            cache_key: clang20
          # Ajouter un build C++23 uniquement pour Clang Release
          - compiler: clang
            build_type: Release
            cpp_standard: '23'
            cxx: clang++-20
            cc: clang-20
            cache_key: clang20
        exclude:
          # Exclure GCC Debug pour accélérer les PR
          - compiler: gcc
            build_type: Debug
    env:
      CXX: ${{ matrix.cxx }}
      CC: ${{ matrix.cc }}
      BUILD_TYPE: ${{ matrix.build_type }}
      CPP_STANDARD: ${{ matrix.cpp_standard }}
    steps:
      - uses: actions/checkout@v4
      # ... (installation, cache, build)
```

Cette matrice génère les jobs suivants :

1. **GCC Release C++20** — combinaison de base
2. **Clang Debug C++20** — combinaison de base
3. **Clang Release C++20** — combinaison de base
4. **Clang Release C++23** — ajouté par `include`

La combinaison GCC Debug est retirée par `exclude`. Le résultat est une matrice asymétrique qui concentre les ressources CI sur les combinaisons les plus pertinentes.

Chaque variable de la matrice est accessible via `${{ matrix.xxx }}` dans les blocs `env`, `steps`, et même `name` — ce dernier point permet d'afficher un nom lisible pour chaque instance de la matrice dans l'interface GitHub.

## Workflow complet commenté

Voici un workflow CI complet et fonctionnel pour un projet C++ CMake + Ninja + Google Test, intégrant tous les concepts abordés :

```yaml
# .github/workflows/ci.yml
# ============================================================
# Workflow CI — Build, test et analyse statique
# ============================================================
name: CI

on:
  push:
    branches: [main]
    paths-ignore: ['docs/**', '**.md', 'LICENSE']
  pull_request:
    branches: [main]
    paths:
      - 'src/**'
      - 'include/**'
      - 'tests/**'
      - 'CMakeLists.txt'
      - 'cmake/**'
      - '.github/workflows/ci.yml'

# ── Variables globales ──────────────────────────────────────
env:
  BUILD_DIR: build
  CPP_STANDARD: '20'
  CCACHE_MAXSIZE: 2G
  CCACHE_COMPILERCHECK: content

# ── Permissions minimales ───────────────────────────────────
permissions:
  contents: read
  pull-requests: write          # Pour les rapports de test

# ── Jobs ────────────────────────────────────────────────────
jobs:

  # ── Lint : Formatage et analyse statique ──────────────────
  lint:
    name: Lint
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4

      - name: Install clang tools
        run: |
          wget https://apt.llvm.org/llvm.sh
          chmod +x llvm.sh
          sudo ./llvm.sh 20
          sudo apt-get install -y clang-format-20 clang-tidy-20

      - name: Check formatting
        run: |
          find src include tests -type f \( -name '*.cpp' -o -name '*.hpp' -o -name '*.h' \) \
            | xargs clang-format-20 --dry-run --Werror

      - name: Generate compile_commands.json
        run: |
          cmake -B ${{ env.BUILD_DIR }} \
            -G Ninja \
            -DCMAKE_CXX_COMPILER=clang++-20 \
            -DCMAKE_EXPORT_COMPILE_COMMANDS=ON

      - name: Run clang-tidy
        run: |
          find src include -type f \( -name '*.cpp' -o -name '*.hpp' \) \
            | xargs -P $(nproc) clang-tidy-20 \
              -p ${{ env.BUILD_DIR }}/compile_commands.json \
              --warnings-as-errors='*' --quiet

  # ── Build : Compilation multi-compilateur ─────────────────
  build:
    name: "Build — ${{ matrix.compiler }} ${{ matrix.build_type }}"
    runs-on: ubuntu-latest
    timeout-minutes: 30
    needs: [lint]
    strategy:
      fail-fast: false
      matrix:
        compiler: [gcc, clang]
        build_type: [Release]
        include:
          - compiler: gcc
            cxx: g++-15
            cc: gcc-15
            cache_key: gcc15
          - compiler: clang
            cxx: clang++-20
            cc: clang-20
            cache_key: clang20
          # Build Debug uniquement sur GCC
          - compiler: gcc
            build_type: Debug
            cxx: g++-15
            cc: gcc-15
            cache_key: gcc15

    env:
      CXX: ${{ matrix.cxx }}
      CC: ${{ matrix.cc }}
      BUILD_TYPE: ${{ matrix.build_type }}

    steps:
      - uses: actions/checkout@v4
        with:
          submodules: recursive
          fetch-depth: 1

      - name: Install GCC 15
        if: ${{ matrix.compiler == 'gcc' }}
        run: |
          sudo add-apt-repository ppa:ubuntu-toolchain-r/test -y
          sudo apt-get update
          sudo apt-get install -y g++-15 gcc-15

      - name: Install Clang 20
        if: ${{ matrix.compiler == 'clang' }}
        run: |
          wget https://apt.llvm.org/llvm.sh
          chmod +x llvm.sh
          sudo ./llvm.sh 20

      - name: Install build tools
        run: sudo apt-get install -y ninja-build ccache

      - name: Setup ccache
        uses: hendrikmuhs/ccache-action@v1
        with:
          key: ${{ matrix.cache_key }}-${{ matrix.build_type }}
          max-size: ${{ env.CCACHE_MAXSIZE }}

      - name: CMake configure
        run: |
          cmake -B ${{ env.BUILD_DIR }} \
            -G Ninja \
            -DCMAKE_BUILD_TYPE=${{ env.BUILD_TYPE }} \
            -DCMAKE_CXX_STANDARD=${{ env.CPP_STANDARD }} \
            -DCMAKE_CXX_COMPILER=${{ env.CXX }} \
            -DCMAKE_C_COMPILER=${{ env.CC }} \
            -DCMAKE_CXX_COMPILER_LAUNCHER=ccache \
            -DCMAKE_EXPORT_COMPILE_COMMANDS=ON

      - name: CMake build
        run: cmake --build ${{ env.BUILD_DIR }} --parallel $(nproc)

      - name: Show ccache stats
        run: ccache --show-stats

      - name: Upload build artifacts
        uses: actions/upload-artifact@v4
        with:
          name: build-${{ matrix.compiler }}-${{ matrix.build_type }}
          path: |
            ${{ env.BUILD_DIR }}/bin/
            ${{ env.BUILD_DIR }}/lib/
            ${{ env.BUILD_DIR }}/tests/
            ${{ env.BUILD_DIR }}/CTestTestfile.cmake
            ${{ env.BUILD_DIR }}/tests/CTestTestfile.cmake
          retention-days: 1

  # ── Test : Exécution des tests unitaires ──────────────────
  test:
    name: "Test — ${{ matrix.compiler }} ${{ matrix.build_type }}"
    runs-on: ubuntu-latest
    timeout-minutes: 15
    needs: [build]
    strategy:
      fail-fast: false
      matrix:
        compiler: [gcc, clang]
        build_type: [Release]
        include:
          - compiler: gcc
            build_type: Debug

    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 1

      - name: Download build artifacts
        uses: actions/download-artifact@v4
        with:
          name: build-${{ matrix.compiler }}-${{ matrix.build_type }}
          path: ${{ env.BUILD_DIR }}/

      - name: Fix permissions on test binaries
        run: chmod +x ${{ env.BUILD_DIR }}/tests/* || true

      - name: Run CTest
        working-directory: ${{ env.BUILD_DIR }}
        env:
          CTEST_OUTPUT_ON_FAILURE: 1
        run: ctest --parallel $(nproc) --timeout 120

  # ── Sanitizers : ASan + UBSan ─────────────────────────────
  sanitizers:
    name: "Sanitizers — ASan + UBSan"
    runs-on: ubuntu-latest
    timeout-minutes: 30
    needs: [lint]
    steps:
      - uses: actions/checkout@v4
        with:
          submodules: recursive

      - name: Install Clang 20 and tools
        run: |
          wget https://apt.llvm.org/llvm.sh
          chmod +x llvm.sh
          sudo ./llvm.sh 20
          sudo apt-get install -y ninja-build ccache

      - name: Setup ccache
        uses: hendrikmuhs/ccache-action@v1
        with:
          key: clang20-asan
          max-size: ${{ env.CCACHE_MAXSIZE }}

      - name: Build with sanitizers
        run: |
          cmake -B ${{ env.BUILD_DIR }} \
            -G Ninja \
            -DCMAKE_BUILD_TYPE=Debug \
            -DCMAKE_CXX_COMPILER=clang++-20 \
            -DCMAKE_CXX_COMPILER_LAUNCHER=ccache \
            -DCMAKE_CXX_FLAGS="-fsanitize=address,undefined -fno-omit-frame-pointer" \
            -DCMAKE_EXE_LINKER_FLAGS="-fsanitize=address,undefined"
          cmake --build ${{ env.BUILD_DIR }} --parallel $(nproc)

      - name: Run tests with sanitizers
        working-directory: ${{ env.BUILD_DIR }}
        env:
          ASAN_OPTIONS: detect_leaks=1:halt_on_error=1:check_initialization_order=1
          UBSAN_OPTIONS: print_stacktrace=1:halt_on_error=1
          CTEST_OUTPUT_ON_FAILURE: 1
        run: ctest --parallel $(nproc) --timeout 300
```

### Points clés de ce workflow

**Le job `sanitizers` est indépendant de la matrice `build`.** Plutôt que d'ajouter une variante sanitizer dans la matrice (ce qui compliquerait la logique), le job sanitizers est autonome : il compile et teste dans le même job, avec ses propres flags. Il dépend de `lint` (pas de `build`) et s'exécute donc en parallèle avec les builds normaux. C'est un choix de conception fréquent : les builds sanitizers n'ont pas besoin de partager des artifacts avec les tests normaux.

**`fix permissions on test binaries`** est un step nécessaire car `actions/upload-artifact` ne préserve pas les permissions d'exécution des fichiers. Les binaires de test uploadés perdent leur bit exécutable et `ctest` ne peut pas les lancer sans un `chmod +x` préalable. C'est un piège classique de GitHub Actions qui n'existe pas dans GitLab CI (où les artifacts sont transférés via le cache du runner sans modification de permissions).

**La matrice `test` duplique la matrice `build`.** C'est une conséquence de l'architecture GitHub Actions : le job `test` doit connaître les noms exacts des artifacts pour les downloader, et ces noms dépendent des paramètres de la matrice du job `build`. La matrice du job `test` doit donc être alignée avec celle du job `build`. Cette duplication est l'un des inconvénients de GitHub Actions par rapport à GitLab CI, où un job de test peut simplement déclarer `needs: [build-gcc-release]` sans recréer une matrice.

**`working-directory`** évite de préfixer chaque commande avec `cd build &&`. C'est une directive de lisibilité qui s'applique à toutes les commandes `run:` du step.

## Organisation multi-fichiers

Pour un projet C++ mature, découper les workflows en plusieurs fichiers améliore la lisibilité et permet des déclencheurs indépendants :

```
.github/
└── workflows/
    ├── ci.yml              # Build + test (push, PR)
    ├── release.yml         # Package + release (tags v*)
    ├── docs.yml            # Documentation (push sur main)
    └── nightly.yml         # Build complet + sanitizers (schedule)
```

**`ci.yml`** — Le workflow présenté ci-dessus. Déclenché sur push et PR, il couvre lint, build multi-compilateur et tests.

**`release.yml`** — Déclenché uniquement sur les tags `v*`. Il compile en Release, génère les paquets DEB/RPM, construit l'image Docker et crée une release GitHub avec les assets.

**`docs.yml`** — Déclenché sur push vers `main`. Il génère la documentation Doxygen et la déploie sur GitHub Pages.

**`nightly.yml`** — Déclenché par un cron nocturne. Il exécute la matrice complète (tous les compilateurs, tous les standards, tous les sanitizers) sans contrainte de temps. C'est le filet de sécurité qui attrape les problèmes que le CI rapide des PR ne couvre pas.

Cette séparation évite qu'un tag de version déclenche le workflow de CI complet (inutile puisque le code a déjà été testé sur la branche), et qu'un push sur `main` déclenche le packaging (qui ne doit se faire que sur les tags).

## Équivalences avec le pipeline GitLab CI

Pour conclure, voici la correspondance entre les jobs du pipeline GitLab CI de la section 38.1.2 et les jobs du workflow GitHub Actions présenté ici :

| GitLab CI (section 38.1) | GitHub Actions | Différence notable |
|---------------------------|----------------|-------------------|
| `format-check` (stage: lint) | Job `lint`, step "Check formatting" | Un seul job regroupe format + tidy |
| `static-analysis` (stage: lint) | Job `lint`, step "Run clang-tidy" | Steps séquentiels dans le même job |
| `build-gcc-release` (stage: build) | Job `build`, matrice compiler=gcc | Matrice vs jobs explicites |
| `build-clang-release` (stage: build) | Job `build`, matrice compiler=clang | Matrice vs jobs explicites |
| `build-asan` (stage: build) | Job `sanitizers` (build + test) | Job autonome, pas dans la matrice |
| `test-gcc-release` (stage: test) | Job `test`, matrice compiler=gcc | Download explicite des artifacts |
| `test-asan` (stage: test) | Job `sanitizers` (intégré) | Build + test dans le même job |
| `package-deb` (stage: package) | Workflow `release.yml` séparé | Fichier workflow distinct |
| `create-release` (stage: deploy) | Workflow `release.yml` séparé | Fichier workflow distinct |

La logique est la même, l'implémentation diffère. Le choix entre les deux plateformes dépend rarement de considérations techniques CI/CD et davantage de l'écosystème dans lequel vit le projet : GitHub pour l'open source et les organisations centrées GitHub, GitLab pour les entreprises avec infrastructure auto-hébergée.
