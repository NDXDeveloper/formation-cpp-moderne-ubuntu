# 38.3.2 sccache : Cache distribué cloud

## Introduction

ccache résout le problème du cache de compilation pour un runner donné : les résultats sont stockés localement et réutilisés d'une exécution à l'autre. Mais cette approche atteint ses limites lorsque l'infrastructure CI repose sur des **runners éphémères** (pods Kubernetes, instances auto-scaling, runners GitHub hébergés) où le système de fichiers local disparaît à la fin de chaque job, ou lorsque **plusieurs runners** doivent partager le même cache pour éviter de recompiler le même code chacun de leur côté.

sccache (Shared Compilation Cache), développé par Mozilla, répond à ces limitations en ajoutant une couche de stockage distant entre le cache et les runners. Plutôt que de sauvegarder et restaurer une archive de cache à chaque exécution (comme on le fait avec ccache + `actions/cache`), sccache communique directement avec un backend de stockage cloud. Chaque lookup et chaque écriture se font en temps réel, sans étape de sauvegarde/restauration explicite.

Cette section couvre l'architecture de sccache, la configuration des principaux backends cloud, l'intégration avec GitHub Actions et GitLab CI, et les situations où sccache apporte un avantage concret par rapport à ccache.

## Architecture de sccache

sccache fonctionne selon un modèle client-serveur local couplé à un backend de stockage distant :

```
  Runner CI
  ┌─────────────────────────────────────────────────┐
  │                                                 │
  │  Ninja/CMake                                    │
  │      │                                          │
  │      ▼                                          │
  │  sccache (client)                               │
  │      │                                          │
  │      ▼                                          │
  │  sccache-server (daemon local)                  │
  │      │         │                                │
  │      │         ▼                                │
  │      │    Cache mémoire local                   │
  │      │    (LRU, éphémère)                       │
  │      │                                          │
  └──────┼──────────────────────────────────────────┘
         │
         ▼  (réseau)
  ┌──────────────────────┐
  │  Backend de stockage │
  │  ┌─────────────────┐ │
  │  │  Amazon S3      │ │
  │  │  Google GCS     │ │
  │  │  Azure Blob     │ │
  │  │  Redis          │ │
  │  │  GitHub Actions │ │
  │  │  Local (disque) │ │
  │  └─────────────────┘ │
  └──────────────────────┘
        ▲
        │ (réseau)
  ┌─────┴──────┐
  │  Autres    │
  │  runners   │   ← Tous partagent le même backend
  └────────────┘
```

Le **client sccache** remplace le compilateur dans la ligne de commande (via `CMAKE_CXX_COMPILER_LAUNCHER=sccache`). À chaque invocation, il contacte le **serveur sccache** — un daemon qui tourne localement sur le runner. Le serveur calcule le hash des entrées, vérifie d'abord un cache mémoire local (rapide mais éphémère), puis interroge le **backend distant** si nécessaire. En cas de miss, la compilation est déléguée au compilateur réel et le résultat est stocké à la fois localement et sur le backend.

Ce modèle présente deux avantages fondamentaux par rapport à ccache en CI :

**Pas de phase de sauvegarde/restauration.** Avec ccache, il faut télécharger une archive du cache en début de job (10-30 secondes) et la re-uploader en fin de job (10-45 secondes). Avec sccache, chaque entrée est lue et écrite individuellement sur le backend — il n'y a pas de surcoût fixe de transfert en masse. Sur un build incrémental qui ne compile que 3 fichiers, sccache n'effectue que 3 lookups réseau (quelques millisecondes chacun) au lieu de transférer un cache complet de 500 Mo.

**Cache partagé nativement.** Tous les runners qui pointent vers le même backend partagent le même cache. Quand le runner A compile un fichier, le résultat est immédiatement disponible pour le runner B. Avec ccache, chaque runner possède son propre cache et recompile indépendamment les mêmes fichiers.

## Les backends de stockage

### Amazon S3

S3 est le backend le plus mature et le plus utilisé avec sccache. La configuration passe par des variables d'environnement :

```yaml
env:
  SCCACHE_BUCKET: "mon-projet-sccache"
  SCCACHE_REGION: "eu-west-3"            # Paris
  SCCACHE_S3_USE_SSL: "true"
  SCCACHE_S3_KEY_PREFIX: "ci/cpp/"       # Organiser par projet/langage
  AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
  AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
```

**`SCCACHE_BUCKET`** identifie le bucket S3 dédié au cache. Un bucket par organisation (avec des préfixes par projet via `SCCACHE_S3_KEY_PREFIX`) est généralement suffisant et évite la prolifération de buckets.

**`SCCACHE_S3_KEY_PREFIX`** organise les entrées de cache dans le bucket. Le préfixe `ci/cpp/` sépare le cache C++ d'éventuels autres usages du bucket. Vous pouvez aller plus loin avec `ci/cpp/gcc15/release/` pour segmenter par compilateur et type de build, mais sccache gère déjà l'isolation par le hash (qui inclut les flags de compilation) — la segmentation par préfixe est surtout organisationnelle.

**Configuration du bucket S3 recommandée :**

```json
{
  "Rules": [
    {
      "ID": "sccache-expiration",
      "Status": "Enabled",
      "Filter": { "Prefix": "ci/cpp/" },
      "Expiration": { "Days": 30 }
    }
  ]
}
```

Une politique d'expiration de 30 jours sur le bucket évite que le cache ne croisse indéfiniment. Les entrées non accédées depuis 30 jours sont automatiquement supprimées par S3. Contrairement à ccache qui gère sa propre éviction LRU, sccache avec un backend S3 n'a pas de notion de taille maximale — c'est au bucket de gérer le cycle de vie.

**Coût S3.** Le coût d'un cache sccache sur S3 est généralement négligeable : quelques Go de stockage ($0.02/Go/mois) et des milliers de requêtes GET/PUT ($0.004 pour 10 000 requêtes). Pour un projet de taille moyenne avec 200 fichiers source, un build incrémental génère au plus 200 requêtes GET (lookups) et quelques PUT (nouveaux résultats). Le coût mensuel est typiquement inférieur à $1, bien en dessous du coût des minutes CI économisées.

### Google Cloud Storage (GCS)

```yaml
env:
  SCCACHE_GCS_BUCKET: "mon-projet-sccache"
  SCCACHE_GCS_KEY_PREFIX: "ci/cpp/"
  SCCACHE_GCS_RW_MODE: "READ_WRITE"
  SCCACHE_GCS_CREDENTIALS_URL: ""        # Utiliser les credentials par défaut
  # OU
  SCCACHE_GCS_KEY_PATH: "/path/to/service-account.json"
```

L'authentification GCS peut utiliser les Application Default Credentials (ADC) si le runner tourne dans Google Cloud, ou un fichier de clé de service account pour les runners externes. La variable `SCCACHE_GCS_RW_MODE` doit être `READ_WRITE` pour les jobs de build (qui écrivent dans le cache) et peut être `READ_ONLY` pour les jobs qui ne font que lire (tests avec recompilation partielle).

### Azure Blob Storage

```yaml
env:
  SCCACHE_AZURE_CONNECTION_STRING: ${{ secrets.AZURE_STORAGE_CONNECTION_STRING }}
  SCCACHE_AZURE_BLOB_CONTAINER: "sccache"
  SCCACHE_AZURE_KEY_PREFIX: "ci/cpp/"
```

### Backend GitHub Actions (GHA)

sccache peut utiliser le cache intégré de GitHub Actions comme backend, sans nécessiter de compte cloud externe :

```yaml
env:
  SCCACHE_GHA_ENABLED: "true"
  ACTIONS_CACHE_URL: ${{ env.ACTIONS_CACHE_URL }}
  ACTIONS_RUNTIME_TOKEN: ${{ env.ACTIONS_RUNTIME_TOKEN }}
```

Ce backend est le plus simple à configurer (aucun secret cloud à créer) mais hérite des limitations du cache GitHub Actions : 10 Go par dépôt, éviction LRU automatique, accessible uniquement depuis les workflows du même dépôt.

Pour un petit à moyen projet, c'est un excellent point de départ. Pour un projet de grande envergure avec une matrice large (5+ combinaisons) ou un dépôt très actif, les 10 Go peuvent être insuffisants — un backend S3 ou GCS offrira plus de marge.

### Redis et Memcached

sccache supporte également Redis et Memcached comme backends. Ces solutions sont adaptées aux organisations qui disposent déjà d'une infrastructure Redis/Memcached et qui souhaitent un cache très rapide (latence sub-milliseconde sur le réseau local). La configuration est simple :

```yaml
env:
  SCCACHE_REDIS: "redis://cache-server.internal:6379"
  # ou
  SCCACHE_MEMCACHED: "tcp://cache-server.internal:11211"
```

L'inconvénient est que ces backends ne fournissent pas de persistance durable par défaut (Redis peut être configuré avec AOF/RDB, mais Memcached est purement en mémoire). Ils sont plus adaptés à un cache "best-effort" sur des runners auto-hébergés partageant le même réseau local.

## Intégration avec GitHub Actions

### Avec l'action officielle Mozilla

```yaml
jobs:
  build:
    name: "Build — ${{ matrix.compiler }} ${{ matrix.build_type }}"
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      matrix:
        include:
          - compiler: gcc
            cxx: g++-15
            cc: gcc-15
          - compiler: clang
            cxx: clang++-20
            cc: clang-20
        build_type: [Release]

    env:
      CXX: ${{ matrix.cxx }}
      CC: ${{ matrix.cc }}
      CMAKE_BUILD_TYPE: ${{ matrix.build_type }}

    steps:
      - uses: actions/checkout@v4

      - name: Install toolchain
        uses: aminya/setup-cpp@v1
        with:
          compiler: ${{ matrix.compiler == 'gcc' && 'gcc-15' || 'llvm-20' }}
          cmake: true
          ninja: true

      - name: Setup sccache
        uses: mozilla/sccache-action@v0.0.6

      - name: CMake configure
        run: |
          cmake -B build \
            -G Ninja \
            -DCMAKE_BUILD_TYPE=${{ env.CMAKE_BUILD_TYPE }} \
            -DCMAKE_CXX_STANDARD=20 \
            -DCMAKE_CXX_COMPILER=${{ env.CXX }} \
            -DCMAKE_C_COMPILER=${{ env.CC }} \
            -DCMAKE_CXX_COMPILER_LAUNCHER=sccache \
            -DCMAKE_C_COMPILER_LAUNCHER=sccache
        env:
          SCCACHE_GHA_ENABLED: "true"

      - name: CMake build
        run: cmake --build build --parallel $(nproc)
        env:
          SCCACHE_GHA_ENABLED: "true"

      - name: Show sccache stats
        run: sccache --show-stats
```

L'action `mozilla/sccache-action` installe sccache, démarre le serveur daemon, et configure automatiquement les variables d'environnement pour le backend GitHub Actions (`ACTIONS_CACHE_URL`, `ACTIONS_RUNTIME_TOKEN`). En fin de job, elle arrête le serveur et affiche les statistiques.

La variable `SCCACHE_GHA_ENABLED: "true"` doit être définie dans les steps qui invoquent la compilation pour que sccache utilise le backend GitHub Actions. Elle peut être définie au niveau du job entier plutôt que step par step :

```yaml
jobs:
  build:
    env:
      SCCACHE_GHA_ENABLED: "true"
    # ...
```

### Avec un backend S3

Pour utiliser S3 au lieu du cache GitHub Actions, il suffit de remplacer les variables d'environnement :

```yaml
    env:
      SCCACHE_BUCKET: "mon-projet-sccache"
      SCCACHE_REGION: "eu-west-3"
      SCCACHE_S3_KEY_PREFIX: "ci/${{ matrix.compiler }}/"
      AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
      AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}

    steps:
      - uses: actions/checkout@v4

      - name: Install sccache
        run: |
          SCCACHE_VERSION="v0.9.1"
          curl -L "https://github.com/mozilla/sccache/releases/download/${SCCACHE_VERSION}/sccache-${SCCACHE_VERSION}-x86_64-unknown-linux-musl.tar.gz" \
            | tar xz --strip-components=1 -C /usr/local/bin/ \
              "sccache-${SCCACHE_VERSION}-x86_64-unknown-linux-musl/sccache"

      - name: Start sccache server
        run: sccache --start-server

      # ... (configure + build avec CMAKE_CXX_COMPILER_LAUNCHER=sccache)

      - name: Show sccache stats
        if: always()
        run: sccache --show-stats

      - name: Stop sccache server
        if: always()
        run: sccache --stop-server
```

Avec un backend S3, l'action `mozilla/sccache-action` n'est pas nécessaire — sccache est installé et géré manuellement. Le serveur est démarré explicitement en début de job et arrêté en fin de job. Les steps `if: always()` garantissent l'affichage des statistiques et l'arrêt propre du serveur même en cas d'échec du build.

## Intégration avec GitLab CI

Sur GitLab CI, sccache s'intègre via les variables d'environnement du pipeline. Il n'y a pas d'équivalent de l'action Mozilla — l'outil est installé dans l'image Docker de build et configuré via les variables :

```yaml
variables:
  SCCACHE_BUCKET: "mon-projet-sccache"
  SCCACHE_REGION: "eu-west-3"
  SCCACHE_S3_KEY_PREFIX: "ci/cpp/"
  # Les credentials sont définis dans Settings > CI/CD > Variables
  # AWS_ACCESS_KEY_ID et AWS_SECRET_ACCESS_KEY

.cmake_build_sccache:
  stage: build
  image: registry.exemple.com/cpp-build:latest  # Image contenant sccache
  before_script:
    - sccache --start-server
    - sccache --zero-stats
  script:
    - cmake -B ${BUILD_DIR}
        -G Ninja
        -DCMAKE_BUILD_TYPE=${BUILD_TYPE}
        -DCMAKE_CXX_STANDARD=${CPP_STANDARD}
        -DCMAKE_CXX_COMPILER=${CXX}
        -DCMAKE_C_COMPILER=${CC}
        -DCMAKE_CXX_COMPILER_LAUNCHER=sccache
        -DCMAKE_C_COMPILER_LAUNCHER=sccache
        -DCMAKE_EXPORT_COMPILE_COMMANDS=ON
    - cmake --build ${BUILD_DIR} --parallel $(nproc)
  after_script:
    - sccache --show-stats
    - sccache --stop-server
  cache: []              # Pas de cache GitLab nécessaire — sccache gère tout
  artifacts:
    paths:
      - ${BUILD_DIR}/bin/
      - ${BUILD_DIR}/lib/
      - ${BUILD_DIR}/tests/
      - ${BUILD_DIR}/CTestTestfile.cmake
    expire_in: 2 hours
```

Le point notable est **`cache: []`** — le mécanisme de cache GitLab CI est désactivé car sccache gère son propre stockage via S3. Il n'y a plus besoin de sauvegarder et restaurer un répertoire `.ccache` entre les exécutions. C'est l'un des avantages opérationnels de sccache : la suppression de la phase de transfert de cache simplifie la configuration et élimine une source potentielle de lenteur et d'erreurs.

L'image Docker de build doit inclure sccache. Ajoutez-le au Dockerfile :

```dockerfile
# Dans le Dockerfile de build
RUN curl -L "https://github.com/mozilla/sccache/releases/download/v0.9.1/sccache-v0.9.1-x86_64-unknown-linux-musl.tar.gz" \
    | tar xz --strip-components=1 -C /usr/local/bin/ \
      "sccache-v0.9.1-x86_64-unknown-linux-musl/sccache"
```

## Séparation du cache par configuration

Comme pour ccache, les entrées de cache produites par des configurations incompatibles (compilateurs différents, flags différents) ne doivent pas interférer. Avec sccache, la séparation se fait naturellement par le hash — les flags de compilation font partie du hash, donc un fichier compilé avec `-O2` et un fichier compilé avec `-O0 -g` produisent des hash différents et ne se confondent jamais.

Cependant, segmenter le préfixe S3 par configuration reste une bonne pratique organisationnelle pour faciliter le diagnostic et permettre une purge ciblée :

```yaml
variables:
  SCCACHE_S3_KEY_PREFIX: "ci/cpp/${COMPILER}/${BUILD_TYPE}/"

build-gcc-release:
  extends: .cmake_build_sccache
  variables:
    COMPILER: "gcc15"
    BUILD_TYPE: "Release"

build-clang-asan:
  extends: .cmake_build_sccache
  variables:
    COMPILER: "clang20"
    BUILD_TYPE: "asan"
```

Avec cette organisation, le bucket S3 contient une arborescence lisible :

```
ci/cpp/gcc15/Release/...  
ci/cpp/gcc15/Debug/...  
ci/cpp/clang20/Release/...  
ci/cpp/clang20/asan/...  
```

Purger le cache d'une configuration spécifique se fait en supprimant le préfixe correspondant avec `aws s3 rm --recursive s3://bucket/ci/cpp/gcc15/Debug/`.

## Diagnostic et statistiques

sccache fournit des statistiques similaires à ccache :

```bash
$ sccache --show-stats
Compile requests                    203  
Compile requests executed           203  
Cache hits                          187  
Cache hits (C/C++)                  187  
Cache misses                         16  
Cache misses (C/C++)                 16  
Cache timeouts                        0  
Cache errors                          0  
Cache hits rate                    92.1%  
Non-cacheable calls                   0  

Cache location                     S3, bucket: mon-projet-sccache, region: eu-west-3  
Cache size                         1.2 GiB  
Max cache size                     unlimited  
```

Les métriques clés sont les mêmes que pour ccache : **Cache hits rate** (objectif >85%), **Cache timeouts** (doit rester à 0 — des timeouts indiquent un problème réseau ou un backend surchargé), et **Cache errors** (idem, doit rester à 0).

### Diagnostic des problèmes réseau

sccache est plus sensible aux problèmes réseau que ccache car chaque lookup implique une requête HTTP vers le backend. La variable `SCCACHE_LOG` active un log détaillé :

```yaml
env:
  SCCACHE_LOG: "debug"
  SCCACHE_ERROR_LOG: "/tmp/sccache-error.log"
```

Les problèmes courants :

| Symptôme | Cause probable | Solution |
|----------|----------------|----------|
| `Cache timeouts` élevé | Latence réseau vers le bucket | Choisir une région S3 proche du runner |
| `Cache errors` élevé | Credentials expirés ou permissions insuffisantes | Vérifier la politique IAM du bucket |
| 0% de hits malgré un cache peuplé | Préfixe S3 différent entre les jobs | Aligner `SCCACHE_S3_KEY_PREFIX` |
| Compilation plus lente avec sccache qu'en direct | Backend trop lent (Memcached saturé, S3 dans une autre région) | Changer de backend ou de région |
| `server was not started` | Serveur sccache non démarré | Ajouter `sccache --start-server` dans `before_script` |

### Supervision à long terme

Pour un suivi continu, les statistiques sccache peuvent être exportées en JSON et envoyées à un système de monitoring :

```yaml
after_script:
  - |
    STATS_JSON=$(sccache --show-stats --stats-format=json 2>/dev/null)
    HIT_RATE=$(echo "$STATS_JSON" | python3 -c "
    import json, sys
    s = json.load(sys.stdin)['stats']
    hits = s.get('cache_hits', {}).get('counts', {}).get('C/C++', 0)
    misses = s.get('cache_misses', {}).get('counts', {}).get('C/C++', 0)
    total = hits + misses
    print(f'{hits/total*100:.1f}' if total > 0 else '0.0')
    ")
    echo "sccache_hit_rate=${HIT_RATE}%"
  - sccache --stop-server
```

## ccache vs sccache : comparaison des performances en CI

Pour aider au choix entre les deux outils, voici une comparaison sur un projet de 200 fichiers source, mesurant le temps total du job (incluant le surcoût du cache) :

| Scénario | Sans cache | ccache (local) | sccache (S3) | sccache (GHA) |
|----------|------------|----------------|-------------|---------------|
| **Build from scratch** (cache vide) | 8 min | 8 min 15s | 8 min 30s | 8 min 25s |
| **Build incrémental** (3 fichiers modifiés) | 8 min | 1 min 10s | 50s | 55s |
| **Build sans changement** (rebuild identique) | 8 min | 45s | 25s | 30s |
| **Surcoût fixe** (save/restore) | — | 20-40s | 0s | 0s |
| **Première exécution d'une branche** | 8 min | 1 min 30s* | 50s | 55s |

*\* Avec fallback sur le cache de `main`.*

Les observations clés :

**Sur les builds incrémentaux**, sccache est légèrement plus rapide que ccache car il n'a pas le surcoût fixe de sauvegarde/restauration d'une archive de cache. La différence (20-40 secondes) correspond exactement au temps de transfert de l'archive ccache.

**Sur les builds from scratch**, sccache est légèrement plus lent que ccache car chaque miss génère un PUT réseau vers S3. Ce surcoût est marginal (15-30 secondes sur 8 minutes) et amorti dès le deuxième build.

**Sur la première exécution d'une branche**, sccache est nettement avantageux. Avec ccache, le fallback sur le cache de `main` nécessite de télécharger une archive potentiellement volumineuse qui ne correspond pas exactement à la branche. Avec sccache, le cache est immédiatement partagé — les entrées produites par les builds de `main` sont directement accessibles sans transfert.

## Quand migrer de ccache vers sccache

Le passage de ccache à sccache se justifie dans les situations suivantes :

**Runners éphémères sans stockage persistant.** Si vos runners sont des pods Kubernetes ou des instances cloud éphémères, le mécanisme de sauvegarde/restauration d'archive ccache introduit un surcoût incompressible à chaque job. sccache avec un backend cloud élimine ce surcoût.

**Nombreux runners parallèles.** Si votre pipeline exécute 10+ jobs de build en parallèle (matrice large), chaque runner avec ccache maintient son propre cache indépendant. Avec sccache, le premier runner qui compile un fichier alimente le cache pour tous les autres.

**Projets multi-dépôts (monorepo, librairies partagées).** Si plusieurs dépôts compilent les mêmes librairies partagées, un backend sccache commun permet de partager les résultats de compilation entre les projets.

**Cache GitHub Actions saturé.** Si les 10 Go de cache du dépôt sont insuffisants pour héberger les archives ccache de toutes les combinaisons de la matrice, migrer vers sccache avec un backend S3 supprime cette contrainte.

Dans tous les autres cas — projet de taille raisonnable, runners dédiés avec stockage persistant, matrice de build modérée — **ccache reste le choix le plus simple et le plus fiable**. La complexité supplémentaire de sccache (credentials cloud, dépendance réseau, coût d'infrastructure) n'est justifiée que lorsque ses avantages spécifiques sont réellement exploités.

## Sécurité du cache distribué

Un cache de compilation partagé introduit des considérations de sécurité spécifiques :

**Empoisonnement du cache.** Si un acteur malveillant peut écrire dans le backend sccache, il pourrait théoriquement injecter des fichiers objets compromis qui seraient ensuite utilisés par d'autres builds. Ce risque est atténué par le fait que sccache vérifie l'intégrité des entrées par hash, mais il est prudent de restreindre les permissions d'écriture :

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:PutObject"],
      "Resource": "arn:aws:s3:::mon-projet-sccache/ci/cpp/*"
    },
    {
      "Effect": "Deny",
      "Action": ["s3:DeleteObject", "s3:PutBucketPolicy"],
      "Resource": "arn:aws:s3:::mon-projet-sccache/*"
    }
  ]
}
```

**Isolation entre branches.** Sur les projets open source, les pull requests de contributeurs externes s'exécutent avec des permissions réduites. Il est recommandé que les builds de forks n'aient qu'un accès en lecture au cache (via des credentials read-only), tandis que les builds de la branche principale ont un accès en écriture. Cela empêche un fork malveillant de polluer le cache des builds internes.

**Rotation des credentials.** Les clés d'accès S3/GCS stockées dans les secrets CI doivent être rotées régulièrement. Utilisez des rôles IAM avec des sessions temporaires (AWS STS AssumeRole) plutôt que des clés statiques lorsque c'est possible.

## Résumé : arbre de décision complet

```
Quel outil de cache de compilation utiliser en CI ?
│
├── Le projet compile en moins de 2 minutes ?
│   └── OUI ──▶ Le cache de compilation n'est probablement pas nécessaire.
│               Les gains seront faibles par rapport à la complexité ajoutée.
│
├── Les runners disposent de stockage persistant entre les exécutions ?
│   ├── OUI ──▶ ccache (simple, local, éprouvé)
│   │           Configuration : section 38.3.1
│   │
│   └── NON ──▶ Le projet a-t-il accès à un bucket cloud (S3/GCS/Azure) ?
│               ├── OUI ──▶ sccache + backend cloud
│               │           (pas de transfert d'archive, cache partagé)
│               │
│               └── NON ──▶ sccache + backend GitHub Actions
│                           (simple, limité à 10 Go)
│                           OU ccache + actions/cache (éprouvé, transfert d'archive)
│
├── Plusieurs dépôts doivent partager le cache ?
│   └── OUI ──▶ sccache + backend cloud (seul outil supportant le partage inter-dépôts)
│
└── La matrice de build est large (10+ jobs) avec des runners parallèles ?
    └── OUI ──▶ sccache + backend cloud (cache partagé entre runners)
```
