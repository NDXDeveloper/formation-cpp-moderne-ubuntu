# 13.5.2 — Remplacement des paramètres pointeur+taille

## Chapitre 13 : Conteneurs Séquentiels — std::span

---

## Introduction

Le pattern `(T* data, size_t count)` est omniprésent en C et dans le code C++ legacy. Il constitue le moyen le plus répandu de passer un tableau à une fonction sans en imposer le type de conteneur. Mais ce pattern est intrinsèquement fragile : rien dans le système de types ne lie le pointeur à sa taille, et le compilateur ne peut pas détecter une incohérence entre les deux.

`std::span` est conçu comme le remplacement moderne et direct de ce pattern. Cette section présente un guide de migration concret, des patterns les plus simples aux cas limites.

---

## Les problèmes du pattern pointeur+taille

Considérons une fonction utilitaire typique :

```cpp
// API classique : pointeur + taille
void remplir_zeros(int* data, std::size_t count) {
    for (std::size_t i = 0; i < count; ++i) {
        data[i] = 0;
    }
}
```

Cette signature présente plusieurs failles que le compilateur ne détecte pas :

```cpp
#include <vector>

int main() {
    std::vector<int> v{1, 2, 3, 4, 5};

    // ✅ Appel correct
    remplir_zeros(v.data(), v.size());

    // ❌ Taille incorrecte — accès hors bornes, comportement indéfini
    remplir_zeros(v.data(), 100);

    // ❌ Inversion des arguments — compile si count est convertible en int*
    // remplir_zeros(v.size(), v.data());  // erreur ici, mais pas toujours

    // ❌ Pointeur nul avec taille non nulle — UB
    remplir_zeros(nullptr, 5);

    // ❌ Pointeur vers un seul élément, taille > 1 — UB
    int x = 42;
    remplir_zeros(&x, 10);
}
```

Chacun de ces bugs compile sans erreur ni avertissement. Le problème fondamental est que le contrat entre le pointeur et la taille n'est exprimé que dans la documentation, pas dans le type.

---

## Migration de base : un remplacement direct

La transformation la plus simple consiste à remplacer la paire `(T*, size_t)` par un `std::span<T>` :

```cpp
#include <span>
#include <vector>
#include <array>
#include <print>

// Avant
void remplir_zeros_v1(int* data, std::size_t count) {
    for (std::size_t i = 0; i < count; ++i) {
        data[i] = 0;
    }
}

// Après
void remplir_zeros_v2(std::span<int> data) {
    for (auto& val : data) {
        val = 0;
    }
}

int main() {
    std::vector<int> v{1, 2, 3, 4, 5};
    std::array<int, 3> a{10, 20, 30};
    int tab[] = {100, 200};

    // Avant : chaque appel doit fournir la taille manuellement
    remplir_zeros_v1(v.data(), v.size());
    remplir_zeros_v1(a.data(), a.size());
    remplir_zeros_v1(tab, 2);

    // Après : la taille est transportée par le span
    remplir_zeros_v2(v);
    remplir_zeros_v2(a);
    remplir_zeros_v2(tab);
}
```

Le bénéfice est immédiat : il est **impossible** de passer une taille incorrecte, car la taille fait partie du span. L'appelant ne peut plus "oublier" de passer le bon compteur.

## Lecture seule vs lecture-écriture

Le choix entre `std::span<T>` et `std::span<const T>` suit exactement la même logique que `T*` vs `const T*` :

```cpp
#include <span>
#include <print>
#include <numeric>

// Lecture seule — ne modifie pas les données
int somme(std::span<const int> donnees) {
    return std::accumulate(donnees.begin(), donnees.end(), 0);
}

// Lecture-écriture — modifie les données en place
void doubler(std::span<int> donnees) {
    for (auto& val : donnees) {
        val *= 2;
    }
}

// Équivalences avec l'ancien style :
// int somme(const int* data, size_t count)  →  int somme(std::span<const int> data)
// void doubler(int* data, size_t count)     →  void doubler(std::span<int> data)
```

> 💡 **Convention** : par défaut, utilisez `std::span<const T>` pour les paramètres en entrée. Ne passez `std::span<T>` que si la fonction doit modifier les éléments. C'est la même discipline que `const&` vs `&` pour les références.

## Patterns de migration courants

### Buffers d'octets (I/O, réseau, fichiers)

Les API d'entrée/sortie utilisent souvent des buffers bruts `(void*, size_t)` ou `(char*, size_t)`. Le remplacement naturel est `std::span<std::uint8_t>` ou `std::span<std::byte>` :

```cpp
#include <span>
#include <cstdint>
#include <cstddef>
#include <vector>
#include <print>

// Avant
// ssize_t envoyer(int fd, const void* buf, size_t len);

// Après — version avec uint8_t
bool envoyer(int fd, std::span<const std::uint8_t> buf) {
    // write(fd, buf.data(), buf.size());
    std::println("Envoi de {} octets sur fd={}", buf.size(), fd);
    return true;
}

// Après — version avec std::byte (C++17, plus typé)
bool envoyer_bytes(int fd, std::span<const std::byte> buf) {
    // write(fd, buf.data(), buf.size());
    std::println("Envoi de {} bytes sur fd={}", buf.size(), fd);
    return true;
}

int main() {
    std::vector<std::uint8_t> paquet{0x01, 0x02, 0x03, 0x04};
    envoyer(3, paquet);  // Envoi de 4 octets sur fd=3
}
```

### Fonctions de traitement de tableaux numériques

```cpp
#include <span>
#include <cmath>
#include <vector>
#include <print>

// Avant
// double moyenne(const double* valeurs, int n);
// void normaliser(double* valeurs, int n);

// Après
double moyenne(std::span<const double> valeurs) {
    if (valeurs.empty()) return 0.0;
    double total = 0.0;
    for (auto v : valeurs) total += v;
    return total / static_cast<double>(valeurs.size());
}

void normaliser(std::span<double> valeurs) {
    double moy = moyenne(valeurs);  // span<double> → span<const double> implicite
    for (auto& v : valeurs) {
        v -= moy;
    }
}

int main() {
    std::vector<double> data{10.0, 20.0, 30.0, 40.0};

    std::println("Moyenne : {}", moyenne(data));  // 25.0

    normaliser(data);
    for (auto v : data) std::print("{} ", v);
    // Sortie : -15 -5 5 15
}
```

Notez que `normaliser` passe son `span<double>` directement à `moyenne` qui attend un `span<const double>`. La conversion implicite `span<T>` → `span<const T>` est toujours valide et sûre.

### Fonctions de recherche et de comparaison

```cpp
#include <span>
#include <algorithm>
#include <print>

// Avant
// int* trouver(int* data, size_t count, int valeur);
// bool contient(const int* data, size_t count, int valeur);

// Après — retourne un index plutôt qu'un pointeur (plus sûr)
std::size_t trouver_index(std::span<const int> data, int valeur) {
    for (std::size_t i = 0; i < data.size(); ++i) {
        if (data[i] == valeur) return i;
    }
    return data.size();  // convention : size() signifie "non trouvé"
}

bool contient(std::span<const int> data, int valeur) {
    return std::find(data.begin(), data.end(), valeur) != data.end();
}

int main() {
    int tab[] = {10, 20, 30, 40, 50};

    std::println("Index de 30 : {}", trouver_index(tab, 30));  // 2
    std::println("Contient 99 : {}", contient(tab, 99));        // false
}
```

### Callbacks et fonctions de rappel

Les callbacks C passent souvent un pointeur + taille. Avec `std::span`, le callback devient plus sûr :

```cpp
#include <span>
#include <vector>
#include <functional>
#include <print>

// Avant
// typedef void (*callback_t)(const float* data, size_t count, void* user_data);

// Après
using callback_t = std::function<void(std::span<const float>)>;

void generer_signal(std::size_t nb_echantillons, callback_t on_data) {
    std::vector<float> buffer(nb_echantillons);
    for (std::size_t i = 0; i < nb_echantillons; ++i) {
        buffer[i] = static_cast<float>(i) * 0.1f;
    }

    // Envoie le buffer complet au callback
    on_data(buffer);

    // Ou en tranches
    std::span<const float> vue(buffer);
    on_data(vue.first(nb_echantillons / 2));
    on_data(vue.last(nb_echantillons / 2));
}

int main() {
    generer_signal(10, [](std::span<const float> data) {
        std::println("Reçu {} échantillons", data.size());
    });
    // Reçu 10 échantillons
    // Reçu 5 échantillons
    // Reçu 5 échantillons
}
```

## Interopérabilité avec les API C existantes

Lors de la migration, vous devrez souvent appeler des fonctions C qui attendent toujours un `(T*, size_t)`. Le passage se fait via `data()` et `size()` :

```cpp
#include <span>
#include <cstring>
#include <print>

// API C existante qu'on ne peut pas modifier
extern "C" {
    // Simulée pour l'exemple
    int c_process_buffer(const int* data, int count) {
        int total = 0;
        for (int i = 0; i < count; ++i) total += data[i];
        return total;
    }
}

// Wrapper moderne qui utilise span en interne
int traiter(std::span<const int> donnees) {
    // Passage à l'API C via data() et size()
    return c_process_buffer(donnees.data(),
                            static_cast<int>(donnees.size()));
}

int main() {
    std::vector<int> v{1, 2, 3, 4, 5};

    // L'appelant utilise l'API moderne
    std::println("Résultat : {}", traiter(v));  // 15
}
```

Ce pattern de **wrapper** est la stratégie de migration recommandée : on enveloppe l'API C dans une fonction C++ qui utilise `std::span`, et le reste du code C++ n'interagit qu'avec le wrapper.

## Gestion des sous-ensembles (slicing)

Un avantage majeur de `std::span` sur le pattern pointeur+taille est la capacité d'extraire des sous-vues de manière lisible et sûre. Avec l'ancien style, le slicing est source d'erreurs d'arithmétique de pointeur :

```cpp
#include <span>
#include <vector>
#include <print>

void traiter_entete(std::span<const std::uint8_t> data);  
void traiter_corps(std::span<const std::uint8_t> data);  

// Avant — arithmétique de pointeur manuelle, fragile
// void parser_v1(const uint8_t* data, size_t len) {
//     if (len < 12) return;
//     traiter_entete(data, 12);                  // les 12 premiers octets
//     traiter_corps(data + 12, len - 12);        // le reste — facile de se tromper
// }

// Après — sous-vues explicites et lisibles
void parser_v2(std::span<const std::uint8_t> data) {
    if (data.size() < 12) return;

    auto entete = data.first(12);       // les 12 premiers octets
    auto corps  = data.subspan(12);     // tout à partir de l'offset 12

    traiter_entete(entete);
    traiter_corps(corps);
}
```

Les méthodes `first`, `last` et `subspan` rendent l'intention explicite. Il n'y a plus de calcul `data + offset` ni de soustraction `len - offset` à écrire manuellement.

## Fonctions à sorties multiples

L'ancien pattern utilisait souvent un paramètre de sortie `(T* out, size_t max_count)` avec une valeur de retour indiquant combien d'éléments ont été réellement écrits. `std::span` peut servir de buffer de sortie, et on retourne un sous-span représentant la partie effectivement remplie :

```cpp
#include <span>
#include <vector>
#include <print>

// Avant
// size_t lire_donnees(int* buffer, size_t buffer_size);

// Après — retourne un span sur la partie remplie du buffer
std::span<int> lire_donnees(std::span<int> buffer) {
    // Simulation : on remplit les 3 premiers éléments
    std::size_t nb_lus = std::min(buffer.size(), std::size_t{3});
    for (std::size_t i = 0; i < nb_lus; ++i) {
        buffer[i] = static_cast<int>((i + 1) * 100);
    }
    // Retourne un span sur la partie effectivement remplie
    return buffer.first(nb_lus);
}

int main() {
    std::vector<int> buf(10, 0);

    auto resultat = lire_donnees(buf);
    // resultat est un span sur les éléments remplis

    std::println("Lus : {} éléments", resultat.size());
    for (auto val : resultat) std::print("{} ", val);
    // Sortie : Lus : 3 éléments
    //          100 200 300
}
```

Ce pattern est plus expressif que le retour d'un `size_t` : le span de retour est directement itérable et passable à d'autres fonctions sans recalcul.

## Checklist de migration

Voici un guide pas à pas pour migrer une API `(T*, size_t)` vers `std::span` :

**1. Identifier la const-correctness.** Si la fonction ne modifie pas le buffer, utilisez `std::span<const T>`. Sinon, `std::span<T>`.

**2. Remplacer la paire de paramètres par un seul `std::span`.** Supprimer le paramètre de taille.

**3. Remplacer les accès `data[i]` par `span[i]` ou un range-based for.** Le code interne change peu — `std::span` fournit `operator[]`, `begin()`/`end()`, et `data()`.

**4. Mettre à jour les sites d'appel.** Dans la plupart des cas, les appels se simplifient : `f(v.data(), v.size())` devient `f(v)`.

**5. Pour les appels vers des API C non modifiables, utiliser `data()` et `size()`.** C'est la passerelle entre le monde `std::span` et le monde `(T*, size_t)`.

**6. Profiter du slicing.** Remplacer les calculs `ptr + offset, len - offset` par `first()`, `last()`, `subspan()`.

---

## Bonnes pratiques

**Passez `std::span` par valeur.** C'est un objet léger (8 ou 16 octets). Le passer par référence ajoute une indirection inutile.

**Utilisez `std::span<const T>` par défaut.** Ne passez `std::span<T>` (mutable) que si la fonction modifie effectivement les données.

**Ne stockez pas un `std::span` comme membre de classe sans précaution.** Le span ne possède pas les données. Si l'objet qui possède les données est détruit ou réalloué, le span devient dangling. Si vous devez stocker un span, documentez clairement la contrainte de durée de vie.

**Migrez de l'extérieur vers l'intérieur.** Commencez par modifier les API publiques de vos bibliothèques pour accepter `std::span`, puis migrez progressivement les fonctions internes. Les wrappers autour des API C permettent une coexistence temporaire.

**N'utilisez pas `std::span` pour remplacer `std::string_view`.** Pour les chaînes de caractères, `std::string_view` est le choix approprié — il offre des méthodes spécialisées (`find`, `substr`, `starts_with`, etc.) que `std::span<const char>` ne possède pas.
