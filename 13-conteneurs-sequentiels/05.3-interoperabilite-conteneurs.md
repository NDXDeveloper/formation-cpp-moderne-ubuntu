# 13.5.3 — Interopérabilité avec vector, array et C arrays

## Chapitre 13 : Conteneurs Séquentiels — std::span

---

## Introduction

La promesse centrale de `std::span` est d'unifier l'accès aux données contiguës, quelle que soit leur provenance. Mais les règles de conversion implicite varient selon la source — `std::vector`, `std::array`, tableau C, ou pointeur brut — et certaines combinaisons réservent des subtilités. Cette section détaille les mécanismes de conversion pour chaque type de source, les cas limites à connaître, et les patterns d'interopérabilité entre conteneurs via `std::span`.

---

## Règles de conversion par source

### Depuis `std::vector<T>`

`std::vector<T>` se convertit implicitement en `std::span<T>` (mutable) ou `std::span<const T>` (lecture seule). Le span résultant est toujours **dynamique** puisque la taille d'un vector est connue uniquement à l'exécution :

```cpp
#include <span>
#include <vector>
#include <print>

void lire(std::span<const int> s) {
    std::println("Lecture de {} éléments", s.size());
}

void modifier(std::span<int> s) {
    for (auto& val : s) val *= 2;
}

int main() {
    std::vector<int> v{10, 20, 30};

    lire(v);       // vector<int> → span<const int>  ✅
    modifier(v);   // vector<int> → span<int>         ✅

    // const vector ne se convertit qu'en span<const T>
    const std::vector<int> cv{1, 2, 3};
    lire(cv);      // const vector<int> → span<const int>  ✅
    // modifier(cv);  // ❌ Erreur : const vector → span<int> interdit
}
```

Le span pointe vers le bloc interne du vector (`v.data()`). Il est donc soumis aux mêmes règles d'invalidation : toute opération qui réalloue le vector (push_back, insert, reserve…) rend le span dangling.

```cpp
#include <span>
#include <vector>

void danger() {
    std::vector<int> v{1, 2, 3};
    std::span<int> s(v);

    v.push_back(4);   // réallocation possible

    // s est potentiellement dangling
    // s[0] → comportement indéfini
}
```

> 🔥 **Règle** : ne modifiez jamais un vector pendant qu'un span le référence. Si vous devez modifier le vector, recréez le span après.

### Depuis `std::array<T, N>`

`std::array<T, N>` se convertit implicitement en span **statique** (`std::span<T, N>`) ou **dynamique** (`std::span<T>`). La conversion statique est préférée par CTAD :

```cpp
#include <span>
#include <array>
#include <print>

void lire_dynamique(std::span<const int> s) {
    std::println("Dynamique : {} éléments", s.size());
}

void lire_statique(std::span<const int, 4> s) {
    std::println("Statique  : {} éléments (extent={})", s.size(), s.extent);
}

int main() {
    std::array<int, 4> arr{10, 20, 30, 40};

    // Conversion implicite vers span dynamique
    lire_dynamique(arr);   // array<int,4> → span<const int>       ✅

    // Conversion implicite vers span statique de même taille
    lire_statique(arr);    // array<int,4> → span<const int, 4>    ✅

    // CTAD : déduit un span statique
    std::span s(arr);      // std::span<int, 4>
    std::println("sizeof = {}", sizeof(s));  // 8

    // Span statique de taille différente : interdit implicitement
    // std::span<const int, 3> s3 = arr;  // ❌ Erreur : 4 ≠ 3
    // Possible via sous-vue explicite :
    std::span<const int, 3> s3 = std::span(arr).first<3>();  // ✅
}
```

Puisque `std::array` vit sur la stack et ne réalloue jamais, le span reste valide pendant toute la durée de vie du `std::array`. Le seul risque est le cas classique du dangling : retourner un span vers un `std::array` local.

```cpp
#include <span>
#include <array>

// ❌ DANGER : retourne un span vers des données locales
std::span<const int> dangereux() {
    std::array<int, 3> local{1, 2, 3};
    return local;   // local détruit → span dangling
}

// ✅ CORRECT : l'array survit au span
void correct() {
    std::array<int, 3> arr{1, 2, 3};
    std::span<const int> s(arr);
    // s valide tant que arr existe
}
```

### Depuis un tableau C (`T[N]`)

Les tableaux C se convertissent implicitement en `std::span`, et CTAD déduit un span **statique** puisque la taille est connue à la compilation :

```cpp
#include <span>
#include <print>

void lire(std::span<const int> s) {
    for (auto val : s) std::print("{} ", val);
    std::println("");
}

int main() {
    int tab[] = {10, 20, 30, 40, 50};

    // Conversion implicite vers span dynamique
    lire(tab);  // ✅  int[5] → span<const int>

    // CTAD : déduit span statique
    std::span s(tab);  // std::span<int, 5>
    std::println("sizeof = {}, extent = {}", sizeof(s), s.extent);
    // sizeof = 8, extent = 5
}
```

Un piège classique concerne la **décroissance en pointeur** (*array-to-pointer decay*). Quand un tableau C est passé à une fonction par pointeur, la taille est perdue et la conversion en span statique n'est plus possible :

```cpp
#include <span>
#include <print>

void avec_tableau(int tab[5]) {
    // tab est en réalité un int* — la taille 5 est ignorée par le compilateur
    // std::span s(tab);  // ❌ Erreur : ne peut pas déduire la taille d'un pointeur
}

void avec_reference(int (&tab)[5]) {
    // Référence vers un tableau — la taille est préservée
    std::span s(tab);    // ✅ std::span<int, 5>
    std::println("size = {}", s.size());
}

int main() {
    int tab[] = {1, 2, 3, 4, 5};
    avec_reference(tab);  // size = 5
}
```

> 💡 Si vous devez passer un tableau C à une fonction tout en préservant sa taille pour construire un span, utilisez une référence vers un tableau (`T (&)[N]`) ou, mieux encore, acceptez directement un `std::span`.

---

### Depuis un pointeur brut + taille

Un pointeur brut seul ne porte aucune information de taille. La construction d'un span nécessite de fournir explicitement le compteur — aucune conversion implicite n'est possible :

```cpp
#include <span>
#include <print>

int main() {
    int data[] = {10, 20, 30, 40, 50};
    int* ptr = data;

    // std::span s(ptr);           // ❌ Erreur : taille inconnue
    std::span s1(ptr, 5);          // ✅ span dynamique : span<int>
    std::span<int, 5> s2(ptr, 5);  // ✅ span statique (taille vérifiée à l'exécution uniquement)

    std::println("s1.size = {}, s2.size = {}", s1.size(), s2.size());

    // Construction depuis une paire de pointeurs (début, fin)
    std::span s3(ptr, ptr + 3);    // ✅ span<int> de taille 3
    std::println("s3.size = {}", s3.size());  // 3
}
```

C'est le mode de construction le plus utilisé pour interfacer `std::span` avec du code C legacy ou des API qui retournent un pointeur et une taille séparément :

```cpp
#include <span>
#include <cstdint>
#include <print>

// API C legacy qui retourne un pointeur et une taille
struct c_buffer {
    std::uint8_t* data;
    std::size_t size;
};

c_buffer c_get_buffer(); // définie ailleurs

void traiter_buffer() {
    auto buf = c_get_buffer();

    // Construction du span depuis le résultat C
    std::span<const std::uint8_t> s(buf.data, buf.size);

    std::println("Buffer de {} octets", s.size());
}
```

## Conversions de mutabilité

La conversion de `std::span<T>` vers `std::span<const T>` est toujours implicite. L'inverse n'est jamais possible — on ne peut pas acquérir un accès en écriture à partir d'un accès en lecture seule :

```cpp
#include <span>
#include <vector>

int main() {
    std::vector<int> v{1, 2, 3};

    std::span<int> s_mut(v);
    std::span<const int> s_ro = s_mut;   // ✅ T → const T implicite

    // std::span<int> s2 = s_ro;          // ❌ const T → T interdit
}
```

Cela signifie qu'une fonction acceptant `std::span<const T>` reçoit indifféremment des sources mutables et constantes, tandis qu'une fonction acceptant `std::span<T>` rejette les sources constantes à la compilation.

## `std::span` comme passerelle entre conteneurs

Un cas d'usage puissant de `std::span` est de servir de **type intermédiaire** pour passer des données d'un conteneur à un autre sans couplage :

```cpp
#include <span>
#include <vector>
#include <array>
#include <print>

// Cette fonction ne connaît ni vector, ni array, ni tableau C
// Elle travaille uniquement avec un span
double produit_scalaire(std::span<const double> a, std::span<const double> b) {
    double resultat = 0.0;
    auto taille = std::min(a.size(), b.size());
    for (std::size_t i = 0; i < taille; ++i) {
        resultat += a[i] * b[i];
    }
    return resultat;
}

int main() {
    // Les données viennent de sources différentes
    std::vector<double> mesures{1.0, 2.0, 3.0};
    std::array<double, 3> poids{0.5, 0.3, 0.2};
    double coefficients[] = {10.0, 20.0, 30.0};

    // Toutes les combinaisons fonctionnent
    std::println("vec × array : {}", produit_scalaire(mesures, poids));
    std::println("vec × C arr : {}", produit_scalaire(mesures, coefficients));
    std::println("array × C   : {}", produit_scalaire(poids, coefficients));

    // Sous-vues croisées
    std::println("partiel     : {}", produit_scalaire(
        std::span(mesures).first(2),
        std::span(poids).last(2)
    ));
}
```

La fonction `produit_scalaire` est compilée **une seule fois** (pas d'instanciation template par type de conteneur) et accepte toute combinaison de sources contiguës. C'est l'avantage de `std::span` sur une approche template pure.

## Conteneurs non supportés

`std::span` ne fonctionne qu'avec les données **contiguës en mémoire**. Les conteneurs dont les éléments sont dispersés ne peuvent pas être convertis en span :

```cpp
#include <span>
#include <list>
#include <deque>
#include <vector>

int main() {
    std::vector<int> vec{1, 2, 3};
    std::span<int> s1(vec);        // ✅ vector — contigu

    std::list<int> lst{1, 2, 3};
    // std::span<int> s2(lst);     // ❌ list — nœuds dispersés

    std::deque<int> dq{1, 2, 3};
    // std::span<int> s3(dq);      // ❌ deque — blocs non contigus
}
```

Techniquement, la conversion est possible pour tout type satisfaisant le concept `std::ranges::contiguous_range` (C++20). En pratique, cela inclut `std::vector`, `std::array`, `std::string`, `std::string_view`, les tableaux C, et tout conteneur personnalisé avec des données contiguës et un `data()` qui retourne un pointeur compatible.

## `std::span` et `std::string` / `std::string_view`

`std::string` stocke ses caractères de manière contiguë et peut se convertir en `std::span<char>` ou `std::span<const char>`. Cependant, pour les chaînes, préférez `std::string_view` qui offre une API spécialisée :

```cpp
#include <span>
#include <string>
#include <string_view>
#include <print>

// Pour du texte : préférez string_view
void traiter_texte(std::string_view sv) {
    std::println("Texte : {}", sv);
}

// Pour des octets bruts : span est approprié
void traiter_octets(std::span<const char> data) {
    std::println("{} octets bruts", data.size());
}

int main() {
    std::string s = "Hello, World!";

    traiter_texte(s);    // ✅ string → string_view
    traiter_octets(s);   // ✅ string → span<const char>

    // Conversion span ↔ string_view (données partagées, zéro copie)
    std::span<const char> sp(s);
    std::string_view sv(sp.data(), sp.size());
    std::println("Via span→sv : {}", sv);
}
```

La règle est simple : si vous manipulez du **texte**, utilisez `std::string_view`. Si vous manipulez des **octets bruts** qui se trouvent être des `char`, utilisez `std::span<const char>` ou `std::span<const std::uint8_t>`.

## Tableau récapitulatif des conversions

| Source | → `span<T>` | → `span<const T>` | → `span<T, N>` | Type de span (CTAD) |  
|---|---|---|---|---|  
| `vector<T>` | ✅ implicite | ✅ implicite | ❌ | Dynamique |  
| `const vector<T>` | ❌ | ✅ implicite | ❌ | Dynamique |  
| `array<T, N>` | ✅ implicite | ✅ implicite | ✅ si même N | Statique (N) |  
| `const array<T, N>` | ❌ | ✅ implicite | ✅ si même N (const) | Statique (N) |  
| `T[N]` | ✅ implicite | ✅ implicite | ✅ si même N | Statique (N) |  
| `const T[N]` | ❌ | ✅ implicite | ✅ si même N (const) | Statique (N) |  
| `T*` + `size_t` | ✅ explicite | ✅ explicite | ✅ explicite | Dynamique |  
| `span<T>` | ✅ identité | ✅ implicite | ✅ via `first<N>` | — |  
| `span<const T>` | ❌ | ✅ identité | ✅ via `first<N>` (const) | — |  
| `span<T, N>` | ✅ implicite (→ dyn) | ✅ implicite | ✅ si même N | — |  
| `string` | ✅ (`span<char>`) | ✅ (`span<const char>`) | ❌ | Dynamique |  
| `list<T>` | ❌ | ❌ | ❌ | — |  
| `deque<T>` | ❌ | ❌ | ❌ | — |

## Bonnes pratiques

**Acceptez `std::span<const T>` dynamique dans vos API publiques.** C'est la forme la plus flexible : elle accepte toutes les sources contiguës, mutables ou constantes, statiques ou dynamiques.

**Réservez les spans statiques aux API où la taille est un invariant du domaine.** Les fonctions internes travaillant sur des structures de taille fixe (vecteurs 3D, headers de protocole) bénéficient d'un span statique pour encoder cette contrainte dans le type.

**Ne mélangez pas modification du conteneur source et utilisation du span.** Le span ne détecte pas les réallocations ou destructions du conteneur sous-jacent. Structurez votre code pour que le span soit créé, utilisé, puis abandonné avant toute modification du conteneur.

**Préférez `std::string_view` à `std::span<const char>` pour le texte.** Les deux sont des vues non-propriétaires, mais `std::string_view` offre des méthodes spécialisées pour la manipulation de chaînes (`find`, `substr`, `starts_with`, `ends_with`) que `std::span` ne possède pas.

**Documentez les contraintes de durée de vie.** Quand une fonction stocke un span (dans un membre, une variable capturée, etc.), documentez explicitement que l'appelant doit garantir la survie des données. Cette contrainte n'est pas vérifiable par le compilateur.
