# 13.1.1 — Fonctionnement interne et capacité

## Chapitre 13 : Conteneurs Séquentiels — std::vector

---

## Introduction

Pour utiliser `std::vector` efficacement, il ne suffit pas de connaître son API — il faut comprendre **comment il fonctionne sous le capot**. Le mécanisme de croissance dynamique, la distinction entre taille et capacité, et les stratégies de réallocation ont un impact direct et mesurable sur les performances de votre programme. Cette section vous donne les clés pour maîtriser ces mécanismes et écrire du code qui coopère avec eux au lieu de les subir.

---

## Structure interne

En interne, un `std::vector<T>` est remarquablement simple. Il se résume à trois pointeurs (ou l'équivalent logique de trois pointeurs) :

```
std::vector<int> v{10, 20, 30, 40, 50};    // capacity = 8

Bloc alloué sur le heap :
┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐
│  10 │  20 │  30 │  40 │  50 │  .  │  .  │  .  │
└─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘
▲                             ▲                    ▲
│                             │                    │
begin_                       end_              end_cap_

◄───── size() = 5 ───────────▶
◄───────────── capacity() = 8 ────────────────────▶
```

Les trois membres internes correspondent à :

- **`begin_`** (ou `data_`) : pointeur vers le premier élément du bloc alloué sur le heap.  
- **`end_`** : pointeur vers la position juste après le dernier élément construit. La différence `end_ - begin_` donne `size()`.  
- **`end_cap_`** : pointeur vers la fin du bloc alloué. La différence `end_cap_ - begin_` donne `capacity()`.

La taille de l'objet `std::vector` lui-même sur la stack est donc typiquement de **24 octets** sur une architecture 64 bits (trois pointeurs de 8 octets chacun), quelle que soit la quantité de données qu'il contient.

> 💡 Les noms exacts des membres varient selon l'implémentation (libstdc++, libc++, MSVC STL), mais le principe reste identique. Ces membres sont privés et inaccessibles directement.

## Le mécanisme de croissance

Lorsqu'on insère un élément dans un `std::vector` qui a atteint sa capacité (`size() == capacity()`), une **réallocation** se produit. Ce processus se déroule en quatre étapes :

**1. Allocation d'un nouveau bloc plus grand.** Le vecteur demande à l'allocateur un nouveau bloc de mémoire dont la taille est un multiple de l'ancienne capacité.

**2. Déplacement (ou copie) des éléments existants.** Les éléments sont transférés de l'ancien bloc vers le nouveau. Si le type `T` possède un constructeur de déplacement `noexcept`, le déplacement est utilisé (rapide). Sinon, le vecteur recourt à la copie (plus lent, mais exception-safe).

**3. Destruction des anciens éléments.** Les destructeurs sont appelés sur les éléments dans l'ancien bloc.

**4. Libération de l'ancien bloc.** La mémoire de l'ancien bloc est rendue à l'allocateur.

```
Avant réallocation (capacity = 4, size = 4) :
┌─────┬─────┬─────┬─────┐
│  10 │  20 │  30 │  40 │   ← ancien bloc (plein)
└─────┴─────┴─────┴─────┘

push_back(50) déclenche une réallocation :

1. Allocation nouveau bloc (capacity = 8) :
┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐
│     │     │     │     │     │     │     │     │   ← nouveau bloc vide
└─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘

2. Déplacement des éléments + insertion du nouveau :
┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐
│  10 │  20 │  30 │  40 │  50 │     │     │     │   ← nouveau bloc
└─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘

3-4. Destruction + libération de l'ancien bloc :
┌─────┬─────┬─────┬─────┐
│ ░░░ │ ░░░ │ ░░░ │ ░░░ │   ← mémoire libérée
└─────┴─────┴─────┴─────┘
```

## Facteur de croissance

Le standard C++ ne prescrit pas de facteur de croissance spécifique. Il exige seulement que `push_back` soit en **O(1) amorti**, ce qui impose mathématiquement un facteur de croissance **multiplicatif** (et non additif). En pratique, les implémentations majeures diffèrent :

| Implémentation | Facteur | Séquence de capacités |  
|---|---|---|  
| GCC (libstdc++) | ×2 | 1, 2, 4, 8, 16, 32, 64… |  
| Clang (libc++) | ×2 | 1, 2, 4, 8, 16, 32, 64… |  
| MSVC (MS STL) | ×1.5 | 1, 2, 3, 4, 6, 9, 13, 19… |

Le facteur 2 est simple et offre un bon amortissement. Le facteur 1.5 gaspille moins de mémoire (au maximum 33% d'espace inutilisé contre 50% pour le facteur 2) et présente un avantage théorique : les blocs libérés précédemment peuvent être fusionnés par l'allocateur pour satisfaire une allocation future, ce qui n'est jamais possible avec un facteur 2 (car la somme de toutes les capacités précédentes est toujours strictement inférieure à la capacité suivante).

---

## Amortissement : pourquoi push_back est O(1) amorti

L'analyse amortie démontre que le coût moyen d'un `push_back` est constant, même si certaines insertions individuelles coûtent O(n) à cause des réallocations.

Prenons l'exemple d'un facteur de croissance de 2 et d'une séquence de n insertions dans un vecteur initialement vide. Les réallocations se produisent aux tailles 1, 2, 4, 8, …, n. À chaque réallocation de taille k, on copie/déplace k éléments. Le coût total des copies est donc :

```
1 + 2 + 4 + 8 + ... + n = 2n - 1
```

En ajoutant les n insertions elles-mêmes (coût 1 chacune), le coût total est de `3n - 1`, soit un coût amorti de **~3 opérations par insertion** — une constante, indépendante de n.

```cpp
#include <vector>
#include <print>

int main() {
    std::vector<int> v;
    int nb_reallocs = 0;
    std::size_t ancienne_cap = 0;

    for (int i = 0; i < 1'000'000; ++i) {
        v.push_back(i);
        if (v.capacity() != ancienne_cap) {
            ++nb_reallocs;
            ancienne_cap = v.capacity();
        }
    }

    std::println("Éléments insérés : 1 000 000");
    std::println("Réallocations    : {}", nb_reallocs);
    std::println("Capacité finale  : {}", v.capacity());
}
```

Résultat typique avec GCC (facteur ×2) :

```
Éléments insérés : 1 000 000
Réallocations    : 21  
Capacité finale  : 1048576  
```

Seulement 21 réallocations pour un million d'insertions. C'est la puissance de la croissance géométrique : le nombre de réallocations croît en O(log n).

## Contrôler la capacité : `reserve()`

Lorsque vous connaissez à l'avance le nombre d'éléments (ou une borne supérieure raisonnable), `reserve()` vous permet d'allouer la capacité nécessaire en une seule fois, éliminant **toutes** les réallocations intermédiaires :

```cpp
#include <vector>
#include <print>

int main() {
    std::vector<int> v;
    v.reserve(1'000'000);  // alloue pour 1M éléments, size reste 0

    std::println("size={}, capacity={}", v.size(), v.capacity());
    // Sortie : size=0, capacity=1000000

    for (int i = 0; i < 1'000'000; ++i) {
        v.push_back(i);    // aucune réallocation
    }

    std::println("size={}, capacity={}", v.size(), v.capacity());
    // Sortie : size=1000000, capacity=1000000
}
```

L'impact sur les performances peut être considérable. Voici une mesure comparative typique :

```cpp
#include <vector>
#include <chrono>
#include <print>

int main() {
    constexpr int N = 10'000'000;

    // Sans reserve
    auto t1 = std::chrono::high_resolution_clock::now();
    {
        std::vector<int> v;
        for (int i = 0; i < N; ++i) v.push_back(i);
    }
    auto t2 = std::chrono::high_resolution_clock::now();

    // Avec reserve
    auto t3 = std::chrono::high_resolution_clock::now();
    {
        std::vector<int> v;
        v.reserve(N);
        for (int i = 0; i < N; ++i) v.push_back(i);
    }
    auto t4 = std::chrono::high_resolution_clock::now();

    auto sans = std::chrono::duration_cast<std::chrono::milliseconds>(t2 - t1);
    auto avec = std::chrono::duration_cast<std::chrono::milliseconds>(t4 - t3);

    std::println("Sans reserve : {} ms", sans.count());
    std::println("Avec reserve : {} ms", avec.count());
}
```

Résultats typiques sur une machine moderne (GCC 15, -O2) :

```
Sans reserve : 45 ms  
Avec reserve : 18 ms  
```

Le gain vient de l'élimination des réallocations et des copies, mais aussi de la **prédictibilité** : pas de fragmentation mémoire ni de surcoûts d'allocation intermédiaires.

> 🔥 **Règle pratique** : appelez `reserve()` systématiquement lorsque vous pouvez estimer le nombre final d'éléments. C'est l'une des optimisations les plus simples et les plus efficaces en C++.

## `reserve()` vs `resize()` : une confusion fréquente

Ces deux méthodes sont souvent confondues, mais leur sémantique est fondamentalement différente :

```cpp
#include <vector>
#include <print>

int main() {
    // reserve() : alloue de la capacité, ne crée PAS d'éléments
    std::vector<int> v1;
    v1.reserve(100);
    std::println("reserve(100) → size={}, capacity={}", v1.size(), v1.capacity());
    // Sortie : reserve(100) → size=0, capacity=100
    // v1[0] serait un comportement indéfini !

    // resize() : change la taille, CRÉE des éléments (value-initialized)
    std::vector<int> v2;
    v2.resize(100);
    std::println("resize(100)  → size={}, capacity={}", v2.size(), v2.capacity());
    // Sortie : resize(100)  → size=100, capacity=100
    // v2[0] est valide et vaut 0

    // resize() avec valeur
    std::vector<int> v3;
    v3.resize(100, -1);
    std::println("v3[50] = {}", v3[50]);
    // Sortie : v3[50] = -1
}
```

Après `reserve(100)`, le vecteur a de la place pour 100 éléments mais en contient zéro. Tenter d'accéder à `v[0]` est un **comportement indéfini**. Après `resize(100)`, le vecteur contient effectivement 100 éléments construits et accessibles.

La règle est simple : utilisez `reserve()` quand vous allez remplir le vecteur progressivement avec `push_back` ou `emplace_back`, et `resize()` quand vous avez besoin d'éléments immédiatement accessibles par index.

## Réduire la capacité : `shrink_to_fit()`

Après des suppressions massives, un vecteur peut conserver une capacité bien supérieure à sa taille. `shrink_to_fit()` (C++11) demande à l'implémentation de libérer la mémoire excédentaire :

```cpp
#include <vector>
#include <print>

int main() {
    std::vector<int> v(10'000);

    v.clear();  // size=0, mais la capacité reste à 10000
    std::println("Après clear : size={}, capacity={}", v.size(), v.capacity());
    // Sortie : Après clear : size=0, capacity=10000

    v.shrink_to_fit();  // demande de réduire la capacité
    std::println("Après shrink : size={}, capacity={}", v.size(), v.capacity());
    // Sortie probable : Après shrink : size=0, capacity=0
}
```

Deux points importants à retenir sur `shrink_to_fit()`. Premièrement, c'est une **requête non contraignante** (*non-binding request*) : l'implémentation est libre de l'ignorer. En pratique, les trois implémentations majeures l'honorent, mais le standard ne le garantit pas. Deuxièmement, si la capacité est effectivement réduite, l'opération implique une réallocation (allocation d'un bloc plus petit, déplacement des éléments, libération de l'ancien bloc), ce qui **invalide tous les itérateurs et pointeurs**.

---

## L'idiome swap pour libérer la mémoire (pré-C++11)

Avant `shrink_to_fit()`, l'idiome classique pour libérer la mémoire excédentaire d'un vecteur était le *swap trick* :

```cpp
#include <vector>
#include <print>

int main() {
    std::vector<int> v(10'000);
    v.clear();

    // Idiome pré-C++11 : swap avec un vecteur temporaire
    std::vector<int>(v).swap(v);
    // Ou de manière équivalente :
    // std::vector<int>().swap(v);  // pour vider complètement

    std::println("size={}, capacity={}", v.size(), v.capacity());
}
```

Cette technique crée un vecteur temporaire (dont la capacité est ajustée à la taille), échange les contenus internes (opération O(1) — seuls les trois pointeurs sont échangés), puis le temporaire est détruit en emportant l'ancien bloc surdimensionné. En C++ moderne, préférez `shrink_to_fit()` qui exprime la même intention de manière plus lisible.

## L'importance de `noexcept` pour le move constructor

Lors d'une réallocation, `std::vector` doit transférer les éléments de l'ancien bloc vers le nouveau. Deux stratégies sont possibles : le **déplacement** (move) et la **copie**.

Le vecteur préfère le déplacement car il est généralement beaucoup plus rapide (transfert de pointeurs internes au lieu de duplication de données). Cependant, il ne peut utiliser le déplacement que si le move constructor du type `T` est déclaré `noexcept`. La raison est liée aux garanties d'exception :

Si un move constructor lance une exception à mi-parcours (par exemple, au 500ᵉ élément sur 1000), les 499 éléments déjà déplacés dans le nouveau bloc ont altéré de manière irréversible les originaux dans l'ancien bloc. Le vecteur ne peut ni restaurer l'ancien état ni terminer le transfert. L'opération est dans un état incohérent.

Avec la copie, si une exception survient au 500ᵉ élément, les 499 premiers éléments dans l'ancien bloc sont toujours intacts. Le vecteur peut détruire les copies partielles dans le nouveau bloc, libérer ce dernier, et propager l'exception — le vecteur reste dans son état d'origine. C'est la **garantie forte d'exception** (*strong exception guarantee*).

```cpp
#include <vector>
#include <print>
#include <string>

class Lent {
    std::string donnees_;
public:
    Lent(std::string s) : donnees_(std::move(s)) {}

    // Move constructor SANS noexcept → le vector utilisera la COPIE
    Lent(Lent&& other) : donnees_(std::move(other.donnees_)) {}

    Lent(const Lent& other) : donnees_(other.donnees_) {
        std::println("  Copie !");
    }
};

class Rapide {
    std::string donnees_;
public:
    Rapide(std::string s) : donnees_(std::move(s)) {}

    // Move constructor AVEC noexcept → le vector utilisera le MOVE
    Rapide(Rapide&& other) noexcept : donnees_(std::move(other.donnees_)) {}

    Rapide(const Rapide& other) : donnees_(other.donnees_) {
        std::println("  Copie !");
    }
};

int main() {
    std::println("=== Sans noexcept (copies lors des réallocations) ===");
    {
        std::vector<Lent> v;
        v.reserve(2);
        v.emplace_back("aaa");
        v.emplace_back("bbb");
        std::println("-- Réallocation imminente --");
        v.emplace_back("ccc");  // réallocation → copies de "aaa" et "bbb"
    }

    std::println("\n=== Avec noexcept (moves lors des réallocations) ===");
    {
        std::vector<Rapide> v;
        v.reserve(2);
        v.emplace_back("aaa");
        v.emplace_back("bbb");
        std::println("-- Réallocation imminente --");
        v.emplace_back("ccc");  // réallocation → moves de "aaa" et "bbb"
    }
}
```

Sortie :

```
=== Sans noexcept (copies lors des réallocations) ===
-- Réallocation imminente --
  Copie !
  Copie !

=== Avec noexcept (moves lors des réallocations) ===
-- Réallocation imminente --
```

Aucune copie dans le second cas. Pour des types contenant des chaînes, des vecteurs ou d'autres ressources lourdes, la différence de performance est massive.

> 🔥 **Règle absolue** : déclarez toujours vos move constructors et move assignment operators `noexcept` si votre implémentation ne lance pas d'exception. C'est une condition nécessaire pour que `std::vector` exploite la sémantique de mouvement lors des réallocations.

## `emplace_back` vs `push_back` : construction en place

`push_back` prend un objet déjà construit (par copie ou par déplacement), tandis que `emplace_back` **construit l'objet directement dans le stockage du vecteur** en transférant ses arguments au constructeur de `T` :

```cpp
#include <vector>
#include <string>
#include <print>

struct Connexion {
    std::string host;
    int port;

    Connexion(std::string h, int p) : host(std::move(h)), port(p) {
        std::println("  Constructeur({}, {})", host, port);
    }

    Connexion(const Connexion& other) : host(other.host), port(other.port) {
        std::println("  Copie({}, {})", host, port);
    }

    Connexion(Connexion&& other) noexcept
        : host(std::move(other.host)), port(other.port) {
        std::println("  Move({}, {})", host, port);
    }
};

int main() {
    std::vector<Connexion> conns;
    conns.reserve(4);  // éviter les réallocations pour l'exemple

    std::println("--- push_back avec temporaire ---");
    conns.push_back(Connexion{"localhost", 8080});
    // Constructeur + Move

    std::println("\n--- emplace_back ---");
    conns.emplace_back("127.0.0.1", 3306);
    // Constructeur uniquement — pas de temporaire, pas de move
}
```

Sortie :

```
--- push_back avec temporaire ---
  Constructeur(localhost, 8080)
  Move(localhost, 8080)

--- emplace_back ---
  Constructeur(127.0.0.1, 3306)
```

Avec `emplace_back`, un seul appel au constructeur est effectué : l'objet est construit directement à l'emplacement final dans le bloc mémoire du vecteur. C'est un appel de moins, et pour des types coûteux à déplacer, le gain peut être significatif.

En pratique, pour des types simples comme `int` ou `double`, la différence est négligeable. Mais pour des types contenant des ressources (chaînes, conteneurs imbriqués, handles système), `emplace_back` est le choix idiomatique en C++ moderne.

## Visualiser le comportement interne

Le programme suivant illustre le cycle de vie complet du stockage d'un `std::vector`, en mettant en évidence les réallocations, les changements d'adresse du bloc interne, et la relation entre taille et capacité :

```cpp
#include <vector>
#include <print>

int main() {
    std::vector<int> v;
    const int* ancien_bloc = nullptr;

    std::println("{:<12} {:<8} {:<10} {:<18} {}",
                 "Opération", "size", "capacity", "data()", "Réalloc?");
    std::println("{:-<65}", "");

    for (int i = 0; i < 20; ++i) {
        v.push_back(i * 10);
        bool realloc = (v.data() != ancien_bloc);
        if (realloc || i < 5 || v.size() == v.capacity()) {
            std::println("push_back({:<3}) {:<8} {:<10} {:<18} {}",
                         i * 10, v.size(), v.capacity(),
                         static_cast<const void*>(v.data()),
                         realloc ? "OUI" : "non");
        }
        ancien_bloc = v.data();
    }

    std::println("{:-<65}", "");
    std::println("\nReserve puis remplissage :");
    std::vector<int> v2;
    v2.reserve(20);
    const int* bloc_fixe = v2.data();
    for (int i = 0; i < 20; ++i) {
        v2.push_back(i);
    }
    std::println("Adresse constante : {} (changée : {})",
                 bloc_fixe == v2.data() ? "oui" : "non",
                 bloc_fixe != v2.data() ? "oui" : "non");
}
```

Ce type de programme est un excellent outil d'apprentissage pour observer concrètement les mécanismes décrits dans cette section. L'adresse retournée par `data()` change à chaque réallocation, confirmant qu'un nouveau bloc est alloué et que l'ancien est libéré. Avec `reserve`, l'adresse reste stable sur toute la durée du remplissage.

## Récapitulatif des méthodes liées à la capacité

| Méthode | Effet | Invalidation |  
|---|---|---|  
| `size()` | Retourne le nombre d'éléments | Aucune |  
| `capacity()` | Retourne la capacité allouée | Aucune |  
| `empty()` | Retourne `true` si `size() == 0` | Aucune |  
| `reserve(n)` | Garantit `capacity() >= n`. Si réallocation nécessaire, déplace les éléments | Tous itérateurs si réallocation |  
| `resize(n)` | Ajuste `size()` à n. Ajoute ou détruit des éléments | Tous itérateurs si réallocation |  
| `resize(n, val)` | Comme `resize(n)`, nouveaux éléments initialisés à `val` | Tous itérateurs si réallocation |  
| `shrink_to_fit()` | Requête pour réduire `capacity()` à `size()` | Tous itérateurs si réallocation |  
| `max_size()` | Retourne le nombre maximum théorique d'éléments | Aucune |

> 💡 **Point clé** : `reserve()` ne réduit jamais la capacité. Appeler `reserve(10)` sur un vecteur de capacité 1000 n'a aucun effet. Pour réduire la capacité, utilisez `shrink_to_fit()`.

---

## Bonnes pratiques

**Appelez `reserve()` quand vous connaissez la taille cible.** C'est l'optimisation la plus rentable et la plus simple sur `std::vector`. Même une estimation approximative (arrondie vers le haut) est bénéfique.

**Déclarez vos move constructors `noexcept`.** Sans cela, `std::vector` sera contraint de copier vos objets à chaque réallocation au lieu de les déplacer — un coût potentiellement prohibitif.

**Préférez `emplace_back` à `push_back`.** Pour les types non triviaux, `emplace_back` évite la construction d'un temporaire. Pour les types simples, les deux sont équivalents.

**Ne comptez jamais sur une adresse stable dans un `std::vector`.** Toute opération susceptible de déclencher une réallocation (`push_back`, `emplace_back`, `insert`, `resize`, `reserve`) peut changer l'adresse du bloc interne. Les itérateurs, pointeurs et références vers les éléments deviennent invalides. Ce sujet est traité en détail dans la section 13.1.3.

**Utilisez `clear()` + `shrink_to_fit()` pour libérer la mémoire.** Un simple `clear()` remet la taille à zéro mais conserve la capacité. Si vous ne réutilisez pas le vecteur, ajoutez `shrink_to_fit()` pour libérer la mémoire.
