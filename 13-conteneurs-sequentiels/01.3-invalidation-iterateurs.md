# 13.1.3 — Invalidation des itérateurs

## Chapitre 13 : Conteneurs Séquentiels — std::vector

---

## Introduction

L'invalidation des itérateurs est l'une des sources de bugs les plus insidieuses en C++. Un itérateur invalidé ressemble à un itérateur valide — il compile, il peut même sembler fonctionner lors de tests rapides — mais son utilisation est un **comportement indéfini**. Le programme peut crasher, corrompre silencieusement des données, ou fonctionner correctement pendant des mois avant de tomber en production.

Ce problème est particulièrement aigu avec `std::vector` car deux mécanismes distincts provoquent l'invalidation : la **réallocation** (qui invalide *tout*) et le **décalage d'éléments** (qui invalide les itérateurs au-delà du point de modification).

---

## Qu'est-ce qu'un itérateur invalide ?

Un itérateur devient invalide quand l'élément qu'il référence a été détruit, déplacé, ou que la mémoire sous-jacente a été libérée. Concrètement, un itérateur sur un `std::vector` est un wrapper autour d'un pointeur vers le bloc de stockage interne. Si ce bloc est libéré lors d'une réallocation, le pointeur interne de l'itérateur pointe vers de la mémoire libérée — exactement comme un *dangling pointer*.

L'invalidation s'applique aussi aux **pointeurs** et aux **références** vers les éléments du vecteur. Si vous stockez un `int& ref = v[3]` ou un `int* ptr = &v[3]`, ces références et pointeurs sont soumis aux mêmes règles que les itérateurs.

```cpp
#include <vector>
#include <print>

int main() {
    std::vector<int> v{10, 20, 30};

    int* ptr = &v[1];           // pointeur vers 20
    int& ref = v[1];            // référence vers 20
    auto it = v.begin() + 1;    // itérateur vers 20

    // Les trois pointent vers le même élément
    std::println("ptr={}, ref={}, *it={}", *ptr, ref, *it);
    // Sortie : ptr=20, ref=20, *it=20

    // Force une réallocation
    v.reserve(1000);

    // ptr, ref et it sont TOUS invalidés
    // Les lignes suivantes sont un COMPORTEMENT INDÉFINI :
    // std::println("{}", *ptr);   // dangling pointer
    // std::println("{}", ref);    // dangling reference
    // std::println("{}", *it);    // dangling iterator
}
```

## Règles d'invalidation pour `std::vector`

Les règles se décomposent en deux cas selon qu'une réallocation se produit ou non.

### Cas 1 : une réallocation se produit

Quand `size()` dépasse `capacity()`, le vecteur alloue un nouveau bloc, déplace les éléments, et libère l'ancien. Dans ce cas :

> **Tous les itérateurs, pointeurs et références vers les éléments sont invalidés — sans exception.**

L'adresse de chaque élément a changé puisqu'ils ont été déplacés vers un nouveau bloc mémoire.

### Cas 2 : pas de réallocation

Quand la capacité est suffisante, le bloc mémoire ne change pas. L'invalidation dépend alors de l'opération effectuée et de la position concernée :

| Opération | Invalidation (sans réallocation) |  
|---|---|  
| `push_back`, `emplace_back` | Seul `end()` est invalidé |  
| `insert(pos, ...)`, `emplace(pos, ...)` | `pos` et tous les itérateurs après `pos` |  
| `erase(pos)` | `pos` et tous les itérateurs après `pos` |  
| `erase(first, last)` | `first` et tous les itérateurs après `first` |  
| `pop_back()` | L'itérateur vers le dernier élément et `end()` |  
| `clear()` | Tous les itérateurs |  
| `resize(n)` (si n < size) | Itérateurs vers les éléments supprimés et `end()` |  
| `swap()` | Aucune invalidation (les itérateurs suivent leur conteneur d'origine) |

Pour les opérations d'insertion (`push_back`, `insert`, `emplace_back`, `emplace`), il faut toujours considérer les **deux cas** : la réallocation peut survenir ou non. Si vous ne contrôlez pas la capacité, traitez toujours l'insertion comme invalidant **tous** les itérateurs.

## Piège classique n°1 : suppression en boucle

Le bug le plus fréquent lié à l'invalidation est la suppression d'éléments pendant un parcours avec un itérateur. L'itérateur utilisé pour la boucle est invalidé par `erase`, mais le code continue de l'utiliser :

```cpp
#include <vector>
#include <print>

int main() {
    std::vector<int> v{1, 2, 3, 4, 5, 6, 7, 8};

    // ❌ BUG : l'itérateur est invalidé par erase
    // for (auto it = v.begin(); it != v.end(); ++it) {
    //     if (*it % 2 == 0) {
    //         v.erase(it);  // it est invalidé ici
    //         // ++it dans le for va utiliser un itérateur invalide → UB
    //     }
    // }

    // ✅ CORRECT : utiliser le retour de erase
    for (auto it = v.begin(); it != v.end(); /* pas de ++it ici */) {
        if (*it % 2 == 0) {
            it = v.erase(it);  // erase retourne l'itérateur suivant valide
        } else {
            ++it;              // avancer manuellement seulement si pas de suppression
        }
    }

    for (auto val : v) std::print("{} ", val);
    // Sortie : 1 3 5 7
}
```

La clé est que `erase` retourne un itérateur vers l'élément qui suit celui supprimé. Ce retour est **le seul moyen sûr** de continuer l'itération après une suppression.

Cependant, comme vu dans la section 13.1.2, la solution la plus propre en C++20 est d'éviter entièrement la boucle manuelle :

```cpp
std::erase_if(v, [](int n) { return n % 2 == 0; });
```

## Piège classique n°2 : stockage d'itérateurs à long terme

Stocker un itérateur ou un pointeur vers un élément d'un vecteur pour le réutiliser plus tard est dangereux, car toute insertion ultérieure peut déclencher une réallocation :

```cpp
#include <vector>
#include <print>

int main() {
    std::vector<std::string> noms{"Alice", "Bob", "Charlie"};

    // On stocke un pointeur vers "Bob"
    std::string* ptr_bob = &noms[1];
    std::println("Avant : {}", *ptr_bob);  // Bob

    // Plus tard dans le programme, quelqu'un ajoute des éléments...
    for (int i = 0; i < 100; ++i) {
        noms.emplace_back("Utilisateur_" + std::to_string(i));
    }

    // ptr_bob est maintenant un dangling pointer
    // std::println("Après : {}", *ptr_bob);  // COMPORTEMENT INDÉFINI
}
```

Si vous avez besoin de références stables vers les éléments, plusieurs alternatives existent :

```cpp
#include <vector>
#include <print>

int main() {
    std::vector<std::string> noms{"Alice", "Bob", "Charlie"};

    // Solution 1 : stocker l'INDEX plutôt que le pointeur
    std::size_t idx_bob = 1;
    // L'index reste valide après réallocation (tant qu'on n'insère pas avant)

    for (int i = 0; i < 100; ++i) {
        noms.emplace_back("Utilisateur_" + std::to_string(i));
    }

    std::println("Bob via index : {}", noms[idx_bob]);  // OK

    // Solution 2 : utiliser reserve() pour garantir l'absence de réallocation
    std::vector<std::string> prenoms;
    prenoms.reserve(200);  // capacité suffisante
    prenoms.push_back("Alice");
    prenoms.push_back("Bob");

    std::string* ptr = &prenoms[1];
    for (int i = 0; i < 100; ++i) {
        prenoms.emplace_back("Prenom_" + std::to_string(i));
        // Pas de réallocation car capacity >= 200
    }
    std::println("Bob via pointeur (reserve garanti) : {}", *ptr);  // OK
}
```

> ⚠️ La solution 2 (avec `reserve`) est fragile : elle dépend du fait que vous **ne dépasserez jamais** la capacité réservée. Si un futur développeur ajoute des insertions au-delà de la capacité prévue, le bug réapparaît silencieusement. Préférez les index quand c'est possible.

## Piège classique n°3 : insertion pendant un parcours

Insérer des éléments pendant un parcours est tout aussi dangereux que la suppression, et souvent plus subtil car le bug peut ne se manifester que lorsque la réallocation se produit :

```cpp
#include <vector>
#include <print>

int main() {
    std::vector<int> v{1, 2, 3};

    // ❌ BUG : insert peut invalider tous les itérateurs (réallocation)
    // for (auto it = v.begin(); it != v.end(); ++it) {
    //     if (*it == 2) {
    //         v.insert(it, 99);  // réallocation possible → it invalide
    //     }
    // }

    // ✅ CORRECT : travailler par index
    for (std::size_t i = 0; i < v.size(); ++i) {
        if (v[i] == 2) {
            v.insert(v.begin() + static_cast<std::ptrdiff_t>(i), 99);
            ++i;  // sauter l'élément inséré pour éviter une boucle infinie
        }
    }

    for (auto val : v) std::print("{} ", val);
    // Sortie : 1 99 2 3
}
```

L'approche par index est robuste car les index ne sont pas invalidés par les réallocations — contrairement aux itérateurs, un index ne contient pas d'adresse mémoire.

## Piège classique n°4 : `end()` invalide après modification

Même sans réallocation, `end()` est invalidé par toute opération qui modifie la taille du vecteur. Un schéma de bug courant consiste à calculer `end()` une seule fois avant une boucle qui modifie le vecteur :

```cpp
#include <vector>
#include <print>

int main() {
    std::vector<int> v{1, 2, 3, 4, 5};
    v.reserve(100);  // pas de réallocation possible

    // ❌ BUG : fin est invalidé dès le premier push_back
    // auto fin = v.end();
    // for (auto it = v.begin(); it != fin; ++it) {
    //     v.push_back(*it * 10);  // fin n'est plus l'end() actuel
    // }

    // ✅ CORRECT : recalculer end() ou travailler avec la taille initiale
    auto taille_initiale = v.size();
    for (std::size_t i = 0; i < taille_initiale; ++i) {
        v.push_back(v[i] * 10);
    }

    for (auto val : v) std::print("{} ", val);
    // Sortie : 1 2 3 4 5 10 20 30 40 50
}
```

## Détecter les invalidations : outils et techniques

Le comportement indéfini lié aux itérateurs invalidés est difficile à reproduire et à diagnostiquer avec les outils classiques. Voici les techniques disponibles.

### Mode debug de la bibliothèque standard

Les trois implémentations majeures proposent un mode debug qui détecte l'utilisation d'itérateurs invalidés à l'exécution :

```bash
# GCC (libstdc++) — mode debug complet
g++ -D_GLIBCXX_DEBUG -D_GLIBCXX_DEBUG_PEDANTIC -g programme.cpp

# Clang (libc++) — mode debug
clang++ -D_LIBCPP_HARDENING_MODE=_LIBCPP_HARDENING_MODE_DEBUG -g programme.cpp
```

Avec ces flags, le programme ci-dessous affiche un message d'erreur explicite au lieu de produire un comportement indéfini :

```cpp
#include <vector>

int main() {
    std::vector<int> v{1, 2, 3};
    auto it = v.begin();

    v.push_back(4);  // peut invalider it

    // Avec _GLIBCXX_DEBUG, cette ligne produit une erreur détaillée :
    int val = *it;   // utilisation d'un itérateur potentiellement invalide
}
```

Message typique avec libstdc++ en mode debug :

```
Error: attempt to dereference a singular iterator.
```

Ce mode ajoute un surcoût significatif (les itérateurs sont enregistrés et suivis), il est donc réservé au développement et aux tests.

### AddressSanitizer

AddressSanitizer (`-fsanitize=address`) détecte les accès à la mémoire libérée, ce qui couvre le cas des réallocations :

```bash
g++ -fsanitize=address -g programme.cpp
```

Cependant, ASan ne détecte pas tous les cas d'invalidation. Si le vecteur n'a pas réalloué (la capacité était suffisante) mais que l'itérateur pointe vers un élément décalé, le pointeur sous-jacent reste dans une zone mémoire valide — ASan ne signalera rien, même si la valeur pointée n'est plus celle attendue.

### Analyse statique

`clang-tidy` propose des checks qui détectent certains patterns d'invalidation à la compilation :

```bash
clang-tidy --checks='bugprone-use-after-move,misc-use-after-free' programme.cpp
```

L'analyse statique ne couvre pas tous les cas (elle ne peut pas toujours savoir si une réallocation aura lieu), mais elle attrape les erreurs les plus évidentes.

## Comparaison avec les autres conteneurs

L'invalidation des itérateurs varie considérablement d'un conteneur à l'autre. Cette comparaison sera approfondie dans les sections suivantes du chapitre, mais voici un aperçu qui motive le choix d'un conteneur alternatif quand la stabilité des itérateurs est critique :

| Conteneur | Insertion | Suppression |  
|---|---|---|  
| `std::vector` | Tous si réalloc ; sinon à partir du point d'insertion | À partir du point de suppression |  
| `std::deque` | Tous (mais pointeurs/références stables si insertion aux extrémités uniquement) | Tous si au milieu ; sinon seuls les itérateurs d'extrémité |  
| `std::list` | **Aucune invalidation** | **Seul l'élément supprimé** |  
| `std::forward_list` | **Aucune invalidation** | **Seul l'élément supprimé** |

C'est l'avantage majeur de `std::list` : les itérateurs, pointeurs et références vers les éléments restent valides après toute insertion ou suppression (sauf pour l'élément supprimé lui-même). Si votre programme dépend fortement de la stabilité des références — par exemple, un graphe où les nœuds contiennent des pointeurs vers d'autres éléments du même conteneur — `std::list` peut être le bon choix malgré ses défauts en termes de localité mémoire.

## Bonnes pratiques

**Préférez les algorithmes aux boucles manuelles.** Les algorithmes STL (`std::remove_if` + `erase`, ou `std::erase_if` en C++20) gèrent correctement les itérateurs. Les boucles manuelles avec suppression ou insertion sont une source permanente de bugs.

**Utilisez `reserve()` pour rendre les insertions prévisibles.** Si vous savez combien d'éléments vous allez insérer, `reserve()` garantit l'absence de réallocation, ce qui préserve les itérateurs, pointeurs et références pour les opérations `push_back`/`emplace_back`.

**Stockez des index plutôt que des itérateurs.** Les index ne sont pas invalidés par les réallocations. Ils restent valides tant que l'élément à cette position n'a pas été supprimé et qu'aucune insertion avant cette position n'a eu lieu.

**Activez le mode debug en développement.** Compilez régulièrement avec `-D_GLIBCXX_DEBUG` (GCC) ou l'équivalent Clang. Le surcoût est acceptable en développement et les erreurs détectées sont précieuses.

**Traitez toute insertion comme potentiellement invalidante.** Sauf si vous avez la garantie explicite que `size() < capacity()` (via un `reserve()` antérieur), considérez que toute insertion invalide tous les itérateurs. C'est la posture la plus sûre.

**Limitez la durée de vie des itérateurs.** Ne stockez un itérateur que pour la durée d'une opération locale. Plus un itérateur vit longtemps, plus le risque qu'une opération intermédiaire l'invalide est élevé.
