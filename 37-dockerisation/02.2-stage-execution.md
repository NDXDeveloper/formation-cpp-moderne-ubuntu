# 37.2.2 — Stage d'exécution minimal

> **Sous-section** : 37.2.2  
> **Section parente** : 37.2 — Multi-stage builds : Optimisation de la taille  
> **Prérequis** : Section 37.2.1 (Stage de compilation), Section 27.4 (Linkage statique vs dynamique)

---

## Rôle du stage runtime

Le stage runtime est l'image qui sera effectivement déployée, stockée dans votre registry, téléchargée par vos nœuds Kubernetes, et exécutée en production. Chaque octet superflu dans cette image a un coût réel : bande passante réseau, temps de démarrage des pods, espace de stockage dans le registry, et — surtout — surface d'attaque exposée.

L'objectif est de construire une image qui ne contient **strictement que** ce qui est nécessaire à l'exécution du binaire C++ : le binaire lui-même, ses librairies dynamiques, les certificats TLS si le binaire fait des appels HTTPS, et un éventuel fichier de configuration. Tout le reste — compilateur, headers, sources, outils de build — est resté dans le stage de compilation et n'existe pas ici.

---

## Choix de l'image de base runtime

Ce choix est distinct de celui du stage de build. Il est tout à fait courant (et recommandé) de compiler dans un conteneur Ubuntu et d'exécuter dans un conteneur Debian slim, ou même dans une image distroless. La seule contrainte est la **compatibilité ABI** : l'image runtime doit fournir une glibc de version supérieure ou égale à celle contre laquelle le binaire a été compilé.

### Les options principales

```dockerfile
# Option 1 : Debian slim — le choix par défaut recommandé
FROM debian:bookworm-slim AS runtime
# ~75 MB compressée. Shell disponible. APT utilisable si besoin.

# Option 2 : Ubuntu minimal
FROM ubuntu:24.04 AS runtime
# ~78 MB compressée. Familier si le build est aussi Ubuntu.

# Option 3 : Distroless (détaillé en §37.5)
FROM gcr.io/distroless/cc-debian12 AS runtime
# ~30 MB compressée. Pas de shell, pas d'APT. Sécurité maximale.

# Option 4 : scratch (binaire statique uniquement)
FROM scratch AS runtime
# 0 MB. Image vide. Exige un binaire 100% statique.

# Option 5 : Alpine (si le build cible musl)
FROM alpine:3.20 AS runtime
# ~7 MB compressée. Uniquement si compilé contre musl (voir §37.1).
```

### Matrice de décision

| Critère | Debian slim | Distroless | scratch | Alpine |
|---------|------------|------------|---------|--------|
| Taille image | ~75 MB | ~30 MB | 0 MB | ~7 MB |
| Shell disponible | Oui | Non | Non | Oui |
| Debug en production | Facile | Difficile | Impossible | Facile |
| Librairie C | glibc | glibc | Aucune | musl |
| Compatibilité binaire | Excellente | Excellente | Statique requis | Recompilation musl |
| Surface d'attaque | Modérée | Faible | Minimale | Modérée |
| Installation de paquets | APT | Impossible | Impossible | apk |

Pour la majorité des projets, **Debian slim** est le point de départ recommandé. Il offre le meilleur équilibre entre légèreté, compatibilité, et capacité de diagnostic en production. La section 37.5 détaillera les images distroless pour les contextes où la sécurité prime sur la commodité de debugging.

## Structure de référence du stage runtime

Voici le stage runtime complet correspondant au stage de build de la section 37.2.1 :

```dockerfile
# ============================================================
# Stage RUNTIME
# ============================================================
FROM debian:bookworm-slim AS runtime

# ── 1. Dépendances runtime (sans les -dev) ────────────────
RUN apt-get update && apt-get install -y --no-install-recommends \
    libssl3 \
    libcurl4 \
    ca-certificates \
    && rm -rf /var/lib/apt/lists/*

# ── 2. Utilisateur non-root ───────────────────────────────
RUN groupadd --gid 1000 appgroup \
    && useradd --uid 1000 --gid appgroup \
       --create-home --shell /bin/false appuser

# ── 3. Copie des artefacts depuis le stage build ──────────
COPY --from=build /install/bin/myapp /usr/local/bin/myapp  
COPY --from=build /install/etc/myapp/ /etc/myapp/  

# ── 4. Permissions et répertoires de travail ──────────────
RUN mkdir -p /var/log/myapp && chown appuser:appgroup /var/log/myapp

# ── 5. Passage en utilisateur non-root ────────────────────
USER appuser  
WORKDIR /home/appuser  

# ── 6. Métadonnées ────────────────────────────────────────
EXPOSE 8080  
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s \  
    CMD ["/usr/local/bin/myapp", "--healthcheck"]

# ── 7. Point d'entrée ────────────────────────────────────
ENTRYPOINT ["myapp"]  
CMD ["--config", "/etc/myapp/config.yaml"]  
```

Détaillons chaque étape.

## Étape 1 : Dépendances runtime

```dockerfile
RUN apt-get update && apt-get install -y --no-install-recommends \
    libssl3 \
    libcurl4 \
    ca-certificates \
    && rm -rf /var/lib/apt/lists/*
```

### Le piège `-dev` vs runtime

C'est l'erreur la plus fréquente dans les stages runtime C++ : installer les paquets de **développement** (`-dev`) au lieu des paquets **runtime**. La différence est considérable :

| Paquet | Contenu | Taille |
|--------|---------|--------|
| `libssl-dev` | Headers `.h`, `.so` symlinks, fichiers `.a`, docs | ~10 MB |
| `libssl3` | Uniquement les `.so` runtime | ~4 MB |
| `libcurl4-openssl-dev` | Headers, pkg-config, liens statiques | ~6 MB |
| `libcurl4` | Uniquement `libcurl.so.4` | ~1 MB |

En production, le binaire est déjà compilé — il n'a besoin que des librairies dynamiques chargées à l'exécution, pas des headers, pas des fichiers de linkage statique, et encore moins de la documentation.

### Comment identifier les paquets runtime corrects

La sortie de `ldd` dans le stage de build (section 37.2.1) vous donne les noms des fichiers `.so` nécessaires. Pour trouver le paquet Debian qui fournit un fichier `.so` spécifique :

```bash
# Depuis une machine Debian/Ubuntu ou le stage build
dpkg -S libssl.so.3
# libssl3:amd64: /usr/lib/x86_64-linux-gnu/libssl.so.3

apt-file search libcurl.so.4
# libcurl4:amd64: /usr/lib/x86_64-linux-gnu/libcurl.so.4
```

Il existe une correspondance quasi systématique : le paquet de développement `libfoo-dev` a un paquet runtime associé `libfoo` ou `libfooN` (où N est le numéro de version SONAME).

### Le cas `ca-certificates`

Le paquet `ca-certificates` installe le bundle de certificats racine du système. Il est indispensable si votre binaire effectue des connexions TLS sortantes (appels à des API HTTPS, connexion à une base de données chiffrée, téléchargement de fichiers). Sans ce paquet, les appels TLS échoueront avec des erreurs de vérification de certificat.

Ce paquet ne pèse que ~400 KB mais son oubli est une source fréquente de bugs en production. Si votre application n'effectue aucune connexion TLS sortante, vous pouvez l'omettre.

## Étape 2 : Utilisateur non-root

```dockerfile
RUN groupadd --gid 1000 appgroup \
    && useradd --uid 1000 --gid appgroup \
       --create-home --shell /bin/false appuser
```

Par défaut, les processus dans un conteneur Docker s'exécutent en tant que `root`. C'est un risque de sécurité : si un attaquant parvient à exploiter une vulnérabilité dans votre binaire, il dispose des privilèges root dans le conteneur. Même si les namespaces Docker limitent l'impact, le principe de moindre privilège impose de créer et d'utiliser un utilisateur dédié.

Les UID/GID fixes (`1000`) facilitent la gestion des permissions dans les environnements Kubernetes, où les `securityContext` peuvent imposer un UID spécifique. Le shell `/bin/false` empêche toute connexion interactive avec cet utilisateur — une précaution supplémentaire.

L'instruction `USER appuser` (étape 5) active ce changement pour toutes les instructions suivantes et pour l'exécution du conteneur. Tout ce qui doit être fait en root (création de répertoires, installation de paquets, changement de permissions) doit être placé **avant** cette instruction.

La section 37.4 reviendra en détail sur les bonnes pratiques de sécurité, y compris les contraintes Kubernetes `runAsNonRoot` et `readOnlyRootFilesystem`.

## Étape 3 : Copie des artefacts

```dockerfile
COPY --from=build /install/bin/myapp /usr/local/bin/myapp  
COPY --from=build /install/etc/myapp/ /etc/myapp/  
```

C'est ici que le multi-stage build prend tout son sens. L'instruction `COPY --from=build` traverse la frontière entre les deux stages : elle extrait des fichiers de l'image de build (qui contient la toolchain complète, les sources, les fichiers objets) et ne copie que les artefacts spécifiés dans l'image runtime.

### Copier depuis `/install` vs depuis l'arborescence de build

En section 37.2.1, nous avons utilisé `cmake --install` pour placer les artefacts dans `/install`. L'avantage est double. D'une part, la structure est propre et prévisible : le binaire est dans `/install/bin/`, les librairies dans `/install/lib/`, la configuration dans `/install/etc/`. D'autre part, seuls les fichiers explicitement déclarés dans les directives `install()` de CMake sont présents — pas de fichiers objets parasites, pas de fichiers temporaires CMake.

L'alternative directe fonctionne aussi pour un projet simple, mais elle est plus fragile :

```dockerfile
# Moins propre : dépend de la structure interne du build
COPY --from=build /src/build/myapp /usr/local/bin/myapp
```

### Placement dans l'image runtime

La convention est de placer les binaires applicatifs dans `/usr/local/bin/`. Ce répertoire est dans le `PATH` par défaut, ce qui permet d'écrire simplement `ENTRYPOINT ["myapp"]` sans chemin absolu. Les fichiers de configuration vont dans `/etc/myapp/`, et les données variables dans `/var/lib/myapp/` ou `/var/log/myapp/`.

### Copie de librairies partagées du projet

Si votre projet produit ses propres librairies partagées (`.so`) en plus du binaire principal, elles doivent aussi être copiées :

```dockerfile
COPY --from=build /install/lib/ /usr/local/lib/

# Reconstruire le cache du linker dynamique
RUN ldconfig
```

L'appel à `ldconfig` met à jour le cache du dynamic linker (`ld-linux-x86-64.so.2`), qui référence les chemins des librairies partagées disponibles. Sans cette étape, les `.so` copiées dans `/usr/local/lib/` pourraient ne pas être trouvées à l'exécution. La section 37.3 traite ce sujet en profondeur.

## Étape 4 : Permissions et répertoires

```dockerfile
RUN mkdir -p /var/log/myapp && chown appuser:appgroup /var/log/myapp
```

Si votre application écrit des fichiers (logs, données temporaires, cache), les répertoires correspondants doivent exister et être accessibles par l'utilisateur non-root. Cette étape se fait **avant** l'instruction `USER` puisqu'elle nécessite les privilèges root pour `chown`.

En pratique, il est préférable que l'application écrive ses logs sur `stdout`/`stderr` (approche twelve-factor, captée nativement par Docker et Kubernetes). Si vous suivez ce principe, cette étape peut être omise. Mais pour les applications qui gèrent un fichier de log local (typique avec spdlog en mode fichier, voir section 40.1), la création du répertoire est nécessaire.

---

## Étape 5 : Passage en utilisateur non-root

```dockerfile
USER appuser  
WORKDIR /home/appuser  
```

À partir de cette ligne, toutes les instructions `RUN`, `CMD`, et `ENTRYPOINT` s'exécutent en tant que `appuser`. Le `WORKDIR` est défini sur le home directory de cet utilisateur — un répertoire dont il est propriétaire.

Il est impératif que **rien après cette ligne** ne nécessite de privilèges root. Si vous devez installer un paquet ou modifier un fichier système, faites-le avant `USER`.

## Étape 6 : Métadonnées

### EXPOSE

```dockerfile
EXPOSE 8080
```

`EXPOSE` est une instruction documentaire : elle indique aux opérateurs que le conteneur écoute sur le port 8080. Elle ne publie pas le port — c'est `docker run -p` ou la configuration Kubernetes `containerPort` qui le font. Néanmoins, c'est une bonne pratique de la maintenir à jour car elle sert de documentation intégrée et est exploitée par certains outils (Docker Compose, reverse proxies automatiques).

### HEALTHCHECK

```dockerfile
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s \
    CMD ["/usr/local/bin/myapp", "--healthcheck"]
```

Le `HEALTHCHECK` permet à Docker de vérifier périodiquement que le processus est fonctionnel. Pour un service C++, on implémente typiquement un flag `--healthcheck` qui effectue un auto-diagnostic rapide (vérification de la connectivité base de données, état des queues internes, etc.) et retourne un code de sortie 0 en cas de succès.

L'alternative, plus courante dans les déploiements Kubernetes, est d'exposer un endpoint HTTP `/health` et d'utiliser les readiness/liveness probes de Kubernetes plutôt que le `HEALTHCHECK` Docker (voir section 40.4). Dans ce cas, l'instruction `HEALTHCHECK` peut être omise du Dockerfile.

Si vous n'avez pas de mécanisme de healthcheck implémenté dans votre binaire, il est préférable d'omettre cette instruction plutôt que d'utiliser un contournement fragile comme `CMD curl localhost:8080/health` — qui nécessiterait d'installer curl dans l'image runtime.

## Étape 7 : Point d'entrée

```dockerfile
ENTRYPOINT ["myapp"]  
CMD ["--config", "/etc/myapp/config.yaml"]  
```

### ENTRYPOINT vs CMD

La distinction est importante. `ENTRYPOINT` définit la commande de base qui sera toujours exécutée. `CMD` fournit les arguments par défaut, qui peuvent être remplacés au `docker run`. Concrètement :

```bash
# Exécution par défaut : myapp --config /etc/myapp/config.yaml
docker run myapp-image

# Override des arguments CMD uniquement
docker run myapp-image --config /etc/myapp/prod.yaml --verbose

# Override complet (ENTRYPOINT + CMD)
docker run --entrypoint /bin/sh myapp-image
```

Cette séparation permet de configurer l'application sans modifier l'image : les arguments de lancement sont passés en ligne de commande ou via la configuration Kubernetes.

### Forme exec vs forme shell

Utilisez toujours la **forme exec** (tableau JSON) pour `ENTRYPOINT` et `CMD` :

```dockerfile
# ✅ Forme exec — le binaire reçoit les signaux directement
ENTRYPOINT ["myapp"]

# ❌ Forme shell — lance /bin/sh -c "myapp", les signaux sont interceptés
ENTRYPOINT myapp
```

La forme shell interpose un processus shell entre Docker et votre binaire. Ce shell intercepte les signaux Unix : quand Docker envoie `SIGTERM` pour arrêter le conteneur (timeout de 10 secondes par défaut avant `SIGKILL`), c'est le shell qui reçoit le signal, pas votre application. Votre code de shutdown propre (fermeture de connexions, flush des buffers, sauvegarde d'état) ne s'exécute jamais.

Avec la forme exec, `myapp` est le PID 1 du conteneur et reçoit directement les signaux. C'est ce qui permet d'implémenter un arrêt gracieux dans votre code C++ :

```cpp
#include <csignal>
#include <atomic>

std::atomic<bool> running{true};

void signal_handler(int signum) {
    // SIGTERM reçu : arrêt propre
    running = false;
}

int main() {
    std::signal(SIGTERM, signal_handler);
    std::signal(SIGINT, signal_handler);

    while (running) {
        // ... boucle principale du service ...
    }

    // Cleanup : fermer les connexions, flusher les logs
    return 0;
}
```

## Vérification de l'image finale

Une fois l'image runtime construite, plusieurs vérifications permettent de s'assurer qu'elle est correcte et optimale.

### Taille de l'image

```bash
docker images myapp
# REPOSITORY   TAG       IMAGE ID       CREATED         SIZE
# myapp        latest    a1b2c3d4e5f6   2 minutes ago   97.4MB
```

Si la taille dépasse vos attentes, `docker history` permet d'identifier les layers les plus lourds :

```bash
docker history myapp --no-trunc --format "{{.Size}}\t{{.CreatedBy}}"
```

### Exécution et vérification fonctionnelle

```bash
# Le binaire démarre et affiche sa version
docker run --rm myapp --version

# Le service démarre et répond
docker run --rm -p 8080:8080 myapp  
curl http://localhost:8080/health  
```

### Inspection des librairies dynamiques dans le conteneur

```bash
docker run --rm --entrypoint ldd myapp /usr/local/bin/myapp
```

Chaque `.so` listée doit être résolue (un chemin réel, pas `not found`). Si une librairie affiche `not found`, elle est manquante dans l'image runtime — il faut soit installer le paquet correspondant, soit la copier depuis le stage de build (voir section 37.3).

### Vérification de l'utilisateur

```bash
docker run --rm --entrypoint whoami myapp
# appuser

docker run --rm --entrypoint id myapp
# uid=1000(appuser) gid=1000(appgroup) groups=1000(appgroup)
```

## Le stage runtime complet assemblé

Voici le Dockerfile complet — stage de build et stage de runtime réunis — tel qu'il serait utilisé pour un projet réel :

```dockerfile
# ============================================================
# Stage BUILD
# ============================================================
FROM ubuntu:24.04 AS build

ENV DEBIAN_FRONTEND=noninteractive  
ENV CXX=g++  
ENV CC=gcc  

RUN apt-get update && apt-get install -y --no-install-recommends \
    g++ cmake ninja-build pkg-config \
    libssl-dev libcurl4-openssl-dev \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /src  
COPY CMakeLists.txt cmake/ ./  
COPY src/ src/  
COPY include/ include/  

RUN cmake -B build -G Ninja \
    -DCMAKE_BUILD_TYPE=Release \
    -DCMAKE_INSTALL_PREFIX=/install \
    && cmake --build build --parallel $(nproc) \
    && cmake --install build

# ============================================================
# Stage RUNTIME
# ============================================================
FROM debian:bookworm-slim AS runtime

RUN apt-get update && apt-get install -y --no-install-recommends \
    libssl3 libcurl4 ca-certificates \
    && rm -rf /var/lib/apt/lists/*

RUN groupadd --gid 1000 appgroup \
    && useradd --uid 1000 --gid appgroup \
       --create-home --shell /bin/false appuser

COPY --from=build /install/bin/myapp /usr/local/bin/myapp

USER appuser  
WORKDIR /home/appuser  

EXPOSE 8080

ENTRYPOINT ["myapp"]  
CMD ["--config", "/etc/myapp/config.yaml"]  
```

Ce template produit une image d'environ **95 MB** pour un serveur HTTP C++ typique — contre 720 MB en single-stage. Le binaire tourne en utilisateur non-root, reçoit les signaux Unix directement, et ne contient aucun outil de build superflu.

Les sections suivantes affinent ce résultat : la section 37.3 traite en détail la gestion des librairies partagées lorsque APT ne suffit pas, la section 37.4 ajoute les couches de sécurité avancées, et la section 37.5 pousse l'optimisation encore plus loin avec les images distroless.
