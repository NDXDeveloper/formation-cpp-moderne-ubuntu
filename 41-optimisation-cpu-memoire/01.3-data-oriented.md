# 41.1.3 — Data-oriented design

> **Chapitre 41 : Optimisation CPU et Mémoire** · Section 41.1 · Sous-section 3  
> **Niveau** : Expert · **Prérequis** : Sections 41.1.1–41.1.2 (hiérarchie cache, cache lines), Chapitre 7 (POO), Chapitre 16 (templates)

---

## Introduction

Les sections 41.1.1 et 41.1.2 ont établi deux faits :

1. Le cache fonctionne par blocs de 64 octets et récompense les accès séquentiels à des données contiguës.
2. Le coût d'un cache miss est 50–200× celui d'un cache hit.

La conclusion logique est brutale : **l'organisation des données en mémoire est souvent plus déterminante pour la performance que l'algorithme lui-même**. Un algorithme O(n log n) sur des données dispersées dans le heap peut être plus lent qu'un algorithme O(n²) sur des données contiguës, pour des tailles réalistes.

Le ***data-oriented design*** (DOD) est une philosophie de conception qui tire les conséquences de ce constat. Au lieu de structurer le code autour des *objets* et de leurs *comportements* (comme le fait la programmation orientée objet classique), le DOD structure le code autour des *données* et de leurs *patterns d'accès*. L'objectif : maximiser la localité spatiale et minimiser les cache misses dans les boucles critiques.

Cette approche ne remplace pas la POO — elle la complète, en offrant une alternative là où la performance est le facteur dominant.

---

## Le problème de la POO classique : AoS et ses conséquences

### Array of Structures (AoS)

La POO encourage naturellement un pattern appelé ***Array of Structures*** (AoS) : on définit une classe regroupant toutes les propriétés d'une entité, puis on stocke un tableau d'instances de cette classe.

```cpp
// Approche POO classique : Array of Structures (AoS)
struct Particle {
    float x, y, z;          // position (12 octets)
    float vx, vy, vz;       // vélocité (12 octets)
    float r, g, b, a;       // couleur (16 octets)
    float mass;              // masse (4 octets)
    float lifetime;          // durée de vie (4 octets)
    std::uint32_t flags;     // état (4 octets)
    std::uint32_t texture_id;// texture (4 octets)
};
// sizeof(Particle) = 56 octets (sans padding supplémentaire)

std::vector<Particle> particles(100'000);
```

En mémoire, le layout est le suivant :

```
┌──────────────────────────────────────────────────────────────┐
│ Particle 0                                                   │
│ [x y z] [vx vy vz] [r g b a] [mass] [lifetime] [flags] [tex] │
├──────────────────────────────────────────────────────────────┤
│ Particle 1                                                   │
│ [x y z] [vx vy vz] [r g b a] [mass] [lifetime] [flags] [tex] │
├──────────────────────────────────────────────────────────────┤
│ Particle 2                                                   │
│ ...                                                          │
└──────────────────────────────────────────────────────────────┘
```

### Le gaspillage de bande passante

Considérons maintenant une boucle de simulation physique qui met à jour uniquement les positions à partir des vélocités :

```cpp
void update_positions(std::vector<Particle>& particles, float dt) {
    for (auto& p : particles) {
        p.x += p.vx * dt;
        p.y += p.vy * dt;
        p.z += p.vz * dt;
    }
}
```

Cette boucle n'accède qu'à **24 octets** par particule (6 floats : x, y, z, vx, vy, vz). Mais chaque particule occupe **56 octets**. Le cache charge des cache lines de 64 octets, donc environ 1 particule par cache line — et 32 octets de cette cache line contiennent des données (couleur, masse, lifetime, flags, texture) dont la boucle n'a **aucun besoin**.

Le résultat : la boucle transfère depuis la RAM **2,3× plus de données** que nécessaire. Sur un système limité par la bande passante mémoire (*bandwidth-bound*), cela signifie que la boucle tourne 2,3× plus lentement que son potentiel théorique.

Et c'est un cas modéré. Dans un moteur de jeu réel, une entité peut contenir des centaines d'octets (mesh, matériaux, scripts, état d'animation, données réseau), dont seule une fraction est nécessaire pour chaque passe de traitement.

## La transformation SoA : Structure of Arrays

### Principe

Le *data-oriented design* propose d'inverser l'organisation : au lieu de stocker un tableau de structures, on stocke **une structure de tableaux** — chaque propriété dans son propre tableau contigu.

```cpp
// Approche DOD : Structure of Arrays (SoA)
struct ParticleSystem {
    // Chaque propriété dans son propre tableau contigu
    std::vector<float> x, y, z;          // positions
    std::vector<float> vx, vy, vz;       // vélocités
    std::vector<float> r, g, b, a;       // couleurs
    std::vector<float> mass;             // masses
    std::vector<float> lifetime;         // durées de vie
    std::vector<std::uint32_t> flags;    // états
    std::vector<std::uint32_t> texture_id; // textures
    
    std::size_t count{0};
    
    void resize(std::size_t n) {
        x.resize(n); y.resize(n); z.resize(n);
        vx.resize(n); vy.resize(n); vz.resize(n);
        r.resize(n); g.resize(n); b.resize(n); a.resize(n);
        mass.resize(n); lifetime.resize(n);
        flags.resize(n); texture_id.resize(n);
        count = n;
    }
};
```

En mémoire, le layout devient :

```
x:   [x₀ x₁ x₂ x₃ x₄ x₅ x₆ x₇ x₈ x₉ x₁₀ x₁₁ x₁₂ x₁₃ x₁₄ x₁₅ ...]  
y:   [y₀ y₁ y₂ y₃ y₄ y₅ y₆ y₇ y₈ y₉ y₁₀ y₁₁ y₁₂ y₁₃ y₁₄ y₁₅ ...]  
z:   [z₀ z₁ z₂ z₃ z₄ z₅ z₆ z₇ z₈ z₉ z₁₀ z₁₁ z₁₂ z₁₃ z₁₄ z₁₅ ...]  
vx:  [vx₀ vx₁ vx₂ ...]  
vy:  [vy₀ vy₁ vy₂ ...]  
vz:  [vz₀ vz₁ vz₂ ...]  
r:   [r₀ r₁ r₂ ...]  
...
```

### La même boucle, transformée

```cpp
void update_positions(ParticleSystem& ps, float dt) {
    for (std::size_t i = 0; i < ps.count; ++i) {
        ps.x[i] += ps.vx[i] * dt;
        ps.y[i] += ps.vy[i] * dt;
        ps.z[i] += ps.vz[i] * dt;
    }
}
```

Maintenant, chaque cache line de 64 octets chargée pour le tableau `x` contient **16 floats** — 16 positions x consécutives, toutes utiles. Aucun octet n'est gaspillé. Le prefetcher matériel détecte le pattern séquentiel et précharge les cache lines suivantes en avance. Les tableaux `r`, `g`, `b`, `a`, `mass`, `lifetime`, `flags`, `texture_id` ne sont même pas touchés par cette boucle — ils restent en RAM sans polluer le cache.

### Impact mesuré

Sur 100 000 particules, boucle `update_positions`, GCC 15, `-O2 -march=native`, AMD Zen 4 :

| Layout | Temps | Bande passante utile | Cache misses L1 |
|--------|-------|---------------------|-----------------|
| AoS | 145 µs | ~38 % | 4,2 % |
| SoA | 52 µs | ~95 % | 0,3 % |

Le SoA est **~2,8× plus rapide**, exclusivement grâce à une meilleure utilisation du cache et de la bande passante mémoire. L'algorithme est identique — seul le layout des données a changé.

De plus, le layout SoA est nettement plus favorable à l'**auto-vectorisation**. Le compilateur peut facilement appliquer des instructions SIMD (SSE/AVX) sur les tableaux de floats homogènes. Avec AoS, les champs entrelacés rendent la vectorisation difficile, voire impossible, sans *gather/scatter* coûteux.

Avec `-O2 -march=native` et auto-vectorisation AVX2 (8 floats par instruction) :

| Layout | Temps (vectorisé) | Speedup vs AoS scalar |
|--------|-------------------|----------------------|
| AoS | 130 µs (vectorisation partielle) | 1,1× |
| SoA | 18 µs (vectorisation complète) | **8×** |

Le SoA combiné à la vectorisation atteint un speedup de **8×** par rapport au AoS scalaire — un facteur qui transforme un budget de 16 ms (limite d'une frame à 60 FPS) en 2 ms, libérant du temps pour le rendu, l'IA et la physique.

## Approche hybride : AoSoA (Array of Structures of Arrays)

Le SoA pur a un inconvénient : lorsqu'une boucle doit accéder à *plusieurs* propriétés d'une même entité (par exemple position + vélocité + masse), les données sont réparties dans des tableaux distincts et potentiellement éloignés en mémoire. Chaque propriété supplémentaire ajoute un flux de lecture séparé, ce qui peut saturer le nombre de *line fill buffers* du processeur.

Le compromis ***AoSoA*** (*Array of Structures of Arrays*) regroupe les données par blocs — typiquement de la taille d'un registre SIMD ou d'un multiple de cache line :

```cpp
constexpr std::size_t BLOCK = 16;  // 16 floats = 64 octets = 1 cache line

struct ParticleBlock {
    // 16 particules, chaque propriété contiguë dans le bloc
    float x[BLOCK], y[BLOCK], z[BLOCK];
    float vx[BLOCK], vy[BLOCK], vz[BLOCK];
    float mass[BLOCK];
    float lifetime[BLOCK];
};
// sizeof(ParticleBlock) = 8 × 16 × 4 = 512 octets = 8 cache lines

std::vector<ParticleBlock> blocks(num_particles / BLOCK);
```

En mémoire :

```
Block 0:
  x:  [x₀  x₁  x₂  ... x₁₅ ]  ← 1 cache line (64 octets)
  y:  [y₀  y₁  y₂  ... y₁₅ ]  ← 1 cache line
  z:  [z₀  z₁  z₂  ... z₁₅ ]  ← 1 cache line
  vx: [vx₀ vx₁ vx₂ ... vx₁₅]  ← 1 cache line
  ...
Block 1:
  x:  [x₁₆ x₁₇ ... x₃₁]
  ...
```

Le traitement itère sur les blocs, et à l'intérieur de chaque bloc, les 16 valeurs d'une même propriété sont contiguës — parfaitement alignées pour le SIMD et tenant dans une seule cache line. Les différentes propriétés d'un même bloc sont proches en mémoire (quelques cache lines d'écart), ce qui préserve la localité spatiale pour les boucles multi-propriétés.

```cpp
void update_positions(std::vector<ParticleBlock>& blocks, float dt) {
    for (auto& blk : blocks) {
        for (std::size_t i = 0; i < BLOCK; ++i) {
            blk.x[i] += blk.vx[i] * dt;
            blk.y[i] += blk.vy[i] * dt;
            blk.z[i] += blk.vz[i] * dt;
        }
    }
}
```

L'AoSoA offre souvent le meilleur compromis entre performance cache, vectorisation SIMD et lisibilité du code. C'est le layout de choix pour les moteurs de simulation à haute performance.

## Le pattern ECS : Entity-Component-System

### Origine et motivation

Le ***Entity-Component-System*** (ECS) est un pattern architectural né dans l'industrie du jeu vidéo, où des milliers d'entités (personnages, projectiles, effets visuels, éléments de décor) doivent être traités chaque frame en moins de 16 ms.

La POO classique modélise une entité de jeu comme un objet avec héritage :

```cpp
// Approche POO classique — hiérarchie d'héritage
class GameObject {  
public:  
    virtual void update(float dt) = 0;
    virtual void render() = 0;
    virtual ~GameObject() = default;
};

class Enemy : public GameObject {
    float x, y, z;
    float health;
    AIState ai_state;
    Mesh* mesh;
    // ...
    void update(float dt) override { /* physique + IA + animation */ }
    void render() override { /* rendu */ }
};
```

Cette approche génère plusieurs problèmes de performance :

- **Données dispersées** : chaque `Enemy` est alloué séparément sur le heap → cache misses systématiques lors du parcours.  
- **vtable overhead** : chaque appel `update()` est un appel virtuel indirect → branch misprediction potentielle, inlining impossible.  
- **Traitement monolithique** : `update()` touche *toutes* les données de l'entité, même si seule la physique ou seule l'IA a besoin d'être mise à jour ce frame.

### Architecture ECS

L'ECS décompose le problème en trois concepts orthogonaux :

- **Entity** : un simple identifiant (typiquement un `uint32_t` ou `uint64_t`). Aucune donnée, aucun comportement.  
- **Component** : une structure de données pure (pas de méthodes), associée à une entité. Exemples : `Position`, `Velocity`, `Health`, `Sprite`.  
- **System** : une fonction qui opère sur toutes les entités possédant un ensemble spécifique de components.

```cpp
// Components : structures de données pures
struct Position { float x, y, z; };  
struct Velocity { float vx, vy, vz; };  
struct Health   { float current, max; };  
struct Sprite   { std::uint32_t texture_id; float scale; };  

// Entity : simple identifiant
using Entity = std::uint32_t;
```

Les components du même type sont stockés dans des **tableaux contigus** — exactement le pattern SoA :

```cpp
// Stockage SoA interne (simplifié)
struct World {
    // Chaque type de component dans son propre tableau dense
    std::vector<Position> positions;
    std::vector<Velocity> velocities;
    std::vector<Health>   healths;
    std::vector<Sprite>   sprites;
    
    // Bitmask ou sparse set pour savoir quelles entités ont quel component
    // (implémentation réelle plus sophistiquée)
};
```

Les *systems* sont des boucles serrées qui itèrent sur les tableaux de components :

```cpp
// System : physique — opère sur Position + Velocity
void physics_system(World& world, float dt) {
    // Itère uniquement sur les entités ayant Position ET Velocity
    for (std::size_t i = 0; i < world.positions.size(); ++i) {
        world.positions[i].x += world.velocities[i].vx * dt;
        world.positions[i].y += world.velocities[i].vy * dt;
        world.positions[i].z += world.velocities[i].vz * dt;
    }
}

// System : rendu — opère sur Position + Sprite
void render_system(const World& world) {
    for (std::size_t i = 0; i < world.positions.size(); ++i) {
        draw_sprite(world.positions[i], world.sprites[i]);
    }
}

// System : dégâts — opère uniquement sur Health
void damage_system(World& world) {
    for (std::size_t i = 0; i < world.healths.size(); ++i) {
        if (world.healths[i].current <= 0.0f)
            mark_for_removal(i);
    }
}
```

### Pourquoi l'ECS est performant

L'ECS tire sa performance de la convergence de plusieurs facteurs :

**Localité mémoire maximale.** Chaque system itère sur des tableaux contigus de petites structures. Le `physics_system` ne touche que `positions` et `velocities` — deux tableaux de floats compacts. Les données de `Health`, `Sprite` et tout le reste ne polluent pas le cache.

**Vectorisation naturelle.** Les tableaux homogènes de `Position` et `Velocity` sont directement vectorisables en SIMD. Le compilateur peut traiter 4 ou 8 entités par instruction AVX.

**Parallélisation triviale.** Les systems qui opèrent sur des ensembles de components disjoints peuvent s'exécuter en parallèle sans synchronisation. `physics_system` (Position + Velocity) et `damage_system` (Health) n'ont aucune donnée en commun.

**Pas de vtable.** Les systems sont des fonctions libres, pas des méthodes virtuelles. Le compilateur peut les inliner et les optimiser sans obstacle.

### Frameworks ECS en C++

Le pattern ECS a donné naissance à des frameworks matures :

| Framework | Caractéristiques |
|-----------|-----------------|
| **EnTT** | Header-only, léger, très utilisé dans l'indie et le middleware. Archetype-based depuis v3. |
| **flecs** | C/C++, excellent support du multi-threading, requêtes déclaratives. Utilisé dans la simulation et le gaming. |
| **Unity DOTS** | ECS de Unity (C#), mais illustre le même pattern DOD. A popularisé l'approche auprès d'un large public. |

Exemple avec EnTT :

```cpp
#include <entt/entt.hpp>

struct Position { float x, y, z; };  
struct Velocity { float vx, vy, vz; };  

int main() {
    entt::registry registry;
    
    // Création de 100 000 entités avec Position + Velocity
    for (int i = 0; i < 100'000; ++i) {
        auto entity = registry.create();
        registry.emplace<Position>(entity, 0.f, 0.f, 0.f);
        registry.emplace<Velocity>(entity, 1.f, 0.f, 0.f);
    }
    
    float dt = 0.016f;
    
    // System : itère sur toutes les entités ayant Position ET Velocity
    // EnTT stocke les components en layout SoA/archetype → cache-friendly
    auto view = registry.view<Position, Velocity>();
    view.each([dt](Position& pos, const Velocity& vel) {
        pos.x += vel.vx * dt;
        pos.y += vel.vy * dt;
        pos.z += vel.vz * dt;
    });
}
```

EnTT organise les components en ***archétypes*** : toutes les entités ayant le même ensemble de components sont regroupées dans des tableaux contigus. Cela garantit un parcours cache-friendly sans nécessiter que le développeur gère manuellement le layout SoA.

## ECS et DOD au-delà du jeu vidéo

Le pattern DOD n'est pas réservé aux moteurs de jeu. Partout où l'on traite de grands ensembles d'entités homogènes avec des boucles de transformation, le DOD apporte des gains substantiels :

**Serveurs réseau haute performance.** Les connexions TCP sont des « entités » avec des components (`SocketState`, `ReadBuffer`, `WriteBuffer`, `TLSContext`). Un system `poll_system` itère sur les `SocketState` pour vérifier l'activité, un `write_system` itère sur les `WriteBuffer` pour flush les données. Le traitement batch d'un tableau contigu de sockets est nettement plus rapide que l'approche classique un-objet-par-connexion.

**Calcul scientifique et simulation.** Les simulations de particules, de dynamique moléculaire, de fluides (SPH), et de corps rigides manipulent des millions d'entités avec des propriétés séparées (position, vélocité, force, masse, charge). Le layout SoA est le standard dans ces domaines depuis des décennies — bien avant que le terme « data-oriented design » ne soit populaire.

**Bases de données en mémoire.** Les bases de données colonnes (*columnar databases* : ClickHouse, DuckDB, Apache Arrow) appliquent exactement le même principe : stocker chaque colonne dans un tableau contigu plutôt que chaque ligne dans un enregistrement. Le parallèle avec SoA vs AoS est direct.

**Traitement de données et ETL.** Les pipelines de transformation de données gagnent à organiser les enregistrements en SoA pour les passes de filtrage, d'agrégation et de projection, où chaque passe ne touche qu'un sous-ensemble des champs.

---

## Quand le DOD n'est PAS la bonne approche

Le DOD a un coût en lisibilité et en complexité de code. Il ne se justifie pas dans tous les contextes :

**Lorsque le nombre d'entités est faible.** Pour quelques dizaines ou centaines d'objets, la différence de performance entre AoS et SoA est négligeable. La clarté du code orienté objet l'emporte.

**Lorsque les accès sont centrés sur une entité.** Si le pattern d'accès dominant est « charger toutes les propriétés de l'entité N et les traiter ensemble » (par exemple, sérialisation JSON d'un objet complet), le AoS est plus adapté car toutes les données sont contiguës par entité.

**Lorsque les types de données sont hétérogènes et polymorphiques.** Le DOD excelle avec des données homogènes (10 000 particules identiques). Il est moins naturel pour des hiérarchies de types très variées avec des comportements différents par entité.

**Pour le code non critique en performance.** Le code de configuration, d'initialisation, de parsing, d'interface utilisateur — tout ce qui ne se trouve pas dans un hotspot identifié par le profiler — n'a pas besoin d'être restructuré en SoA.

La règle reste la même : **profiler d'abord**, restructurer ensuite.

## Guide de transformation AoS → SoA

Pour les cas où le profiling confirme qu'une boucle est *bandwidth-bound* ou souffre de cache misses excessifs, voici la démarche de transformation :

### Étape 1 : identifier les boucles critiques

Utiliser `perf record` + flamegraph pour localiser les hotspots, puis `perf stat` pour vérifier le taux de cache misses.

### Étape 2 : analyser les accès par boucle

Pour chaque boucle critique, lister les champs effectivement lus et écrits. Si la boucle n'accède qu'à un sous-ensemble des champs de la structure, le SoA est profitable.

### Étape 3 : séparer les champs par affinité d'accès

Regrouper les champs qui sont toujours accédés ensemble dans le même tableau (ou la même sous-structure). Les champs accédés par des boucles différentes vont dans des tableaux séparés.

```cpp
// Avant (AoS) :
struct Entity {
    float x, y, z;       // accédés par physics + render
    float vx, vy, vz;    // accédés par physics uniquement
    float r, g, b, a;    // accédés par render uniquement
    int health;           // accédé par combat uniquement
};

// Après (SoA groupé par affinité) :
struct EntityStore {
    // Groupe "physics" : position + vélocité
    struct { std::vector<float> x, y, z; }  pos;
    struct { std::vector<float> vx, vy, vz; } vel;
    
    // Groupe "render" : couleur
    struct { std::vector<float> r, g, b, a; } color;
    
    // Groupe "combat"
    std::vector<int> health;
    
    std::size_t count;
};
```

### Étape 4 : adapter les boucles

Remplacer les itérations sur objets par des itérations indexées sur les tableaux :

```cpp
// Avant :
for (auto& e : entities) {
    e.x += e.vx * dt;
}

// Après :
for (std::size_t i = 0; i < store.count; ++i) {
    store.pos.x[i] += store.vel.vx[i] * dt;
}
```

### Étape 5 : mesurer le gain

Comparer les performances avant/après avec Google Benchmark et `perf stat`. Vérifier que les cache misses ont effectivement diminué et que la boucle est désormais *bandwidth-bound* plutôt que *latency-bound*.

## Cohérence entre index et gestion des suppressions

Un défi pratique du SoA est la **cohérence des index** : si l'entité `i` est supprimée, il faut supprimer l'élément `i` de *tous* les tableaux simultanément. Les approches courantes :

### Swap-and-pop

L'élément à supprimer est échangé avec le dernier élément du tableau, puis le tableau est réduit d'un élément. C'est O(1) mais change l'ordre des entités.

```cpp
void remove_entity(EntityStore& store, std::size_t idx) {
    std::size_t last = store.count - 1;
    if (idx != last) {
        // Swap avec le dernier dans CHAQUE tableau
        std::swap(store.pos.x[idx], store.pos.x[last]);
        std::swap(store.pos.y[idx], store.pos.y[last]);
        std::swap(store.pos.z[idx], store.pos.z[last]);
        std::swap(store.vel.vx[idx], store.vel.vx[last]);
        // ... tous les autres tableaux
    }
    store.count--;
}
```

### Génération d'index (tombstones)

L'élément supprimé est marqué comme invalide (via un champ `generation` ou un flag), et les boucles le sautent. L'avantage est de préserver la stabilité des index. L'inconvénient est la fragmentation progressive — il faut compacter périodiquement.

### Index indirect (sparse set)

Un tableau d'indirection mappe les identifiants stables vers les positions dans les tableaux denses. C'est l'approche d'EnTT et de nombreux frameworks ECS. Elle offre un bon compromis entre stabilité des identifiants et densité du stockage.

## Résumé

| Aspect | AoS (POO classique) | SoA (DOD) | AoSoA (hybride) |
|--------|---------------------|-----------|-----------------|
| **Localité par entité** | Excellente | Mauvaise | Bonne (par bloc) |
| **Localité par propriété** | Mauvaise | Excellente | Excellente (dans le bloc) |
| **Vectorisation SIMD** | Difficile | Naturelle | Naturelle |
| **Bande passante utile** | Faible (champs inutiles chargés) | Maximale | Maximale |
| **Lisibilité** | Naturelle (POO) | Moins intuitive | Intermédiaire |
| **Ajout de propriétés** | Trivial (nouveau champ) | Nouveau tableau + sync | Nouveau champ dans le bloc |
| **Cas d'usage idéal** | Peu d'entités, accès par entité | Beaucoup d'entités, accès par propriété | Boucles SIMD intensives |

Le DOD n'est pas un dogme — c'est un outil. Utilisé là où le profiling le justifie, il transforme la performance des boucles critiques. Appliqué aveuglément, il complique le code sans bénéfice mesurable. La prochaine section (41.2) explore un autre levier d'optimisation : la prédiction de branchement et les techniques pour réduire les *mispredictions*.
