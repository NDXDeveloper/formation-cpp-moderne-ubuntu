# 41.3.2 — Auto-vectorisation du compilateur

> **Chapitre 41 : Optimisation CPU et Mémoire** · Section 41.3 · Sous-section 2  
> **Niveau** : Expert · **Prérequis** : Sections 41.3 et 41.3.1 (SIMD, intrinsics), Section 41.1.3 (data-oriented design)

---

## Introduction

La sous-section 41.3.1 a présenté les intrinsics — l'artillerie lourde du SIMD. Mais dans la grande majorité des cas, **le compilateur peut vectoriser automatiquement** vos boucles si le code est structuré correctement. L'auto-vectorisation est la voie recommandée : le code reste portable, lisible, maintenable, et le compilateur peut ré-optimiser pour de nouvelles architectures sans modification du source.

Le problème, c'est que l'auto-vectorisation est **fragile**. Un détail apparemment anodin — un pointeur qui *pourrait* aliaser un autre, un appel de fonction non-inline, un type entier signé dans un compteur de boucle — peut convaincre le compilateur d'abandonner la vectorisation. Le résultat : une boucle scalaire 4 à 8 fois plus lente, sans aucun warning par défaut.

Cette sous-section vous apprend à **écrire du C++ que le compilateur vectorise systématiquement** — et à diagnostiquer les cas où il échoue.

---

## Le workflow de l'auto-vectorisation

L'approche se résume en quatre étapes itératives :

```
  ┌──────────────────────────────┐
  │ 1. Écrire la boucle          │
  │    (code C++ naturel)        │
  └──────────────┬───────────────┘
                 │
  ┌──────────────▼───────────────┐
  │ 2. Compiler avec rapports    │
  │    -Rpass=loop-vectorize     │
  │    -Rpass-missed=...         │
  └──────────────┬───────────────┘
                 │
          ┌──────┴──────┐
          │             │
       Vectorisé    Pas vectorisé
          │             │
          │    ┌────────▼────────┐
          │    │ 3. Lire la      │
          │    │ raison du refus │
          │    └────────┬────────┘
          │             │
          │    ┌────────▼─────────┐
          │    │ 4. Restructurer  │
          │    │ le code / ajouter│
          │    │ des annotations  │
          │    └────────┬─────────┘
          │             │
          └──────┬──────┘
                 │
  ┌──────────────▼───────────────┐
  │ 5. Vérifier le speedup       │
  │    (perf stat, benchmark)    │
  └──────────────────────────────┘
```

L'étape 2 est la plus importante : **toujours activer les rapports de vectorisation** pendant le développement des boucles critiques. Sans eux, on travaille à l'aveugle.

## Activer les rapports de vectorisation

### Clang

```bash
clang++ -O2 -march=native \
    -Rpass=loop-vectorize \
    -Rpass-missed=loop-vectorize \
    -Rpass-analysis=loop-vectorize \
    -c fichier.cpp
```

| Flag | Rôle |
|------|------|
| `-Rpass=loop-vectorize` | Signale les boucles **vectorisées avec succès** |
| `-Rpass-missed=loop-vectorize` | Signale les boucles **non vectorisées** avec la raison |
| `-Rpass-analysis=loop-vectorize` | Détails supplémentaires sur les décisions (coûts, alias) |

Exemple de sortie réussie :

```
fichier.cpp:15:5: remark: vectorized loop (vectorization width: 8, 
    interleaved count: 2) [-Rpass=loop-vectorize]
```

Exemple d'échec :

```
fichier.cpp:28:5: remark: loop not vectorized: cannot identify array bounds
    [-Rpass-missed=loop-vectorize]
```

### GCC

```bash
g++ -O2 -march=native \
    -fopt-info-vec-optimized \
    -fopt-info-vec-missed \
    -fopt-info-vec-note \
    -c fichier.cpp
```

| Flag | Rôle |
|------|------|
| `-fopt-info-vec-optimized` | Boucles vectorisées |
| `-fopt-info-vec-missed` | Boucles non vectorisées avec raison |
| `-fopt-info-vec-note` | Informations supplémentaires |
| `-fopt-info-vec-all` | Tout (verbeux) |

Exemple de sortie GCC :

```
fichier.cpp:15:21: optimized: loop vectorized using 32 byte vectors  
fichier.cpp:28:21: missed: couldn't vectorize loop  
fichier.cpp:28:21: missed: not vectorized: complicated access pattern.  
```

### Astuce CMake

Pour activer les rapports systématiquement dans un projet CMake pendant le développement :

```cmake
# Dans CMakeLists.txt (build type Debug ou RelWithDebInfo)
if(CMAKE_CXX_COMPILER_ID MATCHES "Clang")
    target_compile_options(my_target PRIVATE
        -Rpass=loop-vectorize
        -Rpass-missed=loop-vectorize
    )
elseif(CMAKE_CXX_COMPILER_ID MATCHES "GNU")
    target_compile_options(my_target PRIVATE
        -fopt-info-vec-missed
    )
endif()
```

## Les 7 raisons principales d'échec de vectorisation

Lorsque le compilateur refuse de vectoriser une boucle, il en donne la raison. Voici les 7 causes les plus fréquentes, par ordre de prévalence, avec le diagnostic et la correction pour chacune.

---

### Raison 1 : aliasing de pointeurs

**Le problème** : le compilateur ne peut pas prouver que deux pointeurs ne désignent pas la même mémoire (ou des zones qui se chevauchent). Si `a` et `c` pointent vers la même zone, écrire dans `c[i]` pourrait modifier `a[i+1]`, ce qui introduirait une dépendance entre itérations.

```cpp
// Le compilateur ne sait pas si a et c se chevauchent
void add_arrays(const float* a, const float* b, float* c, std::size_t n) {
    for (std::size_t i = 0; i < n; ++i)
        c[i] = a[i] + b[i];   // c pourrait aliaser a ou b → pas vectorisable ?
}
```

Message typique :

```
remark: loop not vectorized: cannot identify array bounds [-Rpass-missed]
```
```
missed: not vectorized: possible dependence between data-refs
```

**Solution 1 : le mot-clé `__restrict`**

Le qualificateur `__restrict` (extension GCC/Clang/MSVC, empruntée au `restrict` de C99) promet au compilateur que le pointeur ne partage pas de mémoire avec les autres paramètres :

```cpp
void add_arrays(const float* __restrict a,
                const float* __restrict b,
                float* __restrict c,
                std::size_t n) {
    for (std::size_t i = 0; i < n; ++i)
        c[i] = a[i] + b[i];   // ✅ Le compilateur sait qu'il n'y a pas d'aliasing
}
```

> ⚠️ `__restrict` est une **promesse du développeur**, pas une vérification du compilateur. Si les pointeurs se chevauchent réellement, le comportement est indéfini. C'est l'équivalent de « je sais ce que je fais » — à utiliser avec la certitude que les buffers sont distincts.

**Solution 2 : utiliser `std::vector` et des références**

Le compilateur sait que deux `std::vector` distincts ne se chevauchent pas :

```cpp
void add_arrays(const std::vector<float>& a,
                const std::vector<float>& b,
                std::vector<float>& c) {
    for (std::size_t i = 0; i < a.size(); ++i)
        c[i] = a[i] + b[i];   // ✅ Trois vectors distincts → pas d'aliasing
}
```

**Solution 3 : copier dans des variables locales**

```cpp
void process(float* data, float* output, std::size_t n) {
    // Copier les pointeurs en locaux pour signaler l'absence d'alias
    const float* __restrict in = data;
    float* __restrict out = output;
    for (std::size_t i = 0; i < n; ++i)
        out[i] = in[i] * 2.0f;
}
```

---

### Raison 2 : dépendances entre itérations

**Le problème** : le résultat de l'itération `i` dépend du résultat de l'itération `i-1` (ou d'une itération antérieure). Les itérations ne peuvent pas être exécutées simultanément.

```cpp
// Dépendance portée par la boucle (loop-carried dependency)
for (std::size_t i = 1; i < n; ++i)
    a[i] = a[i-1] * 2.0f;    // ❌ a[i] dépend de a[i-1]
```

Message typique :

```
missed: not vectorized: unsafe dependent memory operations in loop body
```

**Solutions possibles :**

Certaines dépendances peuvent être **reformulées**. Par exemple, la récurrence `a[i] = a[i-1] * 2` est une progression géométrique calculable directement :

```cpp
// Reformulation : la valeur de a[i] = a[0] * 2^i
float base = a[0];  
for (std::size_t i = 0; i < n; ++i) {  
    a[i] = base;     // ✅ Pas de dépendance inter-itérations
    base *= 2.0f;
}
// Ou mieux : a[i] = a[0] * std::pow(2.0f, i) si la précision le permet
```

Pour les dépendances de type *réduction* (somme, produit, min, max), le compilateur sait les gérer si on lui en donne la permission :

```cpp
// Réduction : dépendance sur sum, mais le compilateur sait paralléliser
float sum = 0.0f;  
for (std::size_t i = 0; i < n; ++i)  
    sum += data[i];    // ✅ vectorisable avec -ffast-math ou #pragma omp simd reduction
```

La réduction est vectorisée en utilisant un accumulateur vectoriel (8 sommes partielles en AVX) qui est réduit horizontalement à la fin. Pour les flottants, cette transformation change l'ordre des additions et donc le résultat (à cause des erreurs d'arrondi). Le compilateur ne le fait donc qu'avec `-ffast-math` ou un pragma explicite.

---

### Raison 3 : appels de fonctions dans la boucle

**Le problème** : un appel de fonction non-inline est une boîte noire pour le compilateur. Il ne sait pas si la fonction modifie de la mémoire globale, lance des exceptions, ou contient des effets de bord.

```cpp
float transform(float x);  // déclarée dans un autre fichier .cpp

void apply(float* data, std::size_t n) {
    for (std::size_t i = 0; i < n; ++i)
        data[i] = transform(data[i]);    // ❌ appel non-inline → pas vectorisable
}
```

Message typique :

```
missed: not vectorized: call instruction cannot be vectorized
```

**Solutions :**

- **Inliner la fonction** : la définir dans le header ou dans le même fichier, ou utiliser LTO (section 41.5) pour que le compilateur voie la définition à travers les unités de compilation.

```cpp
// Définition visible → le compilateur peut inliner et vectoriser
inline float transform(float x) {
    return x * x + 1.0f;
}
```

- **Utiliser les fonctions mathématiques vectorisées** : GCC et Clang reconnaissent les fonctions de `<cmath>` et les remplacent par des versions SIMD quand c'est possible (avec `-ffast-math` ou `-mveclibabi`).

```cpp
#include <cmath>

void apply_sqrt(float* data, std::size_t n) {
    for (std::size_t i = 0; i < n; ++i)
        data[i] = std::sqrt(data[i]);
    // ✅ Le compilateur remplace par vsqrtps (AVX) avec -O2 -march=native
}
```

Les fonctions reconnues automatiquement incluent : `sqrt`, `sin`, `cos`, `exp`, `log`, `pow`, `fabs`, `fmin`, `fmax`, `floor`, `ceil`, `round`.

- **Marquer la fonction comme `[[gnu::pure]]` ou `[[gnu::const]]`** si elle n'a pas d'effets de bord :

```cpp
[[gnu::const]]    // promet : pas d'effets de bord, résultat dépend uniquement des arguments
float transform(float x);
// Le compilateur peut maintenant vectoriser les appels
```

---

### Raison 4 : branchements dans le corps de boucle

**Le problème** : un `if` / `else` dans la boucle empêche le compilateur d'appliquer la même instruction à toutes les lanes SIMD.

```cpp
for (std::size_t i = 0; i < n; ++i) {
    if (data[i] > threshold)
        output[i] = data[i] * 2.0f;
    else
        output[i] = 0.0f;
}
```

**En réalité**, les compilateurs modernes gèrent souvent ce cas en utilisant des masques SIMD (*predication*) : ils calculent les deux branches et fusionnent les résultats. Mais cela ne fonctionne que si les deux branches sont **simples** (pas d'appels de fonctions, pas d'effets de bord différents).

Si le compilateur refuse, on peut reformuler en opération branchless (section 41.2) :

```cpp
for (std::size_t i = 0; i < n; ++i) {
    // Opérateur ternaire — le compilateur le transforme en blend SIMD
    output[i] = (data[i] > threshold) ? data[i] * 2.0f : 0.0f;
}
```

Ou utiliser la technique du masque arithmétique :

```cpp
for (std::size_t i = 0; i < n; ++i) {
    float mask = (data[i] > threshold) ? 1.0f : 0.0f;
    output[i] = data[i] * 2.0f * mask;
}
```

---

### Raison 5 : compteur de boucle ou bornes non analysables

**Le problème** : le compilateur doit connaître (ou borner) le nombre d'itérations pour générer le prélude, le corps vectorisé, et l'épilogue. Si les bornes sont incertaines, il abandonne.

```cpp
// ❌ Borne inconnue : le compilateur ne peut pas analyser la boucle
void process(float* data, bool* flags, std::size_t n) {
    for (std::size_t i = 0; i < n && flags[i]; ++i)   // sortie anticipée
        data[i] *= 2.0f;
}
```

**Solutions :**

- Séparer la condition de sortie anticipée du compteur de boucle :

```cpp
// ✅ Boucle comptée + test de sortie séparé
std::size_t limit = n;  
for (std::size_t i = 0; i < n; ++i) {  
    if (!flags[i]) { limit = i; break; }
}
// Maintenant on peut vectoriser la boucle de 0 à limit
for (std::size_t i = 0; i < limit; ++i)
    data[i] *= 2.0f;   // ✅ Boucle simple, bornes connues
```

- Utiliser un compteur de type `std::size_t` ou `int` (signé ou non), pas un itérateur complexe dont le compilateur ne peut pas analyser le pas.

---

### Raison 6 : stride non unitaire ou accès non contigu

**Le problème** : le SIMD charge des blocs de mémoire consécutifs. Si la boucle accède aux éléments avec un pas > 1, le compilateur doit émettre des opérations de *gather* (lentes) ou renoncer.

```cpp
// AoS : stride de sizeof(Particle) entre deux positions x consécutives
struct Particle { float x, y, z, mass; };  
std::vector<Particle> particles(N);  

// ❌ Stride de 16 octets entre deux x consécutifs
for (auto& p : particles)
    p.x += p.y * dt;
```

**Solution** : transformer en SoA (section 41.1.3) pour obtenir des accès continus :

```cpp
// ✅ SoA : x[] est un tableau contigu de floats
struct ParticleSystem {
    std::vector<float> x, y, z, mass;
};

for (std::size_t i = 0; i < ps.count; ++i)
    ps.x[i] += ps.y[i] * dt;   // ✅ Accès séquentiel → vectorisation naturelle
```

C'est le lien direct entre le data-oriented design (section 41.1.3) et la vectorisation. Le SoA n'est pas qu'une question de cache — c'est aussi la condition nécessaire pour que le compilateur vectorise les boucles de traitement.

---

### Raison 7 : types non supportés ou taille exotique

**Le problème** : certains types ne sont pas directement supportés en SIMD, ou le compilateur ne dispose pas d'instructions efficaces pour eux.

```cpp
// ❌ long double : pas de support SIMD (80/128-bit selon la plateforme)
for (std::size_t i = 0; i < n; ++i)
    result[i] = static_cast<long double>(a[i]) + b[i];

// ❌ Opérations sur des types de taille non standard
struct RGB { uint8_t r, g, b; };  // 3 octets — non aligné sur une puissance de 2
```

**Solutions :**

- Utiliser des types standards (`float`, `double`, `int32_t`, `int16_t`, `int8_t`).  
- Éviter `long double` dans les boucles critiques — utiliser `double` à la place.  
- Pour les structures de petite taille (comme `RGB`), ajouter un padding pour atteindre une puissance de 2 (`RGBA` avec 4 octets), ou passer en SoA.

## Annotations et pragmas pour guider le compilateur

Quand la restructuration du code ne suffit pas, des annotations explicites peuvent débloquer la vectorisation.

### `#pragma omp simd` (OpenMP 4.0+)

Le pragma le plus portable et le plus puissant. Il **ordonne** au compilateur de vectoriser la boucle, même s'il a des doutes sur l'aliasing ou les dépendances :

```cpp
#pragma omp simd
for (std::size_t i = 0; i < n; ++i)
    c[i] = a[i] + b[i];
```

Variantes courantes :

```cpp
// Indiquer une réduction (nécessaire pour les accumulateurs)
float sum = 0.0f;
#pragma omp simd reduction(+:sum)
for (std::size_t i = 0; i < n; ++i)
    sum += data[i];

// Spécifier l'alignement des pointeurs
#pragma omp simd aligned(a, b, c : 32)
for (std::size_t i = 0; i < n; ++i)
    c[i] = a[i] + b[i];

// Combiné
#pragma omp simd reduction(+:sum) simdlen(8)
for (std::size_t i = 0; i < n; ++i)
    sum += data[i];
```

Activation :

```bash
# GCC : -fopenmp-simd active les pragmas SIMD sans activer le runtime OpenMP complet
g++ -O2 -march=native -fopenmp-simd -c fichier.cpp

# Clang : même flag
clang++ -O2 -march=native -fopenmp-simd -c fichier.cpp
```

Le flag `-fopenmp-simd` est léger — il n'ajoute aucune dépendance au runtime OpenMP. Il active uniquement l'interprétation des pragmas `#pragma omp simd`.

> ⚠️ `#pragma omp simd` est un **ordre**, pas un conseil. Si le développeur se trompe (les itérations ne sont pas réellement indépendantes), le code vectorisé produira des résultats incorrects silencieusement. C'est l'équivalent SIMD de `__restrict` : une promesse du développeur.

### `#pragma GCC ivdep`

Spécifique à GCC (et supporté par Clang par compatibilité). Indique qu'il n'y a pas de dépendances portées par la boucle (*Ignore Vector DEPendencies*) :

```cpp
#pragma GCC ivdep
for (std::size_t i = 0; i < n; ++i)
    a[i] = b[i] + c[i];
```

Moins portable que `#pragma omp simd`, mais ne nécessite pas `-fopenmp-simd`. Utile pour les projets qui ne veulent pas de dépendance OpenMP, même minimale.

### `#pragma clang loop`

Spécifique à Clang. Permet un contrôle fin sur la vectorisation et le déroulement :

```cpp
#pragma clang loop vectorize(enable) vectorize_width(8)
for (std::size_t i = 0; i < n; ++i)
    data[i] *= 2.0f;

// Forcer le déroulement
#pragma clang loop unroll_count(4)
for (std::size_t i = 0; i < n; ++i)
    data[i] *= 2.0f;

// Désactiver la vectorisation (utile pour le debugging)
#pragma clang loop vectorize(disable)
for (std::size_t i = 0; i < n; ++i)
    data[i] *= 2.0f;
```

### `#pragma GCC optimize`

Activer `-O3` ou `-ffast-math` localement pour une seule fonction :

```cpp
#pragma GCC push_options
#pragma GCC optimize("O3,fast-math")

void hot_loop(float* __restrict data, std::size_t n) {
    for (std::size_t i = 0; i < n; ++i)
        data[i] = std::sin(data[i]);
}

#pragma GCC pop_options
// Le reste du fichier garde ses options de compilation normales
```

## `-ffast-math` : le déverrouilleur controversé

Le flag `-ffast-math` est souvent la différence entre une boucle vectorisée et une boucle scalaire pour les opérations flottantes. Il autorise le compilateur à :

- **Réassocier les additions/multiplications** : `(a + b) + c` → `a + (b + c)`. Nécessaire pour les réductions parallèles.  
- **Supposer que les NaN et infinis n'existent pas** : simplifie les comparaisons.  
- **Remplacer les divisions par des multiplications réciproques** : plus rapide mais moins précis.  
- **Utiliser des approximations pour les fonctions transcendantes** : `sin`, `cos`, `exp`, `log`.

**Impact concret sur la vectorisation :**

```cpp
// SANS -ffast-math : souvent NON vectorisée (réduction flottante)
float sum = 0.0f;  
for (std::size_t i = 0; i < n; ++i)  
    sum += data[i];

// AVEC -ffast-math : vectorisée (8 accumulateurs partiels en AVX)
// → speedup de 6-8×
```

**Les risques :**

- Les résultats numériques changent légèrement (erreurs d'arrondi différentes).  
- Le code ne détecte plus les NaN ni les infinis.  
- Certaines identités mathématiques utilisées pour la stabilité numérique ne sont plus respectées.

**Recommandation** : ne pas activer `-ffast-math` globalement. L'activer ciblé sur les fonctions où la performance prime et où la conformité IEEE stricte n'est pas requise, via `#pragma GCC optimize("fast-math")` ou l'attribut de fonction.

Pour les réductions seules, préférer le pragma OpenMP qui est plus ciblé :

```cpp
float sum = 0.0f;
#pragma omp simd reduction(+:sum)
for (std::size_t i = 0; i < n; ++i)
    sum += data[i];
// Vectorisé SANS -ffast-math global — la réassociation est limitée à cette boucle
```

## Patterns favorables à la vectorisation

Voici un récapitulatif des patterns que le compilateur vectorise le plus facilement :

### Le cas idéal

```cpp
// Boucle comptée, accès séquentiels, pas de dépendance, pas de branchement,
// types simples, pointeurs restrict
void ideal(const float* __restrict a, const float* __restrict b,
           float* __restrict c, std::size_t n) {
    for (std::size_t i = 0; i < n; ++i)
        c[i] = a[i] * b[i] + 1.0f;
}
// → vectorisé systématiquement avec -O2 -march=native
```

### Réductions

```cpp
float sum = 0.0f;
#pragma omp simd reduction(+:sum)
for (std::size_t i = 0; i < n; ++i)
    sum += data[i];
// → vectorisé avec -fopenmp-simd
```

### Sélection conditionnelle simple

```cpp
for (std::size_t i = 0; i < n; ++i)
    output[i] = (input[i] > 0.0f) ? input[i] : 0.0f;
// → vectorisé via masque SIMD (blend)
```

### Fonctions mathématiques standards

```cpp
for (std::size_t i = 0; i < n; ++i)
    output[i] = std::sqrt(input[i]);
// → vectorisé (vsqrtps sur AVX)
```

### Conversion de types

```cpp
// int → float
for (std::size_t i = 0; i < n; ++i)
    floats[i] = static_cast<float>(ints[i]);
// → vectorisé (vcvtdq2ps sur AVX)
```

## Patterns bloquants : checklist de diagnostic

Quand une boucle n'est pas vectorisée, passer en revue cette checklist :

| Question | Si la réponse est "oui" | Correction |
|----------|------------------------|------------|
| Les pointeurs pourraient-ils aliaser ? | Aliasing suspecté | Ajouter `__restrict` |
| Y a-t-il un appel de fonction non-inline ? | Appel opaque | Inliner, LTO, ou `[[gnu::const]]` |
| Y a-t-il une dépendance `a[i] = f(a[i-1])` ? | Dépendance portée | Reformuler l'algorithme |
| Y a-t-il un `if` avec un corps lourd ? | Branchement complexe | Réécrire en branchless ou séparer la boucle |
| Les données sont-elles non contiguës (AoS, list) ? | Stride ou scatter | Passer en SoA |
| La boucle a-t-elle une condition de sortie anticipée ? | Bornes non analysables | Séparer en deux boucles |
| Le type est-il `long double` ou une struct exotique ? | Type non supporté | Utiliser `float` / `double` / `intN_t` |
| L'accumulateur est-il flottant sans `-ffast-math` ? | Réduction non autorisée | `#pragma omp simd reduction` |
| Le pas de la boucle est-il variable ? | Stride non constant | Réécrire avec un pas fixe |
| Y a-t-il un `try`/`catch` dans la boucle ? | Exceptions | Sortir le `try` de la boucle |

## Interleaving : l'optimisation cachée

Au-delà de la vectorisation pure, les compilateurs appliquent souvent l'***interleaving*** (ou *unroll-and-jam*) : traiter 2 à 4 vecteurs SIMD par itération au lieu d'un seul. Cela masque la latence des instructions en remplissant le pipeline avec des opérations indépendantes.

```
Sans interleaving (vectorization width: 8, interleaved count: 1) :
  Itération 1 : charge v0, calcule v0, stocke v0
  Itération 2 : charge v1, calcule v1, stocke v1
  → Le pipeline attend entre charge et calcul

Avec interleaving ×2 (vectorization width: 8, interleaved count: 2) :
  Itération 1 : charge v0, charge v1, calcule v0, calcule v1, stocke v0, stocke v1
  → Pendant que v0 est en cours de calcul, v1 est chargé — pipeline mieux rempli
```

Le rapport de Clang indique le facteur d'interleaving :

```
remark: vectorized loop (vectorization width: 8, interleaved count: 2)
```

Un `interleaved count: 1` signifie que le compilateur n'a pas pu interleaver — souvent à cause de registres insuffisants ou de dépendances. Un `interleaved count: 2` ou `4` est le signe d'une vectorisation efficace.

## Exemple complet : diagnostic et correction

### Code initial

```cpp
#include <cmath>
#include <vector>

struct Point {
    float x, y, z;
};

float total_distance(const std::vector<Point>& points, Point origin) {
    float sum = 0.0f;
    for (const auto& p : points) {
        float dx = p.x - origin.x;
        float dy = p.y - origin.y;
        float dz = p.z - origin.z;
        sum += std::sqrt(dx*dx + dy*dy + dz*dz);
    }
    return sum;
}
```

### Diagnostic

```bash
$ clang++ -O2 -march=native -Rpass-missed=loop-vectorize -c example.cpp
example.cpp:10:5: remark: loop not vectorized: cannot prove it is safe to 
    reorder floating-point operations; allow reordering by specifying 
    '#pragma omp simd reduction' before the loop or by providing the 
    compiler option '-ffast-math' [-Rpass-missed=loop-vectorize]
```

Deux problèmes : la réduction flottante (`sum +=`) et le layout AoS.

### Correction progressive

**Étape 1** : autoriser la réduction

```cpp
float total_distance(const std::vector<Point>& points, Point origin) {
    float sum = 0.0f;
    #pragma omp simd reduction(+:sum)
    for (std::size_t i = 0; i < points.size(); ++i) {
        float dx = points[i].x - origin.x;
        float dy = points[i].y - origin.y;
        float dz = points[i].z - origin.z;
        sum += std::sqrt(dx*dx + dy*dy + dz*dz);
    }
    return sum;
}
```

Recompiler : le compilateur vectorise maintenant, mais avec des opérations de gather (stride de 12 octets entre deux `x` consécutifs) — sous-optimal.

**Étape 2** : transformer en SoA

```cpp
struct Points {
    std::vector<float> x, y, z;
    std::size_t count;
};

float total_distance(const Points& pts, float ox, float oy, float oz) {
    float sum = 0.0f;
    #pragma omp simd reduction(+:sum)
    for (std::size_t i = 0; i < pts.count; ++i) {
        float dx = pts.x[i] - ox;
        float dy = pts.y[i] - oy;
        float dz = pts.z[i] - oz;
        sum += std::sqrt(dx*dx + dy*dy + dz*dz);
    }
    return sum;
}
```

Recompiler :

```
remark: vectorized loop (vectorization width: 8, interleaved count: 2)
```

Vectorisé en AVX2 avec interleaving ×2. Trois flux de lecture continus (`x[]`, `y[]`, `z[]`) et des instructions FMA pour les produits-sommes.

### Résultats

| Version | Temps (1M points) | Speedup |
|---------|--------------------|---------|
| AoS, pas de pragma | 3,2 ms | 1× |
| AoS + `#pragma omp simd` | 1,8 ms | 1,8× |
| SoA + `#pragma omp simd` | 0,45 ms | 7,1× |

Le passage AoS → SoA combiné au pragma de réduction produit un speedup de **7×** — proche du maximum théorique de 8× pour AVX2 sur des floats.

## Résumé

| Principe | Action |
|----------|--------|
| **Toujours activer les rapports** | `-Rpass-missed=loop-vectorize` (Clang), `-fopt-info-vec-missed` (GCC) |
| **Éliminer l'aliasing** | `__restrict` sur les pointeurs, ou utiliser des `std::vector` distincts |
| **Supprimer les appels opaques** | Inliner, LTO, `[[gnu::const]]` |
| **Assurer la contiguïté** | Layout SoA pour les boucles multi-propriétés |
| **Gérer les réductions flottantes** | `#pragma omp simd reduction(+:var)` ou `-ffast-math` ciblé |
| **Simplifier les branchements** | Opérateur ternaire, masques arithmétiques |
| **Forcer si nécessaire** | `#pragma omp simd`, `#pragma GCC ivdep` |
| **Vérifier le résultat** | Compiler Explorer (chercher `vmulps ymm`), `perf stat` |

L'auto-vectorisation est le **rapport effort/gain le plus élevé** de tout le chapitre 41 : quelques annotations et une restructuration SoA peuvent multiplier les performances par 4–8× sur les boucles critiques, sans écrire une seule ligne d'assembleur ou d'intrinsic.
