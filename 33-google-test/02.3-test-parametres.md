# 33.2.3 TEST_P : Tests paramétrés

## Le problème de la duplication par les données

En fin de section 33.2.2, nous avons identifié un signal de duplication caractéristique : plusieurs tests partagent exactement la même logique et ne diffèrent que par les données d'entrée. Reprenons cet exemple de validation d'email et imaginons une couverture réaliste :

```cpp
// ❌ 12 tests avec le même corps — seul l'argument change
TEST_F(ValidatorTest, AcceptsSimpleEmail) {
    EXPECT_TRUE(validator.is_valid_email("user@example.com"));
}
TEST_F(ValidatorTest, AcceptsEmailWithPlus) {
    EXPECT_TRUE(validator.is_valid_email("user+tag@example.com"));
}
TEST_F(ValidatorTest, AcceptsEmailWithDots) {
    EXPECT_TRUE(validator.is_valid_email("first.last@example.com"));
}
TEST_F(ValidatorTest, RejectsEmailWithoutAt) {
    EXPECT_FALSE(validator.is_valid_email("userexample.com"));
}
TEST_F(ValidatorTest, RejectsEmailWithoutDomain) {
    EXPECT_FALSE(validator.is_valid_email("user@"));
}
// ... et ainsi de suite
```

Douze tests, douze fonctions, un seul appel qui change. C'est exactement le cas d'usage de `TEST_P` : écrire la logique du test **une seule fois** et la nourrir avec un jeu de données.

## Anatomie d'un test paramétré

Un test paramétré repose sur trois éléments :

1. **Une classe fixture** qui hérite de `::testing::TestWithParam<T>` où `T` est le type du paramètre.
2. **Un ou plusieurs `TEST_P`** qui utilisent `GetParam()` pour récupérer la valeur courante.
3. **Un appel `INSTANTIATE_TEST_SUITE_P`** qui fournit le jeu de données.

### Cas simple : un paramètre scalaire

Commençons par le cas le plus élémentaire — un paramètre de type `std::string` :

```cpp
#include <gtest/gtest.h>
#include "mon_projet/validator.hpp"

// ── 1. Fixture paramétrée ─────────────────────────────────
class ValidEmailTest : public ::testing::TestWithParam<std::string> {  
protected:  
    mp::EmailValidator validator;
};

// ── 2. Test paramétré ─────────────────────────────────────
TEST_P(ValidEmailTest, AcceptsWellFormedEmail) {
    const std::string& email = GetParam();
    EXPECT_TRUE(validator.is_valid(email)) << "Rejeté à tort : " << email;
}

// ── 3. Instanciation avec les données ─────────────────────
INSTANTIATE_TEST_SUITE_P(
    CommonEmails,                     // Préfixe d'instanciation
    ValidEmailTest,                   // Classe fixture
    ::testing::Values(                // Générateur de valeurs
        "user@example.com",
        "first.last@example.com",
        "user+tag@example.com",
        "user@subdomain.example.com",
        "user123@example.co.uk",
        "x@y.zz"
    )
);
```

GTest génère automatiquement un test distinct pour chaque valeur. La sortie console ressemble à ceci :

```
[----------] 6 tests from CommonEmails/ValidEmailTest
[ RUN      ] CommonEmails/ValidEmailTest.AcceptsWellFormedEmail/0
[       OK ] CommonEmails/ValidEmailTest.AcceptsWellFormedEmail/0 (0 ms)
[ RUN      ] CommonEmails/ValidEmailTest.AcceptsWellFormedEmail/1
[       OK ] CommonEmails/ValidEmailTest.AcceptsWellFormedEmail/1 (0 ms)
...
[ RUN      ] CommonEmails/ValidEmailTest.AcceptsWellFormedEmail/5
[       OK ] CommonEmails/ValidEmailTest.AcceptsWellFormedEmail/5 (0 ms)
```

Le format de nommage est `Préfixe/Fixture.NomDuTest/Index`. L'index numérique identifie le paramètre. Ce nommage permet le filtrage en ligne de commande :

```bash
# Tous les tests de l'instanciation CommonEmails
./mon_projet_tests --gtest_filter="CommonEmails/*"

# Un paramètre spécifique (le troisième, index 2)
./mon_projet_tests --gtest_filter="*/ValidEmailTest.AcceptsWellFormedEmail/2"
```

### Le message d'échec personnalisé

La ligne `<< "Rejeté à tort : " << email` dans le `EXPECT_TRUE` est importante. Par défaut, quand un test paramétré échoue, GTest affiche la valeur du paramètre uniquement si elle est d'un type que le framework sait afficher (types primitifs, `std::string`). Le message personnalisé via `<<` garantit que le rapport identifie clairement **quel** paramètre a causé l'échec, indépendamment du type.

## Paramètres structurés

Les tests paramétrés prennent toute leur dimension quand le paramètre est une structure qui porte à la fois l'entrée et le résultat attendu.

### Avec std::tuple

L'approche la plus directe utilise `std::tuple` :

```cpp
using EmailTestCase = std::tuple<std::string, bool>;
//                               ^input      ^expected

class EmailValidationTest : public ::testing::TestWithParam<EmailTestCase> {  
protected:  
    mp::EmailValidator validator;
};

TEST_P(EmailValidationTest, ValidatesCorrectly) {
    const auto& [email, expected] = GetParam();  // Structured binding (C++17)
    EXPECT_EQ(validator.is_valid(email), expected)
        << "Email: " << email
        << " | Attendu: " << std::boolalpha << expected;
}

INSTANTIATE_TEST_SUITE_P(
    EmailCases,
    EmailValidationTest,
    ::testing::Values(
        // Cas valides
        EmailTestCase{"user@example.com",            true},
        EmailTestCase{"first.last@example.com",      true},
        EmailTestCase{"user+tag@example.com",        true},
        EmailTestCase{"user@sub.domain.example.com", true},
        // Cas invalides
        EmailTestCase{"",                            false},
        EmailTestCase{"userexample.com",             false},
        EmailTestCase{"user@",                       false},
        EmailTestCase{"@example.com",                false},
        EmailTestCase{"user @example.com",           false},
        EmailTestCase{"user@example",                false}
    )
);
```

Les structured bindings (section 12.1) rendent l'extraction propre. Le tuple rassemble l'entrée et la sortie attendue en une seule ligne lisible.

### Avec une struct nommée

Quand le paramètre dépasse deux ou trois champs, un tuple devient illisible. Une struct nommée est préférable :

```cpp
struct ConversionTestCase {
    double celsius;
    double expected_fahrenheit;
    std::string description;  // Pour les messages d'échec
};

// Surcharge de operator<< pour l'affichage dans les rapports
std::ostream& operator<<(std::ostream& os, const ConversionTestCase& tc) {
    return os << tc.description 
              << " (" << tc.celsius << "°C -> " 
              << tc.expected_fahrenheit << "°F)";
}

class TemperatureConversionTest 
    : public ::testing::TestWithParam<ConversionTestCase> {
};

TEST_P(TemperatureConversionTest, ConvertsCelsiusToFahrenheit) {
    const auto& tc = GetParam();
    mp::Temperature temp{tc.celsius};
    EXPECT_NEAR(temp.fahrenheit(), tc.expected_fahrenheit, 0.01)
        << "Cas : " << tc.description;
}

INSTANTIATE_TEST_SUITE_P(
    StandardConversions,
    TemperatureConversionTest,
    ::testing::Values(
        ConversionTestCase{0.0,    32.0,   "Point de congélation"},
        ConversionTestCase{100.0,  212.0,  "Point d'ébullition"},
        ConversionTestCase{-40.0,  -40.0,  "Point de croisement"},
        ConversionTestCase{37.0,   98.6,   "Température corporelle"},
        ConversionTestCase{-273.15, -459.67, "Zéro absolu"}
    )
);
```

La surcharge de `operator<<` est un détail qui fait la différence en pratique. Sans elle, GTest affiche la représentation brute de la struct (souvent un dump hexadécimal inutile). Avec elle, chaque test paramétré s'identifie clairement dans la sortie console et les rapports XML.

### Noms personnalisés avec un name generator

Par défaut, les tests paramétrés sont numérotés `/0`, `/1`, `/2`. Pour des rapports encore plus lisibles, GTest permet de fournir un **name generator** qui transforme le paramètre en nom de test :

```cpp
struct EmailNameGenerator {
    std::string operator()(
        const ::testing::TestParamInfo<EmailTestCase>& info) const 
    {
        const auto& [email, expected] = info.param;
        // Le nom doit être un identifiant C++ valide : pas d'@, de ., etc.
        std::string name = email;
        std::ranges::replace_if(name, 
            [](char c) { return !std::isalnum(c); }, '_');
        return name;
    }
};

INSTANTIATE_TEST_SUITE_P(
    EmailCases,
    EmailValidationTest,
    ::testing::Values(
        EmailTestCase{"user@example.com",       true},
        EmailTestCase{"userexample.com",        false},
        EmailTestCase{"@example.com",           false}
    ),
    EmailNameGenerator{}   // 4e argument optionnel
);
```

La sortie devient alors :

```
[ RUN      ] EmailCases/EmailValidationTest.ValidatesCorrectly/user_example_com
[ RUN      ] EmailCases/EmailValidationTest.ValidatesCorrectly/userexample_com
[ RUN      ] EmailCases/EmailValidationTest.ValidatesCorrectly/_example_com
```

Le nom doit être un identifiant C++ valide — d'où le remplacement des caractères spéciaux. Ce mécanisme est particulièrement utile dans les rapports CI où un nom descriptif évite de devoir consulter le code source pour comprendre quel cas a échoué.

## Les générateurs de valeurs

`::testing::Values(...)` est le générateur le plus courant, mais GTest en propose plusieurs autres qui couvrent des cas d'usage distincts.

### Values : liste explicite

```cpp
::testing::Values(1, 2, 3, 5, 8, 13)
```

Usage : liste finie et connue de cas de test. C'est le générateur par défaut.

### ValuesIn : depuis un conteneur

```cpp
const std::vector<std::string> forbidden_words = 
    mp::load_forbidden_words("test_data/blacklist.txt");

INSTANTIATE_TEST_SUITE_P(
    ForbiddenWords,
    ContentFilterTest,
    ::testing::ValuesIn(forbidden_words)
);
```

Usage : les données de test proviennent d'un fichier, d'une base de données ou d'un conteneur construit dynamiquement. Particulièrement utile pour les tests data-driven où le jeu de données évolue indépendamment du code de test.

### Range : plage numérique

```cpp
// Génère 0, 1, 2, ..., 255
::testing::Range(0, 256)

// Génère 0.0, 0.5, 1.0, 1.5, ..., 9.5
::testing::Range(0.0, 10.0, 0.5)
```

Usage : tester un comportement sur une plage continue. Le troisième argument est le pas (step), qui vaut 1 par défaut.

### Bool : true et false

```cpp
::testing::Bool()
// Équivalent à Values(false, true)
```

Usage : tester un comportement avec un flag activé/désactivé.

### Combine : produit cartésien

```cpp
INSTANTIATE_TEST_SUITE_P(
    AllConfigurations,
    CompressionTest,
    ::testing::Combine(
        ::testing::Values("gzip", "lz4", "zstd"),  // Algorithme
        ::testing::Values(1, 5, 9),                  // Niveau
        ::testing::Bool()                             // Avec checksum
    )
);
```

`Combine` génère le **produit cartésien** de tous les générateurs. Dans cet exemple, 3 × 3 × 2 = 18 combinaisons, soit 18 tests générés automatiquement. Le paramètre est un `std::tuple` dont les types correspondent aux générateurs :

```cpp
class CompressionTest 
    : public ::testing::TestWithParam<
          std::tuple<std::string, int, bool>> {
protected:
    void SetUp() override {
        const auto& [algo, level, checksum] = GetParam();
        compressor.configure(algo, level, checksum);
    }
    mp::Compressor compressor;
};

TEST_P(CompressionTest, RoundTripPreservesData) {
    std::string original = "Test data for compression round-trip";
    auto compressed = compressor.compress(original);
    auto decompressed = compressor.decompress(compressed);
    EXPECT_EQ(decompressed, original);
}
```

`Combine` est un outil extrêmement puissant pour l'exploration combinatoire, mais il faut surveiller l'explosion du nombre de tests. 4 générateurs de 5 valeurs chacun produisent 625 tests — chacun impliquant une compilation, une exécution et une ligne dans le rapport. Si les tests sont lents, le temps de la suite peut exploser.

## Instanciations multiples

Un même `TEST_P` peut être instancié plusieurs fois avec des jeux de données différents. Chaque `INSTANTIATE_TEST_SUITE_P` produit un groupe de tests distinct, identifié par son préfixe :

```cpp
// Même fixture, même TEST_P, mais deux jeux de données
INSTANTIATE_TEST_SUITE_P(
    AsciiEmails,
    EmailValidationTest,
    ::testing::Values(
        EmailTestCase{"user@example.com",    true},
        EmailTestCase{"admin@localhost",      true}
    )
);

INSTANTIATE_TEST_SUITE_P(
    InternationalEmails,
    EmailValidationTest,
    ::testing::Values(
        EmailTestCase{"用户@例え.jp",         true},
        EmailTestCase{"user@münchen.de",     true}
    )
);
```

La sortie distingue clairement les deux groupes :

```
[----------] 2 tests from AsciiEmails/EmailValidationTest
[----------] 2 tests from InternationalEmails/EmailValidationTest
```

Ce mécanisme est utile pour organiser les données par catégorie sémantique : cas nominaux, cas limites, données internationalisées, cas issus de bugs reportés (regression tests).

## Fixture paramétrée avec setup

`TestWithParam<T>` hérite de `::testing::Test`, ce qui signifie que toutes les fonctionnalités des fixtures classiques restent disponibles : `SetUp()`, `TearDown()`, membres protégés et méthodes helper :

```cpp
struct HttpTestCase {
    std::string method;
    std::string path;
    int expected_status;
};

class HttpRouterTest : public ::testing::TestWithParam<HttpTestCase> {  
protected:  
    void SetUp() override {
        router.add_route("GET",  "/users",    handler_list);
        router.add_route("POST", "/users",    handler_create);
        router.add_route("GET",  "/users/:id", handler_detail);
    }

    mp::Router router;
    mp::Handler handler_list   = [](auto&) { return mp::Response{200}; };
    mp::Handler handler_create = [](auto&) { return mp::Response{201}; };
    mp::Handler handler_detail = [](auto&) { return mp::Response{200}; };
};

TEST_P(HttpRouterTest, RoutesToCorrectHandler) {
    const auto& tc = GetParam();
    auto response = router.dispatch(tc.method, tc.path);
    EXPECT_EQ(response.status(), tc.expected_status)
        << tc.method << " " << tc.path;
}

INSTANTIATE_TEST_SUITE_P(
    DefinedRoutes,
    HttpRouterTest,
    ::testing::Values(
        HttpTestCase{"GET",    "/users",     200},
        HttpTestCase{"POST",   "/users",     201},
        HttpTestCase{"GET",    "/users/42",  200},
        HttpTestCase{"DELETE", "/users",     405},  // Method not allowed
        HttpTestCase{"GET",    "/unknown",   404}   // Not found
    )
);
```

Le `SetUp()` configure le routeur une fois par test (rappel : une instance fraîche par paramètre), et `GetParam()` fournit le cas spécifique. Le résultat est un test qui couvre cinq scénarios de routage avec une seule fonction de test.

## Quand utiliser TEST_P et quand l'éviter

### TEST_P est le bon choix quand

- **Même logique, données multiples.** Le corps du test est identique pour tous les cas — seules les entrées et sorties attendues changent.
- **Couverture exhaustive.** Vous voulez tester un comportement sur une large plage de valeurs (caractères Unicode, formats de date, plages numériques).
- **Tests de régression data-driven.** Les cas de test sont issus de bugs reportés et stockés dans un fichier externe que `ValuesIn` consomme.
- **Exploration combinatoire.** `Combine` génère automatiquement toutes les combinaisons d'une matrice de configuration.

### TEST_P n'est pas le bon choix quand

- **La logique diffère entre les cas.** Si chaque cas de test nécessite un arrangement ou une assertion différente, les forcer dans un `TEST_P` conduit à des `if/else` dans le corps du test — un anti-pattern qui défait l'intérêt de la paramétrisation. Utilisez plutôt des `TEST_F` distincts avec des noms descriptifs.
- **Trois cas ou moins.** Pour deux ou trois valeurs, trois `TEST` ou `TEST_F` simples sont plus lisibles que la machinerie `TestWithParam` + `INSTANTIATE_TEST_SUITE_P`. La paramétrisation se justifie à partir de quatre ou cinq cas, ou quand le nombre de cas est amené à croître.
- **Débogage d'un cas spécifique.** Un test paramétré qui échoue sur le cas `/7` parmi vingt nécessite de compter mentalement les paramètres pour identifier la valeur fautive. Les noms personnalisés (name generator) atténuent ce problème, mais un `TEST` nommé `RejectsEmailWithConsecutiveDots` reste intrinsèquement plus clair qu'un `ValidatesCorrectly/7`.

## Résumé des trois macros

| Macro | Héritage | Cas d'usage | Quand l'utiliser |
|-------|----------|-------------|------------------|
| `TEST` | Aucun (implicite `::testing::Test`) | Fonctions pures, cas isolés | Setup trivial (0-2 lignes) |
| `TEST_F` | `::testing::Test` | Tests partageant un contexte | Setup commun, ressources à gérer |
| `TEST_P` | `::testing::TestWithParam<T>` | Même logique, données multiples | 4+ cas de données, combinatoire |

Les trois macros ne sont pas mutuellement exclusives dans un même fichier de test. Un fichier typique peut contenir des `TEST` pour les cas triviaux, une fixture avec des `TEST_F` pour les scénarios nécessitant un setup, et un `TEST_P` pour la validation exhaustive sur des données — chacun au bon niveau d'abstraction.
