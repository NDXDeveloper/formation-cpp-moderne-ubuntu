# 33.2.2 TEST_F : Tests avec fixtures

## Le problème que résolvent les fixtures

En section 33.2.1, nous avons vu que `TEST` fonctionne parfaitement pour les cas simples. Mais dès que plusieurs tests partagent une même séquence d'initialisation, la duplication s'installe :

```cpp
// ❌ Duplication — chaque test recrée le même contexte
TEST(ShoppingCart, TotalIsZeroWhenEmpty) {
    mp::ProductCatalog catalog;
    catalog.add("Widget", 9.99);
    catalog.add("Gadget", 24.99);
    mp::ShoppingCart cart{catalog};

    EXPECT_DOUBLE_EQ(cart.total(), 0.0);
}

TEST(ShoppingCart, TotalReflectsAddedItems) {
    mp::ProductCatalog catalog;
    catalog.add("Widget", 9.99);
    catalog.add("Gadget", 24.99);
    mp::ShoppingCart cart{catalog};

    cart.add_item("Widget", 2);
    EXPECT_DOUBLE_EQ(cart.total(), 19.98);
}

TEST(ShoppingCart, RemoveItemUpdatesTotalCorrectly) {
    mp::ProductCatalog catalog;
    catalog.add("Widget", 9.99);
    catalog.add("Gadget", 24.99);
    mp::ShoppingCart cart{catalog};

    cart.add_item("Widget", 3);
    cart.remove_item("Widget", 1);
    EXPECT_DOUBLE_EQ(cart.total(), 19.98);
}
```

Trois tests, trois fois le même bloc de quatre lignes. Et ce n'est qu'avec deux produits — dans un projet réel, l'initialisation peut impliquer des fichiers de configuration, des connexions réseau ou des structures de données complexes. La fixture résout ce problème en factorisant le setup dans une classe dédiée.

## Anatomie d'une fixture

Une fixture GTest est une classe qui hérite de `::testing::Test`. Ses membres protégés (ou publics) sont directement accessibles dans le corps de chaque `TEST_F` :

```cpp
class ShoppingCartTest : public ::testing::Test {  
protected:  
    // ── Setup ─────────────────────────────────────────────
    void SetUp() override {
        catalog.add("Widget", 9.99);
        catalog.add("Gadget", 24.99);
        catalog.add("Doohickey", 49.99);
    }

    // ── Teardown (optionnel) ──────────────────────────────
    void TearDown() override {
        // Nettoyage si nécessaire
    }

    // ── Membres partagés ──────────────────────────────────
    mp::ProductCatalog catalog;
    mp::ShoppingCart cart{catalog};
};
```

Les tests s'écrivent alors avec `TEST_F`, dont le premier argument est **exactement** le nom de la classe fixture :

```cpp
TEST_F(ShoppingCartTest, TotalIsZeroWhenEmpty) {
    EXPECT_DOUBLE_EQ(cart.total(), 0.0);
}

TEST_F(ShoppingCartTest, TotalReflectsAddedItems) {
    cart.add_item("Widget", 2);
    EXPECT_DOUBLE_EQ(cart.total(), 19.98);
}

TEST_F(ShoppingCartTest, RemoveItemUpdatesTotalCorrectly) {
    cart.add_item("Widget", 3);
    cart.remove_item("Widget", 1);
    EXPECT_DOUBLE_EQ(cart.total(), 19.98);
}
```

Le code de setup a disparu des tests. Chaque cas de test ne contient plus que la phase **Act** et la phase **Assert**, ce qui améliore considérablement la lisibilité.

## Cycle de vie détaillé

Comme introduit en section 33.2, GTest crée une **instance fraîche** de la fixture pour chaque `TEST_F`. Le cycle complet mérite d'être examiné en détail, car il conditionne la conception de fixtures correctes.

### Séquence d'exécution pour chaque TEST_F

```
1.  Allocation mémoire pour l'instance de la fixture
2.  Appel du constructeur de la fixture
3.  Appel de SetUp()
4.  Exécution du corps du TEST_F
5.  Appel de TearDown()
6.  Appel du destructeur de la fixture
7.  Libération mémoire
```

Cette séquence se répète intégralement pour chaque test. Concrètement, si une suite `ShoppingCartTest` contient cinq tests, cinq instances distinctes de `ShoppingCartTest` sont créées et détruites. Aucun état ne persiste d'un test à l'autre.

### Conséquence : isolation garantie

Cette isolation est le bénéfice fondamental des fixtures. Observons ce que cela implique :

```cpp
TEST_F(ShoppingCartTest, AddingItemsModifiesCart) {
    cart.add_item("Widget", 10);
    cart.add_item("Gadget", 5);
    EXPECT_EQ(cart.item_count(), 15);
    // cart contient maintenant 15 items
}

TEST_F(ShoppingCartTest, CartStartsEmpty) {
    // Ce test passe : il reçoit un cart NEUF
    EXPECT_EQ(cart.item_count(), 0);
}
```

Même si `AddingItemsModifiesCart` s'exécute avant `CartStartsEmpty`, le second test reçoit un `cart` fraîchement construit. L'ordre d'exécution n'a aucune importance — et `--gtest_shuffle` le prouvera.

## Constructeur vs SetUp : guide de choix

La question revient fréquemment : faut-il initialiser dans le constructeur ou dans `SetUp()` ? Les deux approches sont valides, mais elles n'ont pas les mêmes caractéristiques.

### Initialiser dans le constructeur

```cpp
class DatabaseClientTest : public ::testing::Test {  
protected:  
    // Initialisation dans la liste d'initialisation
    mp::Config config{"test_db.toml"};
    mp::DatabaseClient client{config};
};
```

Le constructeur est idéal pour les initialisations **qui ne peuvent pas échouer** : assignation de valeurs primitives, construction d'objets via leur constructeur, allocation via smart pointers. L'avantage principal est la concision — la liste d'initialisation des membres (section 6.2.5) suffit souvent, sans même écrire un corps de constructeur.

### Initialiser dans SetUp

```cpp
class ServerConnectionTest : public ::testing::Test {  
protected:  
    void SetUp() override {
        connection = mp::ServerConnection::connect("localhost", 8080);
        ASSERT_NE(connection, nullptr) 
            << "Impossible de se connecter au serveur de test";
    }

    void TearDown() override {
        if (connection) {
            connection->disconnect();
        }
    }

    std::unique_ptr<mp::ServerConnection> connection;
};
```

`SetUp()` est nécessaire quand l'initialisation **peut échouer** et que vous souhaitez utiliser une assertion fatale pour court-circuiter le test. Rappel du modèle d'exécution : si un `ASSERT_*` échoue dans `SetUp()`, GTest saute le corps du test, appelle `TearDown()` puis le destructeur. Ce comportement est bien défini et prévisible. En revanche, un `ASSERT_*` dans un constructeur ne peut pas interrompre proprement la construction de l'objet — le résultat est imprévisible.

### Résumé des critères de choix

| Critère | Constructeur | SetUp() |
|---------|-------------|---------|
| Initialisation simple, sans échec possible | ✅ Idéal | Fonctionnel mais verbeux |
| Assertions fatales (ASSERT_*) nécessaires | ❌ Comportement indéfini | ✅ Fonctionne correctement |
| Membres const ou références | ✅ Via liste d'initialisation | ❌ Impossible (déjà construits) |
| Besoin de méthodes virtuelles | ❌ Appel virtuel dans constructeur piège classique | ✅ L'objet est pleinement construit |

En pratique, beaucoup de fixtures combinent les deux : le constructeur (via les initialiseurs de membres) gère les allocations simples, et `SetUp()` gère les opérations faillibles ou les assertions.

## Destructeur vs TearDown : même logique

La symétrie s'applique au nettoyage :

- **Destructeur** — Préféré pour le nettoyage standard, en accord avec le RAII. Les smart pointers, les conteneurs et les objets RAII se nettoient automatiquement. Si tous vos membres se détruisent proprement, vous n'avez même pas besoin d'écrire de destructeur.
- **`TearDown()`** — Nécessaire si le nettoyage peut échouer et que vous voulez le signaler via une assertion, ou si vous avez besoin de signaler un problème de teardown dans le rapport de test. Les assertions dans un destructeur déclenchent un comportement indéfini (une exception dans un destructeur, même implicite).

Le cas le plus fréquent en pratique est de ne définir **ni destructeur ni `TearDown()`** explicites, en s'appuyant entièrement sur le RAII des membres. C'est la beauté du modèle C++ : une fixture dont tous les membres sont des smart pointers ou des objets à durée de vie automatique se nettoie toute seule.

```cpp
class FileParserTest : public ::testing::Test {  
protected:  
    void SetUp() override {
        // Crée un fichier temporaire pour les tests
        tmp_path = std::filesystem::temp_directory_path() / "gtest_input.txt";
        std::ofstream out{tmp_path};
        ASSERT_TRUE(out.is_open());
        out << "line1\nline2\nline3\n";
    }

    void TearDown() override {
        // Nettoyage explicite : supprimer le fichier temporaire
        std::filesystem::remove(tmp_path);
    }

    std::filesystem::path tmp_path;
    mp::FileParser parser;
};

TEST_F(FileParserTest, CountsLinesCorrectly) {
    auto result = parser.parse(tmp_path);
    EXPECT_EQ(result.line_count(), 3);
}

TEST_F(FileParserTest, ReturnsEmptyOnMissingFile) {
    auto result = parser.parse("/nonexistent/path.txt");
    EXPECT_TRUE(result.is_empty());
}
```

Ici, `TearDown()` supprime le fichier temporaire créé par `SetUp()`. Le fichier doit être supprimé même si le test échoue — et c'est exactement ce que garantit GTest : `TearDown()` est toujours appelé, que le test réussisse ou échoue.

## Fixtures à plusieurs niveaux

Quand une famille de tests partage un setup de base mais que certains sous-groupes nécessitent un setup supplémentaire, l'héritage de fixtures offre une solution élégante :

```cpp
// ── Fixture de base : un utilisateur authentifié ──────────
class AuthenticatedUserTest : public ::testing::Test {  
protected:  
    void SetUp() override {
        session.login("testuser", "testpass");
        ASSERT_TRUE(session.is_authenticated());
    }

    mp::Session session;
};

// ── Fixture spécialisée : un utilisateur avec des données ─
class UserWithDataTest : public AuthenticatedUserTest {  
protected:  
    void SetUp() override {
        AuthenticatedUserTest::SetUp();  // ⚠️ Appeler le parent d'abord
        session.create_document("draft.txt", "Hello");
        session.create_document("notes.md", "# Notes");
    }
};
```

La règle cruciale est d'**appeler explicitement `SetUp()` du parent** dans la classe dérivée. GTest n'effectue pas cet appel automatiquement — c'est une erreur fréquente qui conduit à des tests non initialisés. En revanche, pour `TearDown()`, l'appel au parent doit se faire **en dernier** :

```cpp
void TearDown() override {
    // Nettoyage spécialisé d'abord
    session.delete_all_documents();
    // Puis nettoyage du parent
    AuthenticatedUserTest::TearDown();
}
```

Les tests utilisent ensuite la fixture du niveau approprié :

```cpp
TEST_F(AuthenticatedUserTest, CanAccessProfile) {
    auto profile = session.get_profile();
    EXPECT_EQ(profile.username(), "testuser");
}

TEST_F(UserWithDataTest, ListsAllDocuments) {
    auto docs = session.list_documents();
    EXPECT_EQ(docs.size(), 2u);
}
```

L'héritage de fixtures est puissant mais doit être utilisé avec modération. Au-delà de deux niveaux, la chaîne d'initialisation devient difficile à suivre et les tests perdent en lisibilité. Si vous atteignez trois niveaux ou plus, c'est souvent le signe que le code de production est trop couplé et qu'un refactoring serait plus bénéfique qu'une fixture plus profonde.

## Méthodes helper dans les fixtures

Une fixture n'est pas limitée aux données et au setup — elle peut aussi exposer des méthodes utilitaires qui factorisent des séquences de vérification complexes :

```cpp
class HttpClientTest : public ::testing::Test {  
protected:  
    void SetUp() override {
        client.set_base_url("http://localhost:8080");
        client.set_timeout(std::chrono::seconds{5});
    }

    // ── Helper : vérification structurée d'une réponse ────
    void ExpectSuccessResponse(const mp::HttpResponse& resp, int code = 200) {
        EXPECT_EQ(resp.status_code(), code);
        EXPECT_FALSE(resp.body().empty());
        EXPECT_TRUE(resp.has_header("Content-Type"));
    }

    void ExpectJsonResponse(const mp::HttpResponse& resp) {
        ExpectSuccessResponse(resp);
        EXPECT_EQ(resp.header("Content-Type"), "application/json");
    }

    mp::HttpClient client;
};

TEST_F(HttpClientTest, GetReturnsJsonOnApiEndpoint) {
    auto resp = client.get("/api/users");
    ExpectJsonResponse(resp);
}

TEST_F(HttpClientTest, PostReturnsCreatedStatus) {
    auto resp = client.post("/api/users", R"({"name":"Alice"})");
    ExpectSuccessResponse(resp, 201);
}
```

Les méthodes helper de la fixture utilisent `EXPECT_*` (et non `ASSERT_*`) pour que toutes les vérifications s'exécutent. Si l'une des sous-vérifications de `ExpectJsonResponse` échoue, le rapport indiquera précisément laquelle, avec le fichier et la ligne dans la méthode helper.

> ⚠️ **Attention au `ASSERT_*` dans les helpers.** Un `ASSERT_*` dans une méthode helper non-void ne compile pas si la méthode retourne une valeur — GTest utilise un `return;` en interne, ce qui est incompatible avec une fonction retournant un type. Si vous avez besoin d'assertions fatales dans un helper, déclarez-le `void` ou utilisez le pattern `ASSERT_*` + `HasFatalFailure()` documenté par Google.

## Piège classique : partager une fixture entre fichiers

Une erreur fréquente dans les projets de taille moyenne est de déclarer une fixture dans un header partagé entre plusieurs fichiers `.cpp` de test :

```cpp
// ❌ Tentant mais problématique
// tests/fixtures/shared_fixture.hpp
class SharedDatabaseTest : public ::testing::Test { ... };
```

Cela fonctionne techniquement, mais pose deux problèmes. D'abord, la moindre modification de la fixture déclenche la recompilation de tous les fichiers de test qui l'incluent. Ensuite, cela crée un couplage entre des suites de test qui devraient être indépendantes — le test du module A ne devrait pas être recompilé quand la fixture du module B change.

L'approche recommandée est de garder chaque fixture **locale à son fichier de test**. Si deux fichiers de test ont besoin d'un setup similaire, c'est généralement le signe qu'ils devraient être un seul fichier, ou que le setup commun devrait être extrait dans une **classe utilitaire** (pas une fixture) que chaque fixture instancie comme membre :

```cpp
// tests/helpers/test_database.hpp — Classe utilitaire, PAS une fixture
class TestDatabase {  
public:  
    TestDatabase() { /* setup */ }
    ~TestDatabase() { /* teardown */ }
    void seed_with_users(int count);
    mp::DatabaseClient& client() { return client_; }
private:
    mp::DatabaseClient client_;
};

// tests/test_user_service.cpp
class UserServiceTest : public ::testing::Test {  
protected:  
    TestDatabase db;  // Utilitaire comme membre
    mp::UserService service{db.client()};
};

// tests/test_admin_service.cpp
class AdminServiceTest : public ::testing::Test {  
protected:  
    TestDatabase db;  // Même utilitaire, fixture indépendante
    mp::AdminService service{db.client()};
};
```

Chaque fixture reste locale et autonome, mais le code de setup de la base de données est factorisé dans une classe utilitaire réutilisable.

## Quand passer à TEST_P

La fixture `TEST_F` atteint ses limites quand vous constatez que plusieurs tests ont le même corps et ne diffèrent que par les données d'entrée :

```cpp
// ❌ Signal : même logique, données différentes
TEST_F(ValidatorTest, AcceptsValidEmail_Simple) {
    EXPECT_TRUE(validator.is_valid_email("user@example.com"));
}

TEST_F(ValidatorTest, AcceptsValidEmail_WithPlus) {
    EXPECT_TRUE(validator.is_valid_email("user+tag@example.com"));
}

TEST_F(ValidatorTest, AcceptsValidEmail_WithDots) {
    EXPECT_TRUE(validator.is_valid_email("first.last@example.com"));
}
```

Ce pattern de duplication — même structure, paramètres différents — est exactement le cas d'usage des tests paramétrés `TEST_P`, que la section 33.2.3 couvre en détail.
