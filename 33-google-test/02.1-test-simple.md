# 33.2.1 TEST : Tests simples

## La macro TEST en détail

`TEST` est la macro la plus élémentaire de Google Test. Elle déclare un cas de test autonome, sans état partagé ni initialisation préalable. C'est l'outil naturel pour tester des fonctions libres, des méthodes statiques ou tout comportement qui ne nécessite pas de setup complexe.

La signature est la suivante :

```cpp
TEST(TestSuiteName, TestName) {
    // Corps du test : assertions EXPECT_* et ASSERT_*
}
```

En coulisses, GTest génère une classe anonyme héritant de `::testing::Test` et enregistre le test dans son registre global. Vous n'avez pas à connaître ce mécanisme pour l'utiliser, mais il explique pourquoi les noms de suite et de test doivent être des identifiants C++ valides — pas d'espaces, pas de caractères spéciaux, pas de tirets.

## Tester des fonctions libres

Le cas d'usage le plus direct de `TEST` est la validation de fonctions libres (free functions). Prenons un module utilitaire de calcul mathématique :

```cpp
// include/mon_projet/math.hpp
#pragma once
#include <stdexcept>

namespace mp {

int add(int a, int b);  
int divide(int numerator, int denominator);  
double average(const std::vector<double>& values);  
int factorial(int n);  

} // namespace mp
```

Les tests correspondants couvrent le comportement nominal, les cas limites et les erreurs :

```cpp
// tests/test_math.cpp
#include <gtest/gtest.h>
#include "mon_projet/math.hpp"

// ── Addition ──────────────────────────────────────────────

TEST(Add, ReturnsSumOfTwoPositiveIntegers) {
    EXPECT_EQ(mp::add(2, 3), 5);
}

TEST(Add, HandlesNegativeNumbers) {
    EXPECT_EQ(mp::add(-1, -1), -2);
    EXPECT_EQ(mp::add(-5, 3), -2);
}

TEST(Add, HandlesZero) {
    EXPECT_EQ(mp::add(0, 0), 0);
    EXPECT_EQ(mp::add(42, 0), 42);
    EXPECT_EQ(mp::add(0, 42), 42);
}

// ── Division ──────────────────────────────────────────────

TEST(Divide, ReturnsIntegerQuotient) {
    EXPECT_EQ(mp::divide(10, 3), 3);
    EXPECT_EQ(mp::divide(9, 3), 3);
}

TEST(Divide, ThrowsOnZeroDivisor) {
    EXPECT_THROW(mp::divide(10, 0), std::invalid_argument);
}

TEST(Divide, HandlesNegativeDividend) {
    EXPECT_EQ(mp::divide(-10, 3), -3);
}
```

Plusieurs points structurels méritent attention dans cet exemple.

**Un comportement par test.** Chaque `TEST` vérifie un aspect précis et porte un nom qui le décrit. `HandlesNegativeNumbers` ne teste pas simultanément le cas zéro et les négatifs. Cette granularité a un coût (plus de tests à écrire) mais un bénéfice majeur : quand un test échoue en CI, le nom seul indique immédiatement le comportement cassé, sans avoir à lire le corps du test ni à analyser quelle assertion parmi dix a échoué.

**Plusieurs assertions par test, quand elles portent sur le même comportement.** `HandlesNegativeNumbers` contient deux `EXPECT_EQ` : les deux vérifient le comportement avec des nombres négatifs. Regrouper des assertions thématiquement liées dans un même test est parfaitement acceptable — c'est la raison pour laquelle `EXPECT_*` (non fatal) est préféré à `ASSERT_*`. Si la première assertion échoue, la seconde s'exécute quand même et le rapport montre toutes les combinaisons problématiques.

**Organisation par suite logique.** Les tests sont regroupés par fonction (`Add`, `Divide`). Une convention alternative tout aussi valide est de nommer la suite d'après le module entier (`MathUtils`) et de préfixer les noms de test :

```cpp
TEST(MathUtils, Add_ReturnsSumOfPositives)  
TEST(MathUtils, Add_HandlesNegatives)  
TEST(MathUtils, Divide_ThrowsOnZero)  
```

Le choix est affaire de préférence d'équipe. L'important est la cohérence au sein du projet.

## Tester des méthodes de classe sans fixture

`TEST` peut aussi servir à tester des classes quand chaque test est suffisamment court pour créer son instance localement. Considérons une classe `Temperature` :

```cpp
// include/mon_projet/temperature.hpp
#pragma once
#include <stdexcept>

namespace mp {

class Temperature {  
public:  
    explicit Temperature(double celsius);

    double celsius() const;
    double fahrenheit() const;
    double kelvin() const;

    bool is_freezing() const;
    bool is_boiling() const;

private:
    double celsius_;
};

} // namespace mp
```

```cpp
// tests/test_temperature.cpp
#include <gtest/gtest.h>
#include "mon_projet/temperature.hpp"

TEST(Temperature, StoresCelsiusValue) {
    mp::Temperature t{36.6};
    EXPECT_DOUBLE_EQ(t.celsius(), 36.6);
}

TEST(Temperature, ConvertsCelsiusToFahrenheit) {
    mp::Temperature t{100.0};
    EXPECT_DOUBLE_EQ(t.fahrenheit(), 212.0);
}

TEST(Temperature, ConvertsCelsiusToKelvin) {
    mp::Temperature t{0.0};
    EXPECT_DOUBLE_EQ(t.kelvin(), 273.15);
}

TEST(Temperature, DetectsFreezingPoint) {
    EXPECT_TRUE(mp::Temperature{0.0}.is_freezing());
    EXPECT_TRUE(mp::Temperature{-10.0}.is_freezing());
    EXPECT_FALSE(mp::Temperature{0.1}.is_freezing());
}

TEST(Temperature, DetectsBoilingPoint) {
    EXPECT_TRUE(mp::Temperature{100.0}.is_boiling());
    EXPECT_TRUE(mp::Temperature{150.0}.is_boiling());
    EXPECT_FALSE(mp::Temperature{99.9}.is_boiling());
}

TEST(Temperature, RejectsTemperatureBelowAbsoluteZero) {
    EXPECT_THROW(mp::Temperature{-274.0}, std::out_of_range);
}
```

Cet exemple illustre un cas où `TEST` suffit largement. Chaque test crée sa propre instance de `Temperature` en une ligne — il n'y a aucun setup commun à factoriser, donc une fixture serait de la sur-ingénierie.

Remarquez l'utilisation de `EXPECT_DOUBLE_EQ` au lieu de `EXPECT_EQ` pour les comparaisons de flottants. La section 33.3 détaillera les assertions spécifiques aux flottants, mais le principe est simple : `EXPECT_DOUBLE_EQ` tolère une différence de l'ordre de 4 ULP (*Units in the Last Place*), ce qui absorbe les erreurs d'arrondi inhérentes à l'arithmétique flottante IEEE 754. Comparer deux `double` avec `EXPECT_EQ` est presque toujours une erreur — le test passera parfois et échouera d'autres fois selon le compilateur, le niveau d'optimisation ou l'architecture.

## Pattern Arrange-Act-Assert

Lorsque le corps d'un test dépasse deux ou trois lignes, le pattern **AAA** (Arrange-Act-Assert) apporte une structure lisible et prédictible :

```cpp
TEST(InvoiceCalculator, AppliesDiscountAboveThreshold) {
    // Arrange — Mise en place des données
    std::vector<double> items{150.0, 200.0, 75.0};
    double discount_rate = 0.10;
    double threshold = 400.0;

    // Act — Exécution du comportement testé
    double total = mp::calculate_total(items, discount_rate, threshold);

    // Assert — Vérification du résultat
    double expected = (150.0 + 200.0 + 75.0) * (1.0 - 0.10);  // 382.50
    EXPECT_DOUBLE_EQ(total, expected);
}
```

Les trois phases sont clairement séparées. Cette convention est si répandue qu'elle est devenue un réflexe dans la communauté testing, tous langages confondus. En C++, elle a un bénéfice supplémentaire : elle rend explicite la durée de vie des objets créés dans la phase Arrange, qui sont tous détruits à la fin du test.

Les commentaires `// Arrange`, `// Act`, `// Assert` ne sont pas obligatoires quand le test est court et évident. Ils deviennent utiles quand la phase Arrange est complexe (plusieurs objets, configuration multi-étapes) ou quand le test est relu par quelqu'un qui n'a pas écrit le code.

## Tester les exceptions

GTest fournit trois macros dédiées aux exceptions qui s'intègrent naturellement dans un `TEST` :

```cpp
TEST(JsonParser, ThrowsOnMalformedJson) {
    mp::JsonParser parser;

    // Vérifie qu'une exception du bon type est levée
    EXPECT_THROW(parser.parse("{invalid}"), mp::ParseError);
}

TEST(JsonParser, ThrowsStdExceptionOnEmptyInput) {
    mp::JsonParser parser;

    // Vérifie qu'une exception (n'importe quel type) est levée
    EXPECT_ANY_THROW(parser.parse(""));
}

TEST(JsonParser, DoesNotThrowOnValidJson) {
    mp::JsonParser parser;

    // Vérifie l'absence d'exception
    EXPECT_NO_THROW(parser.parse(R"({"key": "value"})"));
}
```

`EXPECT_THROW` est de loin la plus utilisée. Elle vérifie à la fois que l'expression lève une exception *et* que cette exception est du type spécifié (ou d'un type dérivé). Si vous avez besoin d'inspecter le contenu de l'exception (son message, un code d'erreur), le pattern est le suivant :

```cpp
TEST(JsonParser, ThrowsWithDescriptiveMessage) {
    mp::JsonParser parser;

    try {
        parser.parse("{invalid}");
        FAIL() << "parse() aurait dû lever une exception";
    } catch (const mp::ParseError& e) {
        EXPECT_THAT(std::string(e.what()), 
                     ::testing::HasSubstr("unexpected token"));
    }
}
```

`FAIL()` est une assertion fatale inconditionnelle : si l'exécution atteint cette ligne (c'est-à-dire si aucune exception n'a été levée), le test échoue avec le message fourni. Le `catch` permet ensuite d'inspecter l'exception avec n'importe quelle assertion, ici un matcher Google Mock (`HasSubstr`, détaillé en section 33.3).

## Tester du code qui utilise std::expected (C++23)

Avec la montée en puissance de `std::expected` (section 12.8) comme alternative aux exceptions, les tests doivent aussi couvrir les chemins d'erreur par valeur de retour :

```cpp
TEST(ConfigLoader, ReturnsConfigOnValidFile) {
    auto result = mp::load_config("test_data/valid.toml");

    ASSERT_TRUE(result.has_value());
    EXPECT_EQ(result->port, 8080);
    EXPECT_EQ(result->host, "localhost");
}

TEST(ConfigLoader, ReturnsErrorOnMissingFile) {
    auto result = mp::load_config("nonexistent.toml");

    ASSERT_FALSE(result.has_value());
    EXPECT_EQ(result.error().code, mp::ErrorCode::FileNotFound);
}
```

Remarquez l'utilisation de `ASSERT_TRUE(result.has_value())` avant d'accéder à `result->port`. C'est un cas typique où `ASSERT_*` (fatal) est préférable à `EXPECT_*` : si le résultat est une erreur, accéder à la valeur via `operator->` est un comportement indéfini. L'assertion fatale garantit que le test s'arrête avant de déclencher un UB.

## Quand passer à TEST_F

`TEST` atteint ses limites lorsque vous observez l'un de ces signaux :

- **Duplication du setup.** Trois tests ou plus commencent par les mêmes lignes d'initialisation — créer un objet, ouvrir une connexion, charger un fichier de données.
- **Ressources à libérer.** Le test alloue des ressources qui nécessitent un nettoyage explicite (fichier temporaire, socket, verrou). Le RAII via des smart pointers peut suffire dans certains cas, mais une fixture rend le cleanup systématique.
- **Configuration combinatoire.** Vous testez un objet dans plusieurs états (vide, partiellement rempli, plein) et chaque état nécessite une séquence d'initialisation.

Lorsque ces signaux apparaissent, il est temps de factoriser le setup dans une fixture `TEST_F`, que la section 33.2.2 couvre en détail.
