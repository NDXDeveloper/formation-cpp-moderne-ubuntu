# 26.2.1 `add_executable` et `add_library`

> **Objectif** : Maîtriser les deux commandes fondamentales de création de cibles dans CMake — `add_executable()` pour les binaires exécutables et `add_library()` pour les bibliothèques — avec leurs variantes (STATIC, SHARED, MODULE, OBJECT, INTERFACE) et les bonnes pratiques de déclaration.

---

## `add_executable` : créer un binaire

La commande `add_executable()` crée une cible qui produit un fichier binaire exécutable. C'est la commande la plus intuitive de CMake : vous lui donnez un nom et une liste de sources, elle produit un programme.

### Syntaxe de base

```cmake
add_executable(<nom> [source1] [source2] ...)
```

Appliqué à notre projet exemple :

```cmake
# apps/CMakeLists.txt
add_executable(my_app main.cpp)
```

Cette ligne crée une cible nommée `my_app`. Lors du build, CMake demandera au compilateur de produire un exécutable à partir de `main.cpp` (et de toutes les dépendances déclarées via `target_link_libraries()`). Sur Linux, le binaire résultant sera `build/apps/my_app`.

### Exécutables avec plusieurs sources

Un exécutable peut parfaitement avoir plusieurs fichiers sources. Tous les fichiers listés sont compilés en fichiers objets puis liés ensemble :

```cmake
add_executable(my_app
    main.cpp
    cli_parser.cpp
    app_config.cpp
)
```

Cependant, si ces fichiers représentent de la logique réutilisable (parsing CLI, configuration), il est souvent préférable de les extraire dans une bibliothèque interne que l'exécutable consomme. L'exécutable reste alors un point d'entrée mince :

```cmake
# L'exécutable ne contient que main()
add_executable(my_app main.cpp)  
target_link_libraries(my_app PRIVATE my_project::core my_project::cli)  
```

Cette approche, discutée en section 26.1, facilite les tests unitaires : la logique métier est dans les bibliothèques, testables indépendamment.

### Chemins des sources

Les chemins des fichiers sources sont **relatifs au répertoire contenant le `CMakeLists.txt` courant**, représenté par la variable `CMAKE_CURRENT_SOURCE_DIR`. Dans notre structure, `apps/CMakeLists.txt` contient `add_executable(my_app main.cpp)`, et CMake cherche `main.cpp` dans `apps/`.

Vous pouvez aussi utiliser des chemins relatifs avec des sous-répertoires :

```cmake
add_executable(my_app
    main.cpp
    handlers/signal_handler.cpp
    handlers/shutdown_handler.cpp
)
```

Les chemins absolus fonctionnent aussi, mais sont à éviter car ils brisent la portabilité du projet entre machines. Si vous avez besoin de référencer des fichiers hors du répertoire courant, préférez les variables CMake :

```cmake
# ✅ Portable : utilise une variable CMake
add_executable(my_app
    main.cpp
    ${PROJECT_SOURCE_DIR}/generated/version_info.cpp
)

# ❌ Non portable : chemin absolu en dur
add_executable(my_app
    main.cpp
    /home/dev/my_project/generated/version_info.cpp
)
```

## `add_library` : créer une bibliothèque

La commande `add_library()` est plus riche que `add_executable()` car elle produit différents **types** de bibliothèques. Le type se spécifie comme premier argument après le nom :

```cmake
add_library(<nom> [STATIC | SHARED | MODULE | OBJECT | INTERFACE] [sources...])
```

Chaque type a des caractéristiques distinctes et des cas d'usage différents. Examinons-les.

---

### STATIC : bibliothèque statique (.a)

```cmake
add_library(my_project_core STATIC
    core.cpp
    network.cpp
)
```

Une bibliothèque statique est une **archive** de fichiers objets, regroupés dans un fichier `.a` (sous Linux). Lors du linkage, le linker extrait les fichiers objets nécessaires et les copie directement dans l'exécutable final. Le résultat est un binaire autonome qui n'a pas besoin de la bibliothèque `.a` au moment de l'exécution.

**Avantages** : déploiement simplifié (un seul fichier à distribuer), pas de problème de versions de bibliothèques au runtime, légèrement plus rapide au démarrage (pas de résolution dynamique).

**Inconvénients** : le binaire est plus gros car il embarque le code de toutes ses dépendances statiques. Si plusieurs exécutables lient la même bibliothèque statique, le code est dupliqué dans chaque binaire. Les mises à jour de la bibliothèque nécessitent une recompilation de tous les exécutables.

Le fichier produit suit la convention de nommage Linux :

```
libmy_project_core.a
```

### SHARED : bibliothèque partagée (.so)

```cmake
add_library(my_project_core SHARED
    core.cpp
    network.cpp
)
```

Une bibliothèque partagée (*shared library*) est un fichier `.so` (*shared object*) chargé en mémoire à l'exécution par le *dynamic linker* (`ld-linux.so`). Plusieurs processus peuvent partager la même bibliothèque en mémoire, et la mise à jour de la bibliothèque ne nécessite pas de recompiler les exécutables qui en dépendent (à condition que l'ABI reste stable).

**Avantages** : binaires plus petits, mémoire partagée entre processus, mises à jour indépendantes.

**Inconvénients** : la bibliothèque `.so` doit être présente au runtime (problèmes classiques de « .so not found »), résolution des symboles au démarrage (léger surcoût), gestion de versions plus complexe (soname, symlinks).

Le fichier produit :

```
libmy_project_core.so
```

Avec le versioning activé (via `set_target_properties()`), CMake génère le triplet classique :

```
libmy_project_core.so       → symlink vers .so.1  
libmy_project_core.so.1     → symlink vers .so.1.2.0  
libmy_project_core.so.1.2.0 → fichier réel  
```

### Laisser CMake choisir : `BUILD_SHARED_LIBS`

Si vous ne spécifiez pas le type, CMake utilise la variable `BUILD_SHARED_LIBS` pour décider :

```cmake
# Pas de type explicite — CMake consulte BUILD_SHARED_LIBS
add_library(my_project_core
    core.cpp
    network.cpp
)
```

```bash
# STATIC (par défaut, BUILD_SHARED_LIBS est OFF)
cmake -B build -G Ninja

# SHARED
cmake -B build -G Ninja -DBUILD_SHARED_LIBS=ON
```

Cette approche donne le choix à l'utilisateur final sans changer le `CMakeLists.txt`. C'est la stratégie recommandée pour les bibliothèques destinées à être consommées par d'autres projets : vous ne présumez pas du mode de linkage que le consommateur préfère.

Toutefois, si votre bibliothèque a une contrainte technique forte (par exemple, elle ne peut pas être compilée en `SHARED` à cause de symboles non exportés), spécifiez le type explicitement.

---

### OBJECT : bibliothèque d'objets

```cmake
add_library(my_project_core OBJECT
    core.cpp
    network.cpp
)
```

Une bibliothèque `OBJECT` ne produit ni `.a` ni `.so`. Elle compile les fichiers sources en fichiers objets (`.o`) et les rend disponibles pour être inclus dans d'autres cibles. C'est un mécanisme de **partage de fichiers objets** entre cibles sans passer par une archive intermédiaire.

Le cas d'usage principal est le partage de code entre un exécutable et ses tests sans créer de bibliothèque « réelle » :

```cmake
# Compile les .o une seule fois
add_library(my_project_core OBJECT
    core.cpp
    network.cpp
)

# L'exécutable et les tests utilisent les mêmes .o
add_executable(my_app main.cpp)  
target_link_libraries(my_app PRIVATE my_project_core)  

add_executable(test_core test_core.cpp)  
target_link_libraries(test_core PRIVATE my_project_core GTest::gtest_main)  
```

Depuis CMake 3.12, les bibliothèques `OBJECT` se comportent comme des cibles normales dans `target_link_libraries()` — elles propagent leurs fichiers objets, include directories et autres propriétés. Avant cette version, il fallait utiliser la syntax `$<TARGET_OBJECTS:my_project_core>`, aujourd'hui obsolète.

**Quand utiliser OBJECT ?** Pour les petits projets où la création d'une bibliothèque statique ou partagée semble excessive, mais où vous avez besoin de partager les sources entre plusieurs cibles. Pour les projets de taille moyenne et grande, une bibliothèque `STATIC` est généralement préférable car elle simplifie la gestion des dépendances.

---

### INTERFACE : bibliothèque sans sources

```cmake
add_library(my_project_headers INTERFACE)

target_include_directories(my_project_headers
    INTERFACE
        $<BUILD_INTERFACE:${PROJECT_SOURCE_DIR}/include>
        $<INSTALL_INTERFACE:include>
)

target_compile_features(my_project_headers INTERFACE cxx_std_23)
```

Une bibliothèque `INTERFACE` ne contient **aucune source à compiler**. Elle n'existe que pour transporter des propriétés vers les cibles qui en dépendent. C'est le mécanisme idéal pour trois cas de figure.

**Les bibliothèques header-only.** Si votre bibliothèque est entièrement implémentée dans les headers (templates, constexpr, inline), elle n'a pas de `.cpp` à compiler. La cible `INTERFACE` porte les include directories et les exigences de compilation, et les propage aux consommateurs :

```cmake
# Bibliothèque header-only
add_library(json_utils INTERFACE)  
target_include_directories(json_utils  
    INTERFACE ${CMAKE_CURRENT_SOURCE_DIR}/include
)
target_compile_features(json_utils INTERFACE cxx_std_17)
```

**Les collections de flags.** Vous pouvez créer une cible `INTERFACE` qui ne contient que des options de compilation, et la lier aux cibles qui doivent les recevoir :

```cmake
# Cible "flags" — pas de sources, pas de headers, juste des options
add_library(project_warnings INTERFACE)  
target_compile_options(project_warnings  
    INTERFACE
        -Wall -Wextra -Wpedantic -Wshadow -Wconversion
)

# Toute cible qui lie project_warnings reçoit ces flags
target_link_libraries(my_project_core PRIVATE project_warnings)  
target_link_libraries(my_project_utils PRIVATE project_warnings)  
```

C'est une alternative élégante au module `CompilerWarnings.cmake` montré en section 26.1 — les deux approches sont valides.

**Les cibles de regroupement.** Une cible `INTERFACE` peut servir de façade qui regroupe plusieurs dépendances sous un seul nom :

```cmake
add_library(my_project_all_deps INTERFACE)  
target_link_libraries(my_project_all_deps  
    INTERFACE
        OpenSSL::SSL
        ZLIB::ZLIB
        Threads::Threads
)

# Lier une seule cible au lieu de trois
target_link_libraries(my_project_core PRIVATE my_project_all_deps)
```

Notez que pour une cible `INTERFACE`, toutes les propriétés doivent être déclarées avec la visibilité `INTERFACE` — `PUBLIC` et `PRIVATE` n'ont pas de sens puisque la cible elle-même ne compile rien.

---

### MODULE : bibliothèque chargée dynamiquement

```cmake
add_library(my_plugin MODULE
    plugin_impl.cpp
)
```

Une bibliothèque `MODULE` est similaire à `SHARED`, mais avec une différence fondamentale : elle n'est pas destinée à être liée à la compilation via `target_link_libraries()`. Elle est conçue pour être chargée **à l'exécution** via `dlopen()` (Linux) ou `LoadLibrary()` (Windows).

C'est le type adapté aux **systèmes de plugins** : l'application charge les modules dynamiquement, à la demande, sans connaître leur existence à la compilation.

```cpp
// Chargement dynamique d'un plugin — côté application
void* handle = dlopen("./libmy_plugin.so", RTLD_LAZY);  
auto create = (Plugin*(*)())dlsym(handle, "create_plugin");  
Plugin* p = create();  
```

Ce type est spécialisé et moins courant que `STATIC` ou `SHARED`. Vous ne le rencontrerez principalement que dans les architectures à plugins explicites.

---

## Résumé comparatif des types de bibliothèques

| Type | Fichier produit | Lié à la compilation ? | Présent au runtime ? | Cas d'usage principal |
|------|----------------|:---------------------:|:-------------------:|----------------------|
| `STATIC` | `.a` | ✅ | ❌ (embarqué dans le binaire) | Bibliothèques internes, déploiement simplifié |
| `SHARED` | `.so` | ✅ | ✅ | Bibliothèques système, mises à jour indépendantes |
| `OBJECT` | `.o` (fichiers objets) | ✅ | ❌ | Partage de sources entre cibles, petits projets |
| `INTERFACE` | Rien | ✅ (propagation de propriétés) | ❌ | Header-only, collections de flags, façades |
| `MODULE` | `.so` | ❌ (chargé via dlopen) | ✅ | Systèmes de plugins |
| *(omis)* | `.a` ou `.so` | ✅ | Dépend | Laisser le choix à l'utilisateur via `BUILD_SHARED_LIBS` |

## Ajouter des sources après la déclaration

Il est parfois pratique d'ajouter des sources à une cible après sa création — par exemple pour gérer des fichiers spécifiques à une plateforme :

```cmake
add_library(my_project_core
    core.cpp
    network.cpp
)

# Sources spécifiques Linux
if(CMAKE_SYSTEM_NAME STREQUAL "Linux")
    target_sources(my_project_core PRIVATE
        platform/linux_impl.cpp
    )
endif()

# Sources générées (par un outil, un proto compiler, etc.)
target_sources(my_project_core PRIVATE
    ${CMAKE_CURRENT_BINARY_DIR}/generated/config.cpp
)
```

La commande `target_sources()` ajoute des fichiers à une cible existante. Elle accepte les mêmes niveaux de visibilité que les autres commandes `target_*` :

- `PRIVATE` : les sources sont compilées pour cette cible uniquement (le cas normal) ;
- `PUBLIC` : les sources sont compilées pour cette cible et propagées aux cibles dépendantes (rare, usage avancé) ;
- `INTERFACE` : les sources sont propagées aux cibles dépendantes mais pas compilées par cette cible (utile pour les bibliothèques `INTERFACE` avec des fichiers source « consommés » par le dépendant).

L'utilisation de `target_sources()` est préférable au mélange de logique conditionnelle dans la commande `add_library()` elle-même, car elle garde la déclaration initiale propre et lisible.

## Headers dans les sources : oui ou non ?

Une question récurrente : faut-il lister les fichiers `.h` / `.hpp` dans `add_library()` ou `add_executable()` ?

```cmake
# Avec headers listés
add_library(my_project_core
    core.cpp
    network.cpp
    ${PROJECT_SOURCE_DIR}/include/my_project/core.h
    ${PROJECT_SOURCE_DIR}/include/my_project/network.h
)

# Sans headers listés
add_library(my_project_core
    core.cpp
    network.cpp
)
```

Techniquement, CMake **ignore les headers** lors de la compilation — seuls les `.cpp` produisent des fichiers objets. Lister les headers n'a aucun effet sur le build lui-même. Cependant, les inclure présente un avantage pratique : les **IDE** (CLion, VS Code, Visual Studio) utilisent la liste des sources d'une cible pour peupler l'arborescence du projet. Si un header n'est listé dans aucune cible, certains IDE ne l'afficheront pas ou ne l'associeront pas correctement au projet.

La recommandation est pragmatique : listez les headers si votre IDE en bénéficie, ne les listez pas si cela encombre vos `CMakeLists.txt` sans apporter de valeur. Dans les deux cas, le résultat de la compilation est identique.

Si vous choisissez de lister les headers, regroupez-les visuellement pour maintenir la lisibilité :

```cmake
add_library(my_project_core
    # Sources
    core.cpp
    network.cpp
    # Headers (pour l'IDE)
    ${PROJECT_SOURCE_DIR}/include/my_project/core.h
    ${PROJECT_SOURCE_DIR}/include/my_project/network.h
)
```

## Nommage des cibles : conventions et pièges

Le nom passé à `add_executable()` ou `add_library()` est le **nom logique** de la cible dans CMake. Par défaut, c'est aussi le nom du fichier produit (sans l'extension). Ces deux aspects peuvent être dissociés si nécessaire.

### Le nom logique doit être unique et descriptif

```cmake
# ❌ Noms génériques — collision garantie dans un projet complexe
add_library(utils ...)  
add_library(core ...)  
add_executable(app ...)  

# ✅ Noms préfixés — sans ambiguïté
add_library(my_project_utils ...)  
add_library(my_project_core ...)  
add_executable(my_project_app ...)  
```

### Le nom de fichier peut différer du nom logique

La propriété `OUTPUT_NAME` permet de dissocier le nom de la cible CMake du nom du fichier produit. C'est utile quand le nom logique doit être unique (avec préfixe), mais le fichier final doit avoir un nom plus court :

```cmake
add_executable(my_project_app main.cpp)  
set_target_properties(my_project_app PROPERTIES  
    OUTPUT_NAME "my-app"
)
# Produit : build/apps/my-app (au lieu de my_project_app)
```

De même pour les bibliothèques, si vous voulez un nom de fichier qui ne suit pas la convention `lib<target>.a` :

```cmake
add_library(my_project_core STATIC core.cpp network.cpp)  
set_target_properties(my_project_core PROPERTIES  
    OUTPUT_NAME "mycore"
)
# Produit : libmycore.a
```

### Cibles alias avec namespace

Comme introduit en section 26.2, les alias à namespace sont une pratique recommandée pour chaque bibliothèque :

```cmake
add_library(my_project_core STATIC core.cpp network.cpp)  
add_library(my_project::core ALIAS my_project_core)  
```

Le nom d'alias `my_project::core` sert uniquement dans les `CMakeLists.txt` (dans `target_link_libraries()` notamment). Il ne peut pas recevoir de propriétés directement — toutes les commandes `target_*` doivent viser le nom réel `my_project_core`.

```cmake
# ✅ Correct — propriétés sur le nom réel
target_include_directories(my_project_core PUBLIC ...)

# ❌ Erreur — on ne peut pas modifier un alias
target_include_directories(my_project::core PUBLIC ...)

# ✅ Correct — l'alias est utilisable dans les dépendances
target_link_libraries(my_app PRIVATE my_project::core)
```

## Exemple intégré : les cibles de notre projet

Reprenons l'arborescence de la section 26.1 et listons toutes les cibles déclarées dans le projet, avec leur type et leur rôle :

```cmake
# libs/utils/CMakeLists.txt
add_library(my_project_utils
    src/string_utils.cpp
)
add_library(my_project::utils ALIAS my_project_utils)


# src/CMakeLists.txt
add_library(my_project_core
    core.cpp
    network.cpp
)
add_library(my_project::core ALIAS my_project_core)


# apps/CMakeLists.txt
add_executable(my_project_app main.cpp)  
set_target_properties(my_project_app PROPERTIES OUTPUT_NAME "my-app")  


# tests/CMakeLists.txt
add_executable(my_project_test_core test_core.cpp)  
add_executable(my_project_test_network test_network.cpp)  
```

Le graphe de cibles résultant :

```
my_project_app (exécutable → "my-app")
  └── my_project_core (bibliothèque)
        └── my_project_utils (bibliothèque)

my_project_test_core (exécutable de test)
  ├── my_project_core
  └── GTest::gtest_main (cible importée)

my_project_test_network (exécutable de test)
  ├── my_project_core
  └── GTest::gtest_main
```

Chaque cible est déclarée une seule fois, dans le `CMakeLists.txt` du répertoire qui la « possède ». Les dépendances entre cibles sont établies par `target_link_libraries()`, que nous détaillons en section 26.2.2.
