# 26.3.3 `add_subdirectory`

> **Objectif** : Maîtriser `add_subdirectory()` — le mécanisme le plus direct de CMake pour intégrer des sources dans le build — qu'il s'agisse de bibliothèques internes au projet, de dépendances embarquées dans un répertoire `third_party/`, ou de sous-modules Git.

---

## Principe de fonctionnement

`add_subdirectory()` est la commande la plus simple et la plus ancienne de CMake pour structurer un projet en modules. Son fonctionnement est élémentaire : elle dit à CMake « va lire le `CMakeLists.txt` qui se trouve dans ce répertoire et exécute-le ».

```cmake
add_subdirectory(src)
```

CMake ouvre `src/CMakeLists.txt`, l'exécute entièrement (ce qui peut créer des cibles, définir des variables, appeler d'autres `add_subdirectory`), puis revient au fichier appelant et continue la lecture. Les cibles créées dans le sous-répertoire deviennent disponibles globalement dans le projet.

C'est le mécanisme que nous utilisons depuis la section 26.1 pour organiser notre projet en sous-répertoires (`libs/utils`, `src`, `apps`, `tests`). Mais `add_subdirectory()` sert aussi à intégrer des **dépendances tierces** dont les sources sont présentes dans l'arborescence du projet.

---

## Syntaxe

```cmake
add_subdirectory(<source_dir> [<binary_dir>] [EXCLUDE_FROM_ALL] [SYSTEM])
```

### Forme simple : répertoire enfant

```cmake
add_subdirectory(libs/utils)
```

Le chemin est relatif au `CMakeLists.txt` courant. CMake crée automatiquement un répertoire de build correspondant dans `build/libs/utils/`. C'est la forme que vous utiliserez 90% du temps.

### Forme avec répertoire de build explicite

```cmake
add_subdirectory(libs/utils build/custom_utils_dir)
```

Le second argument spécifie le répertoire de build. Rarement nécessaire — utile uniquement quand vous intégrez un répertoire situé hors de l'arborescence du projet (voir plus loin).

### `EXCLUDE_FROM_ALL`

```cmake
add_subdirectory(benchmarks EXCLUDE_FROM_ALL)
```

Par défaut, toutes les cibles d'un sous-répertoire font partie du build global — un `cmake --build build` les compile toutes. `EXCLUDE_FROM_ALL` exclut les cibles du sous-répertoire du build par défaut. Elles ne sont compilées que si elles sont explicitement demandées (en tant que dépendance d'une autre cible, ou via `cmake --build build --target bench_core`).

C'est utile pour les benchmarks, les exemples ou les outils optionnels que vous ne voulez pas compiler à chaque build :

```cmake
add_subdirectory(benchmarks EXCLUDE_FROM_ALL)  
add_subdirectory(examples EXCLUDE_FROM_ALL)  
add_subdirectory(tools EXCLUDE_FROM_ALL)  
```

### `SYSTEM` (CMake 3.25+)

```cmake
add_subdirectory(third_party/some_lib SYSTEM)
```

Le mot-clé `SYSTEM` marque les include directories du sous-répertoire comme **headers système**. Concrètement, le compilateur utilisera `-isystem` au lieu de `-I` pour ces chemins, ce qui supprime les warnings provenant de la dépendance. C'est l'équivalent de `target_include_directories(... SYSTEM ...)`, mais appliqué automatiquement à toutes les cibles du sous-répertoire.

C'est précieux pour les dépendances tierces embarquées : vous voulez compiler votre code avec `-Wall -Wextra -Werror`, sans que les warnings d'une bibliothèque tierce ne fassent échouer votre build.

```cmake
# Sans SYSTEM : les warnings de some_lib polluent votre build
add_subdirectory(third_party/some_lib)

# Avec SYSTEM : les warnings de some_lib sont supprimés
add_subdirectory(third_party/some_lib SYSTEM)
```

## Cas d'usage n°1 : structurer un projet en modules internes

C'est l'usage principal d'`add_subdirectory`, que nous pratiquons depuis le début de ce chapitre. Le `CMakeLists.txt` racine délègue aux sous-répertoires :

```cmake
# CMakeLists.txt racine
cmake_minimum_required(VERSION 3.25)  
project(my_project VERSION 1.2.0 LANGUAGES CXX)  

add_subdirectory(libs/utils)    # Bibliothèque utilitaire interne  
add_subdirectory(src)           # Bibliothèque principale  
add_subdirectory(apps)          # Exécutables  

if(MY_PROJECT_BUILD_TESTS)
    enable_testing()
    add_subdirectory(tests)
endif()
```

Chaque sous-répertoire contient son propre `CMakeLists.txt` qui déclare ses cibles. La hiérarchie `add_subdirectory` reflète la structure logique du projet.

### Ordre des appels

L'ordre des `add_subdirectory()` est important. CMake exécute les fichiers séquentiellement : une cible doit être **créée** avant d'être **référencée**. Si `src/CMakeLists.txt` fait `target_link_libraries(my_project_core PRIVATE my_project::utils)`, alors `add_subdirectory(libs/utils)` doit précéder `add_subdirectory(src)` :

```cmake
# ✅ Correct : utils est créé avant d'être référencé par src
add_subdirectory(libs/utils)  
add_subdirectory(src)  

# ❌ Erreur : src référence utils qui n'existe pas encore
add_subdirectory(src)  
add_subdirectory(libs/utils)  
```

Pour les projets avec de nombreux sous-répertoires, un commentaire rappelant les dépendances entre modules aide à la maintenance :

```cmake
# Ordre imposé par les dépendances :
# utils → (aucune dépendance interne)
# core  → utils
# apps  → core
# tests → core
add_subdirectory(libs/utils)  
add_subdirectory(src)  
add_subdirectory(apps)  
add_subdirectory(tests)  
```

## Cas d'usage n°2 : dépendances embarquées dans `third_party/`

L'approche la plus directe pour intégrer une dépendance tierce est de placer ses sources dans un répertoire `third_party/` et d'appeler `add_subdirectory()` dessus.

### Copie directe

```
my_project/
├── CMakeLists.txt
├── src/
├── third_party/
│   ├── fmt/                    ← Sources de fmt copiées ici
│   │   ├── CMakeLists.txt
│   │   ├── include/
│   │   └── src/
│   └── cli11/                  ← Sources de CLI11 copiées ici
│       ├── CMakeLists.txt
│       └── include/
└── ...
```

```cmake
# CMakeLists.txt racine
add_subdirectory(third_party/fmt)  
add_subdirectory(third_party/cli11)  
```

L'avantage est la simplicité et la totale indépendance vis-à-vis du réseau au moment du build. L'inconvénient majeur est que les sources de la dépendance vivent dans votre dépôt — vous devez les mettre à jour manuellement, et elles alourdissent l'historique Git.

### Sous-modules Git (recommandé pour third_party)

Les sous-modules Git résolvent le problème de la mise à jour. La dépendance reste dans son propre dépôt ; votre projet ne stocke qu'une référence (un commit précis) :

```bash
# Ajouter un sous-module
git submodule add https://github.com/fmtlib/fmt.git third_party/fmt  
cd third_party/fmt  
git checkout 11.1.4  
cd ../..  
git add third_party/fmt  
git commit -m "Add fmt 11.1.4 as submodule"  
```

```cmake
# CMakeLists.txt — identique à la copie directe
add_subdirectory(third_party/fmt SYSTEM)
```

L'arborescence est la même, mais `third_party/fmt/` est un sous-module Git pointant vers un commit précis. La mise à jour se fait par :

```bash
cd third_party/fmt  
git fetch && git checkout 11.2.0  
cd ../..  
git add third_party/fmt  
git commit -m "Update fmt to 11.2.0"  
```

Le clone du projet nécessite `--recurse-submodules` pour récupérer les sous-modules :

```bash
git clone --recurse-submodules https://github.com/you/my_project.git
# Ou après un clone classique :
git submodule update --init --recursive
```

> 💡 **Conseil** : ajoutez un rappel dans votre README ou un check dans votre CI pour détecter les sous-modules non initialisés. Un `CMakeLists.txt` vide dans `third_party/fmt/` (parce que le sous-module n'a pas été cloné) produit des erreurs CMake cryptiques.  
>
> ```cmake
> # Vérification rapide dans le CMakeLists.txt racine
> if(NOT EXISTS "${CMAKE_CURRENT_SOURCE_DIR}/third_party/fmt/CMakeLists.txt")
>     message(FATAL_ERROR
>         "Le sous-module third_party/fmt n'est pas initialisé.\n"
>         "Exécutez : git submodule update --init --recursive"
>     )
> endif()
> ```

## Cas d'usage n°3 : répertoire externe à l'arborescence

`add_subdirectory()` peut aussi intégrer un répertoire situé **en dehors** de votre arbre source. Dans ce cas, vous devez fournir explicitement le répertoire de build (second argument) :

```cmake
add_subdirectory(/home/dev/shared_libs/math_lib  ${CMAKE_BINARY_DIR}/math_lib)
```

Le répertoire de build est nécessaire car CMake ne peut pas placer les fichiers générés dans un sous-répertoire de `build/` qui correspondrait à un chemin hors de l'arbre source.

Ce mécanisme est utile pendant le développement local, quand vous travaillez simultanément sur un projet et une bibliothèque et voulez tester les modifications sans réinstaller la bibliothèque à chaque changement. Mais il est non portable (chemin absolu) et ne doit pas apparaître dans du code versionné. Pour les cas de développement local, `FetchContent` avec `SOURCE_DIR` (section 26.3.2) ou les CMake Presets avec des surcharges locales (section 27.6) sont des alternatives plus propres.

## Scope des variables et isolation

Comme évoqué en section 26.2, `add_subdirectory()` crée un **nouveau scope de variables**. Le sous-répertoire hérite des variables du parent au moment de l'appel, mais ses modifications locales ne remontent pas :

```cmake
# CMakeLists.txt racine
set(MY_VAR "valeur_parent")  
add_subdirectory(src)  
message(STATUS "MY_VAR = ${MY_VAR}")  # Affiche : valeur_parent  
```

```cmake
# src/CMakeLists.txt
message(STATUS "MY_VAR = ${MY_VAR}")  # Affiche : valeur_parent  
set(MY_VAR "valeur_enfant")  
message(STATUS "MY_VAR = ${MY_VAR}")  # Affiche : valeur_enfant  
# La modification reste locale à ce scope
```

Cette isolation est une protection contre les effets de bord involontaires. Cependant, les **cibles** et les **variables de cache** sont globales — une cible créée dans un sous-répertoire est visible partout.

### Isolation vis-à-vis des dépendances tierces

L'isolation de scope est particulièrement importante quand vous intégrez des dépendances via `add_subdirectory()`. Le `CMakeLists.txt` de la dépendance peut modifier des variables (`CMAKE_CXX_FLAGS`, `CMAKE_BUILD_TYPE`, etc.) sans affecter votre projet — à condition qu'il ne manipule pas les variables de cache de manière agressive.

C'est une différence clé entre la théorie et la pratique. En théorie, le scope protège votre projet. En pratique, certaines bibliothèques mal écrites utilisent `set(CMAKE_CXX_FLAGS "..." CACHE STRING "" FORCE)` ou appellent `add_definitions()` (portée globale descendante), ce qui peut contaminer votre build. Le mot-clé `SYSTEM` (CMake 3.25+) atténue les problèmes d'include directories, mais ne résout pas les manipulations de variables.

## `add_subdirectory` vs `FetchContent` : quand utiliser lequel ?

Les deux mécanismes appellent le même code interne — `FetchContent_MakeAvailable` appelle `add_subdirectory()` en coulisse. La différence est dans la **provenance des sources** :

| Critère | `add_subdirectory` | `FetchContent` |
|---------|-------------------|----------------|
| Sources déjà présentes localement | ✅ Oui — c'est sa raison d'être | Possible (avec `SOURCE_DIR`) mais non idiomatique |
| Téléchargement automatique | ❌ Non | ✅ Oui (Git clone, archive URL) |
| Sous-modules Git | ✅ Naturel | Non pertinent |
| Reproductibilité sans réseau | ✅ Les sources sont dans le dépôt | ❌ Nécessite un accès réseau au premier build |
| Taille du dépôt | ⚠️ Augmentée par les sources embarquées | ✅ Les sources sont dans le cache de build |
| Mise à jour de la dépendance | Manuelle (sous-module) ou copie | Changer le `GIT_TAG` |
| Patchs locaux | ✅ Facile (modifier directement les sources) | Possible mais plus délicat |

**En résumé** : utilisez `add_subdirectory()` quand les sources sont déjà dans votre arborescence (sous-modules, copies, modules internes). Utilisez `FetchContent` quand vous voulez que CMake télécharge les sources automatiquement.

## Patterns avancés

### Wrapper conditionnel find_package / add_subdirectory

Un pattern courant pour les bibliothèques internes réutilisables est de supporter les deux modes de consommation :

```cmake
# Le consommateur peut utiliser la bibliothèque via find_package (si installée)
# ou via add_subdirectory (si intégrée comme sous-projet)

# Détection automatique dans le CMakeLists.txt racine :
if(EXISTS "${CMAKE_CURRENT_SOURCE_DIR}/libs/utils/CMakeLists.txt")
    # Mode sous-projet : les sources sont disponibles
    add_subdirectory(libs/utils)
else()
    # Mode installé : chercher sur le système
    find_package(my_project_utils REQUIRED)
endif()
```

### Désactiver l'installation des dépendances

Quand vous intégrez une dépendance tierce via `add_subdirectory()`, ses règles `install()` s'ajoutent aux vôtres. Un `cmake --install build` installerait alors non seulement votre projet, mais aussi toutes les dépendances — rarement souhaitable.

La convention pour désactiver l'installation d'un sous-projet est de ne pas appeler `install()` quand le projet n'est pas le projet racine :

```cmake
# CMakeLists.txt d'une bibliothèque bien écrite
if(CMAKE_CURRENT_SOURCE_DIR STREQUAL CMAKE_SOURCE_DIR)
    # Ce projet est le projet racine — inclure les règles d'installation
    install(TARGETS my_lib ...)
endif()
```

Pour les bibliothèques qui ne font pas cette vérification, `EXCLUDE_FROM_ALL` empêche la plupart des installations indésirables :

```cmake
add_subdirectory(third_party/some_lib EXCLUDE_FROM_ALL)
```

Depuis CMake 3.21, la variable `<Package>_INSTALL_SKIP` ou le pattern `FetchContent` + `OVERRIDE_FIND_PACKAGE` offrent des solutions plus chirurgicales — mais `EXCLUDE_FROM_ALL` reste la méthode la plus simple et la plus largement supportée.

### Regrouper les dépendances dans un CMakeLists.txt dédié

Pour garder le `CMakeLists.txt` racine concis, les dépendances tierces peuvent être regroupées dans un fichier dédié :

```cmake
# third_party/CMakeLists.txt
# Toutes les dépendances embarquées sont déclarées ici

add_subdirectory(fmt SYSTEM)  
add_subdirectory(spdlog SYSTEM)  
add_subdirectory(cli11 SYSTEM)  

# Désactiver les composants non nécessaires
set(BENCHMARK_ENABLE_TESTING OFF CACHE BOOL "" FORCE)  
add_subdirectory(benchmark EXCLUDE_FROM_ALL SYSTEM)  
```

```cmake
# CMakeLists.txt racine — une seule ligne pour toutes les dépendances tierces
add_subdirectory(third_party)
```

## Récapitulatif

| Aspect | Recommandation |
|--------|---------------|
| Modules internes | `add_subdirectory(libs/module)` — usage principal |
| Dépendances embarquées | `add_subdirectory(third_party/lib SYSTEM)` + sous-modules Git |
| Répertoire externe | Éviter en code versionné — préférer FetchContent avec `SOURCE_DIR` |
| Ordre des appels | Respecter le graphe de dépendances (créer avant de référencer) |
| Warnings des dépendances | Mot-clé `SYSTEM` (CMake 3.25+) |
| Installation des dépendances | `EXCLUDE_FROM_ALL` pour empêcher l'installation |
| Vérification des sous-modules | Test d'existence du `CMakeLists.txt` + message explicite |
| Organisation | Regrouper les dépendances tierces dans `third_party/CMakeLists.txt` |
