# 26.3.2 `FetchContent`

> **Objectif** : Maîtriser le module `FetchContent` — le mécanisme de CMake pour télécharger et intégrer automatiquement des dépendances au moment de la configuration — en comprenant son workflow, ses options, ses pièges, et les bonnes pratiques pour un build reproductible.

---

## Le problème que FetchContent résout

`find_package()` suppose que la bibliothèque est déjà installée sur le système. C'est parfait pour les dépendances système courantes (OpenSSL, ZLIB, Threads), mais contraignant pour les bibliothèques qui ne sont pas dans les dépôts de la distribution ou qui nécessitent une version très spécifique.

Sans `FetchContent`, le workflow classique ressemble à :

```bash
# L'utilisateur doit installer manuellement chaque dépendance
git clone https://github.com/google/googletest.git  
cd googletest && mkdir build && cd build  
cmake .. -DCMAKE_INSTALL_PREFIX=/usr/local  
cmake --build . && sudo cmake --install .  
# ... et recommencer pour chaque dépendance
```

`FetchContent` automatise tout cela : CMake télécharge les sources, les configure et les intègre dans votre build, en une seule étape, de manière transparente pour l'utilisateur.

---

## Workflow en deux étapes

`FetchContent` fonctionne en deux phases distinctes : la **déclaration** et la **mise à disposition**.

### Étape 1 : `FetchContent_Declare` — décrire la dépendance

```cmake
include(FetchContent)

FetchContent_Declare(googletest
    GIT_REPOSITORY https://github.com/google/googletest.git
    GIT_TAG        v1.17.0
)
```

Cette commande **ne télécharge rien**. Elle enregistre une description : le nom logique de la dépendance (`googletest`), l'URL du dépôt, et la version (tag Git). CMake stocke ces informations pour les utiliser plus tard.

### Étape 2 : `FetchContent_MakeAvailable` — télécharger et intégrer

```cmake
FetchContent_MakeAvailable(googletest)
```

C'est ici que tout se passe. `FetchContent_MakeAvailable` effectue séquentiellement :

1. **Téléchargement** : clone le dépôt Git (ou télécharge l'archive) dans le répertoire de build, sous `_deps/googletest-src/`.
2. **Configuration** : exécute le `CMakeLists.txt` de la dépendance, comme si vous aviez fait `add_subdirectory()` sur ses sources.
3. **Exposition des cibles** : les cibles définies par la dépendance (ici `GTest::gtest`, `GTest::gtest_main`, `GTest::gmock`) deviennent disponibles dans votre projet.

Après cet appel, vous consommez la dépendance exactement comme si elle était installée sur le système :

```cmake
target_link_libraries(my_test PRIVATE GTest::gtest_main)
```

## Sources de téléchargement

`FetchContent_Declare` supporte plusieurs types de sources.

### Dépôt Git (le plus courant)

```cmake
FetchContent_Declare(spdlog
    GIT_REPOSITORY https://github.com/gabime/spdlog.git
    GIT_TAG        v1.15.3
)
```

`GIT_TAG` accepte un tag, un hash de commit, ou un nom de branche. Les tags et les hashs de commit sont **reproductibles** — ils pointent toujours vers le même code. Les noms de branches (`main`, `develop`) sont mobiles et à éviter pour les builds de production.

```cmake
# ✅ Reproductible — tag fixe
GIT_TAG v1.15.3

# ✅ Reproductible — hash de commit
GIT_TAG 8fc1b20e5a0b7b5c9e9e75c7f3b9f6c8e2e0d1a3

# ❌ Non reproductible — branche mobile
GIT_TAG main
```

### Option `GIT_SHALLOW`

Pour les dépôts volumineux, le clone complet peut être lent. L'option `GIT_SHALLOW ON` effectue un clone superficiel (équivalent de `git clone --depth 1`) qui ne télécharge que le commit ciblé :

```cmake
FetchContent_Declare(llvm
    GIT_REPOSITORY https://github.com/llvm/llvm-project.git
    GIT_TAG        llvmorg-19.1.0
    GIT_SHALLOW    ON
)
```

Le gain de temps et de bande passante peut être considérable pour les gros dépôts. L'inconvénient est que `git log` dans le répertoire cloné ne montrera qu'un seul commit — sans impact sur le build, mais potentiellement gênant pour le debug.

### Archive URL

```cmake
FetchContent_Declare(nlohmann_json
    URL      https://github.com/nlohmann/json/releases/download/v3.11.3/json.tar.xz
    URL_HASH SHA256=d6c65aca6b1ed68e7a182f4757f0f1c0e0c4cf09a362c05da44e23467c4e617c
)
```

Télécharger une archive est souvent plus rapide qu'un clone Git (pas d'historique, pas de métadonnées `.git`). Le paramètre `URL_HASH` est **fortement recommandé** : il garantit l'intégrité du téléchargement et protège contre les modifications silencieuses du fichier distant. CMake vérifie le hash et échoue si la correspondance n'est pas exacte.

Pour obtenir le hash d'une archive :

```bash
wget -q https://github.com/nlohmann/json/releases/download/v3.11.3/json.tar.xz  
sha256sum json.tar.xz  
```

### Répertoire local

```cmake
FetchContent_Declare(my_local_dep
    SOURCE_DIR /home/dev/projects/my_other_lib
)
```

Utile pendant le développement, quand vous travaillez simultanément sur un projet et sa dépendance. Aucun téléchargement n'est effectué — CMake utilise directement les sources locales. À ne pas utiliser en production car le chemin est non portable.

---

## Où arrivent les fichiers ?

`FetchContent` organise les fichiers téléchargés dans le répertoire de build :

```
build/
└── _deps/
    ├── googletest-src/      ← Sources téléchargées (clone Git ou archive extraite)
    ├── googletest-build/    ← Répertoire de build de la dépendance
    ├── googletest-subbuild/ ← Scripts internes de FetchContent
    ├── spdlog-src/
    ├── spdlog-build/
    └── spdlog-subbuild/
```

Ces répertoires sont entièrement contenus dans `build/` et n'apparaissent ni dans votre arbre source ni dans votre historique Git. Un `rm -rf build/` nettoie tout, y compris les dépendances téléchargées.

Le téléchargement n'a lieu qu'à la première configuration. Les exécutions suivantes de `cmake` réutilisent les sources déjà présentes (sauf si vous changez le `GIT_TAG` ou l'URL).

---

## Contrôler les options de la dépendance

Les bibliothèques téléchargées via `FetchContent` sont configurées par leur propre `CMakeLists.txt`. Elles exposent souvent des options CMake que vous pouvez vouloir modifier. La technique est de positionner les variables **avant** l'appel à `FetchContent_MakeAvailable` :

```cmake
include(FetchContent)

# Désactiver les tests de spdlog (on ne veut pas compiler les tests d'une dépendance)
set(SPDLOG_BUILD_TESTS OFF CACHE BOOL "" FORCE)  
set(SPDLOG_BUILD_EXAMPLE OFF CACHE BOOL "" FORCE)  

FetchContent_Declare(spdlog
    GIT_REPOSITORY https://github.com/gabime/spdlog.git
    GIT_TAG        v1.15.3
)
FetchContent_MakeAvailable(spdlog)
```

Le pattern `set(... CACHE BOOL "" FORCE)` est nécessaire car les options de la dépendance sont des variables de cache. Un simple `set()` local pourrait être écrasé par la valeur de cache existante. `FORCE` garantit que votre valeur prend le dessus.

### Désactiver systématiquement les tests et exemples des dépendances

C'est une bonne pratique quasi universelle : vous ne voulez pas compiler les tests, les exemples ni les benchmarks de vos dépendances. Ils alourdissent le build et n'apportent rien à votre projet. Malheureusement, il n'y a pas de convention universelle pour les noms d'options. Voici les patterns les plus courants :

```cmake
# Google Test
set(BUILD_GMOCK OFF CACHE BOOL "" FORCE)  
set(INSTALL_GTEST OFF CACHE BOOL "" FORCE)  

# spdlog
set(SPDLOG_BUILD_TESTS OFF CACHE BOOL "" FORCE)  
set(SPDLOG_BUILD_EXAMPLE OFF CACHE BOOL "" FORCE)  

# fmt
set(FMT_TEST OFF CACHE BOOL "" FORCE)  
set(FMT_DOC OFF CACHE BOOL "" FORCE)  

# nlohmann/json
set(JSON_BuildTests OFF CACHE BOOL "" FORCE)
```

Consultez le `CMakeLists.txt` racine de chaque dépendance pour identifier les noms exacts de ses options.

## `FetchContent_MakeAvailable` vs le workflow manuel

`FetchContent_MakeAvailable` est un raccourci introduit dans CMake 3.14 qui combine plusieurs opérations. Pour comprendre ce qu'il fait, voici l'équivalent détaillé :

```cmake
# Ce que FetchContent_MakeAvailable(googletest) fait en coulisse :
FetchContent_GetProperties(googletest)  
if(NOT googletest_POPULATED)  
    FetchContent_Populate(googletest)
    add_subdirectory(
        ${googletest_SOURCE_DIR}
        ${googletest_BINARY_DIR}
    )
endif()
```

Le workflow manuel est parfois nécessaire pour des cas avancés :

### Intégration conditionnelle : FetchContent comme fallback de find_package

Un pattern élégant consiste à chercher d'abord la bibliothèque sur le système, et à la télécharger uniquement si elle n'est pas trouvée :

```cmake
find_package(spdlog 1.15 QUIET)

if(NOT spdlog_FOUND)
    message(STATUS "spdlog non trouvé sur le système — téléchargement via FetchContent")
    FetchContent_Declare(spdlog
        GIT_REPOSITORY https://github.com/gabime/spdlog.git
        GIT_TAG        v1.15.3
    )
    FetchContent_MakeAvailable(spdlog)
endif()

# Dans les deux cas, spdlog::spdlog est disponible
target_link_libraries(my_lib PRIVATE spdlog::spdlog)
```

Ce pattern offre le meilleur des deux mondes : si l'utilisateur a déjà spdlog installé (via `apt`, Conan ou vcpkg), il est réutilisé sans téléchargement. Sinon, `FetchContent` prend le relais. Le code consommateur est identique dans les deux cas.

Depuis CMake 3.24, ce pattern est nativement supporté via `FIND_PACKAGE_ARGS` :

```cmake
FetchContent_Declare(spdlog
    GIT_REPOSITORY https://github.com/gabime/spdlog.git
    GIT_TAG        v1.15.3
    FIND_PACKAGE_ARGS 1.15
)
FetchContent_MakeAvailable(spdlog)
# CMake essaie d'abord find_package(spdlog 1.15),
# puis télécharge via FetchContent si find_package échoue
```

### Manipulation pré-intégration

Le workflow manuel permet d'intervenir entre le téléchargement et l'intégration :

```cmake
FetchContent_GetProperties(some_lib)  
if(NOT some_lib_POPULATED)  
    FetchContent_Populate(some_lib)

    # Patcher un fichier avant l'intégration
    file(READ ${some_lib_SOURCE_DIR}/CMakeLists.txt CONTENT)
    string(REPLACE "-Werror" "" CONTENT "${CONTENT}")
    file(WRITE ${some_lib_SOURCE_DIR}/CMakeLists.txt "${CONTENT}")

    add_subdirectory(${some_lib_SOURCE_DIR} ${some_lib_BINARY_DIR})
endif()
```

C'est un dernier recours pour corriger une bibliothèque dont le `CMakeLists.txt` pose problème (un `-Werror` en PUBLIC, par exemple). L'approche propre reste de contribuer un correctif au projet amont.

## Déclarer plusieurs dépendances

`FetchContent_MakeAvailable` accepte plusieurs noms en un seul appel :

```cmake
include(FetchContent)

FetchContent_Declare(spdlog
    GIT_REPOSITORY https://github.com/gabime/spdlog.git
    GIT_TAG        v1.15.3
)

FetchContent_Declare(googletest
    GIT_REPOSITORY https://github.com/google/googletest.git
    GIT_TAG        v1.17.0
)

FetchContent_Declare(nlohmann_json
    GIT_REPOSITORY https://github.com/nlohmann/json.git
    GIT_TAG        v3.11.3
)

# Un seul appel pour tout télécharger et intégrer
FetchContent_MakeAvailable(spdlog googletest nlohmann_json)
```

Regrouper les déclarations au début du `CMakeLists.txt` racine (ou dans un fichier dédié `cmake/Dependencies.cmake`) améliore la lisibilité. Toutes les dépendances externes sont visibles en un coup d'œil, avec leurs versions.

## Impact sur les temps de configuration et de build

`FetchContent` a des implications sur les performances qu'il faut connaître.

### Première configuration : téléchargement

La première exécution de `cmake -B build` déclenche le téléchargement de toutes les dépendances. Selon leur taille et la bande passante disponible, cela peut prendre de quelques secondes à plusieurs minutes. Les configurations suivantes sont quasi instantanées (les sources sont déjà présentes dans `build/_deps/`).

**En CI/CD** (section 38.3), ce téléchargement se produit à chaque build propre. Deux stratégies d'atténuation existent : mettre en cache le répertoire `build/_deps/` entre les jobs, ou utiliser `FETCHCONTENT_BASE_DIR` pour pointer vers un cache partagé :

```bash
cmake -B build -G Ninja -DFETCHCONTENT_BASE_DIR=/cache/fetchcontent
```

### Temps de compilation augmenté

Contrairement à `find_package()` qui utilise des bibliothèques pré-compilées, `FetchContent` intègre les sources de la dépendance dans votre build. Ces sources sont compilées à chaque build propre. Pour une bibliothèque volumineuse (Boost, Protobuf, gRPC), cela peut représenter des dizaines de minutes.

C'est un compromis explicite : la reproductibilité et l'autonomie du build contre le temps de compilation. Pour les dépendances lourdes, `find_package()` combiné à un gestionnaire de paquets (Conan, vcpkg) est souvent préférable.

### Atténuation avec ccache

L'utilisation de ccache (section 2.3) atténue considérablement le problème. Les fichiers objets des dépendances sont mis en cache après la première compilation. Les builds suivants — même après un `rm -rf build/` — retrouvent les objets en cache et la compilation est quasi instantanée :

```bash
# Configuration avec ccache
cmake -B build -G Ninja \
    -DCMAKE_C_COMPILER_LAUNCHER=ccache \
    -DCMAKE_CXX_COMPILER_LAUNCHER=ccache
```

## Pièges courants

### Piège n°1 : collision de noms de cibles

Si deux dépendances définissent des cibles avec le même nom, CMake signale une erreur. C'est rare avec les bibliothèques bien écrites (qui préfixent leurs cibles), mais peut arriver avec des projets moins rigoureux.

La solution est généralement de ne pas intégrer les deux dépendances en collision, ou de patcher l'une d'elles. `FetchContent` ne fournit pas de mécanisme d'isolation des cibles — toutes les cibles vivent dans le même espace global.

### Piège n°2 : pollution des options de cache

Les options CMake de la dépendance apparaissent dans votre cache CMake et dans l'interface de `ccmake` ou `cmake-gui`. Pour un projet avec cinq dépendances FetchContent, cela peut représenter des dizaines d'options qui n'ont rien à voir avec votre projet.

C'est un inconvénient cosmétique plus que fonctionnel, mais cela complique l'utilisation de `ccmake` pour naviguer dans les options. La variable `FETCHCONTENT_QUIET` (positionnée à `ON` par défaut) réduit le bruit dans la sortie de configuration.

### Piège n°3 : versions conflictuelles

Si votre projet et une de vos dépendances FetchContent utilisent tous les deux la même bibliothèque tierce (par exemple, spdlog), mais avec des versions différentes, vous avez un conflit. `FetchContent` suit la règle **first-come, first-served** : la première déclaration d'une dépendance (par nom) gagne. Les déclarations suivantes du même nom sont ignorées.

```cmake
# Votre projet déclare spdlog v1.15
FetchContent_Declare(spdlog GIT_TAG v1.15.3 ...)

# Une dépendance FetchContent essaie de déclarer spdlog v1.12
# → Ignoré, v1.15 est déjà déclarée
```

C'est un problème fondamental du modèle « sources intégrées » — il n'y a pas de résolution de dépendances au sens d'un gestionnaire de paquets. Pour les projets avec des graphes de dépendances complexes, un vrai gestionnaire de paquets (Conan, vcpkg) est plus adapté.

### Piège n°4 : dépendance non conçue pour add_subdirectory

`FetchContent_MakeAvailable` appelle `add_subdirectory()` en interne. Certaines bibliothèques ne sont pas conçues pour être intégrées de cette manière — elles supposent être le projet racine et utilisent `CMAKE_SOURCE_DIR` au lieu de `PROJECT_SOURCE_DIR`, ou elles installent des cibles globales de manière agressive.

Les bibliothèques C++ modernes populaires (GoogleTest, spdlog, fmt, nlohmann/json, Catch2, CLI11) sont toutes compatibles avec `FetchContent`. Pour les bibliothèques moins courantes, un test rapide révèle les problèmes éventuels.

## Dépendances FetchContent courantes

Voici les bibliothèques les plus fréquemment intégrées via `FetchContent` dans les projets C++ modernes :

```cmake
include(FetchContent)

# ── Tests ──────────────────────────────────────────────────────
FetchContent_Declare(googletest
    GIT_REPOSITORY https://github.com/google/googletest.git
    GIT_TAG        v1.17.0
)

# ── Benchmarking ───────────────────────────────────────────────
FetchContent_Declare(benchmark
    GIT_REPOSITORY https://github.com/google/benchmark.git
    GIT_TAG        v1.9.5
)
set(BENCHMARK_ENABLE_TESTING OFF CACHE BOOL "" FORCE)

# ── Logging ────────────────────────────────────────────────────
FetchContent_Declare(spdlog
    GIT_REPOSITORY https://github.com/gabime/spdlog.git
    GIT_TAG        v1.15.3
)

# ── Formatage (pré-C++23) ─────────────────────────────────────
FetchContent_Declare(fmt
    GIT_REPOSITORY https://github.com/fmtlib/fmt.git
    GIT_TAG        11.1.4
)

# ── JSON ───────────────────────────────────────────────────────
FetchContent_Declare(nlohmann_json
    GIT_REPOSITORY https://github.com/nlohmann/json.git
    GIT_TAG        v3.11.3
)

# ── CLI parsing ────────────────────────────────────────────────
FetchContent_Declare(cli11
    GIT_REPOSITORY https://github.com/CLIUtils/CLI11.git
    GIT_TAG        v2.6.0
)

FetchContent_MakeAvailable(googletest benchmark spdlog fmt nlohmann_json cli11)
```

> 💡 **Conseil** : ces déclarations peuvent être extraites dans un fichier `cmake/Dependencies.cmake` inclus depuis le `CMakeLists.txt` racine, pour garder ce dernier concis :  
>
> ```cmake
> # CMakeLists.txt racine
> include(cmake/Dependencies.cmake)
> ```

## Récapitulatif

| Aspect | Recommandation |
|--------|---------------|
| Source | Dépôt Git avec `GIT_TAG` fixe (tag ou hash), ou archive avec `URL_HASH` |
| Version | Toujours un tag ou un hash, jamais une branche |
| Gros dépôts | Activer `GIT_SHALLOW ON` |
| Options de la dépendance | Désactiver tests, exemples et documentation via `set(... CACHE BOOL "" FORCE)` |
| Fallback de find_package | Utiliser `FIND_PACKAGE_ARGS` (CMake 3.24+) ou le pattern find-then-fetch |
| Performances CI | Mettre en cache `_deps/` ou utiliser `FETCHCONTENT_BASE_DIR` + ccache |
| Dépendances lourdes | Préférer `find_package()` + gestionnaire de paquets |
| Dépendances légères / tests | `FetchContent` est idéal |
| Organisation | Regrouper les déclarations dans un fichier `cmake/Dependencies.cmake` |
