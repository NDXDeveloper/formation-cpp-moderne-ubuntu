# 29.4.4 — MemorySanitizer (`-fsanitize=memory`)

## Section 29.4 : Sanitizers · Chapitre 29

---

## Introduction

Quand vous déclarez une variable locale en C++ sans l'initialiser, sa valeur est indéterminée. Elle contient ce qui traînait sur la pile à cet emplacement — un résidu d'un appel de fonction précédent, un fragment d'adresse de retour, des octets aléatoires. Lire cette valeur est un comportement indéfini.

Le problème, c'est que ça fonctionne presque toujours. La valeur indéterminée est un nombre — pas une erreur, pas un crash, pas un signal. Le programme continue avec cette valeur et produit un résultat. Un résultat faux, certes, mais qui peut paraître plausible. Et comme la valeur dépend de l'état de la pile, elle change selon le chemin d'exécution, le niveau d'optimisation, le compilateur, et la phase de la lune. Le bug se manifeste de manière intermittente, avec des symptômes différents à chaque occurrence.

MemorySanitizer — MSan — traque chaque octet de mémoire et signale toute lecture d'un octet qui n'a jamais été explicitement écrit. Il ne vérifie pas si la valeur est "raisonnable" — il vérifie si elle a été initialisée. C'est une distinction fondamentale : une variable initialisée à 0 puis jamais modifiée est correcte (même si sa valeur est 0). Une variable jamais initialisée qui vaut 0 par chance est un bug.

MSan est le sanitizer le plus spécialisé et le plus exigeant des quatre. Il n'est disponible que sous Clang, il nécessite que l'ensemble du code — y compris les bibliothèques — soit compilé avec MSan, et son surcoût est significatif. Mais quand il détecte un bug, c'est un bug que rien d'autre n'aurait trouvé.

---

## Activation

MSan est exclusif à Clang. GCC ne le supporte pas.

```bash
clang++ -fsanitize=memory -g -O1 -fno-omit-frame-pointer -o prog main.cpp
```

### Pourquoi Clang uniquement ?

MSan a été développé par Google dans le cadre du projet LLVM/compiler-rt. Son implémentation est profondément intégrée à l'infrastructure de Clang. Porter MSan vers GCC nécessiterait un effort considérable pour un gain marginal — les développeurs qui ont besoin de MSan utilisent Clang pour ce build spécifique, même si leur build de production utilise GCC.

En pratique, ce n'est pas un obstacle : dans un pipeline CI, rien n'empêche d'avoir un job MSan compilé avec Clang en parallèle d'un job ASan compilé avec GCC.

### Incompatibilités

MSan est incompatible avec ASan et TSan. C'est un troisième build séparé :

```bash
# Build 1 : ASan + UBSan (GCC ou Clang)
g++ -fsanitize=address,undefined -g -O1 -fno-omit-frame-pointer -o prog-asan main.cpp

# Build 2 : TSan (GCC ou Clang)
g++ -fsanitize=thread -g -O1 -fno-omit-frame-pointer -o prog-tsan main.cpp

# Build 3 : MSan (Clang uniquement)
clang++ -fsanitize=memory -g -O1 -fno-omit-frame-pointer -o prog-msan main.cpp
```

### Surcoût de performance

| Métrique | Surcoût typique |
|---|---|
| Temps d'exécution | ~3x plus lent |
| Mémoire | ~2-3x plus de mémoire |
| Taille du binaire | ~2x plus gros |

Le surcoût est comparable à celui d'ASan. La mécanique est similaire — shadow memory pour traquer l'état d'initialisation de chaque octet — mais le volume de vérifications est potentiellement plus grand car MSan vérifie chaque lecture, pas seulement les accès aux zones spéciales (redzones, quarantaine).

## Comment MSan fonctionne

### Shadow memory : initialisé vs non initialisé

MSan associe un **bit d'ombre** à chaque octet de mémoire applicative. Ce bit indique si l'octet a été initialisé (écrit au moins une fois) ou non :

- **Octet initialisé** (shadow = 0) : a été écrit par le programme. Sa lecture est valide.
- **Octet non initialisé** (shadow = 1) : n'a jamais été écrit. Sa lecture est un bug.

Quand le programme alloue de la mémoire (variable locale, `new`, `malloc`), MSan marque tous les octets correspondants comme non initialisés. Quand le programme écrit dans cette mémoire, MSan marque les octets écrits comme initialisés. Quand le programme lit un octet, MSan vérifie son état d'ombre.

### Propagation de l'état non initialisé

MSan ne signale pas une erreur dès qu'un octet non initialisé est lu. Il **propage** l'état non initialisé à travers les calculs. Si `a` est non initialisé et que vous calculez `b = a + 1`, alors `b` est aussi marqué comme "dérivé de mémoire non initialisée". L'erreur n'est signalée que quand la valeur non initialisée est **utilisée de manière observable** :

- Comme opérande d'une instruction de branchement (`if`, `while`, `switch`)
- Comme argument d'un appel système
- Comme adresse d'un accès mémoire

Cette propagation réduit les faux positifs. Beaucoup de code copie de la mémoire non initialisée de manière bénigne (par exemple, `memcpy` d'une structure avec du padding — le padding n'est pas initialisé mais n'est jamais utilisé). MSan ne signale pas ces copies ; il signale uniquement le moment où la valeur non initialisée influence réellement le comportement du programme.

```
┌─────────────────────────────────────────────────────────────┐
│  int x;           // MSan : x marqué non initialisé         │
│  int y = x;       // MSan : y marqué non initialisé (copie) │
│  int z = y + 1;   // MSan : z marqué non initialisé (dérivé)│
│                                                             │
│  if (z > 0) {     // MSan : ERREUR — branchement sur valeur │
│                   //        non initialisée                 │
│  }                                                          │
└─────────────────────────────────────────────────────────────┘
```

L'erreur est signalée au `if`, pas au `int x;`. C'est le point où la valeur non initialisée a un impact observable.

## Exemple fondamental

### Code avec lecture de mémoire non initialisée

```cpp
// uninit_read.cpp
#include <cstdio>

int compute(int mode) {
    int result;
    
    if (mode == 1) {
        result = 42;
    } else if (mode == 2) {
        result = 84;
    }
    // Bug : si mode != 1 et mode != 2, result n'est pas initialisé
    
    return result;
}

int main() {
    int value = compute(3);    // Déclenche le chemin non initialisé
    
    if (value > 50) {          // Branchement sur valeur non initialisée
        std::printf("Grand : %d\n", value);
    } else {
        std::printf("Petit : %d\n", value);
    }
    
    return 0;
}
```

### Compilation et exécution

```bash
clang++ -fsanitize=memory -g -O1 -fno-omit-frame-pointer -o uninit_read uninit_read.cpp
./uninit_read
```

### Rapport MSan

```
==54321==WARNING: MemorySanitizer: use-of-uninitialized-value
    #0 0x4f4c10 in main uninit_read.cpp:18
    #1 0x7f8a12345890 in __libc_start_call_main (...)
    #2 0x7f8a12345949 in __libc_start_main (...)

  Uninitialized value was created by an allocation of 'result' in the stack frame 
  of function 'compute'
    #0 0x4f4b00 in compute(int) uninit_read.cpp:4

SUMMARY: MemorySanitizer: use-of-uninitialized-value uninit_read.cpp:18 in main
```

### Lecture du rapport

**La première pile d'appels** montre où la valeur non initialisée est utilisée : `uninit_read.cpp:18` — le `if (value > 50)`. C'est le point d'impact observable.

**Le bloc "Uninitialized value was created by"** montre l'origine : la variable `result` dans la fonction `compute` (ligne 4). MSan a retracé la propagation depuis l'allocation non initialisée jusqu'à son utilisation dans le branchement.

Les deux informations ensemble donnent le diagnostic complet : `result` n'est pas initialisé quand `mode != 1 && mode != 2`, et cette valeur non initialisée remonte jusqu'au `if` dans `main`.

### Traquer l'origine avec `-fsanitize-memory-track-origins`

Par défaut, MSan indique où la valeur non initialisée est utilisée, mais pas toujours où elle a été créée (le bloc "Uninitialized value was created by" n'est pas toujours présent, surtout pour des chaînes de propagation longues). Le flag `-fsanitize-memory-track-origins` active un suivi complet de l'origine :

```bash
# Niveau 1 : traquer l'allocation d'origine
clang++ -fsanitize=memory -fsanitize-memory-track-origins \
    -g -O1 -fno-omit-frame-pointer -o prog main.cpp

# Niveau 2 : traquer aussi les copies et propagations intermédiaires
clang++ -fsanitize=memory -fsanitize-memory-track-origins=2 \
    -g -O1 -fno-omit-frame-pointer -o prog main.cpp
```

Le niveau 2 est plus coûteux (surcoût supplémentaire de ~1.5-2x) mais produit des rapports beaucoup plus détaillés pour les chaînes de propagation longues. C'est le niveau recommandé quand le rapport de base ne suffit pas à identifier l'origine du problème.

## Patterns courants de lectures non initialisées

### Pattern 1 : Variable locale sans valeur par défaut

Le cas le plus fréquent — une variable déclarée sans initialisation dans un chemin conditionnel incomplet :

```cpp
int classify(int input) {
    int category;              // Non initialisé
    if (input > 100)
        category = 3;
    else if (input > 50)
        category = 2;
    else if (input > 0)
        category = 1;
    // Pas de else → category non initialisé si input <= 0
    return category;
}
```

La correction est simple — initialisez la variable ou ajoutez le `else` manquant :

```cpp
int classify(int input) {
    int category = 0;          // Initialisé par défaut
    if (input > 100)
        category = 3;
    else if (input > 50)
        category = 2;
    else if (input > 0)
        category = 1;
    return category;
}
```

### Pattern 2 : Struct avec des membres non initialisés

```cpp
struct Packet {
    uint32_t src_addr;
    uint32_t dst_addr;
    uint16_t src_port;
    uint16_t dst_port;
    uint8_t  protocol;
    uint8_t  flags;
    // 2 octets de padding implicite
};

void send_packet() {
    Packet pkt;
    pkt.src_addr = 0x0A000001;
    pkt.dst_addr = 0x0A000002;
    pkt.src_port = 8080;
    pkt.dst_port = 80;
    pkt.protocol = 6;    // TCP
    // Bug : flags non initialisé
    
    transmit(&pkt, sizeof(pkt));    // Envoie des octets non initialisés
}
```

MSan détecte que `flags` (et le padding) ne sont pas initialisés quand `transmit` les utilise. La correction :

```cpp
void send_packet() {
    Packet pkt{};    // Initialisation zéro de toute la structure
    pkt.src_addr = 0x0A000001;
    // ... reste des champs ...
}
```

L'initialisation par `{}` met à zéro tous les membres, y compris le padding. C'est la pratique recommandée pour les structures envoyées sur le réseau ou écrites sur disque.

### Pattern 3 : Buffer partiellement rempli

```cpp
void read_config(char* buffer, size_t buffer_size) {
    char temp[256];
    size_t bytes_read = read_from_file(temp, sizeof(temp));
    
    // Bug potentiel : si bytes_read < buffer_size,
    // les octets après bytes_read ne sont pas initialisés
    std::memcpy(buffer, temp, buffer_size);
}
```

Si `read_from_file` ne remplit que 100 octets de `temp` mais que `buffer_size` est 256, les 156 octets restants sont copiés non initialisés. MSan détecte le problème quand `buffer` est ensuite lu au-delà de la position 100.

```cpp
void read_config(char* buffer, size_t buffer_size) {
    char temp[256] = {};    // Initialisation zéro
    size_t bytes_read = read_from_file(temp, sizeof(temp));
    
    size_t copy_size = std::min(bytes_read, buffer_size);
    std::memcpy(buffer, temp, copy_size);
}
```

### Pattern 4 : Allocation heap non initialisée

```cpp
int* data = new int[100];       // Non initialisé (contrairement à std::vector)  
int sum = 0;  
for (int i = 0; i < 100; ++i) {  
    sum += data[i];             // Lecture de mémoire non initialisée
}
```

`new int[100]` n'initialise pas le tableau. `new int[100]()` (avec les parenthèses) l'initialise à zéro. La solution moderne est d'utiliser `std::vector<int>(100, 0)` ou simplement `std::vector<int>(100)` (qui value-initialize les éléments à 0 pour les types primitifs).

### Pattern 5 : Réutilisation d'un buffer après resize

```cpp
std::vector<char> buffer(1024);
// ... buffer rempli avec des données ...

buffer.resize(2048);
// Les 1024 nouveaux octets sont value-initialized (0 pour char)
// → Pas de problème ici avec std::vector

// Mais avec realloc :
char* raw = (char*)std::malloc(1024);
// ... rempli ...
raw = (char*)std::realloc(raw, 2048);
// Les 1024 nouveaux octets ne sont PAS initialisés
process(raw, 2048);    // MSan détecte la lecture des octets 1024-2047
```

## La contrainte majeure : tout doit être compilé avec MSan

C'est le point qui distingue MSan de tous les autres sanitizers et qui limite son adoption en pratique.

### Le problème

MSan traque l'état d'initialisation de chaque octet via la shadow memory. Pour que ce suivi soit correct, **chaque écriture en mémoire doit être interceptée par MSan** afin de marquer les octets comme initialisés. Si une bibliothèque non instrumentée écrit en mémoire, MSan ne voit pas l'écriture — les octets restent marqués "non initialisés" dans la shadow memory, alors qu'ils ont été écrits par la bibliothèque.

Le résultat : des faux positifs massifs. Chaque appel à une fonction de la libc, de la libstdc++, ou de toute bibliothèque système produit des données que MSan considère comme non initialisées.

### La solution : recompiler avec MSan

Pour un fonctionnement correct, l'ensemble de l'arbre de dépendances doit être compilé avec `-fsanitize=memory` :

- Votre code
- La bibliothèque standard C++ (libstdc++ ou libc++)
- La bibliothèque standard C (libc)
- Toutes les bibliothèques tierces

En pratique, la méthode la plus viable est d'utiliser **libc++ instrumentée** (la bibliothèque standard C++ de LLVM), qui peut être compilée avec MSan :

```bash
# Compiler avec libc++ instrumentée MSan
clang++ -fsanitize=memory \
    -stdlib=libc++ \
    -g -O1 -fno-omit-frame-pointer \
    -o prog main.cpp
```

La libc++ de LLVM est compatible avec MSan. La libstdc++ de GCC ne l'est pas (sauf si vous la recompilez vous-même avec MSan, ce qui est un effort considérable).

### Construire un environnement MSan complet

Pour les projets sérieux, la méthode recommandée est de construire une toolchain MSan dédiée. Le projet LLVM fournit des scripts pour cela :

```bash
# Compiler libc++ avec MSan (simplifié)
cmake -S llvm-project/runtimes -B build-msan-libcxx \
    -DCMAKE_C_COMPILER=clang \
    -DCMAKE_CXX_COMPILER=clang++ \
    -DLLVM_ENABLE_RUNTIMES="libcxx;libcxxabi;libunwind" \
    -DLLVM_USE_SANITIZER=MemoryWithOrigins

cmake --build build-msan-libcxx

# Utiliser la libc++ MSan pour compiler votre programme
clang++ -fsanitize=memory -fsanitize-memory-track-origins \
    -stdlib=libc++ \
    -I/path/to/build-msan-libcxx/include/c++/v1 \
    -L/path/to/build-msan-libcxx/lib \
    -Wl,-rpath,/path/to/build-msan-libcxx/lib \
    -g -O1 -fno-omit-frame-pointer \
    -o prog main.cpp
```

C'est un investissement initial significatif, mais une fois la toolchain en place, elle se réutilise pour tous les builds MSan du projet.

### Quand la recompilation complète est impossible

Si vous ne pouvez pas recompiler toutes les dépendances avec MSan (bibliothèques propriétaires, dépendances système complexes), vous pouvez atténuer les faux positifs de deux manières :

**Intercepteurs intégrés.** MSan inclut des intercepteurs pour les fonctions les plus courantes de la libc (`memcpy`, `strlen`, `read`, `write`, etc.). Ces intercepteurs marquent correctement la mémoire écrite par ces fonctions. La libc standard est donc partiellement couverte sans recompilation.

**Le flag `-fsanitize-memory-use-after-dtor`.** Détecte les accès à des membres d'un objet après l'appel de son destructeur — un cas qui échappe à ASan quand la mémoire n'est pas encore libérée.

```bash
clang++ -fsanitize=memory -fsanitize-memory-use-after-dtor \
    -g -O1 -fno-omit-frame-pointer -o prog main.cpp
```

## MSan et GDB

Comme pour les autres sanitizers, MSan se combine avec GDB :

```bash
MSAN_OPTIONS="halt_on_error=1" gdb ./prog-msan
(gdb) run
# MSan détecte une lecture non initialisée et appelle abort()
(gdb) backtrace
(gdb) info locals
```

Ou via un breakpoint sur le handler MSan :

```
(gdb) break __msan_warning_with_origin
(gdb) run
```

GDB vous arrête à l'instant exact de la lecture non initialisée. Vous pouvez inspecter les variables environnantes pour comprendre pourquoi la variable n'a pas été initialisée — quel chemin conditionnel a été emprunté, quelle boucle ne s'est pas exécutée, quel appel de fonction n'a pas rempli le buffer.

## Options d'exécution

```bash
# Configuration recommandée pour le CI
MSAN_OPTIONS="halt_on_error=1:log_path=/tmp/msan:print_stats=1"

# Configuration pour le développement (plus de contexte)
MSAN_OPTIONS="halt_on_error=1:poison_in_dtor=1"
```

| Option | Défaut | Effet |
|---|---|---|
| `halt_on_error` | `0` | `1` = arrêter à la première erreur |
| `log_path` | stderr | Redirige les rapports vers un fichier |
| `poison_in_dtor` | `0` | `1` = empoisonner la mémoire des objets dans leur destructeur |
| `print_stats` | `0` | `1` = afficher les statistiques de shadow memory à la fin |
| `wrap_signals` | `1` | Intercepter les handlers de signaux |
| `exitcode` | `77` | Code de retour quand une erreur est détectée |

L'option `poison_in_dtor` est intéressante : elle marque la mémoire d'un objet comme non initialisée après l'appel de son destructeur. Toute lecture ultérieure (use-after-destroy) est détectée. C'est un filet de sécurité pour les cas où l'objet est détruit mais la mémoire n'est pas encore libérée (par exemple, quand l'objet est dans un pool de mémoire).

```cmake
option(ENABLE_MSAN "Enable MemorySanitizer (Clang only)" OFF)

if(ENABLE_MSAN)
    # Vérifier que le compilateur est Clang
    if(NOT CMAKE_CXX_COMPILER_ID STREQUAL "Clang")
        message(FATAL_ERROR "MemorySanitizer requires Clang")
    endif()
    
    # Vérifier l'incompatibilité avec ASan et TSan
    if(ENABLE_ASAN)
        message(FATAL_ERROR "MSan and ASan cannot be enabled simultaneously")
    endif()
    if(ENABLE_TSAN)
        message(FATAL_ERROR "MSan and TSan cannot be enabled simultaneously")
    endif()
    
    add_compile_options(
        -fsanitize=memory
        -fsanitize-memory-track-origins=2
        -fno-omit-frame-pointer
    )
    add_link_options(-fsanitize=memory)
    
    # Recommandé : utiliser libc++ pour une couverture MSan complète
    add_compile_options(-stdlib=libc++)
    add_link_options(-stdlib=libc++)
endif()
```

## Alternatives à MSan pour détecter les lectures non initialisées

Étant donné la contrainte de recompilation complète, MSan n'est pas toujours pratique. Voici les alternatives, avec leurs compromis :

### Valgrind (Memcheck)

Valgrind détecte les lectures de mémoire non initialisée sans recompilation — il instrumente le binaire dynamiquement. C'est souvent le choix pragmatique quand MSan n'est pas viable.

```bash
valgrind --track-origins=yes ./prog
```

Avantages : fonctionne avec n'importe quel binaire, pas de recompilation. Inconvénient : 20-50x de ralentissement, détection parfois moins fine que MSan.

> 📎 *Valgrind est couvert en détail au chapitre 30.*

### Compilation avec `-Wuninitialized` et `-Wmaybe-uninitialized`

Les warnings du compilateur détectent statiquement certains cas de lectures non initialisées :

```bash
g++ -Wall -Wextra -Wuninitialized -Wmaybe-uninitialized -O1 -o prog main.cpp
```

Le flag `-O1` (ou supérieur) est nécessaire : l'analyse de flux de données qui détecte les variables non initialisées ne fonctionne qu'avec les optimisations activées.

Avantages : détection à la compilation, pas de surcoût à l'exécution. Inconvénient : couverture partielle — le compilateur ne peut pas toujours prouver statiquement qu'un chemin mène à une lecture non initialisée, surtout à travers des appels de fonctions.

### Initialisation systématique

L'approche préventive : initialisez toujours vos variables.

```cpp
int x = 0;                          // Scalaire : valeur par défaut  
Packet pkt{};                       // Struct : initialisation zéro  
std::vector<int> data(100, 0);      // Conteneur : valeur initiale  
auto ptr = std::make_unique<Config>();  // Smart pointer : constructeur  
```

Les C++ Core Guidelines recommandent explicitement cette pratique (règle ES.20 : "Always initialize an object"). Combinée avec le flag GCC/Clang `-ftrivial-auto-var-init=pattern` (qui initialise les variables locales non initialisées à un pattern détectable), c'est une défense en profondeur :

```bash
# Initialiser automatiquement les variables locales non initialisées
clang++ -ftrivial-auto-var-init=pattern -g -O1 -o prog main.cpp
# Les variables non initialisées reçoivent un pattern reconnaissable (0xAA...)
# au lieu d'une valeur aléatoire de la pile
```

Ce flag ne corrige pas le bug (la variable est toujours "initialisée" à une valeur absurde), mais il rend le symptôme reproductible et facilite le diagnostic.

## Quand utiliser MSan

MSan n'est pas un sanitizer que vous activez par défaut au quotidien — contrairement à ASan et UBSan. Ses contraintes de déploiement (Clang uniquement, recompilation de toutes les dépendances) en font un outil de vérification périodique plutôt qu'un compagnon permanent.

### Recommandations d'usage

**En CI, si vous pouvez maintenir un build MSan** : ajoutez un job MSan à votre pipeline. La couverture de détection justifie l'effort de maintenance de la toolchain MSan.

**Ponctuellement, lors d'audits de qualité** : quand vous révisez un composant critique (parser, protocole réseau, sérialisation), compilez-le avec MSan et exécutez les tests. Ces composants sont les plus susceptibles de manipuler des buffers partiellement remplis.

**Systématiquement pour du code nouveau** : quand vous démarrez un nouveau projet ou un nouveau module, intégrer MSan dès le début (avec libc++) est beaucoup plus facile que de l'ajouter rétroactivement.

**En complément de Valgrind** : MSan et Valgrind (Memcheck) détectent des catégories similaires de bugs mais avec des techniques différentes. Certains cas sont détectés par l'un et pas l'autre. Les deux outils sont complémentaires.

## Récapitulatif des quatre sanitizers

Avec cette section, la couverture des quatre sanitizers est complète. Voici la vue d'ensemble pour guider votre choix :

| Sanitizer | Flag | Détecte | Surcoût CPU | Compilateur | Priorité |
|---|---|---|---|---|---|
| ASan | `-fsanitize=address` | Buffer overflow, use-after-free, double-free, leaks | ~2x | GCC, Clang | **Essentiel** |
| UBSan | `-fsanitize=undefined` | Comportements indéfinis (overflow signé, shift, null, etc.) | ~1.1-1.2x | GCC, Clang | **Essentiel** |
| TSan | `-fsanitize=thread` | Data races, deadlocks potentiels | ~5-15x | GCC, Clang | **Essentiel si multithreadé** |
| MSan | `-fsanitize=memory` | Lectures de mémoire non initialisée | ~3x | Clang seul | **Recommandé si viable** |

**La configuration minimale** pour tout projet C++ sérieux : ASan + UBSan en développement et CI, TSan en CI pour le code concurrent. MSan quand la toolchain le permet.

**La configuration maximale** : les trois builds en parallèle dans le CI, chacun exécutant la suite de tests complète. Tout rapport de sanitizer fait échouer le pipeline. Zéro tolérance.

---

> **À retenir** : MSan détecte les lectures de mémoire non initialisée — des bugs silencieux qui produisent des résultats imprévisibles sans crash ni warning. Sa contrainte principale (tout doit être compilé avec MSan) limite son adoption, mais les alternatives existent : Valgrind pour la détection dynamique, `-Wuninitialized` pour la détection statique, et l'initialisation systématique comme pratique préventive. Quand MSan est viable, il trouve des bugs que rien d'autre ne trouve.
