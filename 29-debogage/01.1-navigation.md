# 29.1.1 — Navigation : run, step, next, continue

## Section 29.1 : GDB — Commandes essentielles et breakpoints · Chapitre 29

---

## Introduction

Naviguer dans un programme avec GDB, c'est contrôler le temps. Vous décidez quand le programme démarre, où il s'arrête, et à quelle granularité il avance — ligne par ligne, instruction par instruction, ou d'un bond jusqu'au prochain point d'intérêt.

Cette sous-section couvre les commandes de contrôle d'exécution. Ce sont les premières que vous utiliserez dans toute session GDB, et celles que vous taperez le plus souvent. Elles se répartissent en trois groupes : démarrer le programme, avancer dans le code, et gérer les points d'arrêt de base.

Tout au long de cette section, nous utilisons le programme `config_parser.cpp` introduit en section 29.1. Si vous ne l'avez pas encore compilé :

```bash
g++ -ggdb3 -O0 -std=c++20 -Wall -Wextra -o config_parser config_parser.cpp
```

---

## Démarrer et arrêter l'exécution

### `run` — Lancer le programme

La commande `run` (abrégée `r`) démarre l'exécution du programme depuis le début :

```
$ gdb ./config_parser
(gdb) run database.conf
Starting program: /home/dev/config_parser database.conf  
Entrées parsées : 5  
[L2] db_host = localhost
[L3] db_port = 5432
[L4] db_name = production
[L6] timeout = 30
[L7] max_connections = 100
Connexion à : localhost
[Inferior 1 (process 12345) exited normally]
```

Les arguments passés à `run` sont transmis au programme comme `argv`. Si vous avez lancé GDB avec `gdb --args ./config_parser database.conf`, un simple `run` sans arguments reprend ceux spécifiés au lancement.

Quelques variantes utiles :

```
(gdb) run                          # Lance avec les derniers arguments utilisés
(gdb) run database.conf --verbose  # Lance avec de nouveaux arguments
(gdb) run < input.txt              # Redirige stdin
(gdb) run > /dev/null 2>&1         # Redirige stdout et stderr
```

Si le programme est déjà en cours d'exécution (arrêté sur un breakpoint, par exemple), `run` le relance depuis le début. GDB demande une confirmation :

```
(gdb) run
The program being debugged has been started already.  
Start it from the beginning? (y or n) y  
```

### `start` — Lancer et s'arrêter à main()

La commande `start` est un raccourci qui combine `break main` et `run`. Elle lance le programme et s'arrête automatiquement à la première ligne de `main()` :

```
(gdb) start database.conf
Temporary breakpoint 1, main (argc=2, argv=0x7fffffffe3a8)
    at config_parser.cpp:48
48          if (argc < 2) {
```

C'est le point de départ le plus courant pour une session de débogage exploratoire — vous voulez comprendre le flux d'exécution depuis le début sans savoir encore où poser vos breakpoints.

### `kill` — Arrêter le programme

```
(gdb) kill
Kill the program being debugged? (y or n) y
[Inferior 1 (process 12345) killed]
```

Le programme est arrêté, mais GDB reste ouvert. Vous pouvez poser de nouveaux breakpoints et relancer avec `run`.

### `quit` — Quitter GDB

```
(gdb) quit
```

Abrégé `q`. Si un programme est en cours d'exécution, GDB demande confirmation avant de le tuer et de quitter.

## Poser des breakpoints

Avant d'avancer pas à pas, il faut d'abord arrêter le programme là où ça vous intéresse. Un breakpoint est un marqueur qui dit à GDB : "quand l'exécution atteint ce point, mets le programme en pause".

### `break` — Poser un breakpoint

La commande `break` (abrégée `b`) accepte plusieurs formes :

```
(gdb) break main                        # Sur une fonction
(gdb) break config_parser.cpp:25        # Sur un fichier:ligne
(gdb) break 25                          # Sur une ligne du fichier courant
(gdb) break parse_config                # Sur la première ligne d'une fonction
(gdb) break ConfigEntry::ConfigEntry    # Sur un constructeur
```

Quand un breakpoint est posé, GDB confirme :

```
(gdb) break parse_config
Breakpoint 1 at 0x5555555552a1: file config_parser.cpp, line 16.
```

Chaque breakpoint reçoit un numéro (ici, 1). Ce numéro sert à le manipuler par la suite.

### `info breakpoints` — Lister les breakpoints

```
(gdb) info breakpoints
Num     Type           Disp Enb Address            What
1       breakpoint     keep y   0x5555555552a1     in parse_config(...)
                                                   at config_parser.cpp:16
2       breakpoint     keep y   0x555555555401     in find_value(...)
                                                   at config_parser.cpp:40
```

Les colonnes importantes : `Num` (identifiant), `Enb` (y = activé, n = désactivé), et `What` (localisation dans le source).

### `delete`, `disable`, `enable` — Gérer les breakpoints

```
(gdb) disable 2          # Désactive le breakpoint 2 (sans le supprimer)
(gdb) enable 2           # Réactive le breakpoint 2
(gdb) delete 1           # Supprime le breakpoint 1
(gdb) delete             # Supprime TOUS les breakpoints (avec confirmation)
```

La distinction entre `disable` et `delete` est importante. Quand vous investiguez un bug, vous posez souvent plusieurs breakpoints puis vous en désactivez certains temporairement pour vous concentrer sur une zone. `disable` préserve le breakpoint — vous pouvez le réactiver plus tard sans retaper la localisation.

### `tbreak` — Breakpoint temporaire

```
(gdb) tbreak parse_config
Temporary breakpoint 3 at 0x5555555552a1: file config_parser.cpp, line 16.
```

Un breakpoint temporaire se supprime automatiquement après le premier arrêt. Utile quand vous voulez vous arrêter une seule fois à un endroit — par exemple, pour observer l'état à l'entrée d'une fonction — sans polluer votre liste de breakpoints.

## Avancer dans le code

Une fois le programme arrêté sur un breakpoint, vous avez quatre commandes principales pour contrôler l'avancement. Leur différence tient à la granularité du pas.

### `next` — Ligne suivante, sans entrer dans les fonctions

La commande `next` (abrégée `n`) exécute la ligne courante et s'arrête à la ligne suivante dans la même fonction. Si la ligne contient un appel de fonction, la fonction s'exécute entièrement — vous ne "descendez" pas dedans.

```
(gdb) start database.conf
48          if (argc < 2) {
(gdb) next
53          auto entries = parse_config(argv[1]);
(gdb) next
54          std::cout << "Entrées parsées : " << entries.size() << "\n";
```

À la ligne 53, `next` a exécuté `parse_config` en entier et vous a placé sur la ligne suivante. Vous n'avez pas vu les instructions internes de `parse_config` — vous avez sauté par-dessus.

`next` est la commande par défaut quand vous parcourez le flux principal d'une fonction. Vous l'utilisez pour observer ce qui se passe à un niveau d'abstraction donné, sans descendre dans les détails de chaque sous-fonction.

### `step` — Ligne suivante, en entrant dans les fonctions

La commande `step` (abrégée `s`) exécute la ligne courante, mais si cette ligne contient un appel de fonction, GDB entre dans cette fonction et s'arrête à sa première ligne :

```
(gdb) start database.conf
48          if (argc < 2) {
(gdb) next
53          auto entries = parse_config(argv[1]);
(gdb) step
parse_config (filename="database.conf") at config_parser.cpp:16
16          std::vector<ConfigEntry> entries;
```

Vous êtes maintenant à l'intérieur de `parse_config`. Chaque `step` ou `next` supplémentaire vous fera avancer dans cette fonction.

La distinction entre `step` et `next` est la décision la plus fréquente en débogage interactif : est-ce que le bug est dans cette fonction (→ `step`) ou ailleurs (→ `next`) ? En général, vous commencez par `next` pour localiser la zone problématique, puis vous relancez avec `step` à l'endroit suspect pour descendre dans les détails.

#### Piège courant : `step` dans la bibliothèque standard

Si la ligne courante contient un appel à une fonction de la STL, `step` vous emmène dans le code source de la bibliothèque standard :

```
(gdb) step
std::basic_ifstream<char>::basic_ifstream (this=0x7fffffffe0a0, ...) 
    at /usr/include/c++/15/fstream:582
582         { this->init(&_M_filebuf); }
```

Ce n'est presque jamais ce que vous voulez. Pour revenir à votre code, utilisez `finish` (voir ci-dessous) ou configurez GDB pour ignorer les sources de la bibliothèque standard :

```
(gdb) skip -gfi /usr/include/c++/*
```

Cette commande dit à GDB de ne jamais entrer dans les fichiers situés sous `/usr/include/c++/`. Vous pouvez l'ajouter à votre `~/.gdbinit` pour en bénéficier dans toutes vos sessions :

```bash
# ~/.gdbinit — Ignorer les sources de la STL lors du step
skip -gfi /usr/include/c++/*  
skip -gfi /usr/include/x86_64-linux-gnu/c++/*  
```

### `continue` — Reprendre l'exécution normale

La commande `continue` (abrégée `c`) reprend l'exécution normale du programme. Il s'exécute à pleine vitesse jusqu'au prochain breakpoint rencontré, ou jusqu'à sa terminaison :

```
(gdb) break find_value
Breakpoint 2 at 0x555555555401: file config_parser.cpp, line 40.
(gdb) continue
Continuing.  
Entrées parsées : 5  
[L2] db_host = localhost
[... affichage ...]

Breakpoint 2, find_value (entries=..., key="db_host")
    at config_parser.cpp:40
40          for (const auto& entry : entries) {
```

Le programme a tourné normalement (la boucle d'affichage s'est exécutée) puis s'est arrêté en atteignant le breakpoint dans `find_value`.

`continue` est ce que vous utilisez quand vous avez fini d'inspecter une zone et que vous voulez avancer rapidement jusqu'au prochain point d'intérêt. La combinaison breakpoints + `continue` est la façon la plus efficace de naviguer dans un programme — bien plus rapide que de faire `next` des centaines de fois.

### `finish` — Exécuter jusqu'au retour de la fonction

La commande `finish` (abrégée `fin`) exécute le reste de la fonction courante et s'arrête juste après son retour dans la fonction appelante :

```
(gdb) step
parse_config (filename="database.conf") at config_parser.cpp:16
16          std::vector<ConfigEntry> entries;
(gdb) finish
Run till exit from #0  parse_config (filename="database.conf")
    at config_parser.cpp:16
0x0000555555555512 in main (argc=2, argv=0x7fffffffe3a8)
    at config_parser.cpp:53
53          auto entries = parse_config(argv[1]);
Value returned is $1 = std::vector of length 5, capacity 8 = { ... }
```

GDB vous montre la valeur de retour. C'est extrêmement pratique : vous êtes entré dans une fonction avec `step`, vous avez constaté qu'elle ne contenait pas le bug, et `finish` vous ramène à l'appelant sans devoir parcourir tout le reste de la fonction ligne par ligne.

### `until` — Avancer jusqu'à une ligne donnée

La commande `until` (abrégée `u`) exécute le programme jusqu'à atteindre une ligne spécifiée :

```
(gdb) until 35
```

Sans argument, `until` avance jusqu'à la prochaine ligne dont le numéro est supérieur au numéro courant. C'est particulièrement utile pour sortir d'une boucle sans poser de breakpoint après :

```
(gdb) next
22              while (std::getline(file, line)) {
(gdb) next
23                  ++line_num;
(gdb) until 37
```

Ici, vous étiez à l'intérieur de la boucle `while` de `parse_config`. Au lieu de faire `next` à travers chaque itération (il y en a 7 pour notre fichier de test), `until 37` avance l'exécution jusqu'à la ligne 37, après la boucle. C'est un gain de temps considérable sur les boucles longues.

## Naviguer dans la pile d'appels

Quand le programme est en pause, vous n'êtes pas limité à observer la fonction courante. Vous pouvez remonter dans la pile d'appels pour examiner le contexte des fonctions appelantes.

### `backtrace` — Afficher la pile d'appels

La commande `backtrace` (abrégée `bt`) affiche la pile d'appels complète :

```
(gdb) backtrace
#0  find_value (entries=..., key="db_host") at config_parser.cpp:41
#1  0x0000555555555512 in main (argc=2, argv=0x7fffffffe3a8)
    at config_parser.cpp:60
```

Le frame `#0` est la fonction courante. Le frame `#1` est son appelant. La pile se lit de bas en haut : `main` a appelé `find_value`.

Pour un programme plus complexe avec de l'imbrication profonde :

```
(gdb) backtrace
#0  validate_entry (entry=...) at validator.cpp:15
#1  0x00005555555554a2 in process_line (line="db_host=localhost", num=2)
    at parser.cpp:87
#2  0x0000555555555301 in parse_config (filename="database.conf")
    at config_parser.cpp:25
#3  0x0000555555555512 in main (argc=2, argv=0x7fffffffe3a8)
    at config_parser.cpp:53
```

Le `backtrace` est souvent la première commande à taper quand le programme s'arrête de manière inattendue (segfault, breakpoint non anticipé). Il répond à la question fondamentale : "comment suis-je arrivé ici ?"

### `frame` — Changer de contexte dans la pile

```
(gdb) frame 1
#1  0x0000555555555512 in main (argc=2, argv=0x7fffffffe3a8)
    at config_parser.cpp:60
60          auto db_host = find_value(entries, "db_host");
```

Vous êtes maintenant dans le contexte de `main` — les commandes `print` et `info locals` afficheront les variables de `main`, pas celles de `find_value`. Cela ne change pas l'exécution du programme, seulement votre point de vue.

### `up` et `down` — Navigation relative dans la pile

```
(gdb) up          # Remonte d'un frame (de #0 à #1)
(gdb) up          # Remonte encore (de #1 à #2)
(gdb) down        # Redescend (de #2 à #1)
```

Plus rapide que `frame <n>` quand vous voulez simplement "remonter d'un cran" pour voir qui a appelé la fonction courante et avec quels arguments.

## Exécution au niveau des instructions machine

Dans la grande majorité des cas, vous naviguez au niveau des lignes de code source. Mais parfois — corruption mémoire, optimisation agressive, interopérabilité C/assembleur — vous avez besoin de descendre au niveau des instructions machine.

### `stepi` et `nexti` — Instruction par instruction

```
(gdb) stepi       # Exécute une instruction machine (entre dans les call)
(gdb) nexti       # Exécute une instruction machine (saute par-dessus les call)
```

Abrégés `si` et `ni`. Vous les combinez avec `disassemble` pour voir le code assembleur :

```
(gdb) disassemble
Dump of assembler code for function parse_config(std::string const&):
   0x00005555555552a1 <+0>:     push   %rbp
   0x00005555555552a2 <+1>:     mov    %rsp,%rbp
=> 0x00005555555552a5 <+4>:     sub    $0x120,%rsp
   ...
```

La flèche `=>` indique l'instruction courante. Chaque `stepi` avance d'une instruction.

Ce niveau de détail est rarement nécessaire pour le débogage courant, mais il devient indispensable quand vous analysez un crash dans une zone sans information de débogage, ou quand vous investiguez un problème de performance lié à la génération de code.

## Raccourci fondamental : la touche Entrée

GDB a un comportement extrêmement pratique : **appuyer sur Entrée sans rien taper répète la dernière commande**. C'est anodin en apparence, mais ça change radicalement l'ergonomie.

Quand vous avancez ligne par ligne, vous tapez `next` une fois, puis vous appuyez sur Entrée, Entrée, Entrée pour avancer de trois lignes supplémentaires. Pas besoin de retaper `next` à chaque fois. Cela fonctionne avec `step`, `next`, `continue`, `stepi`, `nexti` — toutes les commandes de navigation.

```
(gdb) next
53          auto entries = parse_config(argv[1]);
(gdb)                         ← Entrée seule = répète next
54          std::cout << "Entrées parsées : " << entries.size() << "\n";
(gdb)                         ← Entrée seule = répète next
56          for (const auto& e : entries) {
```

C'est un détail ergonomique qui, cumulé sur des centaines de sessions de débogage, représente un gain de productivité tangible.

| Commande | Abrégé | Effet |
|---|---|---|
| `run [args]` | `r` | Lance le programme depuis le début |
| `start [args]` | — | Lance et s'arrête à `main()` |
| `kill` | `k` | Arrête le programme en cours |
| `quit` | `q` | Quitte GDB |
| `break <loc>` | `b` | Pose un breakpoint |
| `tbreak <loc>` | `tb` | Breakpoint temporaire (supprimé après premier arrêt) |
| `info breakpoints` | `i b` | Liste les breakpoints |
| `delete [n]` | `d` | Supprime un ou tous les breakpoints |
| `disable [n]` | `dis` | Désactive un breakpoint |
| `enable [n]` | `en` | Réactive un breakpoint |
| `next` | `n` | Ligne suivante (sans entrer dans les fonctions) |
| `step` | `s` | Ligne suivante (entre dans les fonctions) |
| `continue` | `c` | Reprend l'exécution jusqu'au prochain breakpoint |
| `finish` | `fin` | Exécute jusqu'au retour de la fonction |
| `until [line]` | `u` | Avance jusqu'à la ligne spécifiée |
| `backtrace` | `bt` | Affiche la pile d'appels |
| `frame <n>` | `f` | Change le contexte vers le frame n |
| `up` | — | Remonte d'un frame dans la pile |
| `down` | — | Descend d'un frame dans la pile |
| `stepi` | `si` | Exécute une instruction machine |
| `nexti` | `ni` | Exécute une instruction machine (saute les `call`) |

## Stratégie de navigation : quand utiliser quoi

Le choix entre les commandes n'est pas arbitraire. Voici les réflexes à développer selon le scénario :

**"Je ne sais pas où est le bug."**
→ `start`, puis `next` dans `main()` pour identifier quelle fonction produit le résultat incorrect. Une fois la fonction suspecte identifiée, relancez avec un breakpoint dessus et utilisez `step` pour y entrer.

**"Je sais dans quelle fonction se trouve le bug."**
→ `break <fonction>`, `run`, puis `next` ligne par ligne dans la fonction. Si elle appelle d'autres fonctions suspectes, `step` pour descendre.

**"Le bug survient à la Nième itération d'une boucle."**
→ Posez un breakpoint conditionnel dans la boucle (section 29.1.3) et `continue`. Ou utilisez `until` pour sauter les itérations non pertinentes.

**"Je suis entré dans une fonction par erreur (ou dans la STL)."**
→ `finish` pour remonter à l'appelant immédiatement.

**"Le programme crash (segfault)."**
→ `run` sans breakpoint. GDB s'arrête automatiquement au signal SIGSEGV. Tapez `backtrace` pour voir la pile d'appels au moment du crash, puis `frame` pour naviguer dans les différents niveaux d'appel et inspecter les variables.

**"Je veux observer le programme à un point précis, puis le laisser continuer."**
→ `tbreak` (breakpoint temporaire) + `run` + inspection + `continue`.

---

> **À retenir** : `next` et `step` sont vos deux modes fondamentaux de déplacement. `next` reste au même niveau d'abstraction, `step` descend d'un niveau. `continue` saute au prochain breakpoint, `finish` remonte d'un niveau. Avec ces quatre commandes et la touche Entrée, vous couvrez 90 % de la navigation dans GDB.
