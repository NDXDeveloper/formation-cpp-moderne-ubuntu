# 29.1.3 — Breakpoints conditionnels

## Section 29.1 : GDB — Commandes essentielles et breakpoints · Chapitre 29

---

## Introduction

Vous savez poser un breakpoint sur une ligne, lancer le programme, et l'inspecter quand il s'arrête (sections 29.1.1 et 29.1.2). Mais imaginez ce scénario : votre programme parse un fichier de 50 000 lignes, et le bug se manifeste à la ligne 38 742. Avec un breakpoint simple dans la boucle de parsing, vous devriez taper `continue` 38 741 fois avant d'atteindre le cas intéressant.

Les breakpoints conditionnels résolvent ce problème. Ils ne déclenchent un arrêt que lorsqu'une condition que vous spécifiez est vraie. Au lieu de vous arrêter à chaque passage, vous dites à GDB : "arrête-toi ici, mais uniquement quand `line_num == 38742`" — et le programme tourne à pleine vitesse jusqu'à cet instant précis.

C'est la différence entre filtrer manuellement des milliers d'événements et laisser le débogueur faire le tri pour vous. Pour les bugs qui ne se manifestent que dans des conditions spécifiques — une certaine valeur, une certaine itération, un certain état — les breakpoints conditionnels sont l'outil décisif.

---

## Poser un breakpoint conditionnel

### Syntaxe directe : `break ... if <condition>`

La manière la plus naturelle est de spécifier la condition au moment de la création du breakpoint :

```
(gdb) break config_parser.cpp:23 if line_num == 5
Breakpoint 1 at 0x555555555302: file config_parser.cpp, line 23.
(gdb) run database.conf
Starting program: /home/dev/config_parser database.conf

Breakpoint 1, parse_config (filename="database.conf")
    at config_parser.cpp:23
23              ++line_num;
(gdb) print line_num
$1 = 5
(gdb) print line
$2 = "# Timeout en secondes"
```

Le programme a traversé les 4 premières itérations de la boucle sans s'arrêter, et GDB a déclenché l'arrêt uniquement quand `line_num` valait 5. Les itérations ignorées se sont exécutées à vitesse normale — aucune pénalité de performance.

La condition est une expression C++ évaluée dans le contexte de la ligne du breakpoint. Tout ce que vous pouvez écrire dans un `if` en C++ est utilisable :

```
(gdb) break parse_config if filename == "production.yaml"
(gdb) break config_parser.cpp:30 if pos == std::string::npos
(gdb) break config_parser.cpp:23 if line_num > 100 && line[0] != '#'
(gdb) break find_value if key == "db_host"
(gdb) break config_parser.cpp:34 if entry.key.size() == 0
```

### Ajouter une condition à un breakpoint existant : `condition`

Si vous avez déjà posé un breakpoint simple et que vous voulez le rendre conditionnel après coup, utilisez la commande `condition` :

```
(gdb) break config_parser.cpp:23
Breakpoint 2 at 0x555555555302: file config_parser.cpp, line 23.
(gdb) condition 2 line_num == 5
```

Le breakpoint numéro 2 est désormais conditionnel. `info breakpoints` le confirme :

```
(gdb) info breakpoints
Num     Type           Disp Enb Address            What
2       breakpoint     keep y   0x555555555302     in parse_config(...)
                                                   at config_parser.cpp:23
        stop only if line_num == 5
```

La ligne `stop only if ...` indique la condition active.

### Retirer une condition

Pour transformer un breakpoint conditionnel en breakpoint simple (qui s'arrête à chaque passage) :

```
(gdb) condition 2
```

Appeler `condition` sans expression retire la condition du breakpoint spécifié.

## Types de conditions courantes

### Conditions sur les valeurs

Le cas le plus fréquent — s'arrêter quand une variable atteint une valeur précise :

```
(gdb) break config_parser.cpp:23 if line_num == 38742
(gdb) break process_request if request_id == 1057
(gdb) break calculate if result < 0
(gdb) break allocator.cpp:88 if size > 1048576          # > 1 Mo
```

### Conditions sur les chaînes

La comparaison de `std::string` fonctionne directement grâce à l'opérateur `==` surchargé :

```
(gdb) break config_parser.cpp:30 if entry.key == "timeout"
(gdb) break parse_config if line == ""
(gdb) break handler.cpp:45 if request.method == "POST"
```

Pour les chaînes C (`char*`), la comparaison `==` compare les adresses, pas le contenu. Utilisez `strcmp` :

```
(gdb) break main if strcmp(argv[1], "--debug") == 0
```

### Conditions sur les conteneurs

Vous pouvez appeler des méthodes dans les conditions :

```
(gdb) break config_parser.cpp:35 if entries.size() > 10
(gdb) break handler.cpp:60 if buffer.empty()
(gdb) break parser.cpp:22 if line.find("error") != std::string::npos
```

Attention : chaque évaluation de la condition implique un appel de fonction. Sur un breakpoint traversé des milliers de fois (dans une boucle serrée), l'appel de `entries.size()` à chaque itération ralentit l'exécution de manière perceptible. Pour les conditions simples sur des variables scalaires (`int`, `bool`, `pointer`), l'évaluation est quasi instantanée.

### Conditions combinées

Les opérateurs logiques C++ fonctionnent dans les conditions :

```
(gdb) break config_parser.cpp:30 if line_num > 100 && pos == std::string::npos
(gdb) break server.cpp:55 if status != 200 || response_time > 5000
(gdb) break allocator.cpp:42 if ptr == nullptr && size > 0
```

### Conditions sur les pointeurs

```
(gdb) break process if ptr == 0x0            # Pointeur null
(gdb) break process if ptr != nullptr        # Pointeur non-null
(gdb) break tree.cpp:30 if node->left == nullptr && node->right == nullptr  # Feuille
```

## Compter les passages : `ignore`

Parfois, vous ne savez pas quelle condition déclenche le bug, mais vous savez qu'il survient "vers la 500e itération". La commande `ignore` dit à GDB de ne pas s'arrêter les N premiers passages :

```
(gdb) break config_parser.cpp:23
Breakpoint 1 at 0x555555555302: file config_parser.cpp, line 23.
(gdb) ignore 1 499
Will ignore next 499 crossings of breakpoint 1.
(gdb) run large_config.conf
[... le programme tourne ...]

Breakpoint 1, parse_config (filename="large_config.conf")
    at config_parser.cpp:23
23              ++line_num;
(gdb) print line_num
$1 = 500
```

Le breakpoint 1 a été traversé 499 fois sans s'arrêter. Au 500e passage, GDB s'est arrêté. Après cet arrêt, le compteur d'ignore est épuisé — les passages suivants déclencheront aussi un arrêt. Si vous voulez sauter à nouveau, retapez `ignore`.

### Vérifier le compteur restant

```
(gdb) info breakpoints
Num     Type           Disp Enb Address            What
1       breakpoint     keep y   0x555555555302     in parse_config(...)
                                                   at config_parser.cpp:23
        ignore next 347 hits
```

La ligne `ignore next 347 hits` indique qu'il reste 347 passages à ignorer.

### Combiner `ignore` et `condition`

Les deux mécanismes sont indépendants et se cumulent : GDB évalue d'abord le compteur `ignore`, puis la condition. Le breakpoint ne s'arrête que si le compteur est épuisé **et** la condition est vraie.

```
(gdb) break config_parser.cpp:23 if line[0] != '#'
(gdb) ignore 1 3
```

Ce breakpoint ignore les 3 premiers passages où `line[0] != '#'` est vrai, puis s'arrête au 4e passage qui satisfait la condition. Les passages où la condition est fausse ne décomptent pas.

## Exécuter des commandes automatiquement : `commands`

La fonctionnalité la plus puissante — et la plus sous-utilisée — de GDB. La commande `commands` associe une séquence de commandes GDB à un breakpoint. Chaque fois que le breakpoint est atteint, ces commandes s'exécutent automatiquement.

### Syntaxe de base

```
(gdb) break config_parser.cpp:23
Breakpoint 1 at 0x555555555302: file config_parser.cpp, line 23.
(gdb) commands 1
Type commands for breakpoint(s) 1, one per line.  
End with a line saying just "end".  
> print line_num
> print line
> print entries.size()
> continue
> end
```

Désormais, à chaque passage par le breakpoint 1, GDB affiche `line_num`, `line`, `entries.size()`, puis reprend automatiquement l'exécution. Le programme ne s'arrête jamais — GDB exécute le bloc et continue. Vous obtenez un log automatique sans modifier le code source :

```
(gdb) run database.conf

Breakpoint 1, parse_config (...) at config_parser.cpp:23
$1 = 1
$2 = "# database.conf"
$3 = 0

Breakpoint 1, parse_config (...) at config_parser.cpp:23
$4 = 2
$5 = "db_host=localhost"
$6 = 0

Breakpoint 1, parse_config (...) at config_parser.cpp:23
$7 = 3
$8 = "db_port=5432"
$9 = 1

[... etc pour chaque itération ...]
```

Vous venez de créer un point de trace (tracepoint) sans recompiler. C'est une technique extrêmement efficace pour observer le comportement d'un programme sur des milliers d'itérations sans intervention manuelle.

### `silent` — Supprimer le message de breakpoint

Par défaut, GDB affiche un message à chaque arrêt ("Breakpoint 1, parse_config..."). Pour un tracepoint, c'est du bruit. La commande `silent` en première position le supprime :

```
(gdb) commands 1
> silent
> printf "Ligne %d : %s (entries: %d)\n", line_num, line.c_str(), entries.size()
> continue
> end
```

La sortie devient propre :

```
(gdb) run database.conf
Ligne 1 : # database.conf (entries: 0)  
Ligne 2 : db_host=localhost (entries: 0)  
Ligne 3 : db_port=5432 (entries: 1)  
Ligne 4 : db_name=production (entries: 2)  
Ligne 5 : # Timeout en secondes (entries: 3)  
Ligne 6 : timeout=30 (entries: 3)  
Ligne 7 : max_connections=100 (entries: 4)  
Entrées parsées : 5  
[...]
```

Vous avez un log structuré de chaque itération, généré par GDB, sans avoir touché au code source. Quand vous avez fini l'investigation, vous supprimez le breakpoint et le programme redevient vierge.

### `commands` sans `continue` — Arrêt conditionnel enrichi

Si vous omettez `continue` dans le bloc, le programme s'arrête et vous rend la main après avoir exécuté les commandes. C'est un breakpoint normal, mais qui pré-affiche les informations pertinentes :

```
(gdb) break config_parser.cpp:30 if pos == std::string::npos
(gdb) commands 2
> silent
> printf "=== ERREUR PARSING ligne %d ===\n", line_num
> print line
> info locals
> end
```

Quand le breakpoint se déclenche (une ligne sans `=`), GDB affiche automatiquement le contexte complet — numéro de ligne, contenu, toutes les variables locales — avant de vous rendre la main. Vous n'avez rien à taper, l'information est déjà là.

### Combiner `commands` avec des conditions

La combinaison de breakpoints conditionnels et de `commands` crée des scénarios de débogage sophistiqués :

```
(gdb) break config_parser.cpp:34 if entry.value.empty()
Breakpoint 3 at 0x555555555380: file config_parser.cpp, line 34.
(gdb) commands 3
> silent
> printf "ALERTE : Clé '%s' avec valeur vide à la ligne %d\n", entry.key.c_str(), line_num
> backtrace 3
> continue
> end
```

Ce breakpoint fonctionne comme un système de monitoring intégré : il ne se déclenche que quand une entrée a une valeur vide, il log l'alerte avec la clé et le numéro de ligne, il affiche les 3 premiers frames de la pile d'appels, et il reprend automatiquement. Vous pouvez le laisser tourner sur un jeu de test volumineux et analyser les alertes après coup.

## Breakpoints sur des événements système

Au-delà des breakpoints sur des lignes de code, GDB peut s'arrêter sur des événements liés au système d'exploitation.

### `catch throw` — S'arrêter quand une exception est lancée

```
(gdb) catch throw
Catchpoint 1 (throw)
(gdb) run database.conf
Catchpoint 1 (exception thrown), 0x00007ffff7c89342 in __cxa_throw ()
    from /lib/x86_64-linux-gnu/libstdc++.so.6
(gdb) backtrace
#0  0x00007ffff7c89342 in __cxa_throw ()
#1  0x0000555555555a01 in parse_config (filename="missing.conf")
    at config_parser.cpp:18
#2  0x0000555555555512 in main (argc=2, argv=0x7fffffffe3a8)
    at config_parser.cpp:53
```

GDB intercepte toute exception C++ au moment où elle est lancée — avant qu'elle ne soit attrapée par un `catch`. C'est la technique pour localiser l'origine d'une exception quand le message d'erreur ne suffit pas à comprendre d'où elle vient.

### Variantes de `catch`

```
(gdb) catch throw                     # Toute exception lancée
(gdb) catch throw std::runtime_error  # Uniquement ce type d'exception
(gdb) catch catch                     # Quand une exception est attrapée
(gdb) catch syscall open              # Appel système open()
(gdb) catch fork                      # Appel à fork()
(gdb) catch load libcurl.so           # Chargement d'une bibliothèque partagée
```

Les catchpoints sur les appels système (`catch syscall`) sont particulièrement utiles pour comprendre les interactions entre votre programme et le noyau — par exemple, identifier tous les fichiers qu'un programme ouvre, ou détecter un `mmap` inattendu.

## Breakpoints temporaires et jetables

### `tbreak` avec condition

Les breakpoints temporaires (qui se suppriment après le premier arrêt) acceptent aussi des conditions :

```
(gdb) tbreak config_parser.cpp:23 if line_num == 5
Temporary breakpoint 1 at 0x555555555302: file config_parser.cpp, line 23.
(gdb) run database.conf
Temporary breakpoint 1, parse_config (...) at config_parser.cpp:23
23              ++line_num;
(gdb) info breakpoints
No breakpoints or watchpoints.     ← supprimé automatiquement
```

Utile quand vous voulez observer un instant précis une seule fois, sans encombrer votre liste de breakpoints.

### `rbreak` — Breakpoints par expression régulière

Pour poser un breakpoint sur toutes les fonctions correspondant à un pattern :

```
(gdb) rbreak parse_.*
Breakpoint 1 at 0x5555555552a1: file config_parser.cpp, line 16.
    void parse_config(std::string const&);
Breakpoint 2 at 0x5555555558a0: file config_parser.cpp, line 72.
    void parse_line(std::string const&, int);
```

GDB a posé un breakpoint sur chaque fonction dont le nom commence par `parse_`. C'est une manière rapide de surveiller une famille de fonctions sans les énumérer manuellement.

Combiné avec `commands`, vous pouvez instrumenter toute une couche de votre application :

```
(gdb) rbreak parse_.*
(gdb) commands 1-2
> silent
> printf ">> Appel à %s\n", __FUNCTION__
> continue
> end
```

## Sauvegarder et restaurer des breakpoints

Quand vous avez passé du temps à configurer un ensemble de breakpoints conditionnels avec des `commands`, vous ne voulez pas les perdre en quittant GDB.

### `save breakpoints` — Exporter vers un fichier

```
(gdb) save breakpoints debug_session.gdb
Saved to file 'debug_session.gdb'.
```

Le fichier généré contient les commandes GDB nécessaires pour recréer tous vos breakpoints :

```bash
# debug_session.gdb (généré automatiquement)
break config_parser.cpp:23 if line_num == 5
  commands
    silent
    printf "Ligne %d : %s\n", line_num, line.c_str()
    continue
  end
break config_parser.cpp:30 if pos == std::string::npos  
break find_value if key == "db_host"  
```

### `source` — Restaurer depuis un fichier

```
(gdb) source debug_session.gdb
Breakpoint 1 at 0x555555555302: file config_parser.cpp, line 23.  
Breakpoint 2 at 0x555555555360: file config_parser.cpp, line 30.  
Breakpoint 3 at 0x555555555401: file find_value, line 40.  
```

Vous retrouvez l'intégralité de votre configuration. C'est indispensable pour les sessions de débogage qui s'étalent sur plusieurs jours — vous sauvegardez votre setup en fin de journée et le restaurez le lendemain.

Vous pouvez aussi charger un fichier au lancement de GDB :

```bash
gdb -x debug_session.gdb ./config_parser
```

## Breakpoints dans du code templaté et surchargé

### Fonctions surchargées

Si une fonction a plusieurs surcharges, GDB vous demande laquelle cibler :

```
(gdb) break process
Function "process" has multiple definitions:
  1. process(int) at handler.cpp:10
  2. process(std::string const&) at handler.cpp:25
  3. process(Config const&) at handler.cpp:40
> 2
Breakpoint 1 at 0x5555555554a0: file handler.cpp, line 25.
```

Pour cibler directement une surcharge sans passer par le menu interactif :

```
(gdb) break process(std::string const&)
```

### Templates

Pour les fonctions template instanciées, utilisez la signature complète :

```
(gdb) break 'std::vector<ConfigEntry>::push_back(ConfigEntry&&)'
```

Les guillemets simples sont nécessaires quand le nom contient des caractères spéciaux (chevrons, virgules). En pratique, il est souvent plus simple de poser le breakpoint par numéro de ligne plutôt que par nom de fonction template.

La commande `info functions` aide à trouver le nom exact :

```
(gdb) info functions push_back
All functions matching regular expression "push_back":
  File /usr/include/c++/15/bits/stl_vector.h:
    void std::vector<ConfigEntry>::push_back(ConfigEntry const&);
    void std::vector<ConfigEntry>::push_back(ConfigEntry&&);
```

## Récapitulatif des commandes

| Commande | Effet |
|---|---|
| `break <loc> if <cond>` | Breakpoint conditionnel (s'arrête si la condition est vraie) |
| `condition <n> <cond>` | Ajoute ou modifie la condition du breakpoint n |
| `condition <n>` | Retire la condition du breakpoint n |
| `ignore <n> <count>` | Ignore les count prochains passages du breakpoint n |
| `commands <n>` | Associe des commandes automatiques au breakpoint n |
| `silent` | (dans `commands`) Supprime le message de breakpoint |
| `catch throw [type]` | S'arrête quand une exception est lancée |
| `catch catch` | S'arrête quand une exception est attrapée |
| `catch syscall <name>` | S'arrête sur un appel système |
| `tbreak <loc> if <cond>` | Breakpoint conditionnel temporaire |
| `rbreak <regex>` | Breakpoint sur toutes les fonctions matchant l'expression |
| `save breakpoints <file>` | Sauvegarde les breakpoints dans un fichier |
| `source <file>` | Charge des commandes GDB depuis un fichier |

## Patterns de débogage avec breakpoints conditionnels

### Le bug intermittent dans une boucle

Scénario : votre programme traite des milliers d'éléments. Un résultat incorrect apparaît "quelque part" dans le traitement, mais vous ne savez pas à quelle itération.

Stratégie en deux passes :

**Passe 1 — Localiser l'itération.** Posez un breakpoint avec `commands` + `silent` + `continue` pour logger l'état à chaque itération sans vous arrêter :

```
(gdb) break process_item
(gdb) commands 1
> silent
> printf "Item %d: result=%d\n", index, result
> continue
> end
(gdb) run > trace.log 2>&1
```

Analysez le log pour identifier l'itération problématique (par exemple, `index == 4217`).

**Passe 2 — Inspecter en détail.** Relancez avec un breakpoint conditionnel ciblé :

```
(gdb) break process_item if index == 4217
(gdb) run
```

Vous arrivez directement sur le cas problématique, prêt à inspecter et à avancer pas à pas.

### La corruption progressive

Scénario : une structure est correcte au début, puis se retrouve corrompue à un moment indéterminé. Vous ne savez pas quel code la corrompt.

Stratégie : combinez un watchpoint (section 29.1.2) avec un breakpoint conditionnel sur une valeur sentinelle :

```
(gdb) break init_config
(gdb) run
(gdb) watch config.port
(gdb) condition 2 config.port > 65535 || config.port < 0
(gdb) continue
```

Le watchpoint surveille `config.port`, mais ne se déclenche que quand la valeur sort de la plage valide. Vous obtenez directement la ligne qui introduit la valeur corrompue.

### L'exception non localisée

Scénario : votre programme lance une exception, mais le message d'erreur ne vous dit pas d'où elle vient. Le `catch` est trop loin de la source.

Stratégie :

```
(gdb) catch throw std::runtime_error
(gdb) run
[... GDB s'arrête au moment du throw ...]
(gdb) backtrace
(gdb) frame 1       # Remonter dans votre code (frame 0 est __cxa_throw)
(gdb) info locals
```

Vous voyez l'état exact au moment du `throw`, avant que le stack unwinding n'ait nettoyé les variables locales.

## Limites et pièges

### Performance des conditions complexes

Chaque fois que le programme atteint un breakpoint conditionnel, GDB évalue la condition. Pour une condition simple (`line_num == 5`), l'impact est négligeable. Pour une condition impliquant un appel de fonction (`line.find("error") != npos`), l'évaluation peut prendre significativement plus de temps — surtout dans une boucle exécutée des millions de fois.

Si votre breakpoint conditionnel ralentit l'exécution de manière inacceptable, une alternative est de modifier temporairement le code source pour ajouter une condition avec un marqueur :

```cpp
// Temporaire — supprimer après débogage
if (line.find("error") != std::string::npos) {
    volatile int gdb_trap = 1;  // break ici avec un breakpoint simple
}
```

Le breakpoint simple sur `gdb_trap` ne nécessite aucune évaluation de condition par GDB — le compilateur a déjà fait le travail.

### Conditions et optimisation

Si le binaire est compilé avec `-O2` ou `-O3`, le compilateur peut avoir éliminé des variables ou réordonné du code. Une condition qui référence une variable "optimized out" échouera :

```
(gdb) break process.cpp:42 if count == 10
Error in testing breakpoint condition:  
value has been optimized out  
```

La solution est de compiler avec `-O0` pour le débogage (section 29.1, prérequis).

### Conditions dans du code multithreadé

En multithreading, un breakpoint conditionnel s'applique à tous les threads par défaut. Si la boucle problématique tourne dans un thread spécifique :

```
(gdb) info threads
  Id   Target Id                  Frame
* 1    Thread 0x7ffff7c2e740      main () at main.cpp:15
  2    Thread 0x7ffff6c2d700      worker () at worker.cpp:22
  3    Thread 0x7ffff5c2c700      worker () at worker.cpp:22

(gdb) break worker.cpp:30 if item_count > 1000 thread 2
```

Le suffixe `thread 2` restreint le breakpoint au thread d'identifiant 2. Les autres threads passent par cette ligne sans s'arrêter, quelle que soit la valeur de `item_count`.

---

> **À retenir** : un breakpoint simple dit "arrête-toi ici". Un breakpoint conditionnel dit "arrête-toi ici quand c'est intéressant". Combiné avec `commands`, il devient un outil de scripting qui transforme GDB en système d'observation programmable — capable de logger, filtrer et alerter sans modifier une seule ligne de votre code source.
