# 24.2.2 — Écriture YAML

## Sous-section de 24.2 — YAML : Parsing avec yaml-cpp

---

## Vue d'ensemble

La section 24.2.1 couvrait la lecture — transformer un fichier YAML en structures C++. Cette section aborde l'opération inverse : générer du YAML depuis un programme C++. Les cas d'usage sont nombreux : produire des fichiers de configuration par défaut, générer des manifestes Kubernetes ou Docker Compose, exporter un état applicatif, ou créer des rapports lisibles par un humain.

yaml-cpp propose deux approches complémentaires pour l'écriture. La première repose sur la **manipulation directe de `YAML::Node`**, identique à la construction d'un arbre JSON avec nlohmann/json. La seconde utilise le **`YAML::Emitter`**, un générateur séquentiel qui offre un contrôle fin sur le formatage de sortie. Chaque approche a ses forces, et un projet réaliste les combine souvent.

---

## Approche 1 : Construction via `YAML::Node`

Un `YAML::Node` peut être construit programmatiquement en lui assignant des valeurs, exactement comme on builderait un objet `nlohmann::json` :

```cpp
#include <yaml-cpp/yaml.h>
#include <fstream>
#include <print>

int main() {
    YAML::Node config;

    // Scalaires
    config["server"]["host"] = "0.0.0.0";
    config["server"]["port"] = 8080;
    config["server"]["workers"] = 4;

    // Séquence
    config["server"]["allowed_origins"].push_back("https://app.example.com");
    config["server"]["allowed_origins"].push_back("https://admin.example.com");

    // Sous-map
    config["database"]["host"] = "db.internal";
    config["database"]["port"] = 5432;
    config["database"]["name"] = "production";

    // Booléen
    config["database"]["ssl"] = true;

    // Écriture vers stdout
    std::print("{}\n", YAML::Dump(config));
}
```

Sortie :

```yaml
server:
  host: 0.0.0.0
  port: 8080
  workers: 4
  allowed_origins:
    - https://app.example.com
    - https://admin.example.com
database:
  host: db.internal
  port: 5432
  name: production
  ssl: true
```

### Écriture vers un fichier

`YAML::Dump` retourne une `std::string`. L'écriture vers un fichier suit le pattern RAII standard :

```cpp
void save_config(const YAML::Node& config, const std::string& path) {
    std::ofstream file(path);
    if (!file.is_open()) {
        throw std::runtime_error("Impossible d'ouvrir " + path);
    }
    file << YAML::Dump(config);
}
```

L'opérateur `<<` est également supporté directement sur le nœud :

```cpp
std::ofstream file("config.yaml");  
file << config;  // équivalent à file << YAML::Dump(config)  
```

### Écriture depuis des types utilisateur

Si des spécialisations `YAML::convert<T>::encode` sont définies (cf. section 24.2), la construction de l'arbre depuis des objets métier est transparente :

```cpp
ServerConfig server{
    .host = "0.0.0.0",
    .port = 443,
    .workers = 8,
    .tls = TlsConfig{"/etc/ssl/cert.pem", "/etc/ssl/key.pem", true},
    .allowed_origins = {"https://app.example.com"}
};

YAML::Node root;  
root["server"] = server;  // appelle convert<ServerConfig>::encode  

std::ofstream file("generated.yaml");  
file << root;  
```

### Limites de l'approche `Node`

La construction par `Node` est simple et suffisante pour la majorité des cas. Cependant, elle ne permet pas de contrôler finement le formatage de sortie : l'indentation est fixe, l'ordre des clés suit l'ordre d'insertion, les commentaires ne peuvent pas être ajoutés, et le choix entre style bloc et style flow est automatique. Pour ces besoins, le `YAML::Emitter` prend le relais.

## Approche 2 : Génération avec `YAML::Emitter`

Le `YAML::Emitter` est un générateur séquentiel : on lui envoie des événements (début de map, clé, valeur, début de séquence, etc.) et il produit le YAML formaté au fil de l'eau. C'est l'équivalent conceptuel d'un XML writer ou d'un JSON serializer événementiel, mais côté écriture.

### Exemple de base

```cpp
#include <yaml-cpp/yaml.h>
#include <print>

int main() {
    YAML::Emitter out;

    out << YAML::BeginMap;

    out << YAML::Key << "server" << YAML::Value;
    out << YAML::BeginMap;
    out << YAML::Key << "host" << YAML::Value << "0.0.0.0";
    out << YAML::Key << "port" << YAML::Value << 8080;
    out << YAML::Key << "workers" << YAML::Value << 4;
    out << YAML::EndMap;

    out << YAML::Key << "database" << YAML::Value;
    out << YAML::BeginMap;
    out << YAML::Key << "host" << YAML::Value << "db.internal";
    out << YAML::Key << "port" << YAML::Value << 5432;
    out << YAML::EndMap;

    out << YAML::EndMap;

    // out.c_str() retourne le YAML généré
    std::print("{}\n", out.c_str());

    // Vérification de validité
    if (!out.good()) {
        std::print(stderr, "Erreur d'émission : {}\n", out.GetLastError());
    }
}
```

Le pattern est toujours le même : `BeginMap`/`EndMap` pour les objets, `BeginSeq`/`EndSeq` pour les séquences, `Key`/`Value` pour structurer les paires clé-valeur, et des valeurs scalaires insérées directement via `<<`.

### Séquences

```cpp
YAML::Emitter out;  
out << YAML::BeginMap;  

out << YAML::Key << "ports" << YAML::Value;  
out << YAML::BeginSeq;  
out << 8080 << 8443 << 9090;  
out << YAML::EndSeq;  

out << YAML::Key << "tags" << YAML::Value;  
out << YAML::BeginSeq;  
out << "production" << "eu-west" << "critical";  
out << YAML::EndSeq;  

out << YAML::EndMap;
```

Sortie :

```yaml
ports:
  - 8080
  - 8443
  - 9090
tags:
  - production
  - eu-west
  - critical
```

## Contrôle du formatage

Le `YAML::Emitter` offre des manipulateurs pour contrôler le style de sortie. C'est son principal avantage sur l'approche `Node`.

### Style flow vs bloc

YAML supporte deux styles de notation. Le style **bloc** (par défaut) utilise l'indentation :

```yaml
ports:
  - 8080
  - 8443
```

Le style **flow** utilise les accolades et crochets, identique à JSON :

```yaml
ports: [8080, 8443]
```

Le manipulateur `YAML::Flow` bascule un nœud en style flow :

```cpp
YAML::Emitter out;  
out << YAML::BeginMap;  

// Séquence en style flow (compact, sur une ligne)
out << YAML::Key << "ports" << YAML::Value;  
out << YAML::Flow << YAML::BeginSeq;  
out << 8080 << 8443 << 9090;  
out << YAML::EndSeq;  

// Map en style flow
out << YAML::Key << "metadata" << YAML::Value;  
out << YAML::Flow << YAML::BeginMap;  
out << YAML::Key << "env" << YAML::Value << "prod";  
out << YAML::Key << "region" << YAML::Value << "eu-west";  
out << YAML::EndMap;  

// Séquence en style bloc (par défaut)
out << YAML::Key << "features" << YAML::Value;  
out << YAML::Block << YAML::BeginSeq;  
out << "logging" << "metrics" << "tracing";  
out << YAML::EndSeq;  

out << YAML::EndMap;
```

Sortie :

```yaml
ports: [8080, 8443, 9090]  
metadata: {env: prod, region: eu-west}  
features:  
  - logging
  - metrics
  - tracing
```

Le mélange des styles au sein d'un même document est courant et améliore la lisibilité : les données courtes et denses gagnent au style flow, tandis que les structures profondes restent plus claires en style bloc.

### Indentation

L'indentation par défaut est de 2 espaces. Elle se modifie via `SetIndent` :

```cpp
YAML::Emitter out;  
out.SetIndent(4);  // 4 espaces par niveau  

out << YAML::BeginMap;  
out << YAML::Key << "server" << YAML::Value;  
out << YAML::BeginMap;  
out << YAML::Key << "host" << YAML::Value << "localhost";  
out << YAML::Key << "port" << YAML::Value << 8080;  
out << YAML::EndMap;  
out << YAML::EndMap;  
```

Sortie :

```yaml
server:
    host: localhost
    port: 8080
```

La convention de 2 espaces est la plus répandue dans l'écosystème YAML (Kubernetes, Ansible, GitHub Actions). Sauf contrainte spécifique, il est préférable de conserver cette valeur par défaut.

### Quotation des chaînes

Par défaut, yaml-cpp quote les chaînes uniquement quand c'est nécessaire (présence de caractères spéciaux, ambiguïté avec un autre type). On peut forcer un style de quotation :

```cpp
YAML::Emitter out;  
out << YAML::BeginMap;  

// Double quotes forcées
out << YAML::Key << "name" << YAML::Value;  
out << YAML::DoubleQuoted << "my-service";  

// Single quotes forcées
out << YAML::Key << "version" << YAML::Value;  
out << YAML::SingleQuoted << "1.0";  

// Literal block (préserve les retours à la ligne)
out << YAML::Key << "description" << YAML::Value;  
out << YAML::Literal << "Première ligne\nDeuxième ligne\nTroisième ligne";  

out << YAML::EndMap;
```

Sortie :

```yaml
name: "my-service"  
version: '1.0'  
description: |  
  Première ligne
  Deuxième ligne
  Troisième ligne
```

Le style `YAML::Literal` est particulièrement utile pour les blocs de texte multi-lignes : scripts shell embarqués, descriptions longues, templates. Le style `YAML::DoubleQuoted` est recommandé pour les valeurs susceptibles d'être mal interprétées par le typage implicite YAML (cf. le piège `NO` → `false` en section 24.2).

### Forcer la quotation globalement

Pour éliminer tout risque de typage implicite dans le YAML généré, on peut activer la quotation systématique de toutes les chaînes avec `SetStringFormat` :

```cpp
YAML::Emitter out;  
out.SetStringFormat(YAML::DoubleQuoted);  

out << YAML::BeginMap;  
out << YAML::Key << "country" << YAML::Value << "NO";  
out << YAML::Key << "enabled" << YAML::Value << "yes";  
out << YAML::Key << "version" << YAML::Value << "1.0";  
out << YAML::EndMap;  
```

Sortie :

```yaml
"country": "NO"
"enabled": "yes"
"version": "1.0"
```

Toutes les chaînes sont explicitement quotées, éliminant toute ambiguïté pour le parser qui relira le fichier. C'est une approche défensive adaptée aux fichiers générés automatiquement.

### Autres manipulateurs

| Manipulateur | Effet |
|-------------|-------|
| `YAML::Null` | Émet la valeur `~` (null YAML) |
| `YAML::BoolFormat(YAML::TrueFalseBool)` | Booléens en `true`/`false` (défaut) |
| `YAML::BoolFormat(YAML::YesNoBool)` | Booléens en `yes`/`no` |
| `YAML::BoolFormat(YAML::OnOffBool)` | Booléens en `on`/`off` |
| `YAML::FloatPrecision(n)` | Précision des flottants |
| `YAML::DoublePrecision(n)` | Précision des doubles |

## Émission de `YAML::Node` existants

Le `YAML::Emitter` accepte directement un `YAML::Node`, ce qui permet de charger un fichier, le modifier, puis le réémettre avec un formatage contrôlé :

```cpp
// Charger, modifier, sauvegarder avec formatage personnalisé
YAML::Node config = YAML::LoadFile("config.yaml");  
config["server"]["workers"] = 16;  
config["server"]["debug"] = false;  

YAML::Emitter out;  
out.SetIndent(2);  
out.SetStringFormat(YAML::DoubleQuoted);  
out << config;  

std::ofstream file("config_updated.yaml");  
file << out.c_str();  
```

> ⚠️ *Un cycle load → modify → emit ne préserve pas les commentaires du fichier original. Les commentaires YAML sont ignorés par le parser et ne font pas partie du DOM. C'est une limitation fondamentale partagée par toutes les librairies YAML dans tous les langages. Si la préservation des commentaires est critique, il faut opérer au niveau textuel (regex, remplacement de lignes) plutôt qu'au niveau du DOM.*

## Génération de documents multiples

Pour produire un fichier multi-documents (séparés par `---`), on insère le manipulateur `YAML::BeginDoc` et `YAML::EndDoc` :

```cpp
YAML::Emitter out;

// Premier document : Deployment
out << YAML::BeginDoc;  
out << YAML::BeginMap;  
out << YAML::Key << "apiVersion" << YAML::Value << "apps/v1";  
out << YAML::Key << "kind" << YAML::Value << "Deployment";  
out << YAML::Key << "metadata" << YAML::Value;  
out << YAML::BeginMap;  
out << YAML::Key << "name" << YAML::Value << "api";  
out << YAML::EndMap;  
out << YAML::Key << "spec" << YAML::Value;  
out << YAML::BeginMap;  
out << YAML::Key << "replicas" << YAML::Value << 3;  
out << YAML::EndMap;  
out << YAML::EndMap;  
out << YAML::EndDoc;  

// Deuxième document : Service
out << YAML::BeginDoc;  
out << YAML::BeginMap;  
out << YAML::Key << "apiVersion" << YAML::Value << "v1";  
out << YAML::Key << "kind" << YAML::Value << "Service";  
out << YAML::Key << "metadata" << YAML::Value;  
out << YAML::BeginMap;  
out << YAML::Key << "name" << YAML::Value << "api-svc";  
out << YAML::EndMap;  
out << YAML::EndMap;  
out << YAML::EndDoc;  

std::ofstream file("manifests.yaml");  
file << out.c_str();  
```

Sortie :

```yaml
---
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: api
spec:
  replicas: 3
...
---
apiVersion: v1  
kind: Service  
metadata:  
  name: api-svc
...
```

Ce pattern est directement applicable à la génération de manifestes Kubernetes, de fichiers Helm, ou de tout format multi-documents.

## Factorisation : helper d'émission

Pour les projets qui génèrent beaucoup de YAML, une couche d'abstraction au-dessus du `YAML::Emitter` réduit le code boilerplate et les erreurs de `BeginMap`/`EndMap` déséquilibrés :

```cpp
class YamlWriter {
    YAML::Emitter out_;

public:
    YamlWriter() {
        out_.SetIndent(2);
    }

    // RAII scope pour les maps
    class MapScope {
        YAML::Emitter& out_;
    public:
        explicit MapScope(YAML::Emitter& out) : out_(out) {
            out_ << YAML::BeginMap;
        }
        ~MapScope() { out_ << YAML::EndMap; }
    };

    // RAII scope pour les séquences
    class SeqScope {
        YAML::Emitter& out_;
    public:
        explicit SeqScope(YAML::Emitter& out) : out_(out) {
            out_ << YAML::BeginSeq;
        }
        ~SeqScope() { out_ << YAML::EndSeq; }
    };

    [[nodiscard]] MapScope begin_map() { return MapScope(out_); }
    [[nodiscard]] SeqScope begin_seq() { return SeqScope(out_); }

    template <typename T>
    void field(const std::string& key, const T& value) {
        out_ << YAML::Key << key << YAML::Value << value;
    }

    void field_flow_seq(const std::string& key,
                        const std::vector<std::string>& values) {
        out_ << YAML::Key << key << YAML::Value;
        out_ << YAML::Flow << YAML::BeginSeq;
        for (const auto& v : values) {
            out_ << v;
        }
        out_ << YAML::EndSeq;
    }

    // Accès au résultat
    std::string str() const { return out_.c_str(); }
    bool good() const { return out_.good(); }

    void save(const std::string& path) const {
        std::ofstream file(path);
        file << out_.c_str();
    }
};
```

Utilisation :

```cpp
YamlWriter w;
{
    auto root = w.begin_map();
    w.field("name", "my-service");
    w.field("version", "1.0.0");
    w.field("replicas", 3);
    w.field_flow_seq("tags", {"production", "eu-west"});
}

std::print("{}\n", w.str());
// name: my-service
// version: 1.0.0
// replicas: 3
// tags: [production, eu-west]
```

Pour les structures plus complexes avec imbrication, la combinaison `Node` + `Emitter` est souvent plus lisible que le helper seul :

```cpp
// Combiner Node et Emitter
YAML::Node deployment;  
deployment["apiVersion"] = "apps/v1";  
deployment["kind"] = "Deployment";  
deployment["metadata"]["name"] = "api";  
deployment["spec"]["replicas"] = 3;  

YAML::Emitter out;  
out.SetIndent(2);  
out << deployment;  
```

En pratique, la combinaison des deux approches est souvent la plus pragmatique : construire l'arbre avec `YAML::Node` (plus lisible), puis émettre avec `YAML::Emitter` quand un contrôle de formatage est nécessaire.

## Gestion des erreurs d'émission

Le `YAML::Emitter` accumule les erreurs sans lever d'exception. Après avoir construit le document, il faut vérifier l'état :

```cpp
YAML::Emitter out;
// ... construction du document ...

if (!out.good()) {
    std::print(stderr, "Erreur de génération YAML : {}\n",
        out.GetLastError());
}
```

Les erreurs typiques sont les structures déséquilibrées (un `BeginMap` sans `EndMap`, ou un `Key` sans `Value`). Ces bugs se manifestent au moment de l'émission, pas à la compilation. C'est la principale faiblesse de l'API séquentielle du `YAML::Emitter` : les erreurs de structure sont détectées tardivement. Le pattern RAII avec les scopes `MapScope`/`SeqScope` présenté ci-dessus atténue ce risque en garantissant que chaque ouverture a sa fermeture correspondante.

## Comparaison des deux approches

| Critère | `YAML::Node` + `Dump` | `YAML::Emitter` |
|---------|----------------------|-----------------|
| Lisibilité du code | Bonne (déclaratif) | Moyenne (séquentiel) |
| Contrôle du formatage | Limité (indentation auto) | Total (flow/bloc, quotation, précision) |
| Multi-documents | Non supporté | Supporté (`BeginDoc`/`EndDoc`) |
| Modification avant écriture | Naturelle | Impossible (write-only) |
| Risque d'erreur | Faible | Moyen (structures déséquilibrées) |
| Usage recommandé | Configuration, export simple | Manifestes, formatage précis, multi-docs |

Pour la majorité des cas — générer un fichier de configuration par défaut, exporter un état, sérialiser un objet — **`YAML::Node` avec `Dump` ou `<<` est suffisant et plus simple**. Le `YAML::Emitter` se justifie quand le contrôle du formatage est une exigence (fichiers destinés à être lus par des humains avec des conventions précises, manifestes multi-documents, chaînes devant être systématiquement quotées).

## Résumé

| Opération | Méthode | Exemple |
|-----------|---------|---------|
| Construire un arbre | `YAML::Node` + assignation | `node["key"] = value;` |
| Émettre en chaîne | `YAML::Dump(node)` ou `<<` | `std::string s = YAML::Dump(node);` |
| Écrire dans un fichier | `ofstream << node` | `file << config;` |
| Formatage personnalisé | `YAML::Emitter` | `out.SetIndent(4);` |
| Style flow | `YAML::Flow` | `out << YAML::Flow << YAML::BeginSeq;` |
| Quotation forcée | `YAML::DoubleQuoted` | `out << YAML::DoubleQuoted << "NO";` |
| Multi-documents | `BeginDoc` / `EndDoc` | `out << YAML::BeginDoc;` |
| Vérification d'erreur | `out.good()` | `if (!out.good()) { ... }` |

La section 24.3 aborde TOML, un format conçu dès l'origine pour la configuration, qui évite les pièges de YAML tout en conservant la lisibilité humaine.
