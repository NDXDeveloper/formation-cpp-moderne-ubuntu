# 24.1.2 — Parsing de fichiers JSON

## Sous-section de 24.1 — JSON : Lecture/Écriture avec nlohmann/json

---

## Vue d'ensemble

Le parsing est la première étape de toute interaction avec des données JSON : transformer un flux de caractères bruts en une structure navigable en mémoire. nlohmann/json propose plusieurs modes de parsing adaptés à des contextes différents — depuis le chargement simple d'un fichier de configuration de quelques Ko jusqu'au traitement en streaming de flux JSON volumineux.

Cette section couvre les sources de parsing, les options de contrôle, la navigation dans le DOM résultant et le mode SAX pour les cas où le chargement intégral en mémoire n'est pas souhaitable.

---

## Parsing depuis différentes sources

### Depuis un fichier

Le cas le plus courant est la lecture d'un fichier de configuration ou de données. On passe un `std::ifstream` directement à `json::parse` :

```cpp
#include <nlohmann/json.hpp>
#include <fstream>
#include <print>

using json = nlohmann::json;

int main() {
    std::ifstream file("config.json");
    if (!file.is_open()) {
        std::print(stderr, "Impossible d'ouvrir config.json\n");
        return 1;
    }

    json config = json::parse(file);

    std::print("Host : {}\n", config["server"]["host"].get<std::string>());
    std::print("Port : {}\n", config["server"]["port"].get<int>());
}
```

Un raccourci fréquemment utilisé passe par l'opérateur `>>`, hérité de la convention des flux C++ :

```cpp
std::ifstream file("config.json");  
json config;  
file >> config;  
```

Les deux formes sont fonctionnellement équivalentes. La forme `json::parse()` est cependant préférable dans un code professionnel car elle offre un contrôle explicite sur le comportement en cas d'erreur, comme on le verra plus loin.

### Depuis une chaîne de caractères

Lorsque le JSON provient d'une réponse HTTP, d'un message réseau ou d'une chaîne construite en mémoire, on parse directement la `std::string` :

```cpp
std::string response = R"({
    "status": "success",
    "data": {
        "user_id": 42,
        "username": "alice",
        "roles": ["admin", "dev"]
    }
})";

json j = json::parse(response);
```

Les raw string literals (`R"(...)"`) sont particulièrement pratiques ici car ils évitent d'échapper les guillemets et permettent d'écrire le JSON sur plusieurs lignes tel qu'il apparaîtrait dans un fichier.

### Depuis une paire d'itérateurs

Pour les cas où les données JSON ne sont qu'une portion d'un buffer plus large, ou lorsqu'elles proviennent d'un conteneur non standard, le parsing accepte une paire d'itérateurs :

```cpp
std::vector<uint8_t> buffer = receive_from_network();

// Parser uniquement les octets [begin, end)
json j = json::parse(buffer.begin(), buffer.end());
```

Cette forme est utile dans les contextes réseau où le JSON arrive dans un buffer binaire (`std::vector<uint8_t>`, `std::array<char, N>`, ou même un `char*` brut issu d'une API C).

### Depuis stdin ou tout `std::istream`

Tout flux d'entrée conforme à `std::istream` est une source valide. Cela inclut `std::cin`, ce qui permet de construire des outils CLI qui consomment du JSON depuis un pipe :

```cpp
// Utilisation : cat data.json | ./process_json
json input = json::parse(std::cin);
```

Cela fonctionne également avec `std::istringstream`, ce qui peut être utile dans les tests unitaires pour simuler un flux d'entrée sans fichier physique.

## Navigation dans le DOM

Une fois le JSON parsé, l'objet `json` résultant est un arbre navigable. La librairie offre plusieurs mécanismes d'accès selon le niveau de sécurité souhaité.

### Accès par clé et par index

L'opérateur `[]` fonctionne sur les objets JSON (accès par clé `std::string`) et les tableaux JSON (accès par index entier) :

```cpp
json j = json::parse(R"({
    "database": {
        "host": "db.example.com",
        "port": 5432,
        "replicas": ["replica-1", "replica-2", "replica-3"]
    }
})");

// Accès imbriqué par clé
std::string db_host = j["database"]["host"];

// Accès par index dans un tableau
std::string first_replica = j["database"]["replicas"][0];
```

Un point important : l'opérateur `[]` sur un objet **crée la clé si elle n'existe pas** (comme `std::map::operator[]`). Si l'objet `json` n'est pas `const` et qu'on accède à une clé inexistante, un élément `null` est inséré silencieusement. C'est un piège classique :

```cpp
json config = json::parse(R"({"host": "localhost"})");

// ⚠️ Crée la clé "port" avec la valeur null !
auto val = config["port"];
// config contient maintenant {"host": "localhost", "port": null}
```

Pour éviter ce comportement, trois alternatives existent.

### Accès sécurisé avec `.at()`

La méthode `.at()` ne crée jamais de clé. Elle lève une exception `json::out_of_range` si la clé ou l'index n'existe pas :

```cpp
try {
    std::string host = j.at("database").at("host");
    int port = j.at("database").at("port");
} catch (const json::out_of_range& e) {
    std::print(stderr, "Champ manquant : {}\n", e.what());
}
```

C'est la méthode à privilégier quand le champ est **obligatoire** et que son absence constitue une erreur que le programme doit traiter.

### Accès avec valeur par défaut via `.value()`

La méthode `.value()` retourne la valeur associée à la clé si elle existe, ou une valeur par défaut fournie par l'appelant :

```cpp
// "timeout" absent → retourne 30
int timeout = j["database"].value("timeout", 30);

// "host" présent → retourne "db.example.com"
std::string host = j["database"].value("host", "localhost");
```

C'est la méthode idéale pour les champs **optionnels** avec un comportement par défaut raisonnable. Elle ne modifie jamais l'objet JSON et ne lève pas d'exception pour une clé absente.

> ⚠️ *`.value()` n'est disponible que sur les objets JSON, pas sur les tableaux. De plus, le type de la valeur par défaut doit correspondre exactement au type attendu — `value("port", 8080)` retourne un `int`, pas un `json`.*

### Test d'existence avec `.contains()`

Pour les cas où la logique dépend de la présence ou non d'un champ, sans valeur par défaut :

```cpp
if (j["database"].contains("ssl")) {
    bool use_ssl = j["database"]["ssl"];
    // configurer SSL...
}
```

Cette méthode est disponible depuis la version 3.6.0 et remplace avantageusement le pattern plus ancien `j.find("key") != j.end()`.

## Extraction typée avec `.get<T>()`

L'opérateur `[]` retourne une référence vers un objet `json`. Pour obtenir un type C++ concret, on utilise `.get<T>()` qui effectue la conversion et lève `json::type_error` si le type JSON ne correspond pas :

```cpp
json j = json::parse(R"({
    "name": "monitoring-agent",
    "version": 3,
    "latency_ms": 12.5,
    "active": true,
    "tags": ["infra", "prod"],
    "metadata": null
})");

// Extraction typée explicite
std::string name    = j["name"].get<std::string>();  
int version         = j["version"].get<int>();  
double latency      = j["latency_ms"].get<double>();  
bool active         = j["active"].get<bool>();  
auto tags           = j["tags"].get<std::vector<std::string>>();  
```

La conversion implicite fonctionne également — `std::string name = j["name"];` — mais `.get<T>()` est préférable dans le code de production car l'intention est explicite et les erreurs de type sont plus faciles à diagnostiquer.

### Vérification du type avant extraction

Pour les données dont la structure n'est pas garantie, la librairie fournit des prédicats de type :

```cpp
if (j["version"].is_number_integer()) {
    int v = j["version"].get<int>();
} else if (j["version"].is_string()) {
    // Certaines API retournent la version comme chaîne
    std::string v = j["version"].get<std::string>();
}
```

La liste complète des prédicats : `is_null()`, `is_boolean()`, `is_number()`, `is_number_integer()`, `is_number_unsigned()`, `is_number_float()`, `is_string()`, `is_array()`, `is_object()`, `is_structured()` (objet ou tableau) et `is_primitive()` (tout sauf objet et tableau).

## Itération sur les structures JSON

### Itération sur un objet

La méthode `.items()` retourne des paires clé-valeur, compatibles avec les structured bindings C++17 :

```cpp
json server = json::parse(R"({
    "host": "0.0.0.0",
    "port": 8080,
    "workers": 4
})");

for (auto& [key, value] : server.items()) {
    std::print("{} → {}\n", key, value.dump());
}
// host → "0.0.0.0"
// port → 8080
// workers → 4
```

### Itération sur un tableau

Un tableau JSON se parcourt directement comme un conteneur STL :

```cpp
json replicas = json::parse(R"(["replica-1", "replica-2", "replica-3"])");

for (const auto& replica : replicas) {
    std::print("Replica : {}\n", replica.get<std::string>());
}
```

Pour un accès indexé, `.size()` retourne le nombre d'éléments :

```cpp
for (std::size_t i = 0; i < replicas.size(); ++i) {
    std::print("[{}] {}\n", i, replicas[i].get<std::string>());
}
```

### Itération récursive

Pour parcourir l'intégralité d'un document JSON quelle que soit sa profondeur, la méthode `.flatten()` produit une représentation plate avec des JSON Pointers comme clés :

```cpp
json nested = json::parse(R"({
    "server": {
        "host": "0.0.0.0",
        "ports": [8080, 8443]
    }
})");

for (auto& [pointer, value] : nested.flatten().items()) {
    std::print("{} = {}\n", pointer, value.dump());
}
// /server/host = "0.0.0.0"
// /server/ports/0 = 8080
// /server/ports/1 = 8443
```

## JSON Pointer : accès par chemin

La RFC 6901 définit une syntaxe de chemin pour naviguer dans un document JSON. nlohmann/json la supporte nativement via le type `json::json_pointer` et l'opérateur `/` sous forme de littéral :

```cpp
json config = json::parse(R"({
    "services": {
        "api": {
            "endpoints": [
                {"path": "/health", "method": "GET"},
                {"path": "/data",   "method": "POST"}
            ]
        }
    }
})");

// Accès direct par JSON Pointer
std::string method = config["/services/api/endpoints/1/method"_json_pointer];
// method == "POST"

// Équivalent à l'accès chaîné
std::string same = config["services"]["api"]["endpoints"][1]["method"];
```

Les JSON Pointers sont particulièrement utiles pour accéder à des chemins profonds de manière concise, ou lorsque le chemin d'accès est lui-même une donnée (stockée dans une variable ou un fichier de configuration).

## Parsing strict vs tolérant

### Mode strict (par défaut)

Par défaut, `json::parse()` lève une exception `json::parse_error` si le JSON est invalide. Ce comportement est approprié quand le programme ne peut pas fonctionner avec des données corrompues :

```cpp
try {
    json j = json::parse("{ invalid json }");
} catch (const json::parse_error& e) {
    // e.what() contient :
    // [json.exception.parse_error.101] parse error at line 1, column 3:
    // syntax error while parsing object key - invalid literal;
    // last read: '{ i'
    std::print(stderr, "Erreur de parsing : {}\n", e.what());
    std::print(stderr, "Position : octet {}\n", e.byte);
}
```

L'exception `parse_error` fournit deux informations exploitables : le message détaillé (`what()`) qui inclut la ligne et la colonne de l'erreur, et le champ `byte` qui indique la position de l'octet fautif dans le flux d'entrée. Ces informations sont précieuses pour produire des messages d'erreur utiles à l'utilisateur qui a écrit le fichier de configuration.

### Mode tolérant (sans exception)

Dans certains contextes — parsing de données réseau non fiables, traitement par lots où un document invalide ne doit pas interrompre le flux — il est préférable d'éviter les exceptions. Le troisième paramètre de `json::parse()` contrôle ce comportement :

```cpp
// allow_exceptions = false
json j = json::parse(input_string, nullptr, false);

if (j.is_discarded()) {
    std::print(stderr, "JSON invalide, document ignoré\n");
    // continuer le traitement...
} else {
    // document valide, utilisation normale
    process(j);
}
```

Quand `allow_exceptions` vaut `false`, un JSON invalide produit un objet `json` dont la méthode `.is_discarded()` retourne `true`. Aucune exception n'est levée, mais l'information sur la nature de l'erreur est perdue. Ce mode est adapté aux cas où la seule question est « valide ou pas ? », sans besoin de diagnostic.

### Callback de parsing

Le deuxième paramètre de `json::parse()` accepte un callback invoqué à chaque événement de parsing. Il permet de filtrer ou transformer les données pendant le parsing, avant la construction du DOM :

```cpp
// Callback qui rejette les clés commençant par un underscore
auto filter = [](int depth, json::parse_event_t event, json& parsed) {
    if (event == json::parse_event_t::key) {
        const std::string& key = parsed.get<std::string>();
        if (!key.empty() && key[0] == '_') {
            return false;  // exclure cette clé et sa valeur
        }
    }
    return true;  // accepter tout le reste
};

std::string input = R"({"name": "app", "_internal": "secret", "version": 2})";  
json j = json::parse(input, filter);  
// j == {"name": "app", "version": 2}
// "_internal" a été filtré pendant le parsing
```

Les événements disponibles sont `object_start`, `object_end`, `array_start`, `array_end`, `key` et `value`. Le callback retourne `true` pour accepter l'élément ou `false` pour l'exclure du DOM résultant.

## Parsing SAX : traitement en streaming

Pour les documents JSON volumineux (dizaines ou centaines de Mo), charger l'intégralité du DOM en mémoire peut être prohibitif. nlohmann/json fournit une interface SAX (Simple API for XML, concept transposé au JSON) qui parse le document en émettant des événements, sans construire d'arbre en mémoire.

Le principe est d'implémenter une classe handler dont les méthodes sont appelées au fil du parsing :

```cpp
#include <nlohmann/json.hpp>
#include <print>

using json = nlohmann::json;

struct StatsHandler : json::json_sax_t {
    int object_count = 0;
    int array_count = 0;
    int string_count = 0;
    int number_count = 0;

    bool null() override { return true; }
    bool boolean(bool) override { return true; }

    bool number_integer(json::number_integer_t) override {
        ++number_count;
        return true;
    }

    bool number_unsigned(json::number_unsigned_t) override {
        ++number_count;
        return true;
    }

    bool number_float(json::number_float_t, const json::string_t&) override {
        ++number_count;
        return true;
    }

    bool string(json::string_t&) override {
        ++string_count;
        return true;
    }

    bool binary(json::binary_t&) override { return true; }

    bool start_object(std::size_t) override {
        ++object_count;
        return true;
    }

    bool end_object() override { return true; }
    bool key(json::string_t&) override { return true; }

    bool start_array(std::size_t) override {
        ++array_count;
        return true;
    }

    bool end_array() override { return true; }

    bool parse_error(std::size_t position, const std::string& last_token,
                     const json::exception& ex) override {
        std::print(stderr, "Erreur parsing position {} : {}\n",
                   position, ex.what());
        return false;  // arrêter le parsing
    }
};
```

L'utilisation du handler se fait via `json::sax_parse()` :

```cpp
std::ifstream large_file("events_100mb.json");

StatsHandler handler;  
bool success = json::sax_parse(large_file, &handler);  

if (success) {
    std::print("Objets  : {}\n", handler.object_count);
    std::print("Tableaux: {}\n", handler.array_count);
    std::print("Chaînes : {}\n", handler.string_count);
    std::print("Nombres : {}\n", handler.number_count);
}
```

### Cas d'usage du mode SAX

Le parsing SAX est adapté à des scénarios spécifiques où le mode DOM standard ne convient pas :

**Extraction ciblée.** Si un fichier JSON de 200 Mo contient des millions d'enregistrements mais que seuls quelques champs sont nécessaires, un handler SAX peut extraire uniquement les données pertinentes sans allouer de mémoire pour le reste du document.

**Validation sans chargement.** Vérifier qu'un fichier JSON est syntaxiquement valide sans construire le DOM. Un handler minimal qui retourne `true` pour tous les événements consomme une mémoire constante quelle que soit la taille du document.

**Transformation en streaming.** Lire un flux JSON entrant et produire une sortie transformée au fil de l'eau, sans jamais charger l'intégralité des données en mémoire. Ce pattern est courant dans les pipelines de traitement de données.

**Comptage et statistiques.** Comme dans l'exemple ci-dessus, calculer des métriques sur un document sans le matérialiser.

Dans la grande majorité des cas d'usage courants — fichiers de configuration, réponses d'API, données de taille modérée — le parsing DOM standard est largement suffisant et beaucoup plus simple à utiliser. Le mode SAX ne se justifie que lorsque la mémoire ou la performance sont des contraintes mesurées.

## Gestion des encodages

JSON est défini par la RFC 8259 comme étant encodé en **UTF-8** (les encodages UTF-16 et UTF-32 étaient autorisés par l'ancienne RFC 4627, mais la spécification actuelle les déconseille fortement). nlohmann/json attend de l'UTF-8 en entrée et produit de l'UTF-8 en sortie.

```cpp
// Les caractères Unicode fonctionnent directement en UTF-8
json j = json::parse(R"({"ville": "Héricourt-en-Caux", "emoji": "🚀"})");

std::string ville = j["ville"];  // "Héricourt-en-Caux"  
std::string emoji = j["emoji"];  // "🚀"  
```

Les séquences d'échappement Unicode (`\uXXXX`) sont automatiquement décodées lors du parsing et réencodées en UTF-8 :

```cpp
json j = json::parse(R"({"text": "\u00e9t\u00e9"})");  
std::string text = j["text"];  // "été"  
```

Si le flux d'entrée contient des octets invalides en UTF-8, le parser lève une exception `json::parse_error`. La librairie ne tente pas de corriger ou de deviner l'encodage — c'est la responsabilité de l'appelant de s'assurer que les données sont en UTF-8 valide avant le parsing.

## Parsing de JSON non standard : commentaires et trailing commas

La spécification JSON est stricte : ni les commentaires (`//`, `/* */`) ni les virgules finales (`[1, 2, 3,]`) ne sont autorisés. nlohmann/json respecte cette spécification par défaut et rejette ces constructions.

Cependant, de nombreux fichiers de configuration « JSON-like » dans la pratique utilisent ces extensions (les fichiers `tsconfig.json` de TypeScript, les fichiers `settings.json` de VS Code). nlohmann/json fournit un mode étendu pour les gérer :

```cpp
// JSON avec commentaires
std::string jsonc = R"({
    // Serveur principal
    "host": "0.0.0.0",
    "port": 8080,      /* port HTTP */
    "features": [
        "logging",
        "metrics"
    ]
})";

// Mode strict : lèverait une parse_error à cause des commentaires
// json j = json::parse(jsonc);

// Accepter les commentaires : utiliser le paramètre ignore_comments
json j = json::parse(jsonc,
                     nullptr,             // pas de callback
                     true,                // exceptions activées
                     true);               // ignorer les commentaires
```

> ⚠️ *Le paramètre `ignore_comments` gère les commentaires `//` et `/* */`, mais **pas les trailing commas** (`[1, 2,]` ou `{"a": 1,}`). Un JSON avec trailing commas provoquera une `parse_error` même avec `ignore_comments` activé. Pour un support complet du format JSONC, il faut pré-traiter le fichier ou utiliser un parser dédié. Si le format de configuration du projet nécessite commentaires et trailing commas, TOML (section 24.3) est souvent une meilleure alternative.*

## Résumé des méthodes de parsing

| Méthode | Source | Usage typique |
|---------|--------|---------------|
| `json::parse(std::ifstream&)` | Fichier | Configuration, données persistées |
| `json::parse(std::string)` | Chaîne | Réponses HTTP, messages réseau |
| `json::parse(begin, end)` | Itérateurs | Buffers binaires, portions de données |
| `json::parse(std::cin)` | stdin | Outils CLI, pipes Unix |
| `file >> j` | Fichier | Raccourci (moins de contrôle sur les erreurs) |
| `json::sax_parse(source, &handler)` | Toute source | Documents volumineux, streaming |

La section 24.1.4 détaillera la gestion complète des erreurs de parsing, incluant la hiérarchie des exceptions, les stratégies de récupération et les patterns de validation structurelle du contenu JSON après le parsing syntaxique.
