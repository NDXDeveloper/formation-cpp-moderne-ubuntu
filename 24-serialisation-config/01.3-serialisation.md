# 24.1.3 — Sérialisation d'objets C++

## Sous-section de 24.1 — JSON : Lecture/Écriture avec nlohmann/json

---

## Vue d'ensemble

La section 24.1.2 a montré comment parser du JSON en un objet `nlohmann::json` générique, puis naviguer dans l'arbre résultant. Mais dans un projet professionnel, on ne travaille pas avec des objets `json` bruts à travers toute la base de code. On définit des structures C++ qui représentent le domaine métier — `ServerConfig`, `UserProfile`, `DeploymentSpec` — et on convertit entre ces types et leur représentation JSON.

nlohmann/json fournit un système de conversion bidirectionnelle extensible qui permet d'écrire `json j = myObject;` et `auto obj = j.get<MyType>();` pour n'importe quel type utilisateur. Cette section couvre les différentes approches, de la plus simple (macros) à la plus flexible (fonctions manuelles), ainsi que la gestion des cas courants : champs optionnels, valeurs par défaut, héritage et types imbriqués.

---

## Le mécanisme fondamental : `to_json` / `from_json`

Le système de conversion repose sur deux fonctions libres que l'utilisateur définit dans le même namespace que son type. Lorsque nlohmann/json doit convertir un objet en JSON (ou inversement), il recherche ces fonctions par ADL (Argument-Dependent Lookup) :

```cpp
#include <nlohmann/json.hpp>
#include <string>

using json = nlohmann::json;

struct Endpoint {
    std::string path;
    std::string method;
    int timeout_ms;
};

// Sérialisation : Endpoint → JSON
void to_json(json& j, const Endpoint& e) {
    j = json{
        {"path", e.path},
        {"method", e.method},
        {"timeout_ms", e.timeout_ms}
    };
}

// Désérialisation : JSON → Endpoint
void from_json(const json& j, Endpoint& e) {
    j.at("path").get_to(e.path);
    j.at("method").get_to(e.method);
    j.at("timeout_ms").get_to(e.timeout_ms);
}
```

Une fois ces fonctions définies, la conversion est transparente :

```cpp
// Sérialisation
Endpoint ep{"/api/users", "GET", 5000};  
json j = ep;  // appelle to_json(j, ep) via ADL  
std::print("{}\n", j.dump(2));  
// {
//   "method": "GET",
//   "path": "/api/users",
//   "timeout_ms": 5000
// }

// Désérialisation
std::string input = R"({"path": "/health", "method": "GET", "timeout_ms": 1000})";  
auto ep2 = json::parse(input).get<Endpoint>();  // appelle from_json(j, ep2)  
std::print("{} {} ({}ms)\n", ep2.method, ep2.path, ep2.timeout_ms);  
```

### Pourquoi ADL et pas une interface/trait ?

Le choix d'ADL (fonctions libres dans le namespace du type) plutôt qu'une interface à implémenter ou un trait à spécialiser est délibéré. Il rend le mécanisme **non intrusif** : on peut définir `to_json`/`from_json` pour des types qu'on ne contrôle pas (types de librairies tierces, types du système), sans modifier leur définition. C'est un pattern central du C++ moderne, similaire à l'approche de `std::hash` ou des concepts C++20.

## Les macros : sérialisation déclarative

Écrire manuellement `to_json` et `from_json` pour chaque type est fastidieux et sujet à des erreurs de synchronisation (oublier un champ, se tromper de nom de clé). Pour les cas simples où chaque membre de la structure correspond directement à une clé JSON du même nom, nlohmann/json fournit des macros qui génèrent automatiquement ces fonctions.

### `NLOHMANN_DEFINE_TYPE_NON_INTRUSIVE`

La macro non intrusive se place **en dehors** de la définition de la classe. Elle ne nécessite aucune modification du type et ne peut accéder qu'aux membres publics :

```cpp
struct DatabaseConfig {
    std::string host;
    int port;
    std::string database;
    bool ssl;
};

// Génère to_json et from_json pour les 4 champs
NLOHMANN_DEFINE_TYPE_NON_INTRUSIVE(DatabaseConfig, host, port, database, ssl)
```

Tous les champs listés sont **obligatoires** à la désérialisation. Si le JSON ne contient pas l'un d'entre eux, une exception `json::out_of_range` est levée.

### `NLOHMANN_DEFINE_TYPE_NON_INTRUSIVE_WITH_DEFAULT`

Variante qui utilise les valeurs actuelles de l'objet comme valeurs par défaut. Les champs absents du JSON conservent leur valeur d'initialisation :

```cpp
struct DatabaseConfig {
    std::string host = "localhost";
    int port = 5432;
    std::string database = "app";
    bool ssl = false;
};

// Les champs absents du JSON prendront les valeurs par défaut de la struct
NLOHMANN_DEFINE_TYPE_NON_INTRUSIVE_WITH_DEFAULT(DatabaseConfig, host, port, database, ssl)
```

```cpp
// JSON partiel — seuls host et ssl sont spécifiés
auto cfg = json::parse(R"({"host": "db.prod.internal", "ssl": true})")
               .get<DatabaseConfig>();

// cfg.host     == "db.prod.internal"  (depuis le JSON)
// cfg.port     == 5432                (valeur par défaut)
// cfg.database == "app"               (valeur par défaut)
// cfg.ssl      == true                (depuis le JSON)
```

Cette variante est particulièrement adaptée aux fichiers de configuration où la plupart des champs ont des valeurs sensibles par défaut et où l'utilisateur ne spécifie que ce qu'il souhaite personnaliser.

### `NLOHMANN_DEFINE_TYPE_INTRUSIVE`

La macro intrusive se place **à l'intérieur** de la définition de la classe. Elle a accès aux membres privés, ce qui permet de sérialiser des classes avec encapsulation :

```cpp
class UserSession {
    std::string token_;
    std::string username_;
    int64_t expires_at_;

public:
    UserSession() = default;
    UserSession(std::string token, std::string user, int64_t exp)
        : token_(std::move(token)), username_(std::move(user)), expires_at_(exp) {}

    const std::string& token() const { return token_; }
    const std::string& username() const { return username_; }

    // Macro intrusive : accède aux membres privés
    NLOHMANN_DEFINE_TYPE_INTRUSIVE(UserSession, token_, username_, expires_at_)
};
```

> ⚠️ *Les noms passés à la macro deviennent les clés JSON. Avec la macro intrusive, les clés JSON auront le suffixe `_` (ex: `"token_"` au lieu de `"token"`). Si les noms de clés JSON doivent différer des noms de membres, il faut écrire les fonctions manuellement.*

La variante `NLOHMANN_DEFINE_TYPE_INTRUSIVE_WITH_DEFAULT` existe également et fonctionne de manière analogue à sa version non intrusive.

### Récapitulatif des macros

| Macro | Placement | Accès privé | Champs absents |
|-------|-----------|-------------|----------------|
| `NLOHMANN_DEFINE_TYPE_NON_INTRUSIVE` | Hors classe | Non | Exception |
| `NLOHMANN_DEFINE_TYPE_NON_INTRUSIVE_WITH_DEFAULT` | Hors classe | Non | Valeur par défaut |
| `NLOHMANN_DEFINE_TYPE_INTRUSIVE` | Dans classe | Oui | Exception |
| `NLOHMANN_DEFINE_TYPE_INTRUSIVE_WITH_DEFAULT` | Dans classe | Oui | Valeur par défaut |

## Fonctions manuelles : contrôle total

Les macros couvrent les cas simples où les noms de champs C++ correspondent aux clés JSON et où aucune logique de transformation n'est nécessaire. Pour tout le reste, on écrit `to_json` et `from_json` à la main.

### Renommage de clés

Quand le JSON utilise des conventions de nommage différentes du C++ (snake_case vs camelCase, ou noms imposés par une API externe) :

```cpp
struct ServiceHealth {
    std::string service_name;
    bool is_healthy;
    double response_time_ms;
};

void to_json(json& j, const ServiceHealth& s) {
    j = json{
        {"serviceName", s.service_name},         // camelCase pour l'API
        {"healthy", s.is_healthy},                // nom raccourci
        {"responseTimeMs", s.response_time_ms}
    };
}

void from_json(const json& j, ServiceHealth& s) {
    j.at("serviceName").get_to(s.service_name);
    j.at("healthy").get_to(s.is_healthy);
    j.at("responseTimeMs").get_to(s.response_time_ms);
}
```

### Champs optionnels avec `std::optional`

Pour les champs qui peuvent légitimement être absents du JSON, `std::optional` est le type C++ naturel. nlohmann/json le supporte nativement depuis la version 3.2.0 — un `std::optional<T>` est sérialisé comme la valeur `T` si présent, ou comme `null` si vide :

```cpp
struct DeploymentSpec {
    std::string image;
    int replicas;
    std::optional<std::string> namespace_override;
    std::optional<int> max_memory_mb;
};

void to_json(json& j, const DeploymentSpec& d) {
    j = json{
        {"image", d.image},
        {"replicas", d.replicas}
    };

    // N'inclure les champs optionnels que s'ils ont une valeur
    if (d.namespace_override.has_value()) {
        j["namespace"] = *d.namespace_override;
    }
    if (d.max_memory_mb.has_value()) {
        j["maxMemoryMb"] = *d.max_memory_mb;
    }
}

void from_json(const json& j, DeploymentSpec& d) {
    j.at("image").get_to(d.image);
    j.at("replicas").get_to(d.replicas);

    // Champs optionnels : présent → valeur, absent → std::nullopt
    if (j.contains("namespace")) {
        d.namespace_override = j["namespace"].get<std::string>();
    } else {
        d.namespace_override = std::nullopt;
    }

    if (j.contains("maxMemoryMb")) {
        d.max_memory_mb = j["maxMemoryMb"].get<int>();
    } else {
        d.max_memory_mb = std::nullopt;
    }
}
```

```cpp
// JSON sans les champs optionnels
auto spec = json::parse(R"({"image": "nginx:1.27", "replicas": 3})")
                .get<DeploymentSpec>();

// spec.namespace_override == std::nullopt
// spec.max_memory_mb      == std::nullopt

// JSON avec les champs optionnels
auto spec2 = json::parse(R"({
    "image": "api:2.1.0",
    "replicas": 5,
    "namespace": "staging",
    "maxMemoryMb": 512
})").get<DeploymentSpec>();

// spec2.namespace_override == "staging"
// spec2.max_memory_mb      == 512
```

### Pattern utilitaire pour les champs optionnels

Le code de désérialisation des champs optionnels est répétitif. Un petit helper template le factorise :

```cpp
template <typename T>  
void optional_from_json(const json& j, const std::string& key,  
                        std::optional<T>& target) {
    if (auto it = j.find(key); it != j.end() && !it->is_null()) {
        target = it->get<T>();
    } else {
        target = std::nullopt;
    }
}

// Utilisation dans from_json
void from_json(const json& j, DeploymentSpec& d) {
    j.at("image").get_to(d.image);
    j.at("replicas").get_to(d.replicas);
    optional_from_json(j, "namespace", d.namespace_override);
    optional_from_json(j, "maxMemoryMb", d.max_memory_mb);
}
```

### Transformation de valeurs

Certains champs nécessitent une conversion entre la représentation JSON et la représentation C++. Un cas courant est celui des énumérations :

```cpp
enum class LogLevel { trace, debug, info, warn, error, fatal };

struct LogConfig {
    std::string output;
    LogLevel level;
    bool colorize;
};

// Conversion enum ↔ string
NLOHMANN_JSON_SERIALIZE_ENUM(LogLevel, {
    {LogLevel::trace, "trace"},
    {LogLevel::debug, "debug"},
    {LogLevel::info,  "info"},
    {LogLevel::warn,  "warn"},
    {LogLevel::error, "error"},
    {LogLevel::fatal, "fatal"},
})

void to_json(json& j, const LogConfig& c) {
    j = json{
        {"output", c.output},
        {"level", c.level},       // sérialisé comme "info", "warn", etc.
        {"colorize", c.colorize}
    };
}

void from_json(const json& j, LogConfig& c) {
    j.at("output").get_to(c.output);
    j.at("level").get_to(c.level);  // désérialisé depuis la chaîne
    j.at("colorize").get_to(c.colorize);
}
```

```cpp
auto cfg = json::parse(R"({"output": "stdout", "level": "warn", "colorize": true})")
               .get<LogConfig>();
// cfg.level == LogLevel::warn

json j = cfg;  
std::print("{}\n", j.dump(2));  
// {
//   "colorize": true,
//   "level": "warn",
//   "output": "stdout"
// }
```

La macro `NLOHMANN_JSON_SERIALIZE_ENUM` génère les fonctions `to_json`/`from_json` pour l'enum. Si la chaîne JSON ne correspond à aucune valeur connue, la première entrée de la table de correspondance est utilisée comme valeur par défaut.

## Types imbriqués et composition

Le système de conversion est composable : si chaque type impliqué dispose de ses fonctions `to_json`/`from_json`, les types composites fonctionnent automatiquement.

```cpp
struct TlsConfig {
    std::string cert_path;
    std::string key_path;
    bool verify_client;
};

NLOHMANN_DEFINE_TYPE_NON_INTRUSIVE_WITH_DEFAULT(TlsConfig,
    cert_path, key_path, verify_client)

struct ServerConfig {
    std::string host;
    int port;
    std::optional<TlsConfig> tls;       // composition avec type optionnel
    std::vector<Endpoint> endpoints;     // vecteur de types sérialisables
};

void to_json(json& j, const ServerConfig& s) {
    j = json{
        {"host", s.host},
        {"port", s.port},
        {"endpoints", s.endpoints}       // vector<Endpoint> → JSON array
    };
    if (s.tls.has_value()) {
        j["tls"] = *s.tls;              // TlsConfig → JSON object
    }
}

void from_json(const json& j, ServerConfig& s) {
    j.at("host").get_to(s.host);
    j.at("port").get_to(s.port);
    j.at("endpoints").get_to(s.endpoints);  // JSON array → vector<Endpoint>
    optional_from_json(j, "tls", s.tls);
}
```

Ce qui rend possible la désérialisation d'un document complet en une seule ligne :

```cpp
std::string config_json = R"({
    "host": "0.0.0.0",
    "port": 443,
    "tls": {
        "cert_path": "/etc/ssl/server.crt",
        "key_path": "/etc/ssl/server.key",
        "verify_client": false
    },
    "endpoints": [
        {"path": "/health", "method": "GET", "timeout_ms": 1000},
        {"path": "/api/data", "method": "POST", "timeout_ms": 5000}
    ]
})";

auto server = json::parse(config_json).get<ServerConfig>();
// Toute l'arborescence est désérialisée récursivement
```

La composition fonctionne aussi avec les conteneurs standard de la STL. nlohmann/json fournit nativement les conversions pour `std::vector`, `std::list`, `std::deque`, `std::set`, `std::unordered_set`, `std::map`, `std::unordered_map`, `std::pair`, `std::tuple`, `std::optional`, `std::variant` et `std::array`. Tant que les types éléments sont eux-mêmes sérialisables, le conteneur l'est aussi.

## Héritage

La sérialisation de hiérarchies de classes nécessite un peu plus d'attention. Le cas le plus courant est une structure de base avec des champs communs, étendue par des types spécialisés.

### Héritage simple

```cpp
struct Resource {
    std::string id;
    std::string name;
    std::string created_at;
};

void to_json(json& j, const Resource& r) {
    j = json{{"id", r.id}, {"name", r.name}, {"created_at", r.created_at}};
}

void from_json(const json& j, Resource& r) {
    j.at("id").get_to(r.id);
    j.at("name").get_to(r.name);
    j.at("created_at").get_to(r.created_at);
}

struct Volume : Resource {
    int size_gb;
    std::string mount_path;
};

void to_json(json& j, const Volume& v) {
    // Sérialiser la partie base
    to_json(j, static_cast<const Resource&>(v));
    // Ajouter les champs spécifiques
    j["size_gb"] = v.size_gb;
    j["mount_path"] = v.mount_path;
}

void from_json(const json& j, Volume& v) {
    // Désérialiser la partie base
    from_json(j, static_cast<Resource&>(v));
    // Désérialiser les champs spécifiques
    j.at("size_gb").get_to(v.size_gb);
    j.at("mount_path").get_to(v.mount_path);
}
```

L'appel explicite aux fonctions de la classe de base avec un `static_cast` garantit que les champs hérités sont correctement traités sans duplication de code.

> 💡 *Depuis la version 3.12.0, nlohmann/json fournit des macros dédiées à l'héritage : `NLOHMANN_DEFINE_DERIVED_TYPE_NON_INTRUSIVE(Derived, Base, field1, field2, ...)` et sa variante `_WITH_DEFAULT`. Elles génèrent automatiquement les fonctions `to_json`/`from_json` en incluant les champs de la classe de base, éliminant le besoin du `static_cast` manuel.*

### Polymorphisme avec champ discriminant

Pour désérialiser un JSON en un type concret parmi plusieurs, on utilise un champ discriminant (souvent `"type"` ou `"kind"`) :

```cpp
struct Notification {
    std::string type;
    std::string message;
    std::string timestamp;
};

struct EmailNotification : Notification {
    std::string recipient;
    std::string subject;
};

struct SlackNotification : Notification {
    std::string channel;
    bool mention_here;
};

// Factory de désérialisation basée sur le champ "type"
std::unique_ptr<Notification> parse_notification(const json& j) {
    std::string type = j.at("type").get<std::string>();

    if (type == "email") {
        auto n = std::make_unique<EmailNotification>();
        j.at("message").get_to(n->message);
        j.at("timestamp").get_to(n->timestamp);
        j.at("recipient").get_to(n->recipient);
        j.at("subject").get_to(n->subject);
        n->type = type;
        return n;
    }

    if (type == "slack") {
        auto n = std::make_unique<SlackNotification>();
        j.at("message").get_to(n->message);
        j.at("timestamp").get_to(n->timestamp);
        j.at("channel").get_to(n->channel);
        j.at("mention_here").get_to(n->mention_here);
        n->type = type;
        return n;
    }

    throw std::invalid_argument("Type de notification inconnu : " + type);
}
```

Ce pattern de factory est préférable à une intégration directe dans `from_json` car le polymorphisme dynamique (retour de `unique_ptr<Base>`) ne se prête pas naturellement au mécanisme `get<T>()` qui instancie un type concret connu à la compilation.

## Écriture JSON vers fichier

La sérialisation vers un fichier est l'opération symétrique du parsing. Deux approches sont courantes :

```cpp
ServerConfig config{/* ... */};  
json j = config;  

// Méthode 1 : via std::ofstream et dump()
{
    std::ofstream file("config_output.json");
    file << j.dump(4);  // indenté avec 4 espaces
}

// Méthode 2 : via l'opérateur <<
{
    std::ofstream file("config_output.json");
    file << std::setw(4) << j;  // std::setw contrôle l'indentation
}
```

Les deux méthodes produisent le même résultat. La première est plus explicite, la seconde plus idiomatique pour les habitués des flux C++. Dans les deux cas, le fichier est automatiquement fermé à la sortie du bloc grâce au destructeur du `std::ofstream` (RAII, section 6.3).

### Contrôle de la sortie

La méthode `.dump()` accepte plusieurs paramètres pour contrôler le format de sortie :

```cpp
json j = config;

// Compact (une seule ligne, taille minimale)
std::string compact = j.dump();

// Indenté (lisible, pour configuration ou débogage)
std::string pretty = j.dump(4);

// Indenté avec tabulations
std::string tabbed = j.dump(1, '\t');

// Échappement des caractères non-ASCII (pour transport 7-bit safe)
std::string ascii_safe = j.dump(4, ' ', true);
// "Héricourt" → "H\u00e9ricourt"
```

## Sérialisation de `std::variant`

`std::variant` est supporté nativement à partir de la version 3.12.0, mais la désérialisation pose un défi : le JSON ne porte pas d'information de type explicite, et la librairie doit deviner quel type alternatif utiliser. Elle tente les conversions dans l'ordre de déclaration des types du variant :

```cpp
using ConfigValue = std::variant<std::string, int, double, bool>;

json j_str = "hello";  
json j_int = 42;  
json j_bool = true;  

auto v1 = j_str.get<ConfigValue>();   // → std::string  
auto v2 = j_int.get<ConfigValue>();   // → int  
auto v3 = j_bool.get<ConfigValue>();  // → bool (pas string)  
```

Ce comportement peut être surprenant avec certaines combinaisons de types. Si un contrôle précis est nécessaire, mieux vaut écrire des fonctions `to_json`/`from_json` manuelles avec un champ discriminant explicite plutôt que de se reposer sur la déduction automatique.

## Anti-patterns à éviter

### Exposer `nlohmann::json` dans les interfaces publiques

Faire apparaître le type `json` dans les headers publics du projet crée un couplage fort entre la librairie de sérialisation et l'ensemble du code :

```cpp
// ❌ Anti-pattern : json dans l'interface publique
class ConfigService {  
public:  
    nlohmann::json load_raw(const std::string& path);
    void save_raw(const nlohmann::json& data);
};

// ✅ Préférer : types métier dans l'interface, json dans l'implémentation
class ConfigService {  
public:  
    std::optional<ServerConfig> load(const std::string& path);
    bool save(const ServerConfig& config, const std::string& path);
};
```

La seconde approche isole la dépendance à nlohmann/json dans le `.cpp` de `ConfigService` et permet de changer de librairie de sérialisation (ou de format) sans impacter les consommateurs.

### Sérialiser sans tester le round-trip

Un piège courant est de tester la sérialisation et la désérialisation séparément, sans vérifier que le cycle complet préserve les données. Un test de round-trip minimal devrait toujours accompagner les fonctions de conversion :

```cpp
// Pattern de test round-trip
ServerConfig original{
    .host = "0.0.0.0",
    .port = 443,
    .tls = TlsConfig{"/etc/ssl/cert.pem", "/etc/ssl/key.pem", true},
    .endpoints = {
        {"/health", "GET", 1000},
        {"/api", "POST", 5000}
    }
};

json j = original;                         // sérialisation  
auto restored = j.get<ServerConfig>();     // désérialisation  

// Vérifier l'égalité champ par champ ou via operator==
assert(original.host == restored.host);  
assert(original.port == restored.port);  
// ...
```

### Ignorer les champs JSON inconnus

Par défaut, `from_json` avec les macros ou les fonctions manuelles basées sur `.at()` ignore silencieusement les clés JSON qui ne correspondent à aucun champ de la structure C++. Ce comportement est souvent souhaitable (compatibilité ascendante), mais peut masquer des fautes de frappe dans les fichiers de configuration. Si la stricte correspondance est requise, une validation explicite des clés est nécessaire — c'est le sujet de la section 24.5 (validation de schémas).

## Résumé des approches

| Approche | Complexité | Flexibilité | Cas d'usage |
|----------|-----------|-------------|-------------|
| Macro `NON_INTRUSIVE` | Minimale | Faible | Structs simples, champs publics, noms identiques |
| Macro `NON_INTRUSIVE_WITH_DEFAULT` | Minimale | Faible | Idem + champs optionnels avec défauts |
| Macro `INTRUSIVE` | Minimale | Faible | Classes avec membres privés |
| `SERIALIZE_ENUM` | Minimale | Faible | Enums ↔ chaînes |
| `to_json`/`from_json` manuelles | Moyenne | Totale | Renommage, transformation, logique conditionnelle |
| Factory + discriminant | Élevée | Totale | Polymorphisme dynamique |

Pour la majorité des types d'un projet, les macros `WITH_DEFAULT` couvrent le besoin. Les fonctions manuelles interviennent pour les cas nécessitant un mapping non trivial entre la structure C++ et la représentation JSON. La factory avec discriminant est réservée aux hiérarchies polymorphiques.
