# 24.6.3 — RE2 et PCRE2 : Alternatives runtime performantes

> **Niveau** : Avancé  
> **Prérequis** : Section 24.6.1 (std::regex), section 24.6.2 (CTRE), chapitre 27 (gestion des dépendances)  
> **Bibliothèques** : [RE2](https://github.com/google/re2) (Google), [PCRE2](https://github.com/PCRE2Project/pcre2) (Philip Hazel et al.)  
> **Licences** : BSD-3-Clause (RE2), BSD-3-Clause (PCRE2)

## Contexte : quand le pattern n'est pas connu à la compilation

CTRE (section 24.6.2) est la solution idéale quand le pattern est un littéral du code source. Mais dans de nombreuses situations réelles, le pattern est **dynamique** : lu depuis un fichier de configuration, saisi par l'utilisateur dans un champ de recherche, construit par programme à partir de paramètres variables.

Dans ces cas, il faut un moteur qui compile le pattern à l'exécution — mais qui le fasse de manière **performante et sûre**, ce que `std::regex` ne garantit pas. C'est le créneau de RE2 et PCRE2 : deux bibliothèques C matures, largement déployées en production, qui offrent chacune un compromis différent entre fonctionnalités et garanties de performance.

---

## RE2 : le choix de la sécurité et de la prévisibilité

### Philosophie

RE2 a été créé chez Google par Russ Cox pour résoudre un problème concret : les moteurs regex à backtracking (dont `std::regex`) sont vulnérables au backtracking catastrophique, rendant tout service acceptant des regex utilisateur exposé à des attaques ReDoS.

La réponse de RE2 est radicale : **tout pattern est exécuté en temps linéaire** par rapport à la taille de l'entrée, quelle que soit la complexité du pattern. Pour garantir cette propriété, RE2 utilise un automate DFA (Deterministic Finite Automaton) construit à la demande, combiné à un NFA pour les cas où le DFA deviendrait trop volumineux. En contrepartie, RE2 renonce aux fonctionnalités qui ne sont pas exprimables par un automate fini : les backreferences et certains types de lookaround.

### API de base

L'API C++ de RE2 est concise et conçue pour couvrir 90 % des cas avec un minimum de code. Le header principal est `<re2/re2.h>`.

**Correspondance totale avec `RE2::FullMatch`** :

```cpp
#include <re2/re2.h>
#include <string>
#include <print>

void demo_fullmatch() {
    std::string input = "2026-03-21";

    // Validation simple (pas de capture)
    if (RE2::FullMatch(input, R"(\d{4}-\d{2}-\d{2})")) {
        std::println("Format de date valide");
    }

    // Avec extraction de captures
    int year, month, day;
    if (RE2::FullMatch(input, R"((\d{4})-(\d{2})-(\d{2}))", &year, &month, &day)) {
        std::println("Date : {}/{}/{}", day, month, year);
    }
}
```

Plusieurs choses à noter dans cet exemple. Les captures sont extraites directement dans des variables typées via des pointeurs. RE2 effectue la conversion automatiquement pour les types `int`, `float`, `double`, `std::string` et `re2::StringPiece`. Si la conversion échoue (par exemple un nombre trop grand pour un `int`), le match échoue.

**Recherche partielle avec `RE2::PartialMatch`** :

```cpp
std::string log = "[ERROR] Connection refused on port 5432";

std::string level;
int port;

if (RE2::PartialMatch(log, R"(\[(\w+)\].*port (\d+))", &level, &port)) {
    std::println("Niveau: {}, Port: {}", level, port);
    // Sortie : Niveau: ERROR, Port: 5432
}
```

`PartialMatch` cherche le pattern n'importe où dans le texte — c'est l'équivalent de `std::regex_search` ou `ctre::search`.

**Ignorer certaines captures** :

Quand on ne s'intéresse pas à tous les groupes, il suffit de passer `nullptr` pour les captures à ignorer — ou d'utiliser des groupes non-capturants `(?:...)` :

```cpp
std::string text = "John Doe, age 42";

// On ne veut que l'âge, pas le nom
int age;
if (RE2::PartialMatch(text, R"((\w+) (\w+), age (\d+))", nullptr, nullptr, &age)) {
    std::println("Âge : {}", age);
}
```

### Pré-compiler un pattern : l'objet `RE2`

Les fonctions statiques `FullMatch` et `PartialMatch` acceptent aussi bien un littéral `const char*` qu'un objet `RE2` pré-compilé. Quand le même pattern est utilisé dans une boucle, la pré-compilation est indispensable :

```cpp
// ❌ Anti-pattern : recompilation implicite à chaque appel
for (const auto& line : lines) {
    RE2::PartialMatch(line, R"(ERROR|FATAL)");  // Pattern recompilé à chaque itération
}

// ✅ Correct : compilation unique
static const RE2 error_pattern(R"(ERROR|FATAL)");
// Vérification de la validité du pattern
if (!error_pattern.ok()) {
    std::println(stderr, "Pattern invalide : {}", error_pattern.error());
    return;
}

for (const auto& line : lines) {
    if (RE2::PartialMatch(line, error_pattern)) {
        process_error(line);
    }
}
```

L'objet `RE2` est thread-safe en lecture : un même pattern compilé peut être utilisé simultanément par plusieurs threads sans synchronisation. C'est un avantage important pour les serveurs multi-threadés.

### `re2::StringPiece` : vues sans copie

RE2 utilise `re2::StringPiece` comme type de vue sur les chaînes, similaire à `std::string_view`. Ce type permet d'extraire des sous-chaînes sans allocation :

```cpp
re2::StringPiece input("Date: 2026-03-21, Rev: 42");
re2::StringPiece date_part;

if (RE2::PartialMatch(input, R"((\d{4}-\d{2}-\d{2}))", &date_part)) {
    // date_part pointe dans la mémoire de input — pas de copie
    std::println("Date : {}", std::string_view(date_part.data(), date_part.size()));
}
```

> 💡 Dans les versions récentes de RE2, `re2::StringPiece` est un alias de `std::string_view` si le compilateur le supporte. La conversion entre les deux est transparente.

### Itération avec `RE2::FindAndConsume`

Pour trouver toutes les correspondances d'un pattern dans un texte, RE2 propose `FindAndConsume` qui avance un curseur (`StringPiece`) à travers le texte :

```cpp
std::string text = "Scores: 95, 42, 78, 100, 60";
re2::StringPiece input(text);

int score;
static const RE2 pattern(R"((\d+))");

std::println("Tous les scores :");
while (RE2::FindAndConsume(&input, pattern, &score)) {
    std::println("  {}", score);
}
// Sortie :
//   95
//   42
//   78
//   100
//   60
```

Le curseur `input` est avancé après chaque correspondance. La boucle se termine quand il ne reste plus de texte à examiner ou quand aucune correspondance n'est trouvée. Il existe aussi `FindAndConsumeN` pour un contrôle plus fin du nombre de captures extraites par itération.

### Remplacement

RE2 propose des fonctions de remplacement efficaces :

```cpp
std::string text = "Hello World, Hello Universe";

// Remplacer la première occurrence
RE2::Replace(&text, R"(Hello)", "Goodbye");
std::println("{}", text);  // "Goodbye World, Hello Universe"

// Remplacer toutes les occurrences
text = "Hello World, Hello Universe";
RE2::GlobalReplace(&text, R"(Hello)", "Goodbye");
std::println("{}", text);  // "Goodbye World, Goodbye Universe"

// Avec des références aux captures
text = "Doe, John";
RE2::Replace(&text, R"((\w+), (\w+))", R"(\2 \1)");
std::println("{}", text);  // "John Doe"
```

> ⚠️ Contrairement à `std::regex_replace` qui utilise `$1`, `$2` pour les captures, RE2 utilise `\1`, `\2` dans les chaînes de remplacement.

### Options de configuration

L'objet `RE2::Options` permet de contrôler finement le comportement du moteur :

```cpp
RE2::Options opts;
opts.set_case_sensitive(false);       // Insensible à la casse
opts.set_dot_nl(true);                // '.' correspond aussi à '\n'
opts.set_max_mem(1 << 24);            // Limite mémoire DFA : 16 Mo (défaut : 8 Mo)
opts.set_log_errors(false);           // Désactiver les logs d'erreur sur stderr

RE2 pattern(R"(error:\s+(.+))", opts);
```

L'option `max_mem` est particulièrement importante. Elle limite la mémoire que le DFA peut consommer. Si la limite est atteinte, RE2 bascule sur son moteur NFA (plus lent mais borné en mémoire). Cela garantit que même des patterns complexes ne peuvent pas épuiser la mémoire du processus.

### Fonctionnalités regex supportées et restrictions

La principale restriction de RE2 découle de son architecture DFA :

| Fonctionnalité | Supportée ? | Notes |
|---|---|---|
| Classes de caractères | ✅ | `\d`, `\w`, `\s`, `[a-z]`, Unicode |
| Quantificateurs | ✅ | `*`, `+`, `?`, `{n,m}`, non-gourmands |
| Alternation | ✅ | `a\|b` |
| Groupes de capture | ✅ | `(...)`, nommés `(?P<name>...)` |
| Groupes non-capturants | ✅ | `(?:...)` |
| Ancres | ✅ | `^`, `$`, `\b`, `\B` |
| Lookahead | ❌ | Non exprimable en DFA |
| Lookbehind | ❌ | Non exprimable en DFA |
| Backreferences | ❌ | Non exprimable en DFA |
| Récursion | ❌ | Non exprimable en DFA |
| Possessive quantifiers | ❌ | `*+`, `++`, etc. |
| Unicode properties | ✅ | `\p{L}`, `\p{Greek}`, etc. |

L'absence de lookaround et de backreferences est la limitation la plus fréquemment rencontrée en pratique. Si votre pattern en a besoin, PCRE2 est l'alternative.

## PCRE2 : puissance maximale avec JIT

### Philosophie

PCRE2 (Perl Compatible Regular Expressions, version 2) est le successeur de PCRE, la bibliothèque de référence pour les regex compatibles Perl. Elle est utilisée, entre autres, par PHP, Nginx, Apache, R et de nombreux outils système.

Là où RE2 sacrifie des fonctionnalités pour garantir un temps linéaire, PCRE2 fait le choix inverse : **offrir le jeu de fonctionnalités le plus complet** (lookahead, lookbehind, backreferences, récursion, conditionnels, callouts) avec un moteur à backtracking hautement optimisé et un compilateur JIT (Just-In-Time) qui transforme le pattern en code machine natif à l'exécution.

Le compromis : PCRE2 est théoriquement vulnérable au backtracking catastrophique sur des patterns pathologiques. Mais elle offre des mécanismes de mitigation (limites de backtracking, limites de récursion) qui rendent ce risque gérable quand les patterns sont contrôlés.

### L'API C de PCRE2

PCRE2 est une bibliothèque C. Son API est fonctionnelle mais verbeuse — typique du C avec gestion manuelle des ressources. L'usage direct en C++ bénéficie grandement d'un wrapper RAII.

**Utilisation directe (API C)** :

```cpp
#define PCRE2_CODE_UNIT_WIDTH 8
#include <pcre2.h>
#include <string>
#include <print>

void demo_pcre2_raw() {
    const std::string pattern_str = R"((\d{4})-(\d{2})-(\d{2}))";
    const std::string subject = "Date: 2026-03-21";

    int errornumber;
    PCRE2_SIZE erroroffset;

    // Compilation du pattern
    pcre2_code* re = pcre2_compile(
        reinterpret_cast<PCRE2_SPTR>(pattern_str.c_str()),
        PCRE2_ZERO_TERMINATED,
        0,                          // Options
        &errornumber,
        &erroroffset,
        nullptr                     // Contexte de compilation
    );

    if (re == nullptr) {
        PCRE2_UCHAR buffer[256];
        pcre2_get_error_message(errornumber, buffer, sizeof(buffer));
        std::println(stderr, "Erreur compilation offset {} : {}",
                     erroroffset, reinterpret_cast<char*>(buffer));
        return;
    }

    // Activation du JIT (fortement recommandé)
    pcre2_jit_compile(re, PCRE2_JIT_COMPLETE);

    // Création du bloc de résultats
    pcre2_match_data* match_data = pcre2_match_data_create_from_pattern(re, nullptr);

    // Exécution
    int rc = pcre2_match(
        re,
        reinterpret_cast<PCRE2_SPTR>(subject.c_str()),
        subject.size(),
        0,                          // Offset de départ
        0,                          // Options
        match_data,
        nullptr                     // Contexte de match
    );

    if (rc > 0) {
        PCRE2_SIZE* ovector = pcre2_get_ovector_pointer(match_data);

        for (int i = 0; i < rc; ++i) {
            PCRE2_SIZE start = ovector[2 * i];
            PCRE2_SIZE end   = ovector[2 * i + 1];
            std::println("Groupe {} : {}", i,
                         subject.substr(start, end - start));
        }
    }

    // Nettoyage obligatoire
    pcre2_match_data_free(match_data);
    pcre2_code_free(re);
}
```

Ce code fonctionne mais illustre pourquoi un wrapper est souhaitable : il y a cinq appels d'API pour un simple match, plus la gestion manuelle de deux ressources.

### Wrapper RAII pour un usage idiomatique en C++

Voici un wrapper minimaliste qui rend l'utilisation de PCRE2 naturelle en C++ moderne :

```cpp
#define PCRE2_CODE_UNIT_WIDTH 8
#include <pcre2.h>
#include <string>
#include <string_view>
#include <vector>
#include <stdexcept>
#include <optional>

class Regex {
public:
    explicit Regex(std::string_view pattern, uint32_t options = 0) {
        int errornumber;
        PCRE2_SIZE erroroffset;

        code_ = pcre2_compile(
            reinterpret_cast<PCRE2_SPTR>(pattern.data()),
            pattern.size(),
            options,
            &errornumber,
            &erroroffset,
            nullptr
        );

        if (!code_) {
            PCRE2_UCHAR buf[256];
            pcre2_get_error_message(errornumber, buf, sizeof(buf));
            throw std::runtime_error(
                "Regex compile error at offset " + std::to_string(erroroffset) +
                ": " + reinterpret_cast<char*>(buf)
            );
        }

        // JIT — silencieusement ignoré si non disponible
        pcre2_jit_compile(code_, PCRE2_JIT_COMPLETE);

        match_data_ = pcre2_match_data_create_from_pattern(code_, nullptr);
    }

    ~Regex() {
        if (match_data_) pcre2_match_data_free(match_data_);
        if (code_)       pcre2_code_free(code_);
    }

    // Non copiable, déplaçable
    Regex(const Regex&) = delete;
    Regex& operator=(const Regex&) = delete;

    Regex(Regex&& other) noexcept
        : code_(other.code_), match_data_(other.match_data_) {
        other.code_ = nullptr;
        other.match_data_ = nullptr;
    }

    Regex& operator=(Regex&& other) noexcept {
        if (this != &other) {
            if (match_data_) pcre2_match_data_free(match_data_);
            if (code_)       pcre2_code_free(code_);
            code_ = other.code_;
            match_data_ = other.match_data_;
            other.code_ = nullptr;
            other.match_data_ = nullptr;
        }
        return *this;
    }

    /// Recherche partielle — retourne les captures ou nullopt
    auto search(std::string_view subject, PCRE2_SIZE start = 0) const
        -> std::optional<std::vector<std::string_view>>
    {
        int rc = pcre2_match(
            code_,
            reinterpret_cast<PCRE2_SPTR>(subject.data()),
            subject.size(),
            start, 0, match_data_, nullptr
        );

        if (rc < 0) return std::nullopt;

        PCRE2_SIZE* ov = pcre2_get_ovector_pointer(match_data_);
        std::vector<std::string_view> captures;
        captures.reserve(static_cast<std::size_t>(rc));

        for (int i = 0; i < rc; ++i) {
            if (ov[2 * i] == PCRE2_UNSET) {
                captures.emplace_back();
            } else {
                captures.emplace_back(
                    subject.data() + ov[2 * i],
                    ov[2 * i + 1] - ov[2 * i]
                );
            }
        }
        return captures;
    }

    /// Correspondance totale
    auto match(std::string_view subject) const
        -> std::optional<std::vector<std::string_view>>
    {
        auto result = search(subject);
        if (result && !result->empty() && (*result)[0] == subject) {
            return result;
        }
        return std::nullopt;
    }

private:
    pcre2_code*       code_       = nullptr;
    pcre2_match_data* match_data_ = nullptr;
};
```

Avec ce wrapper, l'utilisation devient concise :

```cpp
#include <print>

void demo_pcre2_wrapper() {
    // Pattern avec lookbehind — impossible avec RE2
    Regex price_pattern(R"((?<=\$)\d+(?:\.\d{2})?)");

    std::string_view text = "Prix: $42.99 et $100 et €50";

    // Itération manuelle sur toutes les correspondances
    PCRE2_SIZE offset = 0;
    while (auto captures = price_pattern.search(text, offset)) {
        auto full = (*captures)[0];
        std::println("Prix USD trouvé : {}", full);
        offset = static_cast<PCRE2_SIZE>(
            full.data() - text.data() + full.size()
        );
    }
    // Sortie :
    //   Prix USD trouvé : 42.99
    //   Prix USD trouvé : 100
}
```

> 💡 **Note architecturale** : ce wrapper n'est pas thread-safe pour les appels concurrents sur la **même instance** car `match_data_` est partagé. Pour un usage multi-thread, il faut soit un `match_data_` par thread (via `thread_local`), soit un verrou. L'objet `pcre2_code*` compilé, lui, est thread-safe en lecture — comme pour RE2.

### Le JIT : ce qui fait la différence en performance

L'appel `pcre2_jit_compile()` est la clé des performances de PCRE2. Il transforme le pattern compilé en code machine natif via un compilateur JIT intégré (disponible sur x86, x86_64, ARM, ARM64, MIPS, PowerPC, s390x).

Sans JIT, PCRE2 interprète un bytecode interne — similaire en vitesse à un bon moteur à backtracking. Avec JIT, les performances sont typiquement **3 à 10 fois meilleures** et rivalisent avec RE2 sur les patterns sans backtracking.

```
Benchmark — Pattern: (\d{4})-(\d{2})-(\d{2})
Entrée  : ligne de log de 120 caractères

  std::regex_search     :  ~2 800 ns/op
  PCRE2 (interprété)    :    ~350 ns/op
  PCRE2 (JIT)           :    ~120 ns/op
  RE2::PartialMatch     :    ~150 ns/op
  ctre::search          :     ~30 ns/op
```

Le JIT est disponible sur Ubuntu par défaut dans le paquet `libpcre2-dev`. Pour vérifier qu'il est activé :

```cpp
uint32_t jit_available = 0;
pcre2_config(PCRE2_CONFIG_JIT, &jit_available);
// jit_available == 1 si le JIT est supporté
```

### Fonctionnalités avancées exclusives à PCRE2

PCRE2 est le seul moteur de cette section à supporter l'intégralité des fonctionnalités Perl :

**Backreferences** — référencer une capture précédente dans le pattern :

```cpp
// Trouver les balises HTML avec ouverture/fermeture correspondantes
Regex tag_pattern(R"(<(\w+)[^>]*>.*?</\1>)", PCRE2_DOTALL);

std::string_view html = "<div>contenu</div> et <span>texte</span>";
auto result = tag_pattern.search(html);
// Capture[0]: "<div>contenu</div>", Capture[1]: "div"
```

**Lookbehind de longueur variable** (depuis PCRE2 10.36) :

```cpp
// Extraire un montant précédé de "prix:" ou "coût:" de longueur variable
Regex amount(R"((?<=(?:prix|coût)\s*:\s*)\d+(?:\.\d+)?)");
```

**Récursion** — patterns récursifs pour des structures imbriquées :

```cpp
// Matcher des parenthèses équilibrées de profondeur arbitraire
Regex balanced_parens(R"(\((?:[^()]*|(?R))*\))");

std::string_view expr = "f(a, g(b, h(c)), d)";
auto result = balanced_parens.search(expr);
// Capture[0]: "(a, g(b, h(c)), d)"
```

C'est une capacité que ni RE2, ni CTRE, ni `std::regex` ne peuvent offrir.

**Conditionnels** — branches basées sur la réussite d'une capture :

```cpp
// Si le groupe 1 a capturé une quote ouvrante, exiger la fermante correspondante
Regex quoted_or_bare(R"((["'])?mot(?(1)\1))");
// Matche: "mot", 'mot', mot
// Ne matche pas: "mot', 'mot"
```

**Limites de backtracking** — protection contre ReDoS :

```cpp
pcre2_match_context* mctx = pcre2_match_context_create(nullptr);
pcre2_set_match_limit(mctx, 100'000);           // Limite d'itérations
pcre2_set_depth_limit(mctx, 500);               // Limite de profondeur récursive

int rc = pcre2_match(code, subject, len, 0, 0, match_data, mctx);

if (rc == PCRE2_ERROR_MATCHLIMIT) {
    // Le matching a été interrompu — pattern potentiellement dangereux
}

pcre2_match_context_free(mctx);
```

## RE2 vs PCRE2 : guide de choix

### Arbre de décision

La question centrale est : **le pattern peut-il provenir d'une source non fiable ?**

Si **oui** (utilisateur, API externe, fichier non contrôlé) : choisir RE2. Sa garantie de temps linéaire élimine le risque ReDoS sans configuration supplémentaire. C'est le choix par défaut pour les serveurs, les API publiques, et tout système exposé.

Si **non** (pattern contrôlé par le développeur, lu depuis une config interne) : le choix dépend des fonctionnalités nécessaires. Si le pattern utilise des backreferences, du lookbehind, de la récursion ou des conditionnels, PCRE2 est la seule option. Sinon, RE2 reste un bon choix pour sa simplicité, ou PCRE2 avec JIT pour des performances maximales.

### Comparaison synthétique

| Critère | RE2 | PCRE2 |
|---|---|---|
| **Garantie temps linéaire** | ✅ Toujours | ❌ Backtracking (atténuable) |
| **Protection ReDoS** | ✅ Native | ⚠️ Via `match_limit` |
| **Backreferences** | ❌ | ✅ |
| **Lookahead** | ❌ | ✅ |
| **Lookbehind** | ❌ | ✅ (longueur variable) |
| **Récursion** | ❌ | ✅ |
| **Conditionnels** | ❌ | ✅ |
| **Compilateur JIT** | ❌ (DFA à la volée) | ✅ |
| **API C++** | ✅ Native | ❌ API C (wrapper nécessaire) |
| **Thread-safety** | ✅ Pattern partageable | ✅ Code partageable |
| **Unicode** | ✅ Complet | ✅ Complet |
| **Taille binaire** | ~1-2 Mo | ~0.5-1 Mo (+ JIT) |
| **Projets notables** | Google, Envoy, Istio, Prometheus | Nginx, PHP, grep (GNU), R |

### Quand les deux se valent

Pour la majorité des patterns du quotidien (extraction de champs, validation de formats, tokenisation simple), RE2 et PCRE2 offrent des performances et une ergonomie comparables. Dans ces cas, la recommandation est de choisir en fonction de l'écosystème du projet :

- Déjà dépendant de l'écosystème Google (Protobuf, gRPC, Abseil) → RE2 s'intègre naturellement.  
- Besoin de fonctionnalités Perl avancées ou déjà dépendant de PCRE → PCRE2.  
- Incertitude → RE2, car sa garantie de temps linéaire est une assurance gratuite.

## Exemple comparatif : le même problème avec les deux moteurs

Pour illustrer les différences d'API, voici le même cas d'usage — extraction de paires clé-valeur depuis une chaîne de configuration — implémenté avec RE2 puis PCRE2 :

### Version RE2

```cpp
#include <re2/re2.h>
#include <string>
#include <map>
#include <print>

auto parse_config_re2(std::string_view input) -> std::map<std::string, std::string> {
    static const RE2 pattern(R"((\w+)\s*=\s*"([^"]*)")");

    std::map<std::string, std::string> config;
    re2::StringPiece text(input.data(), input.size());

    std::string key, value;
    while (RE2::FindAndConsume(&text, pattern, &key, &value)) {
        config[key] = value;
    }
    return config;
}

void demo_re2() {
    auto cfg = parse_config_re2(
        R"(host = "db.local"  port = "5432"  db = "myapp")"
    );
    for (const auto& [k, v] : cfg) {
        std::println("{} = {}", k, v);
    }
}
```

### Version PCRE2 (avec le wrapper)

```cpp
#include <print>
#include <map>

auto parse_config_pcre2(std::string_view input) -> std::map<std::string, std::string> {
    Regex pattern(R"((\w+)\s*=\s*"([^"]*)")");

    std::map<std::string, std::string> config;
    PCRE2_SIZE offset = 0;

    while (auto captures = pattern.search(input, offset)) {
        auto& caps = *captures;
        config[std::string(caps[1])] = std::string(caps[2]);
        offset = static_cast<PCRE2_SIZE>(
            caps[0].data() - input.data() + caps[0].size()
        );
    }
    return config;
}

void demo_pcre2() {
    auto cfg = parse_config_pcre2(
        R"(host = "db.local"  port = "5432"  db = "myapp")"
    );
    for (const auto& [k, v] : cfg) {
        std::println("{} = {}", k, v);
    }
}
```

Les deux versions produisent le même résultat. On observe que l'API RE2 est plus concise grâce à l'extraction typée directe et à `FindAndConsume`. L'API PCRE2 brute est plus verbeuse, mais le wrapper RAII la rend comparable. Le choix entre les deux se fait donc sur les critères fonctionnels et de sécurité, pas sur l'ergonomie.

---

## Bonnes pratiques communes aux deux moteurs

**1. Toujours pré-compiler les patterns.** Créer un objet `RE2` ou `pcre2_code*` une seule fois et le réutiliser. Le coût de compilation d'un pattern est de l'ordre de la microseconde pour RE2 et de la dizaine de microsecondes pour PCRE2 avec JIT — négligeable pour un seul appel, significatif dans une boucle.

**2. Activer le JIT pour PCRE2.** L'appel `pcre2_jit_compile()` est simple et les gains sont substantiels. Il n'y a pratiquement aucune raison de ne pas le faire.

**3. Configurer des limites de backtracking pour PCRE2** quand les patterns ou les entrées ne sont pas entièrement contrôlés. `pcre2_set_match_limit()` est votre filet de sécurité.

**4. Vérifier la validité du pattern après compilation.** `RE2::ok()` pour RE2, le code de retour de `pcre2_compile` pour PCRE2. Ne jamais ignorer ces vérifications, surtout avec des patterns dynamiques.

**5. Utiliser des vues (`StringPiece`, `string_view`) plutôt que des `std::string`** pour les captures quand c'est possible. Cela évite des allocations inutiles quand on ne fait que lire les résultats.

**6. Préférer les groupes non-capturants** `(?:...)` quand une capture n'est pas nécessaire. Les deux moteurs optimisent mieux les patterns avec moins de captures actives.

---

> 💡 **Résumé** : RE2 est le choix sûr par défaut pour les patterns dynamiques — temps linéaire garanti, API C++ propre, thread-safe. PCRE2 est le recours quand les fonctionnalités avancées de Perl sont nécessaires — backreferences, lookaround, récursion — avec des performances excellentes grâce au JIT. Les deux surpassent `std::regex` d'au moins un ordre de grandeur. Le choix entre les deux est un compromis entre garanties de sécurité (RE2) et richesse fonctionnelle (PCRE2).
