# 24.1.4 — Gestion des erreurs de parsing

## Sous-section de 24.1 — JSON : Lecture/Écriture avec nlohmann/json

---

## Vue d'ensemble

Les sections précédentes ont montré comment parser, naviguer et convertir du JSON. Mais dans un programme en production, les données ne sont pas toujours conformes à ce qu'on attend. Un fichier de configuration peut contenir une faute de syntaxe, une réponse d'API peut omettre un champ obligatoire, un message réseau peut arriver tronqué, ou un utilisateur peut fournir une chaîne là où un entier est attendu.

nlohmann/json distingue clairement **trois catégories d'erreurs**, chacune représentée par un type d'exception dédié. Comprendre cette hiérarchie et savoir quand capturer chaque type est essentiel pour produire du code robuste et des messages d'erreur exploitables.

---

## Hiérarchie des exceptions

Toutes les exceptions de nlohmann/json héritent de `json::exception`, elle-même dérivée de `std::exception`. Chaque exception porte un identifiant numérique unique (accessible via `.id`) et un message détaillé (accessible via `.what()`) :

```
std::exception
  └── json::exception
        ├── json::parse_error      (id: 101)
        ├── json::invalid_iterator (id: 201-214)
        ├── json::type_error       (id: 301-313)
        ├── json::out_of_range     (id: 401-403)
        └── json::other_error      (id: 501)
```

En pratique, trois de ces types couvrent la quasi-totalité des situations rencontrées dans un projet.

### `json::parse_error` — Le JSON est syntaxiquement invalide

Levée lorsque le flux d'entrée ne constitue pas du JSON valide. C'est l'erreur la plus fondamentale : le parser ne parvient pas à transformer les caractères en arbre JSON.

```cpp
try {
    // Accolade fermante manquante
    json j = json::parse(R"({"host": "localhost", "port": 8080)");
} catch (const json::parse_error& e) {
    std::print(stderr, "Erreur de syntaxe JSON\n");
    std::print(stderr, "  Message  : {}\n", e.what());
    std::print(stderr, "  ID       : {}\n", e.id);
    std::print(stderr, "  Position : octet {}\n", e.byte);
}
```

Sortie typique :

```
Erreur de syntaxe JSON
  Message  : [json.exception.parse_error.101] parse error at line 1, column 35:
             syntax error while parsing object - unexpected end of input; expected '}'
  ID       : 101
  Position : octet 35
```

Le champ `byte` indique la position exacte dans le flux où l'erreur a été détectée. Combiné avec le message, il permet de construire un diagnostic précis pour l'utilisateur.

**Causes courantes :** virgule manquante ou en trop, guillemets non fermés, accolades/crochets déséquilibrés, utilisation de guillemets simples (`'`) au lieu de doubles (`"`), caractères de contrôle non échappés dans les chaînes, fichier tronqué.

### `json::type_error` — Le type JSON ne correspond pas au type C++ demandé

Levée lorsqu'une opération est appliquée à un type JSON incompatible. C'est l'erreur la plus fréquente après le parsing, typiquement lors de l'extraction typée.

```cpp
json j = json::parse(R"({"port": "not_a_number"})");

try {
    int port = j["port"].get<int>();  // "not_a_number" n'est pas un entier
} catch (const json::type_error& e) {
    std::print(stderr, "Type inattendu : {}\n", e.what());
    // [json.exception.type_error.302] type must be number, but is string
}
```

```cpp
json j = json::parse(R"({"items": "not_an_array"})");

try {
    auto items = j["items"].get<std::vector<int>>();
} catch (const json::type_error& e) {
    std::print(stderr, "Type inattendu : {}\n", e.what());
    // [json.exception.type_error.302] type must be array, but is string
}
```

**Causes courantes :** un champ numérique transmis comme chaîne par une API, un tableau attendu mais un objet reçu, `null` là où une valeur concrète est attendue, confusion entre entier et flottant.

### `json::out_of_range` — La clé ou l'index n'existe pas

Levée par `.at()` lorsqu'une clé est absente d'un objet ou qu'un index dépasse la taille d'un tableau. Non levée par `operator[]`, qui crée silencieusement la clé (cf. section 24.1.2).

```cpp
json j = json::parse(R"({"host": "localhost"})");

try {
    std::string db = j.at("database").get<std::string>();
} catch (const json::out_of_range& e) {
    std::print(stderr, "Champ manquant : {}\n", e.what());
    // [json.exception.out_of_range.403] key 'database' not found
}
```

```cpp
json arr = json::parse(R"([10, 20, 30])");

try {
    int val = arr.at(5);
} catch (const json::out_of_range& e) {
    std::print(stderr, "Index invalide : {}\n", e.what());
    // [json.exception.out_of_range.401] array index 5 is out of range
}
```

**Causes courantes :** champ obligatoire absent du fichier de configuration, faute de frappe dans un nom de clé, version d'API différente qui n'inclut pas certains champs, tableau plus court que prévu.

## Stratégies de gestion des erreurs

### Stratégie 1 : Capture granulaire par type d'exception

La méthode la plus expressive consiste à capturer chaque type d'exception séparément, pour produire des messages d'erreur adaptés au contexte :

```cpp
std::optional<ServerConfig> load_config(const std::string& path) {
    std::ifstream file(path);
    if (!file.is_open()) {
        std::print(stderr, "Fichier introuvable : {}\n", path);
        return std::nullopt;
    }

    try {
        json j = json::parse(file);
        return j.get<ServerConfig>();

    } catch (const json::parse_error& e) {
        std::print(stderr,
            "Erreur de syntaxe dans {} (octet {}) :\n  {}\n",
            path, e.byte, e.what());

    } catch (const json::out_of_range& e) {
        std::print(stderr,
            "Champ obligatoire manquant dans {} :\n  {}\n",
            path, e.what());

    } catch (const json::type_error& e) {
        std::print(stderr,
            "Type de valeur incorrect dans {} :\n  {}\n",
            path, e.what());
    }

    return std::nullopt;
}
```

Cette approche est recommandée pour les points d'entrée du programme (chargement de configuration, réception de messages) où un diagnostic précis aide l'utilisateur ou l'opérateur à corriger le problème.

### Stratégie 2 : Capture globale avec `json::exception`

Quand le programme ne peut pas faire grand-chose de plus que logger l'erreur et abandonner l'opération, capturer la classe de base suffit :

```cpp
try {
    auto config = load_and_validate(path);
    run_server(config);
} catch (const json::exception& e) {
    std::print(stderr, "Erreur JSON [{}] : {}\n", e.id, e.what());
    return 1;
}
```

Le champ `e.id` permet tout de même de catégoriser l'erreur dans les logs (101 = syntaxe, 302 = type, 403 = clé manquante, etc.) sans multiplier les blocs `catch`.

### Stratégie 3 : Parsing sans exception

Pour les contextes où les exceptions ne sont pas souhaitables — code temps réel, boucle de traitement à haut débit, politique de projet interdisant les exceptions — le parsing peut fonctionner entièrement sans exception :

```cpp
bool process_message(const std::string& raw_message) {
    // Parsing sans exception
    json j = json::parse(raw_message, nullptr, false);
    if (j.is_discarded()) {
        log_warning("Message JSON invalide, ignoré");
        return false;
    }

    // Navigation sans exception avec .contains() et .value()
    if (!j.contains("event_type") || !j["event_type"].is_string()) {
        log_warning("Champ 'event_type' manquant ou invalide");
        return false;
    }

    std::string event_type = j["event_type"].get<std::string>();
    int priority = j.value("priority", 0);
    std::string source = j.value("source", "unknown");

    dispatch_event(event_type, priority, source);
    return true;
}
```

Ce style est plus verbeux mais offre un contrôle total sur le flux d'exécution. Chaque accès est protégé par un test explicite, et aucun chemin de code ne dépend d'un `catch`.

### Stratégie 4 : Retour avec `std::expected` (C++23)

Pour les projets en C++23, `std::expected` (section 12.8) permet de combiner la clarté d'un type de retour explicite avec un diagnostic structuré :

```cpp
#include <expected>

enum class ConfigError {
    file_not_found,
    syntax_error,
    missing_field,
    type_mismatch
};

struct ConfigErrorInfo {
    ConfigError code;
    std::string detail;
};

std::expected<ServerConfig, ConfigErrorInfo>  
load_config(const std::string& path) {  
    std::ifstream file(path);
    if (!file.is_open()) {
        return std::unexpected(ConfigErrorInfo{
            ConfigError::file_not_found,
            std::format("Fichier introuvable : {}", path)
        });
    }

    try {
        json j = json::parse(file);
        return j.get<ServerConfig>();

    } catch (const json::parse_error& e) {
        return std::unexpected(ConfigErrorInfo{
            ConfigError::syntax_error,
            std::format("Syntaxe invalide (octet {}) : {}", e.byte, e.what())
        });

    } catch (const json::out_of_range& e) {
        return std::unexpected(ConfigErrorInfo{
            ConfigError::missing_field, e.what()
        });

    } catch (const json::type_error& e) {
        return std::unexpected(ConfigErrorInfo{
            ConfigError::type_mismatch, e.what()
        });
    }
}
```

L'appelant traite le résultat sans `try`/`catch` :

```cpp
auto result = load_config("server.json");

if (!result) {
    const auto& err = result.error();
    std::print(stderr, "[{}] {}\n",
        static_cast<int>(err.code), err.detail);
    return 1;
}

const ServerConfig& config = *result;
// utilisation normale...
```

Ce pattern est particulièrement efficace quand la fonction de chargement est appelée depuis du code qui ne souhaite pas propager d'exceptions.

## Messages d'erreur exploitables

La qualité des messages d'erreur détermine directement la vitesse à laquelle un problème de configuration est résolu. nlohmann/json fournit des messages détaillés, mais ils sont formulés en termes techniques (noms de types JSON, positions d'octets). Dans un outil destiné à des utilisateurs non-développeurs, il est utile de les enrichir avec du contexte applicatif.

### Enrichissement avec le contexte fichier

```cpp
void report_config_error(const std::string& filepath,
                         const json::parse_error& e) {
    std::print(stderr, "\n╔══ Erreur de configuration ══════════════════\n");
    std::print(stderr, "║ Fichier : {}\n", filepath);
    std::print(stderr, "║ Position: octet {}\n", e.byte);
    std::print(stderr, "║\n");
    std::print(stderr, "║ Le fichier contient une erreur de syntaxe JSON.\n");
    std::print(stderr, "║ Vérifiez les virgules, guillemets et accolades\n");
    std::print(stderr, "║ autour de la position indiquée.\n");
    std::print(stderr, "║\n");
    std::print(stderr, "║ Détail technique : {}\n", e.what());
    std::print(stderr, "╚═════════════════════════════════════════════\n\n");
}
```

### Localisation de l'erreur dans le fichier

Le champ `byte` de `parse_error` est un offset depuis le début du flux. Pour le convertir en numéro de ligne et colonne — bien plus utile pour l'utilisateur — il faut parcourir le contenu du fichier :

```cpp
struct FilePosition {
    std::size_t line;
    std::size_t column;
};

FilePosition byte_to_position(const std::string& content, std::size_t byte_offset) {
    std::size_t line = 1;
    std::size_t col = 1;

    for (std::size_t i = 0; i < byte_offset && i < content.size(); ++i) {
        if (content[i] == '\n') {
            ++line;
            col = 1;
        } else {
            ++col;
        }
    }

    return {line, col};
}

void report_with_location(const std::string& filepath,
                          const std::string& content,
                          const json::parse_error& e) {
    auto [line, col] = byte_to_position(content, e.byte);
    std::print(stderr, "{}:{}:{}: erreur de syntaxe JSON\n",
               filepath, line, col);
}
```

Ce format `fichier:ligne:colonne:` est le standard des compilateurs et des linters. Les éditeurs de code et les terminaux modernes le reconnaissent et permettent de naviguer directement à la position de l'erreur.

### Affichage de la ligne fautive avec marqueur

Pour les outils CLI, afficher la ligne incriminée avec un marqueur visuel accélère considérablement le diagnostic :

```cpp
void show_error_context(const std::string& content,
                        const std::string& filepath,
                        const json::parse_error& e) {
    auto [line, col] = byte_to_position(content, e.byte);

    // Extraire la ligne fautive
    std::istringstream stream(content);
    std::string current_line;
    for (std::size_t i = 0; i < line; ++i) {
        std::getline(stream, current_line);
    }

    std::print(stderr, "{}:{}:{}: erreur de syntaxe JSON\n",
               filepath, line, col);
    std::print(stderr, "  {} | {}\n", line, current_line);

    // Marqueur sous la position de l'erreur
    std::string marker(col - 1, ' ');
    std::print(stderr, "  {} | {}^\n",
               std::string(std::to_string(line).size(), ' '), marker);
}
```

Sortie pour un fichier `config.json` avec une virgule manquante :

```
config.json:4:25: erreur de syntaxe JSON
  4 |     "port": 8080
    |                         ^
```

## Validation structurelle post-parsing

Le parsing syntaxique (le JSON est-il bien formé ?) ne garantit pas la validité sémantique (le JSON contient-il les données attendues avec les bons types ?). Une fois le document parsé avec succès, il reste à vérifier que sa structure correspond à ce que le programme attend.

### Validation manuelle des champs requis

```cpp
struct ValidationResult {
    bool valid = true;
    std::vector<std::string> errors;

    void add_error(std::string msg) {
        valid = false;
        errors.push_back(std::move(msg));
    }
};

ValidationResult validate_server_config(const json& j) {
    ValidationResult result;

    // Vérifier que c'est un objet
    if (!j.is_object()) {
        result.add_error("Le document racine doit être un objet JSON");
        return result;  // pas de point à continuer
    }

    // Champs obligatoires avec vérification de type
    if (!j.contains("host")) {
        result.add_error("Champ obligatoire 'host' manquant");
    } else if (!j["host"].is_string()) {
        result.add_error("'host' doit être une chaîne de caractères");
    }

    if (!j.contains("port")) {
        result.add_error("Champ obligatoire 'port' manquant");
    } else if (!j["port"].is_number_integer()) {
        result.add_error("'port' doit être un entier");
    } else {
        int port = j["port"];
        if (port < 1 || port > 65535) {
            result.add_error(
                std::format("'port' doit être entre 1 et 65535 (reçu: {})", port));
        }
    }

    // Champ optionnel avec contrainte de type si présent
    if (j.contains("workers") && !j["workers"].is_number_unsigned()) {
        result.add_error("'workers' doit être un entier positif");
    }

    return result;
}
```

Utilisation :

```cpp
json j = json::parse(file);  
auto validation = validate_server_config(j);  

if (!validation.valid) {
    std::print(stderr, "Configuration invalide ({} erreur(s)) :\n",
               validation.errors.size());
    for (const auto& err : validation.errors) {
        std::print(stderr, "  - {}\n", err);
    }
    return std::nullopt;
}

// La structure est validée, la désérialisation ne devrait pas échouer
return j.get<ServerConfig>();
```

L'intérêt de la validation explicite est de **collecter toutes les erreurs en une seule passe** plutôt que de s'arrêter à la première. Un opérateur qui reçoit la liste complète des problèmes peut tous les corriger d'un coup, au lieu d'itérer erreur par erreur.

### Validation avec JSON Schema

Pour les projets à grande échelle ou les formats partagés entre équipes, la validation manuelle devient difficile à maintenir. La norme **JSON Schema** (RFC draft, largement adoptée) permet de décrire la structure attendue d'un document JSON dans un format déclaratif, lui-même en JSON :

```json
{
    "$schema": "https://json-schema.org/draft/2020-12/schema",
    "type": "object",
    "required": ["host", "port"],
    "properties": {
        "host": {
            "type": "string",
            "minLength": 1
        },
        "port": {
            "type": "integer",
            "minimum": 1,
            "maximum": 65535
        },
        "workers": {
            "type": "integer",
            "minimum": 1,
            "default": 4
        }
    },
    "additionalProperties": false
}
```

nlohmann/json ne fournit pas de validateur JSON Schema intégré, mais plusieurs librairies C++ tierces s'en chargent, notamment **json-schema-validator** (basée sur nlohmann/json) et **valijson**. L'intégration avec un schéma formel fait l'objet de la section 24.5 (bonnes pratiques de validation).

## Pattern complet : chargement robuste d'une configuration

En combinant les techniques de cette section, voici le pattern recommandé pour le chargement d'un fichier de configuration dans un programme en production :

```cpp
#include <nlohmann/json.hpp>
#include <filesystem>
#include <fstream>
#include <optional>
#include <print>

using json = nlohmann::json;  
namespace fs = std::filesystem;  

std::optional<ServerConfig> load_server_config(const fs::path& config_path) {
    // 1. Vérification de l'existence du fichier
    if (!fs::exists(config_path)) {
        std::print(stderr, "Configuration introuvable : {}\n",
                   config_path.string());
        return std::nullopt;
    }

    // 2. Lecture du contenu brut (pour les diagnostics d'erreur)
    std::string content;
    {
        std::ifstream file(config_path);
        if (!file.is_open()) {
            std::print(stderr, "Impossible de lire : {}\n",
                       config_path.string());
            return std::nullopt;
        }
        content.assign(std::istreambuf_iterator<char>(file),
                       std::istreambuf_iterator<char>());
    }

    // 3. Parsing syntaxique
    json j;
    try {
        j = json::parse(content);
    } catch (const json::parse_error& e) {
        auto [line, col] = byte_to_position(content, e.byte);
        std::print(stderr, "{}:{}:{}: erreur de syntaxe JSON\n",
                   config_path.string(), line, col);
        std::print(stderr, "  {}\n", e.what());
        return std::nullopt;
    }

    // 4. Validation structurelle
    auto validation = validate_server_config(j);
    if (!validation.valid) {
        std::print(stderr, "Configuration invalide ({}) :\n",
                   config_path.string());
        for (const auto& err : validation.errors) {
            std::print(stderr, "  - {}\n", err);
        }
        return std::nullopt;
    }

    // 5. Désérialisation vers le type métier
    try {
        return j.get<ServerConfig>();
    } catch (const json::exception& e) {
        std::print(stderr, "Erreur de conversion inattendue : {}\n",
                   e.what());
        return std::nullopt;
    }
}
```

Les cinq étapes forment un pipeline de validation incrémentale. Chaque étape ne s'exécute que si la précédente a réussi, et chaque erreur produit un diagnostic adapté à sa nature :

1. **Existence du fichier** — erreur système, message simple.
2. **Lecture du contenu** — erreur de permissions ou d'I/O.
3. **Parsing syntaxique** — erreur de format, avec position ligne:colonne.
4. **Validation structurelle** — erreurs sémantiques collectées en une passe.
5. **Désérialisation** — filet de sécurité pour les cas non couverts par la validation.

L'étape 5 est un filet de sécurité : si la validation de l'étape 4 est correctement implémentée, la désérialisation ne devrait jamais échouer. Mais le `try`/`catch` reste présent par principe de défense en profondeur, car un oubli dans la validation ne doit pas provoquer un crash non géré.

## Récapitulatif des erreurs et réponses

| Erreur | Exception | ID | Réponse recommandée |
|--------|-----------|-----|---------------------|
| Syntaxe JSON invalide | `parse_error` | 101 | Afficher position (ligne:colonne) et extrait du fichier |
| Type JSON incompatible | `type_error` | 302 | Indiquer le champ, le type attendu et le type reçu |
| Clé absente (`.at()`) | `out_of_range` | 403 | Lister les champs obligatoires manquants |
| Index hors limites | `out_of_range` | 401 | Indiquer la taille du tableau et l'index demandé |
| Itérateur invalide | `invalid_iterator` | 2xx | Généralement un bug — corriger le code |
| Toute erreur JSON | `json::exception` | * | Capture générique pour le logging |

La section 24.5 étendra ces concepts avec les stratégies de validation de schéma formelles, applicables à tous les formats textuels couverts dans ce chapitre.
