# 24.1.1 — Installation et intégration

## Sous-section de 24.1 — JSON : Lecture/Écriture avec nlohmann/json

---

## Vue d'ensemble

nlohmann/json est une librairie **header-only** : elle ne nécessite aucune compilation préalable ni aucun fichier `.a` ou `.so` à linker. Son intégration se résume à rendre le header accessible au compilateur, puis à l'inclure dans le code source. Cette simplicité est un avantage considérable, mais il existe plusieurs façons de procéder, chacune adaptée à un contexte de projet différent.

Cette section présente les quatre méthodes d'installation courantes, de la plus recommandée à la plus rudimentaire, puis détaille la configuration CMake associée et les stratégies pour réduire l'impact sur le temps de compilation.

---

## Méthode 1 : Conan 2 (recommandée pour les projets professionnels)

Conan est le gestionnaire de paquets C++ couvert en détail à la section 27.2. C'est la méthode recommandée pour tout projet sérieux car elle garantit la reproductibilité des builds, gère les versions de manière explicite, et s'intègre naturellement dans une chaîne CI/CD.

### Configuration dans `conanfile.py`

```python
from conan import ConanFile  
from conan.tools.cmake import cmake_layout  

class MyProjectConan(ConanFile):
    name = "myproject"
    version = "1.0.0"
    settings = "os", "compiler", "build_type", "arch"
    generators = "CMakeDeps", "CMakeToolchain"

    def requirements(self):
        self.requires("nlohmann_json/3.12.0")

    def layout(self):
        cmake_layout(self)
```

Pour un projet simple, un `conanfile.txt` suffit :

```ini
[requires]
nlohmann_json/3.12.0

[generators]
CMakeDeps  
CMakeToolchain  
```

### Installation des dépendances

```bash
# Depuis la racine du projet
conan install . --output-folder=build --build=missing
```

### Intégration CMake

Avec les générateurs `CMakeDeps` et `CMakeToolchain`, Conan produit les fichiers de configuration nécessaires. Le `CMakeLists.txt` utilise ensuite `find_package` de manière standard :

```cmake
cmake_minimum_required(VERSION 3.20)  
project(myproject LANGUAGES CXX)  

set(CMAKE_CXX_STANDARD 23)  
set(CMAKE_CXX_STANDARD_REQUIRED ON)  

# Conan a généré le fichier de config pour nlohmann_json
find_package(nlohmann_json REQUIRED)

add_executable(myapp src/main.cpp)  
target_link_libraries(myapp PRIVATE nlohmann_json::nlohmann_json)  
```

La commande de build utilise le preset généré par Conan :

```bash
cd build  
cmake .. -G Ninja -DCMAKE_TOOLCHAIN_FILE=conan_toolchain.cmake  
ninja  
```

> 💡 *Avec CMake Presets (section 27.6), la ligne de commande se simplifie encore : `cmake --preset conan-release && cmake --build --preset conan-release`.*

## Méthode 2 : vcpkg

vcpkg est le gestionnaire de paquets C++ de Microsoft, couvert à la section 27.3. Il fonctionne selon un modèle différent de Conan : les paquets sont installés dans un répertoire central et intégrés via une toolchain CMake.

### Installation du paquet

```bash
# Si vcpkg est installé dans ~/vcpkg
vcpkg install nlohmann-json
```

### Intégration CMake

```cmake
cmake_minimum_required(VERSION 3.20)  
project(myproject LANGUAGES CXX)  

set(CMAKE_CXX_STANDARD 23)  
set(CMAKE_CXX_STANDARD_REQUIRED ON)  

find_package(nlohmann_json CONFIG REQUIRED)

add_executable(myapp src/main.cpp)  
target_link_libraries(myapp PRIVATE nlohmann_json::nlohmann_json)  
```

Le `CMakeLists.txt` est identique à la version Conan. La différence se situe à l'invocation de CMake, où l'on passe la toolchain vcpkg :

```bash
cmake -B build -G Ninja -DCMAKE_TOOLCHAIN_FILE=~/vcpkg/scripts/buildsystems/vcpkg.cmake  
cmake --build build  
```

Pour les projets utilisant un fichier `vcpkg.json` (mode manifest, recommandé), la dépendance est déclarée explicitement :

```json
{
    "name": "myproject",
    "version-string": "1.0.0",
    "dependencies": [
        "nlohmann-json"
    ]
}
```

Avec le mode manifest, vcpkg installe automatiquement les dépendances lors de la configuration CMake.

## Méthode 3 : CMake FetchContent

`FetchContent` permet de télécharger et intégrer une dépendance directement depuis son dépôt Git au moment de la configuration CMake, sans gestionnaire de paquets externe. C'est un bon compromis pour les petits projets ou les prototypes où installer Conan ou vcpkg serait disproportionné.

```cmake
cmake_minimum_required(VERSION 3.20)  
project(myproject LANGUAGES CXX)  

set(CMAKE_CXX_STANDARD 23)  
set(CMAKE_CXX_STANDARD_REQUIRED ON)  

include(FetchContent)

FetchContent_Declare(
    json
    GIT_REPOSITORY https://github.com/nlohmann/json.git
    GIT_TAG        v3.12.0
    GIT_SHALLOW    TRUE       # ne clone pas tout l'historique
)

# Désactive les tests et benchmarks de la librairie
set(JSON_BuildTests OFF CACHE BOOL "" FORCE)

FetchContent_MakeAvailable(json)

add_executable(myapp src/main.cpp)  
target_link_libraries(myapp PRIVATE nlohmann_json::nlohmann_json)  
```

Le point important est `GIT_SHALLOW TRUE` qui évite de cloner l'intégralité de l'historique Git du dépôt (plusieurs centaines de Mo avec les données de test). De même, `JSON_BuildTests OFF` empêche la compilation des tests unitaires de la librairie elle-même, ce qui économise un temps de build considérable.

**Limitations de FetchContent :**

- Le téléchargement a lieu à chaque première configuration dans un répertoire de build vierge, ce qui peut poser problème en CI si le réseau est lent ou restreint.
- Aucun mécanisme de cache global entre projets — chaque projet télécharge sa propre copie.
- Pour un projet avec de nombreuses dépendances, la gestion devient vite plus complexe qu'avec Conan ou vcpkg.

## Méthode 4 : Header-only direct (single include)

La méthode la plus simple, adaptée au prototypage rapide ou à l'évaluation de la librairie. On télécharge le header unique et on le place dans le projet.

### Téléchargement

```bash
# Créer le répertoire d'inclusion
mkdir -p third_party/nlohmann

# Télécharger le single header
curl -L -o third_party/nlohmann/json.hpp \
    https://github.com/nlohmann/json/releases/download/v3.12.0/json.hpp
```

### Intégration CMake

```cmake
cmake_minimum_required(VERSION 3.20)  
project(myproject LANGUAGES CXX)  

set(CMAKE_CXX_STANDARD 23)  
set(CMAKE_CXX_STANDARD_REQUIRED ON)  

add_executable(myapp src/main.cpp)

# Rendre le répertoire third_party visible comme chemin d'inclusion
target_include_directories(myapp PRIVATE ${CMAKE_SOURCE_DIR}/third_party)
```

L'inclusion dans le code source fonctionne alors normalement :

```cpp
#include <nlohmann/json.hpp>
using json = nlohmann::json;
```

**Quand utiliser cette méthode :**

- Prototypage rapide ou exploration de la librairie.
- Environnements contraints où aucun gestionnaire de paquets n'est disponible.
- Projets ponctuels à durée de vie limitée.

**Quand l'éviter :**

- Tout projet destiné à durer. La mise à jour de la librairie devient manuelle, le fichier n'est pas versionné par un gestionnaire de dépendances, et le risque d'avoir des versions incohérentes entre développeurs est réel.

## Installation système (apt)

Sur Ubuntu, nlohmann/json est disponible dans les dépôts officiels :

```bash
sudo apt install nlohmann-json3-dev
```

Le paquet installe les headers dans `/usr/include/nlohmann/` et fournit un fichier de configuration CMake. L'intégration se fait avec `find_package` exactement comme avec Conan ou vcpkg :

```cmake
find_package(nlohmann_json REQUIRED)  
target_link_libraries(myapp PRIVATE nlohmann_json::nlohmann_json)  
```

Cette méthode est pratique pour démarrer rapidement, mais la version disponible dans les dépôts Ubuntu peut être en retard par rapport à la dernière release. Sur Ubuntu 24.04 LTS, la version fournie est la 3.11.3 — suffisante pour la plupart des usages, mais il faut le vérifier si une fonctionnalité récente est requise (la version 3.12.0 apporte notamment le support natif de `std::optional` et des macros pour les classes dérivées). Pour les projets où le contrôle précis de la version compte, Conan ou vcpkg restent préférables.

## Réduction du temps de compilation

Le header `json.hpp` en version single-include pèse environ 24 000 lignes de code C++. Chaque unité de traduction (fichier `.cpp`) qui l'inclut doit le parser intégralement, ce qui peut ajouter plusieurs secondes à la compilation unitaire. Sur un projet avec des dizaines de fichiers source incluant le header, l'impact cumulé est significatif.

Plusieurs stratégies permettent de maîtriser ce coût.

### Stratégie 1 : Limiter la propagation de l'include

La première mesure, et la plus importante, est de ne pas inclure `json.hpp` dans les fichiers headers du projet. Concrètement, si une classe manipule du JSON dans son implémentation mais n'expose pas le type `json` dans son interface publique, l'include doit se trouver dans le `.cpp` uniquement :

```cpp
// config_reader.h — PAS de #include <nlohmann/json.hpp> ici
#pragma once
#include <string>
#include <optional>

struct AppConfig {
    std::string host;
    int port;
    bool debug;
};

class ConfigReader {  
public:  
    explicit ConfigReader(const std::string& path);
    std::optional<AppConfig> load() const;

private:
    std::string filepath_;
};
```

```cpp
// config_reader.cpp — l'include est ici, invisible du reste du projet
#include "config_reader.h"
#include <nlohmann/json.hpp>
#include <fstream>

using json = nlohmann::json;

ConfigReader::ConfigReader(const std::string& path)
    : filepath_(path) {}

std::optional<AppConfig> ConfigReader::load() const {
    std::ifstream file(filepath_);
    if (!file.is_open()) return std::nullopt;

    try {
        auto j = json::parse(file);
        return AppConfig{
            .host  = j.value("host", "localhost"),
            .port  = j.value("port", 8080),
            .debug = j.value("debug", false)
        };
    } catch (const json::parse_error&) {
        return std::nullopt;
    }
}
```

Ce pattern — isoler la dépendance JSON dans les fichiers d'implémentation — est la mesure la plus rentable. Elle réduit le nombre de fichiers recompilés lors d'un changement de version de la librairie et maintient les temps de compilation incrémentale courts.

### Stratégie 2 : Forward declaration

La librairie fournit un header de forward declaration qui permet de déclarer le type `json` dans un header sans inclure la définition complète :

```cpp
// mon_header.h
#include <nlohmann/json_fwd.hpp>  // ~50 lignes au lieu de ~24 000

void process_config(const nlohmann::json& config);
```

L'inclusion complète de `json.hpp` se fait uniquement dans le `.cpp` correspondant. Cette technique est particulièrement utile quand le type `json` apparaît dans une signature de fonction ou comme membre de classe via un pointeur ou une référence.

### Stratégie 3 : ccache

Comme détaillé à la section 2.3, ccache met en cache les résultats de compilation. Les unités de traduction qui incluent `json.hpp` sans modification bénéficient directement du cache, ramenant la recompilation à un lookup quasi-instantané. C'est une mesure systémique qui profite à toutes les dépendances header-only, pas seulement nlohmann/json.

### Stratégie 4 : Precompiled headers (PCH)

CMake 3.16+ supporte nativement les precompiled headers. Si `json.hpp` est inclus dans de nombreux fichiers du projet, le placer dans un PCH permet de ne le parser qu'une seule fois :

```cmake
target_precompile_headers(myapp PRIVATE
    <nlohmann/json.hpp>
    <string>
    <vector>
)
```

Le gain est notable sur les gros projets. En contrepartie, le PCH lui-même prend du temps à générer et peut masquer des dépendances d'inclusion implicites. C'est un outil à utiliser de manière ciblée.

## Vérification de l'installation

Quel que soit le mode d'installation choisi, un programme minimal permet de vérifier que tout fonctionne correctement :

```cpp
// verify_json.cpp
#include <nlohmann/json.hpp>
#include <print>

using json = nlohmann::json;

int main() {
    // Création
    json j = {
        {"library", "nlohmann/json"},
        {"version", 3},
        {"header_only", true}
    };

    // Sérialisation
    std::print("JSON généré :\n{}\n", j.dump(2));

    // Parsing
    auto parsed = json::parse(R"({"status": "ok", "code": 200})");
    std::print("Status : {}\n", parsed["status"].get<std::string>());
    std::print("Code   : {}\n", parsed["code"].get<int>());

    return 0;
}
```

Compilation et exécution :

```bash
# Avec CMake (méthode recommandée)
cmake -B build -G Ninja  
cmake --build build  
./build/verify_json

# Ou directement avec g++ pour un test rapide
g++ -std=c++23 -O2 -o verify_json verify_json.cpp \
    -I/chemin/vers/nlohmann
./verify_json
```

Sortie attendue :

```
JSON généré :
{
  "header_only": true,
  "library": "nlohmann/json",
  "version": 3
}
Status : ok  
Code   : 200  
```

> ⚠️ *Notez que les clés dans l'objet JSON sont triées par ordre alphabétique dans la sortie. C'est le comportement par défaut de nlohmann/json, qui utilise `std::map<std::string, json>` en interne pour les objets. Ce choix garantit un dump déterministe mais ne préserve pas l'ordre d'insertion. La librairie propose un type alternatif `nlohmann::ordered_json` qui utilise une map ordonnée par insertion si ce comportement est nécessaire.*

## Récapitulatif des méthodes

| Méthode | Gestion des versions | Reproductibilité | Mise en place | Recommandé pour |
|---------|---------------------|-------------------|---------------|-----------------|
| **Conan 2** | Explicite, lockfile | Excellente | Moyenne | Projets professionnels, CI/CD |
| **vcpkg** | Manifest ou globale | Très bonne | Moyenne | Écosystème Microsoft, projets multi-plateforme |
| **FetchContent** | Tag Git | Bonne | Simple | Petits projets, prototypes |
| **Header direct** | Manuelle | Faible | Triviale | Prototypage, évaluation |
| **apt** | Version distro | Limitée | Triviale | Démarrage rapide sur Ubuntu |

Pour un projet destiné à être maintenu dans la durée, avec une chaîne CI/CD et plusieurs développeurs, **Conan 2 ou vcpkg** sont les choix appropriés. Pour un prototype ou un script one-shot, le **header direct** ou **FetchContent** font parfaitement l'affaire.
