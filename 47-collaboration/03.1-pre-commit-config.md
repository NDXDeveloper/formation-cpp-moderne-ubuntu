# 47.3.1 — `.pre-commit-config.yaml`

> **Chapitre 47 : Collaboration et Maintenance** · Section 47.3 : Configuration pre-commit

## Introduction

Cette sous-section assemble la configuration `.pre-commit-config.yaml` complète pour un projet C++ professionnel. Plutôt que de présenter un fichier monolithique sans contexte, nous construisons la configuration bloc par bloc, en expliquant chaque décision. Le fichier complet est fourni à la fin, prêt à être copié et adapté.

La configuration présentée ici correspond au profil **"projet d'équipe professionnel"** identifié en section 47.2.2 — le profil le plus courant. Les adaptations pour les profils plus légers (projet personnel) ou plus stricts (projet critique 10+ développeurs) sont indiquées en fin de section.

## En-tête et options globales

```yaml
# .pre-commit-config.yaml
# ═══════════════════════════════════════════════════════════════
# Pre-commit hooks — Projet C++ moderne
# Documentation : https://pre-commit.com
# Usage :
#   pre-commit install && pre-commit install --hook-type commit-msg
#   pre-commit run --all-files    (vérification complète)
#   pre-commit autoupdate         (mise à jour des hooks)
# ═══════════════════════════════════════════════════════════════

# Version minimale requise du framework
minimum_pre_commit_version: '3.5.0'

# Langue par défaut pour les hooks qui utilisent un runtime
default_language_version:
  python: python3

# Exclure globalement les répertoires hors du contrôle du projet
exclude: |
  (?x)^(
    build/|
    cmake-build-.*/|
    third_party/|
    vendor/|
    \.cache/|
    .*\.pb\.(h|cc)$|
    .*\.generated\.\w+$
  )
```

### Détail des options globales

**`minimum_pre_commit_version`** — Garantit que tous les développeurs utilisent une version du framework suffisamment récente. Si un développeur a une version antérieure à `3.5.0`, `pre-commit` affiche un message d'erreur explicite plutôt que d'échouer de manière cryptique sur une syntaxe non supportée.

**`default_language_version`** — Fixe la version Python utilisée par les hooks écrits en Python (codespell, commitizen, cmake-format). `python3` utilise le Python 3 par défaut du système. Sur Ubuntu 24.04+, c'est Python 3.12.

**`exclude` global** — L'expression régulière exclut les fichiers et répertoires qui ne doivent jamais être analysés par les hooks :

| Pattern | Raison de l'exclusion |
|---|---|
| `build/` | Artefacts de compilation |
| `cmake-build-.*/` | Répertoires de build CLion |
| `third_party/` | Code externe non maintenu par l'équipe |
| `vendor/` | Dépendances vendored |
| `\.cache/` | Cache clangd, ccache local |
| `.*\.pb\.(h\|cc)$` | Code généré par Protobuf |
| `.*\.generated\.\w+$` | Tout fichier généré par convention |

La syntaxe `(?x)` active le mode *verbose* des regex Python, ce qui permet d'écrire l'expression sur plusieurs lignes avec des commentaires. C'est beaucoup plus lisible qu'une regex monoligne pour les patterns complexes.

> **Point important** : le `exclude` global s'applique à **tous** les hooks. Si un hook spécifique a besoin d'un pattern d'exclusion différent, il peut définir son propre `exclude` qui s'ajoute au global (il ne le remplace pas — les deux sont évalués).

## Bloc 1 : Hooks d'hygiène générale

```yaml
repos:
  # ── Bloc 1 : Hygiène générale ─────────────────────────────
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.6.0
    hooks:
      # Supprime les espaces en fin de ligne
      - id: trailing-whitespace

      # Garantit un newline en fin de fichier
      - id: end-of-file-fixer

      # Bloque les fichiers > 500 Ko (évite les binaires accidentels)
      - id: check-added-large-files
        args: ['--maxkb=500']

      # Détecte les marqueurs de conflit Git oubliés
      - id: check-merge-conflict

      # Valide la syntaxe YAML (.clang-format, CI, Docker Compose...)
      - id: check-yaml
        args: ['--allow-multiple-documents']

      # Valide la syntaxe JSON (CMakePresets.json, configs...)
      - id: check-json

      # Vérifie que les symlinks pointent vers des cibles existantes
      - id: check-symlinks

      # Normalise les fins de ligne vers LF (Unix)
      - id: mixed-line-ending
        args: ['--fix=lf']

      # Bloque les clés privées/certificats accidentellement stagés
      - id: detect-private-key

      # Vérifie que les fichiers exécutables ont un shebang
      - id: check-executables-have-shebangs
```

### Notes sur ce bloc

**`check-yaml` avec `--allow-multiple-documents`** — Certains fichiers YAML légitimes contiennent plusieurs documents séparés par `---` (fichiers Kubernetes, certaines configurations CI). Sans cet argument, le hook les rejette. Si votre projet n'utilise pas de YAML multi-documents, vous pouvez retirer cet argument pour une validation plus stricte.

**Ordre d'exécution** — Les hooks s'exécutent dans l'ordre de déclaration. Placer les hooks d'hygiène en premier est une convention : s'ils modifient les fichiers (trailing-whitespace, end-of-file-fixer, mixed-line-ending), les hooks suivants (clang-format) opèrent sur des fichiers déjà nettoyés. Cela évite que `clang-format` ne reformate un fichier puis que `trailing-whitespace` ne le modifie à nouveau, forçant deux restages.

**Temps total de ce bloc** : < 500 ms, quel que soit le nombre de fichiers. Ces hooks opèrent sur le texte brut sans parsing sémantique.

## Bloc 2 : Formatage et analyse C++

```yaml
  # ── Bloc 2 : C++ — Formatage et analyse ───────────────────
  - repo: https://github.com/pocc/pre-commit-hooks
    rev: v1.3.5
    hooks:
      # Formatage automatique du code C++ (utilise .clang-format)
      - id: clang-format
        args: ['-i', '--style=file']
        files: '\.(cpp|hpp|h|cc|cxx|c|tpp)$'

      # Analyse statique C++ (utilise .clang-tidy)
      # Limité aux fichiers sources (pas les headers isolés)
      - id: clang-tidy
        args:
          - '--config-file=.clang-tidy'
          - '-p=build'
          - '--warnings-as-errors=*'
        files: '\.(cpp|cc|cxx)$'
        stages: [pre-commit]
```

### Détail de la configuration `clang-format`

**`args: ['-i', '--style=file']`** — L'argument `-i` reformate les fichiers en place (*in-place*). L'argument `--style=file` indique à `clang-format` de lire les règles depuis le fichier `.clang-format` le plus proche dans l'arborescence parent. Si aucun `.clang-format` n'est trouvé, `clang-format` utilise son style par défaut (LLVM) — ce qui n'est généralement pas souhaitable. Vérifiez toujours qu'un `.clang-format` existe à la racine du projet.

**`files: '\.(cpp|hpp|h|cc|cxx|c|tpp)$'`** — Le pattern cible les extensions C++ courantes. L'extension `.tpp` est incluse pour les fichiers d'implémentation de templates (convention utilisée dans certains projets pour séparer la déclaration template dans le `.hpp` de son implémentation dans un `.tpp` inclus par le header). Adaptez cette liste aux conventions de votre projet.

**Ce hook modifie les fichiers.** Si un fichier stagé n'est pas correctement formaté, `clang-format` le corrige et le hook échoue. Le développeur voit le diff, re-stage et re-commit. Ce cycle prend quelques secondes.

### Détail de la configuration `clang-tidy`

**`args: ['--config-file=.clang-tidy', '-p=build', '--warnings-as-errors=*']`** — Trois arguments critiques :

- `--config-file=.clang-tidy` : pointe explicitement vers le fichier de configuration plutôt que de laisser `clang-tidy` le chercher. Cela évite les surprises si un `.clang-tidy` existe dans un sous-répertoire.  
- `-p=build` : indique le répertoire contenant `compile_commands.json`. C'est le répertoire standard de build CMake. Si votre projet utilise un autre chemin (par exemple `cmake-build-debug/`), ajustez en conséquence. La section 47.3.3 détaille les stratégies pour gérer ce chemin.  
- `--warnings-as-errors=*` : transforme toutes les violations en erreurs, ce qui force le hook à échouer. Sans cet argument, `clang-tidy` affiche les warnings mais laisse le commit passer.

**`files: '\.(cpp|cc|cxx)$'`** — Volontairement restreint aux fichiers sources, pas aux headers. Comme expliqué en section 47.3, analyser un `.h` isolé avec `clang-tidy` produit souvent des faux positifs car le header n'est pas une unité de traduction autonome. Les headers sont analysés indirectement via les fichiers sources qui les incluent. L'analyse complète des headers est réservée à la CI.

**`stages: [pre-commit]`** — Explicitement déclaré pour la lisibilité. C'est la valeur par défaut, mais la déclarer rend l'intention claire dans le fichier de configuration.

### Le hook `clang-tidy` est-il optionnel ici ?

Oui. Si votre équipe constate que `clang-tidy` rend le hook trop lent (> 15 secondes sur un commit typique), retirez-le de ce bloc et laissez la CI s'en charger. `clang-format` seul couvre déjà 80 % du bénéfice des hooks C++ en termes de réduction du bruit en code review. Pour désactiver temporairement `clang-tidy` sans le supprimer du fichier, utilisez le mécanisme `SKIP` :

```bash
# Contourner clang-tidy pour un commit spécifique
SKIP=clang-tidy git commit -m "feat: work in progress, tidy later"
```

C'est plus granulaire que `--no-verify` qui désactive *tous* les hooks.

## Bloc 3 : Formatage CMake

```yaml
  # ── Bloc 3 : CMake — Formatage et linting ─────────────────
  - repo: https://github.com/cheshirekow/cmake-format-precommit
    rev: v0.6.13
    hooks:
      # Formatage des fichiers CMake (utilise .cmake-format.yaml)
      - id: cmake-format

      # Linting CMake (détecte les mauvaises pratiques)
      - id: cmake-lint
        args: ['--disabled-codes=C0103,C0301']
```

### Notes sur `cmake-lint`

Les codes désactivés dans l'exemple :

- `C0103` : convention de nommage des variables. Les projets CMake utilisent des conventions variées (`lower_case`, `UPPER_CASE`, `camelCase`) qui ne correspondent pas toujours au défaut de `cmake-lint`.  
- `C0301` : longueur de ligne. Si votre `.cmake-format.yaml` gère déjà la longueur via le reformatage, ce check devient redondant.

Ajustez ces codes selon les conventions de votre projet. Pour voir la liste complète des checks disponibles :

```bash
cmake-lint --list-lint-codes
```

## Bloc 4 : Détection de typos

```yaml
  # ── Bloc 4 : Détection de fautes de frappe ────────────────
  - repo: https://github.com/codespell-project/codespell
    rev: v2.3.0
    hooks:
      - id: codespell
        args:
          - '--skip=build,third_party,*.pb.cc,*.pb.h,*.json,*.svg'
          - '--ignore-words-list=dout,ot,crate,te,ans'
        files: '\.(cpp|hpp|h|cc|cxx|c|cmake|md|txt|yaml|yml)$'
```

### Gestion des faux positifs

Le paramètre `--ignore-words-list` est la clé pour éviter la frustration avec `codespell`. Les projets C++ contiennent des identifiants qui ressemblent à des erreurs mais n'en sont pas :

| Terme | Détection codespell | Raison de l'exclusion |
|---|---|---|
| `dout` | → "doubt" ? | Nom de variable courant (debug output) |
| `ot` | → "to" ? | Abréviation légitime |
| `crate` | → "create" ? | Terme Rust dans les projets interop C++/Rust |
| `te` | → "the" ? | Variable mathématique |
| `ans` | → "and" ? | Variable pour "answer" |

Alimentez cette liste progressivement. Quand un développeur rencontre un faux positif, il ajoute le mot à la liste et commit le changement dans `.pre-commit-config.yaml`. Au bout de quelques semaines, les faux positifs disparaissent.

Pour les projets avec beaucoup de termes spécifiques, déplacez la liste dans un fichier dédié :

```yaml
args:
  - '--skip=build,third_party,*.pb.cc,*.pb.h'
  - '--ignore-words-list-file=.codespell-ignore'
```

```
# .codespell-ignore (un mot par ligne)
dout  
ot  
crate  
te  
ans  
numer  
```

## Bloc 5 : Validation des messages de commit

```yaml
  # ── Bloc 5 : Messages de commit (Conventional Commits) ────
  - repo: https://github.com/commitizen-tools/commitizen
    rev: v3.29.0
    hooks:
      - id: commitizen
        stages: [commit-msg]
```

Ce hook nécessite que le hook `commit-msg` ait été installé (`pre-commit install --hook-type commit-msg`). Il valide le format Conventional Commits décrit en section 47.1.

La configuration de `commitizen` est définie dans `.cz.toml` :

```toml
# .cz.toml
[tool.commitizen]
name = "cz_conventional_commits"  
version = "0.1.0"  
tag_format = "v$version"  
version_files = [  
    "CMakeLists.txt:project\\(.*VERSION",
]
```

Le champ `version_files` permet à `commitizen` de mettre à jour automatiquement le numéro de version dans `CMakeLists.txt` lors d'un bump de version (section 47.6).

## Bloc 6 : Hooks locaux personnalisés

```yaml
  # ── Bloc 6 : Vérifications spécifiques au projet ──────────
  - repo: local
    hooks:
      # Vérifier que compile_commands.json existe pour clang-tidy
      - id: check-compile-commands
        name: "Check compile_commands.json exists"
        entry: bash -c 'if [ ! -f build/compile_commands.json ]; then echo "⚠️  build/compile_commands.json manquant."; echo "   Lancez : cmake -B build -DCMAKE_EXPORT_COMPILE_COMMANDS=ON"; echo "   clang-tidy sera ignoré pour ce commit."; fi; exit 0'
        language: system
        always_run: true
        pass_filenames: false
        stages: [pre-commit]

      # Interdire les TODO sans référence de ticket
      - id: todo-with-ticket
        name: "TODOs must reference a ticket"
        entry: bash -c 'FOUND=$(grep -rnE "//\s*TODO[^(]|//\s*TODO$" --include="*.cpp" --include="*.hpp" --include="*.h" "$@" 2>/dev/null); if [ -n "$FOUND" ]; then echo "❌ TODOs sans référence de ticket :"; echo "$FOUND"; echo ""; echo "Format attendu : // TODO(PROJ-123): description"; exit 1; fi; exit 0'
        language: system
        files: '\.(cpp|hpp|h|cc|cxx)$'
```

### Notes sur les hooks locaux

**`check-compile-commands`** — Ce hook ne bloque pas le commit (`exit 0` dans tous les cas). Il affiche un avertissement si `compile_commands.json` n'existe pas, ce qui signifie que le hook `clang-tidy` échouera ou produira des résultats peu fiables. C'est un rappel non bloquant, pas une contrainte — un développeur qui vient de cloner le projet et n'a pas encore lancé CMake ne doit pas être bloqué.

**`todo-with-ticket`** — Optionnel mais très utile dans les équipes structurées. Ce hook impose que tout commentaire `TODO` soit associé à un ticket (`TODO(PROJ-123): ...`). Les TODOs orphelins s'accumulent indéfiniment ; les TODOs avec ticket sont traçables et peuvent être planifiés. Le pattern `TODO[^(]|TODO$` détecte les TODOs suivis d'un caractère qui n'est pas une parenthèse ouvrante, ou les TODOs en fin de ligne.

## Variantes selon le profil de projet

### Profil minimal (projet personnel, prototype)

Retirez les blocs 3 (CMake), 5 (commitizen), 6 (hooks locaux) et le hook `clang-tidy` du bloc 2. Ne gardez que l'hygiène de base et `clang-format` :

```yaml
repos:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.6.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-added-large-files
        args: ['--maxkb=500']

  - repo: https://github.com/pocc/pre-commit-hooks
    rev: v1.3.5
    hooks:
      - id: clang-format
        args: ['-i', '--style=file']
        files: '\.(cpp|hpp|h|cc|cxx|c)$'
```

Temps total : **< 1 seconde**.

### Profil strict (projet critique, 10+ développeurs)

Ajoutez au fichier complet :

```yaml
      # Bloc 1 — ajout
      - id: check-case-conflict     # Détecte les conflits de casse (Linux vs macOS)
      - id: forbid-submodules        # Interdit les sous-modules Git accidentels

  # Bloc supplémentaire — vérification des licences
  - repo: https://github.com/Lucas-C/pre-commit-hooks
    rev: v1.5.5
    hooks:
      - id: insert-license
        files: '\.(cpp|hpp|h|cc|cxx)$'
        args:
          - '--license-filepath=.license-header'
          - '--comment-style=//'
```

Et renforcez le hook `clang-tidy` avec des checks explicites si la version du `.clang-tidy` est jugée trop lente :

```yaml
      - id: clang-tidy
        args:
          - '--checks=-*,bugprone-*,clang-analyzer-*,misc-*'
          - '-p=build'
          - '--warnings-as-errors=*'
        files: '\.(cpp|cc|cxx)$'
```

L'argument `--checks` sur la ligne de commande **surcharge** la configuration du fichier `.clang-tidy` uniquement pour le hook pre-commit. La CI continue d'utiliser le `.clang-tidy` complet. C'est la technique clé pour maintenir deux niveaux de rigueur avec un seul fichier de configuration.

## Mise en service sur un projet existant

### Le commit initial de reformatage

Quand vous ajoutez `pre-commit` à un projet existant qui n'utilisait pas `clang-format`, le premier `pre-commit run --all-files` va reformater potentiellement des centaines de fichiers. Ce reformatage massif doit être géré proprement pour ne pas polluer l'historique Git.

**Étape 1 : Créer le commit de reformatage isolé**

```bash
# Lancer le reformatage sur tout le projet
pre-commit run --all-files

# Stager tous les fichiers reformatés
git add -A

# Commit dédié, clairement identifié
git commit -m "style: apply clang-format to entire codebase

This is a one-time formatting commit. Use 'git blame --ignore-rev'  
to skip this commit in blame output.  
Ref: .git-blame-ignore-revs"  
```

**Étape 2 : Créer `.git-blame-ignore-revs`**

Ce fichier permet à `git blame` d'ignorer les commits de reformatage massif, pour que le blame continue de pointer vers les commits fonctionnels :

```bash
# Récupérer le hash du commit de reformatage
HASH=$(git rev-parse HEAD)

# Créer le fichier .git-blame-ignore-revs
echo "# Commit de reformatage clang-format initial" > .git-blame-ignore-revs  
echo "$HASH" >> .git-blame-ignore-revs  

git add .git-blame-ignore-revs  
git commit -m "chore: add git-blame-ignore-revs for formatting commit"  
```

**Étape 3 : Configurer Git pour utiliser ce fichier automatiquement**

```bash
# Configuration locale (par projet)
git config blame.ignoreRevsFile .git-blame-ignore-revs
```

Documentez cette configuration dans le `CONTRIBUTING.md` pour que chaque développeur l'applique, ou ajoutez-la dans le script `setup-dev.sh` :

```bash
# Dans scripts/setup-dev.sh — après pre-commit install
git config blame.ignoreRevsFile .git-blame-ignore-revs
```

GitHub et GitLab prennent en charge nativement `.git-blame-ignore-revs` dans leur interface web — le blame en ligne ignorera automatiquement les commits listés.

### L'ordre d'adoption recommandé

Pour un projet existant, n'introduisez pas tous les hooks en même temps. Procédez par paliers pour éviter la résistance de l'équipe :

**Semaine 1** — Hygiène de base uniquement (trailing-whitespace, end-of-file-fixer, check-added-large-files). Impact quasi nul sur le workflow, les développeurs s'habituent au mécanisme.

**Semaine 2** — Ajout de `clang-format` avec le commit de reformatage initial. C'est le changement le plus visible, il mérite une communication à l'équipe.

**Semaine 3** — Ajout de `cmake-format`, `codespell` et `commitizen`. Ces hooks ont un impact modéré et sont bien tolérés.

**Semaine 4+** — Ajout de `clang-tidy` (en commençant par un jeu de checks minimal), hooks locaux personnalisés. Ajustez en fonction du retour de l'équipe.

## Vérification et diagnostic

### Valider la configuration

```bash
# Vérifier que le fichier YAML est syntaxiquement valide
pre-commit validate-config

# Lister tous les hooks configurés
pre-commit run --all-files --verbose 2>&1 | head -50
```

### Diagnostiquer un hook qui échoue

```bash
# Exécuter un hook spécifique en mode verbose
pre-commit run clang-format --all-files --verbose

# Exécuter sur un fichier précis
pre-commit run clang-tidy --files src/core/engine.cpp --verbose
```

Le mode `--verbose` affiche la commande exacte exécutée par le hook, les arguments passés et la sortie complète. C'est indispensable pour comprendre pourquoi un hook échoue de manière inattendue.

### Problèmes courants et solutions

**"clang-format not found"** — Le binaire n'est pas dans le `PATH`. Installez-le (`sudo apt install clang-format-19`) et vérifiez avec `which clang-format`.

**"Error reading compile_commands.json"** — Le fichier n'existe pas ou le chemin `-p=build` ne correspond pas. Lancez `cmake -B build -DCMAKE_EXPORT_COMPILE_COMMANDS=ON` pour le générer.

**Hook `clang-tidy` extrêmement lent** — Trop de checks activés ou fichiers avec des templates lourds. Restreignez les checks via l'argument `--checks` ou déplacez `clang-tidy` vers la CI.

**"Failed to fetch"** lors de `pre-commit install-hooks` — Problème réseau. Vérifiez l'accès à github.com. Derrière un proxy, configurez les variables `HTTP_PROXY`/`HTTPS_PROXY`.

**Hooks qui modifient les fichiers en boucle** — Deux hooks modifient le même fichier de manière contradictoire (par exemple `end-of-file-fixer` ajoute un newline que `trailing-whitespace` retire ensuite). Vérifiez l'ordre des hooks et testez avec `pre-commit run --all-files` jusqu'à stabilisation (deux passes suffisent normalement).

## Résumé

Le fichier `.pre-commit-config.yaml` est le contrat de qualité automatisé de votre projet. Il est versionné, partagé, et appliqué mécaniquement à chaque commit. Les six blocs présentés — hygiène, C++, CMake, typos, messages de commit, hooks locaux — forment une configuration modulable que vous adaptez en retirant ou ajoutant des blocs selon votre contexte. Le point essentiel reste la rapidité : si le total dépasse 15 secondes, allégez. Un hook rapide que tout le monde utilise vaut mieux qu'un hook exhaustif que personne ne respecte.
