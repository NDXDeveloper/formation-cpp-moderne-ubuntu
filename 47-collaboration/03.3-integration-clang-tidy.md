# 47.3.3 — Intégration clang-tidy

> **Chapitre 47 : Collaboration et Maintenance** · Section 47.3 : Configuration pre-commit  
> **Niveau** : Expert  
> **Prérequis** : Section 47.3.2 (Intégration clang-format), section 32.1 (clang-tidy — analyse statique moderne), section 26.5 (CMake et génération Ninja)

---

## Introduction

Si `clang-format` (section 47.3.2) est le hook C++ le plus simple à intégrer — rapide, déterministe, sans dépendance externe — `clang-tidy` est son exact opposé. C'est un outil d'analyse *sémantique* : il ne se contente pas de lire le texte du fichier, il le compile partiellement pour comprendre les types, résoudre les overloads, instancier les templates et évaluer le flux de contrôle. Cette profondeur d'analyse est sa force — elle lui permet de détecter des bugs réels, des violations de const-correctness, des use-after-move, des conversions implicites dangereuses. Mais elle est aussi la source de toutes les difficultés d'intégration dans un hook pre-commit.

Cette sous-section résout les trois problèmes fondamentaux identifiés en section 47.3 :

1. **`compile_commands.json`** — comment le générer, le localiser, et le maintenir à jour pour que `clang-tidy` fonctionne correctement dans le hook.
2. **Temps d'exécution** — comment sélectionner un jeu de checks qui reste rapide en pre-commit tout en capturant les bugs critiques.
3. **Faux positifs** — comment exclure le code tiers, les headers isolés et les constructions qui déclenchent des diagnostics non pertinents.

## Prérequis : `compile_commands.json`

### Pourquoi c'est indispensable

`clang-tidy` a besoin de savoir comment chaque fichier est compilé : quels flags de compilation sont utilisés, quels chemins d'include sont configurés, quelles macros sont définies. Sans ces informations, l'outil ne peut pas résoudre les `#include`, ne comprend pas les `#ifdef`, et produit des diagnostics absurdes.

Ces informations sont stockées dans le fichier `compile_commands.json` — une base de données de compilation au format JSON, standardisée par le projet Clang. Chaque entrée décrit la commande de compilation d'un fichier source :

```json
[
  {
    "directory": "/home/dev/project/build",
    "command": "g++ -std=c++23 -Wall -Wextra -I/home/dev/project/include -I/home/dev/project/third_party/json/include -DNDEBUG -O2 -c /home/dev/project/src/core/engine.cpp -o src/core/engine.cpp.o",
    "file": "/home/dev/project/src/core/engine.cpp"
  },
  {
    "directory": "/home/dev/project/build",
    "command": "g++ -std=c++23 -Wall -Wextra -I/home/dev/project/include ...",
    "file": "/home/dev/project/src/network/connection_pool.cpp"
  }
]
```

### Génération avec CMake

CMake génère automatiquement `compile_commands.json` avec l'option `CMAKE_EXPORT_COMPILE_COMMANDS` :

```bash
cmake -B build -DCMAKE_EXPORT_COMPILE_COMMANDS=ON
```

Le fichier est créé dans le répertoire de build (`build/compile_commands.json`). Pour éviter de devoir passer cette option à chaque configuration, intégrez-la dans le `CMakePresets.json` :

```json
{
    "version": 6,
    "configurePresets": [
        {
            "name": "dev",
            "displayName": "Development",
            "generator": "Ninja",
            "binaryDir": "${sourceDir}/build",
            "cacheVariables": {
                "CMAKE_BUILD_TYPE": "Debug",
                "CMAKE_EXPORT_COMPILE_COMMANDS": "ON"
            }
        },
        {
            "name": "release",
            "displayName": "Release",
            "generator": "Ninja",
            "binaryDir": "${sourceDir}/build-release",
            "cacheVariables": {
                "CMAKE_BUILD_TYPE": "Release",
                "CMAKE_EXPORT_COMPILE_COMMANDS": "ON"
            }
        }
    ]
}
```

Ainsi, tout développeur qui utilise `cmake --preset dev` obtient automatiquement `compile_commands.json` dans `build/`.

### Le problème du chemin : comment `clang-tidy` trouve le fichier

Par défaut, `clang-tidy` cherche `compile_commands.json` dans le répertoire courant ou dans un répertoire parent. Le hook pre-commit exécute `clang-tidy` depuis la racine du projet. Si `compile_commands.json` se trouve dans `build/`, `clang-tidy` ne le trouve pas sans aide.

Trois solutions existent, par ordre de préférence :

**Solution 1 : l'argument `-p` dans le hook (recommandée)**

C'est la solution utilisée dans notre `.pre-commit-config.yaml` (section 47.3.1) :

```yaml
- id: clang-tidy
  args:
    - '--config-file=.clang-tidy'
    - '-p=build'
    - '--warnings-as-errors=*'
  files: '\.(cpp|cc|cxx)$'
```

L'argument `-p=build` indique explicitement le répertoire contenant `compile_commands.json`. C'est simple, explicite, et ne nécessite aucun fichier supplémentaire.

**Inconvénient** : si un développeur utilise un répertoire de build différent (`cmake-build-debug/` sous CLion, `out/` par convention personnelle), le hook échoue. La standardisation du répertoire de build via `CMakePresets.json` résout ce problème.

**Solution 2 : symlink à la racine du projet**

Un lien symbolique à la racine pointe vers le fichier dans le répertoire de build :

```bash
ln -sf build/compile_commands.json compile_commands.json
```

`clang-tidy` trouve alors le fichier automatiquement dans le répertoire courant. Le symlink peut être créé par le script `setup-dev.sh` :

```bash
# Dans scripts/setup-dev.sh
if [ -f build/compile_commands.json ]; then
    ln -sf build/compile_commands.json compile_commands.json
    echo "  ✅ compile_commands.json symlink créé"
fi
```

**Inconvénient** : le symlink doit être ajouté au `.gitignore` (c'est un artefact local, pas un fichier du projet). Et si le build n'a pas encore été lancé, le symlink pointe vers une cible inexistante.

**Solution 3 : hook local de préparation**

Le hook `check-compile-commands` défini en section 47.3.1 avertit quand le fichier est absent. On peut aller plus loin avec un hook qui crée le symlink automatiquement :

```yaml
- repo: local
  hooks:
    - id: ensure-compile-commands
      name: "Ensure compile_commands.json is accessible"
      entry: bash -c '
        if [ -f build/compile_commands.json ]; then
          if [ ! -e compile_commands.json ]; then
            ln -sf build/compile_commands.json compile_commands.json
          fi
        else
          echo "⚠️  build/compile_commands.json absent"
          echo "   clang-tidy peut produire des faux positifs"
          echo "   Lancez : cmake --preset dev"
        fi
        exit 0'
      language: system
      always_run: true
      pass_filenames: false
```

Ce hook ne bloque jamais le commit (`exit 0`), mais garantit que le symlink existe si le build a été effectué.

### Maintenir `compile_commands.json` à jour

Le fichier `compile_commands.json` reflète l'état du build au moment où CMake a été exécuté. Si vous ajoutez un nouveau fichier source, modifiez les `target_include_directories`, ou changez les options de compilation dans `CMakeLists.txt`, il faut relancer CMake pour régénérer le fichier :

```bash
cmake --preset dev    # Régénère compile_commands.json
```

En pratique, avec Ninja (section 28.3), CMake se relance automatiquement si un `CMakeLists.txt` a changé. Le fichier `compile_commands.json` est mis à jour en conséquence. Mais si vous travaillez sur une branche qui ajoute de nouveaux fichiers sources sans jamais lancer le build, `compile_commands.json` ne les connaîtra pas et `clang-tidy` les ignorera silencieusement.

> **Bonne pratique** : prenez l'habitude de lancer `cmake --preset dev` après un rebase ou un merge important qui a modifié la structure du projet.

## Le fichier `.clang-tidy`

### Structure et syntaxe

Le fichier `.clang-tidy` est un fichier YAML placé à la racine du projet. Il définit les checks activés, les options de ces checks, et les filtres d'exclusion :

```yaml
# .clang-tidy
# ═══════════════════════════════════════════════════════════════
# Configuration clang-tidy — Projet C++ moderne
# Requiert : clang-tidy 19+
# ═══════════════════════════════════════════════════════════════
---
# Checks activés (configuration CI complète)
Checks: >
  -*,
  bugprone-*,
  clang-analyzer-*,
  cppcoreguidelines-*,
  misc-*,
  modernize-*,
  performance-*,
  readability-*,
  -bugprone-easily-swappable-parameters,
  -cppcoreguidelines-avoid-magic-numbers,
  -cppcoreguidelines-pro-bounds-array-to-pointer-decay,
  -modernize-use-trailing-return-type,
  -readability-magic-numbers,
  -readability-identifier-length,
  -readability-named-parameter

# Transformer les warnings en erreurs (en CI)
WarningsAsErrors: ''

# En-têtes à analyser (regex)
HeaderFilterRegex: 'include/.*\.h(pp)?$'

# Checks nécessitant des options spécifiques
CheckOptions:
  # ── readability ────────────────────────────────────────────
  - key: readability-identifier-naming.ClassCase
    value: CamelCase
  - key: readability-identifier-naming.FunctionCase
    value: camelCase
  - key: readability-identifier-naming.VariableCase
    value: lower_case
  - key: readability-identifier-naming.ConstantCase
    value: UPPER_CASE
  - key: readability-identifier-naming.NamespaceCase
    value: lower_case
  - key: readability-identifier-naming.PrivateMemberSuffix
    value: '_'
  - key: readability-function-cognitive-complexity.Threshold
    value: 25

  # ── modernize ──────────────────────────────────────────────
  - key: modernize-use-override.IgnoreDestructors
    value: true
  - key: modernize-loop-convert.MinConfidence
    value: reasonable
  - key: modernize-use-nullptr.NullMacros
    value: 'NULL'
  - key: modernize-use-auto.MinTypeNameLength
    value: 5

  # ── performance ────────────────────────────────────────────
  - key: performance-unnecessary-value-param.AllowedTypes
    value: 'std::shared_ptr;std::unique_ptr'
  - key: performance-move-const-arg.CheckTriviallyCopyableMove
    value: false

  # ── misc ───────────────────────────────────────────────────
  - key: misc-non-private-member-variables-in-classes.IgnoreClassesWithAllMemberVariablesBeingPublic
    value: true

  # ── cppcoreguidelines ─────────────────────────────────────
  - key: cppcoreguidelines-special-member-functions.AllowSoleDefaultDtor
    value: true

# Exclure les répertoires hors du contrôle du projet
ExcludeHeaderFilterRegex: '(third_party|build|vendor)/.*'

SystemHeaders: false  
UseColor: true  
```

### Anatomie de la chaîne `Checks`

La chaîne `Checks` est le cœur de la configuration. Elle fonctionne par ajout/retrait séquentiel :

```
-*                                          ← Désactive tout
bugprone-*                                  ← Active tous les checks bugprone-*  
clang-analyzer-*                            ← Active tous les checks clang-analyzer-*  
...
-bugprone-easily-swappable-parameters       ← Désactive ce check spécifique
-readability-identifier-length              ← Désactive ce check spécifique
```

L'ordre compte : les règles ultérieures écrasent les précédentes. Le pattern `−*` suivi d'activations explicites crée une *whitelist* (seuls les checks listés sont actifs). C'est plus sûr qu'une approche *blacklist* (`*` puis `-check1,-check2`) car une mise à jour de `clang-tidy` n'active pas automatiquement de nouveaux checks inconnus.

### Checks désactivés et justification

Chaque check désactivé dans la configuration mérite une justification. Sans cela, les futurs mainteneurs ne savent pas si le check a été désactivé par choix réfléchi ou par exaspération :

| Check désactivé | Raison |
|---|---|
| `bugprone-easily-swappable-parameters` | Produit un volume massif de warnings sur les fonctions avec plusieurs paramètres du même type. Le rapport signal/bruit est trop faible. |
| `cppcoreguidelines-avoid-magic-numbers` | Alias de `readability-magic-numbers` — désactiver les deux évite la double détection. Trop de faux positifs dans le code de test et les constantes locales évidentes. |
| `cppcoreguidelines-pro-bounds-array-to-pointer-decay` | Incompatible avec l'utilisation d'`assert()` et des macros de logging qui prennent des string literals. |
| `modernize-use-trailing-return-type` | Imposer `auto f() -> int` partout est un choix de style controversé que beaucoup d'équipes rejettent. |
| `readability-identifier-length` | Les variables courtes (`i`, `j`, `n`, `it`) sont idiomatiques en C++, surtout dans les boucles et les lambdas. |
| `readability-named-parameter` | En conflit avec la convention de laisser les paramètres sans nom dans les déclarations quand le nom n'apporte pas d'information. |

> **Bonne pratique** : maintenez ces justifications dans un commentaire au-dessus de chaque désactivation dans le `.clang-tidy`, ou dans un fichier `docs/clang-tidy-decisions.md` pour les cas complexes.

### `HeaderFilterRegex` : contrôler le scope d'analyse des headers

Par défaut, `clang-tidy` n'analyse que le fichier source passé en argument. Les diagnostics provenant des headers inclus sont supprimés. L'option `HeaderFilterRegex` contrôle quels headers inclus sont également analysés :

```yaml
HeaderFilterRegex: 'include/.*\.h(pp)?$'
```

Ce pattern analyse les headers situés dans le répertoire `include/` du projet. Les headers système (`<vector>`, `<string>`) et les headers tiers (`third_party/`) sont exclus.

**Attention** : un `HeaderFilterRegex` trop large (comme `.*`) inclut les headers système et tiers dans l'analyse, ce qui produit des milliers de diagnostics hors de votre contrôle et ralentit considérablement l'exécution. Soyez aussi précis que possible.

`ExcludeHeaderFilterRegex` est le complément : il exclut explicitement certains chemins même s'ils correspondent au `HeaderFilterRegex`. C'est une sécurité supplémentaire.

### `CheckOptions` : conventions de nommage

Le bloc `readability-identifier-naming.*` dans `CheckOptions` est particulièrement utile dans un contexte de collaboration. Il impose automatiquement les conventions de nommage du projet :

```cpp
// Avec la configuration ci-dessus :
class NetworkManager {          // ✅ CamelCase (ClassCase)
    void connectToServer();     // ✅ camelCase (FunctionCase)
    int retry_count_;           // ✅ lower_case + suffix _ (PrivateMemberSuffix)
    static constexpr int MAX_RETRIES = 5;  // ✅ UPPER_CASE (ConstantCase)
};

// Violations détectées :
class network_manager {};       // ❌ ClassCase: expected CamelCase  
void ConnectToServer();         // ❌ FunctionCase: expected camelCase  
int retryCount;                 // ❌ VariableCase: expected lower_case  
```

Adaptez ces conventions à votre projet. Le système est très flexible — `clang-tidy` supporte des dizaines de catégories d'identifiants (paramètres, variables locales, membres publics, constantes globales, enums, typedefs, concepts, etc.).

## Stratégie à deux niveaux : pre-commit vs CI

C'est le point central de cette sous-section. Le même fichier `.clang-tidy` est utilisé à la fois en pre-commit et en CI, mais le hook pre-commit **restreint les checks activés** via l'argument `--checks` sur la ligne de commande, qui surcharge la configuration du fichier.

### Niveau 1 : Pre-commit (rapide, critique)

Le hook pre-commit n'active que les checks qui détectent des **bugs réels** et des **comportements indéfinis** — les catégories où le rapport bénéfice/temps est maximal :

```yaml
# Dans .pre-commit-config.yaml
- id: clang-tidy
  args:
    - '--checks=-*,bugprone-*,clang-analyzer-*,misc-use-after-move,misc-unconventional-assign-operator'
    - '--config-file=.clang-tidy'
    - '-p=build'
    - '--warnings-as-errors=*'
  files: '\.(cpp|cc|cxx)$'
```

L'argument `--checks=-*,bugprone-*,clang-analyzer-*,...` **surcharge** le champ `Checks` du fichier `.clang-tidy` sans le modifier. Le fichier `.clang-tidy` conserve la configuration complète pour la CI, mais le hook n'en active qu'un sous-ensemble.

**Checks activés en pre-commit et ce qu'ils détectent :**

| Catégorie | Exemples de détection |
|---|---|
| `bugprone-*` | Use-after-move, argument swapping, forwarding reference overload, integer narrowing, infinite loop, string constructor ambiguity |
| `clang-analyzer-*` | Null pointer dereference, use-after-free, division par zéro, fuite mémoire, dead code prouvé par analyse de flux |
| `misc-use-after-move` | Utilisation d'un objet après `std::move` |
| `misc-unconventional-assign-operator` | Opérateur `=` qui ne retourne pas `*this` |

Ce sous-ensemble représente typiquement 40-50 checks. Il couvre les bugs les plus dangereux — ceux qui compilent sans warning mais produisent un comportement indéfini ou un crash en runtime.

**Temps d'exécution attendu** : 2-10 secondes pour 1 à 3 fichiers `.cpp` modifiés. Acceptable dans un hook pre-commit.

### Niveau 2 : CI (exhaustif, complet)

La CI utilise le fichier `.clang-tidy` directement, sans restriction de checks :

```yaml
# .gitlab-ci.yml (extrait)
clang-tidy:
  stage: analyze
  script:
    - cmake --preset dev
    - cmake --build --preset dev
    - >
      run-clang-tidy
      -p build
      -config-file .clang-tidy
      -warnings-as-errors '*'
      -header-filter 'include/.*\.h(pp)?$'
```

`run-clang-tidy` (fourni avec le package `clang-tools`) parallélise l'exécution de `clang-tidy` sur tous les fichiers du projet. C'est bien plus efficace que d'exécuter `clang-tidy` fichier par fichier.

**Checks supplémentaires activés uniquement en CI :**

| Catégorie | Rôle |
|---|---|
| `modernize-*` | Détecte le code pré-C++17 qui pourrait être modernisé (`auto`, `override`, `nullptr`, range-for, structured bindings) |
| `performance-*` | Copies inutiles, moves manquants, `reserve()` oublié, passage par valeur quand const-ref serait préférable |
| `readability-*` | Complexité cognitive excessive, fonctions trop longues, nommage, `else` après `return` |
| `cppcoreguidelines-*` | Conformité aux C++ Core Guidelines (rule of five, slicing, gsl::owner) |

**Temps d'exécution attendu** : 5-30 minutes selon la taille du projet, sur un runner CI. C'est acceptable en CI car l'exécution est asynchrone — le développeur n'attend pas.

### Pourquoi un seul fichier `.clang-tidy` et pas deux

Il serait tentant de maintenir deux fichiers : `.clang-tidy` pour la CI et `.clang-tidy.precommit` pour le hook. Cela crée un problème de synchronisation : quand un développeur modifie une option dans l'un, il oublie de la reporter dans l'autre. La divergence s'installe et les diagnostics deviennent incohérents.

L'approche à un seul fichier + override par `--checks` en ligne de commande est plus robuste : le fichier `.clang-tidy` est la source de vérité unique pour les `CheckOptions` (conventions de nommage, seuils, options de checks). Le hook pre-commit se contente de restreindre le *périmètre* des checks exécutés.

## Gestion des headers dans le hook pre-commit

### Le problème

Un header `.h` ou `.hpp` n'est pas une unité de traduction. Le compiler isolément nécessite de deviner les flags de compilation, les includes transitifs et les macros de configuration. `clang-tidy` utilise les informations de `compile_commands.json`, mais ce fichier ne contient que des entrées pour les fichiers `.cpp` — pas pour les headers.

Quand `clang-tidy` analyse un header sans contexte, trois choses se passent :

1. Les includes du header qui dépendent de macros définies par le `.cpp` échouent.
2. Les types définis dans des headers inclus transitifs ne sont pas résolus.
3. Les diagnostics produits sont souvent des faux positifs.

### La solution : exclure les headers du hook pre-commit

C'est pourquoi la configuration du hook (sections 47.2.2 et 47.3.1) restreint le scope aux fichiers sources :

```yaml
files: '\.(cpp|cc|cxx)$'    # Pas de .h, .hpp
```

Les headers sont analysés *indirectement* quand `clang-tidy` traite un fichier `.cpp` qui les inclut. Grâce au `HeaderFilterRegex` dans `.clang-tidy`, les diagnostics dans les headers du projet sont rapportés — mais uniquement dans le contexte de compilation du `.cpp` qui les inclut, ce qui garantit des diagnostics fiables.

L'analyse complète des headers est réservée à la CI, où `run-clang-tidy` traite tous les fichiers `.cpp` du projet et couvre donc tous les headers transitifs.

### Cas particulier : headers-only libraries

Si votre projet est une bibliothèque header-only (tous les fichiers sont des `.hpp`), le hook pre-commit sur `clang-tidy` ne s'exécute jamais (aucun `.cpp` modifié). Deux approches :

**Approche 1 : créer des fichiers `.cpp` de compilation** — un fichier par module qui inclut simplement le header :

```cpp
// tests/compile/test_compile_core.cpp
// Ce fichier n'est utilisé que pour l'analyse statique et la vérification de compilation
#include "mylib/core.hpp"
```

Ces fichiers apparaissent dans `compile_commands.json` et servent de points d'entrée pour `clang-tidy`.

**Approche 2 : inclure les headers dans le hook** avec un flag de tolérance — accepter que certains diagnostics puissent être des faux positifs et désactiver `--warnings-as-errors` :

```yaml
- id: clang-tidy
  args:
    - '--checks=-*,bugprone-*,clang-analyzer-*'
    - '--config-file=.clang-tidy'
    - '-p=build'
    # Pas de --warnings-as-errors pour les headers
  files: '\.(hpp|h)$'
```

La première approche est préférable — elle donne des diagnostics fiables.

## Suppressions ciblées : `NOLINT`

Même avec une configuration soignée, certains diagnostics sont des faux positifs ou des cas intentionnels. `clang-tidy` fournit des annotations pour les supprimer ponctuellement :

### `NOLINT` — suppression sur une ligne

```cpp
// Supprime tous les diagnostics sur cette ligne
auto raw_ptr = new Connection();  // NOLINT

// Supprime un check spécifique (préférable — plus explicite)
auto raw_ptr = new Connection();  // NOLINT(cppcoreguidelines-owning-memory)

// Supprime plusieurs checks spécifiques
reinterpret_cast<char*>(buf);  // NOLINT(cppcoreguidelines-pro-type-reinterpret-cast,performance-no-int-to-ptr)
```

### `NOLINTNEXTLINE` — suppression sur la ligne suivante

```cpp
// NOLINTNEXTLINE(bugprone-narrowing-conversions)
int width = static_cast<int>(precise_width);
```

### `NOLINTBEGIN` / `NOLINTEND` — suppression sur un bloc

```cpp
// NOLINTBEGIN(cppcoreguidelines-pro-type-reinterpret-cast)
void serialize(const Data& data, std::byte* buffer) {
    auto* ptr = reinterpret_cast<const std::byte*>(&data);
    std::memcpy(buffer, ptr, sizeof(Data));
}
// NOLINTEND(cppcoreguidelines-pro-type-reinterpret-cast)
```

### Règles d'usage

**Toujours spécifier le check.** Un `NOLINT` sans argument supprime *tous* les diagnostics — y compris des bugs réels que `clang-tidy` pourrait trouver à l'avenir. `NOLINT(check-name)` ne supprime que le diagnostic visé.

**Justifier la suppression.** Un commentaire expliquant *pourquoi* la suppression est nécessaire aide les futurs lecteurs :

```cpp
// Performance critique : on utilise reinterpret_cast pour le zero-copy serialization.
// La donnée est trivially copyable et correctement alignée.
// NOLINTNEXTLINE(cppcoreguidelines-pro-type-reinterpret-cast)
auto* ptr = reinterpret_cast<const std::byte*>(&data);
```

**Surveiller l'accumulation.** Si un fichier accumule plus de 5-10 suppressions `NOLINT`, c'est un signal : soit le code mériterait un refactoring, soit le check est trop strict pour votre contexte et devrait être désactivé globalement dans `.clang-tidy`.

**Rechercher les NOLINT obsolètes.** Quand du code change, les NOLINT peuvent devenir inutiles. `clang-tidy` 17+ supporte l'option `--verify-ignore-nolint` qui avertit quand un `NOLINT` ne supprime aucun diagnostic actif — un nettoyage utile à lancer périodiquement en CI.

## Traitement du code tiers et généré

### Headers tiers (`third_party/`, `vendor/`)

Le `exclude` global du `.pre-commit-config.yaml` (section 47.3.1) empêche déjà les hooks de traiter les fichiers dans ces répertoires. Mais `clang-tidy` peut quand même rapporter des diagnostics dans des headers tiers *inclus* par votre code, si le `HeaderFilterRegex` n'est pas assez restrictif.

La configuration proposée règle le problème à deux niveaux :

```yaml
# Dans .clang-tidy
HeaderFilterRegex: 'include/.*\.h(pp)?$'           # N'analyse QUE vos headers  
ExcludeHeaderFilterRegex: '(third_party|build|vendor)/.*'  # Exclut explicitement  
```

### Code généré (Protobuf, gRPC, flatbuffers)

Les fichiers générés par `protoc` (`.pb.h`, `.pb.cc`), `grpc_cpp_plugin` ou `flatc` contiennent du code produit par un outil, pas par un humain. Les analyser avec `clang-tidy` est inutile (vous ne pouvez pas corriger les violations) et coûteux en temps.

Le `exclude` global dans `.pre-commit-config.yaml` gère les patterns courants :

```yaml
exclude: |
  (?x)^(
    ...
    .*\.pb\.(h|cc)$|
    .*\.grpc\.pb\.(h|cc)$|
    .*\.generated\.\w+$
  )
```

Si votre projet utilise d'autres générateurs, ajoutez leurs patterns.

## Comportement quand `compile_commands.json` est absent

Un nouveau développeur qui clone le projet et tente de commiter immédiatement n'aura pas de `compile_commands.json` (il n'a pas encore lancé CMake). Que se passe-t-il ?

**Comportement par défaut** : `clang-tidy` échoue ou produit des résultats erratiques. Le hook bloque le commit pour des raisons non pertinentes. Mauvaise expérience.

**Comportement souhaité** : le hook `clang-tidy` est silencieusement ignoré si `compile_commands.json` n'existe pas, et un avertissement est affiché.

Pour obtenir ce comportement, remplacez le hook `clang-tidy` du dépôt `pocc/pre-commit-hooks` par un hook local qui vérifie l'existence du fichier avant d'exécuter l'outil :

```yaml
- repo: local
  hooks:
    - id: clang-tidy-safe
      name: "clang-tidy (skipped if no compile_commands.json)"
      entry: bash -c '
        if [ ! -f build/compile_commands.json ]; then
          echo "⏭️  clang-tidy ignoré (build/compile_commands.json absent)"
          echo "   Lancez cmake --preset dev pour activer l analyse statique"
          exit 0
        fi
        exec clang-tidy \
          --checks="-*,bugprone-*,clang-analyzer-*" \
          --config-file=.clang-tidy \
          -p=build \
          --warnings-as-errors="*" \
          "$@"'
      language: system
      files: '\.(cpp|cc|cxx)$'
      types: [c++]
```

Ce wrapper Bash vérifie l'existence de `compile_commands.json` et :
- S'il est absent : affiche un message informatif et quitte avec succès (`exit 0`) — le commit n'est pas bloqué.  
- S'il est présent : exécute `clang-tidy` normalement avec les arguments de niveau pre-commit.

C'est le compromis idéal entre robustesse et accessibilité : le nouveau développeur n'est pas pénalisé, et dès qu'il lance son premier build, `clang-tidy` s'active automatiquement.

## Intégration avec l'éditeur

### clangd et `clang-tidy` en temps réel

`clangd`, le serveur LSP de LLVM, intègre `clang-tidy` et exécute les checks en temps réel dans l'éditeur. Il utilise automatiquement le fichier `.clang-tidy` du projet. Cela signifie que le développeur voit les diagnostics **pendant qu'il code**, avant même le commit.

Configuration dans VS Code (`.vscode/settings.json`) :

```jsonc
{
    "clangd.arguments": [
        "--clang-tidy",
        "--header-insertion=iwyu",
        "--completion-style=detailed",
        "--compile-commands-dir=build"
    ]
}
```

L'argument `--clang-tidy` active l'intégration. `clangd` utilise le `.clang-tidy` du projet et le `compile_commands.json` spécifié par `--compile-commands-dir`.

**Effet vertueux** : comme pour `clang-format` à la sauvegarde, l'intégration éditeur rend le hook pre-commit quasi transparent. Les diagnostics sont corrigés en temps réel, et le hook ne fait que confirmer que tout est propre.

### Cohérence entre `clangd` et le hook

`clangd` utilise le fichier `.clang-tidy` complet (tous les checks). Le hook pre-commit n'utilise qu'un sous-ensemble via `--checks`. Cela signifie que le développeur peut voir des warnings `modernize-*` dans son éditeur qui ne bloquent pas le hook. C'est le comportement souhaité : l'éditeur informe, le hook bloque uniquement sur les bugs critiques, et la CI applique la rigueur complète.

## Diagnostic et résolution de problèmes

### `clang-tidy` est trop lent en pre-commit

**Symptôme** : le hook prend > 30 secondes pour 2-3 fichiers.

**Causes et solutions** :

1. **Trop de checks activés** — Vérifiez que l'argument `--checks` du hook restreint bien les catégories. `performance-*` et `readability-*` sont coûteux et n'ont pas leur place en pre-commit.

2. **Fichiers avec des templates lourds** — L'instanciation de templates (Boost, Eigen, templates variadiques profonds) ralentit considérablement l'analyse. Solution : exclure ces fichiers spécifiques du hook :

```yaml
- id: clang-tidy
  ...
  exclude: 'src/math/eigen_wrappers\.cpp$'
```

3. **`compile_commands.json` obsolète** — Si le fichier pointe vers un répertoire de build invalide, `clang-tidy` passe du temps à résoudre des chemins inexistants. Régénérez avec `cmake --preset dev`.

### Faux positifs récurrents

**Symptôme** : un check produit des diagnostics non pertinents dans votre contexte.

**Approche graduée** :

1. **Vérifier si c'est un vrai bug** — Ne rejetez pas un diagnostic trop vite. `clang-analyzer-*` a un taux de faux positifs faible ; si l'outil signale un null pointer dereference, investiguez avant de supprimer.

2. **Suppression locale** — Si c'est un faux positif isolé, utilisez `NOLINT(check-name)` avec une justification.

3. **Ajustement d'option** — Si le check est utile mais trop strict, ajustez ses options dans `CheckOptions` (seuil de complexité, types autorisés, etc.).

4. **Désactivation globale** — Si le check produit systématiquement des faux positifs dans votre codebase, désactivez-le dans `.clang-tidy` et documentez la raison.

### `error: no matching compilation database entry found`

**Symptôme** : `clang-tidy` ne trouve pas d'entrée pour un fichier dans `compile_commands.json`.

**Causes** :

- Le fichier est nouveau et CMake n'a pas été relancé depuis.  
- Le fichier n'est pas listé dans un `add_executable` ou `add_library` de `CMakeLists.txt`.  
- Le chemin dans `compile_commands.json` ne correspond pas au chemin utilisé par le hook (relatif vs absolu, symlink vs chemin réel).

**Solution** : relancez `cmake --preset dev` et vérifiez que le fichier apparaît dans `compile_commands.json`.

## Résumé

L'intégration de `clang-tidy` dans les pre-commit hooks repose sur trois piliers : un `compile_commands.json` à jour et accessible, une stratégie à deux niveaux de checks (pre-commit rapide, CI exhaustive), et une gestion rigoureuse du scope d'analyse (fichiers sources uniquement, exclusion du code tiers et généré). Le hook local avec fallback gracieux quand `compile_commands.json` est absent garantit que l'expérience développeur reste fluide, même pour un nouveau venu qui n'a pas encore lancé son premier build. Combiné à l'intégration `clangd` dans l'éditeur, `clang-tidy` devient un filet de sécurité quasi invisible qui détecte les bugs critiques avant qu'ils n'atteignent la branche partagée.
