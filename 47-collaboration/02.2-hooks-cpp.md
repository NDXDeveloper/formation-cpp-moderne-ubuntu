# 47.2.2 — Hooks essentiels pour C++

> **Chapitre 47 : Collaboration et Maintenance** · Section 47.2 : Pre-commit hooks  
> **Niveau** : Expert  
> **Prérequis** : Section 47.2.1 (Installation du framework pre-commit), chapitre 32 (Analyse statique et linting), chapitre 26 (CMake)

---

## Introduction

La section 47.2.1 a installé le framework `pre-commit` et expliqué la structure du fichier `.pre-commit-config.yaml`. Cette sous-section détaille les hooks concrets pertinents pour un projet C++, organisés par catégorie. L'objectif est de constituer un pipeline de vérification local qui couvre les besoins typiques sans sacrifier la rapidité d'exécution.

La configuration `.pre-commit-config.yaml` complète et intégrée sera assemblée en section 47.3, avec un focus particulier sur `clang-format` et `clang-tidy`. Ici, nous couvrons le *pourquoi* et le *quoi* de chaque hook, pour permettre un choix éclairé.

## Catégorie 1 : Hygiène générale du dépôt

Ces hooks ne sont pas spécifiques au C++ mais sont indispensables dans tout projet. Ils sont fournis par le dépôt officiel `pre-commit/pre-commit-hooks` et s'exécutent en quelques millisecondes.

### `trailing-whitespace`

Supprime les espaces et tabulations en fin de ligne dans les fichiers stagés.

```yaml
- id: trailing-whitespace
```

**Pourquoi c'est important en C++** : les espaces en fin de ligne polluent les diffs et provoquent des conflits de merge artificiels. Quand deux développeurs modifient le même fichier et que l'un d'eux ajoute un espace invisible en fin de ligne, Git signale un conflit qui n'a aucun rapport avec le code. Ce hook élimine cette catégorie de bruit.

Le hook modifie les fichiers en place (supprime les espaces). Si une modification est effectuée, le commit est bloqué et le développeur doit re-stager les fichiers corrigés.

### `end-of-file-fixer`

Garantit que chaque fichier texte se termine par exactement une ligne vide (newline final).

```yaml
- id: end-of-file-fixer
```

**Pourquoi c'est important en C++** : le standard C++ exige qu'un fichier source se termine par un newline (la norme parle de *non-empty source file shall end in a new-line character*). Les compilateurs émettent un warning si ce newline est absent (`-Wnewline-eof` avec Clang). Au-delà du standard, l'absence de newline final pollue les diffs : la dernière ligne apparaît comme modifiée à chaque ajout en fin de fichier.

### `check-added-large-files`

Bloque le commit si un fichier dépassant une taille seuil est stagé.

```yaml
- id: check-added-large-files
  args: ['--maxkb=500']
```

**Pourquoi c'est critique en C++** : les projets C++ sont particulièrement exposés au risque d'ajout accidentel de fichiers binaires volumineux — un exécutable compilé oublié dans `build/`, une bibliothèque `.so` copiée manuellement, un core dump, un fichier `.o` non exclu par le `.gitignore`. Un seul fichier de 50 Mo dans l'historique Git ne peut plus être retiré proprement sans réécrire l'historique (opération destructive). Ce hook est la dernière ligne de défense contre ce type d'erreur.

Le seuil de 500 Ko est un bon point de départ. Ajustez-le selon votre projet : si vous versionnez des fichiers de test binaires légitimes (fixtures protobuf, fichiers de données), augmentez le seuil ou excluez les répertoires concernés :

```yaml
- id: check-added-large-files
  args: ['--maxkb=1000']
  exclude: '^tests/fixtures/'
```

### `check-merge-conflict`

Détecte les marqueurs de conflit Git oubliés dans les fichiers (`<<<<<<<`, `=======`, `>>>>>>>`).

```yaml
- id: check-merge-conflict
```

Un oubli de résolution de conflit compile parfois sans erreur (le marqueur peut se trouver dans un commentaire ou une chaîne) mais produit un comportement absurde. Ce hook coûte zéro en performance et évite un piège embarrassant.

### `check-yaml` et `check-json`

Valident la syntaxe des fichiers YAML et JSON présents dans le dépôt.

```yaml
- id: check-yaml
- id: check-json
```

**Pertinence en C++** : un projet C++ moderne contient souvent des fichiers YAML (CI/CD, `.clang-format`, `.clang-tidy`, `CMakePresets.json`, configurations Conan, Docker Compose) et JSON (presets CMake, `compile_commands.json` versionné, fichiers de configuration applicatifs). Une erreur de syntaxe YAML dans `.gitlab-ci.yml` ou `.clang-tidy` peut bloquer la CI ou rendre silencieusement inopérant un outil d'analyse — des problèmes frustrants à diagnostiquer.

### `check-symlinks`

Vérifie que les liens symboliques pointent vers des cibles existantes.

```yaml
- id: check-symlinks
```

Les projets C++ sur Linux utilisent fréquemment des symlinks (bibliothèques partagées avec `libfoo.so → libfoo.so.2.1`, liens vers des headers tiers). Un symlink cassé versionné provoque des erreurs de compilation mystérieuses lors du clone sur une autre machine.

### `mixed-line-ending`

Détecte les fichiers contenant un mélange de fins de ligne `\n` (Unix) et `\r\n` (Windows).

```yaml
- id: mixed-line-ending
  args: ['--fix=lf']
```

L'argument `--fix=lf` normalise vers le format Unix, qui est le standard pour le développement C++ sur Linux. Les fins de ligne mixtes peuvent provoquer des problèmes subtils avec certains préprocesseurs et outils de build.

### Récapitulatif de la catégorie

```yaml
# ── Hygiène générale ──
- repo: https://github.com/pre-commit/pre-commit-hooks
  rev: v4.6.0
  hooks:
    - id: trailing-whitespace
    - id: end-of-file-fixer
    - id: check-added-large-files
      args: ['--maxkb=500']
    - id: check-merge-conflict
    - id: check-yaml
    - id: check-json
    - id: check-symlinks
    - id: mixed-line-ending
      args: ['--fix=lf']
```

Temps d'exécution total de cette catégorie : **< 1 seconde**, même sur un projet de grande taille. Il n'y a aucune raison de s'en priver.

## Catégorie 2 : Formatage et analyse du code C++

Ces hooks constituent le cœur de la vérification pour un projet C++. Ils s'appuient sur les outils du chapitre 32 (clang-format, clang-tidy) et les rendent automatiques.

### `clang-format` — Formatage automatique du code

Le hook `clang-format` vérifie (et optionnellement corrige) le formatage du code C++ selon les règles définies dans `.clang-format`.

```yaml
- repo: https://github.com/pocc/pre-commit-hooks
  rev: v1.3.5
  hooks:
    - id: clang-format
      args: ['--style=file']       # Utilise le .clang-format du projet
      files: '\.(cpp|hpp|h|cc|cxx|c)$'
```

**Comportement** : le hook lance `clang-format -i` sur chaque fichier C++ stagé. Si le fichier n'était pas correctement formaté, il est modifié en place et le commit est bloqué. Le développeur n'a plus qu'à re-stager et re-commiter.

**Temps d'exécution** : quasi instantané. `clang-format` traite des milliers de lignes en quelques millisecondes. C'est le hook le plus rapide et le plus impactant — il élimine à lui seul la majorité des commentaires de style en code review.

**L'argument `--style=file`** indique à `clang-format` de chercher le fichier `.clang-format` le plus proche dans l'arborescence. Sans cet argument, le style par défaut de l'outil serait utilisé, ce qui ne correspond pas forcément aux conventions du projet.

**Version de `clang-format`** : le hook utilise le binaire `clang-format` présent sur le système. Si votre projet exige une version spécifique (par exemple `clang-format-19` pour bénéficier des derniers styles C++23/26), vous pouvez le spécifier. L'approche dépend du dépôt de hooks utilisé — le détail est couvert en section 47.3.

> ⚠️ **Point critique** : tout le monde dans l'équipe doit utiliser la **même version** de `clang-format`. Deux versions différentes peuvent produire un formatage légèrement différent sur le même fichier, provoquant des commits de reformatage en ping-pong. Documentez la version requise dans le `CONTRIBUTING.md` et, idéalement, utilisez des DevContainers (section 2.4.3) pour garantir l'homogénéité.

### `clang-tidy` — Analyse statique rapide

Le hook `clang-tidy` exécute l'analyseur statique sur les fichiers C++ modifiés, en utilisant la configuration `.clang-tidy` du projet.

```yaml
- repo: https://github.com/pocc/pre-commit-hooks
  rev: v1.3.5
  hooks:
    - id: clang-tidy
      args: ['--config-file=.clang-tidy']
      files: '\.(cpp|cc|cxx)$'
```

**Comportement** : `clang-tidy` analyse chaque fichier source stagé et rapporte les violations. Contrairement à `clang-format`, il ne modifie pas les fichiers par défaut — il signale les problèmes et bloque le commit si des erreurs sont trouvées.

**Temps d'exécution** : c'est le hook le plus lent de la catégorie. `clang-tidy` doit parser le fichier, résoudre les includes, et appliquer les checks configurés. Sur un fichier unique, cela prend typiquement 2 à 10 secondes. Sur 5 fichiers modifiés, on atteint 10 à 30 secondes — la limite haute de ce qui est acceptable dans un hook pre-commit.

**Stratégies pour maintenir la rapidité** :

Limiter le scope aux fichiers `.cpp` (pas aux `.h`). Analyser un header isolé sans son contexte de compilation est souvent peu fiable et génère des faux positifs. La CI se chargera de l'analyse complète.

Sélectionner les checks activés avec soin. Un `.clang-tidy` trop ambitieux (toutes les catégories activées) ralentit l'analyse. En section 47.3, nous verrons comment configurer un jeu de checks adapté au pre-commit — plus restreint que celui de la CI.

**Nécessite `compile_commands.json`** : `clang-tidy` a besoin de la base de données de compilation pour résoudre les includes et les flags. Ce fichier est généré par CMake :

```bash
cmake -B build -DCMAKE_EXPORT_COMPILE_COMMANDS=ON
```

Le hook `clang-tidy` doit pouvoir localiser ce fichier. La configuration détaillée est couverte en section 47.3.3.

### Faut-il inclure `clang-tidy` dans les hooks pre-commit ?

C'est une question que chaque équipe doit trancher en fonction de son contexte :

**Arguments pour** : détecter les bugs et les mauvaises pratiques *avant* le push réduit drastiquement les allers-retours en review et en CI. C'est le shift-left poussé à son maximum.

**Arguments contre** : si le hook prend 30 secondes, les développeurs finiront par utiliser `--no-verify`. Le rapport coût/bénéfice est moins favorable que pour `clang-format` (qui est instantané).

**Compromis courant** : activer `clang-tidy` en hook pre-commit avec un jeu de checks restreint (checks critiques uniquement : bugs, undefined behavior, memory safety), et réserver l'analyse complète (modernisation, readability, performance) à la CI. Si même la version restreinte est trop lente, déplacer `clang-tidy` vers un hook `pre-push` ou le laisser exclusivement en CI.

## Catégorie 3 : Formatage CMake

Les fichiers `CMakeLists.txt` et les modules `.cmake` sont du code à part entière et méritent le même traitement que les sources C++.

### `cmake-format` — Formatage des fichiers CMake

```yaml
- repo: https://github.com/cheshirekow/cmake-format-precommit
  rev: v0.6.13
  hooks:
    - id: cmake-format
```

**Comportement** : reformate les fichiers CMake selon les règles définies dans `.cmake-format.yaml` (ou `.cmake-format.json`, `.cmake-format.py`). Comme `clang-format`, il modifie les fichiers en place et bloque le commit en cas de changement.

**Pourquoi c'est utile** : les fichiers CMake deviennent rapidement complexes dans les projets C++ de taille significative. Sans formatage cohérent, les `CMakeLists.txt` accumulent des styles hétérogènes — indentation incohérente, arguments alignés ou non, parenthèses sur des lignes différentes. `cmake-format` normalise tout cela.

Exemple de configuration `.cmake-format.yaml` :

```yaml
# .cmake-format.yaml
format:
  line_width: 100
  tab_size: 4
  use_tabchars: false
  max_subgroups_hwrap: 2
  max_pargs_hwrap: 6
  dangle_parens: true
  dangle_align: prefix
```

### `cmake-lint` — Linting des fichiers CMake

```yaml
- repo: https://github.com/cheshirekow/cmake-format-precommit
  rev: v0.6.13
  hooks:
    - id: cmake-lint
```

`cmake-lint` détecte les mauvaises pratiques CMake : utilisation de commandes dépréciées, variables non définies, arguments manquants, conventions de nommage non respectées. C'est l'équivalent de `clang-tidy` pour CMake.

**Temps d'exécution** : très rapide (< 1 seconde). Les fichiers CMake sont petits comparés aux sources C++.

## Catégorie 4 : Détection de problèmes transversaux

### `codespell` — Fautes de frappe dans le code et les commentaires

```yaml
- repo: https://github.com/codespell-project/codespell
  rev: v2.3.0
  hooks:
    - id: codespell
      args: ['--skip=build,third_party,*.pb.cc,*.pb.h',
             '--ignore-words-list=dout,ot,crate']
```

`codespell` détecte les fautes d'orthographe courantes dans le code source, les commentaires et la documentation. Il ne s'agit pas d'un correcteur orthographique complet mais d'un dictionnaire de typos fréquentes (par exemple `acount` → `account`, `occured` → `occurred`, `recieve` → `receive`).

**Pertinence en C++** : les commentaires et les noms de variables contiennent souvent des fautes non détectées par le compilateur. Un `// Retreive the connexion from the pool` n'empêche pas la compilation mais dégrade la lisibilité et le professionnalisme du code. Les typos dans les messages de log sont particulièrement gênantes car elles se retrouvent visibles par les utilisateurs ou les opérateurs en production.

**L'argument `--ignore-words-list`** permet d'exclure les faux positifs — les identifiants techniques que `codespell` confond avec des erreurs. Alimentez cette liste progressivement en fonction des faux positifs rencontrés.

**L'argument `--skip`** exclut les répertoires et les fichiers générés (code Protobuf, third_party) qui contiennent des termes hors de votre contrôle.

### `detect-private-key` — Clés privées et secrets

```yaml
- repo: https://github.com/pre-commit/pre-commit-hooks
  rev: v4.6.0
  hooks:
    - id: detect-private-key
```

Bloque le commit si un fichier contient une clé privée SSH, PGP ou un certificat. En C++, les projets qui impliquent du TLS (clients HTTPS, serveurs gRPC — chapitre 22) manipulent parfois des certificats de test. Ce hook empêche de les versionner accidentellement.

### `check-executables-have-shebangs`

```yaml
- repo: https://github.com/pre-commit/pre-commit-hooks
  rev: v4.6.0
  hooks:
    - id: check-executables-have-shebangs
```

Vérifie que les fichiers marqués exécutables (`chmod +x`) possèdent un shebang (`#!/usr/bin/env bash`, `#!/usr/bin/env python3`). Cela évite de versionner accidentellement un binaire compilé avec le bit exécutable, ou un script sans interpréteur défini.

## Catégorie 5 : Validation des messages de commit

### `commitizen` — Conventional Commits

```yaml
- repo: https://github.com/commitizen-tools/commitizen
  rev: v3.29.0
  hooks:
    - id: commitizen
      stages: [commit-msg]
```

Ce hook valide que le message de commit respecte le format Conventional Commits (section 47.1). Il s'exécute au stade `commit-msg` (et non `pre-commit`), c'est-à-dire après la saisie du message mais avant la création effective du commit.

**Messages acceptés** :

```
feat(parser): add TOML support           ✅  
fix(network): resolve timeout on ARM     ✅  
refactor(core)!: rename Config to Opts   ✅  (breaking change)  
docs: update installation guide          ✅  
```

**Messages rejetés** :

```
fixed stuff                              ❌  (pas de type)  
Feature: add parser                      ❌  (majuscule, mauvais format)  
WIP                                      ❌  (pas de type ni de description)  
```

> **Rappel** : pour que ce hook fonctionne, il faut avoir exécuté `pre-commit install --hook-type commit-msg` lors de l'installation (section 47.2.1).

**Flexibilité** : `commitizen` est configurable via un fichier `.cz.toml` ou une section `[tool.commitizen]` dans `pyproject.toml`. Vous pouvez personnaliser les types autorisés, les scopes valides, et la longueur maximale du sujet :

```toml
# .cz.toml
[tool.commitizen]
name = "cz_conventional_commits"  
version = "2.4.0"  
tag_format = "v$version"  
max_header_length = 72  
```

## Catégorie 6 : Hooks locaux personnalisés

Le framework `pre-commit` permet de définir des hooks locaux — des scripts spécifiques au projet qui ne proviennent d'aucun dépôt externe. C'est utile pour des vérifications propres à votre projet.

### Vérification d'inclusion de headers interdits

Certains projets interdisent des headers spécifiques (par exemple `<iostream>` dans du code performance-critical, ou `<windows.h>` dans du code portable) :

```yaml
- repo: local
  hooks:
    - id: no-iostream-in-lib
      name: "No <iostream> in library code"
      entry: bash -c 'grep -rln "#include <iostream>" --include="*.cpp" --include="*.h" src/lib/ && echo "❌ <iostream> interdit dans src/lib/. Utilisez spdlog." && exit 1 || exit 0'
      language: system
      pass_filenames: false
      files: '^src/lib/.*\.(cpp|h|hpp)$'
```

### Vérification des `#pragma once` vs include guards

Si votre projet impose `#pragma once` (ou au contraire des include guards classiques), un hook local peut le vérifier :

```yaml
- repo: local
  hooks:
    - id: pragma-once-check
      name: "Headers must use #pragma once"
      entry: bash -c 'for f in "$@"; do head -5 "$f" | grep -q "#pragma once" || { echo "❌ $f : #pragma once manquant"; exit 1; }; done'
      language: system
      files: '\.(h|hpp)$'
      exclude: '^third_party/'
```

### Vérification du copyright header

De nombreuses organisations exigent un en-tête de copyright dans chaque fichier source :

```yaml
- repo: local
  hooks:
    - id: copyright-header
      name: "Copyright header present"
      entry: bash -c 'for f in "$@"; do head -3 "$f" | grep -q "Copyright" || { echo "❌ $f : copyright header manquant"; exit 1; }; done'
      language: system
      files: '\.(cpp|hpp|h|cc|cxx)$'
      exclude: '^(third_party/|build/)'
```

### Anatomie d'un hook local

Les hooks locaux partagent la même structure que les hooks externes, avec quelques champs spécifiques :

```yaml
- repo: local
  hooks:
    - id: mon-hook              # Identifiant unique
      name: "Description lisible"  # Affiché à l'exécution
      entry: bash -c '...'       # Commande à exécuter
      language: system            # Interpréteur (system, python, node, etc.)
      files: '...'               # Regex des fichiers ciblés
      exclude: '...'             # Regex des fichiers exclus
      pass_filenames: true        # Passer les fichiers modifiés en argument
      stages: [pre-commit]       # Phase d'exécution
      always_run: false           # Exécuter même sans fichier correspondant
```

Le choix `language: system` exécute la commande directement via le shell système. C'est le plus simple pour des scripts Bash courts. Pour des vérifications plus complexes, écrivez un script Python ou Bash dédié dans `scripts/` et référencez-le :

```yaml
- repo: local
  hooks:
    - id: custom-check
      name: "Project-specific validation"
      entry: python scripts/check_project_conventions.py
      language: python
      files: '\.(cpp|hpp|h)$'
```

## Configuration recommandée par taille de projet

Tous les hooks ne sont pas nécessaires pour tous les projets. Voici des configurations graduées :

### Projet personnel ou prototype

Privilégiez la simplicité — les hooks d'hygiène et le formatage suffisent :

```
✅ trailing-whitespace
✅ end-of-file-fixer
✅ check-added-large-files
✅ clang-format
```

Temps d'exécution total : **< 1 seconde**.

### Projet d'équipe (3-10 développeurs)

Ajoutez l'analyse CMake, la validation des messages de commit et la détection de typos :

```
✅ trailing-whitespace
✅ end-of-file-fixer
✅ check-added-large-files
✅ check-merge-conflict
✅ check-yaml
✅ mixed-line-ending
✅ clang-format
✅ cmake-format
✅ codespell
✅ commitizen (commit-msg)
```

Temps d'exécution total : **1-3 secondes**.

### Projet professionnel (10+ développeurs, CI robuste)

Ajoutez `clang-tidy` (checks restreints), les hooks de sécurité et les vérifications personnalisées :

```
✅ trailing-whitespace
✅ end-of-file-fixer
✅ check-added-large-files
✅ check-merge-conflict
✅ check-yaml / check-json
✅ check-symlinks
✅ mixed-line-ending
✅ detect-private-key
✅ check-executables-have-shebangs
✅ clang-format
✅ clang-tidy (checks critiques)
✅ cmake-format
✅ cmake-lint
✅ codespell
✅ commitizen (commit-msg)
✅ hooks locaux (copyright, conventions)
```

Temps d'exécution total : **3-15 secondes** (dominé par `clang-tidy`).

## Résumé

Le choix des hooks suit une logique de couches concentriques. Le noyau — hygiène et `clang-format` — est non négociable et prend moins d'une seconde. Chaque couche supplémentaire ajoute de la protection au prix d'un temps d'exécution légèrement accru. La règle reste la même : si le total dépasse 15 secondes, allégez la configuration locale et déportez les vérifications lentes vers la CI.

La section suivante assemble ces hooks dans une configuration `.pre-commit-config.yaml` complète et opérationnelle, avec un focus sur l'intégration fine de `clang-format` et `clang-tidy`.
