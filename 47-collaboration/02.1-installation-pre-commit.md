# 47.2.1 — Installation du framework pre-commit

> **Chapitre 47 : Collaboration et Maintenance** · Section 47.2 : Pre-commit hooks  
> **Niveau** : Expert  
> **Prérequis** : Section 47.2 (Pre-commit hooks — concepts), Git installé et configuré, Python 3.9+ disponible

---

## Introduction

La section 47.2 a présenté le framework `pre-commit` et ses avantages par rapport aux hooks Git natifs. Cette sous-section couvre l'installation concrète sur Ubuntu, l'activation dans un projet C++, et les pratiques d'onboarding pour que chaque membre de l'équipe bénéficie des hooks dès son premier clone.

---

## Installation de `pre-commit` sur Ubuntu

Le framework `pre-commit` est un outil Python. Plusieurs méthodes d'installation sont possibles sur Ubuntu. Le choix dépend de votre gestion de l'environnement Python.

### Méthode recommandée : `pipx`

`pipx` installe les outils Python en ligne de commande dans des environnements virtuels isolés, sans polluer le Python système ni les virtualenvs de vos projets. C'est la méthode la plus propre pour un outil global comme `pre-commit` :

```bash
# Installer pipx si ce n'est pas déjà fait
sudo apt update  
sudo apt install pipx  
pipx ensurepath  

# Installer pre-commit via pipx
pipx install pre-commit

# Vérifier l'installation
pre-commit --version
# pre-commit 4.1.0 (ou version plus récente)
```

L'avantage de `pipx` est double : `pre-commit` est disponible globalement (pas besoin d'activer un virtualenv), et il est isolé du reste du système Python. Les mises à jour se font simplement avec `pipx upgrade pre-commit`.

### Méthode alternative : `pip` avec virtualenv utilisateur

Si `pipx` n'est pas disponible ou si vous préférez `pip` classique :

```bash
# Installation dans l'espace utilisateur (pas de sudo)
pip install --user pre-commit

# Vérifier que ~/.local/bin est dans le PATH
echo $PATH | grep -q "$HOME/.local/bin" || echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc  
source ~/.bashrc  

pre-commit --version
```

> **Attention** : évitez `sudo pip install pre-commit`. Installer des paquets Python avec `sudo pip` peut casser le gestionnaire de paquets système sur les versions récentes d'Ubuntu (24.04+), qui protègent l'environnement Python système par défaut (PEP 668).

### Méthode alternative : paquet APT

Ubuntu propose `pre-commit` dans ses dépôts, mais la version est souvent en retard de plusieurs releases :

```bash
sudo apt install pre-commit
```

Cette méthode est acceptable pour un usage rapide, mais la version disponible peut manquer de fonctionnalités récentes ou de corrections de bugs. Pour un projet professionnel, préférez `pipx` qui donne accès à la dernière version stable.

### Dans un DevContainer

Si votre projet utilise des DevContainers (section 2.4.3), l'installation de `pre-commit` doit être intégrée dans l'image Docker pour que chaque développeur dispose de l'outil sans intervention manuelle :

```dockerfile
# .devcontainer/Dockerfile (extrait)
FROM mcr.microsoft.com/devcontainers/cpp:ubuntu-24.04

# Installer pipx et pre-commit
RUN apt-get update && apt-get install -y pipx \
    && pipx install pre-commit \
    && pipx ensurepath

# Les outils C++ nécessaires aux hooks
RUN apt-get install -y clang-format-19 clang-tidy-19
```

Complété par un script post-création qui active les hooks automatiquement :

```jsonc
// .devcontainer/devcontainer.json (extrait)
{
    "postCreateCommand": "pre-commit install && pre-commit install --hook-type commit-msg"
}
```

Avec cette configuration, un développeur qui ouvre le projet dans un DevContainer a les hooks actifs immédiatement, sans aucune action manuelle.

## Activation des hooks dans un projet

### Initialisation : `pre-commit install`

Une fois `pre-commit` installé sur la machine, il faut l'activer pour chaque projet. Cette commande installe les hooks Git dans `.git/hooks/` :

```bash
cd /chemin/vers/mon-projet-cpp  
pre-commit install  
# pre-commit installed at .git/hooks/pre-commit
```

À partir de cet instant, chaque `git commit` dans ce dépôt déclenche automatiquement les hooks définis dans `.pre-commit-config.yaml`.

### Activer le hook `commit-msg`

Par défaut, `pre-commit install` n'installe que le hook `pre-commit`. Pour activer également la validation des messages de commit (Conventional Commits, section 47.1), il faut spécifier le type :

```bash
pre-commit install --hook-type commit-msg
# pre-commit installed at .git/hooks/commit-msg
```

On peut combiner les deux en une seule ligne dans la documentation ou le script d'onboarding :

```bash
pre-commit install && pre-commit install --hook-type commit-msg
```

### Activer le hook `pre-push` (optionnel)

Pour des vérifications plus lourdes avant le push (par exemple, lancer les tests unitaires rapides), un hook `pre-push` peut être ajouté :

```bash
pre-commit install --hook-type pre-push
```

> **Prudence** : un hook `pre-push` lent (compilation + tests) ralentit significativement le cycle push. Réservez-le aux vérifications qui prennent moins de 30 secondes, ou rendez-le optionnel.

### Vérifier ce qui est installé

Pour confirmer quels hooks sont actifs :

```bash
ls -la .git/hooks/
# -rwxr-xr-x  pre-commit      (géré par pre-commit)
# -rwxr-xr-x  commit-msg      (géré par pre-commit)
```

Les fichiers installés par `pre-commit` contiennent un shebang et un appel au framework. Ils remplacent les hooks par défaut de Git (les `.sample` présents à l'initialisation du dépôt).

## Le fichier `.pre-commit-config.yaml`

Le cœur de la configuration est le fichier `.pre-commit-config.yaml` à la racine du projet. Ce fichier **doit être versionné** dans Git — c'est lui qui garantit que tous les développeurs utilisent les mêmes hooks avec les mêmes versions.

Voici un squelette minimal pour un projet C++ :

```yaml
# .pre-commit-config.yaml
# Documentation : https://pre-commit.com

# Langue par défaut pour les hooks locaux
default_language_version:
  python: python3

# Appliquer les hooks uniquement aux fichiers stagés (comportement par défaut)
# Exclure les répertoires générés
exclude: '^(build/|third_party/|vendor/|cmake-build-.*/)' 

repos:
  # ── Hooks génériques (hygiène de base) ──
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.6.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-added-large-files
        args: ['--maxkb=500']
      - id: check-merge-conflict
      - id: check-yaml
      - id: check-json

  # ── Hooks C++ (détaillés en section 47.2.2 et 47.3) ──
  - repo: https://github.com/pocc/pre-commit-hooks
    rev: v1.3.5
    hooks:
      - id: clang-format
        args: ['--style=file']
      - id: clang-tidy
        args: ['--config-file=.clang-tidy']
```

Chaque entrée dans `repos` référence un dépôt Git contenant des définitions de hooks. Le champ `rev` fixe la version exacte utilisée — c'est un point crucial pour la reproductibilité. La configuration détaillée des hooks C++ est couverte en section 47.2.2 et 47.3.

### Structure d'une entrée de hook

Chaque hook peut être paramétré finement :

```yaml
- repo: https://github.com/pocc/pre-commit-hooks
  rev: v1.3.5
  hooks:
    - id: clang-format              # Identifiant du hook dans le dépôt
      name: "C++ formatting"         # Nom affiché à l'exécution (optionnel)
      args: ['--style=file']         # Arguments passés à l'outil
      files: '\.(cpp|hpp|h|cc|cxx)$' # Regex des fichiers ciblés
      exclude: '^third_party/'       # Regex des fichiers exclus
      stages: [pre-commit]           # À quel moment s'exécuter
      language_version: system       # Utiliser le binaire système
```

Les champs les plus utilisés :

| Champ | Rôle |
|---|---|
| `id` | Identifiant unique du hook dans le dépôt source |
| `args` | Arguments en ligne de commande passés à l'outil |
| `files` | Expression régulière pour filtrer les fichiers ciblés |
| `exclude` | Expression régulière pour exclure des fichiers |
| `stages` | Phase Git où le hook s'exécute (`pre-commit`, `commit-msg`, `pre-push`) |
| `pass_filenames` | Si `true` (défaut), les noms de fichiers stagés sont passés en argument |
| `always_run` | Si `true`, exécuter même si aucun fichier ne correspond au filtre |

## Premier lancement et cache

### Exécution initiale

Lors du premier commit après l'installation, `pre-commit` télécharge et prépare les environnements de chaque hook. Cette initialisation prend 10-30 secondes selon le nombre de hooks et la connexion réseau :

```bash
git commit -m "feat(core): add initial project structure"

# Premier lancement — installation des environnements
# [INFO] Installing environment for https://github.com/pre-commit/pre-commit-hooks
# [INFO] Once installed this environment will be reused.
# [INFO] Installing environment for https://github.com/pocc/pre-commit-hooks
# [INFO] Once installed this environment will be reused.
#
# trailing-whitespace .............................. Passed
# end-of-file-fixer ............................... Passed
# check-added-large-files ......................... Passed
# check-merge-conflict ............................ Passed
# clang-format .................................... Passed
# clang-tidy ...................................... Passed
```

### Cache des environnements

Les environnements téléchargés sont mis en cache dans `~/.cache/pre-commit/`. Les exécutions suivantes sont quasi instantanées car elles réutilisent ce cache. Le répertoire de cache peut être configuré via la variable d'environnement `PRE_COMMIT_HOME` :

```bash
# Déplacer le cache (utile si le home est sur un FS lent)
export PRE_COMMIT_HOME=/opt/pre-commit-cache
```

Pour nettoyer le cache (rarement nécessaire) :

```bash
pre-commit clean
```

### Pré-remplir le cache

Pour éviter l'attente du premier commit, on peut pré-installer les environnements de hooks dès le clone :

```bash
git clone https://github.com/mon-org/mon-projet.git  
cd mon-projet  
pre-commit install && pre-commit install --hook-type commit-msg  
pre-commit install-hooks    # Télécharge tous les environnements immédiatement  
```

C'est particulièrement utile dans les scripts d'onboarding ou les DevContainers.

## Exécution manuelle et CI

### Lancer les hooks manuellement

Il n'est pas nécessaire de commiter pour tester les hooks. `pre-commit run` permet de les déclencher à la demande :

```bash
# Exécuter tous les hooks sur les fichiers stagés
pre-commit run

# Exécuter tous les hooks sur TOUS les fichiers du projet
pre-commit run --all-files

# Exécuter un hook spécifique
pre-commit run clang-format

# Exécuter un hook spécifique sur tous les fichiers
pre-commit run clang-format --all-files

# Exécuter sur des fichiers précis
pre-commit run clang-format --files src/core/engine.cpp src/core/engine.h
```

La commande `pre-commit run --all-files` est essentielle lors de l'ajout de `pre-commit` à un projet existant : elle permet de formater l'ensemble du code en une seule passe, dans un commit dédié.

### Intégration dans la CI

Les pre-commit hooks doivent être dupliqués en CI pour garantir que le code qui atteint la branche principale respecte les standards, même si un développeur a contourné les hooks locaux. L'exécution en CI se fait avec `pre-commit run --all-files` :

```yaml
# .gitlab-ci.yml (extrait)
pre-commit:
  stage: lint
  image: python:3.12-slim
  before_script:
    - pip install pre-commit
    - apt-get update && apt-get install -y clang-format-19 clang-tidy-19
  script:
    - pre-commit run --all-files
  cache:
    paths:
      - ~/.cache/pre-commit/
```

```yaml
# .github/workflows/lint.yml (extrait)
name: Pre-commit checks  
on: [pull_request]  

jobs:
  pre-commit:
    runs-on: ubuntu-24.04
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'
      - name: Install tools
        run: |
          sudo apt-get update
          sudo apt-get install -y clang-format-19 clang-tidy-19
      - uses: pre-commit/action@v3.0.1
```

L'action officielle `pre-commit/action` pour GitHub Actions gère automatiquement l'installation, le cache et l'exécution. C'est la méthode la plus simple pour GitHub.

> **Point clé** : la CI exécute `--all-files` (pas seulement les fichiers modifiés). Cela détecte les cas où un changement de `.clang-format` ou `.clang-tidy` rend du code existant non conforme. Localement, les hooks ne vérifient que les fichiers stagés pour rester rapides.

## Mise à jour des hooks

Les hooks référencent des versions précises via le champ `rev`. Au fil du temps, ces versions deviennent obsolètes. `pre-commit` fournit une commande dédiée pour les mettre à jour :

```bash
# Mettre à jour tous les hooks vers leur dernière version taguée
pre-commit autoupdate
```

Cette commande parcourt chaque entrée `repos` du fichier `.pre-commit-config.yaml` et remplace le `rev` par le dernier tag disponible :

```yaml
# Avant autoupdate
- repo: https://github.com/pre-commit/pre-commit-hooks
  rev: v4.5.0    # ancienne version

# Après autoupdate
- repo: https://github.com/pre-commit/pre-commit-hooks
  rev: v4.6.0    # dernière version
```

### Fréquence recommandée

Une mise à jour mensuelle est un bon compromis. Intégrez-la dans votre routine de maintenance :

```bash
# Script de mise à jour périodique
pre-commit autoupdate  
pre-commit run --all-files    # Vérifier que rien ne casse  
git add .pre-commit-config.yaml  
git commit -m "chore: update pre-commit hooks"  
```

Testez toujours avec `--all-files` après une mise à jour. Un nouveau `rev` de `clang-format` pourrait appliquer des changements de style mineurs, ou un `clang-tidy` mis à jour pourrait activer de nouveaux checks. Mieux vaut découvrir ces changements dans un commit dédié que dispersés dans les commits de feature.

### Figer les versions pour la reproductibilité

Le champ `rev` accepte un tag ou un hash de commit. Pour une reproductibilité maximale, les tags sont préférables (ils sont lisibles et correspondent aux releases stables). Évitez de pointer sur une branche (`rev: main`) — cela rendrait les builds non reproductibles.

## Onboarding d'un nouveau développeur

L'adoption des pre-commit hooks ne fonctionne que si chaque membre de l'équipe les a installés. Voici la procédure recommandée, à documenter dans le `CONTRIBUTING.md` (section 47.1) et dans le `README.md` du projet :

### Procédure standard

```bash
# 1. Cloner le projet
git clone https://github.com/mon-org/mon-projet.git  
cd mon-projet  

# 2. Installer pre-commit (une fois par machine)
pipx install pre-commit

# 3. Activer les hooks pour ce projet
pre-commit install  
pre-commit install --hook-type commit-msg  

# 4. (Optionnel) Pré-installer les environnements de hooks
pre-commit install-hooks
```

### Script d'initialisation

Pour réduire les étapes à un minimum, fournissez un script `scripts/setup-dev.sh` :

```bash
#!/usr/bin/env bash
# scripts/setup-dev.sh — Configuration de l'environnement de développement
set -euo pipefail

echo "=== Configuration de l'environnement de développement ==="

# Vérifier que pre-commit est installé
if ! command -v pre-commit &> /dev/null; then
    echo "pre-commit n'est pas installé."
    echo "Installation via pipx..."
    if command -v pipx &> /dev/null; then
        pipx install pre-commit
    else
        echo "pipx non trouvé. Installez-le avec : sudo apt install pipx"
        echo "Puis relancez ce script."
        exit 1
    fi
fi

echo "→ pre-commit $(pre-commit --version)"

# Installer les hooks Git
echo "Installation des hooks Git..."  
pre-commit install  
pre-commit install --hook-type commit-msg  

# Pré-installer les environnements
echo "Pré-installation des environnements de hooks..."  
pre-commit install-hooks  

# Vérifier les outils C++ requis
echo ""  
echo "Vérification des outils C++..."  
for tool in clang-format clang-tidy cmake-format; do  
    if command -v "$tool" &> /dev/null; then
        echo "  ✅ $tool trouvé"
    else
        echo "  ⚠️  $tool non trouvé — certains hooks ne fonctionneront pas"
    fi
done

echo ""  
echo "=== Environnement prêt ==="  
echo "Les hooks pre-commit sont actifs. Bon développement !"  
```

Ce script est idempotent : le relancer ne casse rien. Documentez-le dans le README :

```markdown
## Démarrage rapide

```bash
git clone https://github.com/mon-org/mon-projet.git  
cd mon-projet  
./scripts/setup-dev.sh
```
```

### Rappel automatique pour les développeurs sans hooks

Une astuce utile : ajouter un commentaire dans le `CMakeLists.txt` principal qui vérifie la présence des hooks et affiche un avertissement lors de la configuration CMake si ils ne sont pas installés :

```cmake
# CMakeLists.txt (à la fin, après project())
if(EXISTS "${CMAKE_SOURCE_DIR}/.pre-commit-config.yaml"
   AND NOT EXISTS "${CMAKE_SOURCE_DIR}/.git/hooks/pre-commit")
    message(WARNING
        "Pre-commit hooks are not installed!\n"
        "Run: pre-commit install && pre-commit install --hook-type commit-msg\n"
        "Or:  ./scripts/setup-dev.sh"
    )
endif()
```

Ainsi, même un développeur qui a oublié le script d'initialisation verra un rappel à chaque configuration CMake.

## Désinstallation et désactivation

### Désactiver temporairement (un seul commit)

```bash
git commit --no-verify -m "wip: work in progress, hooks skipped"
```

### Désinstaller les hooks du projet

```bash
pre-commit uninstall  
pre-commit uninstall --hook-type commit-msg  
```

Les hooks sont retirés de `.git/hooks/`. Le fichier `.pre-commit-config.yaml` reste intact (il est versionné). Un simple `pre-commit install` les réactive.

### Désinstaller `pre-commit` du système

```bash
# Via pipx
pipx uninstall pre-commit

# Via pip
pip uninstall pre-commit
```

## Résumé des commandes essentielles

| Commande | Rôle |
|---|---|
| `pipx install pre-commit` | Installer le framework |
| `pre-commit install` | Activer le hook `pre-commit` dans le dépôt courant |
| `pre-commit install --hook-type commit-msg` | Activer le hook de validation du message de commit |
| `pre-commit install-hooks` | Pré-télécharger les environnements de hooks |
| `pre-commit run` | Exécuter les hooks sur les fichiers stagés |
| `pre-commit run --all-files` | Exécuter les hooks sur tout le projet |
| `pre-commit run <hook-id>` | Exécuter un hook spécifique |
| `pre-commit autoupdate` | Mettre à jour les `rev` vers les dernières versions |
| `pre-commit clean` | Nettoyer le cache des environnements |
| `pre-commit uninstall` | Désactiver les hooks du dépôt courant |
