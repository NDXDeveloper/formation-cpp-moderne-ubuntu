## Introduction

La section 42.4 a présenté le compare-and-swap comme la brique élémentaire de la programmation lock-free, et la section 42.4.1 l'a mis en œuvre dans des piles et des files. Cette section approfondit le CAS lui-même : son fonctionnement au niveau matériel, ses variantes (single-word, double-word, multi-word émulé), et les patterns avancés qui en découlent — accumulateurs lock-free, flags composites, et descriptors d'opérations.

Comprendre le CAS en profondeur permet de concevoir des algorithmes lock-free sur mesure pour des problèmes qui ne se réduisent pas aux structures classiques de la littérature. C'est aussi la clé pour analyser la performance des algorithmes existants et identifier pourquoi un CAS échoue trop souvent ou pourquoi une boucle de réessai ne converge pas.

## Le CAS au niveau matériel

### x86-64 : `LOCK CMPXCHG` et `LOCK CMPXCHG16B`

Sur x86-64, le CAS est implémenté par l'instruction `CMPXCHG` (Compare and Exchange) préfixée par `LOCK`. Le préfixe `LOCK` garantit que l'instruction s'exécute atomiquement sur le bus mémoire — aucun autre cœur ne peut lire ou écrire l'emplacement mémoire ciblé pendant toute la durée de l'instruction.

Pour un CAS sur 32 ou 64 bits :

```nasm
; CMPXCHG [mem], reg
; Compare EAX/RAX avec [mem].
; Si égaux : [mem] = reg, ZF = 1 (succès)
; Si différents : EAX/RAX = [mem], ZF = 0 (échec)

lock cmpxchg [rdi], rsi    ; CAS 64 bits
```

Pour un CAS sur 128 bits (nécessaire pour les tagged pointers — section 42.4.1) :

```nasm
; CMPXCHG16B [mem]
; Compare RDX:RAX avec [mem] (128 bits).
; Si égaux : [mem] = RCX:RBX, ZF = 1
; Si différents : RDX:RAX = [mem], ZF = 0

lock cmpxchg16b [rdi]      ; CAS 128 bits — requiert -mcx16
```

`CMPXCHG16B` opère sur 128 bits en utilisant deux paires de registres : `RDX:RAX` pour la valeur attendue et `RCX:RBX` pour la valeur souhaitée. Cette instruction est disponible sur tous les processeurs x86-64 depuis 2005 (AMD Athlon 64 révision E, Intel depuis le Pentium 4 model F), mais le compilateur ne l'émet que si le flag `-mcx16` est passé explicitement.

### ARM aarch64 : LL/SC et LSE

ARM utilise historiquement un mécanisme différent : la paire **Load-Linked / Store-Conditional** (LL/SC), appelée `LDXR`/`STXR` (Load/Store Exclusive Register) sur aarch64.

```nasm
; Boucle CAS simulée par LL/SC
retry:
    ldxr  x0, [x1]         ; Load-Exclusive : lit [x1] et "réserve" la ligne de cache
    cmp   x0, x2            ; Compare avec la valeur attendue
    b.ne  fail              ; Si différent, échec
    stxr  w3, x4, [x1]     ; Store-Conditional : écrit x4 dans [x1] SI la réservation tient
    cbnz  w3, retry         ; Si stxr a échoué (w3 != 0), réessaie
    ; ... succès ...
fail:
    ; ... échec, x0 contient la valeur courante ...
```

Le `STXR` peut échouer pour des raisons qui n'ont rien à voir avec la valeur comparée : un autre cœur a accédé à la même ligne de cache, une interruption a eu lieu entre le `LDXR` et le `STXR`, le scheduler a migré le thread vers un autre cœur, etc. C'est la source des **spurious failures** de `compare_exchange_weak` mentionnés en section 42.4.

À partir d'ARMv8.1-a, les **Large System Extensions (LSE)** ajoutent des instructions CAS natives (`CAS`, `CASP`, `SWP`, `LDADD`…) qui ne souffrent pas de spurious failures :

```nasm
; CAS natif ARMv8.1-a (LSE)
cas   x0, x4, [x1]         ; Si [x1] == x0, alors [x1] = x4 ; sinon x0 = [x1]
```

Les processeurs ARM récents (Apple M1+, AWS Graviton 2+, Ampere Altra) supportent tous LSE. Sur ces processeurs, `compare_exchange_strong` et `compare_exchange_weak` ont des performances identiques, car le CAS matériel ne produit plus de spurious failures. GCC 15 et Clang 20 émettent automatiquement les instructions LSE quand `-march=armv8.1-a` ou supérieur est spécifié.

### Coût du CAS

Le CAS n'est pas une opération gratuite. Son coût dépend de l'état de la ligne de cache ciblée :

| Scénario | Latence approximative (x86-64) |
|----------|--------------------------------|
| CAS sur une ligne de cache déjà détenue en exclusivité (état M ou E dans MESI) | ~10–20 cycles |
| CAS sur une ligne de cache partagée entre cœurs du même socket (état S) | ~30–60 cycles |
| CAS sur une ligne de cache détenue par un cœur d'un autre socket (NUMA) | ~100–300 cycles |

Sous forte contention (beaucoup de threads CAS-ant la même adresse), le coût explose car chaque CAS déclenche un transfert de ligne de cache entre cœurs via le protocole de cohérence. C'est pourquoi les structures lock-free performantes minimisent le nombre de CAS par opération et répartissent les accès sur plusieurs lignes de cache.

## Anatomie d'un CAS-loop performant

Le CAS-loop (section 42.4) est le pattern universel de la programmation lock-free. Sa performance dépend de plusieurs facteurs qui méritent d'être examinés en détail.

### Structure canonique

```cpp
void cas_loop_operation(std::atomic<State>& target) {
    State expected = target.load(std::memory_order_acquire);  // (1) Load initial

    do {
        State desired = compute_new_state(expected);           // (2) Calcul

        if (!is_valid(desired)) return;                        // (3) Abandon conditionnel

    } while (!target.compare_exchange_weak(                    // (4) CAS
                  expected,           // Mis à jour en cas d'échec
                  desired,
                  std::memory_order_release,
                  std::memory_order_relaxed));
    // (5) Succès — l'opération est terminée
}
```

Cinq points clés :

**(1) Load initial.** Un seul load avant la boucle. Le load est `acquire` car il sert de point de synchronisation pour lire l'état courant de manière cohérente. Si l'opération ne synchronise pas d'autres données, `relaxed` suffit.

**(2) Calcul du nouvel état.** Cette étape doit être aussi légère que possible. Tout temps passé ici augmente la probabilité qu'un autre thread modifie `target` entre le load et le CAS, ce qui fera échouer le CAS et forcera un réessai. Pour les structures complexes, cela implique de préparer les allocations et les calculs lourds *en dehors* de la boucle quand c'est possible.

**(3) Abandon conditionnel.** Certaines opérations doivent pouvoir abandonner si l'état courant ne le permet pas (file pleine, compteur au maximum, condition de garde non remplie). L'abandon se fait avant le CAS, pas après un échec.

**(4) CAS weak.** Dans une boucle, `weak` est toujours préférable à `strong` — un spurious failure est absorbé par le réessai sans surcoût. Le memory order de succès est typiquement `release` (pour publier le nouvel état). Le memory order d'échec est `relaxed` (aucune synchronisation nécessaire, `expected` est juste mis à jour localement).

**(5) Mise à jour automatique de `expected`.** C'est l'élégance du CAS C++ : en cas d'échec, `expected` contient la valeur courante fraîche. Pas besoin d'un load explicite avant le réessai — la boucle recommence directement à l'étape (2) avec la valeur à jour.

### Backoff exponentiel

Sous forte contention, un CAS-loop naïf peut dégénérer en « tempête de CAS » : tous les threads lisent la même valeur, calculent le même résultat (ou des résultats différents), tentent le CAS simultanément, un seul réussit, et tous les autres recommencent immédiatement — ce qui provoque encore plus de contention sur le bus mémoire.

Le **backoff exponentiel** atténue ce problème en introduisant un délai croissant entre les réessais :

```cpp
#include <atomic>
#include <thread>
#include <cstdint>

class ExponentialBackoff {
    static constexpr int MinDelay = 1;
    static constexpr int MaxDelay = 1024;

    int current_delay_ = MinDelay;

public:
    void operator()() {
        // Attente active calibrée — pas de yield au noyau
        for (int i = 0; i < current_delay_; ++i) {
            #if defined(__x86_64__)
                asm volatile("pause");     // Réduit la consommation et la contention
            #elif defined(__aarch64__)
                asm volatile("isb");       // Instruction Synchronization Barrier (léger)
            #endif
        }

        // Double le délai pour le prochain échec, plafonné
        current_delay_ = std::min(current_delay_ * 2, MaxDelay);
    }

    void reset() {
        current_delay_ = MinDelay;
    }
};

// Utilisation dans un CAS-loop
void lock_free_update(std::atomic<int>& target) {
    int expected = target.load(std::memory_order_relaxed);
    ExponentialBackoff backoff;

    while (!target.compare_exchange_weak(
                expected,
                expected + 1,
                std::memory_order_relaxed,
                std::memory_order_relaxed)) {
        backoff();  // Attend de plus en plus longtemps entre les réessais
    }
}
```

Le backoff réduit la pression sur le bus mémoire en espaçant les tentatives, ce qui laisse le temps au thread gagnant de terminer son opération et à la ligne de cache de se stabiliser. L'instruction `pause` sur x86-64 est spécialement conçue pour ce cas : elle signale au processeur qu'il est dans une boucle de spin-wait, ce qui réduit la consommation d'énergie et évite les pénalités de pipeline liées à la spéculation.

## Double-word CAS (DWCAS)

Le double-word CAS (DWCAS) opère sur deux mots machine adjacents en une seule opération atomique. Sur x86-64, cela signifie un CAS sur 128 bits via `CMPXCHG16B`. Sur ARM aarch64, `CASP` (CAS Pair) remplit le même rôle.

### Motivation

Le CAS simple (64 bits) ne peut modifier atomiquement qu'un seul mot. Pour les structures qui nécessitent de modifier deux champs simultanément (pointeur + compteur, pointeur + flag, deux pointeurs adjacents), le DWCAS est indispensable.

L'exemple le plus courant est le **tagged pointer** vu en section 42.4.1 : un pointeur (64 bits) couplé à un compteur de version (64 bits) pour résoudre le problème ABA.

### Implémentation en C++

C++ ne fournit pas de primitive DWCAS explicite. L'approche standard est d'utiliser `std::atomic<T>` sur une structure de 128 bits :

```cpp
#include <atomic>
#include <cstdint>
#include <cassert>

struct DoubleWord {
    uint64_t lo;
    uint64_t hi;

    bool operator==(const DoubleWord&) const = default;
};

static_assert(sizeof(DoubleWord) == 16);

// Vérifie que le CAS 128 bits est lock-free sur cette plateforme
static_assert(std::atomic<DoubleWord>::is_always_lock_free,
              "DWCAS requis — compiler avec -mcx16 sur x86-64");

class TaggedCounter {
    // Stocke un pointeur (lo) et un compteur de version (hi)
    std::atomic<DoubleWord> state_{{0, 0}};

public:
    // Lecture atomique du pointeur et du compteur
    DoubleWord load() const {
        return state_.load(std::memory_order_acquire);
    }

    // CAS atomique sur les deux mots simultanément
    bool try_update(DoubleWord& expected, DoubleWord desired) {
        return state_.compare_exchange_weak(
            expected, desired,
            std::memory_order_acq_rel,
            std::memory_order_acquire);
    }
};
```

### Alignement

Pour que `CMPXCHG16B` fonctionne correctement, l'opérande mémoire doit être aligné sur 16 octets. Le standard C++ garantit que `std::atomic<T>` est suffisamment aligné, mais si vous stockez la structure dans un conteneur ou l'allouez manuellement, vérifiez l'alignement :

```cpp
struct alignas(16) AlignedDoubleWord {
    uint64_t lo;
    uint64_t hi;
};

// Allocation dynamique alignée
auto* ptr = new (std::align_val_t{16}) AlignedDoubleWord{0, 0};
```

### Limites du DWCAS

Le DWCAS opère sur deux mots **adjacents en mémoire**. Il ne peut pas modifier atomiquement deux emplacements mémoire arbitraires. Si vous avez besoin de modifier atomiquement deux pointeurs situés à des adresses différentes (par exemple `head` et `tail` d'une file), le DWCAS ne suffit pas — il faut recourir à des techniques de multi-word CAS émulé (voir plus bas) ou repenser le design.

## Accumulateurs lock-free

Un accumulateur lock-free est une généralisation du compteur atomique : au lieu d'un simple incrément, il applique une fonction arbitraire à une valeur partagée. Le pattern CAS-loop est son implémentation naturelle.

### Accumulateur générique

```cpp
#include <atomic>
#include <concepts>
#include <type_traits>

template <typename T>
    requires std::is_trivially_copyable_v<T>
class LockFreeAccumulator {
    std::atomic<T> value_;

public:
    explicit LockFreeAccumulator(T initial) : value_(initial) {}

    // Applique une transformation arbitraire de manière lock-free
    // Retourne l'ancienne valeur
    template <std::invocable<T> F>
    T transform(F&& func) {
        T expected = value_.load(std::memory_order_relaxed);
        T desired;

        do {
            desired = func(expected);
        } while (!value_.compare_exchange_weak(
                      expected, desired,
                      std::memory_order_acq_rel,
                      std::memory_order_relaxed));

        return expected;  // Ancienne valeur
    }

    T load() const {
        return value_.load(std::memory_order_acquire);
    }
};
```

Cet accumulateur est remarquablement flexible. Quelques exemples d'utilisation :

```cpp
LockFreeAccumulator<int> counter(0);

// Incrément simple (équivalent à fetch_add)
counter.transform([](int n) { return n + 1; });

// Maximum atomique : met à jour seulement si la nouvelle valeur est plus grande
counter.transform([new_val](int current) {
    return std::max(current, new_val);
});

// Multiplication atomique
counter.transform([factor](int n) { return n * factor; });
```

### Application : statistiques lock-free

Un cas d'usage fréquent en production est l'agrégation de statistiques par plusieurs threads. Un accumulateur de min/max lock-free évite le mutex sur le chemin critique :

```cpp
#include <atomic>
#include <limits>

class MinMaxTracker {
    std::atomic<int64_t> min_{std::numeric_limits<int64_t>::max()};
    std::atomic<int64_t> max_{std::numeric_limits<int64_t>::min()};

public:
    void observe(int64_t value) {
        // Mise à jour du minimum
        int64_t current_min = min_.load(std::memory_order_relaxed);
        while (value < current_min) {
            if (min_.compare_exchange_weak(
                        current_min, value,
                        std::memory_order_relaxed,
                        std::memory_order_relaxed)) {
                break;
            }
            // current_min a été mis à jour — on revérifie la condition
        }

        // Mise à jour du maximum
        int64_t current_max = max_.load(std::memory_order_relaxed);
        while (value > current_max) {
            if (max_.compare_exchange_weak(
                        current_max, value,
                        std::memory_order_relaxed,
                        std::memory_order_relaxed)) {
                break;
            }
        }
    }

    int64_t min() const { return min_.load(std::memory_order_acquire); }
    int64_t max() const { return max_.load(std::memory_order_acquire); }
};
```

Notez le pattern de la boucle : la condition `value < current_min` est revérifiée après chaque échec du CAS, car `current_min` a pu être mis à jour par un autre thread avec une valeur encore plus petite. Si `value >= current_min` après un échec, l'opération s'arrête — un autre thread a déjà enregistré une valeur plus petite ou égale, ce qui rend notre mise à jour inutile. C'est un CAS-loop avec **abandon conditionnel**, plus efficace qu'un CAS-loop inconditionnel car il ne réessaie que si l'opération a encore un sens.

### Application : bitset lock-free (flags composites)

Les flags atomiques composites sont un pattern courant dans les systèmes concurrents — par exemple, un ensemble de capacités annoncées par plusieurs threads :

```cpp
#include <atomic>
#include <cstdint>

class AtomicBitSet {
    std::atomic<uint64_t> bits_{0};

public:
    // Active un bit de manière lock-free (OR atomique)
    void set(int position) {
        // fetch_or est wait-free — pas besoin de CAS-loop
        bits_.fetch_or(uint64_t{1} << position, std::memory_order_acq_rel);
    }

    // Désactive un bit de manière lock-free (AND atomique)
    void clear(int position) {
        bits_.fetch_and(~(uint64_t{1} << position), std::memory_order_acq_rel);
    }

    // Active un bit et retourne true s'il était précédemment inactif
    // (test-and-set atomique)
    bool test_and_set(int position) {
        uint64_t mask = uint64_t{1} << position;
        uint64_t old = bits_.fetch_or(mask, std::memory_order_acq_rel);
        return (old & mask) == 0;  // true si le bit n'était pas set avant
    }

    // Active un ensemble de bits conditionnellement :
    // "Active ces flags SEULEMENT SI ces autres flags sont déjà actifs"
    // Retourne true si l'opération a réussi
    bool set_if_all_set(uint64_t required, uint64_t to_set) {
        uint64_t expected = bits_.load(std::memory_order_relaxed);
        do {
            if ((expected & required) != required) {
                return false;  // Prérequis non remplis
            }
        } while (!bits_.compare_exchange_weak(
                      expected,
                      expected | to_set,
                      std::memory_order_acq_rel,
                      std::memory_order_relaxed));
        return true;
    }

    bool test(int position) const {
        return (bits_.load(std::memory_order_acquire) &
                (uint64_t{1} << position)) != 0;
    }

    uint64_t load() const {
        return bits_.load(std::memory_order_acquire);
    }
};
```

Les opérations simples (`set`, `clear`, `test_and_set`) utilisent `fetch_or` / `fetch_and` qui sont wait-free — pas besoin de CAS-loop. L'opération conditionnelle (`set_if_all_set`) nécessite un CAS-loop car elle doit lire l'état, vérifier une condition, et écrire le nouvel état en une seule étape atomique.

## Multi-word CAS émulé : le pattern Descriptor

Le CAS matériel (y compris le DWCAS) est limité à un ou deux mots adjacents. Certains algorithmes nécessitent de modifier atomiquement plusieurs emplacements mémoire non adjacents — par exemple, insérer un nœud dans une liste doublement chaînée lock-free (qui exige de modifier `prev->next` et `node->prev` atomiquement), ou transférer un élément d'une structure à une autre (retirer d'une file et insérer dans une autre en une seule opération atomique).

Le **multi-word CAS** (MCAS, aussi appelé k-CAS) est impossible au niveau matériel, mais il peut être **émulé en logiciel** à l'aide du pattern *descriptor*.

### Principe du pattern Descriptor

L'idée centrale est de remplacer la modification directe de plusieurs emplacements par une **publication d'intention** :

1. Le thread crée un **descriptor** — un objet qui décrit l'ensemble des modifications à effectuer (adresses, valeurs attendues, nouvelles valeurs).
2. Le thread utilise un CAS simple pour installer le descriptor dans le premier emplacement à modifier, signalant aux autres threads qu'une opération multi-mot est en cours.
3. Le thread (ou tout autre thread qui rencontre le descriptor — *helping*) applique les modifications restantes une par une.
4. Le thread marque le descriptor comme terminé.

Tout thread qui accède à un emplacement contenant un descriptor plutôt qu'une valeur normale sait qu'une opération est en cours. Il aide à la compléter avant de poursuivre sa propre opération. C'est le même principe de helping vu dans l'algorithme de Michael & Scott (section 42.4.1), généralisé à des opérations arbitraires.

### Implémentation simplifiée d'un 2-CAS

Voici une implémentation simplifiée d'un double-CAS non adjacent — un CAS qui modifie atomiquement deux emplacements mémoire quelconques. Cette version illustre le principe du descriptor sans viser l'exhaustivité d'une implémentation en production :

```cpp
#include <atomic>
#include <cstdint>

// Descriptor : décrit une opération 2-CAS
struct Descriptor {
    enum class Status : int { Active, Succeeded, Failed };

    std::atomic<Status> status{Status::Active};

    // Premier emplacement
    std::atomic<uintptr_t>* addr1;
    uintptr_t expected1;
    uintptr_t desired1;

    // Second emplacement
    std::atomic<uintptr_t>* addr2;
    uintptr_t expected2;
    uintptr_t desired2;
};

// Pour distinguer un pointeur vers un descriptor d'une valeur normale,
// on utilise le bit de poids faible (les pointeurs alloués sont alignés
// sur au moins 2 octets, donc le bit 0 est toujours 0)
constexpr uintptr_t DescriptorBit = 1;

bool is_descriptor(uintptr_t val) {
    return (val & DescriptorBit) != 0;
}

Descriptor* to_descriptor(uintptr_t val) {
    return reinterpret_cast<Descriptor*>(val & ~DescriptorBit);
}

uintptr_t from_descriptor(Descriptor* desc) {
    return reinterpret_cast<uintptr_t>(desc) | DescriptorBit;
}

// Aide à compléter une opération décrite par un descriptor
void help(Descriptor* desc) {
    if (desc->status.load(std::memory_order_acquire) != Descriptor::Status::Active) {
        return;  // Déjà terminé
    }

    // Tente d'installer le descriptor dans le second emplacement
    uintptr_t exp2 = desc->expected2;
    bool cas2_ok = desc->addr2->compare_exchange_strong(
        exp2, from_descriptor(desc),
        std::memory_order_acq_rel,
        std::memory_order_relaxed);

    if (cas2_ok || exp2 == from_descriptor(desc)) {
        // Le second emplacement contient notre descriptor → succès
        desc->status.store(Descriptor::Status::Succeeded,
                           std::memory_order_release);
    } else {
        // Le second CAS a échoué → l'opération entière échoue
        desc->status.store(Descriptor::Status::Failed,
                           std::memory_order_release);
    }

    // Nettoyage : remplace les descriptors par les valeurs finales
    auto final_status = desc->status.load(std::memory_order_acquire);
    if (final_status == Descriptor::Status::Succeeded) {
        // Remplace le descriptor par desired dans les deux emplacements
        uintptr_t d = from_descriptor(desc);
        desc->addr1->compare_exchange_strong(
            d, desc->desired1, std::memory_order_release,
            std::memory_order_relaxed);
        d = from_descriptor(desc);
        desc->addr2->compare_exchange_strong(
            d, desc->desired2, std::memory_order_release,
            std::memory_order_relaxed);
    } else {
        // Restaure les valeurs originales
        uintptr_t d = from_descriptor(desc);
        desc->addr1->compare_exchange_strong(
            d, desc->expected1, std::memory_order_release,
            std::memory_order_relaxed);
        d = from_descriptor(desc);
        desc->addr2->compare_exchange_strong(
            d, desc->expected2, std::memory_order_release,
            std::memory_order_relaxed);
    }
}

// Opération 2-CAS principale
bool double_cas(std::atomic<uintptr_t>* addr1, uintptr_t exp1, uintptr_t des1,
                std::atomic<uintptr_t>* addr2, uintptr_t exp2, uintptr_t des2) {

    // Alloue un descriptor (en production, utiliser un pool)
    auto* desc = new Descriptor{
        .status = Descriptor::Status::Active,
        .addr1 = addr1, .expected1 = exp1, .desired1 = des1,
        .addr2 = addr2, .expected2 = exp2, .desired2 = des2
    };

    // Phase 1 : installe le descriptor dans le premier emplacement
    uintptr_t e1 = exp1;
    if (!addr1->compare_exchange_strong(
                e1, from_descriptor(desc),
                std::memory_order_acq_rel,
                std::memory_order_relaxed)) {
        // Le premier CAS a échoué
        // Si e1 est un autre descriptor, on pourrait l'aider d'abord (omis ici)
        delete desc;
        return false;
    }

    // Phase 2 : complète l'opération (ou un autre thread le fera via help)
    help(desc);

    bool succeeded =
        desc->status.load(std::memory_order_acquire)
            == Descriptor::Status::Succeeded;

    // En production : le descriptor doit être libéré de manière safe
    // (hazard pointers, epoch-based reclamation, etc.)
    delete desc;  // ⚠️ Simplifié — unsafe en multi-thread

    return succeeded;
}
```

### Analyse du pattern Descriptor

Le pattern Descriptor est la technique la plus puissante de la boîte à outils lock-free, mais aussi la plus complexe :

**Avantages.** Il permet de réaliser des opérations atomiques sur un nombre arbitraire d'emplacements mémoire. Le mécanisme de helping garantit la progression lock-free : si un thread est préempté au milieu d'une opération, tout autre thread qui rencontre le descriptor le complète avant de poursuivre.

**Inconvénients.** Chaque lecture d'un emplacement mémoire doit vérifier si la valeur est un descriptor ou une valeur réelle, ce qui ajoute un branchement sur chaque accès. L'allocation et la libération des descriptors posent le même problème de réclamation mémoire que les nœuds lock-free (hazard pointers, EBR nécessaires). Le marquage du bit de poids faible pour distinguer descriptors et valeurs nécessite que les valeurs normales aient le bit 0 à zéro (ce qui est vrai pour les pointeurs alignés mais pas pour les entiers arbitraires). Enfin, le nombre de CAS par opération est proportionnel au nombre d'emplacements modifiés, ce qui augmente le coût et la probabilité de contention.

**En production.** Le multi-word CAS émulé est rarement implémenté à la main. Des bibliothèques comme libcds et des travaux de recherche (Harris et al., 2002 ; Guerraoui et al., 2020) fournissent des implémentations optimisées et vérifiées. Le pattern Descriptor est présenté ici pour sa valeur pédagogique : comprendre comment il fonctionne permet de comprendre la structure interne des algorithmes lock-free les plus avancés.

## CAS et l'illusion de l'atomicité composable

Un piège fréquent chez les développeurs qui découvrent le CAS est de croire que deux opérations CAS successives forment ensemble une opération atomique. Ce n'est **jamais** le cas :

```cpp
std::atomic<int> balance_a{100};  
std::atomic<int> balance_b{50};  

// ❌ PAS atomique — un thread peut observer un état incohérent
// entre les deux CAS
void transfer_broken(int amount) {
    int a = balance_a.load(std::memory_order_relaxed);
    balance_a.compare_exchange_strong(
        a, a - amount, std::memory_order_acq_rel,
        std::memory_order_relaxed);  // (1) Débite A

    int b = balance_b.load(std::memory_order_relaxed);
    balance_b.compare_exchange_strong(
        b, b + amount, std::memory_order_acq_rel,
        std::memory_order_relaxed);  // (2) Crédite B

    // Entre (1) et (2), la somme totale est incorrecte.
    // Pire : si (2) échoue, l'argent a disparu.
}
```

Entre les CAS (1) et (2), un observateur externe voit un état où `amount` a été retiré de A mais pas encore ajouté à B. Ce sont exactement les problèmes que les transactions de base de données résolvent avec le mécanisme ACID.

Les solutions correctes sont :

- **Un seul CAS** sur un état composite (structure contenant les deux balances — mais elles doivent être adjacentes et ≤ 128 bits).  
- **Le pattern Descriptor** (2-CAS émulé) pour des emplacements non adjacents.  
- **Un mutex** — parfois la solution la plus simple et la plus correcte.  
- **Des designs qui évitent le problème** : repenser l'architecture pour qu'une seule donnée atomique suffise (journal d'opérations, CRDT, etc.).

## Read-Modify-Write : au-delà du CAS

`std::atomic` fournit plusieurs opérations read-modify-write (RMW) qui sont des alternatives spécialisées au CAS-loop. Quand une de ces opérations correspond au besoin, elle est toujours préférable au CAS-loop car elle est plus concise, souvent wait-free au niveau matériel, et moins sujette à la contention.

### Opérations RMW disponibles

| Opération | Instruction x86-64 | Garantie | Utilisation typique |
|-----------|-------------------|----------|---------------------|
| `fetch_add(n)` | `LOCK XADD` | Wait-free | Compteurs, indices |
| `fetch_sub(n)` | `LOCK XADD` (avec négation) | Wait-free | Comptage de références |
| `fetch_or(mask)` | `LOCK OR` | Wait-free | Activation de flags |
| `fetch_and(mask)` | `LOCK AND` | Wait-free | Désactivation de flags |
| `fetch_xor(mask)` | `LOCK XOR` | Wait-free | Toggle de flags |
| `exchange(val)` | `LOCK XCHG` | Wait-free | Remplacement, spin-locks |
| `compare_exchange_*` | `LOCK CMPXCHG` | Lock-free | Transformations arbitraires |

Les opérations `fetch_*` sont **wait-free** : elles s'exécutent en un nombre borné de cycles quelles que soient les actions des autres threads. Le CAS est **lock-free** : il peut échouer et nécessiter un réessai, mais la progression globale est garantie.

### Règle de choix

```
La transformation est-elle une addition/soustraction ?
    → fetch_add / fetch_sub

La transformation est-elle un OR/AND/XOR sur des bits ?
    → fetch_or / fetch_and / fetch_xor

La transformation est-elle un remplacement inconditionnel ?
    → exchange

La transformation nécessite-t-elle de lire l'ancienne valeur,  
de vérifier une condition, et de calculer une nouvelle valeur ?  
    → CAS-loop (compare_exchange_weak)
```

### `fetch_max` et `fetch_min` (C++26)

C++26 ajoute `fetch_max` et `fetch_min` à `std::atomic`, qui implémentent atomiquement un maximum ou minimum. Avant C++26, ces opérations nécessitaient un CAS-loop (comme dans le `MinMaxTracker` présenté plus haut). Avec C++26 :

```cpp
// C++26 — wait-free si le matériel le supporte
std::atomic<int64_t> global_max{0};  
global_max.fetch_max(new_value, std::memory_order_relaxed);  
```

> ⚠️ **Support** : `fetch_max` / `fetch_min` sont ratifiés dans le standard C++26. Le support dans GCC 15 et Clang 20 est en cours d'implémentation — vérifiez la disponibilité pour votre version.

## Performances : mesurer le coût du CAS

Le coût d'un CAS dépend fortement du scénario de contention. Voici comment mesurer le comportement réel avec Google Benchmark (chapitre 35) :

```cpp
#include <atomic>
#include <benchmark/benchmark.h>

std::atomic<int64_t> shared_counter{0};

// Benchmark : CAS-loop sous contention croissante
static void BM_CAS_Increment(benchmark::State& state) {
    for (auto _ : state) {
        int64_t expected = shared_counter.load(std::memory_order_relaxed);
        while (!shared_counter.compare_exchange_weak(
                    expected, expected + 1,
                    std::memory_order_relaxed,
                    std::memory_order_relaxed)) {}
    }
}
BENCHMARK(BM_CAS_Increment)->Threads(1)->Threads(2)->Threads(4)
                            ->Threads(8)->Threads(16)->Threads(32);

// Benchmark de référence : fetch_add (wait-free)
static void BM_FetchAdd_Increment(benchmark::State& state) {
    for (auto _ : state) {
        shared_counter.fetch_add(1, std::memory_order_relaxed);
    }
}
BENCHMARK(BM_FetchAdd_Increment)->Threads(1)->Threads(2)->Threads(4)
                                 ->Threads(8)->Threads(16)->Threads(32);
```

Les résultats typiques sur un serveur à 32 cœurs montrent :

- **1 thread** : CAS-loop et `fetch_add` ont des performances similaires (~2–5 ns/op).  
- **4 threads** : `fetch_add` reste rapide (~10–15 ns/op), le CAS-loop ralentit (~20–40 ns/op) à cause des réessais.  
- **32 threads** : `fetch_add` ralentit modérément (~30–50 ns/op), le CAS-loop peut exploser (~200–500 ns/op) si le calcul dans la boucle est non trivial, car le taux d'échec du CAS approche les 100%.

Ces chiffres varient considérablement selon l'architecture (Intel vs AMD vs ARM), la topologie NUMA, et la nature du calcul dans la boucle. La seule vérité absolue est celle du benchmark sur votre matériel cible.

## Résumé

Le compare-and-swap est à la programmation lock-free ce que le pointeur est au C : une primitive simple qui requiert une discipline rigoureuse pour être utilisée correctement. Les points essentiels à retenir :

- Le CAS matériel est `LOCK CMPXCHG` sur x86-64 et `CAS`/`LDXR+STXR` sur ARM. Son coût dépend de l'état de la ligne de cache — de ~10 cycles sans contention à ~300 cycles en NUMA cross-socket.  
- Le CAS-loop (lire → calculer → CAS → réessayer) est le pattern universel. Utilisez `compare_exchange_weak` dans les boucles, `strong` hors boucles. Ajoutez un backoff exponentiel sous forte contention.  
- Le double-word CAS (`CMPXCHG16B`, `-mcx16`) permet d'opérer atomiquement sur 128 bits. C'est la base des tagged pointers (anti-ABA) et de toute structure nécessitant la modification atomique de deux champs adjacents.  
- Le multi-word CAS émulé via le pattern Descriptor permet de modifier atomiquement des emplacements mémoire non adjacents, au prix d'une complexité significative et d'un surcoût par accès.  
- Deux CAS successifs ne forment **jamais** une opération atomique composée. Utilisez un CAS unique sur un état composite, un Descriptor, ou un mutex.  
- Préférez les opérations RMW spécialisées (`fetch_add`, `fetch_or`, `exchange`) au CAS-loop quand elles couvrent le besoin — elles sont souvent wait-free et toujours plus performantes.  
- C++26 ajoute `fetch_max` et `fetch_min`, comblant deux lacunes qui nécessitaient un CAS-loop jusqu'ici.  
- Mesurez toujours avec Google Benchmark sous contention réaliste. Le coût théorique du CAS et son coût mesuré peuvent différer d'un ordre de grandeur selon le scénario.
