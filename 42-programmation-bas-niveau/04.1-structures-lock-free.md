## Introduction

La section précédente a posé les fondations conceptuelles de la programmation lock-free : le CAS-loop, le problème ABA, les garanties de progression, et les ordres mémoire. Cette section passe à la pratique avec l'implémentation détaillée de structures de données lock-free, en commençant par les plus simples (compteur, pile) pour progresser vers les plus élaborées (file MPMC), en abordant à chaque étape le problème critique de la gestion mémoire des nœuds retirés.

Chaque structure est présentée dans une démarche itérative : une version naïve d'abord, puis l'identification de ses défauts, et enfin une version corrigée. Cette approche reflète la réalité du développement lock-free, où la première implémentation qui « semble correcte » contient presque toujours un bug subtil.

> ⚠️ **Avertissement** : les implémentations de cette section sont pédagogiques. Elles illustrent les patterns fondamentaux et les pièges à éviter. Pour du code en production, utilisez des bibliothèques éprouvées comme Folly, Boost.Lockfree, libcds ou moodycamel::ConcurrentQueue, qui ont été vérifiées formellement et testées à grande échelle.

## Compteur lock-free : le cas trivial

Le compteur atomique est la structure lock-free la plus simple. Il sert de point de départ pour vérifier la compréhension du CAS-loop avant d'attaquer les structures à pointeurs.

### Version `fetch_add` (wait-free)

La manière la plus directe d'implémenter un compteur atomique est d'utiliser `fetch_add`, qui est **wait-free** — chaque appel termine en un nombre borné d'étapes, sans boucle de réessai :

```cpp
#include <atomic>
#include <cstdint>

class WaitFreeCounter {
    std::atomic<int64_t> count_{0};

public:
    void increment() {
        count_.fetch_add(1, std::memory_order_relaxed);
    }

    void decrement() {
        count_.fetch_sub(1, std::memory_order_relaxed);
    }

    int64_t get() const {
        return count_.load(std::memory_order_relaxed);
    }
};
```

`memory_order_relaxed` suffit car le compteur ne synchronise pas d'autres données — la seule chose qui importe est l'atomicité du read-modify-write.

### Version CAS-loop (lock-free)

Pour illustrer le pattern CAS-loop, voici le même compteur implémenté manuellement. Cette version est **lock-free** (pas wait-free) car le CAS peut échouer et nécessiter un réessai :

```cpp
class LockFreeCounter {
    std::atomic<int64_t> count_{0};

public:
    void increment() {
        int64_t expected = count_.load(std::memory_order_relaxed);
        while (!count_.compare_exchange_weak(
                    expected,
                    expected + 1,
                    std::memory_order_relaxed,
                    std::memory_order_relaxed)) {
            // expected a été mis à jour automatiquement
            // On boucle avec la valeur fraîche
        }
    }

    int64_t get() const {
        return count_.load(std::memory_order_relaxed);
    }
};
```

Dans la pratique, utilisez toujours `fetch_add` pour un simple compteur — le CAS-loop est strictement inférieur en performance et en garantie de progression. Mais ce pattern deviendra indispensable dès que la transformation de l'ancien état vers le nouveau sera plus complexe qu'une addition.

### Version CAS-loop avec transformation arbitraire

Le vrai intérêt du CAS-loop apparaît quand on doit appliquer une transformation conditionnelle ou non-triviale :

```cpp
class ClampedCounter {
    std::atomic<int64_t> count_{0};

    static constexpr int64_t MaxValue = 1000;

public:
    // Incrémente seulement si le compteur n'a pas atteint le maximum
    // Retourne true si l'incrémentation a réussi
    bool try_increment() {
        int64_t expected = count_.load(std::memory_order_relaxed);
        do {
            if (expected >= MaxValue) {
                return false;  // Limite atteinte — pas de réessai
            }
        } while (!count_.compare_exchange_weak(
                      expected,
                      expected + 1,
                      std::memory_order_relaxed,
                      std::memory_order_relaxed));
        return true;
    }
};
```

Ce pattern — « lire, vérifier une condition, calculer la nouvelle valeur, CAS » — est le socle de toutes les structures lock-free qui suivent.

## Pile lock-free (Lock-free Stack)

La pile est la structure lock-free la plus classique. Elle est relativement simple à implémenter car les deux opérations (push et pop) modifient le même point d'accès : le sommet de la pile (`head`).

### Conception

Une pile lock-free est une liste chaînée dont la tête est un pointeur atomique. Push insère un nœud en tête, pop retire le nœud en tête. Les deux opérations utilisent un CAS sur le pointeur `head` :

```
Push(D) :
    Avant :  head → [A] → [B] → [C] → nullptr
    Après :  head → [D] → [A] → [B] → [C] → nullptr

Pop() → retourne D :
    Avant :  head → [D] → [A] → [B] → [C] → nullptr
    Après :  head → [A] → [B] → [C] → nullptr
```

### Implémentation naïve

```cpp
#include <atomic>
#include <optional>
#include <memory>

template <typename T>  
class NaiveLockFreeStack {  
    struct Node {
        T data;
        Node* next;

        // Constructeur template pour le perfect forwarding
        template <typename U>
        Node(U&& val, Node* n)
            : data(std::forward<U>(val)), next(n) {}
    };

    std::atomic<Node*> head_{nullptr};

public:
    void push(const T& value) {
        Node* new_node = new Node(value, nullptr);
        new_node->next = head_.load(std::memory_order_relaxed);

        // CAS-loop : tente de remplacer head par new_node
        while (!head_.compare_exchange_weak(
                    new_node->next,      // expected (mis à jour en cas d'échec)
                    new_node,            // desired
                    std::memory_order_release,
                    std::memory_order_relaxed)) {
            // Échec : un autre thread a modifié head
            // new_node->next a été mis à jour avec le nouveau head
            // On réessaie — le chaînage est automatiquement correct
        }
    }

    std::optional<T> pop() {
        Node* expected = head_.load(std::memory_order_acquire);

        while (expected != nullptr) {
            // Tente de remplacer head par head->next
            if (head_.compare_exchange_weak(
                        expected,             // expected (mis à jour en cas d'échec)
                        expected->next,       // desired : le nœud suivant
                        std::memory_order_acquire,
                        std::memory_order_relaxed)) {
                // Succès : expected pointe vers le nœud retiré
                T value = std::move(expected->data);
                delete expected;  // ⚠️ DANGER — voir ci-dessous
                return value;
            }
            // Échec : expected a été mis à jour, on réessaie
        }

        return std::nullopt;  // Pile vide
    }

    ~NaiveLockFreeStack() {
        while (pop().has_value()) {}
    }
};
```

### Analyse du memory ordering

Le choix des ordres mémoire dans cette pile mérite d'être explicité :

**`push` — release sur le CAS de succès.** Le release garantit que l'initialisation du nœud (`data` et `next`) est visible par tout thread qui lira `head` via un acquire. Sans release, un thread faisant un pop pourrait voir un nœud partiellement construit.

**`pop` — acquire sur le load initial et le CAS de succès.** L'acquire en lecture garantit que le thread voit les écritures effectuées par le thread qui a poussé le nœud (le release du push). Lire `expected->next` et `expected->data` après un acquire load de `head` garantit la cohérence de ces champs.

**Relaxed en cas d'échec du CAS.** Quand le CAS échoue, aucune modification n'a été effectuée — il suffit de relire la valeur courante (ce que `compare_exchange_weak` fait automatiquement en mettant à jour `expected`). Aucune synchronisation n'est nécessaire pour le cas d'échec.

### Les trois bugs de cette implémentation

Cette implémentation « naïve » contient trois bugs fondamentaux que toute structure lock-free à base de pointeurs doit résoudre :

**Bug 1 : Use-after-free dans `pop()`.** Le `delete expected` dans `pop` est dangereux. Entre le moment où le CAS réussit (retirant le nœud de la liste) et le moment où `delete` est appelé, un autre thread pourrait être en train de lire `expected->next` dans sa propre boucle CAS. Ce thread accéderait alors à de la mémoire libérée.

Scénario concret :

1. Thread A et Thread B appellent `pop()` simultanément.
2. Les deux lisent `head == &X` et `X.next == &Y`.
3. Thread A réussit son CAS : `head` passe de `&X` à `&Y`.
4. Thread A exécute `delete &X`.
5. Thread B, qui avait déjà lu `expected == &X`, tente son CAS avec `expected->next`. Mais `expected` a été libéré — **use-after-free**.

**Bug 2 : Problème ABA.** Si un nœud est retiré puis réinséré (ou si un nouveau nœud est alloué à la même adresse que l'ancien — ce qui est courant avec les allocateurs standards), le CAS peut réussir alors que la structure de la pile a changé (voir section 42.4 pour l'illustration détaillée).

**Bug 3 : Exception safety.** Si le constructeur de copie/mouvement de `T` lance une exception dans `T value = std::move(expected->data)`, le nœud est perdu — il a été retiré de la pile (CAS réussi) mais sa donnée n'a pas été transférée. Ce problème n'a pas de solution simple dans le cas général et constitue l'une des raisons pour lesquelles les structures lock-free stockent souvent des `shared_ptr<T>` plutôt que des `T` directement.

## Pile lock-free avec tagged pointer

Pour résoudre les bugs 1 et 2, la technique la plus directe est le **tagged pointer** combiné avec une stratégie de gestion mémoire. Commençons par résoudre le problème ABA avec un compteur de version.

### Tagged pointer sur x86-64

Sur x86-64, les adresses virtuelles en espace utilisateur n'utilisent que 48 bits (ou 57 avec la 5-level paging, mais les bits hauts sont encore disponibles en pratique). On peut encoder un compteur de version dans les bits inutilisés. Cependant, l'approche la plus portable consiste à utiliser une structure de 128 bits avec un CAS 128 bits (`CMPXCHG16B`) :

```cpp
#include <atomic>
#include <cstdint>
#include <optional>

template <typename T>  
class TaggedLockFreeStack {  
    struct Node {
        T data;
        Node* next;

        template <typename U>
        Node(U&& val, Node* n)
            : data(std::forward<U>(val)), next(n) {}
    };

    // Pointeur taggé : pointeur + compteur de version
    struct TaggedPtr {
        Node* ptr = nullptr;
        uint64_t tag = 0;

        bool operator==(const TaggedPtr&) const = default;
    };

    // Doit être lock-free pour être utile
    // Requiert -mcx16 sur x86-64 pour CMPXCHG16B
    std::atomic<TaggedPtr> head_{TaggedPtr{nullptr, 0}};

    static_assert(std::atomic<TaggedPtr>::is_always_lock_free,
                  "TaggedPtr doit etre lock-free (compiler avec -mcx16)");

public:
    void push(const T& value) {
        Node* new_node = new Node(value, nullptr);
        TaggedPtr old_head = head_.load(std::memory_order_relaxed);
        TaggedPtr new_head;

        do {
            new_node->next = old_head.ptr;
            new_head = {new_node, old_head.tag + 1};
        } while (!head_.compare_exchange_weak(
                      old_head, new_head,
                      std::memory_order_release,
                      std::memory_order_relaxed));
    }

    std::optional<T> pop() {
        TaggedPtr old_head = head_.load(std::memory_order_acquire);
        TaggedPtr new_head;

        while (old_head.ptr != nullptr) {
            new_head = {old_head.ptr->next, old_head.tag + 1};

            if (head_.compare_exchange_weak(
                        old_head, new_head,
                        std::memory_order_acquire,
                        std::memory_order_relaxed)) {
                T value = std::move(old_head.ptr->data);
                // ⚠️ La libération du nœud reste problématique
                // (use-after-free possible — voir section suivante)
                delete old_head.ptr;
                return value;
            }
            // Échec : old_head a été mis à jour automatiquement
        }

        return std::nullopt;
    }

    ~TaggedLockFreeStack() {
        TaggedPtr current = head_.load(std::memory_order_relaxed);
        while (current.ptr != nullptr) {
            Node* next = current.ptr->next;
            delete current.ptr;
            current.ptr = next;
        }
    }
};
```

Le compteur `tag` est incrémenté à chaque modification de `head`. Même si un nœud est retiré puis réinséré à la même adresse, le tag sera différent — le CAS échouera et le thread réessaiera avec l'état frais. Le problème ABA est résolu.

Mais le use-after-free (bug 1) persiste : `old_head.ptr->next` est lu *avant* le CAS, et un autre thread pourrait avoir libéré ce nœud entre le load de `head` et la lecture de `->next`. Pour résoudre ce problème, il faut une stratégie de **réclamation mémoire différée**.

## Gestion mémoire : le problème central du lock-free

Le problème de la libération mémoire est le défi le plus difficile de la programmation lock-free. Dans un algorithme à verrous, c'est simple : le thread qui détient le verrou peut libérer un nœud en toute sécurité, car aucun autre thread ne peut y accéder. En lock-free, il n'y a pas de verrou — n'importe quel thread peut être en train de lire n'importe quel nœud à tout moment.

La question fondamentale est : **quand est-il sûr de libérer un nœud qui a été retiré de la structure ?**

### Approche 1 : Compteur de lecteurs actifs

L'idée la plus simple est de compter le nombre de threads qui sont actuellement « à l'intérieur » d'une opération sur la structure. Un nœud retiré n'est libéré que lorsque plus aucun thread ne peut le référencer.

```cpp
#include <atomic>
#include <optional>
#include <vector>
#include <functional>

template <typename T>  
class SafeLockFreeStack {  
    struct Node {
        T data;
        Node* next;

        template <typename U>
        Node(U&& val, Node* n)
            : data(std::forward<U>(val)), next(n) {}
    };

    std::atomic<Node*> head_{nullptr};

    // Compteur de threads actuellement dans pop()
    std::atomic<int> threads_in_pop_{0};

    // Liste des nœuds en attente de suppression
    std::atomic<Node*> to_be_deleted_{nullptr};

    // Tente de libérer les nœuds en attente
    void try_reclaim(Node* old_head) {
        if (threads_in_pop_.load(std::memory_order_acquire) == 1) {
            // Nous sommes le seul thread dans pop()
            // On peut libérer les nœuds en attente

            // Récupère atomiquement la liste des nœuds en attente
            Node* pending = to_be_deleted_.exchange(
                nullptr, std::memory_order_acquire);

            // Revérifie que nous sommes toujours seuls
            if (threads_in_pop_.fetch_sub(1, std::memory_order_release) == 1) {
                // Toujours seul — on libère tout
                delete_nodes(pending);
            } else if (pending != nullptr) {
                // Un autre thread est arrivé — on remet les nœuds en attente
                chain_pending(pending);
            }

            // Libère le nœud qu'on vient de retirer
            delete old_head;
        } else {
            // D'autres threads sont dans pop()
            // On ajoute le nœud à la liste d'attente
            chain_node(old_head);
            threads_in_pop_.fetch_sub(1, std::memory_order_release);
        }
    }

    void chain_node(Node* node) {
        node->next = to_be_deleted_.load(std::memory_order_relaxed);
        while (!to_be_deleted_.compare_exchange_weak(
                    node->next, node,
                    std::memory_order_release,
                    std::memory_order_relaxed)) {}
    }

    void chain_pending(Node* nodes) {
        Node* last = nodes;
        while (last->next != nullptr) {
            last = last->next;
        }
        last->next = to_be_deleted_.load(std::memory_order_relaxed);
        while (!to_be_deleted_.compare_exchange_weak(
                    last->next, nodes,
                    std::memory_order_release,
                    std::memory_order_relaxed)) {}
    }

    static void delete_nodes(Node* nodes) {
        while (nodes != nullptr) {
            Node* next = nodes->next;
            delete nodes;
            nodes = next;
        }
    }

public:
    void push(const T& value) {
        Node* new_node = new Node(value, nullptr);
        new_node->next = head_.load(std::memory_order_relaxed);
        while (!head_.compare_exchange_weak(
                    new_node->next, new_node,
                    std::memory_order_release,
                    std::memory_order_relaxed)) {}
    }

    std::optional<T> pop() {
        // Signale que nous sommes dans pop()
        threads_in_pop_.fetch_add(1, std::memory_order_acquire);

        Node* old_head = head_.load(std::memory_order_acquire);
        while (old_head != nullptr) {
            if (head_.compare_exchange_weak(
                        old_head, old_head->next,
                        std::memory_order_acquire,
                        std::memory_order_relaxed)) {
                T value = std::move(old_head->data);
                try_reclaim(old_head);
                return value;
            }
        }

        // Pile vide — on décrémente et on sort
        threads_in_pop_.fetch_sub(1, std::memory_order_release);
        return std::nullopt;
    }

    ~SafeLockFreeStack() {
        while (pop().has_value()) {}
        delete_nodes(to_be_deleted_.load(std::memory_order_relaxed));
    }
};
```

### Analyse de l'approche par compteur

Cette approche fonctionne correctement, mais elle a une faiblesse importante : si un thread reste dans `pop()` pendant longtemps (parce qu'il est préempté par le scheduler, par exemple), le compteur `threads_in_pop_` ne retombe jamais à zéro, et les nœuds s'accumulent dans `to_be_deleted_` sans jamais être libérés. Sous forte charge continue, cela peut provoquer une fuite mémoire effective.

C'est le compromis fondamental de cette technique : elle est simple à comprendre et correcte, mais elle ne garantit pas de borne sur la mémoire non réclamée. Pour des systèmes à longue durée de vie sous charge soutenue, les techniques plus sophistiquées (hazard pointers, epoch-based reclamation) sont nécessaires.

### Approche 2 : Hazard pointers (aperçu)

Les hazard pointers (Michael, 2004) résolvent le problème de la réclamation mémoire de manière plus fine. Chaque thread publie les pointeurs qu'il utilise activement dans un tableau partagé. Avant de libérer un nœud, le thread vérifie qu'il n'apparaît dans aucun hazard pointer d'aucun thread.

Le principe simplifié :

```cpp
// Chaque thread possède un ou plusieurs hazard pointers
thread_local Node* my_hazard_ptr = nullptr;

std::optional<T> pop() {
    Node* old_head;
    do {
        old_head = head_.load(std::memory_order_acquire);
        if (old_head == nullptr) return std::nullopt;

        // Publie le pointeur que nous sommes en train de lire
        my_hazard_ptr = old_head;

        // Revérifie que head n'a pas changé pendant la publication
        // (sinon, notre hazard pointer protège le mauvais nœud)
    } while (old_head != head_.load(std::memory_order_acquire));

    // À ce point, old_head est protégé par notre hazard pointer :
    // aucun autre thread ne le libérera tant que my_hazard_ptr == old_head

    Node* next = old_head->next;  // Sûr : old_head est protégé

    if (head_.compare_exchange_strong(
                old_head, next,
                std::memory_order_acquire,
                std::memory_order_relaxed)) {
        T value = std::move(old_head->data);

        // Retire la protection
        my_hazard_ptr = nullptr;

        // old_head est retiré de la pile mais peut-être encore
        // référencé par d'autres hazard pointers → mise en attente
        retire_node(old_head);

        return value;
    }

    my_hazard_ptr = nullptr;
    return std::nullopt;  // Simplifié — en réalité, on réessaierait
}

void retire_node(Node* node) {
    // Ajoute à la liste locale de nœuds retirés
    retired_list.push_back(node);

    // Périodiquement, scanne les hazard pointers de tous les threads
    // et libère les nœuds retirés qui ne sont plus protégés
    if (retired_list.size() > threshold) {
        scan_and_reclaim();
    }
}
```

L'implémentation complète des hazard pointers est substantielle (gestion du tableau partagé, amortissement du scan, thread registration/deregistration) et dépasse le cadre de cette section. Le comité C++ travaille sur `std::hazard_pointer` pour une future inclusion dans le standard.

### Approche 3 : Epoch-based reclamation (aperçu)

L'epoch-based reclamation divise le temps en époques (0, 1, 2). Chaque thread signale dans quelle époque il se trouve. Un nœud retiré pendant l'époque N ne peut être libéré que lorsque tous les threads ont progressé au-delà de l'époque N.

```
Époque 0 :  Thread A retire le nœud X → X mis en attente (époque 0)
Époque 1 :  Tous les threads ont quitté l'époque 0
Époque 2 :  Tous les threads ont quitté l'époque 1
            → Les nœuds retirés à l'époque 0 peuvent être libérés
```

L'EBR est plus simple à implémenter que les hazard pointers et offre un meilleur throughput agrégé, car le « coût » de la réclamation est amorti sur de nombreux nœuds. Son inconvénient est qu'un thread bloqué dans une époque empêche la réclamation de tous les nœuds retirés depuis cette époque — un thread bloqué longtemps peut causer une accumulation mémoire significative.

### Approche 4 : `std::shared_ptr` atomique

C++20 a introduit `std::atomic<std::shared_ptr<T>>`, qui offre une solution prête à l'emploi pour la gestion mémoire lock-free. Le compteur de références du `shared_ptr` gère automatiquement la durée de vie :

```cpp
#include <atomic>
#include <memory>
#include <optional>

template <typename T>  
class SharedPtrStack {  
    struct Node {
        T data;
        std::shared_ptr<Node> next;

        template <typename U>
        Node(U&& val, std::shared_ptr<Node> n)
            : data(std::forward<U>(val)), next(std::move(n)) {}
    };

    std::atomic<std::shared_ptr<Node>> head_{nullptr};

public:
    void push(const T& value) {
        auto new_node = std::make_shared<Node>(
            value, head_.load(std::memory_order_relaxed));

        while (!head_.compare_exchange_weak(
                    new_node->next, new_node,
                    std::memory_order_release,
                    std::memory_order_relaxed)) {}
    }

    std::optional<T> pop() {
        auto old_head = head_.load(std::memory_order_acquire);

        while (old_head != nullptr) {
            if (head_.compare_exchange_weak(
                        old_head, old_head->next,
                        std::memory_order_acquire,
                        std::memory_order_relaxed)) {
                return std::move(old_head->data);
                // old_head (shared_ptr) sera détruit automatiquement
                // quand plus personne ne le référence
            }
        }

        return std::nullopt;
    }
};
```

Cette version est considérablement plus simple et exempte de use-after-free par construction. L'inconvénient est la performance : `std::atomic<std::shared_ptr<T>>` utilise en interne un mécanisme de synchronisation (souvent un spinlock ou un compteur atomique 128 bits) qui ajoute un surcoût par rapport à un CAS simple sur un pointeur nu. Le problème ABA est également résolu implicitement, car le nœud n'est pas libéré tant qu'un `shared_ptr` le référence — sa réallocation à la même adresse est donc impossible.

> ⚠️ **Support** : `std::atomic<std::shared_ptr<T>>` est supporté depuis GCC 12 et Clang 16. Vérifiez que l'implémentation est lock-free sur votre plateforme — sur certaines, elle utilise un mutex interne.

## File lock-free (Lock-free Queue)

La file (FIFO) est plus complexe qu'une pile car elle possède deux points d'accès : la tête (`head`) pour les consommateurs (dequeue) et la queue (`tail`) pour les producteurs (enqueue). La coordination entre ces deux extrémités sans verrou est le défi principal.

### Contexte : SPSC, MPSC, MPMC

Le design d'une file lock-free dépend fortement du nombre de producteurs et de consommateurs :

- **SPSC** (Single Producer, Single Consumer) — Un seul thread enfile, un seul défile. C'est le cas le plus simple : pas de contention, pas besoin de CAS.  
- **MPSC** (Multiple Producers, Single Consumer) — Plusieurs threads enfilent, un seul défile. Plus simple que MPMC car le côté consommateur n'a pas de contention.  
- **MPMC** (Multiple Producers, Multiple Consumers) — Le cas général, le plus complexe.

### File SPSC : ring buffer sans CAS

La file SPSC est un cas particulier important, car elle ne nécessite pas de CAS — des loads et stores atomiques avec acquire/release suffisent. C'est la structure la plus rapide possible pour la communication entre deux threads :

```cpp
#include <atomic>
#include <array>
#include <optional>
#include <cstddef>

template <typename T, size_t Capacity>  
class SPSCQueue {  
    static_assert((Capacity & (Capacity - 1)) == 0,
                  "Capacity doit etre une puissance de 2");

    std::array<T, Capacity> buffer_;

    // Indices : seul le producteur écrit dans write_idx_,
    //           seul le consommateur écrit dans read_idx_
    alignas(64) std::atomic<size_t> write_idx_{0};  // Cache line séparée
    alignas(64) std::atomic<size_t> read_idx_{0};   // Cache line séparée

    static constexpr size_t Mask = Capacity - 1;

public:
    // Appelé uniquement par le thread producteur
    bool try_push(const T& value) {
        size_t write = write_idx_.load(std::memory_order_relaxed);
        size_t read  = read_idx_.load(std::memory_order_acquire);

        if (write - read >= Capacity) {
            return false;  // File pleine
        }

        buffer_[write & Mask] = value;

        // Release : garantit que l'écriture dans buffer_ est visible
        // avant que le consommateur ne voie le nouvel index
        write_idx_.store(write + 1, std::memory_order_release);
        return true;
    }

    // Appelé uniquement par le thread consommateur
    std::optional<T> try_pop() {
        size_t read  = read_idx_.load(std::memory_order_relaxed);
        size_t write = write_idx_.load(std::memory_order_acquire);

        if (read == write) {
            return std::nullopt;  // File vide
        }

        T value = buffer_[read & Mask];

        // Release : garantit que la lecture de buffer_ est terminée
        // avant que le producteur ne voie le nouvel index
        // (et ne réécrive dans le même slot)
        read_idx_.store(read + 1, std::memory_order_release);
        return value;
    }
};
```

Points clés de cette implémentation :

**Capacité en puissance de 2.** L'opération modulo est remplacée par un AND binaire (`& Mask`), qui est une seule instruction CPU. Cela élimine une division coûteuse à chaque opération.

**`alignas(64)` sur les indices.** Les deux indices sont placés sur des lignes de cache séparées (64 octets sur la plupart des architectures x86-64). Sans cet alignement, les deux indices tomberaient sur la même ligne de cache et provoqueraient du **false sharing** : chaque écriture par un thread invaliderait la ligne de cache de l'autre thread, même si les deux écrivent dans des variables différentes. Le false sharing peut diviser les performances par un facteur 10 ou plus.

**Pas de CAS.** Chaque index est écrit par un seul thread et lu par l'autre. Un simple store release / load acquire suffit. C'est ce qui rend la file SPSC si rapide — elle est effectivement **wait-free**.

**Indices monotones croissants.** Les indices ne sont jamais ramenés à zéro (pas de wraparound explicite). L'opération `& Mask` gère le wraparound implicitement. Cela simplifie la détection de file pleine/vide : `write - read == Capacity` signifie plein, `write == read` signifie vide. Les indices de type `size_t` déborderont après 2⁶⁴ opérations — soit environ 585 ans à 1 milliard d'opérations par seconde.

### File MPMC : algorithme de Michael & Scott

Pour le cas général multi-producteurs, multi-consommateurs, l'algorithme de référence est celui de Michael et Scott (1996). Il utilise une liste chaînée avec un nœud sentinelle (dummy node) pour séparer les opérations sur `head` et `tail`.

Le principe est le suivant :

- La file contient toujours au moins un nœud : le sentinelle. Une file « vide » est une file où `head` et `tail` pointent tous les deux vers le sentinelle.  
- **Enqueue** : alloue un nouveau nœud, attache-le après le nœud pointé par `tail`, puis avance `tail`.  
- **Dequeue** : avance `head` et retourne la donnée du nœud qui était juste après l'ancien `head` (l'ancien sentinelle).

```cpp
#include <atomic>
#include <optional>

template <typename T>  
class MSQueue {  
    struct Node {
        T data;
        std::atomic<Node*> next{nullptr};

        Node() = default;  // Pour le sentinelle

        template <typename U>
        explicit Node(U&& val) : data(std::forward<U>(val)) {}
    };

    // head et tail sur des lignes de cache séparées
    alignas(64) std::atomic<Node*> head_;
    alignas(64) std::atomic<Node*> tail_;

public:
    MSQueue() {
        Node* sentinel = new Node();  // Nœud sentinelle
        head_.store(sentinel, std::memory_order_relaxed);
        tail_.store(sentinel, std::memory_order_relaxed);
    }

    void enqueue(const T& value) {
        Node* new_node = new Node(value);

        while (true) {
            Node* tail = tail_.load(std::memory_order_acquire);
            Node* next = tail->next.load(std::memory_order_acquire);

            // Vérifie que tail est toujours le dernier nœud
            if (tail == tail_.load(std::memory_order_relaxed)) {
                if (next == nullptr) {
                    // tail->next est nullptr : tail est bien le dernier nœud
                    // Tente d'attacher new_node après tail
                    if (tail->next.compare_exchange_weak(
                                next, new_node,
                                std::memory_order_release,
                                std::memory_order_relaxed)) {
                        // Succès — avance tail (best effort)
                        tail_.compare_exchange_strong(
                            tail, new_node,
                            std::memory_order_release,
                            std::memory_order_relaxed);
                        return;
                    }
                } else {
                    // tail est en retard : un autre thread a attaché un nœud
                    // mais n'a pas encore avancé tail
                    // On l'aide en avançant tail nous-mêmes (helping)
                    tail_.compare_exchange_weak(
                        tail, next,
                        std::memory_order_release,
                        std::memory_order_relaxed);
                }
            }
        }
    }

    std::optional<T> dequeue() {
        while (true) {
            Node* head = head_.load(std::memory_order_acquire);
            Node* tail = tail_.load(std::memory_order_acquire);
            Node* next = head->next.load(std::memory_order_acquire);

            if (head == head_.load(std::memory_order_relaxed)) {
                if (head == tail) {
                    if (next == nullptr) {
                        return std::nullopt;  // File vide
                    }
                    // tail est en retard — on l'aide
                    tail_.compare_exchange_weak(
                        tail, next,
                        std::memory_order_release,
                        std::memory_order_relaxed);
                } else {
                    if (next == nullptr) continue;  // État transitoire

                    // Lit la donnée AVANT le CAS
                    // (après le CAS, un autre thread pourrait libérer next)
                    T value = next->data;

                    // Avance head au-delà du sentinelle actuel
                    if (head_.compare_exchange_weak(
                                head, next,
                                std::memory_order_acquire,
                                std::memory_order_relaxed)) {
                        // Succès : l'ancien sentinelle (head) peut être libéré
                        // ⚠️ Même problème de réclamation mémoire qu'avec la pile
                        delete head;
                        return value;
                    }
                }
            }
        }
    }

    ~MSQueue() {
        while (dequeue().has_value()) {}
        // Libère le sentinelle restant
        delete head_.load(std::memory_order_relaxed);
    }
};
```

### Le pattern « helping »

L'un des aspects les plus remarquables de l'algorithme de Michael & Scott est le mécanisme d'**aide mutuelle** (*helping*). Dans `enqueue`, l'opération se fait en deux temps : d'abord attacher le nouveau nœud (`tail->next = new_node`), puis avancer `tail`. Si un thread est préempté entre ces deux étapes, `tail` est en retard par rapport à la véritable fin de la file.

Les autres threads ne se bloquent pas — ils **aident** le thread en retard en avançant `tail` eux-mêmes. Cette aide est la clé de la garantie lock-free : même si un thread est bloqué indéfiniment après le premier CAS, les autres threads détectent l'incohérence (`tail->next != nullptr`) et la corrigent. Le système global progresse toujours.

Ce pattern de helping se retrouve dans toutes les structures lock-free non triviales. Il est la manifestation concrète de la garantie de progression : si un thread ne peut pas terminer son opération seul (parce qu'un autre thread est bloqué au milieu de la sienne), il aide l'autre thread à terminer avant de poursuivre sa propre opération.

## Bonnes pratiques pour les structures lock-free

### 1. Séparer les lignes de cache

Le false sharing est l'ennemi silencieux de la performance lock-free. Toute donnée atomique accédée fréquemment par des threads différents doit être placée sur sa propre ligne de cache :

```cpp
// ❌ head_ et tail_ sur la même ligne de cache → false sharing
struct BadLayout {
    std::atomic<Node*> head_;  // Offset 0
    std::atomic<Node*> tail_;  // Offset 8 — même ligne de cache
};

// ✅ Chacun sur sa propre ligne de cache
struct GoodLayout {
    alignas(64) std::atomic<Node*> head_;  // Ligne de cache A
    alignas(64) std::atomic<Node*> tail_;  // Ligne de cache B
};
```

C++17 fournit `std::hardware_destructive_interference_size` pour obtenir la taille de la ligne de cache de manière portable, mais sa valeur est une constante de compilation qui peut ne pas correspondre au matériel réel. `alignas(64)` est le choix pragmatique — 64 octets est la taille de ligne de cache sur la quasi-totalité des processeurs x86-64 et ARM modernes.

### 2. Lire la donnée avant le CAS de retrait

Dans une opération de retrait (pop, dequeue), la donnée du nœud doit être lue **avant** le CAS qui retire le nœud de la structure. Après le CAS, le nœud pourrait être libéré par un autre thread à tout moment :

```cpp
// ✅ CORRECT : lecture avant CAS
T value = old_head->data;  
if (head.compare_exchange_weak(old_head, old_head->next, ...)) {  
    return value;  // old_head peut être libéré par un autre thread
}

// ❌ BUG : lecture après CAS
if (head.compare_exchange_weak(old_head, old_head->next, ...)) {
    T value = old_head->data;  // Use-after-free possible !
    return value;
}
```

### 3. Valider les hypothèses avec des re-lectures

Dans l'algorithme de Michael & Scott, la valeur de `tail` est relue après le load initial pour vérifier qu'elle n'a pas changé. Ce pattern de « double-check » est courant dans les algorithmes lock-free : une lecture atomique donne un instantané qui peut devenir périmé immédiatement. Avant d'agir sur cet instantané, revérifier qu'il est toujours valide réduit les CAS inutiles.

### 4. Préférer `compare_exchange_weak` dans les boucles

Comme expliqué en section 42.4, `weak` est plus efficace que `strong` dans les boucles de réessai, car il n'a pas besoin de gérer les faux échecs en interne sur les architectures LL/SC. Réservez `strong` pour les CAS isolés (pas de boucle).

### 5. Envisager les allocateurs spécialisés

Les structures lock-free font beaucoup d'allocations (`new Node`) et de libérations (`delete node`). L'allocateur standard (`operator new`) utilise souvent un mutex interne, ce qui peut devenir un goulot d'étranglement et défaire l'intérêt du lock-free. Envisagez des allocateurs lock-free comme jemalloc, tcmalloc, ou mimalloc, ou utilisez un pool de nœuds préalloués.

## Comparaison des approches

| Structure | Complexité | Performance | Gestion mémoire | Cas d'usage |
|-----------|-----------|-------------|------------------|-------------|
| Compteur `fetch_add` | Triviale | Wait-free | N/A | Compteurs, IDs |
| Pile (CAS simple) | Faible | Lock-free | Problématique (ABA, use-after-free) | Pools d'objets |
| Pile (tagged ptr) | Modérée | Lock-free | ABA résolu, réclamation à traiter | Usage général |
| Pile (`shared_ptr`) | Faible | Lock-free* | Automatique | Prototypage, charge modérée |
| File SPSC (ring buffer) | Faible | Wait-free | Pas d'allocation | Communication 1→1 |
| File MPMC (Michael & Scott) | Élevée | Lock-free | Problématique | Communication N→N |

*\* Lock-free si `std::atomic<std::shared_ptr<T>>` est lock-free sur la plateforme — ce qui n'est pas garanti.*

## Résumé

Les structures lock-free reposent sur un petit nombre de patterns fondamentaux — le CAS-loop, le helping, le tagged pointer — combinés avec une gestion mémoire rigoureuse. Les points essentiels à retenir :

- La pile lock-free est le point d'entrée : un seul pointeur atomique (`head`), un CAS par opération. Elle illustre clairement le CAS-loop, le problème ABA et le use-after-free.  
- La file SPSC (ring buffer) est la structure lock-free la plus performante et la plus simple. Elle est wait-free, n'utilise pas de CAS, et n'a pas de problème de réclamation mémoire. Si votre scénario le permet (un seul producteur, un seul consommateur), c'est toujours le premier choix.  
- La file MPMC (Michael & Scott) introduit le pattern de helping : les threads aident les opérations en cours des autres threads pour garantir la progression globale.  
- La gestion mémoire des nœuds retirés est le défi central. Le compteur de lecteurs actifs est la solution la plus simple. Les hazard pointers et l'epoch-based reclamation sont nécessaires pour les systèmes à longue durée de vie. `std::atomic<std::shared_ptr<T>>` offre une alternative automatique mais avec un surcoût.  
- Le false sharing détruit la performance lock-free. Utilisez `alignas(64)` sur toutes les données atomiques accédées par des threads différents.  
- Utilisez des implémentations existantes en production. La file SPSC de Folly (`folly::ProducerConsumerQueue`), `moodycamel::ConcurrentQueue`, et `boost::lockfree::queue` ont été testées à l'échelle de milliards d'opérations.
