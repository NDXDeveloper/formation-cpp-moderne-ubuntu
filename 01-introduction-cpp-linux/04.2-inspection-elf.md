## Introduction

La section 1.4.1 a décrit la structure interne du format ELF — headers, sections, segments, tables de symboles, GOT, PLT. Tous ces composants sont des structures binaires, invisibles si l'on se contente d'exécuter le programme. Pour les voir, il faut des **outils d'inspection**.

Linux fournit une suite d'outils en ligne de commande, installés par défaut sur Ubuntu (paquet `binutils`), qui permettent de disséquer un fichier ELF sous tous les angles. Chaque outil a sa spécialité, et savoir lequel utiliser dans quelle situation est une compétence qui vous servira tout au long de votre carrière de développeur C++.

Cette section est résolument pratique. Nous reprenons le programme de référence de la section 1.4.1 et nous l'interrogeons avec chaque outil, en expliquant ce que révèle chaque commande et dans quel contexte elle est utile.

### Programme de référence

```cpp
// reference.cpp
#include <cstdio>

const char* message = "Formation C++ moderne";  
int compteur_global = 42;  
int tableau_zero[1000];  

void afficher() {
    printf("%s (compteur: %d)\n", message, compteur_global);
}

int main() {
    afficher();
    return 0;
}
```

Compilation en deux variantes — avec et sans informations de débogage :

```bash
g++ -std=c++23 -O0 -g -o reference_debug reference.cpp  
g++ -std=c++23 -O2 -o reference_release reference.cpp  
```

## `readelf` — L'outil de référence pour l'analyse ELF

`readelf` est l'outil le plus complet pour lire la structure d'un fichier ELF. Il affiche les données brutes des headers, tables et sections sans les interpréter ni les transformer. C'est l'outil à utiliser quand vous voulez **comprendre la structure** d'un binaire.

### `readelf -h` — L'en-tête ELF

```bash
readelf -h ./reference_debug
```

Sortie (extraits) :

```
ELF Header:
  Magic:   7f 45 4c 46 02 01 01 00 00 00 00 00 00 00 00 00
  Class:                             ELF64
  Data:                              2's complement, little endian
  Version:                           1 (current)
  OS/ABI:                            UNIX - System V
  Type:                              DYN (Position-Independent Executable)
  Machine:                           Advanced Micro Devices X86-64
  Version:                           0x1
  Entry point address:               0x1060
  Start of program headers:          64 (bytes into file)
  Start of section headers:          21048 (bytes into file)
  Flags:                             0x0
  Size of this header:               64 (bytes)
  Size of program headers:           56 (bytes)
  Number of program headers:         13
  Size of section headers:           64 (bytes)
  Number of section headers:         38
  Section header string table index: 37
```

**Quand utiliser cette commande :**

- Vérifier l'**architecture** d'un binaire (`Machine`). Un binaire compilé pour ARM sur une machine x86_64 affichera `AArch64` ici — c'est la première chose à vérifier face à un `cannot execute binary file: Exec format error`.
- Identifier le **type** du fichier (`Type`). `REL` = fichier objet, `DYN` = exécutable PIE ou bibliothèque partagée, `EXEC` = exécutable à adresse fixe.
- Retrouver le **point d'entrée** (`Entry point address`) pour vérifier que le binaire est correctement lié.
- Compter le **nombre de sections** pour repérer un binaire strippé (très peu de sections) vs un binaire complet avec debug (beaucoup de sections).

### `readelf -S` — La table des sections

C'est l'une des commandes les plus utilisées. Elle affiche la liste complète des sections avec leur type, taille, offset et flags.

```bash
readelf -S ./reference_debug
```

Sortie (extraits significatifs) :

```
There are 38 section headers, starting at offset 0x5238:

Section Headers:
  [Nr] Name              Type             Address           Offset    Size
  [ 0]                   NULL             0000000000000000  00000000  0000000000000000
  [ 1] .interp           PROGBITS         0000000000000318  00000318  000000000000001c
  ...
  [14] .text             PROGBITS         0000000000001060  00001060  0000000000000145
  [15] .fini             PROGBITS         00000000000011a8  000011a8  000000000000000d
  [16] .rodata           PROGBITS         0000000000001200  00001200  0000000000000030
  ...
  [23] .data             PROGBITS         0000000000004000  00003000  0000000000000020
  [24] .bss              NOBITS           0000000000004020  00003020  0000000000000fc0
  [25] .comment          PROGBITS         0000000000000000  00003020  000000000000002b
  [26] .debug_aranges    PROGBITS         0000000000000000  00003050  0000000000000030
  [27] .debug_info       PROGBITS         0000000000000000  00003080  00000000000001a5
  [28] .debug_abbrev     PROGBITS         0000000000000000  00003225  00000000000000c8
  [29] .debug_line       PROGBITS         0000000000000000  000032ed  0000000000000090
  [30] .debug_str        PROGBITS         0000000000000000  0000337d  00000000000000f5
  ...

Key to Flags:
  W (write), A (alloc), X (execute), ...
```

**Lecture de la sortie :**

Chaque ligne est une section. Les colonnes essentielles sont :

- **Name** : le nom de la section (`.text`, `.data`, `.bss`…).
- **Type** : `PROGBITS` (contient des données définies par le programme), `NOBITS` (pas de contenu sur disque — `.bss`), `SYMTAB`, `STRTAB`, etc.
- **Address** : l'adresse virtuelle où la section sera chargée en mémoire. Les sections non-chargées (debug, symboles) ont une adresse à `0`.
- **Offset** : la position de la section dans le fichier sur disque.
- **Size** : la taille de la section en octets (en hexadécimal).

**Observations sur notre programme :**

La section `.text` fait `0x145` octets (325 octets) — c'est le code machine de `afficher()` et `main()`. La section `.bss` fait `0xfc0` octets (4032 octets) — notre `tableau_zero` de 4000 octets plus quelques variables internes. Son type est `NOBITS`, confirmant qu'elle n'occupe pas de place sur disque. Les sections `.debug_*` ont une adresse à `0` (non chargées en mémoire) et totalisent plusieurs centaines d'octets — sur un vrai projet, elles peuvent représenter la majorité de la taille du fichier.

**Quand utiliser cette commande :**

- Diagnostiquer un **binaire anormalement gros** : identifier quelles sections consomment l'espace (souvent `.debug_info`).
- Vérifier que les **informations de débogage** sont présentes avant une session GDB.
- Comparer la structure d'un binaire **debug vs release**.
- Vérifier la présence de sections spécifiques (`.dynamic` pour un binaire dynamique, `.interp` pour le chemin du chargeur).

```bash
readelf -l ./reference_debug
```

Sortie (extraits) :

```
  INTERP         0x0000000000000318 0x0000000000000318 0x0000000000000318
                 0x000000000000001c 0x000000000000001c  R      0x1
      [Requesting program interpreter: /lib64/ld-linux-x86-64.so.2]
  LOAD           0x0000000000000000 0x0000000000000000 0x0000000000000000
                 0x0000000000000628 0x0000000000000628  R      0x1000
  LOAD           0x0000000000001000 0x0000000000001000 0x0000000000001000
                 0x00000000000001b5 0x00000000000001b5  R E    0x1000
  LOAD           0x0000000000002000 0x0000000000002000 0x0000000000002000
                 0x0000000000000150 0x0000000000000150  R      0x1000
  LOAD           0x0000000000002de8 0x0000000000003de8 0x0000000000003de8
                 0x0000000000000238 0x0000000000001200  RW     0x1000
  DYNAMIC        0x0000000000002df8 0x0000000000003df8 0x0000000000003df8
                 0x00000000000001e0 0x00000000000001e0  RW     0x8
  ...
  GNU_STACK      0x0000000000000000 0x0000000000000000 0x0000000000000000
                 0x0000000000000000 0x0000000000000000  RW     0x10
  GNU_RELRO      0x0000000000002de8 0x0000000000003de8 0x0000000000003de8
                 0x0000000000000218 0x0000000000000218  R      0x1

 Section to Segment mapping:
  Segment Sections...
   00
   01     .interp
   02     .interp .note.gnu.property .note.gnu.build-id .note.ABI-tag ...
   03     .init .plt .plt.got .plt.sec .text .fini
   04     .rodata .eh_frame_hdr .eh_frame
   05     .init_array .fini_array .dynamic .got .data .bss
   06     .dynamic
   ...
```

**L'information la plus précieuse** se trouve en bas : le **mapping sections → segments**. Il montre exactement quelles sections sont regroupées dans chaque segment. Le segment 03 (flags `R E`, lecture + exécution) contient `.text` — le code. Le segment 05 (flags `RW`, lecture + écriture) contient `.data` et `.bss` — les données modifiables.

Observez le segment 05 : `FileSiz = 0x238` mais `MemSiz = 0x1200`. La différence (`0x1200 - 0x238 = 0xfc8` octets) correspond à `.bss`, qui sera allouée en mémoire mais n'existe pas dans le fichier.

Le segment `GNU_STACK` a les flags `RW` (pas de `X`) : la pile est **non exécutable**, une protection de sécurité essentielle.

**Quand utiliser cette commande :**

- Comprendre la **disposition mémoire** d'un programme.
- Vérifier les **permissions** des segments (pile non-exécutable, RELRO activé).
- Diagnostiquer un problème de **chargement dynamique** (segment `INTERP` manquant ou incorrect).
- Analyser la différence `FileSiz` / `MemSiz` pour comprendre l'usage de `.bss`.

### `readelf -s` — La table des symboles

```bash
readelf -s ./reference_debug
```

Sortie (extraits) :

```
Symbol table '.dynsym' contains 8 entries:
   Num:    Value          Size Type    Bind   Vis      Ndx Name
     0: 0000000000000000     0 NOTYPE  LOCAL  DEFAULT  UND
     1: 0000000000000000     0 FUNC    GLOBAL DEFAULT  UND printf@GLIBC_2.2.5
     ...

Symbol table '.symtab' contains 42 entries:
   Num:    Value          Size Type    Bind   Vis      Ndx Name
    ...
    28: 0000000000004010     8 OBJECT  GLOBAL DEFAULT   23 message
    29: 0000000000004018     4 OBJECT  GLOBAL DEFAULT   23 compteur_global
    30: 0000000000004020  4000 OBJECT  GLOBAL DEFAULT   24 tableau_zero
    31: 0000000000001149    52 FUNC    GLOBAL DEFAULT   14 _Z8afficherv
    32: 000000000000117d    27 FUNC    GLOBAL DEFAULT   14 main
    ...
```

**Lecture de la sortie :**

Deux tables sont affichées. `.dynsym` contient les symboles nécessaires au **chargement dynamique** — uniquement les symboles importés des bibliothèques partagées et les symboles exportés. `.symtab` contient la table **complète**, y compris les symboles locaux, les fonctions et les variables globales de votre code.

Les colonnes essentielles :

- **Value** : l'adresse virtuelle du symbole.
- **Size** : la taille en octets. `tableau_zero` fait 4000 octets, `compteur_global` fait 4 octets (un `int`), `message` fait 8 octets (un pointeur 64 bits).
- **Type** : `FUNC` (fonction), `OBJECT` (variable), `NOTYPE` (indéterminé).
- **Bind** : `LOCAL` (visible uniquement dans ce fichier objet), `GLOBAL` (visible partout), `WEAK` (peut être remplacé par un symbole global du même nom).
- **Ndx** : l'index de la section qui contient le symbole. `14` correspond à `.text`, `23` à `.data`, `24` à `.bss`. `UND` signifie *undefined* — le symbole est importé.
- **Name** : le nom du symbole, avec mangling pour les fonctions C++ (`_Z8afficherv` = `afficher()`).

> 💡 **Note** — La table `.symtab` est supprimée par `strip --strip-all`. Après cette opération, seule `.dynsym` subsiste (car elle est nécessaire à l'exécution). C'est pourquoi un binaire strippé est plus difficile à déboguer — les noms de fonctions locales et de variables ont disparu.

**Quand utiliser cette commande :**

- Chercher si un **symbole est défini** dans un binaire ou un fichier objet.
- Vérifier le **mangling** d'une fonction C++.
- Diagnostiquer un `undefined reference` en comparant les symboles de plusieurs fichiers objet.
- Comprendre la **taille** de chaque variable globale et fonction.

### `readelf -d` — La section `.dynamic`

```bash
readelf -d ./reference_debug
```

Sortie (extraits) :

```
Dynamic section at offset 0x2df8 contains 27 entries:
  Tag        Type                         Name/Value
 0x0000000000000001 (NEEDED)             Shared library: [libc.so.6]
 0x000000000000000c (INIT)               0x1000
 0x000000000000000d (FINI)               0x11a8
 0x0000000000000019 (INIT_ARRAY)         0x3de8
 0x000000000000001b (INIT_ARRAYSZ)       8 (bytes)
 0x000000000000001a (FINI_ARRAY)         0x3df0
 0x000000000000001c (FINI_ARRAYSZ)       8 (bytes)
 0x000000006ffffef5 (GNU_HASH)           0x3b0
 0x0000000000000005 (STRTAB)             0x480
 0x0000000000000006 (SYMTAB)             0x3c8
 ...
 0x000000000000001e (FLAGS)              BIND_NOW
 0x000000006ffffffb (FLAGS_1)            Flags: NOW PIE
 ...
```

La ligne `NEEDED` liste les bibliothèques partagées requises — ici uniquement `libc.so.6`. Le flag `BIND_NOW` confirme que le Full RELRO est activé (tous les symboles dynamiques sont résolus au chargement, pas de *lazy binding*). Le flag `PIE` confirme que l'exécutable est à position indépendante.

**Quand utiliser cette commande :**

- Lister les **dépendances** d'un binaire (alternative à `ldd` qui ne nécessite pas d'exécuter le binaire).
- Vérifier les **flags de sécurité** (BIND_NOW, PIE).
- Inspecter les chemins `RPATH` / `RUNPATH` encodés dans le binaire.

> 🔥 **Point clé** — `readelf -d` est plus sûr que `ldd` pour inspecter un binaire non fiable. `ldd` exécute partiellement le binaire pour résoudre les dépendances, ce qui peut déclencher du code malveillant. `readelf` lit le fichier passivement, sans aucune exécution.

## `objdump` — Désassemblage et analyse du code

`objdump` est complémentaire de `readelf`. Là où `readelf` excelle dans l'analyse de la structure ELF, `objdump` se spécialise dans le **désassemblage** — la conversion du code machine binaire en instructions assembleur lisibles.

### `objdump -d` — Désassemblage du code

```bash
objdump -d ./reference_debug
```

Sortie (extrait de la fonction `main`) :

```
000000000000117d <main>:
    117d:       f3 0f 1e fa             endbr64
    1181:       55                      push   %rbp
    1182:       48 89 e5                mov    %rsp,%rbp
    1185:       e8 bf ff ff ff          call   1149 <_Z8afficherv>
    118a:       b8 00 00 00 00          mov    $0x0,%eax
    118f:       5d                      pop    %rbp
    1190:       c3                      ret
```

Chaque ligne montre l'**adresse**, les **octets du code machine** (en hexadécimal), et l'**instruction assembleur** correspondante (syntaxe AT&T par défaut). On voit clairement l'appel à `afficher()` via `call 1149`, le chargement de la valeur de retour `0` dans `eax`, et le `ret` final.

Pour utiliser la syntaxe Intel (souvent jugée plus lisible) :

```bash
objdump -d -M intel ./reference_debug
```

Sortie équivalente :

```
000000000000117d <main>:
    117d:       f3 0f 1e fa             endbr64
    1181:       55                      push   rbp
    1182:       48 89 e5                mov    rbp,rsp
    1185:       e8 bf ff ff ff          call   1149 <_Z8afficherv>
    118a:       b8 00 00 00 00          mov    eax,0x0
    118f:       5d                      pop    rbp
    1190:       c3                      ret
```

Pour désassembler **uniquement une fonction spécifique**, combinez avec `grep` ou utilisez l'option `--disassemble=` (disponible dans les versions récentes de binutils) :

```bash
objdump -d -M intel ./reference_debug | grep -A 20 '<main>:'
```

### `objdump -d` avec `-S` — Code source intercalé

Si le binaire contient des informations de débogage (`-g`), l'option `-S` intercale le code source C++ avec l'assembleur :

```bash
objdump -d -S -M intel ./reference_debug
```

Sortie (extrait) :

```
int main() {
    117d:       f3 0f 1e fa             endbr64
    1181:       55                      push   rbp
    1182:       48 89 e5                mov    rbp,rsp
    afficher();
    1185:       e8 bf ff ff ff          call   1149 <_Z8afficherv>
    return 0;
    118a:       b8 00 00 00 00          mov    eax,0x0
    118f:       5d                      pop    rbp
    1190:       c3                      ret
```

C'est un outil remarquable pour comprendre **ce que le compilateur a fait de votre code**. Chaque ligne C++ est suivie des instructions machine correspondantes. C'est particulièrement instructif quand on compare la sortie en `-O0` (pas d'optimisation, correspondance directe) avec la sortie en `-O2` (optimisations, le code peut être radicalement réorganisé).

**Quand utiliser `objdump -d` :**

- Comprendre le **code machine généré** par le compilateur.
- Vérifier qu'une **optimisation** a bien été appliquée (inline, vectorisation, élimination de code mort).
- Analyser un **crash** quand vous n'avez que l'adresse de l'instruction fautive.
- Comparer le code généré par **GCC vs Clang** ou par **différents niveaux d'optimisation**.

### `objdump -t` — Table des symboles

```bash
objdump -t ./reference_debug
```

Cette commande affiche la table des symboles dans un format légèrement différent de `readelf -s`, avec les flags de section et la section d'appartenance en clair. Elle est utile comme alternative quand on veut voir rapidement les symboles avec leur section.

### `objdump -s -j` — Dump hexadécimal d'une section

Pour voir le contenu brut d'une section spécifique :

```bash
objdump -s -j .rodata ./reference_debug
```

Sortie (extrait) :

```
Contents of section .rodata:
 2000 01000200 00000000 00000000 00000000  ................
 2010 466f726d 6174696f 6e204300 2b2b206d  Formation C++ m
 2020 6f646572 6e650025 73202863 6f6d7074  oderne.%s (compt
 2030 6575723a 20256429 0a00               eur: %d)..
```

On voit clairement nos chaînes littérales dans la section `.rodata` : `"Formation C++ moderne"` et la chaîne de format `"%s (compteur: %d)\n"`. La colonne de droite montre l'interprétation ASCII (les octets non-imprimables sont remplacés par des points).

```bash
objdump -s -j .data ./reference_debug
```

Sortie (extrait) :

```
Contents of section .data:
 4000 00000000 00000000 08400000 00000000  .........@......
 4010 10200000 00000000 2a000000           . ......*...
```

L'octet `2a` à l'offset 4018 est la valeur hexadécimale de `42` — c'est notre `compteur_global`. Les 8 octets à l'offset 4010 sont le pointeur `message`, qui contient l'adresse de la chaîne dans `.rodata`.

**Quand utiliser `objdump -s -j` :**

- Inspecter le **contenu exact** d'une section (données constantes, valeurs initiales de variables globales).
- Vérifier que des **chaînes sensibles** (mots de passe, clés) ne sont pas présentes en clair dans le binaire.
- Comprendre la **représentation mémoire** des données (endianness, alignement, padding).

## `nm` — Exploration rapide des symboles

`nm` est un outil plus léger que `readelf -s`, spécialisé dans l'affichage de la table des symboles. C'est souvent le **premier réflexe** pour diagnostiquer un problème de liaison.

### Utilisation de base

```bash
nm ./reference_debug
```

Sortie (extraits) :

```
0000000000004018 D compteur_global
0000000000004010 D message
0000000000004020 B tableau_zero
0000000000001149 T _Z8afficherv
000000000000117d T main
                 U printf@GLIBC_2.2.5
```

Les lettres sont des codes de section :

| Code | Signification |
|------|---------------|
| `T` / `t` | Symbole défini dans `.text` (code). Majuscule = global, minuscule = local. |
| `D` / `d` | Symbole défini dans `.data` (données initialisées). |
| `B` / `b` | Symbole défini dans `.bss` (données non-initialisées). |
| `R` / `r` | Symbole défini dans `.rodata` (données en lecture seule). |
| `U` | Symbole **non défini** (undefined) — importé d'une bibliothèque externe. |
| `W` / `w` | Symbole faible (*weak*). |
| `A` | Adresse absolue. |

### Options essentielles

```bash
nm -C ./reference_debug       # -C : demangling des noms C++
```

Sortie avec demangling :

```
0000000000004018 D compteur_global
0000000000004010 D message
0000000000004020 B tableau_zero
0000000000001149 T afficher()
000000000000117d T main
                 U printf@GLIBC_2.2.5
```

`_Z8afficherv` est devenu `afficher()` — beaucoup plus lisible.

```bash
nm -C --defined-only ./reference_debug    # Uniquement les symboles définis  
nm -C --undefined-only ./reference_debug  # Uniquement les symboles non-définis  
nm -C -S ./reference_debug                # -S : affiche aussi la taille des symboles  
```

### Scénario concret : diagnostiquer un `undefined reference`

Vous obtenez l'erreur suivante à l'édition de liens :

```
undefined reference to `parse_config(std::string const&)'
```

Première étape — vérifier dans quel fichier objet le symbole est censé être défini :

```bash
nm -C config.o | grep parse_config
```

Si rien ne s'affiche, la fonction n'est pas définie dans `config.o`. Soit le fichier n'a pas été compilé avec la bonne version du code, soit la signature de la fonction dans le `.cpp` ne correspond pas à la déclaration dans le `.h`.

Si le symbole apparaît avec le code `T`, le fichier objet est correct — vérifiez alors que `config.o` est bien inclus dans la commande de liaison.

## `file` — Identification rapide

`file` est la commande la plus simple mais souvent la plus utile en première intention. Elle identifie le type d'un fichier en lisant ses premiers octets :

```bash
file ./reference_debug
```

```
reference_debug: ELF 64-bit LSB pie executable, x86-64, version 1 (SYSV),  
dynamically linked, interpreter /lib64/ld-linux-x86-64.so.2,  
BuildID[sha1]=a1b2c3..., for GNU/Linux 3.2.0, with debug_info, not stripped  
```

En une seule ligne, `file` révèle :

- le format (`ELF 64-bit`),
- l'endianness (`LSB` — little-endian),
- le type (`pie executable`),
- l'architecture (`x86-64`),
- le mode de liaison (`dynamically linked`),
- le chargeur dynamique (`interpreter /lib64/ld-linux-x86-64.so.2`),
- la présence d'informations de débogage (`with debug_info`),
- le fait que le binaire n'est pas strippé (`not stripped`).

Comparons avec le binaire release strippé :

```bash
strip --strip-all -o reference_stripped reference_release  
file ./reference_stripped  
```

```
reference_stripped: ELF 64-bit LSB pie executable, x86-64, version 1 (SYSV),  
dynamically linked, interpreter /lib64/ld-linux-x86-64.so.2,  
BuildID[sha1]=d4e5f6..., for GNU/Linux 3.2.0, stripped  
```

Les mentions `with debug_info` et `not stripped` ont disparu.

## `ldd` — Dépendances dynamiques

Déjà rencontré dans les sections précédentes, `ldd` mérite un rappel ici car il est complémentaire des outils d'inspection ELF :

```bash
ldd ./reference_debug
```

```
linux-vdso.so.1 (0x00007ffce9ffe000)  
libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6 (0x00007f8a3c000000)  
/lib64/ld-linux-x86-64.so.2 (0x00007f8a3c400000)
```

Notre programme ne dépend que de `libc.so.6` (la bibliothèque C standard, qui inclut aussi les fonctionnalités C++ de base quand la libstdc++ est liée statiquement ou intégrée). Les binaires C++ plus complexes listeront aussi `libstdc++.so.6`, `libm.so.6`, `libpthread.so.0`, etc.

> 💡 **Rappel** — Pour les binaires non fiables, préférez `readelf -d` à `ldd` (voir l'encadré dans la sous-section `readelf -d` ci-dessus).

```
   text    data     bss     dec     hex filename
   1953     632    4032    6617    19d9 reference_debug
   1749     600    4016    6365    18dd reference_release
   1749     600    4016    6365    18dd reference_stripped
```

Observations :

- Le binaire release (`-O2`) a une section `text` légèrement plus petite : les optimisations ont produit un code machine plus compact.
- Le stripping n'affecte pas les colonnes `text`, `data` et `bss` — il supprime les sections non-chargées (symboles, debug), qui ne sont pas comptées ici.
- La section `bss` reste autour de 4000 octets dans tous les cas — c'est notre `tableau_zero`.

Comparons maintenant la taille réelle des fichiers sur disque :

```bash
ls -l reference_debug reference_release reference_stripped
```

```
-rwxr-xr-x 1 user user  22480 mar  9 14:30 reference_debug
-rwxr-xr-x 1 user user  15960 mar  9 14:30 reference_release
-rwxr-xr-x 1 user user   6120 mar  9 14:30 reference_stripped
```

Le fichier debug fait 22 Ko, le release 16 Ko, et le strippé 6 Ko. La différence entre debug et release provient essentiellement des sections `.debug_*`. La différence entre release et stripped provient de `.symtab`, `.strtab` et d'autres sections de métadonnées.

## `strings` — Extraire les chaînes lisibles

`strings` n'est pas un outil ELF à proprement parler, mais il est précieux pour inspecter le contenu textuel d'un binaire :

```bash
strings ./reference_debug | grep -i formation
```

```
Formation C++ moderne
```

La chaîne littérale est visible en clair dans le binaire. C'est un rappel important : **tout ce que vous écrivez en dur dans votre code source se retrouve tel quel dans le binaire**. Mots de passe, clés d'API, URLs internes — tout est accessible avec un simple `strings`.

```bash
strings ./reference_debug | grep -i gcc
```

```
GCC: (Ubuntu 14.2.0-4ubuntu2) 14.2.0
```

Le compilateur laisse sa signature dans le binaire (section `.comment`). Utile pour identifier quelle version de GCC ou Clang a produit un binaire.

## `hexdump` / `xxd` — Inspection brute

Pour les cas où vous avez besoin de voir les octets bruts d'un fichier ELF :

```bash
# Les 64 premiers octets (l'en-tête ELF complet)
xxd -l 64 ./reference_debug
```

```
00000000: 7f45 4c46 0201 0100 0000 0000 0000 0000  .ELF............
00000010: 0300 3e00 0100 0000 6010 0000 0000 0000  ..>.....`.......
00000020: 4000 0000 0000 0000 3852 0000 0000 0000  @.......8R......
00000030: 0000 0000 4000 3800 0d00 4000 2600 2500  ....@.8...@.&.%.
```

Les quatre premiers octets (`7f 45 4c 46`) sont le nombre magique `\x7fELF`. L'octet à l'offset 4 est `02` (ELFCLASS64). L'octet à l'offset 5 est `01` (little-endian). L'octet à l'offset 16 est `03` (ET_DYN, en little-endian `03 00`).

`hexdump` et `xxd` sont rarement nécessaires au quotidien, mais ils sont irremplaçables quand vous devez vérifier un comportement à l'octet près — par exemple, vérifier le nombre magique d'un fichier corrompu ou analyser l'encoding exact d'une valeur en mémoire.

## Récapitulatif : quel outil pour quelle question ?

| Question | Outil | Commande |
|----------|-------|----------|
| Quelle architecture, quel type de binaire ? | `file` | `file ./programme` |
| Quelles sont les sections et leur taille ? | `readelf` | `readelf -S ./programme` |
| Comment le programme est-il chargé en mémoire ? | `readelf` | `readelf -l ./programme` |
| De quelles bibliothèques le binaire dépend-il ? | `readelf` / `ldd` | `readelf -d ./programme` / `ldd ./programme` |
| Quels symboles sont définis / non-définis ? | `nm` | `nm -C ./programme` |
| Quel code assembleur a été généré ? | `objdump` | `objdump -d -M intel -S ./programme` |
| Que contient une section spécifique ? | `objdump` | `objdump -s -j .rodata ./programme` |
| Quelle est la taille des sections code/data/bss ? | `size` | `size ./programme` |
| Quelles chaînes de texte sont dans le binaire ? | `strings` | `strings ./programme` |
| Quels sont les octets bruts à un offset donné ? | `xxd` | `xxd -l 64 ./programme` |

## Ce qu'il faut retenir

Les outils d'inspection ELF sont les **stéthoscopes du développeur C++**. `file` pour l'identification rapide, `readelf` pour l'analyse structurelle complète (headers, sections, segments, symboles, section dynamique), `objdump` pour le désassemblage et l'inspection du contenu des sections, `nm` pour l'exploration rapide des symboles, `size` pour la vue synthétique, `ldd` pour les dépendances dynamiques, `strings` pour les chaînes lisibles, et `xxd` pour l'inspection brute octet par octet. Ces outils sont installés par défaut sur Ubuntu, ne nécessitent aucune configuration, et fonctionnent sur n'importe quel fichier ELF — exécutable, fichier objet ou bibliothèque partagée.

Avec cette sous-section, nous concluons le **chapitre 1** — Introduction au C++ et à l'écosystème Linux. Vous disposez maintenant d'une vision complète du voyage d'un programme C++ : de son contexte historique (section 1.1) à sa pertinence en 2026 (section 1.2), en passant par chaque étape de sa compilation (sections 1.3.x) jusqu'à la structure interne du binaire produit (sections 1.4.x). Le **chapitre 2** passera à la pratique en installant et configurant l'ensemble de la toolchain sur votre machine Ubuntu.
