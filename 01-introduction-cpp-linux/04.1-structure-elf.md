## Introduction

La section 1.4 a présenté la vue d'ensemble du format ELF et ses principes fondamentaux. Cette sous-section descend d'un cran dans le détail : nous allons parcourir chaque composant structurel d'un fichier ELF — l'en-tête principal, la table des sections, la table des segments — en expliquant le rôle de chaque champ et la manière dont ces composants s'articulent entre eux.

L'objectif n'est pas de mémoriser chaque champ par cœur. C'est de construire un **modèle mental** suffisamment précis pour que, lorsque vous exécuterez `readelf` ou `objdump` (section 1.4.2), chaque ligne de la sortie ait un sens pour vous.

Pour rendre les choses concrètes, nous travaillerons avec un programme de référence tout au long de cette section :

```cpp
// reference.cpp
#include <cstdio>

const char* message = "Formation C++ moderne";  
int compteur_global = 42;  
int tableau_zero[1000];  // initialisé à zéro → .bss  

void afficher() {
    printf("%s (compteur: %d)\n", message, compteur_global);
}

int main() {
    afficher();
    return 0;
}
```

Compilons-le :

```bash
g++ -std=c++23 -g -O0 -o reference reference.cpp
```

Ce programme est volontairement simple mais contient des éléments variés : une constante chaîne (`.rodata`), un pointeur global initialisé (`.data`), un entier global initialisé (`.data`), un tableau global initialisé à zéro (`.bss`), du code exécutable (`.text`), et des informations de débogage (sections `.debug_*` grâce au flag `-g`). Il nous servira de support pour illustrer chaque composant du format.

## L'en-tête ELF (*ELF Header*)

L'en-tête ELF est une structure de taille fixe située **au tout début du fichier** (offset 0). Sur un système 64 bits, il occupe exactement **64 octets**. C'est le premier élément lu par tout outil ou par le noyau quand il doit interpréter un fichier ELF.

### Les champs de l'en-tête

Voici les champs de l'en-tête ELF 64 bits, dans l'ordre où ils apparaissent dans le fichier :

| Champ | Taille | Description |
|-------|--------|-------------|
| `e_ident[0..3]` | 4 octets | **Nombre magique** : `0x7f` suivi de `E`, `L`, `F` en ASCII (`7f 45 4c 46`). Identifie sans ambiguïté un fichier ELF. |
| `e_ident[4]` | 1 octet | **Classe** : `1` = 32 bits (ELFCLASS32), `2` = 64 bits (ELFCLASS64). |
| `e_ident[5]` | 1 octet | **Encodage des données** : `1` = little-endian (ELFDATA2LSB), `2` = big-endian (ELFDATA2MSB). x86_64 est toujours little-endian. |
| `e_ident[6]` | 1 octet | **Version ELF** : toujours `1` (EV_CURRENT). |
| `e_ident[7]` | 1 octet | **OS/ABI** : identifie l'ABI cible. `0` = ELFOSABI_NONE (System V, le plus courant sur Linux). `3` = ELFOSABI_LINUX. |
| `e_ident[8..15]` | 8 octets | **Padding** : octets réservés, remplis de zéros. |
| `e_type` | 2 octets | **Type de fichier** : `ET_REL` (1) = fichier objet, `ET_EXEC` (2) = exécutable à adresse fixe, `ET_DYN` (3) = shared object ou exécutable PIE. |
| `e_machine` | 2 octets | **Architecture** : `EM_X86_64` (62), `EM_AARCH64` (183), `EM_RISCV` (243), etc. |
| `e_version` | 4 octets | **Version** : toujours `1`. |
| `e_entry` | 8 octets | **Point d'entrée** : adresse virtuelle de la première instruction à exécuter (`_start`). Vaut `0` pour les fichiers objet. |
| `e_phoff` | 8 octets | **Offset de la Program Header Table** (table des segments) dans le fichier. `0` si absent (fichiers objet). |
| `e_shoff` | 8 octets | **Offset de la Section Header Table** (table des sections) dans le fichier. |
| `e_flags` | 4 octets | **Flags spécifiques à l'architecture**. Souvent `0` sur x86_64. |
| `e_ehsize` | 2 octets | **Taille de cet en-tête** : `64` octets pour ELF64. |
| `e_phentsize` | 2 octets | **Taille d'une entrée** dans la Program Header Table. |
| `e_phnum` | 2 octets | **Nombre d'entrées** dans la Program Header Table. |
| `e_shentsize` | 2 octets | **Taille d'une entrée** dans la Section Header Table. |
| `e_shnum` | 2 octets | **Nombre d'entrées** dans la Section Header Table. |
| `e_shstrndx` | 2 octets | **Index de la section** qui contient les noms des sections (`.shstrtab`). |

### Lecture de l'en-tête avec readelf

```bash
readelf -h ./reference
```

Sortie (extraits significatifs) :

```
ELF Header:
  Magic:   7f 45 4c 46 02 01 01 00 00 00 00 00 00 00 00 00
  Class:                             ELF64
  Data:                              2's complement, little endian
  Version:                           1 (current)
  OS/ABI:                            UNIX - System V
  Type:                              DYN (Position-Independent Executable)
  Machine:                           Advanced Micro Devices X86-64
  Entry point address:               0x1060
  Start of program headers:          64 (bytes into file)
  Start of section headers:          17824 (bytes into file)
  ...
  Number of program headers:         13
  Number of section headers:         36
  Section header string table index: 35
```

Chaque ligne correspond directement à un champ de la structure. Quelques observations :

Le **type** est `DYN`, pas `EXEC`. C'est normal : GCC compile par défaut en mode PIE (exécutable à position indépendante), qui utilise le type `ET_DYN`. Le binaire est néanmoins un exécutable.

Le **point d'entrée** est `0x1060`. Ce n'est pas l'adresse de `main()` — c'est l'adresse de `_start`, la fonction d'initialisation fournie par la libc.

La **Program Header Table** commence à l'offset 64 — c'est-à-dire immédiatement après l'en-tête ELF. C'est la convention habituelle.

La **Section Header Table** est à l'offset 17824, vers la fin du fichier. Elle contient 36 entrées (36 sections), et l'index 35 (la dernière section) est la table de noms de sections `.shstrtab`.

## La table des sections (*Section Header Table*)

La Section Header Table est un **tableau d'entrées** situé vers la fin du fichier ELF. Chaque entrée décrit une section : son nom, son type, ses attributs, sa position dans le fichier, et sa taille.

### Structure d'une entrée de section

Chaque entrée (*Section Header*) occupe 64 octets sur un système 64 bits et contient les champs suivants :

| Champ | Taille | Description |
|-------|--------|-------------|
| `sh_name` | 4 octets | Offset du nom de la section dans la table de chaînes `.shstrtab`. |
| `sh_type` | 4 octets | Type de la section (voir tableau ci-dessous). |
| `sh_flags` | 8 octets | Attributs de la section (voir tableau ci-dessous). |
| `sh_addr` | 8 octets | Adresse virtuelle en mémoire (si la section est chargée, sinon `0`). |
| `sh_offset` | 8 octets | Offset de la section dans le fichier sur disque. |
| `sh_size` | 8 octets | Taille de la section en octets. |
| `sh_link` | 4 octets | Lien vers une autre section (dépend du type, ex : `.symtab` pointe vers `.strtab`). |
| `sh_info` | 4 octets | Information supplémentaire (dépend du type). |
| `sh_addralign` | 8 octets | Alignement requis (puissance de 2). |
| `sh_entsize` | 8 octets | Taille de chaque entrée si la section contient une table (ex : `.symtab`). |

### Types de sections courants

| Type (`sh_type`) | Valeur | Description |
|-----------------|--------|-------------|
| `SHT_NULL` | 0 | Section invalide (la première entrée est toujours `SHT_NULL`). |
| `SHT_PROGBITS` | 1 | Données définies par le programme (code, données, constantes…). `.text`, `.data`, `.rodata` sont de ce type. |
| `SHT_SYMTAB` | 2 | Table des symboles complète. |
| `SHT_STRTAB` | 3 | Table de chaînes de caractères (noms de symboles, noms de sections). |
| `SHT_RELA` | 4 | Entrées de relocation avec addend. |
| `SHT_DYNAMIC` | 6 | Informations pour le chargement dynamique. |
| `SHT_NOTE` | 7 | Notes (informations auxiliaires : build ID, version ABI…). |
| `SHT_NOBITS` | 8 | Section qui n'occupe pas de place sur disque. `.bss` est de ce type. |
| `SHT_DYNSYM` | 11 | Table des symboles dynamiques. |

### Flags de sections

Les flags (`sh_flags`) déterminent les permissions mémoire et le comportement de la section :

| Flag | Valeur | Signification |
|------|--------|---------------|
| `SHF_WRITE` | 0x1 | La section est modifiable en mémoire (écriture autorisée). |
| `SHF_ALLOC` | 0x2 | La section occupe de la mémoire à l'exécution (elle est chargée). |
| `SHF_EXECINSTR` | 0x4 | La section contient du code exécutable. |

La combinaison de ces flags détermine le comportement de chaque section :

| Section | Flags | Interprétation |
|---------|-------|----------------|
| `.text` | `SHF_ALLOC + SHF_EXECINSTR` | Chargée en mémoire, exécutable, non modifiable. |
| `.rodata` | `SHF_ALLOC` | Chargée en mémoire, lecture seule. |
| `.data` | `SHF_ALLOC + SHF_WRITE` | Chargée en mémoire, modifiable. |
| `.bss` | `SHF_ALLOC + SHF_WRITE` | Chargée en mémoire, modifiable, zéro place sur disque. |
| `.symtab` | *(aucun de ces flags)* | Non chargée en mémoire. Présente uniquement dans le fichier. |
| `.debug_info` | *(aucun de ces flags)* | Non chargée en mémoire. Utilisée uniquement par le debugger. |

> 💡 **Note** — La distinction entre `SHF_ALLOC` et l'absence de ce flag est importante. Les sections sans `SHF_ALLOC` (tables de symboles, informations de débogage, tables de chaînes) existent dans le fichier sur disque mais ne sont **jamais chargées en mémoire** pendant l'exécution. Elles n'ont aucun impact sur l'empreinte mémoire du processus. C'est pourquoi un binaire de 200 Mo avec des informations de débogage peut n'utiliser que 2 Mo de RAM à l'exécution.

### La section `.shstrtab` — Noms des sections

Les noms des sections (`.text`, `.data`, `.rodata`…) ne sont pas stockés directement dans les entrées de la Section Header Table. Chaque entrée contient un **offset** (`sh_name`) dans une section spéciale appelée `.shstrtab` (*section header string table*). Cette section est un bloc de texte contenant tous les noms, séparés par des octets nuls (`\0`) :

```
Offset 0:   \0  
Offset 1:   .text\0  
Offset 7:   .data\0  
Offset 13:  .bss\0  
Offset 18:  .rodata\0  
Offset 26:  .symtab\0  
...
```

L'entrée de la section `.text` contient `sh_name = 1`, ce qui pointe vers la chaîne `.text` à l'offset 1 dans `.shstrtab`. Ce système d'indirection par table de chaînes est un pattern récurrent dans le format ELF — il est également utilisé pour les noms de symboles (`.strtab`, `.dynstr`).

## Cartographie des sections de notre programme de référence

Revenons à notre programme `reference.cpp` et voyons comment le compilateur et le linker ont distribué ses éléments dans les sections ELF.

### Le code source et ses sections cibles

```cpp
const char* message = "Formation C++ moderne";  // "Formation..." → .rodata
                                                  // message (le pointeur) → .data
int compteur_global = 42;                         // → .data  
int tableau_zero[1000];                           // → .bss  

void afficher() { ... }                           // → .text  
int main() { ... }                                // → .text  
```

Détaillons chaque élément :

**La chaîne littérale `"Formation C++ moderne"`** est une constante en lecture seule. Elle est placée dans la section `.rodata`. Le programme n'a pas le droit de la modifier — tenter de le faire provoquerait un segmentation fault.

**Le pointeur `message`** est une variable globale initialisée avec l'adresse de la chaîne. Comme c'est une variable modifiable avec une valeur initiale non nulle, elle est placée dans `.data`. Sa valeur sur disque est l'adresse (relative) de la chaîne dans `.rodata`, ajustée par une relocation au chargement.

**L'entier `compteur_global = 42`** est une variable globale initialisée avec une valeur non nulle. Il est placé dans `.data`. Ses 4 octets contiennent la valeur `42` (en little-endian : `2a 00 00 00`) directement dans le fichier sur disque.

**Le tableau `tableau_zero[1000]`** est un tableau global de 1000 entiers (4000 octets) implicitement initialisé à zéro. Il est placé dans `.bss`. Le fichier sur disque enregistre uniquement sa taille (4000 octets) — les zéros ne sont pas stockés. C'est le chargeur qui allouera 4000 octets de mémoire remplis de zéros au moment du chargement.

**Les fonctions `afficher()` et `main()`** contiennent du code machine. Elles sont placées dans `.text`.

### Impact sur la taille

```bash
size ./reference
```

Sortie typique :

```
   text    data     bss     dec     hex filename
   1953     632    4032    6617    19d9 reference
```

Observations :

- `text` (1953 octets) : le code machine des fonctions plus les données constantes.
- `data` (632 octets) : les variables globales initialisées, plus diverses métadonnées du runtime.
- `bss` (4032 octets) : principalement notre `tableau_zero` de 4000 octets, plus quelques variables internes du runtime.

Le point important : les 4032 octets de `.bss` n'occupent **aucune place** dans le fichier sur disque. Vous pouvez vérifier la taille du fichier :

```bash
ls -l ./reference
# La taille du fichier est bien inférieure à text + data + bss
```

> 🔥 **Point clé** — La distinction entre `.data` et `.bss` a un impact direct sur la taille des binaires et des images Docker. Si vous avez de grands tableaux ou buffers globaux initialisés à zéro, ils n'alourdissent pas le binaire. En revanche, si vous les initialisez avec des valeurs non nulles (même identiques), ils passent dans `.data` et augmentent la taille du fichier d'autant. En C++ moderne, cette situation est rare car on évite les variables globales au profit d'allocations locales ou dynamiques.

## La table des segments (*Program Header Table*)

La Program Header Table est la **vue exécution** du fichier ELF. Elle est lue par le noyau et par l'éditeur de liens dynamique pour charger le programme en mémoire. Elle n'est présente que dans les exécutables et les bibliothèques partagées (pas dans les fichiers objet `.o`).

### Structure d'une entrée de segment

Chaque entrée (*Program Header*) occupe 56 octets sur un système 64 bits :

| Champ | Taille | Description |
|-------|--------|-------------|
| `p_type` | 4 octets | Type du segment (voir tableau ci-dessous). |
| `p_flags` | 4 octets | Permissions : `PF_R` (lecture), `PF_W` (écriture), `PF_X` (exécution). |
| `p_offset` | 8 octets | Offset du segment dans le fichier sur disque. |
| `p_vaddr` | 8 octets | Adresse virtuelle où le segment doit être chargé en mémoire. |
| `p_paddr` | 8 octets | Adresse physique (ignorée sur les systèmes avec mémoire virtuelle). |
| `p_filesz` | 8 octets | Taille du segment dans le fichier sur disque. |
| `p_memsz` | 8 octets | Taille du segment en mémoire. |
| `p_align` | 8 octets | Alignement requis (doit être un multiple de la taille de page). |

### Différence entre `p_filesz` et `p_memsz`

La distinction entre ces deux champs est subtile mais cruciale. Pour la plupart des segments, `p_filesz` et `p_memsz` sont identiques : le contenu sur disque est copié tel quel en mémoire.

Mais pour le segment qui contient la section `.bss`, `p_memsz` est **supérieur** à `p_filesz`. La différence correspond aux données `.bss` : elles n'existent pas dans le fichier (d'où `p_filesz` plus petit), mais elles occupent de la mémoire (d'où `p_memsz` plus grand). Le chargeur comble la différence en allouant des pages remplies de zéros.

```
Segment LOAD (données) :
  p_filesz = 632 octets   ← taille sur disque (.data uniquement)
  p_memsz  = 4664 octets  ← taille en mémoire (.data + .bss)
                              La différence (4032 octets) = .bss, rempli de zéros
```

### Types de segments

| Type (`p_type`) | Description |
|-----------------|-------------|
| `PT_NULL` | Entrée inactive (ignorée). |
| `PT_LOAD` | Segment à charger en mémoire. C'est le type le plus important — il contient le code et les données. Un exécutable typique a deux segments `PT_LOAD` : un pour le code (R+X) et un pour les données (R+W). |
| `PT_DYNAMIC` | Contient les informations pour le chargeur dynamique (section `.dynamic`). Indique les bibliothèques requises, les tables de symboles dynamiques, etc. |
| `PT_INTERP` | Contient le chemin de l'interpréteur (éditeur de liens dynamique) — typiquement `/lib64/ld-linux-x86-64.so.2`. Le noyau lit ce segment pour savoir quel programme charger en premier. |
| `PT_NOTE` | Métadonnées auxiliaires : identifiant de build, version ABI, informations du système. |
| `PT_GNU_EH_FRAME` | Table d'informations pour le déroulement des exceptions C++ (*exception handling*). Permet au runtime de remonter la pile d'appels lors d'un `throw`. |
| `PT_GNU_STACK` | Indique les permissions de la pile. Par défaut, la pile est non-exécutable (RW sans X), ce qui protège contre certaines attaques par injection de code. |
| `PT_GNU_RELRO` | *Relocation Read-Only* : après le chargement et la résolution des relocations dynamiques, la zone mémoire correspondante est marquée en lecture seule. Protège la GOT contre l'écrasement par un attaquant. |

### Mapping sections → segments

Un segment regroupe une ou plusieurs sections partageant les mêmes permissions. Voici le mapping typique pour un exécutable :

```
Segment LOAD #1 (permissions R + X) :
  ├── .text        (code machine)
  ├── .rodata      (données constantes)
  ├── .plt         (trampolines d'appels dynamiques)
  └── .eh_frame    (informations d'exception handling)

Segment LOAD #2 (permissions R + W) :
  ├── .data        (variables initialisées)
  ├── .got         (Global Offset Table)
  ├── .got.plt     (GOT pour la PLT)
  └── .bss         (variables non-initialisées, p_memsz > p_filesz)
```

Le regroupement par permissions est dicté par le fonctionnement de la mémoire virtuelle du noyau Linux. La protection mémoire s'applique par **page** (4096 octets sur x86_64). Toutes les pages d'un même segment reçoivent les mêmes permissions. Séparer le code (exécutable, non modifiable) des données (modifiables, non exécutables) est une mesure de sécurité fondamentale : si un attaquant parvient à modifier des données, il ne peut pas les exécuter ; s'il parvient à détourner le flux d'exécution, il ne peut pas modifier le code.

> 💡 **Note** — L'option de linkage `-z separate-code` (activée par défaut dans les distributions modernes) garantit que les segments de code et de données sont placés dans des pages distinctes, sans chevauchement. Les options `-z relro` et `-z now` renforcent la protection en marquant la GOT en lecture seule après le chargement (*Full RELRO*). Ces protections sont détaillées en section 45.4.

## La section `.dynamic` — Pont entre les deux vues

La section `.dynamic` mérite une attention particulière car elle est à la croisée des deux vues. Elle est référencée à la fois comme **section** (dans la Section Header Table, consultable par les outils d'analyse) et comme **segment** de type `PT_DYNAMIC` (dans la Program Header Table, consulté par le chargeur dynamique).

Elle contient un tableau d'entrées clé-valeur qui pilotent le chargement dynamique :

| Tag | Description |
|-----|-------------|
| `DT_NEEDED` | Nom d'une bibliothèque partagée requise (une entrée par bibliothèque). |
| `DT_SONAME` | Nom canonique de la bibliothèque (pour les `.so` uniquement). |
| `DT_RPATH` / `DT_RUNPATH` | Chemins de recherche des bibliothèques encodés dans le binaire. |
| `DT_SYMTAB` | Adresse de la table des symboles dynamiques. |
| `DT_STRTAB` | Adresse de la table de chaînes dynamiques. |
| `DT_PLTGOT` | Adresse de la GOT (Global Offset Table). |
| `DT_INIT` | Adresse de la fonction d'initialisation (appelée avant `main`). |
| `DT_FINI` | Adresse de la fonction de finalisation (appelée après `main`). |

Les entrées `DT_INIT` et `DT_FINI` sont particulièrement pertinentes en C++. La fonction d'initialisation est responsable de l'appel des **constructeurs de variables globales** — les objets C++ déclarés en portée globale dont le constructeur doit s'exécuter avant `main()`. Symétriquement, la fonction de finalisation appelle les destructeurs de ces objets après le retour de `main()`.

## La chaîne de résolution des symboles dynamiques : GOT et PLT

Quand un exécutable lié dynamiquement appelle une fonction d'une bibliothèque partagée (comme `printf` de la libc), l'appel ne peut pas pointer directement vers l'adresse de la fonction — car cette adresse n'est pas connue au moment de la compilation et change à chaque exécution (grâce à l'ASLR). Deux structures ELF résolvent ce problème :

**La GOT (*Global Offset Table*)** est un tableau d'adresses dans la section `.got`. Chaque entrée correspond à un symbole externe (fonction ou variable). Au démarrage, le chargeur dynamique remplit la GOT avec les adresses réelles des symboles.

**La PLT (*Procedure Linkage Table*)** est un ensemble de petites fonctions trampolines dans la section `.plt`. Quand votre code appelle `printf`, il appelle en réalité une entrée de la PLT. La PLT consulte la GOT pour trouver l'adresse réelle et saute vers celle-ci.

```
Votre code (.text) :
  call printf@PLT          ← appel indirect via la PLT
         │
         ▼
PLT (.plt) :
  [entrée printf]
  jmp *GOT[printf]         ← lecture de l'adresse dans la GOT
         │
         ▼
GOT (.got.plt) :
  [adresse de printf]      ← remplie par ld-linux au chargement
         │
         ▼
libc.so (.text) :
  printf:                   ← le vrai code de printf
    push rbp
    ...
```

Sur les systèmes modernes avec **Full RELRO** (activé par défaut), toutes les entrées de la GOT sont résolues au chargement (pas de *lazy binding*), puis la GOT est marquée en lecture seule. Cela empêche un attaquant de modifier les adresses dans la GOT pour détourner le flux d'exécution.

> 💡 **Note** — Le mécanisme GOT/PLT explique le `@PLT` que vous avez vu dans la sortie assembleur de la section 1.3. Chaque appel à une fonction externe passe par ce mécanisme d'indirection. Le coût est minimal (un saut indirect supplémentaire) mais mesurable dans les boucles très serrées — c'est l'une des raisons pour lesquelles le linkage statique peut être plus performant que le linkage dynamique dans certains cas critiques.

## Synthèse : les trois composants et leurs relations

```
                    ┌──────────────────────┐
                    │     ELF Header       │
                    │  ┌────────────────┐  │
                    │  │ e_phoff ───────┼──┼──→ Program Header Table
                    │  │ e_shoff ───────┼──┼──→ Section Header Table
                    │  │ e_entry        │  │     (point d'entrée : _start)
                    │  │ e_shstrndx ────┼──┼──→ .shstrtab (noms des sections)
                    │  └────────────────┘  │
                    └──────────────────────┘
                              │
              ┌───────────────┴────────────────┐
              ▼                                ▼
   Program Header Table               Section Header Table
   (vue noyau/chargeur)               (vue linker/outils)
   ┌─────────────────┐                ┌──────────────────┐
   │ LOAD R+X ───────┼── regroupe ──→ │ .text            │
   │                 │                │ .rodata          │
   │                 │                │ .plt             │
   ├─────────────────┤                ├──────────────────┤
   │ LOAD R+W ───────┼── regroupe ──→ │ .data            │
   │                 │                │ .got             │
   │                 │                │ .bss             │
   ├─────────────────┤                ├──────────────────┤
   │ DYNAMIC ────────┼── correspond → │ .dynamic         │
   ├─────────────────┤                ├──────────────────┤
   │ INTERP ─────────┼── correspond → │ .interp          │
   ├─────────────────┤                ├──────────────────┤
   │ GNU_STACK       │                │ .symtab          │
   │ GNU_RELRO       │                │ .strtab          │
   │ NOTE            │                │ .debug_info      │
   └─────────────────┘                │ .debug_line      │
                                      │ ...              │
                                      └──────────────────┘
```

Les deux vues sont complémentaires et coexistent dans le même fichier. La Program Header Table dit au noyau **comment charger** le programme. La Section Header Table dit aux outils **ce que contient** le programme. Le noyau n'a pas besoin de la Section Header Table pour charger et exécuter un binaire — vous pouvez la supprimer avec `strip` sans affecter l'exécution. Inversement, un fichier objet (`.o`) n'a pas de Program Header Table car il n'est jamais chargé directement.

---

## Ce qu'il faut retenir

Le format ELF est construit autour de trois composants : un en-tête fixe qui identifie le fichier et pointe vers les deux tables, une table des sections (vue granulaire pour les outils et le linker) et une table des segments (vue mémoire pour le noyau). Les sections de code (`.text`) et de données constantes (`.rodata`) sont regroupées dans un segment exécutable non-modifiable. Les sections de données modifiables (`.data`, `.bss`) sont regroupées dans un segment modifiable non-exécutable. La section `.bss` illustre une optimisation élégante : les données initialisées à zéro n'occupent aucune place sur disque. Le mécanisme GOT/PLT gère l'indirection nécessaire aux appels de fonctions dynamiques, et la section `.dynamic` pilote le chargement des bibliothèques partagées.

La sous-section suivante (1.4.2) met tout cela en pratique avec les outils `readelf` et `objdump`.
