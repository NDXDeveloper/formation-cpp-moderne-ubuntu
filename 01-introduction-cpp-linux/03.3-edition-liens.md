## Introduction

Le compilateur a fait son travail : chaque fichier `.cpp` a été transformé en un fichier objet `.o` contenant du code machine. Mais ces fichiers objet sont des pièces de puzzle incomplètes. Le fichier `main.o` appelle une fonction `parse_config()` qui est définie dans `config.o`. Le fichier `config.o` utilise `std::vector`, dont l'implémentation se trouve dans la bibliothèque standard. Aucun de ces fichiers ne peut fonctionner seul — il faut assembler les pièces.

C'est le rôle de l'**éditeur de liens** (*linker*). Il prend l'ensemble des fichiers objet et des bibliothèques, résout les références croisées entre eux, et produit un binaire final : un exécutable ou une bibliothèque partagée. C'est la dernière étape du pipeline de compilation, et c'est aussi celle qui génère les erreurs les plus déroutantes pour les débutants.

---

## Le rôle fondamental du linker

### Résolution des symboles

Chaque fichier objet contient une **table des symboles** — un répertoire des noms (fonctions et variables globales) qu'il **définit** et des noms qu'il **utilise sans les définir**. Les symboles définis sont marqués comme *définis* (ou *exportés*). Les symboles utilisés mais non définis sont marqués comme *non définis* (*undefined*, abrégé `U`).

Reprenons l'outil `nm` pour observer cela sur un exemple concret. Supposons deux fichiers source :

```cpp
// math_utils.cpp
int ajouter(int a, int b) {
    return a + b;
}

int multiplier(int a, int b) {
    return a * b;
}
```

```cpp
// main.cpp
#include <cstdio>

int ajouter(int a, int b);     // déclaration (pas de définition)  
int multiplier(int a, int b);  // déclaration (pas de définition)  

int main() {
    int somme = ajouter(3, 4);
    int produit = multiplier(3, 4);
    printf("Somme: %d, Produit: %d\n", somme, produit);
    return 0;
}
```

Compilons chaque fichier séparément :

```bash
g++ -c main.cpp -o main.o  
g++ -c math_utils.cpp -o math_utils.o  
```

Inspectons les tables de symboles :

```bash
nm main.o
```

Sortie (simplifiée) :

```
                 U _Z7ajouterii       # ajouter(int, int) — non défini
0000000000000000 T main               # main — défini (section .text)
                 U _Z10multiplierii   # multiplier(int, int) — non défini
                 U printf             # printf — non défini
```

```bash
nm math_utils.o
```

Sortie (simplifiée) :

```
0000000000000000 T _Z7ajouterii       # ajouter(int, int) — défini
0000000000000014 T _Z10multiplierii   # multiplier(int, int) — défini
```

Le travail du linker est de **mettre en correspondance chaque symbole `U` (non défini) avec un symbole `T` (défini) dans un autre fichier objet ou dans une bibliothèque**. Pour `main.o`, le linker doit trouver les définitions de `ajouter`, `multiplier` (dans `math_utils.o`) et `printf` (dans la bibliothèque C standard, `libc`).

```bash
g++ main.o math_utils.o -o programme
```

Si la résolution réussit pour tous les symboles, le linker produit l'exécutable final. Si un symbole reste non résolu, il émet l'erreur la plus célèbre du C++ :

```
undefined reference to `fonction_manquante(int)'
```

### Le name mangling

Vous avez probablement remarqué les noms étranges dans la sortie de `nm` : `_Z7ajouterii` au lieu de `ajouter`. C'est le **name mangling** (*décoration de noms*), un mécanisme par lequel le compilateur C++ encode dans le nom du symbole des informations supplémentaires : le namespace, le nom de la classe (pour les méthodes), le nombre et le type des paramètres.

Le name mangling est nécessaire parce que le C++ autorise la **surcharge de fonctions** — plusieurs fonctions peuvent avoir le même nom si leurs paramètres diffèrent. Au niveau du linker, chaque symbole doit avoir un nom unique. Le mangling résout cette ambiguïté :

```cpp
int ajouter(int a, int b);         // → _Z7ajouterii  
double ajouter(double a, double b); // → _Z7ajouterdd  
```

Chaque compilateur a son propre schéma de mangling (GCC et Clang utilisent le même, défini par l'Itanium ABI ; MSVC utilise un schéma différent). L'outil `c++filt` permet de décoder un nom manglé en un nom lisible :

```bash
echo "_Z7ajouterii" | c++filt
# Sortie : ajouter(int, int)
```

Ou directement avec `nm` :

```bash
nm -C main.o    # -C : demangling automatique
```

> 💡 **Note** — La directive `extern "C"` désactive le name mangling pour les fonctions déclarées dans son bloc. C'est ce qui permet au C++ d'appeler des fonctions C (et inversement), car le C n'utilise pas le mangling :  
>
> ```cpp
> extern "C" {
>     int fonction_c(int x);  // symbole : fonction_c (pas de mangling)
> }
> ```
>  
> Ce mécanisme est détaillé dans la section 43.1 (C++ et C : extern "C" et ABI compatibility).

### Fusion des sections et relocation

Au-delà de la résolution des symboles, le linker effectue deux autres opérations essentielles :

**Fusion des sections.** Chaque fichier objet contient ses propres sections `.text`, `.data`, `.bss`, `.rodata`. Le linker les concatène pour former les sections finales de l'exécutable. Toutes les sections `.text` sont regroupées en une seule section `.text` dans le binaire final, et ainsi de suite :

```
main.o:        .text (code de main)         ─┐  
math_utils.o:  .text (code d'ajouter, ...)  ─┼──→  .text (exécutable final)  
libc.a:        .text (code de printf, ...)  ─┘  
```

**Relocation.** Pendant la compilation, le compilateur ne connaît pas l'adresse finale de chaque fonction et variable dans l'exécutable. Il utilise des adresses provisoires (souvent à partir de zéro) et enregistre des **entrées de relocation** qui indiquent au linker quels emplacements devront être ajustés. Le linker calcule les adresses finales, puis parcourt toutes les entrées de relocation pour insérer les bonnes adresses dans le code machine.

Par exemple, quand `main.o` appelle `ajouter()`, l'instruction d'appel contient une adresse provisoire. Le linker remplace cette adresse par l'adresse réelle de `ajouter()` dans la section `.text` fusionnée.

## Bibliothèques statiques et dynamiques

Le linker ne travaille pas uniquement avec vos fichiers objet. Il lie aussi votre code à des **bibliothèques** — des collections de fichiers objet précompilés empaquetés ensemble. Linux utilise deux types de bibliothèques, et la distinction entre les deux a des implications majeures sur le déploiement.

### Bibliothèques statiques (`.a`)

Une bibliothèque statique est une **archive de fichiers objet**, créée avec l'outil `ar` (*archiver*). Son extension est `.a` (*archive*).

```bash
# Création d'une bibliothèque statique
ar rcs libmath.a math_utils.o geometry.o algebra.o
```

Quand le linker utilise une bibliothèque statique, il **extrait les fichiers objet nécessaires** (ceux qui contiennent des symboles requis) et les incorpore directement dans l'exécutable final. Le résultat est un binaire autonome qui n'a aucune dépendance externe sur cette bibliothèque.

```bash
g++ main.o -L. -lmath -o programme
# -L.     : chercher les bibliothèques dans le répertoire courant
# -lmath  : lier avec libmath.a (ou libmath.so, voir la priorité ci-dessous)
```

**Avantages du linkage statique** : le binaire est autonome, facile à déployer (un seul fichier), pas de problème de versions de bibliothèques sur la machine cible. C'est l'approche privilégiée pour les conteneurs Docker minimaux et les environnements embarqués.

**Inconvénients** : la taille du binaire augmente (le code de la bibliothèque est dupliqué dans chaque exécutable qui l'utilise), et les mises à jour de la bibliothèque (correctifs de sécurité, bugs) nécessitent de recompiler et redéployer chaque exécutable.

### Bibliothèques dynamiques (partagées) (`.so`)

Une bibliothèque dynamique (*shared library* ou *shared object*) est un fichier binaire ELF avec l'extension `.so` (*shared object*). Contrairement à une bibliothèque statique, son code n'est **pas copié** dans l'exécutable. Le linker enregistre seulement une **référence** vers la bibliothèque, et le code est chargé en mémoire au moment de l'exécution par l'**éditeur de liens dynamique** (`ld-linux-x86-64.so.2`).

```bash
# Création d'une bibliothèque partagée
g++ -shared -fPIC -o libmath.so math_utils.o geometry.o algebra.o
```

Le flag `-fPIC` (*Position-Independent Code*) demande au compilateur de générer du code qui peut être chargé à n'importe quelle adresse en mémoire — une exigence pour les bibliothèques partagées.

**Avantages du linkage dynamique** : la bibliothèque est partagée en mémoire entre tous les processus qui l'utilisent (économie de RAM), les mises à jour de la bibliothèque ne nécessitent pas de recompiler les exécutables, et la taille des binaires est réduite.

**Inconvénients** : la bibliothèque doit être présente sur le système cible au moment de l'exécution, les problèmes de compatibilité de versions (le fameux *"DLL hell"* sous Windows, ou *"dependency hell"* sous Linux) peuvent survenir, et le chargement dynamique a un léger coût au démarrage.

### Priorité de résolution

Quand vous spécifiez `-lmath`, le linker cherche **d'abord** `libmath.so` (bibliothèque dynamique), **puis** `libmath.a` (bibliothèque statique). Pour forcer le linkage statique, utilisez l'option `-static` :

```bash
g++ main.o -static -lmath -o programme    # Force le linkage statique
```

Ou, pour un contrôle plus fin, spécifiez le chemin complet :

```bash
g++ main.o ./libmath.a -o programme        # Lie explicitement la version statique
```

> 🔥 **Point clé** — La distinction entre linkage statique et dynamique est fondamentale pour le déploiement. En contexte DevOps, le choix affecte directement la taille des images Docker, la gestion des dépendances en production, et la stratégie de mise à jour. Le sujet est approfondi en section 27.4 (Linkage statique vs dynamique).

## Vérifier les dépendances dynamiques

Quand un exécutable est lié dynamiquement, il est essentiel de pouvoir vérifier ses dépendances. Deux outils sont indispensables :

### `ldd` — Lister les dépendances

```bash
ldd ./programme
```

Sortie typique :

```
linux-vdso.so.1 (0x00007ffcc3bfe000)  
libstdc++.so.6 => /lib/x86_64-linux-gnu/libstdc++.so.6 (0x00007f2a1c000000)  
libm.so.6 => /lib/x86_64-linux-gnu/libm.so.6 (0x00007f2a1be00000)  
libgcc_s.so.1 => /lib/x86_64-linux-gnu/libgcc_s.so.1 (0x00007f2a1bdc0000)  
libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6 (0x00007f2a1ba00000)  
/lib64/ld-linux-x86-64.so.2 (0x00007f2a1c400000)
```

Chaque ligne montre le nom de la bibliothèque requise, le chemin où elle a été trouvée sur le système, et l'adresse à laquelle elle sera chargée en mémoire. Si une bibliothèque est introuvable, `ldd` affiche `not found` — c'est l'origine de l'erreur de chargement :

```
error while loading shared libraries: libcustom.so: cannot open shared object file:  
No such file or directory  
```

Les bibliothèques courantes d'un programme C++ sont :

| Bibliothèque | Rôle |
|-------------|------|
| `libc.so.6` | Bibliothèque C standard (printf, malloc, appels système…) |
| `libstdc++.so.6` | Bibliothèque standard C++ de GCC (conteneurs STL, streams, threads…) |
| `libm.so.6` | Fonctions mathématiques (sin, cos, sqrt…) |
| `libgcc_s.so.1` | Support runtime GCC (gestion des exceptions, arithmétique 128 bits…) |
| `libpthread.so.0` | Threads POSIX (souvent intégré dans libc sur les systèmes récents) |
| `ld-linux-x86-64.so.2` | L'éditeur de liens dynamique lui-même |

### `readelf -d` — En-têtes dynamiques

Pour une inspection plus détaillée que `ldd`, la commande `readelf -d` affiche les entrées de la section dynamique du binaire :

```bash
readelf -d ./programme
```

Cette commande montre les bibliothèques requises (`NEEDED`), le chemin de recherche des bibliothèques (`RPATH`, `RUNPATH`), et d'autres métadonnées du chargement dynamique.

## Le chemin de recherche des bibliothèques

Quand l'éditeur de liens dynamique charge un exécutable, il doit trouver chaque bibliothèque `.so` référencée. La recherche suit un ordre précis :

1. **`RPATH`** ou **`RUNPATH`** — un chemin encodé dans le binaire lui-même au moment de la compilation (via `-Wl,-rpath,/chemin`).
2. **`LD_LIBRARY_PATH`** — une variable d'environnement qui spécifie des répertoires de recherche supplémentaires.
3. **Le cache `ldconfig`** — un cache système maintenu par la commande `ldconfig`, basé sur les répertoires listés dans `/etc/ld.so.conf` et `/etc/ld.so.conf.d/`.
4. **Les répertoires par défaut** — `/lib`, `/usr/lib`, et leurs variantes 64 bits (`/lib/x86_64-linux-gnu/`, `/usr/lib/x86_64-linux-gnu/`).

Quand vous installez une nouvelle bibliothèque partagée sur le système, il faut souvent exécuter `sudo ldconfig` pour mettre à jour le cache et rendre la bibliothèque trouvable.

> 💡 **Note** — L'utilisation de `LD_LIBRARY_PATH` est commode pour le développement et les tests, mais elle est déconseillée en production car elle affecte globalement tous les programmes lancés dans le même environnement. Préférez `RUNPATH` (encodé dans le binaire) ou l'installation dans les répertoires standard du système.

## Les erreurs d'édition de liens

Les erreurs de linker sont parmi les plus fréquentes et les plus frustrantes pour les développeurs C++. Contrairement aux erreurs de compilation, elles ne pointent pas vers une ligne précise du code source — elles concernent des **symboles**, c'est-à-dire des noms de fonctions ou de variables.

### `undefined reference to`

C'est l'erreur la plus courante. Elle signifie qu'un symbole est utilisé (un appel de fonction, un accès à une variable) mais que sa définition n'a été trouvée dans aucun fichier objet ni bibliothèque fourni au linker.

```
/usr/bin/ld: main.o: in function `main':
main.cpp:(.text+0x15): undefined reference to `parse_config(std::string const&)'  
collect2: error: ld returned 1 exit status  
```

**Causes fréquentes et diagnostic :**

**Un fichier objet manquant dans la commande de liaison.** La fonction est définie dans `config.cpp`, mais `config.o` n'a pas été passé au linker. Vérifiez que tous les fichiers `.o` nécessaires sont inclus.

**Une bibliothèque manquante.** La fonction provient d'une bibliothèque externe qui n'a pas été liée. Par exemple, si vous utilisez des fonctions mathématiques de `<cmath>` comme `sqrt`, il peut être nécessaire d'ajouter `-lm` sur certaines configurations.

**Une déclaration sans définition.** Le fichier d'en-tête déclare la fonction (`int parse_config(const std::string& path);`) mais aucun fichier `.cpp` ne la définit. Vérifiez que le fichier source contenant la définition existe et est compilé.

**Un problème de signature.** La déclaration dans le `.h` ne correspond pas exactement à la définition dans le `.cpp` (types de paramètres différents, `const` manquant, namespace différent). Le mangling produit des noms de symboles différents, et le linker ne les met pas en correspondance. Utilisez `nm -C` pour comparer les symboles dans les différents fichiers objet.

**Oubli de `extern "C"` pour du code C.** Si vous appelez une fonction C depuis du C++ sans `extern "C"`, le compilateur applique le mangling au nom, et le linker cherche un symbole manglé qui n'existe pas dans la bibliothèque C.

### `multiple definition of`

L'erreur inverse : un symbole est défini dans **plusieurs** fichiers objet.

```
/usr/bin/ld: config.o: in function `helper()':
config.cpp:(.text+0x0): multiple definition of `helper()';  
main.o:main.cpp:(.text+0x0): first defined here  
```

**Causes fréquentes :**

**Une fonction définie dans un fichier d'en-tête** (pas seulement déclarée). Si le `.h` contient le corps de la fonction et qu'il est inclus dans plusieurs `.cpp`, chaque unité de compilation contient une copie de la définition, et le linker voit des doublons. La solution est de déclarer la fonction `inline` (ce qui autorise les définitions multiples identiques) ou de déplacer la définition dans un seul fichier `.cpp`.

**Une variable globale définie dans un en-tête.** Même problème : une variable `int compteur = 0;` dans un `.h` inclus par plusieurs `.cpp` produit plusieurs définitions. La solution en C++17 est d'utiliser `inline` :

```cpp
// header.h
inline int compteur = 0;  // C++17 : une seule instance, définitions multiples autorisées
```

### Erreurs de chargement dynamique

Ces erreurs se manifestent non pas à la compilation mais à l'**exécution** :

```bash
./programme
```

```
./programme: error while loading shared libraries: libcustom.so:
cannot open shared object file: No such file or directory
```

Le binaire a été lié avec une bibliothèque partagée qui n'est pas trouvable sur le système au moment de l'exécution.

**Diagnostic :**

```bash
ldd ./programme          # Identifier la bibliothèque manquante  
find / -name "libcustom.so*" 2>/dev/null   # La chercher sur le système  
```

**Solutions :**

```bash
# 1. Installer la bibliothèque et mettre à jour le cache
sudo cp libcustom.so /usr/local/lib/  
sudo ldconfig  

# 2. Ou spécifier le chemin temporairement
LD_LIBRARY_PATH=/chemin/vers/lib ./programme

# 3. Ou encoder le chemin dans le binaire à la compilation
g++ main.o -lcustom -Wl,-rpath,/chemin/vers/lib -o programme
```

## Les linkers disponibles sur Linux

Le linker par défaut sur Ubuntu est **GNU ld** (invoqué automatiquement par `g++`). Mais il existe des alternatives plus rapides, ce qui est pertinent car l'édition de liens peut devenir un goulot d'étranglement sur les gros projets :

**GNU ld** — Le linker historique. Fiable, mature, mais relativement lent sur les gros projets (des dizaines de milliers de fichiers objet).

**GNU gold** — Un linker alternatif développé par Google, conçu pour être plus rapide que GNU ld. Il fait partie du paquet `binutils` et peut être sélectionné avec `-fuse-ld=gold`.

**mold** — Un linker moderne, open-source, créé par Rui Ueyama (qui avait déjà créé le linker `lld` de LLVM). `mold` est significativement plus rapide que tous ses prédécesseurs — souvent d'un facteur 5 à 10 sur les gros projets. Sur un projet conséquent, la phase de liaison qui prenait 30 secondes avec GNU ld peut descendre à 3 secondes avec `mold`.

```bash
# Installation de mold
sudo apt install mold

# Utilisation avec g++ ou clang++
g++ main.o utils.o -fuse-ld=mold -o programme
```

**lld** — Le linker du projet LLVM, conçu comme compagnon de Clang. Il est plus rapide que GNU ld et gold, comparable à mold sur certains projets.

```bash
# Utilisation avec clang++
clang++ main.o utils.o -fuse-ld=lld -o programme
```

> 💡 **Note** — Le choix du linker est transparent pour votre code source — les quatre linkers produisent des exécutables fonctionnellement identiques. La différence est purement dans le temps de liaison. Sur un petit projet, la différence est négligeable. Sur un projet de plusieurs millions de lignes, le choix du linker peut économiser des minutes à chaque build. L'intégration du linker avec CMake est traitée en section 26.

## Visualiser le processus complet

Pour conclure cette sous-section, reconstituons le processus complet avec notre exemple de deux fichiers :

```bash
# 1. Compilation séparée (préprocesseur + compilation + assemblage)
g++ -std=c++23 -Wall -Wextra -c main.cpp -o main.o  
g++ -std=c++23 -Wall -Wextra -c math_utils.cpp -o math_utils.o  

# 2. Édition de liens
g++ main.o math_utils.o -o programme

# 3. Vérification du résultat
file programme           # Type du binaire (ELF, exécutable, dynamiquement lié)  
nm -C programme          # Table des symboles (noms démanglés)  
ldd programme            # Dépendances dynamiques  
./programme              # Exécution
```

Le flux complet ressemble à ceci :

```
main.cpp ──→ [préprocesseur] ──→ [compilateur] ──→ [assembleur] ──→ main.o ─────────────┐
                                                                                        │
math_utils.cpp ──→ [préprocesseur] ──→ [compilateur] ──→ [assembleur] ──→ math_utils.o ─┤
                                                                                        │
libc.so (bibliothèque C standard) ──────────────────────────────────────────────────────┤  
libstdc++.so (bibliothèque C++ standard) ───────────────────────────────────────────────┤  
                                                                                        │
                                                                                  [linker (ld)]
                                                                                        │
                                                                                        ▼
                                                                                    programme
                                                                                 (exécutable ELF)
```

## Ce qu'il faut retenir

L'éditeur de liens est la dernière étape du pipeline de compilation. Il assemble les fichiers objet et les bibliothèques en un binaire final en résolvant les références croisées entre symboles. Le C++ utilise le name mangling pour encoder les informations de surcharge dans les noms de symboles, et l'outil `c++filt` (ou `nm -C`) permet de les décoder. Les bibliothèques peuvent être liées statiquement (code copié dans le binaire) ou dynamiquement (code chargé à l'exécution), avec des implications directes sur le déploiement. Les erreurs de linker (`undefined reference`, `multiple definition`) ne pointent pas vers des lignes de code mais vers des symboles, et leur diagnostic passe par l'inspection des tables de symboles avec `nm`. Enfin, le choix du linker (`ld`, `gold`, `lld`, `mold`) peut significativement réduire les temps de build sur les gros projets.

La section suivante (1.4) descend d'un niveau pour examiner le format du binaire produit par le linker : le format ELF, le standard de Linux pour les exécutables, les bibliothèques partagées et les fichiers objet.
