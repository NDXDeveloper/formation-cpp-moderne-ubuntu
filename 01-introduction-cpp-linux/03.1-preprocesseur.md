# 1.3.1 — Le préprocesseur : #include, #define, macros

> **Section parente** : [1.3 — Le cycle de compilation](./03-cycle-compilation.md)  
> **Niveau** : Débutant  
> **Durée estimée** : 25 à 35 minutes

## Introduction

Le préprocesseur est la première étape de la chaîne de compilation. C'est aussi la plus ancienne : il est hérité directement du langage C et n'a pratiquement pas évolué depuis les années 1970. Son principe est simple en apparence — **transformer du texte avant que le compilateur ne le voie** — mais ses implications sont profondes et parfois piégeuses.

Le préprocesseur ne comprend pas le C++. Il ne connaît ni les types, ni les classes, ni les namespaces, ni la portée des variables. Il manipule le texte du fichier source de manière purement mécanique : il copie, remplace, inclut et exclut des blocs de texte selon des règles définies par les **directives préprocesseur** — ces lignes qui commencent par le caractère `#`.

Cette section détaille les trois grandes familles de directives : l'inclusion de fichiers (`#include`), la définition de macros (`#define`), et la compilation conditionnelle (`#ifdef`, `#if`, etc.).

---

## `#include` — L'inclusion de fichiers

### Le mécanisme

La directive `#include` est l'instruction la plus utilisée du préprocesseur. Son fonctionnement est d'une simplicité brutale : **elle remplace la ligne `#include` par le contenu intégral du fichier référencé**, tel quel, caractère pour caractère.

```cpp
// main.cpp
#include <iostream>

int main() {
    std::cout << "Hello\n";
    return 0;
}
```

Quand le préprocesseur traite ce fichier, il cherche le fichier `iostream` dans les répertoires d'en-têtes du système, lit son contenu complet (qui inclut lui-même d'autres fichiers via d'autres `#include`), et insère tout ce texte à la place de la ligne `#include <iostream>`. Le résultat prétraité peut facilement dépasser **50 000 lignes** pour un simple programme de 5 lignes, simplement parce que les en-têtes de la bibliothèque standard sont volumineux et interconnectés.

Vous pouvez observer ce phénomène directement :

```bash
g++ -E main.cpp | wc -l
```

### Les deux formes de `#include`

Il existe deux syntaxes, et la différence n'est pas cosmétique :

**`#include <fichier>`** — Chevrons. Le préprocesseur cherche le fichier dans les **répertoires système** (typiquement `/usr/include`, `/usr/include/c++/14/`, etc.) et dans les répertoires spécifiés par l'option `-isystem`. C'est la forme utilisée pour les en-têtes de la bibliothèque standard et des bibliothèques tierces installées sur le système.

**`#include "fichier"`** — Guillemets. Le préprocesseur cherche d'abord dans le **répertoire du fichier source courant**, puis, s'il ne trouve pas, dans les répertoires spécifiés par l'option `-I`, et enfin dans les répertoires système. C'est la forme utilisée pour les en-têtes de votre propre projet.

```cpp
#include <vector>           // en-tête système (STL)
#include <nlohmann/json.hpp> // bibliothèque tierce installée
#include "config.h"          // en-tête du projet
#include "utils/parser.h"    // en-tête du projet, sous-répertoire
```

L'option `-I` de GCC et Clang permet d'ajouter des répertoires de recherche pour les `#include` avec guillemets :

```bash
g++ -I./include -I../libs/json/include -c main.cpp
```

> 💡 **Note** — La distinction chevrons/guillemets est une convention et un mécanisme de recherche, pas une garantie de sécurité. Un `#include "vector"` fonctionnera aussi (le fichier sera trouvé dans les répertoires système après avoir échoué dans le répertoire local), mais c'est une mauvaise pratique qui brouille l'intention du programmeur.

### Le problème de l'inclusion multiple

Puisque `#include` est une copie textuelle pure, rien n'empêche un même fichier d'être inclus plusieurs fois dans la même unité de compilation. Considérons cette situation :

```
main.cpp  →  #include "a.h"
              #include "b.h"

a.h       →  #include "types.h"

b.h       →  #include "types.h"
```

Le fichier `types.h` est inclus deux fois dans `main.cpp` : une fois via `a.h` et une fois via `b.h`. Si `types.h` contient des définitions (une classe, une structure, une énumération), le compilateur verra ces définitions deux fois et signalera une erreur de redéfinition.

La solution classique est le **garde d'inclusion** (*include guard*) :

```cpp
// types.h
#ifndef TYPES_H
#define TYPES_H

struct Point {
    double x;
    double y;
};

#endif // TYPES_H
```

Le mécanisme est le suivant. La première fois que `types.h` est inclus, la macro `TYPES_H` n'est pas définie, donc `#ifndef TYPES_H` est vrai, le contenu est inclus, et la macro est définie par `#define TYPES_H`. La deuxième fois que le fichier est inclus, `TYPES_H` est déjà définie, donc `#ifndef` est faux et tout le contenu est ignoré.

L'alternative moderne est la directive `#pragma once`, supportée par tous les compilateurs majeurs (GCC, Clang, MSVC) bien qu'elle ne fasse pas partie du standard ISO C++ :

```cpp
// types.h
#pragma once

struct Point {
    double x;
    double y;
};
```

`#pragma once` demande au préprocesseur de n'inclure le fichier qu'une seule fois, sans avoir besoin de définir manuellement une macro de garde. Elle est plus concise et évite les collisions de noms de macros.

> 🔥 **Point clé** — Chaque fichier d'en-tête `.h` ou `.hpp` de votre projet doit avoir un garde d'inclusion (`#ifndef`/`#define`/`#endif`) ou un `#pragma once`. Oublier cette protection est l'une des erreurs les plus fréquentes chez les débutants et produit des erreurs de compilation confuses.

### L'impact sur les temps de compilation

L'inclusion textuelle a un coût direct et mesurable sur les temps de compilation. Chaque fichier `.cpp` qui inclut `<iostream>` demande au préprocesseur de copier et au compilateur d'analyser des dizaines de milliers de lignes d'en-têtes. Si votre projet contient 200 fichiers `.cpp` qui incluent chacun les mêmes en-têtes, ce travail est répété 200 fois.

Plusieurs techniques atténuent ce problème :

- **Les en-têtes précompilés** (*Precompiled Headers*, PCH) : le compilateur analyse les en-têtes une seule fois et stocke le résultat sous forme binaire, réutilisable par tous les fichiers source.
- **Les modules C++20** : le remplacement à long terme du système `#include`. Les modules sont compilés une seule fois et exportent des interfaces binaires sans copie textuelle. Leur adoption progresse mais reste incomplète en 2026 (voir section 12.13).
- **Le cache de compilation** : `ccache` (section 2.3) détecte quand la sortie de la compilation est identique à une exécution précédente et réutilise le résultat mis en cache.

## `#define` — Définition de macros

### Macros constantes

La forme la plus simple de `#define` crée une **substitution textuelle** : chaque occurrence de l'identifiant est remplacée par le texte de remplacement.

```cpp
#define MAX_CONNECTIONS 1024
#define VERSION "3.2.1"
#define PI 3.14159265358979323846
```

Après le passage du préprocesseur, toute occurrence de `MAX_CONNECTIONS` dans le code sera remplacée par `1024`, textuellement. Le compilateur ne verra jamais le nom `MAX_CONNECTIONS` — il ne verra que le nombre `1024`.

> 🔥 **Point clé — En C++ moderne, les macros constantes sont largement obsolètes.** Préférez `constexpr` pour les valeurs numériques et `std::string_view` ou les constantes `const` pour les chaînes :  
>
> ```cpp
> // ❌ Style hérité (C / C++98)
> #define MAX_CONNECTIONS 1024
> #define VERSION "3.2.1"
>
> // ✅ Style C++ moderne
> constexpr int max_connections = 1024;
> constexpr std::string_view version = "3.2.1";
> ```
>  
> Les constantes `constexpr` sont typées, respectent les namespaces et les portées, et sont visibles par le debugger. Les macros ne bénéficient d'aucun de ces avantages.

### Macros avec paramètres (macros fonctions)

`#define` peut aussi créer des macros paramétrées, qui ressemblent à des fonctions mais n'en sont pas :

```cpp
#define CARRE(x) ((x) * (x))
#define MIN(a, b) ((a) < (b) ? (a) : (b))
```

Utilisation :

```cpp
int r = CARRE(5);       // → int r = ((5) * (5));  
int m = MIN(a + 1, b);  // → int m = ((a + 1) < (b) ? (a + 1) : (b));  
```

Les parenthèses autour de chaque paramètre et autour de l'expression complète sont **indispensables**. Sans elles, la substitution textuelle peut produire des résultats catastrophiques :

```cpp
// ❌ Macro mal parenthésée
#define CARRE_MAUVAIS(x) x * x

int r = CARRE_MAUVAIS(3 + 1);
// Expansion : int r = 3 + 1 * 3 + 1;
// Résultat  : r = 7  (au lieu de 16 !)
// Raison    : la multiplication a priorité sur l'addition
```

Les macros paramétrées posent d'autres problèmes subtils :

**Évaluation multiple des arguments.** Contrairement à une vraie fonction, les arguments d'une macro sont substitués textuellement et peuvent être évalués plusieurs fois :

```cpp
#define MIN(a, b) ((a) < (b) ? (a) : (b))

int x = MIN(compteur++, seuil);
// Expansion : ((compteur++) < (seuil) ? (compteur++) : (seuil))
// compteur est incrémenté deux fois si compteur < seuil !
```

**Absence de typage.** Le préprocesseur ne vérifie pas les types. Une macro `MIN` accepte n'importe quoi, y compris des arguments de types incompatibles, et l'erreur ne sera détectée qu'à la compilation du code expansé, avec un message d'erreur qui pointe vers la ligne d'utilisation de la macro, pas vers sa définition.

> 🔥 **Point clé — En C++ moderne, les macros paramétrées sont presque toujours remplaçables** par des fonctions `constexpr`, des fonctions `inline`, ou des templates :  
>
> ```cpp
> // ❌ Macro
> #define CARRE(x) ((x) * (x))
>
> // ✅ Fonction constexpr (évaluable à la compilation ET à l'exécution)
> constexpr auto carre(auto x) { return x * x; }
>
> // ✅ Ou template explicite pour C++17 et antérieur
> template<typename T>
> constexpr T carre(T x) { return x * x; }
> ```
>  
> La fonction `constexpr` est typée, n'évalue chaque argument qu'une seule fois, respecte les portées et les namespaces, et produit des messages d'erreur clairs.

### Macros multi-lignes

Une macro peut s'étendre sur plusieurs lignes grâce au caractère de continuation `\` en fin de ligne :

```cpp
#define LOG_ERROR(msg) \
    do { \
        std::cerr << "[ERROR] " << __FILE__ << ":" << __LINE__ \
                  << " - " << msg << std::endl; \
    } while(0)
```

L'idiome `do { ... } while(0)` est une astuce classique qui garantit que la macro se comporte comme une instruction unique, même dans un `if` sans accolades. C'est une convention héritée du C que vous rencontrerez fréquemment dans le code existant.

### `#undef` — Annulation d'une macro

La directive `#undef` supprime la définition d'une macro :

```cpp
#define BUFFER_SIZE 4096
// ... code utilisant BUFFER_SIZE ...
#undef BUFFER_SIZE
// À partir d'ici, BUFFER_SIZE n'est plus défini
```

`#undef` est parfois utilisé pour éviter les conflits entre des macros définies dans des en-têtes différents, ou pour redéfinir une macro avec une valeur différente.

### Macros prédéfinies

Le compilateur et le préprocesseur définissent automatiquement un certain nombre de macros utiles :

| Macro | Description | Exemple de valeur |
|-------|-------------|-------------------|
| `__FILE__` | Nom du fichier source courant | `"main.cpp"` |
| `__LINE__` | Numéro de ligne courant | `42` |
| `__func__` | Nom de la fonction courante (C++11) | `"main"` |
| `__DATE__` | Date de compilation | `"Mar  9 2026"` |
| `__TIME__` | Heure de compilation | `"14:30:00"` |
| `__cplusplus` | Version du standard C++ | `202602L` (C++26) |
| `__GNUC__` | Version majeure de GCC | `15` |
| `__clang__` | Défini si le compilateur est Clang | `1` |

La macro `__cplusplus` est particulièrement utile pour écrire du code qui s'adapte au standard utilisé :

```cpp
#if __cplusplus >= 202302L
    // Code spécifique C++23 ou plus
    #include <print>
#else
    // Fallback pour standards antérieurs
    #include <iostream>
#endif
```

### L'opérateur `#` (stringification) et `##` (concaténation)

Le préprocesseur dispose de deux opérateurs spéciaux utilisables dans les macros :

**`#` (stringification)** — Transforme un argument de macro en chaîne de caractères littérale :

```cpp
#define TO_STRING(x) #x

std::cout << TO_STRING(42 + 3);    // Affiche : 42 + 3  
std::cout << TO_STRING(hello);     // Affiche : hello  
```

**`##` (concaténation de tokens)** — Fusionne deux tokens en un seul :

```cpp
#define MAKE_VAR(prefix, num) prefix##num

int MAKE_VAR(value, 1) = 10;   // → int value1 = 10;  
int MAKE_VAR(value, 2) = 20;   // → int value2 = 20;  
```

Ces opérateurs sont principalement rencontrés dans du code existant et dans certaines bibliothèques qui les utilisent pour la génération de code. En C++ moderne, les templates et `constexpr` remplacent la majorité de ces usages.

## Compilation conditionnelle

### Principe

Les directives de compilation conditionnelle permettent d'**inclure ou d'exclure des portions de code** avant que le compilateur ne les voie. Le texte exclu n'existe tout simplement pas dans le fichier prétraité — il n'est ni compilé, ni analysé, ni vérifié syntaxiquement.

### `#ifdef` / `#ifndef` / `#endif`

La forme la plus courante teste si une macro est définie ou non :

```cpp
#ifdef DEBUG
    std::cerr << "Mode debug activé" << std::endl;
#endif

#ifndef NDEBUG
    assert(ptr != nullptr);
#endif
```

La macro peut être définie dans le code (`#define DEBUG`) ou sur la ligne de commande du compilateur avec l'option `-D` :

```bash
g++ -DDEBUG -c main.cpp          # Définit la macro DEBUG  
g++ -DNDEBUG -c main.cpp         # Définit la macro NDEBUG  
g++ -DMAX_SIZE=2048 -c main.cpp  # Définit MAX_SIZE avec la valeur 2048  
```

### `#if` / `#elif` / `#else` / `#endif`

La forme étendue permet des conditions numériques et des expressions :

```cpp
#if __cplusplus >= 202302L
    // Utiliser les fonctionnalités C++23
    std::println("C++23 ou plus récent");
#elif __cplusplus >= 202002L
    // Fallback C++20
    std::cout << std::format("C++20\n");
#elif __cplusplus >= 201703L
    // Fallback C++17
    std::cout << "C++17" << std::endl;
#else
    // Standard antérieur
    #error "Ce projet nécessite au minimum C++17"
#endif
```

La directive `#error` interrompt la compilation avec un message d'erreur explicite. Elle est utile pour signaler qu'une configuration non supportée a été détectée.

### Cas d'usage légitimes en 2026

La compilation conditionnelle reste pertinente dans plusieurs situations :

- **Portabilité entre systèmes d'exploitation** : adapter le code aux API de Linux, macOS ou Windows.
- **Portabilité entre compilateurs** : contourner les différences entre GCC, Clang et MSVC.
- **Adaptation au standard C++** : proposer des implémentations différentes selon la version du standard disponible.
- **Modes de compilation** : activer du code de debug, de profiling ou de test uniquement dans certaines configurations.
- **Feature flags** : activer ou désactiver des fonctionnalités au moment de la compilation.

## Les limites et les dangers du préprocesseur

Le préprocesseur est un outil puissant mais primitif. Ses principales faiblesses, qu'il est crucial de connaître, expliquent pourquoi le C++ moderne cherche à limiter son usage :

**Pas de notion de portée.** Une macro définie dans un fichier d'en-tête est visible dans **tous** les fichiers qui l'incluent, directement ou indirectement. Elle peut entrer en conflit avec des noms de variables, de fonctions ou de classes définis ailleurs. Un exemple tristement célèbre : l'en-tête `<windows.h>` définit des macros `min` et `max` qui entrent en conflit avec `std::min` et `std::max`.

**Pas de typage.** Le préprocesseur ne vérifie aucun type. Les erreurs de type ne seront détectées que par le compilateur, dans le code expansé, avec des messages d'erreur qui peuvent être très difficiles à interpréter.

**Débogage obscurci.** Le code que vous voyez dans votre éditeur n'est pas nécessairement le code que le compilateur traite. Les macros complexes rendent le débogage difficile car le debugger montre le code expansé, qui peut être très différent du code source original.

**Pas de namespaces.** Les macros vivent dans un espace de noms global et plat. Il n'y a aucun moyen de les encapsuler dans un namespace.

## Bonnes pratiques en C++ moderne

Le préprocesseur reste nécessaire pour certains usages (gardes d'inclusion, compilation conditionnelle, interopérabilité avec du code C), mais le C++ moderne offre des alternatives supérieures pour la majorité des cas :

| Usage | Préprocesseur (hérité) | C++ moderne (préféré) |
|-------|----------------------|----------------------|
| Constantes | `#define PI 3.14` | `constexpr double pi = 3.14;` |
| Fonctions simples | `#define CARRE(x) ((x)*(x))` | `constexpr auto carre(auto x) { return x*x; }` |
| Conditions sur les types | `#ifdef` + macros | `if constexpr` (C++17) |
| Inclusion de code | `#include` | `import` (modules C++20, adoption progressive) |
| Assertions compile-time | `#if ... #error` | `static_assert` (C++11) |

> 🔥 **Règle d'or** — Utilisez le préprocesseur uniquement quand il n'existe pas d'alternative C++ satisfaisante. Pour tout le reste, préférez les mécanismes du langage (`constexpr`, `consteval`, `if constexpr`, `static_assert`, templates). Ces mécanismes sont typés, respectent les portées, et produisent des erreurs compréhensibles.

## Ce qu'il faut retenir

Le préprocesseur est un héritage du C qui reste présent dans chaque compilation C++. Il opère par substitution textuelle avant que le compilateur n'entre en jeu. Ses trois mécanismes principaux — l'inclusion de fichiers (`#include`), la définition de macros (`#define`) et la compilation conditionnelle (`#ifdef`, `#if`) — sont des outils que vous rencontrerez quotidiennement, que ce soit dans votre propre code ou dans celui des bibliothèques que vous utilisez. Maîtriser leur fonctionnement vous permet de comprendre les erreurs qu'ils produisent et, surtout, de savoir quand les remplacer par les alternatives modernes du langage.
