# 48.4.4 — Compiler Explorer (godbolt.org)

## Section 48.4 : Communautés et forums

---

## Présentation

Compiler Explorer — universellement connu sous le nom de **godbolt.org** d'après son créateur Matt Godbolt — est un outil en ligne gratuit qui permet de compiler du code C++ en temps réel contre des dizaines de compilateurs dans des centaines de versions, et de visualiser le résultat : code assembleur généré, diagnostics du compilateur, sortie d'exécution, et bien plus encore. C'est l'outil le plus utilisé et le plus cité de l'écosystème C++, à tel point qu'un "lien godbolt" est devenu le standard de facto pour illustrer un point technique dans n'importe quelle discussion C++.

Compiler Explorer n'est pas un forum ni une communauté à proprement parler, mais il est si profondément intégré dans la pratique communautaire — sur Stack Overflow, Reddit, Discord, dans les talks de conférence, dans les code reviews — qu'il constitue une pièce indispensable de l'infrastructure sociale et technique du C++.

**URL** : [godbolt.org](https://godbolt.org) (aussi accessible via [compiler-explorer.com](https://compiler-explorer.com))  
**Code source** : [github.com/compiler-explorer/compiler-explorer](https://github.com/compiler-explorer/compiler-explorer) (open source, licence BSD)  
**Créateur** : Matt Godbolt  

---

## Origine et philosophie

Matt Godbolt a créé Compiler Explorer en 2012 alors qu'il travaillait dans le trading haute fréquence, un domaine où chaque cycle CPU compte. Son besoin initial était simple : vérifier que le compilateur générait bien le code assembleur attendu pour un morceau de C++ donné, sans avoir à mettre en place manuellement une chaîne de compilation et de désassemblage. L'outil a été mis en ligne et a rapidement été adopté par la communauté entière.

La philosophie de Compiler Explorer se résume en un principe : **rendre le comportement du compilateur observable et partageable**. Avant godbolt.org, vérifier le code assembleur généré par un compilateur demandait d'installer le compilateur, de configurer les flags, de compiler en mode assembleur, et de lire le fichier `.s` résultant. Compiler Explorer réduit cette opération à quelques secondes dans un navigateur web. Et surtout, il permet de **partager** le résultat via un simple lien — ce qui a transformé la façon dont la communauté C++ communique sur les questions de performance, de comportement du langage et de support compilateur.

Son talk CppCon "What Has My Compiler Done for Me Lately?" (mentionné en section 48.2.1) est devenu un classique qui montre comment utiliser l'outil pour comprendre les optimisations du compilateur.

## Fonctionnalités principales

### Compilation multi-compilateur

C'est la fonctionnalité fondatrice. Compiler Explorer donne accès à un nombre impressionnant de compilateurs et de versions :

- **GCC** — des versions anciennes (GCC 4.x) jusqu'à GCC 15 et aux snapshots trunk  
- **Clang** — de Clang 3.x jusqu'à Clang 20 et trunk  
- **MSVC** — plusieurs versions de Visual C++  
- **ICC/ICX** — les compilateurs Intel  
- **Forks spécialisés** — des forks de Clang avec des fonctionnalités expérimentales, comme l'implémentation de la réflexion C++26 par Bloomberg ou EDG

Chaque compilateur peut être configuré avec des flags arbitraires (`-std=c++26`, `-O2`, `-Wall`, `-fsanitize=address`, etc.). Plusieurs compilateurs peuvent être affichés côte à côte pour comparer le code généré — par exemple, comparer la sortie de GCC 15 et Clang 20 pour le même code, ou comparer `-O0` et `-O3` sur le même compilateur.

### Visualisation de l'assembleur

Le panneau de sortie affiche le code assembleur généré, avec un **mapping couleur** entre les lignes de code source et les instructions assembleur correspondantes. En survolant une ligne de code C++, les instructions assembleur qui en résultent sont surlignées, et vice versa. Cette correspondance visuelle rend la lecture de l'assembleur accessible même aux développeurs qui ne maîtrisent pas couramment les jeux d'instructions x86, ARM ou RISC-V.

Des filtres permettent de masquer les directives d'assemblage, les labels non pertinents et les commentaires du compilateur pour ne garder que les instructions significatives. Le résultat est un affichage lisible qui montre immédiatement si une fonction a été inlinée, si une boucle a été vectorisée, ou si une abstraction a généré du code machine supplémentaire.

### Exécution du code

Au-delà de la compilation, Compiler Explorer peut **exécuter** le programme et afficher sa sortie standard. C'est un ajout qui transforme l'outil d'un simple visualiseur d'assembleur en un véritable environnement de test en ligne. On peut écrire un programme complet avec un `main()`, le compiler et l'exécuter, le tout dans le navigateur.

L'exécution est sandboxée avec des limites de temps et de mémoire pour des raisons de sécurité, mais les contraintes sont suffisamment généreuses pour tester la plupart des snippets raisonnables.

### Outils d'analyse intégrés

Compiler Explorer intègre plusieurs outils d'analyse en plus de la compilation et de l'exécution :

- **Clang-Tidy** — Analyse statique en ligne, directement sur le code saisi. Les warnings et suggestions de clang-tidy apparaissent dans un panneau dédié (voir section 32.1).  
- **Sanitizers** — Compilation avec `-fsanitize=address`, `-fsanitize=undefined`, etc. et visualisation des diagnostics à l'exécution (voir section 29.4).  
- **opt-viewer** — Visualisation des décisions d'optimisation du compilateur (quelles fonctions ont été inlinées, quelles boucles vectorisées, et pourquoi).  
- **Include tree** — Arbre des inclusions de headers, utile pour diagnostiquer les temps de compilation.  
- **Différence de code généré** — Comparaison diff entre le code assembleur de deux compilateurs ou de deux versions de votre code, pour mesurer l'impact d'une modification.

### Liens partageables et short links

Chaque configuration — code source, choix du compilateur, flags, outils actifs — peut être sauvegardée et partagée via un **short link** (une URL courte commençant par `godbolt.org/z/`). Ces liens encapsulent l'intégralité de l'état de l'éditeur et sont stables dans le temps : un lien créé aujourd'hui restera fonctionnel dans un an.

Les short links sont devenus le moyen standard de communication technique dans la communauté C++. Sur Stack Overflow, Reddit, Discord, dans les issues GitHub, et même dans les papers du comité de standardisation, un lien godbolt vaut mille mots. Il dit : "Voici le code, voici le compilateur, voici les flags, voici le résultat. Vérifiez vous-même."

## Cas d'usage concrets

### Vérifier le support d'une fonctionnalité

"Est-ce que GCC 15 supporte `std::expected` ?" Au lieu de chercher dans les tableaux de support compilateur (qui ne sont pas toujours à jour), écrivez un petit programme qui utilise `std::expected`, sélectionnez GCC 15 avec `-std=c++23`, et compilez. Le résultat est immédiat et sans ambiguïté.

Ce cas d'usage est particulièrement pertinent pour les fonctionnalités C++23 et C++26, dont le support compilateur est en cours de déploiement (voir section 12.14.5). Compiler Explorer dispose des versions les plus récentes des compilateurs, y compris les snapshots trunk qui implémentent des fonctionnalités pas encore incluses dans les releases stables.

Pour la réflexion statique C++26 en particulier, Compiler Explorer héberge les forks spécialisés de Clang (dont celui de Bloomberg) qui permettent de tester les fonctionnalités de réflexion avant même leur intégration dans les branches principales des compilateurs.

### Comprendre le coût d'une abstraction

"Est-ce que `std::optional` génère du code supplémentaire par rapport à un pointeur nu ?" Écrivez deux versions d'une fonction — l'une avec `std::optional`, l'autre avec un pointeur — compilez avec `-O2`, et comparez l'assembleur. Si les deux versions génèrent un code identique, l'abstraction est véritablement "zero-cost". Si elles diffèrent, l'assembleur montre exactement où se situe le surcoût.

Ce type de vérification est au cœur de la philosophie C++ d'abstraction sans surcoût. Les talks de Chandler Carruth et de Jason Turner (mentionnés en section 48.2.1) utilisent Compiler Explorer de manière intensive pour démontrer — ou infirmer — le caractère zero-cost d'abstractions courantes.

### Diagnostiquer un bug de compilation

"Ce code compile avec Clang mais pas avec GCC — qui a raison ?" Saisissez le code dans Compiler Explorer, compilez-le avec les deux compilateurs côte à côte, et comparez les diagnostics. Si les deux compilateurs divergent sur la validité d'un programme, c'est potentiellement un bug de l'un d'eux — ou un cas d'ambiguïté dans le standard.

Ce cas d'usage est courant quand on travaille avec des fonctionnalités récentes du langage, où les implémentations ne sont pas encore alignées. Le résultat peut être partagé dans un bug report au compilateur ou dans une question Stack Overflow pour demander l'interprétation correcte.

### Illustrer un point technique

Vous êtes en code review et un collègue doute que `std::move` ne génère aucune instruction assembleur (ce n'est qu'un cast). Au lieu d'un long argument verbal, envoyez un lien godbolt qui montre le code et l'assembleur correspondant. La preuve est visuelle, vérifiable, et indiscutable.

Ce réflexe — transformer un argument technique en lien Compiler Explorer — est une habitude qui améliore drastiquement la qualité des discussions techniques. Il remplace les spéculations par des faits observables.

### Tester du code sans installation locale

Vous lisez un article de blog qui présente une fonctionnalité C++23 que votre environnement local ne supporte pas encore. Au lieu d'installer un nouveau compilateur, copiez le code dans Compiler Explorer, sélectionnez un compilateur récent, et testez. En quelques secondes, vous passez de la lecture passive à l'expérimentation active.

Ce cas d'usage est particulièrement précieux pour les lecteurs de cette formation. Chaque fois qu'un snippet de code est présenté dans les sections sur les nouveautés C++20/23/26 (sections 12.4 à 12.14), il peut être testé immédiatement sur Compiler Explorer sans attendre que votre toolchain locale soit à jour.

## Lire l'assembleur : les bases

Compiler Explorer affiche du code assembleur, ce qui peut être intimidant pour un développeur qui n'a jamais lu d'assembleur. Quelques repères essentiels suffisent pourtant pour tirer une valeur considérable de l'outil, sans devenir expert en jeu d'instructions.

### Ce qu'il faut observer

**Le volume de code généré.** Le premier indicateur est simplement la quantité d'assembleur produit. Si une fonction d'une ligne génère 50 instructions, quelque chose mérite investigation — une allocation dynamique inattendue, un appel de constructeur de copie, ou un branchement complexe.

**Les appels de fonctions.** Les instructions `call` (x86) ou `bl` (ARM) indiquent des appels de fonctions. Si vous voyez un `call` vers `operator new` alors que vous n'attendiez aucune allocation dynamique, c'est un signal d'alerte. Si une fonction que vous pensiez inlinée apparaît comme un `call`, l'optimiseur n'a pas réussi à l'inliner.

**L'absence de code.** Parfois, le résultat le plus informatif est l'absence totale de code assembleur pour une construction C++ donnée. Cela signifie que le compilateur a entièrement optimisé la construction — c'est la preuve ultime de l'abstraction zero-cost. Par exemple, un `std::unique_ptr` bien utilisé génère exactement le même assembleur qu'un raw pointer avec `delete` manuel.

**Les instructions SIMD.** Les instructions utilisant les registres `xmm`, `ymm` ou `zmm` (x86) indiquent de la vectorisation — le compilateur a converti une boucle scalaire en opérations parallèles sur plusieurs données simultanément (voir section 41.3).

### Le filtre de bruit

Activez systématiquement les filtres de Compiler Explorer pour masquer les directives assembleur, les labels non pertinents, et les métadonnées. Le résultat filtré est beaucoup plus lisible et se concentre sur les instructions qui importent.

### Ne pas suroptimiser

Un avertissement important : Compiler Explorer rend l'assembleur tellement accessible qu'il peut pousser à une obsession contreproductive de la micro-optimisation. Vérifier l'assembleur est pertinent pour les chemins critiques de performance identifiés par le profiling (voir section 31). L'utiliser systématiquement pour chaque ligne de code est une perte de temps. L'outil est un microscope — précieux quand on sait où regarder, inutile quand on l'utilise au hasard.

## Fonctionnalités avancées

### Environnements multi-fichiers (CMake)

Compiler Explorer supporte les projets CMake simples, ce qui permet de tester des configurations de build, des dépendances entre fichiers, et des options de linkage directement dans le navigateur. Cette fonctionnalité est utile pour reproduire des problèmes de linkage ou de visibilité qui ne se manifestent pas dans un fichier unique.

### Librairies disponibles

Compiler Explorer met à disposition un ensemble de librairies C++ populaires directement utilisables sans installation : Boost, nlohmann/json, fmt, Catch2, Google Test, Abseil, Range-v3, et bien d'autres. Cela permet de tester du code qui dépend de ces librairies sans configuration locale. Quand vous lisez les sections de cette formation sur JSON (section 24.1), les tests (section 33), ou fmt (section 36.3), les exemples peuvent souvent être testés directement sur Compiler Explorer.

### Conformance view

La "conformance view" compile le même code avec tous les compilateurs majeurs simultanément et affiche une matrice de résultats : compilation réussie ou échouée pour chaque combinaison compilateur/version. C'est l'outil idéal pour vérifier la portabilité d'un code et identifier les compilateurs ou versions qui ne supportent pas encore une fonctionnalité donnée.

### API et intégration

Compiler Explorer dispose d'une API REST qui permet l'intégration programmatique. Des plugins pour VS Code, Vim et d'autres éditeurs permettent d'envoyer du code à Compiler Explorer directement depuis l'IDE, sans ouvrir un navigateur. Cette intégration réduit encore la friction entre l'écriture de code et la vérification du résultat.

## Contribuer à Compiler Explorer

Compiler Explorer est un projet open source sous licence BSD, hébergé sur GitHub. Les contributions sont bienvenues à plusieurs niveaux : ajout de compilateurs ou de librairies, amélioration de l'interface, corrections de bugs, documentation, et financement (le projet accepte les donations via Patreon et GitHub Sponsors pour financer l'infrastructure serveur).

Le projet est un exemple remarquable de contribution individuelle à l'écosystème. Matt Godbolt a créé un outil qui a changé la façon dont des millions de développeurs interagissent avec le C++. C'est un rappel que les contributions techniques les plus impactantes ne sont pas toujours des fonctionnalités du langage ou des librairies de production — parfois, c'est un outil qui rend le langage plus accessible.
