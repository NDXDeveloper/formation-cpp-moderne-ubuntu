# 14.2.2 — Custom hash functions

## Module 5 — La Bibliothèque Standard (STL) | Niveau Intermédiaire

---

## Introduction

La sous-section précédente a montré que la performance d'une table de hachage dépend directement de la qualité de sa fonction de hachage. Pour les types standard (`int`, `std::string`, pointeurs…), la bibliothèque standard fournit des spécialisations de `std::hash<T>` prêtes à l'emploi. Mais dès que l'on utilise un **type personnalisé** comme clé — un `struct`, une `class`, un `std::pair` ou un `enum class` composite —, la compilation échoue : il n'existe pas de `std::hash` par défaut pour ces types.

```cpp
struct Coordinate {
    int x;
    int y;
};

// ❌ Ne compile pas : pas de std::hash<Coordinate>
std::unordered_map<Coordinate, std::string> labels;
```

Cette sous-section explique comment écrire des fonctions de hachage personnalisées, quelles techniques utiliser pour combiner les hash de plusieurs champs, et quels pièges éviter.

---

## Les deux exigences pour une clé de unordered_map

Pour qu'un type `K` soit utilisable comme clé de `std::unordered_map<K, V>`, il doit satisfaire deux exigences :

1. **Une fonction de hachage** — un callable qui prend un `K` et retourne un `std::size_t`.
2. **Un opérateur d'égalité** — `operator==` entre deux `K`, utilisé pour distinguer les clés dans un même bucket après collision.

Si `operator==` est défini (ou généré par le compilateur en C++20 avec `= default`), il ne reste qu'à fournir la fonction de hachage.

---

## Approche 1 : spécialiser std::hash

La méthode la plus idiomatique consiste à fournir une spécialisation de `std::hash` dans le namespace `std`. Cela permet d'utiliser le type directement dans `std::unordered_map` sans paramètre supplémentaire.

```cpp
#include <functional>
#include <unordered_map>
#include <string>

struct Coordinate {
    int x;
    int y;

    bool operator==(const Coordinate&) const = default; // C++20
};

// Spécialisation de std::hash pour Coordinate
template <>  
struct std::hash<Coordinate> {  
    std::size_t operator()(const Coordinate& c) const noexcept {
        std::size_t h1 = std::hash<int>{}(c.x);
        std::size_t h2 = std::hash<int>{}(c.y);
        // Combinaison des deux hash (technique détaillée plus bas)
        return h1 ^ (h2 << 1);
    }
};

// Fonctionne directement — std::hash<Coordinate> est trouvé automatiquement
std::unordered_map<Coordinate, std::string> labels;  
labels[{10, 20}] = "Point A";  
```

Cette approche a l'avantage de la transparence : tout code qui utilise `std::unordered_map<Coordinate, ...>` ou `std::unordered_set<Coordinate>` bénéficie automatiquement du hash, sans paramètre explicite.

> ⚠️ Le standard autorise l'ajout de spécialisations dans le namespace `std` uniquement pour des types définis par l'utilisateur. Spécialiser `std::hash` pour un type de la bibliothèque standard (comme `std::pair<int, int>`) est techniquement un **comportement indéfini** — même si cela fonctionne en pratique sur tous les compilateurs courants. Pour ces types, préférez l'approche 2 (hash functor externe).

## Approche 2 : fournir un hash functor externe

On peut passer un callable de hachage comme troisième paramètre template de `std::unordered_map`. C'est la méthode recommandée quand on ne veut pas (ou ne peut pas) toucher au namespace `std`.

### Avec un struct callable

```cpp
struct CoordinateHash {
    std::size_t operator()(const Coordinate& c) const noexcept {
        std::size_t h1 = std::hash<int>{}(c.x);
        std::size_t h2 = std::hash<int>{}(c.y);
        return h1 ^ (h2 << 1);
    }
};

std::unordered_map<Coordinate, std::string, CoordinateHash> labels;
```

### Avec une lambda (C++20)

Depuis C++20, les lambdas sans capture sont utilisables comme paramètres template non-type. Mais pour `std::unordered_map`, le type du hasher doit être default-constructible, ce qui rend les lambdas peu pratiques dans ce contexte. La forme la plus lisible reste le struct callable.

On peut toutefois passer une lambda au constructeur :

```cpp
auto hasher = [](const Coordinate& c) {
    return std::hash<int>{}(c.x) ^ (std::hash<int>{}(c.y) << 1);
};

std::unordered_map<Coordinate, std::string, decltype(hasher)> labels(16, hasher);
```

Cette syntaxe est plus lourde et rarement préférée au struct dédié.

## Combiner les hash : le problème central

La vraie difficulté n'est pas d'écrire un hash pour un type à un seul champ — `std::hash<int>{}(c.x)` suffit. Le problème survient quand la clé contient **plusieurs champs** qu'il faut combiner en un seul `std::size_t`.

### Le XOR naïf — et pourquoi il est insuffisant

```cpp
// ❌ Mauvaise combinaison
return std::hash<int>{}(c.x) ^ std::hash<int>{}(c.y);
```

Le XOR (`^`) est commutatif : `hash(x) ^ hash(y) == hash(y) ^ hash(x)`. Cela signifie que `{3, 7}` et `{7, 3}` produisent le **même hash**. Pire encore, toute paire `{n, n}` produit 0 (`hash(n) ^ hash(n) == 0`). Avec des données réelles où ces symétries sont fréquentes, les collisions explosent.

### La technique Boost : hash_combine

La fonction `hash_combine` popularisée par Boost est la technique de combinaison la plus répandue en C++. Elle mélange un hash existant avec une nouvelle valeur en utilisant un décalage, un XOR et la constante magique dérivée du nombre d'or :

```cpp
inline void hash_combine(std::size_t& seed, std::size_t value) {
    seed ^= value + 0x9e3779b9 + (seed << 6) + (seed >> 2);
}
```

La constante `0x9e3779b9` est `⌊2³² / φ⌋` (où φ est le nombre d'or ≈ 1.618). Elle assure une bonne dispersion des bits même pour des entrées séquentielles.

Utilisée dans un hash de structure :

```cpp
struct CoordinateHash {
    std::size_t operator()(const Coordinate& c) const noexcept {
        std::size_t seed = 0;
        hash_combine(seed, std::hash<int>{}(c.x));
        hash_combine(seed, std::hash<int>{}(c.y));
        return seed;
    }
};
```

Cette technique est **non commutative** (`{3, 7}` ≠ `{7, 3}`), résistante aux entrées triviales (`{n, n}` ne produit pas 0), et offre une bonne distribution en pratique.

### Version 64 bits

Sur les plateformes 64 bits (la majorité aujourd'hui), il est préférable d'utiliser la constante adaptée :

```cpp
inline void hash_combine(std::size_t& seed, std::size_t value) {
    if constexpr (sizeof(std::size_t) == 8) {
        // Constante 64 bits : ⌊2⁶⁴ / φ⌋
        seed ^= value + 0x9e3779b97f4a7c15ULL + (seed << 12) + (seed >> 4);
    } else {
        // Constante 32 bits originale
        seed ^= value + 0x9e3779b9 + (seed << 6) + (seed >> 2);
    }
}
```

## Un utilitaire hash_combine réutilisable

En pratique, on écrit `hash_combine` une seule fois dans un header utilitaire et on le réutilise pour tous les types personnalisés. Voici une version variadic qui combine un nombre arbitraire de champs en un seul appel :

```cpp
// hash_utils.h
#pragma once
#include <functional>
#include <cstddef>

namespace util {

inline void hash_combine(std::size_t& seed, std::size_t value) {
    seed ^= value + 0x9e3779b97f4a7c15ULL + (seed << 12) + (seed >> 4);
}

template <typename T, typename... Rest>  
void hash_combine(std::size_t& seed, const T& first, const Rest&... rest) {  
    hash_combine(seed, std::hash<T>{}(first));
    (hash_combine(seed, std::hash<Rest>{}(rest)), ...); // Fold expression C++17
}

// Helper : retourne directement le hash combiné
template <typename... Args>  
std::size_t combined_hash(const Args&... args) {  
    std::size_t seed = 0;
    hash_combine(seed, args...);
    return seed;
}

} // namespace util
```

L'utilisation devient triviale :

```cpp
struct Coordinate {
    int x, y;
    bool operator==(const Coordinate&) const = default;
};

struct Coordinate3D {
    int x, y, z;
    bool operator==(const Coordinate3D&) const = default;
};

template <>  
struct std::hash<Coordinate> {  
    std::size_t operator()(const Coordinate& c) const noexcept {
        return util::combined_hash(c.x, c.y);
    }
};

template <>  
struct std::hash<Coordinate3D> {  
    std::size_t operator()(const Coordinate3D& c) const noexcept {
        return util::combined_hash(c.x, c.y, c.z);
    }
};
```

## Cas courants : std::pair et std::tuple

La bibliothèque standard ne fournit pas de `std::hash` pour `std::pair` ni `std::tuple`. C'est un manque fréquemment rencontré. Voici comment le combler proprement avec un hash functor externe (pour éviter le comportement indéfini de la spécialisation dans `std`).

### Hash pour std::pair

```cpp
struct PairHash {
    template <typename T1, typename T2>
    std::size_t operator()(const std::pair<T1, T2>& p) const noexcept {
        return util::combined_hash(p.first, p.second);
    }
};

// Utilisation
std::unordered_map<std::pair<int, int>, std::string, PairHash> grid;  
grid[{0, 0}] = "origin";  
grid[{1, 2}] = "point A";  
```

### Hash pour std::tuple

```cpp
struct TupleHash {
    template <typename... Ts>
    std::size_t operator()(const std::tuple<Ts...>& t) const noexcept {
        return std::apply([](const auto&... args) {
            return util::combined_hash(args...);
        }, t);
    }
};

// Utilisation — clé composite (département, ville, code postal)
using LocationKey = std::tuple<std::string, std::string, int>;  
std::unordered_map<LocationKey, double, TupleHash> data;  
```

## Hash pour les enum class

Les `enum class` n'ont pas de spécialisation `std::hash` automatique dans certaines implémentations antérieures à C++14. Depuis C++14, le standard garantit `std::hash` pour les types énumérés, mais en pratique certains compilateurs plus anciens ne la fournissent pas. Un hash générique pour enums est simple à écrire :

```cpp
struct EnumHash {
    template <typename E>
    std::size_t operator()(E e) const noexcept {
        return std::hash<std::underlying_type_t<E>>{}(
            static_cast<std::underlying_type_t<E>>(e)
        );
    }
};
```

> En C++14 et au-delà, avec GCC et Clang récents, `std::hash` fonctionne nativement pour les `enum class`. Vérifiez le support de votre compilateur avant d'ajouter ce workaround.

## Hash pour un type avec des champs complexes

Quand une structure contient des champs qui sont eux-mêmes des types composites (vecteurs, strings, sous-structures), on compose les hash de manière récursive :

```cpp
struct Employee {
    std::string name;
    std::string department;
    int id;

    bool operator==(const Employee&) const = default;
};

template <>  
struct std::hash<Employee> {  
    std::size_t operator()(const Employee& e) const noexcept {
        return util::combined_hash(e.name, e.department, e.id);
    }
};

// Fonctionne car std::hash<std::string> et std::hash<int> existent
std::unordered_set<Employee> team;
```

Si un champ est lui-même un type personnalisé, il suffit que ce type ait aussi un `std::hash` défini — la composition est naturellement récursive.

## Qualités d'une bonne fonction de hachage

Toutes les fonctions de hachage ne se valent pas. Une bonne fonction de hachage pour une table de hachage doit posséder plusieurs propriétés :

### Uniformité de distribution

Le hash doit répartir les clés uniformément dans l'espace des `std::size_t`. Des clés similaires doivent produire des hash très différents. L'**effet avalanche** — un seul bit de différence dans l'entrée modifie environ la moitié des bits de sortie — est le critère classique.

```
hash({0, 0}) = 0x3a7c5e1b9f4d2c08   ← Bits bien répartis  
hash({0, 1}) = 0xd1e8a347c62f5b90   ← Très différent malgré une entrée proche  
hash({1, 0}) = 0x85b4f2c0e9a71d63   ← Aussi très différent  
```

### Déterminisme

Le même input doit toujours produire le même output au sein d'une exécution. C'est garanti par construction si la fonction est pure (pas d'état mutable, pas de dépendance à l'adresse mémoire).

> ⚠️ Le hash n'a pas besoin d'être stable **entre exécutions**. En fait, certaines implémentations (Python, Rust) ajoutent un *seed* aléatoire au démarrage pour prévenir les attaques par hash flooding. En C++, `std::hash` est stable entre exécutions sur les implémentations courantes, mais le standard ne le garantit pas formellement.

### Rapidité

La fonction de hachage est appelée à chaque insertion, recherche et suppression. Si elle est lente, elle annule l'avantage du O(1). Pour des clés simples (entiers, petites structures), la fonction doit être en O(1). Pour des clés de taille variable (`std::string`), elle est nécessairement en O(k) où k est la longueur de la clé.

### Cohérence avec operator==

C'est une exigence du standard, pas une simple recommandation :

> **Invariant** : si `a == b`, alors `hash(a) == hash(b)`.

La réciproque n'est pas requise (deux clés différentes peuvent avoir le même hash — c'est une collision). Mais si cet invariant est violé, la table de hachage est **cassée** : un élément inséré avec la clé `a` ne sera jamais retrouvé par une recherche avec la clé `b`, même si `a == b`.

Un cas classique de violation : utiliser un champ dans le hash mais pas dans `operator==`, ou inversement.

```cpp
// ❌ BUG : le hash utilise name, mais operator== compare uniquement id
struct Item {
    int id;
    std::string name;
    bool operator==(const Item& other) const { return id == other.id; }
};

template <>  
struct std::hash<Item> {  
    std::size_t operator()(const Item& i) const noexcept {
        return std::hash<std::string>{}(i.name); // Incohérent avec operator==
    }
};
```

> ⭐ **Règle d'or** : le hash doit utiliser exactement les mêmes champs que `operator==`, ni plus, ni moins. Si `operator==` compare `{id}`, le hash doit hasher `{id}`. Si `operator==` compare `{name, department}`, le hash doit combiner les hash de `{name, department}`.

## Tester la qualité de son hash

Un test simple consiste à mesurer la distribution des buckets après insertion d'un jeu de données représentatif (cf. la fonction `print_bucket_distribution` de la section 14.2.1). Les indicateurs clés sont :

- **Longueur maximale d'un bucket** — devrait rester faible (≤ 3-5 pour un facteur de charge de 1.0).
- **Variance de la longueur des buckets** — une faible variance indique une distribution uniforme.
- **Nombre de collisions** — `size() - buckets_non_vides` donne le nombre total de collisions.

Si la longueur maximale d'un bucket est disproportionnée (dizaines ou centaines), c'est un signal clair que la fonction de hachage distribue mal les données réelles.

## Résumé des approches

| Approche | Quand l'utiliser | Avantage | Inconvénient |
|---|---|---|---|
| Spécialiser `std::hash<T>` | Type utilisateur propre | Transparent, pas de paramètre | Interdit pour les types de la STL |
| Struct callable externe | `std::pair`, `std::tuple`, types STL | Légal pour tous les types | Paramètre template explicite |
| `hash_combine` variadic | Clés multi-champs | Réutilisable, concis | Utilitaire à maintenir |

---

## En bref

Utiliser un type personnalisé comme clé de `std::unordered_map` nécessite de fournir une fonction de hachage et un `operator==`. La technique `hash_combine` issue de Boost est la méthode standard de facto pour combiner les hash de plusieurs champs. Les deux pièges critiques à éviter sont le XOR naïf (symétrique, collisions sur les paires identiques) et l'incohérence entre les champs utilisés dans le hash et dans `operator==`. Une bonne fonction de hachage distribue uniformément, est rapide, et respecte l'invariant `a == b ⟹ hash(a) == hash(b)`.
