# 5.1.3 — Caractéristiques du Heap

## Une mémoire à la demande

Là où la stack est un mécanisme rigide, rapide et automatique, le heap est une zone mémoire **flexible, vaste et entièrement manuelle**. C'est l'espace où votre programme peut demander de la mémoire au moment de son choix, en quantité déterminée à l'exécution, et la conserver aussi longtemps que nécessaire — indépendamment des appels de fonctions et des scopes.

Cette liberté a un prix. Chaque allocation sur le heap implique un travail non trivial de la part d'un composant logiciel appelé l'**allocateur mémoire**. Et chaque octet alloué devient votre responsabilité : si vous oubliez de le libérer, il reste occupé jusqu'à la fin du programme. C'est la source de la grande majorité des bugs mémoire en C++.

---

## L'allocateur mémoire : le gestionnaire invisible

Quand vous écrivez `new int(42)` en C++, voici ce qui se passe en coulisses :

1. L'opérateur `new` appelle la fonction `operator new()`, qui appelle en interne `malloc()` de la bibliothèque C standard (glibc sous Ubuntu).
2. L'allocateur de glibc (`ptmalloc2`) cherche un bloc libre de taille suffisante dans ses structures internes.
3. S'il en trouve un, il le marque comme occupé et retourne son adresse.
4. S'il n'en trouve pas, il demande de la mémoire au noyau Linux via l'appel système `brk()` (pour étendre le heap) ou `mmap()` (pour les grandes allocations).
5. Le noyau met à jour les tables de pages du processus et retourne de la mémoire virtuelle.
6. L'adresse du bloc est retournée à votre programme.

Le chemin inverse se produit lors d'un `delete` : `operator delete()` appelle `free()`, qui marque le bloc comme libre dans les structures de l'allocateur. La mémoire n'est généralement **pas rendue au système d'exploitation** immédiatement — l'allocateur la conserve en prévision de futures allocations.

Ce mécanisme en couches explique pourquoi une allocation heap est plus coûteuse qu'une allocation stack :

```
Votre code C++
     │
     ▼
operator new / operator delete       ← Couche C++
     │
     ▼
malloc / free  (glibc ptmalloc2)     ← Allocateur espace utilisateur
     │                                  Gère les blocs libres/occupés
     ▼                                  Évite les appels système quand possible
brk / mmap  (appels système)         ← Noyau Linux
     │                                  Alloue des pages de mémoire virtuelle
     ▼
Tables de pages (MMU)                ← Matériel
                                       Traduit adresses virtuelles → physiques
```

## Le coût réel d'une allocation

Combien coûte un `new` par rapport à une variable sur la stack ? La réponse dépend des circonstances, mais les ordres de grandeur sont les suivants.

Une allocation sur la stack prend **1 à 2 cycles CPU** — c'est un simple ajustement du registre RSP.

Une allocation sur le heap, dans le cas favorable (un bloc libre de la bonne taille est immédiatement disponible dans le cache de l'allocateur), prend environ **50 à 100 nanosecondes**, soit quelques dizaines de cycles sur un processeur moderne.

Dans le cas défavorable (l'allocateur doit chercher dans ses listes, fusionner des blocs, ou pire, faire un appel système au noyau), le coût peut monter à **plusieurs microsecondes**.

Ces chiffres peuvent sembler négligeables pour une allocation isolée. Mais dans une boucle qui traite des millions d'éléments, ou dans un serveur qui gère des milliers de requêtes par seconde, la différence entre stack et heap devient un facteur de performance mesurable. C'est la raison pour laquelle le C++ haute performance cherche systématiquement à **minimiser les allocations dynamiques** — en réservant la mémoire à l'avance (`std::vector::reserve`), en utilisant des allocateurs personnalisés, ou en privilégiant les objets sur la stack quand c'est possible.

## La fragmentation : l'ennemi silencieux

Contrairement à la stack, où la structure LIFO garantit que la mémoire est toujours compacte, le heap peut devenir **fragmenté** au fil du temps. La fragmentation se produit quand des blocs sont alloués et libérés dans un ordre imprévisible, laissant des trous entre les blocs encore occupés.

Considérez cette séquence :

```cpp
// Étape 1 : trois allocations consécutives
int* a = new int[1000];    // bloc de 4 000 octets  
int* b = new int[1000];    // bloc de 4 000 octets, juste après a  
int* c = new int[1000];    // bloc de 4 000 octets, juste après b  

// Étape 2 : on libère le bloc du milieu
delete[] b;                 // 4 000 octets libérés entre a et c

// Étape 3 : on demande un gros bloc
int* d = new int[2000];    // besoin de 8 000 octets
```

Après l'étape 2, le heap ressemble à ceci :

```
┌────────────┬────────────┬────────────┐
│  a (occupé)│  (libre)   │  c (occupé)│
│  4 000 o.  │  4 000 o.  │  4 000 o.  │
└────────────┴────────────┴────────────┘
```

Le trou de 4 000 octets laissé par `b` ne peut pas accueillir `d` qui en demande 8 000. L'allocateur doit donc aller chercher de l'espace plus loin, même si la quantité totale de mémoire libre est suffisante. C'est la **fragmentation externe** : l'espace libre existe, mais il est dispersé en morceaux trop petits pour être utiles.

Il existe aussi une **fragmentation interne** : l'allocateur arrondit souvent les tailles demandées à des multiples de 8 ou 16 octets (pour des raisons d'alignement), et chaque bloc contient un en-tête de métadonnées (typiquement 8 à 16 octets). Si vous allouez un seul `int` de 4 octets, l'allocateur utilise en réalité 16 à 32 octets — la différence est de la fragmentation interne.

Dans un programme qui tourne pendant des heures ou des jours (un serveur, un daemon système), la fragmentation peut faire grimper la consommation mémoire bien au-delà de ce que les données actives nécessiteraient. C'est un problème classique des applications long-running.

## Durée de vie : liberté et responsabilité

La grande force du heap est que la durée de vie des données n'est liée à aucun scope. Un objet alloué dans une fonction peut survivre au retour de cette fonction, être passé à une autre fonction, stocké dans une structure de données, et libéré des minutes plus tard dans un tout autre contexte :

```cpp
#include <string>

struct Utilisateur {
    std::string nom;
    int age;
};

Utilisateur* creer_utilisateur(const std::string& nom, int age) {
    Utilisateur* u = new Utilisateur{nom, age};
    return u;   // l'objet survit au retour de la fonction
}

void traiter() {
    Utilisateur* alice = creer_utilisateur("Alice", 30);

    // ... alice est utilisé pendant toute la durée de traiter() ...
    // ... il pourrait même être stocké dans un conteneur global ...

    delete alice;  // libération explicite — le programmeur choisit le moment
}
```

Cette flexibilité est indispensable dans de nombreuses situations : structures de données dynamiques (arbres, graphes, listes chaînées), objets polymorphiques manipulés par pointeur de base, buffers de taille déterminée à l'exécution, données partagées entre threads.

Mais cette liberté implique que chaque `new` doit être équilibré par exactement un `delete`. Oubliez le `delete` → fuite mémoire. Appelez `delete` deux fois → comportement indéfini. Utilisez le pointeur après le `delete` → comportement indéfini. La section 5.4 catalogue ces erreurs en détail.

---

## Le heap est partagé entre threads

Contrairement à la stack, où chaque thread possède sa propre pile isolée, le heap est une **zone commune** à tous les threads d'un processus. Deux threads peuvent allouer de la mémoire sur le même heap, et un objet alloué par un thread peut être libéré par un autre.

Cette propriété est à la fois utile (c'est le mécanisme principal de partage de données entre threads) et dangereuse (elle nécessite une synchronisation pour éviter les data races).

L'allocateur de glibc (`ptmalloc2`) gère cette situation en utilisant des **arenas** — des sous-ensembles du heap attribués à chaque thread pour réduire la contention sur les verrous. Quand un thread alloue de la mémoire, il utilise de préférence son arena locale, évitant ainsi de bloquer les autres threads. Ce mécanisme est transparent pour le programmeur, mais il explique pourquoi la consommation mémoire d'un programme multi-thread peut être plus élevée que ce que la somme des allocations individuelles suggérerait — chaque arena maintient ses propres blocs libres.

```
Thread 1                Thread 2               Thread 3
    │                      │                     │
    ▼                      ▼                     ▼
┌─────────┐           ┌─────────┐           ┌─────────┐
│ Arena 1 │           │ Arena 2 │           │ Arena 3 │
└────┬────┘           └────┬────┘           └────┬────┘
     │                     │                     │
     └─────────────────────┼─────────────────────┘
                           ▼
                    ┌─────────────┐
                    │    HEAP     │
                    │  (partagé)  │
                    └─────────────┘
```

## Allocation paresseuse et overcommit

Un aspect souvent méconnu du heap sous Linux est que le noyau pratique l'**overcommit** par défaut. Quand `malloc` demande de la mémoire au noyau via `brk()` ou `mmap()`, le noyau réserve de l'espace d'adressage virtuel mais ne lui attribue pas immédiatement de mémoire physique. Les pages physiques ne sont allouées que lorsque le programme **écrit effectivement** dans cette zone — c'est l'allocation paresseuse (*lazy allocation* ou *demand paging*).

Conséquence surprenante : `new` peut réussir même si la mémoire physique est insuffisante. Le programme ne découvrira le problème que plus tard, lorsqu'il tentera d'écrire dans les pages allouées. Si la mémoire physique est épuisée à ce moment-là, le noyau invoque l'**OOM Killer** (*Out Of Memory Killer*), qui termine un processus pour libérer de la mémoire — et ce processus n'est pas forcément celui qui a provoqué le problème.

```bash
# Vérifier la politique d'overcommit
cat /proc/sys/vm/overcommit_memory
# 0 = heuristique (défaut), 1 = toujours, 2 = jamais
```

Ce comportement signifie qu'en C++ sous Linux, vérifier le retour de `new` (qui lance `std::bad_alloc` en cas d'échec) ne suffit pas à garantir que la mémoire sera disponible au moment de l'utilisation. C'est une réalité système qu'il faut avoir en tête, même si en pratique, l'OOM Killer n'intervient que dans des situations extrêmes.

## Observer le heap en temps réel

Plusieurs outils permettent de surveiller la consommation heap d'un programme. Le plus simple est `/proc/<PID>/status` :

```bash
# Pendant l'exécution du programme
cat /proc/$(pidof mon_programme)/status | grep -i vm
```

```
VmPeak:    25432 kB    ← pic d'utilisation de mémoire virtuelle  
VmSize:    25432 kB    ← mémoire virtuelle actuelle  
VmRSS:     3280 kB    ← mémoire physique réellement utilisée (Resident Set Size)  
VmData:    1156 kB    ← taille du segment data + heap  
VmStk:      136 kB    ← taille de la stack  
```

La distinction entre `VmSize` et `VmRSS` illustre l'allocation paresseuse : `VmSize` est l'espace virtuel réservé, `VmRSS` est la mémoire physique effectivement touchée. Un programme peut avoir un `VmSize` de plusieurs gigaoctets mais un `VmRSS` modeste si une grande partie de sa mémoire virtuelle n'a jamais été accédée.

Pour une vue plus détaillée de chaque allocation, Valgrind et l'outil Massif (que nous verrons en section 5.5.1 et au chapitre 30) permettent de tracer chaque `new` et chaque `delete`, et de visualiser l'évolution du heap dans le temps.

## Quand utiliser le heap

Le heap est le bon choix dans quatre situations principales.

**La taille est déterminée à l'exécution.** Si la quantité de mémoire dépend d'une entrée utilisateur, d'un fichier de configuration, ou d'une condition dynamique, la stack ne convient pas :

```cpp
int n;  
std::cin >> n;  
// int tableau[n];          // ❌ VLA — non standard en C++, taille stack limitée
std::vector<int> tableau(n); // ✅ allocation heap via vector
```

**L'objet doit survivre à son scope de création.** Si une fonction crée un objet qui doit être utilisé après son retour, cet objet ne peut pas vivre sur la stack de la fonction :

```cpp
std::unique_ptr<Config> charger_config(const std::string& chemin) {
    auto cfg = std::make_unique<Config>();  // heap — survit au retour
    cfg->charger(chemin);
    return cfg;
}
```

**L'objet est trop volumineux pour la stack.** Au-delà de quelques kilo-octets, la stack devient un territoire risqué. Les buffers, les matrices, les structures de données volumineuses doivent aller sur le heap :

```cpp
// ❌ 8 Mo sur la stack → stack overflow quasi certain
double matrice[1000][1000];

// ✅ Sur le heap, géré automatiquement par vector
std::vector<std::vector<double>> matrice(1000, std::vector<double>(1000));
```

**Le polymorphisme est nécessaire.** Le polymorphisme dynamique en C++ fonctionne via des pointeurs ou des références vers une classe de base. Les objets polymorphiques sont presque toujours alloués sur le heap et manipulés via des smart pointers :

```cpp
std::unique_ptr<Animal> creer_animal(const std::string& type) {
    if (type == "chat") return std::make_unique<Chat>();
    if (type == "chien") return std::make_unique<Chien>();
    return std::make_unique<AnimalInconnu>();
}
```

## Synthèse : les propriétés du heap

**La flexibilité.** Aucune contrainte sur la durée de vie, la taille ou l'ordre de libération. Le heap s'adapte à tous les besoins que la stack ne peut pas satisfaire.

**Le volume.** Limité uniquement par la mémoire disponible du système — plusieurs gigaoctets sont courants, contre 8 Mo pour la stack.

**Le partage.** Zone commune entre threads, permettant la communication et le partage de données.

**Le coût.** Chaque allocation est significativement plus lente qu'une allocation stack. L'allocateur doit maintenir des structures de données complexes pour gérer les blocs libres et occupés.

**La fragmentation.** L'ordre imprévisible des allocations et libérations crée des trous inutilisables, dégradant progressivement l'efficacité mémoire.

**La responsabilité.** Chaque allocation doit être explicitement libérée. C'est la source de trois catégories entières d'erreurs — les fuites mémoire, les dangling pointers et les double free — que nous allons maintenant explorer dans les sections 5.2 à 5.4. Et c'est précisément pour neutraliser cette responsabilité dangereuse que le C++ moderne a inventé les smart pointers (chapitre 9) et le RAII (chapitre 6).
