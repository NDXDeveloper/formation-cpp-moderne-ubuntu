# 5.5.1 — Valgrind : Installation et utilisation

## Qu'est-ce que Valgrind ?

Valgrind est un framework d'instrumentation dynamique pour les programmes Linux. Son outil le plus utilisé, **Memcheck**, intercepte chaque opération mémoire de votre programme — chaque lecture, chaque écriture, chaque allocation, chaque libération — et vérifie sa validité. Il ne nécessite aucune modification de votre code source ni recompilation spéciale (bien qu'une compilation avec symboles de débogage soit fortement recommandée pour obtenir des rapports lisibles).

Valgrind fonctionne en traduisant le code machine de votre exécutable en une représentation intermédiaire, en y insérant des vérifications, puis en exécutant cette version instrumentée. C'est cette traduction dynamique qui explique le ralentissement de 10 à 30×, mais aussi la capacité de Valgrind à détecter des erreurs invisibles autrement.

---

## Installation sur Ubuntu

L'installation est directe via le gestionnaire de paquets :

```bash
sudo apt update  
sudo apt install valgrind  
```

Vérifiez l'installation :

```bash
valgrind --version
# Sortie attendue : valgrind-3.22.0 (ou version ultérieure)
```

Valgrind est disponible sur toutes les versions récentes d'Ubuntu. Sur les architectures x86_64 et ARM64, il fonctionne sans configuration particulière.

---

## Compiler pour Valgrind : les bonnes options

Valgrind travaille sur l'exécutable final, mais la qualité de ses rapports dépend directement des informations de débogage présentes dans le binaire. La compilation idéale pour une session Valgrind :

```bash
g++ -std=c++17 -g -O0 -o programme programme.cpp
```

Chaque option a son importance :

**`-g`** inclut les symboles de débogage (noms de fonctions, numéros de lignes, noms de fichiers). Sans ce flag, les rapports Valgrind n'affichent que des adresses hexadécimales — exploitables, mais pénibles à lire.

**`-O0`** désactive toutes les optimisations. L'optimiseur peut supprimer des variables, réordonner des instructions, ou inliner des fonctions, rendant la correspondance entre le rapport et votre code source confuse. Pour le diagnostic, la clarté prime sur la vitesse.

> **Note :** il est possible d'utiliser Valgrind sur un binaire compilé avec `-O2`, mais les numéros de lignes dans les rapports peuvent être décalés et certaines variables locales disparaissent du fait de l'optimisation. Réservez cette pratique aux cas où le bug ne se reproduit qu'en mode optimisé.

## Anatomie d'un rapport Valgrind

Chaque message Valgrind suit une structure identique. Apprendre à la lire est la compétence clé.

### Le préfixe PID

Chaque ligne commence par `==12345==` où `12345` est le PID du processus. Ce préfixe permet de distinguer les messages Valgrind de la sortie normale du programme, et de séparer les rapports quand plusieurs processus sont instrumentés simultanément (par exemple un programme qui fait un `fork`).

### Les erreurs d'accès

Le premier bloc d'erreur signale notre buffer overflow :

```
==12345== Invalid write of size 4
==12345==    at 0x109205: main (bugs_memoire.cpp:12)
==12345==  Address 0x4a47054 is 0 bytes after a block of size 20 alloc'd
==12345==    at 0x484021F: operator new[](unsigned long) (vg_replace_malloc.c:652)
==12345==    by 0x1091F1: main (bugs_memoire.cpp:11)
```

Décomposons ligne par ligne :

- **`Invalid write of size 4`** — une écriture invalide de 4 octets (un `int`). Valgrind qualifie l'opération (read/write) et sa taille.
- **`at 0x109205: main (bugs_memoire.cpp:12)`** — l'erreur se produit à la ligne 12 de `bugs_memoire.cpp`, dans la fonction `main`. C'est la ligne `tableau[5] = 999;`.
- **`Address 0x4a47054 is 0 bytes after a block of size 20 alloc'd`** — l'adresse écrite est située exactement après la fin d'un bloc de 20 octets (5 × `sizeof(int)`). Valgrind sait que ce bloc a été alloué et vous dit où commence le dépassement.
- **`at ... by 0x1091F1: main (bugs_memoire.cpp:11)`** — la pile d'appels de l'allocation d'origine, pointant vers la ligne 11 : `int* tableau = new int[5];`.

La pile d'appels d'allocation est une information précieuse. Elle vous dit non seulement *où* l'erreur se produit, mais aussi *quel bloc* est concerné et *qui l'a alloué*.

### Le use-after-free

Le deuxième bloc signale notre lecture après libération :

```
==12345== Invalid read of size 4
==12345==    at 0x109236: main (bugs_memoire.cpp:17)
==12345==  Address 0x4a470a0 is 0 bytes inside a block of size 4 free'd
==12345==    at 0x4840B0F: operator delete(void*, unsigned long) (...)
==12345==    by 0x10922D: main (bugs_memoire.cpp:16)
==12345==  Block was alloc'd at
==12345==    at 0x4840183: operator new(unsigned long) (...)
==12345==    by 0x10921B: main (bugs_memoire.cpp:15)
```

Valgrind fournit trois informations : où l'accès invalide se produit (ligne 17), où le bloc a été libéré (ligne 16), et où il avait été alloué (ligne 15). Cette triple localisation est exactement ce qu'il faut pour comprendre le cycle de vie complet du bloc problématique.

### Le double free

Le troisième bloc :

```
==12345== Invalid free() / delete / delete[] / realloc()
==12345==    at ...: main (bugs_memoire.cpp:21)
==12345==  Address 0x4a470a0 is 0 bytes inside a block of size 4 free'd
==12345==    at ...: main (bugs_memoire.cpp:20)
==12345==  Block was alloc'd at
==12345==    at ...: main (bugs_memoire.cpp:19)
```

Même structure : Valgrind montre le second `delete` (ligne 21), le premier `delete` (ligne 20) et l'allocation originale (ligne 19).

### Le résumé des fuites

Le LEAK SUMMARY en fin de rapport résume les fuites :

```
==12345== LEAK SUMMARY:
==12345==    definitely lost: 400 bytes in 1 blocks
==12345==    indirectly lost: 0 bytes in 0 blocks
==12345==      possibly lost: 0 bytes in 0 blocks
==12345==    still reachable: 0 bytes in 0 blocks
```

Les quatre catégories ont des significations précises :

**definitely lost** — aucun pointeur dans le programme ne pointe vers ce bloc au moment de la sortie. C'est une fuite certaine. Dans notre cas, 400 octets (100 × `sizeof(int)`) correspondent au `new int[100]` jamais libéré.

**indirectly lost** — le bloc n'est pas directement référencé, mais il est accessible via un autre bloc lui-même perdu. Typique des structures chaînées : si vous perdez la tête d'une liste, tous les nœuds suivants sont *indirectly lost*.

**possibly lost** — Valgrind a trouvé un pointeur qui pointe à l'intérieur du bloc (pas au début). Cela peut être un pointeur qui a fait de l'arithmétique, ou un faux positif.

**still reachable** — un pointeur vers le bloc existe encore au moment de la sortie. Ce n'est techniquement pas une fuite (le programme pourrait encore faire un `delete`), mais en pratique, ne pas libérer la mémoire avant la sortie indique souvent un oubli. C'est courant avec les singletons et les caches globaux.

## Options essentielles de Memcheck

### Détail complet des fuites

Par défaut, Valgrind résume les fuites sans montrer les piles d'appels. Pour obtenir le détail :

```bash
valgrind --leak-check=full ./bugs_memoire
```

Le rapport inclut alors la pile d'appels de chaque allocation ayant fuité :

```
==12345== 400 bytes in 1 blocks are definitely lost in loss record 1 of 1
==12345==    at 0x484021F: operator new[](unsigned long) (vg_replace_malloc.c:652)
==12345==    by 0x1091C5: main (bugs_memoire.cpp:7)
```

C'est l'option que vous utiliserez le plus souvent. Ajoutez-la systématiquement.

### Origine des données non initialisées

Si votre programme produit des résultats imprévisibles, les lectures de mémoire non initialisée peuvent en être la cause. Par défaut, Valgrind signale *où* la valeur non initialisée est *utilisée*, mais pas *où* elle a été *créée*. L'option `--track-origins=yes` ajoute cette information :

```bash
valgrind --leak-check=full --track-origins=yes ./programme
```

Cette option augmente le ralentissement (environ 2× supplémentaire), mais elle est précieuse pour ce type de bug.

### Afficher les fuites "still reachable"

Par défaut, Valgrind ne montre pas les blocs *still reachable* (mémoire non libérée mais encore accessible). Pour les voir :

```bash
valgrind --leak-check=full --show-reachable=yes ./programme
```

### Limiter la profondeur des piles d'appels

Dans un programme avec de profondes chaînes d'appels, les piles par défaut (12 niveaux) peuvent être insuffisantes. Augmentez la profondeur :

```bash
valgrind --leak-check=full --num-callers=30 ./programme
```

### Rediriger le rapport dans un fichier

Pour les programmes qui produisent beaucoup de sortie sur `stderr`, il est pratique de séparer le rapport Valgrind :

```bash
valgrind --leak-check=full --log-file=valgrind_rapport.txt ./programme
```

Le rapport est écrit dans `valgrind_rapport.txt` au lieu de `stderr`.

## La commande de référence

En combinant les options les plus utiles, voici la commande que vous utiliserez le plus souvent :

```bash
valgrind --leak-check=full \
         --show-leak-kinds=all \
         --track-origins=yes \
         --num-callers=20 \
         ./programme
```

Cette commande active le détail complet des fuites, affiche toutes les catégories de fuites (y compris *still reachable*), trace l'origine des valeurs non initialisées, et affiche jusqu'à 20 niveaux de pile d'appels.

## Fichiers de suppression

Certaines bibliothèques (comme la libc, libstdc++, ou des bibliothèques tierces) contiennent des allocations que Valgrind signale comme des fuites, mais qui sont en réalité intentionnelles — mémoire allouée une fois au démarrage et jamais libérée, par design. Ces faux positifs polluent le rapport.

Valgrind utilise des **fichiers de suppression** (`.supp`) pour masquer ces avertissements connus. Un fichier de suppression par défaut est fourni avec Valgrind, mais vous pouvez créer le vôtre pour les bibliothèques spécifiques à votre projet.

Pour générer les suppressions automatiquement :

```bash
valgrind --leak-check=full --gen-suppressions=all ./programme 2> suppressions_brutes.txt
```

Puis extraire et organiser les blocs de suppression dans un fichier `.supp`, et l'utiliser :

```bash
valgrind --leak-check=full --suppressions=mon_projet.supp ./programme
```

En pratique, les suppressions sont surtout utiles dans les projets matures qui cherchent un rapport "zéro erreur" dans leur pipeline CI — un objectif qui vaut largement l'investissement initial de configuration.

## Intégration dans un workflow

### Valgrind en script de test

Un usage courant est d'encapsuler Valgrind dans un script qui échoue si des erreurs sont détectées. L'option `--error-exitcode` fait retourner un code d'erreur non nul si Valgrind trouve des problèmes :

```bash
valgrind --leak-check=full \
         --error-exitcode=1 \
         ./mon_programme_tests
```

Si Memcheck détecte une erreur, la commande retourne le code de sortie `1`, ce qui fait échouer le job dans un pipeline CI.

### Valgrind avec CTest (CMake)

Si votre projet utilise CMake et CTest pour les tests, vous pouvez exécuter toute la suite de tests sous Valgrind :

```bash
cd build  
ctest -T memcheck  
```

CTest lance automatiquement chaque test via Valgrind et produit des rapports XML exploitables par les outils de CI. La configuration se fait dans `CTestCustom.cmake` ou via les propriétés `MEMCHECK_COMMAND` de CMake.

## Limites de Valgrind

Valgrind est un outil remarquable, mais il a ses limites qu'il faut connaître pour ne pas se reposer aveuglément dessus.

**Couverture de code.** Valgrind ne détecte que les erreurs sur les chemins d'exécution effectivement empruntés. Si un bug mémoire ne se manifeste que dans une branche `else` que vos tests ne couvrent jamais, Valgrind ne le verra pas. La qualité de la détection dépend directement de la qualité de votre suite de tests.

**Performance.** Le ralentissement de 10-30× rend Valgrind impraticable pour les programmes interactifs, les tests de charge ou les benchmarks. Pour ces usages, AddressSanitizer (section suivante) est préférable.

**Multi-threading.** Memcheck fonctionne avec les programmes multi-threadés, mais le ralentissement peut modifier le timing des threads et masquer certaines data races (ou en provoquer de nouvelles). Pour les problèmes de concurrence, Valgrind propose un autre outil, **Helgrind**, et le sanitizer dédié est **ThreadSanitizer** (section 29.4.3).

**Architecture.** Valgrind ne supporte que Linux (et marginalement macOS et Solaris). Il ne fonctionne pas sous Windows ni dans les conteneurs Docker basés sur des images trop minimales (il a besoin de bibliothèques de débogage).

Malgré ces limites, Valgrind reste l'outil de référence pour l'analyse mémoire en C++ sous Linux. Son rapport coût/bénéfice est imbattable pour un premier diagnostic : aucune recompilation, aucune modification du code, et un rapport détaillé en quelques secondes.
