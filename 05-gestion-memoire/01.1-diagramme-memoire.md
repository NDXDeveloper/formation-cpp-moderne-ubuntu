# 5.1.1 — Diagramme mémoire d'un processus

## L'espace d'adressage virtuel

Quand vous lancez un programme compilé sur Ubuntu, le noyau Linux ne lui donne pas un accès direct à la RAM physique. Il lui attribue un **espace d'adressage virtuel** — une vision privée et isolée de la mémoire, traduite en adresses physiques par le matériel (MMU, *Memory Management Unit*) en collaboration avec le noyau.

Sur un système 64 bits, cet espace virtuel est théoriquement immense (2⁴⁸ octets soit 256 To sur x86_64 en pratique), mais seules les régions effectivement utilisées par le programme sont rattachées à de la mémoire physique. Le reste n'existe tout simplement pas du point de vue matériel — tout accès à une adresse non allouée provoque un `Segmentation fault`.

Cette virtualisation a deux conséquences importantes pour nous. Premièrement, chaque processus croit avoir toute la mémoire pour lui — deux programmes peuvent utiliser la même adresse virtuelle sans conflit, car elles pointent vers des emplacements physiques différents. Deuxièmement, la disposition des régions mémoire est **standardisée** : le noyau et le compilateur s'accordent sur un agencement conventionnel que nous allons maintenant détailler.

## Détail de chaque région

### Le segment `.text` (code)

C'est ici que résident les instructions machine de votre programme — le résultat de la compilation. Cette zone est en **lecture seule** : toute tentative d'écriture provoque un `Segmentation fault`. Cette protection empêche le code de se modifier lui-même accidentellement (ou malicieusement).

Sur Linux, si plusieurs instances du même programme tournent simultanément, elles **partagent** les mêmes pages physiques pour `.text`. Le noyau est assez intelligent pour ne charger le code machine qu'une seule fois en RAM.

### Le segment `.rodata` (données en lecture seule)

Les littéraux de chaînes (`"Hello, world!"`), les constantes globales `const`, et certaines données `constexpr` atterrissent ici. Comme `.text`, cette zone est protégée en écriture :

```cpp
const char* msg = "Bonjour";  // "Bonjour" est dans .rodata
                                // msg (le pointeur) est sur la stack ou dans .data
```

Tenter de modifier le contenu pointé — par exemple via un `const_cast` maladroit — est un **comportement indéfini** qui provoque généralement un crash.

### Le segment `.data` (données initialisées)

Les variables globales et statiques **explicitement initialisées** à une valeur non nulle sont stockées ici. Leurs valeurs initiales occupent de la place dans l'exécutable ELF sur disque :

```cpp
int compteur_global = 42;           // .data  
static double ratio = 3.14;         // .data  

void fonction() {
    static int appels = 1;          // .data (static locale, initialisée à 1)
}
```

### Le segment `.bss` (données non initialisées)

Les variables globales et statiques qui ne sont pas initialisées, ou qui sont initialisées à zéro, sont placées dans `.bss`. La particularité de ce segment est qu'il **n'occupe pas de place dans l'exécutable** sur disque — seule sa taille est enregistrée. Au chargement du programme, le noyau alloue la zone et la remplit de zéros :

```cpp
int tableau_global[10000];          // .bss — 40 000 octets en mémoire,
                                    //        mais ~0 octet dans l'exécutable

static long compteur;               // .bss — initialisé à 0 par le système
```

C'est une optimisation importante : un programme qui déclare de grands tableaux globaux non initialisés ne produit pas un exécutable de plusieurs mégaoctets.

### La région mmap (memory-mapped)

Entre le heap et la stack se trouve une zone utilisée pour les **mappings mémoire**. C'est ici que le chargeur dynamique (`ld-linux`) place les bibliothèques partagées (`.so`) comme `libc.so` et `libstdc++.so`. Cette zone sert également pour les fichiers mappés en mémoire (`mmap()`) et pour les très grandes allocations dynamiques — `malloc` (et donc `new`) bascule automatiquement vers `mmap` au-delà d'un seuil (typiquement 128 Ko avec glibc).

### Le heap

Zone de mémoire dynamique, gérée par l'allocateur (glibc `malloc`/`free`, que `new`/`delete` utilisent en interne). Le heap croît vers les adresses hautes. Nous l'explorerons en profondeur dans la section 5.1.3.

### La stack

Zone de mémoire automatique pour les appels de fonctions. Chaque thread possède sa propre stack. Elle croît vers les adresses basses. Nous l'étudierons en détail dans la section 5.1.2.

## Voir la carte mémoire d'un vrai processus

Linux expose la carte mémoire de chaque processus via le pseudo-système de fichiers `/proc`. Prenons un programme minimal :

```cpp
// memoire_demo.cpp
#include <iostream>
#include <unistd.h>  // getpid(), pause()

int variable_bss[1000];                // .bss  
int variable_data = 42;                // .data  
const char* message = "Hello";         // pointeur dans .data, chaîne dans .rodata  

int main() {
    int variable_stack = 10;            // stack
    int* variable_heap = new int(99);   // heap

    std::cout << "PID: " << getpid() << "\n";
    std::cout << "Adresses :\n";
    std::cout << "  code  (main)    : " << reinterpret_cast<void*>(&main) << "\n";
    std::cout << "  .rodata (msg)   : " << static_cast<const void*>(message) << "\n";
    std::cout << "  .data           : " << &variable_data << "\n";
    std::cout << "  .bss            : " << &variable_bss << "\n";
    std::cout << "  heap            : " << variable_heap << "\n";
    std::cout << "  stack           : " << &variable_stack << "\n";

    std::cout << "\nAppuyez sur Entrée pour quitter...\n";
    std::cin.get();

    delete variable_heap;
    return 0;
}
```

Compilez et lancez ce programme :

```bash
g++ -std=c++17 -o memoire_demo memoire_demo.cpp
./memoire_demo
```

Vous obtiendrez une sortie ressemblant à ceci (les adresses exactes varient à cause de l'ASLR) :

```
PID: 12345  
Adresses :  
  code  (main)    : 0x5612a3401189
  .rodata (msg)   : 0x5612a3402004
  .data           : 0x5612a3404010
  .bss            : 0x5612a3404040
  heap            : 0x5612a4e912a0
  stack           : 0x7ffd3b2a1e4c
```

Observez la hiérarchie des adresses : `.text` et `.rodata` sont dans les adresses basses (préfixe `0x56...`), le heap est un peu plus haut, et la stack est dans les adresses très hautes (préfixe `0x7ffd...`). C'est exactement le diagramme vu plus haut.

## Inspection avec `/proc/<PID>/maps`

Pendant que le programme est en pause (avant d'appuyer sur Entrée), ouvrez un second terminal et inspectez sa carte mémoire :

```bash
cat /proc/12345/maps
```

La sortie affiche chaque région avec ses permissions et son origine :

```
5612a3400000-5612a3402000 r-xp  00000000  08:01  1234  /home/user/memoire_demo   ← .text
5612a3402000-5612a3403000 r--p  00002000  08:01  1234  /home/user/memoire_demo   ← .rodata
5612a3403000-5612a3405000 rw-p  00003000  08:01  1234  /home/user/memoire_demo   ← .data + .bss
5612a4e91000-5612a4eb2000 rw-p  00000000  00:00  0     [heap]
7f8c12a00000-7f8c12be8000 r-xp  00000000  08:01  5678  /lib/x86_64.../libc.so.6 ← mmap (libc)
...
7ffd3b282000-7ffd3b2a3000 rw-p  00000000  00:00  0     [stack]
```

Les colonnes de permissions se lisent ainsi : `r` = lecture, `w` = écriture, `x` = exécution, `p` = privé (copy-on-write). On retrouve bien `.text` en `r-xp` (exécutable, non modifiable), `.data`/`.bss` en `rw-p` (modifiable), et le heap et la stack en `rw-p`.

## L'outil `pmap` : une vue plus lisible

La commande `pmap` reformate les informations de `/proc/<PID>/maps` de manière plus compacte :

```bash
pmap -x 12345
```

```
Address           Kbytes  RSS  Dirty  Mode  Mapping
00005612a3400000       8    8      0  r-x-- memoire_demo
00005612a3402000       4    4      0  r---- memoire_demo
00005612a3403000       8    8      8  rw--- memoire_demo
00005612a4e91000     132   12     12  rw---   [ anon ]         ← heap
...
00007ffd3b282000     132   16     16  rw---   [ stack ]
                  ------  ---  -----
total kB          ...
```

La colonne **RSS** (*Resident Set Size*) indique la mémoire physique réellement utilisée. La colonne **Dirty** montre les pages modifiées en mémoire. On constate que `.text` n'a aucune page *dirty* (lecture seule), alors que le heap et la stack en ont (elles contiennent des données modifiées par le programme).

## L'ASLR : pourquoi les adresses changent à chaque exécution

Si vous relancez le programme plusieurs fois, vous constaterez que les adresses changent. Ce n'est pas un hasard : c'est l'**ASLR** (*Address Space Layout Randomization*), une protection de sécurité activée par défaut sur les systèmes Linux modernes.

L'ASLR randomise la position de la stack, du heap, des bibliothèques partagées et même du code exécutable (quand il est compilé en PIE, *Position-Independent Executable*, ce qui est le cas par défaut avec GCC et Clang sur Ubuntu). L'objectif est de rendre les attaques par corruption de mémoire (buffer overflow, return-oriented programming) beaucoup plus difficiles, car l'attaquant ne peut pas prédire les adresses.

Pour désactiver temporairement l'ASLR à des fins de débogage ou d'apprentissage :

```bash
# Désactive l'ASLR pour cette session shell uniquement
setarch $(uname -m) -R ./memoire_demo
```

Avec l'ASLR désactivé, les adresses deviennent reproductibles d'une exécution à l'autre, ce qui peut faciliter les expérimentations sur la disposition mémoire.

> ⚠️ Ne désactivez jamais l'ASLR en production. C'est une protection de sécurité critique.

## Synthèse

Le diagramme mémoire d'un processus n'est pas un concept théorique : c'est une réalité que vous pouvez observer et inspecter sur votre machine. Chaque variable que vous déclarez, chaque objet que vous allouez, chaque fonction que vous appelez aboutit dans une région précise de ce diagramme.
