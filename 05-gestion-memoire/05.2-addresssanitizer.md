# 5.5.2 — AddressSanitizer : Compilation avec -fsanitize=address

## Principe de fonctionnement

AddressSanitizer (souvent abrégé **ASan**) est un détecteur d'erreurs mémoire intégré directement dans les compilateurs GCC et Clang. Contrairement à Valgrind qui instrumente le binaire au moment de l'exécution, ASan insère du code de vérification **pendant la compilation**. Chaque accès mémoire, chaque allocation et chaque libération sont encadrés par des vérifications générées par le compilateur lui-même.

Le mécanisme repose sur un concept de **shadow memory** : pour chaque tranche de 8 octets de la mémoire de votre programme, ASan maintient 1 octet de métadonnées qui indique quels octets de cette tranche sont accessibles. Avant chaque lecture ou écriture, le code instrumenté consulte la shadow memory pour vérifier la légitimité de l'accès. Si l'accès est invalide, ASan arrête immédiatement le programme et produit un rapport détaillé.

De plus, ASan place des **zones empoisonnées** (*redzones*) autour de chaque allocation — avant et après le bloc sur le heap, autour des variables locales sur la stack, et autour des variables globales. Toute écriture dans une redzone signale un buffer overflow.

```
Mémoire vue par ASan :

  [redzone][  votre bloc alloué  ][redzone][  bloc suivant  ][redzone]
  ^^^^^^^^                        ^^^^^^^^
  empoisonné                      empoisonné
  écriture ici → erreur détectée
```

Ce mécanisme explique les deux caractéristiques clés d'ASan : un **ralentissement modéré** (2-3× au lieu de 10-30× pour Valgrind, car les vérifications sont de simples consultations mémoire sans traduction binaire) et une **surconsommation mémoire** (environ 2-3× la mémoire originale, à cause de la shadow memory et des redzones).

---

## Compilation avec ASan

Activer ASan ne demande qu'un flag de compilation et de liaison :

```bash
g++ -std=c++17 -g -O1 -fsanitize=address -o programme programme.cpp
```

Ou avec Clang :

```bash
clang++ -std=c++17 -g -O1 -fsanitize=address -o programme programme.cpp
```

Le flag `-fsanitize=address` doit être présent **à la fois à la compilation et à l'édition de liens**. Si vous compilez et liez en deux étapes, ajoutez-le aux deux :

```bash
g++ -std=c++17 -g -O1 -fsanitize=address -c fichier.cpp -o fichier.o  
g++ -fsanitize=address fichier.o -o programme  
```

### Choix du niveau d'optimisation

Contrairement à Valgrind où `-O0` est recommandé, ASan fonctionne bien avec `-O1`. Ce niveau offre un bon compromis : il active les optimisations légères qui rendent le programme suffisamment rapide pour les tests, tout en préservant une correspondance lisible entre le rapport et le code source. Avec `-O2` ou `-O3`, les rapports restent exploitables mais certains numéros de lignes peuvent être décalés à cause de l'inlining.

### Combiner avec d'autres sanitizers

ASan se combine très bien avec **UndefinedBehaviorSanitizer** (UBSan). C'est la combinaison recommandée pour le développement quotidien :

```bash
g++ -std=c++17 -g -O1 -fsanitize=address,undefined -o programme programme.cpp
```

En revanche, ASan **ne peut pas** être combiné avec **ThreadSanitizer** (`-fsanitize=thread`) ni avec **MemorySanitizer** (`-fsanitize=memory`) dans la même compilation. Ces sanitizers utilisent des mécanismes de shadow memory incompatibles. Si vous avez besoin des trois, créez des configurations de build séparées.

## Anatomie d'un rapport ASan

Les rapports ASan sont denses mais très structurés. Décortiquons chaque section.

### Le type d'erreur

La première ligne identifie précisément le problème :

```
==12345==ERROR: AddressSanitizer: heap-buffer-overflow on address 0x602000000054
```

ASan nomme ses erreurs de manière explicite. Les types que vous rencontrerez le plus souvent :

- **`heap-buffer-overflow`** — écriture ou lecture au-delà d'un bloc alloué sur le heap
- **`stack-buffer-overflow`** — dépassement d'un tableau local sur la stack
- **`global-buffer-overflow`** — dépassement d'un tableau global
- **`heap-use-after-free`** — accès à un bloc déjà libéré
- **`double-free`** — appel de `delete`/`free` sur un bloc déjà libéré
- **`stack-use-after-return`** — accès à une variable locale après le retour de sa fonction
- **`alloc-dealloc-mismatch`** — mélange `new`/`free` ou `malloc`/`delete`

### La pile d'appels de l'erreur

```
WRITE of size 4 at 0x602000000054 thread T0
    #0 0x401233 in main /home/user/bugs_memoire.cpp:12:15
```

ASan indique l'opération (`WRITE of size 4`), le thread (`T0` = thread principal), et la pile d'appels complète avec fichiers et numéros de lignes. Le format `fichier.cpp:12:15` signifie ligne 12, colonne 15 — plus précis que Valgrind qui ne donne que la ligne.

### Le contexte de l'allocation

```
0x602000000054 is located 0 bytes after 20-byte region [0x602000000040,0x602000000054)
allocated by thread T0 here:
    #0 0x7f12345abcde in operator new[](unsigned long)
    #1 0x4011ef in main /home/user/bugs_memoire.cpp:11:21
```

ASan montre que l'adresse fautive se trouve juste après un bloc de 20 octets (notre `new int[5]`), et fournit la pile d'appels de l'allocation. Pour un use-after-free, ASan montre également la pile d'appels de la libération — trois piles au total (accès, libération, allocation), comme Valgrind.

### La shadow memory

La section en bas du rapport affiche un extrait de la shadow memory autour de l'adresse problématique :

```
Shadow bytes around the buggy address:
  0x602000000040: 00 fa fa fa 04[fa]fa fa
```

Chaque octet de shadow représente 8 octets de mémoire réelle. La valeur `00` signifie "8 octets accessibles", `04` signifie "4 octets accessibles sur 8" (fin du bloc de 20 octets = 2×8 + 4), et `fa` signifie "heap redzone — accès interdit". Les crochets `[fa]` marquent l'octet de shadow correspondant à l'adresse de l'erreur. Cette visualisation est rarement nécessaire pour le diagnostic courant, mais elle peut être précieuse pour les cas complexes.

## Les rapports pour chaque type d'erreur

### Use-after-free

Si on corrige le buffer overflow et qu'on relance, ASan détecte le use-after-free :

```
==12345==ERROR: AddressSanitizer: heap-use-after-free on address 0x602000000010
READ of size 4 at 0x602000000010 thread T0
    #0 0x401267 in main /home/user/bugs_memoire.cpp:17:19

0x602000000010 is located 0 bytes inside of 4-byte region [0x602000000010,0x602000000014)
freed by thread T0 here:
    #0 0x7f1234500000 in operator delete(void*, unsigned long)
    #1 0x401254 in main /home/user/bugs_memoire.cpp:16:5

previously allocated by thread T0 here:
    #0 0x7f1234500000 in operator new(unsigned long)
    #1 0x401240 in main /home/user/bugs_memoire.cpp:15:25
```

Trois piles d'appels : l'accès invalide (ligne 17), la libération (ligne 16) et l'allocation originale (ligne 15). Le diagnostic est immédiat.

### Double free

```
==12345==ERROR: AddressSanitizer: attempting double-free on 0x602000000010
    #0 0x7f1234500000 in operator delete(void*, unsigned long)
    #1 0x40128a in main /home/user/bugs_memoire.cpp:21:5

0x602000000010 is located 0 bytes inside of 4-byte region [0x602000000010,0x602000000014)
freed by thread T0 here:
    #0 0x7f1234500000 in operator delete(void*, unsigned long)
    #1 0x401276 in main /home/user/bugs_memoire.cpp:20:5

previously allocated by thread T0 here:
    #0 0x7f1234500000 in operator new(unsigned long)
    #1 0x401262 in main /home/user/bugs_memoire.cpp:19:25
```

ASan distingue clairement le second `delete` (ligne 21), le premier `delete` (ligne 20) et l'allocation (ligne 19).

### Fuites mémoire (LeakSanitizer)

ASan intègre automatiquement **LeakSanitizer** (LSan), qui s'exécute à la fin du programme et signale les blocs non libérés :

```
==12345==ERROR: LeakSanitizer: detected memory leaks

Direct leak of 400 byte(s) in 1 object(s) allocated from:
    #0 0x7f12345abcde in operator new[](unsigned long)
    #1 0x4011c0 in main /home/user/bugs_memoire.cpp:7:21

SUMMARY: AddressSanitizer: 400 byte(s) leaked in 1 allocation(s).
```

LeakSanitizer est activé par défaut avec ASan sous Linux. Si vous voulez uniquement la détection de fuites sans l'instrumentation complète d'ASan, vous pouvez utiliser LSan seul :

```bash
g++ -std=c++17 -g -O1 -fsanitize=leak -o programme programme.cpp
```

## Détection des erreurs stack

Un avantage majeur d'ASan sur Valgrind est la détection des buffer overflows **sur la stack**. Valgrind ne les détecte que partiellement, car il ne place pas de redzones autour des variables locales. ASan le fait :

```cpp
// stack_overflow.cpp
int main() {
    int tableau[5] = {1, 2, 3, 4, 5};
    return tableau[10];    // lecture hors limites sur la stack
}
```

```bash
g++ -std=c++17 -g -O1 -fsanitize=address -o stack_overflow stack_overflow.cpp
./stack_overflow
```

```
==12345==ERROR: AddressSanitizer: stack-buffer-overflow on address 0x7ffd12345698
READ of size 4 at 0x7ffd12345698 thread T0
    #0 0x401156 in main /home/user/stack_overflow.cpp:3:12

Address 0x7ffd12345698 is located in stack of thread T0 at offset 56 in frame
    #0 0x401100 in main /home/user/stack_overflow.cpp:1

  This frame has 1 object(s):
    [32, 52) 'tableau' (line 2)    <== Memory access at offset 56 overflows this variable
```

ASan identifie précisément la variable locale concernée (`tableau`, ligne 2) et la distance du dépassement. C'est une information que Valgrind ne peut pas fournir.

## Configuration via ASAN_OPTIONS

Le comportement d'ASan se personnalise via la variable d'environnement `ASAN_OPTIONS`, qui accepte une liste d'options séparées par des deux-points :

### Continuer après la première erreur

Par défaut, ASan arrête le programme à la première erreur. Pour continuer et détecter plusieurs erreurs en une seule exécution :

```bash
ASAN_OPTIONS=halt_on_error=0 ./programme
```

Attention : après une première erreur mémoire, l'état du programme est potentiellement corrompu. Les erreurs suivantes peuvent être des conséquences de la première plutôt que des bugs indépendants.

### Augmenter la quarantaine

Quand un bloc est libéré, ASan ne le rend pas immédiatement à l'allocateur — il le place en **quarantaine**, une zone empoisonnée qui permet de détecter les use-after-free même si la réallocation est différée. La taille par défaut de la quarantaine est de 256 Mo. Pour les programmes avec des use-after-free tardifs :

```bash
ASAN_OPTIONS=quarantine_size_mb=512 ./programme
```

### Désactiver la détection de fuites

Si vous ne vous intéressez qu'aux erreurs d'accès et pas aux fuites :

```bash
ASAN_OPTIONS=detect_leaks=0 ./programme
```

### Combinaison courante

```bash
ASAN_OPTIONS=halt_on_error=0:detect_stack_use_after_return=1:check_initialization_order=1 ./programme
```

L'option `detect_stack_use_after_return=1` active la détection des accès aux variables locales après le retour de la fonction — un type d'erreur que Valgrind ne détecte pas du tout. Cette option augmente la consommation mémoire car ASan déplace les variables locales sur le heap pour les surveiller.

## Intégration dans CMake

Pour un projet CMake, la manière propre d'activer ASan est de créer une configuration de build dédiée. Ajoutez dans votre `CMakeLists.txt` :

```cmake
option(ENABLE_ASAN "Enable AddressSanitizer" OFF)

if(ENABLE_ASAN)
    add_compile_options(-fsanitize=address,undefined -fno-omit-frame-pointer)
    add_link_options(-fsanitize=address,undefined)
endif()
```

Le flag `-fno-omit-frame-pointer` n'est pas strictement requis, mais il améliore la qualité des piles d'appels dans les rapports en s'assurant que le frame pointer est toujours disponible.

Utilisation :

```bash
mkdir build-asan && cd build-asan  
cmake -DENABLE_ASAN=ON -DCMAKE_BUILD_TYPE=Debug ..  
cmake --build .  
ctest    # tous les tests sont exécutés avec ASan  
```

Une approche alternative utilise les **CMake Presets** (section 27.6) pour encapsuler cette configuration :

```json
{
    "name": "asan",
    "displayName": "Debug + AddressSanitizer",
    "inherits": "debug",
    "cacheVariables": {
        "ENABLE_ASAN": "ON"
    }
}
```

## Intégration dans un pipeline CI/CD

ASan est suffisamment rapide pour être exécuté dans chaque pipeline. Voici un exemple de job GitHub Actions :

```yaml
sanitizer-check:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4

    - name: Configure with ASan
      run: |
        cmake -B build-asan \
              -DCMAKE_BUILD_TYPE=Debug \
              -DENABLE_ASAN=ON

    - name: Build
      run: cmake --build build-asan

    - name: Run tests under ASan
      run: |
        cd build-asan
        ASAN_OPTIONS=halt_on_error=1:detect_leaks=1 ctest --output-on-failure
```

Si ASan détecte une erreur, le test échoue avec un code de sortie non nul, ce qui fait échouer le job. Le rapport complet est capturé dans la sortie du job grâce à `--output-on-failure`.

La pratique recommandée est de maintenir **deux jobs de test** dans votre pipeline : un sans sanitizer (pour la vitesse et la mesure de performance) et un avec ASan+UBSan (pour la détection d'erreurs). Les projets les plus rigoureux ajoutent un troisième job avec ThreadSanitizer pour la concurrence, et un passage ponctuel de Valgrind pour la mémoire non initialisée.

## ASan en pratique : les pièges à connaître

### Toutes les unités de compilation doivent être instrumentées

Si votre programme lie des bibliothèques compilées **sans** ASan, les accès mémoire dans ces bibliothèques ne sont pas vérifiés. ASan détectera quand même les problèmes liés aux allocations (car il intercepte `malloc`/`free`), mais les buffer overflows internes à une bibliothèque non instrumentée passeront inaperçus.

Pour les bibliothèques système (libc, libstdc++), ce n'est généralement pas un problème — ASan fournit des intercepteurs pour les fonctions standard (`memcpy`, `strcpy`, etc.). Mais pour vos propres bibliothèques partagées, assurez-vous qu'elles sont compilées avec le même flag `-fsanitize=address`.

### Impact sur les performances des tests

Le ralentissement de 2-3× est généralement acceptable pour les tests unitaires. Mais pour les tests d'intégration impliquant du réseau, des bases de données ou des timeouts serrés, le ralentissement peut provoquer des échecs spurieux. La solution est d'augmenter les timeouts dans la configuration de test ASan, ou d'exclure certains tests longs de l'exécution sous sanitizer.

### Ne jamais déployer un binaire ASan en production

Un binaire compilé avec ASan est plus lent, consomme plus de mémoire, et s'arrête à la première erreur détectée. Ce n'est pas un binaire de production. Maintenez des configurations de build séparées : `Debug`, `Release`, et `Sanitize`.

## Synthèse : ASan dans votre workflow

AddressSanitizer est l'outil qui offre le meilleur rapport entre facilité d'utilisation, couverture de détection et impact sur les performances. Sa place naturelle est dans la boucle de développement quotidienne et dans le pipeline CI/CD.

En combinaison avec Valgrind pour les audits ponctuels et MemorySanitizer pour les lectures de mémoire non initialisée, il forme un filet de sécurité qui attrape la grande majorité des erreurs mémoire avant qu'elles n'atteignent la production. Mais la meilleure défense reste la prévention : les smart pointers (chapitre 9), le RAII (chapitre 6), les conteneurs de la STL (chapitres 13-14) et `std::span` (section 13.5) éliminent structurellement les catégories entières d'erreurs que ces outils détectent. Les outils sont un filet de sécurité ; le C++ moderne est la corde de rappel.
