# 5.1.2 — Caractéristiques de la Stack

## Un mécanisme matériel, pas logiciel

La stack n'est pas une invention du langage C++ ni même du système d'exploitation. C'est un mécanisme supporté directement par le **processeur**. Sur x86_64, un registre dédié — `RSP` (*Register Stack Pointer*) — pointe en permanence vers le sommet de la pile. Un second registre, `RBP` (*Register Base Pointer*), pointe vers la base du stack frame courant, ce qui permet de naviguer facilement entre les variables locales.

Allouer de la mémoire sur la stack revient littéralement à **soustraire une valeur de RSP**. Libérer cette mémoire revient à **ajouter une valeur à RSP**. Ces deux opérations prennent chacune un cycle CPU. Il n'y a pas de recherche d'espace libre, pas de structure de données à mettre à jour, pas d'appel système. C'est ce qui rend la stack extraordinairement rapide.

---

## Le stack frame : anatomie d'un appel de fonction

Chaque fois que vous appelez une fonction, le processeur crée un **stack frame** (ou *cadre d'activation*) sur la pile. Ce cadre contient tout ce dont la fonction a besoin pour s'exécuter et tout ce qu'il faut pour revenir proprement à l'appelant.

Prenons un exemple concret :

```cpp
int additionner(int a, int b) {
    int resultat = a + b;
    return resultat;
}

int main() {
    int x = 10;
    int y = 20;
    int somme = additionner(x, y);
    return 0;
}
```

Au moment où `additionner` s'exécute, la stack ressemble à ceci (les adresses décroissent vers le bas, puisque la stack croît vers les adresses basses) :

```
Adresses hautes
┌──────────────────────────────────────────┐
│           Stack frame de main()          │
│  ┌────────────────────────────────────┐  │
│  │  x = 10                            │  │
│  │  y = 20                            │  │
│  │  somme = (pas encore assigné)      │  │
│  └────────────────────────────────────┘  │
│  RBP de main ──────────────────────────► │
├──────────────────────────────────────────┤
│       Stack frame de additionner()       │
│  ┌────────────────────────────────────┐  │
│  │  Adresse de retour → main          │  │
│  │  Ancien RBP (sauvegardé)           │  │
│  │  a = 10  (paramètre)               │  │
│  │  b = 20  (paramètre)               │  │
│  │  resultat = 30                     │  │
│  └────────────────────────────────────┘  │
│  RSP (sommet de pile) ──────────────────►│
└──────────────────────────────────────────┘
Adresses basses
```

> **Note sur les conventions d'appel :** sur x86_64 Linux (convention System V AMD64), les premiers paramètres entiers sont passés dans les registres (`RDI`, `RSI`, `RDX`, `RCX`, `R8`, `R9`) et non sur la stack. Le schéma ci-dessus est simplifié pour illustrer le concept. Dans la réalité, `a` et `b` arrivent dans `RDI` et `RSI`, et le compilateur peut les placer sur la stack uniquement s'il manque de registres ou si leur adresse est prise.

Quand `additionner` se termine, son stack frame est détruit en restaurant l'ancien `RSP`. L'exécution reprend dans `main` à l'adresse de retour sauvegardée. Les octets occupés par le frame de `additionner` ne sont pas "effacés" — ils restent en mémoire, mais ils sont considérés comme libres et seront écrasés par le prochain appel de fonction.

## La durée de vie est liée au scope

La propriété la plus importante de la stack du point de vue du programmeur C++ est que la durée de vie des variables est **automatiquement liée à leur portée lexicale** (scope). Quand l'exécution quitte un bloc `{}`, toutes les variables déclarées dans ce bloc sont détruites — et si ce sont des objets C++, leurs **destructeurs sont appelés** :

```cpp
#include <string>
#include <iostream>

void demonstrer_scope() {
    std::string prenom = "Alice";       // construit sur la stack
    std::cout << prenom << "\n";

    {
        std::string nom = "Dupont";     // construit sur la stack (scope interne)
        std::cout << nom << "\n";
    }   // ← destructeur de 'nom' appelé ici — mémoire interne libérée

    // 'nom' n'existe plus, mais 'prenom' est toujours valide
    std::cout << prenom << "\n";

}   // ← destructeur de 'prenom' appelé ici
```

C'est cette mécanique de destruction automatique en fin de scope qui constitue le socle du **RAII** (*Resource Acquisition Is Initialization*), le pattern le plus fondamental du C++ moderne. Nous l'étudierons en détail au chapitre 6, mais l'idée essentielle est que la stack garantit que tout objet correctement construit sera détruit de manière déterministe — pas "un jour", pas "quand le garbage collector passera", mais **exactement à la fermeture du bloc englobant**, dans l'ordre inverse de construction.

---

## Ordre de destruction : LIFO garanti

Les objets sur la stack sont détruits dans l'**ordre inverse** de leur construction. C'est une conséquence directe de la structure LIFO (*Last In, First Out*) de la pile, et c'est une garantie du langage :

```cpp
#include <iostream>

struct Trace {
    std::string nom;
    Trace(std::string n) : nom(std::move(n)) {
        std::cout << "  Construction de " << nom << "\n";
    }
    ~Trace() {
        std::cout << "  Destruction de " << nom << "\n";
    }
};

void ordre_lifo() {
    std::cout << "Début\n";
    Trace a("A");
    Trace b("B");
    Trace c("C");
    std::cout << "Fin du bloc\n";
}
```

Sortie :

```
Début
  Construction de A
  Construction de B
  Construction de C
Fin du bloc
  Destruction de C
  Destruction de B
  Destruction de A
```

Cet ordre inversé n'est pas un détail anecdotique. Il garantit que si un objet `B` dépend d'un objet `A` construit avant lui, `B` sera détruit **avant** `A` — au moment de sa destruction, `A` est toujours valide. C'est une propriété cruciale pour la gestion correcte des ressources.

## Taille de la stack : limitée et configurable

La stack n'est pas extensible à l'infini. Sous Linux, chaque thread dispose d'une stack de taille fixe, déterminée au démarrage. La valeur par défaut est généralement de **8 Mo** pour le thread principal :

```bash
# Afficher la taille maximale de la stack (en Ko)
ulimit -s
# Sortie typique : 8192 (soit 8 Mo)
```

Cette limite peut être modifiée :

```bash
# Augmenter à 16 Mo pour la session shell courante
ulimit -s 16384

# Taille illimitée (déconseillé — supprime la détection de stack overflow)
ulimit -s unlimited
```

Pour les threads créés avec `std::thread` ou `pthread_create`, la taille de stack par défaut est également de 8 Mo sur la plupart des distributions, mais elle est configurable via les attributs pthread :

```cpp
#include <pthread.h>

pthread_attr_t attr;  
pthread_attr_init(&attr);  
pthread_attr_setstacksize(&attr, 2 * 1024 * 1024);  // 2 Mo  

pthread_t thread;  
pthread_create(&thread, &attr, ma_fonction, nullptr);  
pthread_attr_destroy(&attr);  
```

Pourquoi cette limite ? Deux raisons. D'abord, elle permet au système de **détecter les récursions infinies** : quand la stack déborde, le programme reçoit un signal `SIGSEGV` plutôt que de consommer silencieusement toute la mémoire. Ensuite, dans un programme multi-thread, chaque thread a sa propre stack. Si vous créez 100 threads avec des stacks de 8 Mo, cela représente déjà 800 Mo d'espace d'adressage réservé (même si les pages physiques ne sont allouées que lorsqu'elles sont effectivement utilisées, grâce à l'allocation paresseuse du noyau).

---

## Stack overflow : quand la pile déborde

Un **stack overflow** se produit quand un programme tente d'utiliser plus d'espace que ce que la stack peut offrir. La cause la plus fréquente est une **récursion non bornée** :

```cpp
// ⚠️ Récursion infinie → stack overflow garanti
void recursion_infinie() {
    int tableau[1000];            // 4 000 octets par appel
    recursion_infinie();          // chaque appel empile un nouveau frame
}
```

Chaque appel récursif ajoute un stack frame. Avec un tableau local de 4 000 octets par frame, la stack de 8 Mo est saturée après environ 2 000 appels. Le résultat est un signal `SIGSEGV` — le fameux *Segmentation fault*.

Mais la récursion infinie n'est pas la seule cause. L'allocation de **structures volumineuses** sur la stack peut aussi provoquer un débordement :

```cpp
void allocation_excessive() {
    double matrice[1000][1000];   // 8 Mo d'un coup → stack overflow probable
}
```

Un tableau de 1 000 × 1 000 `double` occupe 8 000 000 octets, soit la quasi-totalité de la stack par défaut. En pratique, le frame de la fonction appelante occupe déjà une partie de la pile, donc ce code provoque très probablement un crash.

La règle pratique est : **n'allouez jamais de grandes structures sur la stack**. Si vous avez besoin d'un tableau de taille significative (au-delà de quelques kilo-octets), utilisez un `std::vector` qui stocke ses données sur le heap, ou allouez explicitement sur le heap.

## Récursion correcte et profondeur raisonnable

La récursion n'est pas à proscrire — c'est un outil puissant quand la profondeur est maîtrisée. Un parcours d'arbre binaire équilibré de 1 million de nœuds ne nécessite qu'environ 20 niveaux de récursion (log₂(1 000 000) ≈ 20), ce qui est parfaitement gérable :

```cpp
struct Noeud {
    int valeur;
    Noeud* gauche;
    Noeud* droite;
};

int somme(const Noeud* n) {
    if (n == nullptr) return 0;
    return n->valeur + somme(n->gauche) + somme(n->droite);
    // Profondeur max ~20 pour un arbre équilibré de 1M nœuds → sans risque
}
```

En revanche, un parcours de liste chaînée de 1 million d'éléments par récursion empile 1 million de frames — c'est un stack overflow assuré. La version itérative est alors impérative :

```cpp
// ❌ Récursif — stack overflow pour une longue liste
int somme_recursive(const Noeud* n) {
    if (n == nullptr) return 0;
    return n->valeur + somme_recursive(n->suivant);  // 1M appels → crash
}

// ✅ Itératif — utilise un espace stack constant
int somme_iterative(const Noeud* n) {
    int total = 0;
    while (n != nullptr) {
        total += n->valeur;
        n = n->suivant;
    }
    return total;
}
```

> **Note :** contrairement à certains langages fonctionnels, le C++ ne garantit pas l'optimisation de la récursion terminale (*tail call optimization*). GCC et Clang **peuvent** l'appliquer à `-O2` ou supérieur, mais ce n'est pas garanti par le standard. Ne comptez pas dessus pour éviter un stack overflow.

## Observer la stack avec GDB

GDB permet d'inspecter la stack en temps réel. Voici les commandes les plus utiles :

```bash
g++ -std=c++17 -g -O0 -o programme programme.cpp  
gdb ./programme  
```

```gdb
(gdb) break main
(gdb) run
(gdb) backtrace              # Affiche la pile d'appels complète
(gdb) info frame             # Détails du stack frame courant
(gdb) info locals            # Variables locales du frame courant
(gdb) print &ma_variable     # Adresse d'une variable locale (adresse stack)
(gdb) info registers rsp rbp # Valeurs des registres stack pointer et base pointer
```

La commande `backtrace` (abrégée `bt`) est particulièrement précieuse. Elle affiche l'enchaînement complet des appels de fonctions, du frame courant jusqu'à `main` — c'est le premier réflexe à avoir lors du débogage d'un crash.

## Synthèse : les propriétés de la stack

Récapitulons les caractéristiques essentielles de la stack qui en font la zone mémoire privilégiée pour les variables locales :

**La vitesse.** Allouer et libérer de la mémoire sur la stack coûte un ou deux cycles CPU. Aucun autre mécanisme d'allocation ne peut rivaliser.

**Le déterminisme.** La destruction se produit à un point précis et prévisible du programme (la sortie du scope), dans un ordre garanti (LIFO inverse). Il n'y a aucune incertitude sur le "quand" ni le "dans quel ordre".

**La sécurité.** Pas de fuite mémoire possible — tout ce qui est alloué sur la stack est automatiquement libéré. Pas de double free — la libération n'est pas sous votre contrôle. Pas de fragmentation — la structure LIFO l'empêche structurellement.

**Les limites.** La taille est fixe et relativement modeste (8 Mo par défaut). La durée de vie est rigidement liée au scope — impossible de faire survivre une variable au-delà de sa fonction sans la copier ou la déplacer. Et le partage entre threads est impossible : chaque thread a sa propre stack.

C'est précisément quand ces limites deviennent un obstacle — données trop volumineuses, durée de vie flexible, partage entre composants — que le heap entre en jeu. C'est l'objet de la section suivante.
