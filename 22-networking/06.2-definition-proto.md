# 22.6.2 — Définition de services avec `.proto`

## Section 22.6 : gRPC et Protocol Buffers

---

## Introduction

Le fichier `.proto` est la **source de vérité** de votre API gRPC. C'est à partir de ce fichier que tout est généré — le code client, le code serveur, la documentation, les mocks de test. Un `.proto` bien conçu produit une API claire, maintenable et évolutive. Un `.proto` mal conçu vous poursuivra pendant des années à travers des problèmes de compatibilité et des contorsions de code.

La section 22.6 a présenté la syntaxe de base. Cette section approfondit les fonctionnalités avancées du langage Protobuf et les bonnes pratiques de design d'API gRPC que l'industrie a dégagées après une décennie d'utilisation à grande échelle.

---

## Structure d'un fichier `.proto`

Un fichier `.proto` bien organisé suit une structure conventionnelle :

```protobuf
// ─── 1. Déclarations globales ───
syntax = "proto3";                    // Toujours proto3 pour les nouveaux projets

package mycompany.myservice.v1;       // Namespace hiérarchique + version

// ─── 2. Imports ───
import "google/protobuf/timestamp.proto";  
import "google/protobuf/duration.proto";  
import "common/pagination.proto";  

// ─── 3. Options ───
// Le namespace C++ est dérivé du package :
// mycompany.myservice.v1 → mycompany::myservice::v1
option cc_enable_arenas = true;  // Optionnel : active les memory arenas

// ─── 4. Services ───
service UserService {
    rpc GetUser (GetUserRequest) returns (GetUserResponse);
    rpc ListUsers (ListUsersRequest) returns (ListUsersResponse);
    rpc CreateUser (CreateUserRequest) returns (CreateUserResponse);
    rpc DeleteUser (DeleteUserRequest) returns (DeleteUserResponse);
    rpc WatchUsers (WatchUsersRequest) returns (stream UserEvent);
}

// ─── 5. Messages de requête/réponse ───
message GetUserRequest {
    string user_id = 1;
}

message GetUserResponse {
    User user = 1;
}

// ─── 6. Messages de domaine ───
message User {
    string id = 1;
    string name = 2;
    string email = 3;
    UserRole role = 4;
    google.protobuf.Timestamp created_at = 5;
}

// ─── 7. Enums ───
enum UserRole {
    USER_ROLE_UNSPECIFIED = 0;
    USER_ROLE_ADMIN = 1;
    USER_ROLE_MEMBER = 2;
    USER_ROLE_VIEWER = 3;
}
```

## Le package : namespacing et versioning

### Conventions de nommage

Le package Protobuf détermine le namespace C++ généré et évite les collisions entre messages de projets différents :

```protobuf
package mycompany.inventory.v1;
```

La convention établie (utilisée par Google, Buf, et la majorité de l'écosystème) est :

```
<organisation>.<service>.<version>
```

La **version dans le package** est une pratique fondamentale en gRPC. Elle permet de faire coexister plusieurs versions d'une API sans conflit — le client v1 et le client v2 peuvent être déployés simultanément, parlant chacun à la version du service qui leur correspond.

### Mapping vers C++

```protobuf
package mycompany.inventory.v1;
```

Génère par défaut le namespace C++ :

```cpp
namespace mycompany::inventory::v1 {
    class GetItemRequest : public google::protobuf::Message { ... };
    // ...
}
```

Le namespace C++ est toujours dérivé du package Protobuf (les points deviennent `::`) — il n'y a pas d'option standard pour le personnaliser. Si vos conventions de namespace C++ diffèrent, utilisez un alias de namespace dans votre code :

```cpp
// Si le package proto est inventory.v1 mais que vous préférez un alias
namespace inv = inventory::v1;
```

## Types de messages avancés

### Messages imbriqués

Les messages peuvent être définis à l'intérieur d'autres messages pour créer une hiérarchie logique :

```protobuf
message Order {
    string id = 1;
    string customer_id = 2;
    repeated LineItem items = 3;
    ShippingInfo shipping = 4;
    
    // Message imbriqué — accessible comme Order.LineItem
    message LineItem {
        string product_id = 1;
        int32 quantity = 2;
        int64 price_cents = 3;
    }
    
    message ShippingInfo {
        string address = 1;
        string city = 2;
        string country = 3;
    }
}
```

En C++, les types imbriqués deviennent des classes internes :

```cpp
Order::LineItem item;  
item.set_product_id("SKU-001");  
item.set_quantity(3);  
item.set_price_cents(1999);  

Order order;
*order.add_items() = item;  // add_items() retourne un pointeur
```

Utilisez l'imbrication quand un type n'a de sens que dans le contexte de son parent. Si un type est réutilisé par plusieurs messages, définissez-le au niveau global.

### Enums

Les enums Protobuf ont une règle cruciale : **la valeur 0 doit toujours être la valeur « non spécifiée »**. C'est parce qu'en proto3, les champs non remplis prennent leur valeur par défaut — et la valeur par défaut d'un enum est 0. Sans une valeur « non spécifiée » explicite, il est impossible de distinguer « le client a choisi la première valeur de l'enum » de « le client n'a pas rempli ce champ ».

```protobuf
enum OrderStatus {
    ORDER_STATUS_UNSPECIFIED = 0;  // OBLIGATOIRE — valeur par défaut
    ORDER_STATUS_PENDING = 1;
    ORDER_STATUS_CONFIRMED = 2;
    ORDER_STATUS_SHIPPED = 3;
    ORDER_STATUS_DELIVERED = 4;
    ORDER_STATUS_CANCELLED = 5;
}
```

La convention de nommage est `ENUM_NAME_VALUE_NAME` — le préfixe du nom de l'enum est répété sur chaque valeur. C'est verbeux mais nécessaire : en Protobuf, les valeurs d'enum sont dans le scope du package, pas de l'enum. Sans préfixe, `PENDING` et `CANCELLED` de deux enums différents entreraient en conflit.

### `oneof` — Union discriminée

`oneof` modélise un champ qui peut prendre **un et un seul** type parmi plusieurs possibilités. C'est l'équivalent Protobuf d'un `std::variant` :

```protobuf
message PaymentMethod {
    string id = 1;
    
    oneof method {
        CreditCard credit_card = 2;
        BankTransfer bank_transfer = 3;
        DigitalWallet digital_wallet = 4;
    }
}

message CreditCard {
    string number_last4 = 1;
    string expiry = 2;
}

message BankTransfer {
    string iban = 1;
    string bic = 2;
}

message DigitalWallet {
    string provider = 1;  // "apple_pay", "google_pay"
    string account_id = 2;
}
```

En C++ :

```cpp
PaymentMethod pm;  
pm.mutable_credit_card()->set_number_last4("4242");  
pm.mutable_credit_card()->set_expiry("12/27");  

// Vérifier quel champ est défini
switch (pm.method_case()) {
    case PaymentMethod::kCreditCard:
        process_card(pm.credit_card());
        break;
    case PaymentMethod::kBankTransfer:
        process_transfer(pm.bank_transfer());
        break;
    case PaymentMethod::kDigitalWallet:
        process_wallet(pm.digital_wallet());
        break;
    case PaymentMethod::METHOD_NOT_SET:
        // Aucun moyen de paiement spécifié
        break;
}
```

`oneof` est essentiel pour les types polymorphiques : réponses qui peuvent être succès ou erreur, événements de différents types, configurations avec des variantes.

### `map` — Tables associatives

```protobuf
message Config {
    map<string, string> labels = 1;
    map<string, int32> feature_flags = 2;
}
```

En C++ :

```cpp
Config config;
(*config.mutable_labels())["env"] = "production";
(*config.mutable_labels())["region"] = "eu-west-1";

// Lecture
for (const auto& [key, value] : config.labels()) {
    std::println("{} = {}", key, value);
}
```

Les clés de map peuvent être des types entiers ou des strings — pas des flottants, des bytes, ni des messages. Les valeurs peuvent être de n'importe quel type (sauf une autre map).

### `optional` — Champs explicitement optionnels

En proto3, tous les champs scalaires ont une valeur par défaut (0, `""`, `false`) et il est impossible de savoir si un champ a été explicitement rempli avec la valeur par défaut ou s'il n'a pas été rempli du tout. Le mot-clé `optional` restaure cette distinction :

```protobuf
message UpdateUserRequest {
    string user_id = 1;
    optional string name = 2;      // Peut distinguer "non fourni" de ""
    optional string email = 3;
    optional int32 age = 4;        // Peut distinguer "non fourni" de 0
}
```

En C++ :

```cpp
UpdateUserRequest req;

// Vérifier si le champ a été explicitement défini
if (req.has_name()) {
    // Le client a fourni un nom (même s'il est vide)
    update_name(req.name());
}
// Sinon : le client n'a pas touché ce champ — ne pas modifier
```

`optional` est indispensable pour les opérations de mise à jour partielle (PATCH semantics) où l'absence d'un champ signifie « ne pas modifier » et non « mettre à la valeur par défaut ».

### `repeated` — Listes

```protobuf
message SearchResponse {
    repeated User users = 1;    // Liste de User (peut être vide)
    int32 total_count = 2;
}
```

En C++ :

```cpp
SearchResponse res;

// Ajouter des éléments
User* user = res.add_users();  
user->set_name("Alice");  

// Itérer
for (const auto& user : res.users()) {
    std::println("User: {}", user.name());
}

// Taille
int count = res.users_size();
```

## Design de services gRPC

### Nommage des RPC

La convention suit le pattern `VerbNoun` :

```protobuf
service ProductService {
    rpc GetProduct (GetProductRequest) returns (GetProductResponse);
    rpc ListProducts (ListProductsRequest) returns (ListProductsResponse);
    rpc CreateProduct (CreateProductRequest) returns (CreateProductResponse);
    rpc UpdateProduct (UpdateProductRequest) returns (UpdateProductResponse);
    rpc DeleteProduct (DeleteProductRequest) returns (DeleteProductResponse);
    rpc SearchProducts (SearchProductsRequest) returns (SearchProductsResponse);
}
```

Les verbes standards sont : `Get` (lecture unitaire), `List` (lecture multiple), `Create`, `Update`, `Delete`, `Search` (recherche complexe), `Watch`/`Stream` (streaming), `Batch` (opérations groupées).

### Un message par RPC

Chaque méthode RPC a ses propres types de requête et de réponse, même s'ils sont similaires à ceux d'une autre méthode. Cette règle paraît verbeux mais elle est fondamentale pour l'évolutivité :

```protobuf
// BON — chaque RPC a ses types dédiés
rpc GetUser (GetUserRequest) returns (GetUserResponse);  
rpc ListUsers (ListUsersRequest) returns (ListUsersResponse);  

// MAUVAIS — réutilisation de types entre RPCs
rpc GetUser (UserRequest) returns (User);  
rpc ListUsers (UserRequest) returns (UserList);  
```

Si `GetUser` et `ListUsers` partagent le même type `UserRequest`, le jour où `ListUsers` a besoin d'un champ de pagination, vous devez modifier un type utilisé par `GetUser` — qui n'a aucun besoin de pagination. Des types dédiés permettent à chaque RPC d'évoluer indépendamment.

### Wrapping des réponses

Ne retournez jamais directement un message de domaine comme réponse — wrappez-le :

```protobuf
// BON — enveloppe de réponse
message GetUserResponse {
    User user = 1;
}

// MAUVAIS — message de domaine comme réponse directe
rpc GetUser (GetUserRequest) returns (User);  // Pas de place pour ajouter
                                               // des métadonnées plus tard
```

Le wrapper permet d'ajouter des métadonnées à la réponse sans casser la compatibilité : un champ `request_id` pour le tracing, un champ `cache_hit` pour le monitoring, un champ `deprecated_fields` pour les migrations.

### Pagination

Pour les listes potentiellement longues, utilisez la pagination par curseur :

```protobuf
message ListOrdersRequest {
    int32 page_size = 1;           // Nombre max de résultats (ex: 50)
    string page_token = 2;         // Token opaque pour la page suivante
    string filter = 3;             // Filtre optionnel
}

message ListOrdersResponse {
    repeated Order orders = 1;
    string next_page_token = 2;    // Vide si c'est la dernière page
    int32 total_count = 3;         // Nombre total (optionnel, coûteux)
}
```

Le `page_token` est un curseur opaque (typiquement un ID encodé en base64 ou un offset chiffré). La pagination par curseur est plus fiable que la pagination par offset (page 1, 2, 3) car elle n'est pas affectée par les insertions/suppressions entre deux pages.

### Field masks pour les mises à jour partielles

Pour les opérations de mise à jour, un `FieldMask` indique précisément quels champs modifier :

```protobuf
import "google/protobuf/field_mask.proto";

message UpdateUserRequest {
    User user = 1;
    google.protobuf.FieldMask update_mask = 2;
}
```

Côté client :

```protobuf
// "Je veux modifier uniquement le nom et l'email"
update_mask: { paths: ["name", "email"] }
```

Côté serveur C++ :

```cpp
#include <google/protobuf/field_mask.util.h>

void UpdateUser(const UpdateUserRequest& req) {
    User existing = load_from_db(req.user().id());
    
    // Appliquer uniquement les champs spécifiés dans le mask
    google::protobuf::util::FieldMaskUtil::MergeOptions options;
    options.set_replace_message_fields(true);
    google::protobuf::util::FieldMaskUtil::MergeMessageFields(
        req.user(), req.update_mask(), &existing);
    
    save_to_db(existing);
}
```

## Gestion d'erreurs gRPC

### Status codes

gRPC définit un ensemble de codes d'erreur standardisés, distincts des codes HTTP :

| Code gRPC | Signification | Équivalent HTTP |
|-----------|---------------|-----------------|
| `OK` | Succès | 200 |
| `INVALID_ARGUMENT` | Requête invalide | 400 |
| `NOT_FOUND` | Ressource non trouvée | 404 |
| `ALREADY_EXISTS` | Ressource existe déjà | 409 |
| `PERMISSION_DENIED` | Non autorisé | 403 |
| `UNAUTHENTICATED` | Non authentifié | 401 |
| `RESOURCE_EXHAUSTED` | Quota dépassé | 429 |
| `FAILED_PRECONDITION` | Précondition non remplie | 400 |
| `UNAVAILABLE` | Service indisponible (retry recommandé) | 503 |
| `DEADLINE_EXCEEDED` | Timeout | 504 |
| `INTERNAL` | Erreur serveur | 500 |
| `UNIMPLEMENTED` | Méthode non implémentée | 501 |

### Erreurs structurées avec `google.rpc.Status`

Pour les erreurs nécessitant des détails (quels champs sont invalides, quelle est la limite dépassée), gRPC supporte le modèle d'erreurs riches de Google :

```protobuf
import "google/rpc/status.proto";  
import "google/rpc/error_details.proto";  
```

En C++, les erreurs riches sont attachées au status gRPC via les **trailing metadata**. Les détails d'implémentation sont couverts en section 22.6.3.

## Compatibilité et évolution du schéma

L'un des atouts majeurs de Protobuf est sa capacité à évoluer sans casser la compatibilité. Mais cela repose sur le respect de règles précises.

### Ce que vous pouvez faire (compatible)

**Ajouter un nouveau champ** — Les anciens clients ignorent les champs inconnus. Les nouveaux clients utilisent la valeur par défaut pour les champs absents.

```protobuf
message User {
    string id = 1;
    string name = 2;
    string email = 3;
    // Ajouté en v2 — les clients v1 l'ignorent
    string phone = 4;
}
```

**Ajouter une nouvelle méthode RPC** — Les anciens clients ne l'appellent jamais, les nouveaux clients peuvent l'utiliser.

**Ajouter une valeur à un enum** — Les anciens clients recevront la valeur numérique et la traiteront comme inconnue (selon le langage).

**Supprimer un champ** en le réservant — Le champ disparaît mais son numéro ne sera jamais réutilisé.

### Ce que vous ne devez JAMAIS faire (casse la compatibilité)

**Changer le numéro d'un champ** — Le numéro est l'identifiant binaire. Le changer fait interpréter les données existantes avec le mauvais type.

**Changer le type d'un champ** — Un `int32` réinterprété comme `string` produit des données corrompues.

**Réutiliser un numéro de champ supprimé** — D'anciens messages sérialisés contiennent encore ce numéro avec l'ancien type.

### `reserved` — Protéger les champs supprimés

```protobuf
message User {
    string id = 1;
    string name = 2;
    // Le champ 3 (anciennement "phone_number") a été supprimé
    // Le champ 5 (anciennement "legacy_role") aussi
    reserved 3, 5;
    reserved "phone_number", "legacy_role";
    
    string email = 4;
    UserRole role = 6;
}
```

`reserved` empêche quiconque de réutiliser accidentellement ces numéros ou noms dans le futur. `protoc` émet une erreur de compilation si un numéro ou nom réservé est réutilisé.

### Versioning par package

Pour les changements incompatibles majeurs (refonte de l'API, changement de sémantique), créez une nouvelle version du package :

```
proto/
├── myservice/
│   ├── v1/
│   │   └── service.proto    # package myservice.v1
│   └── v2/
│       └── service.proto    # package myservice.v2
```

Les deux versions coexistent. Les clients migrent progressivement de v1 vers v2. Le serveur peut implémenter les deux versions simultanément pendant la période de transition.

## Options utiles pour C++

```protobuf
// Le namespace C++ est dérivé automatiquement du package :
// package myapp.api.v1; → namespace myapp::api::v1 { ... }

// Active les memory arenas pour la performance
option cc_enable_arenas = true;  // Active les memory arenas pour la performance
```

### Memory arenas

L'option `cc_enable_arenas` est un levier de performance important pour les serveurs à haut débit. Par défaut, chaque message Protobuf alloue ses champs sur le heap de manière indépendante. Avec les arenas, tous les messages d'une même requête partagent un bloc mémoire unique, alloué en une seule fois et libéré en une seule fois :

```cpp
// Sans arena : chaque message fait ses propres allocations
GetUserResponse response;  
response.mutable_user()->set_name("Alice");  // allocation heap  

// Avec arena : un seul bloc mémoire pour tout
google::protobuf::Arena arena;  
auto* response = google::protobuf::Arena::CreateMessage<GetUserResponse>(&arena);  
response->mutable_user()->set_name("Alice");  // allocation dans l'arena  
// Tout est libéré quand l'arena est détruite — rapide et sans fragmentation
```

## Exemple complet : API d'un service de gestion de tâches

Voici un fichier `.proto` réaliste intégrant les bonnes pratiques couvertes dans cette section :

```protobuf
// task_service/v1/task.proto
syntax = "proto3";

package taskservice.v1;

import "google/protobuf/timestamp.proto";  
import "google/protobuf/field_mask.proto";  
import "google/protobuf/empty.proto";  

// Namespace C++ généré : taskservice::v1

// ──────── Service ────────

service TaskService {
    // CRUD
    rpc CreateTask (CreateTaskRequest) returns (CreateTaskResponse);
    rpc GetTask (GetTaskRequest) returns (GetTaskResponse);
    rpc UpdateTask (UpdateTaskRequest) returns (UpdateTaskResponse);
    rpc DeleteTask (DeleteTaskRequest) returns (google.protobuf.Empty);
    
    // Liste avec pagination
    rpc ListTasks (ListTasksRequest) returns (ListTasksResponse);
    
    // Streaming : notifications en temps réel
    rpc WatchTasks (WatchTasksRequest) returns (stream TaskEvent);
}

// ──────── Messages de domaine ────────

message Task {
    string id = 1;
    string title = 2;
    string description = 3;
    TaskStatus status = 4;
    TaskPriority priority = 5;
    string assignee_id = 6;
    repeated string labels = 7;
    google.protobuf.Timestamp created_at = 8;
    google.protobuf.Timestamp updated_at = 9;
    optional google.protobuf.Timestamp due_date = 10;
}

enum TaskStatus {
    TASK_STATUS_UNSPECIFIED = 0;
    TASK_STATUS_TODO = 1;
    TASK_STATUS_IN_PROGRESS = 2;
    TASK_STATUS_IN_REVIEW = 3;
    TASK_STATUS_DONE = 4;
    TASK_STATUS_CANCELLED = 5;
}

enum TaskPriority {
    TASK_PRIORITY_UNSPECIFIED = 0;
    TASK_PRIORITY_LOW = 1;
    TASK_PRIORITY_MEDIUM = 2;
    TASK_PRIORITY_HIGH = 3;
    TASK_PRIORITY_CRITICAL = 4;
}

// ──────── Requêtes / Réponses ────────

message CreateTaskRequest {
    string title = 1;
    string description = 2;
    TaskPriority priority = 3;
    optional string assignee_id = 4;
    repeated string labels = 5;
    optional google.protobuf.Timestamp due_date = 6;
}

message CreateTaskResponse {
    Task task = 1;
}

message GetTaskRequest {
    string task_id = 1;
}

message GetTaskResponse {
    Task task = 1;
}

message UpdateTaskRequest {
    Task task = 1;
    google.protobuf.FieldMask update_mask = 2;
}

message UpdateTaskResponse {
    Task task = 1;
}

message DeleteTaskRequest {
    string task_id = 1;
}

message ListTasksRequest {
    int32 page_size = 1;
    string page_token = 2;
    TaskStatus status_filter = 3;
    string assignee_filter = 4;
}

message ListTasksResponse {
    repeated Task tasks = 1;
    string next_page_token = 2;
    int32 total_count = 3;
}

// ──────── Streaming ────────

message WatchTasksRequest {
    // Filtres optionnels pour limiter les événements reçus
    repeated string task_ids = 1;
    optional string assignee_id = 2;
}

message TaskEvent {
    EventType type = 1;
    Task task = 2;
    google.protobuf.Timestamp event_time = 3;
    
    enum EventType {
        EVENT_TYPE_UNSPECIFIED = 0;
        EVENT_TYPE_CREATED = 1;
        EVENT_TYPE_UPDATED = 2;
        EVENT_TYPE_DELETED = 3;
        EVENT_TYPE_STATUS_CHANGED = 4;
        EVENT_TYPE_ASSIGNED = 5;
    }
}
```

Ce fichier illustre en contexte réel : le versioning par package, les types well-known de Google, les enums avec `_UNSPECIFIED`, `optional` pour les champs nullable, `repeated` pour les listes, `FieldMask` pour les mises à jour partielles, la pagination par curseur, le streaming serveur pour les événements, et les messages wrappés pour chaque RPC.

## Outils de validation

### `buf` — Le linter pour `.proto`

[Buf](https://buf.build) est un outil qui lint, formate, et détecte les incompatibilités dans vos fichiers `.proto`. Il remplace `protoc` pour la validation et est largement adopté dans l'industrie :

```bash
# Installation
curl -sSL https://github.com/bufbuild/buf/releases/latest/download/buf-Linux-x86_64 \
    -o /usr/local/bin/buf && chmod +x /usr/local/bin/buf

# Linting
buf lint proto/

# Détection d'incompatibilités entre deux versions
buf breaking proto/ --against .git#branch=main
```

`buf breaking` est particulièrement puissant : il compare vos modifications `.proto` avec la version précédente et signale tout changement incompatible (numéro de champ modifié, type changé, champ required supprimé).

Intégrez `buf lint` et `buf breaking` dans votre CI pour garantir la qualité et la compatibilité de vos fichiers `.proto` à chaque commit.

## Résumé

Le fichier `.proto` est le contrat de votre API gRPC. Les principes clés pour un design robuste et évolutif :

- **Un message dédié par requête et par réponse** — Chaque RPC évolue indépendamment.
- **Wrappez les réponses** — Ne retournez jamais un message de domaine directement.
- **Enum : toujours `_UNSPECIFIED = 0`** — Pour distinguer « non rempli » de la première valeur.
- **`optional`** pour les champs où l'absence a un sens (mises à jour partielles, filtres).
- **`oneof`** pour les unions discriminées (types polymorphiques, variantes).
- **`reserved`** pour les champs supprimés — ne jamais réutiliser un numéro.
- **Pagination par curseur** — Plus fiable que la pagination par offset.
- **`FieldMask`** pour les mises à jour partielles — le client dit exactement quels champs modifier.
- **Versioning par package** (`v1`, `v2`) pour les changements incompatibles.
- **Types well-known de Google** (`Timestamp`, `Duration`, `Empty`) plutôt que des types maison.
