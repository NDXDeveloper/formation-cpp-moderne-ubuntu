# 22.1.3 — send, recv, sendto, recvfrom

## Section 22.1 : Sockets TCP/UDP — API POSIX

---

## Introduction

Le socket est créé, configuré, bindé. La connexion est établie (TCP) ou le socket est prêt à émettre (UDP). Il est temps de faire ce pour quoi tout cela a été construit : **envoyer et recevoir des données**.

Cette étape, en apparence simple, est en réalité celle qui concentre le plus de subtilités et de pièges. Un `send()` ne garantit pas que toutes vos données ont été envoyées. Un `recv()` ne garantit pas que vous recevrez un « message » complet — ni même un seul octet. Les signaux peuvent interrompre vos appels. Le pair peut fermer la connexion à tout moment. Le buffer noyau peut être plein.

Cette section détaille les quatre fonctions d'échange de données, leurs comportements réels (pas ceux qu'on imagine), et les patterns robustes pour les utiliser correctement.

---

## Les quatre fonctions

L'API sockets propose deux paires de fonctions pour l'échange de données :

```
                    TCP (connecté)          UDP (avec ou sans connect)
                    ──────────────          ──────────────────────────
Envoyer             send()                  sendto()  
Recevoir            recv()                  recvfrom()  
```

`send()` et `recv()` travaillent sur un socket **déjà connecté** — c'est le cas naturel en TCP après `connect()` ou `accept()`, et en UDP après un `connect()` optionnel.

`sendto()` et `recvfrom()` acceptent en plus une **adresse de destination/source** — c'est le mode naturel en UDP, où chaque datagramme peut aller vers (ou venir de) une adresse différente.

En réalité, les alternatives POSIX génériques `read()` / `write()` fonctionnent aussi sur les sockets connectés, puisqu'un socket est un file descriptor. Mais `send()` / `recv()` offrent un paramètre `flags` supplémentaire qui donne un contrôle plus fin — c'est pourquoi on les préfère pour le code réseau.

---

## `send()` — Envoyer des données (TCP)

### Signature

```cpp
#include <sys/socket.h>

ssize_t send(int sockfd, const void* buf, size_t len, int flags);
```

### Ce que `send()` fait réellement

Un malentendu très répandu : `send()` **n'envoie pas de données sur le réseau**. Ce qu'il fait, c'est copier vos données dans le **buffer d'envoi du noyau** associé au socket. C'est le noyau qui se chargera ensuite, de manière asynchrone, de segmenter ces données en paquets TCP, de les envoyer, de gérer les retransmissions et les acquittements.

```
Votre code              Noyau Linux                    Réseau
──────────              ───────────                    ──────

buf ──► send() ──────► Buffer d'envoi ──────► Pile TCP/IP ──────► Paquets
        │               (SO_SNDBUF)
        │
        └─ Retourne le nombre d'octets
           copiés dans le buffer noyau
           (pas envoyés sur le réseau !)
```

La valeur retournée est le nombre d'octets que le noyau a accepté dans son buffer — pas le nombre d'octets effectivement reçus par le pair. La réception par le destinataire n'est confirmée qu'au niveau TCP (via les ACK), et cette information n'est pas remontée à l'application.

### Envois partiels : le piège numéro un

Quand le buffer d'envoi du noyau n'a pas assez de place pour accepter toutes vos données, `send()` n'échoue pas — il accepte **ce qu'il peut** et retourne le nombre d'octets effectivement pris en charge :

```cpp
const char* data = /* 10 000 octets */;  
ssize_t sent = send(sockfd, data, 10000, 0);  
// sent pourrait valoir 4096, 8192, ou n'importe quelle valeur ≤ 10000
```

Si vous envoyez 10 000 octets et que `send()` retourne 4096, il reste 5 904 octets à envoyer. Les ignorer signifie perdre silencieusement des données — un bug qui peut passer inaperçu pendant des mois en développement (où le réseau local est rapide et les buffers rarement pleins) et exploser en production.

### La boucle `send_all` : pattern obligatoire

Pour garantir l'envoi complet, il faut boucler :

```cpp
// Envoie TOUTES les données ou lance une exception
void send_all(int sockfd, const void* buf, size_t len) {
    auto* ptr = static_cast<const std::byte*>(buf);
    size_t remaining = len;

    while (remaining > 0) {
        ssize_t sent = send(sockfd, ptr, remaining, 0);

        if (sent == -1) {
            if (errno == EINTR) {
                continue;  // Interrompu par un signal — réessayer
            }
            throw std::system_error(errno, std::system_category(), "send()");
        }

        ptr += sent;
        remaining -= static_cast<size_t>(sent);
    }
}
```

Ce pattern est si fondamental que vous devriez considérer `send_all` comme votre fonction d'envoi par défaut, et `send()` brut comme un appel bas niveau que vous n'utilisez directement que dans des cas spécifiques (sockets non-bloquants, intégration avec `epoll`).

### Le flag `MSG_NOSIGNAL`

Sur Linux, écrire sur un socket dont le pair a fermé sa connexion déclenche un signal `SIGPIPE` qui, par défaut, **tue votre processus**. C'est un comportement hérité d'Unix qui surprend régulièrement les développeurs.

Le flag `MSG_NOSIGNAL` empêche l'envoi du signal. À la place, `send()` retourne `-1` avec `errno = EPIPE` — une erreur que vous pouvez gérer proprement :

```cpp
ssize_t sent = send(sockfd, data, len, MSG_NOSIGNAL);  
if (sent == -1 && errno == EPIPE) {  
    // Le pair a fermé la connexion — gérer proprement
}
```

**Utilisez `MSG_NOSIGNAL` sur tous vos appels `send()`** en C++. L'alternative est d'ignorer `SIGPIPE` globalement avec `signal(SIGPIPE, SIG_IGN)` en début de programme, ce que font la plupart des serveurs. Les deux approches sont valides — `MSG_NOSIGNAL` est plus ciblé, `SIG_IGN` est plus global et évite d'oublier le flag sur un appel.

### Version robuste de `send_all`

En intégrant `MSG_NOSIGNAL` et une gestion d'erreur plus fine :

```cpp
enum class SendResult { ok, peer_closed, error };

SendResult send_all(int sockfd, std::span<const std::byte> data) {
    auto* ptr = data.data();
    size_t remaining = data.size();

    while (remaining > 0) {
        ssize_t sent = send(sockfd, ptr, remaining, MSG_NOSIGNAL);

        if (sent == -1) {
            if (errno == EINTR) continue;
            if (errno == EPIPE) return SendResult::peer_closed;
            return SendResult::error;
        }

        // sent == 0 ne devrait pas arriver en TCP, mais soyons prudents
        ptr += sent;
        remaining -= static_cast<size_t>(sent);
    }

    return SendResult::ok;
}
```

> 💡 L'utilisation de `std::span<const std::byte>` (C++20) rend l'interface plus expressive qu'un couple `void* + size_t`. Le span encapsule le pointeur et la taille, et `std::byte` explicite qu'on manipule des données brutes.

### Autres flags utiles pour `send()`

| Flag | Effet |
|------|-------|
| `MSG_NOSIGNAL` | Empêche `SIGPIPE` — retourne `EPIPE` à la place |
| `MSG_DONTWAIT` | Rend cet appel non-bloquant (équivalent ponctuel de `SOCK_NONBLOCK`) |
| `MSG_MORE` | Indique au noyau que d'autres données suivront — active le cork TCP pour agréger en un seul paquet. Utile pour construire un message en plusieurs `send()`. |

## `recv()` — Recevoir des données (TCP)

### Signature

```cpp
#include <sys/socket.h>

ssize_t recv(int sockfd, void* buf, size_t len, int flags);
```

### Ce que `recv()` fait réellement

Symétriquement à `send()`, `recv()` **ne lit pas des données du réseau** — il lit des données depuis le **buffer de réception du noyau**. Le noyau accumule les données entrantes au fur et à mesure qu'elles arrivent du réseau, et `recv()` en copie une partie dans votre buffer applicatif.

```
Réseau                    Noyau Linux              Votre code
──────                    ───────────              ──────────

Paquets ──────► Pile TCP/IP ──────► Buffer de réception ──────► recv() ──► buf
                                    (SO_RCVBUF)                  │
                                                                 └─ Retourne le nombre
                                                                    d'octets copiés
```

### Les trois valeurs de retour

`recv()` a trois résultats possibles, et chacun a une signification précise :

**`n > 0`** — `n` octets ont été copiés dans votre buffer. C'est le cas normal.

**`n == 0`** — Le pair a effectué une **fermeture ordonnée** (`close()` ou `shutdown(SHUT_WR)`). C'est l'équivalent réseau de EOF pour un fichier. La connexion est terminée côté pair — il n'y aura plus de données. Ce n'est pas une erreur, c'est un événement normal qu'il faut gérer.

**`n == -1`** — Une erreur s'est produite. Vérifiez `errno` pour connaître la cause.

```cpp
ssize_t n = recv(sockfd, buffer, sizeof(buffer), 0);

if (n > 0) {
    // Données reçues — les traiter
    process_data(buffer, n);
} else if (n == 0) {
    // Le pair a fermé la connexion — nettoyage
    std::println("Client déconnecté");
} else {
    // Erreur
    if (errno == EINTR) {
        // Signal reçu — réessayer
    } else if (errno == EAGAIN || errno == EWOULDBLOCK) {
        // Socket non-bloquant, rien à lire — normal
    } else {
        throw std::system_error(errno, std::system_category(), "recv()");
    }
}
```

> ⚠️ **Ne confondez jamais `recv()` retournant 0 avec « pas de données disponibles ».** Un retour de 0 signifie que la connexion est fermée — définitivement. « Pas de données disponibles » se manifeste par un blocage (mode bloquant) ou par `-1` avec `EAGAIN` (mode non-bloquant).

### TCP est un flux d'octets, pas un flux de messages

C'est le point le plus important de cette section, et probablement le piège le plus fréquent en programmation réseau TCP.

Quand un client envoie deux messages successifs de 100 octets chacun, le serveur peut les recevoir de n'importe laquelle de ces manières :

```
Envoyé par le client :    [── 100 octets ──][── 100 octets ──]

Reçu par le serveur, possibilités :

  1) [── 200 octets ──────────────────]         Un seul recv()
  2) [── 100 octets ──][── 100 octets ──]       Deux recv() parfaits (rare)
  3) [── 50 ──][── 50 ──][── 100 octets ──]     Trois recv()
  4) [── 150 octets ──────][── 50 ──]           Deux recv() qui coupent au milieu
  5) [─ 1 ─][─ 1 ─][─ 1 ─]...[─ 1 ─]          200 recv() d'un octet (théorique)
```

TCP ne préserve pas les frontières de messages. Les données sont un flux continu d'octets, et `recv()` retourne ce qui est disponible dans le buffer noyau au moment de l'appel — de 1 octet à la taille de votre buffer, selon l'état du réseau et du timing.

C'est à l'application de **reconstruire les messages** à partir du flux. On parle de **framing** (encadrement de messages). Les stratégies courantes sont :

**Longueur préfixée** — Chaque message commence par un header de taille fixe qui indique la longueur du corps. Le récepteur lit d'abord le header, puis exactement le nombre d'octets annoncé. C'est la méthode la plus courante et la plus fiable.

```
┌──────────┬────────────────────────────┐
│ len (4B) │  payload (len octets)      │
└──────────┴────────────────────────────┘
```

**Délimiteur** — Les messages sont séparés par un caractère ou une séquence spéciale (par exemple `\r\n` pour HTTP, `\0` pour certains protocoles). Le récepteur lit jusqu'à trouver le délimiteur. Simple, mais fragile si le délimiteur peut apparaître dans les données.

**Taille fixe** — Tous les messages ont la même taille, connue à l'avance. Le récepteur lit toujours exactement N octets. Rigide, mais simple et efficace quand c'est applicable.

### Lecture exacte de N octets : `recv_exact`

Pour le framing par longueur préfixée, vous avez besoin de lire **exactement** N octets — ni plus, ni moins. Comme `recv()` peut retourner moins que demandé, il faut boucler :

```cpp
// Lit exactement len octets ou échoue
// Retourne false si le pair ferme la connexion avant que len octets soient lus
bool recv_exact(int sockfd, void* buf, size_t len) {
    auto* ptr = static_cast<std::byte*>(buf);
    size_t remaining = len;

    while (remaining > 0) {
        ssize_t n = recv(sockfd, ptr, remaining, 0);

        if (n == -1) {
            if (errno == EINTR) continue;
            throw std::system_error(errno, std::system_category(), "recv()");
        }

        if (n == 0) {
            // Connexion fermée avant d'avoir tout reçu
            return false;
        }

        ptr += n;
        remaining -= static_cast<size_t>(n);
    }

    return true;
}
```

### Framing par longueur préfixée : exemple complet

Voici un protocole minimal utilisant un header de 4 octets (uint32_t en network byte order) suivi du payload :

```cpp
#include <cstdint>
#include <vector>
#include <arpa/inet.h>  // htonl, ntohl

// --- Envoi d'un message framé ---
void send_message(int sockfd, std::span<const std::byte> payload) {
    // 1. Envoyer le header : taille du payload en network byte order
    uint32_t len_net = htonl(static_cast<uint32_t>(payload.size()));
    send_all(sockfd, &len_net, sizeof(len_net));

    // 2. Envoyer le payload
    send_all(sockfd, payload.data(), payload.size());
}

// --- Réception d'un message framé ---
std::vector<std::byte> recv_message(int sockfd) {
    // 1. Lire le header (exactement 4 octets)
    uint32_t len_net = 0;
    if (!recv_exact(sockfd, &len_net, sizeof(len_net))) {
        return {};  // Connexion fermée
    }
    uint32_t len = ntohl(len_net);

    // 2. Vérification de sécurité : limiter la taille
    constexpr uint32_t max_message_size = 16 * 1024 * 1024;  // 16 MiB
    if (len > max_message_size) {
        throw std::runtime_error("Message trop grand : " + std::to_string(len));
    }

    // 3. Lire le payload (exactement len octets)
    std::vector<std::byte> payload(len);
    if (!recv_exact(sockfd, payload.data(), len)) {
        throw std::runtime_error("Connexion fermée pendant la lecture du payload");
    }

    return payload;
}
```

> ⚠️ **Toujours valider la taille reçue dans le header.** Un client malveillant (ou un bug) pourrait envoyer une longueur de 4 GiB, provoquant une allocation mémoire massive et potentiellement un crash. Imposez une limite raisonnable et rejetez les messages qui la dépassent.

### Le flag `MSG_PEEK`

`MSG_PEEK` permet de lire les données dans le buffer de réception **sans les consommer**. Les données restent disponibles pour un prochain `recv()` :

```cpp
// Regarder ce qui est disponible sans le retirer du buffer
char preview[4];  
ssize_t n = recv(sockfd, preview, sizeof(preview), MSG_PEEK);  
// Les mêmes données seront retournées par le prochain recv() sans MSG_PEEK
```

C'est utile pour inspecter un header avant de décider comment traiter le message, ou pour implémenter un protocole multi-format où les premiers octets identifient le type de message.

### Le flag `MSG_WAITALL`

`MSG_WAITALL` demande à `recv()` d'attendre que le buffer soit **entièrement rempli** (ou qu'une erreur/EOF survienne) avant de retourner :

```cpp
// Tente de lire exactement sizeof(buffer) octets
ssize_t n = recv(sockfd, buffer, sizeof(buffer), MSG_WAITALL);
```

En théorie, cela remplace la boucle `recv_exact`. En pratique, `MSG_WAITALL` n'est **pas parfaitement fiable** : il peut retourner moins d'octets que demandé si un signal est reçu ou si la connexion est fermée. La boucle explicite `recv_exact` reste plus robuste et portable — préférez-la.

### Récapitulatif des flags pour `recv()`

| Flag | Effet |
|------|-------|
| `MSG_PEEK` | Lit sans consommer les données du buffer |
| `MSG_WAITALL` | Attend que le buffer soit plein (pas garanti à 100 %) |
| `MSG_DONTWAIT` | Rend cet appel non-bloquant ponctuellement |
| `MSG_TRUNC` | (UDP) Retourne la taille réelle du datagramme même s'il est tronqué |

## `sendto()` — Envoyer des datagrammes (UDP)

### Signature

```cpp
#include <sys/socket.h>

ssize_t sendto(int sockfd, const void* buf, size_t len, int flags,
               const struct sockaddr* dest_addr, socklen_t addrlen);
```

### Rôle

`sendto()` est la fonction d'envoi pour les sockets UDP **non connectés**. Les deux derniers paramètres spécifient l'adresse de destination de chaque datagramme individuellement — ce qui permet d'envoyer des datagrammes à différentes destinations depuis le même socket.

```cpp
sockaddr_in6 dest{};  
dest.sin6_family = AF_INET6;  
dest.sin6_port = htons(53);  // DNS  
inet_pton(AF_INET6, "2001:4860:4860::8888", &dest.sin6_addr);  

const char* query = /* données DNS */;  
ssize_t sent = sendto(sockfd, query, query_len, 0,  
                      reinterpret_cast<sockaddr*>(&dest), sizeof(dest));

if (sent == -1) {
    throw std::system_error(errno, std::system_category(), "sendto()");
}
```

### Différences fondamentales avec `send()` TCP

**Pas d'envoi partiel** — En UDP, `sendto()` envoie le datagramme en entier ou échoue. Il n'y a pas de « j'ai envoyé 4096 octets sur les 10 000 demandés ». Si le datagramme est trop gros pour être transmis en un seul paquet IP (après prise en compte du MTU), le comportement dépend du flag `DF` (Don't Fragment) :

- Avec `DF` positionné (défaut pour IPv6) : `sendto()` échoue avec `EMSGSIZE`.  
- Sans `DF` : le datagramme est fragmenté au niveau IP. Mais la fragmentation UDP est déconseillée — elle est source de pertes et de problèmes de performance.

En pratique, **limitez vos datagrammes UDP à environ 1400 octets** pour rester sous le MTU Ethernet standard (1500) avec marge pour les headers. Pour les applications qui ont besoin de gros messages, utilisez TCP ou implémentez votre propre fragmentation applicative.

**Aucune garantie de livraison** — `sendto()` retournant un nombre positif signifie uniquement que le noyau a accepté le datagramme pour envoi. Le datagramme peut être perdu sur le réseau, arriver en double, ou arriver dans le désordre. Aucune erreur ne sera reportée à l'application.

**Pas de contrôle de flux** — Si vous envoyez des datagrammes plus vite que le réseau ne peut les absorber, ils seront simplement perdus — silencieusement. C'est à l'application de gérer le rythme d'envoi.

### `sendto()` sur un socket connecté

Si vous avez appelé `connect()` sur un socket UDP, vous pouvez utiliser `send()` au lieu de `sendto()` — l'adresse de destination est implicite. Vous pouvez aussi utiliser `sendto()` avec les paramètres d'adresse à `nullptr` et `0` :

```cpp
// Socket UDP connecté — les deux sont équivalents :
send(sockfd, data, len, 0);  
sendto(sockfd, data, len, 0, nullptr, 0);  
```

## `recvfrom()` — Recevoir des datagrammes (UDP)

### Signature

```cpp
#include <sys/socket.h>

ssize_t recvfrom(int sockfd, void* buf, size_t len, int flags,
                 struct sockaddr* src_addr, socklen_t* addrlen);
```

### Rôle

`recvfrom()` est la fonction de réception pour UDP. En plus des données, elle remplit la structure `src_addr` avec l'adresse de l'expéditeur — information essentielle en UDP puisque les datagrammes peuvent venir de sources différentes.

```cpp
sockaddr_storage sender{};  
socklen_t sender_len = sizeof(sender);  
char buffer[1500];  

ssize_t n = recvfrom(sockfd, buffer, sizeof(buffer), 0,
                     reinterpret_cast<sockaddr*>(&sender), &sender_len);

if (n == -1) {
    throw std::system_error(errno, std::system_category(), "recvfrom()");
}

// n octets reçus, sender contient l'adresse de l'expéditeur
// On peut répondre avec sendto() vers cette adresse
sendto(sockfd, response, resp_len, 0,
       reinterpret_cast<sockaddr*>(&sender), sender_len);
```

### Préservation des frontières de messages

Contrairement à TCP, **chaque `recvfrom()` retourne exactement un datagramme**. Si l'expéditeur envoie un datagramme de 500 octets, le récepteur recevra exactement 500 octets en un seul appel — jamais fragmenté en plusieurs `recvfrom()`, jamais fusionné avec un autre datagramme.

C'est à la fois un avantage (pas besoin de framing applicatif) et une contrainte : votre buffer doit être assez grand pour contenir le datagramme le plus gros que vous attendez.

### Troncation de datagrammes

Si votre buffer est plus petit que le datagramme reçu, le comportement par défaut est que les octets excédentaires sont **silencieusement perdus**. `recvfrom()` retourne la taille du buffer, pas la taille du datagramme original :

```cpp
char small_buffer[100];
// Un datagramme de 500 octets arrive
ssize_t n = recvfrom(sockfd, small_buffer, sizeof(small_buffer), 0, ...);
// n == 100, les 400 octets restants sont perdus sans avertissement
```

Le flag `MSG_TRUNC` (spécifique Linux) modifie ce comportement : `recvfrom()` retourne la taille **réelle** du datagramme, même s'il a été tronqué. Cela permet de détecter la troncation :

```cpp
ssize_t n = recvfrom(sockfd, buffer, sizeof(buffer), MSG_TRUNC, ...);  
if (static_cast<size_t>(n) > sizeof(buffer)) {  
    std::println(stderr, "Datagramme tronqué : {} octets, buffer de {} octets",
                 n, sizeof(buffer));
}
```

### Pattern serveur UDP typique

Un serveur UDP est structurellement plus simple qu'un serveur TCP — pas de `listen`, pas de `accept`, pas de socket par client :

```cpp
void udp_server(uint16_t port) {
    auto addr = resolve(nullptr, std::to_string(port).c_str(),
                        AF_INET6, SOCK_DGRAM);

    Socket sock{addr->ai_family, addr->ai_socktype, addr->ai_protocol};
    sock.enable_reuse_addr();

    if (bind(sock.fd(), addr->ai_addr, addr->ai_addrlen) == -1) {
        throw std::system_error(errno, std::system_category(), "bind()");
    }

    std::println("Serveur UDP en écoute sur le port {}...", port);

    char buffer[65535];  // Taille max d'un datagramme UDP

    while (true) {
        sockaddr_storage sender{};
        socklen_t sender_len = sizeof(sender);

        ssize_t n = recvfrom(sock.fd(), buffer, sizeof(buffer), 0,
                             reinterpret_cast<sockaddr*>(&sender),
                             &sender_len);

        if (n == -1) {
            if (errno == EINTR) continue;
            throw std::system_error(errno, std::system_category(), "recvfrom()");
        }

        // Traiter le datagramme et répondre
        // (chaque datagramme est indépendant — pas d'état de connexion)
        process_and_reply(sock.fd(), buffer, n, sender, sender_len);
    }
}
```

## `read()` / `write()` vs `send()` / `recv()`

Puisqu'un socket est un file descriptor, les fonctions de fichier POSIX fonctionnent aussi :

```cpp
ssize_t read(int fd, void* buf, size_t count);  
ssize_t write(int fd, const void* buf, size_t count);  
```

Elles sont équivalentes à `recv()` / `send()` avec `flags = 0`. La différence est uniquement l'absence du paramètre `flags`. En pratique :

- Utilisez `send()` / `recv()` quand vous avez besoin de flags (`MSG_NOSIGNAL`, `MSG_PEEK`, `MSG_DONTWAIT`, etc.) — c'est le cas le plus fréquent en code réseau.  
- `read()` / `write()` peuvent être utiles quand votre code doit fonctionner de manière générique sur des file descriptors (fichiers, pipes, sockets) sans distinction.

## Comportement en mode non-bloquant

En mode non-bloquant (`SOCK_NONBLOCK` ou `O_NONBLOCK` via `fcntl`), les fonctions d'envoi et réception ne bloquent jamais. Leur comportement change de manière importante :

### `send()` non-bloquant

| Situation | Retour |
|-----------|--------|
| Buffer d'envoi a de la place | Nombre d'octets acceptés (peut être < `len`) |
| Buffer d'envoi plein | `-1` avec `errno = EAGAIN` |
| Connexion fermée par le pair | `-1` avec `errno = EPIPE` |

### `recv()` non-bloquant

| Situation | Retour |
|-----------|--------|
| Données disponibles dans le buffer | Nombre d'octets lus (peut être < `len`) |
| Aucune donnée disponible | `-1` avec `errno = EAGAIN` |
| Connexion fermée par le pair | `0` |

Le pattern d'utilisation typique est d'intégrer ces sockets avec un mécanisme de multiplexage (`epoll`, section 22.3) qui vous notifie quand le socket est prêt en lecture ou écriture, plutôt que de boucler en polling actif :

```cpp
// MAUVAIS — polling actif, consomme 100% CPU
while (true) {
    ssize_t n = recv(sockfd, buf, sizeof(buf), 0);
    if (n == -1 && errno == EAGAIN) {
        continue;  // Boucle serrée — gaspillage de CPU
    }
    // ...
}

// BON — attendre avec epoll que le socket soit prêt
// (voir section 22.3 pour le code complet)
epoll_wait(epfd, events, max_events, timeout);
// Seuls les sockets prêts sont traités
```

## Gestion de `EINTR` : le pattern universel

Les appels système bloquants (`send`, `recv`, `accept`, `connect`) peuvent être interrompus par un signal POSIX. Quand cela arrive, l'appel retourne `-1` avec `errno = EINTR`, même si aucune erreur n'a eu lieu.

La gestion correcte est de **réessayer l'appel** :

```cpp
ssize_t safe_recv(int sockfd, void* buf, size_t len, int flags) {
    ssize_t n;
    do {
        n = recv(sockfd, buf, len, flags);
    } while (n == -1 && errno == EINTR);
    return n;
}
```

Ce pattern `do { ... } while (errno == EINTR)` est omniprésent dans le code réseau POSIX. Vous pouvez l'encapsuler dans un helper générique :

```cpp
template<typename Func, typename... Args>  
auto retry_on_eintr(Func fn, Args&&... args) {  
    decltype(fn(std::forward<Args>(args)...)) result;
    do {
        result = fn(std::forward<Args>(args)...);
    } while (result == -1 && errno == EINTR);
    return result;
}

// Utilisation
ssize_t n = retry_on_eintr(recv, sockfd, buffer, sizeof(buffer), 0);
```

## Gestion des erreurs : récapitulatif par `errno`

Voici les erreurs les plus fréquentes sur les opérations d'envoi/réception, classées par gravité :

### Erreurs récupérables (réessayer)

| `errno` | Fonction | Signification |
|---------|----------|---------------|
| `EINTR` | Toutes | Signal reçu — relancer l'appel |
| `EAGAIN` / `EWOULDBLOCK` | Toutes (non-bloquant) | Ressource temporairement indisponible — réessayer plus tard |

### Erreurs de connexion (le pair a un problème)

| `errno` | Fonction | Signification |
|---------|----------|---------------|
| `EPIPE` | `send` | Le pair a fermé sa réception — plus possible d'écrire |
| `ECONNRESET` | `send`, `recv` | Le pair a envoyé un RST — connexion brutalement fermée |
| `ETIMEDOUT` | `send`, `recv` | Timeout TCP — le pair ne répond plus depuis longtemps |

### Erreurs graves (bug dans le code ou problème système)

| `errno` | Fonction | Signification |
|---------|----------|---------------|
| `EBADF` | Toutes | File descriptor invalide — le socket n'existe pas ou est déjà fermé |
| `ENOTSOCK` | Toutes | Le fd n'est pas un socket |
| `ENOTCONN` | `send`, `recv` | Le socket n'est pas connecté |
| `EMSGSIZE` | `sendto` (UDP) | Le datagramme est trop gros pour être envoyé |

## `sendmsg()` / `recvmsg()` : les fonctions avancées

Pour compléter le tableau, l'API sockets propose aussi `sendmsg()` et `recvmsg()`, qui utilisent une structure `msghdr` pour regrouper tous les paramètres :

```cpp
ssize_t sendmsg(int sockfd, const struct msghdr* msg, int flags);  
ssize_t recvmsg(int sockfd, struct msghdr* msg, int flags);  
```

Ces fonctions sont plus complexes mais offrent des capacités supplémentaires :

- **Scatter/gather I/O** — Envoyer ou recevoir depuis/vers plusieurs buffers en un seul appel système (vectored I/O avec `struct iovec`), évitant une copie intermédiaire.  
- **Données auxiliaires (ancillary data)** — Transmettre des informations de contrôle à travers le socket : file descriptors (pour le passage de fd entre processus via `AF_UNIX`), informations de routage, timestamps du noyau.  
- **Envoi multi-message** — `sendmmsg()` et `recvmmsg()` (Linux spécifique) envoient/reçoivent **plusieurs messages en un seul appel système**, réduisant l'overhead des context switches pour les serveurs UDP haute performance.

Ces fonctions sont essentielles pour les cas d'usage avancés (serveurs DNS haute performance, proxies réseau, passage de file descriptors entre processus), mais dépassent le cadre de cette section introductive.

## Résumé

L'envoi et la réception de données sur les sockets POSIX se résument à quatre fonctions, mais leur utilisation correcte demande de comprendre plusieurs subtilités fondamentales :

- **TCP est un flux d'octets** — Ni `send()` ni `recv()` ne préservent les frontières de messages. Implémentez un protocole de framing (longueur préfixée recommandé) pour délimiter vos messages.  
- **`send()` peut être partiel** — Bouclez toujours avec `send_all` pour garantir l'envoi complet. Utilisez `MSG_NOSIGNAL` pour éviter `SIGPIPE`.  
- **`recv()` retournant 0 signifie EOF** — Le pair a fermé la connexion. Ce n'est pas une erreur, c'est un événement à gérer.  
- **UDP préserve les frontières** — Chaque `sendto()` / `recvfrom()` correspond à un datagramme entier. Pas besoin de framing, mais pas de garantie de livraison.  
- **Gérez `EINTR` systématiquement** — Les signaux peuvent interrompre tout appel bloquant. Le pattern `do/while (errno == EINTR)` est universel.  
- **Le mode non-bloquant change les règles** — `EAGAIN` n'est pas une erreur mais une indication à attendre. Combinez avec `epoll` (section 22.3), jamais avec du polling actif.
