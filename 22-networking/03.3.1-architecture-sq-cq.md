# 22.3.3.1 Architecture : Submission Queue et Completion Queue

## Vue d'ensemble

L'architecture d'`io_uring` repose sur un mécanisme de **ring buffers en mémoire partagée** entre l'application (userspace) et le noyau (kernel). Cette mémoire partagée est le cœur de la conception : elle permet à l'application de soumettre des requêtes et de récupérer des résultats sans effectuer d'appel système dans le cas courant.

Le système se compose de trois structures distinctes :

- **La Submission Queue (SQ)** : un tableau circulaire d'indices. L'application y pousse les indices des SQE (Submission Queue Entries) qu'elle souhaite soumettre.  
- **Le tableau de SQE (SQ Entries array)** : un tableau plat contenant les descriptions des opérations. Chaque SQE décrit une opération complète (type, descripteur cible, buffer, offset, flags...).  
- **La Completion Queue (CQ)** : un tableau circulaire de CQE (Completion Queue Entries). Le noyau y dépose les résultats des opérations terminées.

La séparation entre la SQ (tableau d'indices) et le tableau de SQE est un choix de conception délibéré : elle permet à l'application de préparer des SQE dans le désordre puis de les soumettre dans l'ordre souhaité, sans copie ni réorganisation.

```
APPLICATION                          MÉMOIRE PARTAGÉE                          NOYAU
═══════════                          ════════════════                          ═════

                              ┌─────────────────────────┐
  Préparer le SQE  ──────────►│  SQ Entries Array       │
  dans le tableau             │  ┌─────┬─────┬─────┐    │
                              │  │SQE 0│SQE 1│SQE 2│... │
                              │  └─────┴─────┴─────┘    │
                              │         ▲               │
                              │         │ (index)       │
                              ├─────────────────────────┤
  Pousser l'index    ────────►│  Submission Queue (SQ)  │────────►  Le noyau consomme
  dans la SQ                  │  ┌───┬───┬───┬───┐      │           les indices, lit
                              │  │ 2 │ 0 │   │   │      │           les SQE, exécute
                              │  └───┴───┴───┴───┘      │           les opérations
                              │  head ──►    ◄── tail   │                │
                              ├─────────────────────────┤                │
  Récolter les       ◄────────│  Completion Queue (CQ)  │◄───────  Le noyau dépose
  résultats                   │  ┌─────┬─────┬─────┐    │          les résultats
                              │  │CQE 0│CQE 1│     │    │
                              │  └─────┴─────┴─────┘    │
                              │  head ──►    ◄── tail   │
                              └─────────────────────────┘
```

## Les appels système fondamentaux

Avant d'examiner les structures en détail, présentons les deux appels système bruts qui sous-tendent `io_uring`. En pratique, on utilise la librairie `liburing` (section 22.3.3.2) qui les encapsule, mais comprendre la couche basse est essentiel pour saisir l'architecture.

### io_uring_setup : création et mmap

```cpp
#include <linux/io_uring.h>
#include <sys/syscall.h>
#include <unistd.h>

// Appel système brut (pas de wrapper glibc standard)
int io_uring_setup(unsigned entries, struct io_uring_params *params);
```

`io_uring_setup` crée une instance `io_uring` et retourne un descripteur de fichier. Le paramètre `entries` indique le nombre de SQE souhaités (arrondi à la puissance de 2 supérieure par le noyau). La structure `params` est à la fois une entrée (flags de configuration) et une sortie (offsets mémoire pour le `mmap`).

Après l'appel, l'application doit effectuer **trois `mmap`** pour accéder aux structures en mémoire partagée :

```cpp
// 1. Mapper la Submission Queue (indices + métadonnées)
void *sq_ptr = mmap(nullptr, sq_size,
                    PROT_READ | PROT_WRITE,
                    MAP_SHARED | MAP_POPULATE,
                    ring_fd,
                    IORING_OFF_SQ_RING);

// 2. Mapper le tableau de SQE
void *sqe_ptr = mmap(nullptr, sqe_size,
                     PROT_READ | PROT_WRITE,
                     MAP_SHARED | MAP_POPULATE,
                     ring_fd,
                     IORING_OFF_SQES);

// 3. Mapper la Completion Queue (CQE + métadonnées)
void *cq_ptr = mmap(nullptr, cq_size,
                    PROT_READ | PROT_WRITE,
                    MAP_SHARED | MAP_POPULATE,
                    ring_fd,
                    IORING_OFF_CQ_RING);
```

Le flag `MAP_POPULATE` pré-alloue les pages physiques pour éviter les fautes de page pendant l'I/O — un détail important pour la performance.

> 📝 Depuis Linux 5.12, le flag `IORING_SETUP_CQ_AFF` permet de fusionner les deux premiers `mmap` (SQ ring et CQ ring) en un seul, simplifiant l'initialisation. Avec `IORING_SETUP_SINGLE_MMAP`, les trois zones peuvent même être mappées en un seul appel.

### io_uring_enter : soumettre et/ou attendre

```cpp
int io_uring_enter(unsigned int fd,
                   unsigned int to_submit,    // Nombre de SQE à soumettre
                   unsigned int min_complete,  // Attendre au moins N complétions
                   unsigned int flags,
                   sigset_t *sig);
```

C'est l'appel système central. Il remplit deux rôles selon les paramètres :

- **Soumission** (`to_submit > 0`) : informe le noyau que de nouvelles entrées sont disponibles dans la SQ.  
- **Attente** (`min_complete > 0` avec `IORING_ENTER_GETEVENTS`) : bloque jusqu'à ce qu'au moins `min_complete` CQE soient disponibles.

On peut combiner les deux dans un seul appel : « soumets ces 5 requêtes et attends qu'au moins 1 soit terminée ». C'est plus efficace que deux appels séparés.

## Anatomie d'un Submission Queue Entry (SQE)

Chaque SQE est une structure de 64 octets qui décrit intégralement une opération I/O :

```cpp
struct io_uring_sqe {
    __u8    opcode;     // Type d'opération (IORING_OP_READ, _WRITE, _RECV, ...)
    __u8    flags;      // Flags par SQE (IOSQE_FIXED_FILE, IOSQE_IO_LINK, ...)
    __u16   ioprio;     // Priorité I/O
    __s32   fd;         // Descripteur de fichier cible
    union {
        __u64 off;      // Offset dans le fichier (pour read/write fichier)
        __u64 addr2;    // Deuxième adresse (opérations avancées)
    };
    union {
        __u64 addr;     // Adresse du buffer userspace
        __u64 splice_off_in;
    };
    __u32   len;        // Taille du buffer / nombre d'octets
    union {
        /* Flags spécifiques à l'opération */
        __kernel_rwf_t  rw_flags;      // Pour read/write
        __u32           fsync_flags;    // Pour fsync
        __u32           poll_events;    // Pour poll
        __u32           sync_range_flags;
        __u32           msg_flags;      // Pour send/recv
        __u32           timeout_flags;
        __u32           accept_flags;   // Pour accept
        __u32           cancel_flags;
        __u32           open_flags;
        __u32           statx_flags;
        __u32           fadvise_advice;
        __u32           splice_flags;
    };
    __u64   user_data;  // Donnée opaque retournée dans le CQE correspondant
    /* ... champs additionnels pour les buffers groupés, etc. */
};
```

Les champs les plus importants pour l'utilisation courante :

**`opcode`** détermine le type d'opération. Voici les opcodes les plus utilisés :

| Opcode | Opération équivalente | Catégorie |
|--------|-----------------------|-----------|
| `IORING_OP_READ` | `pread` / `readv` | Fichier |
| `IORING_OP_WRITE` | `pwrite` / `writev` | Fichier |
| `IORING_OP_RECV` | `recv` | Réseau |
| `IORING_OP_SEND` | `send` | Réseau |
| `IORING_OP_ACCEPT` | `accept4` | Réseau |
| `IORING_OP_CONNECT` | `connect` | Réseau |
| `IORING_OP_CLOSE` | `close` | Système |
| `IORING_OP_OPENAT` | `openat` | Système |
| `IORING_OP_STATX` | `statx` | Système |
| `IORING_OP_FSYNC` | `fsync` | Fichier |
| `IORING_OP_TIMEOUT` | (timer interne) | Contrôle |
| `IORING_OP_CANCEL` | (annulation de SQE) | Contrôle |
| `IORING_OP_SEND_ZC` | `send` zero-copy | Réseau (6.0+) |

**`fd`** est le descripteur sur lequel l'opération porte. Pour les fichiers et sockets pré-enregistrés via `io_uring_register`, on utilise un index fixe au lieu du fd réel (avec le flag `IOSQE_FIXED_FILE`), ce qui évite au noyau de résoudre le descripteur à chaque opération.

**`addr`** et **`len`** décrivent le buffer userspace dans lequel lire ou à partir duquel écrire.

**`user_data`** est un entier 64 bits opaque que le noyau copie tel quel dans le CQE correspondant. C'est le mécanisme de corrélation entre soumissions et complétions — l'équivalent du `data.ptr` d'`epoll`, mais en plus flexible. On y place typiquement un identifiant de connexion, un pointeur vers un contexte, ou un tag encodant le type d'opération.

**`flags`** contrôle le comportement de la soumission individuelle :

- `IOSQE_IO_LINK` : chaîne cette SQE à la suivante — la suivante ne sera exécutée que si celle-ci réussit (dependency chain).  
- `IOSQE_IO_DRAIN` : attend que toutes les SQE précédentes soient terminées avant d'exécuter celle-ci (barrière).  
- `IOSQE_FIXED_FILE` : utilise un index de fichier pré-enregistré au lieu d'un fd classique.  
- `IOSQE_ASYNC` : force l'exécution dans un worker thread du noyau, même si l'opération pourrait être complétée de manière synchrone.

## Anatomie d'un Completion Queue Entry (CQE)

Le CQE est nettement plus simple — 16 octets :

```cpp
struct io_uring_cqe {
    __u64   user_data;  // Copié tel quel depuis le SQE correspondant
    __s32   res;        // Résultat de l'opération (ou code d'erreur négatif)
    __u32   flags;      // Flags additionnels (ex: IORING_CQE_F_MORE)
};
```

**`user_data`** est la valeur que vous avez placée dans le SQE. C'est ce qui vous permet de savoir *à quelle requête* ce résultat correspond.

**`res`** contient le résultat, avec la même sémantique que l'appel système correspondant. Pour un `IORING_OP_READ`, c'est le nombre d'octets lus (positif) ou un code d'erreur négatif (par exemple `-EAGAIN`, `-ECONNRESET`). Pour un `IORING_OP_ACCEPT`, c'est le nouveau descripteur de fichier. La valeur 0 pour une lecture indique une fin de fichier (EOF) / déconnexion.

**`flags`** fournit des métadonnées additionnelles. Le flag le plus notable est `IORING_CQE_F_MORE`, qui indique que d'autres CQE suivront pour la même SQE (utilisé avec les opérations *multishot* comme `accept` multishot ou `recv` multishot).

## Fonctionnement des ring buffers

Les SQ et CQ sont des **files circulaires** (ring buffers) basées sur deux compteurs atomiques : `head` et `tail`. Le protocole est le schéma classique producteur/consommateur en lock-free :

### Submission Queue

```
Producteur : APPLICATION              Consommateur : NOYAU
────────────────────────              ────────────────────

L'application écrit des               Le noyau lit les indices  
indices dans sq_array[tail]            depuis sq_array[head]  
puis incrémente tail.                  puis incrémente head.  

    head                tail
     │                   │
     ▼                   ▼
    ┌────┬────┬────┬────┬────┬────┬────┬────┐
    │    │ 3  │ 1  │ 7  │    │    │    │    │
    └────┴────┴────┴────┴────┴────┴────┴────┘
           ▲              ▲
           │              │
        à consommer    prochaine
        par le noyau   écriture app

    Entrées utilisées : tail - head
    Entrées libres   : taille_ring - (tail - head)
```

### Completion Queue

```
Producteur : NOYAU                    Consommateur : APPLICATION
──────────────────                    ──────────────────────────

Le noyau écrit des CQE                L'application lit les CQE
à cq_array[tail]                      depuis cq_array[head]
puis incrémente tail.                  puis incrémente head.

    head                tail
     │                   │
     ▼                   ▼
    ┌─────┬─────┬─────┬─────┬─────┬─────┐
    │     │CQE a│CQE b│CQE c│     │     │
    └─────┴─────┴─────┴─────┴─────┴─────┘
            ▲                 ▲
            │                 │
         à lire par        prochaine
         l'application     écriture noyau
```

L'accès aux compteurs utilise des **opérations atomiques avec memory ordering** approprié :

- Le producteur écrit les données *avant* d'incrémenter `tail` (avec une barrière `store-release`).  
- Le consommateur lit `tail` (avec une barrière `load-acquire`) *avant* de lire les données.

Ce protocole garantit que le consommateur ne lit jamais une entrée partiellement écrite, sans aucun mutex ni lock — c'est du lock-free pur.

```cpp
// Pseudocode simplifié — côté application, soumission d'un SQE

// 1. Obtenir un slot libre dans le tableau de SQE
unsigned idx = sq_tail & sq_mask;  // Masque = taille_ring - 1 (puissance de 2)

// 2. Remplir le SQE
sqe_array[idx].opcode    = IORING_OP_RECV;  
sqe_array[idx].fd        = client_fd;  
sqe_array[idx].addr      = reinterpret_cast<__u64>(buffer);  
sqe_array[idx].len       = buffer_size;  
sqe_array[idx].user_data = my_context_id;  

// 3. Publier l'index dans la SQ
sq_array[sq_tail & sq_mask] = idx;

// 4. Barrière mémoire + incrémenter tail
io_uring_smp_store_release(sq_tail_ptr, sq_tail + 1);
```

```cpp
// Pseudocode simplifié — côté application, récolte d'un CQE

// 1. Lire head avec acquire
unsigned head = io_uring_smp_load_acquire(cq_head_ptr);

// 2. Vérifier s'il y a des complétions
if (head == *cq_tail_ptr) {
    // Rien à récolter — la CQ est vide
    return;
}

// 3. Lire le CQE
struct io_uring_cqe *cqe = &cq_array[head & cq_mask];  
uint64_t context = cqe->user_data;  
int32_t  result  = cqe->res;  

// 4. Avancer head (consommer le CQE)
io_uring_smp_store_release(cq_head_ptr, head + 1);
```

> ⚠️ Ce pseudocode illustre le mécanisme. En pratique, utilisez `liburing` (section 22.3.3.2) qui encapsule correctement les barrières mémoire et la gestion des compteurs.

## Dimensionnement des queues

Le nombre d'entrées de la SQ est spécifié à la création via `io_uring_setup` et arrondi à la puissance de 2 supérieure. La CQ est par défaut **deux fois** la taille de la SQ (car une SQE peut potentiellement générer plusieurs CQE dans certains cas avancés).

| Paramètre | Valeur par défaut | Personnalisation |
|-----------|-------------------|------------------|
| Taille SQ | `entries` (arrondi à puissance de 2) | Paramètre de `io_uring_setup` |
| Taille CQ | 2 × taille SQ | `IORING_SETUP_CQSIZE` + champ `cq_entries` dans `params` |
| Taille SQE | 64 octets (fixe) | Non modifiable |
| Taille CQE | 16 octets (32 avec `IORING_SETUP_CQE32`) | Flag à la création |

**Que se passe-t-il si la CQ est pleine ?** Si le noyau termine une opération mais que la CQ n'a plus de place (l'application n'a pas consommé les CQE assez vite), le comportement dépend de la configuration :

- Par défaut, le noyau entre en état de **CQ overflow** : il stocke les CQE en excès dans une liste interne et positionne un flag. L'application doit drainer la CQ puis appeler `io_uring_enter` avec `IORING_ENTER_GETEVENTS` pour récupérer les CQE débordés.  
- Avec le flag `IORING_SETUP_CQ_NODROP` (Linux 5.5+), le noyau refuse de compléter de nouvelles SQE tant que la CQ est pleine, ce qui crée une contre-pression naturelle (back-pressure).

En pratique, dimensionnez la CQ suffisamment grande pour ne jamais déborder dans le cas courant, et traitez le débordement comme un cas d'erreur à logguer.

## Modes de soumission

### Mode normal (par défaut)

L'application prépare ses SQE, les pousse dans la SQ, puis appelle `io_uring_enter` pour notifier le noyau. C'est un appel système, mais un seul peut soumettre un lot arbitrairement grand de SQE — le coût est **amorti sur le nombre d'opérations**.

```
App: préparer SQE[0], SQE[1], SQE[2]  
App: io_uring_enter(to_submit=3, min_complete=1)  ← 1 seul appel système  
                                                      pour 3 opérations + attente
Kernel: exécute les 3 opérations  
Kernel: dépose 2 CQE (2 sont terminées)  
App: récolte les 2 CQE  
```

### Mode SQPOLL : zéro appel système

Avec le flag `IORING_SETUP_SQPOLL`, le noyau crée un **kernel thread dédié** qui scrute la SQ en permanence. L'application n'a qu'à écrire dans la SQ — pas besoin d'appeler `io_uring_enter` pour soumettre.

```
App: préparer SQE[0]      ← simple écriture mémoire  
App: préparer SQE[1]      ← simple écriture mémoire  
Kernel thread: détecte les nouvelles SQE, les exécute  
App: récolte les CQE       ← simple lecture mémoire  
```

Le kernel thread se met en veille après un délai d'inactivité configurable (`sq_thread_idle` dans `params`). Si l'application détecte que le thread dort (via un flag dans la SQ), elle doit appeler `io_uring_enter` pour le réveiller.

Ce mode est optimal pour les workloads à très faible latence (trading haute fréquence, serveurs de jeu) où chaque microseconde d'appel système compte. Le compromis est la consommation CPU du kernel thread en polling actif.

> 📝 Le mode SQPOLL nécessite la capability `CAP_SYS_NICE` ou que le processus s'exécute en root, car le kernel thread peut monopoliser un cœur CPU.

## Linked SQEs : chaîner les opérations

Le flag `IOSQE_IO_LINK` permet de créer des **chaînes de dépendances** entre SQE. Une SQE chaînée n'est exécutée que si la précédente a réussi. Si une SQE échoue, toutes les suivantes dans la chaîne sont annulées avec `-ECANCELED`.

Un cas d'usage classique : « lire depuis un fichier, puis envoyer sur un socket » :

```
SQE[0]: IORING_OP_READ   (fichier → buffer)    flags = IOSQE_IO_LINK  
SQE[1]: IORING_OP_SEND   (buffer → socket)     flags = 0 (fin de chaîne)  
```

Si la lecture échoue, l'envoi n'est jamais tenté. Sans le chaînage, il faudrait attendre le CQE de la lecture, vérifier le résultat, puis soumettre l'envoi — ce qui double la latence.

On peut aussi ajouter un **timeout** à une chaîne :

```
SQE[0]: IORING_OP_RECV        flags = IOSQE_IO_LINK  
SQE[1]: IORING_OP_LINK_TIMEOUT (5 secondes)  
```

Si le `recv` ne se termine pas dans les 5 secondes, il est annulé. Ce pattern remplace les timers manuels courants avec `epoll`.

## Multishot operations : accept et recv en continu

Les opérations classiques sont **singleshot** : une SQE produit exactement un CQE. Il faut resoumettre une nouvelle SQE après chaque complétion pour continuer à surveiller un descripteur (par exemple, resoumettre un `recv` après chaque réception).

Les opérations **multishot** (Linux 5.18+) restent actives après chaque complétion : une seule SQE produit des CQE *en continu* jusqu'à annulation ou erreur. Le CQE porte le flag `IORING_CQE_F_MORE` pour indiquer que d'autres suivront.

C'est particulièrement utile pour :

- **`IORING_OP_ACCEPT` multishot** : une seule SQE accepte toutes les connexions entrantes, indéfiniment. Chaque nouvelle connexion produit un CQE avec le nouveau fd.  
- **`IORING_OP_RECV` multishot** : une seule SQE reçoit tous les messages sur un socket, indéfiniment.

Le gain est double : moins de SQE à soumettre (réduit la pression sur la SQ) et moins de transitions user/kernel.

## Récapitulatif du flux complet

Pour synthétiser, voici le cycle de vie complet d'une opération I/O avec `io_uring` :

```
1. PRÉPARER          L'application remplit un SQE dans le tableau partagé :
                     opcode, fd, buffer, taille, user_data.

2. SOUMETTRE         L'application pousse l'index du SQE dans la SQ
                     et notifie le noyau (io_uring_enter ou SQPOLL).

3. EXÉCUTER          Le noyau retire l'index de la SQ, lit le SQE,
                     effectue l'opération I/O (potentiellement via
                     un worker thread interne pour les opérations
                     qui bloqueraient).

4. COMPLÉTER         Le noyau écrit un CQE dans la CQ avec :
                     - user_data (copié du SQE)
                     - res (résultat ou code d'erreur)

5. RÉCOLTER          L'application lit le CQE, identifie la requête
                     via user_data, traite le résultat.

6. LIBÉRER           L'application avance le head de la CQ pour
                     indiquer que le CQE a été consommé.
```

Ce flux se répète en continu dans la boucle événementielle. Le batching intervient naturellement : l'application peut préparer les étapes 1-2 pour plusieurs opérations, puis récolter plusieurs résultats (étapes 5-6) en lot — le tout avec au plus un seul appel système (`io_uring_enter`), voire aucun en mode SQPOLL.

La section suivante (22.3.3.2) montre comment `liburing` encapsule toute cette mécanique derrière une API ergonomique qui gère les `mmap`, les barrières mémoire et les compteurs de manière transparente.
