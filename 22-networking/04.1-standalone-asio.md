# 22.4.1 — Standalone Asio : Networking sans Boost

## Section 22.4 : Librairies réseau modernes

---

## Introduction

Standalone Asio est la version autonome de la librairie Asio — **header-only**, sans aucune dépendance sur Boost. C'est la forme recommandée pour les nouveaux projets : vous obtenez toute la puissance d'Asio (I/O asynchrone, timers, résolution DNS, support coroutines C++20) sans embarquer l'écosystème Boost complet.

Cette section vous guide de l'installation jusqu'à la construction d'un serveur TCP asynchrone complet, en couvrant les deux styles de programmation principaux : callbacks et coroutines C++20.

## Configuration CMake

Standalone Asio est header-only, mais nécessite quelques définitions et la liaison avec les threads POSIX :

```cmake
cmake_minimum_required(VERSION 3.20)  
project(asio_demo LANGUAGES CXX)  

set(CMAKE_CXX_STANDARD 20)       # Minimum C++20 pour les coroutines  
set(CMAKE_CXX_STANDARD_REQUIRED ON)  

find_package(Threads REQUIRED)

# Si installé via Conan/vcpkg :
find_package(asio CONFIG REQUIRED)

add_executable(server server.cpp)  
target_link_libraries(server PRIVATE asio::asio Threads::Threads)  

# Définition nécessaire pour standalone Asio (pas Boost)
target_compile_definitions(server PRIVATE ASIO_STANDALONE)
```

Si vous utilisez FetchContent ou une installation manuelle :

```cmake
add_executable(server server.cpp)  
target_include_directories(server PRIVATE ${asio_SOURCE_DIR}/asio/include)  
target_compile_definitions(server PRIVATE ASIO_STANDALONE)  
target_link_libraries(server PRIVATE Threads::Threads)
```

> 💡 La définition `ASIO_STANDALONE` indique à Asio d'utiliser les types de la bibliothèque standard (`std::error_code`, `std::system_error`) au lieu des types Boost. Avec les versions récentes d'Asio (1.22+), ce mode est le défaut, mais la définition reste une bonne pratique pour l'explicité.

## Premiers pas : les briques fondamentales

### L'`io_context`

Tout programme Asio commence par un `io_context`. C'est la boucle événementielle — l'équivalent de l'instance `epoll` + la boucle `epoll_wait` de la section 22.3 :

```cpp
#include <asio.hpp>

int main() {
    asio::io_context ctx;

    // ... configurer des opérations asynchrones ...

    ctx.run();  // Bloque et exécute la boucle événementielle
    // run() retourne quand il n'y a plus de travail en attente
}
```

`ctx.run()` ne retourne que lorsque toutes les opérations asynchrones sont terminées et qu'aucun handler ne reste en attente. Si vous voulez que la boucle tourne indéfiniment (comme un serveur), il suffit qu'il y ait toujours au moins une opération en cours — typiquement, un `async_accept` relancé après chaque connexion.

### Les types réseau fondamentaux

```cpp
using asio::ip::tcp;    // Protocole TCP  
using asio::ip::udp;    // Protocole UDP  

tcp::socket socket(ctx);           // Un socket TCP  
tcp::acceptor acceptor(ctx);       // Un socket d'écoute (serveur)  
tcp::resolver resolver(ctx);       // Résolution DNS  
tcp::endpoint endpoint;            // Couple adresse:port  

udp::socket udp_sock(ctx);         // Un socket UDP
```

Ces types encapsulent les file descriptors POSIX dans des objets RAII. Un `tcp::socket` est automatiquement fermé à sa destruction — plus besoin du wrapper `Socket` de la section 22.1.1.

## Client TCP synchrone

Commençons par un client synchrone (bloquant) pour comprendre l'API de base avant de passer à l'asynchrone :

```cpp
// sync_client.cpp
#include <asio.hpp>
#include <print>
#include <string>

using asio::ip::tcp;

int main() {
    try {
        asio::io_context ctx;

        // 1. Résolution DNS (synchrone)
        tcp::resolver resolver(ctx);
        auto endpoints = resolver.resolve("localhost", "8080");

        // 2. Connexion (synchrone, itère automatiquement sur les résultats)
        tcp::socket socket(ctx);
        asio::connect(socket, endpoints);

        std::println("Connecté à {}:{}",
                     socket.remote_endpoint().address().to_string(),
                     socket.remote_endpoint().port());

        // 3. Envoi
        std::string message = "Hello from Asio!\n";
        asio::write(socket, asio::buffer(message));

        // 4. Réception
        std::array<char, 4096> buf{};
        std::error_code ec;
        size_t n = socket.read_some(asio::buffer(buf), ec);

        if (!ec) {
            std::println("Réponse: {}", std::string_view(buf.data(), n));
        }

    } catch (const std::exception& e) {
        std::println(stderr, "Erreur: {}", e.what());
        return 1;
    }
}
```

### Points à noter

**`asio::connect`** itère automatiquement sur tous les endpoints retournés par le resolver (IPv4, IPv6, etc.) et tente de se connecter à chacun — exactement le pattern d'itération `getaddrinfo` que nous avons écrit manuellement en section 22.1.2.

**`asio::write`** est une fonction composée qui boucle jusqu'à ce que **toutes** les données soient envoyées — c'est l'équivalent de notre `send_all()`. La version bas niveau `socket.write_some()` peut faire un envoi partiel, comme `send()`.

**`asio::buffer`** crée une vue (non-owning) sur vos données. Il ne copie rien et ne gère pas la durée de vie du buffer — c'est votre responsabilité de garder les données vivantes jusqu'à la fin de l'opération.

## Serveur TCP avec callbacks

Voici un serveur echo asynchrone utilisant le modèle callbacks. C'est le style historique d'Asio, compatible C++11 :

```cpp
// callback_server.cpp
#include <asio.hpp>
#include <print>
#include <memory>

using asio::ip::tcp;

class Session : public std::enable_shared_from_this<Session> {  
public:  
    explicit Session(tcp::socket socket)
        : socket_(std::move(socket)) {}

    void start() { do_read(); }

private:
    void do_read() {
        auto self = shared_from_this();  // Garder la session vivante

        socket_.async_read_some(
            asio::buffer(buffer_),
            [self](std::error_code ec, size_t length) {
                if (!ec) {
                    self->do_write(length);
                } else {
                    std::println("Client déconnecté: {}", ec.message());
                }
            }
        );
    }

    void do_write(size_t length) {
        auto self = shared_from_this();

        asio::async_write(
            socket_,
            asio::buffer(buffer_.data(), length),
            [self](std::error_code ec, size_t /*bytes_written*/) {
                if (!ec) {
                    self->do_read();  // Boucle : lire → écrire → lire → ...
                } else {
                    std::println("Erreur écriture: {}", ec.message());
                }
            }
        );
    }

    tcp::socket socket_;
    std::array<char, 4096> buffer_{};
};

class Server {  
public:  
    Server(asio::io_context& ctx, uint16_t port)
        : acceptor_(ctx) {

        // Options AVANT bind (v6_only doit être positionné avant le bind)
        acceptor_.open(tcp::v6());
        acceptor_.set_option(tcp::acceptor::reuse_address(true));
        acceptor_.set_option(asio::ip::v6_only(false));  // Dual-stack IPv4+IPv6
        acceptor_.bind(tcp::endpoint(tcp::v6(), port));
        acceptor_.listen();

        std::println("Serveur echo (callbacks) sur le port {}", port);
        do_accept();
    }

private:
    void do_accept() {
        acceptor_.async_accept(
            [this](std::error_code ec, tcp::socket socket) {
                if (!ec) {
                    std::println("Nouveau client: {}:{}",
                        socket.remote_endpoint().address().to_string(),
                        socket.remote_endpoint().port());

                    std::make_shared<Session>(std::move(socket))->start();
                }
                do_accept();  // Relancer l'accept pour le prochain client
            }
        );
    }

    tcp::acceptor acceptor_;
};

int main() {
    try {
        asio::io_context ctx;
        Server server(ctx, 8080);
        ctx.run();
    } catch (const std::exception& e) {
        std::println(stderr, "Erreur fatale: {}", e.what());
        return 1;
    }
}
```

### Analyse du pattern `Session` avec `shared_ptr`

Ce pattern mérite une explication détaillée car il est **omniprésent** dans le code Asio :

```
                 shared_ptr<Session>
                 ┌─────────────────┐
                 │  socket_         │
                 │  buffer_         │
                 │                  │
    ┌───────────►│  do_read()  ─────┼──► async_read_some(handler)
    │            │                  │         │
    │            │  do_write() ─────┼──► async_write(handler)
    │            │                  │         │
    │            └──────────────────┘         │
    │                                         │
    └── handler capture self = shared_from_this()
        (maintient l'objet vivant pendant l'opération async)
```

**Le problème** — Une opération asynchrone est lancée, puis le flot d'exécution retourne à `io_context::run()`. Le handler sera appelé plus tard, quand l'I/O sera terminée. Si l'objet `Session` est détruit entre-temps (par exemple, parce que personne ne le référence plus), le handler accédera à de la mémoire libérée — undefined behavior.

**La solution** — `std::enable_shared_from_this` permet à l'objet de créer un `shared_ptr` vers lui-même. Chaque handler capture `self = shared_from_this()`, ce qui incrémente le compteur de références et empêche la destruction tant qu'une opération est en cours.

**Le cycle de vie** — L'objet `Session` est créé avec `std::make_shared`, `start()` lance la première lecture, et la chaîne `read → write → read → ...` maintient un handler actif en permanence. Quand une erreur survient (déconnexion, erreur I/O), le handler ne relance pas d'opération, le `shared_ptr` n'est plus capturé nulle part, le compteur de références tombe à zéro, et l'objet est détruit — emportant le socket avec lui (fermeture RAII).

C'est élégant mais subtil. Si vous oubliez de capturer `self` dans un seul handler, l'objet peut être détruit prématurément. C'est l'une des raisons pour lesquelles les coroutines sont recommandées : elles possèdent naturellement leur état et éliminent ce problème.

## Serveur TCP avec coroutines C++20

Voici le même serveur echo, réécrit avec les coroutines C++20. La différence de lisibilité est frappante :

```cpp
// coroutine_server.cpp
#include <asio.hpp>
#include <asio/co_spawn.hpp>
#include <asio/detached.hpp>
#include <asio/use_awaitable.hpp>
#include <print>

using asio::ip::tcp;  
using asio::awaitable;  
using asio::use_awaitable;  

awaitable<void> handle_client(tcp::socket socket) {
    auto endpoint = socket.remote_endpoint();
    std::println("Client connecté: {}:{}",
                 endpoint.address().to_string(), endpoint.port());

    try {
        std::array<char, 4096> buffer{};

        while (true) {
            // co_await suspend la coroutine — le thread n'est PAS bloqué
            size_t n = co_await socket.async_read_some(
                asio::buffer(buffer), use_awaitable);

            co_await asio::async_write(
                socket, asio::buffer(buffer.data(), n), use_awaitable);
        }
    } catch (const std::system_error& e) {
        if (e.code() == asio::error::eof ||
            e.code() == asio::error::connection_reset) {
            std::println("Client {}:{} déconnecté",
                         endpoint.address().to_string(), endpoint.port());
        } else {
            std::println(stderr, "Erreur: {}", e.what());
        }
    }
    // socket détruit ici → fermeture automatique
}

awaitable<void> listener(tcp::acceptor acceptor) {
    std::println("Serveur echo (coroutines) sur le port {}",
                 acceptor.local_endpoint().port());

    while (true) {
        tcp::socket socket = co_await acceptor.async_accept(use_awaitable);

        // Lancer une coroutine indépendante pour ce client
        co_spawn(acceptor.get_executor(), handle_client(std::move(socket)),
                 asio::detached);
    }
}

int main() {
    try {
        asio::io_context ctx;

        tcp::acceptor acceptor(ctx);
        acceptor.open(tcp::v6());
        acceptor.set_option(tcp::acceptor::reuse_address(true));
        acceptor.set_option(asio::ip::v6_only(false));
        acceptor.bind(tcp::endpoint(tcp::v6(), 8080));
        acceptor.listen();

        co_spawn(ctx, listener(std::move(acceptor)), asio::detached);

        ctx.run();
    } catch (const std::exception& e) {
        std::println(stderr, "Erreur fatale: {}", e.what());
        return 1;
    }
}
```

### Pourquoi les coroutines changent tout

Comparez les deux versions. Le serveur avec coroutines :

- **Se lit de haut en bas** — Le flux `read → write → read` est une simple boucle `while`. Pas de callbacks imbriqués, pas de méthodes `do_read()` / `do_write()` qui se rappellent mutuellement.
- **N'a pas besoin de `shared_from_this`** — La coroutine possède son propre frame qui contient le socket et le buffer. Tant que la coroutine est vivante (suspendue sur un `co_await`), ses variables locales existent. Pas de problème de durée de vie.
- **Gère les erreurs naturellement** — Un `try/catch` englobe toute la logique. Avec les callbacks, chaque handler doit vérifier `ec` individuellement.
- **Est entièrement asynchrone** — Malgré son apparence séquentielle, `co_await` **suspend** la coroutine sans bloquer le thread. Le thread retourne à `io_context::run()` et peut traiter d'autres coroutines. Quand l'I/O est prête, la coroutine reprend exactement où elle s'était arrêtée.

### `co_spawn` — Lancer des coroutines

`co_spawn` lance une coroutine dans un executor (typiquement celui de l'`io_context`). Le troisième paramètre est un **completion token** qui détermine ce qui se passe quand la coroutine se termine :

```cpp
// La coroutine tourne de manière autonome — les erreurs non capturées
// appellent std::terminate
co_spawn(ctx, my_coroutine(), asio::detached);

// Les erreurs sont passées au handler
co_spawn(ctx, my_coroutine(),
    [](std::exception_ptr ep) {
        if (ep) {
            try { std::rethrow_exception(ep); }
            catch (const std::exception& e) {
                std::println(stderr, "Coroutine error: {}", e.what());
            }
        }
    });
```

### `use_awaitable` vs `as_tuple`

Par défaut, `use_awaitable` fait que les erreurs I/O sont lancées comme des exceptions. C'est pratique pour la plupart des cas, mais parfois vous voulez inspecter l'erreur sans exception (par exemple, pour distinguer EOF d'une vraie erreur) :

```cpp
// Avec use_awaitable : l'erreur lance une exception
size_t n = co_await socket.async_read_some(buffer, use_awaitable);

// Avec as_tuple : l'erreur est retournée dans un tuple
auto [ec, n] = co_await socket.async_read_some(
    buffer, asio::as_tuple(use_awaitable));

if (ec == asio::error::eof) {
    // Déconnexion propre — pas une erreur
    co_return;
}
if (ec) {
    std::println(stderr, "Erreur: {}", ec.message());
    co_return;
}
// n octets lus avec succès
```

`as_tuple` est particulièrement utile dans les boucles de lecture où EOF est un événement attendu et non une condition exceptionnelle.

## Timers asynchrones

Les timers sont des citoyens de première classe dans Asio, intégrés nativement dans la boucle événementielle (pas besoin de `timerfd` manuel comme en section 22.3) :

```cpp
#include <asio.hpp>
#include <asio/steady_timer.hpp>
#include <chrono>

using namespace std::chrono_literals;

// --- Style callbacks ---
void start_timer_callback(asio::io_context& ctx) {
    auto timer = std::make_shared<asio::steady_timer>(ctx, 3s);

    timer->async_wait([timer](std::error_code ec) {
        if (!ec) {
            std::println("Timer expiré (callback)");
        }
    });
}

// --- Style coroutine ---
awaitable<void> periodic_timer() {
    auto executor = co_await asio::this_coro::executor;
    asio::steady_timer timer(executor);

    for (int i = 0; i < 5; ++i) {
        timer.expires_after(2s);
        co_await timer.async_wait(use_awaitable);
        std::println("Tick {} à t+{}s", i + 1, (i + 1) * 2);
    }

    std::println("Timer terminé");
}
```

### Timeout sur une opération I/O

Un cas d'usage fondamental : imposer un timeout sur une lecture réseau. Sans timeout, un `recv` sur un client inactif bloquerait la coroutine indéfiniment.

Asio ne propose pas de paramètre timeout directement sur les opérations I/O, mais le pattern idiomatique utilise un timer concurrent :

```cpp
awaitable<void> read_with_timeout(tcp::socket& socket,
                                   std::span<char> buffer,
                                   std::chrono::seconds timeout) {
    asio::steady_timer timer(socket.get_executor());
    timer.expires_after(timeout);

    // Lancer le timer et la lecture en parallèle
    // Le premier qui termine annule l'autre

    using namespace asio::experimental::awaitable_operators;

    auto result = co_await (
        socket.async_read_some(asio::buffer(buffer), use_awaitable)
        || timer.async_wait(use_awaitable)
    );

    // result.index() == 0 si la lecture a gagné
    // result.index() == 1 si le timer a gagné (timeout)
    if (result.index() == 1) {
        socket.cancel();  // Annuler la lecture en cours
        throw std::runtime_error("Lecture timeout");
    }

    // La lecture a réussi — result contient le nombre d'octets
}
```

> 💡 L'opérateur `||` (expérimental) lance les deux opérations en parallèle et attend la première qui termine. C'est l'équivalent de `select` ou `Promise.race` dans d'autres langages. Cet opérateur est disponible dans Asio 1.28+ via `<asio/experimental/awaitable_operators.hpp>`.

## Résolution DNS asynchrone

La résolution DNS peut prendre des centaines de millisecondes (voire des secondes avec un serveur DNS lent). En mode synchrone, cela bloque un thread. Asio offre une résolution asynchrone :

```cpp
awaitable<void> connect_async(asio::io_context& ctx,
                               const std::string& host,
                               const std::string& port) {
    tcp::resolver resolver(ctx);

    // Résolution asynchrone — ne bloque pas le thread
    auto endpoints = co_await resolver.async_resolve(
        host, port, use_awaitable);

    tcp::socket socket(ctx);

    // Connexion asynchrone — itère sur les endpoints
    auto connected = co_await asio::async_connect(
        socket, endpoints, use_awaitable);

    std::println("Connecté à {}:{}", 
                 connected.address().to_string(), connected.port());

    // Utiliser le socket...
}
```

`async_resolve` et `async_connect` sont les versions asynchrones des opérations synchrones vues plus haut. Elles libèrent le thread pendant l'attente, permettant à d'autres coroutines ou handlers de s'exécuter.

## Multi-threading avec `io_context`

Un seul thread appelant `ctx.run()` suffit pour gérer des milliers de connexions asynchrones. Mais si vos handlers effectuent des traitements CPU-intensifs (parsing complexe, compression, chiffrement), un seul thread peut devenir un goulot d'étranglement.

La solution est d'appeler `ctx.run()` depuis **plusieurs threads** :

```cpp
int main() {
    asio::io_context ctx;

    // ... configurer le serveur ...

    // Lancer un pool de threads sur le même io_context
    unsigned int num_threads = std::thread::hardware_concurrency();
    std::vector<std::jthread> threads;
    threads.reserve(num_threads);

    for (unsigned int i = 0; i < num_threads; ++i) {
        threads.emplace_back([&ctx] { ctx.run(); });
    }

    std::println("Serveur démarré avec {} threads", num_threads);

    // Les jthreads sont rejoints automatiquement à la destruction
}
```

### Strands : sérialisation sans mutex

Quand plusieurs threads exécutent la même boucle événementielle, deux handlers peuvent être invoqués **simultanément** sur deux threads différents. Si ces handlers accèdent aux mêmes données (par exemple, le buffer d'une même connexion), il y a une race condition.

Les **strands** sont le mécanisme d'Asio pour garantir qu'un ensemble de handlers s'exécute de manière sérialisée, sans avoir recours aux mutex :

```cpp
class Session : public std::enable_shared_from_this<Session> {  
public:  
    Session(tcp::socket socket)
        : socket_(std::move(socket)),
          strand_(asio::make_strand(socket_.get_executor())) {}

    void start() { do_read(); }

private:
    void do_read() {
        auto self = shared_from_this();

        // Binder le handler au strand garantit qu'un seul handler
        // de cette session s'exécute à la fois
        socket_.async_read_some(
            asio::buffer(buffer_),
            asio::bind_executor(strand_,
                [self](std::error_code ec, size_t n) {
                    if (!ec) self->do_write(n);
                })
        );
    }
    // ...

    tcp::socket socket_;
    asio::strand<asio::io_context::executor_type> strand_;
    std::array<char, 4096> buffer_{};
};
```

Avec les coroutines, le strand est implicite : une coroutine s'exécute séquentiellement par nature. Les `co_await` libèrent le thread, mais la reprise se fait dans le contexte de l'executor de la coroutine — pas besoin de `bind_executor` explicite dans la majorité des cas.

## Configuration avancée des sockets

Les options de sockets vues en section 22.1.1 (`SO_REUSEADDR`, `TCP_NODELAY`, `SO_KEEPALIVE`) ont leurs équivalents Asio typés :

```cpp
// v6_only et reuse_address doivent être positionnés AVANT bind
tcp::acceptor acceptor(ctx);  
acceptor.open(tcp::v6());  

// SO_REUSEADDR
acceptor.set_option(tcp::acceptor::reuse_address(true));

// IPv6 dual-stack (doit être avant bind)
acceptor.set_option(asio::ip::v6_only(false));

acceptor.bind(tcp::endpoint(tcp::v6(), 8080));  
acceptor.listen();  

// Sur un socket client :
tcp::socket socket(ctx);
// ... connect ...

// TCP_NODELAY
socket.set_option(tcp::no_delay(true));

// SO_KEEPALIVE
socket.set_option(asio::socket_base::keep_alive(true));

// Buffer de réception
socket.set_option(asio::socket_base::receive_buffer_size(65536));

// Buffer d'envoi
socket.set_option(asio::socket_base::send_buffer_size(65536));

// Mode non-bloquant (géré automatiquement par les opérations async,
// mais utile si vous mélangez appels sync et async)
socket.non_blocking(true);
```

L'avantage de ces wrappers typés est la sécurité : le compilateur vérifie que vous appliquez l'option au bon niveau de protocole. Pas de risque de confondre `SOL_SOCKET` et `IPPROTO_TCP`.

## Serveur echo complet : version production

Voici un serveur echo en coroutines intégrant les bonnes pratiques pour un usage proche de la production — arrêt propre, gestion des signaux, multi-thread et logging :

```cpp
// production_server.cpp
#include <asio.hpp>
#include <asio/co_spawn.hpp>
#include <asio/detached.hpp>
#include <asio/signal_set.hpp>
#include <asio/use_awaitable.hpp>
#include <print>
#include <atomic>

using asio::ip::tcp;  
using asio::awaitable;  
using asio::use_awaitable;  

std::atomic<int> active_connections{0};

awaitable<void> handle_client(tcp::socket socket) {
    auto ep = socket.remote_endpoint();
    auto client = ep.address().to_string() + ":" + std::to_string(ep.port());
    active_connections++;
    std::println("[+] {} (actives: {})", client, active_connections.load());

    try {
        socket.set_option(tcp::no_delay(true));
        std::array<char, 8192> buffer{};

        while (true) {
            auto [ec, n] = co_await socket.async_read_some(
                asio::buffer(buffer),
                asio::as_tuple(use_awaitable));

            if (ec) break;  // EOF ou erreur

            co_await asio::async_write(
                socket, asio::buffer(buffer.data(), n), use_awaitable);
        }
    } catch (const std::exception& e) {
        std::println("[!] {} erreur: {}", client, e.what());
    }

    active_connections--;
    std::println("[-] {} (actives: {})", client, active_connections.load());
}

awaitable<void> listener(tcp::acceptor acceptor) {
    while (true) {
        auto [ec, socket] = co_await acceptor.async_accept(
            asio::as_tuple(use_awaitable));

        if (ec) {
            if (ec == asio::error::operation_aborted) {
                break;  // Arrêt demandé
            }
            std::println(stderr, "Accept error: {}", ec.message());
            continue;
        }

        co_spawn(acceptor.get_executor(),
                 handle_client(std::move(socket)),
                 asio::detached);
    }
}

int main() {
    try {
        asio::io_context ctx;

        // Gestion des signaux d'arrêt
        asio::signal_set signals(ctx, SIGINT, SIGTERM);
        signals.async_wait([&ctx](std::error_code, int sig) {
            std::println("\nSignal {} reçu — arrêt en cours...", sig);
            ctx.stop();
        });

        // Configurer l'acceptor (options avant bind)
        tcp::acceptor acceptor(ctx);
        acceptor.open(tcp::v6());
        acceptor.set_option(tcp::acceptor::reuse_address(true));
        acceptor.set_option(asio::ip::v6_only(false));
        acceptor.bind(tcp::endpoint(tcp::v6(), 8080));
        acceptor.listen();

        std::println("Serveur echo (production) sur le port 8080");

        co_spawn(ctx, listener(std::move(acceptor)), asio::detached);

        // Multi-thread
        unsigned int num_threads = std::max(1u,
            std::thread::hardware_concurrency());
        std::vector<std::jthread> threads;

        for (unsigned int i = 1; i < num_threads; ++i) {
            threads.emplace_back([&ctx] { ctx.run(); });
        }

        ctx.run();  // Le thread principal participe aussi

        std::println("Serveur arrêté. {} connexions encore actives.",
                     active_connections.load());

    } catch (const std::exception& e) {
        std::println(stderr, "Erreur fatale: {}", e.what());
        return 1;
    }
}
```

### Points notables

**`asio::signal_set`** — Remplace `signal()` / `sigaction()` et le self-pipe trick de la section 22.2. Le handler de signal est intégré dans la boucle événementielle Asio, pas besoin de pipe ni de `signalfd`.

**`ctx.stop()`** — Arrête la boucle événementielle. Tous les appels à `ctx.run()` retournent. Les opérations en cours sont annulées (les handlers reçoivent `asio::error::operation_aborted`).

**`as_tuple` dans le listener** — Permet de détecter l'annulation (`operation_aborted`) sans lancer d'exception, pour une sortie propre de la boucle d'accept.

**Multi-thread naturel** — Plusieurs threads appellent `ctx.run()`. Asio distribue les handlers entre les threads disponibles. Les coroutines étant séquentielles par nature, chaque session est traitée de manière sérialisée — pas de race condition sur les buffers de session.

## Client TCP asynchrone avec coroutines

```cpp
// async_client.cpp
#include <asio.hpp>
#include <asio/co_spawn.hpp>
#include <asio/use_awaitable.hpp>
#include <print>

using asio::ip::tcp;  
using asio::awaitable;  
using asio::use_awaitable;  

awaitable<void> run_client(asio::io_context& ctx,
                            std::string host, std::string port) {
    // Résolution DNS asynchrone
    tcp::resolver resolver(ctx);
    auto endpoints = co_await resolver.async_resolve(host, port, use_awaitable);

    // Connexion asynchrone
    tcp::socket socket(ctx);
    co_await asio::async_connect(socket, endpoints, use_awaitable);

    std::println("Connecté à {}:{}", host, port);

    // Envoi
    std::string msg = "Hello from async client!\n";
    co_await asio::async_write(socket, asio::buffer(msg), use_awaitable);

    // Réception
    std::array<char, 4096> buffer{};
    auto [ec, n] = co_await socket.async_read_some(
        asio::buffer(buffer), asio::as_tuple(use_awaitable));

    if (!ec) {
        std::println("Réponse: {}", std::string_view(buffer.data(), n));
    }
}

int main() {
    try {
        asio::io_context ctx;
        co_spawn(ctx, run_client(ctx, "localhost", "8080"),
            [](std::exception_ptr ep) {
                if (ep) {
                    try { std::rethrow_exception(ep); }
                    catch (const std::exception& e) {
                        std::println(stderr, "Erreur: {}", e.what());
                    }
                }
            });
        ctx.run();
    } catch (const std::exception& e) {
        std::println(stderr, "Erreur fatale: {}", e.what());
        return 1;
    }
}
```

## UDP avec Asio

L'API UDP en Asio est plus simple que TCP — pas de connexion, pas d'acceptor :

```cpp
awaitable<void> udp_echo_server(asio::io_context& ctx, uint16_t port) {
    udp::socket socket(ctx);
    socket.open(udp::v6());
    socket.set_option(asio::ip::v6_only(false));
    socket.bind(udp::endpoint(udp::v6(), port));

    std::println("Serveur UDP echo sur le port {}", port);

    std::array<char, 65535> buffer{};

    while (true) {
        udp::endpoint sender;
        auto [ec, n] = co_await socket.async_receive_from(
            asio::buffer(buffer), sender,
            asio::as_tuple(use_awaitable));

        if (ec) {
            std::println(stderr, "Erreur recv: {}", ec.message());
            continue;
        }

        co_await socket.async_send_to(
            asio::buffer(buffer.data(), n), sender, use_awaitable);
    }
}
```

La correspondance avec l'API POSIX est directe : `async_receive_from` ↔ `recvfrom`, `async_send_to` ↔ `sendto`. Asio gère le non-bloquant et `epoll` en interne.

## Comparaison avec l'approche `epoll` brute

Pour mesurer concrètement ce qu'Asio apporte, voici un tableau comparatif entre le serveur `epoll` de la section 22.3 et le serveur Asio coroutines :

```
Aspect                  epoll brut (22.3)           Asio coroutines
──────────────────────  ──────────────────────      ────────────────────────
Lignes de code          ~120                        ~60  
Gestion des buffers     Manuelle                    asio::buffer + async_write  
  partiels en écriture
Timers                  timerfd + epoll_ctl          steady_timer, une ligne  
Résolution DNS          getaddrinfo (bloquant)      async_resolve  
                        ou thread dédié
Gestion des signaux     self-pipe ou signalfd       signal_set, 3 lignes  
Portabilité             Linux uniquement            Linux, macOS, Windows  
TLS/SSL                 Implémentation manuelle     asio::ssl::stream (wrapper)  
Gestion d'erreurs       errno, vérification         error_code / exceptions  
                        après chaque appel          unifiées
Lisibilité              Boucle switch/if            Code séquentiel naturel  
Durée de vie objets     Gestion manuelle            RAII + coroutine frame  
Multi-thread            EPOLLONESHOT + mutex         Strands (automatique  
                                                     avec coroutines)
```

L'approche `epoll` brute n'est pas « mauvaise » — elle est juste **inappropriée** pour du code applicatif. C'est le niveau que les *auteurs de librairies* manipulent. En tant qu'utilisateur, Asio (ou une abstraction équivalente) est le bon niveau.

## Pièges courants et bonnes pratiques

### Ne pas oublier `ctx.run()`

L'erreur de débutant classique : lancer des opérations asynchrones puis terminer `main()` sans appeler `ctx.run()`. Rien ne se passe — les opérations sont enregistrées mais jamais exécutées.

### Durée de vie des buffers

Le buffer passé à une opération async doit rester vivant et stable **jusqu'à l'appel du handler**. Ceci est incorrect :

```cpp
// DANGEREUX — le string peut être détruit avant la fin de l'envoi
void send_bad(tcp::socket& socket) {
    std::string msg = "Hello";
    asio::async_write(socket, asio::buffer(msg), [](auto, auto) {});
    // msg est détruit ici, mais l'écriture async n'est peut-être pas finie
}
```

Solutions correctes :

```cpp
// Solution 1 : capturer le buffer dans le handler
void send_ok(tcp::socket& socket) {
    auto msg = std::make_shared<std::string>("Hello");
    asio::async_write(socket, asio::buffer(*msg),
        [msg](auto, auto) { /* msg reste vivant */ });
}

// Solution 2 : buffer membre de la classe Session
// Solution 3 : variable locale dans une coroutine (la plus propre)
awaitable<void> send_coro(tcp::socket& socket) {
    std::string msg = "Hello";  // Vivant tant que la coroutine est suspendue
    co_await asio::async_write(socket, asio::buffer(msg), use_awaitable);
    // msg est valide pendant tout le co_await
}
```

### Ne pas mélanger sync et async sur le même socket

Un socket utilisé en mode asynchrone ne doit pas être utilisé en mode synchrone en parallèle — le comportement est indéfini. Choisissez un mode et restez-y.

### `cancel()` et `close()`

Pour arrêter proprement les opérations sur un socket :

- `socket.cancel()` — Annule les opérations async en cours. Les handlers sont invoqués avec `asio::error::operation_aborted`. Le socket reste ouvert.
- `socket.close()` — Ferme le socket et annule les opérations en cours. Le socket ne peut plus être utilisé.

## Résumé

Standalone Asio transforme la programmation réseau C++ en offrant une abstraction asynchrone portable au-dessus des mécanismes systèmes (`epoll`, `kqueue`, IOCP) :

- **Header-only, zéro dépendance** — Facile à intégrer via Conan, vcpkg ou FetchContent.
- **Coroutines C++20** — Le style recommandé en 2026. Du code asynchrone qui se lit comme du synchrone, sans les pièges des callbacks (`shared_from_this`, durée de vie).
- **Callbacks** — Toujours disponibles pour le code C++11/14/17 ou les cas où les coroutines ne sont pas souhaitées.
- **Timers, DNS, signaux** — Intégrés nativement dans la boucle événementielle, sans bricolage de `timerfd` ou self-pipe.
- **Multi-thread** — Plusieurs threads sur un `io_context`, sérialisés par strands quand nécessaire.
- **Portabilité** — Même code sur Linux, macOS et Windows.

Pour les nouveaux projets en 2026, **standalone Asio avec coroutines C++20 est le point de départ recommandé** pour toute programmation réseau en C++.
