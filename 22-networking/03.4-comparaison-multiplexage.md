# 22.3.4 Comparaison : select vs poll vs epoll vs io_uring

## Introduction

Les sections 22.3.1 à 22.3.3 ont présenté chaque mécanisme de multiplexage en détail. Cette section les met en regard de manière systématique — architecture, performances, complexité de code, et critères de choix — pour vous permettre de prendre une décision informée dans vos projets.

---

## Comparaison architecturale

### Modèle d'interaction avec le noyau

Les quatre mécanismes diffèrent fondamentalement dans leur manière d'interagir avec le noyau :

```
select / poll             epoll                     io_uring
─────────────             ─────                     ────────

Chaque itération :        Initialisation :           Initialisation :
  copier N fds ───►         epoll_create1()            io_uring_setup()
  kernel parcourt N         epoll_ctl(ADD, fd1)        mmap SQ, SQE, CQ
  copier résultats ◄──      epoll_ctl(ADD, fd2)
                            ...                      Chaque itération :
Coût : O(N) × 2 copies                                écrire SQE en mémoire
       par itération      Chaque itération :           (pas de copie)
                            epoll_wait()               io_uring_enter()
                            → retourne K fds prêts       ou SQPOLL (0 syscall)
                            read()/write() par fd      → retourne K CQE
                                                         (résultats complets)
                          Coût : O(K) + 1 syscall
                                 + 1 syscall par I/O   Coût : O(K) + 0-1 syscall
                                                              pour N opérations
```

La progression est claire : de la copie intégrale à chaque appel (select/poll), à l'enregistrement persistant avec notification des prêts (epoll), jusqu'à la soumission/complétion en mémoire partagée sans appel système (io_uring).

### Modèle de notification

| | select | poll | epoll (LT) | epoll (ET) | io_uring |
|---|--------|------|------------|------------|----------|
| **Paradigme** | Readiness | Readiness | Readiness | Readiness | Completion |
| **Notification** | Level-triggered | Level-triggered | Level-triggered | Edge-triggered | N/A (résultat direct) |
| **Qui effectue l'I/O** | L'application | L'application | L'application | L'application | Le noyau |
| **Appels système par I/O** | 2 (select + read) | 2 (poll + read) | 2 (epoll_wait + read) | 2 (epoll_wait + read) | 0-1 (submit batché) |

Avec les mécanismes readiness, le noyau vous informe que le descripteur est *prêt*, mais c'est votre code qui effectue le `read()` ou `write()`. Avec io_uring, vous décrivez l'opération souhaitée et le noyau l'exécute — vous récupérez directement le résultat.

## Comparaison des performances

### Scalabilité en fonction du nombre de connexions

Le comportement asymptotique est le facteur le plus déterminant à grande échelle :

| Métrique | select | poll | epoll | io_uring |
|----------|--------|------|-------|----------|
| **Ajout/suppression d'un fd** | O(1) (bit set) | O(1) (append) | O(log n) (rb-tree) | O(1) (SQE array) |
| **Coût par appel d'attente** | O(N total) | O(N total) | O(K actifs) | O(K actifs) |
| **Copie user↔kernel par appel** | 3 bitsets de N bits | N × sizeof(pollfd) | 0 (état dans le noyau) | 0 (mémoire partagée) |
| **Appels système par opération I/O** | 2 | 2 | 2 | 0-1 (batché/SQPOLL) |

Où **N** = nombre total de descripteurs surveillés et **K** = nombre de descripteurs actifs dans l'itération courante. La distinction est cruciale : pour un serveur avec 100 000 connexions ouvertes dont 50 sont actives à un instant donné, N = 100 000 et K = 50.

### Ordres de grandeur pratiques

Les chiffres suivants sont indicatifs et proviennent de benchmarks publics sur du matériel serveur récent (Xeon, NVMe). Ils illustrent les tendances relatives, pas des valeurs absolues :

**Débit réseau (requêtes/seconde, serveur echo, 1 thread) :**

| Connexions simultanées | select | poll | epoll | io_uring |
|------------------------|--------|------|-------|----------|
| 10 | ~300K | ~300K | ~310K | ~320K |
| 100 | ~250K | ~260K | ~305K | ~320K |
| 1 000 | ~80K | ~90K | ~300K | ~315K |
| 10 000 | ~9K | ~10K | ~290K | ~310K |
| 50 000 | — (FD_SETSIZE) | ~2K | ~270K | ~305K |

**Observations :**

- **À faible nombre de connexions** (< 100), les quatre mécanismes sont comparables. La complexité algorithmique n'a pas d'impact mesurable, et les constantes (overhead d'initialisation, allocation) dominent. Choisir `select` pour 5 connexions est parfaitement raisonnable.

- **Entre 100 et 1 000 connexions**, l'écart se creuse. `select` et `poll` commencent à montrer leur coût O(N), tandis qu'`epoll` et `io_uring` restent stables.

- **Au-delà de 10 000 connexions**, `select` devient inutilisable (limite `FD_SETSIZE`), `poll` s'effondre, et seuls `epoll` et `io_uring` restent viables.

- **L'écart epoll/io_uring en pur networking** est relativement modeste (5-15 %). L'avantage majeur d'`io_uring` se manifeste dans les workloads mixtes (réseau + fichier) et dans la réduction du nombre d'appels système.

**I/O fichier (lectures aléatoires 4K sur NVMe, 1 thread) :**

| Mécanisme | IOPS (opérations/seconde) |
|-----------|---------------------------|
| `read()` synchrone en boucle | ~100K |
| Thread pool + epoll (4 workers) | ~250K |
| io_uring (queue depth 64) | ~400K+ |

C'est sur les I/O fichier que l'avantage d'`io_uring` est le plus spectaculaire : le noyau peut réordonner et paralléliser les opérations sur les queues NVMe, ce qu'aucune approche basée sur des appels `read()` séquentiels ne permet.

## Comparaison de la complexité de code

### Squelette d'un serveur echo (pseudocode simplifié)

**select :**
```
boucle:
    reconstruire fd_set            ← O(N), à chaque itération
    reconstruire timeout
    select(max_fd+1, &fds, ...)
    pour chaque fd de 0 à max_fd:  ← O(N)
        si FD_ISSET(fd, &fds):
            si fd == server: accept()
            sinon: recv() puis send()
```

**poll :**
```
boucle:
    poll(fds, nfds, timeout)
    pour chaque fd dans fds[]:     ← O(N)
        si fd.revents & POLLIN:
            si fd == server: accept(), ajouter à fds[]
            sinon: recv() puis send()
```

**epoll :**
```
init:
    epoll_create1()
    epoll_ctl(ADD, server_fd)

boucle:
    n = epoll_wait(events, ...)
    pour i de 0 à n-1:             ← O(K)
        si events[i].fd == server: accept(), epoll_ctl(ADD, client)
        sinon: recv() puis send()
```

**io_uring :**
```
init:
    io_uring_queue_init()
    soumettre accept

boucle:
    wait_cqe(&cqe)
    selon cqe.user_data:           ← O(K)
        Accept: soumettre recv + nouvel accept
        Recv:   soumettre send
        Send:   soumettre recv
    submit()
```

Le modèle io_uring est conceptuellement différent : il n'y a pas de « boucle qui vérifie quels fd sont prêts », mais une machine à états où chaque CQE déclenche la soumission de l'opération suivante. C'est plus proche de la programmation par continuations.

### Lignes de code pour un serveur echo fonctionnel

| Mécanisme | Lignes approximatives | Complexité conceptuelle |
|-----------|----------------------|------------------------|
| select | ~80 | Faible (mais pièges FD_SETSIZE, reconstruction) |
| poll | ~70 | Faible |
| epoll (LT) | ~75 | Moyenne (gestion de epoll_ctl) |
| epoll (ET) | ~90 | Élevée (drainage EAGAIN obligatoire) |
| io_uring | ~100 | Élevée (encodage user_data, machine à états) |

La complexité de code d'`io_uring` est plus élevée pour un cas d'usage simple (serveur echo réseau pur). L'investissement se justifie quand les besoins dépassent le multiplexage réseau basique.

| Fonctionnalité | select | poll | epoll | io_uring |
|----------------|--------|------|-------|----------|
| Limite de descripteurs | 1024 (FD_SETSIZE) | ulimit -n | ulimit -n | ulimit -n |

| Support conteneurs (Docker) | Oui | Oui | Oui | Restreint par défaut |

## Portabilité

La portabilité est souvent le facteur décisif dans le choix initial :

| Système | select | poll | epoll | kqueue | IOCP | io_uring |
|---------|--------|------|-------|--------|------|----------|
| Linux | ✅ | ✅ | ✅ | ❌ | ❌ | ✅ (5.1+) |
| macOS / FreeBSD | ✅ | ✅ | ❌ | ✅ | ❌ | ❌ |
| Windows | ✅ (Winsock) | ❌ | ❌ | ❌ | ✅ | ❌ |
| Solaris / illumos | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ |

Si votre code doit tourner sur plusieurs systèmes d'exploitation, trois stratégies s'offrent à vous :

- **Dénominateur commun** : utiliser `select` (partout) ou `poll` (partout sauf Windows natif). Acceptable pour un petit nombre de connexions.  
- **Compilation conditionnelle** : implémenter une couche d'abstraction avec `epoll` sous Linux, `kqueue` sous macOS/BSD, `IOCP` sous Windows. C'est exactement ce que font libevent, libev et libuv.  
- **Librairie d'abstraction** : utiliser Boost.Asio ou standalone Asio (section 22.4), qui encapsule le mécanisme optimal de chaque plateforme derrière une API unifiée. C'est l'approche recommandée pour les projets multiplateformes.

## Arbre de décision

```
Votre application a besoin de multiplexage I/O :
│
├── Portabilité multiplateforme requise ?
│   │
│   ├── OUI → Peu de connexions (< 50) ?
│   │         ├── OUI → select ou poll
│   │         └── NON → Librairie d'abstraction (Asio, libuv)
│   │                   → Voir section 22.4
│   │
│   └── NON (Linux uniquement)
│       │
│       ├── Workload réseau pur ?
│       │   │
│       │   ├── < 100 connexions → poll ou epoll (peu de différence)
│       │   │
│       │   ├── 100 — 100K connexions → epoll
│       │   │
│       │   └── > 100K connexions ou latence critique
│       │       → epoll (éprouvé) ou io_uring (optimal)
│       │
│       ├── I/O fichier asynchrone nécessaire ?
│       │   └── OUI → io_uring (seul mécanisme viable)
│       │
│       ├── Mixte réseau + fichier ?
│       │   └── io_uring (une seule boucle, pas de thread pool)
│       │
│       └── Besoin de features avancées ?
│           (zero-copy, linked ops, multishot, cancel)
│           └── io_uring
```

## Matrice de recommandation par contexte

| Contexte | Recommandation | Justification |
|----------|---------------|---------------|
| Outil CLI surveillant stdin + 1 socket | `poll` | Simple, suffisant, portable |
| Prototype / preuve de concept | `poll` | API la plus simple, pas de dépendance |
| Serveur réseau de production (Linux) | `epoll` | Éprouvé, écosystème mature, documentation abondante |
| Serveur haute performance (> 100K conn.) | `epoll` ou `io_uring` | Profiler pour choisir |
| Base de données / stockage (Linux) | `io_uring` | I/O fichier async indispensable |
| Proxy / reverse proxy (Linux) | `io_uring` | Mixte réseau + fichier + zero-copy |
| Application multiplateforme | Asio | Abstraction du mécanisme natif |
| Code legacy à maintenir | Conserver l'existant | Migration seulement si bottleneck identifié |
| Environnement conteneurisé restreint | `epoll` | io_uring bloqué par défaut dans Docker |

## Évolution probable

Le paysage du multiplexage I/O sous Linux converge progressivement vers `io_uring` comme mécanisme universel. Plusieurs tendances renforcent cette trajectoire :

- **Adoption croissante dans les projets majeurs** : PostgreSQL, RocksDB, et de nombreux frameworks intègrent progressivement des backends `io_uring`.  
- **Maturité sécuritaire** : les restrictions et le sandboxing introduits depuis Linux 5.11 répondent aux préoccupations initiales. Les profils seccomp Docker évoluent pour autoriser `io_uring` de manière contrôlée.  
- **Convergence des librairies** : Boost.Asio propose un backend `io_uring` expérimental. Les futures versions en feront probablement le backend par défaut sous Linux.  
- **std::execution (C++26)** : le framework Senders/Receivers standardisé en C++26 (section 12.14.4) fournira un modèle d'asynchronisme de haut niveau qui pourra s'appuyer sur `io_uring` comme backend d'exécution, achevant de démocratiser l'accès à ces performances.

Cela dit, `epoll` n'est pas menacé de disparition. Il restera le choix pragmatique et éprouvé pour la majorité des serveurs réseau pendant de nombreuses années, tout comme `poll` reste pertinent pour les cas simples plus de 40 ans après son introduction.

---

## Résumé en une phrase

- **select** : le plus portable, le plus limité — pour les scripts et les outils simples.  
- **poll** : select sans la limite FD_SETSIZE — pour les prototypes et la portabilité.  
- **epoll** : le standard de production Linux — pour 99 % des serveurs réseau.  
- **io_uring** : le futur du I/O Linux — pour les workloads qui exigent le maximum de performance ou qui mélangent réseau et fichier.
