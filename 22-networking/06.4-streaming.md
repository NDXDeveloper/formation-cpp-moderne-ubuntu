# 22.6.4 — Streaming bidirectionnel

## Section 22.6 : gRPC et Protocol Buffers

---

## Introduction

La section 22.6.3 a couvert les appels RPC unaires — une requête, une réponse. Mais gRPC offre trois autres patterns de communication qui exploitent le multiplexage natif d'HTTP/2 : le **streaming serveur**, le **streaming client** et le **streaming bidirectionnel**. Ces patterns ouvrent des possibilités impossibles en REST classique : flux de données en temps réel, uploads progressifs, protocoles conversationnels, et synchronisation continue entre services.

Cette section implémente les quatre patterns à travers des exemples concrets, couvre les subtilités de gestion du cycle de vie des streams, et aborde le backpressure — le mécanisme qui empêche un producteur rapide de submerger un consommateur lent.

---

## Les quatre patterns : rappel

```protobuf
service DataPipeline {
    // 1. Unaire — couvert en section 22.6.3
    rpc GetStatus (StatusRequest) returns (StatusResponse);

    // 2. Streaming serveur — le serveur envoie un flux
    rpc Subscribe (SubscribeRequest) returns (stream Event);

    // 3. Streaming client — le client envoie un flux
    rpc Upload (stream DataChunk) returns (UploadSummary);

    // 4. Bidirectionnel — les deux envoient des flux simultanément
    rpc Chat (stream ChatMessage) returns (stream ChatMessage);
}
```

Le mot-clé `stream` devant un type active le streaming pour ce côté. Les combinaisons donnent les quatre patterns. Le unaire a été couvert en section 22.6.3 — cette section se concentre sur les trois patterns avec streaming.

## Le fichier `.proto` de référence

Les exemples de cette section utilisent le service suivant :

```protobuf
// proto/streaming_demo.proto
syntax = "proto3";

package streaming.v1;

import "google/protobuf/timestamp.proto";

// Namespace C++ généré : streaming::v1

service MetricsService {
    // Streaming serveur : le client s'abonne, le serveur envoie les métriques
    rpc Subscribe (SubscribeRequest) returns (stream MetricEvent);

    // Streaming client : le client envoie un flux de métriques
    rpc Ingest (stream MetricSample) returns (IngestSummary);

    // Bidirectionnel : échange continu de commandes et résultats
    rpc InteractiveQuery (stream QueryRequest) returns (stream QueryResult);
}

// ── Messages ──

message SubscribeRequest {
    repeated string metric_names = 1;  // Filtrer par nom (vide = tout)
    int32 interval_ms = 2;            // Intervalle entre les envois
}

message MetricEvent {
    string name = 1;
    double value = 2;
    google.protobuf.Timestamp timestamp = 3;
}

message MetricSample {
    string name = 1;
    double value = 2;
    map<string, string> labels = 3;
}

message IngestSummary {
    int64 samples_received = 1;
    int64 samples_accepted = 2;
    int64 samples_rejected = 3;
    double duration_seconds = 4;
}

message QueryRequest {
    string query = 1;          // Ex: "cpu_usage > 80"
    int32 max_results = 2;
}

message QueryResult {
    string query = 1;          // La requête correspondante
    repeated MetricEvent results = 2;
    string error = 3;          // Vide si succès
}
```

## Streaming serveur

### Concept

Le client envoie une requête unique, le serveur répond avec un flux de messages. Le client lit les messages un par un jusqu'à ce que le serveur ferme le stream. C'est le pattern le plus courant en streaming — il couvre les notifications push, les flux de données, les résultats paginés par stream, et le monitoring temps réel.

```
Client                          Serveur
──────                          ───────
  ── SubscribeRequest ────────►
  ◄── MetricEvent (cpu: 42%) ──
  ◄── MetricEvent (mem: 71%) ──
  ◄── MetricEvent (cpu: 38%) ──
  ◄── MetricEvent (mem: 68%) ──
  ...                           (continue tant que le serveur veut)
  ◄── (fin du stream) ─────────
```

### Implémentation serveur

```cpp
#include <grpcpp/grpcpp.h>
#include "streaming_demo.grpc.pb.h"
#include <print>
#include <thread>
#include <chrono>
#include <random>

using namespace streaming::v1;

class MetricsServiceImpl final : public MetricsService::Service {  
public:  
    grpc::Status Subscribe(
        grpc::ServerContext* context,
        const SubscribeRequest* request,
        grpc::ServerWriter<MetricEvent>* writer) override
    {
        std::println("[Subscribe] Client connecté, intervalle={}ms",
                     request->interval_ms());

        int interval = request->interval_ms() > 0 ? request->interval_ms() : 1000;
        std::mt19937 rng{std::random_device{}()};
        std::uniform_real_distribution<double> cpu_dist(10.0, 95.0);
        std::uniform_real_distribution<double> mem_dist(40.0, 90.0);

        // Construire le set de métriques demandées
        std::set<std::string> filter(
            request->metric_names().begin(),
            request->metric_names().end());
        bool send_all = filter.empty();

        while (!context->IsCancelled()) {
            auto now = std::chrono::system_clock::now();
            auto seconds = std::chrono::duration_cast<std::chrono::seconds>(
                now.time_since_epoch()).count();

            // Générer et envoyer les métriques
            if (send_all || filter.contains("cpu_usage")) {
                MetricEvent event;
                event.set_name("cpu_usage");
                event.set_value(cpu_dist(rng));
                event.mutable_timestamp()->set_seconds(seconds);

                if (!writer->Write(event)) {
                    // Write retourne false si le client a fermé le stream
                    std::println("[Subscribe] Client déconnecté (write failed)");
                    return grpc::Status::OK;
                }
            }

            if (send_all || filter.contains("memory_usage")) {
                MetricEvent event;
                event.set_name("memory_usage");
                event.set_value(mem_dist(rng));
                event.mutable_timestamp()->set_seconds(seconds);

                if (!writer->Write(event)) {
                    return grpc::Status::OK;
                }
            }

            std::this_thread::sleep_for(std::chrono::milliseconds(interval));
        }

        std::println("[Subscribe] Client annulé");
        return grpc::Status::OK;
    }
    
    // ... autres méthodes implémentées plus bas ...
};
```

### Points clés du streaming serveur

**`grpc::ServerWriter<T>`** — L'objet qui permet d'écrire des messages dans le stream. Chaque appel à `Write()` envoie un message au client. Le stream reste ouvert entre les appels.

**`writer->Write()` retourne `false`** — C'est le signal que le client s'est déconnecté ou a annulé le stream. Vous devez vérifier le retour à chaque écriture et arrêter proprement si le client est parti. Ignorer ce retour et continuer à écrire est une erreur courante qui gaspille des ressources.

**`context->IsCancelled()`** — Vérifie si le client a annulé l'appel (deadline dépassée, annulation explicite, déconnexion). C'est la condition de sortie naturelle pour les streams longue durée. Attention : cette vérification a un léger coût — ne l'appelez pas à chaque nanoseconde, mais à chaque itération de votre boucle.

**Retour du `Status`** — Le `Status` retourné par la méthode est envoyé au client après la fermeture du stream. `Status::OK` indique que le stream s'est terminé normalement.

### Implémentation client

```cpp
void subscribe_to_metrics(MetricsService::Stub* stub) {
    SubscribeRequest request;
    request.add_metric_names("cpu_usage");
    request.add_metric_names("memory_usage");
    request.set_interval_ms(2000);

    grpc::ClientContext context;
    // Pas de deadline stricte — le stream peut durer longtemps
    // Mais on peut annuler manuellement avec context.TryCancel()

    // Obtenir le reader
    auto reader = stub->Subscribe(&context, request);

    // Lire les événements un par un
    MetricEvent event;
    int count = 0;
    while (reader->Read(&event)) {
        std::println("[{}] {} = {:.1f}%",
                     event.timestamp().seconds(),
                     event.name(),
                     event.value());

        // Exemple : arrêter après 10 événements
        if (++count >= 10) {
            context.TryCancel();  // Annuler le stream côté client
            break;
        }
    }

    // Récupérer le status final
    grpc::Status status = reader->Finish();
    if (status.ok() || status.error_code() == grpc::StatusCode::CANCELLED) {
        std::println("Stream terminé après {} événements", count);
    } else {
        std::println(stderr, "Erreur: {}", status.error_message());
    }
}
```

**`reader->Read()`** — Bloque jusqu'au prochain message du serveur. Retourne `true` si un message a été lu, `false` si le stream est terminé (le serveur a retourné son `Status`).

**`context.TryCancel()`** — Permet au client d'annuler le stream à tout moment. Le serveur verra `IsCancelled()` retourner `true` et le `Write()` suivant retourner `false`.

**`reader->Finish()`** — Récupère le `Status` final envoyé par le serveur. Doit être appelé après que `Read()` ait retourné `false`. Appeler `Finish()` avant d'avoir consommé tous les messages est une erreur.

## Streaming client

### Concept

Le client envoie un flux de messages, le serveur ne répond qu'une seule fois, quand le client signale la fin du stream. C'est le pattern naturel pour l'upload de données, l'envoi batch, et l'agrégation côté serveur.

```
Client                          Serveur
──────                          ───────
  ── MetricSample (cpu) ──────►
  ── MetricSample (mem) ──────►
  ── MetricSample (disk) ─────►
  ── MetricSample (net) ──────►
  ── (fin du stream) ─────────►
  ◄── IngestSummary ───────────
```

### Implémentation serveur

```cpp
grpc::Status Ingest(
    grpc::ServerContext* context,
    grpc::ServerReader<MetricSample>* reader,
    IngestSummary* response) override
{
    auto start = std::chrono::steady_clock::now();

    MetricSample sample;
    int64_t received = 0;
    int64_t accepted = 0;
    int64_t rejected = 0;

    // Lire les samples un par un jusqu'à la fin du stream
    while (reader->Read(&sample)) {
        received++;

        // Validation simple
        if (sample.name().empty()) {
            rejected++;
            continue;
        }

        if (sample.value() < 0 || sample.value() > 100) {
            rejected++;
            continue;
        }

        // Traiter le sample (ici, juste compter)
        accepted++;

        // Vérifier l'annulation périodiquement
        if (received % 1000 == 0 && context->IsCancelled()) {
            return grpc::Status(grpc::StatusCode::CANCELLED,
                                "Ingestion annulée");
        }
    }

    auto end = std::chrono::steady_clock::now();
    double duration = std::chrono::duration<double>(end - start).count();

    response->set_samples_received(received);
    response->set_samples_accepted(accepted);
    response->set_samples_rejected(rejected);
    response->set_duration_seconds(duration);

    std::println("[Ingest] Reçu: {}, Accepté: {}, Rejeté: {}, Durée: {:.2f}s",
                 received, accepted, rejected, duration);

    return grpc::Status::OK;
}
```

**`grpc::ServerReader<T>`** — L'objet de lecture côté serveur. `Read()` bloque jusqu'au prochain message du client ou retourne `false` quand le client a fermé son stream (`WritesDone()`).

### Implémentation client

```cpp
void ingest_metrics(MetricsService::Stub* stub) {
    IngestSummary summary;
    grpc::ClientContext context;
    context.set_deadline(
        std::chrono::system_clock::now() + std::chrono::seconds(30));

    // Obtenir le writer
    auto writer = stub->Ingest(&context, &summary);

    // Envoyer un flux de samples
    std::mt19937 rng{std::random_device{}()};
    std::uniform_real_distribution<double> dist(0.0, 100.0);

    for (int i = 0; i < 10000; ++i) {
        MetricSample sample;
        sample.set_name("sensor_" + std::to_string(i % 10));
        sample.set_value(dist(rng));
        (*sample.mutable_labels())["host"] = "server-01";
        (*sample.mutable_labels())["region"] = "eu-west";

        if (!writer->Write(sample)) {
            // Le serveur a fermé le stream prématurément
            std::println(stderr, "Écriture interrompue à l'itération {}", i);
            break;
        }
    }

    // Signaler la fin du stream client
    writer->WritesDone();

    // Récupérer la réponse du serveur
    grpc::Status status = writer->Finish();

    if (status.ok()) {
        std::println("Ingestion terminée:");
        std::println("  Reçus:    {}", summary.samples_received());
        std::println("  Acceptés: {}", summary.samples_accepted());
        std::println("  Rejetés:  {}", summary.samples_rejected());
        std::println("  Durée:    {:.2f}s", summary.duration_seconds());
    } else {
        std::println(stderr, "Erreur: {}", status.error_message());
    }
}
```

**`writer->Write()`** — Envoie un message dans le stream. Retourne `false` si le serveur a annulé ou fermé la connexion.

**`writer->WritesDone()`** — Signale au serveur que le client a fini d'envoyer. C'est cette opération qui provoque le retour `false` de `reader->Read()` côté serveur. Sans `WritesDone()`, le serveur attend indéfiniment.

**`writer->Finish()`** — Attend la réponse du serveur (le message unique de retour + le status). Doit être appelé après `WritesDone()`.

## Streaming bidirectionnel

### Concept

Les deux côtés envoient et reçoivent des flux de messages **simultanément et indépendamment**. Chaque côté peut écrire et lire à son propre rythme. C'est le pattern le plus flexible et le plus complexe — il permet les protocoles conversationnels, le chat, la synchronisation en temps réel, et les pipelines de traitement interactifs.

```
Client                          Serveur
──────                          ───────
  ── QueryRequest ────────────►
  ◄── QueryResult ─────────────
  ── QueryRequest ────────────►
  ── QueryRequest ────────────►
  ◄── QueryResult ─────────────
  ◄── QueryResult ─────────────
  ── QueryRequest ────────────►
  ◄── QueryResult ─────────────
  ── (fin) ───────────────────►
  ◄── (fin) ───────────────────
```

L'ordre des messages est indépendant : le serveur peut répondre à la requête 2 avant la requête 1, envoyer plusieurs réponses pour une seule requête, ou envoyer des messages proactivement sans attendre de requête.

### Implémentation serveur

```cpp
grpc::Status InteractiveQuery(
    grpc::ServerContext* context,
    grpc::ServerReaderWriter<QueryResult, QueryRequest>* stream) override
{
    std::println("[InteractiveQuery] Session démarrée");

    QueryRequest request;
    while (stream->Read(&request)) {
        std::println("[Query] '{}'", request.query());

        QueryResult result;
        result.set_query(request.query());

        // Simuler le traitement de la requête
        if (request.query().empty()) {
            result.set_error("Requête vide");
            stream->Write(result);
            continue;
        }

        // Simuler des résultats
        int max = request.max_results() > 0 ? request.max_results() : 5;
        std::mt19937 rng{std::random_device{}()};
        std::uniform_real_distribution<double> dist(0.0, 100.0);

        for (int i = 0; i < max; ++i) {
            MetricEvent* event = result.add_results();
            event->set_name("metric_" + std::to_string(i));
            event->set_value(dist(rng));

            auto now = std::chrono::system_clock::now();
            event->mutable_timestamp()->set_seconds(
                std::chrono::duration_cast<std::chrono::seconds>(
                    now.time_since_epoch()).count());
        }

        if (!stream->Write(result)) {
            std::println("[InteractiveQuery] Client déconnecté");
            break;
        }
    }

    std::println("[InteractiveQuery] Session terminée");
    return grpc::Status::OK;
}
```

**`grpc::ServerReaderWriter<W, R>`** — L'objet bidirectionnel. Il combine les capacités de `ServerReader` (méthode `Read()`) et `ServerWriter` (méthode `Write()`). Le premier type template (`W`) est le type des messages écrits, le second (`R`) est le type des messages lus.

Dans cet exemple, le serveur adopte un modèle requête-réponse au sein du stream bidirectionnel — il lit une requête, construit une réponse, l'envoie, puis attend la requête suivante. Mais ce n'est qu'un des patterns possibles. Le serveur pourrait aussi envoyer des messages proactivement, ou traiter les requêtes en parallèle sur plusieurs threads.

### Implémentation client

```cpp
void interactive_query(MetricsService::Stub* stub) {
    grpc::ClientContext context;

    // Obtenir le stream bidirectionnel
    auto stream = stub->InteractiveQuery(&context);

    // ── Thread d'écriture : envoie les requêtes ──
    std::jthread writer([&stream] {
        std::vector<std::string> queries = {
            "cpu_usage > 80",
            "memory_usage > 70",
            "",                     // Requête invalide (pour tester l'erreur)
            "disk_io > 50",
            "network_latency > 100"
        };

        for (const auto& q : queries) {
            QueryRequest request;
            request.set_query(q);
            request.set_max_results(3);

            if (!stream->Write(request)) {
                std::println(stderr, "Écriture échouée");
                break;
            }

            // Petit délai pour simuler un client interactif
            std::this_thread::sleep_for(std::chrono::milliseconds(500));
        }

        // Signaler la fin des requêtes
        stream->WritesDone();
    });

    // ── Thread principal : lit les réponses ──
    QueryResult result;
    while (stream->Read(&result)) {
        if (!result.error().empty()) {
            std::println("[Erreur] Requête '{}': {}",
                         result.query(), result.error());
            continue;
        }

        std::println("[Résultat] Requête '{}' — {} résultats:",
                     result.query(), result.results_size());

        for (const auto& event : result.results()) {
            std::println("  {} = {:.1f}", event.name(), event.value());
        }
    }

    grpc::Status status = stream->Finish();
    if (!status.ok()) {
        std::println(stderr, "Stream terminé avec erreur: {}",
                     status.error_message());
    }
}
```

### Pattern lecture/écriture concurrente

L'exemple ci-dessus illustre un point fondamental du streaming bidirectionnel : **les lectures et écritures peuvent se faire sur des threads différents**. gRPC garantit que :

- Un seul `Read()` peut être en cours à la fois.
- Un seul `Write()` peut être en cours à la fois.
- Mais `Read()` et `Write()` peuvent être en cours **simultanément** sur deux threads différents.

```
Thread 1 (écriture)           Thread 2 (lecture)
───────────────────           ──────────────────
stream->Write(req1)           stream->Read(&result)  ← bloque  
stream->Write(req2)           // résultat reçu  
                              stream->Read(&result)  ← bloque
stream->WritesDone()          // résultat reçu
                              stream->Read(&result)  ← retourne false
                              stream->Finish()
```

C'est la raison pour laquelle le client lance un thread séparé pour l'écriture. Le thread principal se charge de la lecture. Sans cette séparation, le client devrait alterner lecture et écriture — ce qui ne fonctionne que si le protocole est strictement ping-pong.

> ⚠️ **Ne faites jamais deux `Write()` simultanés ni deux `Read()` simultanés.** gRPC ne supporte pas la concurrence au sein d'une même direction. Protégez les appels avec un mutex si plusieurs threads peuvent écrire.

## Patterns avancés

### Stream serveur avec notification par événements

L'implémentation du `Subscribe` montrée plus haut utilise un polling (`sleep` + boucle). En production, un pattern plus efficace utilise une **queue d'événements** partagée entre les producteurs de données et les streams :

```cpp
#include <queue>
#include <mutex>
#include <condition_variable>

template<typename T>  
class EventBus {  
public:  
    // Les producteurs publient des événements
    void publish(const T& event) {
        std::lock_guard lock{mutex_};
        // Nettoyer les abonnés expirés et publier aux actifs
        std::erase_if(subscribers_, [](auto& w) { return w.expired(); });
        for (auto& weak : subscribers_) {
            if (auto sub = weak.lock()) {
                sub->queue_.push(event);
            }
        }
        cv_.notify_all();
    }

    // Chaque stream s'abonne et reçoit sa propre queue
    class Subscription {
    public:
        bool wait_and_pop(T& event, std::chrono::milliseconds timeout) {
            std::unique_lock lock{bus_.mutex_};
            if (!bus_.cv_.wait_for(lock, timeout,
                    [this] { return !queue_.empty(); })) {
                return false;  // Timeout
            }
            event = std::move(queue_.front());
            queue_.pop();
            return true;
        }

    private:
        friend class EventBus;
        Subscription(EventBus& bus) : bus_(bus) {}
        EventBus& bus_;
        std::queue<T> queue_;
    };

    std::shared_ptr<Subscription> subscribe() {
        std::lock_guard lock{mutex_};
        auto sub = std::shared_ptr<Subscription>(
            new Subscription(*this));
        subscribers_.push_back(sub);
        return sub;
    }

private:
    std::vector<std::weak_ptr<Subscription>> subscribers_;
    std::mutex mutex_;
    std::condition_variable cv_;
};

// Utilisation dans le service
EventBus<MetricEvent> metric_bus_;  // Partagé entre le service et les producteurs

grpc::Status Subscribe(
    grpc::ServerContext* context,
    const SubscribeRequest* request,
    grpc::ServerWriter<MetricEvent>* writer) override
{
    auto sub = metric_bus_.subscribe();

    while (!context->IsCancelled()) {
        MetricEvent event;
        if (sub->wait_and_pop(event, std::chrono::milliseconds(500))) {
            if (!writer->Write(event)) {
                break;  // Client déconnecté
            }
        }
        // Timeout → revérifier IsCancelled et reboucler
    }

    return grpc::Status::OK;
}
```

Ce pattern découple la production d'événements de leur distribution aux clients. Les producteurs (threads de collecte de métriques, handlers d'événements) publient dans le bus, et chaque stream client reçoit sa propre copie des événements via sa subscription.

### Streaming avec annulation et deadline

La gestion correcte de l'annulation est cruciale pour les streams longue durée :

```cpp
// Côté client : annuler après un certain temps ou un nombre de messages
void subscribe_with_limit(MetricsService::Stub* stub,
                           int max_events,
                           std::chrono::seconds max_duration) {
    SubscribeRequest request;
    request.set_interval_ms(500);

    grpc::ClientContext context;
    // Deadline absolue
    context.set_deadline(std::chrono::system_clock::now() + max_duration);

    auto reader = stub->Subscribe(&context, request);

    MetricEvent event;
    int count = 0;
    while (reader->Read(&event)) {
        std::println("[{}/{}] {} = {:.1f}",
                     ++count, max_events, event.name(), event.value());

        if (count >= max_events) {
            context.TryCancel();
            break;
        }
    }

    grpc::Status status = reader->Finish();
    // CANCELLED est attendu si on a annulé manuellement
    // DEADLINE_EXCEEDED si la deadline a expiré
    if (!status.ok() &&
        status.error_code() != grpc::StatusCode::CANCELLED &&
        status.error_code() != grpc::StatusCode::DEADLINE_EXCEEDED) {
        std::println(stderr, "Erreur inattendue: {}", status.error_message());
    }
}
```

## Backpressure

### Le problème

Que se passe-t-il quand le serveur envoie des messages plus vite que le client ne peut les consommer ? Sans mécanisme de régulation, les buffers internes de gRPC (côté serveur et dans la pile HTTP/2) se remplissent, la consommation mémoire explose, et le système finit par crasher ou perdre des messages.

### Le mécanisme de gRPC

gRPC gère le backpressure automatiquement grâce au **contrôle de flux HTTP/2**. Quand les buffers de réception du client sont pleins, la couche HTTP/2 signale au serveur de ralentir. Concrètement :

- `Write()` côté serveur devient **bloquant** quand les buffers sont pleins. L'appel ne retourne que quand de la place se libère (le client a consommé des données).
- Cela crée naturellement un ralentissement du producteur — le serveur ne peut pas écrire plus vite que le client ne lit.

```
Serveur                     HTTP/2                      Client
───────                     ──────                      ──────
Write(msg1) → rapide        buffer: [msg1] ──────────►  Read() → msg1  
Write(msg2) → rapide        buffer: [msg2] ──────────►  Read() → msg2  
Write(msg3) → rapide        buffer: [msg3, ...]         (client lent)  
Write(msg4) → BLOQUE ⏳     buffer: [PLEIN]             Read() → msg3  
Write(msg4) → retourne      buffer: [msg4] ──────────►  (place libérée)  
```

### Implications pour le design

**Côté serveur**, si `Write()` bloque trop longtemps (client très lent ou déconnecté), le thread serveur est monopolisé. Pour les serveurs à haut débit avec l'API synchrone, cela peut épuiser le pool de threads. Solutions :

- Utiliser l'API asynchrone (callbacks) pour ne pas bloquer de thread.
- Imposer un timeout sur les écritures via la deadline du contexte.
- Détecter les clients lents et les déconnecter proactivement.

**Côté client**, `Read()` ne pose pas de problème de backpressure — il bloque naturellement en attendant des données. Mais si le client traite les messages lentement, c'est le serveur qui sera ralenti par le backpressure HTTP/2 — ce qui peut être acceptable ou non selon votre architecture.

### Buffering côté application

Pour un contrôle plus fin, vous pouvez introduire un buffer applicatif entre la production et l'envoi :

```cpp
grpc::Status Subscribe(
    grpc::ServerContext* context,
    const SubscribeRequest* request,
    grpc::ServerWriter<MetricEvent>* writer) override
{
    constexpr size_t MAX_BUFFER = 1000;
    std::queue<MetricEvent> buffer;

    auto sub = metric_bus_.subscribe();

    while (!context->IsCancelled()) {
        MetricEvent event;
        if (sub->wait_and_pop(event, std::chrono::milliseconds(100))) {
            if (buffer.size() >= MAX_BUFFER) {
                // Buffer plein — supprimer le plus ancien (drop policy)
                buffer.pop();
                // Alternative : rejeter le nouveau, ou signaler une erreur
            }
            buffer.push(std::move(event));
        }

        // Drainer le buffer vers le stream
        while (!buffer.empty()) {
            if (!writer->Write(buffer.front())) {
                return grpc::Status::OK;  // Client déconnecté
            }
            buffer.pop();
        }
    }

    return grpc::Status::OK;
}
```

Ce pattern de « drop oldest » est courant dans les systèmes de monitoring : il vaut mieux perdre quelques anciennes métriques que de bloquer le pipeline entier. La politique de drop (oldest, newest, random) dépend de votre cas d'usage.

## Cycle de vie complet d'un stream

### Côté serveur

```
                    ┌── Read() en boucle ──┐
                    │                      │    (stream client)
                    ▼                      │
  Appel méthode → Traitement → Write() ────┘
       │                          │
       │                          ▼
       │                    Write retourne false
       │                    (client parti)
       │                          │
       ▼                          ▼
  Read retourne false       Sortir du handler
  (client a fait                  │
   WritesDone)                    ▼
       │                   return Status::OK
       ▼                   (envoyé au client)
  Sortir du handler
```

### Côté client

```
  Obtenir stream ──► Write() en boucle ──► WritesDone()
       │                                       │
       │              (en parallèle)           │
       │                                       ▼
       └──────────► Read() en boucle ──► Read retourne false
                                               │
                                               ▼
                                          Finish()
                                          (récupère le Status)
```

### Règles à respecter

1. **Toujours appeler `Finish()`** côté client — c'est lui qui récupère le status et nettoie les ressources.
2. **Appeler `WritesDone()` avant `Finish()`** si vous avez un stream client — sinon le serveur attend indéfiniment des messages.
3. **Consommer tous les `Read()`** avant `Finish()` — les messages non lus seront perdus.
4. **Vérifier `Write()` retournant `false`** — toujours, sur les deux côtés.
5. **Un `ClientContext` par appel** — ne réutilisez jamais un contexte entre deux appels RPC.

## Quand utiliser chaque pattern

| Pattern | Cas d'usage | Exemples concrets |
|---------|-------------|-------------------|
| **Unaire** | Opérations CRUD, requêtes ponctuelles | GetUser, CreateOrder, DeleteItem |
| **Stream serveur** | Flux de données serveur → client | Notifications, logs en temps réel, cours de bourse, résultats de recherche progressifs |
| **Stream client** | Envoi batch ou progressif | Upload de fichiers, ingestion de métriques, envoi de traces |
| **Bidirectionnel** | Protocoles conversationnels | Chat, jeux multijoueurs, synchronisation d'état, shell interactif distant |

Le pattern unaire couvre la majorité des besoins (80 % des méthodes gRPC dans un projet typique). Le streaming serveur est le deuxième plus courant (notifications, événements). Le streaming client et le bidirectionnel sont réservés aux cas où la nature du problème l'exige — ne les utilisez pas par défaut « parce que c'est plus puissant ». La complexité supplémentaire (gestion du cycle de vie, concurrence, backpressure) n'est justifiée que quand le pattern unaire ne peut pas répondre au besoin.

## Résumé

Le streaming gRPC exploite HTTP/2 pour offrir des patterns de communication impossibles en REST classique :

- **Streaming serveur** (`ServerWriter`) — Le serveur envoie un flux de réponses. Vérifiez toujours le retour de `Write()` et `IsCancelled()`.
- **Streaming client** (`ServerReader` / `ClientWriter`) — Le client envoie un flux de requêtes. `WritesDone()` signale la fin côté client.
- **Bidirectionnel** (`ServerReaderWriter` / `ClientReaderWriter`) — Les deux côtés lisent et écrivent simultanément. Séparez lecture et écriture sur des threads distincts côté client.
- **Backpressure** — Géré automatiquement par HTTP/2. `Write()` bloque quand le consommateur est lent. Introduisez un buffer applicatif avec politique de drop si le blocage est inacceptable.
- **Annulation** — `TryCancel()` côté client, `IsCancelled()` côté serveur. Les deadlines s'appliquent aussi aux streams.
- **Règle pragmatique** — Utilisez le pattern unaire par défaut. Le streaming n'est justifié que quand la nature du problème l'exige.
