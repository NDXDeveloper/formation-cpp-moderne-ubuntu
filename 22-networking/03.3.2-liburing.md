# 22.3.3.2 liburing : Interface C/C++ simplifiée

## Pourquoi liburing ?

La section précédente a montré la mécanique interne d'`io_uring` : `mmap` de trois régions, manipulation de compteurs atomiques avec des barrières mémoire, gestion manuelle des indices dans les ring buffers. Ce niveau de détail est essentiel pour comprendre l'architecture, mais écrire du code de production directement contre les appels système bruts serait fastidieux et fragile.

**liburing** est la librairie officielle créée et maintenue par Jens Axboe (l'auteur d'`io_uring` lui-même). Elle encapsule toute la plomberie — `mmap`, barrières mémoire, gestion des compteurs, soumission et récolte — derrière une API C propre et ergonomique. C'est la manière recommandée d'utiliser `io_uring` dans tout code C et C++.

Ce que liburing gère pour vous :

- L'appel à `io_uring_setup` et les trois `mmap` sont remplacés par un seul appel `io_uring_queue_init`.  
- L'allocation et la publication des SQE se font via `io_uring_get_sqe` + des fonctions `io_uring_prep_*`.  
- La soumission et l'attente sont encapsulées dans `io_uring_submit` et `io_uring_wait_cqe`.  
- Les barrières mémoire sont correctement placées en interne — vous n'avez jamais à y penser.  
- Le nettoyage (unmap + close) est un seul appel `io_uring_queue_exit`.

## Installation sur Ubuntu

### Depuis les paquets système

```bash
# Ubuntu 22.04+
sudo apt update
sudo apt install liburing-dev
```

Le paquet installe les headers (`<liburing.h>`) et la librairie statique/dynamique. Vérifiez la version installée :

```bash
apt show liburing-dev | grep Version
```

Sur Ubuntu 24.04 LTS, le paquet fournit liburing 2.5+, ce qui est suffisant pour toutes les fonctionnalités courantes.

### Depuis les sources (dernière version)

Pour accéder aux toutes dernières fonctionnalités (notamment le support des opcodes récents) :

```bash
git clone https://github.com/axboe/liburing.git
cd liburing
./configure --prefix=/usr/local
make -j$(nproc)
sudo make install
```

### Intégration CMake

```cmake
# Recherche du paquet système
find_package(PkgConfig REQUIRED)
pkg_check_modules(URING REQUIRED liburing)

add_executable(my_server main.cpp)
target_link_libraries(my_server PRIVATE ${URING_LIBRARIES})
target_include_directories(my_server PRIVATE ${URING_INCLUDE_DIRS})
```

Alternative minimaliste si liburing est installée dans un chemin standard :

```cmake
add_executable(my_server main.cpp)
target_link_libraries(my_server PRIVATE uring)
```

```cmake

    liburing
```

## L'API liburing : vue d'ensemble

L'API suit un cycle en quatre phases qui correspond au flux décrit en section 22.3.3.1 :

```
    1. INITIALISER   io_uring_queue_init()
           │
           ▼                                          ───┐
    2. PRÉPARER      sqe = io_uring_get_sqe()            │
                     io_uring_prep_recv(sqe, ...)        │
                     io_uring_sqe_set_data(sqe, ctx)     │
           │                                             │
           ▼                                             │
    3. SOUMETTRE     io_uring_submit()                   │  boucle
       + ATTENDRE    io_uring_wait_cqe() / peek_cqe()    │
           │                                             │
           ▼                                             │
    4. RÉCOLTER      cqe->res, cqe->user_data            │
       + LIBÉRER     io_uring_cqe_seen()                 │
           │                                             │
           └──────── retour à l'étape 2 ─────────────────┘

    NETTOYAGE        io_uring_queue_exit()
```

## Référence des fonctions essentielles

### Cycle de vie de l'instance

```cpp
#include <liburing.h>

// Créer une instance avec `entries` SQE (arrondi à puissance de 2)
int io_uring_queue_init(unsigned entries,
                        struct io_uring *ring,
                        unsigned flags);

// Variante avec contrôle fin via io_uring_params
int io_uring_queue_init_params(unsigned entries,
                               struct io_uring *ring,
                               struct io_uring_params *params);

// Détruire l'instance (unmap + close)
void io_uring_queue_exit(struct io_uring *ring);
```

Le flag le plus courant à la création est `0` (mode normal). Pour le mode SQPOLL :

```cpp
io_uring_queue_init(256, &ring, IORING_SETUP_SQPOLL);
```

### Obtenir et préparer un SQE

```cpp
// Obtient un SQE libre. Retourne nullptr si la SQ est pleine.
struct io_uring_sqe *io_uring_get_sqe(struct io_uring *ring);
```

Une fois le SQE obtenu, on le prépare avec l'une des nombreuses fonctions `io_uring_prep_*`. Chacune correspond à un opcode et remplit les champs du SQE pour vous :

```cpp
// Réseau
void io_uring_prep_recv(struct io_uring_sqe *sqe,
                        int sockfd, void *buf, size_t len, int flags);

void io_uring_prep_send(struct io_uring_sqe *sqe,
                        int sockfd, const void *buf, size_t len, int flags);

void io_uring_prep_accept(struct io_uring_sqe *sqe,
                          int sockfd, struct sockaddr *addr,
                          socklen_t *addrlen, int flags);

void io_uring_prep_connect(struct io_uring_sqe *sqe,
                           int sockfd, const struct sockaddr *addr,
                           socklen_t addrlen);

// Fichier
void io_uring_prep_read(struct io_uring_sqe *sqe,
                        int fd, void *buf, unsigned nbytes, __u64 offset);

void io_uring_prep_write(struct io_uring_sqe *sqe,
                         int fd, const void *buf, unsigned nbytes, __u64 offset);

void io_uring_prep_fsync(struct io_uring_sqe *sqe, int fd, unsigned flags);

// Système
void io_uring_prep_openat(struct io_uring_sqe *sqe,
                          int dfd, const char *path, int flags, mode_t mode);

void io_uring_prep_close(struct io_uring_sqe *sqe, int fd);

// Contrôle
void io_uring_prep_timeout(struct io_uring_sqe *sqe,
                           struct __kernel_timespec *ts,
                           unsigned count, unsigned flags);

void io_uring_prep_cancel(struct io_uring_sqe *sqe,
                          __u64 user_data, int flags);
```

### Associer un contexte (user_data)

```cpp
// Associer un pointeur (ou un entier casté) comme donnée de corrélation
void io_uring_sqe_set_data(struct io_uring_sqe *sqe, void *data);

// Variante pour un entier 64 bits
void io_uring_sqe_set_data64(struct io_uring_sqe *sqe, __u64 data);

// Récupérer le contexte depuis un CQE
void *io_uring_cqe_get_data(const struct io_uring_cqe *cqe);
__u64 io_uring_cqe_get_data64(const struct io_uring_cqe *cqe);
```

### Soumettre

```cpp
// Soumet toutes les SQE en attente. Retourne le nombre soumis ou -errno.
int io_uring_submit(struct io_uring *ring);

// Soumet et attend au moins `wait_nr` complétions.
int io_uring_submit_and_wait(struct io_uring *ring, unsigned wait_nr);
```

### Attendre et récolter les CQE

```cpp
// Bloque jusqu'à ce qu'au moins 1 CQE soit disponible
int io_uring_wait_cqe(struct io_uring *ring,
                      struct io_uring_cqe **cqe_ptr);

// Bloque avec timeout
int io_uring_wait_cqe_timeout(struct io_uring *ring,
                              struct io_uring_cqe **cqe_ptr,
                              struct __kernel_timespec *ts);

// Non bloquant : retourne un CQE s'il y en a un, sinon -EAGAIN
int io_uring_peek_cqe(struct io_uring *ring,
                      struct io_uring_cqe **cqe_ptr);

// Récolter plusieurs CQE d'un coup
unsigned io_uring_peek_batch_cqe(struct io_uring *ring,
                                 struct io_uring_cqe **cqes,
                                 unsigned count);

// Marquer un CQE comme consommé (avance le head de la CQ)
void io_uring_cqe_seen(struct io_uring *ring,
                       const struct io_uring_cqe *cqe);

// Avancer le head de `count` CQE d'un coup
void io_uring_cq_advance(struct io_uring *ring, unsigned count);
```

> ⚠️ **Oublier d'appeler `io_uring_cqe_seen` (ou `io_uring_cq_advance`) est un bug classique.** Sans cela, le head de la CQ n'avance pas, la queue se remplit, et le noyau finit par ne plus pouvoir déposer de nouveaux résultats (CQ overflow).

## Exemple complet : serveur echo avec liburing

Voici un serveur echo fonctionnel qui utilise liburing pour gérer les connexions, les lectures et les écritures via `io_uring`. C'est l'équivalent des serveurs `select`, `poll` et `epoll` des sections précédentes.

```cpp
#include <liburing.h>
#include <sys/socket.h>
#include <netinet/in.h>
#include <unistd.h>
#include <fcntl.h>
#include <cstring>
#include <print>

// ── Identification du type d'opération via user_data ──
// On encode le type d'opération et le fd dans un uint64_t.
// Bits [63:56] = type, Bits [55:0] = fd ou index.

enum class OpType : uint8_t {
    Accept = 1,
    Recv   = 2,
    Send   = 3,
};

uint64_t encode_user_data(OpType type, int fd) {
    return (static_cast<uint64_t>(type) << 56) | static_cast<uint64_t>(fd);
}

OpType get_op_type(uint64_t user_data) {
    return static_cast<OpType>(user_data >> 56);
}

int get_fd(uint64_t user_data) {
    return static_cast<int>(user_data & 0x00FFFFFFFFFFFFFF);
}

// ── Buffers de lecture (un par connexion possible) ──
// Simplification : tableau statique indicé par fd.
constexpr int    max_conns = 4096;
constexpr size_t buf_size  = 4096;
char buffers[max_conns][buf_size];

// ── Helpers pour préparer les opérations ──

void prep_accept(io_uring *ring, int server_fd) {
    io_uring_sqe *sqe = io_uring_get_sqe(ring);
    io_uring_prep_accept(sqe, server_fd, nullptr, nullptr, 0);
    io_uring_sqe_set_data64(sqe, encode_user_data(OpType::Accept, server_fd));
}

void prep_recv(io_uring *ring, int client_fd) {
    io_uring_sqe *sqe = io_uring_get_sqe(ring);
    io_uring_prep_recv(sqe, client_fd, buffers[client_fd], buf_size, 0);
    io_uring_sqe_set_data64(sqe, encode_user_data(OpType::Recv, client_fd));
}

void prep_send(io_uring *ring, int client_fd, size_t len) {
    io_uring_sqe *sqe = io_uring_get_sqe(ring);
    io_uring_prep_send(sqe, client_fd, buffers[client_fd], len, 0);
    io_uring_sqe_set_data64(sqe, encode_user_data(OpType::Send, client_fd));
}

// ── Création du serveur (identique aux sections précédentes) ──

int create_server(uint16_t port) {
    int server_fd = socket(AF_INET, SOCK_STREAM, 0);
    if (server_fd == -1) return -1;

    int opt = 1;
    setsockopt(server_fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));

    sockaddr_in addr{};
    addr.sin_family      = AF_INET;
    addr.sin_addr.s_addr = INADDR_ANY;
    addr.sin_port        = htons(port);

    if (bind(server_fd, reinterpret_cast<sockaddr*>(&addr), sizeof(addr)) == -1) {
        close(server_fd);
        return -1;
    }
    if (listen(server_fd, SOMAXCONN) == -1) {
        close(server_fd);
        return -1;
    }
    return server_fd;
}

int main() {
    constexpr uint16_t port = 8080;
    int client_count = 0;

    int server_fd = create_server(port);
    if (server_fd == -1) {
        std::print(stderr, "Erreur création serveur\n");
        return 1;
    }

    // ── Initialisation d'io_uring ──
    io_uring ring{};
    if (int ret = io_uring_queue_init(256, &ring, 0); ret < 0) {
        std::print(stderr, "io_uring_queue_init : {}\n", strerror(-ret));
        close(server_fd);
        return 1;
    }

    std::print("Serveur io_uring en écoute sur le port {}\n", port);

    // ── Première soumission : accepter une connexion ──
    prep_accept(&ring, server_fd);
    io_uring_submit(&ring);

    // ── Boucle événementielle ──
    while (true) {
        io_uring_cqe *cqe = nullptr;
        int ret = io_uring_wait_cqe(&ring, &cqe);
        if (ret < 0) {
            if (ret == -EINTR) continue;
            std::print(stderr, "io_uring_wait_cqe : {}\n", strerror(-ret));
            break;
        }

        uint64_t ud  = io_uring_cqe_get_data64(cqe);
        OpType   op  = get_op_type(ud);
        int      fd  = get_fd(ud);
        int      res = cqe->res;

        // Marquer le CQE comme consommé IMMÉDIATEMENT
        io_uring_cqe_seen(&ring, cqe);

        switch (op) {

        case OpType::Accept: {
            // Toujours resoumettre un accept pour la prochaine connexion
            prep_accept(&ring, server_fd);

            if (res < 0) {
                std::print(stderr, "accept : {}\n", strerror(-res));
                break;
            }

            int client_fd = res;  // res contient le nouveau fd
            ++client_count;
            std::print("Nouveau client fd={} (total: {})\n",
                       client_fd, client_count);

            // Lancer une lecture sur le nouveau client
            prep_recv(&ring, client_fd);
            break;
        }

        case OpType::Recv: {
            if (res <= 0) {
                // res == 0 : déconnexion propre
                // res <  0 : erreur
                if (res < 0) {
                    std::print("Client fd={} erreur recv : {}\n",
                               fd, strerror(-res));
                } else {
                    std::print("Client fd={} déconnecté\n", fd);
                }
                close(fd);
                --client_count;
                break;
            }

            // Echo : envoyer les données reçues
            prep_send(&ring, fd, res);
            break;
        }

        case OpType::Send: {
            if (res < 0) {
                std::print("Client fd={} erreur send : {}\n",
                           fd, strerror(-res));
                close(fd);
                --client_count;
                break;
            }

            // Envoi terminé : relancer une lecture
            prep_recv(&ring, fd);
            break;
        }

        }  // switch

        // Soumettre toutes les SQE préparées dans cette itération
        io_uring_submit(&ring);
    }

    io_uring_queue_exit(&ring);
    close(server_fd);
    return 0;
}
```

### Analyse du design

Plusieurs choix de conception méritent d'être explicités :

**L'encodage dans `user_data`.** Contrairement à `epoll` où un événement concerne un descripteur et on sait implicitement qu'il est « prêt en lecture », avec `io_uring` on soumet des *opérations spécifiques*. Le CQE retourné ne porte pas le type d'opération — il ne contient que `user_data` et `res`. C'est donc à l'application de maintenir la corrélation. L'encodage type+fd dans un `uint64_t` est un pattern simple et efficace. Pour des applications plus complexes, on utilise `io_uring_sqe_set_data` avec un pointeur vers une structure de contexte allouée dynamiquement.

**La chaîne accept → recv → send → recv → ...** Le serveur forme un cycle par connexion : après chaque `send` terminé, on resoumet un `recv`. Après chaque `recv` terminé, on soumet un `send`. L'accept est resoumis immédiatement pour être prêt pour la prochaine connexion. Il n'y a pas de boucle événementielle « classique » avec reconstruction d'état — chaque CQE déclenche directement l'opération suivante.

**Pas de mode non bloquant explicite.** Contrairement à `epoll`, où les sockets doivent être configurées en `O_NONBLOCK`, avec `io_uring` c'est optionnel. Les opérations sont intrinsèquement asynchrones — c'est le noyau qui gère le blocage en interne (via des worker threads si nécessaire). Cela dit, configurer `O_NONBLOCK` reste une bonne pratique si le code peut aussi être utilisé en dehors d'`io_uring`.

**Un seul `io_uring_submit` par itération.** On accumule toutes les SQE préparées dans le traitement d'un CQE, puis on soumet en lot. Si le traitement d'un `Accept` prépare un `Accept` + un `Recv` (2 SQE), les deux sont soumis ensemble en un seul appel système.

## Récolte par lot : traiter plusieurs CQE par itération

L'exemple précédent traite un seul CQE par itération de boucle. Sous forte charge, il est plus efficace de drainer tous les CQE disponibles à chaque tour :

```cpp
while (true) {
    // Attendre au moins 1 CQE
    io_uring_cqe *cqe = nullptr;
    int ret = io_uring_wait_cqe(&ring, &cqe);
    if (ret < 0) {
        if (ret == -EINTR) continue;
        break;
    }

    // Récolter tous les CQE disponibles (batch)
    unsigned head;
    unsigned count = 0;
    io_uring_for_each_cqe(&ring, head, cqe) {
        // Traiter le CQE (même switch que précédemment)
        handle_cqe(&ring, cqe);
        ++count;
    }

    // Avancer le head de la CQ en une seule opération
    io_uring_cq_advance(&ring, count);

    // Soumettre toutes les SQE accumulées pendant le traitement
    io_uring_submit(&ring);
}
```

La macro `io_uring_for_each_cqe` itère sur tous les CQE disponibles sans les consommer. L'appel unique à `io_uring_cq_advance` en fin de boucle est plus efficace que `count` appels individuels à `io_uring_cqe_seen`, car il ne met à jour le compteur atomique qu'une seule fois.

> ⚠️ Pendant l'itération avec `io_uring_for_each_cqe`, les CQE ne sont pas encore libérés. Si les opérations que vous préparez dans `handle_cqe` remplissent la SQ au point que le noyau ne peut plus déposer de CQE, vous risquez un interblocage. En pratique, dimensionnez la CQ suffisamment grande (au moins 2× la SQ, ce qui est le défaut).

## Pattern avancé : multishot accept

Plutôt que de resoumettre un `accept` après chaque connexion acceptée, on peut utiliser l'accept **multishot** (Linux 5.18+). Une seule SQE reste active et produit un CQE pour chaque nouvelle connexion :

```cpp
void prep_multishot_accept(io_uring *ring, int server_fd) {
    io_uring_sqe *sqe = io_uring_get_sqe(ring);
    io_uring_prep_multishot_accept(sqe, server_fd, nullptr, nullptr, 0);
    io_uring_sqe_set_data64(sqe, encode_user_data(OpType::Accept, server_fd));
}
```

Dans le traitement du CQE, on vérifie le flag `IORING_CQE_F_MORE` :

```cpp
case OpType::Accept: {
    if (res < 0) {
        std::print(stderr, "accept : {}\n", strerror(-res));
        // Si IORING_CQE_F_MORE n'est PAS présent, le multishot s'est arrêté :
        // il faut le resoumettre.
        if (!(cqe->flags & IORING_CQE_F_MORE)) {
            prep_multishot_accept(&ring, server_fd);
        }
        break;
    }

    int client_fd = res;
    ++client_count;
    std::print("Nouveau client fd={} (total: {})\n", client_fd, client_count);
    prep_recv(&ring, client_fd);

    // Pas besoin de resoumettre l'accept :
    // IORING_CQE_F_MORE indique que le multishot est toujours actif.
    break;
}
```

L'avantage est double : une seule SQE consommée dans la SQ pour un nombre illimité d'acceptations, et aucun appel `io_uring_submit` supplémentaire pour relancer l'accept.

## Gestion des erreurs : les codes de retour liburing

Les fonctions liburing suivent deux conventions selon la catégorie :

**Fonctions de préparation** (`io_uring_prep_*`, `io_uring_sqe_set_data`) : elles ne retournent rien (`void`). Elles ne peuvent pas échouer car elles ne font qu'écrire dans un SQE déjà alloué.

**Fonctions d'interaction avec le noyau** (`io_uring_queue_init`, `io_uring_submit`, `io_uring_wait_cqe`) : elles retournent 0 en cas de succès ou un **`-errno` négatif** en cas d'erreur. C'est la convention noyau Linux, différente de la convention POSIX (qui retourne -1 et positionne `errno`).

```cpp
// Convention liburing : le code d'erreur est dans la valeur de retour
int ret = io_uring_submit(&ring);
if (ret < 0) {
    std::print(stderr, "submit : {}\n", strerror(-ret));  // Notez le -ret
}

// Convention identique pour les résultats dans les CQE
if (cqe->res < 0) {
    std::print(stderr, "opération échouée : {}\n", strerror(-cqe->res));
}
```

**`io_uring_get_sqe`** retourne `nullptr` si la SQ est pleine. Ce cas peut survenir si vous préparez plus de SQE que la taille de la SQ sans soumettre entre-temps. La réponse est soit d'augmenter la taille du ring, soit de soumettre plus fréquemment :

```cpp
io_uring_sqe *sqe = io_uring_get_sqe(&ring);
if (!sqe) {
    // SQ pleine : soumettre les SQE en attente pour libérer de la place
    io_uring_submit(&ring);
    sqe = io_uring_get_sqe(&ring);
    if (!sqe) {
        // Toujours plein : le ring est trop petit pour ce workload
        std::print(stderr, "SQ pleine même après submit\n");
        return;
    }
}
```

## liburing et C++ : encapsulation RAII

liburing est une bibliothèque C. En C++ idiomatique, on souhaite une gestion RAII de l'instance `io_uring`. Voici un wrapper minimal :

```cpp
#include <liburing.h>
#include <stdexcept>
#include <cstring>
#include <utility>

class IoUring {
public:
    explicit IoUring(unsigned entries, unsigned flags = 0) {
        if (int ret = io_uring_queue_init(entries, &ring_, flags); ret < 0) {
            throw std::runtime_error{
                std::string{"io_uring_queue_init: "} + strerror(-ret)
            };
        }
    }

    ~IoUring() {
        io_uring_queue_exit(&ring_);
    }

    // Non copiable
    IoUring(const IoUring&)            = delete;
    IoUring& operator=(const IoUring&) = delete;

    // Déplaçable
    IoUring(IoUring&& other) noexcept : ring_(other.ring_) {
        other.ring_ = {};  // Empêcher le double-free dans le destructeur
    }

    IoUring& operator=(IoUring&& other) noexcept {
        if (this != &other) {
            io_uring_queue_exit(&ring_);
            ring_ = other.ring_;
            other.ring_ = {};
        }
        return *this;
    }

    // Accès direct pour compatibilité avec l'API C
    io_uring*       raw()       { return &ring_; }
    const io_uring* raw() const { return &ring_; }

    // Wrappers typés pour les opérations courantes
    io_uring_sqe* get_sqe() { return io_uring_get_sqe(&ring_); }

    int submit() { return io_uring_submit(&ring_); }

    int wait_cqe(io_uring_cqe** cqe) {
        return io_uring_wait_cqe(&ring_, cqe);
    }

    void cqe_seen(const io_uring_cqe* cqe) {
        io_uring_cqe_seen(&ring_, cqe);
    }

    void cq_advance(unsigned count) {
        io_uring_cq_advance(&ring_, count);
    }

private:
    io_uring ring_{};
};
```

Ce wrapper garantit que `io_uring_queue_exit` est toujours appelé, y compris en cas d'exception. Il reste mince — l'objectif n'est pas de ré-abstraire toute l'API, mais d'ajouter la sécurité RAII tout en gardant l'accès à `raw()` pour les fonctions `io_uring_prep_*` qui prennent un `io_uring_sqe*` directement.

Usage :

```cpp
int main() {
    IoUring ring{256};

    auto* sqe = ring.get_sqe();
    io_uring_prep_accept(sqe, server_fd, nullptr, nullptr, 0);
    io_uring_sqe_set_data64(sqe, encode_user_data(OpType::Accept, server_fd));

    ring.submit();

    io_uring_cqe* cqe = nullptr;
    ring.wait_cqe(&cqe);
    // ... traitement ...
    ring.cqe_seen(cqe);
}
// ring est automatiquement nettoyé ici
```
