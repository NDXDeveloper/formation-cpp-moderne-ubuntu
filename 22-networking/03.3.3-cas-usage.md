# 22.3.3.3 Cas d'usage : networking, fichiers, timeouts

## Introduction

Les deux sections précédentes ont posé l'architecture d'`io_uring` (SQ/CQ, SQE/CQE) et présenté l'API `liburing` avec un serveur echo minimaliste. Cette section approfondit trois familles de cas d'usage concrets qui illustrent les forces spécifiques d'`io_uring` par rapport à `epoll` : le networking avancé, les opérations fichier véritablement asynchrones, et la gestion intégrée des timeouts et annulations.

Chaque cas d'usage est accompagné de code complet et compilable utilisant `liburing`.

---

## Cas 1 : Networking — serveur TCP avec gestion de l'écriture partielle

Le serveur echo de la section précédente avait une simplification importante : il supposait que chaque `send` transmettait toutes les données en une seule opération. En réalité, le noyau peut n'envoyer qu'une partie des données si le buffer TCP est saturé (congestion réseau, client lent). Un serveur de production doit gérer les **écritures partielles** et maintenir un buffer d'envoi par connexion.

### Structure de contexte par connexion

```cpp
#include <liburing.h>
#include <sys/socket.h>
#include <netinet/in.h>
#include <unistd.h>
#include <cstring>
#include <memory>
#include <vector>
#include <print>

constexpr size_t buf_size = 4096;

enum class OpType : uint8_t {
    Accept = 1,
    Recv   = 2,
    Send   = 3,
};

struct Connection {
    int    fd;
    char   recv_buf[buf_size];

    // Buffer d'envoi pour gérer les écritures partielles
    std::vector<char> send_buf;
    size_t            send_offset = 0;

    bool is_sending() const { return send_offset < send_buf.size(); }

    // Données restant à envoyer
    const char* send_data() const { return send_buf.data() + send_offset; }
    size_t      send_remaining() const { return send_buf.size() - send_offset; }
};

// Encodage du type d'opération et d'un pointeur dans user_data.
// On utilise les bits de poids fort pour le type (les adresses
// userspace sur x86_64 n'utilisent que 48 bits en pratique).
uint64_t encode(OpType type, Connection* conn) {
    auto addr = reinterpret_cast<uint64_t>(conn);
    return (static_cast<uint64_t>(type) << 56) | (addr & 0x00FFFFFFFFFFFFFF);
}

OpType get_type(uint64_t ud) {
    return static_cast<OpType>(ud >> 56);
}

Connection* get_conn(uint64_t ud) {
    // Restaurer l'adresse canonique (extension de signe pour x86_64)
    uint64_t addr = ud & 0x00FFFFFFFFFFFFFF;
    if (addr & (1ULL << 47)) addr |= 0xFFFF000000000000;  // Adresses noyau
    return reinterpret_cast<Connection*>(addr);
}
```

### Boucle événementielle avec écritures partielles

```cpp
void prep_accept(io_uring* ring, int server_fd) {
    auto* sqe = io_uring_get_sqe(ring);
    io_uring_prep_accept(sqe, server_fd, nullptr, nullptr, 0);
    io_uring_sqe_set_data64(sqe, encode(OpType::Accept, nullptr));
}

void prep_recv(io_uring* ring, Connection* conn) {
    auto* sqe = io_uring_get_sqe(ring);
    io_uring_prep_recv(sqe, conn->fd, conn->recv_buf, buf_size, 0);
    io_uring_sqe_set_data64(sqe, encode(OpType::Recv, conn));
}

void prep_send(io_uring* ring, Connection* conn) {
    auto* sqe = io_uring_get_sqe(ring);
    io_uring_prep_send(sqe, conn->fd, conn->send_data(),
                       conn->send_remaining(), MSG_NOSIGNAL);
    io_uring_sqe_set_data64(sqe, encode(OpType::Send, conn));
}

void handle_cqe(io_uring* ring, io_uring_cqe* cqe, int server_fd) {
    uint64_t ud  = io_uring_cqe_get_data64(cqe);
    OpType   op  = get_type(ud);
    int      res = cqe->res;

    switch (op) {

    case OpType::Accept: {
        prep_accept(ring, server_fd);  // Toujours relancer

        if (res < 0) {
            std::print(stderr, "accept: {}\n", strerror(-res));
            break;
        }
        auto* conn = new Connection{.fd = res};
        std::print("Client fd={} connecté\n", conn->fd);
        prep_recv(ring, conn);
        break;
    }

    case OpType::Recv: {
        auto* conn = get_conn(ud);
        if (res <= 0) {
            std::print("Client fd={} déconnecté\n", conn->fd);
            close(conn->fd);
            delete conn;
            break;
        }

        // Copier les données reçues dans le buffer d'envoi
        conn->send_buf.assign(conn->recv_buf, conn->recv_buf + res);
        conn->send_offset = 0;
        prep_send(ring, conn);
        break;
    }

    case OpType::Send: {
        auto* conn = get_conn(ud);
        if (res < 0) {
            std::print(stderr, "send fd={}: {}\n", conn->fd, strerror(-res));
            close(conn->fd);
            delete conn;
            break;
        }

        conn->send_offset += res;

        if (conn->is_sending()) {
            // Écriture partielle : resoumettre pour le reste
            prep_send(ring, conn);
        } else {
            // Tout envoyé : relancer une lecture
            conn->send_buf.clear();
            conn->send_offset = 0;
            prep_recv(ring, conn);
        }
        break;
    }

    }  // switch
}
```

Le point clé est le traitement de `OpType::Send` : si `res` est inférieur à la quantité restante, on avance `send_offset` et on resoumet un `send` pour la suite. Ce pattern est plus simple qu'avec `epoll`, où il faut jongler entre `EPOLLIN` et `EPOLLOUT` dynamiquement — ici, on soumet explicitement l'opération suivante.

## Cas 2 : Opérations fichier asynchrones

C'est le domaine où `io_uring` offre l'avantage le plus marqué par rapport à `epoll`. Sous Linux, les I/O fichier bloquent presque toujours le thread appelant, même avec `O_NONBLOCK` — le mode non bloquant n'est pas supporté de manière fiable par les systèmes de fichiers (ext4, XFS, etc.). Avec `epoll`, la seule solution est de dédier des threads d'I/O fichier et de coordonner via des files de messages.

`io_uring` résout ce problème nativement : les opérations fichier sont soumises dans la SQ comme n'importe quelle autre opération, et le noyau les exécute via ses propres worker threads internes. L'application reste entièrement non bloquante.

### Lecture asynchrone de fichiers

```cpp
#include <liburing.h>
#include <fcntl.h>
#include <unistd.h>
#include <cstring>
#include <print>

constexpr size_t block_size = 64 * 1024;  // 64 Ko par lecture

struct ReadCtx {
    int    fd;
    char*  buffer;
    size_t offset;
    size_t total_read;
};

int main(int argc, char* argv[]) {
    if (argc < 2) {
        std::print(stderr, "Usage: {} <fichier>\n", argv[0]);
        return 1;
    }

    int fd = open(argv[1], O_RDONLY);
    if (fd == -1) {
        std::print(stderr, "open: {}\n", strerror(errno));
        return 1;
    }

    // Taille du fichier
    off_t file_size = lseek(fd, 0, SEEK_END);
    if (file_size <= 0) {
        std::print(stderr, "Fichier vide ou erreur lseek\n");
        close(fd);
        return 1;
    }

    io_uring ring{};
    if (int ret = io_uring_queue_init(64, &ring, 0); ret < 0) {
        std::print(stderr, "io_uring_queue_init: {}\n", strerror(-ret));
        close(fd);
        return 1;
    }

    // Soumettre toutes les lectures en parallèle par blocs
    size_t remaining = file_size;
    size_t offset    = 0;
    int    pending   = 0;

    // Allouer un buffer contigu pour tout le fichier
    auto* file_buf = new char[file_size];

    while (remaining > 0) {
        size_t chunk = std::min(remaining, block_size);

        auto* sqe = io_uring_get_sqe(&ring);
        if (!sqe) {
            // SQ pleine : soumettre ce qu'on a et attendre
            io_uring_submit(&ring);
            sqe = io_uring_get_sqe(&ring);
        }

        io_uring_prep_read(sqe, fd, file_buf + offset, chunk, offset);
        io_uring_sqe_set_data64(sqe, offset);  // Stocker l'offset comme contexte

        offset    += chunk;
        remaining -= chunk;
        ++pending;
    }

    std::print("Soumission de {} lectures asynchrones ({} octets)\n",
               pending, file_size);
    io_uring_submit(&ring);

    // Récolter toutes les complétions
    size_t total_read = 0;
    int    errors     = 0;

    while (pending > 0) {
        io_uring_cqe* cqe = nullptr;
        int ret = io_uring_wait_cqe(&ring, &cqe);
        if (ret < 0) {
            std::print(stderr, "wait_cqe: {}\n", strerror(-ret));
            break;
        }

        if (cqe->res < 0) {
            std::print(stderr, "read à l'offset {}: {}\n",
                       io_uring_cqe_get_data64(cqe), strerror(-cqe->res));
            ++errors;
        } else {
            total_read += cqe->res;
        }

        io_uring_cqe_seen(&ring, cqe);
        --pending;
    }

    std::print("Lecture terminée : {} octets lus, {} erreurs\n",
               total_read, errors);

    delete[] file_buf;
    io_uring_queue_exit(&ring);
    close(fd);
    return 0;
}
```

### Analyse des performances

L'intérêt de cette approche est que **toutes les lectures sont soumises avant même que la première ne se termine**. Le noyau peut les réordonner pour minimiser les déplacements de tête de disque (sur HDD) ou les paralléliser sur les canaux du SSD. Avec une approche synchrone classique (`read` en boucle), chaque lecture attend la fin de la précédente — il n'y a aucun parallélisme I/O.

Sur un SSD NVMe, la différence est spectaculaire pour les lectures dispersées (random reads). Sur des lectures séquentielles d'un gros fichier, le gain est plus modeste car le readahead du noyau compense partiellement.

### Copie de fichier asynchrone avec linked SQE

Un cas d'usage naturel : copier un fichier bloc par bloc, en chaînant lecture et écriture :

```cpp
void submit_copy_block(io_uring* ring, int src_fd, int dst_fd,
                       char* buffer, size_t size, off_t offset) {
    // SQE 1 : lire depuis src
    auto* sqe_read = io_uring_get_sqe(ring);
    io_uring_prep_read(sqe_read, src_fd, buffer, size, offset);
    io_uring_sqe_set_data64(sqe_read, offset);
    sqe_read->flags |= IOSQE_IO_LINK;  // Chaîner avec le SQE suivant

    // SQE 2 : écrire vers dst (exécuté SEULEMENT si la lecture réussit)
    auto* sqe_write = io_uring_get_sqe(ring);
    io_uring_prep_write(sqe_write, dst_fd, buffer, size, offset);
    io_uring_sqe_set_data64(sqe_write, offset);
    // Pas de IOSQE_IO_LINK : fin de la chaîne
}
```

Chaque paire read→write forme une chaîne atomique. Si la lecture échoue, l'écriture est automatiquement annulée (`-ECANCELED`). On peut soumettre plusieurs dizaines de ces paires en parallèle pour des offsets différents, saturant ainsi la bande passante du device.

> ⚠️ Chaque SQE dans une chaîne génère son propre CQE. Pour une chaîne read→write, vous recevrez **deux CQE** : un pour le read, un pour le write (ou `-ECANCELED` si le read a échoué). Il faut traiter les deux.

## Cas 3 : Timeouts et annulations

La gestion des timeouts est un besoin universel en programmation réseau : annuler une opération si elle ne se termine pas dans un délai raisonnable. Avec `epoll`, cela nécessite des timers manuels (`timerfd_create`, `timer_settime`) enregistrés comme des descripteurs supplémentaires, ou des calculs de timeout dans `epoll_wait`. Avec `io_uring`, les timeouts sont des **opérations de première classe**.

### Timeout global sur epoll_wait-like

L'opcode `IORING_OP_TIMEOUT` attend soit qu'un délai expire, soit qu'un nombre donné d'autres CQE soient produits :

```cpp
// Timeout de 5 secondes : produit un CQE après 5s
// OU après que 10 autres CQE aient été générés (selon ce qui arrive en premier)
void prep_timeout(io_uring* ring, __kernel_timespec* ts, unsigned count) {
    auto* sqe = io_uring_get_sqe(ring);
    io_uring_prep_timeout(sqe, ts, count, 0);
    io_uring_sqe_set_data64(sqe, 0);  // Identifiant spécial pour le timeout
}

// Usage :
__kernel_timespec ts{.tv_sec = 5, .tv_nsec = 0};
prep_timeout(&ring, &ts, 0);  // count=0 : ne pas compter les CQE, juste le temps
io_uring_submit(&ring);

// Le CQE du timeout aura res == -ETIME si le délai a expiré,
// ou res == 0 si le count a été atteint avant.
```

### Timeout lié à une opération spécifique (linked timeout)

Le pattern le plus puissant : attacher un timeout directement à une opération via le chaînage `IOSQE_IO_LINK`. Si l'opération ne se termine pas dans le délai, elle est **automatiquement annulée** :

```cpp
void prep_recv_with_timeout(io_uring* ring, Connection* conn,
                            unsigned timeout_sec) {
    // SQE 1 : la réception
    auto* sqe_recv = io_uring_get_sqe(ring);
    io_uring_prep_recv(sqe_recv, conn->fd, conn->recv_buf, buf_size, 0);
    io_uring_sqe_set_data64(sqe_recv, encode(OpType::Recv, conn));
    sqe_recv->flags |= IOSQE_IO_LINK;  // Chaîner avec le timeout

    // SQE 2 : le timeout (annule le recv s'il expire avant)
    __kernel_timespec ts{.tv_sec = timeout_sec, .tv_nsec = 0};
    auto* sqe_timeout = io_uring_get_sqe(ring);
    io_uring_prep_link_timeout(sqe_timeout, &ts, 0);
    io_uring_sqe_set_data64(sqe_timeout, 0);  // Identifiant timeout
}
```

Le comportement est le suivant :

- **Le recv termine avant le timeout** : le CQE du recv contient les données normalement (`res > 0`). Le CQE du timeout contient `res == -ECANCELED` (le timeout a été annulé car le recv a réussi).  
- **Le timeout expire avant le recv** : le CQE du timeout contient `res == -ETIME`. Le CQE du recv contient `res == -ECANCELED` (le recv a été annulé par le timeout).

Dans les deux cas, vous recevez **exactement deux CQE**. Il faut les gérer tous les deux :

```cpp
// Dans le switch de traitement des CQE :
case OpType::Recv: {
    auto* conn = get_conn(ud);
    if (res == -ECANCELED) {
        // Le recv a été annulé par son linked timeout
        std::print("Client fd={} : timeout de réception\n", conn->fd);
        close(conn->fd);
        delete conn;
        break;
    }
    if (res <= 0) {
        // Déconnexion ou erreur classique
        close(conn->fd);
        delete conn;
        break;
    }
    // Données reçues normalement
    // ...
    break;
}
```

### Annulation explicite d'une opération en cours

Indépendamment des timeouts, on peut annuler n'importe quelle opération soumise mais pas encore complétée, en la ciblant par son `user_data` :

```cpp
void cancel_operation(io_uring* ring, uint64_t target_user_data) {
    auto* sqe = io_uring_get_sqe(ring);
    io_uring_prep_cancel64(sqe, target_user_data, 0);
    io_uring_sqe_set_data64(sqe, 0);  // CQE de l'annulation elle-même
}
```

Le CQE de l'annulation indique si elle a réussi (`res == 0`) ou si l'opération cible n'a pas été trouvée (`res == -ENOENT`, par exemple si elle s'est terminée entre-temps). L'opération annulée produit elle aussi un CQE avec `res == -ECANCELED`.

Cas d'usage typique : un client envoie un message d'annulation applicatif, et le serveur doit abandonner une lecture fichier en cours pour ce client.

## Cas 4 : Combiner networking et fichiers dans une même boucle

Le cas d'usage qui démontre le mieux la supériorité d'`io_uring` est un serveur qui mélange I/O réseau et I/O fichier — un serveur HTTP servant des fichiers statiques, un proxy avec cache disque, ou un serveur de base de données.

Avec `epoll`, cette combinaison nécessite une architecture complexe : un thread pool dédié aux I/O fichier, des files de messages pour les communications inter-threads, et une synchronisation soigneuse. Avec `io_uring`, tout passe par la même boucle événementielle.

### Serveur de fichiers minimaliste

Voici un exemple simplifié : un serveur TCP qui reçoit un nom de fichier, le lit de manière asynchrone, et renvoie son contenu.

```cpp
#include <liburing.h>
#include <sys/socket.h>
#include <netinet/in.h>
#include <fcntl.h>
#include <unistd.h>
#include <cstring>
#include <string>
#include <print>

constexpr size_t buf_size = 8192;

enum class OpType : uint8_t {
    Accept   = 1,
    Recv     = 2,   // Réception du nom de fichier
    Open     = 3,   // Ouverture du fichier
    Read     = 4,   // Lecture du fichier
    Send     = 5,   // Envoi du contenu
    Close    = 6,   // Fermeture du fichier
};

struct Request {
    int    client_fd;
    int    file_fd = -1;
    char   recv_buf[buf_size];
    char   file_buf[buf_size];
    size_t file_bytes = 0;
};

uint64_t encode(OpType type, Request* req) {
    return (static_cast<uint64_t>(type) << 56) |
           (reinterpret_cast<uint64_t>(req) & 0x00FFFFFFFFFFFFFF);
}

OpType   get_type(uint64_t ud) { return static_cast<OpType>(ud >> 56); }
Request* get_req(uint64_t ud) {
    uint64_t addr = ud & 0x00FFFFFFFFFFFFFF;
    return reinterpret_cast<Request*>(addr);
}

// ── Création du serveur (identique aux sections précédentes) ──

int create_server(uint16_t port) {
    int sfd = socket(AF_INET, SOCK_STREAM, 0);
    int opt = 1;
    setsockopt(sfd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));
    sockaddr_in addr{};
    addr.sin_family      = AF_INET;
    addr.sin_addr.s_addr = INADDR_ANY;
    addr.sin_port        = htons(port);
    bind(sfd, reinterpret_cast<sockaddr*>(&addr), sizeof(addr));
    listen(sfd, SOMAXCONN);
    return sfd;
}

int main() {
    constexpr uint16_t port = 8080;
    int server_fd = create_server(port);

    io_uring ring{};
    io_uring_queue_init(256, &ring, 0);

    // Premier accept
    auto* sqe = io_uring_get_sqe(&ring);
    io_uring_prep_accept(sqe, server_fd, nullptr, nullptr, 0);
    io_uring_sqe_set_data64(sqe, encode(OpType::Accept, nullptr));
    io_uring_submit(&ring);

    std::print("Serveur de fichiers io_uring sur le port {}\n", port);

    while (true) {
        io_uring_cqe* cqe = nullptr;
        if (io_uring_wait_cqe(&ring, &cqe) < 0) continue;

        uint64_t ud  = io_uring_cqe_get_data64(cqe);
        OpType   op  = get_type(ud);
        int      res = cqe->res;
        io_uring_cqe_seen(&ring, cqe);

        switch (op) {

        case OpType::Accept: {
            // Relancer l'accept
            auto* s = io_uring_get_sqe(&ring);
            io_uring_prep_accept(s, server_fd, nullptr, nullptr, 0);
            io_uring_sqe_set_data64(s, encode(OpType::Accept, nullptr));

            if (res >= 0) {
                auto* req = new Request{.client_fd = res};
                auto* s2 = io_uring_get_sqe(&ring);
                io_uring_prep_recv(s2, req->client_fd,
                                   req->recv_buf, buf_size, 0);
                io_uring_sqe_set_data64(s2, encode(OpType::Recv, req));
            }
            break;
        }

        case OpType::Recv: {
            auto* req = get_req(ud);
            if (res <= 0) {
                close(req->client_fd);
                delete req;
                break;
            }

            // Le client a envoyé un nom de fichier — ouvrir le fichier
            req->recv_buf[res] = '\0';
            // Retirer le newline éventuel
            if (auto* nl = strchr(req->recv_buf, '\n')) *nl = '\0';

            std::print("Client fd={} demande '{}'\n",
                       req->client_fd, req->recv_buf);

            auto* s = io_uring_get_sqe(&ring);
            io_uring_prep_openat(s, AT_FDCWD, req->recv_buf,
                                 O_RDONLY, 0);
            io_uring_sqe_set_data64(s, encode(OpType::Open, req));
            break;
        }

        case OpType::Open: {
            auto* req = get_req(ud);
            if (res < 0) {
                // Fichier introuvable : envoyer une erreur
                const char* err = "ERREUR: fichier introuvable\n";
                auto* s = io_uring_get_sqe(&ring);
                // Copier le message dans le buffer pour que le pointeur reste valide
                std::memcpy(req->file_buf, err, strlen(err));
                req->file_bytes = strlen(err);
                io_uring_prep_send(s, req->client_fd,
                                   req->file_buf, req->file_bytes,
                                   MSG_NOSIGNAL);
                io_uring_sqe_set_data64(s, encode(OpType::Send, req));
                break;
            }

            req->file_fd = res;

            // Lire le contenu du fichier (simplifié : un seul bloc)
            auto* s = io_uring_get_sqe(&ring);
            io_uring_prep_read(s, req->file_fd,
                               req->file_buf, buf_size, 0);
            io_uring_sqe_set_data64(s, encode(OpType::Read, req));
            break;
        }

        case OpType::Read: {
            auto* req = get_req(ud);
            if (res < 0) {
                std::print(stderr, "read file: {}\n", strerror(-res));
                close(req->file_fd);
                close(req->client_fd);
                delete req;
                break;
            }

            req->file_bytes = res;

            // Fermer le fichier de manière asynchrone (fire-and-forget)
            auto* s_close = io_uring_get_sqe(&ring);
            io_uring_prep_close(s_close, req->file_fd);
            io_uring_sqe_set_data64(s_close, encode(OpType::Close, req));

            // Envoyer le contenu au client
            auto* s_send = io_uring_get_sqe(&ring);
            io_uring_prep_send(s_send, req->client_fd,
                               req->file_buf, req->file_bytes,
                               MSG_NOSIGNAL);
            io_uring_sqe_set_data64(s_send, encode(OpType::Send, req));
            break;
        }

        case OpType::Send: {
            auto* req = get_req(ud);
            // Réponse envoyée : fermer la connexion client
            close(req->client_fd);
            delete req;
            break;
        }

        case OpType::Close: {
            // Fermeture du fichier terminée — rien à faire
            // (le Request peut déjà être supprimé par le Send)
            break;
        }

        }  // switch

        io_uring_submit(&ring);
    }

    io_uring_queue_exit(&ring);
    close(server_fd);
    return 0;
}
```

### Ce que cet exemple démontre

Le flux complet pour une requête traverse **cinq types d'opérations I/O** :

```
accept (réseau) → recv (réseau) → openat (fichier)
    → read (fichier) → send (réseau) + close (fichier)
```

Toutes ces opérations passent par la même boucle événementielle, la même SQ et la même CQ. Il n'y a aucun thread dédié aux I/O fichier, aucune file de messages inter-threads, aucun mutex. L'`openat` et le `read` asynchrones sont soumis exactement comme un `recv` ou un `send` — c'est l'unification promise par `io_uring`.

Avec `epoll`, les opérations `openat` et `read` sur un fichier auraient bloqué le thread de la boucle événementielle, gelant **toutes les autres connexions** pendant la durée de l'I/O disque. La solution classique (thread pool + file de messages) ajouterait typiquement 50 à 100 lignes de code de coordination et de synchronisation.

## Résumé des patterns io_uring

| Pattern | Opcodes impliqués | Avantage par rapport à epoll |
|---------|-------------------|-------------------------------|
| Serveur TCP avec écritures partielles | `accept`, `recv`, `send` | Soumission explicite, pas de gestion EPOLLOUT dynamique |
| Lecture fichier parallélisée | `read` × N | Vrai parallélisme I/O, pas de blocking |
| Copie fichier chaînée | `read` → `write` (linked) | Atomicité read→write sans code applicatif |
| Timeout par opération | `recv` → `link_timeout` (linked) | Intégré, pas de timerfd externe |
| Annulation explicite | `cancel` | Ciblage par user_data, pas de close forcé |
| Serveur mixte réseau + fichier | Tous | Une seule boucle, zéro thread pool |

Ces patterns se combinent naturellement dans des applications de production. La section 22.3.4 compare les performances et les compromis de tous les mécanismes de multiplexage présentés dans ce chapitre.
