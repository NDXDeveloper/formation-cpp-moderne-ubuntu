# 22.6.1 — Installation et configuration gRPC

## Section 22.6 : gRPC et Protocol Buffers

---

## Introduction

Avant d'écrire la moindre ligne de code gRPC, il faut installer et configurer une toolchain spécifique. Contrairement à une librairie classique qu'on inclut et qu'on link, gRPC nécessite un **compilateur de fichiers `.proto`** (`protoc`), un **plugin de génération C++** (`grpc_cpp_plugin`), et la **librairie runtime gRPC** elle-même. Le tout doit être correctement orchestré dans CMake pour que la génération de code s'intègre dans votre workflow de build.

Cette section couvre l'installation sur Ubuntu et la configuration CMake complète, des paquets système jusqu'à une configuration reproductible prête pour la CI/CD.

---

## La toolchain gRPC

Le build d'un projet gRPC en C++ implique plusieurs composants qui interagissent :

```
Votre code                         Outils                  Librairies
──────────                         ──────                  ──────────

service.proto ──► protoc ─────────► service.pb.h/.cc       libprotobuf
                     │               (messages Protobuf)
                     │
                     └─ grpc_cpp   ► service.grpc.pb.h/.cc libgrpc++
                        _plugin      (stubs RPC)           libgrpc
                                                           libabsl
server.cpp ────────────────────────────────────────────►   libgrpc++  
client.cpp ────────────────────────────────────────────►   libprotobuf  
```

**`protoc`** — Le compilateur Protocol Buffers. Il lit les fichiers `.proto` et génère le code de sérialisation/désérialisation (les fichiers `.pb.h` et `.pb.cc`).

**`grpc_cpp_plugin`** — Un plugin pour `protoc` qui génère le code spécifique gRPC : les stubs client, les classes de service serveur, et le code de marshalling RPC (les fichiers `.grpc.pb.h` et `.grpc.pb.cc`).

**`libprotobuf`** — La librairie runtime Protocol Buffers. Le code généré par `protoc` en dépend pour la sérialisation binaire, la réflexion, et les types well-known.

**`libgrpc++`** — La librairie runtime gRPC pour C++. Elle fournit les classes `Channel`, `Server`, `CompletionQueue`, et toute la mécanique de transport HTTP/2.

**`libabsl`** — Abseil, la librairie fondamentale de Google utilisée en interne par gRPC. C'est une dépendance transitive que vous n'utiliserez pas directement, mais qui doit être présente au link.

## Méthode 1 : Paquets système (Ubuntu)

C'est la méthode la plus rapide pour démarrer :

```bash
# Compilateur et plugin
sudo apt install protobuf-compiler protobuf-compiler-grpc

# Librairies de développement
sudo apt install libprotobuf-dev libgrpc++-dev libgrpc-dev

# Vérification des versions installées
protoc --version
# libprotoc 3.x.x ou 4.x.x selon la version d'Ubuntu

which grpc_cpp_plugin
# /usr/bin/grpc_cpp_plugin
```

### Vérification rapide

```bash
# Tester que protoc fonctionne avec le plugin gRPC
echo 'syntax = "proto3"; package test; service Svc { rpc Ping(Empty) returns (Empty); } message Empty {}' > /tmp/test.proto

protoc --cpp_out=/tmp --grpc_out=/tmp \
    --plugin=protoc-gen-grpc=$(which grpc_cpp_plugin) \
    /tmp/test.proto

# Vérifier que les fichiers ont été générés
ls /tmp/test.pb.h /tmp/test.pb.cc /tmp/test.grpc.pb.h /tmp/test.grpc.pb.cc
```

### Limites des paquets système

Les paquets Ubuntu ont un inconvénient : la version est figée pour chaque release d'Ubuntu et peut être en retard de plusieurs versions majeures par rapport aux releases officielles de gRPC. Si vous avez besoin de fonctionnalités récentes (nouvelles API asynchrones, support de fonctionnalités Protobuf récentes), préférez Conan ou vcpkg.

De plus, les versions de `protoc` et de `libprotobuf` doivent correspondre exactement. Mélanger un `protoc` système avec une `libprotobuf` installée via Conan provoquera des erreurs de link subtiles.

## Méthode 2 : Conan 2 (recommandé pour les projets)

Conan garantit la cohérence des versions et la reproductibilité du build :

```ini
# conanfile.txt
[requires]
grpc/1.67.1  
protobuf/5.28.3  

[generators]
CMakeDeps  
CMakeToolchain  

[tool_requires]
protobuf/5.28.3  
grpc/1.67.1  
```

La section `[tool_requires]` est importante : elle installe `protoc` et `grpc_cpp_plugin` comme outils de build, pas seulement les librairies. Cela garantit que les versions du compilateur et du runtime sont synchronisées.

```bash
conan install . --output-folder=build --build=missing
```

> 💡 La compilation de gRPC depuis les sources (ce que Conan fait au premier `--build=missing`) peut prendre **15 à 30 minutes** selon votre machine. C'est un build lourd qui compile Abseil, Protobuf, gRPC core, et le binding C++. Le résultat est mis en cache — les builds suivants sont instantanés.

---

## Méthode 3 : vcpkg

```bash
vcpkg install grpc protobuf

# vcpkg installe automatiquement les dépendances transitives (abseil, etc.)
```

Configuration CMake avec vcpkg :

```cmake
# Passer le toolchain vcpkg à CMake
cmake -B build -DCMAKE_TOOLCHAIN_FILE=$VCPKG_ROOT/scripts/buildsystems/vcpkg.cmake
```

---

## Méthode 4 : CMake FetchContent (compilation depuis les sources)

Pour un contrôle total, vous pouvez compiler gRPC directement dans votre projet :

```cmake
include(FetchContent)

FetchContent_Declare(
    grpc
    GIT_REPOSITORY https://github.com/grpc/grpc
    GIT_TAG        v1.67.1
    GIT_SHALLOW    TRUE
)

set(FETCHCONTENT_QUIET OFF)  
set(gRPC_BUILD_TESTS OFF CACHE BOOL "" FORCE)  
set(gRPC_INSTALL OFF CACHE BOOL "" FORCE)  

FetchContent_MakeAvailable(grpc)
```

> ⚠️ **Attention** : FetchContent pour gRPC télécharge et compile l'intégralité de gRPC et ses dépendances (Abseil, Protobuf, c-ares, re2, zlib). Le premier build peut dépasser **30 minutes** et consommer plusieurs GiB d'espace disque. Cette méthode est appropriée pour les projets qui ont besoin d'un build hermétique (CI, distribution), mais Conan ou vcpkg sont plus pragmatiques pour le développement quotidien.

## Configuration CMake complète

Quelle que soit la méthode d'installation choisie, la configuration CMake suit le même pattern. Voici un `CMakeLists.txt` complet et commenté pour un projet gRPC :

### Structure du projet

```
my_grpc_project/
├── CMakeLists.txt
├── proto/
│   └── greeter.proto
├── src/
│   ├── server.cpp
│   └── client.cpp
└── conanfile.txt          # (si Conan)
```

### CMakeLists.txt

```cmake
cmake_minimum_required(VERSION 3.20)  
project(my_grpc_project LANGUAGES CXX)  

set(CMAKE_CXX_STANDARD 20)  
set(CMAKE_CXX_STANDARD_REQUIRED ON)  

# ──────────────────────────────────────────────
# 1. Trouver les dépendances
# ──────────────────────────────────────────────
find_package(Threads REQUIRED)  
find_package(Protobuf CONFIG REQUIRED)  
find_package(gRPC CONFIG REQUIRED)  

# ──────────────────────────────────────────────
# 2. Localiser les exécutables protoc et plugin
# ──────────────────────────────────────────────
# Avec Conan/vcpkg, les targets importées fournissent les chemins
find_program(PROTOC_EXECUTABLE protoc
    HINTS ${Protobuf_PROTOC_EXECUTABLE}
          ${CONAN_BIN_DIRS_PROTOBUF}
)
find_program(GRPC_CPP_PLUGIN grpc_cpp_plugin
    HINTS ${CONAN_BIN_DIRS_GRPC}
)

if(NOT PROTOC_EXECUTABLE)
    message(FATAL_ERROR "protoc non trouvé")
endif()  
if(NOT GRPC_CPP_PLUGIN)  
    message(FATAL_ERROR "grpc_cpp_plugin non trouvé")
endif()

message(STATUS "protoc: ${PROTOC_EXECUTABLE}")  
message(STATUS "grpc_cpp_plugin: ${GRPC_CPP_PLUGIN}")  

# ──────────────────────────────────────────────
# 3. Générer le code depuis les fichiers .proto
# ──────────────────────────────────────────────
set(PROTO_SRC_DIR "${CMAKE_SOURCE_DIR}/proto")  
set(PROTO_GEN_DIR "${CMAKE_BINARY_DIR}/generated")  
file(MAKE_DIRECTORY ${PROTO_GEN_DIR})  

set(PROTO_FILE "${PROTO_SRC_DIR}/greeter.proto")

# Fichiers générés par protoc (Protobuf)
set(PB_GENERATED_SRCS "${PROTO_GEN_DIR}/greeter.pb.cc")  
set(PB_GENERATED_HDRS "${PROTO_GEN_DIR}/greeter.pb.h")  

# Fichiers générés par grpc_cpp_plugin (gRPC)
set(GRPC_GENERATED_SRCS "${PROTO_GEN_DIR}/greeter.grpc.pb.cc")  
set(GRPC_GENERATED_HDRS "${PROTO_GEN_DIR}/greeter.grpc.pb.h")  

# Commande custom pour la génération
add_custom_command(
    OUTPUT
        ${PB_GENERATED_SRCS} ${PB_GENERATED_HDRS}
        ${GRPC_GENERATED_SRCS} ${GRPC_GENERATED_HDRS}
    COMMAND ${PROTOC_EXECUTABLE}
        --proto_path=${PROTO_SRC_DIR}
        --cpp_out=${PROTO_GEN_DIR}
        --grpc_out=${PROTO_GEN_DIR}
        --plugin=protoc-gen-grpc=${GRPC_CPP_PLUGIN}
        ${PROTO_FILE}
    DEPENDS ${PROTO_FILE}
    COMMENT "Génération du code Protobuf/gRPC pour greeter.proto"
    VERBATIM
)

# ──────────────────────────────────────────────
# 4. Librairie avec le code généré
# ──────────────────────────────────────────────
add_library(greeter_proto
    ${PB_GENERATED_SRCS}
    ${GRPC_GENERATED_SRCS}
)

target_include_directories(greeter_proto PUBLIC ${PROTO_GEN_DIR})

target_link_libraries(greeter_proto PUBLIC
    protobuf::libprotobuf
    gRPC::grpc++
)

# Supprimer les warnings dans le code généré (hors de notre contrôle)
target_compile_options(greeter_proto PRIVATE -Wno-unused-parameter)

# ──────────────────────────────────────────────
# 5. Exécutables serveur et client
# ──────────────────────────────────────────────
add_executable(greeter_server src/server.cpp)  
target_link_libraries(greeter_server PRIVATE  
    greeter_proto
    Threads::Threads
)

add_executable(greeter_client src/client.cpp)  
target_link_libraries(greeter_client PRIVATE  
    greeter_proto
    Threads::Threads
)
```

### Analyse des choix

**`add_custom_command`** — C'est le mécanisme CMake pour exécuter `protoc` automatiquement quand les fichiers `.proto` changent. Le `DEPENDS ${PROTO_FILE}` garantit que la régénération se déclenche uniquement si le `.proto` a été modifié. Ninja (section 28.3) gère ces dépendances de manière particulièrement efficace.

**Librairie `greeter_proto`** — Le code généré est isolé dans une librairie dédiée. C'est une bonne pratique : le serveur et le client linkent contre la même librairie, et le code généré (potentiellement volumineux) n'est compilé qu'une seule fois.

**`-Wno-unused-parameter`** — Le code généré par `protoc` contient des paramètres non utilisés dans certaines fonctions. Plutôt que de désactiver ce warning globalement, on le supprime uniquement sur la target du code généré.

**`PUBLIC` sur les dépendances** — Les headers générés (`.pb.h`, `.grpc.pb.h`) incluent des headers de Protobuf et gRPC. Les targets qui linkent `greeter_proto` ont besoin de ces includes — d'où la propagation `PUBLIC`.

## Fonction CMake réutilisable

Pour les projets avec plusieurs fichiers `.proto`, encapsulez la génération dans une fonction :

```cmake
# grpc_generate.cmake

function(grpc_generate_cpp TARGET_NAME PROTO_FILE)
    get_filename_component(PROTO_NAME ${PROTO_FILE} NAME_WE)
    get_filename_component(PROTO_DIR ${PROTO_FILE} DIRECTORY)

    set(GEN_DIR "${CMAKE_BINARY_DIR}/generated")
    file(MAKE_DIRECTORY ${GEN_DIR})

    set(PB_SRC "${GEN_DIR}/${PROTO_NAME}.pb.cc")
    set(PB_HDR "${GEN_DIR}/${PROTO_NAME}.pb.h")
    set(GRPC_SRC "${GEN_DIR}/${PROTO_NAME}.grpc.pb.cc")
    set(GRPC_HDR "${GEN_DIR}/${PROTO_NAME}.grpc.pb.h")

    add_custom_command(
        OUTPUT ${PB_SRC} ${PB_HDR} ${GRPC_SRC} ${GRPC_HDR}
        COMMAND ${PROTOC_EXECUTABLE}
            --proto_path=${PROTO_DIR}
            --cpp_out=${GEN_DIR}
            --grpc_out=${GEN_DIR}
            --plugin=protoc-gen-grpc=${GRPC_CPP_PLUGIN}
            ${PROTO_FILE}
        DEPENDS ${PROTO_FILE}
        COMMENT "Generating gRPC code for ${PROTO_NAME}.proto"
        VERBATIM
    )

    add_library(${TARGET_NAME} ${PB_SRC} ${GRPC_SRC})
    target_include_directories(${TARGET_NAME} PUBLIC ${GEN_DIR})
    target_link_libraries(${TARGET_NAME} PUBLIC
        protobuf::libprotobuf
        gRPC::grpc++
    )
    target_compile_options(${TARGET_NAME} PRIVATE -Wno-unused-parameter)
endfunction()
```

Utilisation :

```cmake
include(grpc_generate.cmake)

grpc_generate_cpp(greeter_proto  proto/greeter.proto)  
grpc_generate_cpp(metrics_proto  proto/metrics.proto)  
grpc_generate_cpp(auth_proto     proto/auth.proto)  

add_executable(server src/server.cpp)  
target_link_libraries(server PRIVATE  
    greeter_proto
    metrics_proto
    auth_proto
)
```

## Imports et dépendances entre fichiers `.proto`

Dans les projets réels, les fichiers `.proto` s'importent mutuellement :

```protobuf
// common/types.proto
syntax = "proto3";  
package common;  

message Timestamp {
    int64 seconds = 1;
    int32 nanos = 2;
}
```

```protobuf
// services/greeter.proto
syntax = "proto3";  
package greeter;  

import "common/types.proto";

message HelloReply {
    string message = 1;
    common.Timestamp created_at = 2;
}
```

Pour que `protoc` résolve les imports, le `--proto_path` doit pointer vers la racine commune :

```cmake
# Structure :
# proto/
# ├── common/
# │   └── types.proto
# └── services/
#     └── greeter.proto

add_custom_command(
    OUTPUT ...
    COMMAND ${PROTOC_EXECUTABLE}
        --proto_path=${CMAKE_SOURCE_DIR}/proto    # Racine commune
        --cpp_out=${GEN_DIR}
        --grpc_out=${GEN_DIR}
        --plugin=protoc-gen-grpc=${GRPC_CPP_PLUGIN}
        ${CMAKE_SOURCE_DIR}/proto/services/greeter.proto
    DEPENDS
        ${CMAKE_SOURCE_DIR}/proto/services/greeter.proto
        ${CMAKE_SOURCE_DIR}/proto/common/types.proto  # Dépendance !
    ...
)
```

Le `DEPENDS` doit inclure **tous** les fichiers `.proto` importés, sinon la régénération ne se déclenchera pas quand un fichier importé change.

### Types well-known de Google

Protobuf fournit des types standard pour les cas courants :

```protobuf
import "google/protobuf/timestamp.proto";  
import "google/protobuf/duration.proto";  
import "google/protobuf/empty.proto";  
import "google/protobuf/wrappers.proto";  
import "google/protobuf/struct.proto";  

message Event {
    string name = 1;
    google.protobuf.Timestamp created_at = 2;
    google.protobuf.Duration ttl = 3;
}

service EventService {
    // google.protobuf.Empty pour les RPC sans requête ou sans réponse
    rpc Ping (google.protobuf.Empty) returns (google.protobuf.Empty);
}
```

Ces fichiers `.proto` sont inclus avec l'installation de Protobuf. `protoc` les trouve automatiquement dans son chemin d'include par défaut. Utilisez-les plutôt que de réinventer vos propres types pour les timestamps, durées, valeurs nullable, etc.

## Compilation et vérification

### Build complet

```bash
# Avec Conan
cd my_grpc_project  
conan install . --output-folder=build --build=missing  
cd build  
cmake .. -G Ninja \  
    -DCMAKE_TOOLCHAIN_FILE=conan_toolchain.cmake \
    -DCMAKE_BUILD_TYPE=Release
cmake --build .

# Avec les paquets système
cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Release  
cmake --build build  
```

### Vérifier les fichiers générés

```bash
ls build/generated/
# greeter.pb.h        — Messages Protobuf (classes C++)
# greeter.pb.cc       — Implémentation sérialisation/désérialisation
# greeter.grpc.pb.h   — Stubs client + classe de base serveur
# greeter.grpc.pb.cc  — Implémentation transport gRPC
```

### Structure du code généré

Pour un fichier `greeter.proto` contenant un service `Greeter` avec une méthode `SayHello`, le code généré contient :

**`greeter.pb.h`** — Les classes de messages (`HelloRequest`, `HelloReply`) avec leurs accesseurs, constructeurs, méthodes de sérialisation, et opérateurs de comparaison.

**`greeter.grpc.pb.h`** — Trois éléments clés :
- `Greeter::Service` — Classe de base abstraite côté serveur. Vous en héritez et implémentez les méthodes.
- `Greeter::Stub` — Classe client. Vous l'instanciez avec un channel et appelez les méthodes.
- `Greeter::AsyncService` — Variante asynchrone du service (pour les serveurs haute performance).

## Problèmes courants et diagnostic

### `protoc: not found` ou `grpc_cpp_plugin: not found`

```bash
# Vérifier les installations
which protoc  
which grpc_cpp_plugin  

# Si installé via Conan, les binaires sont dans le cache Conan
# Vérifier que conan_toolchain.cmake est bien utilisé
```

### Version mismatch entre `protoc` et `libprotobuf`

Le symptôme typique est une erreur de compilation ou de link dans les fichiers `.pb.cc` générés, du type :

```
error: 'InternalMetadata' is not a member of 'google::protobuf'
```

ou des erreurs de symboles non résolus au link. La cause est presque toujours un `protoc` d'une version différente de `libprotobuf`. La solution : s'assurer que les deux proviennent de la même source (même paquet système, même version Conan, même FetchContent).

```bash
# Vérifier la version de protoc
protoc --version

# Vérifier la version de libprotobuf
pkg-config --modversion protobuf
```

### Fichiers générés non trouvés

Si CMake ne trouve pas les fichiers `.pb.h` :

```bash
# Vérifier que la génération a eu lieu
ls build/generated/

# Vérifier que le include_directories est correct
# Le PROTO_GEN_DIR doit être dans les include paths de la target
```

### Erreurs de link avec Abseil

gRPC dépend d'Abseil (la librairie C++ interne de Google). Les erreurs de link mentionnant `absl::` sont typiques d'une installation incomplète :

```bash
# Avec apt
sudo apt install libabsl-dev

# Avec Conan/vcpkg, Abseil est installé automatiquement comme dépendance transitive
```

## Configuration pour la CI/CD

En environnement CI (GitHub Actions, GitLab CI), l'installation de gRPC doit être reproductible et rapide :

### GitHub Actions

```yaml
# .github/workflows/build.yml
name: Build

on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-24.04
    steps:
      - uses: actions/checkout@v4

      - name: Install dependencies
        run: |
          sudo apt-get update
          sudo apt-get install -y \
            protobuf-compiler protobuf-compiler-grpc \
            libprotobuf-dev libgrpc++-dev libabsl-dev \
            cmake ninja-build

      - name: Configure
        run: cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Release

      - name: Build
        run: cmake --build build

      - name: Test
        run: cd build && ctest --output-on-failure
```

### Avec Conan en CI (plus reproductible)

```yaml
      - name: Install Conan
        run: pip install conan

      - name: Configure Conan
        run: conan profile detect --force

      - name: Install dependencies
        run: conan install . --output-folder=build --build=missing

      - name: Build
        run: |
          cd build
          cmake .. -G Ninja \
            -DCMAKE_TOOLCHAIN_FILE=conan_toolchain.cmake \
            -DCMAKE_BUILD_TYPE=Release
          cmake --build .
```

Pour accélérer les builds CI, utilisez le cache des dépendances :

```yaml
      - name: Cache Conan packages
        uses: actions/cache@v4
        with:
          path: ~/.conan2
          key: conan-${{ hashFiles('conanfile.txt') }}
```

## Résumé

L'installation de gRPC en C++ est plus complexe que celle d'une librairie ordinaire en raison de l'étape de génération de code. Les points clés :

- **Quatre méthodes d'installation** : paquets système (rapide mais version figée), Conan (recommandé, reproductible), vcpkg (alternative), FetchContent (hermétique mais lent).
- **Cohérence des versions** : `protoc`, `grpc_cpp_plugin` et `libprotobuf` doivent provenir de la même source et être de versions compatibles.
- **`add_custom_command`** dans CMake pour déclencher `protoc` automatiquement quand les `.proto` changent.
- **Librairie dédiée** pour le code généré, avec propagation `PUBLIC` des dépendances Protobuf et gRPC.
- **Fonction CMake réutilisable** pour les projets multi-`.proto`.

Une fois la toolchain en place, elle fonctionne de manière transparente : vous modifiez un `.proto`, vous rebuildiez, et le code C++ est régénéré automatiquement.
