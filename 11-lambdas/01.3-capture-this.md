# 11.1.3 — Capture de `this`

## Le problème : accéder aux membres d'une classe depuis une lambda

Lorsqu'une lambda est définie à l'intérieur d'une méthode de classe, elle a naturellement besoin d'accéder aux membres de l'instance courante — attributs et méthodes. Mais les membres ne sont pas des variables locales de la méthode : ils vivent dans l'objet pointé par `this`. La lambda doit donc capturer `this` d'une manière ou d'une autre pour y accéder.

```cpp
class Sensor {
    std::string name_ = "temp_01";
    double value_ = 23.5;
public:
    auto create_reporter() {
        // Comment accéder à name_ et value_ ?
        return [/* ??? */]() {
            std::print("{}: {}\n", name_, value_);
        };
    }
};
```

C++ offre trois approches, introduites dans des standards successifs, chacune avec des caractéristiques de durée de vie différentes.

---

## `[this]` — Capture du pointeur (C++11)

La syntaxe `[this]` capture le **pointeur `this`** par valeur. La lambda stocke une copie du pointeur, ce qui lui donne accès à tous les membres de l'objet — exactement comme n'importe quelle méthode.

```cpp
class Sensor {
    std::string name_ = "temp_01";
    double value_ = 23.5;
public:
    auto create_reporter() {
        return [this]() {
            // Accès transparent aux membres via le pointeur this capturé
            std::print("{}: {}\n", name_, value_);
        };
    }

    void update(double v) { value_ = v; }
};
```

L'accès aux membres est **transparent** — on écrit `name_` et non `this->name_`, exactement comme dans une méthode ordinaire. Le compilateur résout implicitement les accès via le pointeur capturé.

### Équivalent foncteur

Pour comprendre ce que le compilateur génère :

```cpp
class __lambda {
    Sensor* __this;  // Pointeur vers l'objet
public:
    __lambda(Sensor* p) : __this(p) {}
    void operator()() const {
        std::print("{}: {}\n", __this->name_, __this->value_);
    }
};
```

La lambda ne possède pas les données — elle possède un **pointeur** vers un objet qui vit ailleurs. La conséquence directe : si l'objet est détruit avant l'appel de la lambda, le pointeur devient dangling.

### Mutation des membres

Puisque la lambda accède aux membres via un pointeur, elle peut les modifier sans avoir besoin du mot-clé `mutable` — c'est le même mécanisme que la capture par référence :

```cpp
class Counter {
    int count_ = 0;
public:
    auto make_incrementer() {
        return [this]() {
            ++count_;  // ✅ Modification directe du membre — pas besoin de mutable
            return count_;
        };
    }
};

Counter c;
auto inc = c.make_incrementer();
std::print("{}\n", inc());  // 1
std::print("{}\n", inc());  // 2
```

## Le danger de `[this]` : durée de vie de l'objet

`[this]` est un pointeur brut. Il ne participe à aucun mécanisme de gestion de durée de vie — pas de comptage de références, pas de vérification de validité. Si l'objet est détruit alors que la lambda existe encore, tout accès aux membres est un comportement indéfini.

### Scénario 1 : Objet détruit avant l'appel du callback

```cpp
class Button {
    std::string label_ = "OK";
public:
    auto on_click() {
        return [this]() {
            std::print("Clicked: {}\n", label_);
        };
    }
};

std::function<void()> callback;

{
    Button btn;
    callback = btn.on_click();  // Capture this → &btn
}  // btn est détruit ici

// callback();  // ⚠️ UNDEFINED BEHAVIOR : this pointe vers un objet détruit
```

### Scénario 2 : Objet déplacé après la capture

Un cas plus subtil survient lorsque l'objet est **déplacé** après la création de la lambda. Le déplacement ne met pas à jour le pointeur `this` capturé — il continue de pointer vers l'ancien emplacement :

```cpp
class Task {
    std::string name_ = "backup";
    std::function<void()> action_;
public:
    void setup() {
        action_ = [this]() {
            std::print("Running: {}\n", name_);
        };
    }

    void run() { if (action_) action_(); }
};

Task t1;
t1.setup();   // La lambda capture &t1

Task t2 = std::move(t1);  // t2 existe, mais la lambda pointe toujours vers t1
// t2.run();  // ⚠️ La lambda dans t2 accède aux membres de t1, pas de t2
```

### Scénario 3 : Callback asynchrone sur un objet temporaire

```cpp
class Downloader {
    std::string url_ = "https://example.com";
public:
    void start(std::vector<std::function<void()>>& queue) {
        queue.push_back([this]() {
            std::print("Downloading: {}\n", url_);
        });
    }
};

std::vector<std::function<void()>> task_queue;

{
    Downloader dl;
    dl.start(task_queue);
}  // dl détruit

// Exécution différée des tâches
for (auto& task : task_queue) {
    // task();  // ⚠️ UNDEFINED BEHAVIOR
}
```

## `[*this]` — Capture par copie de l'objet (C++17)

C++17 introduit la syntaxe `[*this]` qui capture une **copie complète** de l'objet `*this`. La lambda possède sa propre instance indépendante — elle ne dépend plus de la durée de vie de l'objet original.

```cpp
class Sensor {
    std::string name_ = "temp_01";
    double value_ = 23.5;
public:
    auto create_reporter() const {
        // Copie l'objet Sensor entier dans la lambda
        return [*this]() {
            std::print("{}: {}\n", name_, value_);
        };
    }
};
```

### Équivalent foncteur

```cpp
class __lambda {
    Sensor __this_copy;  // Copie complète de l'objet
public:
    __lambda(const Sensor& s) : __this_copy(s) {}
    void operator()() const {
        std::print("{}: {}\n", __this_copy.name_, __this_copy.value_);
    }
};
```

### Sécurité de durée de vie

La copie élimine le risque de dangling pointer — la lambda est autonome :

```cpp
std::function<void()> callback;

{
    Sensor s;
    callback = s.create_reporter();  // Copie de s dans la lambda
}  // s est détruit, mais la copie dans la lambda est intacte

callback();  // ✅ Affiche : temp_01: 23.5
```

### Indépendance de la copie

La copie est indépendante — les modifications de l'objet original ne sont pas visibles depuis la lambda, et inversement :

```cpp
class Config {
    int timeout_ = 30;
public:
    auto snapshot() const {
        return [*this]() { return timeout_; };
    }
    void set_timeout(int t) { timeout_ = t; }
};

Config cfg;
auto snap = cfg.snapshot();  // Copie avec timeout_ = 30

cfg.set_timeout(60);         // Modification de l'original

std::print("{}\n", snap());  // Affiche : 30 — la copie est indépendante
```

### Mutation de la copie avec `mutable`

Par défaut, l'`operator()` est `const`, donc les membres de la copie ne sont pas modifiables. Le mot-clé `mutable` lève cette restriction — il permet de modifier la copie interne, sans affecter l'objet original :

```cpp
class Counter {
    int count_ = 0;
public:
    auto make_independent_counter() {
        return [*this]() mutable {
            return ++count_;  // Modifie la copie interne
        };
    }
};

Counter c;
auto counter = c.make_independent_counter();
std::print("{}\n", counter());  // 1
std::print("{}\n", counter());  // 2
// c.count_ est toujours 0 — l'original n'est pas affecté
```

## `[this]` vs `[*this]` — Comparaison détaillée

| Aspect | `[this]` | `[*this]` |
|---|---|---|
| Standard | C++11 | C++17 |
| Ce qui est stocké | Pointeur vers l'objet (8 octets) | Copie complète de l'objet |
| Accès aux membres | Via indirection (pointeur) | Directement sur la copie |
| Modification des membres originaux | Oui | Non — la copie est indépendante |
| Dangling si l'objet est détruit | Oui — undefined behavior | Non — la copie survit |
| Coût mémoire | Minimal (1 pointeur) | Taille de l'objet entier |
| Coût de création | Négligeable | Coût du constructeur de copie |
| Objet non-copiable | Fonctionne | Erreur de compilation |
| Usage recommandé | Lambda locale, synchrone | Lambda stockée, asynchrone, retournée |

## Le piège de `[=]` dans les méthodes

Avant C++20, la capture par défaut `[=]` dans une méthode de classe avait un comportement surprenant : elle capturait **`this` par pointeur**, pas les membres par valeur. Le `=` donnait un faux sentiment de sécurité en suggérant que tout était copié :

```cpp
class Logger {
    std::string prefix_ = "[INFO]";
public:
    auto create_log_function() {
        // ⚠️ [=] capture this par pointeur, pas prefix_ par valeur !
        return [=]() {
            std::print("{} message\n", prefix_);
        };
    }
};

std::function<void()> log_fn;

{
    Logger logger;
    log_fn = logger.create_log_function();
}  // logger détruit

// log_fn();  // ⚠️ UNDEFINED BEHAVIOR — this est dangling malgré [=]
```

Le développeur lit `[=]` et pense "tout est copié, c'est sûr". En réalité, `prefix_` n'est pas une variable locale — c'est un membre accessible via `this`. La capture `[=]` capture `this` par pointeur, et l'accès à `prefix_` passe par ce pointeur dangling.

### Dépréciation en C++20

Le comité C++ a reconnu ce piège et a **déprécié la capture implicite de `this` via `[=]`** en C++20. Avec les compilateurs conformes, le code suivant émet un warning :

```cpp
auto create_log_function() {
    // Warning C++20 : implicit capture of 'this' via '[=]' is deprecated
    return [=]() {
        std::print("{} message\n", prefix_);
    };
}
```

Le fix consiste à rendre la capture de `this` explicite :

```cpp
auto create_log_function() {
    // ✅ Intention claire : on capture this par pointeur
    return [=, this]() {
        std::print("{} message\n", prefix_);
    };

    // ✅ Ou mieux : copie de l'objet pour la sécurité
    return [*this]() {
        std::print("{} message\n", prefix_);
    };
}
```

La syntaxe `[=, this]` est valide depuis C++20. Elle signifie "capture par défaut par valeur pour les variables locales, et capture `this` par pointeur explicitement". Elle ne change pas le comportement, mais elle documente l'intention.

## Capturer des membres individuels

Il est impossible de capturer un membre spécifique directement — la syntaxe `[member_]` ne compile pas car les membres ne sont pas des variables locales :

```cpp
class Processor {
    int threshold_ = 100;
    std::vector<double> data_;
public:
    auto make_filter() {
        // ERREUR : threshold_ n'est pas une variable locale
        // return [threshold_](double v) { return v > threshold_; };

        // ✅ Solution 1 : init capture — copie du membre dans une variable locale
        return [threshold = threshold_](double v) {
            return v > threshold;
        };
    }
};
```

L'init capture (C++14) est la solution idiomatique. Elle crée une variable locale `threshold` initialisée avec la valeur de `threshold_`, indépendante de l'objet :

```cpp
class Connection {
    std::string host_ = "localhost";
    int port_ = 8080;
    std::vector<char> buffer_;  // Gros — on ne veut pas le copier
public:
    auto make_reconnector() {
        // Copie uniquement host_ et port_ — pas le buffer_
        return [host = host_, port = port_]() {
            std::print("Reconnecting to {}:{}...\n", host, port);
        };
    }
};
```

Cette approche a un avantage majeur sur `[*this]` : elle permet de choisir **quels membres** copier, évitant le coût de la copie complète de l'objet quand seuls quelques champs sont nécessaires.

## `[this]` et les hiérarchies de classes

La capture de `this` dans une classe dérivée donne accès aux membres de toute la hiérarchie, y compris les membres protégés de la classe de base :

```cpp
class Base {
protected:
    int id_ = 0;
};

class Derived : public Base {
    std::string label_ = "default";
public:
    auto create_identifier() {
        return [this]() {
            // Accès aux membres de Base et Derived
            return std::format("{}:{}", id_, label_);
        };
    }
};
```

Le polymorphisme dynamique fonctionne également — si la lambda appelle une méthode virtuelle via `this`, c'est la version la plus dérivée qui est invoquée :

```cpp
class Shape {
public:
    virtual std::string name() const { return "Shape"; }

    auto make_printer() {
        return [this]() {
            // Appel virtuel — résolu dynamiquement
            std::print("I am a {}\n", name());
        };
    }

    virtual ~Shape() = default;
};

class Circle : public Shape {
public:
    std::string name() const override { return "Circle"; }
};

Circle c;
auto printer = c.make_printer();
printer();  // Affiche : I am a Circle — pas "I am a Shape"
```

Avec `[*this]`, en revanche, la copie est effectuée selon le **type statique** — si la méthode est appelée sur un objet de type `Shape` qui est en réalité un `Circle`, la copie via `[*this]` crée un objet `Shape`, pas `Circle`. C'est le problème classique du *slicing* :

```cpp
Shape& shape_ref = c;
// Si create_reporter utilisait [*this], la copie serait un Shape, pas un Circle
```

Pour cette raison, `[*this]` dans un contexte polymorphique nécessite de la prudence. Si le polymorphisme dynamique est important, `[this]` avec une gestion explicite de la durée de vie (via `std::shared_ptr` par exemple) est souvent préférable.

---

## Pattern sûr : `shared_from_this`

Pour les objets à durée de vie gérée par `std::shared_ptr`, le pattern `shared_from_this` permet de capturer un shared_ptr vers `this`, garantissant que l'objet reste vivant tant que la lambda existe :

```cpp
class Session : public std::enable_shared_from_this<Session> {
    std::string id_ = "sess_001";
public:
    auto create_handler() {
        // Capture un shared_ptr — l'objet ne sera pas détruit tant que la lambda vit
        auto self = shared_from_this();
        return [self]() {
            std::print("Handling session: {}\n", self->id_);
        };
    }
};

std::function<void()> handler;

{
    auto session = std::make_shared<Session>();
    handler = session->create_handler();
}  // Le shared_ptr local est détruit, mais la lambda en détient un autre

handler();  // ✅ Sûr — l'objet Session est toujours vivant
```

Ce pattern est omniprésent dans le code réseau asynchrone (Boost.Asio, gRPC) où les callbacks doivent maintenir l'objet en vie. La section 22.4 (Librairies réseau modernes) l'utilise intensivement.

Il faut cependant noter que `shared_from_this()` ne fonctionne que si l'objet est déjà géré par un `shared_ptr`. L'appeler sur un objet créé sur la stack ou via `new` sans shared_ptr provoque un comportement indéfini.

## Arbre de décision pour la capture de `this`

Le choix du mode de capture dépend principalement de deux facteurs : la durée de vie relative de la lambda et de l'objet, et le besoin de voir les mutations de l'objet.

**La lambda est locale et éphémère** (passée à un algorithme, exécutée dans la même méthode) : `[this]` est le choix naturel. La durée de vie est garantie et la lambda voit l'état le plus récent de l'objet.

```cpp
void Processor::run() {
    // ✅ Lambda éphémère — this est garanti vivant
    std::for_each(items_.begin(), items_.end(), [this](auto& item) {
        process_item(item);  // Appel d'une méthode privée
    });
}
```

**La lambda est retournée ou stockée, et n'a besoin que de quelques membres** : init capture des membres nécessaires. Pas de dépendance sur l'objet, pas de copie inutile.

```cpp
auto Sensor::create_alert() {
    // ✅ Copie sélective — indépendant de l'objet
    return [name = name_, threshold = threshold_](double value) {
        if (value > threshold) std::print("ALERT {}: {}\n", name, value);
    };
}
```

**La lambda est retournée ou stockée, et a besoin de tout l'objet** : `[*this]` si l'objet est copiable et de taille raisonnable.

```cpp
auto Config::snapshot_checker() const {
    // ✅ Copie complète — snapshot indépendant
    return [*this]() { return validate(); };
}
```

**La lambda est asynchrone et l'objet est géré par `shared_ptr`** : `shared_from_this()` pour garantir la durée de vie.

```cpp
auto Session::on_data_received() {
    auto self = shared_from_this();
    return [self](std::span<const char> data) {
        self->handle_data(data);
    };
}
```

## Récapitulatif

| Syntaxe | Standard | Ce qui est capturé | Durée de vie | Mutation de l'original | Coût |
|---|---|---|---|---|---|
| `[this]` | C++11 | Pointeur vers l'objet | Dangling si objet détruit | Oui | 8 octets |
| `[*this]` | C++17 | Copie de l'objet entier | Sûre — copie indépendante | Non | sizeof(objet) |
| `[=]` (dans méthode) | C++11 | `this` implicitement (pointeur) | Dangling si objet détruit | Oui | 8 octets |
| `[=, this]` | C++20 | `this` explicitement (pointeur) | Dangling si objet détruit | Oui | 8 octets |
| `[member = member_]` | C++14 | Copie d'un membre spécifique | Sûre — copie indépendante | Non | sizeof(membre) |
| `[self = shared_from_this()]` | C++11 | shared_ptr vers l'objet | Sûre — ref counting | Oui | ~16 octets |
