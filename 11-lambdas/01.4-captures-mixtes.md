# 11.1.4 — Captures mixtes et init captures

## Pourquoi aller au-delà des captures simples

Les sections précédentes ont couvert les captures par valeur et par référence, chacune avec ses forces et ses limites. En pratique, les lambdas du monde réel ont rarement besoin d'un mode de capture uniforme pour toutes leurs variables. Un callback réseau peut avoir besoin de **posséder** un buffer (capture par valeur ou déplacement), de **référencer** un compteur partagé (capture par référence), et de **figer** un identifiant de requête (capture par valeur d'un type léger) — le tout dans la même lambda.

C++ offre deux mécanismes complémentaires pour répondre à ce besoin : les **captures mixtes** (C++11), qui combinent différents modes dans une même clause, et les **init captures** (C++14), aussi appelées captures généralisées, qui permettent de créer de nouvelles variables directement dans la clause de capture.

---

## Captures mixtes (C++11)

### Principe

Une capture mixte combine une capture par défaut (`=` ou `&`) avec des exceptions explicites pour certaines variables. La capture par défaut s'applique à toutes les variables utilisées dans le corps, sauf celles listées explicitement avec un mode différent.

### Défaut par valeur, exceptions par référence

La syntaxe `[=, &var1, &var2]` capture tout par valeur sauf les variables nommées, capturées par référence :

```cpp
void process_batch(const std::vector<int>& data, int threshold) {
    int match_count = 0;
    int total_sum = 0;

    std::for_each(data.begin(), data.end(),
        [=, &match_count, &total_sum](int value) {
            //  threshold → copié (valeur)
            //  match_count → référence (modifiable)
            //  total_sum → référence (modifiable)
            if (value > threshold) {
                ++match_count;
                total_sum += value;
            }
        });

    std::print("Matches: {}, Sum: {}\n", match_count, total_sum);
}
```

Le pattern est clair : `threshold` est un paramètre en lecture seule (copié), tandis que `match_count` et `total_sum` sont des accumulateurs qui doivent être modifiés (référencés). C'est le pattern "paramètres par valeur, résultats par référence" mentionné en section 11.1.2.

### Défaut par référence, exceptions par valeur

La syntaxe `[&, var1, var2]` fait l'inverse — tout par référence sauf les variables nommées, capturées par valeur :

```cpp
void monitor_system(SystemStats& stats, Logger& logger) {
    const auto snapshot_time = std::chrono::steady_clock::now();

    auto check = [&, snapshot_time]() {
        // stats → référence (on veut l'état actuel)
        // logger → référence (on veut le logger actuel)
        // snapshot_time → copie (on fige le moment de la création)
        auto elapsed = std::chrono::steady_clock::now() - snapshot_time;
        logger.log("CPU: {}%, uptime: {}s",
                    stats.cpu_usage(),
                    std::chrono::duration_cast<std::chrono::seconds>(elapsed).count());
    };

    // ... utilisation de check ...
}
```

Ici, `snapshot_time` est figé au moment de la création (copie), tandis que `stats` et `logger` sont accédés par référence pour refléter l'état le plus récent.

### Combinaison avec `this`

Depuis C++20, on peut combiner la capture par défaut avec une capture explicite de `this` :

```cpp
class Pipeline {
    std::string name_ = "default";
    int stage_ = 0;
public:
    auto create_step(int step_id) {
        // Variables locales par valeur, this explicitement par pointeur
        return [=, this]() {
            std::print("[{}] Executing step {} at stage {}\n",
                       name_, step_id, stage_);
        };
    }
};
```

Et avec `*this` pour une copie de l'objet (disponible depuis C++17) :

```cpp
auto create_safe_step(int step_id) {
    // Variables locales par valeur, copie de l'objet
    return [=, *this]() {
        std::print("[{}] Step {} (snapshot at stage {})\n",
                   name_, step_id, stage_);
    };
}
```

### Règles de composition

Les règles syntaxiques pour les captures mixtes sont strictes. Le compilateur refuse toute ambiguïté ou redondance :

```cpp
int a = 1, b = 2, c = 3;

auto ok1 = [=, &a]() {};       // ✅ Tout par valeur, sauf a par référence
auto ok2 = [&, a]() {};        // ✅ Tout par référence, sauf a par valeur
auto ok3 = [=, &a, &b]() {};   // ✅ Multiples exceptions
auto ok4 = [&, a, b]() {};     // ✅ Multiples exceptions

// auto err1 = [=, a]() {};    // ❌ a est déjà capturé par = (redondant)
// auto err2 = [&, &a]() {};   // ❌ a est déjà capturé par & (redondant)
// auto err3 = [=, &]() {};    // ❌ Deux défauts
// auto err4 = [&a, =]() {};   // ❌ Le défaut doit être en première position
```

Un seul défaut est autorisé, et il doit apparaître en premier. Chaque exception doit utiliser le mode **opposé** au défaut.

## Init captures — captures généralisées (C++14)

### Le problème que les init captures résolvent

Avant C++14, la clause de capture ne pouvait référencer que des variables existantes dans la portée englobante. Trois limitations en découlaient :

Il était impossible de **déplacer** une ressource dans une lambda. Les types move-only comme `std::unique_ptr` ne pouvaient tout simplement pas être capturés :

```cpp
auto ptr = std::make_unique<int>(42);
// auto lambda = [ptr]() { return *ptr; };  // ❌ unique_ptr non copiable
// auto lambda = [&ptr]() { return *ptr; };  // ⚠️ Dangling si la lambda survit
```

Il était impossible de **renommer** une variable capturée ou de capturer le résultat d'une **expression** :

```cpp
// Pas de moyen de capturer x*2 ou de donner un nom différent
int x = 10;
// auto lambda = [double_x = x * 2]() { ... };  // Pas disponible en C++11
```

Il était impossible de capturer un membre de classe **individuellement** sans capturer `this` (comme vu en section 11.1.3).

### Syntaxe

Une init capture introduit une **nouvelle variable** dans la closure, initialisée par une expression arbitraire. La syntaxe est `nom = expression` dans la clause de capture :

```cpp
// Init capture par valeur
auto lambda = [value = some_expression]() {
    // value est une nouvelle variable locale à la closure
    return value;
};

// Init capture par référence
auto lambda2 = [&ref = some_lvalue]() {
    ref = 42;
};
```

La variable créée n'existe que dans la closure — elle est **indépendante** de toute variable de la portée englobante, même si l'expression d'initialisation en utilise une.

### Exemples fondamentaux

**Renommer une capture** pour améliorer la lisibilité :

```cpp
class DatabasePool {
    std::string connection_string_ = "postgres://localhost/mydb";
public:
    auto create_query_runner() {
        // Plus clair que d'accéder à connection_string_ via this
        return [conn_str = connection_string_]() {
            std::print("Connecting to: {}\n", conn_str);
        };
    }
};
```

**Capturer le résultat d'une expression** :

```cpp
int width = 800;
int height = 600;

auto report = [area = width * height, ratio = static_cast<double>(width) / height]() {
    std::print("Area: {}, Aspect ratio: {:.2f}\n", area, ratio);
};

report();  // Area: 480000, Aspect ratio: 1.33
```

Les expressions `width * height` et `static_cast<double>(width) / height` sont évaluées **une seule fois**, au moment de la création de la lambda. Les variables `area` et `ratio` vivent dans la closure.

**Capturer un index de boucle transformé** :

```cpp
std::vector<std::function<std::string()>> generators;

for (int i = 0; i < 5; ++i) {
    generators.push_back([label = "item_" + std::to_string(i)]() {
        return label;
    });
}

for (auto& gen : generators) {
    std::print("{}\n", gen());
}
// item_0, item_1, item_2, item_3, item_4
```

Chaque itération crée une nouvelle `std::string` `label`, initialisée avec la valeur courante de `i`. Chaque lambda possède sa propre copie indépendante.

## Init captures et `std::move` — capturer des types move-only

C'est le cas d'usage le plus important des init captures. Elles permettent de **transférer la propriété** d'une ressource dans une lambda via `std::move`, ce qui était impossible en C++11.

### `std::unique_ptr`

```cpp
auto ptr = std::make_unique<std::string>("critical data");

// Déplacement du unique_ptr dans la lambda
auto handler = [p = std::move(ptr)]() {
    std::print("Data: {}\n", *p);
};

// ptr est maintenant nullptr — la lambda possède la ressource
// std::print("{}\n", *ptr);  // ⚠️ Crash : ptr est null

handler();  // Data: critical data
```

Le mécanisme est le suivant : l'expression `std::move(ptr)` produit une rvalue reference, et la variable `p` dans la closure est initialisée par le **constructeur de déplacement** de `std::unique_ptr`. Le résultat est un transfert de propriété sans copie.

### Conteneurs volumineux

Le même mécanisme s'applique aux conteneurs pour éviter des copies coûteuses lorsque la donnée originale n'est plus nécessaire :

```cpp
std::vector<double> measurements(1'000'000);
// ... remplissage de measurements ...

// Transfert du vecteur dans la lambda — zéro copie
auto analyzer = [data = std::move(measurements)]() {
    double sum = 0.0;
    for (double v : data) sum += v;
    return sum / static_cast<double>(data.size());
};

// measurements est maintenant vide
std::print("Size after move: {}\n", measurements.size());  // 0
std::print("Average: {}\n", analyzer());
```

### Ressources système

Les types RAII wrappant des ressources système (file descriptors, sockets, handles) sont souvent move-only. Les init captures permettent de les transférer dans des lambdas-callbacks :

```cpp
class FileHandle {
    int fd_;
public:
    explicit FileHandle(const std::string& path);
    ~FileHandle();
    FileHandle(FileHandle&& other) noexcept;
    FileHandle& operator=(FileHandle&& other) noexcept;
    FileHandle(const FileHandle&) = delete;             // Non copiable
    FileHandle& operator=(const FileHandle&) = delete;

    void write(std::string_view data);
};

auto create_writer(const std::string& path) {
    FileHandle fh(path);
    // Transfert du handle dans la lambda
    return [file = std::move(fh)](std::string_view data) mutable {
        file.write(data);  // mutable car write() est non-const
    };
}

auto writer = create_writer("/tmp/output.log");
writer("Hello\n");
writer("World\n");
```

Le mot-clé `mutable` est nécessaire ici car `file.write()` est une opération non-const sur la ressource possédée.

## Init captures par référence

On peut également créer une init capture par référence en préfixant le nom avec `&`. La variable dans la closure est alors une référence vers l'objet désigné par l'expression :

```cpp
std::vector<int> data = {1, 2, 3, 4, 5};

// ref est une référence vers data — pas une copie
auto appender = [&ref = data](int value) {
    ref.push_back(value);
};

appender(6);
appender(7);
// data contient maintenant {1, 2, 3, 4, 5, 6, 7}
```

Le cas d'usage principal est de capturer une **référence const** vers un objet non-const, en utilisant `std::as_const` :

```cpp
std::map<std::string, int> scores = {{"Alice", 95}, {"Bob", 87}};

// Référence const — lecture seule garantie
auto reader = [&cscores = std::as_const(scores)](const std::string& name) {
    auto it = cscores.find(name);
    return it != cscores.end() ? it->second : -1;
};

std::print("Alice: {}\n", reader("Alice"));  // 95
```

## Combiner captures mixtes et init captures

La véritable puissance apparaît lorsqu'on combine tous les modes dans une même clause de capture. Chaque variable capturée utilise le mode le plus approprié :

```cpp
class TaskScheduler {
    std::string name_ = "main_scheduler";
    int max_retries_ = 3;
public:
    auto schedule_task(std::unique_ptr<Task> task, Logger& logger) {
        return [
            this,                            // Accès au scheduler (pointeur)
            task = std::move(task),           // Possession exclusive de la tâche
            &logger,                         // Référence vers le logger partagé
            created_at = std::chrono::steady_clock::now(),  // Timestamp figé
            attempt = 0                      // Compteur local initialisé à 0
        ]() mutable {
            ++attempt;
            logger.info("[{}] Attempt {}/{} for task",
                        name_, attempt, max_retries_);

            auto elapsed = std::chrono::steady_clock::now() - created_at;
            if (attempt <= max_retries_) {
                task->execute();
            }
        };
    }
};
```

Dans cette lambda, cinq modes de capture coexistent, chacun adapté à son rôle : `this` par pointeur pour les membres du scheduler, une init capture avec `std::move` pour la possession exclusive de la tâche, une référence vers le logger partagé, et deux init captures par valeur pour le timestamp et le compteur.

## Init captures dans les boucles : le pattern factory

Les init captures résolvent élégamment le problème des lambdas créées en boucle. Chaque itération produit une closure avec ses propres valeurs :

```cpp
std::vector<std::function<std::string()>> formatters;

std::vector<std::pair<std::string, double>> metrics = {
    {"cpu", 78.5}, {"memory", 62.3}, {"disk", 91.0}
};

for (const auto& [name, value] : metrics) {
    formatters.push_back(
        [label = name, val = value, severity = (value > 80.0 ? "HIGH" : "OK")]() {
            return std::format("[{}] {}: {:.1f}%", severity, label, val);
        });
}

for (auto& fmt : formatters) {
    std::print("{}\n", fmt());
}
// [OK] cpu: 78.5%
// [OK] memory: 62.3%
// [HIGH] disk: 91.0%
```

Chaque lambda possède ses propres copies de `label`, `val` et `severity`, calculées au moment de l'itération. Aucun risque de dangling reference, aucune confusion sur la valeur capturée.

## Init captures et durée de vie : subtilités

### La variable capturée est indépendante

Une init capture crée une **nouvelle variable** dans la closure. Elle ne crée pas d'alias vers la variable source — même si le nom est identique :

```cpp
std::string message = "original";

auto lambda = [message = message]() {  // Copie dans une NOUVELLE variable
    return message;
};

message = "modified";

std::print("{}\n", lambda());   // original — la copie est indépendante
std::print("{}\n", message);    // modified — l'original est inchangé
```

Cela semble identique à `[message]` et, dans ce cas, le résultat est le même. La différence se manifeste lorsqu'on transforme la valeur :

```cpp
std::string message = "hello";

auto lambda = [msg = message + " world"]() {
    return msg;
};
// message est toujours "hello"
// lambda() retourne "hello world"
```

### Évaluation unique de l'expression

L'expression d'initialisation est évaluée **une seule fois**, au moment de la création. Si l'expression a des effets de bord, ils ne se produisent qu'une fois :

```cpp
int call_count = 0;

auto get_next_id = [&call_count]() { return ++call_count; };

// L'init capture appelle get_next_id() UNE SEULE FOIS
auto lambda = [id = get_next_id()]() {
    return id;
};

std::print("{}\n", lambda());  // 1
std::print("{}\n", lambda());  // 1 — toujours la même valeur
std::print("{}\n", lambda());  // 1
```

## Init captures `constexpr` (C++17)

Les init captures peuvent participer à des lambdas `constexpr` si l'expression d'initialisation est elle-même évaluable à la compilation :

```cpp
constexpr auto make_multiplier(int factor) {
    return [f = factor](int x) { return x * f; };
}

constexpr auto double_it = make_multiplier(2);
static_assert(double_it(21) == 42);
```

Le compilateur évalue la création de la lambda, y compris l'init capture, au moment de la compilation. Le résultat est un calcul entièrement résolu sans aucun coût à l'exécution.

## Anti-patterns et pièges

### Capturer `this` et un membre par init capture simultanément

Capturer `this` donne accès à tous les membres. Capturer en plus un membre par init capture crée une copie **indépendante** qui peut diverger de l'original, ce qui est source de confusion :

```cpp
class Server {
    int port_ = 8080;
public:
    auto make_handler() {
        return [this, port = port_]() {
            // port est une copie figée — port_ est l'original via this
            // Lequel utiliser ? Le code est ambigu.
            std::print("Snapshot port: {}, Current port: {}\n", port, port_);
        };
    }

    void set_port(int p) { port_ = p; }
};

Server s;
auto h = s.make_handler();
s.set_port(9090);
h();  // Snapshot port: 8080, Current port: 9090
```

Ce n'est pas toujours un bug — parfois on veut comparer un état capturé avec l'état courant. Mais sans documentation claire, le code est difficile à maintenir. Préférez l'un ou l'autre mode, pas les deux pour le même champ.

### Déplacer et continuer à utiliser la source

Après un `std::move`, la variable source est dans un état valide mais indéterminé. L'utiliser est une erreur logique :

```cpp
auto buffer = std::make_unique<char[]>(4096);
char* raw = buffer.get();

auto handler = [buf = std::move(buffer)]() mutable {
    // utilise buf...
};

// ⚠️ raw pointe vers la mémoire possédée par la lambda
// Toute utilisation de raw est un alias dangereux
// buffer est nullptr — toute utilisation est un bug
```

La règle est simple : après un `std::move` dans une init capture, considérez la variable source comme **morte**. Ne la réutilisez pas.

### Init capture masquant une variable locale

Une init capture peut avoir le même nom qu'une variable dans la portée englobante. La variable dans la closure **masque** la variable externe. Ce n'est pas un bug, mais cela nuit à la lisibilité si les deux ont des valeurs différentes :

```cpp
int x = 10;
auto lambda = [x = x * 2]() { return x; };  // x dans la lambda vaut 20

std::print("{}\n", x);         // 10 — la variable locale
std::print("{}\n", lambda());  // 20 — la variable dans la closure
```

En cas de doute, utilisez un nom distinct pour la variable capturée :

```cpp
int x = 10;
auto lambda = [double_x = x * 2]() { return double_x; };  // Plus clair
```

## Résumé des patterns recommandés

| Besoin | Pattern | Exemple |
|---|---|---|
| Paramètres en lecture seule | Capture par valeur | `[threshold]` |
| Accumulateur local | Capture par référence | `[&count]` |
| Lecture seule d'un gros objet local | Référence const via init capture | `[&cref = std::as_const(obj)]` |
| Possession d'un type move-only | Init capture + `std::move` | `[p = std::move(ptr)]` |
| Transfert d'un conteneur | Init capture + `std::move` | `[data = std::move(vec)]` |
| Snapshot d'un membre de classe | Init capture depuis membre | `[name = name_]` |
| Valeur calculée | Init capture avec expression | `[area = w * h]` |
| Compteur interne | Init capture + `mutable` | `[n = 0]() mutable { ++n; }` |
| Combinaison multi-mode | Captures mixtes | `[=, &logger, task = std::move(t)]` |

---

## Récapitulatif

| Fonctionnalité | Standard | Description |
|---|---|---|
| Captures mixtes `[=, &x]` | C++11 | Défaut par valeur avec exceptions par référence |
| Captures mixtes `[&, x]` | C++11 | Défaut par référence avec exceptions par valeur |
| Init capture `[x = expr]` | C++14 | Nouvelle variable dans la closure, initialisée par une expression |
| Init capture `[&x = lvalue]` | C++14 | Référence vers un objet existant, avec renommage possible |
| Init capture + `std::move` | C++14 | Transfert de propriété de types move-only |
| `[=, this]` | C++20 | Capture par défaut valeur + `this` explicite |
| `[=, *this]` | C++17 | Capture par défaut valeur + copie de l'objet |
