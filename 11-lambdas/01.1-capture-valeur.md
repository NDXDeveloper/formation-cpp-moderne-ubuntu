# 11.1.1 — Capture par valeur `[=]`

## Principe

La capture par valeur crée une **copie** de la variable capturée à l'intérieur de la lambda. Cette copie est un membre de l'objet closure généré par le compilateur. La lambda possède donc sa propre version indépendante de la donnée — toute modification de la variable originale après la création de la lambda n'a aucun effet sur la copie interne, et inversement.

```cpp
int x = 10;  
auto snapshot = [x]() { return x; };  

x = 999;  
std::print("{}\n", snapshot());  // Affiche : 10 — la copie est indépendante  
```

---

## Capture explicite vs capture par défaut

Il existe deux façons de capturer par valeur : nommer explicitement chaque variable, ou utiliser le défaut `=` qui capture automatiquement toutes les variables utilisées dans le corps de la lambda.

### Capture explicite

On liste chaque variable individuellement dans la clause de capture :

```cpp
int width = 800;  
int height = 600;  

auto area = [width, height]() {
    return width * height;
};
```

Seules les variables nommées sont capturées. Toute tentative d'utiliser une variable non listée dans le corps de la lambda provoque une erreur de compilation :

```cpp
int depth = 32;

auto area = [width, height]() {
    // return width * height * depth;  // ERREUR : depth non capturé
    return width * height;
};
```

C'est la forme la plus **explicite** et la plus **lisible** — en lisant la clause de capture, on sait immédiatement de quelles données la lambda dépend.

### Capture par défaut `[=]`

Le symbole `=` comme capture par défaut demande au compilateur de capturer par valeur **toutes les variables utilisées** dans le corps de la lambda :

```cpp
int width = 800;  
int height = 600;  
int depth = 32;  

auto volume = [=]() {
    return width * height * depth;  // Les trois sont capturés automatiquement
};
```

Le compilateur ne capture que les variables **effectivement référencées** dans le corps — les autres ne sont pas copiées. Ainsi, si `depth` n'apparaît pas dans le corps, il n'est pas capturé, même avec `[=]`.

---

## Le `const` implicite de `operator()`

Les captures par valeur sont accessibles en **lecture seule** par défaut dans le corps de la lambda. Techniquement, l'`operator()` généré par le compilateur est qualifié `const`, ce qui interdit la modification des membres de la closure :

```cpp
int counter = 0;

// ERREUR de compilation : counter est const dans le corps
// auto inc = [counter]() { return ++counter; };
```

Pour comprendre pourquoi, rappelons l'équivalent foncteur :

```cpp
class __lambda {
    const int counter;  // Membre effectivement const (operator() est const)
public:
    __lambda(int c) : counter(c) {}
    int operator()() const {    // ← const ici
        return ++counter;       // ERREUR : modification d'un membre dans un contexte const
    }
};
```

Ce choix de design est délibéré : il encourage l'écriture de lambdas sans effets de bord, conformément aux principes de programmation fonctionnelle. Si vous avez besoin de modifier une capture par valeur, utilisez `mutable` (voir section 11.1, paragraphe dédié) — mais questionnez d'abord si une capture par référence ne serait pas plus appropriée.

---

## Quand la copie a-t-elle lieu ?

La copie se produit **au moment où l'expression lambda est évaluée** — c'est-à-dire au moment où la ligne de code contenant la lambda est exécutée, pas au moment où la lambda est appelée. C'est un piège fréquent dans les boucles :

```cpp
std::vector<std::function<int()>> functions;

for (int i = 0; i < 5; ++i) {
    // La copie de i a lieu à CHAQUE itération — chaque lambda a sa propre valeur
    functions.push_back([i]() { return i; });
}

for (auto& fn : functions) {
    std::print("{} ", fn());
}
// Affiche : 0 1 2 3 4 ✅
```

Chaque itération crée une nouvelle lambda avec une copie distincte de `i`. Comparons avec la capture par référence qui produirait un résultat très différent (et dangereux — voir section 11.1.2).

Un second piège concerne les variables modifiées **entre la création de la lambda et son appel** :

```cpp
std::string name = "Alice";  
auto greet = [name]() { std::print("Hello, {}!\n", name); };  

name = "Bob";       // Modification après la capture  
name.clear();       // Même destruction du contenu  

greet();            // Affiche : Hello, Alice!  — la copie est intacte
```

La lambda détient un `std::string` indépendant dont le contenu est `"Alice"` — les manipulations ultérieures de `name` n'ont aucun impact.

---

## Coût de la copie

Capturer par valeur implique une **copie**. Pour les types primitifs (`int`, `double`, `bool`, pointeurs bruts), le coût est négligeable. Pour des types plus lourds, la copie peut avoir un impact significatif :

```cpp
std::vector<int> large_data(1'000'000, 42);

// ⚠️ Copie d'un vector d'un million d'éléments !
auto process = [large_data]() {
    return large_data.size();
};
```

Chaque création de cette lambda copie le vecteur entier. Si la lambda est créée dans une boucle ou passée à de multiples threads, les copies se multiplient.

### Stratégies pour éviter les copies coûteuses

**Capture par référence constante** quand la lambda est utilisée localement et que la donnée ne doit pas être modifiée — mais attention, la syntaxe `[&large_data]` capture par référence non-const. Pour une sémantique const, on combine avec une init capture :

```cpp
// Capture par référence const via init capture
auto process = [&data = std::as_const(large_data)]() {
    return data.size();
    // data.push_back(1);  // ERREUR : data est const
};
```

**Init capture avec `std::move`** (C++14) quand la lambda doit posséder la donnée sans la copier — le transfert de propriété est traité en détail en section 11.1.4 :

```cpp
std::vector<int> large_data(1'000'000, 42);

// Déplacement : pas de copie, large_data est vidé
auto process = [data = std::move(large_data)]() {
    return data.size();
};
// large_data est maintenant vide
```

**Capture par référence simple** `[&]` quand la lambda est éphémère (passée à un algorithme STL, utilisée dans la même portée) :

```cpp
// ✅ Pas de copie — la lambda est utilisée immédiatement
auto count = std::count_if(large_data.begin(), large_data.end(),
    [&large_data](int v) { return v > 0; }  // Ici &large_data n'est même pas nécessaire
);
```

## Capture par valeur de pointeurs : un faux sentiment de sécurité

Capturer un **pointeur brut** par valeur copie le pointeur lui-même, pas l'objet pointé. La lambda et le contexte englobant partagent donc la même zone mémoire sous-jacente :

```cpp
int* ptr = new int(42);  
auto reader = [ptr]() { return *ptr; };  

*ptr = 99;
std::print("{}\n", reader());  // Affiche : 99 — le pointeur pointe au même endroit

delete ptr;
// reader();  // UNDEFINED BEHAVIOR : le pointeur est maintenant dangling
```

La copie du pointeur ne protège pas contre la désallocation de la mémoire pointée. C'est un problème identique aux dangling references, mais plus insidieux car la syntaxe `[ptr]` donne l'impression d'une capture par valeur sûre.

La même logique s'applique aux itérateurs, aux `string_view`, aux `span`, et à tout type qui est essentiellement une **vue** sur des données possédées ailleurs. Capturer ces types par valeur copie la vue, pas les données.

### Solution avec les smart pointers

Pour les ressources dynamiques, `std::shared_ptr` fournit une capture par valeur véritablement sûre grâce au comptage de références :

```cpp
auto ptr = std::make_shared<int>(42);

auto reader = [ptr]() { return *ptr; };  // Incrémente le compteur de références

// Même si le shared_ptr original est détruit, la lambda maintient l'objet en vie
ptr.reset();

std::print("{}\n", reader());  // ✅ Affiche : 42 — l'objet est toujours vivant
```

## Capture par valeur et types non-copiables

Certains types ne peuvent pas être copiés — `std::unique_ptr`, `std::mutex`, `std::thread`, ou toute classe avec un constructeur de copie supprimé. La capture par valeur directe de ces types provoque une erreur de compilation :

```cpp
auto ptr = std::make_unique<int>(42);

// ERREUR : std::unique_ptr n'est pas copiable
// auto reader = [ptr]() { return *ptr; };

// ERREUR aussi avec le défaut [=]
// auto reader = [=]() { return *ptr; };
```

La solution est l'**init capture avec déplacement**, introduite en C++14 et couverte en section 11.1.4 :

```cpp
auto ptr = std::make_unique<int>(42);  
auto reader = [p = std::move(ptr)]() { return *p; };  
// ptr est maintenant nullptr — la lambda possède la ressource
```

## Capture par valeur et `this`

Dans une méthode de classe, la capture par défaut `[=]` ne copie **pas** les membres de l'objet — elle capture le **pointeur `this`** par valeur. C'est une source fréquente de confusion :

```cpp
class Timer {
    int interval_ = 100;
public:
    auto create_callback() {
        // [=] capture this par valeur (le pointeur), pas interval_ !
        return [=]() { return interval_; };
    }
};
```

La lambda accède à `interval_` via le pointeur `this` capturé. Si l'objet `Timer` est détruit avant l'appel de la lambda, on obtient un comportement indéfini — exactement le même problème qu'avec un pointeur brut dangling.

Ce sujet est traité en profondeur dans la section 11.1.3 (Capture de `this`), qui explique également la différence entre `[this]` et `[*this]` (C++17).

> ⚠️ **Avertissement du compilateur** : depuis C++20, la capture implicite de `this` via `[=]` génère un warning avec GCC et Clang (`-Wdeprecated`). Le comité C++ a jugé que le comportement était trop surprenant et recommande de capturer `this` ou `*this` explicitement.

## `[=]` vs captures explicites : recommandations

Le choix entre la capture par défaut `[=]` et les captures explicites `[x, y]` est un équilibre entre concision et lisibilité.

### Arguments en faveur de `[=]`

La capture par défaut est concise et pratique quand la lambda utilise plusieurs variables du contexte. Elle évite de maintenir manuellement une liste qui doit évoluer avec le corps de la lambda :

```cpp
// Concis — approprié quand le contexte est simple
auto compute = [=]() {
    return base_price * quantity * (1.0 - discount);
};
```

### Arguments en faveur des captures explicites

Les C++ Core Guidelines (item F.51) **recommandent les captures explicites** plutôt que les défauts. Les raisons sont les suivantes :

Elles documentent les dépendances de la lambda. En lisant `[base_price, quantity, discount]`, on sait exactement de quoi la lambda dépend, sans avoir à lire son corps.

Elles évitent les captures accidentelles. Avec `[=]`, l'ajout d'une variable dans le corps de la lambda la capture automatiquement. Si cette variable est un type coûteux à copier ou un pointeur avec des problèmes de durée de vie, la capture silencieuse peut introduire un bug :

```cpp
// Avec [=], on ne voit pas qu'on capture expensive_data
auto transform = [=]() {
    // ... 50 lignes de code ...
    return process(expensive_data);  // Capture silencieuse d'un gros objet
};

// Avec des captures explicites, c'est visible immédiatement
auto transform = [&expensive_data]() {
    // ... 50 lignes de code ...
    return process(expensive_data);
};
```

Elles rendent explicite la capture de `this` dans les méthodes de classe, évitant le piège décrit dans le paragraphe précédent.

### Recommandation pratique

Pour les lambdas **courtes** (1 à 3 lignes) utilisées **localement** avec un contexte simple, `[=]` est acceptable et idiomatique. Pour les lambdas **longues**, **stockées**, ou **utilisées dans un contexte concurrent**, préférez les captures explicites — le coût en verbosité est largement compensé par la clarté et la sécurité.

## Récapitulatif

| Caractéristique | Comportement |
|---|---|
| Mécanisme | Copie de la variable au moment de la création |
| Modification de l'original | Invisible pour la lambda |
| Modification dans la lambda | Interdite sauf `mutable` |
| Durée de vie | Sûre — la copie vit avec la lambda |
| Coût | Dépend du constructeur de copie du type |
| Types non-copiables | Impossible — utiliser init capture + `std::move` |
| Pointeurs bruts | Copie le pointeur, pas l'objet pointé |
| `[=]` et `this` | Capture le pointeur `this`, pas les membres |
