# 36.1.3 — Validation et callbacks

## Section 36.1 : CLI11 — Parsing d'arguments professionnel

---

## Introduction

La section précédente a montré comment déclarer des options, flags et sous-commandes. Mais déclarer une interface ne suffit pas : il faut **valider** les entrées utilisateur et **réagir** au résultat du parsing. Cette section couvre deux mécanismes complémentaires de CLI11 : les **validateurs** (Validators), qui vérifient les valeurs avant qu'elles n'atteignent la logique métier, et les **callbacks**, qui exécutent du code en réponse au parsing.

Un outil CLI robuste ne devrait jamais laisser une valeur invalide atteindre la logique métier. La validation à la frontière — au moment du parsing — est la première ligne de défense, et CLI11 la rend déclarative.

---

## Validateurs intégrés

CLI11 fournit un ensemble de validateurs prêts à l'emploi, couvrant les cas de validation les plus courants. Chacun est un objet que l'on passe à la méthode `->check()` ou `->transform()` d'une option.

### Validateurs numériques

```cpp
int port = 8080;  
app.add_option("--port,-p", port, "Port d'écoute")  
    ->check(CLI::Range(1, 65535));

double ratio = 0.5;  
app.add_option("--ratio", ratio, "Ratio")  
    ->check(CLI::Range(0.0, 1.0));

int workers = 4;  
app.add_option("--workers,-w", workers, "Nombre de workers")  
    ->check(CLI::PositiveNumber);    // > 0

int offset = 0;  
app.add_option("--offset", offset, "Décalage")  
    ->check(CLI::NonNegativeNumber); // >= 0
```

`CLI::Range(min, max)` est inclusif aux deux bornes. En cas de violation, le message d'erreur est précis :

```bash
$ mon-outil --port 70000
Error: --port: Value 70000 not in range [1, 65535]
```

### Validateurs de fichiers et répertoires

Ces validateurs vérifient l'existence (ou la non-existence) de chemins sur le système de fichiers :

```cpp
std::string input;  
app.add_option("--input,-i", input, "Fichier d'entrée")  
    ->check(CLI::ExistingFile);       // Le fichier doit exister

std::string dir;  
app.add_option("--dir,-d", dir, "Répertoire de travail")  
    ->check(CLI::ExistingDirectory);  // Le répertoire doit exister

std::string path;  
app.add_option("--path", path, "Chemin existant")  
    ->check(CLI::ExistingPath);       // Fichier OU répertoire

std::string output;  
app.add_option("--output,-o", output, "Fichier de sortie")  
    ->check(CLI::NonexistentPath);    // Ne doit PAS exister (éviter l'écrasement)
```

```bash
$ mon-outil --input data.csv
# OK si data.csv existe

$ mon-outil --input inexistant.csv
Error: --input: File does not exist: inexistant.csv

$ mon-outil --output rapport.txt
# OK si rapport.txt n'existe pas encore

$ mon-outil --output rapport.txt
Error: --output: Path already exists: rapport.txt
```

Ces validateurs sont particulièrement utiles pour les outils de traitement de fichiers : ils court-circuitent l'exécution immédiatement avec un message clair, plutôt que de laisser le programme échouer plus tard avec un message d'erreur système cryptique.

### Validateur d'appartenance à un ensemble

`CLI::IsMember` restreint la valeur à une liste explicite de choix valides :

```cpp
std::string format = "text";  
app.add_option("--format,-f", format, "Format de sortie")  
    ->check(CLI::IsMember({"text", "json", "csv", "table"}));

std::string log_level = "info";  
app.add_option("--log-level", log_level, "Niveau de log")  
    ->check(CLI::IsMember({"trace", "debug", "info", "warn", "error", "fatal"}));
```

```bash
$ mon-outil --format xml
Error: --format: Check CLI::IsMember FAILED

$ mon-outil --format json
# OK
```

Par défaut, la comparaison est sensible à la casse. Pour une comparaison insensible :

```cpp
app.add_option("--format,-f", format, "Format de sortie")
    ->check(CLI::IsMember({"text", "json", "csv"}, CLI::ignore_case));
```

```bash
$ mon-outil --format JSON    # Accepté
$ mon-outil --format Json    # Accepté aussi
```

### Validateur de chaînes de caractères

```cpp
std::string name;  
app.add_option("--name,-n", name, "Nom de la ressource")  
    ->required();  // Garantit que l'option est fournie (non vide)
```

## Transformeurs (`->transform()`)

Là où `->check()` valide sans modifier la valeur, `->transform()` **transforme** la valeur avant de l'assigner à la variable. CLI11 fournit plusieurs transformeurs intégrés.

### Conversion d'unités avec `AsNumberWithUnit`

Pour les options qui acceptent des valeurs avec unités (tailles de fichier, durées), CLI11 propose un mécanisme de conversion automatique :

```cpp
uint64_t max_size = 0;

// Mapping unité → multiplicateur
std::map<std::string, uint64_t> size_units{
    {"B", 1}, {"KB", 1024}, {"MB", 1024 * 1024},
    {"GB", 1024 * 1024 * 1024}
};

app.add_option("--max-size", max_size, "Taille maximale (ex: 512MB)")
    ->transform(CLI::AsNumberWithUnit(size_units, CLI::AsNumberWithUnit::CASE_INSENSITIVE));
```

```bash
$ mon-outil --max-size 512MB
# max_size = 536870912 (512 * 1024 * 1024)

$ mon-outil --max-size 2GB
# max_size = 2147483648

$ mon-outil --max-size 1024
# max_size = 1024 (sans unité = valeur brute)
```

### Mapping d'énumérations

Pour convertir des chaînes en valeurs d'énumération, `CLI::CheckedTransformer` offre un mapping déclaratif :

```cpp
enum class LogLevel { trace, debug, info, warn, error, fatal };

LogLevel level = LogLevel::info;

std::map<std::string, LogLevel> level_map{
    {"trace", LogLevel::trace}, {"debug", LogLevel::debug},
    {"info",  LogLevel::info},  {"warn",  LogLevel::warn},
    {"error", LogLevel::error}, {"fatal", LogLevel::fatal}
};

app.add_option("--log-level,-l", level, "Niveau de log")
    ->transform(CLI::CheckedTransformer(level_map, CLI::ignore_case))
    ->default_str("info");
```

```bash
$ mon-outil --log-level DEBUG
# level = LogLevel::debug

$ mon-outil --log-level critical
Error: --log-level: Check CLI::CheckedTransformer FAILED
```

La différence avec `IsMember` est fondamentale : `IsMember` valide une `std::string` et la laisse en l'état, tandis que `CheckedTransformer` convertit la chaîne en un type arbitraire (ici une `enum class`). Utilisez `CheckedTransformer` quand vous travaillez avec des types non-string.

### Borner après transformation

`->check()` et `->transform()` peuvent être chaînés. Les transformeurs s'exécutent en premier, puis les validateurs s'appliquent sur le résultat transformé :

```cpp
uint64_t max_size = 0;  
app.add_option("--max-size", max_size, "Taille maximale")  
    ->transform(CLI::AsNumberWithUnit(size_units,
        CLI::AsNumberWithUnit::CASE_INSENSITIVE))
    ->check(CLI::Range(uint64_t{1024}, uint64_t{4ULL * 1024 * 1024 * 1024}));
// Accepte de 1KB à 4GB après conversion
```

## Composition de validateurs

Les validateurs CLI11 sont composables grâce aux opérateurs `|` (OU logique) et `&` (ET logique). Cela permet de construire des règles de validation complexes à partir de briques simples.

### OU logique (`|`)

La valeur est acceptée si **au moins un** des validateurs réussit :

```cpp
std::string source;  
app.add_option("--source,-s", source, "Source (fichier ou répertoire)")  
    ->check(CLI::ExistingFile | CLI::ExistingDirectory);
```

```bash
$ mon-outil --source ./data/          # Répertoire : OK
$ mon-outil --source config.yaml      # Fichier : OK
$ mon-outil --source /inexistant      # Ni l'un ni l'autre : erreur
```

### ET logique (`&`)

La valeur est acceptée seulement si **tous** les validateurs réussissent :

```cpp
int value = 0;  
app.add_option("--value", value, "Valeur paire entre 0 et 100")  
    ->check(CLI::Range(0, 100) & CLI::Validator(
        [](const std::string& s) -> std::string {
            int v = std::stoi(s);
            if (v % 2 != 0) return "La valeur doit être paire";
            return {};  // Chaîne vide = validation réussie
        }, "Paire"));
```

Ce dernier exemple utilise un validateur personnalisé inline — nous y revenons en détail dans la section suivante.

## Validateurs personnalisés

Les validateurs intégrés couvrent les cas courants, mais la logique métier impose souvent des contraintes spécifiques. CLI11 permet de créer des validateurs personnalisés de deux façons.

### Via `CLI::Validator` et une lambda

Un validateur est une fonction qui reçoit une `std::string` (la valeur brute de la ligne de commande) et retourne une `std::string` : vide si la validation réussit, un message d'erreur sinon.

```cpp
// Validateur : le port ne doit pas être un port réservé
auto not_reserved_port = CLI::Validator(
    [](const std::string& input) -> std::string {
        int port = std::stoi(input);
        if (port < 1024)
            return "Le port " + input + " est réservé (< 1024)";
        return {};  // OK
    },
    "Port non réservé (>= 1024)"  // Description pour l'aide
);

int port = 8080;  
app.add_option("--port,-p", port, "Port d'écoute")  
    ->check(CLI::Range(1, 65535) & not_reserved_port);
```

```bash
$ mon-outil --port 80
Error: --port: Le port 80 est réservé (< 1024)

$ mon-outil --port 8080
# OK
```

### Validateurs réutilisables en tant que fonctions

Pour un validateur utilisé dans plusieurs endroits du code, on peut le définir comme une fonction qui retourne un `CLI::Validator` :

```cpp
// validators.hpp
#pragma once
#include <CLI/CLI.hpp>
#include <regex>

inline CLI::Validator SemVer() {
    return CLI::Validator(
        [](const std::string& input) -> std::string {
            std::regex re(R"(^\d+\.\d+\.\d+(-[\w.]+)?(\+[\w.]+)?$)");
            if (!std::regex_match(input, re))
                return "'" + input + "' n'est pas un semver valide "
                       "(attendu: MAJOR.MINOR.PATCH)";
            return {};
        },
        "Semantic Version (MAJOR.MINOR.PATCH)"
    );
}

inline CLI::Validator HostPort() {
    return CLI::Validator(
        [](const std::string& input) -> std::string {
            auto pos = input.rfind(':');
            if (pos == std::string::npos)
                return "Format attendu : host:port";
            std::string port_str = input.substr(pos + 1);
            try {
                int port = std::stoi(port_str);
                if (port < 1 || port > 65535)
                    return "Port hors limites : " + port_str;
            } catch (...) {
                return "Port invalide : " + port_str;
            }
            return {};
        },
        "host:port"
    );
}

inline CLI::Validator KeyValuePair(char delimiter = '=') {
    return CLI::Validator(
        [delimiter](const std::string& input) -> std::string {
            if (input.find(delimiter) == std::string::npos)
                return "Format attendu : clé" + std::string(1, delimiter)
                       + "valeur — reçu : '" + input + "'";
            return {};
        },
        std::string("clé") + delimiter + "valeur"
    );
}
```

Utilisation :

```cpp
#include "validators.hpp"

std::string version;  
app.add_option("--version,-V", version, "Version du paquet")  
    ->check(SemVer());

std::string listen_addr = "0.0.0.0:8080";  
app.add_option("--listen", listen_addr, "Adresse d'écoute")  
    ->check(HostPort());

std::vector<std::string> labels;  
app.add_option("--label,-l", labels, "Labels (clé=valeur)")  
    ->check(KeyValuePair('='));
```

```bash
$ mon-outil --version 1.2
Error: --version: '1.2' n'est pas un semver valide (attendu: MAJOR.MINOR.PATCH)

$ mon-outil --version 1.2.3
# OK

$ mon-outil --listen localhost
Error: --listen: Format attendu : host:port

$ mon-outil --listen localhost:8080
# OK

$ mon-outil --label "env=prod" --label "team=infra"
# OK

$ mon-outil --label "orphelin"
Error: --label: Format attendu : clé=valeur — reçu : 'orphelin'
```

Ce pattern de validateurs réutilisables sous forme de fonctions retournant `CLI::Validator` est l'approche recommandée pour les projets de taille significative. Les validateurs sont testables unitairement, composables et documentés.

## `->check()` vs `->transform()` : quelle méthode choisir ?

Ces deux méthodes acceptent les mêmes types de validateurs/transformeurs, mais leur sémantique diffère :

| Aspect | `->check()` | `->transform()` |
|--------|------------|-----------------|
| **Rôle** | Valider sans modifier | Transformer puis assigner |
| **Valeur stockée** | La valeur originale | La valeur transformée |
| **Cas d'usage** | Vérifier un format, un intervalle | Convertir des unités, mapper une enum |
| **Aide générée** | Affiche le nom du validateur | Affiche le nom du transformeur |

En cas de doute, la règle est simple : si vous vérifiez que la valeur est conforme, utilisez `check()`. Si vous transformez la valeur en quelque chose d'autre, utilisez `transform()`.

Ils peuvent être combinés librement sur la même option — les transformeurs s'exécutent d'abord (dans l'ordre de déclaration), puis les validateurs s'appliquent sur le résultat :

```cpp
app.add_option("--size", size, "Taille")
    ->transform(CLI::AsNumberWithUnit(units))  // "512MB" → 536870912
    ->check(CLI::Range(1024, max_allowed));     // Vérifie la valeur convertie
```

## Contraintes inter-options

Au-delà de la validation individuelle, CLI11 permet de définir des contraintes logiques entre options.

### Exclusion mutuelle (`->excludes()`)

Deux options qui ne peuvent pas coexister :

```cpp
auto* opt_json = app.add_flag("--json", use_json, "Sortie JSON");  
auto* opt_csv = app.add_flag("--csv", use_csv, "Sortie CSV");  
opt_json->excludes(opt_csv);  
opt_csv->excludes(opt_json);  
```

```bash
$ mon-outil --json --csv
Error: --json: excludes --csv
```

### Dépendance (`->needs()`)

Une option qui requiert la présence d'une autre :

```cpp
auto* opt_user = app.add_option("--user,-u", user, "Utilisateur");  
auto* opt_pass = app.add_option("--password,-p", password, "Mot de passe");  
auto* opt_token = app.add_option("--token,-t", token, "Token d'API");  

opt_pass->needs(opt_user);     // --password exige --user  
opt_user->needs(opt_pass);     // --user exige --password (bidirectionnel)  
```

```bash
$ mon-outil --user admin
Error: --user: requires --password

$ mon-outil --user admin --password secret
# OK

$ mon-outil --token abc123
# OK (indépendant du couple user/password)
```

### Groupes d'options mutuellement exclusives

Pour un ensemble de plus de deux options mutuellement exclusives, les appels `excludes()` croisés deviennent verbeux. CLI11 propose les **option groups** :

```cpp
auto* auth_group = app.add_option_group("Authentification");

std::string user, password, token;  
std::string cert_path;  

auto* basic_opt = auth_group->add_option("--user", user, "Utilisateur");  
auth_group->add_option("--password", password, "Mot de passe")  
    ->needs(basic_opt);

auto* token_opt = auth_group->add_option("--token", token, "Token Bearer");

auto* cert_opt = auth_group->add_option("--cert", cert_path, "Certificat client")
    ->check(CLI::ExistingFile);

// Une seule méthode d'authentification à la fois
basic_opt->excludes(token_opt)->excludes(cert_opt);  
token_opt->excludes(basic_opt)->excludes(cert_opt);  
cert_opt->excludes(basic_opt)->excludes(token_opt);  

auth_group->require_option(0, 1);  // 0 ou 1 option du groupe
```

L'aide générée regroupe automatiquement ces options sous le titre "Authentification", ce qui améliore la lisibilité.

## Callbacks

Les callbacks sont des fonctions exécutées automatiquement après le parsing. Ils permettent de réagir au résultat du parsing sans multiplier les blocs `if/else` dans le `main()`.

### Callback sur une sous-commande

Le cas d'usage le plus courant : attacher une action à chaque sous-commande.

```cpp
auto* cmd_init = app.add_subcommand("init", "Initialiser un projet");  
std::string project_name;  
std::string tmpl = "default";  

cmd_init->add_option("name", project_name, "Nom du projet")->required();  
cmd_init->add_option("--template,-t", tmpl, "Template de projet")  
    ->check(CLI::IsMember({"default", "lib", "cli", "server"}));

cmd_init->callback([&]() {
    std::println("Initialisation du projet '{}' (template: {})",
                 project_name, tmpl);
    // Appel à la logique métier...
});

// Même chose pour d'autres sous-commandes
auto* cmd_clean = app.add_subcommand("clean", "Nettoyer les artefacts");  
bool all = false;  
cmd_clean->add_flag("--all,-a", all, "Supprimer aussi le cache");  

cmd_clean->callback([&]() {
    std::println("Nettoyage{}", all ? " complet (cache inclus)" : "");
});

CLI11_PARSE(app, argc, argv);
// Le callback de la sous-commande active est exécuté automatiquement
// après le retour de CLI11_PARSE
```

Le callback est invoqué seulement si la sous-commande a été parsée avec succès. Cela garantit que toutes les variables sont remplies et validées au moment de l'exécution.

### Callback sur une option individuelle

On peut aussi attacher un callback à une option spécifique. Il s'exécute quand l'option est rencontrée pendant le parsing :

```cpp
int verbosity = 0;  
app.add_flag("--verbose,-v", verbosity, "Niveau de verbosité")  
    ->each([](const std::string&) {
        // Exécuté à chaque occurrence de -v
        // Utile pour configurer le logger en temps réel
    });
```

### Callback de pré-parsing sur l'application

L'application elle-même peut avoir un callback, exécuté quand le parsing de ses options directes est terminé (avant les sous-commandes) :

```cpp
app.callback([&]() {
    // Exécuté après le parsing des options globales
    if (verbose) {
        std::println(stderr, "[debug] Configuration chargée depuis: {}",
                     config_file.empty() ? "(défaut)" : config_file);
    }
});
```

### Callback pour la validation croisée complexe

Quand la validation nécessite de combiner plusieurs options avec une logique qui dépasse `needs()` et `excludes()`, un callback sur la sous-commande ou l'application permet d'implémenter la vérification manuellement :

```cpp
auto* cmd_deploy = app.add_subcommand("deploy", "Déployer l'application");

std::string env;  
bool force = false;  
bool dry_run = false;  
int replicas = 1;  

cmd_deploy->add_option("--env,-e", env, "Environnement")->required()
    ->check(CLI::IsMember({"dev", "staging", "production"}));
cmd_deploy->add_flag("--force,-f", force, "Ignorer les garde-fous");  
cmd_deploy->add_flag("--dry-run", dry_run, "Simuler le déploiement");  
cmd_deploy->add_option("--replicas,-r", replicas, "Nombre de réplicas")  
    ->check(CLI::Range(1, 50));

cmd_deploy->callback([&]() {
    // Validation croisée : production + replicas élevé nécessite --force
    if (env == "production" && replicas > 10 && !force) {
        std::println(stderr,
            "Erreur : déployer plus de 10 réplicas en production "
            "requiert --force");
        std::exit(1);
    }

    // Validation croisée : dry-run et force sont contradictoires
    if (dry_run && force) {
        std::println(stderr,
            "Erreur : --dry-run et --force ne peuvent pas être "
            "combinés");
        std::exit(1);
    }

    // Tout est validé, on procède
    std::println("Déploiement sur {} ({} réplicas{}{})",
                 env, replicas,
                 force ? ", forcé" : "",
                 dry_run ? ", simulation" : "");
});
```

Ce pattern est essentiel pour les outils critiques : les validateurs CLI11 couvrent les contraintes individuelles, et le callback de sous-commande couvre les invariants métier qui impliquent plusieurs options ensemble.

## Gestion fine des erreurs de parsing

Par défaut, `CLI11_PARSE` intercepte toutes les erreurs et affiche un message. Pour un contrôle plus fin, on remplace la macro par un parsing explicite :

```cpp
try {
    app.parse(argc, argv);
} catch (const CLI::ParseError& e) {
    // Personnaliser le comportement selon le type d'erreur
    if (dynamic_cast<const CLI::CallForHelp*>(&e)) {
        // L'utilisateur a demandé --help
        std::cout << app.help() << std::endl;
        return 0;
    }
    if (dynamic_cast<const CLI::CallForVersion*>(&e)) {
        // L'utilisateur a demandé --version
        std::println("{}", app.version());
        return 0;
    }
    // Toute autre erreur : afficher sur stderr et quitter
    std::println(stderr, "Erreur : {}", e.what());
    std::println(stderr, "Exécutez '{} --help' pour l'aide.", argv[0]);
    return e.get_exit_code();
}
```

Les types d'exceptions CLI11 les plus courants :

| Exception | Cause | Code de sortie |
|-----------|-------|:--------------:|
| `CLI::CallForHelp` | `--help` invoqué | 0 |
| `CLI::CallForVersion` | `--version` invoqué | 0 |
| `CLI::ParseError` | Classe de base de toutes les erreurs | variable |
| `CLI::RequiredError` | Option obligatoire manquante | 1 |
| `CLI::ValidationError` | Échec d'un validateur | 2 |
| `CLI::ConversionError` | Impossible de convertir la valeur | 2 |
| `CLI::ExtrasError` | Arguments non reconnus | 1 |

Ce parsing explicite est recommandé quand vous avez besoin de logger les erreurs, d'effectuer un nettoyage, ou de personnaliser les messages — par exemple pour afficher les erreurs en français ou pour intégrer la sortie avec un système de logging structuré (voir **section 40.1** sur spdlog).

## Ordre d'exécution complet

Comprendre l'ordre dans lequel CLI11 exécute les différentes phases permet d'éviter les surprises :

```
1. Parsing des arguments (gauche à droite sur la ligne de commande)
   ├── Pour chaque option rencontrée :
   │   ├── a) Transformation (->transform()) — dans l'ordre de déclaration
   │   ├── b) Validation (->check()) — dans l'ordre de déclaration
   │   └── c) Assignation à la variable de destination
   │
2. Vérification des contraintes
   │   ├── Options required() manquantes ?
   │   ├── Contraintes needs() / excludes() respectées ?
   │   └── require_option() / require_subcommand() satisfaits ?
   │
3. Fallback sur les variables d'environnement (envname())
   │   └── Même pipeline transform → check → assign
   │
4. Fallback sur le fichier de configuration (set_config())
   │   └── Même pipeline transform → check → assign
   │
5. Exécution des callbacks
       ├── Callback de l'application principale
       └── Callback de la sous-commande active
```

Un point important : les variables d'environnement et le fichier de configuration ne sont consultés que si l'option n'a **pas** été fournie sur la ligne de commande. L'ordre de priorité est donc :

```
Ligne de commande  >  Variable d'environnement  >  Fichier de config  >  Valeur par défaut
```

## Résumé des bonnes pratiques

**Validez tôt, validez tout.** Chaque option qui accepte une valeur devrait avoir au minimum un validateur. Les fichiers et répertoires passent par `ExistingFile` / `ExistingDirectory`, les nombres par `Range` ou `PositiveNumber`, les choix par `IsMember` ou `CheckedTransformer`.

**Privilégiez les validateurs déclaratifs.** Utilisez `->check()` et `->transform()` plutôt qu'une validation manuelle dans le callback. Le code déclaratif est plus lisible, testable et génère automatiquement une aide informative.

**Réservez les callbacks pour la logique croisée.** Les contraintes simples entre deux options passent par `needs()` et `excludes()`. Les invariants métier impliquant trois options ou plus, ou des conditions dynamiques, passent par un callback de sous-commande.

**Créez des validateurs réutilisables.** Dès qu'un pattern de validation apparaît plus d'une fois (semver, host:port, clé=valeur, identifiant alphanumérique), extrayez-le dans une fonction retournant `CLI::Validator`. Ce sont des briques testables et documentées.

**Utilisez `CheckedTransformer` pour les enums.** C'est le pattern standard pour convertir des chaînes en valeurs typées, avec validation et message d'erreur intégrés — bien plus robuste qu'un `if/else` sur la chaîne brute.
