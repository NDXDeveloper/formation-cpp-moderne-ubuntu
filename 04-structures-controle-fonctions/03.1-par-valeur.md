# 4.3.1 — Passage par valeur

## Chapitre 4 · Structures de Contrôle et Fonctions · Niveau Débutant

---

## Principe

Quand un paramètre est passé **par valeur**, la fonction reçoit une **copie indépendante** de l'argument fourni par l'appelant. Toute modification effectuée sur le paramètre à l'intérieur de la fonction n'a **aucun effet** sur la variable originale.

```cpp
#include <iostream>

void doubler(int x) {
    x *= 2;
    std::cout << "Dans la fonction : x = " << x << "\n";
}

int main() {
    int n = 5;
    doubler(n);
    std::cout << "Après l'appel    : n = " << n << "\n";
}
```

```
Dans la fonction : x = 10  
Après l'appel    : n = 5  
```

Le mécanisme est simple : au moment de l'appel, la valeur de `n` est copiée dans un nouvel emplacement mémoire réservé sur la stack pour le paramètre `x`. Les deux variables vivent dans des zones mémoire distinctes. Quand la fonction se termine, `x` est détruit — `n` n'a jamais été touché.

---

## Ce qui se passe en mémoire

Visualisons la stack au moment de l'appel `doubler(n)` :

```
┌──────────────────────────┐
│  main()                  │
│    n = 5                 │  ← variable originale
├──────────────────────────┤
│  doubler()               │
│    x = 5  (copie de n)   │  ← copie indépendante
└──────────────────────────┘
```

Après `x *= 2` :

```
┌──────────────────────────┐
│  main()                  │
│    n = 5                 │  ← inchangé
├──────────────────────────┤
│  doubler()               │
│    x = 10                │  ← seule la copie est modifiée
└──────────────────────────┘
```

Quand `doubler()` retourne, son cadre de pile (*stack frame*) est libéré. `x` disparaît. `n` conserve sa valeur d'origine.

## Quand le passage par valeur est approprié

### Types primitifs et petits types

Pour les types dont la copie est triviale et peu coûteuse, le passage par valeur est le choix naturel. Le coût de la copie est négligeable — souvent une simple instruction machine — et la sémantique est limpide : la fonction travaille sur sa propre donnée.

```cpp
double aire_cercle(double rayon) {
    return 3.14159265358979 * rayon * rayon;
}

bool est_pair(int n) {
    return n % 2 == 0;
}

char to_upper(char c) {
    if (c >= 'a' && c <= 'z') {
        return c - ('a' - 'A');
    }
    return c;
}
```

En règle générale, les types suivants sont efficacement passés par valeur : `bool`, `char`, `int`, `long`, `float`, `double`, les pointeurs bruts, les itérateurs, et les petites structures contenant uniquement des types primitifs (jusqu'à environ 16 octets, soit deux registres sur une architecture 64 bits).

### Quand on a besoin d'une copie locale de toute façon

Parfois, la fonction a besoin de modifier les données **localement** sans affecter l'appelant. Plutôt que de passer par référence constante et de créer une copie explicite dans le corps, on peut passer par valeur directement — le compilateur se charge de la copie (ou de l'éliminer, voir la section sur la copy elision).

```cpp
// ❌ Passage par référence constante + copie manuelle
std::string normaliser_v1(const std::string& input) {
    std::string result = input;  // copie explicite
    std::transform(result.begin(), result.end(), result.begin(), ::tolower);
    return result;
}

// ✅ Passage par valeur — la copie est faite à l'appel
std::string normaliser_v2(std::string input) {
    std::transform(input.begin(), input.end(), input.begin(), ::tolower);
    return input;
}
```

Les deux versions produisent le même résultat. La seconde est souvent préférée car elle est plus concise et permet au compilateur d'optimiser mieux dans certains cas (notamment quand l'argument est un temporaire — le déplacement remplace alors la copie).

C'est ce qu'on appelle l'idiome **sink parameter** : le paramètre « absorbe » la valeur reçue. Cet idiome prend tout son sens avec la sémantique de mouvement (chapitre 10), mais il est bon de le connaître dès maintenant.

## Le coût de la copie

Le passage par valeur implique une copie, et le coût de cette copie dépend entièrement du type.

### Types trivials — copie quasi gratuite

Pour un `int` ou un `double`, la copie se résume à copier quelques octets entre registres ou emplacements mémoire. Le coût est insignifiant.

```cpp
// Copie de 4 octets (int) — une instruction machine
void process(int value) { /* ... */ }
```

### Types complexes — copie potentiellement coûteuse

Pour un `std::vector<int>` contenant un million d'éléments, le passage par valeur copie **l'intégralité du vecteur** : allocation mémoire sur le heap, copie élément par élément. Le coût peut se chiffrer en millisecondes.

```cpp
// ❌ Copie d'un vecteur de 1 000 000 d'éléments à chaque appel
void analyser(std::vector<int> donnees) {
    // ...
}

int main() {
    std::vector<int> gros_vecteur(1'000'000, 42);
    analyser(gros_vecteur);  // Copie complète — lent
}
```

Pour ce cas, le passage par `const std::vector<int>&` (section 4.3.3) évite toute copie.

### Tableau indicatif du coût

| Type | Taille typique | Coût de la copie | Passer par valeur ? |  
|------|---------------|------------------|---------------------|  
| `int`, `double`, `char`, `bool` | 1–8 octets | Négligeable | Oui |  
| Pointeur (`T*`) | 8 octets (64-bit) | Négligeable | Oui |  
| Petite struct (2–3 champs primitifs) | 8–24 octets | Faible | Généralement oui |  
| `std::string` | ~32 octets + heap | Variable (SSO aide pour les courtes) | Seulement si copie nécessaire |  
| `std::vector<T>` | ~24 octets + heap | Proportionnel à `size()` | Non (sauf sink parameter) |  
| `std::map<K,V>` | ~48 octets + heap | Proportionnel à `size()` | Non |

> 💡 **SSO** (Small String Optimization) — Les implémentations de `std::string` stockent les chaînes courtes (typiquement ≤ 15–22 caractères) directement dans l'objet, sans allocation heap. Pour ces chaînes, la copie est rapide. Au-delà, une allocation et une copie du buffer sont nécessaires.

## Copy elision et RVO

Le compilateur est autorisé (et souvent obligé depuis C++17) à **éliminer certaines copies** quand la valeur de retour d'une fonction est directement utilisée. C'est ce qu'on appelle la **copy elision**, dont le cas le plus courant est la **Return Value Optimization** (RVO).

```cpp
std::string creer_message() {
    std::string msg = "Hello, World!";
    return msg;  // Le compilateur peut éliminer la copie
}

int main() {
    std::string resultat = creer_message();
    // Avec RVO : aucune copie — 'msg' est construit directement dans 'resultat'
}
```

En C++17, la copy elision est **garantie** dans le cas où la fonction retourne un objet temporaire (prvalue). Pour les variables locales nommées (NRVO — Named Return Value Optimization), elle reste une optimisation facultative mais appliquée par tous les compilateurs majeurs.

Concrètement, cela signifie que retourner un objet par valeur depuis une fonction est **efficace** en C++ moderne — il n'y a pas de copie inutile. C'est un changement de mentalité important par rapport au C++98, où retourner un gros objet par valeur était découragé.

Le sujet est couvert en profondeur à la section 10.5 (RVO et Copy Elision).

## Qualifier le paramètre avec `const`

Même un paramètre passé par valeur peut être qualifié `const`. Cela empêche toute modification accidentelle de la copie locale dans le corps de la fonction :

```cpp
int carre(const int x) {
    // x = 10;  // ❌ Erreur de compilation
    return x * x;
}
```

Cette pratique est une question de style. Elle n'a **aucun impact** sur l'appelant (qui fournit de toute façon une copie) ni sur la signature vue de l'extérieur (le compilateur ignore le `const` de premier niveau dans les déclarations pour la résolution de surcharge). Son seul effet est de protéger le développeur de la fonction contre une modification involontaire.

Certains guides de style la recommandent systématiquement (plus de sécurité), d'autres la considèrent comme du bruit inutile (la copie est locale, donc le risque est limité). Les deux positions sont défendables — l'essentiel est d'être cohérent au sein d'un projet.

## Pièges courants

### Passer un gros objet par valeur sans le vouloir

C'est l'erreur la plus fréquente chez les débutants en C++, en particulier ceux venant de langages à ramasse-miettes :

```cpp
// ❌ Copie involontaire d'un vecteur à chaque appel
void afficher_stats(std::vector<double> mesures) {
    double somme = 0;
    for (double m : mesures) somme += m;
    std::cout << "Moyenne : " << somme / mesures.size() << "\n";
}
```

Ici, la fonction ne modifie pas `mesures`. Le paramètre devrait être `const std::vector<double>&`. Le compilateur ne produira **aucun warning** — la copie est parfaitement légale, juste inutilement coûteuse.

### Croire qu'on modifie l'original

```cpp
void reset(int compteur) {
    compteur = 0;  // Modifie la copie locale, pas l'original
}

int main() {
    int c = 42;
    reset(c);
    std::cout << c << "\n";  // Affiche 42, pas 0
}
```

Si l'intention est de modifier la variable de l'appelant, il faut passer par référence (`int&`), comme le montre la section 4.3.2.

### Copie de types polymorphiques : le *slicing*

Quand on passe un objet dérivé par valeur à une fonction qui attend la classe de base, l'objet est **tronqué** (*sliced*) : seule la partie correspondant à la classe de base est copiée, et les données spécifiques à la classe dérivée sont perdues.

```cpp
#include <iostream>

struct Animal {
    virtual void parler() const { std::cout << "...\n"; }
};

struct Chien : Animal {
    void parler() const override { std::cout << "Woof!\n"; }
};

// ❌ Passage par valeur — slicing
void faire_parler(Animal a) {
    a.parler();  // Appelle toujours Animal::parler(), jamais Chien::parler()
}

// ✅ Passage par référence constante — polymorphisme préservé
void faire_parler_ok(const Animal& a) {
    a.parler();  // Appelle la bonne version selon le type réel
}

int main() {
    Chien rex;
    faire_parler(rex);      // Affiche "..." — slicing !
    faire_parler_ok(rex);   // Affiche "Woof!" — polymorphisme correct
}
```

Le slicing est un des pièges les plus insidieux du C++. La règle est simple : **ne passez jamais un objet polymorphique par valeur**. Utilisez une référence ou un pointeur (sections 4.3.2 à 4.3.4). Ce sujet sera approfondi au chapitre 7 (Héritage et Polymorphisme).

---

## Résumé

| Aspect | Passage par valeur |  
|--------|--------------------|  
| Syntaxe | `void f(T x)` |  
| Copie de l'argument ? | Oui |  
| Fonction modifie l'original ? | Non — travaille sur la copie |  
| Adapté aux types primitifs ? | Oui — c'est le mode par défaut |  
| Adapté aux gros objets ? | Non, sauf besoin explicite d'une copie (sink parameter) |  
| Polymorphisme préservé ? | Non — risque de slicing |  
| Copy elision possible ? | Oui (RVO/NRVO, surtout C++17+) |
