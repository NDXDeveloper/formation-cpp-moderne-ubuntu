# 4.1.1 — `if`, `else if`, `else` et `if constexpr`

## Chapitre 4 · Structures de Contrôle et Fonctions · Niveau Débutant

---

## La conditionnelle fondamentale : `if` / `else`

Le bloc `if` est la structure de contrôle la plus élémentaire. Il évalue une expression booléenne et exécute un bloc d'instructions si cette expression vaut `true`. Un bloc `else` optionnel fournit le chemin alternatif.

```cpp
#include <iostream>

int main() {
    int temperature = 38;

    if (temperature > 37) {
        std::cout << "Fièvre détectée\n";
    } else {
        std::cout << "Température normale\n";
    }
}
```

L'expression entre parenthèses doit être convertible en `bool`. En C++, les entiers, les pointeurs et les objets disposant d'un `operator bool()` sont implicitement convertibles : zéro et `nullptr` valent `false`, toute autre valeur vaut `true`.

```cpp
int* ptr = get_resource();

// Idiome courant — le pointeur est implicitement converti en bool
if (ptr) {
    ptr->process();
}
```

## Chaîner les conditions : `else if`

Lorsqu'il faut distinguer plus de deux cas, on chaîne les blocs avec `else if`. Le compilateur évalue les conditions **de haut en bas** et exécute le premier bloc dont la condition est vraie. Si aucune condition n'est satisfaite, le bloc `else` final (s'il existe) est exécuté.

```cpp
#include <iostream>

int main() {
    int score = 73;

    if (score >= 90) {
        std::cout << "Mention : Excellent\n";
    } else if (score >= 75) {
        std::cout << "Mention : Bien\n";
    } else if (score >= 60) {
        std::cout << "Mention : Assez bien\n";
    } else {
        std::cout << "Mention : Insuffisant\n";
    }
}
```

**Ordre des conditions.** Puisque l'évaluation s'arrête au premier `true`, l'ordre est important. Placez toujours les conditions les plus restrictives en premier. Inverser les deux premières lignes dans l'exemple ci-dessus donnerait « Bien » pour un score de 95, ce qui serait incorrect.

## `if` avec initialisation (C++17)

C++17 introduit une syntaxe qui permet de **déclarer une variable directement dans le `if`**, en limitant sa portée au bloc `if` / `else if` / `else`. La forme générale est :

```
if (initialisation; condition) { ... }
```

Le point-virgule sépare la déclaration de la condition. La variable déclarée est accessible dans la condition elle-même et dans tous les blocs `if` / `else` qui suivent, mais **pas en dehors**.

### Avant C++17

Sans cette syntaxe, on est contraint de déclarer la variable dans la portée englobante, où elle reste visible bien au-delà de son utilité :

```cpp
auto it = config.find("timeout");  
if (it != config.end()) {  
    apply_timeout(it->second);
} else {
    use_default_timeout();
}
// 'it' est toujours accessible ici — risque de réutilisation involontaire
```

### Avec C++17

```cpp
if (auto it = config.find("timeout"); it != config.end()) {
    apply_timeout(it->second);
} else {
    use_default_timeout();
}
// 'it' n'existe plus — la portée est parfaitement délimitée
```

Cette syntaxe est particulièrement utile avec les fonctions qui renvoient un itérateur, un `std::optional`, ou un code d'erreur. Quelques exemples courants :

```cpp
// Avec std::optional
if (auto val = parse_int(input); val.has_value()) {
    std::cout << "Valeur : " << val.value() << "\n";
}

// Avec un code retour POSIX
if (int fd = open("/tmp/data.bin", O_RDONLY); fd >= 0) {
    read(fd, buffer, sizeof(buffer));
    close(fd);
} else {
    std::cerr << "Erreur d'ouverture (fd=" << fd << ")\n";
}

// Avec un lock (multithreading)
if (std::lock_guard lock(mutex); shared_queue.empty()) {
    std::cout << "File vide\n";
}
```

> 💡 **Règle pratique** — Utilisez le `if` avec initialisation chaque fois qu'une variable n'est nécessaire que pour tester une condition et traiter le résultat. C'est un réflexe à adopter en C++ moderne.

## `if constexpr` : branchement à la compilation (C++17)

### Le problème

Imaginons une fonction template qui doit se comporter différemment selon le type reçu. Avec un `if` classique, **les deux branches doivent compiler** même si une seule sera exécutée à l'exécution :

```cpp
#include <iostream>
#include <type_traits>

template <typename T>  
void afficher(const T& val) {  
    // ❌ Ne compile pas si T est un entier : .size() n'existe pas
    if (std::is_integral_v<T>) {
        std::cout << "Entier : " << val << "\n";
    } else {
        std::cout << "Autre (taille : " << val.size() << ")\n";
    }
}
```

Si on appelle `afficher(42)`, le compilateur tente quand même d'instancier la branche `else` avec `T = int`. L'appel `val.size()` sur un `int` provoque une erreur de compilation.

Avant C++17, la solution passait par la **spécialisation de template** ou par du SFINAE — deux mécanismes lourds et peu lisibles pour un besoin aussi simple.

### La solution : `if constexpr`

`if constexpr` évalue la condition **à la compilation**. La branche non retenue est **éliminée du code instancié** (on dit qu'elle est *discarded*). Elle n'a pas besoin de compiler pour le type courant :

```cpp
#include <iostream>
#include <string>
#include <type_traits>

template <typename T>  
void afficher(const T& val) {  
    if constexpr (std::is_integral_v<T>) {
        std::cout << "Entier : " << val << "\n";
    } else if constexpr (std::is_floating_point_v<T>) {
        std::cout << "Flottant : " << val << "\n";
    } else {
        std::cout << "Autre (taille : " << val.size() << ")\n";
    }
}

int main() {
    afficher(42);                    // → Entier : 42
    afficher(3.14);                  // → Flottant : 3.14
    afficher(std::string("hello"));  // → Autre (taille : 5)
}
```

Chaque instanciation ne conserve que la branche pertinente. Quand `T = int`, la branche appelant `.size()` est purement et simplement ignorée par le compilateur — aucune erreur.

### Règles de `if constexpr`

**La condition doit être une expression constante.** Elle doit pouvoir être évaluée à la compilation. Les traits de type (`std::is_integral_v`, `std::is_same_v`, etc.), les valeurs `constexpr`, et les paramètres template non-type en sont les candidats naturels.

```cpp
constexpr int version = 2;

void configure() {
    if constexpr (version >= 2) {
        enable_new_feature();
    } else {
        use_legacy_path();
    }
}
```

**La branche éliminée n'est pas instanciée, mais doit rester syntaxiquement valide.** Le compilateur vérifie que le code éliminé est du C++ bien formé en termes de syntaxe (accolades, points-virgules), même s'il ne le compile pas sémantiquement. On ne peut pas y écrire du texte arbitraire.

**`if constexpr` ne fonctionne que dans les templates ou avec des expressions `constexpr`.** Dans une fonction non-template avec une condition qui dépend d'une variable runtime, `if constexpr` n'a pas de sens — le compilateur produira une erreur.

```cpp
void process(int x) {
    // ❌ Erreur : x n'est pas une expression constante
    // if constexpr (x > 0) { ... }

    // ✅ Correct — condition constante
    if constexpr (sizeof(int) == 4) {
        // vrai sur la majorité des plateformes
    }
}
```

### Cas d'usage typiques

`if constexpr` est omniprésent dans le code générique moderne. Voici les situations où il excelle.

**Adapter le comportement selon un trait de type :**

```cpp
template <typename Container>  
void log_content(const Container& c) {  
    if constexpr (std::is_same_v<Container, std::string>) {
        std::cout << "Chaîne : \"" << c << "\"\n";
    } else {
        std::cout << "Conteneur de " << c.size() << " éléments\n";
    }
}
```

**Déplier un parameter pack (variadic templates) :**

```cpp
template <typename T, typename... Rest>  
void print_all(const T& first, const Rest&... rest) {  
    std::cout << first;
    if constexpr (sizeof...(rest) > 0) {
        std::cout << ", ";
        print_all(rest...);
    }
}

// print_all(1, "hello", 3.14);  → 1, hello, 3.14
```

**Sélectionner un algorithme selon la taille d'un type :**

```cpp
template <typename T>  
T byte_swap(T val) {  
    if constexpr (sizeof(T) == 1) {
        return val;
    } else if constexpr (sizeof(T) == 2) {
        return __builtin_bswap16(val);
    } else if constexpr (sizeof(T) == 4) {
        return __builtin_bswap32(val);
    } else if constexpr (sizeof(T) == 8) {
        return __builtin_bswap64(val);
    }
}
```

## `if` classique vs `if constexpr` : quand utiliser lequel ?

| Critère | `if` classique | `if constexpr` |  
|---------|---------------|----------------|  
| Évaluation | À l'exécution (runtime) | À la compilation (compile-time) |  
| Condition | Toute expression convertible en `bool` | Expression constante uniquement |  
| Branches | Les deux doivent compiler | Seule la branche retenue est instanciée |  
| Contexte principal | Code applicatif courant | Templates, code générique |  
| Depuis | C (hérité) | C++17 |

En résumé : utilisez `if constexpr` quand la condition est connue à la compilation **et** que la branche éliminée ne compilerait pas (ou ne devrait pas être compilée) pour tous les types possibles. Dans tous les autres cas, le `if` classique suffit — et reste plus simple à lire.

## Pièges courants

### Oublier les accolades

```cpp
if (x > 0)
    std::cout << "positif\n";
    do_something();  // ⚠️ Exécuté inconditionnellement — l'indentation est trompeuse
```

Le compilateur ne tient aucun compte de l'indentation. Sans accolades, seule la première instruction après le `if` est conditionnelle. Avec `-Wall`, certains compilateurs émettent un warning (*misleading indentation*), mais ce n'est pas garanti.

### Confondre `=` et `==`

```cpp
int status = 0;

// ❌ Affectation au lieu de comparaison — status vaut maintenant 1, condition vraie
if (status = 1) {
    std::cout << "Erreur\n";
}
```

Avec `-Wall`, GCC et Clang émettent le warning `suggest parentheses as a truth value`. Une parade classique est de placer la constante à gauche (*Yoda condition*), mais les warnings modernes rendent cette pratique moins nécessaire :

```cpp
// Style Yoda — l'erreur '1 = status' ne compile pas
if (1 == status) { ... }

// Style moderne — on fait confiance aux warnings du compilateur
if (status == 1) { ... }
```

### Utiliser `if constexpr` sans nécessité

```cpp
// ❌ Inutile — les deux branches compilent pour tout type
if constexpr (true) {
    do_something();
}
```

`if constexpr` a un coût en lisibilité (le lecteur s'attend à du code générique). Ne l'utilisez que quand il apporte une valeur réelle — typiquement quand une branche ne compilerait pas sans lui.

### Tester un flottant avec `==`

```cpp
double result = 0.1 + 0.2;

// ❌ Peut être faux à cause de l'arithmétique flottante
if (result == 0.3) { ... }

// ✅ Comparaison avec une tolérance (epsilon)
if (std::abs(result - 0.3) < 1e-9) { ... }
```

Ce piège n'est pas spécifique au `if`, mais c'est dans les conditionnelles qu'il se manifeste. L'arithmétique en virgule flottante est par nature approximative — ne comparez jamais deux flottants avec `==` sauf si vous savez exactement ce que vous faites.
