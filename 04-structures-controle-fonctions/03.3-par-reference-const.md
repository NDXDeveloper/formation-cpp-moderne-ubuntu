# 4.3.3 — Passage par référence constante (`const &`)

## Chapitre 4 · Structures de Contrôle et Fonctions · Niveau Débutant

---

## Principe

Le passage par référence constante combine les avantages des deux modes précédents : comme la référence (`&`), il **évite toute copie** ; comme le passage par valeur, il **garantit que l'original ne sera pas modifié**. C'est le mode de passage **le plus utilisé** en C++ moderne pour les types non primitifs.

```cpp
#include <iostream>
#include <string>

void afficher(const std::string& message) {
    std::cout << message << "\n";
    // message += "!";  // ❌ Erreur de compilation — const interdit la modification
}

int main() {
    std::string salut = "Bonjour";
    afficher(salut);
    std::cout << salut << "\n";  // "Bonjour" — inchangé, garanti par const
}
```

Le qualificateur `const` est une **promesse** faite par la fonction à l'appelant : « je vais lire cette donnée, mais je ne la modifierai pas ». Le compilateur vérifie cette promesse à la compilation — toute tentative de modification produit une erreur.

---

## Ce qui se passe en mémoire

Le mécanisme est identique au passage par référence simple (section 4.3.2) : la fonction reçoit un alias vers l'objet original, sans copie. La seule différence est la **restriction d'accès** imposée par `const`, qui interdit les opérations d'écriture.

```
┌──────────────────────────┐
│  main()                  │
│    salut = "Bonjour"     │  ← variable originale
├──────────────────────────┤
│  afficher()              │
│    message ────────────────→ (alias en lecture seule vers salut)
└──────────────────────────┘
```

Aucune allocation supplémentaire, aucune copie. Le coût du passage est le même qu'un pointeur — typiquement 8 octets sur une architecture 64 bits — quel que soit le type ou la taille de l'objet.

---

## Pourquoi c'est le mode par défaut pour les types non primitifs

Prenons une fonction qui calcule une statistique sur un vecteur d'un million d'éléments :

```cpp
// ❌ Passage par valeur — copie le million d'éléments
double moyenne(std::vector<double> donnees) {
    double somme = 0;
    for (double d : donnees) somme += d;
    return somme / donnees.size();
}

// ✅ Passage par référence constante — zéro copie
double moyenne(const std::vector<double>& donnees) {
    double somme = 0;
    for (double d : donnees) somme += d;
    return somme / donnees.size();
}
```

La seconde version est strictement équivalente en résultat, mais elle ne copie rien. Sur un vecteur d'un million de `double` (8 Mo de données), la différence est considérable. Et comme `const` interdit toute modification, la sécurité sémantique est identique au passage par valeur.

C'est pour cette raison que la recommandation universelle est :

> **Pour les paramètres en lecture seule de types non primitifs, utilisez `const T&`.**

Cette règle est formalisée par les C++ Core Guidelines (F.16) et adoptée par la quasi-totalité des bases de code professionnelles.

---

## Avantage clé : accepter les temporaires

Contrairement à une référence non-const (`T&`, section 4.3.2), une référence constante **peut se lier à un temporaire** (une rvalue). C'est une propriété fondamentale du langage.

```cpp
void traiter(const std::string& s) {
    std::cout << "Longueur : " << s.size() << "\n";
}

int main() {
    std::string nom = "Alice";

    traiter(nom);                        // ✅ lvalue
    traiter("littéral C");               // ✅ temporaire (conversion const char* → std::string)
    traiter(std::string("temporaire"));  // ✅ rvalue explicite
    traiter(nom + " et Bob");            // ✅ résultat d'une expression (temporaire)
}
```

Tous ces appels compilent et fonctionnent correctement. La durée de vie du temporaire est **prolongée** pour durer aussi longtemps que la référence constante qui le lie — c'est-à-dire jusqu'à la fin de l'instruction ou du bloc dans lequel la référence existe.

Si `traiter` prenait un `std::string&` (sans `const`), les trois derniers appels seraient refusés par le compilateur :

```cpp
void traiter_mut(std::string& s) { /* ... */ }

int main() {
    std::string nom = "Alice";
    traiter_mut(nom);                    // ✅ lvalue
    // traiter_mut("littéral");          // ❌ Erreur — rvalue vers ref non-const
    // traiter_mut(nom + " et Bob");     // ❌ Erreur
}
```

Cette restriction est logique : modifier un temporaire qui va être détruit immédiatement n'a aucun sens. `const &` contourne le problème en garantissant qu'il n'y aura pas de modification.

---

## Exemples courants dans la STL et les API

Le passage par `const &` est omniprésent dans les interfaces C++. Quelques exemples que vous rencontrerez dès les premiers projets :

### Constructeurs et méthodes de la STL

```cpp
// Le constructeur de copie de std::vector
std::vector<int> copie(const std::vector<int>& autre);

// std::map::find
iterator find(const key_type& k);

// std::sort avec comparateur personnalisé
template <typename Iter, typename Comp>  
void sort(Iter first, Iter last, const Comp& comp);  
```

### Opérateur `<<` pour l'affichage

La surcharge de `operator<<` pour un type personnalisé prend toujours l'objet en `const &` :

```cpp
struct Point {
    double x, y;
};

std::ostream& operator<<(std::ostream& os, const Point& p) {
    os << "(" << p.x << ", " << p.y << ")";
    return os;
}
```

Le `std::ostream&` est non-const (on écrit dedans), mais le `Point` est `const &` (on le lit seulement).

### Fonctions utilitaires

```cpp
bool contient(const std::string& texte, const std::string& motif) {
    return texte.find(motif) != std::string::npos;
}

double distance(const Point& a, const Point& b) {
    double dx = a.x - b.x;
    double dy = a.y - b.y;
    return std::sqrt(dx * dx + dy * dy);
}

void log_requete(const HttpRequest& req) {
    std::cout << req.method() << " " << req.path() << "\n";
}
```

Le schéma est toujours le même : la fonction lit les données sans les modifier, et `const &` exprime cette intention tout en évitant une copie.

---

## `const &` vs passage par valeur : le cas des petits types

Pour les types primitifs (`int`, `double`, `char`, `bool`, pointeurs bruts), le passage par valeur est aussi efficace, voire plus efficace, que `const &`. Copier un `int` (4 octets) coûte autant que copier une référence (8 octets sur 64-bit), et le passage par valeur évite une indirection mémoire.

```cpp
// ✅ Correct mais inutile — une référence vers un int ajoute une indirection
double carre(const int& x) {
    return x * x;
}

// ✅ Plus simple et potentiellement plus rapide pour un type primitif
double carre(int x) {
    return x * x;
}
```

La règle est donc : `const T&` pour les types « gros » (conteneurs, chaînes, structures complexes), passage par valeur pour les types « petits » (primitifs, itérateurs, petites structs triviales).

Le seuil exact dépend de l'architecture, mais une heuristique courante est : si `sizeof(T)` est inférieur ou égal à deux fois la taille d'un pointeur (16 octets sur 64-bit) **et** que le type est trivially copyable, le passage par valeur est préférable.

---

## `const` en profondeur : ce que `const &` protège réellement

Le qualificateur `const` empêche d'appeler toute méthode ou opération non-`const` sur l'objet. Concrètement, seules les méthodes marquées `const` dans la définition de la classe sont accessibles.

```cpp
void analyser(const std::vector<int>& v) {
    // ✅ Méthodes const — lecture autorisée
    std::cout << v.size() << "\n";
    std::cout << v[0] << "\n";
    std::cout << v.front() << "\n";
    bool vide = v.empty();

    // ❌ Méthodes non-const — modification interdite
    // v.push_back(42);
    // v.clear();
    // v[0] = 99;
}
```

C'est le mécanisme de **const-correctness** du C++ : chaque méthode d'une classe déclare si elle modifie ou non l'objet, et le compilateur utilise cette information pour vérifier les accès. Ce concept sera approfondi au chapitre 6 (Classes et Encapsulation).

### Attention : `const` est superficiel

`const` protège l'objet directement référencé, mais pas nécessairement les objets qu'il **contient** indirectement via des pointeurs bruts :

```cpp
struct Buffer {
    int* data;
    int size;
};

void lire(const Buffer& buf) {
    // buf.size = 0;       // ❌ Interdit — modification du membre direct
    // buf.data = nullptr;  // ❌ Interdit — modification du pointeur

    buf.data[0] = 999;     // ⚠️ Autorisé ! Le pointeur est const, pas les données pointées
}
```

Ce phénomène est lié à la nature du `const` en C++ : il s'applique au niveau supérieur (le pointeur lui-même), pas aux données vers lesquelles le pointeur pointe. En pratique, ce problème est rare avec les conteneurs standard (`std::vector`, `std::string`, etc.) dont les méthodes `const` propagent correctement la constance. Il se manifeste surtout avec des pointeurs bruts — une raison de plus de les éviter dans le code moderne.

---

## Pièges courants

### Stocker une référence constante vers un temporaire au-delà de sa durée de vie

La prolongation de durée de vie d'un temporaire ne s'applique qu'aux références locales directes. Si la référence est stockée dans un membre de classe ou passée à une autre fonction qui la conserve, le temporaire est détruit et la référence devient pendante :

```cpp
class Logger {
    const std::string& prefix_;  // ⚠️ Référence — ne possède pas la donnée
public:
    Logger(const std::string& prefix) : prefix_(prefix) {}
    void log(const std::string& msg) {
        std::cout << prefix_ << ": " << msg << "\n";
    }
};

int main() {
    Logger logger("APP");  // ❌ "APP" est un temporaire std::string
    // Le temporaire est détruit à la fin de cette instruction
    logger.log("Démarrage");  // Comportement indéfini — prefix_ est un dangling reference
}
```

La solution est de stocker une **copie** (`std::string prefix_`) plutôt qu'une référence. En règle générale, les membres de classe ne devraient quasiment jamais être des références — c'est un sujet abordé au chapitre 6.

### Oublier `const` quand la fonction ne modifie pas l'argument

```cpp
// ❌ Manque de const — interdit les temporaires et ne documente pas l'intention
void afficher(std::string& msg) {
    std::cout << msg << "\n";
}

int main() {
    // afficher("Hello");  // ❌ Erreur — temporaire vers ref non-const
    std::string s = "Hello";
    afficher(s);            // ✅ Compile, mais l'interface est trompeuse
}
```

L'absence de `const` est un signal pour le lecteur : « cette fonction va modifier l'argument ». Si ce n'est pas le cas, c'est une interface mensongère. Ajoutez systématiquement `const` quand la fonction ne modifie pas le paramètre.

### Confusion entre `const T&` et `const T`

Dans une déclaration de paramètre, `const T` et `const T&` sont fondamentalement différents :

```cpp
void f(const std::string  s);   // Copie, mais copie non modifiable  
void g(const std::string& s);   // Pas de copie, lecture seule sur l'original  
```

La première crée une copie puis empêche de la modifier — ce qui est rarement utile. La seconde évite la copie — c'est presque toujours ce qu'on veut. Attention à ne pas confondre les deux, surtout en lisant du code existant.

---

## Résumé

| Aspect | Passage par référence constante |  
|--------|--------------------------------|  
| Syntaxe | `void f(const T& x)` |  
| Copie de l'argument ? | Non — alias en lecture seule |  
| Fonction modifie l'original ? | Non — garanti par `const` |  
| Accepte un temporaire ? | Oui — durée de vie prolongée |  
| Adapté aux types primitifs ? | Peu d'intérêt (préférer par valeur) |  
| Adapté aux types complexes ? | Oui — c'est le mode par défaut |  
| Usage principal | Paramètre en lecture seule (in) |

## Ce qui suit

Hérité du C, il reste utile dans des cas précis — paramètre optionnel (`nullptr`), interopérabilité avec des API C, et manipulation de tableaux dynamiques — mais il est de plus en plus remplacé par les références et les smart pointers dans le code C++ moderne.
