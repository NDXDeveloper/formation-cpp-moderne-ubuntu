# 4.3.4 — Passage par pointeur (`*`)

## Chapitre 4 · Structures de Contrôle et Fonctions · Niveau Débutant

---

## Principe

Quand un paramètre est passé **par pointeur**, la fonction reçoit l'**adresse mémoire** de la variable. Pour accéder à la donnée ou la modifier, la fonction doit **déréférencer** le pointeur avec l'opérateur `*` ou `->`.

```cpp
#include <iostream>

void doubler(int* ptr) {
    *ptr *= 2;  // Déréférencement : modifie la valeur à l'adresse pointée
}

int main() {
    int n = 5;
    doubler(&n);  // L'opérateur & donne l'adresse de n
    std::cout << "n = " << n << "\n";  // n = 10
}
```

Du côté de l'appelant, le passage par pointeur est **explicite** : l'opérateur `&` devant l'argument signale visuellement qu'une adresse est transmise. C'est une différence notable avec le passage par référence, où rien ne distingue syntaxiquement l'appel d'un passage par valeur.

---

## Ce qui se passe en mémoire

Au moment de l'appel `doubler(&n)`, l'adresse de `n` est copiée dans le paramètre `ptr`. Le pointeur lui-même est passé par valeur (c'est une copie d'adresse), mais il permet d'accéder — et de modifier — l'objet original via déréférencement.

```
┌──────────────────────────────┐
│  main()                      │
│    n = 5     [adresse: 0x80] │  ← variable originale
├──────────────────────────────┤
│  doubler()                   │
│    ptr = 0x80                │  ← copie de l'adresse de n
│    *ptr ─────────────────────────→ accède à n (0x80)
└──────────────────────────────┘
```

Après `*ptr *= 2` :

```
┌──────────────────────────────┐
│  main()                      │
│    n = 10    [adresse: 0x80] │  ← modifié via *ptr
├──────────────────────────────┤
│  doubler()                   │
│    ptr = 0x80                │
└──────────────────────────────┘
```

Le pointeur `ptr` est une variable locale à `doubler` — le réaffecter (`ptr = autre_adresse`) ne changerait pas `n`. Seul le déréférencement (`*ptr = ...`) modifie la donnée originale.

---

## Quand le passage par pointeur est justifié en C++ moderne

Les sections précédentes ont montré que les références couvrent la majorité des besoins. Le passage par pointeur reste cependant le bon choix dans quelques situations précises.

### 1. Paramètre optionnel : `nullptr` comme valeur sentinelle

C'est le cas d'usage principal en C++ moderne. Un pointeur peut légitimement valoir `nullptr`, signifiant « pas de donnée fournie ». Une référence ne le peut pas.

```cpp
#include <iostream>
#include <string>

void enregistrer(const std::string& nom, const std::string* commentaire) {
    std::cout << "Enregistrement de " << nom;
    if (commentaire != nullptr) {
        std::cout << " (commentaire : " << *commentaire << ")";
    }
    std::cout << "\n";
}

int main() {
    std::string note = "Urgent";
    enregistrer("Alice", &note);    // Avec commentaire
    enregistrer("Bob", nullptr);    // Sans commentaire
}
```

Le `nullptr` joue ici le rôle de « paramètre absent ». La fonction vérifie la validité du pointeur avant de le déréférencer — c'est une obligation absolue.

> 💡 En C++ moderne, `std::optional<T>` (section 12.2) est souvent une alternative plus expressive pour les paramètres optionnels en entrée. Le pointeur reste pertinent quand l'objet existe déjà en mémoire et qu'on veut simplement signaler qu'il peut ou non être fourni, sans créer de copie.

### 2. Interopérabilité avec du code C et des API système

Les API C, les appels système POSIX et de nombreuses bibliothèques tierces utilisent exclusivement des pointeurs. Communiquer avec elles impose de passer des adresses.

```cpp
#include <cstdio>
#include <cstring>

// API C standard
void lire_fichier(const char* chemin) {
    FILE* f = fopen(chemin, "r");
    if (f == nullptr) {
        perror("fopen");
        return;
    }
    // ...
    fclose(f);
}
```

```cpp
#include <sys/socket.h>
#include <netinet/in.h>

// API POSIX — attend des pointeurs vers des structures
void configurer_socket(int sockfd) {
    struct sockaddr_in addr;
    std::memset(&addr, 0, sizeof(addr));
    addr.sin_family = AF_INET;
    addr.sin_port = htons(8080);
    addr.sin_addr.s_addr = INADDR_ANY;

    // bind() attend un pointeur vers sockaddr
    bind(sockfd, reinterpret_cast<struct sockaddr*>(&addr), sizeof(addr));
}
```

Ces API ne changeront pas — le passage par pointeur restera nécessaire pour s'interfacer avec elles. Le chapitre 22 (Networking) et le chapitre 43 (Interopérabilité) couvrent ces situations en détail.

### 3. Tableaux et buffers

Le passage d'un tableau brut à une fonction se fait nécessairement par pointeur, car un tableau *decay* en pointeur vers son premier élément au moment de l'appel :

```cpp
void remplir(int* buffer, std::size_t taille) {
    for (std::size_t i = 0; i < taille; ++i) {
        buffer[i] = static_cast<int>(i * i);
    }
}

int main() {
    int tableau[10];
    remplir(tableau, 10);  // 'tableau' decay en int*
}
```

En C++ moderne, `std::span` (C++20, section 13.5) est l'alternative recommandée pour ce pattern : il encapsule le pointeur et la taille dans un seul objet, éliminant une source classique d'erreurs.

```cpp
#include <span>

// ✅ C++20 — plus sûr, la taille est portée par le type
void remplir(std::span<int> buffer) {
    for (std::size_t i = 0; i < buffer.size(); ++i) {
        buffer[i] = static_cast<int>(i * i);
    }
}
```

### 4. Structures de données avec liens entre nœuds

Les structures de données chaînées (listes, arbres, graphes) utilisent des pointeurs pour relier les nœuds entre eux. Les smart pointers (chapitre 9) remplacent les pointeurs bruts pour la gestion de la mémoire, mais le concept de « lien vers un nœud » reste fondamentalement un pointeur.

```cpp
struct TreeNode {
    int value;
    TreeNode* left;   // Peut être nullptr (pas d'enfant gauche)
    TreeNode* right;  // Peut être nullptr (pas d'enfant droit)
};

void parcours_infixe(const TreeNode* node) {
    if (node == nullptr) return;
    parcours_infixe(node->left);
    std::cout << node->value << " ";
    parcours_infixe(node->right);
}
```

## Pointeur `const` : les quatre combinaisons

Le qualificateur `const` peut s'appliquer au pointeur lui-même, aux données pointées, ou aux deux. Chaque combinaison a une sémantique différente :

```cpp
int valeur = 42;
```

### Pointeur vers donnée modifiable

```cpp
int* ptr = &valeur;
*ptr = 10;    // ✅ Modification de la donnée
ptr = nullptr; // ✅ Réaffectation du pointeur
```

### Pointeur vers donnée constante

C'est l'équivalent de `const T&` pour les pointeurs — lecture seule sur la donnée, mais le pointeur peut être réaffecté :

```cpp
const int* ptr = &valeur;
// *ptr = 10;    // ❌ Donnée protégée par const
ptr = nullptr;    // ✅ Le pointeur lui-même peut changer
```

C'est la forme la plus courante pour les paramètres en lecture seule passés par pointeur, typiquement dans les API C :

```cpp
// strlen ne modifie pas la chaîne — const char*
std::size_t strlen(const char* s);

// memcpy : destination modifiable, source en lecture seule
void* memcpy(void* dest, const void* src, std::size_t n);
```

### Pointeur constant vers donnée modifiable

Le pointeur ne peut pas être réaffecté, mais la donnée peut être modifiée :

```cpp
int* const ptr = &valeur;
*ptr = 10;    // ✅ Modification de la donnée
// ptr = nullptr; // ❌ Le pointeur est const
```

Ce cas est rare en pratique pour les paramètres de fonctions, car le pointeur étant une copie locale, sa réaffectation n'affecte pas l'appelant.

### Pointeur constant vers donnée constante

Double verrouillage — ni le pointeur ni la donnée ne peuvent être modifiés :

```cpp
const int* const ptr = &valeur;
// *ptr = 10;    // ❌
// ptr = nullptr; // ❌
```

### Astuce de lecture

La règle de lecture est de lire la déclaration **de droite à gauche** :

| Déclaration | Lecture droite → gauche |  
|-------------|------------------------|  
| `int* ptr` | ptr est un pointeur vers int |  
| `const int* ptr` | ptr est un pointeur vers int constant |  
| `int* const ptr` | ptr est un pointeur constant vers int |  
| `const int* const ptr` | ptr est un pointeur constant vers int constant |

## Vérification de `nullptr` : une obligation

Un pointeur peut être nul. Déréférencer un pointeur nul est un **comportement indéfini** — en pratique, un crash (`SIGSEGV` sur Linux). La vérification doit être systématique :

```cpp
void traiter(const Config* config) {
    // ❌ Dangereux — config peut être nullptr
    // std::cout << config->timeout << "\n";

    // ✅ Vérification avant déréférencement
    if (config == nullptr) {
        std::cerr << "Erreur : configuration absente\n";
        return;
    }
    std::cout << "Timeout : " << config->timeout << "\n";
}
```

Depuis C++11, utilisez toujours `nullptr` au lieu de `NULL` ou `0` pour représenter un pointeur nul. `nullptr` a un type dédié (`std::nullptr_t`) qui évite les ambiguïtés de surcharge que `NULL` (défini comme `0`) pouvait provoquer :

```cpp
void f(int x)    { std::cout << "int\n"; }  
void f(int* ptr) { std::cout << "pointeur\n"; }  

int main() {
    f(0);        // Appelle f(int) — ambiguïté historique avec NULL
    f(nullptr);  // Appelle f(int*) — sans ambiguïté
}
```

## Passage par pointeur vs passage par référence

Le tableau suivant résume les critères de choix entre les deux modes. La colonne de droite indique la recommandation pour du code C++ moderne.

| Critère | Référence (`T&` / `const T&`) | Pointeur (`T*` / `const T*`) | Recommandation |  
|---------|-------------------------------|------------------------------|----------------|  
| Peut être nul | Non | Oui | Pointeur si nullable |  
| Syntaxe d'accès | Directe (`x.member`) | Déréférencement (`x->member`) | Référence (plus simple) |  
| Signal au site d'appel | Aucun (`f(x)`) | Visible (`f(&x)`) | Pointeur (plus explicite) |  
| Réaffectable | Non | Oui | Selon le besoin |  
| Interop C | Non | Oui | Pointeur (obligatoire) |  
| Smart pointers | Non applicable | Remplacent les pointeurs bruts | Smart pointers (chapitre 9) |

La visibilité au site d'appel est un argument parfois avancé en faveur des pointeurs : voir `f(&x)` rappelle qu'une adresse est transmise et que `x` pourrait être modifié. C'est un argument stylistique — certaines équipes le valorisent, d'autres préfèrent la simplicité des références. Les C++ Core Guidelines tranchent en faveur des références pour les paramètres non-nullables.

## Pièges courants

### Déréférencer sans vérifier

```cpp
void process(int* data) {
    // ❌ Crash si data est nullptr
    std::cout << *data << "\n";
}
```

Chaque pointeur reçu en paramètre doit être considéré comme potentiellement nul, sauf si une convention claire (documentation, contrats C++26) l'exclut.

### Retourner l'adresse d'une variable locale

Comme pour les références (section 4.3.2), retourner un pointeur vers une variable locale crée un **dangling pointer** — la mémoire est libérée quand la fonction retourne :

```cpp
// ❌ Dangling pointer
int* creer() {
    int local = 42;
    return &local;  // ⚠️ Warning du compilateur
}

int main() {
    int* ptr = creer();
    std::cout << *ptr << "\n";  // Comportement indéfini
}
```

Si la fonction doit créer un objet et en transférer la propriété, utilisez un smart pointer :

```cpp
#include <memory>

// ✅ Transfert de propriété via std::unique_ptr
std::unique_ptr<int> creer() {
    return std::make_unique<int>(42);
}
```

### Confusion entre pointeur et tableau

Un paramètre `int*` peut pointer vers un seul entier ou vers le premier élément d'un tableau. Rien dans le type ne distingue les deux cas — la taille est perdue :

```cpp
// Combien d'éléments ? 1 ? 100 ? Impossible à savoir depuis la signature
void traiter(int* data) { /* ... */ }
```

C'est un défaut fondamental de l'héritage C. Les solutions modernes sont `std::span` (C++20) pour les vues non-propriétaires, ou `std::vector` / `std::array` passés par référence pour les conteneurs propriétaires.

### Arithmétique de pointeur accidentelle

```cpp
void f(int* ptr) {
    ptr++;     // ⚠️ Avance le pointeur local — n'affecte pas l'appelant
    *ptr = 0;  // Écrit à l'adresse suivante — potentiellement hors limites
}
```

L'arithmétique de pointeur est puissante mais dangereuse. En dehors de l'itération explicite sur un buffer (et même dans ce cas, les itérateurs ou `std::span` sont préférables), elle est à éviter.

## Vers le code moderne : réduire l'usage des pointeurs bruts

Le C++ moderne tend à **minimiser** l'usage des pointeurs bruts dans le code applicatif. Voici les alternatives à privilégier :

| Besoin | Pointeur brut (legacy) | Alternative moderne |  
|--------|----------------------|---------------------|  
| Paramètre in-out | `T*` | `T&` |  
| Paramètre en lecture seule | `const T*` | `const T&` |  
| Paramètre optionnel | `T*` (nullptr si absent) | `std::optional<T>` ou `const T*` |  
| Propriété exclusive | `T*` + `new`/`delete` | `std::unique_ptr<T>` |  
| Propriété partagée | `T*` + comptage manuel | `std::shared_ptr<T>` |  
| Vue sur un tableau | `T*` + `size` | `std::span<T>` (C++20) |

Les pointeurs bruts restent légitimes pour l'interopérabilité C, les structures de données internes, et le code de très bas niveau. Mais dans le code applicatif quotidien, les alternatives ci-dessus sont presque toujours préférables — plus sûres, plus expressives, et sans surcoût à l'exécution pour la plupart d'entre elles.

Les smart pointers sont couverts en détail au chapitre 9.

---

## Résumé

| Aspect | Passage par pointeur |  
|--------|---------------------|  
| Syntaxe | `void f(T* ptr)` / `void f(const T* ptr)` |  
| Copie de l'argument ? | Copie de l'adresse (8 octets), pas de l'objet |  
| Fonction modifie l'original ? | Oui (via `*ptr`), sauf si `const T*` |  
| Peut être nul ? | Oui — vérification obligatoire avant déréférencement |  
| Signal au site d'appel | Oui — `f(&x)` rend le passage visible |  
| Usage principal en C++ moderne | Paramètre nullable, interop C, structures chaînées |  
| Danger principal | Déréférencement de `nullptr`, dangling pointer |
