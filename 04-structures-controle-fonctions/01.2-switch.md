# 4.1.2 — `switch` et `switch` avec initialisation (C++17)

## Chapitre 4 · Structures de Contrôle et Fonctions · Niveau Débutant

---

## Le `switch` : aiguillage sur une valeur discrète

Le `switch` est une alternative au chaînage `if` / `else if` quand on compare **une même expression** à plusieurs **constantes entières**. Il produit un code plus lisible et, dans certains cas, plus performant : le compilateur peut générer une *jump table* au lieu d'une suite de comparaisons séquentielles.

### Syntaxe de base

```cpp
#include <iostream>

int main() {
    int day = 3;

    switch (day) {
        case 1:
            std::cout << "Lundi\n";
            break;
        case 2:
            std::cout << "Mardi\n";
            break;
        case 3:
            std::cout << "Mercredi\n";
            break;
        case 4:
            std::cout << "Jeudi\n";
            break;
        case 5:
            std::cout << "Vendredi\n";
            break;
        default:
            std::cout << "Week-end\n";
            break;
    }
}
```

L'expression dans le `switch(...)` est évaluée une seule fois. Le programme saute directement au `case` correspondant. Si aucun `case` ne correspond, le bloc `default` est exécuté (s'il existe).

### Types autorisés dans l'expression

L'expression du `switch` doit être d'un type **entier ou énuméré**. Concrètement, les types suivants sont acceptés : `int`, `short`, `long`, `char`, `unsigned`, `enum`, `enum class`, et tout type implicitement convertible en entier. En revanche, les flottants (`float`, `double`), les chaînes (`std::string`) et les types personnalisés ne sont **pas autorisés** — il faut utiliser un `if` / `else if` dans ces cas.

```cpp
enum class Color { Red, Green, Blue };

Color c = Color::Green;

switch (c) {
    case Color::Red:
        std::cout << "Rouge\n";
        break;
    case Color::Green:
        std::cout << "Vert\n";
        break;
    case Color::Blue:
        std::cout << "Bleu\n";
        break;
}
```

Les `enum class` (enums fortement typés, C++11) sont le compagnon naturel du `switch`. Le compilateur peut même émettre un warning si un cas n'est pas couvert, à condition de ne pas mettre de `default` (voir la section sur les bonnes pratiques plus bas).

## Le `break` et le piège du *fall-through*

Contrairement à ce que l'indentation suggère, le `switch` en C++ n'exécute pas uniquement le bloc du `case` correspondant. Par défaut, une fois qu'un `case` est atteint, **l'exécution continue dans les `case` suivants** jusqu'à rencontrer un `break` ou la fin du `switch`. C'est le comportement de *fall-through*, hérité du C.

```cpp
int x = 2;

switch (x) {
    case 1:
        std::cout << "un\n";
    case 2:
        std::cout << "deux\n";   // ← exécution commence ici
    case 3:
        std::cout << "trois\n";  // ← exécuté aussi (fall-through)
    default:
        std::cout << "autre\n";  // ← exécuté aussi (fall-through)
}
// Sortie :
// deux
// trois
// autre
```

Ce comportement est la source d'innombrables bugs. La règle est simple : **terminez chaque `case` par un `break`** sauf si le fall-through est intentionnel.

### `[[fallthrough]]` : documenter l'intention (C++17)

Dans les rares cas où le fall-through est voulu, C++17 fournit l'attribut `[[fallthrough]]` pour indiquer explicitement au compilateur (et au lecteur) que l'absence de `break` est **délibérée**. Sans cet attribut, les compilateurs modernes avec `-Wimplicit-fallthrough` émettent un warning.

```cpp
#include <iostream>

int main() {
    int level = 2;

    switch (level) {
        case 3:
            std::cout << "Initialisation avancée\n";
            [[fallthrough]];
        case 2:
            std::cout << "Initialisation standard\n";
            [[fallthrough]];
        case 1:
            std::cout << "Initialisation minimale\n";
            break;
        default:
            std::cout << "Niveau inconnu\n";
            break;
    }
}
// Avec level = 2 :
// Initialisation standard
// Initialisation minimale
```

Ce pattern est parfois utilisé pour des initialisations cumulatives (comme ici) ou pour regrouper des cas partageant un traitement commun. Même dans ces situations, `[[fallthrough]]` est obligatoire pour supprimer le warning — c'est une bonne chose, car cela force la documentation de l'intention.

## Regrouper des cas

Quand plusieurs valeurs doivent produire le même traitement, on empile les `case` sans instruction entre eux. Ce n'est techniquement pas du fall-through (il n'y a pas de code entre les labels), donc `[[fallthrough]]` n'est pas nécessaire.

```cpp
char c = 'b';

switch (c) {
    case 'a':
    case 'e':
    case 'i':
    case 'o':
    case 'u':
        std::cout << "Voyelle\n";
        break;
    default:
        std::cout << "Consonne (ou autre)\n";
        break;
}
```

## `switch` avec initialisation (C++17)

Comme pour le `if` (section 4.1.1), C++17 permet de déclarer une variable dans le `switch`, en limitant sa portée au bloc entier. La syntaxe est :

```
switch (initialisation; expression) { ... }
```

### Avant C++17

```cpp
auto result = parse_command(input);  
switch (result.code) {  
    case 0:
        execute(result.payload);
        break;
    case 1:
        std::cerr << "Commande invalide\n";
        break;
    default:
        std::cerr << "Erreur inconnue\n";
        break;
}
// 'result' est toujours dans la portée — potentiellement gênant
```

### Avec C++17

```cpp
switch (auto result = parse_command(input); result.code) {
    case 0:
        execute(result.payload);
        break;
    case 1:
        std::cerr << "Commande invalide\n";
        break;
    default:
        std::cerr << "Erreur " << result.code << "\n";
        break;
}
// 'result' n'existe plus ici
```

La variable `result` est accessible dans tous les `case` et dans le `default`, mais pas en dehors du `switch`. C'est particulièrement utile quand la valeur testée est un champ d'une structure retournée par une fonction, ou le résultat d'une conversion.

Autre exemple courant avec une conversion :

```cpp
switch (int n = std::stoi(token); n) {
    case 200:
        std::cout << "OK\n";
        break;
    case 404:
        std::cout << "Not Found\n";
        break;
    case 500:
        std::cout << "Server Error\n";
        break;
    default:
        std::cout << "Code HTTP : " << n << "\n";
        break;
}
```

## Portée des variables dans un `switch`

Un piège classique du `switch` concerne la **déclaration de variables à l'intérieur des `case`**. En C++, tous les `case` partagent la même portée (le bloc du `switch`). Déclarer une variable dans un `case` sans accolades peut provoquer un saut par-dessus son initialisation, ce que le compilateur refuse.

```cpp
switch (x) {
    case 1:
        int val = 42;  // ❌ Erreur : le case 2 saute par-dessus cette initialisation
        process(val);
        break;
    case 2:
        // 'val' est techniquement dans la portée mais n'a pas été initialisé
        break;
}
```

**La solution : des accolades pour créer une portée locale.**

```cpp
switch (x) {
    case 1: {
        int val = 42;  // ✅ Portée limitée à ce bloc
        process(val);
        break;
    }
    case 2: {
        int val = 100; // ✅ Autre variable, autre portée
        process(val);
        break;
    }
}
```

C'est une bonne pratique d'ajouter systématiquement des accolades à chaque `case` dès qu'il contient une déclaration de variable.

## `switch` vs `if` / `else if` : comment choisir ?

| Critère | `switch` | `if` / `else if` |  
|---------|----------|-------------------|  
| Type de l'expression | Entier ou enum uniquement | Toute expression booléenne |  
| Nature des conditions | Comparaison à des constantes | Conditions arbitraires (plages, fonctions, etc.) |  
| Performance potentielle | Jump table possible (O(1)) | Comparaisons séquentielles (O(n)) |  
| Lisibilité | Excellente pour 4+ cas discrets | Meilleure pour 2-3 cas ou conditions complexes |  
| Détection de cas manquants | Warning si enum sans `default` | Aucune aide du compilateur |

**Règle pratique :** utilisez `switch` quand vous comparez une valeur à un ensemble fini de constantes connues (codes d'erreur, états d'une machine à états, valeurs d'enum). Utilisez `if` / `else if` pour tout le reste — plages de valeurs, conditions combinées, comparaisons de chaînes, résultats de fonctions.

## Bonnes pratiques

### Toujours inclure un `default` — sauf avec les enums

Pour les valeurs numériques, un `default` sert de filet de sécurité face aux valeurs imprévues. En revanche, pour les `enum class`, **omettre le `default`** est souvent préférable : le compilateur émet alors un warning si un cas n'est pas couvert, ce qui protège contre les oublis lors de l'ajout d'une nouvelle valeur à l'enum.

```cpp
enum class State { Idle, Running, Paused, Stopped };

State s = get_state();

// ✅ Pas de default — le compilateur prévient si un cas est oublié
switch (s) {
    case State::Idle:
        start();
        break;
    case State::Running:
        monitor();
        break;
    case State::Paused:
        resume();
        break;
    case State::Stopped:
        cleanup();
        break;
    // Si on ajoute State::Error plus tard, le compilateur émet un warning ici
}
```

Si vous ajoutez un `default` à ce `switch`, le compilateur considère que tous les cas sont couverts et ne signale plus les valeurs manquantes — vous perdez cette protection.

### Activer `-Wimplicit-fallthrough`

Ce flag (inclus dans `-Wextra` sur GCC et Clang) signale tout fall-through non annoté par `[[fallthrough]]`. C'est une des premières options à activer sur tout projet sérieux.

### Garder les `case` courts

Un `case` contenant plus de quelques lignes rend le `switch` difficile à lire. Si le traitement est complexe, extrayez-le dans une fonction dédiée :

```cpp
switch (event.type) {
    case EventType::Connect:
        handle_connect(event);
        break;
    case EventType::Disconnect:
        handle_disconnect(event);
        break;
    case EventType::Message:
        handle_message(event);
        break;
}
```

Ce style transforme le `switch` en table de dispatch lisible d'un coup d'œil, où chaque cas tient sur deux lignes.
