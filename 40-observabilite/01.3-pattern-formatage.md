# 40.1.3 — Pattern de formatage

## Contrôler ce que contient chaque ligne de log

---

## Introduction

Les deux sections précédentes ont posé les fondations : les niveaux de sévérité filtrent *quels* messages sont émis, les sinks déterminent *où* ils atterrissent. Cette section traite du troisième axe : le **format** — *comment* chaque message est structuré.

Le format d'un log détermine son utilité. Un message mal formaté est difficile à lire pour un humain et impossible à parser pour un outil d'agrégation. Un message bien formaté contient l'horodatage précis, le contexte d'émission, le niveau de sévérité, et le contenu — le tout dans un format cohérent et prévisible.

spdlog utilise un système de **patterns** inspiré de log4j et de la librairie `fmt`. Un pattern est une chaîne contenant des **flags** (préfixés par `%`) que spdlog remplace par des valeurs dynamiques au moment de l'émission du message.

---

## Syntaxe des patterns

Un pattern se configure sur un logger ou sur un sink individuel :

```cpp
// Sur le logger (s'applique à tous ses sinks sans pattern propre)
logger->set_pattern("[%Y-%m-%d %H:%M:%S.%e] [%n] [%^%l%$] %v");

// Sur un sink spécifique (surcharge le pattern du logger)
console_sink->set_pattern("[%H:%M:%S] [%^%l%$] %v");  
file_sink->set_pattern("[%Y-%m-%d %H:%M:%S.%e] [%n] [%l] [pid:%P tid:%t] %v");  
```

Un même logger peut ainsi produire des formats différents selon la destination : un format compact et colorisé sur la console, un format complet et parsable dans le fichier.

---

## Référence des flags

### Horodatage

| Flag | Description | Exemple |
|------|-------------|---------|
| `%Y` | Année (4 chiffres) | `2026` |
| `%m` | Mois (01-12) | `03` |
| `%d` | Jour du mois (01-31) | `18` |
| `%H` | Heure (00-23) | `14` |
| `%M` | Minute (00-59) | `23` |
| `%S` | Seconde (00-59) | `07` |
| `%e` | Millisecondes (000-999) | `412` |
| `%f` | Microsecondes (000000-999999) | `412583` |
| `%F` | Nanosecondes (000000000-999999999) | `412583921` |
| `%T` | Raccourci pour `%H:%M:%S` | `14:23:07` |
| `%z` | Offset UTC | `+0200` |
| `%E` | Epoch (secondes depuis 1970-01-01) | `1774048987` |

La précision de l'horodatage est un choix important. Les millisecondes (`%e`) suffisent pour la majorité des applications. Les microsecondes (`%f`) sont utiles pour le profiling de performance et le diagnostic de problèmes de latence. Les nanosecondes (`%F`) sont rarement nécessaires hors du HFT et des systèmes temps réel.

L'horodatage est en heure locale par défaut. Pour utiliser l'UTC (recommandé dans les environnements multi-timezone et les systèmes distribués) :

```cpp
spdlog::set_default_logger(logger);  
spdlog::default_logger()->sinks()[0]->set_pattern("[%Y-%m-%d %H:%M:%S.%e UTC] [%l] %v");  

// Ou globalement via la configuration du pattern sur le logger
```

spdlog ne fournit pas de flag dédié pour forcer UTC dans le pattern. L'approche standard est de configurer la timezone du processus via la variable d'environnement `TZ=UTC` au démarrage, ce qui affecte tous les horodatages :

```bash
TZ=UTC ./syswatch
```

Ou dans le fichier unit systemd :

```ini
[Service]
Environment=TZ=UTC
```

### Identification du message

| Flag | Description | Exemple |
|------|-------------|---------|
| `%n` | Nom du logger | `syswatch` |
| `%l` | Niveau (texte complet) | `info` |
| `%L` | Niveau (lettre unique) | `I` |
| `%^...%$` | Colorisation du texte entre les marqueurs | (codes ANSI) |
| `%v` | Le contenu du message lui-même | `Connection established` |

Le flag `%v` est toujours présent — c'est le message que vous passez à `spdlog::info(...)`. Tous les autres flags ajoutent du contexte autour de ce message.

### Contexte d'exécution

| Flag | Description | Exemple |
|------|-------------|---------|
| `%t` | Thread ID | `14352` |
| `%P` | Process ID | `8847` |
| `%s` | Nom du fichier source (basename) | `server.cpp` |
| `%g` | Chemin complet du fichier source | `/home/user/src/server.cpp` |
| `%#` | Numéro de ligne | `142` |
| `%!` | Nom de la fonction | `handle_request` |

Les flags de source location (`%s`, `%#`, `%!`) nécessitent l'utilisation des macros de logging plutôt que des fonctions directes :

```cpp
// Les fonctions directes n'incluent PAS la source location
spdlog::info("message");  // %s, %#, %! seront vides

// La macro SPDLOG_INFO inclut automatiquement la source location
SPDLOG_INFO("message");   // %s=server.cpp, %#=42, %!=main
```

Les macros `SPDLOG_TRACE`, `SPDLOG_DEBUG`, `SPDLOG_INFO`, `SPDLOG_WARN`, `SPDLOG_ERROR` et `SPDLOG_CRITICAL` capturent `__FILE__`, `__LINE__` et `__func__` au point d'appel et les passent à spdlog. Le coût est négligeable — ce sont des constantes à la compilation.

L'utilisation systématique des macros vs des fonctions est un choix de projet. Les macros ajoutent du contexte précieux pour le débogage mais produisent des lignes de log plus longues. Un compromis courant est d'utiliser les macros pour `debug` et au-dessus, et les fonctions pour `trace` :

```cpp
// trace : volume élevé, pas besoin de source location à chaque ligne
spdlog::trace("loop iteration i={}", i);

// debug et au-dessus : source location utile pour le diagnostic
SPDLOG_DEBUG("Config loaded: {} rules", rules.size());  
SPDLOG_INFO("Server started on port {}", port);  
SPDLOG_ERROR("Failed to connect: {}", ec.message());  
```

### Alignement et padding

| Flag | Description | Exemple |
|------|-------------|---------|
| `%<width><flag>` | Alignement à droite | `%8l` → `    info` |
| `%-<width><flag>` | Alignement à gauche | `%-8l` → `info    ` |

L'alignement produit des colonnes visuellement régulières dans les logs, ce qui facilite la lecture humaine :

```
[14:23:07.412] [info    ] Server started
[14:23:07.415] [debug   ] Loading config
[14:23:07.416] [warning ] Deprecated option used
[14:23:07.420] [error   ] Connection refused
```

Le pattern correspondant est `[%T.%e] [%-8l] %v`.

## Patterns recommandés par contexte

Le bon pattern dépend de l'usage. Voici quatre configurations éprouvées.

### Développement local

```cpp
logger->set_pattern("[%T.%e] [%^%-8l%$] [%s:%#] %v");
```

Résultat :

```
[14:23:07.412] [info    ] [main.cpp:42] Server started on port 8080
[14:23:07.415] [debug   ] [config.cpp:87] Loaded 12 rules from config.yaml
[14:23:07.420] [error   ] [database.cpp:156] Connection refused: timeout
```

Ce format privilégie la lisibilité et le débogage. L'heure sans date suffit pour une session de développement. La source location (`%s:%#`) permet de retrouver immédiatement la ligne de code. La colorisation (`%^...%$`) distingue visuellement les niveaux. Pas de nom de logger ni de PID — inutiles en développement local.

Ce pattern nécessite l'utilisation des macros `SPDLOG_INFO`, `SPDLOG_DEBUG`, etc. pour que `%s` et `%#` soient renseignés.

### Production (fichier)

```cpp
file_sink->set_pattern("[%Y-%m-%d %H:%M:%S.%e] [%n] [%l] [pid:%P tid:%t] %v");
```

Résultat :

```
[2026-03-18 14:23:07.412] [syswatch] [info] [pid:8847 tid:14352] Server started on port 8080
[2026-03-18 14:23:07.415] [network] [debug] [pid:8847 tid:14358] Connection accepted from 10.0.1.42
[2026-03-18 14:23:07.420] [database] [error] [pid:8847 tid:14360] Query timeout after 5032ms
```

Ce format est conçu pour l'investigation post-mortem. La date complète est indispensable quand les logs couvrent plusieurs jours. Le nom du logger identifie le composant. Le PID distingue les instances si plusieurs processus loguent dans le même fichier (ou après un restart). Le thread ID est essentiel pour reconstituer le flux d'exécution dans une application multi-thread.

La source location est volontairement omise dans le pattern de production : elle augmente le volume de logs et expose la structure interne du code. Si vous la souhaitez, ajoutez-la uniquement pour les niveaux `error` et `critical` via un sink séparé.

### Docker / Kubernetes (console)

```cpp
console_sink->set_pattern("[%Y-%m-%dT%H:%M:%S.%e%z] [%l] [%n] %v");
```

Résultat :

```
[2026-03-18T14:23:07.412+0200] [info] [syswatch] Server started on port 8080
[2026-03-18T14:23:07.415+0200] [debug] [network] Connection accepted from 10.0.1.42
```

Ce format suit le standard ISO 8601 pour l'horodatage (avec le `T` séparateur et l'offset UTC). Les drivers de logging Docker et les agrégateurs (Fluentd, Filebeat, Promtail) parsent ce format nativement. Le thread ID et le PID sont omis — dans un conteneur, il n'y a typiquement qu'un seul processus et les threads sont identifiés par d'autres moyens (traces distribuées).

La colorisation est omise volontairement : `stdout` dans Docker n'est souvent pas un TTY, et les codes ANSI apparaîtraient comme du bruit dans les agrégateurs. spdlog désactive automatiquement les couleurs quand la sortie n'est pas un terminal, mais expliciter un pattern sans `%^...%$` rend l'intention claire.

### JSON structuré (agrégation)

```cpp
file_sink->set_pattern(R"({"time":"%Y-%m-%dT%H:%M:%S.%e%z","level":"%l","logger":"%n","pid":%P,"tid":%t,"msg":"%v"})");
```

Résultat :

```json
{"time":"2026-03-18T14:23:07.412+0200","level":"info","logger":"syswatch","pid":8847,"tid":14352,"msg":"Server started on port 8080"}
{"time":"2026-03-18T14:23:07.420+0200","level":"error","logger":"database","pid":8847,"tid":14360,"msg":"Query timeout after 5032ms"}
```

Ce format produit une ligne JSON par message, directement ingérable par Elasticsearch, Loki, Datadog, ou tout autre agrégateur de logs structurés. Chaque champ est indexable et filtrable indépendamment.

Ce pattern a une limitation importante : le contenu du message (`%v`) est inséré tel quel dans la chaîne JSON. Si le message contient des guillemets doubles, des retours à la ligne, ou des caractères spéciaux JSON, le résultat n'est pas du JSON valide. La section 40.5 traite ce problème en détail et présente des approches robustes pour le logging JSON (custom formatter, sinks dédiés, ou librairies d'intégration spdlog/nlohmann).

Pour les cas simples où les messages ne contiennent pas de caractères spéciaux (ce qui est le cas si vous contrôlez tous les appels de logging), ce pattern fonctionne sans complication.

## Custom formatters

Pour les besoins qui dépassent les capacités du système de patterns, spdlog permet de créer un **formatter personnalisé** en dérivant de `spdlog::custom_flag_formatter` :

```cpp
#include <spdlog/pattern_formatter.h>
#include <unistd.h>  // gethostname

// Flag personnalisé %* qui affiche le nom du serveur
class server_name_flag : public spdlog::custom_flag_formatter {  
public:  
    void format(const spdlog::details::log_msg&,
                const std::tm&,
                spdlog::memory_buf_t& dest) override {
        static const std::string hostname = []() {
            char buf[256];
            gethostname(buf, sizeof(buf));
            return std::string(buf);
        }();
        dest.append(hostname.data(), hostname.data() + hostname.size());
    }

    std::unique_ptr<custom_flag_formatter> clone() const override {
        return std::make_unique<server_name_flag>();
    }
};

// Enregistrement du flag personnalisé
auto formatter = std::make_unique<spdlog::pattern_formatter>();  
formatter->add_flag<server_name_flag>('*');  
formatter->set_pattern("[%Y-%m-%d %H:%M:%S.%e] [%*] [%l] %v");  
logger->set_formatter(std::move(formatter));  
```

Résultat :

```
[2026-03-18 14:23:07.412] [prod-web-03] [info] Server started on port 8080
```

Les flags personnalisés permettent d'injecter n'importe quelle métadonnée dans le format sans modifier chaque appel de logging. Cas d'usage courants : nom d'hôte, identifiant de datacenter, version de l'application, identifiant de déploiement, ou request ID extrait d'un thread-local storage.

La méthode `clone()` est nécessaire car spdlog peut copier les formatters en interne (notamment lors de la création de loggers asynchrones). Le hostname est calculé une seule fois (variable `static` dans la lambda) pour éviter un appel système à chaque message.

## Enrichissement contextuel avec les thread-locals

Un besoin fréquent en production est d'associer chaque message de log à un contexte applicatif — typiquement un request ID ou un trace ID — sans le passer manuellement à chaque appel.

spdlog ne fournit pas de mécanisme de contexte thread-local intégré (contrairement à des frameworks comme Log4j avec son MDC). Mais le pattern est facile à implémenter avec un flag personnalisé et un `thread_local` :

```cpp
// Contexte de requête (thread-local)
struct RequestContext {
    std::string request_id;
    std::string user_id;
};

inline thread_local RequestContext g_request_ctx;

// Flag personnalisé pour le request ID
class request_id_flag : public spdlog::custom_flag_formatter {  
public:  
    void format(const spdlog::details::log_msg&,
                const std::tm&,
                spdlog::memory_buf_t& dest) override {
        const auto& rid = g_request_ctx.request_id;
        if (!rid.empty()) {
            dest.append(rid.data(), rid.data() + rid.size());
        } else {
            const char* na = "-";
            dest.append(na, na + 1);
        }
    }

    std::unique_ptr<custom_flag_formatter> clone() const override {
        return std::make_unique<request_id_flag>();
    }
};
```

Utilisation dans le code de traitement des requêtes :

```cpp
void handle_request(const Request& req) {
    // Positionner le contexte au début du traitement
    g_request_ctx.request_id = req.id();
    g_request_ctx.user_id = req.user();

    SPDLOG_INFO("Processing request: method={} path={}", req.method(), req.path());
    
    // ... traitement ...
    // Tous les logs émis dans ce thread incluent automatiquement le request ID
    
    SPDLOG_INFO("Request completed: status={} duration={}ms", status, duration);

    // Nettoyer le contexte
    g_request_ctx = {};
}
```

Résultat avec le pattern `[%T.%e] [%l] [req:%R] %v` (où `%R` est le flag personnalisé) :

```
[14:23:07.412] [info] [req:a1b2c3d4] Processing request: method=GET path=/api/metrics
[14:23:07.415] [debug] [req:a1b2c3d4] Cache miss for key=metrics_5m
[14:23:07.418] [info] [req:a1b2c3d4] Request completed: status=200 duration=6ms
```

Ce pattern est la base de la corrélation de logs : en filtrant sur `req:a1b2c3d4`, un opérateur retrouve tous les messages émis pendant le traitement d'une requête spécifique, même si des dizaines de requêtes concurrentes sont traitées simultanément. La section 40.5 étend ce concept au logging JSON structuré, et la section 40.3 montre comment le trace ID OpenTelemetry s'intègre dans ce même mécanisme.

## Impact du formatage sur les performances

Le formatage a un coût — construction de la chaîne, conversion des nombres, résolution de l'horodatage. Quelques points à garder en tête.

L'horodatage est le composant le plus coûteux du formatage. spdlog optimise ce coût en cachant la partie date/heure et en ne recalculant que les millisecondes à chaque message. L'appel système `clock_gettime()` sous-jacent est rapide sur Linux (implémenté via vDSO, sans vraie transition noyau), mais il reste le poste principal du formatage.

Les flags de source location (`%s`, `%#`, `%!`) n'ont aucun coût au formatage — les valeurs sont des constantes connues à la compilation, injectées par les macros. En revanche, l'utilisation des macros plutôt que des fonctions a un impact marginal sur la taille du code généré (chaque site d'appel embarque le nom de fichier et le numéro de ligne).

Le pattern JSON est marginalement plus coûteux qu'un pattern texte simple car il produit des chaînes plus longues. La différence est mesurable en micro-benchmark mais rarement significative en conditions réelles.

Les flags personnalisés ont un coût proportionnel à leur implémentation. Un flag qui fait un appel système (`gethostname`) à chaque message serait catastrophique — d'où l'utilisation de variables `static` ou `thread_local` pour cacher les valeurs stables.

En pratique, le formatage représente une fraction modeste du coût total du logging. L'écriture sur disque (ou réseau) domine largement. Optimiser le pattern pour la performance n'a de sens que dans les applications où le logging est sur le chemin critique à très haut débit — et dans ce cas, le mode asynchrone (section 40.1.2) a un impact bien supérieur au choix du pattern.

## Résumé

Le système de patterns spdlog offre un contrôle fin sur le contenu de chaque ligne de log, du format le plus compact pour le développement au JSON structuré pour l'agrégation. Les quatre patterns recommandés — développement, production fichier, Docker/K8s, et JSON — couvrent les contextes les plus courants. Les flags personnalisés et le pattern thread-local étendent le système pour l'enrichissement contextuel (request ID, hostname, trace ID), qui est la base de la corrélation de logs dans les systèmes distribués.

Avec les trois sous-sections de 40.1 complètes — installation, niveaux/sinks, et formatage — vous disposez d'une base de logging professionnelle pour votre application C++. La section 40.2 aborde la deuxième dimension de l'observabilité : les métriques Prometheus. La section 40.5 revient sur le logging pour couvrir en profondeur le JSON structuré et l'intégration avec les stacks d'agrégation.
