# 40.1.1 — Installation et configuration de spdlog

## Intégrer spdlog dans un projet CMake

---

## Introduction

spdlog est conçu pour être facile à intégrer. Dans le cas le plus simple, un seul `#include` suffit pour commencer à logger. Mais un projet professionnel exige un peu plus de rigueur : gestion propre de la dépendance via un gestionnaire de paquets ou FetchContent, choix entre le mode header-only et le mode compilé, configuration du logger par défaut, et création de loggers dédiés par composant.

Cette section couvre l'installation de spdlog dans un projet CMake selon les trois méthodes principales (Conan, vcpkg, FetchContent), le choix du mode de compilation, et la configuration initiale pour un projet C++ typique.

---

## Méthode 1 : Installation via Conan 2.0

Conan est le gestionnaire de dépendances couvert au chapitre 27. C'est la méthode recommandée pour les projets professionnels car elle gère les versions, les options de compilation, et la reproductibilité du build.

Dans votre `conanfile.py` :

```python
from conan import ConanFile  
from conan.tools.cmake import cmake_layout  

class SyswatchConan(ConanFile):
    settings = "os", "compiler", "build_type", "arch"
    generators = "CMakeDeps", "CMakeToolchain"

    def requirements(self):
        self.requires("spdlog/1.15.3")

    def layout(self):
        cmake_layout(self)
```

Installez les dépendances et configurez le projet :

```bash
conan install . --build=missing -of=build  
cmake -B build -G Ninja --preset conan-release  
cmake --build build  
```

Dans votre `CMakeLists.txt`, la dépendance est disponible via `find_package` :

```cmake
find_package(spdlog REQUIRED)

add_executable(syswatch src/main.cpp)  
target_link_libraries(syswatch PRIVATE spdlog::spdlog)  
```

Conan gère automatiquement le choix entre `fmt` bundlée et `fmt` externe, ainsi que la configuration header-only ou compilée, selon les options du paquet. Par défaut, Conan fournit spdlog en mode compilé avec `fmt` bundlée — le choix le plus courant et le plus simple.

---

## Méthode 2 : Installation via vcpkg

Si votre projet utilise vcpkg (chapitre 27, section 27.3) :

```bash
vcpkg install spdlog
```

Dans votre `CMakeLists.txt` :

```cmake
find_package(spdlog CONFIG REQUIRED)

add_executable(syswatch src/main.cpp)  
target_link_libraries(syswatch PRIVATE spdlog::spdlog)  
```

La configuration CMake est identique à celle de Conan — seul le mécanisme d'installation diffère. vcpkg installe par défaut spdlog en mode compilé.

---

## Méthode 3 : FetchContent (sans gestionnaire de paquets)

Pour les projets qui ne veulent pas dépendre d'un gestionnaire de paquets externe, CMake FetchContent télécharge et compile spdlog directement :

```cmake
include(FetchContent)

FetchContent_Declare(
    spdlog
    GIT_REPOSITORY https://github.com/gabime/spdlog.git
    GIT_TAG        v1.15.3
)

# Optionnel : forcer le mode compilé (recommandé pour les gros projets)
set(SPDLOG_BUILD_SHARED OFF CACHE BOOL "" FORCE)

FetchContent_MakeAvailable(spdlog)

add_executable(syswatch src/main.cpp)  
target_link_libraries(syswatch PRIVATE spdlog::spdlog)  
```

FetchContent est pratique pour les petits projets et le prototypage. Pour un projet professionnel, Conan ou vcpkg offrent une meilleure gestion des versions et une compilation plus rapide grâce au cache de paquets précompilés.

## Mode header-only vs mode compilé

spdlog propose deux modes de fonctionnement qui méritent une décision explicite.

### Header-only

En mode header-only, toute la librairie est dans les fichiers d'en-tête. Aucune étape de compilation séparée n'est nécessaire — il suffit d'inclure les headers et de compiler votre code.

L'avantage est la simplicité d'intégration : pas de librairie à linker, pas de configuration de build pour spdlog lui-même. L'inconvénient est que chaque unité de compilation (`.cpp`) qui inclut spdlog recompile le code de la librairie. Sur un projet avec 50 fichiers `.cpp` qui loguent, cela signifie que le code de spdlog est compilé 50 fois — un impact direct sur le temps de build.

Le mode header-only est le défaut quand vous incluez simplement `<spdlog/spdlog.h>` sans configuration particulière.

### Compilé (recommandé)

En mode compilé, spdlog est compilé une fois en librairie statique (ou partagée) et linkée à votre projet. Le code de la librairie n'est compilé qu'une seule fois, et les fichiers `.cpp` de votre projet n'incluent que les déclarations.

Pour activer le mode compilé, définissez la macro `SPDLOG_COMPILED_LIB` avant tout include de spdlog. En pratique, cela se fait via CMake :

```cmake
target_compile_definitions(syswatch PRIVATE SPDLOG_COMPILED_LIB)
```

Si vous utilisez Conan ou vcpkg, cette configuration est gérée automatiquement par le paquet — vous n'avez rien à faire. Si vous utilisez FetchContent, spdlog se compile automatiquement comme librairie statique quand il est ajouté comme sous-projet CMake via `FetchContent_MakeAvailable`.

**Recommandation** : utilisez le mode compilé pour tout projet au-delà du prototype. Le gain en temps de compilation est significatif dès que le projet dépasse une dizaine de fichiers source.

## Premier programme avec spdlog

Voici un exemple minimal qui démontre les fonctionnalités de base :

```cpp
#include <spdlog/spdlog.h>

int main() {
    // Le logger par défaut écrit sur stdout en niveau info
    spdlog::info("Application démarrée");
    spdlog::debug("Ce message ne s'affiche pas (niveau info par défaut)");

    // Changer le niveau de logging
    spdlog::set_level(spdlog::level::debug);
    spdlog::debug("Maintenant ce message s'affiche");

    // Formatage positionnel (syntaxe fmt/std::format)
    int port = 8080;
    std::string host = "0.0.0.0";
    spdlog::info("Serveur en écoute sur {}:{}", host, port);

    // Niveaux de sévérité
    spdlog::warn("Fichier de config absent, utilisation des valeurs par défaut");
    spdlog::error("Connexion à la base de données échouée: timeout après {}ms", 5000);
    spdlog::critical("Espace disque insuffisant, arrêt du service");

    return 0;
}
```

La sortie produite (avec le format par défaut) :

```
[2026-03-18 14:23:07.412] [info] Application démarrée
[2026-03-18 14:23:07.412] [debug] Maintenant ce message s'affiche
[2026-03-18 14:23:07.412] [info] Serveur en écoute sur 0.0.0.0:8080
[2026-03-18 14:23:07.413] [warning] Fichier de config absent, utilisation des valeurs par défaut
[2026-03-18 14:23:07.413] [error] Connexion à la base de données échouée: timeout après 5000ms
[2026-03-18 14:23:07.413] [critical] Espace disque insuffisant, arrêt du service
```

Plusieurs choses à noter. L'horodatage est automatique, avec une résolution à la milliseconde. Le niveau de sévérité est affiché et colorisé dans le terminal (non visible dans cet extrait texte). Le message `debug` initial est silencieusement ignoré car le niveau par défaut est `info`. Après `set_level(debug)`, il apparaît. Le formatage utilise la syntaxe `{}` de `fmt` — identique à `std::format` et `std::print`.

## Le logger par défaut vs les loggers nommés

spdlog fonctionne selon deux modèles d'utilisation.

### Fonctions globales (logger par défaut)

Les appels `spdlog::info(...)`, `spdlog::warn(...)`, etc. utilisent un **logger par défaut** global. C'est le modèle le plus simple, suffisant pour les petits projets et les outils CLI :

```cpp
spdlog::info("Message via le logger par défaut");
```

Ce logger par défaut écrit sur `stdout` avec un format standard. Vous pouvez le remplacer par un logger personnalisé :

```cpp
auto file_logger = spdlog::basic_logger_mt("default", "logs/syswatch.log");  
spdlog::set_default_logger(file_logger);  

// Maintenant spdlog::info() écrit dans le fichier
spdlog::info("Ce message va dans le fichier");
```

### Loggers nommés (recommandé pour les projets structurés)

Pour les projets avec plusieurs composants, les **loggers nommés** permettent de distinguer la source de chaque message et de configurer des comportements différents par composant :

```cpp
#include <spdlog/spdlog.h>
#include <spdlog/sinks/stdout_color_sinks.h>

// Créer des loggers dédiés par composant
auto net_logger = spdlog::stdout_color_mt("network");  
auto db_logger  = spdlog::stdout_color_mt("database");  

// Chaque logger a son propre niveau
net_logger->set_level(spdlog::level::debug);  
db_logger->set_level(spdlog::level::warn);  

// Les messages identifient leur source
net_logger->info("Connection accepted from {}", client_addr);  
db_logger->warn("Query slow: {}ms", query_duration);  
```

La sortie inclut le nom du logger :

```
[2026-03-18 14:23:07.412] [network] [info] Connection accepted from 10.0.1.42
[2026-03-18 14:23:07.415] [database] [warning] Query slow: 342ms
```

Les loggers nommés sont récupérables depuis n'importe où dans le code via `spdlog::get()` :

```cpp
// Dans n'importe quel fichier .cpp du composant réseau
auto logger = spdlog::get("network");  
if (logger) {  
    logger->debug("Packet received: {} bytes", packet_size);
}
```

Ce pattern évite de passer le logger en paramètre à travers toute la pile d'appels. L'appel `spdlog::get()` est une recherche dans une registry interne — rapide, mais à éviter dans une boucle critique. La pratique recommandée est de stocker le `shared_ptr` retourné par `spdlog::get()` dans un membre de classe ou une variable locale au début de la fonction.

## Architecture d'initialisation recommandée

Pour un projet professionnel, l'initialisation du logging se fait une seule fois au démarrage de l'application, dans une fonction dédiée. Voici un pattern éprouvé :

```cpp
// logging.h
#pragma once

namespace syswatch::logging {

struct LogConfig {
    std::string level = "info";       // Niveau par défaut
    std::string log_file;             // Vide = stdout uniquement
    std::size_t max_file_size = 10 * 1024 * 1024;  // 10 Mo
    std::size_t max_files = 5;        // Rotation : 5 fichiers
    bool async = false;               // Mode asynchrone
};

void init(const LogConfig& config);  
void shutdown();  

}  // namespace syswatch::logging
```

```cpp
// logging.cpp
#include "logging.h"

#include <spdlog/spdlog.h>
#include <spdlog/sinks/stdout_color_sinks.h>
#include <spdlog/sinks/rotating_file_sink.h>
#include <spdlog/async.h>

namespace syswatch::logging {

void init(const LogConfig& config) {
    // Convertir le niveau textuel
    auto level = spdlog::level::from_str(config.level);

    // Construire la liste de sinks
    std::vector<spdlog::sink_ptr> sinks;

    // Sink console (toujours présent)
    auto console_sink = std::make_shared<spdlog::sinks::stdout_color_sink_mt>();
    console_sink->set_level(level);
    sinks.push_back(console_sink);

    // Sink fichier rotatif (optionnel)
    if (!config.log_file.empty()) {
        auto file_sink = std::make_shared<spdlog::sinks::rotating_file_sink_mt>(
            config.log_file, config.max_file_size, config.max_files
        );
        file_sink->set_level(level);
        sinks.push_back(file_sink);
    }

    // Créer le logger principal avec plusieurs sinks
    auto logger = std::make_shared<spdlog::logger>(
        "syswatch", sinks.begin(), sinks.end()
    );
    logger->set_level(level);

    // Pattern de formatage
    logger->set_pattern("[%Y-%m-%d %H:%M:%S.%e] [%n] [%^%l%$] [tid:%t] %v");

    // Flush automatique à partir de warn
    logger->flush_on(spdlog::level::warn);

    // Enregistrer comme logger par défaut
    spdlog::set_default_logger(logger);

    spdlog::info("Logging initialisé: level={} file={} async={}",
                 config.level,
                 config.log_file.empty() ? "stdout" : config.log_file,
                 config.async);
}

void shutdown() {
    spdlog::info("Arrêt du logging");
    spdlog::shutdown();  // Flush tous les buffers et détruit les loggers
}

}  // namespace syswatch::logging
```

L'utilisation dans le `main()` :

```cpp
#include "logging.h"

int main(int argc, char* argv[]) {
    syswatch::logging::LogConfig log_config;
    log_config.level = "info";
    log_config.log_file = "/var/log/syswatch/syswatch.log";

    syswatch::logging::init(log_config);

    // ... code de l'application ...
    // Tous les appels spdlog::info(), spdlog::warn() etc.
    // utilisent le logger configuré ci-dessus

    syswatch::logging::shutdown();
    return 0;
}
```

Ce pattern offre plusieurs avantages. La configuration du logging est centralisée et testable. Les paramètres (`level`, `log_file`, `async`) peuvent venir d'un fichier YAML, d'arguments CLI, ou de variables d'environnement — la fonction `init()` ne se soucie pas de leur provenance. L'appel `shutdown()` garantit que tous les buffers sont vidés avant la fin du programme — essentiel en mode asynchrone pour ne pas perdre les derniers messages.

## Configuration du niveau via variable d'environnement

Un pattern courant en production est de permettre le changement du niveau de logging via une variable d'environnement, sans modifier la configuration ni redémarrer le service :

```cpp
void init(const LogConfig& config) {
    // La variable d'environnement surcharge la config
    std::string level_str = config.level;
    if (const char* env = std::getenv("SYSWATCH_LOG_LEVEL")) {
        level_str = env;
    }
    auto level = spdlog::level::from_str(level_str);
    // ... suite de l'initialisation
}
```

En production, le niveau est `info`. Quand un problème survient :

```bash
# Redémarrer temporairement en mode debug
sudo systemctl stop syswatch  
SYSWATCH_LOG_LEVEL=debug sudo systemctl start syswatch  
```

Ou mieux, avec un override systemd (sans modifier le fichier unit du paquet) :

```bash
sudo systemctl edit syswatch
# Ajouter :
# [Service]
# Environment=SYSWATCH_LOG_LEVEL=debug

sudo systemctl restart syswatch
```

Après diagnostic, un simple `sudo systemctl revert syswatch` supprime l'override et ramène le niveau à sa valeur par défaut.

## flush_on : quand écrire sur disque

Par défaut, spdlog bufferise les messages et les écrit périodiquement sur disque (ou lors de l'appel à `flush()`). Ce buffering est essentiel pour la performance — un appel système `write()` pour chaque message serait prohibitif sous charge.

Mais si l'application crashe entre un `spdlog::error(...)` et le prochain flush, le message d'erreur est perdu — précisément celui dont vous avez besoin pour diagnostiquer le crash.

La méthode `flush_on()` résout ce dilemme en forçant un flush immédiat à partir d'un certain niveau de sévérité :

```cpp
logger->flush_on(spdlog::level::warn);
```

Avec cette configuration, les messages `trace`, `debug` et `info` sont bufferisés normalement (performance optimale pour les messages à haut débit), tandis que les messages `warn`, `error` et `critical` déclenchent un flush immédiat (les messages importants sont persistés sans délai).

C'est un compromis entre performance et fiabilité que vous pouvez ajuster selon vos besoins. Pour un service où la traçabilité des erreurs est critique, `flush_on(warn)` est un bon défaut. Pour un outil de traitement batch où la performance prime, `flush_on(error)` ou même un flush uniquement à la fin du programme peut être approprié.

Un flush périodique est également disponible :

```cpp
spdlog::flush_every(std::chrono::seconds(3));
```

Cette approche combine les deux stratégies : flush immédiat pour les messages critiques, flush périodique pour garantir que même les messages `info` ne restent pas en buffer indéfiniment.

## Compilation et vérification

Vérifions que tout fonctionne avec un `CMakeLists.txt` complet :

```cmake
cmake_minimum_required(VERSION 3.25)  
project(syswatch VERSION 1.2.0 LANGUAGES CXX)  

set(CMAKE_CXX_STANDARD 23)  
set(CMAKE_CXX_STANDARD_REQUIRED ON)  

find_package(spdlog REQUIRED)

add_executable(syswatch
    src/main.cpp
    src/logging.cpp
)

target_link_libraries(syswatch PRIVATE spdlog::spdlog)
```

Compilation et test :

```bash
cmake -B build -G Ninja  
cmake --build build  
./build/syswatch
```

Si l'application affiche les messages avec horodatage, niveau colorisé, et nom du logger, l'installation est fonctionnelle. La section suivante (40.1.2) explore en détail les niveaux de sévérité et les différents sinks disponibles pour router les logs vers les bonnes destinations.
