# 40.1.2 — Niveaux de log et sinks

## Filtrer les messages et les router vers les bonnes destinations

---

## Introduction

La section précédente a mis en place spdlog dans un projet CMake et présenté le pattern d'initialisation recommandé. Cette section approfondit les deux concepts centraux de tout framework de logging : les **niveaux de sévérité** qui filtrent les messages, et les **sinks** qui déterminent où ces messages atterrissent.

Bien utiliser ces deux mécanismes fait la différence entre un logging qui aide au diagnostic et un logging qui noie l'opérateur sous un flux illisible ou, à l'inverse, qui reste muet quand on a besoin d'informations.

---

## Les six niveaux de sévérité

spdlog définit six niveaux, ordonnés du plus verbeux au plus critique :

```cpp
namespace spdlog::level {
    enum level_enum {
        trace    = 0,   // Le plus verbeux
        debug    = 1,
        info     = 2,
        warn     = 3,
        err      = 4,
        critical = 5,
        off      = 6    // Désactive tout logging
    };
}
```

Quand vous configurez le niveau d'un logger à `info`, tous les messages de niveau `info` et supérieur (`info`, `warn`, `err`, `critical`) sont émis. Les messages `trace` et `debug` sont ignorés — le formatage n'est même pas effectué, ce qui signifie zéro coût en performance pour les messages filtrés.

### trace — Granularité maximale

```cpp
spdlog::trace("Entering parse_config(), file={}", path);  
spdlog::trace("Token read: type={} value='{}'", token.type, token.value);  
spdlog::trace("Loop iteration i={}, accumulator={}", i, acc);  
```

Le niveau `trace` capture le flux d'exécution détaillé : entrée et sortie de fonctions, valeurs intermédiaires dans les boucles, contenu des buffers. Le volume de messages en `trace` est souvent colossal — plusieurs ordres de grandeur au-dessus de `info`. Ce niveau n'est jamais activé en production continue. Il sert au diagnostic ponctuel de bugs difficiles à reproduire, activé temporairement puis désactivé.

Un bon réflexe est de placer des logs `trace` aux points d'entrée et de sortie des composants critiques pendant le développement. Ils ne coûtent rien en production (filtrage avant formatage) et deviennent précieux le jour où un problème surgit.

### debug — Informations de diagnostic

```cpp
spdlog::debug("Config loaded: {} endpoints, {} workers", endpoints.size(), n_workers);  
spdlog::debug("Cache hit for key={}, age={}ms", key, age.count());  
spdlog::debug("Connection pool: active={} idle={} max={}", active, idle, max);  
```

Le niveau `debug` fournit des informations utiles pour comprendre le comportement de l'application sans descendre au niveau instruction par instruction. C'est le niveau activé quand un opérateur investigue un problème signalé en production : il fournit suffisamment de contexte sans le bruit du `trace`.

La frontière entre `trace` et `debug` est une question de jugement. Un guide simple : si le message est utile pour comprendre *pourquoi* l'application a pris une décision, c'est `debug`. Si le message décrit *comment* une opération s'exécute pas à pas, c'est `trace`.

### info — Fonctionnement normal

```cpp
spdlog::info("Service started on port {}", port);  
spdlog::info("Request processed: method={} path={} status={} duration={}ms",  
             method, path, status, duration_ms);
spdlog::info("Config reloaded: {} rules active", rules.size());  
spdlog::info("Graceful shutdown initiated");  
```

Le niveau `info` est le niveau de production par défaut. Il documente les événements significatifs du cycle de vie de l'application : démarrage, arrêt, rechargement de configuration, requêtes traitées (éventuellement en résumé), connexions établies ou perdues. En lisant les logs `info`, un opérateur doit pouvoir comprendre ce que fait l'application sans être submergé.

La tentation est d'en mettre trop. Un service qui traite 10 000 requêtes par seconde ne devrait pas logger chaque requête en `info` — cela produirait un volume ingérable. Loguez les requêtes individuelles en `debug`, et réservez `info` pour les agrégats périodiques (`info("Processed {} requests in last 60s, avg latency={}ms", count, avg)`) ou les événements ponctuels.

### warn — Situations anormales non bloquantes

```cpp
spdlog::warn("Connection to redis lost, retrying in {}s", backoff_seconds);  
spdlog::warn("Config file {} not found, using defaults", config_path);  
spdlog::warn("Request queue at {}% capacity", occupancy_pct);  
spdlog::warn("Certificate expires in {} days", days_remaining);  
```

Le niveau `warn` signale que quelque chose d'inhabituel s'est produit, mais que l'application a pu continuer à fonctionner — typiquement via un mécanisme de fallback, un retry, ou des valeurs par défaut. Un warning isolé n'est pas une urgence. Mais un flux continu de warnings identiques est le signe d'un problème latent qui mérite attention.

Les alertes de monitoring sont souvent configurées sur le volume de warnings : "alerter si plus de 100 warnings par minute sur le composant `database`". C'est pourquoi il est important de ne pas surclasser des événements normaux en `warn` — cela génère des faux positifs qui érodent la confiance dans les alertes.

### error — Opérations échouées

```cpp
spdlog::error("Failed to process request: id={} error='{}'", req_id, ec.message());  
spdlog::error("Database query timeout after {}ms: query='{}'", timeout_ms, query);  
spdlog::error("Unable to write to {}: {}", output_path, strerror(errno));  
```

Le niveau `error` signale qu'une opération a échoué de manière non récupérable — la requête en cours ne sera pas servie, le fichier ne sera pas écrit, la transaction a été annulée. L'application continue de tourner, mais quelque chose a mal fonctionné pour un cas spécifique.

Chaque message `error` doit contenir suffisamment de contexte pour diagnostiquer le problème *sans* avoir besoin de reproduire l'incident : identifiant de la requête ou de l'opération, valeurs des paramètres pertinents, message d'erreur du système ou de la librairie.

### critical — Le service est compromis

```cpp
spdlog::critical("Out of memory: cannot allocate {} bytes", required_bytes);  
spdlog::critical("All database replicas unreachable, service degraded");  
spdlog::critical("Data corruption detected in index file {}", index_path);  
```

Le niveau `critical` est réservé aux situations qui compromettent le fonctionnement global du service — pas seulement une requête, mais le service entier. Un message `critical` doit déclencher une alerte immédiate (PagerDuty, SMS, etc.) et une intervention humaine.

L'utilisation de `critical` doit être parcimonieuse. Si tout est critique, rien ne l'est. Un bon test : "si je reçois cette alerte à 3h du matin, est-ce que ça justifie de me réveiller ?" Si la réponse est non, c'est un `error`, pas un `critical`.

## Configurer le niveau par logger

Chaque logger peut avoir son propre niveau, indépendamment du logger par défaut. Cela permet un filtrage fin par composant :

```cpp
auto net_logger = spdlog::stdout_color_mt("network");  
auto db_logger  = spdlog::stdout_color_mt("database");  
auto cache_log  = spdlog::stdout_color_mt("cache");  

// En production : composant réseau en debug (on investigue un problème),
// reste en info
net_logger->set_level(spdlog::level::debug);  
db_logger->set_level(spdlog::level::info);  
cache_log->set_level(spdlog::level::info);  
```

Ce mécanisme est fondamental pour le diagnostic en production. Plutôt que de passer toute l'application en `debug` (ce qui génère un volume de logs ingérable et peut affecter les performances), vous activez le `debug` uniquement sur le composant suspecté.

Le niveau global `spdlog::set_level()` agit comme un filtre plafond : un message n'est émis que s'il passe à la fois le filtre du logger et le filtre global. En pratique, laissez le filtre global à `trace` (pas de filtrage global) et gérez le filtrage au niveau de chaque logger.

## Les sinks : où vont les messages

Un **sink** est une destination pour les messages de log. spdlog fournit une collection de sinks couvrant les besoins courants, et une interface pour créer des sinks personnalisés.

Chaque sink possède son propre niveau de filtrage, indépendant du niveau du logger. Cela permet des configurations comme "envoyer `info` et au-dessus vers `stdout`, mais envoyer `debug` et au-dessus vers le fichier" :

```cpp
auto console_sink = std::make_shared<spdlog::sinks::stdout_color_sink_mt>();  
console_sink->set_level(spdlog::level::info);  

auto file_sink = std::make_shared<spdlog::sinks::rotating_file_sink_mt>(
    "logs/syswatch.log", 10 * 1024 * 1024, 5
);
file_sink->set_level(spdlog::level::debug);

auto logger = std::make_shared<spdlog::logger>(
    "syswatch",
    spdlog::sinks_init_list{console_sink, file_sink}
);
// Le logger doit être au niveau le plus bas de ses sinks
logger->set_level(spdlog::level::debug);
```

Avec cette configuration, l'opérateur voit les messages `info`+ dans le terminal, et les messages `debug`+ sont disponibles dans le fichier pour une investigation approfondie.

Un point important : le niveau du logger doit être au moins aussi bas que le sink le plus verbeux. Si le logger est configuré en `info` mais qu'un sink est en `debug`, les messages `debug` seront filtrés par le logger avant même d'atteindre le sink. C'est une source de confusion fréquente.

### Console : stdout_color_sink et stderr_color_sink

```cpp
#include <spdlog/sinks/stdout_color_sinks.h>

// Sink stdout avec couleurs (mt = multi-threaded)
auto console = std::make_shared<spdlog::sinks::stdout_color_sink_mt>();

// Sink stderr (pour les erreurs uniquement)
auto err_console = std::make_shared<spdlog::sinks::stderr_color_sink_mt>();  
err_console->set_level(spdlog::level::err);  
```

Les sinks console sont les plus utilisés. La variante `_color_` ajoute des codes ANSI pour coloriser les niveaux dans le terminal (vert pour `info`, jaune pour `warn`, rouge pour `error`, etc.). La colorisation est automatiquement désactivée quand la sortie n'est pas un terminal (pipe, redirection vers un fichier, environnement Docker sans TTY).

La convention Docker est de logger sur `stdout`. Pour les applications qui distinguent la sortie normale de la sortie d'erreur, utiliser `stdout` pour `info`/`debug` et `stderr` pour `warn`/`error`/`critical` est une option — mais cela peut compliquer l'agrégation car les deux flux sont mélangés par le runtime Docker. La plupart des projets utilisent uniquement `stdout`.

Les suffixes `_mt` (multi-threaded) et `_st` (single-threaded) indiquent la politique de thread-safety. Utilisez `_mt` par défaut. Le surcoût de synchronisation est négligeable dans la majorité des cas. `_st` n'est justifié que pour les applications strictement mono-thread où chaque nanoseconde compte.

### Fichiers simples : basic_file_sink

```cpp
#include <spdlog/sinks/basic_file_sink.h>

auto file_sink = std::make_shared<spdlog::sinks::basic_file_sink_mt>(
    "logs/syswatch.log",
    true  // truncate : true = écraser, false = ajouter (append)
);
```

Le sink fichier simple écrit dans un fichier unique sans rotation. L'option `truncate` contrôle si le fichier est écrasé au démarrage (`true`) ou si les nouveaux messages sont ajoutés à la fin (`false`). Le mode append est le défaut pour la production — vous ne voulez pas perdre les logs de la session précédente.

Ce sink convient pour les outils CLI et les traitements batch avec une durée de vie limitée. Pour les services qui tournent en continu, le fichier grossit sans limite — utilisez plutôt un sink rotatif.

### Fichiers rotatifs : rotating_file_sink

```cpp
#include <spdlog/sinks/rotating_file_sink.h>

auto rotating = std::make_shared<spdlog::sinks::rotating_file_sink_mt>(
    "logs/syswatch.log",   // Chemin du fichier principal
    10 * 1024 * 1024,      // Taille max par fichier : 10 Mo
    5                       // Nombre de fichiers de rotation
);
```

Quand le fichier atteint la taille maximale, spdlog le renomme en `syswatch.1.log`, l'ancien `syswatch.1.log` devient `syswatch.2.log`, et ainsi de suite. Le fichier le plus ancien (`syswatch.5.log`) est supprimé. Le fichier principal est recréé vide.

Ce mécanisme garantit que l'espace disque consommé par les logs est borné — dans cet exemple, au maximum 60 Mo (6 fichiers × 10 Mo). C'est indispensable pour les services en production où un disque plein peut mettre l'application (et le système entier) à genoux.

Le dimensionnement dépend de votre contexte. Un service à haut débit qui logue en `info` peut générer plusieurs centaines de Mo par heure. Un daemon tranquille peut tenir dans 5 Mo pour une semaine. Mesurez le volume réel de vos logs en conditions représentatives avant de fixer les paramètres.

### Rotation quotidienne : daily_file_sink

```cpp
#include <spdlog/sinks/daily_file_sink.h>

// Rotation à 00h00 chaque jour
auto daily = std::make_shared<spdlog::sinks::daily_file_sink_mt>(
    "logs/syswatch.log",
    0, 0,    // Heure et minute de rotation
    false,   // truncate
    7        // Conserver 7 jours d'historique
);
```

Le sink quotidien crée un nouveau fichier chaque jour avec un suffixe de date (`syswatch_2026-03-18.log`). Le paramètre de rétention (7 dans cet exemple) supprime automatiquement les fichiers plus anciens.

Ce sink est préférable au rotatif par taille quand les logs doivent être corrélés avec des dates : "que s'est-il passé jeudi dernier ?" est une question plus facile à résoudre quand chaque jour a son fichier.

### Syslog : syslog_sink

```cpp
#include <spdlog/sinks/syslog_sink.h>

auto syslog = std::make_shared<spdlog::sinks::syslog_sink_mt>(
    "syswatch",      // Identifiant dans syslog
    LOG_PID,         // Options syslog (inclure le PID)
    LOG_DAEMON,      // Facilité syslog
    true             // Activer le formatage spdlog
);
```

Le sink syslog envoie les messages vers le daemon syslog du système (`rsyslog`, `syslog-ng`, `journald`). C'est la méthode standard d'intégration dans les infrastructures Linux traditionnelles où les logs sont centralisés par syslog.

Les niveaux spdlog sont mappés automatiquement vers les priorités syslog : `trace`/`debug` → `LOG_DEBUG`, `info` → `LOG_INFO`, `warn` → `LOG_WARNING`, `err` → `LOG_ERR`, `critical` → `LOG_CRIT`.

Sur les systèmes avec systemd (Ubuntu inclus), les messages envoyés à `stdout`/`stderr` par un service systemd sont automatiquement capturés par `journald` — ce qui rend le sink syslog redondant si votre application tourne comme service. Le sink syslog reste utile pour les applications qui ne tournent pas sous systemd ou qui ont besoin de la classification par facilité.

### Callback : callback_sink

```cpp
#include <spdlog/sinks/callback_sink.h>

auto callback = std::make_shared<spdlog::sinks::callback_sink_mt>(
    [](const spdlog::details::log_msg& msg) {
        // Envoyer les erreurs vers un système d'alerte externe
        if (msg.level >= spdlog::level::err) {
            alert_system::notify(
                std::string(msg.payload.begin(), msg.payload.end())
            );
        }
    }
);
```

Le sink callback exécute une fonction définie par l'utilisateur pour chaque message. C'est le point d'extension pour les intégrations personnalisées : envoi vers une file de messages (Kafka, RabbitMQ), stockage dans une base de données, transmission à un service d'alerte, ou toute autre destination non couverte par les sinks fournis.

Attention à la performance : la callback s'exécute dans le thread de logging (ou le thread d'arrière-plan en mode asynchrone). Une callback lente bloque l'émission des messages suivants. Pour les destinations réseau, utilisez une callback qui écrit dans un buffer interne, vidé par un thread séparé.

## Multi-sink : combiner les destinations

La puissance du modèle de sinks vient de la combinaison. Un logger avec plusieurs sinks envoie chaque message à toutes ses destinations, chacune avec son propre niveau de filtrage et son propre formatage :

```cpp
#include <spdlog/spdlog.h>
#include <spdlog/sinks/stdout_color_sinks.h>
#include <spdlog/sinks/rotating_file_sink.h>
#include <spdlog/sinks/syslog_sink.h>

std::vector<spdlog::sink_ptr> sinks;

// Console : info et au-dessus, format lisible humain
auto console = std::make_shared<spdlog::sinks::stdout_color_sink_mt>();  
console->set_level(spdlog::level::info);  
console->set_pattern("[%H:%M:%S.%e] [%^%l%$] %v");  
sinks.push_back(console);  

// Fichier : debug et au-dessus, format complet avec thread ID
auto file = std::make_shared<spdlog::sinks::rotating_file_sink_mt>(
    "logs/syswatch.log", 50 * 1024 * 1024, 10
);
file->set_level(spdlog::level::debug);  
file->set_pattern("[%Y-%m-%d %H:%M:%S.%e] [%n] [%l] [tid:%t] %v");  
sinks.push_back(file);  

// Syslog : error et au-dessus, pour les alertes centralisées
auto sys = std::make_shared<spdlog::sinks::syslog_sink_mt>("syswatch", LOG_PID, LOG_DAEMON);  
sys->set_level(spdlog::level::err);  
sinks.push_back(sys);  

auto logger = std::make_shared<spdlog::logger>("syswatch", sinks.begin(), sinks.end());  
logger->set_level(spdlog::level::debug);  // Laisser passer tout vers les sinks  
spdlog::set_default_logger(logger);  
```

Avec cette configuration, un message `spdlog::debug(...)` apparaît uniquement dans le fichier. Un message `spdlog::info(...)` apparaît dans la console et dans le fichier. Un message `spdlog::error(...)` apparaît partout : console, fichier, et syslog. C'est un pattern de production courant et éprouvé.

Chaque sink peut avoir son propre pattern de formatage. La console utilise un format compact (heure sans date, pas de thread ID) pour la lisibilité interactive. Le fichier utilise un format complet pour l'investigation post-mortem. Syslog utilise le format natif de syslog.

## Logging asynchrone

Pour les applications à haut débit où le coût d'écriture sur disque ne doit pas ralentir le thread applicatif, spdlog propose un mode asynchrone.

### Activation

```cpp
#include <spdlog/async.h>
#include <spdlog/sinks/rotating_file_sink.h>

// Initialiser le thread pool asynchrone (taille du buffer, nombre de threads)
spdlog::init_thread_pool(8192, 1);

auto file_sink = std::make_shared<spdlog::sinks::rotating_file_sink_mt>(
    "logs/syswatch.log", 10 * 1024 * 1024, 5
);

auto async_logger = std::make_shared<spdlog::async_logger>(
    "syswatch",
    file_sink,
    spdlog::thread_pool(),
    spdlog::async_overflow_policy::block  // Politique en cas de buffer plein
);

spdlog::set_default_logger(async_logger);
```

En mode asynchrone, l'appel `spdlog::info(...)` écrit le message dans un ring buffer en mémoire et retourne immédiatement. Un thread dédié (le worker du thread pool) lit les messages du buffer et les écrit sur les sinks. Le thread applicatif n'attend jamais l'écriture disque.

### Politique de débordement

Quand le ring buffer est plein (l'application produit des messages plus vite que le worker ne les écrit), deux politiques sont disponibles :

**`block`** — Le thread applicatif bloque jusqu'à ce qu'une place se libère dans le buffer. Aucun message n'est perdu, mais l'application est ralentie. C'est le choix sûr quand la perte de messages est inacceptable.

**`overrun_oldest`** — Le message le plus ancien dans le buffer est écrasé par le nouveau. L'application n'est jamais ralentie, mais des messages peuvent être perdus sous charge extrême. C'est le choix performant quand la latence prime sur l'exhaustivité des logs.

```cpp
// Ne jamais perdre de messages (défaut recommandé)
spdlog::async_overflow_policy::block

// Ne jamais ralentir l'application (accepte la perte)
spdlog::async_overflow_policy::overrun_oldest
```

Le dimensionnement du buffer (premier paramètre de `init_thread_pool`) est un compromis mémoire/débit. Un buffer de 8192 slots (le défaut) consomme quelques centaines de Ko et absorbe les pics de logging courants. Pour les applications qui émettent des rafales de millions de messages, un buffer de 65536 ou 131072 peut être nécessaire.

### Quand utiliser le mode asynchrone

Le mode synchrone est suffisant pour la majorité des applications. Le mode asynchrone est justifié quand le profiling montre que les appels de logging sont un goulot d'étranglement mesurable — typiquement quand l'application émet plus de 100 000 messages par seconde vers des sinks fichier ou réseau.

N'activez pas le mode asynchrone par défaut "au cas où". Il ajoute de la complexité (thread pool à dimensionner, politique de débordement à choisir, shutdown à gérer correctement) pour un gain qui n'est mesurable que sous forte charge. Commencez en synchrone, profilez, et passez en asynchrone uniquement si les chiffres le justifient.

Un piège courant : oublier d'appeler `spdlog::shutdown()` avant la fin du programme en mode asynchrone. Sans cet appel, les messages encore dans le buffer sont perdus quand le thread pool est détruit.

## Résumé des bonnes pratiques

**Choisissez le bon niveau.** `info` pour les événements de cycle de vie, `warn` pour les anomalies récupérées, `error` pour les échecs d'opération, `critical` pour les situations qui compromettent le service. Ne sur-classez pas et ne sous-classez pas — la fiabilité des alertes en dépend.

**Configurez des sinks adaptés au contexte.** Console pour le développement et Docker. Fichier rotatif pour les serveurs bare metal. Syslog pour les infrastructures centralisées. Combinez-les avec des niveaux différents par sink.

**Laissez le logger au niveau le plus bas de ses sinks.** Si un sink est en `debug`, le logger doit être au moins en `debug`. Sinon le logger filtre avant que le sink n'ait la chance de voir le message.

**Utilisez des loggers nommés par composant.** Cela permet d'activer le `debug` chirurgicalement sur un composant sans noyer le reste. En production, c'est la différence entre une investigation ciblée et un déluge de données.

**Réservez le mode asynchrone aux cas mesurés.** Le mode synchrone est plus simple, plus prévisible, et suffisant pour la plupart des applications. Le mode asynchrone est un outil de performance, pas un réglage par défaut.
