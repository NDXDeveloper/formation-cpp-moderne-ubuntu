# 12.14.2 Réflexion statique (Static Reflection)

## Quand le code s'examine lui-même

La réflexion (*reflection*) est la capacité d'un programme à examiner sa propre structure : quels sont les membres de cette classe ? Quels sont les enumerateurs de cet enum ? Quels paramètres prend cette fonction ? Des langages comme Java, C#, Python et Rust offrent cette capacité depuis longtemps. Le C++, lui, en était dépourvu — forçant les développeurs à recourir à des macros, des X-macros, des bibliothèques de traits, ou des générateurs de code externes pour accomplir des tâches qui auraient dû être triviales.

C++26 introduit la réflexion statique (P2996), un système qui opère **entièrement à la compilation**. Pas de RTTI dynamique coûteux, pas de métadonnées embarquées dans le binaire, pas d'overhead runtime. Le compilateur connaît la structure de tous les types — la réflexion statique expose simplement cette connaissance au programmeur sous forme de valeurs manipulables à la compilation.

C'est la fonctionnalité la plus complexe et la plus transformatrice de C++26. Elle change fondamentalement la manière d'écrire du code générique et de la métaprogrammation en C++.

## Les deux opérateurs fondamentaux

La réflexion C++26 repose sur deux nouveaux opérateurs syntaxiques qui forment un aller-retour entre le monde du code et le monde de la métaprogrammation.

### L'opérateur de réflexion : ^^

L'opérateur `^^` (*reflect*) prend une entité du programme (un type, une variable, une fonction, un namespace, un enumerateur) et produit une valeur de type `std::meta::info` — un objet opaque qui représente cette entité à la compilation :

```cpp
#include <meta>

struct Point {
    double x;
    double y;
    double z;
};

constexpr auto point_reflection = ^^Point;      // Réflexion du type Point  
constexpr auto x_reflection = ^^Point::x;       // Réflexion du membre x  
constexpr auto ns_reflection = ^^std;            // Réflexion du namespace std  
```

L'opérateur `^^` « élève » une entité du programme vers le monde de la métaprogrammation compile-time. L'objet `std::meta::info` retourné est une valeur consteval — il n'existe qu'à la compilation et n'occupe aucun espace dans le binaire.

> 💡 *Le choix du double caret `^^` est intentionnel : le caret simple `^` entre en conflit avec l'extension « blocks » d'Apple/Clang. Le double caret rappelle visuellement l'idée d'« élever » une entité vers un niveau méta.*

### L'opérateur de splice : [: :]

L'opérateur `[: :]` (*splice*) fait l'opération inverse : il prend une valeur `std::meta::info` et la « réinjecte » dans le code comme une entité concrète (un type, un nom, un accès à un membre) :

```cpp
#include <meta>

using T = int;  
constexpr auto t_refl = ^^T;  

// Le splice [: t_refl :] produit le type int
[: t_refl :] value = 42;    // Équivalent à : int value = 42;
```

Le splice est le mécanisme qui ferme la boucle : on réfléchit une entité (^^), on la manipule avec les fonctions de `<meta>`, puis on la réinjecte dans le code ([: :]).

### L'aller-retour : reflect → manipulate → splice

```
Code source                    Métaprogrammation compile-time
                                        
  Point          ──── ^^ ────►  std::meta::info
                                    │
                                    │ fonctions std::meta::
                                    │ (nonstatic_data_members_of,
                                    │  identifier_of, type_of, ...)
                                    │
                                    ▼
  item.x         ◄── [: :] ──  member_reflection
```

## Le type std::meta::info

Toutes les réflexions sont représentées par un type unique : `std::meta::info`. Qu'il s'agisse d'un type, d'une variable, d'une fonction, d'un enumerateur ou d'un namespace, la réflexion est toujours un `std::meta::info`. C'est un design volontairement uniforme — pas de hiérarchie de types de réflexion comme dans d'autres langages.

L'objet est opaque : on ne peut pas inspecter son contenu directement. Toute l'information est extraite via les fonctions `consteval` du namespace `std::meta` :

```cpp
#include <meta>

struct Point {
    double x;
    double y;
};

constexpr auto refl = ^^Point;

// Extraction d'informations via les fonctions std::meta
static_assert(std::meta::is_type(refl));  
static_assert(std::meta::identifier_of(refl) == "Point");  
static_assert(std::meta::nonstatic_data_members_of(refl).size() == 2);  
```

## Fonctions std::meta essentielles

Le header `<meta>` fournit un ensemble riche de fonctions `consteval` pour interroger les réflexions. Voici les plus courantes, regroupées par catégorie.

### Identification

```cpp
std::meta::identifier_of(refl)    // Nom de l'entité ("Point", "x", "Red")  
std::meta::display_string_of(refl) // Représentation lisible pour le debug  
```

### Classification

```cpp
std::meta::is_type(refl)           // Est-ce un type ?  
std::meta::is_function(refl)       // Est-ce une fonction ?  
std::meta::is_variable(refl)       // Est-ce une variable ?  
std::meta::is_namespace(refl)      // Est-ce un namespace ?  
std::meta::is_class_type(refl)     // Est-ce une classe/struct ?  
std::meta::is_enum(refl)           // Est-ce un enum ?  
```

### Inspection de types

```cpp
std::meta::type_of(refl)           // Le type d'une variable ou d'un membre  
std::meta::parent_of(refl)         // Le type parent (pour un membre)  
std::meta::scope_of(refl)          // Le scope englobant  
```

### Itération sur les membres

```cpp
// Retourne un vector<info> des membres de données non-statiques
std::meta::nonstatic_data_members_of(refl, ctx)

// Retourne un vector<info> des enumerateurs d'un enum
std::meta::enumerators_of(refl)

// Retourne un vector<info> des membres (avec contrôle d'accès)
std::meta::members_of(refl, ctx)
```

Le paramètre `ctx` dans `nonstatic_data_members_of` et `members_of` est un `access_context` qui détermine quels membres sont visibles (publics uniquement, ou tous les membres si on a l'accès). L'expression `std::meta::access_context::current()` fournit le contexte d'accès du point d'appel.

### Valeurs et constantes

```cpp
std::meta::constant_of(refl)       // Valeur d'une constante réfléchie
```

## Expansion statements : template for

La réflexion serait limitée sans un moyen d'itérer sur les résultats à la compilation. C++26 introduit les *expansion statements* — une boucle `template for` qui se déroule à la compilation :

```cpp
#include <meta>
#include <print>

struct Config {
    std::string host = "localhost";
    int port = 8080;
    bool debug = false;
};

void print_config(const Config& cfg) {
    constexpr auto members = std::meta::nonstatic_data_members_of(
        ^^Config, std::meta::access_context::current());

    template for (constexpr auto member : members) {
        std::print("  {} = {}\n", 
                   std::meta::identifier_of(member),
                   cfg.[:member:]);
    }
}

// Sortie :
//   host = localhost
//   port = 8080
//   debug = false
```

Le `template for` itère sur la séquence de réflexions `members` à la compilation. À chaque itération, `member` est une constante de compilation représentant un membre différent de `Config`. Le corps de la boucle est **instancié** pour chaque élément — comme un template, pas comme une boucle runtime. L'expression `cfg.[:member:]` utilise le splice pour accéder au membre correspondant sur l'objet `cfg`.

Le résultat est que la fonction `print_config` affiche tous les membres de `Config` sans en nommer aucun explicitement. Si on ajoute un membre à `Config`, la fonction se met à jour automatiquement — pas de code à modifier.

## Cas d'usage : des problèmes enfin résolus

### Enum vers string — le cas emblématique

La conversion d'un `enum` en string est le cas d'usage le plus souvent cité pour la réflexion, parce qu'il est à la fois trivial conceptuellement et pénible sans réflexion :

```cpp
#include <meta>
#include <string>

enum class Color { Red, Green, Blue };

// Avant C++26 : macro, X-macro, switch manuel, ou bibliothèque tierce (magic_enum)
// C++26 : quelques lignes

consteval std::string to_string(Color c) {
    template for (constexpr auto e : std::meta::enumerators_of(^^Color)) {
        if (std::meta::constant_of(e) == c) {
            return std::string(std::meta::identifier_of(e));
        }
    }
    return "<unknown>";
}

// Générique — fonctionne pour n'importe quel enum
template <typename E>
    requires std::is_enum_v<E>
consteval std::string enum_to_string(E value) {
    template for (constexpr auto e : std::meta::enumerators_of(^^E)) {
        if (std::meta::constant_of(e) == value) {
            return std::string(std::meta::identifier_of(e));
        }
    }
    return "<unknown>";
}
```

La version générique fonctionne pour **tout** enum, sans aucune spécialisation. Ce qui nécessitait des hacks comme `magic_enum` (limitée à une plage de valeurs) ou des macros d'enregistrement est désormais résolu dans le langage.

### Sérialisation automatique en JSON

```cpp
#include <meta>
#include <string>
#include <format>

template <typename T>  
std::string to_json(const T& obj) {  
    constexpr auto members = std::meta::nonstatic_data_members_of(
        ^^T, std::meta::access_context::current());

    std::string result = "{";
    bool first = true;

    template for (constexpr auto member : members) {
        if (!first) result += ", ";
        first = false;

        result += std::format("\"{}\":", std::meta::identifier_of(member));

        // Dispatch selon le type du membre
        using MemberType = [: std::meta::type_of(member) :];
        if constexpr (std::is_same_v<MemberType, std::string>) {
            result += std::format("\"{}\"", obj.[:member:]);
        } else if constexpr (std::is_same_v<MemberType, bool>) {
            result += obj.[:member:] ? "true" : "false";
        } else {
            result += std::format("{}", obj.[:member:]);
        }
    }

    result += "}";
    return result;
}

struct User {
    std::string name;
    int age;
    bool active;
};

// Utilisation :
User alice{"Alice", 30, true};  
std::string json = to_json(alice);  
// json == {"name":"Alice", "age":30, "active":true}
```

Ce sérialiseur fonctionne pour **toute** structure dont les membres publics sont des types supportés. Ajouter un champ à `User` met automatiquement à jour le JSON produit. Pas de macro d'enregistrement, pas de mapping manuel, pas de générateur de code.

### Comparaison automatique de structures

```cpp
#include <meta>

template <typename T>  
bool structural_equal(const T& a, const T& b) {  
    constexpr auto members = std::meta::nonstatic_data_members_of(
        ^^T, std::meta::access_context::current());

    template for (constexpr auto member : members) {
        if (a.[:member:] != b.[:member:]) {
            return false;
        }
    }
    return true;
}

// Fonctionne pour toute struct avec des membres comparables
struct Point { double x, y, z; };  
Point a{1.0, 2.0, 3.0};  
Point b{1.0, 2.0, 3.0};  
bool eq = structural_equal(a, b);  // true  
```

### Rendre un type formattable automatiquement

```cpp
#include <meta>
#include <format>

// Spécialisation automatique de std::formatter pour tout type T
// qui a des membres publics
template <typename T>
    requires (std::meta::nonstatic_data_members_of(
        ^^T, std::meta::access_context::current()).size() > 0)
struct std::formatter<T> {
    constexpr auto parse(std::format_parse_context& ctx) {
        return ctx.begin();
    }

    auto format(const T& obj, std::format_context& ctx) const {
        auto out = ctx.out();
        out = std::format_to(out, "{}{{", std::meta::identifier_of(^^T));

        constexpr auto members = std::meta::nonstatic_data_members_of(
            ^^T, std::meta::access_context::current());
        bool first = true;

        template for (constexpr auto m : members) {
            if (!first) out = std::format_to(out, ", ");
            first = false;
            out = std::format_to(out, ".{}={}", 
                                 std::meta::identifier_of(m), obj.[:m:]);
        }

        return std::format_to(out, "}}");
    }
};

// Toute struct est maintenant formattable automatiquement
struct Point { double x, y; };  
std::print("{}\n", Point{3.14, 2.72});  
// Point{.x=3.14, .y=2.72}
```

## Ce que la réflexion ne fait PAS (en C++26)

La proposition P2996 est volontairement un « MVP ». Certaines fonctionnalités sont reportées à C++29 :

**Pas d'injection de code (token injection).** La réflexion C++26 permet d'inspecter et de splicer des entités existantes, mais pas de générer du code arbitraire (nouvelles classes, nouvelles fonctions). La proposition P3294 (code injection via token sequences) est en cours pour C++29.

**Pas de splice template arguments.** La possibilité d'utiliser un splice comme argument template (`std::vector<[: type_refl :]>`) a été reportée à C++29 — le mécanisme est complexe et l'implémentation n'est pas suffisamment mature.

**Pas de réflexion d'expressions.** On ne peut pas réfléchir une expression arbitraire (`^^(a + b)`). Seules les entités nommées (types, variables, fonctions, enumerateurs, namespaces) sont réfléchissables. Le support des expressions est prévu via une syntaxe `^^(expr)` dans une version future.

**Pas de réflexion runtime.** Toute la réflexion opère à la compilation. Il n'y a pas de RTTI étendu ni de métadonnées de réflexion dans le binaire. C'est un choix de design conforme à la philosophie « zero-overhead » du C++.

## Implémentations disponibles

En mars 2026, deux implémentations de référence existent :

- **Bloomberg/Clang-P2996** — Un fork de Clang avec le support le plus complet de P2996. Disponible sur Compiler Explorer. Développé activement. C'est l'implémentation de référence utilisée pour valider la proposition.

- **EDG** — L'implémentation du frontend EDG (utilisé notamment par le compilateur NVIDIA) supporte une grande partie de P2996. Également disponible sur Compiler Explorer.

- **GCC** — Le support de la réflexion a été fusionné dans le trunk de GCC, rendant les fonctionnalités basiques accessibles dans les builds récentes.

Les trois compilateurs de production (GCC, Clang mainline, MSVC) travaillent à l'intégration, mais le support complet et stable prendra vraisemblablement jusqu'en 2027-2028. En attendant, Compiler Explorer avec les forks expérimentaux est le meilleur terrain de jeu.

## Bonnes pratiques

**Utiliser la réflexion pour éliminer le boilerplate, pas pour l'obfusquer.** Le code réflexif doit être plus simple que l'alternative manuelle. Si la version avec réflexion est plus difficile à lire que le code explicite, elle n'a pas de raison d'être.

**Encapsuler la métaprogrammation dans des fonctions utilitaires.** Écrire une fois `enum_to_string<E>`, `to_json<T>`, `structural_equal<T>`, puis les réutiliser partout. Le code applicatif ne devrait pas manipuler directement `std::meta::info` sauf dans ces utilitaires.

**Tester sur Compiler Explorer en attendant le support compilateur.** Le fork Bloomberg/Clang et l'implémentation EDG permettent d'expérimenter dès maintenant. La syntaxe est stable — le code écrit aujourd'hui fonctionnera avec les compilateurs de production quand le support sera disponible.

**Penser à l'accès aux membres.** Le paramètre `access_context` dans `nonstatic_data_members_of` contrôle quels membres sont visibles. Pour une sérialisation qui doit accéder aux membres privés, le code réflexif doit se trouver dans un contexte ayant l'accès approprié (friend, membre de la classe, etc.).

**Anticiper l'injection de code (C++29).** La réflexion C++26 est la fondation. L'injection de code (P3294) en sera l'extension naturelle. Concevoir ses utilitaires réflexifs de manière à être extensibles quand l'injection arrivera.
