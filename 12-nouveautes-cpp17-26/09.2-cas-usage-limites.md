# 12.9.2 Cas d'usage et limites

## Où les flat containers brillent — et où ils ne conviennent pas

La section 12.9.1 a quantifié les performances. Cette section complète le tableau en présentant des cas d'usage concrets issus de projets réels, puis en détaillant les limites techniques et les situations où `std::flat_map` et `std::flat_set` ne sont pas le bon choix.

## Cas d'usage concrets

### Tables de configuration

Le cas d'usage le plus naturel. Les fichiers de configuration sont chargés au démarrage, parsés une fois, puis consultés des milliers de fois pendant l'exécution du programme. Le ratio lecture/écriture est quasi infini :

```cpp
#include <flat_map>
#include <string>
#include <expected>

struct AppConfig {
    std::flat_map<std::string, std::string> settings;

    std::expected<void, std::string> load(const std::string& path) {
        auto entries = parse_config_file(path);  // vector<pair<string,string>>
        if (!entries) return std::unexpected(entries.error());

        std::ranges::sort(*entries, {}, &std::pair<std::string, std::string>::first);

        std::vector<std::string> keys, values;
        keys.reserve(entries->size());
        values.reserve(entries->size());
        for (auto& [k, v] : *entries) {
            keys.push_back(std::move(k));
            values.push_back(std::move(v));
        }
        settings = std::flat_map<std::string, std::string>(
            std::sorted_unique, std::move(keys), std::move(values));
        return {};
    }

    std::string_view get(const std::string& key, 
                          std::string_view default_val = "") const {
        auto it = settings.find(key);   // Recherche binaire — rapide
        return it != settings.end() ? it->second : default_val;
    }
};
```

La construction avec `sorted_unique` élimine le coût de tri interne, et chaque appel à `get()` bénéficie de la localité mémoire. Pour une configuration typique de quelques dizaines à quelques centaines de paramètres, le conteneur tient intégralement en cache L1.

### Registres de handlers et dispatch tables

Les systèmes qui associent des identifiants (codes d'opération, types de message, routes HTTP) à des fonctions de traitement constituent un autre cas idéal. Le registre est construit au démarrage et consulté à chaque requête :

```cpp
#include <flat_map>
#include <functional>
#include <string>

class Router {
    using Handler = std::function<Response(const Request&)>;
    std::flat_map<std::string, Handler> routes_;

public:
    void add_route(std::string path, Handler handler) {
        routes_.emplace(std::move(path), std::move(handler));
    }

    Response dispatch(const Request& req) const {
        if (auto it = routes_.find(req.path()); it != routes_.end()) {
            return it->second(req);
        }
        return Response::not_found();
    }
};
```

Pour un serveur HTTP avec quelques dizaines à quelques centaines de routes, le `flat_map` offre un dispatch plus rapide que `std::map` et une itération ordonnée naturelle pour lister les routes disponibles (documentation, health checks).

### Caches en lecture seule et index

Les index construits à partir de données chargées depuis une base de données, un fichier ou un service distant entrent dans le profil « build once, query many » :

```cpp
#include <flat_map>
#include <flat_set>

// Index des utilisateurs par ID — construit au démarrage
std::flat_map<uint64_t, UserProfile> user_index;

// Ensemble des tags connus — construit depuis la base
std::flat_set<std::string> known_tags;

// Lookup rapide pendant toute la durée de vie du service
bool is_known_tag(const std::string& tag) const {
    return known_tags.contains(tag);  // Recherche binaire cache-friendly
}
```

### Sérialisation et transport de données

Les flat containers se sérialisent naturellement parce que leurs données sous-jacentes sont de simples vecteurs. Cela simplifie considérablement l'écriture vers des formats binaires, le transfert réseau, ou le stockage en mémoire partagée :

```cpp
#include <flat_map>
#include <vector>

// Extraction pour sérialisation
std::flat_map<int, double> measurements = /* ... */;
auto [keys, values] = std::move(measurements).extract();

// keys et values sont des vector<int> et vector<double> ordinaires
// → sérialisation triviale (memcpy, protobuf, flatbuffers, etc.)
serialize(keys);
serialize(values);

// Désérialisation et reconstruction
auto deserialized_keys = deserialize<std::vector<int>>(stream);
auto deserialized_values = deserialize<std::vector<double>>(stream);
std::flat_map<int, double> restored(
    std::sorted_unique,
    std::move(deserialized_keys),
    std::move(deserialized_values));
```

Avec `std::map`, la sérialisation nécessite une itération complète et la désérialisation requiert n insertions dans l'arbre. Avec les flat containers, c'est un transfert de mémoire brut.

### Données embarquées et constexpr

Les flat containers s'intègrent bien dans les environnements contraints en mémoire. En combinant un conteneur sous-jacent de taille fixe (comme `std::array` encapsulé dans un adaptateur approprié), on peut créer des lookup tables sans allocation dynamique.

Pour les tables constantes connues à la compilation, un `std::array` trié avec `std::ranges::lower_bound` offre le même profil de performance qu'un `flat_map` sans aucune dépendance runtime :

```cpp
#include <algorithm>
#include <array>
#include <ranges>
#include <string_view>

struct Entry {
    std::string_view key;
    int value;
};

// Table constante triée — recherche binaire à coût zéro
constexpr std::array<Entry, 4> lookup_table = {{
    {"error",   3},
    {"info",    1},
    {"trace",   0},
    {"warning", 2},
}};

constexpr int log_level(std::string_view name) {
    auto it = std::ranges::lower_bound(
        lookup_table, name, {}, &Entry::key);
    if (it != lookup_table.end() && it->key == name) {
        return it->value;
    }
    return -1;
}

static_assert(log_level("info") == 1);
static_assert(log_level("unknown") == -1);
```

Ce pattern est le pendant compile-time du `flat_map` runtime. Il est utile pour les mappings statiques qui ne changent jamais.

### Conteneurs ordonnés dans les structures de données composites

Quand un flat container est membre d'une structure copiée ou déplacée fréquemment, son comportement de déplacement est optimal — déplacer deux vecteurs est une opération O(1) :

```cpp
struct QueryResult {
    std::flat_map<std::string, double> metrics;
    std::flat_set<std::string> tags;
    std::string query_id;
    // ... déplacement O(1) pour l'ensemble de la structure
};

// Retour par valeur — move semantics optimales
QueryResult execute_query(const Query& q) {
    QueryResult result;
    // ... remplir result ...
    return result;   // NRVO ou move — les flat_map/flat_set sont déplacés en O(1)
}
```

Avec `std::map`, le déplacement est aussi O(1) (transfert de la racine de l'arbre), mais la copie est O(n) avec n allocations individuelles. La copie d'un `flat_map` est O(n) mais avec une seule allocation par vecteur, ce qui est considérablement plus rapide grâce à la copie mémoire contiguë (`memcpy` optimisé).

## Limites techniques

### Insertions et suppressions O(n)

C'est la limite fondamentale. Chaque insertion dans un `flat_map` déjà peuplé nécessite de décaler tous les éléments situés après le point d'insertion. Pour un conteneur de n éléments, une insertion au milieu déplace en moyenne n/2 éléments :

```cpp
std::flat_map<int, std::string> fm;

// Les premières insertions sont rapides (conteneur petit)
for (int i = 0; i < 100; ++i) {
    fm[i] = std::format("value_{}", i);   // OK — conteneur petit
}

// Mais insérer dans un conteneur de 100 000 éléments est coûteux
// Chaque insertion décale potentiellement ~50 000 éléments
```

**Mitigation :** regrouper les insertions en batch. Plutôt que d'insérer un par un, accumuler les éléments dans un vecteur temporaire, trier, fusionner avec les données existantes, puis reconstruire le flat container. Certaines implémentations proposent des méthodes `insert(sorted_range)` optimisées pour ce pattern.

### Types non-déplaçables ou à déplacement coûteux

Les opérations de décalage lors des insertions et suppressions reposent sur le déplacement des éléments. Si le type stocké n'est pas déplaçable (move constructor supprimé) ou si son déplacement est coûteux (copie de buffers internes, par exemple), les performances des flat containers se dégradent fortement :

```cpp
struct HeavyObject {
    std::array<char, 4096> buffer;   // 4 Ko — coûteux à déplacer
    // Le move constructor copie les 4 Ko (pas de pointeur à transférer)
};

// Chaque insertion déplace potentiellement des mégaoctets
std::flat_map<int, HeavyObject> fm;  // Déconseillé si insertions fréquentes
```

Pour ces types, `std::map` est préférable : ses nœuds sont alloués individuellement et jamais déplacés après insertion.

**Mitigation :** stocker des `std::unique_ptr<HeavyObject>` dans le flat container. Le déplacement d'un `unique_ptr` est O(1) quelle que soit la taille de l'objet pointé. Le prix : une indirection supplémentaire lors de l'accès et une perte partielle de la localité mémoire pour les valeurs (les clés restent contiguës).

### Invalidation des itérateurs

Comme détaillé en section 12.9, toute modification structurelle (insertion, suppression, `operator[]` sur une clé inexistante) invalide potentiellement tous les itérateurs, références et pointeurs vers les éléments. Cela interdit certains patterns courants avec `std::map` :

```cpp
// Pattern valide avec std::map, INVALIDE avec std::flat_map
std::flat_map<std::string, int> fm = {{"a", 1}, {"b", 2}, {"c", 3}};

for (auto it = fm.begin(); it != fm.end(); ) {
    if (it->second < 2) {
        it = fm.erase(it);   // Après erase, TOUS les itérateurs sont invalidés
                              // Mais erase retourne un itérateur valide vers le suivant
    } else {
        ++it;
    }
}
```

Le pattern `erase` dans une boucle fonctionne grâce au retour d'itérateur de `erase()`, mais chaque suppression est O(n) — la boucle entière est O(n²) dans le pire cas. Avec `std::map`, elle est O(n log n).

**Mitigation :** utiliser `std::erase_if` (C++20), qui effectue les suppressions en une seule passe optimisée :

```cpp
std::erase_if(fm, [](const auto& pair) {
    return pair.second < 2;
});
```

### Pas de node handles ni merge

`std::map` et `std::set` (depuis C++17) supportent les *node handles* — des objets qui encapsulent un nœud extrait du conteneur et peuvent être insérés dans un autre conteneur sans copie ni allocation. Les flat containers ne supportent pas ce mécanisme :

```cpp
// Possible avec std::map
auto node = map1.extract("key");
map2.insert(std::move(node));     // Transfert sans copie

// Impossible avec std::flat_map — pas de node handles
// Il faut copier ou déplacer l'élément manuellement
```

De même, `std::map::merge` transfère des nœuds d'un conteneur à un autre en O(n log n) sans allocation. Cette opération n'a pas d'équivalent efficace pour les flat containers.

### Pas de garantie d'adresse stable

Certains patterns avancés reposent sur le fait que l'adresse d'un élément dans un `std::map` ne change jamais après insertion (tant que l'élément n'est pas supprimé). C'est garanti par le standard pour les conteneurs à nœuds, mais pas pour les flat containers :

```cpp
std::flat_map<std::string, Config> configs;
configs["database"] = load_db_config();

// DANGER : l'adresse peut changer après une nouvelle insertion
Config* db_ptr = &configs["database"];
configs["cache"] = load_cache_config();   // Réallocation possible
// db_ptr est potentiellement invalide !
```

Ce problème affecte tout code qui stocke des pointeurs ou des références vers les éléments d'un flat container. C'est la même contrainte que pour `std::vector`.

### Multimap et multiset

C++23 fournit aussi `std::flat_multimap` et `std::flat_multiset` pour les cas où des clés dupliquées sont nécessaires. Les mêmes avantages et limites s'appliquent, avec une nuance : la recherche dans un flat_multimap retourne un itérateur vers le premier élément d'un groupe de clés identiques, et l'itération sur ce groupe est contiguë — un avantage par rapport à `std::multimap` où les nœuds de clés identiques peuvent être dispersés en mémoire.

## Grille de décision finale

Pour choisir le bon conteneur associatif ordonné :

```
Le conteneur est-il petit (< 100 éléments) ?
  └─ OUI → std::flat_map (gagne sur tous les axes)

Les données sont-elles construites en batch puis consultées ?
  └─ OUI → std::flat_map avec sorted_unique

Les insertions/suppressions sont-elles fréquentes sur un grand conteneur ?
  └─ OUI → std::map

La stabilité des itérateurs/références est-elle requise ?
  └─ OUI → std::map

Les types stockés sont-ils coûteux à déplacer ?
  └─ OUI → std::map (ou flat_map avec unique_ptr)

L'ordre n'est pas nécessaire et la recherche O(1) est prioritaire ?
  └─ OUI → std::unordered_map

La mémoire est-elle contrainte ?
  └─ OUI → std::flat_map (overhead minimal)

La sérialisation est-elle fréquente ?
  └─ OUI → std::flat_map (extract/replace trivial)
```

En cas de doute, commencer avec `std::flat_map` pour les conteneurs ordonnés. Si le profiling révèle un goulot d'étranglement sur les insertions, migrer vers `std::map` — le changement de type est quasi transparent grâce à la compatibilité d'API.

## Bonnes pratiques

**Profiler avant de choisir.** Les chiffres de la section 12.9.1 sont des ordres de grandeur. Le gain réel dépend de la taille des types, du pattern d'accès, de l'architecture CPU et de l'état du cache. Google Benchmark (section 35.1) est l'outil approprié pour mesurer.

**Construire en batch chaque fois que possible.** Accumuler les données, trier, puis construire avec `sorted_unique`. Éviter les insertions unitaires dans un grand flat container.

**Utiliser `extract()` et `replace()` pour les modifications groupées.** Extraire les vecteurs internes, les modifier avec les algorithmes standard, puis les réinjecter est souvent plus efficace que des modifications individuelles.

**Préférer `std::erase_if` pour les suppressions conditionnelles.** C'est plus efficace et plus sûr qu'une boucle manuelle avec `erase()`.

**Documenter le choix du conteneur.** Un commentaire expliquant pourquoi `flat_map` a été choisi (profil lecture-dominant, taille modérée, construction en batch) aide les mainteneurs futurs à comprendre le compromis et à ne pas introduire accidentellement un pattern d'insertion fréquente.
