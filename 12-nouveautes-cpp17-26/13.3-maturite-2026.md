# 12.13.3 Maturité en 2026 : Ce qui fonctionne, ce qui reste limité

## Le verdict pratique

Les sections précédentes ont détaillé les avantages théoriques des modules (12.13.1) et l'état technique du support compilateur et outillage (12.13.2). Cette section synthétise le tout en un verdict pratique : qu'est-il raisonnable de faire avec les modules en mars 2026 ? Quels projets peuvent les adopter dès maintenant, et lesquels ont intérêt à attendre ?

## Ce qui fonctionne — utilisable en production

### import std; pour les nouveaux projets

Le cas d'usage le plus immédiat et le plus bénéfique est `import std;`. Remplacer la dizaine de `#include` de headers standard par une seule ligne est simple, sans risque, et apporte un gain de compilation mesurable. Les trois compilateurs majeurs le supportent, CMake 3.28+ le gère, et la migration ne nécessite aucune réécriture du code applicatif — juste le remplacement des includes standards.

```cpp
// Avant : 8 lignes d'includes
#include <vector>
#include <string>
#include <map>
#include <algorithm>
#include <ranges>
#include <print>
#include <optional>
#include <expected>

// Après : 1 ligne
import std;
```

Pour un nouveau projet en C++23, `import std;` est la recommandation par défaut. Le gain en temps de compilation est immédiat (les dizaines de milliers de lignes des headers standard sont compilées une seule fois en BMI), et le risque est minimal puisque le module `std` est fourni et maintenu par les implémenteurs de la bibliothèque standard eux-mêmes.

### Modules internes dans un projet mono-compilateur

Pour les projets qui ciblent un seul compilateur (situation courante en entreprise), écrire ses propres modules pour le code interne fonctionne bien. Les modules nommés, les module implementation units et les partitions de base sont stables dans les trois compilateurs. L'encapsulation réelle (`export` vs non-exporté), l'isolation des macros et les gains en compilation incrémentale se concrétisent immédiatement.

Le profil de projet idéal pour cette adoption :

- Un nouveau projet, ou un composant neuf dans un projet existant.
- Un seul compilateur cible (GCC 15 **ou** Clang 20 **ou** MSVC).
- CMake 3.28+ avec Ninja (ou MSBuild pour MSVC).
- Pas de dépendance à des bibliothèques tierces qui exigeraient d'être elles-mêmes modulaires.

### Modularisation « feuille d'abord »

Pour les projets existants, la stratégie qui fonctionne en 2026 est la modularisation progressive en partant des composants « feuilles » — ceux qui n'ont pas de dépendances internes au projet (ou peu) et qui sont consommés par de nombreux autres fichiers :

```
Approche recommandée : bottom-up

  ┌──────────────────────────────────────────────┐
  │  Application (main, logique métier)          │ ← Dernier à migrer
  │  #include "service.h"                        │
  ├──────────────────────────────────────────────┤
  │  Couche service                              │ ← Migrer ensuite
  │  #include "utils.h"                          │
  ├──────────────────────────────────────────────┤
  │  Utilitaires (logging, config, helpers)      │ ← Migrer en premier
  │  export module utils;                        │   (composants feuilles)
  └──────────────────────────────────────────────┘
```

Les utilitaires de bas niveau deviennent des modules. Les couches supérieures les importent, tout en continuant à utiliser `#include` pour leurs propres dépendances non encore modulaires. Le *global module fragment* (section 12.13.2) assure la coexistence des deux modèles.

## Ce qui fonctionne — avec précautions

### Projets multi-compilateurs

Un projet qui doit compiler avec GCC **et** Clang (cas fréquent sur Linux) peut utiliser les modules, mais avec des précautions :

- Les BMI doivent être générés séparément pour chaque compilateur — ils ne sont pas interchangeables.
- Les extensions de fichier diffèrent entre écosystèmes (`.cppm` est le compromis le plus portable).
- Certains patterns peuvent déclencher un ICE sur un compilateur et pas l'autre. Des tests CI sur les deux compilateurs sont indispensables.
- CMake gère cette situation correctement (il régénère les BMI pour chaque toolchain), mais la matrice de test s'élargit.

En pratique, cela fonctionne pour les modules simples (peu de templates, pas de patterns de métaprogrammation exotiques). Pour les modules qui font un usage intensif de templates avec concepts, de CRTP, ou de SFINAE résiduel, le risque de bugs compilateur divergents est plus élevé.

### Templates complexes aux frontières de module

Les templates définis dans un module et instanciés par les importateurs fonctionnent dans les cas courants. Les cas complexes — templates variadiques avec contraintes imbriquées, spécialisation partielle à travers les frontières de module, interaction entre concepts et friend declarations — peuvent encore déclencher des bugs dans certains compilateurs.

La recommandation : pour les bibliothèques template-heavy destinées à être consommées comme modules, tester intensivement avec chaque compilateur cible et prévoir un fallback header pour les cas problématiques.

## Ce qui reste limité

### Bibliothèques tierces : le blocage principal

C'est le frein numéro un à l'adoption généralisée. L'immense majorité des bibliothèques de l'écosystème C++ — Boost, Abseil, Eigen, {fmt}, nlohmann/json, gRPC, OpenSSL, et des milliers d'autres — ne fournissent pas d'interfaces de module et n'en prévoient pas à court terme.

Les raisons sont multiples :

- **Compatibilité ascendante** — Les bibliothèques doivent supporter des compilateurs plus anciens qui n'ont pas de support des modules.
- **Complexité de la migration** — Modulariser une bibliothèque comme Boost, avec ses centaines de headers interdépendants, ses macros de configuration, et ses spécialisations conditionnelles, est un chantier considérable.
- **Absence de standard de distribution** — Il n'existe pas encore de mécanisme standardisé pour distribuer des modules pré-compilés via les gestionnaires de paquets (Conan, vcpkg, apt).

En attendant, le *global module fragment* et les *header units* permettent de consommer ces bibliothèques depuis du code modulaire, mais sans les bénéfices de compilation que des modules natifs apporteraient.

### Distribution et packaging de modules

Comment distribuer une bibliothèque sous forme de module ? En 2026, il n'y a pas de réponse standardisée. Les BMI ne sont pas portables entre compilateurs ni entre versions d'un même compilateur. Distribuer des fichiers `.cppm`/`.ixx` source est possible, mais impose au consommateur de les compiler — ce qui réintroduit une partie de la complexité que les modules étaient censés éliminer.

Le Common Package Specification (CPS), un effort en cours pour standardiser les métadonnées de paquets C++, intègre progressivement la notion de modules, mais le travail n'est pas achevé. Les gestionnaires de paquets (Conan, vcpkg) n'ont pas encore de support mature pour la distribution de modules.

C'est un problème de poule et d'œuf : les bibliothèques n'adoptent pas les modules tant que la distribution n'est pas résolue, et la distribution ne progresse pas tant que les bibliothèques n'adoptent pas les modules.

### Meson et certains build systems

Comme noté en section 12.13.2, Meson — build system utilisé par de nombreux projets Linux majeurs (GNOME, systemd, Mesa, PipeWire) — ne supporte pas encore les modules C++20 dans ses versions stables. Pour les projets qui dépendent de Meson, l'adoption des modules est bloquée par le build system.

Bazel, utilisé à grande échelle chez Google, n'a qu'un support expérimental via des règles communautaires. Pour les projets utilisant Bazel, l'adoption des modules est prématurée.

### Debugging et profiling

Le support du debugging des modules dans GDB et LLDB s'améliore mais n'est pas encore au niveau du debugging du code classique. Les informations de debug (DWARF) pour les entités importées depuis un module peuvent être incomplètes dans certaines configurations. Le « Go to Definition » dans les IDE peut parfois mener au BMI plutôt qu'au fichier source, selon la qualité de l'intégration.

Le profiling avec `perf` et les flamegraphs n'est pas directement affecté par les modules (il opère au niveau du binaire), mais les outils d'analyse statique (clang-tidy, cppcheck) sont encore en train d'adapter leur support.

### Interopérabilité entre modules et headers dans le même target

Mélanger des fichiers modulaires et des fichiers classiques (#include) dans un même target CMake fonctionne, mais peut introduire des subtilités d'ordonnancement. CMake gère correctement la plupart des cas, mais des configurations non triviales (dépendances circulaires entre un module et un header, includes conditionnels dans le global module fragment) peuvent poser des problèmes.

## Grille de décision : adopter les modules en 2026 ?

```
Question                                          → Recommandation
─────────────────────────────────────────────────────────────────────

Nouveau projet C++23, un seul compilateur ?        → OUI, import std;
                                                     + modules internes

Nouveau projet, multi-compilateurs ?               → import std; OUI
                                                     Modules internes
                                                     avec tests CI croisés

Projet existant, composants feuilles isolés ?      → OUI, migration
                                                     progressive bottom-up

Bibliothèque open-source multi-plateforme ?        → ATTENDRE pour les
                                                     interfaces de module
                                                     (garder les headers)

Projet utilisant Meson ou Bazel ?                  → ATTENDRE le support
                                                     du build system

Projet embarqué / compilateur spécialisé ?         → ATTENDRE (vérifier
                                                     le support du compilo)

Projet avec contrainte de support C++17 ?          → NON — les modules
                                                     requièrent C++20 min.
```

## Trajectoire et horizon

La direction est claire : les modules sont le futur du modèle de compilation C++. Les headers ne seront pas retirés du langage (la compatibilité ascendante est un principe fondamental du C++), mais les modules deviendront progressivement le modèle recommandé pour le nouveau code.

L'horizon réaliste pour une adoption large se dessine ainsi :

**2026 (maintenant)** — `import std;` est prêt pour la production. Les modules internes sont utilisables dans les projets contrôlés (mono-compilateur, CMake+Ninja). La migration progressive des composants feuilles commence.

**2027-2028** — Les bibliothèques majeures commencent à fournir des interfaces de module en option, parallèlement à leurs headers. Les gestionnaires de paquets intègrent le support de distribution de modules. Meson et Bazel atteignent la maturité sur les modules.

**2029+** — Les modules deviennent le mode par défaut pour les nouveaux projets. Les formations et les livres enseignent les modules en premier, les headers en second. L'écosystème atteint la masse critique.

Cette trajectoire est cohérente avec l'histoire du C++ : les fonctionnalités majeures (smart pointers, move semantics, lambdas) ont toutes mis 5 à 8 ans entre leur standardisation et leur adoption généralisée. Les modules, étant un changement plus profond (infrastructure de build, pas seulement syntaxe), suivent un cycle naturellement plus long.

## Bonnes pratiques pour l'adoption en 2026

**Adopter `import std;` immédiatement.** C'est le gain le plus simple pour le coût le plus faible. Aucune raison d'attendre pour les projets en C++23.

**Modulariser progressivement, en partant des feuilles.** Ne pas tenter une migration big-bang. Convertir un composant à la fois, tester, valider, puis passer au suivant.

**Garder un fallback header pour les bibliothèques.** Si vous écrivez une bibliothèque destinée à être consommée par d'autres projets, fournissez les headers classiques en plus des interfaces de module. Les consommateurs qui ne peuvent pas encore utiliser les modules (compilateur ancien, build system non supporté) doivent pouvoir continuer à fonctionner.

**Tester avec les dernières versions des compilateurs.** Le support des modules s'améliore à chaque release. Les bugs rencontrés avec GCC 14 sont souvent corrigés dans GCC 15. Rester sur les versions récentes maximise la stabilité.

**Suivre l'écosystème.** Le rapport modules de la communauté (github.com/royjacobson/modules-report) et les discussions du Study Group 15 (SG15) du comité C++ sont les meilleures sources pour suivre l'évolution de l'outillage et des pratiques.

**Ne pas optimiser prématurément la structure des modules.** Commencer avec des modules simples (un module = un composant logique). Les partitions, les sous-modules et les architectures modulaires élaborées pourront être introduits plus tard, quand l'expérience et la stabilité du support le permettront.
