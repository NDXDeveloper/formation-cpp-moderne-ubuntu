# 12.14.3 Pattern Matching : match, is et as

## Inspecter les données, pas les interroger

Le pattern matching est la capacité d'examiner une valeur, de la comparer simultanément à plusieurs formes possibles (patterns), et d'exécuter le code correspondant au pattern qui correspond. C'est un mécanisme fondamental dans les langages fonctionnels (Haskell, ML, Erlang) et dans les langages modernes qui s'en inspirent (Rust `match`, Kotlin `when`, Swift `switch`). Le C++ en est privé depuis toujours — forçant les développeurs à enchaîner des `if`/`else if`, des `switch` limités aux entiers, ou le pattern `std::visit` + `overloaded` (section 12.2) pour les variants.

La proposition P2688 (`match` expression) porte le pattern matching vers le C++. C'est l'une des fonctionnalités les plus attendues par la communauté. La proposition a reçu un consensus favorable du comité, dispose d'une implémentation expérimentale sur Compiler Explorer, et continue d'évoluer activement.

> ⚠️ **État en mars 2026** — Le pattern matching n'a pas atteint la maturité nécessaire pour être intégré dans C++26 avant le gel des fonctionnalités de juin 2025. La proposition est désormais ciblée pour **C++29**. Nous couvrons néanmoins le design proposé car il est suffisamment stable pour être étudié, l'implémentation expérimentale est disponible, et la fonctionnalité arrivera dans le prochain standard. Comprendre le pattern matching dès maintenant prépare à son adoption future.

## Le problème : les limites du dispatch actuel

### switch : limité aux entiers

Le `switch` de C++ est hérité du C et se limite aux types entiers et aux enums. On ne peut pas faire de `switch` sur une `std::string`, un `std::variant`, ou un type structuré :

```cpp
std::string command = get_command();

// Impossible :
// switch (command) {
//     case "quit": ...
//     case "help": ...
// }

// On est forcé d'écrire :
if (command == "quit") { /* ... */ }  
else if (command == "help") { /* ... */ }  
else if (command == "save") { /* ... */ }  
else { /* ... */ }  
```

### std::visit + overloaded : fonctionnel mais lourd

Pour les variants, le pattern `std::visit` avec le helper `overloaded` (section 12.2) est le plus proche du pattern matching, mais la syntaxe est verbeuse et le mécanisme est indirect :

```cpp
using Shape = std::variant<Circle, Rectangle, Triangle>;

double area(const Shape& shape) {
    return std::visit(overloaded{
        [](const Circle& c)    { return 3.14159 * c.radius * c.radius; },
        [](const Rectangle& r) { return r.width * r.height; },
        [](const Triangle& t)  { return 0.5 * t.base * t.height; }
    }, shape);
}
```

Ça fonctionne, mais il faut définir le helper `overloaded`, la syntaxe avec les lambdas imbriquées est lourde, et il n'y a pas de support natif pour les gardes, la déstructuration, ou le matching sur des valeurs constantes.

### if constexpr : compile-time uniquement

`if constexpr` permet de brancher à la compilation sur des conditions de type, mais pas sur des valeurs runtime :

```cpp
template <typename T>  
void process(const T& value) {  
    if constexpr (std::is_integral_v<T>) {
        // ...
    } else if constexpr (std::is_floating_point_v<T>) {
        // ...
    }
}
```

C'est utile mais orthogonal au pattern matching, qui opère sur des valeurs à l'exécution.

## L'expression match : le design proposé

### Syntaxe de base

L'expression `match` évalue un sujet contre une séquence de patterns. Le premier pattern qui correspond est sélectionné :

```cpp
int describe(int x) {
    return x match {
        0     => 0,
        1     => 1,
        let n => n * n    // Capture la valeur dans n
    };
}
```

La syntaxe se lit naturellement : « x match : si c'est 0, retourner 0 ; si c'est 1, retourner 1 ; sinon, capturer la valeur dans n et retourner n × n ».

Chaque branche a la forme `pattern => expression`. L'opérateur `=>` sépare le pattern du résultat. Le `match` est une **expression** — il produit une valeur et peut être utilisé dans une affectation, un retour, un argument de fonction.

### match avec type de retour explicite

Quand les branches produisent des types différents, un type de retour explicite peut être spécifié :

```cpp
auto result = value match -> std::string {
    0     => "zéro",
    1     => "un",
    let n => std::to_string(n)
};
```

### match dans les conditions

Un `match` avec un seul pattern produit un booléen et peut être utilisé dans un `if`, un `while`, ou un `for` :

```cpp
if (point match [0, 0]) {
    std::print("Origine\n");
}

if (opt match some(let value)) {
    std::print("Valeur : {}\n", value);
}
```

Ce pattern à un seul cas est particulièrement utile pour tester et extraire en une seule opération — l'équivalent de `if let` en Rust ou Swift.

## Les types de patterns

### Pattern constant

Compare le sujet à une valeur constante :

```cpp
x match {
    0       => "zéro",
    1       => "un",
    42      => "la réponse",
    let _   => "autre"
};
```

### Pattern wildcard et let

Le mot-clé `let` introduit un nouveau nom dans un pattern. Le wildcard `_` capture sans nommer :

```cpp
x match {
    let n => std::print("valeur : {}\n", n)   // Capture dans n
};

x match {
    let _ => std::print("quelque chose\n")    // Ignore la valeur
};
```

`let` rend le binding explicite — on ne peut pas accidentellement confondre un nom de variable avec une constante. C'est une différence importante avec certains langages où la distinction entre « matcher une constante nommée » et « capturer dans une nouvelle variable » est ambiguë.

### Pattern de déstructuration (structured bindings)

Les tuples, paires et types tuple-like peuvent être déstructurés :

```cpp
std::pair<int, int> point = {3, 4};

point match {
    [0, 0]          => "origine",
    [0, let y]      => std::format("axe Y à {}", y),
    [let x, 0]      => std::format("axe X à {}", x),
    [let x, let y]  => std::format("({}, {})", x, y)
};
```

Les crochets `[...]` déstructurent le sujet en ses composants, exactement comme les structured bindings (section 12.1) mais avec la puissance du matching sur les valeurs individuelles.

### Pattern optionnel (pointer-like)

Les types pointer-like (`std::optional`, pointeurs bruts, `std::unique_ptr`) supportent un pattern `some`/`none` :

```cpp
std::optional<int> opt = get_value();

opt match {
    some(let value) => std::print("Valeur : {}\n", value),
    none            => std::print("Absent\n")
};

// Équivalent pour un pointeur
int* ptr = find_element();  
ptr match {  
    some(let value) => std::print("Trouvé : {}\n", value),
    none            => std::print("Pas trouvé\n")
};
```

### Pattern alternatif (variant-like)

Les types variant-like (`std::variant`, `std::expected`, types polymorphes) supportent le matching sur l'alternative active :

```cpp
using Result = std::variant<int, std::string, std::vector<int>>;

Result r = get_result();

r match {
    let i: int              => std::print("Entier : {}\n", i),
    let s: std::string      => std::print("Chaîne : {}\n", s),
    let v: std::vector<int> => std::print("Vecteur de {} éléments\n", v.size())
};
```

Le pattern `let name: Type` teste si l'alternative active est du type donné et, si oui, lie la valeur à `name`. C'est l'équivalent direct du `std::visit` avec `overloaded`, mais avec une syntaxe native et la possibilité d'ajouter des gardes et de la déstructuration.

Pour `std::expected` (section 12.8) :

```cpp
auto result = load_config("app.yaml");

result match {
    let cfg: Config     => use_config(cfg),
    let err: FileError  => handle_error(err)
};
```

## Gardes : conditions additionnelles

Un pattern peut être accompagné d'une garde — une condition booléenne supplémentaire qui doit être satisfaite pour que le pattern corresponde :

```cpp
int classify(int x) {
    return x match {
        0                     => "zéro",
        let n if (n > 0)      => "positif",
        let n if (n < 0)      => "négatif"
    };
}

// Combinaison déstructuration + garde
point match {
    [let x, let y] if (x == y) => "diagonal",
    [let x, let y] if (x == 0) => "axe Y",
    [let x, let y] if (y == 0) => "axe X",
    [let x, let y]              => "quelconque"
};
```

Les gardes sont entourées de parenthèses (`if (condition)`) et sont évaluées uniquement si le pattern structurel a déjà correspondu. Elles permettent d'exprimer des conditions arbitraires que le pattern seul ne peut pas capturer.

## match constexpr

L'expression `match` peut être évaluée à la compilation avec le qualificateur `constexpr` :

```cpp
constexpr auto result = 42 match constexpr {
    0     => "zéro",
    let n => "non-zéro"
};

static_assert(result == "non-zéro");
```

Cela permet d'utiliser le pattern matching dans des contextes `consteval` et `constexpr`, en cohérence avec la tendance du C++ moderne à déplacer toujours plus de calcul à la compilation.

## Comparaison avec les approches existantes

### match vs std::visit + overloaded

```cpp
// std::visit + overloaded (C++17)
std::visit(overloaded{
    [](int i)                { return std::format("int: {}", i); },
    [](const std::string& s) { return std::format("str: {}", s); },
    [](double d)             { return std::format("dbl: {}", d); }
}, value);

// match (proposé)
value match -> std::string {
    let i: int         => std::format("int: {}", i),
    let s: std::string => std::format("str: {}", s),
    let d: double      => std::format("dbl: {}", d)
};
```

Le `match` est plus concis, ne nécessite pas de helper `overloaded`, et supporte nativement les gardes et les patterns composés. `std::visit` reste disponible et utile — notamment pour les cas où le callable est déjà défini sous forme de fonction ou d'objet fonctionnel — mais `match` sera l'idiome de choix pour le dispatch sur variants dans du code nouveau.

### match vs switch

```cpp
// switch classique (entiers uniquement)
switch (status) {
    case Status::OK:       return handle_ok();
    case Status::Error:    return handle_error();
    case Status::Pending:  return handle_pending();
    default:               return handle_unknown();
}

// match (tout type supporté + déstructuration)
status match {
    Status::OK      => handle_ok(),
    Status::Error   => handle_error(),
    Status::Pending => handle_pending(),
    let _           => handle_unknown()
};
```

Le `match` est une expression (pas un statement), ne nécessite pas de `break`, et supporte des types bien au-delà des entiers. Le `switch` classique reste valide pour les cas simples sur des entiers et des enums, mais `match` le subsume.

### match vs if/else if

```cpp
// Chaîne de if/else if
if (auto* circle = std::get_if<Circle>(&shape)) {
    return area(*circle);
} else if (auto* rect = std::get_if<Rectangle>(&shape)) {
    return area(*rect);
} else if (auto* tri = std::get_if<Triangle>(&shape)) {
    return area(*tri);
}

// match
shape match {
    let c: Circle    => area(c),
    let r: Rectangle => area(r),
    let t: Triangle  => area(t)
};
```

Le `match` élimine la cascade de `if`/`else if` avec des `get_if` manuels, et le compilateur peut vérifier l'exhaustivité (tous les cas couverts).

## Exhaustivité

L'un des avantages les plus importants du pattern matching est la **vérification d'exhaustivité** à la compilation. Si un cas n'est pas couvert, le compilateur produit un avertissement ou une erreur :

```cpp
enum class Direction { North, South, East, West };

Direction dir = get_direction();

// Le compilateur signale que Direction::West n'est pas couvert
dir match {
    Direction::North => go_north(),
    Direction::South => go_south(),
    Direction::East  => go_east()
    // Manque : Direction::West => ...
};
```

C'est le même mécanisme que celui qui rend le `match` de Rust si fiable : quand on ajoute une variante à un enum, le compilateur force la mise à jour de tous les points de matching. Les oublis sont détectés à la compilation, pas en production.

## Implémentation expérimentale

Une implémentation de P2688 basée sur un fork de Clang est disponible sur Compiler Explorer (sous le label « x86-64 clang (pattern matching - P2688) », avec le flag `-fpattern-matching`). La plupart des patterns proposés sont implémentés : wildcards, constantes, déstructuration, optionnels, alternatives, gardes, et évaluation constexpr.

Cette implémentation permet d'expérimenter dès maintenant avec la syntaxe et de se familiariser avec les idiomes du pattern matching en C++, en attendant le support dans les compilateurs de production.

## Perspective : is et as

Parallèlement à P2688 (`match`), Herb Sutter a proposé `is` et `as` (P2392) — deux opérateurs qui étendent le pattern matching au-delà de l'expression `match`, en le rendant utilisable dans les conditions et les expressions courantes :

```cpp
// is : tester si une valeur correspond à un pattern
if (value is int) { /* ... */ }  
if (shape is Circle) { /* ... */ }  
if (x is 42) { /* ... */ }  

// as : extraire une valeur en la castant de manière sûre
auto& c = shape as Circle;   // Lève une exception si shape n'est pas un Circle
```

`is` est un opérateur booléen de test de correspondance, et `as` est un opérateur de conversion sûre. Combinés avec `match`, ils forment un écosystème cohérent pour l'inspection de données. Bjarne Stroustrup a exprimé un soutien fort à l'approche `is`/`as` car elle est extensible par l'utilisateur via la surcharge d'opérateurs — cohérent avec la tradition du C++.

La relation entre `match`, `is` et `as` continue d'évoluer au sein du comité, et leur forme finale pour C++29 sera le résultat de cette convergence.

## Se préparer dès maintenant

Même si le pattern matching est ciblé pour C++29, plusieurs pratiques préparent son adoption :

**Utiliser `std::variant` plutôt que les hiérarchies de classes pour les types somme.** Le pattern matching s'intègre naturellement avec les variants. Concevoir ses types de données en « types somme » (une valeur qui est l'un de plusieurs types) prépare idéalement la migration vers `match`.

**Structurer le code avec `std::visit` + `overloaded`.** C'est le pattern le plus proche de `match` disponible aujourd'hui. Le code utilisant `visit` se convertira naturellement vers `match` quand celui-ci sera disponible.

**Utiliser `std::expected` pour les résultats faillibles.** Le pattern matching avec `expected` sera l'idiome de choix pour la gestion d'erreurs — préparer ses API avec `expected` (section 12.8) maintenant facilitera la migration.

**Expérimenter sur Compiler Explorer.** Le fork Clang avec `-fpattern-matching` permet de tester la syntaxe proposée et de se familiariser avec les idiomes.

**Suivre l'évolution de la proposition.** P2688 continue d'être affinée. Les retours d'expérience de la communauté influencent le design final.
