# 12.14.4.1 Modèle Sender/Receiver : Concepts fondamentaux

## Sous le capot du pipeline

La section 12.14.4 a présenté `std::execution` du point de vue de l'utilisateur : des senders composés avec `|`, exécutés via `sync_wait`. Cette section descend d'un niveau pour expliquer les concepts formels qui font fonctionner le modèle. Comprendre ces mécanismes n'est pas indispensable pour *utiliser* des pipelines simples, mais c'est nécessaire pour *implémenter* des algorithmes sender personnalisés, diagnostiquer des erreurs de compilation, et tirer pleinement parti du framework.

> 💡 *En pratique, les utilisateurs de frameworks construits sur `std::execution` manipulent principalement des senders et des schedulers. Les receivers et les operation states sont des concepts « côté implémenteur ». Cette section les couvre néanmoins car ils forment l'architecture logique du modèle.*

## Les quatre concepts du modèle

Le modèle Sender/Receiver repose sur quatre concepts interconnectés définis dans `<execution>` :

```
                    connect()                start()
  Sender  +  Receiver  ──────►  Operation State  ──────►  Exécution
                                                              │
                                                    ┌─────────┼─────────┐
                                                    ▼         ▼         ▼
                                               set_value  set_error  set_stopped
                                               (succès)   (erreur)   (annulation)
```

### 1. Sender — la description

Un sender est un objet qui *décrit* une opération asynchrone. Il porte deux types d'informations :

- **Les signatures de complétion** — Quels types de valeurs le sender peut produire en cas de succès, quels types d'erreurs en cas d'échec, et s'il peut être annulé.
- **Les attributs** — Des métadonnées sur le sender (par exemple, le scheduler sur lequel il s'exécutera).

Un sender ne fait rien par lui-même. Il est inerte — c'est un plan, pas une action. Cette paresse est fondamentale : elle permet de composer des pipelines arbitrairement complexes avant de les exécuter, et elle permet au compilateur d'optimiser l'ensemble du pipeline comme une seule unité.

Le concept formel est défini dans `<execution>` :

```cpp
namespace ex = std::execution;

// Un type S est un sender s'il :
// - fournit des informations de complétion (completion_signatures)
// - est déplaçable
// - possède un type interne sender_concept = ex::sender_t
```

L'exemple le plus simple de sender est `just`, qui complète immédiatement avec des valeurs données :

```cpp
auto s = ex::just(42, "hello"s);
// s est un sender qui complète avec set_value(42, "hello"s)
// Rien n'est exécuté — s est une pure description
```

### 2. Receiver — le callback structuré

Un receiver est un objet qui *reçoit* le résultat d'un sender. Il implémente les trois canaux de complétion sous forme de fonctions appelables :

```cpp
// Conceptuellement, un receiver fournit :
struct MyReceiver {
    // Canal de succès — appelé quand le sender réussit
    void set_value(auto&&... values);

    // Canal d'erreur — appelé quand le sender échoue
    void set_error(auto&& error);

    // Canal d'annulation — appelé quand le sender est annulé
    void set_stopped();
};
```

Un receiver est un « callback multi-canal » — contrairement à un callback classique (une seule fonction), il gère trois cas de terminaison. Cette exhaustivité est ce qui rend le modèle robuste : chaque opération asynchrone *doit* terminer par exactement l'un des trois canaux. Pas de résultat silencieusement perdu, pas d'erreur ignorée, pas d'annulation non traitée.

En pratique, l'utilisateur final ne crée presque jamais de receiver manuellement. Les algorithmes comme `then`, `upon_error`, `when_all` et `sync_wait` construisent des receivers internes qui connectent les étapes entre elles. Les receivers sont la colle invisible du pipeline.

### 3. Operation State — l'opération connectée

Quand un sender et un receiver sont **connectés** via `ex::connect(sender, receiver)`, le résultat est un **operation state** — un objet qui contient tout le nécessaire pour exécuter l'opération :

```cpp
auto sender = ex::just(42);  
auto receiver = /* ... */;  

// connect() produit un operation state
auto op = ex::connect(std::move(sender), std::move(receiver));
// op contient le sender, le receiver, et tout état intermédiaire
// Rien n'est encore exécuté
```

L'operation state est l'objet qui sera effectivement « démarré ». Il a une durée de vie bien définie : il doit rester valide jusqu'à ce que le receiver soit appelé (c'est-à-dire jusqu'à ce que l'opération complète). C'est le mécanisme qui garantit que toutes les ressources nécessaires à l'opération survivent à son exécution.

### 4. start() — le déclenchement

L'appel à `ex::start(op)` démarre effectivement l'opération. C'est le point où le travail commence réellement — les threads sont sollicités, les I/O sont lancées, les calculs démarrent :

```cpp
ex::start(op);
// L'opération s'exécute maintenant
// À un moment, le receiver sera appelé avec set_value, set_error, ou set_stopped
```

`start()` est une opération non-bloquante. Elle lance le travail et retourne immédiatement. Le résultat sera livré au receiver de manière asynchrone.

## Le protocole complet

L'enchaînement connect → start → complétion forme le **protocole sender/receiver**. Voici le cycle complet :

```
1. Description    :  auto sndr = ex::just(42) | ex::then(f);
                     // Pipeline construit, rien n'est exécuté

2. Connexion      :  auto op = ex::connect(sndr, receiver);
                     // Le sender et le receiver sont liés
                     // L'operation state est créé

3. Démarrage      :  ex::start(op);
                     // L'exécution commence

4. Complétion     :  receiver.set_value(result);
                     // OU receiver.set_error(err);
                     // OU receiver.set_stopped();
                     // Exactement un des trois, une seule fois

5. Destruction    :  // L'operation state peut être détruit
                     // Toutes les ressources sont libérées
```

Ce protocole est strict et les garanties sont fortes :

- Un receiver est appelé **exactement une fois** avec **exactement un** des trois canaux.
- Le receiver n'est appelé qu'**après** `start()`.
- L'operation state doit rester en vie jusqu'à l'appel du receiver.
- Après l'appel du receiver, l'operation state peut être détruit.

Ces garanties éliminent les classes de bugs classiques de la programmation asynchrone : double complétion, complétion avant démarrage, accès à un état détruit, résultat perdu.

## Les trois canaux de complétion en détail

### set_value — le succès

Le canal de succès transmet les valeurs de résultat. Un sender peut compléter avec zéro, une ou plusieurs valeurs :

```cpp
ex::just()                      // set_value()        — zéro valeur  
ex::just(42)                    // set_value(42)      — une valeur  
ex::just(42, "hello"s, 3.14)   // set_value(42, "hello", 3.14) — trois valeurs  
```

Quand un sender complète avec `set_value(a, b, c)`, l'algorithme `then(f)` appelle `f(a, b, c)`. Les types et le nombre de valeurs sont propagés à travers le pipeline via le système de types — vérifiés à la compilation.

### set_error — l'erreur

Le canal d'erreur transmet une valeur d'erreur. Le type de l'erreur est libre — `std::exception_ptr`, `std::error_code`, un enum, une string, un type d'erreur personnalisé :

```cpp
// Un sender qui échoue toujours avec une exception_ptr
auto failing = ex::just_error(std::make_exception_ptr(std::runtime_error("oops")));

// Un sender qui échoue avec un error_code
auto io_fail = ex::just_error(std::make_error_code(std::errc::no_such_file_or_directory));
```

Le canal d'erreur est distinct du canal de succès — pas besoin de `std::expected` ou de `std::variant` pour distinguer les deux cas. Le modèle sépare structurellement succès et erreur.

### set_stopped — l'annulation

Le canal d'annulation est appelé quand l'opération est annulée proprement — par exemple via un *stop token*. L'annulation est un concept de premier plan dans `std::execution`, pas un ajout tardif :

```cpp
namespace ex = std::execution;

// Stop source / stop token — mécanisme d'annulation coopérative
std::stop_source source;

// Passer le stop token dans le pipeline via l'environnement
auto pipeline = ex::schedule(sch)
    | ex::then([] { return long_computation(); });

// Depuis un autre thread :
source.request_stop();
// Le pipeline peut détecter la demande d'arrêt et compléter avec set_stopped()
```

L'annulation est **coopérative** — le code dans le pipeline vérifie périodiquement si un arrêt a été demandé. Le modèle ne force pas une terminaison brutale (pas de `pthread_cancel` ou équivalent), mais fournit un mécanisme propre pour propager les demandes d'annulation à travers un graphe de senders.

## Completion signatures : la carte de visite d'un sender

Chaque sender déclare ses **signatures de complétion** — les différentes manières dont il peut se terminer. Ces signatures sont des informations de type, vérifiées à la compilation :

```cpp
namespace ex = std::execution;

// ex::just(42) a les signatures :
//   set_value(int)      — peut réussir avec un int
//   (pas de set_error)  — ne peut pas échouer
//   (pas de set_stopped) — ne peut pas être annulé

// ex::just(42) | ex::then([](int x) -> std::string { ... }) a les signatures :
//   set_value(std::string)  — le then transforme int → string
```

Les completion signatures permettent au compilateur de vérifier à la compilation que les types sont compatibles entre les étapes du pipeline. Si un sender produit un `int` et que le `then` suivant attend un `std::string`, l'erreur est détectée à la compilation — pas à l'exécution.

C'est un avantage majeur sur les systèmes de callbacks et de futures classiques, où les erreurs de type ne sont détectées qu'à l'exécution (voire jamais, produisant des bugs silencieux).

## Comment then fonctionne en interne

Pour illustrer comment les concepts s'articulent, voyons conceptuellement ce que `then(f)` fait en interne. Quand on écrit :

```cpp
auto pipeline = ex::just(42) | ex::then([](int x) { return x * 2; });
```

L'algorithme `then` crée un nouveau sender qui :

1. **Stocke** le sender d'entrée (`just(42)`) et la fonction `f`.
2. Quand **connecté** à un receiver `R`, crée un receiver intermédiaire qui :
   - Sur `set_value(42)` → appelle `f(42)` → appelle `R.set_value(84)`
   - Sur `set_error(e)` → propage directement `R.set_error(e)`
   - Sur `set_stopped()` → propage directement `R.set_stopped()`
3. **Connecte** le sender d'entrée au receiver intermédiaire.

```
Pipeline : just(42) | then(f)

Connexion avec un receiver final R :

  just(42) ──connect──► ThenReceiver{f, R}
                              │
                    set_value(42)
                              │
                         appelle f(42)
                              │
                         R.set_value(84)
```

Le receiver intermédiaire est la colle qui adapte la complétion du sender d'entrée pour y appliquer la transformation `f` avant de la propager au receiver suivant. C'est le même pattern pour tous les algorithmes : chaque algorithme crée un sender qui, lors de la connexion, interpose un receiver adapté.

## Environnements et propriétés

Les receivers portent un **environnement** (`env`) qui contient des propriétés du contexte d'exécution. Les senders et les algorithmes peuvent interroger cet environnement pour adapter leur comportement :

```cpp
// Exemples de propriétés d'environnement :
ex::get_scheduler(env)     // Le scheduler associé au contexte  
ex::get_stop_token(env)    // Le stop token pour l'annulation  
ex::get_allocator(env)     // L'allocateur à utiliser  
```

L'environnement est propagé de sender en sender à travers le pipeline, permettant à chaque étape de connaître son contexte d'exécution sans le passer explicitement en argument. C'est l'équivalent d'un contexte implicite — similaire aux « execution contexts » d'autres frameworks, mais formalisé dans le système de types.

## Senders vs fonctions : analogie

Pour construire une intuition, voici une analogie entre les fonctions classiques et les senders :

```
Fonctions classiques              Senders
─────────────────────────────     ──────────────────────────────
int f(int x) { return x*2; }     auto s = just(21) | then(f);

Appel : f(21) → 42               connect(s, recv) + start(op)
                                  → recv.set_value(42)

Retour : return 42                set_value(42)  
Exception : throw err             set_error(err)  
(pas d'équivalent)                set_stopped()

Composition : g(f(x))            just(x) | then(f) | then(g)
  → exécution immédiate           → exécution différée (lazy)

Contexte : pile d'appels          Scheduler + environnement
```

La différence clé : les fonctions sont **eagerly evaluated** (exécutées immédiatement à l'appel), tandis que les senders sont **lazily evaluated** (exécutés seulement quand on les démarre). Cette paresse est ce qui permet la composition libre, l'optimisation globale du pipeline, et le contrôle du contexte d'exécution.

## Qui implémente quoi ?

Le modèle sépare clairement les rôles :

**L'utilisateur final** (développeur d'application) :
- Compose des senders avec `|`, `then`, `when_all`, etc.
- Choisit un scheduler
- Appelle `sync_wait` pour démarrer
- Ne touche jamais aux receivers ni aux operation states

**L'auteur de framework** (bibliothèque de concurrence, framework I/O) :
- Implémente des schedulers (thread pool, event loop)
- Implémente des senders personnalisés (I/O asynchrone, timers)
- Manipule les receivers et les operation states
- Doit respecter le protocole connect/start/complétion

**L'auteur d'algorithme** (extension de `std::execution`) :
- Implémente de nouveaux algorithmes (comme `then`, `when_all`, etc.)
- Crée des senders qui interposent des receivers intermédiaires
- Doit respecter les completion signatures et les garanties du protocole

Cette séparation des préoccupations est ce qui rend le modèle extensible. Un framework I/O peut fournir des senders pour les opérations réseau sans connaître les algorithmes de l'utilisateur. Un utilisateur peut composer ces senders sans connaître les détails d'implémentation du framework.

## Bonnes pratiques

**Penser en termes de description, pas d'exécution.** Un sender est un plan. Le construire ne coûte rien. Construire le pipeline d'abord, l'exécuter ensuite — c'est le workflow naturel de `std::execution`.

**Ne pas stocker d'operation states à longue durée de vie.** Les operation states sont des objets temporaires liés à une exécution spécifique. Ils sont créés par `connect`, vivent pendant l'exécution, et sont détruits après complétion. Les stocker dans des structures de données globales est un anti-pattern.

**Utiliser `sync_wait` comme point de transition unique.** Idéalement, `sync_wait` est appelé une seule fois dans le programme — au point de transition entre le monde synchrone (`main`) et le monde asynchrone (le pipeline de senders). Le reste du code est entièrement composé de senders.

**Préférer les algorithmes standard aux receivers manuels.** `then`, `upon_error`, `when_all`, `let_value` couvrent la grande majorité des cas. Écrire un receiver personnalisé n'est nécessaire que pour des abstractions de bas niveau (frameworks, bibliothèques I/O).

**Tester avec stdexec.** L'implémentation de référence de NVIDIA est le meilleur terrain de jeu pour expérimenter avec les concepts Sender/Receiver. Elle compile avec GCC et Clang, et est disponible sur Compiler Explorer.
