# 12.13.2 État du support (GCC 15, Clang 20, MSVC)

## Un paysage en mouvement rapide

Les modules C++20 ont été standardisés fin 2020, mais leur implémentation dans les compilateurs et les build systems a suivi un rythme bien différent de celui des autres fonctionnalités du standard. La raison : les modules ne sont pas une simple fonctionnalité syntaxique — ils bouleversent le modèle de compilation lui-même, et impactent l'ensemble de la chaîne d'outils (compilateur, build system, IDE, analyse statique).

En mars 2026, le support a atteint un point d'inflexion. Les trois grands compilateurs proposent un support fonctionnel, CMake traite les modules comme une fonctionnalité de premier plan, et des projets réels commencent à les adopter. Le support n'est cependant pas uniforme — chaque compilateur a ses forces, ses limitations, et ses particularités. Cette section fait le point, compilateur par compilateur et outil par outil.

## MSVC : le pionnier

Microsoft a été le premier éditeur à investir massivement dans les modules, et MSVC reste le compilateur avec le support le plus mature.

### Ce qui fonctionne

MSVC (toolset 14.34+, fourni avec Visual Studio 2022 17.4 et supérieur) dispose d'une implémentation complète des modules nommés C++20. Les module interface units, les module implementation units, les partitions de modules et les header units fonctionnent. Le support de `import std;` (C++23) est disponible depuis Visual Studio 2022 17.5, permettant d'importer l'intégralité de la bibliothèque standard en une seule instruction.

L'intégration avec MSBuild est native : Visual Studio gère automatiquement l'ordonnancement des compilations, la génération des BMI (fichiers `.ifc` chez Microsoft), et les dépendances inter-modules. Le format P1689 pour le *dependency scanning* est supporté, ce qui permet aussi l'intégration avec CMake.

L'extension de fichier conventionnelle pour les interfaces de module est `.ixx` dans l'écosystème MSVC.

### Limitations connues

Malgré son avance, MSVC n'est pas exempt de problèmes. Des bugs de compilation liés aux modules sont régulièrement rapportés, notamment dans les cas impliquant des templates complexes, des friend declarations à travers les frontières de module, ou des configurations de modules partitionnés non triviales. IntelliSense (l'autocomplétion dans Visual Studio), qui utilise un frontend différent du compilateur (EDG), peut présenter des incohérences avec le comportement réel de MSVC sur les modules.

La portabilité des BMI est un point important : les fichiers `.ifc` ne sont pas portables entre versions du compilateur ni entre configurations de compilation (Debug/Release, options d'optimisation différentes). C'est une limitation fondamentale des BMI dans tous les compilateurs, pas seulement MSVC.

## GCC 15 : un support en progression rapide

GCC a longtemps été en retrait sur les modules, avec un support expérimental jugé instable. La situation s'est considérablement améliorée avec GCC 14 et surtout GCC 15, grâce notamment au travail de Nathaniel Shead sur l'implémentation des modules dans le frontend C++.

### Ce qui fonctionne

GCC 15 supporte les modules nommés, les module implementation units, et les partitions de modules. La compilation avec `-fmodules` (ou via le standard C++20/23 avec les flags appropriés) permet de créer et d'importer des modules. Les BMI sont stockés sous forme de fichiers `.gcm` dans un répertoire `gcm.cache` local.

Le support de `import std;` (C++23) est fonctionnel dans GCC 15 avec libstdc++. La compilation nécessite de builder d'abord le module `std` depuis le fichier source fourni par la bibliothèque standard :

```bash
# Build du module std (une seule fois)
g++-15 -std=c++23 -fmodules -fsearch-include-path bits/std.cc

# Compilation utilisant import std;
g++-15 -std=c++23 -fmodules main.cpp -o app
```

Le répertoire `gcm.cache` sert de cache pour les BMI — une fois le module `std` compilé, les compilations suivantes le réutilisent sans le reconstruire.

Le support du protocole P1689 (dependency scanning) a été intégré dans GCC 14, ce qui a permis à CMake 3.28+ de supporter GCC pour la compilation de modules.

### Limitations connues

GCC 15 représente un progrès considérable, mais des limitations subsistent. Des *internal compiler errors* (ICE) sont encore rapportés dans des cas complexes — templates avec contraintes (concepts) à travers les frontières de module, certains patterns de friend declarations, ou des configurations de partitions élaborées. Le taux de ces crashs a diminué entre GCC 14 et GCC 15, mais la prudence reste de mise pour les projets complexes.

Le mécanisme de *module mapper* de GCC (fichier `module.mapper`) pour contrôler la correspondance entre noms de modules et fichiers BMI est fonctionnel mais moins ergonomique que les approches automatisées de CMake ou MSBuild.

L'extension conventionnelle pour les interfaces de module dans l'écosystème GCC est `.cppm`, mais GCC n'impose pas d'extension spécifique — il se fie au contenu du fichier.

## Clang 20 : l'approche méthodique

Clang a adopté une approche prudente et méthodique pour les modules. Clang possède en fait deux systèmes de modules : les *Clang modules* (un système propriétaire antérieur à C++20, utilisé en interne par le projet LLVM) et les *C++20 named modules* (le standard). Les deux coexistent mais ne sont pas interchangeables.

### Ce qui fonctionne

Clang 20 supporte les modules nommés C++20, les module implementation units, les partitions, et les header units. Le protocole P1689 est supporté depuis Clang 16, permettant l'intégration avec CMake.

Le support de `import std;` est fonctionnel avec libc++ (la bibliothèque standard du projet LLVM). Le fichier source du module std est fourni sous forme de `.cppm` et doit être précompilé en un BMI (fichier `.pcm`) :

```bash
# Précompilation du module std
clang++ -std=c++23 -stdlib=libc++ --precompile \
    -o std.pcm /usr/share/libc++/v1/std.cppm

# Compilation utilisant import std;
clang++ -std=c++23 -stdlib=libc++ -fmodule-file=std=std.pcm \
    main.cpp -o app
```

Clang 20 a également intégré un *module level lookup* amélioré qui corrige des problèmes de résolution de noms qui affectaient les versions précédentes.

Un effort GSoC (Google Summer of Code) 2025 a porté sur le support natif de la compilation de modules directement depuis le driver Clang, sans nécessiter un build system externe pour les cas simples. Ce travail progresse et pourrait simplifier significativement l'expérience pour les petits projets et les tests.

### Limitations connues

Comme pour GCC, des crashs du compilateur (ICE) sont rapportés dans des cas complexes, bien que leur fréquence ait nettement diminué dans les versions récentes. L'auteur principal du support modules de Clang rapporte que le nombre de bugs liés aux modules est désormais inférieur à celui des bugs liés aux coroutines — un signe de stabilisation.

L'utilisation de `import std;` nécessite libc++ — elle ne fonctionne pas avec libstdc++ sous Clang. Sur les distributions Linux qui utilisent libstdc++ par défaut (la majorité des distributions basées sur Debian/Ubuntu), il faut installer libc++ séparément et spécifier `-stdlib=libc++`.

L'import d'un *Clang module* dans un *C++20 named module* (ou vice versa) n'est pas encore supporté, ce qui peut compliquer les projets qui utilisaient les Clang modules historiques.

## Tableau comparatif des compilateurs

```
┌───────────────────────────────┬────────────┬───────────┬────────────┐
│ Fonctionnalité                │ MSVC 17.x  │ GCC 15    │ Clang 20   │
├───────────────────────────────┼────────────┼───────────┼────────────┤
│ Modules nommés                │ ✅ Complet │ ✅ Complet│ ✅ Complet │
│ Module implementation units   │ ✅         │ ✅        │ ✅         │
│ Module partitions             │ ✅         │ ✅        │ ✅         │
│ Header units                  │ ✅         │ ⚠️ Partiel│ ⚠️ Partiel │
│ import std; (C++23)           │ ✅         │ ✅        │ ✅ (libc++)│
│ P1689 (dependency scanning)   │ ✅         │ ✅        │ ✅         │
│ Stabilité (cas complexes)     │ ⚠️ Bugs    │ ⚠️ ICE    │ ⚠️ ICE     │
│ Extension interface           │ .ixx       │ .cppm     │ .cppm      │
│ Extension BMI                 │ .ifc       │ .gcm      │ .pcm       │
└───────────────────────────────┴────────────┴───────────┴────────────┘
✅ = Supporté et fonctionnel
⚠️ = Supporté avec des limitations ou bugs connus
```

Le constat global : les trois compilateurs supportent les fondamentaux des modules. Les différences se situent dans la robustesse face aux cas complexes et dans l'écosystème d'outils associé.

## Build systems

### CMake : le support de référence

CMake est le build system dominant pour les projets C++ et son support des modules est désormais un élément central. L'historique de ce support reflète la complexité du problème :

- **CMake 3.26** (mars 2023) — Support expérimental, nécessitant des variables `CMAKE_EXPERIMENTAL_*` magiques.
- **CMake 3.28** (décembre 2023) — Le support sort de l'expérimental. Les modules fonctionnent avec Ninja et les générateurs Visual Studio, pour les trois compilateurs supportant P1689.
- **CMake 4.0** (2025) — Stabilisation majeure. Le *dependency scanning* est activé par défaut via la policy CMP0155, le support `import std;` progresse, et la documentation couvre les cas d'usage courants.

La configuration CMake pour un projet avec modules reste relativement simple :

```cmake
cmake_minimum_required(VERSION 3.28)  
project(my_project CXX)  

set(CMAKE_CXX_STANDARD 23)

add_executable(app)  
target_sources(app  
    PUBLIC FILE_SET CXX_MODULES FILES
        math_utils.cppm
    PRIVATE
        main.cpp
)
```

La directive `FILE_SET CXX_MODULES` indique à CMake quels fichiers sont des interfaces de module. CMake se charge du scanning des dépendances, de la génération des BMI dans le bon ordre, et de l'intégration avec le générateur de build sous-jacent (Ninja recommandé).

**Limitation importante :** le support nécessite le générateur Ninja ou Visual Studio. Le générateur Makefiles classique ne supporte pas le *dynamic dependency* (dyndep) nécessaire aux modules. Ninja est le choix recommandé sur Linux.

> ⚠️ **Note Ubuntu :** les versions d'Ubuntu antérieures à 26.04 fournissent des fichiers `libstdc++.modules.json` défectueux qui peuvent causer des problèmes avec `import std;` via CMake. Vérifier la documentation CMake pour les workarounds.

### Autres build systems

**Ninja** — Le backend de build recommandé pour les modules sur Linux. Son support du *dynamic dependency* (dyndep) est la fonctionnalité clé qui permet à CMake de gérer l'ordonnancement des compilations de modules.

**MSBuild** — Intégration native avec MSVC dans Visual Studio. C'est le chemin le plus simple pour les projets Windows-only.

**Build2** — Supporte les modules avec GCC depuis 2021, avec une approche intégrée de la gestion des dépendances inter-modules. C'est l'un des build systems les plus avancés sur les modules, bien que moins répandu que CMake.

**xmake** — Support des modules avec les trois compilateurs, ayant implémenté son propre code de découverte des dépendances. C'est une alternative intéressante pour les projets qui l'utilisent déjà.

**Meson** — Le support des modules C++20 est en cours de développement mais n'est pas encore disponible dans les versions stables en mars 2026. C'est une limitation notable car Meson est utilisé par de nombreux projets Linux majeurs (GNOME, systemd, Mesa).

**Bazel** — Support expérimental via des règles communautaires. Le support natif progresse mais n'est pas encore mature.

## IDE et outillage

### clangd (language server)

Le support des modules C++20 dans clangd (le language server de l'écosystème LLVM, utilisé par VS Code, Neovim et d'autres éditeurs) est fonctionnel mais encore en phase de maturation. clangd peut compiler les BMI nécessaires à l'autocomplétion et à la navigation dans le code module, à condition de disposer d'une *compilation database* correcte.

Le principal défi rapporté est le temps de « warm-up » lors de l'ouverture d'un projet — clangd doit scanner l'ensemble des fichiers pour construire le graphe de dépendances inter-modules. Des optimisations sont en cours pour réduire ce temps.

### Visual Studio

Visual Studio offre le meilleur support IDE pour les modules, ce qui est naturel compte tenu de l'avance de MSVC. L'autocomplétion, la navigation (Go to Definition), et le refactoring fonctionnent sur le code modulaire. IntelliSense, qui utilise le frontend EDG plutôt que MSVC, peut cependant présenter des incohérences dans les cas limites.

### CLion

CLion (JetBrains) supporte les modules C++20 depuis la version 2022.3. Le support s'appuie sur le language server interne de JetBrains et sur la compilation database générée par CMake.

## Bibliothèques standard et tierces

### import std; — le module standard

La possibilité d'écrire `import std;` pour importer l'intégralité de la bibliothèque standard est une fonctionnalité C++23 qui simplifie considérablement l'adoption des modules. Son support dépend de la bibliothèque standard utilisée :

- **libstdc++** (GCC) — Le fichier source du module std est disponible depuis GCC 15. La bibliothèque fournit un fichier `bits/std.cc` qui doit être compilé en BMI.
- **libc++** (Clang/LLVM) — Le fichier `std.cppm` est fourni depuis libc++ 17. Il nécessite la précompilation en `.pcm`.
- **MSVC STL** — Le support est intégré depuis VS 2022 17.5. La compilation du module std est gérée automatiquement par MSBuild.

### Bibliothèques tierces

C'est le maillon le plus faible de l'écosystème en mars 2026. L'immense majorité des bibliothèques C++ tierces (Boost, Abseil, Eigen, {fmt}, nlohmann/json, etc.) ne fournissent pas d'interfaces de module. L'adoption par les bibliothèques majeures est encore embryonnaire — Boost n'a pas encore de support officiel des modules, bien que des expérimentations existent.

En attendant, la stratégie de migration recommandée repose sur les **header units** et le **global module fragment** :

```cpp
// my_module.cppm
module;                          // Global module fragment
#include <boost/asio.hpp>        // Headers classiques ici
#include <nlohmann/json.hpp>

export module my_module;

// Le code du module peut utiliser les types des headers inclus
export class MyService {
    // Utilise boost::asio et nlohmann::json en interne
};
```

Le *global module fragment* (la zone entre `module;` et `export module ...;`) permet d'inclure des headers classiques dans un module, offrant une coexistence fluide entre le modèle ancien et le nouveau.

## Bonnes pratiques pour l'adoption

**Commencer avec un compilateur récent.** GCC 15, Clang 20, ou MSVC 17.6+ sont les versions minimales recommandées pour un usage sérieux des modules. Les versions antérieures ont trop de bugs bloquants.

**Utiliser CMake 3.28+ avec Ninja.** C'est la combinaison la plus testée et la plus portable. Éviter les générateurs Makefiles qui ne supportent pas le dyndep.

**Tester `import std;` en premier.** C'est le moyen le plus simple d'évaluer le support des modules dans votre environnement. Si `import std;` fonctionne, les modules nommés du projet fonctionneront aussi.

**Utiliser le global module fragment pour les dépendances tierces.** Ne pas attendre que les bibliothèques tierces fournissent des interfaces de module — les inclure via le global module fragment permet de les utiliser immédiatement dans un module.

**Tester la compilation croisée entre compilateurs.** Les BMI ne sont pas portables entre compilateurs. Si le projet doit compiler avec GCC et Clang, les BMI doivent être régénérés pour chaque compilateur.
