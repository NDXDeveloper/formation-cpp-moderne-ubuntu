# 12.14.4.3 Intégration avec les schedulers et thread pools

## Le scheduler : le chef d'orchestre de l'exécution

Les sections précédentes ont montré comment décrire du travail (senders) et comment le composer (pipelines). Cette section traite du dernier pilier : **où** et **quand** ce travail s'exécute. Le scheduler est l'abstraction qui répond à cette question. C'est le composant qui détermine le contexte d'exécution — un thread pool, une event loop, un GPU, un contexte temps réel — et qui permet de changer ce contexte sans modifier le pipeline de travail.

## Le concept scheduler

Un scheduler est un objet qui sait créer un sender via `schedule()`. Ce sender, quand démarré, transfère l'exécution vers le contexte d'exécution du scheduler :

```cpp
namespace ex = std::execution;

// Obtenir un scheduler
auto sch = /* ... */;

// schedule(sch) crée un sender qui :
// - quand démarré, transfère l'exécution vers le contexte de sch
// - complète avec set_value() (sans valeur) une fois le transfert effectué
auto start_here = ex::schedule(sch);
```

Le sender retourné par `schedule` est le point d'entrée dans un contexte d'exécution. Tout ce qui suit dans le pipeline (via `| then(...)`) s'exécute dans ce contexte :

```cpp
auto pipeline = ex::schedule(sch)          // Transfère vers le contexte de sch
    | ex::then([] { return compute(); })   // S'exécute dans le contexte de sch
    | ex::then([](auto r) { save(r); });   // S'exécute aussi dans le contexte de sch
```

Le concept formel `scheduler` est minimal : un type est un scheduler s'il supporte `ex::schedule(sch)` et que le résultat est un sender. Cette simplicité permet à n'importe quel contexte d'exécution de s'intégrer dans le modèle.

## Les schedulers disponibles

### run_loop : le scheduler minimal du standard

`std::execution::run_loop` est le seul scheduler fourni directement par P2300 dans le standard C++26. C'est un scheduler mono-thread basé sur une file d'attente — les senders programmés dessus sont exécutés par le thread qui appelle `run_loop::run()` :

```cpp
namespace ex = std::execution;

ex::run_loop loop;

// Programmer du travail sur la loop
auto work = ex::schedule(loop.get_scheduler())
    | ex::then([] { 
        std::print("Exécuté sur le thread de la run_loop\n");
        return 42;
    });

// Démarrer le travail
auto op = ex::connect(std::move(work), /* receiver */);  
ex::start(op);  

// Exécuter la loop (bloque jusqu'à ce qu'elle soit terminée)
loop.run();
// Le travail a été exécuté sur CE thread
loop.finish();
```

`run_loop` est utile pour les tests unitaires, le bootstrapping, et les cas où on veut exécuter des senders sur un thread spécifique (le thread principal, par exemple). Mais pour la concurrence réelle, un thread pool est nécessaire.

### system_context : le thread pool de la plateforme

La proposition P2079 (`system_context`) introduit un scheduler qui utilise le thread pool natif de la plateforme — le mécanisme le plus naturel pour distribuer du travail sur plusieurs cœurs :

```cpp
namespace ex = std::execution;

// Obtenir le scheduler du contexte système
auto sch = ex::system_context().get_scheduler();

// Le travail s'exécute sur un thread du pool système
auto pipeline = ex::schedule(sch)
    | ex::then([] { return heavy_computation(); });

auto [result] = *ex::sync_wait(std::move(pipeline));
```

`system_context` est le scheduler que la plupart des utilisateurs emploieront au quotidien. Il est analogue au `std::launch::async` de `std::async`, mais avec un contexte explicite et un contrôle fin sur l'exécution.

> ⚠️ *`system_context` fait partie des propositions complémentaires à P2300 (P2079). Son inclusion dans le standard C++26 finalisé est en cours de résolution. En attendant, les implémentations de référence (stdexec) fournissent des schedulers équivalents.*

### static_thread_pool : contrôle du nombre de threads

Pour les cas où le nombre de threads doit être contrôlé explicitement — par exemple, limiter la parallélisation à un nombre fixe de cœurs — un thread pool à taille fixe est nécessaire :

```cpp
namespace ex = std::execution;

// Thread pool avec exactement 4 threads (stdexec)
exec::static_thread_pool pool(4);  
auto sch = pool.get_scheduler();  

// Le travail est distribué sur les 4 threads du pool
auto pipeline = ex::when_all(
    ex::schedule(sch) | ex::then(task_a),
    ex::schedule(sch) | ex::then(task_b),
    ex::schedule(sch) | ex::then(task_c),
    ex::schedule(sch) | ex::then(task_d)
);

ex::sync_wait(std::move(pipeline));
// Les 4 tâches s'exécutent en parallèle sur les 4 threads
```

`static_thread_pool` n'est pas dans le standard C++26 tel quel, mais il est fourni par stdexec et par d'autres implémentations. Un thread pool standard est un objectif pour une future révision.

## Changer de contexte d'exécution

L'un des aspects les plus puissants des schedulers est la possibilité de **changer de contexte d'exécution** au milieu d'un pipeline. Un calcul peut démarrer sur un thread pool, transférer vers un thread d'I/O pour une opération réseau, puis revenir sur le thread pool pour le post-traitement :

```cpp
namespace ex = std::execution;

auto compute_sch = ex::system_context().get_scheduler();   // Thread pool CPU  
auto io_sch = io_context.get_scheduler();                   // Event loop I/O  

auto pipeline = ex::schedule(compute_sch)
    | ex::then(prepare_request)              // Sur le thread pool CPU

    | ex::continues_on(io_sch)               // Transfère vers le contexte I/O
    | ex::then(send_http_request)            // Sur le thread I/O
    | ex::then(read_response)                // Sur le thread I/O

    | ex::continues_on(compute_sch)          // Retour vers le thread pool CPU
    | ex::then(parse_response)               // Sur le thread pool CPU
    | ex::then(update_cache);                // Sur le thread pool CPU

ex::sync_wait(std::move(pipeline));
```

`continues_on(sch)` (anciennement `transfer`) est l'algorithme qui transfère l'exécution d'un contexte à un autre. Il crée un point de synchronisation dans le pipeline : tout ce qui précède s'exécute dans l'ancien contexte, tout ce qui suit dans le nouveau.

Ce pattern est fondamental pour les applications réseau et les serveurs : le traitement CPU-intensif se fait sur le thread pool de calcul, les opérations I/O sur l'event loop dédiée, et les deux sont composés dans un pipeline unique et lisible.

### starts_on vs continues_on

Deux algorithmes contrôlent le contexte d'exécution :

```cpp
// starts_on : le sender COMMENCE son exécution sur le scheduler donné
auto s1 = ex::starts_on(sch, some_sender);
// Quand s1 est démarré, some_sender démarre sur un thread de sch

// continues_on : le sender CONTINUE sur le scheduler donné après complétion
auto s2 = some_sender | ex::continues_on(sch);
// Quand some_sender complète, la complétion est livrée sur un thread de sch
```

`starts_on` contrôle le début de l'exécution d'un sender. `continues_on` contrôle la suite après la complétion. Les deux permettent un contrôle fin du placement de chaque étape sur le bon contexte d'exécution.

## Schedulers personnalisés

Le concept `scheduler` est ouvert — n'importe quel type qui fournit `schedule()` retournant un sender peut être un scheduler. Cela permet d'intégrer des contextes d'exécution arbitraires dans le modèle `std::execution`.

### Exemple conceptuel : scheduler inline

Le scheduler le plus simple possible — exécute le travail immédiatement sur le thread appelant :

```cpp
namespace ex = std::execution;

struct inline_scheduler {
    struct sender {
        using sender_concept = ex::sender_t;
        using completion_signatures = ex::completion_signatures<
            ex::set_value_t()
        >;

        template <typename Receiver>
        struct operation {
            Receiver recv;

            friend void tag_invoke(ex::start_t, operation& self) noexcept {
                ex::set_value(std::move(self.recv));
            }
        };

        template <typename Receiver>
        auto connect(Receiver recv) const {
            return operation<Receiver>{std::move(recv)};
        }
    };

    sender schedule() const noexcept {
        return {};
    }
};
```

Ce scheduler ne transfère vers aucun thread — il complète immédiatement sur le thread appelant. C'est utile pour les tests et comme fallback quand aucune parallélisation n'est nécessaire. En production, le même pipeline peut utiliser un `system_context` sans aucune modification du code de travail :

```cpp
// En test : exécution inline (pas de thread, déterministe)
auto pipeline = ex::schedule(inline_scheduler{}) | ex::then(compute);

// En production : exécution sur le thread pool (parallélisme réel)
auto pipeline = ex::schedule(system_sch) | ex::then(compute);

// Le code de compute ne change pas — seul le scheduler change
```

### Cas d'usage des schedulers personnalisés

**Thread affinity** — Un scheduler qui exécute toujours sur un thread spécifique (le thread principal pour les opérations UI, un thread dédié pour les accès base de données).

**Priorités** — Un scheduler avec des files de priorité, qui exécute le travail urgent avant le travail en arrière-plan.

**Instrumenté** — Un scheduler qui mesure le temps d'exécution, trace les tâches, ou log les événements — utile pour l'observabilité (section 40).

**GPU** — Un scheduler qui soumet le travail à un GPU via CUDA, OpenCL ou Vulkan. L'implémentation stdexec de NVIDIA inclut des schedulers CUDA.

**Event loop I/O** — Un scheduler basé sur `epoll` (Linux), `io_uring`, ou `IOCP` (Windows) pour les opérations d'entrée/sortie asynchrones.

## Concurrence pratique : exemples composites

### Serveur de traitement avec pipeline I/O + CPU

```cpp
namespace ex = std::execution;

auto cpu_sch = ex::system_context().get_scheduler();  
auto io_sch = io_service.get_scheduler();  

// Pour chaque requête, le pipeline traverse deux contextes
auto handle_request(Request req) {
    return ex::schedule(io_sch)
        | ex::then([req] { return read_body(req); })           // I/O

        | ex::continues_on(cpu_sch)
        | ex::then([](Body body) { return parse_json(body); }) // CPU
        | ex::then([](Json data) { return validate(data); })   // CPU
        | ex::then([](auto valid) { return build_response(valid); })

        | ex::continues_on(io_sch)
        | ex::then([](Response resp) { send_response(resp); }) // I/O
        ;
}
```

Chaque requête traverse naturellement les contextes d'exécution adaptés à chaque étape : I/O pour la réception et l'envoi, CPU pour le parsing et la validation. Le pipeline est lisible, composable, et le contexte d'exécution est explicite à chaque étape.

### Parallélisation de calcul avec agrégation

```cpp
namespace ex = std::execution;

auto sch = exec::static_thread_pool(std::thread::hardware_concurrency())
               .get_scheduler();

// Découper un vecteur en chunks et traiter en parallèle
auto parallel_process(std::span<const double> data) {
    constexpr size_t chunk_size = 1024;
    std::vector<ex::sender auto> senders;

    for (size_t i = 0; i < data.size(); i += chunk_size) {
        auto chunk = data.subspan(i, std::min(chunk_size, data.size() - i));
        senders.push_back(
            ex::schedule(sch) | ex::then([chunk] { return process_chunk(chunk); })
        );
    }

    // when_all attend tous les résultats, then les agrège
    return ex::when_all_with_variant(std::move(senders))
        | ex::then(aggregate_results);
}
```

### Retry avec backoff exponentiel

```cpp
namespace ex = std::execution;

template <typename Sender>  
auto retry_with_backoff(Sender sndr, auto sch, int max_retries) {  
    // let_error : en cas d'erreur, créer un nouveau sender de retry
    return std::move(sndr)
        | ex::let_error([sch, max_retries, attempt = 0](auto err) mutable
              -> ex::sender auto {
            if (++attempt >= max_retries) {
                return ex::just_error(err);   // Propager l'erreur finale
            }
            auto delay = std::chrono::milliseconds(100 * (1 << attempt));
            // Attendre puis réessayer
            return ex::schedule_after(sch, delay)
                | ex::then([] { return fetch_data(); });
        });
}
```

Ce pattern — réessayer une opération faillible avec un délai exponentiel — est un idiome classique des systèmes distribués. Avec `std::execution`, il se compose naturellement dans un pipeline sans callbacks imbriqués.

## Intégration avec les coroutines

Les senders et les coroutines C++20 (section 12.6) sont des abstractions complémentaires, et leur intégration est un objectif explicite du modèle. Un sender peut être utilisé comme expression `co_await` dans une coroutine :

```cpp
namespace ex = std::execution;

ex::task<std::string> fetch_and_process(auto sch) {
    // co_await sur un sender — suspend la coroutine jusqu'à complétion
    auto data = co_await (ex::schedule(sch) | ex::then(fetch_data));
    auto result = co_await (ex::schedule(sch) | ex::then([&] { return process(data); }));
    co_return result;
}
```

Le type `ex::task<T>` (proposé dans les extensions à P2300) est un sender qui encapsule une coroutine. Il combine la lisibilité du style séquentiel des coroutines avec le contrôle d'exécution des schedulers. Le code se lit comme du code synchrone mais s'exécute de manière asynchrone sur les contextes d'exécution spécifiés.

Cette intégration est le futur de la programmation asynchrone en C++ : les coroutines pour la logique séquentielle, les senders pour la composition et l'orchestration, les schedulers pour le contrôle du contexte.

## Intégration avec les bibliothèques réseau

L'intégration entre `std::execution` et les bibliothèques de networking est un objectif majeur du comité. Le scénario cible : Asio (section 22.4) ou une future bibliothèque réseau standard fournit des senders pour les opérations I/O (accept, read, write), et ces senders se composent naturellement avec les algorithmes de `std::execution`.

```cpp
// Scénario cible (conceptuel)
namespace ex = std::execution;

auto handle_connection(tcp::socket sock, auto cpu_sch) {
    return async_read(sock)                               // Sender I/O
        | ex::continues_on(cpu_sch)                       // Vers le pool CPU
        | ex::then([](auto data) { return parse(data); }) // Traitement CPU
        | ex::continues_on(sock.get_scheduler())           // Retour vers l'I/O
        | ex::then([&](auto resp) { return async_write(sock, resp); });
}
```

Ce scénario représente le « graal » de l'asynchronisme en C++ : un vocabulaire unique pour l'I/O, le calcul CPU, et le GPU, composable en pipelines lisibles avec contrôle explicite des contextes d'exécution.

## Le paysage en mars 2026

### Ce qui est disponible

- **stdexec (NVIDIA)** — Fournit `static_thread_pool`, `run_loop`, des schedulers GPU (CUDA), et la majorité des algorithmes sender. C'est l'implémentation la plus complète.
- **libunifex (Meta)** — Fournit un `thread_pool`, un `timed_single_thread_context`, et une intégration avec `io_uring` (Linux).
- **Asio** — L'intégration avec le modèle sender/receiver progresse. Asio fournit déjà ses propres awaitables pour les coroutines ; l'alignement avec `std::execution` est en cours.

### Ce qui est en attente

- **`system_context` dans le standard** — Le scheduler « par défaut » qui utilise le thread pool de la plateforme. Prioritaire pour le standard.
- **Schedulers basés sur io_uring/IOCP** — L'intégration I/O native dans le modèle sender est un objectif C++29.
- **`task<T>` standard** — Le type coroutine + sender qui combine les deux modèles.

## Bonnes pratiques

**Commencer avec un seul scheduler (system_context ou static_thread_pool).** La puissance des schedulers multiples est séduisante, mais un seul scheduler couvre la majorité des besoins. Ajouter des schedulers spécialisés quand le profiling le justifie.

**Rendre le scheduler un paramètre, pas un global.** Passer le scheduler en argument aux fonctions qui créent des pipelines. Cela permet de substituer un inline_scheduler dans les tests pour une exécution déterministe, et un thread pool en production :

```cpp
auto process(auto sch, Data data) {
    return ex::schedule(sch)
        | ex::then([data] { return transform(data); });
}

// Test : déterministe, pas de threading
process(inline_scheduler{}, test_data);

// Production : parallèle
process(system_sch, real_data);
```

**Utiliser `continues_on` pour les transitions de contexte.** Ne pas laisser le contexte d'exécution implicite — le rendre explicite à chaque transition améliore la lisibilité et prévient les bugs de threading (opération I/O sur un thread CPU, opération CPU sur le thread UI).

**Dimensionner les thread pools selon le profil de charge.** Pour du calcul CPU-bound, un thread par cœur physique. Pour de l'I/O-bound, un pool plus petit suffit car les threads passent l'essentiel de leur temps à attendre. Le profiling (section 31) guide le dimensionnement.

**Suivre l'évolution du standard.** `system_context`, `async_scope`, et l'intégration I/O sont les prochaines étapes majeures. Les propositions complémentaires (P2079, P3149, P2762) précisent le plan. Suivre les trip reports du comité et les releases de stdexec pour anticiper les changements.
