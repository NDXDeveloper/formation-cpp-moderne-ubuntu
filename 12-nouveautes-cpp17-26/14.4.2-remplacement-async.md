# 12.14.4.2 Remplacement de std::async et std::future

## Migrer vers le modèle Sender/Receiver

`std::async` et `std::future` restent fonctionnels et ne seront pas retirés du standard. Mais pour tout nouveau code concurrent, `std::execution` est le remplacement recommandé. Cette section montre, pattern par pattern, comment réécrire les idiomes courants de `std::async`/`std::future` avec des senders.

> 💡 *Les exemples de cette section utilisent le namespace alias `namespace ex = std::execution;` et supposent l'accès à un scheduler système. En mars 2026, l'implémentation de référence **stdexec** (NVIDIA) est le moyen le plus direct de les compiler.*

## Pattern 1 : Lancer une tâche et attendre le résultat

Le cas le plus basique — exécuter une fonction sur un autre thread et récupérer le résultat.

### Avec std::async

```cpp
#include <future>

// Lance compute() sur un thread, bloque pour récupérer le résultat
auto future = std::async(std::launch::async, compute);
int result = future.get();   // Bloque jusqu'à complétion
```

### Avec std::execution

```cpp
namespace ex = std::execution;

auto sch = ex::system_context().get_scheduler();

// Décrit le travail : démarrer sur le pool, puis exécuter compute
auto sender = ex::schedule(sch) | ex::then(compute);

// Exécute et attend
auto [result] = *ex::sync_wait(std::move(sender));
```

`sync_wait` retourne un `std::optional<std::tuple<...>>`. Le `*` déréférence l'optional (qui contient une valeur si le sender a complété avec `set_value`), et le structured binding `[result]` extrait la valeur du tuple.

La différence clé : le sender sépare la description (`schedule | then`) de l'exécution (`sync_wait`). Le scheduler est explicite — on sait exactement où le travail s'exécute.

## Pattern 2 : Enchaîner des opérations dépendantes

Le cas le plus courant en pratique — A produit un résultat consommé par B, qui produit un résultat consommé par C.

### Avec std::async — blocage à chaque étape

```cpp
auto futA = std::async(std::launch::async, fetch_data);
Data data = futA.get();                                    // Bloque

auto futB = std::async(std::launch::async, [&] { return process(data); });
Result result = futB.get();                                // Bloque

auto futC = std::async(std::launch::async, [&] { save(result); });
futC.get();                                                // Bloque
```

Trois blocages, quatre threads consommés (le thread courant + trois threads de travail). Le thread appelant passe la majorité de son temps à attendre.

### Avec std::execution — zéro blocage intermédiaire

```cpp
namespace ex = std::execution;

auto sch = ex::system_context().get_scheduler();

auto pipeline = ex::schedule(sch)
    | ex::then(fetch_data)
    | ex::then(process)
    | ex::then(save);

ex::sync_wait(std::move(pipeline));   // Un seul point de blocage
```

Le pipeline s'exécute sur les threads du pool sans jamais bloquer entre les étapes. Le thread qui termine `fetch_data` enchaîne directement sur `process`, qui enchaîne sur `save`. Le thread appelant ne bloque qu'une seule fois, à `sync_wait`. Deux threads suffisent (le thread appelant + un thread du pool) au lieu de quatre.

## Pattern 3 : Exécuter des tâches en parallèle

Lancer plusieurs tâches indépendantes et attendre que toutes terminent.

### Avec std::async — attente manuelle

```cpp
auto fut1 = std::async(std::launch::async, compute_a);
auto fut2 = std::async(std::launch::async, compute_b);
auto fut3 = std::async(std::launch::async, compute_c);

// Attendre chacun séquentiellement
auto a = fut1.get();
auto b = fut2.get();
auto c = fut3.get();

auto merged = merge(a, b, c);
```

Les trois tâches s'exécutent en parallèle, mais l'attente est séquentielle : si `fut2` termine avant `fut1`, on attend quand même `fut1` d'abord. Plus important : il n'y a aucune gestion de l'annulation — si `fut1` échoue, `fut2` et `fut3` continuent inutilement.

### Avec std::execution — when_all + annulation automatique

```cpp
namespace ex = std::execution;

auto sch = ex::system_context().get_scheduler();

auto pipeline = ex::when_all(
        ex::schedule(sch) | ex::then(compute_a),
        ex::schedule(sch) | ex::then(compute_b),
        ex::schedule(sch) | ex::then(compute_c)
    )
    | ex::then([](auto a, auto b, auto c) { return merge(a, b, c); });

auto [merged] = *ex::sync_wait(std::move(pipeline));
```

`when_all` lance les trois senders en parallèle et complète avec les résultats de tous quand ils sont disponibles — pas d'attente séquentielle. Si l'un des senders échoue, `when_all` peut propager une demande d'annulation aux autres (via le stop token), évitant le travail inutile.

Les résultats sont passés directement au `then` suivant comme arguments de la lambda — pas besoin de stocker des futures intermédiaires.

## Pattern 4 : Gestion d'erreurs

### Avec std::async — try/catch sur le get()

```cpp
auto future = std::async(std::launch::async, risky_operation);

try {
    auto result = future.get();   // Lance l'exception si l'opération a échoué
    use(result);
} catch (const std::exception& e) {
    log_error(e.what());
    use(default_value);
}
```

`std::future::get()` relance l'exception capturée pendant l'exécution asynchrone. La gestion d'erreur est séparée du point d'appel par un `try`/`catch`, ce qui casse le flux de lecture.

### Avec std::execution — upon_error dans le pipeline

```cpp
namespace ex = std::execution;

auto sch = ex::system_context().get_scheduler();

auto pipeline = ex::schedule(sch)
    | ex::then(risky_operation)
    | ex::upon_error([](auto err) {
        log_error(err);
        return default_value;
    })
    | ex::then(use);

ex::sync_wait(std::move(pipeline));
```

La gestion d'erreur est intégrée dans le pipeline, au même niveau que la logique métier. Si `risky_operation` échoue (que ce soit via une exception capturée dans `set_error` ou via un code d'erreur), `upon_error` intercepte l'erreur et fournit une valeur de fallback. Le pipeline continue ensuite normalement avec `use`.

Pour des cas plus complexes — où la récupération d'erreur est elle-même asynchrone — `let_error` remplace `upon_error` :

```cpp
auto pipeline = ex::schedule(sch)
    | ex::then(fetch_primary)
    | ex::let_error([sch](auto err) {
        // Récupération asynchrone : essayer un serveur de secours
        return ex::schedule(sch) | ex::then(fetch_fallback);
    })
    | ex::then(process);
```

`let_error` accepte une fonction qui retourne un **sender** (pas une valeur) — permettant de lancer une nouvelle opération asynchrone en cas d'erreur.

## Pattern 5 : Timeout

### Avec std::async — pas de mécanisme natif

```cpp
auto future = std::async(std::launch::async, slow_operation);

// wait_for ne fait que vérifier — il n'annule pas l'opération
if (future.wait_for(std::chrono::seconds(5)) == std::future_status::timeout) {
    // L'opération continue en arrière-plan — impossible de l'annuler !
    log("Timeout, mais l'opération tourne encore...");
}
```

`std::future::wait_for` vérifie si le résultat est disponible dans le délai, mais ne peut pas annuler l'opération. Le thread de travail continue son exécution — c'est une fuite de travail.

### Avec std::execution — annulation propre

Avec `std::execution`, un timeout peut être implémenté en combinant un stop token et un timer. La proposition complémentaire pour les schedulers système et les timers rendra ce pattern standard. En attendant, voici le principe conceptuel :

```cpp
namespace ex = std::execution;

auto sch = ex::system_context().get_scheduler();

// Conceptuel — la syntaxe exacte dépend des propositions complémentaires
auto pipeline = ex::when_all(
    ex::schedule(sch) | ex::then(slow_operation),
    ex::schedule_after(sch, std::chrono::seconds(5))  // Timer
) | ex::then([](auto result, auto) { return result; });
```

Le principe : `when_all` avec un timer crée une course entre l'opération et le timeout. Si le timer complète en premier, il peut déclencher l'annulation de l'opération via le stop token. L'opération vérifie coopérativement le stop token et se termine proprement.

## Pattern 6 : Fire-and-forget (à éviter)

### Avec std::async — le piège du fire-and-forget

```cpp
// PIÈGE : le future temporaire est détruit immédiatement
// Le destructeur de std::future bloque jusqu'à complétion !
std::async(std::launch::async, background_task);   // Bloque ici !

// Solution courante mais dangereuse :
auto future = std::async(std::launch::async, background_task);
// Si on oublie future.get() et que future est détruit avant
// la complétion, le comportement dépend de l'implémentation
```

Le fire-and-forget est l'un des pièges les plus connus de `std::async` : le destructeur d'un `std::future` retourné par `async` avec `launch::async` bloque jusqu'à complétion. Lancer et oublier est impossible sans stocker le future — ce qui contredit le concept même de fire-and-forget.

### Avec std::execution — concurrence structurée

`std::execution` décourage le fire-and-forget par design. La concurrence structurée exige que toute opération asynchrone soit rattachée à un scope et rejointe avant sa destruction. C'est une contrainte intentionnelle — les tâches orphelines sont une source de bugs.

Pour les cas où du travail doit être lancé sans attente immédiate, la proposition **async_scope** fournit un mécanisme de spawn structuré :

```cpp
namespace ex = std::execution;

// Conceptuel — basé sur les propositions complémentaires
ex::async_scope scope;

scope.spawn(ex::schedule(sch) | ex::then(background_task_1));
scope.spawn(ex::schedule(sch) | ex::then(background_task_2));

// ... faire autre chose ...

// Rejoindre TOUTES les tâches spawnées avant de quitter le scope
ex::sync_wait(scope.on_empty());
```

L'async_scope est un conteneur qui garde trace de toutes les tâches spawnées et garantit qu'elles sont toutes terminées avant sa destruction. C'est le fire-and-forget sûr — le travail est découplé du point de lancement, mais la jointure est garantie.

## Tableau récapitulatif de migration

```
┌────────────────────────────┬──────────────────────────────────────────┐
│ Pattern std::async/future  │ Équivalent std::execution                │
├────────────────────────────┼──────────────────────────────────────────┤
│ async(f)                   │ schedule(sch) | then(f)                  │
│ future.get()               │ sync_wait(sender)                        │
│ async(f1); get(); async(f2)│ then(f1) | then(f2)                      │
│ Lancer N tâches en //      │ when_all(s1, s2, ..., sN)                │
│ try { get() } catch(...)   │ upon_error(handler) / let_error(...)     │
│ future.wait_for(timeout)   │ when_all(sender, timer) + stop token     │
│ Fire-and-forget            │ async_scope.spawn(sender)                │
│ (pas d'annulation)         │ set_stopped() + stop tokens              │
└────────────────────────────┴──────────────────────────────────────────┘
```

## Coexistence pendant la transition

La migration n'a pas besoin d'être instantanée. `std::async`/`std::future` et `std::execution` peuvent coexister dans le même projet. Une stratégie de migration progressive :

**Phase 1 — Nouveau code.** Écrire tout nouveau code concurrent avec `std::execution` (via stdexec en attendant le support dans la bibliothèque standard).

**Phase 2 — Points de contact.** Aux frontières entre l'ancien et le nouveau code, convertir les futures en senders ou inversement. Un sender peut encapsuler un `std::future::get()` (en bloquant un thread du pool), et un future peut être créé à partir d'un sender via une promesse/future classique alimentée par le receiver.

**Phase 3 — Migration progressive.** Convertir les usages de `std::async` un par un, en commençant par les plus critiques en performance (les pipelines avec plusieurs `.get()` séquentiels) et ceux qui bénéficient de l'annulation.

**Phase 4 — Retrait.** Quand tout le code concurrent utilise `std::execution`, retirer les dépendances à `<future>`.

## Bonnes pratiques

**Remplacer les chaînes de `.get()` en priorité.** C'est là que le gain est le plus immédiat — chaque `.get()` éliminé supprime un point de blocage et libère un thread.

**Utiliser `when_all` plutôt que des futures indépendantes.** Le groupement structuré est plus sûr (annulation automatique) et plus performant (pas d'attente séquentielle).

**Préférer `upon_error` / `let_error` au try/catch autour de `sync_wait`.** Intégrer la gestion d'erreur dans le pipeline garde le flux de données lisible et évite de mélanger code synchrone et asynchrone.

**Ne pas recréer fire-and-forget.** Si le design nécessite de lancer du travail en arrière-plan, utiliser un async_scope plutôt que des hacks pour « détacher » un sender. La concurrence structurée est une contrainte qui prévient les bugs.

**Mesurer le gain.** La migration vers `std::execution` élimine les blocages inutiles et réduit le nombre de threads nécessaires. Pour les applications I/O-bound ou les pipelines de traitement multi-étapes, le gain peut être significatif. Le profiling (section 31) confirme les bénéfices.
