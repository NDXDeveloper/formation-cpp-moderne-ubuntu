# 32.1.2 — Checks recommandés

## Introduction

La section précédente a présenté la mécanique de configuration du fichier `.clang-tidy`. Cette section aborde la question la plus pratique : **quels checks activer concrètement ?** Avec plus de 400 checks disponibles, le choix peut sembler intimidant. Activer tous les checks produit un flot de diagnostics — certains pertinents, d'autres bruyants — qui noie l'information utile. N'en activer que quelques-uns laisse passer des bugs que l'outil aurait pu détecter.

La clé est de constituer un jeu de checks adapté au contexte du projet, en évaluant chaque check selon trois critères : son **taux de vrais positifs** (signale-t-il des problèmes réels ?), son **rapport signal/bruit** (combien de faux positifs pour un vrai positif ?), et sa **valeur ajoutée** (le problème détecté est-il grave ou cosmétique ?).

Cette section propose des sélections éprouvées, organisées par niveau de priorité, avec la justification de chaque choix.

---

## Socle minimal : les checks que tout projet devrait activer

Ces checks ont un taux de vrais positifs très élevé et détectent des bugs graves. Ils génèrent peu de faux positifs et ne sont pas controversés du point de vue du style. Quel que soit le projet — neuf ou legacy, strict ou permissif — ces checks constituent un filet de sécurité de base.

### Bugs critiques

```yaml
Checks: >
  bugprone-use-after-move,
  bugprone-dangling-handle,
  bugprone-forwarding-reference-overload,
  bugprone-infinite-loop,
  bugprone-move-forwarding-reference,
  bugprone-string-constructor,
  bugprone-stringview-nullptr,
  bugprone-suspicious-memory-comparison,
  bugprone-undefined-memory-manipulation,
  bugprone-virtual-near-miss,
```

| Check | Pourquoi il est essentiel |
|---|---|
| `bugprone-use-after-move` | Utiliser un objet après `std::move` est un comportement indéfini en pratique. Ce bug est silencieux — le code compile sans erreur et peut fonctionner « par chance » pendant des mois. |
| `bugprone-dangling-handle` | Un `std::string_view` ou un `std::span` qui survit à l'objet qu'il référence est un pointeur dangling déguisé. Ce pattern est de plus en plus fréquent avec l'adoption du C++ moderne. |
| `bugprone-forwarding-reference-overload` | Un constructeur template avec une référence universelle (`T&&`) capture des appels qui devraient aller au constructeur de copie. C'est un piège classique des templates. |
| `bugprone-infinite-loop` | Détecte les boucles dont la condition de sortie ne peut jamais être satisfaite. Souvent causé par une erreur de typage ou une variable modifiée dans le mauvais scope. |
| `bugprone-virtual-near-miss` | Une méthode qui ressemble à un override mais qui diffère légèrement dans sa signature (mauvais type, `const` manquant). Le développeur pense surcharger, mais il crée une nouvelle méthode. |

### Analyse de chemin (Clang Static Analyzer)

```yaml
Checks: >
  clang-analyzer-core.*,
  clang-analyzer-cplusplus.*,
  clang-analyzer-deadcode.DeadStores,
  clang-analyzer-unix.Malloc,
  clang-analyzer-unix.MismatchedDeallocator,
```

| Check | Pourquoi il est essentiel |
|---|---|
| `clang-analyzer-core.NullDereference` | Détecte les chemins d'exécution où un pointeur potentiellement nul est déréférencé. Analyse interprocédurale — bien plus puissant qu'un simple pattern matching. |
| `clang-analyzer-core.StackAddressEscape` | Retour d'un pointeur ou d'une référence vers une variable locale. Bug grave qui produit un comportement indéfini dès le premier accès. |
| `clang-analyzer-core.UndefinedBinaryOperatorResult` | Opération arithmétique dont le résultat est indéfini (division par zéro, shift excessif). |
| `clang-analyzer-cplusplus.NewDeleteLeaks` | Fuite mémoire détectable statiquement : un `new` sans `delete` correspondant sur au moins un chemin d'exécution. |
| `clang-analyzer-unix.MismatchedDeallocator` | `new` libéré avec `free`, `malloc` libéré avec `delete` — mismatch de désallocation. |

### Performance évidente

```yaml
Checks: >
  performance-unnecessary-value-param,
  performance-unnecessary-copy-initialization,
  performance-for-range-copy,
  performance-implicit-conversion-in-loop,
  performance-move-const-arg,
```

| Check | Pourquoi il est essentiel |
|---|---|
| `performance-unnecessary-value-param` | Un paramètre copié inutilement alors qu'il n'est lu que par `const` reference. Chaque appel effectue une copie superflue. Ce check a un fix automatique fiable. |
| `performance-unnecessary-copy-initialization` | Une variable locale initialisée par copie alors qu'une `const&` suffit. Même logique : copie évitable, fix automatique disponible. |
| `performance-for-range-copy` | `for (auto x : container)` copie chaque élément au lieu de `for (const auto& x : container)`. Dans une boucle sur un `vector<string>`, l'impact peut être massif. |
| `performance-move-const-arg` | `std::move` sur un argument `const` est silencieusement ignoré (on obtient une copie, pas un move). Le code suggère une intention de performance qui n'est pas réalisée. |

## Deuxième couche : modernisation et lisibilité

Ces checks améliorent la qualité du code sans détecter des bugs à proprement parler. Ils sont recommandés pour les projets qui visent une base de code moderne et maintenable.

### Migration vers le C++ moderne

```yaml
Checks: >
  modernize-use-nullptr,
  modernize-use-override,
  modernize-use-emplace,
  modernize-use-using,
  modernize-loop-convert,
  modernize-use-default-member-init,
  modernize-use-nodiscard,
  modernize-use-starts-ends-with,
  modernize-use-std-print,
  modernize-redundant-void-arg,
  modernize-deprecated-headers,
```

| Check | Transformation | Risque du fix |
|---|---|---|
| `modernize-use-nullptr` | `NULL`/`0` → `nullptr` | Quasi nul. Fix mécanique et sûr. |
| `modernize-use-override` | Ajout de `override` sur les méthodes virtuelles | Nul. Ne change pas le comportement, mais détecte les faux overrides. |
| `modernize-use-emplace` | `push_back(T(...))` → `emplace_back(...)` | Faible. Peut changer le comportement si le constructeur est `explicit`. |
| `modernize-use-using` | `typedef` → `using` | Nul. Équivalent syntaxique. |
| `modernize-loop-convert` | Boucle par index → range-based for | Faible. Vérifier que l'index n'est pas utilisé ailleurs dans la boucle. |
| `modernize-use-nodiscard` | Ajout de `[[nodiscard]]` sur les fonctions qui retournent des valeurs importantes | Nul. Ajoute une protection à la compilation. |
| `modernize-use-starts-ends-with` | `str.find("prefix") == 0` → `str.starts_with("prefix")` (C++20) | Nul. Plus lisible et potentiellement plus efficace. |
| `modernize-use-std-print` | `printf`/`cout <<` → `std::print` (C++23) | Faible. Vérifier le support compilateur (section 2.7). |

### Lisibilité

```yaml
Checks: >
  readability-braces-around-statements,
  readability-redundant-string-cstr,
  readability-redundant-smartptr-get,
  readability-make-member-function-const,
  readability-container-size-empty,
  readability-simplify-boolean-expr,
  readability-avoid-const-params-in-decls,
  readability-qualified-auto,
```

| Check | Ce qu'il améliore |
|---|---|
| `readability-braces-around-statements` | Force les accolades sur les `if`/`for`/`while` à une ligne. Prévient la classe de bugs « goto fail » (Apple SSL, 2014). |
| `readability-redundant-string-cstr` | Supprime les appels `.c_str()` inutiles (passer un `std::string` à une fonction qui accepte `std::string`). |
| `readability-redundant-smartptr-get` | Supprime les `.get()` inutiles sur les smart pointers lorsque la conversion implicite suffit. |
| `readability-make-member-function-const` | Signale les méthodes qui ne modifient pas l'objet et pourraient être `const`. Renforce le contrat de l'interface. |
| `readability-container-size-empty` | Remplace `container.size() == 0` par `container.empty()`. Plus lisible, potentiellement plus efficace sur certains conteneurs. |
| `readability-simplify-boolean-expr` | Simplifie les expressions booléennes inutilement complexes (`if (b == true)` → `if (b)`). |

## Troisième couche : rigueur et sécurité

Ces checks sont plus stricts et plus opinionated. Ils sont recommandés pour les projets à haute exigence de qualité (librairies publiques, code critique, logiciels exposés à des entrées non fiables).

### C++ Core Guidelines (sélection)

L'activation complète de `cppcoreguidelines-*` est très stricte et produit de nombreux diagnostics sur la plupart des bases de code. La sélection suivante retient les checks les plus pertinents :

```yaml
Checks: >
  cppcoreguidelines-owning-memory,
  cppcoreguidelines-no-malloc,
  cppcoreguidelines-slicing,
  cppcoreguidelines-special-member-functions,
  cppcoreguidelines-init-variables,
  cppcoreguidelines-pro-type-member-init,
  cppcoreguidelines-pro-type-static-cast-downcast,
  cppcoreguidelines-avoid-goto,
  cppcoreguidelines-narrowing-conversions,
```

| Check | Guideline |
|---|---|
| `cppcoreguidelines-owning-memory` | Tout pointeur brut qui possède une ressource devrait être un smart pointer. Détecte les `new` non encapsulés. |
| `cppcoreguidelines-no-malloc` | Pas de `malloc`/`free` en C++. Utiliser `new`/`delete` ou, mieux, les smart pointers et conteneurs. |
| `cppcoreguidelines-slicing` | Détecte le *object slicing* : passage d'un objet dérivé par valeur à une fonction attendant la base. La partie dérivée est silencieusement tronquée. |
| `cppcoreguidelines-special-member-functions` | Détecte les violations de la Rule of Five (section 6.5) : si une classe définit un destructeur, elle devrait définir ou supprimer les 4 autres opérations spéciales. |
| `cppcoreguidelines-init-variables` | Les variables locales doivent être initialisées à la déclaration. Prévient l'utilisation de valeurs non initialisées. |
| `cppcoreguidelines-narrowing-conversions` | Conversions implicites qui peuvent perdre des données (`int64_t` → `int32_t`, `double` → `float`). |

### Sécurité (CERT)

```yaml
Checks: >
  cert-err34-c,
  cert-err52-cpp,
  cert-err60-cpp,
  cert-flp30-c,
  cert-msc50-cpp,
  cert-msc51-cpp,
  cert-str34-c,
```

| Check | Règle CERT |
|---|---|
| `cert-err34-c` | Vérifier la valeur de retour des fonctions de conversion (`atoi`, `strtol`). |
| `cert-err52-cpp` | Ne pas utiliser `setjmp`/`longjmp` en C++ (incompatible avec les destructeurs). |
| `cert-err60-cpp` | Les types d'exception doivent être copiables sans lancer d'exception. |
| `cert-flp30-c` | Ne pas utiliser de variables flottantes comme compteur de boucle. |
| `cert-msc50-cpp` | Ne pas utiliser `rand()` pour la génération de nombres aléatoires — utiliser `<random>`. |
| `cert-msc51-cpp` | Ne pas utiliser une seed prévisible pour les générateurs pseudo-aléatoires. |

### Concurrence

```yaml
Checks: >
  concurrency-mt-unsafe,
  bugprone-spuriously-wake-up-functions,
```

| Check | Ce qu'il détecte |
|---|---|
| `concurrency-mt-unsafe` | Utilisation de fonctions non thread-safe de la librairie C (`strtok`, `localtime`, `getenv` sans protection) dans un contexte potentiellement concurrent. |
| `bugprone-spuriously-wake-up-functions` | Utilisation de `condition_variable::wait` sans prédicat — vulnérable aux réveils intempestifs (*spurious wakeups*). |

## Checks déconseillés ou à éviter

Certains checks sont controversés, trop bruyants, ou inadaptés à la majorité des projets. Les désactiver explicitement évite les frustrations.

### Checks trop bruyants

| Check | Problème |
|---|---|
| `readability-identifier-length` | Signale les variables de 1 ou 2 caractères. Les itérateurs `i`, `j`, `k` et les variables de lambda `x` sont des conventions universellement acceptées. |
| `bugprone-easily-swappable-parameters` | Signale chaque paire de paramètres du même type. Produit des dizaines de warnings sur des APIs parfaitement claires comme `void set_range(int min, int max)`. |
| `cppcoreguidelines-avoid-magic-numbers` / `readability-magic-numbers` | Signale chaque littéral numérique. Pertinent pour le code de production, mais insupportable dans les tests et le code d'initialisation. Préférez l'activer uniquement dans la configuration racine et le désactiver dans `tests/.clang-tidy`. |
| `misc-non-private-member-variables-in-classes` | Interdit les membres publics dans les classes. Rend impossible l'utilisation de `struct` comme agrégat de données, un idiome parfaitement légitime en C++. |

### Checks controversés

| Check | Pourquoi il divise |
|---|---|
| `modernize-use-trailing-return-type` | Transforme `int foo()` en `auto foo() -> int`. La syntaxe trailing est utile pour les templates et `decltype`, mais l'imposer partout est un choix de style que beaucoup d'équipes rejettent. |
| `modernize-use-auto` | Remplace les types explicites par `auto`. Utile pour les types longs (`std::vector<std::pair<int,int>>::iterator`), mais controversé sur les types simples (`int`, `double`). L'option `MinTypeNameLength` atténue le problème. |
| `readability-else-after-return` | Signale les `else` après un `return`. Certains styles de codage préfèrent le `else` explicite pour la symétrie visuelle. |

### Checks spécifiques à des projets

Les catégories suivantes implémentent des conventions spécifiques à des projets particuliers. Elles ne sont pertinentes que si vous travaillez sur ces projets :

- **`llvmlibc-*`** : conventions du projet LLVM libc.
- **`fuchsia-*`** : conventions du système d'exploitation Fuchsia (Google).
- **`altera-*`** : conventions pour le développement FPGA Altera/Intel.
- **`android-*`** : conventions spécifiques au développement Android NDK.
- **`abseil-*`** : checks pour la librairie Abseil (Google).

Désactivez-les systématiquement sauf si vous êtes dans le contexte correspondant :

```yaml
Checks: >
  -*,
  # ... vos checks ...
  -llvmlibc-*,
  -fuchsia-*,
  -altera-*
```

## Stratégie d'adoption pour un projet existant

Activer d'un coup les trois couches de checks sur un projet existant de 100 000 lignes produirait des milliers de warnings, décourageant l'équipe et rendant les diagnostics inexploitables. L'adoption progressive est la clé.

### Phase 1 : Socle minimal (semaine 1)

Activez le socle minimal (bugs critiques + clang-analyzer + performance évidente). Corrigez les warnings existants — ils sont peu nombreux et chacun est probablement un vrai bug. Promouvez ces checks en `WarningsAsErrors` pour empêcher l'introduction de nouveaux bugs de ces catégories.

### Phase 2 : Modernisation (semaines 2–4)

Activez les checks `modernize-*` un par un. Utilisez `run-clang-tidy --fix` avec un seul check à la fois pour générer des commits atomiques :

```bash
# Semaine 2 : nullptr
run-clang-tidy -p build/ -fix -checks='-*,modernize-use-nullptr'  
git add -A && git commit -m "refactor: modernize-use-nullptr"  

# Semaine 3 : override
run-clang-tidy -p build/ -fix -checks='-*,modernize-use-override'  
git add -A && git commit -m "refactor: modernize-use-override"  
```

### Phase 3 : Lisibilité et rigueur (mois 2+)

Activez progressivement les checks de lisibilité et les Core Guidelines. Chaque nouveau check est ajouté dans le `.clang-tidy`, les warnings existants sont corrigés dans un commit dédié, puis le check est promu en `WarningsAsErrors`.

### Phase 4 : Stabilisation

Lorsque le nombre de warnings atteint zéro sur l'ensemble du projet, la configuration est gelée. Toute modification future (ajout ou suppression d'un check) passe par une code review et est documentée dans le changelog.

Le rythme exact dépend de la taille de l'équipe et du projet. L'objectif est la progression continue, pas la perfection immédiate. Un projet qui corrige 10 warnings par semaine pendant 6 mois se retrouve avec une base de code nettement plus saine qu'un projet qui tente une correction massive puis abandonne face à l'ampleur de la tâche.
