# 27.2.3 Profils et settings

> **Objectif** : Maîtriser le système de profils de Conan 2.0 — créer et gérer des configurations multiples (Debug/Release, GCC/Clang, x86_64/ARM64), comprendre le modèle double profil host/build, et utiliser les settings, les sections `[conf]` et `[buildenv]` pour un contrôle fin de la compilation.

---

## Le rôle du profil

Un profil Conan est un fichier texte qui décrit un **environnement de compilation complet**. Il répond à la question : « pour quelle plateforme, avec quel compilateur, quel standard et quel type de build les dépendances doivent-elles être compilées ? »

Le profil est l'entrée principale du calcul de **package ID** — le hash qui identifie une variante binaire. Deux profils différents produisent des package IDs différents, donc des binaires différents. C'est ainsi que Conan gère la matrice de configurations décrite en section 27.1.

---

## Anatomie d'un profil

Un profil se compose de plusieurs sections, chacune contrôlant un aspect de la configuration :

```ini
# ~/.conan2/profiles/gcc15-release
[settings]
os=Linux  
arch=x86_64  
compiler=gcc  
compiler.version=15  
compiler.cppstd=23  
compiler.libcxx=libstdc++11  
build_type=Release  

[options]
*:shared=False

[conf]
tools.cmake.cmaketoolchain:generator=Ninja  
tools.build:jobs=8  

[buildenv]
CC=gcc-15  
CXX=g++-15  
```

### Section `[settings]`

Les settings sont les paramètres fondamentaux de la matrice de configurations. Ils déterminent la compatibilité binaire entre paquets.

| Setting | Valeurs courantes | Impact |
|---------|------------------|--------|
| `os` | `Linux`, `Windows`, `Macos`, `Android` | Format binaire, APIs système |
| `arch` | `x86_64`, `armv8`, `riscv64` | Jeu d'instructions |
| `compiler` | `gcc`, `clang`, `msvc`, `apple-clang` | ABI, name mangling |
| `compiler.version` | `14`, `15`, `20` | Changements ABI potentiels |
| `compiler.cppstd` | `17`, `20`, `23`, `gnu23` | Standard C++, layout mémoire |
| `compiler.libcxx` | `libstdc++11`, `libc++` | Implémentation de la bibliothèque standard |
| `build_type` | `Release`, `Debug`, `RelWithDebInfo`, `MinSizeRel` | Optimisations, symboles debug |

Le setting `compiler.libcxx` mérite une attention particulière sous Linux. La valeur `libstdc++11` correspond à la nouvelle ABI de libstdc++ (introduite par GCC 5 en 2015, utilisant le SSO pour `std::string`). La valeur `libstdc++` (sans le `11`) correspond à l'ancienne ABI pré-GCC 5 — vous ne devriez jamais avoir besoin de cette valeur en 2026. Pour Clang avec libc++ (LLVM), utilisez `libc++`.

### Section `[options]`

Les options permettent de configurer les paquets individuellement ou globalement :

```ini
[options]
*:shared=False                    # Toutes les dépendances en statique
openssl/*:no_deprecated=True      # Désactiver les API deprecated d'OpenSSL  
spdlog/*:header_only=False        # spdlog compilé (pas header-only)  
```

Les options définies dans le profil s'appliquent à toutes les installations qui utilisent ce profil. Elles peuvent être surchargées en ligne de commande avec `-o` :

```bash
conan install . --profile=gcc15-release -o "openssl/*:shared=True"
```

### Section `[conf]`

La section `[conf]` configure le comportement des **outils** Conan — les helpers qui génèrent les fichiers d'intégration et pilotent la compilation :

```ini
[conf]
# Générateur CMake utilisé par CMakeToolchain
tools.cmake.cmaketoolchain:generator=Ninja

# Nombre de jobs de compilation parallèle
tools.build:jobs=8

# Exécutables du compilateur
tools.build:compiler_executables={"c": "gcc-15", "cpp": "g++-15"}

# Variables CMake injectées dans le toolchain
tools.cmake.cmaketoolchain:cache_variables={"MY_PROJECT_ENABLE_FEATURE_X": "ON"}
```

La section `[conf]` est la plus flexible du profil. Elle influence directement les fichiers générés par `CMakeToolchain` et `CMakeDeps`.

### Section `[buildenv]`

La section `[buildenv]` définit des **variables d'environnement** appliquées pendant la compilation des dépendances :

```ini
[buildenv]
CC=gcc-15  
CXX=g++-15  
CFLAGS=-march=native  
CXXFLAGS=-march=native  
```

Ces variables sont injectées dans l'environnement quand Conan compile un paquet depuis les sources (`--build=missing`). Elles n'affectent pas votre projet directement — elles influencent la compilation des dépendances tierces.

## Le modèle double profil : host et build

Conan 2.0 introduit un concept fondamental absent de Conan 1.x : la distinction entre la plateforme **host** (cible) et la plateforme **build** (de compilation).

### Build natif : host = build

Pour un build natif (compiler sur x86_64 pour exécuter sur x86_64), les deux profils sont identiques. Par défaut, Conan utilise le profil `default` pour les deux :

```bash
# Implicitement : --profile:host=default --profile:build=default
conan install . --build=missing
```

Vous pouvez spécifier un profil explicitement :

```bash
conan install . --profile:host=gcc15-release --profile:build=gcc15-release --build=missing

# Raccourci : -pr:h et -pr:b
conan install . -pr:h=gcc15-release -pr:b=gcc15-release --build=missing
```

Quand vous ne spécifiez qu'un seul `--profile` (sans `:host` ni `:build`), Conan l'utilise comme profil host. Le profil build prend la valeur `default` :

```bash
# Équivalent à : -pr:h=gcc15-release -pr:b=default
conan install . --profile=gcc15-release --build=missing
```

### Cross-compilation : host ≠ build

La puissance du double profil apparaît en cross-compilation. Votre machine de build est x86_64, mais vous compilez pour ARM64 :

```bash
conan install . -pr:h=arm64-linux -pr:b=default --build=missing
```

Le profil **host** (`arm64-linux`) décrit la cible :

```ini
# ~/.conan2/profiles/arm64-linux
[settings]
os=Linux  
arch=armv8  
compiler=gcc  
compiler.version=14  
compiler.cppstd=23  
compiler.libcxx=libstdc++11  
build_type=Release  

[conf]
tools.build:compiler_executables={"c": "aarch64-linux-gnu-gcc", "cpp": "aarch64-linux-gnu-g++"}
```

Le profil **build** (`default`) décrit la machine de compilation — votre poste x86_64 avec GCC 15.

Conan utilise cette distinction pour les `tool_requires` : les outils comme `protoc` (Protocol Buffers compiler) doivent être compilés pour la machine de build (x86_64) — ils s'exécutent pendant la compilation — tandis que les bibliothèques comme `libprotobuf` doivent être compilées pour la cible (ARM64).

```python
def build_requirements(self):
    self.tool_requires("protobuf/5.27.0")  # Compilé pour BUILD (x86_64)

def requirements(self):
    self.requires("protobuf/5.27.0")       # Compilé pour HOST (ARM64)
```

Sans le double profil, cette distinction serait impossible à exprimer.

## Profils courants pour un projet type

Voici un ensemble de profils couvrant les scénarios les plus fréquents.

### Debug GCC

```ini
# ~/.conan2/profiles/gcc-debug
[settings]
os=Linux  
arch=x86_64  
compiler=gcc  
compiler.version=15  
compiler.cppstd=23  
compiler.libcxx=libstdc++11  
build_type=Debug  

[options]
*:shared=False

[conf]
tools.cmake.cmaketoolchain:generator=Ninja  
tools.build:compiler_executables={"c": "gcc-15", "cpp": "g++-15"}  
```

### Release GCC

```ini
# ~/.conan2/profiles/gcc-release
[settings]
os=Linux  
arch=x86_64  
compiler=gcc  
compiler.version=15  
compiler.cppstd=23  
compiler.libcxx=libstdc++11  
build_type=Release  

[options]
*:shared=False

[conf]
tools.cmake.cmaketoolchain:generator=Ninja  
tools.build:compiler_executables={"c": "gcc-15", "cpp": "g++-15"}  
```

### Clang avec libc++

```ini
# ~/.conan2/profiles/clang-release
[settings]
os=Linux  
arch=x86_64  
compiler=clang  
compiler.version=20  
compiler.cppstd=23  
compiler.libcxx=libc++  
build_type=Release  

[options]
*:shared=False

[conf]
tools.cmake.cmaketoolchain:generator=Ninja  
tools.build:compiler_executables={"c": "clang-20", "cpp": "clang++-20"}  
```

Notez le `compiler.libcxx=libc++` — Clang peut utiliser soit `libstdc++` (la bibliothèque standard de GCC) soit `libc++` (celle de LLVM). Le choix affecte l'ABI et doit être cohérent entre toutes les dépendances.

### Cross-compilation ARM64

```ini
# ~/.conan2/profiles/arm64-release
[settings]
os=Linux  
arch=armv8  
compiler=gcc  
compiler.version=14  
compiler.cppstd=23  
compiler.libcxx=libstdc++11  
build_type=Release  

[options]
*:shared=False

[conf]
tools.build:compiler_executables={"c": "aarch64-linux-gnu-gcc", "cpp": "aarch64-linux-gnu-g++"}  
tools.cmake.cmaketoolchain:generator=Ninja  
```

## Héritage de profils avec `include()`

Pour éviter la duplication entre profils qui ne diffèrent que par un ou deux settings, Conan supporte l'inclusion :

```ini
# ~/.conan2/profiles/base-linux
[settings]
os=Linux  
arch=x86_64  
compiler=gcc  
compiler.version=15  
compiler.cppstd=23  
compiler.libcxx=libstdc++11  

[options]
*:shared=False

[conf]
tools.cmake.cmaketoolchain:generator=Ninja  
tools.build:compiler_executables={"c": "gcc-15", "cpp": "g++-15"}  
```

```ini
# ~/.conan2/profiles/debug
include(base-linux)

[settings]
build_type=Debug
```

```ini
# ~/.conan2/profiles/release
include(base-linux)

[settings]
build_type=Release
```

```ini
# ~/.conan2/profiles/asan
include(debug)

[conf]
tools.build:cflags=["-fsanitize=address", "-fno-omit-frame-pointer"]  
tools.build:cxxflags=["-fsanitize=address", "-fno-omit-frame-pointer"]  
tools.build:sharedlinkflags=["-fsanitize=address"]  
tools.build:exelinkflags=["-fsanitize=address"]  
```

Le profil `asan` hérite de `debug`, qui hérite de `base-linux`. Les settings et sections des profils enfants **écrasent** ceux du parent au niveau de chaque clé. Les listes (comme `cxxflags`) dans `[conf]` remplacent celles du parent — elles ne fusionnent pas.

Ce mécanisme d'héritage est précieux pour les équipes qui maintiennent de nombreuses configurations. Chaque variation est exprimée de manière minimale, et les modifications de la base se propagent automatiquement.

## Surcharges en ligne de commande

Tout setting, option ou conf peut être surchargé en ligne de commande, sans modifier le profil :

```bash
# Surcharger le build_type
conan install . -pr=gcc-release -s build_type=Debug --build=missing

# Surcharger le standard C++
conan install . -pr=gcc-release -s compiler.cppstd=20 --build=missing

# Surcharger une option
conan install . -pr=gcc-release -o "openssl/*:shared=True" --build=missing

# Surcharger une conf
conan install . -pr=gcc-release -c tools.build:jobs=4 --build=missing
```

Les flags `-s` (settings), `-o` (options) et `-c` (conf) s'appliquent au profil **host** par défaut. Pour surcharger le profil build, ajoutez le suffixe `:build` :

```bash
conan install . -pr:h=arm64-release -pr:b=default -s:b build_type=Release --build=missing
```

Les surcharges en ligne de commande sont pratiques pour les tests ponctuels. Pour les configurations récurrentes, un profil dédié est toujours préférable.

## Lister les settings disponibles

Conan embarque un fichier `settings.yml` qui définit l'ensemble des valeurs valides pour chaque setting. Pour consulter les valeurs possibles :

```bash
# Voir la configuration complète des settings
conan config show "settings.yml"
```

Pour connaître les versions de compilateur reconnues par Conan :

```bash
conan config show "settings.yml" | grep -A 20 "gcc:"
```

Si votre version de compilateur n'est pas dans la liste (cas rare avec les compilateurs très récents), vous pouvez étendre `settings.yml` via le répertoire `~/.conan2/extensions/`.

## Profils et CI/CD

En CI/CD, les profils garantissent que les builds sont reproductibles indépendamment de la machine. La stratégie recommandée est de **versionner les profils dans le dépôt du projet** plutôt que de dépendre des profils installés sur les runners CI :

```
my_project/
├── CMakeLists.txt
├── conanfile.py
├── conan/
│   └── profiles/
│       ├── linux-gcc-debug
│       ├── linux-gcc-release
│       ├── linux-clang-release
│       └── linux-arm64-release
└── ...
```

Les profils sont alors référencés par chemin relatif :

```bash
conan install . -pr:h=conan/profiles/linux-gcc-release -pr:b=conan/profiles/linux-gcc-release --build=missing
```

Cette approche offre deux avantages. D'une part, les profils sont versionnés avec le code — tout contributeur obtient exactement les mêmes configurations. D'autre part, un changement de profil (montée de version du compilateur, changement de standard) passe par le même processus de review que le code.

Un pipeline CI typique utilise une **matrix build** pour tester plusieurs profils :

```yaml
# .github/workflows/build.yml (extrait)
strategy:
  matrix:
    profile:
      - linux-gcc-debug
      - linux-gcc-release
      - linux-clang-release

steps:
  - name: Install dependencies
    run: conan install . -pr:h=conan/profiles/${{ matrix.profile }}
                         -pr:b=conan/profiles/${{ matrix.profile }}
                         --build=missing
```

## Récapitulatif

| Concept | Rôle |
|---------|------|
| **Profil** | Fichier décrivant un environnement de compilation complet |
| **Settings** (`[settings]`) | Paramètres de la matrice de configurations (OS, arch, compiler, build_type) |
| **Options** (`[options]`) | Configuration des paquets (shared, features spécifiques) |
| **Conf** (`[conf]`) | Configuration des outils Conan (générateur, jobs, compilateur) |
| **Build env** (`[buildenv]`) | Variables d'environnement pour la compilation des dépendances |
| **Profil host** (`-pr:h`) | Plateforme cible (où le binaire s'exécutera) |
| **Profil build** (`-pr:b`) | Plateforme de compilation (où la compilation s'effectue) |
| **`include()`** | Héritage entre profils pour éviter la duplication |
| **Surcharges CLI** | `-s`, `-o`, `-c` pour les ajustements ponctuels |
