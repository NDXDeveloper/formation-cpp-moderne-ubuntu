# 27.2.4 Intégration CMake

> **Objectif** : Mettre en œuvre le workflow complet Conan + CMake de bout en bout — de `conan install` à `cmake --build` — en exploitant les presets générés automatiquement, et en gérant la cohabitation avec les dépendances FetchContent.

---

## Le workflow complet, pas à pas

Reprenons le projet exemple du chapitre 26 et connectons toutes les pièces. Voici l'arborescence avec les fichiers Conan :

```
my_project/
├── CMakeLists.txt          # Inchangé — find_package() + target_link_libraries()
├── CMakePresets.json        # Presets du projet (optionnel — Conan génère les siens)
├── conanfile.py             # Déclaration des dépendances Conan
├── conan/
│   └── profiles/
│       ├── gcc-debug
│       └── gcc-release
├── src/
│   ├── CMakeLists.txt
│   └── ...
├── apps/
├── tests/
└── ...
```

### Étape 1 : installer les dépendances

```bash
conan install . --profile:host=conan/profiles/gcc-release \
                --profile:build=conan/profiles/gcc-release \
                --build=missing
```

Conan télécharge (ou compile) les dépendances et génère les fichiers d'intégration dans le répertoire de build. Grâce à `cmake_layout()` dans le `conanfile.py`, les fichiers sont placés dans `build/Release/generators/` :

```
build/
└── Release/
    └── generators/
        ├── conan_toolchain.cmake        ← CMakeToolchain
        ├── CMakePresets.json            ← Presets générés par Conan (voir plus bas)
        ├── OpenSSLConfig.cmake          ← CMakeDeps
        ├── OpenSSLTargets.cmake
        ├── spdlogConfig.cmake
        ├── spdlogTargets.cmake
        ├── ZLIBConfig.cmake
        ├── ZLIBTargets.cmake
        ├── GTestConfig.cmake
        ├── GTestTargets.cmake
        └── ...
```

### Étape 2 : configurer CMake

```bash
cmake -B build/Release -G Ninja \
    -DCMAKE_TOOLCHAIN_FILE=build/Release/generators/conan_toolchain.cmake \
    -DCMAKE_BUILD_TYPE=Release
```

CMake charge le toolchain Conan, qui configure le compilateur, le standard C++, les flags, et le `CMAKE_PREFIX_PATH` pour que `find_package()` trouve les fichiers `*Config.cmake` générés par `CMakeDeps`.

### Étape 3 : compiler

```bash
cmake --build build/Release
```

### Étape 4 : tester

```bash
ctest --test-dir build/Release --output-on-failure
```

C'est le workflow de base. Quatre commandes, toujours les mêmes. Voyons maintenant comment l'améliorer.

## Les presets générés par Conan

L'un des apports les plus pratiques de `cmake_layout()` est la génération automatique d'un fichier `CMakePresets.json` (ou `CMakeUserPresets.json`) contenant les presets correspondant à la configuration Conan. Ce fichier est placé dans `build/Release/generators/`.

Conan génère en fait un `CMakeUserPresets.json` à la racine du projet qui inclut les presets générés. Ce fichier contient des presets nommés d'après la configuration :

```bash
# Après conan install, les presets sont disponibles :
cmake --list-presets
# Available configure presets:
#   "conan-release" - 'conan-release' config
```

Le workflow se simplifie alors considérablement :

```bash
# Étape 1 : installer les dépendances
conan install . -pr:h=conan/profiles/gcc-release -pr:b=conan/profiles/gcc-release --build=missing

# Étape 2+3 : configurer ET compiler via preset
cmake --preset conan-release  
cmake --build --preset conan-release  

# Étape 4 : tester
ctest --preset conan-release
```

Plus besoin de spécifier manuellement le toolchain, le générateur ou le répertoire de build — tout est encapsulé dans le preset.

### Presets multiples (Debug + Release)

Pour gérer plusieurs configurations, lancez `conan install` une fois par profil :

```bash
# Debug
conan install . -pr=conan/profiles/gcc-debug --build=missing

# Release
conan install . -pr=conan/profiles/gcc-release --build=missing

# Les deux presets sont maintenant disponibles
cmake --list-presets
# Available configure presets:
#   "conan-debug"   - 'conan-debug' config
#   "conan-release" - 'conan-release' config

# Configurer et compiler en Debug
cmake --preset conan-debug  
cmake --build --preset conan-debug  

# Configurer et compiler en Release
cmake --preset conan-release  
cmake --build --preset conan-release  
```

### Cohabitation avec vos propres presets

Si votre projet définit déjà un `CMakePresets.json` (section 27.6), le `CMakeUserPresets.json` généré par Conan le complète sans le remplacer. CMake fusionne les deux fichiers : les presets de votre `CMakePresets.json` et ceux du `CMakeUserPresets.json` sont tous disponibles.

La convention est la suivante :

- `CMakePresets.json` : versionné dans Git, contient vos presets de projet.
- `CMakeUserPresets.json` : généré par Conan, listé dans `.gitignore`, contient les presets spécifiques à la configuration locale.

```gitignore
# .gitignore
CMakeUserPresets.json  
build/  
```

Vous pouvez aussi créer un preset dans votre `CMakePresets.json` qui **hérite** du preset Conan, en y ajoutant vos propres variables :

```json
{
    "version": 6,
    "configurePresets": [
        {
            "name": "dev-release",
            "inherits": "conan-release",
            "cacheVariables": {
                "MY_PROJECT_BUILD_TESTS": "ON",
                "MY_PROJECT_BUILD_BENCHMARKS": "OFF"
            }
        }
    ]
}
```

```bash
cmake --preset dev-release    # Hérite du toolchain Conan + vos variables
```

## Ce que `conan_toolchain.cmake` configure

Le fichier `conan_toolchain.cmake` généré par `CMakeToolchain` configure une dizaine de variables CMake. Comprendre son contenu aide au diagnostic :

```cmake
# Extrait simplifié de conan_toolchain.cmake (généré, ne pas modifier)

# Standard C++ (correspond au setting compiler.cppstd du profil)
set(CMAKE_CXX_STANDARD 23)  
set(CMAKE_CXX_STANDARD_REQUIRED ON)  
set(CMAKE_CXX_EXTENSIONS OFF)  

# Type de build (correspond au setting build_type)
set(CMAKE_BUILD_TYPE "Release")

# Chemin de recherche pour find_package() — pointe vers les fichiers CMakeDeps
list(PREPEND CMAKE_PREFIX_PATH "${CMAKE_CURRENT_LIST_DIR}")

# Compilateur (si spécifié dans [conf] tools.build:compiler_executables)
set(CMAKE_C_COMPILER "gcc-15")  
set(CMAKE_CXX_COMPILER "g++-15")  

# Variables de cache injectées via generate()
set(MY_PROJECT_BUILD_TESTS ON CACHE BOOL "")
```

Le point clé est la ligne `CMAKE_PREFIX_PATH` : elle ajoute le répertoire `generators/` au chemin de recherche de CMake, de sorte que vos `find_package()` trouvent les fichiers Config générés par Conan. C'est le pont entre Conan et CMake — simple et transparent.

### Conséquence sur les CMakeLists.txt

Vos `CMakeLists.txt` n'ont **aucune connaissance de Conan**. Ils utilisent les mêmes commandes que si les dépendances étaient installées par `apt`, vcpkg, ou compilées manuellement :

```cmake
# CMakeLists.txt racine — aucune mention de Conan
cmake_minimum_required(VERSION 3.25)  
project(my_project VERSION 1.2.0 LANGUAGES CXX)  

set(CMAKE_CXX_STANDARD 23)  
set(CMAKE_CXX_STANDARD_REQUIRED ON)  
set(CMAKE_CXX_EXTENSIONS OFF)  

find_package(OpenSSL REQUIRED)  
find_package(spdlog REQUIRED)  
find_package(ZLIB REQUIRED)  

add_subdirectory(libs/utils)  
add_subdirectory(src)  
add_subdirectory(apps)  

if(MY_PROJECT_BUILD_TESTS)
    enable_testing()
    find_package(GTest REQUIRED)
    add_subdirectory(tests)
endif()
```

```cmake
# src/CMakeLists.txt — aucune mention de Conan
add_library(my_project_core core.cpp network.cpp)  
add_library(my_project::core ALIAS my_project_core)  

target_include_directories(my_project_core
    PUBLIC
        $<BUILD_INTERFACE:${PROJECT_SOURCE_DIR}/include>
        $<INSTALL_INTERFACE:include>
    PRIVATE
        ${CMAKE_CURRENT_SOURCE_DIR}/internal
)

target_link_libraries(my_project_core
    PUBLIC  my_project::utils Threads::Threads
    PRIVATE OpenSSL::SSL OpenSSL::Crypto spdlog::spdlog ZLIB::ZLIB
)
```

C'est le découplage fondamental : le `CMakeLists.txt` décrit **ce que** le projet consomme, le `conanfile.py` décrit **d'où** viennent les dépendances. Les deux sont indépendants.

## Cohabitation Conan + FetchContent

Dans un projet réel, certaines dépendances sont mieux gérées par Conan (bibliothèques lourdes, dépendances système) et d'autres par FetchContent (bibliothèques légères, header-only, projets en développement actif). Les deux mécanismes coexistent naturellement.

### Stratégie recommandée

| Type de dépendance | Outil | Raison |
|-------------------|-------|--------|
| Lourdes à compiler (OpenSSL, Boost, Protobuf, gRPC) | **Conan** | Cache binaire — évite la recompilation |
| Système (Threads, ZLIB, CURL) | **Conan** ou `apt` | Conan gère les versions ; `apt` suffit pour les standards |
| Framework de test (Google Test, Catch2) | **Conan** ou FetchContent | Les deux fonctionnent ; Conan est plus rapide en cache |
| Header-only légères (nlohmann/json, CLI11) | **FetchContent** | Pas de compilation, FetchContent est trivial |
| Dépendances en développement actif (votre propre lib) | **FetchContent** ou `add_subdirectory` | Modifications fréquentes, pas de versioning stable |

### Exemple hybride

```python
# conanfile.py — Conan gère les dépendances lourdes
class MyProjectRecipe(ConanFile):
    settings = "os", "compiler", "build_type", "arch"
    generators = "CMakeToolchain", "CMakeDeps"

    def requirements(self):
        self.requires("openssl/3.2.1")
        self.requires("spdlog/1.15.3")
        self.requires("zlib/1.3.1")

    def build_requirements(self):
        self.test_requires("gtest/1.17.0")

    def layout(self):
        cmake_layout(self)
```

```cmake
# CMakeLists.txt — FetchContent gère les dépendances légères
cmake_minimum_required(VERSION 3.25)  
project(my_project VERSION 1.2.0 LANGUAGES CXX)  

# Dépendances Conan (trouvées via le toolchain Conan)
find_package(OpenSSL REQUIRED)  
find_package(spdlog REQUIRED)  
find_package(ZLIB REQUIRED)  

# Dépendances FetchContent (téléchargées et intégrées)
include(FetchContent)  
FetchContent_Declare(nlohmann_json  
    GIT_REPOSITORY https://github.com/nlohmann/json.git
    GIT_TAG        v3.11.3
)
FetchContent_Declare(cli11
    GIT_REPOSITORY https://github.com/CLIUtils/CLI11.git
    GIT_TAG        v2.6.0
)
FetchContent_MakeAvailable(nlohmann_json cli11)

add_subdirectory(src)  
add_subdirectory(apps)  
```

Le workflow reste le même — `conan install` puis `cmake --preset` :

```bash
conan install . -pr=conan/profiles/gcc-release --build=missing  
cmake --preset conan-release  
cmake --build --preset conan-release  
```

CMake trouve OpenSSL, spdlog et ZLIB via les fichiers Conan, et nlohmann/json et CLI11 via FetchContent. Les deux mécanismes coexistent sans conflit.

### Éviter les doublons

Un risque de la cohabitation est de déclarer une même dépendance dans les deux systèmes. Si `spdlog` est à la fois dans le `conanfile.py` et dans un `FetchContent_Declare`, vous obtiendrez un conflit de cibles CMake.

La règle est simple : **chaque dépendance est gérée par un seul outil**. Si elle est dans Conan, elle ne doit pas être dans FetchContent, et inversement. Le pattern `find_package()` + `FetchContent` comme fallback (section 26.3.2) reste utilisable mais ne devrait pas être nécessaire quand Conan est en place — Conan fournit toutes les bibliothèques qu'il déclare.

## Diagnostic : quand ça ne fonctionne pas

### `find_package()` ne trouve pas la dépendance

**Symptôme** : `Could not find a package configuration file provided by "spdlog"`

**Causes probables** :

1. `conan install` n'a pas été lancé (ou a échoué).
2. Le toolchain Conan n'est pas chargé — vérifiez `-DCMAKE_TOOLCHAIN_FILE=...` ou le preset.
3. Le nom attendu par `find_package()` ne correspond pas au nom généré par Conan.

**Diagnostic** :

```bash
# Vérifier que les fichiers ont été générés
ls build/Release/generators/*Config.cmake

# Vérifier que le toolchain est chargé (mode verbose CMake)
cmake --preset conan-release --log-level=VERBOSE 2>&1 | grep "CMAKE_PREFIX_PATH"
```

### Erreur de linkage : symboles non trouvés

**Symptôme** : `undefined reference to 'SSL_new'`

**Causes probables** :

1. La dépendance est trouvée par `find_package()` mais n'est pas dans `target_link_libraries()`.
2. Conflit de mode de linkage — la dépendance est compilée en statique mais votre code attend la version dynamique (ou inversement).
3. Les settings du profil Conan ne correspondent pas à ceux de votre compilation CMake.

**Diagnostic** :

```bash
# Vérifier les settings utilisés par Conan
conan install . -pr=conan/profiles/gcc-release --build=missing 2>&1 | grep "settings"

# Vérifier les options de linkage
conan list "openssl/3.2.1:*" | grep shared
```

### Incohérence entre profil Conan et CMakeLists.txt

Le toolchain Conan définit `CMAKE_CXX_STANDARD`, `CMAKE_BUILD_TYPE`, etc. Si votre `CMakeLists.txt` les redéfinit avec des valeurs différentes, les settings effectifs seront ceux de CMake (le dernier `set()` gagne), mais les dépendances Conan auront été compilées avec les settings du profil. Cette incohérence peut provoquer des erreurs de linkage ou des violations de l'ODR.

**Solution** : soit vous laissez le toolchain Conan tout configurer (retirez les `set(CMAKE_CXX_STANDARD ...)` de votre CMakeLists.txt), soit vous vous assurez que les valeurs sont identiques. La première approche est plus propre — le profil Conan est la source de vérité unique pour les paramètres de compilation.

```cmake
# Option A : laisser Conan configurer (via conan_toolchain.cmake)
cmake_minimum_required(VERSION 3.25)  
project(my_project VERSION 1.2.0 LANGUAGES CXX)  
# Pas de set(CMAKE_CXX_STANDARD ...) — le toolchain Conan s'en charge

# Option B : configurer explicitement (valeurs identiques au profil)
cmake_minimum_required(VERSION 3.25)  
project(my_project VERSION 1.2.0 LANGUAGES CXX)  
set(CMAKE_CXX_STANDARD 23)  
set(CMAKE_CXX_STANDARD_REQUIRED ON)  
set(CMAKE_CXX_EXTENSIONS OFF)  
# Doit correspondre exactement à compiler.cppstd=23 du profil
```

L'option A est recommandée quand Conan est l'outil principal. L'option B est nécessaire quand le projet doit aussi fonctionner sans Conan (avec les dépendances installées manuellement).

## Résumé du workflow recommandé

```bash
# 1. Installer les dépendances (une fois, ou quand conanfile.py change)
conan install . -pr:h=conan/profiles/gcc-release \
                -pr:b=conan/profiles/gcc-release \
                --build=missing

# 2. Configurer (via preset Conan)
cmake --preset conan-release

# 3. Compiler
cmake --build --preset conan-release

# 4. Tester
ctest --preset conan-release

# --- Pour basculer en Debug ---

# 1. Installer les dépendances Debug
conan install . -pr:h=conan/profiles/gcc-debug \
                -pr:b=conan/profiles/gcc-debug \
                --build=missing

# 2-4. Même workflow
cmake --preset conan-debug  
cmake --build --preset conan-debug  
ctest --preset conan-debug  
```

Quatre commandes par configuration. Les `CMakeLists.txt` ne mentionnent jamais Conan. Les profils sont versionnés avec le projet. Les presets sont générés automatiquement. Le build est reproductible sur n'importe quelle machine.
