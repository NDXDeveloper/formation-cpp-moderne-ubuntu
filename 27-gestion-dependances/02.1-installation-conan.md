# 27.2.1 Installation et configuration

> **Objectif** : Installer Conan 2.0 sur Ubuntu, générer le profil par défaut, comprendre la structure du cache local, et vérifier que tout fonctionne avec une première installation de dépendance.

---

## Prérequis

Conan est une application Python. Vous avez besoin de :

- **Python 3.8+** (Python 3.10+ recommandé en 2026)
- **pip** (le gestionnaire de paquets Python)

Ubuntu 24.04 LTS fournit Python 3.12 par défaut. Vérifiez votre installation :

```bash
python3 --version
# Python 3.12.x

pip3 --version
# pip 24.x.x from /usr/lib/python3/dist-packages/pip (python 3.12)
```

Si `pip3` n'est pas disponible :

```bash
sudo apt install python3-pip
```

---

## Installation de Conan

### Méthode recommandée : pipx

Sur les distributions récentes (Ubuntu 23.04+), `pip install` en mode global est bloqué par la politique `externally-managed-environment`. La méthode recommandée est **pipx**, qui installe Conan dans un environnement virtuel isolé tout en rendant la commande `conan` disponible globalement :

```bash
# Installer pipx
sudo apt install pipx  
pipx ensurepath  

# Installer Conan
pipx install conan

# Vérifier l'installation
conan --version
# Conan version 2.26.2
```

`pipx ensurepath` ajoute le répertoire des binaires pipx à votre `PATH`. Vous devrez peut-être fermer et rouvrir votre terminal (ou exécuter `source ~/.bashrc`) pour que le changement prenne effet.

### Méthode alternative : pip avec --user

Si pipx n'est pas disponible ou si vous préférez pip :

```bash
pip3 install --user conan

# Vérifier que ~/.local/bin est dans le PATH
export PATH="$HOME/.local/bin:$PATH"  # Ajouter à ~/.bashrc si nécessaire

conan --version
```

### Méthode alternative : virtualenv dédié

Pour isoler complètement Conan du système Python :

```bash
python3 -m venv ~/.conan-venv  
source ~/.conan-venv/bin/activate  
pip install conan  
conan --version  
```

L'inconvénient est que vous devez activer le virtualenv à chaque session. Un alias dans `~/.bashrc` peut automatiser cela :

```bash
# ~/.bashrc
alias conan='~/.conan-venv/bin/conan'
```

### Mise à jour

Quelle que soit la méthode d'installation :

```bash
# Avec pipx
pipx upgrade conan

# Avec pip
pip3 install --user --upgrade conan
```

## Première configuration : le profil par défaut

Après l'installation, la première étape est de générer le **profil par défaut**. Ce profil décrit votre environnement de compilation et sera utilisé pour toutes les installations de dépendances :

```bash
conan profile detect
```

Conan inspecte votre système et produit un profil dans `~/.conan2/profiles/default` :

```bash
cat ~/.conan2/profiles/default
```

Le résultat ressemble à :

```ini
[settings]
arch=x86_64  
build_type=Release  
compiler=gcc  
compiler.cppstd=gnu17  
compiler.libcxx=libstdc++11  
compiler.version=15  
os=Linux  
```

Conan a détecté votre architecture (x86_64), votre système d'exploitation (Linux), votre compilateur (GCC 15), et a défini des valeurs par défaut pour le standard C++ (gnu17) et le type de build (Release).

### Ajuster le profil

Le profil auto-détecté est un point de départ, mais il mérite souvent des ajustements. Ouvrez-le dans un éditeur :

```bash
nano ~/.conan2/profiles/default
```

**Changer le standard C++** : si votre projet utilise C++23, modifiez `compiler.cppstd` :

```ini
compiler.cppstd=23
```

Notez la différence entre `23` et `gnu23`. La valeur `23` correspond au flag `-std=c++23` (mode strict, sans extensions GNU). La valeur `gnu23` correspond à `-std=gnu++23` (avec extensions). En cohérence avec les recommandations du chapitre 26 (`CMAKE_CXX_EXTENSIONS OFF`), préférez `23`.

**Changer le type de build par défaut** : pour le développement quotidien en Debug :

```ini
build_type=Debug
```

**Ajouter des variables d'environnement ou des options de compilation** (avancé) :

```ini
[settings]
arch=x86_64  
build_type=Debug  
compiler=gcc  
compiler.cppstd=23  
compiler.libcxx=libstdc++11  
compiler.version=15  
os=Linux  

[conf]
tools.cmake.cmaketoolchain:generator=Ninja  
tools.build:compiler_executables={"c": "gcc-15", "cpp": "g++-15"}  
```

La section `[conf]` permet de configurer des outils et des comportements. Ici, nous demandons à Conan de générer un toolchain qui utilise Ninja comme générateur CMake et des exécutables de compilateur spécifiques.

### Profils multiples

Vous pouvez créer autant de profils que nécessaire. Chaque profil est un fichier texte dans `~/.conan2/profiles/` :

```bash
# Créer un profil Debug
cp ~/.conan2/profiles/default ~/.conan2/profiles/debug  
nano ~/.conan2/profiles/debug  
# → Modifier build_type=Debug

# Créer un profil Clang
cp ~/.conan2/profiles/default ~/.conan2/profiles/clang  
nano ~/.conan2/profiles/clang  
# → Modifier compiler=clang, compiler.version=20

# Créer un profil cross-compilation ARM64
nano ~/.conan2/profiles/arm64
```

Exemple de profil Clang :

```ini
[settings]
arch=x86_64  
build_type=Release  
compiler=clang  
compiler.cppstd=23  
compiler.libcxx=libstdc++11  
compiler.version=20  
os=Linux  

[conf]
tools.build:compiler_executables={"c": "clang", "cpp": "clang++"}
```

Exemple de profil cross-compilation ARM64 :

```ini
[settings]
arch=armv8  
build_type=Release  
compiler=gcc  
compiler.cppstd=23  
compiler.libcxx=libstdc++11  
compiler.version=14  
os=Linux  

[conf]
tools.build:compiler_executables={"c": "aarch64-linux-gnu-gcc", "cpp": "aarch64-linux-gnu-g++"}
```

L'utilisation de ces profils est couverte en section 27.2.3.

## Structure du cache local

Conan stocke toutes ses données dans `~/.conan2/`. Comprendre cette structure aide au diagnostic :

```
~/.conan2/
├── global.conf             # Configuration globale de Conan
├── profiles/               # Profils de compilation
│   ├── default
│   ├── debug
│   └── clang
├── extensions/             # Plugins et extensions
├── p/                      # Cache des paquets (artefacts binaires)
│   ├── openssba1c2d3e/     # Paquet OpenSSL (hash abrégé)
│   │   ├── e/              # Export (recette)
│   │   └── p/              # Package (binaires)
│   ├── spdlof4e5g6h/       # Paquet spdlog
│   └── ...
└── remotes.json            # Liste des serveurs distants
```

Le répertoire `p/` est le **cache de paquets**. Chaque dépendance installée y est stockée avec ses headers, bibliothèques compilées, et métadonnées. Les noms de répertoires utilisent des hashs abrégés pour l'unicité.

### Commandes utiles pour le cache

```bash
# Lister les paquets dans le cache
conan list "*"

# Lister les binaires d'un paquet spécifique
conan list "openssl/3.2.1:*"

# Supprimer un paquet du cache
conan remove "spdlog/1.15.3" -c

# Vider entièrement le cache (attention !)
conan remove "*" -c
```

L'option `-c` confirme la suppression sans demander d'interaction.

### Réinitialiser Conan

En cas de problème de cache ou de configuration corrompue, la solution nucléaire est de supprimer le répertoire complet :

```bash
rm -rf ~/.conan2  
conan profile detect   # Régénérer le profil  
```

C'est sans danger — vous perdez les paquets en cache (qui seront retéléchargés au besoin) et les profils personnalisés (qu'il faudra recréer).

## Les remotes : les serveurs de paquets

Conan télécharge les paquets depuis des **remotes** — des serveurs qui hébergent les recettes et les binaires. Le remote par défaut est **ConanCenter** :

```bash
conan remote list
# conancenter: https://center2.conan.io [Verify SSL: True, Enabled: True]
```

ConanCenter est suffisant pour la plupart des usages. Mais les organisations peuvent ajouter des serveurs privés :

```bash
# Ajouter un remote privé (Artifactory, Conan Server, etc.)
conan remote add company-server https://conan.company.com/artifactory/api/conan/cpp-libs

# Définir la priorité (le premier remote est consulté en premier)
conan remote add company-server https://conan.company.com/... --index 0
```

Lors d'un `conan install`, Conan cherche les binaires dans les remotes par ordre de priorité : d'abord le premier remote, puis le suivant, et ainsi de suite. Le cache local est consulté avant tout remote.

## Vérification : première installation de dépendance

Testons l'installation avec un scénario minimal. Créez un répertoire temporaire avec un `conanfile.txt` simple :

```bash
mkdir /tmp/conan-test && cd /tmp/conan-test
```

```ini
# conanfile.txt
[requires]
fmt/11.1.4

[generators]
CMakeToolchain  
CMakeDeps  

[layout]
cmake_layout
```

Lancez l'installation :

```bash
conan install . --build=missing
```

Si tout est correctement configuré, vous verrez Conan :

1. Résoudre les dépendances (fmt et ses dépendances transitives éventuelles)
2. Chercher un binaire pré-compilé sur ConanCenter pour votre configuration
3. Télécharger le binaire (ou le compiler depuis les sources si `--build=missing` est spécifié et qu'aucun binaire n'est disponible)
4. Générer les fichiers CMake dans le répertoire de build

La sortie se termine par quelque chose comme :

```
Install finished successfully
```

Vous pouvez vérifier les fichiers générés :

```bash
ls build/Release/generators/
# conan_toolchain.cmake
# fmt-config.cmake
# fmt-Targets.cmake
# ...
```

Le fichier `conan_toolchain.cmake` est celui que vous passerez à CMake via `-DCMAKE_TOOLCHAIN_FILE`. Les fichiers `fmt-config.cmake` et `fmt-Targets.cmake` sont ceux que `find_package(fmt)` trouvera.

Nettoyez le répertoire temporaire :

```bash
rm -rf /tmp/conan-test
```

## Configuration globale

Le fichier `~/.conan2/global.conf` permet de définir des configurations qui s'appliquent à toutes les invocations de Conan, sans avoir à les répéter dans chaque profil ou en ligne de commande :

```ini
# ~/.conan2/global.conf

# Toujours utiliser Ninja comme générateur CMake
tools.cmake.cmaketoolchain:generator=Ninja

# Paralléliser la compilation des dépendances
tools.build:jobs={{os.cpu_count()}}

# Répertoire de téléchargement personnalisé (utile en CI)
# core.download:download_cache=/shared/conan-download-cache
```

La variable `tools.build:jobs` utilise une expression Python pour détecter le nombre de cœurs CPU disponibles. Les dépendances compilées depuis les sources (`--build=missing`) bénéficieront automatiquement de la parallélisation.
