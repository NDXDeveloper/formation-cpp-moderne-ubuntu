# 27.2.2 `conanfile.py` vs `conanfile.txt`

> **Objectif** : Comprendre les deux formats de déclaration de dépendances Conan — le `conanfile.txt` simplifié et le `conanfile.py` complet — savoir quand utiliser chacun, et maîtriser les méthodes essentielles du `conanfile.py` pour un projet CMake professionnel.

---

## Deux formats, un seul rôle

Le fichier conan — `conanfile.txt` ou `conanfile.py` — se place à la **racine** de votre projet (à côté du `CMakeLists.txt` principal). Il remplit trois fonctions :

1. **Déclarer les dépendances** : quelles bibliothèques sont nécessaires, à quelles versions.
2. **Configurer les générateurs** : quels fichiers CMake produire pour l'intégration.
3. **Organiser le layout** : où placer les fichiers générés par rapport au répertoire de build.

Les deux formats produisent le même résultat. La différence réside dans l'expressivité : `conanfile.txt` est un format déclaratif limité, `conanfile.py` est un script Python offrant un contrôle total.

---

## `conanfile.txt` : le format simple

Le format texte est adapté aux projets simples avec des dépendances directes, sans logique conditionnelle.

```ini
# conanfile.txt
[requires]
openssl/3.2.1  
spdlog/1.15.3  
zlib/1.3.1  

[test_requires]
gtest/1.17.0

[generators]
CMakeToolchain  
CMakeDeps  

[layout]
cmake_layout

[options]
spdlog/*:shared=False  
openssl/*:shared=False  
```

### Sections du `conanfile.txt`

| Section | Rôle |
|---------|------|
| `[requires]` | Dépendances de production (une par ligne, format `nom/version`) |
| `[test_requires]` | Dépendances de test uniquement (non propagées aux consommateurs) |
| `[tool_requires]` | Outils de build (protoc, flatc, etc.) — exécutés sur la machine de build |
| `[generators]` | Générateurs CMake à activer |
| `[layout]` | Organisation des répertoires de build |
| `[options]` | Options spécifiques aux paquets (format `paquet/*:option=valeur`) |

### Limites du format texte

Le `conanfile.txt` ne permet pas :

- de conditionner des dépendances selon le système d'exploitation ou le profil ;
- de définir des options par défaut pour votre propre paquet ;
- d'exécuter de la logique avant ou après l'installation ;
- de personnaliser la configuration CMake au-delà des générateurs standard ;
- de publier votre projet comme un paquet Conan consommable par d'autres.

Dès qu'un de ces besoins apparaît, il faut passer au `conanfile.py`.

---

## `conanfile.py` : le format recommandé

Le `conanfile.py` est un script Python qui hérite de la classe `ConanFile`. Il offre une expressivité complète tout en restant lisible pour les cas simples. C'est le format recommandé par Conan pour tous les projets, y compris les plus simples — la migration de `.txt` vers `.py` est inévitable dès que le projet grandit.

### Équivalent minimal du `conanfile.txt` ci-dessus

```python
from conan import ConanFile  
from conan.tools.cmake import cmake_layout  


class MyProjectRecipe(ConanFile):
    settings = "os", "compiler", "build_type", "arch"
    generators = "CMakeToolchain", "CMakeDeps"

    def requirements(self):
        self.requires("openssl/3.2.1")
        self.requires("spdlog/1.15.3")
        self.requires("zlib/1.3.1")

    def build_requirements(self):
        self.test_requires("gtest/1.17.0")

    def layout(self):
        cmake_layout(self)
```

Ce fichier fait exactement la même chose que le `conanfile.txt` précédent. Détaillons chaque élément.

## Anatomie du `conanfile.py`

### L'attribut `settings`

```python
settings = "os", "compiler", "build_type", "arch"
```

Cet attribut déclare quels **settings** du profil sont pertinents pour votre projet. Conan utilise ces settings pour calculer le **package ID** de chaque dépendance — le hash qui identifie une variante binaire spécifique.

Les quatre settings listés ici sont les plus courants et couvrent la grande majorité des cas. Ils correspondent aux paramètres de la matrice de configurations (section 27.1). Si votre projet n'est pas sensible à l'un d'eux (par exemple une bibliothèque header-only n'est pas sensible à `build_type`), vous pouvez le retirer.

### L'attribut `generators`

```python
generators = "CMakeToolchain", "CMakeDeps"
```

Déclare les générateurs qui produisent les fichiers d'intégration. Pour un projet CMake, vous utiliserez toujours ce couple :

- **`CMakeToolchain`** : génère `conan_toolchain.cmake` (compilateur, standard, flags, type de build).
- **`CMakeDeps`** : génère les fichiers `*-config.cmake` et `*-Targets.cmake` pour chaque dépendance.

L'alternative est de déclarer les générateurs dans la méthode `generate()` plutôt que comme attribut. Les deux approches sont équivalentes pour le cas standard :

```python
# Équivalent via generate()
from conan.tools.cmake import CMakeToolchain, CMakeDeps

def generate(self):
    tc = CMakeToolchain(self)
    tc.generate()
    deps = CMakeDeps(self)
    deps.generate()
```

La méthode `generate()` devient nécessaire quand vous voulez personnaliser les générateurs (nous y reviendrons).

### La méthode `requirements()`

```python
def requirements(self):
    self.requires("openssl/3.2.1")
    self.requires("spdlog/1.15.3")
    self.requires("zlib/1.3.1")
```

C'est ici que vous déclarez les dépendances de votre projet. Chaque appel à `self.requires()` ajoute une dépendance avec sa version exacte.

#### Plages de versions

Conan supporte les plages de versions, bien que les versions exactes soient recommandées pour la reproductibilité :

```python
self.requires("spdlog/[>=1.14 <2.0]")   # Toute version 1.x ≥ 1.14  
self.requires("zlib/[~1.3]")             # Toute version 1.3.x  
self.requires("openssl/3.2.1")           # Version exacte (recommandé)  
```

#### Dépendances transitives et `override`

Si deux de vos dépendances requièrent la même bibliothèque transitive à des versions différentes, Conan tente de résoudre le conflit. Vous pouvez forcer une version spécifique avec `override` :

```python
def requirements(self):
    self.requires("my_lib_a/1.0")     # Dépend de zlib/1.3.0
    self.requires("my_lib_b/2.0")     # Dépend de zlib/1.3.1
    self.requires("zlib/1.3.1", override=True)  # Forcer zlib 1.3.1 pour tous
```

#### Dépendances optionnelles

```python
def requirements(self):
    self.requires("openssl/3.2.1")
    if self.options.get_safe("with_compression", default=True):
        self.requires("zlib/1.3.1")
```

Ce pattern lie la présence d'une dépendance à une option du projet — nous reviendrons sur les options plus loin.

### La méthode `build_requirements()`

```python
def build_requirements(self):
    self.test_requires("gtest/1.17.0")
```

Cette méthode déclare les dépendances qui ne font pas partie du produit final :

- **`self.test_requires()`** : dépendances de test (Google Test, Catch2, Google Benchmark). Elles ne sont pas propagées aux consommateurs de votre bibliothèque.
- **`self.tool_requires()`** : outils de build nécessaires à la compilation (protobuf compiler, flatbuffers compiler, cmake lui-même). Ils s'exécutent sur la machine de build, pas sur la cible.

```python
def build_requirements(self):
    self.test_requires("gtest/1.17.0")
    self.test_requires("benchmark/1.9.5")
    self.tool_requires("protobuf/5.27.0")  # Pour protoc
```

### La méthode `layout()`

```python
from conan.tools.cmake import cmake_layout

def layout(self):
    cmake_layout(self)
```

La méthode `layout()` indique à Conan comment votre projet est organisé et où placer les fichiers générés. `cmake_layout()` est le helper standard qui configure :

- le répertoire source (racine du projet) ;
- le répertoire de build (`build/Release`, `build/Debug`, etc.) ;
- l'emplacement des fichiers générés (`build/Release/generators/`).

Avec `cmake_layout`, la commande `conan install` produit les fichiers au bon endroit pour que CMake les trouve automatiquement :

```bash
conan install . --build=missing  
cmake --preset conan-release   # Le preset est généré automatiquement par Conan  
cmake --build build/Release  
```

`cmake_layout` génère aussi des **CMake Presets** (fichiers `CMakeUserPresets.json`) qui s'intègrent avec les presets de votre projet (section 27.6).

## Options : configurer les dépendances

Les paquets Conan exposent des **options** qui contrôlent leur comportement de build. L'option la plus universelle est `shared` (linkage statique ou dynamique), mais chaque paquet peut définir ses propres options.

### Définir les options dans `default_options`

```python
class MyProjectRecipe(ConanFile):
    settings = "os", "compiler", "build_type", "arch"
    generators = "CMakeToolchain", "CMakeDeps"

    default_options = {
        "openssl/*:shared": False,
        "spdlog/*:shared": False,
        "zlib/*:shared": False,
        "spdlog/*:header_only": False,
    }

    def requirements(self):
        self.requires("openssl/3.2.1")
        self.requires("spdlog/1.15.3")
        self.requires("zlib/1.3.1")

    def build_requirements(self):
        self.test_requires("gtest/1.17.0")

    def layout(self):
        cmake_layout(self)
```

Le dictionnaire `default_options` utilise le format `"paquet/*:option": valeur`. Le wildcard `*` s'applique à toutes les versions du paquet.

### Forcer le linkage statique global

Pour forcer toutes les dépendances en linkage statique d'un seul coup :

```python
default_options = {
    "*:shared": False,
}
```

Le wildcard `*` sans nom de paquet s'applique à **toutes** les dépendances. C'est la méthode la plus concise pour un déploiement statique.

### Définir ses propres options

Si votre projet est lui-même destiné à être consommé comme un paquet Conan, vous pouvez définir des options propres :

```python
class MyProjectRecipe(ConanFile):
    name = "my_project"
    version = "1.2.0"

    options = {
        "shared": [True, False],
        "with_compression": [True, False],
        "with_ssl": [True, False],
    }
    default_options = {
        "shared": False,
        "with_compression": True,
        "with_ssl": True,
    }

    def requirements(self):
        self.requires("openssl/3.2.1")
        if self.options.with_compression:
            self.requires("zlib/1.3.1")
```

Les consommateurs de votre paquet peuvent alors choisir :

```bash
conan install . -o my_project/*:with_compression=False
```

### Découvrir les options d'un paquet

```bash
# Voir les options disponibles pour un paquet
conan inspect openssl/3.2.1
```

La sortie liste les options du paquet avec leurs valeurs par défaut et leurs valeurs possibles.

## Personnaliser la génération avec `generate()`

Quand l'attribut `generators` ne suffit pas, la méthode `generate()` offre un contrôle fin sur les fichiers produits :

```python
from conan import ConanFile  
from conan.tools.cmake import cmake_layout, CMakeToolchain, CMakeDeps  


class MyProjectRecipe(ConanFile):
    settings = "os", "compiler", "build_type", "arch"

    def requirements(self):
        self.requires("openssl/3.2.1")
        self.requires("spdlog/1.15.3")
        self.requires("zlib/1.3.1")

    def build_requirements(self):
        self.test_requires("gtest/1.17.0")

    def layout(self):
        cmake_layout(self)

    def generate(self):
        # Personnaliser le toolchain
        tc = CMakeToolchain(self)
        tc.cache_variables["MY_PROJECT_BUILD_TESTS"] = True
        tc.cache_variables["MY_PROJECT_USE_SSL"] = bool(self.dependencies["openssl"])
        tc.generate()

        # Personnaliser les dépendances CMake
        deps = CMakeDeps(self)
        # Renommer un paquet si son nom CMake diffère du nom Conan
        deps.set_property("spdlog", "cmake_file_name", "spdlog")
        deps.set_property("spdlog", "cmake_target_name", "spdlog::spdlog")
        deps.generate()
```

`CMakeToolchain` permet d'injecter des **variables CMake** dans le toolchain. Ces variables seront automatiquement définies quand CMake charge `conan_toolchain.cmake`, comme si vous aviez passé `-DMY_PROJECT_BUILD_TESTS=ON` en ligne de commande.

`CMakeDeps` permet de contrôler les noms des fichiers et des cibles CMake générés. C'est utile quand le nom d'un paquet Conan ne correspond pas au nom attendu par `find_package()` dans votre `CMakeLists.txt`. Par exemple, si votre CMake fait `find_package(ZLIB)` mais que le paquet Conan s'appelle `zlib`, Conan gère cette correspondance automatiquement pour les paquets courants — mais `set_property()` permet d'ajuster les cas non standards.

## Exemple complet : le `conanfile.py` de notre projet

Voici le `conanfile.py` complet pour le projet exemple que nous construisons depuis le chapitre 26, intégrant toutes les bonnes pratiques :

```python
from conan import ConanFile  
from conan.tools.cmake import cmake_layout, CMakeToolchain, CMakeDeps  


class MyProjectRecipe(ConanFile):
    settings = "os", "compiler", "build_type", "arch"

    # Options des dépendances
    default_options = {
        "*:shared": False,          # Tout en statique par défaut
        "spdlog/*:header_only": False,
    }

    def requirements(self):
        self.requires("openssl/3.2.1")
        self.requires("spdlog/1.15.3")
        self.requires("zlib/1.3.1")

    def build_requirements(self):
        self.test_requires("gtest/1.17.0")

    def layout(self):
        cmake_layout(self)

    def generate(self):
        tc = CMakeToolchain(self)
        tc.cache_variables["MY_PROJECT_BUILD_TESTS"] = True
        tc.generate()

        deps = CMakeDeps(self)
        deps.generate()
```

Ce fichier se place à la racine du projet, à côté du `CMakeLists.txt`. Le `CMakeLists.txt` n'est **pas modifié** — les mêmes `find_package()` et `target_link_libraries()` fonctionnent :

```cmake
# CMakeLists.txt — inchangé par rapport au chapitre 26
find_package(OpenSSL REQUIRED)  
find_package(spdlog REQUIRED)  
find_package(ZLIB REQUIRED)  

target_link_libraries(my_project_core
    PRIVATE OpenSSL::SSL OpenSSL::Crypto spdlog::spdlog ZLIB::ZLIB
)
```

## Quand utiliser quel format ?

| Critère | `conanfile.txt` | `conanfile.py` |
|---------|:--------------:|:--------------:|
| Projet simple, aucune logique | ✅ Suffisant | ✅ Fonctionne aussi |
| Dépendances conditionnelles (OS, options) | ❌ | ✅ |
| Personnalisation des générateurs | ❌ | ✅ |
| Injection de variables CMake | ❌ | ✅ |
| Publication comme paquet Conan | ❌ | ✅ Obligatoire |
| Migration vers `conanfile.py` inévitable ? | Oui, dès que le projet grandit | — |

**Recommandation** : commencez directement avec `conanfile.py`. Le surcoût d'écriture est minime (quelques lignes de Python en plus), et vous n'aurez jamais à migrer. Le `conanfile.txt` reste utile pour les exemples rapides et les prototypes jetables.
