# 30.1.2 — Lecture des rapports Valgrind

## Introduction

Savoir lancer Valgrind est une chose ; savoir exploiter ses rapports en est une autre. Sur un projet réel — plusieurs dizaines de milliers de lignes de code, des librairies tierces, des templates imbriqués — un rapport Memcheck peut contenir des dizaines d'erreurs, des piles d'appels de 20 niveaux traversant la librairie standard, et des messages qui semblent cryptiques au premier abord. La compétence clé n'est pas de générer le rapport, mais de le *lire* efficacement : identifier l'erreur racine, ignorer le bruit, et remonter du symptôme signalé par Valgrind jusqu'à la ligne de code responsable.

Cette section propose une méthodologie structurée pour décoder les rapports Valgrind, illustrée sur des exemples représentatifs de ce qu'on rencontre en conditions réelles.

---

## Structure d'un rapport Memcheck

Un rapport Memcheck complet suit un ordre précis. Comprendre cette structure permet de naviguer rapidement vers l'information pertinente.

### 1. En-tête

```
==78901== Memcheck, a memory error detector
==78901== Copyright (C) 2002-2024, and GNU GPL'd, by Julian Seward et al.
==78901== Using Valgrind-3.23.0 and LibVEX; rerun with -h for copyright info
==78901== Command: ./mon_serveur --port 8080
==78901== Parent PID: 5432
```

L'en-tête confirme la version de Valgrind, la commande exécutée et ses arguments. En cas de doute sur les conditions de reproduction d'un bug, cette information est précieuse pour vérifier qu'on analyse bien le bon binaire avec les bons paramètres.

### 2. Erreurs détectées en cours d'exécution

Les erreurs mémoire (lectures/écritures invalides, utilisation de valeurs non initialisées, désallocations incorrectes) sont émises **au moment où elles se produisent**, intercalées avec la sortie standard du programme. Elles apparaissent dans l'ordre chronologique d'exécution.

### 3. HEAP SUMMARY

Émis à la fin du programme, il donne la photographie finale de l'état du heap.

### 4. Rapports détaillés des fuites

Affichés après le HEAP SUMMARY lorsque `--leak-check=full` est actif. Les blocs fuyants sont regroupés par pile d'appels et triés par taille croissante.

### 5. LEAK SUMMARY

Le résumé final avec les totaux par catégorie.

### 6. ERROR SUMMARY

```
==78901== ERROR SUMMARY: 14 errors from 5 contexts (suppressed: 2 from 2)
```

La dernière ligne du rapport. Elle indique le nombre total d'erreurs détectées, le nombre de *contextes* distincts (c'est-à-dire de piles d'appels uniques), et le nombre d'erreurs supprimées par les fichiers de suppressions.

**Réflexe de lecture** : commencez toujours par le ERROR SUMMARY et le LEAK SUMMARY en fin de rapport. Si les deux affichent zéro, le programme est sain — inutile de relire le reste. Si des erreurs existent, remontez dans le rapport pour les examiner.

## Décoder une erreur mémoire

Les erreurs détectées en cours d'exécution sont les plus critiques : elles signalent un comportement indéfini qui se produit *maintenant*, pas simplement une ressource non libérée. Examinons les formats les plus courants.

### Lecture ou écriture invalide

```
==78901== Invalid read of size 4
==78901==    at 0x109234: traiter_element(int*, int) (traitement.cpp:47)
==78901==    by 0x109312: boucle_principale() (main.cpp:83)
==78901==    by 0x109401: main (main.cpp:102)
==78901==  Address 0x4A3C190 is 0 bytes after a block of size 400 alloc'd
==78901==    at 0x4845013: operator new[](unsigned long) (vg_replace_malloc.c:652)
==78901==    by 0x109200: traiter_element(int*, int) (traitement.cpp:42)
==78901==    by 0x109312: boucle_principale() (main.cpp:83)
==78901==    by 0x109401: main (main.cpp:102)
```

Ce message se décompose en deux blocs distincts :

**Le premier bloc** décrit l'erreur elle-même. `Invalid read of size 4` signifie qu'une lecture de 4 octets (un `int` sur la plupart des architectures) a été tentée à une adresse invalide. La pile d'appels qui suit montre *où* cette lecture a lieu : dans `traiter_element()` à la ligne 47 de `traitement.cpp`.

**Le second bloc** (introduit par `Address 0x4A3C190 is...`) donne le **contexte mémoire** de l'adresse fautive. Ici, Memcheck explique que l'adresse se trouve `0 bytes after a block of size 400` — c'est-à-dire immédiatement après la fin d'un bloc de 400 octets. C'est un accès hors limites classique : un tableau de 100 `int` (400 octets) est accédé à l'index 100. La pile d'appels de ce second bloc montre où le bloc a été alloué.

Les descriptions d'adresses les plus fréquentes sont :

| Message Valgrind | Signification |
|---|---|
| `X bytes after a block of size Y alloc'd` | Accès après la fin d'un bloc alloué (buffer overflow) |
| `X bytes before a block of size Y alloc'd` | Accès avant le début d'un bloc (buffer underflow) |
| `X bytes inside a block of size Y free'd` | Accès dans un bloc déjà libéré (use-after-free) |
| `Address is on thread X's stack` | Accès à une zone de la pile d'un thread |
| `Address is not stack'd, malloc'd or (recently) free'd` | Adresse complètement inconnue (pointeur corrompu ou non initialisé) |

### Utilisation de valeur non initialisée

```
==78901== Conditional jump or move depends on uninitialised value(s)
==78901==    at 0x109180: filtrer(int) (filtre.cpp:12)
==78901==    by 0x1091F0: main (main.cpp:28)
==78901==  Uninitialised value was created by a stack allocation
==78901==    at 0x109160: filtrer(int) (filtre.cpp:8)
```

Ce type d'erreur signifie qu'une branche conditionnelle (`if`, `while`, opérateur ternaire) dépend d'une valeur qui n'a jamais été initialisée. Le second bloc (présent uniquement avec `--track-origins=yes`) indique *où* la valeur non initialisée a été créée — ici, une variable locale dans `filtrer()` à la ligne 8.

Sans `--track-origins=yes`, Memcheck ne fournit que le premier bloc, ce qui rend le diagnostic nettement plus difficile. C'est pourquoi cette option est recommandée systématiquement malgré son surcoût.

### Désallocation incorrecte

```
==78901== Mismatched free() / delete / delete []
==78901==    at 0x484A20F: operator delete(void*) (vg_replace_malloc.c:935)
==78901==    by 0x1091C0: nettoyer() (cleanup.cpp:15)
==78901==    by 0x109250: main (main.cpp:40)
==78901==  Address 0x4A3C040 is 0 bytes inside a block of size 80 alloc'd
==78901==    at 0x4845013: operator new[](unsigned long) (vg_replace_malloc.c:652)
==78901==    by 0x109190: initialiser() (init.cpp:22)
==78901==    by 0x109230: main (main.cpp:35)
```

Le bloc a été alloué avec `operator new[]` (c'est-à-dire `new[]`) mais libéré avec `operator delete` (c'est-à-dire `delete` sans crochets). Memcheck fournit les deux piles d'appels : celle de la libération incorrecte et celle de l'allocation d'origine, permettant de voir immédiatement le mismatch.

## Décoder les piles d'appels complexes

Sur un projet réel, les piles d'appels sont rarement aussi lisibles que dans les exemples didactiques. Voici les situations les plus courantes et comment les aborder.

### Traversée de la librairie standard

```
==78901== Invalid read of size 8
==78901==    at 0x10A230: std::__cxx11::basic_string<...>::_M_data() const (basic_string.h:195)
==78901==    by 0x10A1F0: std::__cxx11::basic_string<...>::c_str() const (basic_string.h:234)
==78901==    by 0x109F10: Logger::ecrire(std::string const&) (logger.cpp:58)
==78901==    by 0x109E80: ServeurHTTP::traiter_connexion(int) (serveur.cpp:142)
==78901==    by 0x109D50: main (main.cpp:67)
```

Les deux premières frames (`basic_string.h`) sont du code interne à la STL. Elles ne sont pas la cause du problème — elles en sont le *symptôme*. La vraie question est : pourquoi `Logger::ecrire()` passe-t-elle un `std::string` invalide à la ligne 58 de `logger.cpp` ?

**Méthode** : parcourez la pile de haut en bas et identifiez la première frame qui correspond à *votre* code. C'est généralement là que se situe l'erreur. Les frames au-dessus (librairie standard, allocateur) montrent le mécanisme interne qui a déclenché l'accès mémoire, pas le bug.

### Noms de fonctions templates mangled

```
==78901==    by 0x10B450: std::vector<std::unique_ptr<Tache, 
    std::default_delete<Tache>>, std::allocator<std::unique_ptr<Tache, 
    std::default_delete<Tache>>>>::emplace_back<Tache*>(Tache*&&) 
    (vector.tcc:115)
```

Les instanciations de templates produisent des noms de fonctions très verbeux. Valgrind les affiche en version démanglée (lisible), mais la longueur peut être décourageante. En pratique, concentrez-vous sur le type principal (`std::vector`) et l'opération (`emplace_back`). Le reste — les paramètres de template par défaut, les allocateurs — est du bruit dans ce contexte.

### Piles tronquées

```
==78901==    by 0x10C100: ???
==78901==    by 0x10C200: ???
```

Les `???` apparaissent lorsque Valgrind ne parvient pas à résoudre les symboles. Causes possibles :

- Le binaire a été compilé sans `-g` (pas de symboles de débogage).
- Le binaire a été strippé (`strip mon_programme`).
- Les frames proviennent d'une librairie partagée compilée sans symboles.

**Solution** : recompiler avec `-g` et, si le problème vient d'une librairie, installer les paquets de symboles de débogage (sur Ubuntu : `sudo apt install libstdc++-13-dev` ou le paquet `-dbgsym` correspondant).

Si les `???` proviennent d'une librairie tierce pour laquelle les symboles ne sont pas disponibles, augmenter `--num-callers` peut aider à capturer des frames de votre propre code situées plus haut dans la pile.

## Méthodologie de lecture d'un rapport volumineux

Un rapport Valgrind sur un programme complexe peut contenir des centaines de lignes. Voici une méthodologie en cinq étapes pour le traiter efficacement.

### Étape 1 : Commencer par la fin

Lisez d'abord le ERROR SUMMARY et le LEAK SUMMARY. Ils donnent l'ampleur du problème en quelques lignes :

```
==78901== LEAK SUMMARY:
==78901==    definitely lost: 12,288 bytes in 48 blocks
==78901==    indirectly lost: 3,072 bytes in 12 blocks
==78901==      possibly lost: 0 bytes in 0 blocks
==78901==    still reachable: 72,704 bytes in 1 blocks
==78901==         suppressed: 0 bytes in 0 blocks
==78901==
==78901== ERROR SUMMARY: 7 errors from 3 contexts (suppressed: 0 from 0)
```

Ici, 3 contextes d'erreurs et 48 blocs *definitely lost*. L'objectif est de ramener ces chiffres à zéro. Les 72 704 octets *still reachable* en 1 bloc sont probablement le buffer d'I/O de la librairie standard — ignorez-les pour l'instant.

### Étape 2 : Traiter les erreurs d'exécution en premier

Les erreurs d'exécution (lectures/écritures invalides, valeurs non initialisées) sont plus graves que les fuites, car elles provoquent un comportement indéfini. Un use-after-free peut corrompre silencieusement des données ou ouvrir une faille de sécurité. Traitez-les avant de passer aux fuites.

Dans un rapport volumineux, recherchez les mots-clés suivants pour localiser ces erreurs :

- `Invalid read`
- `Invalid write`
- `uninitialised`
- `Mismatched free`
- `Invalid free`

### Étape 3 : Regrouper les fuites par contexte

Memcheck regroupe déjà les fuites partageant la même pile d'appels, mais il est fréquent que plusieurs groupes pointent vers le même pattern de bug. Par exemple, trois rapports distincts de *definitely lost* pourraient tous impliquer la même fonction `creer_connexion()` appelée depuis des sites différents. Corriger la fuite dans `creer_connexion()` élimine les trois.

Parcourez les rapports de fuites et notez les fonctions qui reviennent dans plusieurs piles. Ce sont vos cibles prioritaires.

### Étape 4 : Traiter du plus gros au plus petit

Les loss records sont triés par taille croissante. Le dernier record est la fuite la plus volumineuse. En traitant les fuites de la plus grosse à la plus petite, vous maximisez l'impact de chaque correction sur la consommation mémoire.

De plus, corriger une fuite volumineuse peut résoudre des fuites *indirectly lost* associées, réduisant d'un coup le nombre total de blocs signalés.

### Étape 5 : Relancer après chaque correction

Après avoir corrigé un groupe de fuites, relancez Valgrind. Certaines erreurs masquent ou provoquent d'autres erreurs. Un use-after-free peut corrompre les structures internes de l'allocateur, générant des rapports secondaires qui disparaîtront une fois le bug racine résolu. Des fuites *indirectly lost* disparaîtront avec la correction de leur fuite parente.

Itérer permet de travailler sur un rapport de plus en plus court et de vérifier que chaque correction n'a pas introduit de régression.

## Corrélation avec le code source

Valgrind indique des numéros de ligne, mais le lien entre le rapport et le bug réel n'est pas toujours direct. Voici les patterns de corrélation les plus courants.

### L'allocation est signalée, pas le bug

Memcheck signale *où* la mémoire a été allouée, car c'est l'information la plus exploitable pour tracer une fuite. Mais l'erreur de conception se trouve rarement sur la ligne d'allocation elle-même — elle se trouve sur le chemin d'exécution qui *aurait dû* libérer cette mémoire et qui ne le fait pas.

```
==78901== 1,024 bytes in 1 blocks are definitely lost
==78901==    at 0x4845013: operator new[](unsigned long)
==78901==    by 0x109200: charger_donnees(char const*) (data.cpp:34)
==78901==    by 0x109350: main (main.cpp:55)
```

Le bug n'est pas à la ligne 34 de `data.cpp` (l'allocation). Il est quelque part en aval, là où le pointeur retourné par `charger_donnees()` aurait dû être libéré. Examinez comment la valeur de retour de cette fonction est utilisée dans `main()` à la ligne 55 et après.

### L'erreur est dans le destructeur manquant

Lorsque Memcheck signale une fuite dont l'allocation a lieu dans un constructeur, la cause est presque toujours un destructeur absent ou incomplet. Vérifiez si la classe implémente la Rule of Five (section 6.5) ou, mieux, si les membres à gestion manuelle de mémoire peuvent être remplacés par des types RAII.

### L'erreur est dans un chemin d'exception

Les fuites qui apparaissent de façon intermittente signalent souvent un chemin d'erreur non couvert. Le chemin nominal libère correctement la mémoire, mais une exception ou un retour anticipé contourne le `delete`. Cherchez les `return`, `throw`, `break` et `continue` entre l'allocation et la libération dans la fonction signalée.

## Différencier le bruit du signal

### Allocations de la librairie standard

Les implémentations de libstdc++ et libc++ maintiennent des structures internes (pools de flux, caches de locales, buffers de `std::cout`) qui apparaissent comme *still reachable* dans les rapports. Ce n'est pas un bug — c'est un choix de performance. Ces blocs sont libérés par le système d'exploitation à la fin du processus.

### Librairies tierces

Certaines librairies effectuent des allocations globales qui ne sont libérées qu'à la sortie du programme (ou pas du tout, par design). Si ces rapports polluent vos résultats, créez des fichiers de suppression ciblés (voir section 30.1.1). Documentez chaque suppression avec un commentaire expliquant pourquoi elle est justifiée.

### Faux positifs liés aux optimisations du compilateur

Avec des niveaux d'optimisation élevés (`-O2`, `-O3`), le compilateur peut réorganiser du code, inliner des fonctions ou éliminer des variables, ce qui rend les rapports Valgrind confus : les numéros de ligne ne correspondent plus au code source, des variables semblent non initialisées alors qu'elles le sont, ou des piles d'appels manquent de frames.

C'est pour cette raison que l'analyse Valgrind se fait toujours sur un binaire compilé en `-O0 -g`. Si un bug n'apparaît qu'en `-O2`, il s'agit souvent d'un comportement indéfini que le compilateur exploite lors de l'optimisation — un cas où les sanitizers (section 29.4) sont mieux adaptés, car ils fonctionnent correctement même avec les optimisations activées.

## Formats de sortie alternatifs

### Sortie XML pour l'outillage

Valgrind peut produire un rapport XML exploitable par des outils d'analyse :

```bash
valgrind --leak-check=full --xml=yes --xml-file=rapport.xml ./mon_programme
```

Le format XML est consommé par des outils comme **Valkyrie** (interface graphique pour Valgrind), des plugins CI (Jenkins Valgrind Plugin), ou des scripts d'analyse maison. Chaque erreur est représentée comme un élément structuré avec la pile d'appels, le type d'erreur, les tailles et les adresses.

### Combiner avec `kcachegrind` pour le profiling

Bien que Memcheck ne produise pas de données de profiling, l'outil compagnon **Callgrind** (lancé avec `valgrind --tool=callgrind`) génère des fichiers exploitables par **KCachegrind**, un outil graphique de visualisation de profils d'exécution. Si votre investigation mémoire mène à des questions de performance (« cette fonction alloue-t-elle trop souvent ? »), Callgrind prend le relais.

## Checklist de lecture rapide

Lorsque vous recevez un rapport Valgrind — qu'il provienne de votre propre exécution ou d'un collègue en code review — appliquez cette séquence :

1. **ERROR SUMMARY** : combien d'erreurs, combien de contextes ?
2. **LEAK SUMMARY** : y a-t-il des blocs *definitely* ou *possibly lost* ?
3. **Erreurs d'exécution** : y a-t-il des `Invalid read/write`, des `uninitialised`, des `Mismatched free` ? Si oui, traitez-les en premier.
4. **Fuites** : pour chaque bloc *definitely lost*, identifiez la première frame de *votre* code dans la pile d'appels. C'est votre point d'entrée pour le diagnostic.
5. **Contexte mémoire** : pour les erreurs d'accès invalide, lisez le second bloc (`Address is...`). Il distingue un buffer overflow, un use-after-free, ou un pointeur corrompu.
6. **Corriger, recompiler, relancer** : vérifiez que la correction élimine bien les erreurs signalées et n'en introduit pas de nouvelles.

Cette approche systématique transforme un rapport de plusieurs centaines de lignes en une liste d'actions concrètes, triée par priorité.
