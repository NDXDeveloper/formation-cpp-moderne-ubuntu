# 6.2.2 — Constructeur paramétré

## Chapitre 6 : Classes et Encapsulation

---

## Ce que vous allez apprendre

- Écrire des constructeurs qui acceptent des arguments pour initialiser l'objet.  
- Gérer la surcharge de constructeurs (plusieurs constructeurs avec des signatures différentes).  
- Déléguer la construction d'un constructeur à un autre (C++11).  
- Appliquer `explicit` de manière systématique sur les constructeurs à un argument.  
- Valider les arguments dans le constructeur pour protéger l'invariant de la classe.

---

## Principe

Un constructeur paramétré permet de créer un objet dans un état **choisi par l'appelant**. Là où le constructeur par défaut place l'objet dans un état initial prédéfini, le constructeur paramétré offre le contrôle :

```cpp
class Circle {  
public:  
    Circle() = default;                        // Cercle de rayon 0
    explicit Circle(double radius)             // Cercle de rayon donné
        : radius_(radius) {}

    double area() const {
        return 3.14159265358979 * radius_ * radius_;
    }

private:
    double radius_ = 0.0;
};

Circle unit(1.0);          // Constructeur paramétré → rayon 1.0  
Circle big(42.5);           // Constructeur paramétré → rayon 42.5  
Circle zero;                // Constructeur par défaut → rayon 0.0  
```

Le constructeur paramétré est la forme la plus courante dans du code réel. La majorité des objets ont besoin d'au moins une donnée pour exister dans un état utile : un fichier a besoin d'un chemin, une connexion a besoin d'une adresse, un thread a besoin d'une fonction.

---

## Surcharge de constructeurs

Comme toute fonction en C++ (section 4.4), les constructeurs peuvent être **surchargés**. Le compilateur choisit le constructeur à appeler en fonction du nombre et du type des arguments :

```cpp
class Color {  
public:  
    // Depuis des composantes RGB
    Color(uint8_t r, uint8_t g, uint8_t b)
        : r_(r), g_(g), b_(b) {}

    // Depuis un code hexadécimal
    explicit Color(uint32_t hex)
        : r_((hex >> 16) & 0xFF)
        , g_((hex >> 8) & 0xFF)
        , b_(hex & 0xFF) {}

    // Depuis un niveau de gris
    explicit Color(uint8_t gray)
        : r_(gray), g_(gray), b_(gray) {}

private:
    uint8_t r_, g_, b_;
};

Color red(255, 0, 0);           // Color(uint8_t, uint8_t, uint8_t)  
Color white(0xFFFFFF);          // Color(uint32_t)  
Color mid_gray(uint8_t{128});   // Color(uint8_t)  
```

Chaque constructeur offre une **façon différente** de créer le même type d'objet. L'appelant choisit celle qui correspond à ses données de départ.

### Attention aux ambiguïtés

La surcharge peut mener à des ambiguïtés si les types des arguments sont trop proches :

```cpp
class Ambiguous {  
public:  
    Ambiguous(int x) {}
    Ambiguous(double x) {}
};

// Ambiguous a(3.0f);   // ERREUR potentielle — float → int ou float → double ?
```

Le compilateur signale une erreur quand il ne peut pas choisir sans ambiguïté. La solution est d'utiliser des types explicites (`Ambiguous a(3.0)` ou `Ambiguous a(3)`) ou de repenser la surcharge.

En pratique, si vous avez plus de deux ou trois surcharges, c'est souvent le signe qu'une approche par **factory functions** (fonctions statiques de création) serait plus lisible. Nous y reviendrons en fin de section.

---

## Constructeurs délégués (C++11)

Quand plusieurs constructeurs partagent une logique d'initialisation commune, la **délégation** permet d'éviter la duplication de code. Un constructeur peut appeler un autre constructeur de la même classe dans sa liste d'initialisation :

```cpp
class Connection {  
public:  
    Connection(const std::string& host, uint16_t port, int timeout_ms)
        : host_(host)
        , port_(port)
        , timeout_ms_(timeout_ms) {
        // Logique commune : validation, ouverture, etc.
        validate();
    }

    Connection(const std::string& host, uint16_t port)
        : Connection(host, port, 5000) {}          // Délègue avec timeout par défaut

    Connection(const std::string& host)
        : Connection(host, 8080) {}                // Délègue avec port par défaut

private:
    void validate() {
        if (host_.empty()) throw std::invalid_argument("Host cannot be empty");
        if (port_ == 0) throw std::invalid_argument("Port cannot be 0");
    }

    std::string host_;
    uint16_t port_;
    int timeout_ms_;
};

Connection c1("api.example.com", 443, 10000);   // Les 3 paramètres  
Connection c2("api.example.com", 443);           // Timeout = 5000  
Connection c3("api.example.com");                // Port = 8080, timeout = 5000  
```

La chaîne de délégation forme un entonnoir : les constructeurs "simplifiés" délèguent au constructeur le plus complet, qui contient toute la logique d'initialisation. La validation n'est écrite qu'une seule fois.

**Règles à respecter pour la délégation :**

La délégation s'écrit dans la **liste d'initialisation**, à la place des initialisations de membres. Un constructeur délégant ne peut pas initialiser de membres en plus de la délégation — c'est l'un ou l'autre :

```cpp
// ERREUR — on ne peut pas combiner délégation et initialisation de membres
Connection(const std::string& host)
    : Connection(host, 8080)
    , timeout_ms_(5000) {}     // Interdit !
```

La logique est claire : si vous déléguez à un autre constructeur, c'est lui qui se charge de tout initialiser. Ajouter des initialisations supplémentaires créerait une ambiguïté sur l'état final de l'objet.

### Avant C++11 : le pattern `init()`

Avant la délégation, la solution classique pour factoriser l'initialisation était une méthode privée `init()` :

```cpp
// Style pré-C++11 — à éviter dans du code moderne
class Connection {  
public:  
    Connection(const std::string& host, uint16_t port, int timeout)
        : host_(host), port_(port), timeout_ms_(timeout) {
        init();
    }

    Connection(const std::string& host, uint16_t port)
        : host_(host), port_(port), timeout_ms_(5000) {
        init();    // Même logique dupliquée dans la liste d'initialisation
    }

private:
    void init() { validate(); }
    // ...
};
```

Cette approche fonctionne, mais elle présente un défaut : la liste d'initialisation des membres est dupliquée dans chaque constructeur. Les constructeurs délégués résolvent élégamment ce problème.

## Délégation vs paramètres par défaut

Pour le cas simple de `Connection`, on pourrait aussi utiliser des **valeurs par défaut** :

```cpp
class Connection {  
public:  
    explicit Connection(const std::string& host,
                        uint16_t port = 8080,
                        int timeout_ms = 5000)
        : host_(host), port_(port), timeout_ms_(timeout_ms) {
        validate();
    }
    // ...
};

Connection c1("api.example.com", 443, 10000);   // Tout explicite  
Connection c2("api.example.com", 443);           // timeout = 5000  
Connection c3("api.example.com");                // port = 8080, timeout = 5000  
```

Cette approche est plus concise et convient bien quand les paramètres optionnels forment une séquence naturelle de droite à gauche. Cependant, les paramètres par défaut ne permettent pas de "sauter" un paramètre au milieu : on ne peut pas spécifier le timeout sans spécifier le port.

Le choix entre les deux dépend du contexte :

| Approche | Quand la préférer |  
|----------|-------------------|  
| **Paramètres par défaut** | Arguments optionnels en séquence, logique unique, interface simple |  
| **Constructeurs délégués** | Logiques d'initialisation différentes selon les combinaisons, ou signatures qui ne s'emboîtent pas naturellement |

## Validation des arguments : protéger l'invariant

Le constructeur est le **gardien de l'invariant** de la classe. Si les arguments fournis violent l'invariant, le constructeur doit le signaler immédiatement — pas laisser l'objet naître dans un état corrompu :

```cpp
class PortRange {  
public:  
    PortRange(uint16_t first, uint16_t last)
        : first_(first), last_(last) {
        if (first > last) {
            throw std::invalid_argument(
                "PortRange: first (" + std::to_string(first) +
                ") must be <= last (" + std::to_string(last) + ")"
            );
        }
    }

    uint16_t first() const { return first_; }
    uint16_t last() const { return last_; }
    uint16_t count() const { return last_ - first_ + 1; }

private:
    uint16_t first_;
    uint16_t last_;
};

PortRange http(80, 80);          // OK — un seul port  
PortRange dynamic(49152, 65535); // OK — ports dynamiques  
// PortRange invalid(100, 50);   // Exception — first > last
```

L'invariant de `PortRange` est simple : `first_ <= last_`. Le constructeur vérifie cette condition. Toutes les méthodes (`count()`, par exemple) peuvent ensuite supposer que l'invariant est respecté, sans vérification supplémentaire. C'est tout le bénéfice de l'encapsulation : la validation est centralisée dans le constructeur, pas dispersée dans chaque méthode.

### Quand lancer une exception vs quand refuser la compilation

Deux stratégies de validation coexistent :

**Validation à l'exécution** (exceptions) — pour les valeurs connues uniquement au runtime (entrées utilisateur, données réseau, fichiers de configuration) :

```cpp
explicit DynArray(std::size_t size)
    : data_(new int[size]{}), size_(size) {
    // size est connu à l'exécution → vérification runtime
    // (ici, new lance std::bad_alloc si la mémoire est insuffisante)
}
```

**Validation à la compilation** (`static_assert`, concepts C++20) — pour les valeurs connues à la compilation :

```cpp
template <std::size_t N>  
class FixedBuffer {  
    static_assert(N > 0, "Buffer size must be positive");
    static_assert(N <= 4096, "Buffer size exceeds maximum");
    // ...
};

// FixedBuffer<0> buf;     // ERREUR de compilation
FixedBuffer<256> buf;      // OK
```

Quand la validation peut se faire à la compilation, c'est toujours préférable : l'erreur est détectée plus tôt et le message est plus clair.

## Factory functions : une alternative à la surcharge

Quand le nombre de surcharges devient élevé, ou quand deux constructeurs auraient la même signature mais une sémantique différente, les **fonctions statiques de création** (factory functions) offrent une meilleure lisibilité :

```cpp
class Color {  
public:  
    static Color from_rgb(uint8_t r, uint8_t g, uint8_t b) {
        return Color(r, g, b);
    }

    static Color from_hex(uint32_t hex) {
        return Color((hex >> 16) & 0xFF,
                     (hex >> 8) & 0xFF,
                     hex & 0xFF);
    }

    static Color from_gray(uint8_t level) {
        return Color(level, level, level);
    }

    static Color from_hsv(double h, double s, double v) {
        // Conversion HSV → RGB
        auto [r, g, b] = hsv_to_rgb(h, s, v);   // Structured binding C++17
        return Color(r, g, b);
    }

private:
    Color(uint8_t r, uint8_t g, uint8_t b)      // Constructeur privé
        : r_(r), g_(g), b_(b) {}

    static std::tuple<uint8_t, uint8_t, uint8_t>
    hsv_to_rgb(double h, double s, double v);

    uint8_t r_, g_, b_;
};

// L'appel est auto-documenté — aucune ambiguïté
Color sky   = Color::from_rgb(135, 206, 235);  
Color brand = Color::from_hex(0xFF5733);  
Color dark  = Color::from_gray(30);  
Color vivid = Color::from_hsv(120.0, 1.0, 1.0);  
```

Chaque fonction statique a un **nom explicite** qui documente l'intention. Le constructeur réel est privé — toute la création passe par les factory functions. C'est un pattern très courant dans les bibliothèques professionnelles.

Notez que `from_rgb` et `from_hsv` prennent tous les deux trois arguments numériques. Avec des constructeurs classiques, il serait impossible de les distinguer par surcharge (même nombre et types proches d'arguments). Les factory functions résolvent ce problème élégamment.

## Rappel : `explicit` sur les constructeurs à un argument

La section 6.2 a introduit `explicit`. Un rappel concret dans le contexte des constructeurs paramétrés :

```cpp
class Meter {  
public:  
    Meter(double value) : value_(value) {}       // Pas explicit — dangereux
    double value() const { return value_; }
private:
    double value_;
};

class Kilogram {  
public:  
    explicit Kilogram(double value) : value_(value) {}   // explicit — sûr
    double value() const { return value_; }
private:
    double value_;
};

void process(Meter m, Kilogram kg) {
    // ...
}

process(3.5, Kilogram(70.0));     // 3.5 implicitement converti en Meter
// process(3.5, 70.0);            // ERREUR — pas de conversion implicite vers Kilogram
```

Sans `explicit`, `Meter` accepte silencieusement un `double`. C'est peut-être acceptable pour un type d'unité simple — mais dans le doute, préférez `explicit`. Il est toujours plus facile de retirer une restriction que d'en ajouter une après coup.

## Fil conducteur : constructeur paramétré de `DynArray`

Notre `DynArray` possède déjà un constructeur paramétré depuis la section 6.1. Améliorons-le avec une meilleure validation et une variante à deux paramètres :

**`dynarray.h`** — évolution :

```cpp
class DynArray {  
public:  
    DynArray() = default;

    // Crée un tableau de 'size' éléments initialisés à zéro
    explicit DynArray(std::size_t size);

    // Crée un tableau de 'size' éléments initialisés à 'value'
    DynArray(std::size_t size, int value);

    ~DynArray();

    std::size_t size() const { return size_; }
    bool empty() const { return size_ == 0; }

    int& operator[](std::size_t index);
    const int& operator[](std::size_t index) const;

private:
    int* data_ = nullptr;
    std::size_t size_ = 0;
};
```

**`dynarray.cpp`** — implémentation avec délégation :

```cpp
#include "dynarray.h"
#include <algorithm>   // std::fill_n
#include <stdexcept>

// Constructeur principal — alloue et initialise à zéro
DynArray::DynArray(std::size_t size)
    : data_(size > 0 ? new int[size]{} : nullptr)
    , size_(size) {}

// Constructeur délégué — réutilise le constructeur principal, puis remplit
DynArray::DynArray(std::size_t size, int value)
    : DynArray(size) {                       // Délégation au constructeur principal
    std::fill_n(data_, size_, value);        // Remplace les zéros par 'value'
}

DynArray::~DynArray() {
    delete[] data_;
}

int& DynArray::operator[](std::size_t index) {
    if (index >= size_) {
        throw std::out_of_range("DynArray: index out of range");
    }
    return data_[index];
}

const int& DynArray::operator[](std::size_t index) const {
    if (index >= size_) {
        throw std::out_of_range("DynArray: index out of range");
    }
    return data_[index];
}
```

**Points à noter :**

Le constructeur `DynArray(std::size_t)` est `explicit` — on ne veut pas qu'un entier soit implicitement converti en tableau. Le constructeur `DynArray(std::size_t, int)` a deux paramètres : `explicit` n'est pas nécessaire (la conversion implicite ne s'applique qu'aux constructeurs appelables avec un seul argument).

Le constructeur principal gère le cas `size == 0` avec l'opérateur ternaire : pas d'allocation si le tableau est vide. Le constructeur `DynArray(size, value)` délègue au premier, puis remplit avec `std::fill_n`. L'allocation et l'initialisation de base ne sont écrites qu'une seule fois.

Utilisation :

```cpp
DynArray empty;           // [] — constructeur par défaut  
DynArray zeros(5);        // [0, 0, 0, 0, 0] — constructeur paramétré  
DynArray nines(3, 9);     // [9, 9, 9] — constructeur délégué  
```

> ⚠️ **Rappel** — La copie reste dangereuse. `DynArray a(3); DynArray b = a;` provoque un *double free* car le constructeur de copie implicite copie le pointeur `data_`, pas les données pointées. Nous corrigeons ce problème dans la prochaine sous-section (6.2.3).

---

## Points clés à retenir

- Un constructeur paramétré initialise l'objet selon les arguments de l'appelant. C'est la forme la plus courante en code réel.  
- La **surcharge** permet plusieurs façons de construire le même type. Si les surcharges deviennent nombreuses ou ambiguës, les **factory functions** (méthodes statiques) offrent une alternative plus lisible et auto-documentée.  
- Les **constructeurs délégués** (C++11) factorisent la logique commune en permettant à un constructeur d'en appeler un autre. Le constructeur délégant ne peut pas initialiser de membres en plus de la délégation.  
- Choisissez entre **paramètres par défaut** (simple, séquentiel) et **constructeurs délégués** (logiques différentes, signatures incompatibles) selon le contexte.  
- Le constructeur est le **gardien de l'invariant**. Validez les arguments et lancez une exception si l'objet ne peut pas être construit dans un état valide.  
- Appliquez `explicit` sur tout constructeur appelable avec un seul argument, sauf si la conversion implicite a un sens sémantique clair.
