# 6.5.2 — Constructeur de copie

## Chapitre 6 : Classes et Encapsulation — Section 6.5 : Règle des 5

---

## Objectif de cette sous-section

Le constructeur de copie a été développé en section 6.2.3 (copie superficielle vs profonde, signature, implémentation). Cette sous-section le **revisite sous l'angle de la Règle des 5** : sa cohérence avec le destructeur et l'opérateur d'affectation, le contrat de la copie profonde, et les interactions avec les opérations de déplacement.

---

## Le constructeur de copie dans l'ensemble des cinq

Le constructeur de copie et le destructeur forment une paire indissociable. Le constructeur de copie **duplique** la ressource, le destructeur **la libère**. Si ces deux opérations ne sont pas symétriques, le programme est incorrect :

```
Constructeur de copie :  new int[size]  +  std::copy_n(...)
                              ↕ symétrie
Destructeur :            delete[] data_
```

Si le constructeur de copie alloue avec `new[]`, le destructeur doit libérer avec `delete[]`. Si le constructeur de copie ouvre un fichier, le destructeur doit le fermer. Si le constructeur de copie acquiert un verrou, le destructeur doit le libérer. La symétrie doit être exacte — toute asymétrie est un bug.

Mais le constructeur de copie doit aussi être cohérent avec la **troisième** opération de la famille copie : l'opérateur d'affectation par copie (section 6.5.3). Ces deux opérations ont la même sémantique — créer un clone indépendant — mais dans des contextes différents :

| Opération | Contexte | Difficulté supplémentaire |  
|---|---|---|  
| Constructeur de copie | L'objet cible **n'existe pas encore** | Aucune — juste allouer et copier |  
| Opérateur d'affectation | L'objet cible **existe déjà** | Libérer l'ancienne ressource, gérer l'auto-affectation |

Le résultat observable doit être identique : après `T b = a;` ou `b = a;`, l'objet `b` doit contenir un clone indépendant de `a`. Si votre constructeur de copie et votre opérateur d'affectation produisent des résultats différents, le code client ne peut pas raisonner sur votre classe de manière cohérente.

---

## Impact sur la génération implicite

Rappelons la ligne correspondante de la matrice :

| Vous déclarez… | Destructeur | Copie ctor | Copie `=` | Move ctor | Move `=` |  
|---|---|---|---|---|---|  
| **Copie ctor** | ✅ généré | — | ✅ généré ⚠️ | ❌ supprimé | ❌ supprimé |

Déclarer un constructeur de copie a deux effets critiques :

**Les opérations de déplacement sont supprimées.** Le compilateur considère que si vous avez personnalisé la copie, c'est que votre classe gère une ressource de manière non triviale. Il ne veut pas deviner ce que le déplacement devrait faire. Conséquence : sans constructeur de déplacement explicite, tout `std::move` sur votre objet **se rabat silencieusement sur la copie**.

```cpp
class CopyOnly {  
public:  
    CopyOnly() = default;
    CopyOnly(const CopyOnly& other) {
        std::cout << "copy\n";
        // ... copie profonde ...
    }
    // Pas de move déclaré → supprimé par le compilateur
};

CopyOnly a;  
CopyOnly b = a;              // "copy" — constructeur de copie  
CopyOnly c = std::move(a);   // "copy" — pas de move, fallback sur la copie !  
```

Ce n'est pas une erreur de compilation — c'est un **pessimisme silencieux**. Le code fonctionne mais perd le bénéfice du déplacement O(1). Pour `DynArray` avec un million d'éléments, cela signifie une copie de 4 Mo là où un déplacement de quelques octets suffirait.

**L'opérateur d'affectation par copie est conservé mais déprécié.** Le compilateur le génère (copie superficielle, membre à membre), mais le standard signale que cette génération est dépréciée. Comme pour le destructeur, la copie superficielle implicite est presque certainement incorrecte si vous avez dû écrire un constructeur de copie personnalisé.

## Le contrat de la copie profonde

Quand vous écrivez un constructeur de copie pour une classe qui possède une ressource, vous vous engagez sur un contrat implicite que le code client attend :

### Indépendance complète

Après la copie, les deux objets doivent être **totalement indépendants**. Modifier l'un ne doit jamais affecter l'autre :

```cpp
DynArray a(3, 42);  
DynArray b = a;       // Copie profonde  

b[0] = 99;            // Modifie b uniquement  
assert(a[0] == 42);   // a est intact — contrat respecté  
```

Ce contrat semble évident, mais il peut être subtilement violé si votre classe contient des pointeurs partagés ou des ressources avec un état global. Chaque niveau d'indirection doit être dupliqué.

### Équivalence sémantique

Après la copie, les deux objets doivent être **sémantiquement équivalents** — ils doivent se comporter de manière identique :

```cpp
DynArray a(5, 42);  
DynArray b = a;  

assert(a.size() == b.size());          // Même taille  
for (std::size_t i = 0; i < a.size(); ++i) {  
    assert(a[i] == b[i]);              // Mêmes valeurs
}
```

Cela ne signifie pas que les objets sont identiques bit à bit — `a.data_` et `b.data_` pointent vers des adresses différentes. Mais du point de vue de l'utilisateur, les deux objets sont interchangeables.

### Pas d'effet sur la source

Le constructeur de copie prend un `const T&`. Le `const` est une garantie : l'objet source n'est pas modifié par l'opération de copie. C'est ce qui distingue fondamentalement la copie du déplacement : la copie laisse l'original intact, le déplacement le vide.

## Exception safety du constructeur de copie

Le constructeur de copie est l'une des opérations les plus susceptibles de lancer une exception, car il alloue des ressources. La question clé est : que se passe-t-il si l'allocation échoue ?

```cpp
DynArray::DynArray(const DynArray& other)
    : data_(other.size_ > 0 ? new int[other.size_] : nullptr)   // Peut lancer std::bad_alloc
    , size_(other.size_) {
    std::copy_n(other.data_, size_, data_);
}
```

Si `new int[other.size_]` lance `std::bad_alloc` :

1. L'objet en cours de construction n'est **jamais complètement construit**.
2. Son destructeur n'est **pas appelé** (le standard ne détruit pas les objets dont le constructeur a échoué).
3. Les membres déjà construits avant le `throw` sont détruits automatiquement par le *stack unwinding*.
4. L'exception se propage à l'appelant.
5. L'objet source `other` n'est **pas affecté** — il est passé par `const&`.

Dans notre cas, l'allocation est la première opération. Si elle échoue, rien n'a été alloué, il n'y a rien à nettoyer. C'est la **garantie forte** (*strong exception guarantee*) : soit l'opération réussit complètement, soit rien ne change.

La situation serait plus complexe si le constructeur de copie devait copier des objets dont le constructeur de copie peut lui-même lancer :

```cpp
class ComplexOwner {  
public:  
    ComplexOwner(const ComplexOwner& other)
        : buffer_(new Widget[other.count_])   // 1. Alloue
        , count_(other.count_) {
        for (std::size_t i = 0; i < count_; ++i) {
            buffer_[i] = other.buffer_[i];    // 2. Copie chaque Widget — peut lancer !
        }
        // Si la copie du 5e Widget lance, les 4 premiers sont construits
        // mais buffer_ ne sera pas delete[] → fuite !
    }
```

Ce code a une fuite potentielle : si la copie d'un `Widget` lance, le bloc alloué par `new Widget[]` n'est jamais libéré (le destructeur de `ComplexOwner` n'est pas appelé, car le constructeur n'a pas terminé). La solution correcte utilise un `std::vector` ou un `std::unique_ptr` pour l'allocation :

```cpp
class SafeOwner {  
public:  
    SafeOwner(const SafeOwner& other)
        : buffer_(std::make_unique<Widget[]>(other.count_))
        , count_(other.count_) {
        for (std::size_t i = 0; i < count_; ++i) {
            buffer_[i] = other.buffer_[i];   // Si ça lance, unique_ptr libère buffer_
        }
    }

private:
    std::unique_ptr<Widget[]> buffer_;
    std::size_t count_ = 0;
};
```

C'est encore une illustration de la Règle du Zéro : en utilisant `std::unique_ptr` comme membre, le nettoyage en cas d'exception est automatique. Pour `DynArray` avec des `int` (qui ne lancent jamais lors de la copie), le problème ne se pose pas — mais il est bon de connaître le piège pour des types plus complexes.

## Cohérence avec les quatre autres opérations

Vérifions systématiquement que le constructeur de copie de `DynArray` est cohérent avec chaque autre opération :

### Avec le destructeur

Le constructeur de copie alloue avec `new int[size]`. Le destructeur libère avec `delete[] data_`. La symétrie est parfaite. Chaque objet copié possède son propre bloc, et chaque destructeur libère exactement un bloc.

### Avec l'opérateur d'affectation par copie (section 6.5.3)

Le constructeur de copie et l'opérateur d'affectation par copie doivent produire le même résultat observable. Nous verrons en section 6.5.3 que l'opérateur d'affectation ajoute la libération de l'ancienne ressource et la gestion de l'auto-affectation — mais le résultat final est identique : un clone indépendant.

### Avec le constructeur de déplacement

Le constructeur de copie **duplique** les données (O(n)). Le constructeur de déplacement **transfère** le pointeur (O(1)). Les deux créent un nouvel objet valide, mais par des mécanismes différents. L'un laisse la source intacte, l'autre la vide. La distinction est nette et complémentaire.

### Avec l'opérateur d'affectation par déplacement (section 6.5.5)

Même relation qu'entre constructeur de copie et opérateur d'affectation par copie : le résultat est le même (transfert de ressource), mais l'opérateur d'affectation doit en plus libérer l'ancienne ressource de l'objet cible.

## Relecture du constructeur de copie de `DynArray`

```cpp
DynArray::DynArray(const DynArray& other)
    : data_(other.size_ > 0 ? new int[other.size_] : nullptr)
    , size_(other.size_) {
    std::copy_n(other.data_, size_, data_);
}
```

| Vérification | Résultat |  
|---|---|  
| Copie profonde (nouveau bloc alloué) | ✅ `new int[other.size_]` |  
| Données dupliquées (pas de partage) | ✅ `std::copy_n` |  
| Source non modifiée (`const&`) | ✅ Paramètre `const DynArray&` |  
| Gère le cas vide (`size_ == 0`) | ✅ Ternaire → `nullptr` |  
| Symétrique avec le destructeur (`new[]` ↔ `delete[]`) | ✅ |  
| Exception safety (garantie forte) | ✅ Si `new` lance, rien n'est alloué |  
| Cohérent avec l'opérateur d'affectation par copie | ⏳ À vérifier en 6.5.3 |

L'implémentation est correcte et complète. Elle sera le modèle sur lequel l'opérateur d'affectation par copie s'appuiera.

## Quand la copie doit être interdite

Rappelons (section 6.2.3) que certaines classes ne doivent pas être copiables. Dans le contexte de la Règle des 5, interdire la copie est une décision cohérente qui impacte l'ensemble :

```cpp
class UniqueConnection {  
public:  
    explicit UniqueConnection(const std::string& addr);
    ~UniqueConnection();

    // Copie interdite — une connexion ne se duplique pas
    UniqueConnection(const UniqueConnection&) = delete;
    UniqueConnection& operator=(const UniqueConnection&) = delete;

    // Déplacement autorisé — transfert de propriété
    UniqueConnection(UniqueConnection&& other) noexcept;
    UniqueConnection& operator=(UniqueConnection&& other) noexcept;

private:
    int socket_fd_;
};
```

L'interdiction de copie n'est pas un aveu de faiblesse — c'est une décision de conception. `std::unique_ptr`, `std::thread`, `std::mutex` sont tous non copiables. Ils modélisent des ressources à **propriété exclusive** : il n'y a qu'un seul propriétaire à tout instant, et la propriété se transfère par déplacement.

Quand vous supprimez le constructeur de copie avec `= delete`, supprimez **aussi** l'opérateur d'affectation par copie. Les deux forment une paire : il serait incohérent d'interdire `T b = a;` mais d'autoriser `b = a;` (ou inversement).

## Le cas particulier : copie polymorphique

Un dernier point qui anticipe le chapitre 7. Quand vous manipulez des objets via des pointeurs vers une classe de base, le constructeur de copie classique ne suffit plus — il découpe (*slices*) l'objet dérivé :

```cpp
class Shape {  
public:  
    virtual ~Shape() = default;
    // ... 
};

class Circle : public Shape {
    double radius_;
};

Circle c(5.0);  
Shape s = c;         // SLICING — seule la partie Shape est copiée, radius_ est perdu  
```

La solution idiomatique est le pattern **`clone()`** — une méthode virtuelle qui retourne une copie polymorphique :

```cpp
class Shape {  
public:  
    virtual ~Shape() = default;
    virtual std::unique_ptr<Shape> clone() const = 0;
};

class Circle : public Shape {  
public:  
    explicit Circle(double r) : radius_(r) {}

    std::unique_ptr<Shape> clone() const override {
        return std::make_unique<Circle>(*this);   // Utilise le copy ctor de Circle
    }

private:
    double radius_;
};

// Copie polymorphique correcte
std::unique_ptr<Shape> original = std::make_unique<Circle>(5.0);  
std::unique_ptr<Shape> copy = original->clone();   // Crée bien un Circle, pas un Shape  
```

Le constructeur de copie de `Circle` est utilisé en interne par `clone()`, mais l'interface publique passe par la méthode virtuelle. C'est un pattern fréquent dans les hiérarchies de classes où la copie doit préserver le type dynamique.

---

## Points clés à retenir

- Le constructeur de copie et le destructeur forment une **paire symétrique** : l'un alloue, l'autre libère. Toute asymétrie est un bug.  
- Le constructeur de copie et l'opérateur d'affectation par copie doivent produire le **même résultat observable** — un clone indépendant. Ils sont la même opération dans deux contextes différents (création vs remplacement).  
- Déclarer un constructeur de copie **supprime** les opérations de déplacement. Sans constructeur de déplacement explicite, `std::move` se rabat silencieusement sur la copie — un pessimisme O(n) au lieu d'un transfert O(1).  
- Le constructeur de copie doit offrir la **garantie forte** d'exception : soit la copie réussit complètement, soit rien ne change. Pour les types complexes, utilisez des membres RAII (`unique_ptr`, `vector`) pour garantir le nettoyage en cas d'échec.  
- Si la copie n'a pas de sens, supprimez **les deux** opérations de copie ensemble (`= delete` sur le constructeur et l'opérateur d'affectation).  
- Pour les hiérarchies polymorphiques, le constructeur de copie classique provoque du *slicing*. Utilisez le pattern **`clone()`** avec une méthode virtuelle retournant un `std::unique_ptr<Base>`.
