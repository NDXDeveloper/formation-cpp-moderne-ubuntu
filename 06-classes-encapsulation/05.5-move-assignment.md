# 6.5.5 — Opérateur d'affectation par déplacement

## Chapitre 6 : Classes et Encapsulation — Section 6.5 : Règle des 5

---

## Ce que vous allez apprendre

- Comprendre les différences et les points communs avec l'opérateur d'affectation par copie.

- Assembler les cinq opérations spéciales en un ensemble cohérent.  
- Obtenir la version **finale et complète** de `DynArray`.

---

## Rôle dans la Règle des 5

L'opérateur d'affectation par déplacement est la **dernière pièce du puzzle**. Il combine les responsabilités de deux opérations que nous avons déjà maîtrisées :

- Comme l'**opérateur d'affectation par copie** (6.5.3) : il remplace le contenu d'un objet existant, ce qui implique de libérer l'ancienne ressource et de gérer l'auto-affectation.  
- Comme le **constructeur de déplacement** (6.5.4) : il transfère la propriété de la ressource source au lieu de la dupliquer, et il neutralise la source.

```
Affectation par copie :      libérer l'ancien  +  dupliquer le nouveau     (O(n))  
Affectation par déplacement : libérer l'ancien  +  transférer le nouveau   (O(1))  
```

---

## Signature

```cpp
class DynArray {  
public:  
    DynArray& operator=(DynArray&& other) noexcept;
    // ...
};
```

Le paramètre est une **référence rvalue non-const** (`T&&`). Contrairement à la copie (`const T&`), le déplacement a besoin de modifier la source pour la neutraliser — d'où l'absence de `const`. Le type de retour est `T&` pour le chaînage, comme pour l'affectation par copie. Et `noexcept` est présent, comme pour le constructeur de déplacement — pour les mêmes raisons (optimisations STL).

---

## Implémentation directe

L'approche la plus lisible suit trois étapes explicites :

```cpp
DynArray& DynArray::operator=(DynArray&& other) noexcept {
    if (this == &other) return *this;    // 1. Protection auto-affectation

    delete[] data_;                       // 2. Libérer l'ancienne ressource

    data_ = other.data_;                  // 3a. Voler la ressource
    size_ = other.size_;

    other.data_ = nullptr;                // 3b. Neutraliser la source
    other.size_ = 0;

    return *this;
}
```

Déroulons l'exécution pour `b = std::move(a)` :

```
Avant :
  a.data_ → [42, 42, 42]    a.size_ = 3
  b.data_ → [0, 0, 0, 0]   b.size_ = 4

Étape 2 — delete[] b.data_ :
  b.data_ → [libéré]

Étape 3a — vol :
  b.data_ → [42, 42, 42]    b.size_ = 3

Étape 3b — neutralisation :
  a.data_ → nullptr          a.size_ = 0

Après :
  b.data_ → [42, 42, 42]    b.size_ = 3     (possède le bloc)
  a.data_ → nullptr          a.size_ = 0     (vidé, destructible)
```

Le bloc mémoire a changé de propriétaire sans aucune copie de données. L'ancien bloc de `b` a été libéré. L'opération est O(1).

---

## Auto-affectation par déplacement

L'auto-affectation par déplacement (`a = std::move(a)`) est un cas particulier qui mérite attention. Sans le test `this == &other`, voici ce qui se passerait :

```cpp
// Sans protection : a = std::move(a)
delete[] data_;           // Libère le bloc de a  
data_ = other.data_;      // other EST a → data_ = pointeur pendouillant !  
size_ = other.size_;  
other.data_ = nullptr;    // Met data_ à nullptr (data_ et other.data_ sont le même)  
other.size_ = 0;          // Met size_ à 0  
// Résultat : a est vide → données perdues
```

Sans protection, l'objet perd ses données. Ce n'est pas un crash (pas de *double free*), mais c'est une **perte silencieuse de données** — un bug plus subtil et plus difficile à diagnostiquer.

Le test `this == &other` court-circuite l'opération et laisse l'objet intact. L'auto-affectation par déplacement est rare en pratique, mais elle peut survenir dans du code générique (algorithmes STL, `std::swap` personnalisé) ou via des alias indirects.

> 💡 Certains experts considèrent que `a = std::move(a)` est une erreur de programmation qui ne devrait jamais arriver, et que la protection est superflue. Le standard exige seulement que l'objet reste dans un **état valide** après auto-move-assignment — pas qu'il conserve sa valeur. En pratique, le test est bon marché (une comparaison de pointeurs) et évite les surprises. Nous recommandons de l'inclure.

---

## Alternative : swap-and-reset

Une variante élégante consiste à échanger les ressources avec `other`, puis à laisser le destructeur de `other` (en fin de portée de l'appelant) libérer l'ancien bloc :

```cpp
DynArray& DynArray::operator=(DynArray&& other) noexcept {
    if (this == &other) return *this;

    // Échanger toutes les ressources
    std::swap(data_, other.data_);
    std::swap(size_, other.size_);

    return *this;
}
```

Après le swap, `other` possède l'ancienne ressource de `*this`. Quand `other` sera détruit — ce qui peut arriver immédiatement si c'est un temporaire, ou plus tard s'il a un nom — le destructeur libérera cette ancienne ressource.

Cette approche a un avantage : elle est `noexcept` par construction (swap de scalaires). Elle a un inconvénient subtil : la libération de l'ancienne ressource est **différée** — elle n'a pas lieu immédiatement dans l'opérateur, mais quand `other` est détruit. Dans la grande majorité des cas, cela ne pose aucun problème. Mais si la libération immédiate est importante (par exemple pour réduire la pression mémoire à un instant précis), la version directe avec `delete[]` est préférable.

---

## Unification copie et déplacement via copy-and-swap

En section 6.5.3, nous avons vu la variante de copy-and-swap qui prend le paramètre **par valeur** :

```cpp
DynArray& DynArray::operator=(DynArray other) {   // Copie OU déplacement selon le contexte
    swap(other);
    return *this;
}
```

Cette unique fonction remplace **les deux** opérateurs d'affectation (copie et déplacement). Le compilateur choisit le constructeur approprié pour créer `other` :

```cpp
DynArray a(3, 42);  
DynArray b(5, 0);  

b = a;                   // 'a' est lvalue → other construit par copie → swap  
b = DynArray(10, 7);     // rvalue → other construit par déplacement → swap  
b = std::move(a);        // rvalue → other construit par déplacement → swap  
```

C'est l'approche la plus concise : une seule fonction au lieu de deux, aucune duplication de logique. Le compromis, comme évoqué en section 6.5.3, est un léger surcoût sur l'auto-affectation et l'impossibilité de réutiliser le buffer existant quand les tailles correspondent.

Pour notre `DynArray` pédagogique, nous conservons les **deux opérateurs séparés** pour que chaque mécanisme soit visible et compréhensible. En production, l'approche unifiée par valeur est souvent le meilleur choix.

---

## Relecture et vérification de cohérence

```cpp
DynArray& DynArray::operator=(DynArray&& other) noexcept {
    if (this == &other) return *this;

    delete[] data_;

    data_ = other.data_;
    size_ = other.size_;

    other.data_ = nullptr;
    other.size_ = 0;

    return *this;
}
```

| Vérification | Résultat |  
|---|---|  
| Libère l'ancienne ressource (`delete[] data_`) | ✅ Pas de fuite |  
| Transfère la nouvelle ressource (vol du pointeur) | ✅ O(1) |  
| Neutralise la source | ✅ `nullptr`, `0` |  
| Gère l'auto-affectation (`this == &other`) | ✅ Court-circuit |  
| `noexcept` | ✅ Aucune allocation |  
| Cohérent avec le destructeur | ✅ Source : `delete[] nullptr` → no-op |  
| Résultat identique à l'affectation par copie | ✅ Mêmes données dans la cible |  
| Retourne `*this` pour le chaînage | ✅ |  
| État *moved-from* = construit par défaut | ✅ `nullptr`, `0` |

## La Règle des 5 : mission accomplie

Les cinq opérations spéciales de `DynArray` sont maintenant toutes en place. Voici le **code final complet** — l'aboutissement de tout le chapitre 6 :

**`dynarray.h`** :

```cpp
#pragma once

#include <cstddef>
#include <iosfwd>

class DynArray {  
public:  
    // --- Constructeurs ---
    DynArray() = default;
    explicit DynArray(std::size_t size);
    DynArray(std::size_t size, int value);

    // --- Règle des 5 ---
    ~DynArray();                                         // 6.5.1 — Destructeur
    DynArray(const DynArray& other);                     // 6.5.2 — Copie ctor
    DynArray& operator=(const DynArray& other);          // 6.5.3 — Copie =
    DynArray(DynArray&& other) noexcept;                 // 6.5.4 — Move ctor
    DynArray& operator=(DynArray&& other) noexcept;      // 6.5.5 — Move =

    // --- Interface publique ---
    std::size_t size() const { return size_; }
    bool empty() const { return size_ == 0; }

    int& operator[](std::size_t index);
    const int& operator[](std::size_t index) const;

    void swap(DynArray& other) noexcept;

    // --- Friend ---
    friend std::ostream& operator<<(std::ostream& os, const DynArray& arr);

private:
    int* data_ = nullptr;
    std::size_t size_ = 0;
};

inline void swap(DynArray& a, DynArray& b) noexcept { a.swap(b); }
```

**`dynarray.cpp`** :

```cpp
#include "dynarray.h"

#include <algorithm>
#include <ostream>
#include <stdexcept>

// --- Constructeurs ---

DynArray::DynArray(std::size_t size)
    : data_(size > 0 ? new int[size]{} : nullptr)
    , size_(size) {}

DynArray::DynArray(std::size_t size, int value)
    : DynArray(size) {
    std::fill_n(data_, size_, value);
}

// --- Règle des 5 ---

DynArray::~DynArray() {
    delete[] data_;
}

DynArray::DynArray(const DynArray& other)
    : data_(other.size_ > 0 ? new int[other.size_] : nullptr)
    , size_(other.size_) {
    std::copy_n(other.data_, size_, data_);
}

DynArray& DynArray::operator=(const DynArray& other) {
    if (this == &other) return *this;

    int* new_data = nullptr;
    if (other.size_ > 0) {
        new_data = new int[other.size_];
        std::copy_n(other.data_, other.size_, new_data);
    }

    delete[] data_;
    data_ = new_data;
    size_ = other.size_;

    return *this;
}

DynArray::DynArray(DynArray&& other) noexcept
    : data_(other.data_)
    , size_(other.size_) {
    other.data_ = nullptr;
    other.size_ = 0;
}

DynArray& DynArray::operator=(DynArray&& other) noexcept {
    if (this == &other) return *this;

    delete[] data_;

    data_ = other.data_;
    size_ = other.size_;

    other.data_ = nullptr;
    other.size_ = 0;

    return *this;
}

// --- Interface publique ---

int& DynArray::operator[](std::size_t index) {
    if (index >= size_) throw std::out_of_range("DynArray: index out of range");
    return data_[index];
}

const int& DynArray::operator[](std::size_t index) const {
    if (index >= size_) throw std::out_of_range("DynArray: index out of range");
    return data_[index];
}

void DynArray::swap(DynArray& other) noexcept {
    std::swap(data_, other.data_);
    std::swap(size_, other.size_);
}

// --- Friend ---

std::ostream& operator<<(std::ostream& os, const DynArray& arr) {
    os << "[";
    for (std::size_t i = 0; i < arr.size_; ++i) {
        if (i > 0) os << ", ";
        os << arr.data_[i];
    }
    os << "]";
    return os;
}
```

## Vérification complète : tous les scénarios

Passons en revue chaque scénario d'utilisation pour vérifier que la classe se comporte correctement dans tous les cas :

```cpp
// Construction par défaut
DynArray empty;                          // data_=nullptr, size_=0 ✅

// Construction paramétrée
DynArray a(5, 42);                       // [42, 42, 42, 42, 42] ✅

// Construction par copie
DynArray b = a;                          // b est un clone indépendant de a ✅  
b[0] = 99;  
assert(a[0] == 42);                      // a non modifié ✅  

// Construction par déplacement
DynArray c = std::move(b);              // c = [99, 42, 42, 42, 42], b = [] ✅

// Affectation par copie
DynArray d(3, 0);  
d = a;                                   // d est un clone de a, ancien bloc libéré ✅  
assert(d.size() == 5 && d[0] == 42);  

// Affectation par déplacement
DynArray e(2, 7);  
e = std::move(d);                        // e = [42,42,42,42,42], d = [] ✅  
assert(e.size() == 5 && d.empty());  

// Auto-affectation par copie
a = a;                                   // No-op ✅

// Auto-affectation par déplacement
e = std::move(e);                        // No-op ✅

// Chaînage
DynArray f(1), g(1), h(3, 7);  
f = g = h;                               // f et g sont des clones de h ✅  

// Temporaires (rvalues)
DynArray i;  
i = DynArray(4, 100);                    // Move assignment depuis temporaire ✅  

// Swap
DynArray x(2, 1), y(3, 9);  
swap(x, y);                              // x=[9,9,9], y=[1,1] ✅  

// Sortie
std::cout << a << "\n";                  // [42, 42, 42, 42, 42] ✅
```

Chaque opération maintient l'invariant de la classe. Aucun bloc mémoire ne fuit. Aucun pointeur n'est partagé entre deux objets. La Règle des 5 est pleinement respectée.

## Vue d'ensemble : les cinq opérations côte à côte

Pour conclure, voici un récapitulatif synthétique des cinq opérations et de leur mécanique interne :

| Opération | Alloue | Libère `*this` | Copie données | Vole pointeur | Neutralise source | `noexcept` |  
|---|---|---|---|---|---|---|  
| **Destructeur** | — | ✅ `delete[]` | — | — | — | ✅ |  
| **Copie ctor** | ✅ `new[]` | — | ✅ `copy_n` | — | — | ❌ |  
| **Copie `=`** | ✅ `new[]` | ✅ `delete[]` | ✅ `copy_n` | — | — | ❌ |  
| **Move ctor** | — | — | — | ✅ | ✅ `nullptr` | ✅ |  
| **Move `=`** | — | ✅ `delete[]` | — | ✅ | ✅ `nullptr` | ✅ |

La symétrie est frappante. Les opérations de copie allouent et copient. Les opérations de déplacement volent et neutralisent. Les opérations d'affectation (copie et déplacement) libèrent l'ancienne ressource en plus. Le destructeur ne fait que libérer. Chaque case du tableau est une conséquence logique du rôle de l'opération — rien n'est arbitraire.

---

## Points clés à retenir

- L'opérateur d'affectation par déplacement **libère l'ancienne ressource** et **transfère** la nouvelle depuis la source. C'est la combinaison du destructeur (libérer) et du constructeur de déplacement (voler + neutraliser).  
- L'auto-affectation par déplacement (`a = std::move(a)`) doit laisser l'objet dans un **état valide**. Le test `this == &other` est la protection la plus simple et la plus lisible.  
- L'opérateur doit être **`noexcept`** — il ne fait que libérer et échanger des scalaires, aucune allocation n'est nécessaire.  
- L'approche **copy-and-swap par valeur** unifie les deux opérateurs d'affectation en une seule fonction, au prix d'un léger surcoût sur l'auto-affectation.  
- Avec les cinq opérations en place, `DynArray` est une classe **complète, correcte et robuste**. Chaque scénario (copie, déplacement, affectation, auto-affectation, chaînage, temporaires) est géré.  
- En C++ moderne, visez la **Règle du Zéro** en utilisant des membres RAII (`vector`, `unique_ptr`). La Règle des 5 est le filet de sécurité quand vous gérez des ressources brutes — et maintenant, vous savez exactement comment l'appliquer.
