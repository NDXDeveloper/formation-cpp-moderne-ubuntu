# 6.2.4 — Constructeur de déplacement

## Chapitre 6 : Classes et Encapsulation

## Ce que vous allez apprendre

- Comprendre pourquoi la copie est parfois un gaspillage inutile.  
- Distinguer une *lvalue* d'une *rvalue* à un niveau intuitif.  
- Écrire un constructeur de déplacement qui transfère les ressources sans les dupliquer.  
- Comprendre l'état "vidé" (*moved-from*) d'un objet après un déplacement.  
- Savoir quand le compilateur génère un constructeur de déplacement implicite — et quand il ne le fait pas.

> 💡 Cette sous-section donne une introduction pratique à la sémantique de déplacement dans le contexte des constructeurs. Le chapitre 10 (*Move Semantics*) couvre le sujet en profondeur : lvalues vs rvalues, `std::move`, perfect forwarding, RVO et copy elision.

## Le problème : des copies coûteuses et inutiles

En section 6.2.3, nous avons vu que copier un `DynArray` de 10 millions d'éléments implique d'allouer 40 Mo et de dupliquer toutes les données. C'est le prix à payer quand on veut deux objets indépendants.

Mais il existe des situations où l'objet source **va disparaître juste après** la copie. Dans ce cas, dupliquer les données pour détruire l'original immédiatement après est du gaspillage pur :

```cpp
DynArray create_data() {
    DynArray result(1'000'000, 42);   // 1 million d'éléments
    return result;                     // result va être détruit après le return
}

DynArray data = create_data();
// Sans déplacement : allocation + copie de 4 Mo, puis destruction de result
// Avec déplacement : transfert de 3 scalaires (pointeur + taille), coût quasi nul
```

L'idée du déplacement est simple : au lieu de dupliquer les ressources de l'objet source, on les **vole**. On prend son pointeur, sa taille, et on le laisse dans un état vide. Pas d'allocation, pas de copie de données — juste quelques affectations de scalaires.

C'est comme un déménagement : au lieu de photocopier tous vos meubles pour les mettre dans le nouvel appartement (copie), vous les transportez directement (déplacement). L'ancien appartement se retrouve vide, mais c'est exactement ce qu'on voulait puisqu'on le quitte.

## Lvalues et rvalues : l'intuition

Avant d'écrire un constructeur de déplacement, il faut comprendre *quand* le compilateur choisit de déplacer plutôt que de copier. La réponse repose sur la distinction entre **lvalues** et **rvalues** :

- Une **lvalue** est une expression qui désigne un objet **persistant**, avec un nom et une adresse en mémoire. On peut la retrouver plus tard.  
- Une **rvalue** est une expression qui désigne un objet **temporaire**, sur le point de disparaître. Personne n'y fera plus référence.

```cpp
DynArray a(100);              // 'a' est une lvalue — elle a un nom, elle persiste  
DynArray b = a;               // Copie — 'a' est une lvalue, on ne peut pas la voler  

DynArray c = DynArray(100);   // DynArray(100) est une rvalue — temporaire, va disparaître
                              // → Le compilateur peut déplacer au lieu de copier

DynArray d = create_data();   // create_data() retourne une rvalue
                              // → Déplacement (ou élision, cf. chapitre 10)
```

Le raisonnement du compilateur est le suivant : si l'objet source est une lvalue, quelqu'un d'autre pourrait encore l'utiliser — il faut copier pour ne pas le corrompre. Si l'objet source est une rvalue, personne ne l'utilisera plus — on peut voler ses ressources en toute sécurité.

> 📎 Cette explication est volontairement simplifiée. La taxonomie complète des catégories de valeur (lvalue, prvalue, xvalue, glvalue) et les références rvalue (`&&`) seront traitées en détail au chapitre 10 (section 10.1).

## La référence rvalue : `T&&`

Pour distinguer un constructeur de copie (qui reçoit une lvalue) d'un constructeur de déplacement (qui reçoit une rvalue), le C++11 introduit un nouveau type de référence : la **référence rvalue**, notée `&&` :

```cpp
class T {  
public:  
    T(const T& other);    // Constructeur de copie   — accepte les lvalues
    T(T&& other);         // Constructeur de déplacement — accepte les rvalues
};
```

Le compilateur choisit automatiquement le bon constructeur en fonction de la catégorie de valeur de l'argument :

```cpp
T a;  
T b = a;             // 'a' est une lvalue   → constructeur de copie  
T c = T();           // T() est une rvalue    → constructeur de déplacement  
T d = std::move(a);  // std::move(a) force la catégorie rvalue → constructeur de déplacement  
```

`std::move` ne déplace rien par lui-même — c'est un simple **cast** qui transforme une lvalue en rvalue, signalant au compilateur que vous autorisez le vol des ressources. C'est une déclaration d'intention : "je n'ai plus besoin de cet objet, tu peux le vider." Nous approfondirons `std::move` au chapitre 10 (section 10.2).

## Écrire un constructeur de déplacement

Le constructeur de déplacement prend une **référence rvalue non-const** (`T&&`) et transfère les ressources de l'objet source vers l'objet en construction. Voici le pattern :

```cpp
DynArray::DynArray(DynArray&& other) noexcept
    : data_(other.data_)          // 1. Voler le pointeur
    , size_(other.size_) {        // 2. Voler la taille
    other.data_ = nullptr;        // 3. Neutraliser la source
    other.size_ = 0;              //    pour éviter le double free
}
```

Trois étapes, toujours les mêmes :

**Étape 1 et 2 — Voler les ressources.** On copie les valeurs scalaires (pointeur et taille) de `other` vers le nouvel objet. C'est une simple copie de quelques octets, quel que soit le nombre d'éléments dans le tableau.

**Étape 3 — Neutraliser la source.** On met `other` dans un état **valide mais vide** : `data_ = nullptr`, `size_ = 0`. C'est crucial : quand le destructeur de `other` s'exécutera, il appellera `delete[] nullptr`, qui ne fait rien. Sans cette neutralisation, on retrouverait le *double free* de la section précédente.

Visualisons le transfert :

```
Avant le déplacement :           Après le déplacement :

other.data_ ──→ [42, 42, 42]    other.data_ ──→ nullptr  
other.size_ = 3                  other.size_ = 0  

this->data_ = ???               this->data_ ──→ [42, 42, 42]  
this->size_ = ???               this->size_ = 3  
```

Le bloc mémoire n'a pas bougé. Seuls les pointeurs ont changé de propriétaire. Le coût est **O(1)** — constant, indépendant de la taille du tableau.

## `noexcept` : une promesse importante

Vous avez remarqué le `noexcept` sur le constructeur de déplacement. Ce n'est pas décoratif — c'est une promesse au compilateur et à la bibliothèque standard que le déplacement **ne lancera jamais d'exception**.

Pourquoi est-ce important ? Parce que la STL en dépend pour prendre des décisions d'optimisation. L'exemple le plus courant est `std::vector` lors d'un agrandissement :

```
Scénario : vector<DynArray> doit réallouer son buffer interne

Si le déplacement est noexcept :
  → vector déplace les éléments vers le nouveau buffer (rapide)

Si le déplacement peut lancer :
  → vector copie les éléments (lent mais safe)
  → En cas d'exception pendant la copie, l'ancien buffer est intact
```

Si `std::vector` déplaçait les éléments et qu'une exception survenait à mi-chemin, certains éléments seraient dans le nouveau buffer et d'autres dans l'ancien — un état incohérent irrécupérable. Pour éviter ce risque, `std::vector` utilise `std::move_if_noexcept` : il ne déplace que si le constructeur de déplacement est garanti `noexcept`. Sinon, il copie.

En pratique, un constructeur de déplacement qui se contente de transférer des scalaires et des pointeurs n'a aucune raison de lancer une exception. Marquez-le **toujours** `noexcept`.

---

## L'état *moved-from* : valide mais indéterminé

Après un déplacement, l'objet source se trouve dans un état dit *moved-from*. Le standard C++ impose une seule contrainte : l'objet doit rester dans un **état valide**, c'est-à-dire qu'il doit être **destructible** et éventuellement **réassignable**. Mais son contenu n'est plus garanti.

```cpp
DynArray a(3, 42);  
DynArray b = std::move(a);  

// a est dans l'état moved-from
// Garanti : a est destructible, a peut recevoir une nouvelle valeur
// Non garanti : a.size(), a[0], etc. — le contenu est indéterminé

a = DynArray(5, 0);     // OK — on peut réassigner un objet moved-from  
std::cout << a.size();   // 5 — a est de nouveau utilisable normalement  
```

Pour notre `DynArray`, l'état *moved-from* est bien défini : `data_ == nullptr` et `size_ == 0`. C'est identique à un `DynArray` construit par défaut. Ce choix est délibéré et va au-delà du minimum requis par le standard — il rend le comportement prévisible et facilite le débogage.

**Bonne pratique** : laissez toujours l'objet *moved-from* dans le même état qu'un objet construit par défaut. Ce n'est pas obligatoire, mais c'est ce que font les types de la bibliothèque standard (`std::string`, `std::vector`, `std::unique_ptr`), et les développeurs s'y attendent.

## Génération implicite par le compilateur

Le compilateur génère un constructeur de déplacement implicite **uniquement si** toutes ces conditions sont réunies :

- Aucun constructeur de copie n'est déclaré par l'utilisateur.  
- Aucun opérateur d'affectation par copie n'est déclaré.  
- Aucun opérateur d'affectation par déplacement n'est déclaré.  
- Aucun destructeur n'est déclaré.

Si l'une de ces conditions n'est pas remplie, le compilateur **ne génère pas** de constructeur de déplacement. C'est une règle conservatrice : si vous avez personnalisé l'une des opérations spéciales, le compilateur ne veut pas deviner ce que le déplacement devrait faire.

Concrètement pour notre `DynArray` : dès que nous avons écrit un destructeur (section 6.1) et un constructeur de copie (section 6.2.3), le compilateur a cessé de générer le constructeur de déplacement. C'est pourquoi nous devons l'écrire nous-mêmes.

```cpp
class Widget {  
public:  
    // Aucune opération spéciale déclarée
    // → Le compilateur génère : constructeur par défaut, copie, déplacement,
    //   affectation par copie, affectation par déplacement, destructeur
private:
    std::string name_;
    std::vector<int> data_;
};

Widget a;  
Widget b = std::move(a);   // Constructeur de déplacement implicite  
                           // → déplace name_ (string::move), déplace data_ (vector::move)
```

Le constructeur de déplacement implicite déplace chaque membre individuellement. Pour les types standard, c'est exactement ce qu'on veut. Pour les types primitifs (`int`, `double`, pointeurs bruts), "déplacer" est identique à "copier" — il n'y a pas de ressource à transférer.

## Déplacement et héritage

Si votre classe hérite d'une classe de base, le constructeur de déplacement doit explicitement déplacer la partie de base. Un oubli fréquent chez les débutants est d'écrire :

```cpp
class Base {  
public:  
    Base(Base&& other) noexcept : name_(std::move(other.name_)) {}
private:
    std::string name_;
};

class Derived : public Base {  
public:  
    // ERREUR subtile — other est une lvalue dans le corps du constructeur
    Derived(Derived&& other) noexcept
        : Base(other)                        // Appelle Base(const Base&) — COPIE !
        , extra_(std::move(other.extra_)) {}

    // CORRECT — std::move convertit other en rvalue
    Derived(Derived&& other) noexcept
        : Base(std::move(other))             // Appelle Base(Base&&) — DÉPLACEMENT
        , extra_(std::move(other.extra_)) {}

private:
    std::vector<int> extra_;
};
```

Le piège vient du fait qu'un paramètre `T&& other`, bien qu'il soit de type référence rvalue, est **lui-même une lvalue** à l'intérieur de la fonction — il a un nom (`other`), donc il persiste. Pour le traiter comme une rvalue et déclencher le déplacement, il faut explicitement appliquer `std::move`. C'est un point contre-intuitif que le chapitre 10 expliquera en détail.

## Copie vs déplacement : résumé comparatif

Pour `DynArray` avec un million d'éléments :

| Opération | Ce qui se passe | Coût |  
|-----------|----------------|------|  
| **Copie** | `new int[1'000'000]` + `std::copy_n` de 1M éléments | O(n) — ~4 Mo alloués et copiés |  
| **Déplacement** | Copie de 2 scalaires (`data_`, `size_`) + neutralisation de la source | O(1) — quelques nanosecondes |

Le ratio est frappant : pour un tableau d'un million d'éléments, le déplacement est environ **un million de fois** plus rapide que la copie. C'est pourquoi la sémantique de déplacement, introduite en C++11, a été une révolution pour les performances du C++.

## Fil conducteur : `DynArray` avec déplacement

Ajoutons le constructeur de déplacement à notre classe :

**`dynarray.h`** :

```cpp
#pragma once

#include <cstddef>

class DynArray {  
public:  
    DynArray() = default;
    explicit DynArray(std::size_t size);
    DynArray(std::size_t size, int value);

    // Copie — copie profonde
    DynArray(const DynArray& other);

    // Déplacement — transfert de propriété
    DynArray(DynArray&& other) noexcept;

    ~DynArray();

    std::size_t size() const { return size_; }
    bool empty() const { return size_ == 0; }

    int& operator[](std::size_t index);
    const int& operator[](std::size_t index) const;

private:
    int* data_ = nullptr;
    std::size_t size_ = 0;
};
```

**`dynarray.cpp`** — ajout du constructeur de déplacement :

```cpp
#include "dynarray.h"
#include <algorithm>
#include <stdexcept>

DynArray::DynArray(std::size_t size)
    : data_(size > 0 ? new int[size]{} : nullptr)
    , size_(size) {}

DynArray::DynArray(std::size_t size, int value)
    : DynArray(size) {
    std::fill_n(data_, size_, value);
}

DynArray::DynArray(const DynArray& other)
    : data_(other.size_ > 0 ? new int[other.size_] : nullptr)
    , size_(other.size_) {
    std::copy_n(other.data_, size_, data_);
}

DynArray::DynArray(DynArray&& other) noexcept
    : data_(other.data_)
    , size_(other.size_) {
    other.data_ = nullptr;
    other.size_ = 0;
}

DynArray::~DynArray() {
    delete[] data_;
}

int& DynArray::operator[](std::size_t index) {
    if (index >= size_) throw std::out_of_range("DynArray: index out of range");
    return data_[index];
}

const int& DynArray::operator[](std::size_t index) const {
    if (index >= size_) throw std::out_of_range("DynArray: index out of range");
    return data_[index];
}
```

Vérifions le comportement :

```cpp
DynArray a(5, 42);                // a = [42, 42, 42, 42, 42]

DynArray b = std::move(a);        // Déplacement — b vole les ressources de a  
std::cout << b.size() << "\n";    // 5  
std::cout << b[0] << "\n";       // 42  
std::cout << a.size() << "\n";    // 0 — a est vidé  
std::cout << a.empty() << "\n";   // 1 (true)  

// Le retour de fonction déclenche aussi le déplacement (ou l'élision)
DynArray make_range(std::size_t n) {
    DynArray result(n);
    for (std::size_t i = 0; i < n; ++i) result[i] = static_cast<int>(i);
    return result;     // Déplacement ou RVO
}

DynArray range = make_range(1'000'000);   // Aucune copie d'un million d'éléments
```

> ⚠️ **Danger restant** — Les opérateurs d'affectation (par copie et par déplacement) ne sont toujours pas définis. L'écriture `b = a;` ou `b = std::move(a);` utilise les versions implicites, qui font une copie superficielle et causent des fuites et des *double free*. La section 6.5 (Règle des 5) complètera les cinq opérations spéciales d'un coup.

## Points clés à retenir

- Le constructeur de déplacement **transfère** les ressources au lieu de les dupliquer. Son coût est O(1), indépendant de la taille des données.  
- Il prend une **référence rvalue** (`T&&`) et s'applique automatiquement aux objets temporaires et aux expressions marquées avec `std::move`.  
- Après un déplacement, l'objet source est dans un état *moved-from* : **valide** (destructible, réassignable) mais au **contenu indéterminé**. En pratique, laissez-le dans l'état d'un objet construit par défaut.  
- Marquez **toujours** le constructeur de déplacement `noexcept`. Sans cela, la STL (`std::vector` notamment) préférera copier vos objets plutôt que les déplacer lors des réallocations.  
- Le compilateur ne génère un constructeur de déplacement implicite que si **aucune** des autres opérations spéciales (destructeur, copie, affectation) n'est déclarée par l'utilisateur.  
- Un paramètre `T&& other` est une lvalue à l'intérieur de la fonction. Pour déclencher le déplacement d'un membre ou de la classe de base, appliquez `std::move` explicitement.
