# 6.2.3 — Constructeur de copie

## Chapitre 6 : Classes et Encapsulation

---

## Ce que vous allez apprendre

- Comprendre ce qu'est un constructeur de copie et quand il est invoqué.  
- Distinguer la **copie superficielle** (*shallow copy*) de la **copie profonde** (*deep copy*).  
- Identifier le bug classique du *double free* causé par la copie superficielle de pointeurs.  
- Écrire un constructeur de copie correct pour une classe gérant une ressource.  
- Savoir quand laisser le compilateur générer la copie — et quand l'interdire avec `= delete`.

---

## Le problème : copier un objet qui possède une ressource

Jusqu'ici, notre `DynArray` possède un constructeur par défaut, un constructeur paramétré et un destructeur. Voyons ce qui se passe quand on copie un objet :

```cpp
DynArray a(3, 42);     // a = [42, 42, 42], a.data_ pointe vers un bloc de 3 int  
DynArray b = a;        // Que se passe-t-il ici ?  
```

Le compilateur génère un constructeur de copie implicite qui copie chaque membre, champ par champ. Pour `data_`, il copie **la valeur du pointeur** — pas les données pointées. Après cette copie, `a.data_` et `b.data_` pointent vers **le même bloc mémoire** :

```
Avant la copie :                  Après la copie (shallow) :

a.data_ ──→ [42, 42, 42]         a.data_ ──→ [42, 42, 42] ←── b.data_  
a.size_ = 3                      a.size_ = 3    b.size_ = 3  
```

Tout semble fonctionner — jusqu'à la destruction. Quand `b` sort de portée, son destructeur appelle `delete[] b.data_`. Le bloc mémoire est libéré. Puis `a` sort de portée, et son destructeur appelle `delete[] a.data_` — sur le même bloc, déjà libéré. C'est un **double free** : un comportement indéfini qui provoque typiquement un crash.

Mais le problème ne se limite pas à la destruction. Avant même que les destructeurs ne s'exécutent, modifier `b[0]` modifie aussi `a[0]`, puisqu'ils partagent la même mémoire. Ce n'est pas une copie — c'est un alias involontaire.

---

## Copie superficielle vs copie profonde

Ces deux termes décrivent deux stratégies de copie fondamentalement différentes :

**Copie superficielle** (*shallow copy*) — copie les valeurs des membres tels quels. Pour un pointeur, cela signifie copier l'adresse, pas les données. Les deux objets partagent la même ressource sous-jacente. C'est ce que fait le constructeur de copie implicite.

**Copie profonde** (*deep copy*) — alloue une nouvelle ressource pour l'objet copié et duplique le contenu. Les deux objets sont complètement indépendants après la copie.

```
Copie superficielle (shallow) :      Copie profonde (deep) :

a.data_ ──→ [42, 42, 42]            a.data_ ──→ [42, 42, 42]
                 ↑
b.data_ ─────────┘                   b.data_ ──→ [42, 42, 42]  (nouveau bloc)
```

Pour une classe qui **possède** une ressource (mémoire allouée, descripteur de fichier, connexion réseau), la copie superficielle est presque toujours un bug. La copie profonde est la solution correcte — ou, si la copie n'a pas de sens, l'interdiction pure et simple.

---

## Définition du constructeur de copie

Un constructeur de copie est un constructeur qui prend en paramètre une **référence constante** vers un objet du même type :

```cpp
class T {  
public:  
    T(const T& other);    // Constructeur de copie
};
```

La référence est `const` parce que copier un objet ne devrait pas modifier l'original. La référence (et non la valeur) est obligatoire — passer par valeur déclencherait un appel au constructeur de copie, créant une récursion infinie.

### Quand le constructeur de copie est-il appelé ?

Le constructeur de copie est invoqué dans quatre situations principales :

```cpp
class Widget {  
public:  
    Widget() { std::cout << "default\n"; }
    Widget(const Widget& other) { std::cout << "copy\n"; }
};

// 1. Initialisation par copie
Widget a;  
Widget b = a;          // copy  

// 2. Initialisation directe avec un objet du même type
Widget c(a);           // copy

// 3. Passage par valeur à une fonction
void process(Widget w) {}  
process(a);            // copy  

// 4. Retour par valeur (si le compilateur n'applique pas la RVO)
Widget create() {
    Widget w;
    return w;          // copy (potentiel — souvent élidé, voir plus bas)
}
```

Le cas 3 est celui qui surprend le plus les débutants : chaque appel de fonction avec un paramètre passé **par valeur** déclenche une copie. C'est pourquoi le chapitre 4 insistait sur le passage par `const&` pour les objets non triviaux.

Le cas 4 mérite une nuance importante : les compilateurs modernes appliquent systématiquement la **RVO** (*Return Value Optimization*) et la **NRVO** (*Named Return Value Optimization*), qui éliminent la copie. Depuis C++17, la RVO est même **garantie par le standard** dans certains cas. Nous approfondirons ce sujet au chapitre 10 (section 10.5).

## Écrire un constructeur de copie correct

Pour `DynArray`, la copie profonde signifie allouer un nouveau bloc et y dupliquer les données de l'original :

```cpp
DynArray::DynArray(const DynArray& other)
    : data_(other.size_ > 0 ? new int[other.size_] : nullptr)
    , size_(other.size_) {
    std::copy_n(other.data_, size_, data_);
}
```

Décomposons chaque étape :

1. **Allocation** — `new int[other.size_]` crée un bloc de même taille que l'original. Si l'original est vide, on alloue `nullptr`.
2. **Initialisation des membres** — `size_` prend la même valeur que `other.size_`.
3. **Duplication des données** — `std::copy_n` copie `size_` éléments de `other.data_` vers `data_`.

Après cette opération, `a` et `b` possèdent chacun leur propre bloc mémoire avec les mêmes valeurs. Les modifier indépendamment est sûr, et la destruction de l'un n'affecte pas l'autre.

Vérifions le comportement :

```cpp
DynArray a(3, 42);     // a = [42, 42, 42]  
DynArray b = a;        // Copie profonde → b = [42, 42, 42] (bloc séparé)  

b[0] = 99;             // Modifie b uniquement

std::cout << a[0] << "\n";   // 42 — a est intact  
std::cout << b[0] << "\n";   // 99 — b a sa propre copie  
// Les destructeurs libèrent deux blocs distincts — pas de double free
```

## Quand laisser le compilateur faire le travail

La copie profonde n'est nécessaire que pour les classes qui gèrent **directement** une ressource brute (pointeur vers de la mémoire allouée, descripteur de fichier, handle système). Si votre classe ne contient que des types qui savent déjà se copier correctement, le constructeur de copie implicite fait exactement ce qu'il faut :

```cpp
class UserProfile {  
private:  
    std::string name_;                // std::string gère sa propre copie profonde
    std::string email_;               // Idem
    std::vector<std::string> tags_;   // std::vector aussi
    int age_ = 0;                     // Type primitif — copie triviale
};

// Le constructeur de copie implicite copie chaque membre.
// string et vector font leur propre copie profonde en interne.
// Résultat : la copie de UserProfile est correcte sans rien écrire.

UserProfile alice;  
UserProfile clone = alice;   // Copie correcte — chaque membre se copie proprement  
```

C'est le cas le plus fréquent en C++ moderne. La majorité des classes n'ont pas besoin d'un constructeur de copie personnalisé parce qu'elles s'appuient sur des types de la bibliothèque standard qui gèrent eux-mêmes leurs ressources. C'est le principe de composition : en construisant vos classes à partir de briques qui se copient correctement, votre classe se copie correctement par défaut.

**Règle pratique** : n'écrivez un constructeur de copie que si votre classe gère directement une ressource brute. Si tous vos membres sont des types standard ou des types avec une copie correcte, laissez le compilateur faire.

## Quand interdire la copie

Certaines classes ne devraient **jamais** être copiées. Dupliquer un thread, un verrou mutex, ou un descripteur de fichier unique n'a pas de sens — ou pire, serait dangereux :

```cpp
class UniqueFile {  
public:  
    explicit UniqueFile(const std::string& path)
        : fd_(::open(path.c_str(), O_RDWR)) {
        if (fd_ < 0) throw std::runtime_error("Cannot open: " + path);
    }

    ~UniqueFile() {
        if (fd_ >= 0) ::close(fd_);
    }

    // Copie interdite — un descripteur de fichier ne se duplique pas
    UniqueFile(const UniqueFile&) = delete;
    UniqueFile& operator=(const UniqueFile&) = delete;

    // Le déplacement, en revanche, est logique (transfert de propriété)
    // → section 6.2.4

private:
    int fd_;
};

UniqueFile f("/tmp/data.bin");
// UniqueFile g = f;    // ERREUR de compilation — copie supprimée
```

`= delete` rend l'intention explicite et produit un message d'erreur clair à la compilation. C'est le mécanisme standard depuis C++11 pour interdire une opération.

### Avant C++11 : la technique du privé non défini

Avant `= delete`, on déclarait le constructeur de copie en `private` sans le définir :

```cpp
// Style pré-C++11 — ne plus utiliser
class NonCopyable {  
private:  
    NonCopyable(const NonCopyable&);             // Déclaré mais jamais défini
    NonCopyable& operator=(const NonCopyable&);  // Idem
};
```

Toute tentative de copie depuis l'extérieur échouait à la compilation (accès privé). Toute tentative depuis l'intérieur de la classe échouait à l'édition de liens (symbole non défini). Cette technique fonctionnait, mais le message d'erreur était obscur. Préférez `= delete` systématiquement.

## Copie et exception safety

Un point subtil : que se passe-t-il si l'allocation échoue au milieu du constructeur de copie ? Reprenons notre implémentation :

```cpp
DynArray::DynArray(const DynArray& other)
    : data_(other.size_ > 0 ? new int[other.size_] : nullptr)
    , size_(other.size_) {
    std::copy_n(other.data_, size_, data_);
}
```

Si `new int[other.size_]` lance `std::bad_alloc`, l'objet en cours de construction n'est **jamais complètement construit**. Le standard C++ garantit que le destructeur n'est pas appelé pour un objet dont le constructeur a lancé une exception. Mais les membres déjà construits avant le throw sont correctement détruits — c'est le mécanisme de *stack unwinding*.

Dans notre cas, l'allocation est la première opération de la liste d'initialisation. Si elle échoue, `size_` n'est pas initialisé, et aucune mémoire n'a été allouée — il n'y a rien à nettoyer. L'exception se propage proprement à l'appelant.

Ce scénario deviendrait plus complexe si la classe gérait **plusieurs ressources**. C'est l'une des raisons pour lesquelles le C++ moderne recommande de gérer chaque ressource dans un objet dédié (smart pointer, RAII wrapper) plutôt que d'accumuler des pointeurs bruts dans une même classe. Nous approfondirons ce sujet en section 6.3 (RAII) et au chapitre 9 (smart pointers).

## Le coût de la copie

Copier un objet a un coût proportionnel à la taille des données qu'il possède. Pour un `DynArray` de 10 millions d'éléments, le constructeur de copie alloue 40 Mo de mémoire et copie 10 millions d'entiers. C'est une opération lourde.

C'est pourquoi le C++ offre deux mécanismes complémentaires pour éviter les copies inutiles :

**Le passage par référence constante** (`const&`) — ne copie pas l'objet, donne un accès en lecture seule :

```cpp
// Mauvais — copie le tableau entier à chaque appel
double average(DynArray arr) {
    double sum = 0;
    for (std::size_t i = 0; i < arr.size(); ++i) sum += arr[i];
    return sum / arr.size();
}

// Bon — aucune copie, accès en lecture seule
double average(const DynArray& arr) {
    double sum = 0;
    for (std::size_t i = 0; i < arr.size(); ++i) sum += arr[i];
    return sum / arr.size();
}
```

**La sémantique de déplacement** (`&&`, `std::move`) — transfère les ressources au lieu de les dupliquer. Au lieu de copier 40 Mo, on déplace trois scalaires (pointeur, taille). C'est le sujet de la prochaine sous-section (6.2.4) et du chapitre 10.

## Fil conducteur : `DynArray` avec copie profonde

Voici l'état complet de notre classe après l'ajout du constructeur de copie :

**`dynarray.h`** :

```cpp
#pragma once

#include <cstddef>

class DynArray {  
public:  
    DynArray() = default;
    explicit DynArray(std::size_t size);
    DynArray(std::size_t size, int value);

    // Constructeur de copie — copie profonde
    DynArray(const DynArray& other);

    ~DynArray();

    std::size_t size() const { return size_; }
    bool empty() const { return size_ == 0; }

    int& operator[](std::size_t index);
    const int& operator[](std::size_t index) const;

private:
    int* data_ = nullptr;
    std::size_t size_ = 0;
};
```

**`dynarray.cpp`** — ajout du constructeur de copie :

```cpp
#include "dynarray.h"
#include <algorithm>
#include <stdexcept>

DynArray::DynArray(std::size_t size)
    : data_(size > 0 ? new int[size]{} : nullptr)
    , size_(size) {}

DynArray::DynArray(std::size_t size, int value)
    : DynArray(size) {
    std::fill_n(data_, size_, value);
}

DynArray::DynArray(const DynArray& other)
    : data_(other.size_ > 0 ? new int[other.size_] : nullptr)
    , size_(other.size_) {
    std::copy_n(other.data_, size_, data_);
}

DynArray::~DynArray() {
    delete[] data_;
}

int& DynArray::operator[](std::size_t index) {
    if (index >= size_) throw std::out_of_range("DynArray: index out of range");
    return data_[index];
}

const int& DynArray::operator[](std::size_t index) const {
    if (index >= size_) throw std::out_of_range("DynArray: index out of range");
    return data_[index];
}
```

Le *double free* est résolu. Mais la classe n'est pas encore complète : il manque l'opérateur d'affectation par copie (`operator=`), le constructeur de déplacement, et l'opérateur d'affectation par déplacement. C'est précisément pour cela que la **Règle des 5** (section 6.5) existe : dès que vous écrivez l'une des cinq opérations spéciales, vous devez considérer les quatre autres. Nous y viendrons.

> ⚠️ **Danger restant** — En l'état, `DynArray a(3); DynArray b(5); b = a;` utilise l'opérateur d'affectation par copie **implicite**, qui fait une copie superficielle de `data_`. Le bloc alloué par `b` fuit (memory leak) et `a.data_` sera libéré deux fois. L'opérateur d'affectation sera traité en section 6.5.3.

---

## Points clés à retenir

- Le constructeur de copie prend un paramètre `const T&` et crée un nouvel objet identique à l'original.  
- Le constructeur de copie **implicite** fait une copie membre à membre (*shallow copy*). C'est correct pour les types qui ne gèrent pas de ressources brutes — et dangereux pour ceux qui en gèrent.  
- Pour une classe possédant une ressource (pointeur brut, descripteur), écrivez un constructeur de copie qui effectue une **copie profonde** : allocation d'une nouvelle ressource et duplication du contenu.  
- Si la copie n'a pas de sens pour votre classe, interdisez-la avec `= delete`.  
- Si tous vos membres sont des types standard (string, vector, smart pointers), laissez le compilateur générer la copie — il fera le bon travail.  
- La copie a un coût proportionnel aux données. Préférez le passage par `const&` et envisagez la sémantique de déplacement (section suivante) pour les transferts de propriété.  
- Écrire un constructeur de copie déclenche la **Règle des 5** : vous devez considérer les quatre autres opérations spéciales (section 6.5).
