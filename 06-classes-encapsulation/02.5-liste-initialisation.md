# 6.2.5 — Liste d'initialisation

## Chapitre 6 : Classes et Encapsulation

---

## Ce que vous allez apprendre

- Comprendre la différence fondamentale entre **initialisation** et **affectation** dans un constructeur.  
- Maîtriser la syntaxe de la *member initializer list*.  
- Savoir quand la liste d'initialisation est **obligatoire** (et pas seulement recommandée).  
- Connaître l'ordre d'initialisation des membres et les pièges qui en découlent.  
- Articuler liste d'initialisation et *default member initializers* (C++11).

---

## Le problème : initialiser n'est pas affecter

Considérez ces deux constructeurs. Ils semblent faire la même chose, mais le mécanisme sous-jacent est fondamentalement différent :

```cpp
class Connection {  
public:  
    // Version 1 — Affectation dans le corps
    Connection(const std::string& host, uint16_t port) {
        host_ = host;      // Affectation
        port_ = port;      // Affectation
    }

    // Version 2 — Liste d'initialisation
    Connection(const std::string& host, uint16_t port)
        : host_(host)      // Initialisation
        , port_(port) {}   // Initialisation

private:
    std::string host_;
    uint16_t port_;
};
```

Dans la **version 1**, voici ce qui se passe réellement à la construction d'un objet `Connection` :

1. Le membre `host_` est d'abord **construit par défaut** (constructeur de `std::string` → chaîne vide).
2. Le membre `port_` est laissé non initialisé (type primitif).
3. Le corps du constructeur s'exécute : `host_` est **réaffecté** avec la valeur de `host`.

C'est un aller-retour inutile : on construit `host_` une première fois (chaîne vide), puis on l'écrase immédiatement avec la bonne valeur. Pour un `std::string`, le surcoût est modeste. Pour un type plus lourd — une matrice, un conteneur, une connexion réseau — il peut être significatif.

Dans la **version 2**, chaque membre est **directement construit avec sa valeur finale**. Pas de construction par défaut intermédiaire, pas de réaffectation. Le membre `host_` est construit une seule fois, directement avec la valeur de `host`.

## Syntaxe

La liste d'initialisation se place entre la signature du constructeur et l'accolade ouvrante du corps, introduite par le caractère `:` :

```cpp
class Rectangle {  
public:  
    Rectangle(double width, double height)
        : width_(width)        // Initialise width_ avec width
        , height_(height) {    // Initialise height_ avec height
        // Corps du constructeur — vide dans ce cas
    }

private:
    double width_;
    double height_;
};
```

Chaque élément de la liste utilise la syntaxe d'initialisation directe : `membre_(valeur)` ou `membre_{valeur}`. Les éléments sont séparés par des virgules. La convention de style courante place la virgule en début de ligne pour faciliter la lecture et les diffs dans un système de contrôle de version :

```cpp
// Style recommandé — une initialisation par ligne, virgule en tête
Connection(const std::string& host, uint16_t port, int timeout)
    : host_(host)
    , port_(port)
    , timeout_ms_(timeout) {}
```

Le corps du constructeur (`{}`) s'exécute **après** la liste d'initialisation. À ce moment, tous les membres sont déjà construits. Le corps sert à exécuter une logique supplémentaire (validation, logging, ouverture de ressources) qui ne peut pas se réduire à une simple initialisation.

---

## Quand la liste d'initialisation est obligatoire

Dans certains cas, l'affectation dans le corps du constructeur ne compile tout simplement pas. La liste d'initialisation est alors la seule option.

### Membres `const`

Un membre `const` ne peut pas être modifié après sa construction. L'affectation dans le corps est impossible :

```cpp
class Config {  
public:  
    // ERREUR — max_retries_ est const, l'affectation est interdite
    Config(int retries) {
        max_retries_ = retries;   // Ne compile pas
    }

    // CORRECT — initialisation directe dans la liste
    Config(int retries)
        : max_retries_(retries) {}

private:
    const int max_retries_;
};
```

### Membres de type référence

Une référence doit être liée à un objet dès sa création. Elle ne peut pas être "réassignée" :

```cpp
class Logger {  
public:  
    // CORRECT — la référence est liée dans la liste d'initialisation
    explicit Logger(std::ostream& output)
        : output_(output) {}

    void log(const std::string& msg) {
        output_ << msg << "\n";
    }

private:
    std::ostream& output_;   // Référence — doit être initialisée, pas affectée
};

Logger console_logger(std::cout);  
console_logger.log("Application started");  
```

### Membres sans constructeur par défaut

Si un membre est d'un type qui n'a pas de constructeur par défaut, le compilateur ne peut pas le construire implicitement avant le corps. Il faut fournir les arguments dans la liste :

```cpp
class DatabasePool {  
public:  
    DatabasePool(const std::string& conn_str, int pool_size)
        : primary_(conn_str)          // Connection n'a pas de constructeur par défaut
        , pool_size_(pool_size) {}

private:
    Connection primary_;    // Pas de Connection() → liste d'initialisation obligatoire
    int pool_size_;
};
```

### Classe de base

L'initialisation de la classe de base se fait obligatoirement dans la liste d'initialisation. C'est le seul endroit où vous pouvez passer des arguments au constructeur du parent :

```cpp
class Animal {  
public:  
    Animal(const std::string& name, int legs)
        : name_(name), legs_(legs) {}
protected:
    std::string name_;
    int legs_;
};

class Dog : public Animal {  
public:  
    Dog(const std::string& name, const std::string& breed)
        : Animal(name, 4)         // Initialisation de la classe de base
        , breed_(breed) {}
private:
    std::string breed_;
};
```

En résumé, la liste d'initialisation est **obligatoire** pour les membres `const`, les références, les types sans constructeur par défaut, et les classes de base. Pour tout le reste, elle est **fortement recommandée** pour des raisons de performance et de clarté.

## Ordre d'initialisation : un piège classique

Les membres sont initialisés **dans l'ordre de leur déclaration dans la classe**, pas dans l'ordre de la liste d'initialisation. C'est une source de bugs subtils :

```cpp
class Buffer {  
public:  
    Buffer(std::size_t count)
        : size_(count * sizeof(int))    // 2e — initialisé en second
        , data_(new char[size_]) {}     // 1er — initialisé en premier !
                                        //       mais size_ n'existe pas encore → BUG

private:
    char* data_;           // Déclaré en premier → initialisé en premier
    std::size_t size_;     // Déclaré en second → initialisé en second
};
```

Dans cet exemple, `data_` est déclaré avant `size_` dans la classe. La liste d'initialisation donne l'impression que `size_` est calculé en premier, mais c'est faux. L'initialisation suit l'ordre de déclaration : `data_` est initialisé avec `new char[size_]` alors que `size_` contient encore une valeur indéterminée. Le comportement est indéfini.

La solution est simple — écrivez la liste d'initialisation **dans le même ordre** que la déclaration des membres :

```cpp
class Buffer {  
public:  
    Buffer(std::size_t count)
        : data_(new char[count * sizeof(int)])   // 1er — comme dans la déclaration
        , size_(count * sizeof(int)) {}           // 2e — comme dans la déclaration

private:
    char* data_;           // 1er
    std::size_t size_;     // 2e
};
```

Les compilateurs modernes émettent un avertissement quand l'ordre de la liste ne correspond pas à l'ordre de déclaration. Avec GCC et Clang, ce warning est activé par `-Wall` (flag `-Wreorder`). C'est l'une des raisons pour lesquelles nous recommandons `-Wall -Wextra -Wpedantic` depuis le début de la formation (section 2.6.1).

> 💡 **Astuce** — Si l'initialisation d'un membre dépend de la valeur d'un autre, déclarez le membre dont on dépend **avant** celui qui en dépend. L'ordre de déclaration dans la classe est votre contrat d'initialisation.

## Interaction avec les *default member initializers*

Depuis C++11, les membres peuvent avoir une valeur par défaut directement dans la déclaration (section 6.1). Comment cette valeur interagit-elle avec la liste d'initialisation ?

La règle est simple : **la liste d'initialisation a priorité**. Si un membre apparaît dans la liste, c'est la valeur de la liste qui est utilisée. Sinon, c'est le *default member initializer*.

```cpp
class Server {  
public:  
    // Constructeur 1 — spécifie tout
    Server(const std::string& host, uint16_t port, int timeout)
        : host_(host)
        , port_(port)
        , timeout_ms_(timeout) {}

    // Constructeur 2 — ne spécifie que l'host
    explicit Server(const std::string& host)
        : host_(host) {}
        // port_ → 8080 (default member initializer)
        // timeout_ms_ → 5000 (default member initializer)

private:
    std::string host_;
    uint16_t port_ = 8080;          // Valeur par défaut
    int timeout_ms_ = 5000;         // Valeur par défaut
};

Server s1("api.example.com", 443, 10000);   // host=api..., port=443, timeout=10000  
Server s2("localhost");                       // host=localhost, port=8080, timeout=5000  
```

Cette combinaison est très puissante. Les *default member initializers* servent de filet de sécurité : ils centralisent les valeurs par défaut à un seul endroit, et chaque constructeur ne mentionne dans sa liste que les membres qu'il veut **surcharger**. Si vous ajoutez un nouveau membre avec un *default member initializer*, aucun constructeur existant ne doit être modifié — la valeur par défaut s'applique automatiquement.

### Quelle stratégie adopter ?

La combinaison recommandée par les *C++ Core Guidelines* (C.45 et C.48) est la suivante :

- Donnez un *default member initializer* à **chaque membre** qui a une valeur par défaut naturelle.  
- Utilisez la liste d'initialisation uniquement pour les membres dont la valeur **dépend des arguments** du constructeur.  
- Utilisez `= default` pour le constructeur par défaut quand les *default member initializers* suffisent.

```cpp
class HttpClient {  
public:  
    HttpClient() = default;    // Tous les defaults suffisent

    explicit HttpClient(const std::string& base_url)
        : base_url_(base_url) {}

    HttpClient(const std::string& base_url, int timeout, int retries)
        : base_url_(base_url)
        , timeout_ms_(timeout)
        , max_retries_(retries) {}

private:
    std::string base_url_ = "http://localhost";
    int timeout_ms_ = 30000;
    int max_retries_ = 3;
    bool follow_redirects_ = true;
    std::string user_agent_ = "HttpClient/1.0";
};
```

Cinq membres, trois constructeurs, et les valeurs par défaut ne sont écrites qu'**une seule fois**. Si demain vous changez le timeout par défaut de 30000 à 60000, la modification est à un seul endroit.

## Parenthèses vs accolades dans la liste d'initialisation

La liste d'initialisation accepte les deux syntaxes :

```cpp
class Example {  
public:  
    Example(int size, const std::string& name)
        : size_(size)          // Parenthèses — initialisation directe
        , name_{name}          // Accolades — initialisation uniforme
        , data_(size, 0) {}    // Parenthèses — appelle vector(size_t, int)

private:
    int size_;
    std::string name_;
    std::vector<int> data_;
};
```

Le même piège qu'en section 6.2 s'applique : les accolades sont dirigées vers un constructeur `std::initializer_list` s'il existe. Dans la liste d'initialisation, cela impacte surtout les conteneurs standard :

```cpp
class Container {  
public:  
    Container()
        : v1_(5, 0)     // vector de 5 éléments valant 0 → [0, 0, 0, 0, 0]
        , v2_{5, 0} {}   // vector initialisé avec {5, 0} → [5, 0]

private:
    std::vector<int> v1_;
    std::vector<int> v2_;
};
```

Pour les types sans constructeur `std::initializer_list` (la grande majorité de vos propres classes), les deux syntaxes sont interchangeables. La recommandation pratique : utilisez les **parenthèses** dans la liste d'initialisation pour éviter toute ambiguïté, et réservez les accolades aux cas où vous voulez explicitement une *initializer list*.

## Fil conducteur : relecture de `DynArray`

Notre `DynArray` utilise déjà la liste d'initialisation dans chacun de ses constructeurs. Relisons-les à la lumière de cette section pour vérifier qu'ils suivent les bonnes pratiques :

```cpp
// Constructeur par défaut — = default utilise les default member initializers
// data_ = nullptr, size_ = 0
DynArray() = default;                                              // ✓

// Constructeur paramétré — liste d'initialisation
DynArray::DynArray(std::size_t size)
    : data_(size > 0 ? new int[size]{} : nullptr)                  // ✓ ordre = déclaration
    , size_(size) {}                                                // ✓ data_ puis size_

// Constructeur délégué — pas d'initialisation de membres en plus
DynArray::DynArray(std::size_t size, int value)
    : DynArray(size) {                                              // ✓ délégation
    std::fill_n(data_, size_, value);
}

// Constructeur de copie — liste d'initialisation + copie des données
DynArray::DynArray(const DynArray& other)
    : data_(other.size_ > 0 ? new int[other.size_] : nullptr)      // ✓
    , size_(other.size_) {                                          // ✓
    std::copy_n(other.data_, size_, data_);
}

// Constructeur de déplacement — liste d'initialisation + neutralisation
DynArray::DynArray(DynArray&& other) noexcept
    : data_(other.data_)                                            // ✓
    , size_(other.size_) {                                          // ✓
    other.data_ = nullptr;
    other.size_ = 0;
}
```

Tous respectent l'ordre de déclaration (`data_` avant `size_`). Tous utilisent la liste d'initialisation plutôt que l'affectation dans le corps. Les *default member initializers* (`nullptr` et `0`) servent de filet de sécurité pour le constructeur `= default` et pour tout futur constructeur que nous pourrions ajouter.

## Erreurs fréquentes à éviter

### Confondre `()` et `=` dans le corps

```cpp
class Bad {  
public:  
    Bad(int x) {
        value_(x);     // ERREUR — ceci est un appel de fonction, pas une initialisation
        // Le compilateur cherche une méthode value_() et ne la trouve pas
    }
private:
    int value_;
};
```

La syntaxe `membre_(valeur)` n'est valide que dans la liste d'initialisation, pas dans le corps du constructeur. Dans le corps, c'est `membre_ = valeur;` (affectation).

### Oublier un membre dans la liste

```cpp
class Partial {  
public:  
    Partial(int a, int b)
        : alpha_(a) {}      // beta_ n'est pas dans la liste et n'a pas de default initializer
                             // → beta_ contient une valeur indéterminée
private:
    int alpha_;
    int beta_;               // Oublié ! Pas de default member initializer non plus
};
```

Le compilateur ne signale pas forcément cet oubli par défaut. Avec GCC et Clang, activez `-Weffc++` pour obtenir un avertissement lorsqu'un membre n'est pas initialisé dans la liste d'initialisation. Mieux encore, ajoutez systématiquement des *default member initializers* pour les types primitifs — l'oubli dans la liste n'aura alors plus de conséquence.

### Utiliser un membre avant qu'il ne soit initialisé

```cpp
class Wrong {  
public:  
    Wrong(int n)
        : total_(n * unit_size_)    // BUG — unit_size_ n'est pas encore initialisé
        , unit_size_(64) {}         //       car total_ est déclaré avant unit_size_

private:
    int total_;                     // Initialisé en premier (déclaré en premier)
    int unit_size_;                 // Initialisé en second
};
```

La solution : réordonnez les déclarations pour que les dépendances soient satisfaites, ou calculez dans le corps du constructeur :

```cpp
class Correct {  
public:  
    Correct(int n)
        : unit_size_(64)              // Initialisé en premier
        , total_(n * unit_size_) {}   // Peut utiliser unit_size_ en toute sécurité

private:
    int unit_size_;                   // Déclaré en premier cette fois
    int total_;                       // Déclaré en second
};
```

---

## Points clés à retenir

- La liste d'initialisation **initialise** les membres directement avec leur valeur finale. L'affectation dans le corps **construit par défaut** puis **écrase** — c'est un travail en double.  
- La liste est **obligatoire** pour les membres `const`, les références, les types sans constructeur par défaut, et l'initialisation de la classe de base.  
- Les membres sont initialisés dans l'**ordre de déclaration** dans la classe, pas dans l'ordre de la liste. Écrivez toujours la liste dans le même ordre que les déclarations, et activez `-Wall` pour que le compilateur vous avertisse d'un désaccord.  
- Combinez *default member initializers* et liste d'initialisation : les defaults centralisent les valeurs communes, la liste surcharge au cas par cas. C'est la stratégie recommandée par les *Core Guidelines*.  
- Préférez les **parenthèses** dans la liste d'initialisation pour éviter le piège des `std::initializer_list` avec les conteneurs standard.  
- Si un membre dépend de la valeur d'un autre, déclarez-le **après** sa dépendance dans la classe.
