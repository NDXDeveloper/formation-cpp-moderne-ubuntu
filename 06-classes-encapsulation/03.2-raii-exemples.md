# 6.3.2 — Exemples pratiques de RAII

## Chapitre 6 : Classes et Encapsulation

---

## Ce que vous allez apprendre

- Appliquer le pattern RAII à des ressources variées : mémoire, fichiers, verrous, timers, répertoires temporaires.  
- Reconnaître la structure récurrente du pattern, quelle que soit la ressource encapsulée.  
- Comprendre comment chaque wrapper élimine une catégorie de bugs spécifique.  
- Apprécier pourquoi le C++ moderne privilégie la composition de wrappers RAII plutôt que la gestion manuelle.

---

## La structure commune

Avant de plonger dans les exemples, rappelons le squelette que chaque wrapper RAII respecte. Quel que soit le type de ressource, la structure est identique :

```cpp
class RaiiWrapper {  
public:  
    // 1. Acquisition dans le constructeur — exception si échec
    explicit RaiiWrapper(/* paramètres d'acquisition */) {
        resource_ = acquire(/* ... */);
        if (!resource_) throw std::runtime_error("Acquisition failed");
    }

    // 2. Libération dans le destructeur — jamais d'exception
    ~RaiiWrapper() {
        if (resource_) release(resource_);
    }

    // 3. Copie interdite ou explicitement définie
    RaiiWrapper(const RaiiWrapper&) = delete;
    RaiiWrapper& operator=(const RaiiWrapper&) = delete;

    // 4. Déplacement autorisé (transfert de propriété)
    RaiiWrapper(RaiiWrapper&& other) noexcept;
    RaiiWrapper& operator=(RaiiWrapper&& other) noexcept;

    // 5. Accès contrôlé à la ressource
    ResourceType get() const { return resource_; }

private:
    ResourceType resource_ = INVALID_VALUE;
};
```

Chaque exemple qui suit est une instanciation de ce squelette pour une ressource différente. La répétition est voulue : elle montre que le RAII est un **pattern universel**, pas une technique spécifique à la mémoire.

## Exemple 1 : Wrapper de descripteur de fichier POSIX

Le cas le plus classique en programmation système Linux. L'API POSIX utilise des entiers (`int`) comme descripteurs de fichier. Un descripteur ouvert doit être fermé — sinon le processus finit par épuiser sa table de descripteurs.

```cpp
#include <fcntl.h>
#include <unistd.h>
#include <stdexcept>
#include <string>

class FileDescriptor {  
public:  
    explicit FileDescriptor(const std::string& path, int flags = O_RDONLY)
        : fd_(::open(path.c_str(), flags)) {
        if (fd_ < 0) {
            throw std::runtime_error("Cannot open: " + path);
        }
    }

    ~FileDescriptor() {
        if (fd_ >= 0) ::close(fd_);
    }

    // Copie interdite — un fd ne se duplique pas implicitement
    FileDescriptor(const FileDescriptor&) = delete;
    FileDescriptor& operator=(const FileDescriptor&) = delete;

    // Déplacement — transfert de propriété
    FileDescriptor(FileDescriptor&& other) noexcept
        : fd_(other.fd_) {
        other.fd_ = -1;    // Neutralisation : -1 = invalide
    }

    FileDescriptor& operator=(FileDescriptor&& other) noexcept {
        if (this != &other) {
            if (fd_ >= 0) ::close(fd_);   // Libérer l'ancienne ressource
            fd_ = other.fd_;
            other.fd_ = -1;
        }
        return *this;
    }

    // Accès au descripteur brut
    int fd() const { return fd_; }

    // Opérations de haut niveau
    std::size_t read(void* buf, std::size_t count) const {
        ssize_t n = ::read(fd_, buf, count);
        if (n < 0) throw std::runtime_error("Read failed");
        return static_cast<std::size_t>(n);
    }

private:
    int fd_ = -1;
};
```

### Ce que ce wrapper élimine

Sans le wrapper, le code POSIX typique ressemble à ceci :

```cpp
// Sans RAII — chaque chemin de sortie doit fermer le fichier
void process_file(const char* path) {
    int fd = open(path, O_RDONLY);
    if (fd < 0) return;

    char buf[1024];
    ssize_t n = read(fd, buf, sizeof(buf));
    if (n < 0) {
        close(fd);          // Facile à oublier
        return;
    }

    do_something(buf, n);   // Si ça lance une exception → fd fuit

    close(fd);
}
```

Avec le wrapper RAII :

```cpp
void process_file(const std::string& path) {
    FileDescriptor fd(path);                  // Ouvre ou lance
    char buf[1024];
    auto n = fd.read(buf, sizeof(buf));       // Lit ou lance
    do_something(buf, n);                     // Si ça lance → fd fermé automatiquement
}   // fd fermé automatiquement
```

Trois points de sortie possibles (exception dans le constructeur, exception dans `read`, retour normal), et le descripteur est fermé dans les trois cas. Zéro `close()` explicite dans le code client.

## Exemple 2 : Verrou mutex avec portée (*scoped lock*)

La synchronisation entre threads repose sur des verrous. Un verrou acquis mais jamais relâché bloque tous les autres threads indéfiniment — c'est un **deadlock**. Le RAII élimine ce risque.

```cpp
#include <mutex>
#include <iostream>

class ScopedLock {  
public:  
    explicit ScopedLock(std::mutex& mutex)
        : mutex_(mutex) {
        mutex_.lock();     // Acquisition
    }

    ~ScopedLock() {
        mutex_.unlock();   // Libération — toujours, même si exception
    }

    // Ni copiable ni déplaçable — le verrou est lié à cette portée
    ScopedLock(const ScopedLock&) = delete;
    ScopedLock& operator=(const ScopedLock&) = delete;
    ScopedLock(ScopedLock&&) = delete;
    ScopedLock& operator=(ScopedLock&&) = delete;

private:
    std::mutex& mutex_;    // Référence — pas de propriété sur le mutex
};
```

### Utilisation et comparaison

```cpp
std::mutex g_mutex;  
std::vector<std::string> g_log;  

// Sans RAII — deadlock si process() lance une exception
void unsafe_log(const std::string& msg) {
    g_mutex.lock();
    g_log.push_back(msg);     // Si push_back lance (std::bad_alloc) → deadlock
    g_mutex.unlock();
}

// Avec RAII — le mutex est toujours déverrouillé
void safe_log(const std::string& msg) {
    ScopedLock lock(g_mutex);
    g_log.push_back(msg);     // Si push_back lance → lock détruit → mutex déverrouillé
}
```

> 💡 Cet exemple est pédagogique. En production, utilisez `std::lock_guard` ou `std::scoped_lock` (C++17), qui font exactement la même chose. Nous les étudierons au chapitre 21 (Threads et Programmation Concurrente). Le but ici est de montrer que le pattern est simple à implémenter soi-même — il n'y a rien de magique dans la bibliothèque standard.

## Exemple 3 : Timer de profiling automatique

Le RAII ne se limite pas aux ressources système. Tout ce qui suit le pattern "acquérir au début, libérer à la fin" peut bénéficier du RAII. Un timer qui mesure la durée d'un bloc de code en est un excellent exemple.

```cpp
#include <chrono>
#include <string>
#include <iostream>

class ScopedTimer {  
public:  
    explicit ScopedTimer(const std::string& label)
        : label_(label)
        , start_(std::chrono::steady_clock::now()) {}

    ~ScopedTimer() {
        auto end = std::chrono::steady_clock::now();
        auto elapsed = std::chrono::duration_cast<std::chrono::microseconds>(
            end - start_
        );
        std::cerr << "[TIMER] " << label_ << ": "
                  << elapsed.count() << " µs\n";
    }

    // Ni copiable ni déplaçable — lié à une portée précise
    ScopedTimer(const ScopedTimer&) = delete;
    ScopedTimer& operator=(const ScopedTimer&) = delete;

private:
    std::string label_;
    std::chrono::steady_clock::time_point start_;
};
```

### Utilisation

```cpp
void sort_data(std::vector<int>& data) {
    ScopedTimer timer("sort_data");          // Le chrono démarre
    std::sort(data.begin(), data.end());
}                                             // Le chrono s'arrête, affiche le résultat
// Sortie : [TIMER] sort_data: 4230 µs
```

Ici, la "ressource" est le temps. Le constructeur capture l'instant de départ, le destructeur calcule et affiche la durée écoulée. Le pattern RAII garantit que la mesure est effectuée même si `std::sort` lançait une exception (ce qui est peu probable, mais le principe reste valide).

Ce pattern est très courant en pratique pour le profiling rapide. Il se combine naturellement avec des blocs de portée pour mesurer des sections précises :

```cpp
void complex_pipeline(const std::vector<int>& input) {
    std::vector<int> data;

    {
        ScopedTimer t("copy");
        data = input;
    }   // Affiche la durée de la copie

    {
        ScopedTimer t("sort");
        std::sort(data.begin(), data.end());
    }   // Affiche la durée du tri

    {
        ScopedTimer t("deduplicate");
        auto last = std::unique(data.begin(), data.end());
        data.erase(last, data.end());
    }   // Affiche la durée de la déduplication
}
```

## Exemple 4 : Répertoire temporaire avec nettoyage automatique

Un cas fréquent en DevOps et en CI/CD : créer un répertoire temporaire pour une opération, puis le supprimer quand on a fini — même si l'opération échoue.

```cpp
#include <filesystem>
#include <stdexcept>

namespace fs = std::filesystem;

class TempDirectory {  
public:  
    explicit TempDirectory(const std::string& prefix = "tmp") {
        path_ = fs::temp_directory_path() / (prefix + "_XXXXXX");
        // Créer un nom unique et le répertoire
        path_ = fs::temp_directory_path() / 
                (prefix + "_" + std::to_string(
                    std::chrono::steady_clock::now().time_since_epoch().count()));
        
        if (!fs::create_directories(path_)) {
            throw std::runtime_error("Cannot create temp dir: " + path_.string());
        }
    }

    ~TempDirectory() {
        std::error_code ec;
        fs::remove_all(path_, ec);   // Supprime récursivement
        // ec ignoré volontairement — le destructeur ne lance jamais
    }

    // Copie interdite — le répertoire est unique
    TempDirectory(const TempDirectory&) = delete;
    TempDirectory& operator=(const TempDirectory&) = delete;

    // Déplacement — transfert de propriété
    TempDirectory(TempDirectory&& other) noexcept
        : path_(std::move(other.path_)) {
        other.path_.clear();   // Neutralisation
    }

    TempDirectory& operator=(TempDirectory&& other) noexcept {
        if (this != &other) {
            // Nettoyer l'ancien répertoire
            std::error_code ec;
            if (!path_.empty()) fs::remove_all(path_, ec);
            path_ = std::move(other.path_);
            other.path_.clear();
        }
        return *this;
    }

    const fs::path& path() const { return path_; }

private:
    fs::path path_;
};
```

### Utilisation

```cpp
void run_build() {
    TempDirectory build_dir("build");

    // Copier les sources, compiler, tester...
    fs::copy("src/main.cpp", build_dir.path() / "main.cpp");
    compile(build_dir.path());
    run_tests(build_dir.path());

    // Si compile() ou run_tests() lance → build_dir supprimé automatiquement
}   // build_dir supprimé automatiquement dans tous les cas
```

Notez le destructeur : il utilise la surcharge de `fs::remove_all` qui prend un `std::error_code` au lieu de lancer une exception. C'est la technique standard pour éviter les exceptions dans un destructeur — on traite l'erreur silencieusement (ici, on l'ignore, car le nettoyage d'un répertoire temporaire est un *best effort*).

## Exemple 5 : Restauration d'état (*scope guard*)

Un pattern élégant : restaurer automatiquement un état précédent en fin de portée. C'est utile pour les modifications temporaires de configuration, de locale, de flags, etc.

```cpp
class CoutFlagsGuard {  
public:  
    explicit CoutFlagsGuard(std::ostream& os = std::cout)
        : os_(os)
        , saved_flags_(os.flags())
        , saved_precision_(os.precision())
        , saved_fill_(os.fill()) {}

    ~CoutFlagsGuard() {
        os_.flags(saved_flags_);
        os_.precision(saved_precision_);
        os_.fill(saved_fill_);
    }

    CoutFlagsGuard(const CoutFlagsGuard&) = delete;
    CoutFlagsGuard& operator=(const CoutFlagsGuard&) = delete;

private:
    std::ostream& os_;
    std::ios_base::fmtflags saved_flags_;
    std::streamsize saved_precision_;
    char saved_fill_;
};
```

### Utilisation

```cpp
void print_hex_table(const std::vector<int>& values) {
    CoutFlagsGuard guard;   // Sauvegarde l'état actuel de std::cout

    std::cout << std::hex << std::uppercase << std::setfill('0');
    for (int v : values) {
        std::cout << "0x" << std::setw(4) << v << "\n";
    }
}   // guard restaure les flags — l'appelant retrouve std::cout intact

void caller() {
    std::cout << 255 << "\n";           // Affiche "255" (décimal)
    print_hex_table({10, 255, 4096});   // Affiche en hexa
    std::cout << 255 << "\n";           // Affiche "255" (décimal) — pas "FF" !
}
```

Sans le guard, les modifications de formatage de `std::cout` persistent après l'appel — un bug classique et difficile à traquer. Le RAII garantit que l'état est restauré, même si une exception est lancée au milieu de l'affichage.

Ici, la "ressource" n'est pas une allocation ou un handle — c'est un **état** qui doit être restauré. Le RAII s'applique avec la même élégance.

## Généralisation : le *scope guard*

Les exemples 3 (timer) et 5 (restauration de flags) suivent un pattern plus général : exécuter une action à la fin d'une portée, quoi qu'il arrive. Ce pattern est parfois appelé **scope guard** et peut être généralisé avec une lambda :

```cpp
class ScopeGuard {  
public:  
    explicit ScopeGuard(std::function<void()> on_exit)
        : on_exit_(std::move(on_exit)) {}

    ~ScopeGuard() {
        if (on_exit_) on_exit_();
    }

    ScopeGuard(const ScopeGuard&) = delete;
    ScopeGuard& operator=(const ScopeGuard&) = delete;

    // Permet d'annuler l'action si tout s'est bien passé
    void dismiss() { on_exit_ = nullptr; }

private:
    std::function<void()> on_exit_;
};
```

### Utilisation

```cpp
void transactional_operation() {
    begin_transaction();

    ScopeGuard rollback([&]() {
        rollback_transaction();   // Annule si on sort sans commit
    });

    step_one();       // Si ça lance → rollback automatique
    step_two();       // Si ça lance → rollback automatique
    step_three();     // Si ça lance → rollback automatique

    commit_transaction();
    rollback.dismiss();   // Tout a réussi — on annule le rollback
}
```

La méthode `dismiss()` est le détail qui rend ce pattern transactionnel : le rollback ne se déclenche que si on n'atteint pas le `commit`. C'est du RAII appliqué à une **logique métier**, pas à une ressource système.

> 💡 La bibliothèque standard ne fournit pas de `ScopeGuard` générique, mais de nombreuses bibliothèques tierces le proposent (Boost.ScopeExit, la GSL de Microsoft). Des proposals existent pour l'ajouter au standard dans une future version.

## Le fil rouge : notre `DynArray` est un wrapper RAII

Avec le recul de cette section, notre classe `DynArray` est elle-même un wrapper RAII. Vérifions qu'elle respecte la structure commune :

| Règle | `DynArray` | Statut |  
|---|---|---|  
| Acquisition dans le constructeur | `new int[size]{}` | ✅ |  
| Libération dans le destructeur | `delete[] data_` | ✅ |  
| Exception si acquisition échoue | `new` lance `std::bad_alloc` | ✅ |  
| Destructeur ne lance pas | `delete[]` ne lance jamais | ✅ |  
| Copie définie (profonde) | Section 6.2.3 | ✅ |  
| Déplacement défini (transfert) | Section 6.2.4 | ✅ |  
| Accès contrôlé | `operator[]`, `size()` | ✅ |  
| Une seule ressource | Un bloc `int*` | ✅ |

`DynArray` est un `std::vector<int>` simplifié — et `std::vector` lui-même est un wrapper RAII autour d'un bloc mémoire dynamique. La bibliothèque standard n'est pas magique : elle est construite exactement avec les techniques que vous apprenez dans ce chapitre.

## Points clés à retenir

- Le pattern RAII est **universel** : mémoire, fichiers, verrous, timers, répertoires temporaires, état de formatage, transactions logiques — tout ce qui suit le cycle "acquérir / utiliser / libérer" bénéficie du RAII.  
- La structure est toujours la même : acquisition dans le constructeur, libération dans le destructeur, copie interdite ou profonde, déplacement par transfert de propriété, accès contrôlé via une méthode.  
- Pour éviter les exceptions dans les destructeurs, utilisez les variantes de fonctions qui retournent des codes d'erreur (comme `fs::remove_all(path, ec)`) au lieu de celles qui lancent.  
- Le **scope guard** généralise le RAII à toute action de fin de portée, y compris la logique métier (rollback de transaction, restauration d'état).  
- La bibliothèque standard fournit déjà des wrappers RAII pour les ressources les plus courantes (`vector`, `string`, `fstream`, `unique_ptr`, `lock_guard`). Avant d'écrire votre propre wrapper, vérifiez qu'il n'en existe pas déjà un.  
- Quand vous devez encapsuler une API C ou système, écrivez un wrapper RAII minimal. Le code client ne devrait jamais manipuler de ressources brutes directement.
