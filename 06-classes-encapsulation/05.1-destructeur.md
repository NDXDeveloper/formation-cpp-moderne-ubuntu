# 6.5.1 — Destructeur

## Chapitre 6 : Classes et Encapsulation — Section 6.5 : Règle des 5

---

## Objectif de cette sous-section

Le destructeur a été présenté en détail en section 6.3 (principes, RAII, exemples). Cette sous-section le **revisite sous l'angle de la Règle des 5** : son rôle de déclencheur de la règle, sa cohérence avec les quatre autres opérations, et les pièges spécifiques qui apparaissent quand on l'envisage comme membre d'un ensemble de cinq.

---

## Le destructeur comme signal déclencheur

La Règle des 5 commence presque toujours par le destructeur. C'est le symptôme le plus visible : si vous avez besoin d'écrire `delete`, `close`, `unlock`, ou toute autre forme de libération dans le destructeur, cela signifie que votre classe possède une ressource. Et si elle possède une ressource, les quatre autres opérations implicites sont probablement incorrectes.

C'est pourquoi le destructeur est la **sentinelle** de la Règle des 5. Avant d'écrire le corps d'un destructeur, posez-vous la question : *est-ce que les constructeurs de copie et de déplacement, ainsi que les opérateurs d'affectation, gèrent correctement cette ressource ?*

```cpp
// Le simple fait d'écrire ce destructeur déclenche la Règle des 5
DynArray::~DynArray() {
    delete[] data_;
}
// → Ai-je un constructeur de copie qui fait une copie profonde ?
// → Ai-je un opérateur d'affectation par copie qui libère l'ancien bloc ?
// → Ai-je un constructeur de déplacement qui neutralise la source ?
// → Ai-je un opérateur d'affectation par déplacement qui libère l'ancien bloc ?
```

Si la réponse à l'une de ces questions est "non", il y a un bug latent.

---

## Impact sur la génération implicite

Rappelons la ligne correspondante de la matrice (section 6.5) :

| Vous déclarez… | Destructeur | Copie ctor | Copie `=` | Move ctor | Move `=` |  
|---|---|---|---|---|---|  
| **Destructeur** | — | ✅ généré ⚠️ | ✅ généré ⚠️ | ❌ supprimé | ❌ supprimé |

Déclarer un destructeur a deux effets :

**Les opérations de déplacement sont supprimées.** Le compilateur ne génère ni constructeur de déplacement ni opérateur d'affectation par déplacement. Si vous essayez de déplacer un objet, le compilateur se rabat sur la copie (si elle existe) — ce qui est correct mais potentiellement beaucoup plus lent.

**Les opérations de copie sont conservées, mais dépréciées.** Le compilateur les génère encore par compatibilité avec le C++98, mais le standard marque cette génération comme dépréciée depuis C++11. En pratique, ces opérations implicites font une copie superficielle — exactement le bug que nous avons identifié en section 6.2.3.

Illustrons ce piège avec une classe qui déclare *uniquement* un destructeur :

```cpp
class HalfBaked {  
public:  
    explicit HalfBaked(std::size_t n)
        : data_(new int[n]{}), size_(n) {}

    ~HalfBaked() { delete[] data_; }

    // Rien d'autre déclaré.
    // Le compilateur génère :
    //   - Copie ctor (shallow copy)     ⚠️ BUG — double free
    //   - Copie = (shallow copy)        ⚠️ BUG — double free + fuite
    //   - Move ctor                     ❌ supprimé
    //   - Move =                        ❌ supprimé

private:
    int* data_;
    std::size_t size_;
};

HalfBaked a(100);  
HalfBaked b = a;       // Compile ! Copie superficielle → double free à la destruction  
HalfBaked c(50);  
c = a;                 // Compile ! Copie superficielle → fuite + double free  

HalfBaked d = std::move(a);  // Move supprimé → se rabat sur la copie ⚠️
                              // Pas d'erreur de compilation, mais copie au lieu de move
```

C'est le pire des cas : le code compile sans avertissement, mais chaque copie est un bug silencieux. C'est pour prévenir exactement cette situation que la Règle des 5 existe.

## Propriétés requises du destructeur dans la Règle des 5

Quand le destructeur fait partie d'un ensemble de cinq opérations, il doit respecter des contraintes supplémentaires par rapport à un destructeur isolé :

### Gérer l'état *moved-from*

Après un déplacement, l'objet source est dans un état vidé (section 6.2.4). Le destructeur doit gérer cet état sans erreur. Pour `DynArray`, cela signifie que `delete[] nullptr` doit être sûr — et c'est le cas, par garantie du standard :

```cpp
DynArray a(100);  
DynArray b = std::move(a);   // a.data_ = nullptr, a.size_ = 0  

// Plus tard, a est détruit :
// ~DynArray() appelle delete[] nullptr → no-op, aucun problème
```

Si votre neutralisation *moved-from* laisse la ressource dans un état différent (par exemple un descripteur de fichier à `-1`), le destructeur doit vérifier cet état :

```cpp
~FileDescriptor() {
    if (fd_ >= 0) ::close(fd_);   // Ne ferme pas si fd_ == -1 (moved-from)
}
```

**Règle** : le destructeur doit fonctionner correctement pour **tous** les états valides de l'objet — construit par défaut, construit avec des paramètres, copié, et vidé par un déplacement.

### Gérer l'état après auto-affectation interrompue

L'opérateur d'affectation par copie (section 6.5.3) peut échouer au milieu de l'opération — par exemple si l'allocation du nouveau bloc lance `std::bad_alloc`. Selon l'implémentation choisie, l'objet peut se retrouver dans un état partiellement modifié. Le destructeur doit pouvoir nettoyer cet état sans problème. En pratique, cela revient à la même règle : **le destructeur doit gérer tout état valide**, y compris un état vide.

### Ne jamais lancer d'exception

Ce point a été abordé en section 6.3, mais il prend une importance particulière dans le contexte de la Règle des 5. Les opérations d'affectation peuvent lancer des exceptions (lors de l'allocation, par exemple). Si l'exception se propage et que le *stack unwinding* détruit l'objet, le destructeur doit s'exécuter sans lancer à son tour. Deux exceptions actives simultanément provoquent l'appel à `std::terminate`.

Depuis C++11, les destructeurs sont **implicitement `noexcept`**. Si vous écrivez un destructeur qui lance, le compilateur le marque quand même `noexcept`, et toute exception lancée appelle directement `std::terminate`. C'est une sécurité du langage, pas une excuse pour ignorer le problème.

## Relecture du destructeur de `DynArray`

Vérifions que notre destructeur est cohérent avec les quatre autres opérations :

```cpp
DynArray::~DynArray() {
    delete[] data_;
}
```

| Vérification | Résultat |  
|---|---|  
| Libère la ressource acquise par le constructeur (`new int[]`) | ✅ `delete[] data_` |  
| Gère l'état construit par défaut (`data_ == nullptr`) | ✅ `delete[] nullptr` est un no-op |  
| Gère l'état *moved-from* (`data_ == nullptr`) | ✅ Même cas |  
| Ne lance pas d'exception | ✅ `delete[]` ne lance jamais |  
| Implicitement `noexcept` | ✅ Par défaut depuis C++11 |

Le destructeur est minimal — une seule ligne. C'est le signe d'un bon design : la ressource est simple (un seul bloc mémoire), l'état invalide est représenté par `nullptr`, et `delete[]` gère uniformément les deux cas (bloc valide et `nullptr`).

### Et si la ressource était plus complexe ?

Pour une classe gérant plusieurs ressources ou un état plus riche, le destructeur doit nettoyer dans l'**ordre inverse d'acquisition** et vérifier chaque ressource individuellement :

```cpp
class MultiResource {  
public:  
    MultiResource(const std::string& db_conn, const std::string& log_path)
        : db_(db_connect(db_conn.c_str()))
        , log_fd_(::open(log_path.c_str(), O_WRONLY | O_CREAT, 0644)) {
        if (!db_) throw std::runtime_error("DB connection failed");
        if (log_fd_ < 0) {
            db_disconnect(db_);   // Nettoyer la première ressource
            throw std::runtime_error("Cannot open log file");
        }
    }

    ~MultiResource() {
        // Ordre inverse d'acquisition
        if (log_fd_ >= 0) ::close(log_fd_);
        if (db_) db_disconnect(db_);
    }

    // ... les 4 autres opérations ...

private:
    db_handle_t db_ = nullptr;
    int log_fd_ = -1;
};
```

Ce destructeur vérifie chaque ressource avant de la libérer. Si le constructeur a lancé après l'acquisition de `db_` mais avant l'ouverture de `log_fd_`, le destructeur n'est **pas appelé** (le standard ne détruit pas les objets dont le constructeur a échoué). C'est le constructeur lui-même qui doit nettoyer dans ce cas — d'où le `db_disconnect(db_)` dans le bloc d'erreur du constructeur.

C'est exactement le genre de complexité que la Règle du Zéro élimine. Avec des wrappers RAII individuels, chaque membre se détruit lui-même :

```cpp
class ModernMultiResource {  
public:  
    ModernMultiResource(const std::string& db_conn, const std::string& log_path)
        : db_(db_conn)             // DbConnection RAII wrapper
        , log_(log_path) {}        // FileHandle RAII wrapper

    // Aucun destructeur nécessaire — chaque membre gère sa destruction
    // Si le constructeur de log_ lance, db_ est automatiquement détruit

private:
    DbConnection db_;
    FileHandle log_;
};
```

## Destructeur et héritage : `virtual`

Un point qui anticipe le chapitre 7 mais qui est directement lié à la Règle des 5 : si votre classe est destinée à être utilisée comme **classe de base polymorphique** (c'est-à-dire manipulée via un pointeur ou une référence vers la base), le destructeur doit être `virtual`.

```cpp
class Base {  
public:  
    virtual ~Base() = default;   // Destructeur virtuel
};

class Derived : public Base {  
public:  
    ~Derived() override {
        // Nettoyage spécifique à Derived
    }
};

// Utilisation polymorphique
std::unique_ptr<Base> ptr = std::make_unique<Derived>();
// À la destruction, ~Derived() est appelé grâce au virtual → puis ~Base()
```

Sans le `virtual`, `delete` sur un pointeur `Base*` qui pointe vers un `Derived` n'appelle que `~Base()` — le destructeur de `Derived` n'est jamais exécuté, et ses ressources fuient. C'est un comportement indéfini.

**Règle** : si votre classe a au moins une méthode `virtual`, son destructeur doit être `virtual`. Inversement, si votre classe n'est pas destinée à l'héritage polymorphique, ne rendez pas le destructeur `virtual` inutilement — cela ajoute le coût d'un pointeur de vtable par instance.

Notez que déclarer `virtual ~Base() = default;` **compte comme une déclaration de destructeur** au sens de la Règle des 5. Cela supprime la génération implicite des opérations de déplacement. Si votre classe de base doit être déplaçable, déclarez explicitement les cinq opérations :

```cpp
class Base {  
public:  
    virtual ~Base() = default;
    Base(const Base&) = default;
    Base& operator=(const Base&) = default;
    Base(Base&&) noexcept = default;
    Base& operator=(Base&&) noexcept = default;
};
```

C'est un cas courant où la Règle des 5 avec `= default` est la bonne approche, même si la classe ne gère aucune ressource directement.

---

## Points clés à retenir

- Le destructeur est le **déclencheur** de la Règle des 5. Si vous en écrivez un, considérez immédiatement les quatre autres opérations.  
- Déclarer un destructeur **supprime** la génération implicite des opérations de déplacement et **déprécie** celle des opérations de copie. Le code compile, mais les copies implicites sont presque certainement des bugs.  
- Le destructeur doit gérer correctement **tous les états valides** de l'objet : construit normalement, construit par défaut, et vidé par un déplacement (*moved-from*).  
- Le destructeur ne doit **jamais lancer** d'exception. Depuis C++11, il est implicitement `noexcept`.  
- Pour les classes de base polymorphiques, le destructeur doit être **`virtual`** — et cette déclaration déclenche elle aussi la Règle des 5.  
- Les classes gérant plusieurs ressources brutes directement sont complexes et fragiles. Préférez la **composition de wrappers RAII** (Règle du Zéro) pour que chaque membre gère sa propre destruction.
