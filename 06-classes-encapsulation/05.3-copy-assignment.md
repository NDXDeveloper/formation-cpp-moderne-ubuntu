# 6.5.3 — Opérateur d'affectation par copie

## Chapitre 6 : Classes et Encapsulation — Section 6.5 : Règle des 5

---

## Ce que vous allez apprendre

- Écrire un opérateur d'affectation par copie correct et robuste.  
- Comprendre pourquoi l'affectation est plus complexe que la construction.  
- Identifier et résoudre le piège de l'**auto-affectation**.  
- Appliquer l'idiome **copy-and-swap** pour obtenir une garantie forte d'exception en une seule technique.  
- Vérifier la cohérence de l'opérateur avec les quatre autres opérations spéciales.

---

## Pourquoi l'affectation est plus complexe que la construction

Le constructeur de copie crée un objet **à partir de rien** — il n'y a pas d'état précédent à gérer. L'opérateur d'affectation par copie remplace le contenu d'un objet **qui existe déjà**. Cette différence a trois conséquences directes :

**L'ancienne ressource doit être libérée.** Avant de copier la nouvelle valeur, il faut libérer ce que l'objet possède actuellement. Oublier cette libération provoque une fuite mémoire.

**L'auto-affectation doit être gérée.** L'expression `a = a;` est valide en C++. Si l'opérateur libère la ressource avant de la copier, et que la source est le même objet, on se retrouve à lire une ressource déjà libérée.

**L'opérateur retourne une référence.** Par convention (et pour permettre le chaînage `a = b = c`), l'opérateur d'affectation retourne `*this`.

---

## Signature

```cpp
class DynArray {  
public:  
    DynArray& operator=(const DynArray& other);
    // ...
};
```

La signature est toujours la même : paramètre `const T&` (on ne modifie pas la source), type de retour `T&` (référence vers l'objet courant pour le chaînage). L'opérateur est une fonction membre — l'opérande gauche est `*this`, l'opérande droit est `other`.

```cpp
DynArray a(3, 42);  
DynArray b(5, 0);  

b = a;          // Appelle b.operator=(a)
                // this = &b, other = a
```

---

## Approche naïve : les trois pièges

Commençons par une implémentation naïve pour identifier les problèmes :

```cpp
// ⚠️ IMPLÉMENTATION NAÏVE — trois bugs potentiels
DynArray& DynArray::operator=(const DynArray& other) {
    delete[] data_;                                      // 1. Libérer l'ancien
    data_ = new int[other.size_];                        // 2. Allouer le nouveau
    std::copy_n(other.data_, other.size_, data_);        // 3. Copier les données
    size_ = other.size_;
    return *this;
}
```

Cette implémentation souffre de trois problèmes.

### Piège 1 : l'auto-affectation

```cpp
DynArray a(3, 42);  
a = a;   // Auto-affectation — que se passe-t-il ?  
```

Déroulons l'exécution : `delete[] data_` libère le bloc de `a`. Puis `new int[other.size_]` alloue un nouveau bloc. Puis `std::copy_n(other.data_, ...)` tente de lire depuis `other.data_` — qui est `a.data_` — le pointeur qu'on vient de `delete[]`. C'est une lecture après libération (*use-after-free*), un comportement indéfini.

L'auto-affectation semble artificielle, mais elle apparaît dans du code réel dès que des alias existent :

```cpp
std::vector<DynArray> arrays = { DynArray(3), DynArray(5) };  
arrays[i] = arrays[j];   // Si i == j → auto-affectation  
```

### Piège 2 : fuite si `new` lance

Si `new int[other.size_]` lance `std::bad_alloc`, l'ancien bloc a déjà été libéré par `delete[] data_`. L'objet se retrouve dans un état corrompu : `data_` est un pointeur pendouillant (*dangling pointer*), et `size_` indique encore l'ancienne taille. Le destructeur appellera `delete[]` sur un pointeur invalide.

### Piège 3 : pas de gestion du cas `other.size_ == 0`

Si `other` est vide, `new int[0]` a un comportement défini (il retourne un pointeur non nul ou nul selon l'implémentation), mais le résultat n'est pas cohérent avec notre invariant qui impose `data_ == nullptr` quand `size_ == 0`.

---

## Approche correcte : allouer d'abord, libérer ensuite

La clé pour résoudre les pièges 1 et 2 est d'**inverser l'ordre** : allouer le nouveau bloc *avant* de libérer l'ancien. Ainsi, si l'allocation échoue, l'objet reste intact :

```cpp
DynArray& DynArray::operator=(const DynArray& other) {
    if (this == &other) return *this;   // Protection contre l'auto-affectation

    // 1. Préparer la nouvelle ressource (peut lancer)
    int* new_data = nullptr;
    if (other.size_ > 0) {
        new_data = new int[other.size_];           // Si ça lance, *this est intact
        std::copy_n(other.data_, other.size_, new_data);
    }

    // 2. Libérer l'ancienne ressource (ne lance pas)
    delete[] data_;

    // 3. Installer la nouvelle ressource
    data_ = new_data;
    size_ = other.size_;

    return *this;
}
```

Analysons cette implémentation :

**Auto-affectation** — le test `this == &other` court-circuite l'opération. Ce n'est pas strictement nécessaire avec l'approche "allouer d'abord" (l'ancien bloc n'est libéré qu'après la copie), mais c'est une optimisation qui évite une allocation et une copie inutiles.

**Exception safety** — si `new` lance, on n'a encore rien modifié. `data_` et `size_` ont toujours leurs anciennes valeurs. L'objet est intact. C'est la **garantie forte** : soit l'affectation réussit complètement, soit l'objet n'est pas modifié.

**Cas vide** — si `other.size_ == 0`, `new_data` reste `nullptr`. Après la libération de l'ancien bloc, `data_ = nullptr` et `size_ = 0`. L'invariant est respecté.

## L'idiome copy-and-swap

Il existe une technique plus élégante qui résout tous les problèmes d'un coup : le **copy-and-swap**. L'idée est de réutiliser le constructeur de copie (que nous avons déjà écrit et testé) pour préparer la nouvelle valeur, puis d'échanger les ressources :

```cpp
DynArray& DynArray::operator=(const DynArray& other) {
    DynArray tmp(other);    // 1. Copie via le constructeur de copie (peut lancer)
    swap(tmp);              // 2. Échange des ressources (ne lance pas)
    return *this;           // 3. tmp détruit avec l'ancienne ressource de *this
}
```

Pour que cela fonctionne, nous avons besoin d'une méthode `swap` membre :

```cpp
class DynArray {  
public:  
    void swap(DynArray& other) noexcept {
        std::swap(data_, other.data_);
        std::swap(size_, other.size_);
    }
    // ...
};
```

Déroulons l'exécution pour `b = a` :

```
Avant :    b.data_ → [0,0,0,0,0]    a.data_ → [42,42,42]

Étape 1 :  tmp construit comme copie de a
           tmp.data_ → [42,42,42] (nouveau bloc)

Étape 2 :  swap(b, tmp)
           b.data_ → [42,42,42] (le nouveau bloc)
           tmp.data_ → [0,0,0,0,0] (l'ancien bloc de b)

Étape 3 :  tmp détruit → delete[] [0,0,0,0,0]
           L'ancien bloc de b est correctement libéré

Après :    b.data_ → [42,42,42]     a.data_ → [42,42,42] (intact)
```

### Pourquoi copy-and-swap est élégant

**Auto-affectation** — fonctionne naturellement. Si `a = a`, `tmp` est une copie de `a`, `swap` échange les pointeurs, et `tmp` détruit ce qui était dans `a` — mais `a` pointe maintenant vers la copie. Aucun test `this == &other` n'est nécessaire.

**Exception safety** — la seule opération qui peut lancer est la construction de `tmp` (étape 1). Si elle échoue, `*this` n'a pas été touché. `swap` est `noexcept` (échange de scalaires). La destruction de `tmp` est `noexcept`. La garantie forte est automatique.

**Pas de duplication de code** — la logique de copie profonde est entièrement dans le constructeur de copie. L'opérateur d'affectation ne fait que l'utiliser. Si vous corrigez un bug dans le constructeur de copie, l'opérateur d'affectation en bénéficie automatiquement.

**Libération correcte** — l'ancien bloc est libéré par le destructeur de `tmp`, pas par un `delete[]` explicite. Le RAII fait le travail.

### La variante par valeur

Une variante encore plus concise passe le paramètre **par valeur** au lieu de `const&`. La copie est alors effectuée par le compilateur lors du passage d'argument :

```cpp
DynArray& DynArray::operator=(DynArray other) {   // Copie faite ici par le compilateur
    swap(other);                                    // Échange (noexcept)
    return *this;                                   // other détruit avec l'ancienne ressource
}
```

Cette variante a un avantage supplémentaire : si l'argument est une rvalue, le compilateur utilise le **constructeur de déplacement** au lieu du constructeur de copie pour créer `other`. L'opérateur d'affectation gère alors à la fois la copie et le déplacement en une seule fonction :

```cpp
DynArray a(3, 42);  
DynArray b(5, 0);  

b = a;                  // 'a' est une lvalue → other construit par copie → swap  
b = DynArray(10, 7);    // rvalue → other construit par déplacement → swap (plus rapide)  
```

Le compromis : cette variante effectue toujours une copie, même en cas d'auto-affectation (`a = a` alloue et copie inutilement). Pour la plupart des classes, ce surcoût est négligeable. Pour les classes avec des ressources très coûteuses à copier, la version `const&` avec test d'auto-affectation peut être préférable.

> 💡 Pour `DynArray`, nous garderons la version classique avec `const&` dans le code final, car elle est plus explicite pour un cours pédagogique. En production, la variante par valeur est souvent préférée pour sa concision.

## Fournir `swap` : une bonne pratique

La méthode `swap` n'est pas seulement utile pour copy-and-swap — c'est une opération fondamentale que les algorithmes de la STL utilisent. Fournir un `swap` performant est une bonne pratique pour toute classe qui gère des ressources :

```cpp
class DynArray {  
public:  
    // Swap membre — noexcept, O(1)
    void swap(DynArray& other) noexcept {
        std::swap(data_, other.data_);
        std::swap(size_, other.size_);
    }

    // ...
};

// Swap non-membre dans le même namespace — trouvé par ADL
inline void swap(DynArray& a, DynArray& b) noexcept {
    a.swap(b);
}
```

La fonction libre `swap` dans le namespace de `DynArray` permet aux algorithmes STL de trouver votre `swap` optimisé via l'**ADL** (*Argument-Dependent Lookup*). Sans elle, `std::swap` utiliserait un déplacement temporaire (ou pire, une copie), ce qui est plus coûteux qu'échanger deux pointeurs.

`swap` doit toujours être `noexcept` : il n'échange que des scalaires, il n'y a aucune raison de lancer.

---

## Chaînage d'affectations

Le type de retour `DynArray&` (et non `void`) permet le chaînage :

```cpp
DynArray a(3, 1);  
DynArray b(4, 2);  
DynArray c(5, 3);  

a = b = c;    // Évalué comme : a = (b = c)
              // 1. b.operator=(c) → b est une copie de c, retourne b
              // 2. a.operator=(b) → a est une copie de b (qui vaut c)
              // Résultat : a, b et c contiennent [3, 3, 3, 3, 3]
```

Le chaînage fonctionne parce que `operator=` retourne `*this` par référence. Retourner par valeur déclencherait une copie supplémentaire à chaque `=`. Retourner `void` empêcherait le chaînage. `T&` est la seule bonne option.

---

## Relecture et cohérence avec les quatre autres opérations

Voici l'implémentation finale de l'opérateur d'affectation par copie pour `DynArray`, utilisant l'approche "allouer d'abord" :

```cpp
DynArray& DynArray::operator=(const DynArray& other) {
    if (this == &other) return *this;

    int* new_data = nullptr;
    if (other.size_ > 0) {
        new_data = new int[other.size_];
        std::copy_n(other.data_, other.size_, new_data);
    }

    delete[] data_;
    data_ = new_data;
    size_ = other.size_;

    return *this;
}
```

Vérifions la cohérence :

| Vérification | Résultat |  
|---|---|  
| Libère l'ancienne ressource (`delete[] data_`) | ✅ Pas de fuite |  
| Alloue une nouvelle ressource (`new int[]`) | ✅ Copie profonde |  
| Gère l'auto-affectation (`this == &other`) | ✅ Court-circuit |  
| Exception safety (garantie forte) | ✅ Allocation avant libération |  
| Gère le cas vide (`other.size_ == 0`) | ✅ `new_data = nullptr` |  
| Résultat identique au constructeur de copie | ✅ Même logique de duplication |  
| Symétrique avec le destructeur (`new[]` ↔ `delete[]`) | ✅ |  
| Retourne `*this` pour le chaînage | ✅ |  
| Source non modifiée (`const&`) | ✅ |

## Fil conducteur : état de `DynArray`

Avec l'opérateur d'affectation par copie, nous avons maintenant **quatre des cinq** opérations spéciales :

```cpp
class DynArray {  
public:  
    DynArray() = default;
    explicit DynArray(std::size_t size);
    DynArray(std::size_t size, int value);

    ~DynArray();                                        // ✅ 6.5.1
    DynArray(const DynArray& other);                    // ✅ 6.5.2
    DynArray& operator=(const DynArray& other);         // ✅ 6.5.3 — NOUVEAU
    DynArray(DynArray&& other) noexcept;                // ✅ 6.5.4

    DynArray& operator=(DynArray&& other) noexcept;     // ⏳ 6.5.5 — reste à faire

    void swap(DynArray& other) noexcept;                // Utilitaire

    std::size_t size() const { return size_; }
    bool empty() const { return size_ == 0; }
    int& operator[](std::size_t index);
    const int& operator[](std::size_t index) const;

    friend std::ostream& operator<<(std::ostream& os, const DynArray& arr);

private:
    int* data_ = nullptr;
    std::size_t size_ = 0;
};
```

Le danger signalé depuis la section 6.2.3 — le *double free* sur `b = a` — est enfin résolu. L'affectation par copie libère correctement l'ancien bloc de `b`, alloue un nouveau bloc, et copie les données de `a`. Les deux objets sont indépendants après l'opération.

Il reste une opération : l'opérateur d'affectation par déplacement. C'est le dernier maillon de la Règle des 5.

---

## Points clés à retenir

- L'opérateur d'affectation par copie est plus complexe que le constructeur de copie : il doit **libérer l'ancienne ressource**, **gérer l'auto-affectation**, et offrir la **garantie forte d'exception**.  
- La stratégie "**allouer d'abord, libérer ensuite**" résout les pièges d'auto-affectation et d'exception safety en une seule approche.  
- L'idiome **copy-and-swap** est une alternative élégante qui réutilise le constructeur de copie, élimine la duplication de code, et fournit la garantie forte automatiquement. Sa variante par valeur unifie copie et déplacement en une seule fonction.  
- Fournissez une méthode **`swap` `noexcept`** pour votre classe. Elle est utile pour copy-and-swap, pour la STL, et comme brique de base pour les opérations d'affectation.  
- L'opérateur retourne `T&` (`*this`) pour permettre le **chaînage** (`a = b = c`).  
- Le résultat observable de l'opérateur d'affectation doit être **identique** à celui du constructeur de copie : un clone indépendant de la source.
