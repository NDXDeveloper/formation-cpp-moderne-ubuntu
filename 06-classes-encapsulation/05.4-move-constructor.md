# 6.5.4 — Constructeur de déplacement

## Chapitre 6 : Classes et Encapsulation — Section 6.5 : Règle des 5

---

## Objectif de cette sous-section

Le constructeur de déplacement a été présenté en détail en section 6.2.4 (rvalues, transfert de ressources, `noexcept`, état *moved-from*). Cette sous-section le **revisite sous l'angle de la Règle des 5** : sa complémentarité avec les opérations de copie, sa cohérence avec le destructeur et l'opérateur d'affectation par déplacement, et les conséquences de son absence.

---

## Le constructeur de déplacement dans l'ensemble des cinq

Les cinq opérations spéciales se répartissent en deux familles :

```
Famille copie :                          Famille déplacement :
  Constructeur de copie    (6.5.2)         Constructeur de déplacement  (6.5.4)
  Affectation par copie    (6.5.3)         Affectation par déplacement  (6.5.5)

                    Destructeur (6.5.1)
                    (partagé par les deux familles)
```

Le constructeur de déplacement est le **pendant** du constructeur de copie dans la famille déplacement. Les deux créent un nouvel objet à partir d'un existant, mais par des mécanismes opposés :

| Aspect | Constructeur de copie | Constructeur de déplacement |  
|---|---|---|  
| Paramètre | `const T&` (lvalue) | `T&&` (rvalue) |  
| Effet sur la source | Intacte | Vidée (*moved-from*) |  
| Ressource | Dupliquée | Transférée |  
| Coût | O(n) | O(1) |  
| Peut lancer | Oui (`new` → `bad_alloc`) | Non (`noexcept`) |

La relation entre les deux est celle d'une **optimisation sémantique** : quand le compilateur sait que la source va disparaître (rvalue), il choisit le déplacement plutôt que la copie. Le résultat du point de vue de l'objet cible est identique — il contient les mêmes données — mais le chemin pour y parvenir est radicalement différent.

---

## Impact sur la génération implicite

Rappelons la ligne de la matrice :

| Vous déclarez… | Destructeur | Copie ctor | Copie `=` | Move ctor | Move `=` |  
|---|---|---|---|---|---|  
| **Move ctor** | ✅ généré | ❌ supprimé | ❌ supprimé | — | ❌ supprimé |

Déclarer un constructeur de déplacement est l'opération qui a l'impact le plus **agressif** sur la génération implicite : elle supprime les **trois** autres opérations non triviales (copie ctor, copie `=`, move `=`). Seul le destructeur survit.

Le raisonnement du compilateur est le suivant : si vous personnalisez le déplacement, c'est que votre classe a une sémantique de propriété non triviale. La copie par défaut (superficielle) serait dangereuse. Et l'opérateur d'affectation par déplacement ne peut pas être deviné puisqu'il doit aussi libérer l'ancienne ressource.

Conséquence concrète — si vous déclarez un constructeur de déplacement sans déclarer les opérations de copie, votre classe devient **move-only** :

```cpp
class MoveOnlyByAccident {  
public:  
    MoveOnlyByAccident() = default;
    MoveOnlyByAccident(MoveOnlyByAccident&& other) noexcept { /* ... */ }
    // Copie supprimée implicitement — probablement involontaire !
};

MoveOnlyByAccident a;  
MoveOnlyByAccident b = std::move(a);   // OK — déplacement  
// MoveOnlyByAccident c = b;           // ERREUR — copie supprimée
```

Si votre classe doit être copiable **et** déplaçable, vous devez déclarer les deux familles explicitement. C'est le cœur de la Règle des 5.

---

## Cohérence avec les quatre autres opérations

### Avec le destructeur

Le constructeur de déplacement transfère la propriété de la ressource. L'objet source est neutralisé (`data_ = nullptr`). Quand le destructeur de la source s'exécute, il doit gérer cet état vidé sans erreur :

```
Constructeur de déplacement :  vole data_, met source.data_ = nullptr
                                    ↕ cohérence
Destructeur de la source :     delete[] nullptr → no-op ✅  
Destructeur de la cible :      delete[] data_ (le bloc volé) → libération normale ✅  
```

Le bloc mémoire change de propriétaire, mais il est libéré **exactement une fois** — par le destructeur de l'objet qui le possède au moment de sa destruction. Pas de fuite, pas de *double free*.

### Avec le constructeur de copie

Les deux constructeurs sont mutuellement exclusifs dans l'overload resolution du compilateur. Pour une lvalue, le compilateur choisit la copie. Pour une rvalue, il choisit le déplacement. Il n'y a jamais d'ambiguïté :

```cpp
DynArray a(5, 42);

DynArray b = a;              // 'a' est lvalue → copie (O(n))  
DynArray c = std::move(a);   // std::move(a) est rvalue → déplacement (O(1))  
DynArray d = DynArray(3);    // temporaire rvalue → déplacement (O(1))  
```

Les deux constructeurs doivent produire un objet cible dans un **état équivalent** (mêmes données, même comportement). La différence est uniquement dans ce qui arrive à la source et dans le coût de l'opération.

### Avec l'opérateur d'affectation par copie

Ce sont deux opérations de familles différentes. Le constructeur de déplacement crée un nouvel objet (pas d'ancienne ressource à libérer). L'opérateur d'affectation par copie remplace un objet existant (ancienne ressource à libérer, auto-affectation à gérer). Pas de chevauchement fonctionnel, mais le même invariant doit être maintenu.

### Avec l'opérateur d'affectation par déplacement (section 6.5.5)

Même relation que entre constructeur de copie et opérateur d'affectation par copie : le résultat est identique (transfert de propriété), mais l'opérateur d'affectation doit en plus libérer l'ancienne ressource de l'objet cible. Le constructeur de déplacement est la version "simple" ; l'opérateur d'affectation par déplacement est la version "avec nettoyage".

## `noexcept` : vérification dans le contexte de la Règle des 5

En section 6.2.4, nous avons vu que `noexcept` est crucial pour que `std::vector` déplace au lieu de copier lors des réallocations. Dans le contexte de la Règle des 5, cette exigence prend une dimension supplémentaire : **toutes les opérations de déplacement doivent être `noexcept`**, pas seulement le constructeur.

```cpp
class DynArray {  
public:  
    DynArray(DynArray&& other) noexcept;               // noexcept ✅
    DynArray& operator=(DynArray&& other) noexcept;     // noexcept ✅ (6.5.5)
    void swap(DynArray& other) noexcept;                // noexcept ✅
    ~DynArray();                                         // implicitement noexcept ✅
};
```

L'ensemble forme une famille cohérente : tout ce qui touche au transfert de propriété et à la libération est garanti sans exception. Le constructeur de copie est la seule opération spéciale qui a le droit de lancer (à cause de l'allocation).

Si votre constructeur de déplacement ne peut pas être `noexcept` — par exemple parce qu'un membre effectue une allocation pendant le déplacement — c'est le signe d'un problème de conception. Un déplacement devrait toujours se réduire à des échanges de scalaires et de pointeurs.

---

## Ce qui se passe sans constructeur de déplacement

Si vous déclarez un destructeur et un constructeur de copie mais **pas** de constructeur de déplacement, le compilateur ne le génère pas (règle de la matrice). Les rvalues se rabattent sur la copie :

```cpp
class NoMove {  
public:  
    NoMove() = default;
    ~NoMove() { delete[] data_; }
    NoMove(const NoMove& other)
        : data_(new int[other.size_]), size_(other.size_) {
        std::copy_n(other.data_, size_, data_);
    }
    NoMove& operator=(const NoMove& other);
    // Pas de move ctor, pas de move = → supprimés par la présence du destructeur

private:
    int* data_ = nullptr;
    std::size_t size_ = 0;
};

NoMove create() {
    NoMove tmp(/* ... */);
    return tmp;      // Pas de move → copie (ou RVO si le compilateur l'applique)
}

std::vector<NoMove> vec;  
vec.push_back(NoMove(/* ... */));   // Pas de move → copie à chaque réallocation  
vec.reserve(100);                    // Même chose : copies, pas de déplacements  
```

Le programme est correct, mais il est **inutilement lent**. Chaque opération qui pourrait bénéficier d'un déplacement O(1) effectue une copie O(n). Pour un `vector` de 1000 éléments qui se réalloue, cela signifie potentiellement des milliers de copies profondes là où des déplacements de quelques octets suffiraient.

C'est pourquoi la Règle des 5 insiste : si vous écrivez un destructeur et une copie, écrivez aussi le déplacement. Le surcoût en code est minime (quelques lignes), le gain en performance peut être considérable.

## Relecture du constructeur de déplacement de `DynArray`

```cpp
DynArray::DynArray(DynArray&& other) noexcept
    : data_(other.data_)
    , size_(other.size_) {
    other.data_ = nullptr;
    other.size_ = 0;
}
```

| Vérification | Résultat |  
|---|---|  
| Transfère la ressource (vol du pointeur) | ✅ `data_(other.data_)` |  
| Neutralise la source | ✅ `other.data_ = nullptr`, `other.size_ = 0` |  
| État *moved-from* = état construit par défaut | ✅ `nullptr`, `0` |  
| `noexcept` | ✅ Aucune allocation, aucun appel pouvant lancer |  
| Cohérent avec le destructeur | ✅ Source : `delete[] nullptr` → no-op. Cible : `delete[] data_` → libération normale |  
| Coût O(1) | ✅ Copie de deux scalaires + deux affectations |  
| Résultat équivalent au constructeur de copie | ✅ Même données dans l'objet cible |  
| Ordre d'initialisation = ordre de déclaration | ✅ `data_` puis `size_` |

L'implémentation est minimale et correcte. Chaque ligne a un rôle précis, et rien ne peut être retiré sans introduire un bug.

## Pattern récapitulatif : écrire un constructeur de déplacement

Pour toute classe gérant une ressource, le constructeur de déplacement suit toujours le même pattern en trois temps :

```cpp
T::T(T&& other) noexcept
    : /* 1. Voler chaque ressource de other */
{
    /* 2. Neutraliser other pour chaque ressource volée */
}
```

Pour une classe avec plusieurs ressources :

```cpp
MultiResource::MultiResource(MultiResource&& other) noexcept
    : db_handle_(other.db_handle_)       // Voler
    , socket_fd_(other.socket_fd_)       // Voler
    , buffer_(other.buffer_)             // Voler
    , buffer_size_(other.buffer_size_) { // Voler
    other.db_handle_ = nullptr;          // Neutraliser
    other.socket_fd_ = -1;              // Neutraliser
    other.buffer_ = nullptr;             // Neutraliser
    other.buffer_size_ = 0;              // Neutraliser
}
```

Chaque ressource a sa valeur de neutralisation propre : `nullptr` pour les pointeurs, `-1` pour les descripteurs de fichier, `0` pour les tailles. L'objectif est que le destructeur de `other` puisse s'exécuter sans tenter de libérer une ressource qui ne lui appartient plus.

Avec la Règle du Zéro, ce code mécanique disparaît entièrement. Si chaque ressource est encapsulée dans un wrapper RAII (`unique_ptr`, `FileDescriptor`, etc.), le constructeur de déplacement implicite déplace chaque membre individuellement — et chaque membre sait comment se déplacer :

```cpp
class ModernMulti {
    std::unique_ptr<Database> db_;    // unique_ptr sait se déplacer
    FileDescriptor socket_;           // FileDescriptor sait se déplacer (6.3.2)
    std::vector<char> buffer_;        // vector sait se déplacer
    // Constructeur de déplacement implicite = parfait
};
```

---

## Points clés à retenir

- Le constructeur de déplacement est le **pendant rapide** du constructeur de copie : même résultat pour l'objet cible, mais O(1) au lieu de O(n), au prix du vidage de la source.  
- Déclarer un constructeur de déplacement **supprime** les opérations de copie et l'opérateur d'affectation par déplacement. Si vous voulez une classe copiable et déplaçable, déclarez les cinq opérations.  
- Sans constructeur de déplacement, les rvalues se rabattent **silencieusement** sur la copie. Le code compile, mais les performances sont dégradées — parfois de plusieurs ordres de grandeur.  
- Le constructeur de déplacement et le destructeur forment une paire : le déplacement transfère la propriété, le destructeur de la source gère l'état vidé. La neutralisation de la source doit être cohérente avec ce que le destructeur attend.  
- `noexcept` est **obligatoire** en pratique. Sans lui, la STL préfère copier plutôt que déplacer, annulant le bénéfice du déplacement.  
- Le pattern est mécanique : voler chaque ressource, neutraliser la source. Avec la Règle du Zéro et des membres RAII, le constructeur de déplacement implicite fait ce travail automatiquement.
