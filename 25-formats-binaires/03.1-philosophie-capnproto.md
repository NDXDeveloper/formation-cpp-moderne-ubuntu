# 25.3.1 Philosophie : le format wire EST le format mémoire

## Le problème de la sérialisation traditionnelle

Pour comprendre la philosophie de Cap'n Proto, il faut d'abord visualiser ce que fait un format de sérialisation classique comme Protobuf. Considérons un message simple :

```protobuf
// Protobuf
message Sensor {
  uint32 id = 1;
  float  temperature = 2;
  string location = 3;
}
```

Avec Protobuf, le cycle de vie d'un message implique **trois représentations distinctes** des mêmes données :

```
1. OBJET C++              2. FORMAT WIRE              3. OBJET C++ (distant)
   en mémoire                sur le réseau                en mémoire

┌──────────────┐          ┌──────────────┐           ┌──────────────┐
│ id: 42       │ encode   │ 08 2A 15 00  │  decode   │ id: 42       │
│ temp: 23.5   │ ──────►  │ 00 BC 41 1A  │  ──────►  │ temp: 23.5   │
│ loc: "Paris" │ (copie)  │ 05 50 61 72  │  (copie)  │ loc: "Paris" │
│ + vtable     │          │ 69 73        │  + alloc  │ + vtable     │
│ + std::string│          └──────────────┘           │ + std::string│
│   internals  │          Format compact,            │   internals  │
└──────────────┘          non exploitable            └──────────────┘
                          directement
```

Chaque transition est une **transformation coûteuse** : l'encodage parcourt tous les champs de l'objet C++ et les écrit dans le format wire compact (varints, length-delimited). Le décodage fait l'inverse — il parse le flux d'octets, alloue de la mémoire (notamment pour les strings et les sous-messages), et construit un nouvel objet C++.

Pour un message de 1 Ko, ce coût est négligeable. Pour un pipeline qui traite des millions de messages par seconde, ou un proxy qui doit inspecter un champ puis retransmettre le message, ces transformations deviennent un goulot d'étranglement mesurable.

---

## L'insight de Cap'n Proto : une seule représentation

Cap'n Proto élimine les transformations en fusionnant les trois représentations en une seule. Le message est construit directement dans un buffer dont le format **est** le format wire :

```
Cap'n Proto : UNE SEULE représentation

┌─────────────────────────────────┐
│ Buffer en mémoire               │
│                                 │
│ ┌─────────┬─────────┬────────┐  │
│ │ id: 42  │temp:23.5│→"Paris"│  │  ← Exploitable directement en C++
│ └─────────┴─────────┴────────┘  │    via des accesseurs générés
│                                 │
│ CE BUFFER EST AUSSI :           │
│  • le format envoyé sur réseau  │
│  • le format écrit sur disque   │
│  • le format reçu et lu         │
│    sans aucune transformation   │
└─────────────────────────────────┘

Envoi réseau : send(fd, buffer, size)   ← pas d'encodage  
Réception :    recv(fd, buffer, size)   ← pas de décodage  
Accès champ :  msg.getTemperature()     ← lecture directe dans le buffer  
```

Il n'y a pas d'objet C++ « riche » avec des `std::string`, des vtables et des allocations séparées. Les accesseurs générés par le compilateur Cap'n Proto sont de simples calculs de pointeurs dans le buffer : « le champ `temperature` est un `float` à l'offset 8 depuis le début du struct ».

## Structure interne d'un message

### Segments

Un message Cap'n Proto est composé d'un ou plusieurs **segments** — des blocs de mémoire contigus. Chaque segment est un tableau de **mots de 8 octets** (64 bits), ce qui garantit un alignement naturel pour tous les types primitifs.

```
Message Cap'n Proto
├── Segment 0 (principal)
│   ┌────────┬────────┬────────┬────────┬────────┐
│   │ mot 0  │ mot 1  │ mot 2  │ mot 3  │ mot 4  │  (8 octets chacun)
│   └────────┴────────┴────────┴────────┴────────┘
├── Segment 1 (optionnel, si le message est grand)
│   ┌────────┬────────┬────────┐
│   │ mot 0  │ mot 1  │ mot 2  │
│   └────────┴────────┴────────┘
└── ...
```

La plupart des messages tiennent dans un seul segment. Les segments multiples interviennent quand un message est très grand ou quand on souhaite réutiliser des arènes mémoire (memory arenas). Le format wire d'un message commence par une **table des segments** (segment table) qui liste la taille de chaque segment, suivie des segments eux-mêmes.

### Structs

Le type fondamental de Cap'n Proto est le **struct**. Chaque struct est stocké comme deux sections contiguës dans un segment :

```
Struct en mémoire
┌──────────────────────────────┬───────────────────────────────┐
│        DATA SECTION          │       POINTER SECTION         │
│   (champs de taille fixe)    │   (pointeurs vers les         │
│                              │    données de taille variable)│
│  ┌──────┬──────┬──────────┐  │  ┌──────────┬──────────┐      │
│  │ id   │ temp │ (padding)│  │  │ ptr vers │ ptr vers │      │
│  │ u32  │ f32  │          │  │  │ location │ tags     │      │
│  │4 oct.│4 oct.│          │  │  │ 8 octets │ 8 octets │      │
│  └──────┴──────┴──────────┘  │  └──────────┴──────────┘      │
└──────────────────────────────┴───────────────────────────────┘
```

**La data section** contient les champs de taille fixe (entiers, flottants, booléens, énumérations) à des offsets fixes déterminés à la compilation. Accéder au champ `temperature` revient à lire 4 octets à l'offset 4 depuis le début de la data section — un simple déréférencement de pointeur.

**La pointer section** contient des pointeurs Cap'n Proto (pas des pointeurs machine) vers les données de taille variable : strings (Text), blobs (Data), listes, et sous-structs. Chaque pointeur est un mot de 8 octets qui encode la position relative et la taille de la cible.

### Pointeurs Cap'n Proto

Les pointeurs Cap'n Proto ne sont pas des adresses mémoire absolues. Ce sont des **offsets relatifs** encodés dans 64 bits, ce qui les rend indépendants de la position absolue du buffer en mémoire. C'est cette propriété qui permet le zero-copy : le buffer peut être reçu du réseau, mappé via `mmap`, ou déplacé en mémoire sans invalider les pointeurs internes.

Il existe trois types de pointeurs :

```
Struct pointer (type = 0)
┌──────┬────────────────────────┬───────────────┬───────────────┐
│ 2 bit│ 30 bits                │ 16 bits       │ 16 bits       │
│ type │ offset (en mots,       │ taille data   │ taille ptrs   │
│ = 00 │ relatif à ce pointeur) │ section (mots)│ section (mots)│
└──────┴────────────────────────┴───────────────┴───────────────┘

List pointer (type = 1)
┌──────┬────────────────────────┬───────┬───────────────────────┐
│ 2 bit│ 30 bits                │ 3 bit │ 29 bits               │
│ type │ offset                 │ elem  │ nombre d'éléments     │
│ = 01 │                        │ size  │                       │
└──────┴────────────────────────┴───────┴───────────────────────┘

Far pointer (type = 2) — pour les références inter-segments
┌──────┬────────────────────────┬───────────────────────────────┐
│ 2 bit│ 29 bits                │ 32 bits                       │
│ type │ offset dans le segment │ ID du segment cible           │
│ = 10 │ cible                  │                               │
└──────┴────────────────────────┴───────────────────────────────┘
```

Le fait que les offsets soient **relatifs** est crucial. Quand le noyau copie un message reçu du réseau dans un buffer userspace, tous les pointeurs internes restent valides car ils ne dépendent pas de l'adresse absolue du buffer.

## Démonstration concrète du zero-copy

### Schéma Cap'n Proto

```capnp
# sensor.capnp
@0xdbb9ad1f14bf0b36;   # ID unique du fichier (généré par capnp id)

struct Sensor {
  id          @0 : UInt32;
  temperature @1 : Float32;
  location    @2 : Text;
  tags        @3 : List(Text);
}
```

Chaque champ a un **numéro ordinal** (`@0`, `@1`, ...) qui détermine son offset dans la data section ou la pointer section. Ces numéros sont stables : on peut ajouter `@4`, `@5`, etc. dans les versions futures du schéma sans casser la compatibilité.

### Génération de code

```bash
capnp compile -oc++ sensor.capnp
# Génère : sensor.capnp.h et sensor.capnp.c++
```

### Construction d'un message (écriture zero-copy)

```cpp
#include "sensor.capnp.h"
#include <capnp/message.h>
#include <capnp/serialize.h>
#include <print>

int main() {
    // ── Construction ──
    // MallocMessageBuilder alloue le(s) segment(s) du message.
    // TOUTES les données sont écrites directement dans ces segments.
    capnp::MallocMessageBuilder message;

    // initRoot<Sensor>() retourne un BUILDER : un accesseur
    // qui écrit directement dans le buffer du message.
    auto sensor = message.initRoot<Sensor>();

    sensor.setId(42);               // Écrit 4 octets à offset 0 dans la data section
    sensor.setTemperature(23.5f);   // Écrit 4 octets à offset 4 dans la data section
    sensor.setLocation("Paris");    // Alloue "Paris\0" dans le segment,
                                    // écrit un pointeur dans la pointer section

    auto tags = sensor.initTags(2); // Alloue une liste de 2 Text dans le segment
    tags.set(0, "indoor");
    tags.set(1, "lab-3");

    // À ce stade, le buffer interne du message contient EXACTEMENT
    // les octets qui seront envoyés sur le réseau. Pas d'étape
    // de sérialisation à venir.

    // ── Envoi ──
    // writeMessageToFd() écrit les segments directement via writev().
    // Aucune copie supplémentaire, aucune transformation.
    capnp::writeMessageToFd(STDOUT_FILENO, message);

    return 0;
}
```

Le point clé : après les appels `set*` et `init*`, les données sont déjà dans le format wire final. `writeMessageToFd` ne fait que prépendre la table des segments et appeler `writev()` — il n'y a pas de phase d'encodage.

### Lecture d'un message (lecture zero-copy)

```cpp
#include "sensor.capnp.h"
#include <capnp/message.h>
#include <capnp/serialize.h>
#include <print>

int main() {
    // ── Réception ──
    // StreamFdMessageReader lit les données depuis un fd
    // dans un buffer interne.
    capnp::StreamFdMessageReader message(STDIN_FILENO);

    // getRoot<Sensor>() retourne un READER : un accesseur
    // qui lit directement depuis le buffer reçu.
    // Pas de parsing, pas d'allocation, pas de copie.
    auto sensor = message.getRoot<Sensor>();

    // Chaque getter est un calcul d'offset + déréférencement.
    std::print("ID:          {}\n", sensor.getId());
    std::print("Température: {:.1f}°C\n", sensor.getTemperature());
    std::print("Location:    {}\n", sensor.getLocation().cStr());

    std::print("Tags:\n");
    for (auto tag : sensor.getTags()) {
        std::print("  - {}\n", tag.cStr());
    }

    return 0;
}
```

Quand `sensor.getTemperature()` est appelé, le code généré fait essentiellement :

```cpp
// Pseudocode du getter généré (simplifié)
float Sensor::Reader::getTemperature() const {
    // Lire 4 octets à l'offset 4 dans la data section de ce struct
    return _reader.getDataField<float>(1);  // index 1 = offset 4 octets
}
```

C'est un accès mémoire direct — pas de boucle de parsing, pas de recherche de tag, pas de décodage varint. Le coût est **O(1)** quel que soit le nombre de champs dans le message.

### Test du pipeline

```bash
# Compiler
capnp compile -oc++ sensor.capnp  
g++ -std=c++23 -O2 -o writer writer.cpp sensor.capnp.c++ -lcapnp -lkj  
g++ -std=c++23 -O2 -o reader reader.cpp sensor.capnp.c++ -lcapnp -lkj  

# Le writer écrit le message binaire sur stdout,
# le reader le lit depuis stdin — zero-copy de bout en bout
./writer | ./reader
# ID:          42
# Température: 23.5°C
# Location:    Paris
# Tags:
#   - indoor
#   - lab-3
```

## Visualisation de la mémoire

Pour rendre le zero-copy tangible, examinons les octets réels d'un message. L'outil `capnp decode` permet de visualiser le contenu :

```bash
./writer | capnp decode sensor.capnp Sensor
# ( id = 42,
#   temperature = 23.5,
#   location = "Paris",
#   tags = ["indoor", "lab-3"] )
```

Et `capnp decode --flat` montre la structure brute. Voici une représentation schématique de la disposition en mémoire pour notre message `Sensor{42, 23.5, "Paris", ["indoor", "lab-3"]}` :

```
Offset   Contenu (hex)          Interprétation
──────   ─────────────          ──────────────
[Table des segments]
0x00     00 00 00 00            1 segment (count - 1 = 0)
0x04     0B 00 00 00            Taille : 11 mots (88 octets)

[Segment 0]
─── Struct pointer (racine) ───
0x08     00 00 00 00 01 00 02 00   Struct: offset=0, data=1 mot, ptrs=2 mots

─── Data section du Sensor ───
0x10     2A 00 00 00            id = 42 (little-endian u32)
0x14     00 00 BC 41            temperature = 23.5 (IEEE 754 float32)

─── Pointer section du Sensor ───
0x18     01 00 00 00 2A 00 00 00   Text ptr: offset=0, taille=5+1 octets
0x20     01 00 01 00 15 00 00 00   List ptr: offset=1, 2 éléments Text

─── Données pointées ───
0x28     50 61 72 69 73 00 00 00   "Paris\0" + padding (aligné à 8 octets)

─── Liste de tags ───
0x30     05 00 00 00 3A 00 00 00   Pointeur list element → "indoor"
0x38     01 00 00 00 32 00 00 00   Pointeur list element → "lab-3"
0x40     69 6E 64 6F 6F 72 00 00   "indoor\0" + padding
0x48     6C 61 62 2D 33 00 00 00   "lab-3\0" + padding
```

Chaque champ est à un offset fixe et aligné. Il n'y a pas de tags de champs (comme les field numbers de Protobuf), pas de lengths encodés en varint, pas d'indirection variable. L'accès à n'importe quel champ scalaire est un déréférencement à offset constant depuis la racine.

## Builders et Readers : les deux faces du zero-copy

Cap'n Proto distingue strictement deux types d'accesseurs :

### Builders (écriture)

Un **Builder** est un accesseur mutable qui écrit directement dans le buffer du message en construction. On obtient un Builder via `initRoot<T>()` sur un `MessageBuilder`, ou via les méthodes `init*` et `get*` d'un Builder parent.

```cpp
capnp::MallocMessageBuilder message;  
Sensor::Builder sensor = message.initRoot<Sensor>();  
sensor.setId(42);           // Écriture directe dans le buffer  
sensor.setTemperature(23.5);  

// Sous-structures
auto tags = sensor.initTags(3);  // Alloue dans le même buffer  
tags.set(0, "a");  
```

Les Builders ne peuvent exister que tant que le `MessageBuilder` est vivant — ils ne sont que des vues sur le buffer du message.

### Readers (lecture)

Un **Reader** est un accesseur en lecture seule qui pointe directement dans le buffer reçu. On obtient un Reader via `getRoot<T>()` sur un `MessageReader`.

```cpp
capnp::StreamFdMessageReader message(fd);  
Sensor::Reader sensor = message.getRoot<Sensor>();  
uint32_t id = sensor.getId();     // Lecture directe depuis le buffer  
```

Les Readers sont extrêmement légers : ce sont essentiellement un pointeur vers le buffer et quelques métadonnées de validation. Copier un Reader ne copie pas les données — il copie juste le pointeur.

### Durée de vie : le compromis fondamental

Le corollaire du zero-copy est que **les données appartiennent au buffer**, pas au Reader. Un Reader devient invalide si le `MessageReader` (et donc le buffer sous-jacent) est détruit :

```cpp
Sensor::Reader get_sensor() {
    capnp::StreamFdMessageReader message(fd);
    auto sensor = message.getRoot<Sensor>();
    return sensor;  // ⚠️ DANGLING : message est détruit en sortant du scope
}
// Le Reader retourné pointe vers de la mémoire libérée
```

Si vous devez conserver les données au-delà de la durée de vie du buffer, deux options :

```cpp
// Option 1 : Copier dans un nouveau message (copie profonde)
capnp::MallocMessageBuilder copy;  
copy.setRoot(original_reader);  
// copy possède maintenant ses propres segments

// Option 2 : Extraire les champs dans des types C++ classiques
std::string location{sensor.getLocation().cStr()};  
uint32_t id = sensor.getId();  
// Les valeurs C++ survivent à la destruction du buffer
```

Cette contrainte de durée de vie est le prix du zero-copy. Elle n'est pas intrinsèquement problématique — c'est le même modèle que `std::string_view` ou `std::span` — mais elle exige une discipline que Protobuf ne demande pas (puisque `ParseFromString` produit un objet C++ indépendant).

## Comparaison des chemins de données

Pour synthétiser, comparons le chemin complet d'un message entre deux processus :

```
Protobuf :
  Émetteur                                  Récepteur
  obj.set_id(42)                            
  obj.set_temp(23.5)                        
  obj.set_loc("Paris")                      
  string wire = obj.SerializeAsString()  ← allocation + encodage
  send(fd, wire.data(), wire.size())        recv(fd, buf, size)
                                            Obj obj;
                                            obj.ParseFromString(buf)  ← alloc + décodage
                                            obj.id()  → 42

  Allocations : 2 (émetteur + récepteur)
  Copies :      2 (encode + decode)
  Parsing :     O(N) où N = taille du message


Cap'n Proto :
  Émetteur                                  Récepteur
  sensor.setId(42)         ← écrit dans     
  sensor.setTemperature(23.5)  le buffer     
  sensor.setLocation("Paris")  directement  
  writeMessageToFd(fd, msg)  ← writev()    readMessageFromFd(fd)  ← read()
                                            sensor.getId()  → déréférencement direct
                                            sensor.getTemperature()  → idem

  Allocations : 1 (le buffer initial, côté émetteur)
  Copies :      0 (le buffer reçu EST le message)
  Parsing :     O(1) par accès de champ
```

La différence est d'autant plus marquée pour les **gros messages** ou les messages qu'on ne lit que partiellement. Avec Protobuf, même si vous n'avez besoin que d'un seul champ, le message entier doit être parsé (sauf à utiliser les optimisations lazy de proto3, qui restent limitées). Avec Cap'n Proto, vous payez uniquement pour les champs que vous accédez effectivement.

## Sécurité de la lecture zero-copy

Lire directement dans un buffer réseau pose une question légitime de sécurité : que se passe-t-il si le buffer est malformé, tronqué, ou délibérément crafté pour exploiter une vulnérabilité ?

Cap'n Proto intègre plusieurs mécanismes de protection, actifs par défaut :

**Validation des bornes de pointeurs.** Chaque pointeur est vérifié avant déréférencement : il doit pointer à l'intérieur d'un segment valide, et la zone pointée ne doit pas dépasser les limites du segment. Un pointeur invalide retourne une valeur par défaut (zéro, string vide) au lieu de provoquer un accès mémoire hors bornes.

**Limite de profondeur de traversée (nesting limit).** Par défaut, la profondeur maximale de traversée des structures imbriquées est de 64. Cela empêche les messages contenant des structures récursivement imbriquées de provoquer un stack overflow.

**Limite de traversée totale (traversal limit).** Le nombre total de mots (8 octets) lus lors du parcours d'un message est limité (par défaut : 64 Mo de données). Cela empêche les « amplification attacks » où un petit message contient des pointeurs qui se chevauchent, forçant le lecteur à traverser bien plus de données que la taille réelle du message.

```cpp
// Personnaliser les limites de sécurité
capnp::ReaderOptions options;  
options.traversalLimitInWords = 128 * 1024 * 1024 / 8;  // 128 Mo  
options.nestingLimit = 32;  // Profondeur max réduite  

capnp::StreamFdMessageReader message(fd, options);
```

Ces protections font de Cap'n Proto un format plus sûr que la lecture directe de structures C (`struct` castées depuis un buffer), qui n'offre aucune validation des pointeurs ni des bornes.

## Résumé

La philosophie « le format wire est le format mémoire » de Cap'n Proto n'est pas un slogan marketing — c'est une propriété architecturale vérifiable dans la disposition des octets en mémoire. Les données scalaires sont à des offsets fixes, les données variables sont adressées par des pointeurs relatifs validés, et aucune transformation ne sépare la représentation mémoire de la représentation réseau.

Cette propriété a des implications concrètes : accès O(1) aux champs, pas d'allocation à la réception, possibilité de `mmap` des fichiers, et coût de « sérialisation » réduit à un simple `writev()`. Le compromis est une taille wire plus grande (alignement, pointeurs 64 bits) et une contrainte de durée de vie sur les Readers.
