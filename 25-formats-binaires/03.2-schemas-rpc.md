# 25.3.2 Schémas, génération de code et RPC intégré

## Le langage de schéma Cap'n Proto

Les fichiers `.capnp` définissent la structure des messages et des interfaces RPC. La syntaxe est volontairement plus expressive que celle de Protobuf, avec un typage riche, des valeurs par défaut, des annotations et des constantes.

### Identifiant de fichier

Chaque fichier `.capnp` commence par un **identifiant unique de 64 bits** qui le distingue de tout autre schéma dans le monde. Cet identifiant est essentiel pour le système RPC et pour détecter les conflits de schéma :

```capnp
@0xa1b2c3d4e5f60718;
```

On le génère avec la commande :

```bash
capnp id
# @0xa1b2c3d4e5f60718;
```

> ⚠️ Ne réutilisez jamais un identifiant d'un autre fichier et ne l'inventez pas — utilisez toujours `capnp id` qui produit un identifiant cryptographiquement aléatoire.

## Types de données

### Types primitifs

Cap'n Proto offre des types primitifs de taille explicite, sans ambiguïté :

| Cap'n Proto | Taille | Équivalent C++ |
|-------------|--------|-----------------|
| `Void` | 0 bits | (aucune donnée) |
| `Bool` | 1 bit | `bool` |
| `Int8`, `Int16`, `Int32`, `Int64` | 8–64 bits | `int8_t` ... `int64_t` |
| `UInt8`, `UInt16`, `UInt32`, `UInt64` | 8–64 bits | `uint8_t` ... `uint64_t` |
| `Float32`, `Float64` | 32–64 bits | `float`, `double` |
| `Text` | variable | `capnp::Text::Reader` (UTF-8, nul-terminé) |
| `Data` | variable | `capnp::Data::Reader` (blob binaire) |

`Text` est toujours UTF-8 et nul-terminé, ce qui permet d'appeler `.cStr()` pour obtenir un `const char*` compatible C sans copie.

### Structs

```capnp
struct Person {
  name     @0 : Text;
  age      @1 : UInt16;
  email    @2 : Text;
  isActive @3 : Bool = true;   # Valeur par défaut
}
```

Chaque champ a un **numéro ordinal** (`@0`, `@1`, ...) attribué séquentiellement. Ces numéros sont permanents : ils déterminent la position physique du champ dans le message et ne doivent jamais être réattribués, même si un champ est supprimé.

### Listes

```capnp
struct Team {
  members  @0 : List(Person);
  scores   @1 : List(Float64);
  tags     @2 : List(Text);
  matrix   @3 : List(List(UInt32));  # Liste de listes
}
```

Les listes sont homogènes (tous les éléments du même type). Les éléments de types primitifs sont stockés de manière compacte (pas de pointeur par élément), ce qui rend les listes de scalaires aussi denses qu'un tableau C.

### Enums

```capnp
enum LogLevel {
  debug   @0;
  info    @1;
  warning @2;
  error   @3;
  fatal   @4;
}

struct LogEntry {
  level   @0 : LogLevel;
  message @1 : Text;
}
```

Les enums sont stockés comme des `UInt16` dans la data section. Comme pour les champs de struct, les numéros ordinaux sont stables.

### Unions et groupes

Les **unions** permettent de représenter des types alternatifs — l'équivalent de `oneof` en Protobuf, mais plus puissant :

```capnp
struct Shape {
  name @0 : Text;

  union {
    circle   @1 : Float64;    # rayon
    rectangle :group {
      width  @2 : Float64;
      height @3 : Float64;
    }
    triangle :group {
      base   @4 : Float64;
      height @5 : Float64;
      angle  @6 : Float64;
    }
  }
}
```

Plusieurs points importants :

- Un struct peut avoir **au plus une union anonyme**. Pour plusieurs variantes, on utilise des unions nommées ou des structs imbriqués.  
- Les **groupes** (`:group`) regroupent plusieurs champs sous un même variant de l'union, sans coût supplémentaire (pas d'indirection pointeur — les champs sont dans la data section du struct parent).  
- La union consomme 16 bits dans la data section pour stocker le discriminant (quel variant est actif).

### Constantes et annotations

```capnp
const maxRetries : UInt32 = 5;
const defaultHost : Text = "localhost";

annotation deprecated(*) : Text;
# Utilisable comme : $deprecated("Utiliser PersonV2 à la place")
```

### Imports

```capnp
using import "common.capnp".Timestamp;
using import "common.capnp".UUID;

struct Event {
  id        @0 : UUID;
  timestamp @1 : Timestamp;
  payload   @2 : Data;
}
```

### Types génériques

Cap'n Proto supporte les **types paramétrés** (generics), une fonctionnalité absente de Protobuf :

```capnp
struct Map(Key, Value) {
  entries @0 : List(Entry);

  struct Entry {
    key   @0 : Key;
    value @1 : Value;
  }
}

struct Config {
  settings @0 : Map(Text, Text);
  limits   @1 : Map(Text, UInt64);
}
```

## Évolution de schéma

L'évolution de schéma est l'une des forces majeures de Cap'n Proto. Les règles sont strictes et appliquées par le compilateur, ce qui élimine une classe entière de bugs de compatibilité.

### Modifications autorisées

- **Ajouter un nouveau champ** à la fin d'un struct (avec le prochain numéro ordinal). Les anciens lecteurs ignorent le champ inconnu.  
- **Renommer un champ** sans changer son numéro ordinal — le nom n'est pas encodé dans le format wire.  
- **Ajouter un variant** à une union existante.  
- **Changer une valeur par défaut** (elle est encodée dans le schéma, pas dans le message).  
- **Promouvoir un champ non-union en union** — le champ existant devient le premier variant.

### Modifications interdites

- **Changer le type d'un champ** (même entre types « compatibles » comme `UInt32` → `UInt64`).  
- **Réutiliser un numéro ordinal** supprimé.  
- **Réordonner les numéros ordinaux** existants.  
- **Supprimer un variant d'union** (on peut le marquer obsolète, mais pas le retirer).

### Dépréciation propre

On ne peut pas supprimer un champ, mais on peut le remplacer par un champ `Void` du même numéro pour signaler l'obsolescence :

```capnp
struct PersonV2 {
  name         @0 : Text;
  age          @1 : UInt16;
  deprecatedX  @2 : Void;    # Était : email @2 : Text;
  isActive     @3 : Bool;
  phone        @4 : Text;    # Nouveau champ
}
```

Le numéro `@2` est préservé dans le schéma (pas de trou), l'espace mémoire est récupéré (un `Void` n'occupe aucun espace dans la data section), et les anciens lecteurs qui lisent `@2` comme un `Text` recevront une string vide (la valeur par défaut de Text).

## Génération de code C++

### Compilation du schéma

```bash
capnp compile -oc++ schema/sensor.capnp
# Produit :
#   schema/sensor.capnp.h
#   schema/sensor.capnp.c++
```

Avec CMake, la macro `capnp_generate_cpp` automatise cette étape :

```cmake
find_package(CapnProto REQUIRED)

capnp_generate_cpp(CAPNP_SRCS CAPNP_HDRS
    schema/sensor.capnp
    schema/service.capnp
)

add_executable(my_app main.cpp ${CAPNP_SRCS})
target_link_libraries(my_app PRIVATE CapnProto::capnp)
target_include_directories(my_app PRIVATE ${CMAKE_CURRENT_BINARY_DIR})
```

### Code généré : anatomie

Pour un struct `Sensor` avec les champs `id`, `temperature`, `location`, `tags`, le compilateur génère deux classes imbriquées dans le namespace correspondant :

```cpp
// Simplifié — le code réel est plus verbeux mais suit cette structure

class Sensor {
public:
    class Reader {
    public:
        uint32_t getId() const;
        float    getTemperature() const;
        bool     hasLocation() const;      // Teste si le pointeur est non-nul
        capnp::Text::Reader getLocation() const;
        bool     hasTags() const;
        capnp::List<capnp::Text>::Reader getTags() const;
    };

    class Builder {
    public:
        void     setId(uint32_t value);
        void     setTemperature(float value);
        void     setLocation(capnp::Text::Reader value);
        capnp::Text::Builder initLocation(unsigned size);
        void     adoptLocation(capnp::Orphan<capnp::Text>&& orphan);
        capnp::List<capnp::Text>::Builder initTags(unsigned size);
        // ...
    };
};
```

### Patterns d'utilisation du code généré

**Construction complète :**

```cpp
#include "sensor.capnp.h"
#include <capnp/message.h>
#include <capnp/serialize.h>

capnp::MallocMessageBuilder message;
auto sensor = message.initRoot<Sensor>();

sensor.setId(42);
sensor.setTemperature(23.5f);
sensor.setLocation("Paris");

auto tags = sensor.initTags(3);
tags.set(0, "indoor");
tags.set(1, "lab-3");
tags.set(2, "calibrated");
```

**Lecture avec vérification :**

```cpp
auto sensor = messageReader.getRoot<Sensor>();

if (sensor.hasLocation()) {
    auto loc = sensor.getLocation();
    std::print("Location: {} ({} octets)\n", loc.cStr(), loc.size());
}

if (sensor.hasTags()) {
    for (auto tag : sensor.getTags()) {
        std::print("  Tag: {}\n", tag.cStr());
    }
}
```

**Manipulation d'unions :**

```cpp
// Schéma :
// struct Shape {
//   union { circle @0 :Float64; rectangle :group { w @1 :Float64; h @2 :Float64; } }
// }

// Construction
auto shape = message.initRoot<Shape>();
auto rect = shape.initRectangle();
rect.setW(10.0);
rect.setH(5.0);

// Lecture
auto shape = reader.getRoot<Shape>();
switch (shape.which()) {
    case Shape::CIRCLE:
        std::print("Cercle, rayon={}\n", shape.getCircle());
        break;
    case Shape::RECTANGLE:
        std::print("Rectangle {}×{}\n",
                   shape.getRectangle().getW(),
                   shape.getRectangle().getH());
        break;
}
```

## Sérialisation et transport

### Écriture/lecture sur descripteur de fichier

```cpp
#include <capnp/serialize.h>

// Écriture (utilise writev en interne — zéro-copie si possible)
capnp::writeMessageToFd(fd, message);

// Lecture
capnp::StreamFdMessageReader reader(fd);
auto root = reader.getRoot<MyStruct>();
```

### Écriture/lecture sur tableau d'octets

```cpp
#include <capnp/serialize.h>

// Sérialiser dans un tableau
auto words = capnp::messageToFlatArray(message);
auto bytes = words.asBytes();  // kj::ArrayPtr<const kj::byte>
// bytes.begin() et bytes.size() pour envoyer via n'importe quel transport

// Désérialiser depuis un tableau
capnp::FlatArrayMessageReader reader(
    kj::ArrayPtr<const capnp::word>(
        reinterpret_cast<const capnp::word*>(data),
        size / sizeof(capnp::word)
    )
);
auto root = reader.getRoot<MyStruct>();
```

### Format packed (optionnel)

Cap'n Proto offre un format **packed** qui comprime les séquences de zéros dans les messages (fréquentes à cause du padding d'alignement). Ce format réduit la taille wire de 30 à 50 % pour les messages typiques, au prix d'un passage de compression/décompression qui brise le zero-copy pur :

```cpp
#include <capnp/serialize-packed.h>

// Écriture packed
capnp::writePackedMessageToFd(fd, message);

// Lecture packed
capnp::PackedFdMessageReader reader(fd);
auto root = reader.getRoot<MyStruct>();
```

Le format packed est un compromis : il ne remplace pas une compression généraliste (zstd, lz4), mais il est beaucoup plus léger et ne nécessite pas de librairie externe. Il est recommandé pour les transports WAN où la bande passante est limitée, tout en gardant le format non-packed pour l'IPC locale ou le stockage disque.

## Orphans : manipulation avancée des sous-messages

Un **orphan** est un sous-message construit indépendamment de sa position finale dans le message parent. C'est utile quand on veut construire des parties du message dans le désordre ou déplacer des sous-structures :

```cpp
capnp::MallocMessageBuilder message;

// Créer un orphan Text
auto orphan = message.getOrphanage().newOrphan<capnp::Text>(5);
auto builder = orphan.get();
memcpy(builder.begin(), "Hello", 5);

// L'adopter dans le message
auto root = message.initRoot<MyStruct>();
root.adoptName(kj::mv(orphan));
// orphan est maintenant vide — la donnée appartient au message
```

Les orphans sont alloués dans le même segment que le message, ce qui préserve le zero-copy. Ils sont l'équivalent des « nœuds détachés » dans un DOM.

## Le système RPC natif

### Pourquoi un RPC intégré ?

Protobuf et gRPC sont deux projets distincts : Protobuf gère la sérialisation, gRPC gère le transport, le streaming, les deadlines et les métadonnées. Cette séparation a des avantages (modularité), mais aussi des coûts : gRPC impose HTTP/2 comme transport, ce qui ajoute du framing, du header compression (HPACK), et une couche de complexité significative.

Le RPC de Cap'n Proto est **intégré au format** et au runtime. Il utilise le même format wire pour les messages et les appels de méthode, sans couche HTTP intermédiaire. Le transport est un simple flux bidirectionnel (TCP socket, Unix socket, pipe) sur lequel les messages sont multiplexés.

### Définition d'interfaces

Les interfaces RPC sont définies dans les fichiers `.capnp` aux côtés des structs :

```capnp
@0xb14ce8c4f1a982e3;

struct SensorReading {
  sensorId    @0 : UInt32;
  temperature @1 : Float64;
  timestamp   @2 : UInt64;
}

struct SensorFilter {
  minTemp @0 : Float64;
  maxTemp @1 : Float64;
}

interface SensorService {
  record @0 (reading : SensorReading) -> (id : UInt64);
  getLatest @1 (sensorId : UInt32) -> (reading : SensorReading);
  listFiltered @2 (filter : SensorFilter) -> (readings : List(SensorReading));
}
```

Chaque méthode a un numéro ordinal (comme les champs de struct), des paramètres nommés et typés, et un type de retour structuré. Les paramètres et le retour sont eux-mêmes des structs anonymes générés par le compilateur.

### Implémentation du serveur

```cpp
#include "sensor.capnp.h"
#include <capnp/ez-rpc.h>
#include <print>
#include <vector>
#include <unordered_map>

class SensorServiceImpl final : public SensorService::Server {
public:
    // Chaque méthode RPC reçoit un contexte typé contenant
    // les paramètres (Reader) et le résultat (Builder).

    kj::Promise<void> record(RecordContext context) override {
        auto params = context.getParams();
        auto reading = params.getReading();

        uint64_t id = next_id_++;
        readings_[reading.getSensorId()] = {
            reading.getTemperature(),
            reading.getTimestamp()
        };

        context.getResults().setId(id);
        std::print("Enregistré sensor {} : {:.1f}°C (id={})\n",
                   reading.getSensorId(), reading.getTemperature(), id);
        return kj::READY_NOW;
    }

    kj::Promise<void> getLatest(GetLatestContext context) override {
        auto sensor_id = context.getParams().getSensorId();

        auto it = readings_.find(sensor_id);
        if (it != readings_.end()) {
            auto result = context.getResults().initReading();
            result.setSensorId(sensor_id);
            result.setTemperature(it->second.temp);
            result.setTimestamp(it->second.ts);
        }
        return kj::READY_NOW;
    }

    kj::Promise<void> listFiltered(ListFilteredContext context) override {
        auto filter = context.getParams().getFilter();
        double min_t = filter.getMinTemp();
        double max_t = filter.getMaxTemp();

        std::vector<std::pair<uint32_t, StoredReading>> matches;
        for (auto& [sid, r] : readings_) {
            if (r.temp >= min_t && r.temp <= max_t) {
                matches.push_back({sid, r});
            }
        }

        auto list = context.getResults().initReadings(matches.size());
        for (size_t i = 0; i < matches.size(); ++i) {
            list[i].setSensorId(matches[i].first);
            list[i].setTemperature(matches[i].second.temp);
            list[i].setTimestamp(matches[i].second.ts);
        }
        return kj::READY_NOW;
    }

private:
    struct StoredReading { double temp; uint64_t ts; };
    uint64_t next_id_ = 1;
    std::unordered_map<uint32_t, StoredReading> readings_;
};

int main() {
    capnp::EzRpcServer server(
        kj::heap<SensorServiceImpl>(),
        "0.0.0.0:9000"
    );
    std::print("Serveur RPC Cap'n Proto sur le port 9000\n");

    auto& wait_scope = server.getWaitScope();
    kj::NEVER_DONE.wait(wait_scope);  // Boucle événementielle infinie
    return 0;
}
```

### Implémentation du client

```cpp
#include "sensor.capnp.h"
#include <capnp/ez-rpc.h>
#include <print>

int main() {
    capnp::EzRpcClient client("localhost:9000");
    auto sensor_svc = client.getMain<SensorService>();
    auto& ws = client.getWaitScope();

    // ── Enregistrer une lecture ──
    {
        auto request = sensor_svc.recordRequest();
        auto reading = request.getReading();
        reading.setSensorId(1);
        reading.setTemperature(23.5);
        reading.setTimestamp(1711036800);

        auto response = request.send().wait(ws);
        std::print("Enregistré avec id={}\n", response.getId());
    }

    // ── Récupérer la dernière lecture ──
    {
        auto request = sensor_svc.getLatestRequest();
        request.setSensorId(1);

        auto response = request.send().wait(ws);
        auto r = response.getReading();
        std::print("Dernière lecture sensor {} : {:.1f}°C\n",
                   r.getSensorId(), r.getTemperature());
    }

    // ── Liste filtrée ──
    {
        auto request = sensor_svc.listFilteredRequest();
        auto filter = request.initFilter();
        filter.setMinTemp(20.0);
        filter.setMaxTemp(30.0);

        auto response = request.send().wait(ws);
        for (auto reading : response.getReadings()) {
            std::print("  Sensor {} : {:.1f}°C\n",
                       reading.getSensorId(), reading.getTemperature());
        }
    }

    return 0;
}
```

### Le pipelining de promesses

La fonctionnalité la plus distinctive du RPC Cap'n Proto est le **pipelining de promesses** (promise pipelining). Considérons un scénario où un appel RPC retourne une capability (un objet distant), et on veut immédiatement appeler une méthode sur cet objet :

```capnp
interface Directory {
  lookup @0 (name : Text) -> (file : File);
}

interface File {
  read @0 (offset : UInt64, size : UInt32) -> (data : Data);
}
```

**Sans pipelining** (modèle gRPC classique) — il faut attendre chaque réponse avant d'envoyer la requête suivante :

```
Client                              Serveur
  │                                   │
  │── lookup("config.yaml") ────────► │
  │                                   │  (traitement)
  │◄── file: <handle> ──────────────  │
  │                                   │
  │── file.read(0, 1024) ──────────►  │
  │                                   │  (traitement)
  │◄── data: <bytes> ───────────────  │
  │                                   │

  Latence totale : 2 allers-retours réseau
```

**Avec pipelining** — on appelle `read()` sur la *promesse* du fichier, avant même de recevoir la réponse du `lookup()` :

```
Client                              Serveur
  │                                   │
  │── lookup("config.yaml") ────────► │
  │── file.read(0, 1024) ───────────► │  ← envoyé immédiatement,
  │                                   │    sans attendre la réponse
  │                                   │  (traitement lookup)
  │                                   │  (traitement read)
  │◄── data: <bytes> ───────────────  │
  │                                   │

  Latence totale : 1 aller-retour réseau
```

En code C++ :

```cpp
auto lookup_promise = directory.lookupRequest();
lookup_promise.setName("config.yaml");

auto file = lookup_promise.send().getFile();
// file est une PROMESSE, pas encore le résultat

auto read_promise = file.readRequest();
read_promise.setOffset(0);
read_promise.setSize(1024);

// Les deux requêtes sont envoyées ensemble (pipelinées)
auto response = read_promise.send().wait(ws);
auto data = response.getData();
```

Le serveur reçoit les deux requêtes et les exécute séquentiellement sans aller-retour intermédiaire. Pour des chaînes de N appels, le pipelining réduit la latence de N allers-retours à **un seul**, ce qui est particulièrement significatif sur des liens à latence élevée (WAN, inter-datacenter).

## La librairie KJ

Le RPC et le runtime Cap'n Proto s'appuient sur **KJ** (Kenton's Junkyard), une librairie utilitaire C++ incluse dans Cap'n Proto. KJ fournit les primitives d'asynchronisme (promises, event loop), les I/O, et les structures de base utilisées en interne.

Les types KJ que vous rencontrerez le plus souvent :

| Type KJ | Rôle | Équivalent approximatif |
|---------|------|-------------------------|
| `kj::Promise<T>` | Promesse asynchrone | `std::future<T>` (mais non-bloquant) |
| `kj::String` | Chaîne possédée | `std::string` |
| `kj::StringPtr` | Vue sur chaîne | `std::string_view` |
| `kj::ArrayPtr<T>` | Vue sur tableau | `std::span<T>` |
| `kj::Own<T>` | Pointeur possédant | `std::unique_ptr<T>` |
| `kj::Maybe<T>` | Valeur optionnelle | `std::optional<T>` |
| `kj::heap<T>(args...)` | Allocation sur le tas | `std::make_unique<T>(args...)` |

KJ utilise ses propres types plutôt que ceux de la STL pour des raisons historiques (Cap'n Proto a précédé l'adoption généralisée de C++14/17) et pour un contrôle fin sur la gestion mémoire. Dans du code applicatif, vous pouvez convertir entre les types KJ et STL facilement :

```cpp
// KJ → STL
kj::StringPtr kj_str = "hello";
std::string_view sv{kj_str.cStr(), kj_str.size()};
std::string s{kj_str.cStr()};

// STL → KJ
std::string std_str = "world";
kj::StringPtr kj_view{std_str.c_str()};
```

## Résumé

Le langage de schéma Cap'n Proto offre un système de types riche (generics, unions avec groupes, constantes) avec des règles d'évolution strictes appliquées à la compilation. Le code généré produit des Builders et Readers zero-copy qui manipulent directement le buffer wire.

Le RPC intégré élimine la couche HTTP/2 de gRPC au profit d'un protocole binaire léger sur connexion directe, avec le pipelining de promesses comme fonctionnalité différenciante majeure pour les scénarios multi-étapes.
