# 25.1.2 — Génération de code C++

## Sous-section de 25.1 — Protocol Buffers (Protobuf) : Sérialisation Google

---

## Vue d'ensemble

La section 25.1.1 a montré comment décrire des messages dans des fichiers `.proto`. Cette section couvre l'étape suivante : la transformation de ces descriptions en code C++ exploitable. Le compilateur `protoc` lit les fichiers `.proto` et produit, pour chaque message, une classe C++ complète avec des accesseurs typés, des méthodes de sérialisation/désérialisation, et une gestion mémoire intégrée.

Comprendre la structure du code généré est important pour deux raisons. D'abord, c'est ce code que l'on appelle quotidiennement dans l'application — connaître ses conventions permet de l'utiliser efficacement. Ensuite, le code généré est volumineux et possède ses propres caractéristiques de performance, ce qui influence les décisions d'architecture du build system.

---

## Invocation de `protoc`

### Génération manuelle (ligne de commande)

La commande de base pour générer du C++ depuis un fichier `.proto` :

```bash
protoc --cpp_out=generated/ proto/config.proto
```

Cette commande produit deux fichiers dans le répertoire `generated/` :

- `config.pb.h` — déclarations des classes (header).
- `config.pb.cc` — implémentations (sérialisation, accesseurs, etc.).

Si le fichier `.proto` importe d'autres fichiers, `protoc` doit savoir où les trouver via l'option `--proto_path` (ou `-I`) :

```bash
protoc \
    --proto_path=proto/ \
    --proto_path=/usr/include/ \
    --cpp_out=generated/ \
    proto/config.proto \
    proto/common/types.proto
```

Le `--proto_path=/usr/include/` est nécessaire pour résoudre les imports des types Well-Known de Google (`google/protobuf/timestamp.proto`, etc.), dont les fichiers `.proto` sont installés dans `/usr/include/` sur Ubuntu (via le paquet `libprotobuf-dev`).

### Options de génération

Plusieurs options contrôlent le code C++ produit :

```bash
# Génération standard
protoc --cpp_out=generated/ config.proto

# Génération avec Lite runtime (binaire plus petit, moins de fonctionnalités)
protoc --cpp_out=lite:generated/ config.proto
```

L'option `lite` génère du code qui linke contre `libprotobuf-lite` au lieu de `libprotobuf`. Le runtime lite ne supporte pas la réflexion (`Descriptor`, `Reflection`), la conversion JSON, ni le décodage par nom de champ. En contrepartie, il réduit la taille du binaire d'environ 1-2 Mo. C'est le choix approprié pour les applications embarquées ou mobiles.

## Intégration CMake

La génération manuelle ne s'intègre pas dans un workflow de build reproductible. CMake fournit des mécanismes pour invoquer `protoc` automatiquement lors de la phase de configuration ou de build.

### Méthode 1 : `protobuf_generate_cpp` (basique)

```cmake
cmake_minimum_required(VERSION 3.20)  
project(myapp LANGUAGES CXX)  

set(CMAKE_CXX_STANDARD 23)  
set(CMAKE_CXX_STANDARD_REQUIRED ON)  

find_package(Protobuf REQUIRED)

# Génère les .pb.h et .pb.cc dans CMAKE_CURRENT_BINARY_DIR
protobuf_generate_cpp(PROTO_SRCS PROTO_HDRS
    proto/config.proto
    proto/common/types.proto
)

add_executable(myapp
    src/main.cpp
    ${PROTO_SRCS}
)

target_include_directories(myapp PRIVATE ${CMAKE_CURRENT_BINARY_DIR})  
target_link_libraries(myapp PRIVATE protobuf::libprotobuf)  
```

Les variables `PROTO_SRCS` et `PROTO_HDRS` sont remplies par la macro avec les chemins des fichiers générés. Le `target_include_directories` vers `CMAKE_CURRENT_BINARY_DIR` est nécessaire car les fichiers `.pb.h` y sont produits.

### Méthode 2 : `protobuf_generate` sur une cible (recommandée)

Depuis CMake 3.13, la fonction `protobuf_generate` peut être associée directement à une cible. C'est l'approche recommandée pour les projets modernes :

```cmake
cmake_minimum_required(VERSION 3.20)  
project(myapp LANGUAGES CXX)  

set(CMAKE_CXX_STANDARD 23)  
set(CMAKE_CXX_STANDARD_REQUIRED ON)  

find_package(Protobuf REQUIRED)

# Librairie dédiée au code Protobuf
add_library(proto_lib
    proto/config.proto
    proto/common/types.proto
)

target_link_libraries(proto_lib PUBLIC protobuf::libprotobuf)  
target_include_directories(proto_lib PUBLIC ${CMAKE_CURRENT_BINARY_DIR})  

protobuf_generate(
    TARGET proto_lib
    LANGUAGE cpp
    IMPORT_DIRS ${CMAKE_CURRENT_SOURCE_DIR}/proto
)

# Application principale
add_executable(myapp src/main.cpp)  
target_link_libraries(myapp PRIVATE proto_lib)  
```

Cette approche présente plusieurs avantages. Isoler le code Protobuf dans une librairie dédiée (`proto_lib`) évite de recompiler les fichiers `.pb.cc` quand seul le code applicatif change. Le `PUBLIC` sur `target_include_directories` et `target_link_libraries` propage automatiquement les chemins d'inclusion et la dépendance à `libprotobuf` vers les cibles consommatrices. Enfin, `IMPORT_DIRS` indique à `protoc` où chercher les fichiers importés.

### Méthode 3 : Intégration avec Conan 2

Quand Protobuf est installé via Conan, le binaire `protoc` est fourni comme outil de build. La configuration CMake s'adapte légèrement :

```python
# conanfile.py
def requirements(self):
    self.requires("protobuf/5.27.0")

def build_requirements(self):
    self.tool_requires("protobuf/5.27.0")
```

Le `tool_requires` garantit que `protoc` est disponible sur la machine de build (il peut différer de la version runtime en cas de cross-compilation). Le `CMakeLists.txt` reste identique — `find_package(Protobuf)` résout les chemins fournis par le générateur CMakeDeps de Conan.

### Organisation recommandée des fichiers

Pour un projet de taille moyenne, la structure suivante est typique :

```
myproject/
├── CMakeLists.txt
├── proto/
│   ├── myapp/
│   │   ├── config.proto
│   │   ├── user.proto
│   │   └── events.proto
│   └── common/
│       └── types.proto
├── src/
│   ├── main.cpp
│   ├── config_loader.cpp
│   └── config_loader.h
└── build/
    └── (fichiers .pb.h et .pb.cc générés ici)
```

Les fichiers `.proto` sont regroupés dans un répertoire `proto/`, organisés en sous-répertoires reflétant le package Protobuf. Les fichiers générés ne sont jamais commités dans le dépôt Git — ils sont produits à chaque build. Un `.gitignore` à la racine doit exclure `*.pb.h` et `*.pb.cc` du suivi de version.

## Structure du code généré

Pour chaque message défini dans un fichier `.proto`, `protoc` génère une classe C++ complète. Examinons le code produit pour un message concret :

```protobuf
// user.proto
syntax = "proto3";  
package myapp;  

enum Role {
    ROLE_UNSPECIFIED = 0;
    ROLE_ADMIN = 1;
    ROLE_USER = 2;
    ROLE_VIEWER = 3;
}

message User {
    int32 id = 1;
    string name = 2;
    string email = 3;
    Role role = 4;
    optional string department = 5;
    repeated string tags = 6;
}
```

Le code généré dans `user.pb.h` contient (en simplifié) :

```cpp
namespace myapp {

// Enum généré
enum Role : int {
    ROLE_UNSPECIFIED = 0,
    ROLE_ADMIN = 1,
    ROLE_USER = 2,
    ROLE_VIEWER = 3,
};

// Fonctions utilitaires pour l'enum
bool Role_IsValid(int value);  
const std::string& Role_Name(Role value);  
bool Role_Parse(const std::string& name, Role* value);  

class User final : public google::protobuf::Message {  
public:  
    // Constructeurs et destructeur
    User();
    ~User() override;
    User(const User& from);            // copie
    User(User&& from) noexcept;        // mouvement
    User& operator=(const User& from); // affectation copie
    User& operator=(User&& from) noexcept; // affectation mouvement

    // --- Accesseurs pour 'id' (int32, champ 1) ---
    int32_t id() const;
    void set_id(int32_t value);
    void clear_id();

    // --- Accesseurs pour 'name' (string, champ 2) ---
    const std::string& name() const;
    void set_name(const std::string& value);
    void set_name(std::string&& value);
    void set_name(const char* value);
    std::string* mutable_name();
    void clear_name();

    // --- Accesseurs pour 'email' (string, champ 3) ---
    // (même pattern que name)

    // --- Accesseurs pour 'role' (enum, champ 4) ---
    myapp::Role role() const;
    void set_role(myapp::Role value);
    void clear_role();

    // --- Accesseurs pour 'department' (optional string, champ 5) ---
    bool has_department() const;        // ← présence explicite (optional)
    const std::string& department() const;
    void set_department(const std::string& value);
    void set_department(std::string&& value);
    std::string* mutable_department();
    void clear_department();

    // --- Accesseurs pour 'tags' (repeated string, champ 6) ---
    int tags_size() const;
    const std::string& tags(int index) const;
    std::string* mutable_tags(int index);
    void set_tags(int index, const std::string& value);
    std::string* add_tags();
    void add_tags(const std::string& value);
    void add_tags(std::string&& value);
    const google::protobuf::RepeatedPtrField<std::string>& tags() const;
    google::protobuf::RepeatedPtrField<std::string>* mutable_tags();
    void clear_tags();

    // --- Sérialisation / Désérialisation ---
    bool SerializeToString(std::string* output) const;
    bool SerializeToOstream(std::ostream* output) const;
    bool SerializeToArray(void* data, int size) const;
    size_t ByteSizeLong() const;

    bool ParseFromString(const std::string& data);
    bool ParseFromIstream(std::istream* input);
    bool ParseFromArray(const void* data, int size);

    // --- Utilitaires ---
    void CopyFrom(const User& from);
    void MergeFrom(const User& from);
    void Clear();
    bool IsInitialized() const;
    std::string DebugString() const;     // représentation texte pour le debug
    std::string ShortDebugString() const;

    // --- Comparaison (protobuf 4+) ---
    bool operator==(const User& other) const;
    bool operator!=(const User& other) const;
    // Note : en protobuf 3.x, utiliser
    // google::protobuf::util::MessageDifferencer::Equals(a, b)

    // --- Swap (efficace, sans copie) ---
    void Swap(User* other);

    // ...
};

}  // namespace myapp
```

### Patterns d'accesseurs par type de champ

Le code généré suit des patterns systématiques selon le type et le qualificateur du champ :

**Scalaires (`int32`, `bool`, `double`, `enum`)** :

```cpp
T field() const;             // lecture  
void set_field(T value);     // écriture  
void clear_field();          // reset à la valeur par défaut  
```

**Chaînes (`string`, `bytes`)** :

```cpp
const std::string& field() const;     // lecture (référence const)  
void set_field(const std::string&);   // écriture (copie)  
void set_field(std::string&&);        // écriture (move)  
void set_field(const char*);          // écriture (C string)  
std::string* mutable_field();         // accès modifiable direct  
void clear_field();                   // reset à ""  
```

`mutable_field()` retourne un pointeur vers la chaîne interne, permettant de la modifier sur place sans copie. C'est utile pour construire des chaînes de manière incrémentale ou pour les remplir depuis une API C.

**Sous-messages** :

```cpp
bool has_field() const;                     // test de présence  
const SubMessage& field() const;            // lecture (référence const)  
SubMessage* mutable_field();                // accès modifiable (crée si absent)  
void clear_field();                         // supprime le sous-message  
SubMessage* release_field();                // transfère la propriété à l'appelant  
void set_allocated_field(SubMessage* msg);  // prend la propriété d'un message existant  
```

`mutable_field()` crée le sous-message s'il n'existe pas encore et retourne un pointeur modifiable. Le message parent possède le sous-message — le pointeur est valide tant que le parent existe et que `clear_field()` n'est pas appelé.

`release_field()` et `set_allocated_field()` gèrent le transfert de propriété, utiles pour déplacer un sous-message d'un parent à un autre sans copie.

**Champs repeated** :

```cpp
int field_size() const;                      // nombre d'éléments  
const T& field(int index) const;             // accès par index  
T* mutable_field(int index);                 // accès modifiable par index  
T* add_field();                              // ajoute un élément vide, retourne un pointeur  
void add_field(const T& value);              // ajoute une copie (scalaires et strings)  
const RepeatedField<T>& field() const;       // accès au conteneur entier  
RepeatedField<T>* mutable_field();           // accès modifiable au conteneur  
void clear_field();                          // vide le conteneur  
```

Pour les types scalaires, le conteneur est `RepeatedField<T>`. Pour les messages et les chaînes, c'est `RepeatedPtrField<T>`. Les deux supportent les itérateurs STL et sont compatibles avec les range-based for loops.

**Maps** :

```cpp
const Map<K, V>& field() const;              // lecture  
Map<K, V>* mutable_field();                  // accès modifiable  
int field_size() const;                      // nombre d'entrées  
```

Le type `Map<K, V>` de Protobuf supporte l'interface standard d'un conteneur associatif (`operator[]`, `find`, `insert`, itérateurs).

## Utilisation courante du code généré

### Construction et sérialisation

```cpp
#include "user.pb.h"
#include <print>
#include <cassert>

myapp::User create_user() {
    myapp::User user;
    user.set_id(42);
    user.set_name("Alice Martin");
    user.set_email("alice@example.com");
    user.set_role(myapp::ROLE_ADMIN);
    user.set_department("Engineering");

    user.add_tags("senior");
    user.add_tags("on-call");

    return user;  // move semantics supportées
}

int main() {
    auto user = create_user();

    // Sérialisation binaire
    std::string binary;
    bool ok = user.SerializeToString(&binary);
    assert(ok);
    std::print("Taille binaire : {} octets\n", binary.size());

    // Debug string (lisible, pour le logging)
    std::print("Debug :\n{}\n", user.DebugString());
}
```

La méthode `DebugString()` produit une représentation texte lisible, indispensable pour le logging et le débogage :

```
id: 42  
name: "Alice Martin"  
email: "alice@example.com"  
role: ROLE_ADMIN  
department: "Engineering"  
tags: "senior"  
tags: "on-call"  
```

### Désérialisation et lecture

```cpp
myapp::User restored;  
if (!restored.ParseFromString(binary)) {  
    std::print(stderr, "Désérialisation échouée\n");
    return 1;
}

std::print("Nom  : {}\n", restored.name());  
std::print("Role : {}\n", myapp::Role_Name(restored.role()));  

if (restored.has_department()) {
    std::print("Dept : {}\n", restored.department());
}

for (const auto& tag : restored.tags()) {
    std::print("Tag  : {}\n", tag);
}
```

### Copie, move et comparaison

Les classes générées supportent la copie, le mouvement et la comparaison :

```cpp
myapp::User user1 = create_user();

// Copie
myapp::User user2 = user1;  
assert(user1 == user2);  // protobuf 4+ : operator== généré  
// En protobuf 3.x : google::protobuf::util::MessageDifferencer::Equals(user1, user2)

// Modification de la copie
user2.set_name("Bob");  
assert(user1 != user2);  

// Move (efficace, sans copie profonde)
myapp::User user3 = std::move(user1);
// user1 est maintenant dans un état valide mais non spécifié

// Swap (très efficace, échange les pointeurs internes)
user2.Swap(&user3);
```

`Swap` est l'opération la plus efficace pour échanger deux messages. Elle échange les pointeurs internes sans copier les données, quelle que soit la taille du message.

### MergeFrom : fusion de messages

`MergeFrom` fusionne les champs d'un message source dans le message courant. Les champs scalaires non-default écrasent les valeurs existantes, les champs repeated sont concaténés, et les sous-messages sont fusionnés récursivement :

```cpp
myapp::User base;  
base.set_name("Alice");  
base.set_role(myapp::ROLE_USER);  
base.add_tags("team-a");  

myapp::User override_msg;  
override_msg.set_role(myapp::ROLE_ADMIN);  // écrase  
override_msg.add_tags("on-call");          // concaténé  

base.MergeFrom(override_msg);
// base.name() == "Alice"          (inchangé, override n'avait pas de name)
// base.role() == ROLE_ADMIN       (écrasé)
// base.tags() == ["team-a", "on-call"]  (concaténé)
```

Ce mécanisme est utile pour les configurations en couches (défauts → overrides) ou pour assembler un message à partir de plusieurs sources.

## Conversion JSON

Le runtime Protobuf standard (pas lite) fournit une conversion bidirectionnelle entre messages Protobuf et JSON, via `google::protobuf::util::MessageToJsonString` et `JsonStringToMessage` :

```cpp
#include "user.pb.h"
#include <google/protobuf/util/json_util.h>
#include <print>

int main() {
    myapp::User user = create_user();

    // Protobuf → JSON
    std::string json_output;
    google::protobuf::util::JsonPrintOptions options;
    options.add_whitespace = true;              // pretty-print
    options.always_print_primitive_fields = true; // inclure les champs à valeur zéro
    options.preserve_proto_field_names = true;   // snake_case (sinon camelCase)

    auto status = google::protobuf::util::MessageToJsonString(
        user, &json_output, options);

    if (status.ok()) {
        std::print("JSON :\n{}\n", json_output);
    }

    // JSON → Protobuf
    std::string json_input = R"({
        "id": 99,
        "name": "Bob",
        "email": "bob@example.com",
        "role": "ROLE_USER",
        "tags": ["new-hire"]
    })";

    myapp::User from_json;
    google::protobuf::util::JsonParseOptions parse_options;
    parse_options.ignore_unknown_fields = true;  // tolérer les champs inconnus

    status = google::protobuf::util::JsonStringToMessage(
        json_input, &from_json, parse_options);

    if (status.ok()) {
        std::print("Depuis JSON : {} ({})\n",
            from_json.name(), myapp::Role_Name(from_json.role()));
    }
}
```

Cette conversion est précieuse pour l'interopérabilité. Un service C++ peut accepter des requêtes JSON d'un client web et les convertir en messages Protobuf pour le traitement interne, puis retourner les résultats en JSON. Les noms de champs suivent le mapping JSON canonique de Protobuf (camelCase par défaut, snake_case avec `preserve_proto_field_names`), et les enums sont représentés par leurs noms de chaîne.

## Arenas : allocation groupée

Pour les applications à haute performance qui créent et détruisent de nombreux messages Protobuf (serveurs gRPC traitant des milliers de requêtes par seconde), les allocations mémoire individuelles peuvent devenir un goulot d'étranglement. Les **arenas** regroupent les allocations d'un message et de tous ses sous-messages dans un bloc mémoire contigu, libéré en une seule opération :

```protobuf
// Dans le .proto, activer le support arenas
option cc_enable_arenas = true;
```

```cpp
#include <google/protobuf/arena.h>

void process_request() {
    google::protobuf::Arena arena;

    // Alloué sur l'arena — pas de new/delete individuel
    auto* request = google::protobuf::Arena::CreateMessage<myapp::User>(&arena);
    request->set_name("Alice");
    request->set_id(42);

    // Les sous-messages alloués via mutable_*() sont aussi sur l'arena
    // ...

    // Toute la mémoire est libérée quand 'arena' sort du scope
    // Pas de destructeurs individuels appelés
}
```

Les arenas sont particulièrement efficaces quand un message contient de nombreux sous-messages ou des champs repeated volumineux. Le gain typique est une réduction de 20 à 50% du temps passé dans l'allocateur pour les workloads message-intensive.

> ⚠️ *Les messages alloués sur une arena ne doivent pas être utilisés après la destruction de l'arena. Ils ne peuvent pas non plus être déplacés (`std::move`) hors de l'arena. L'arena possède toute la mémoire — c'est un pattern RAII à granularité large.*

## Taille du code généré et impact sur la compilation

Le code généré par `protoc` est volumineux. Un fichier `.proto` de 50 lignes peut produire un `.pb.h` de 1000-2000 lignes et un `.pb.cc` de taille comparable. Pour un projet avec de nombreux messages, le code Protobuf généré peut représenter une portion significative du code total compilé.

### Stratégies de mitigation

**Isoler dans une librairie dédiée.** Comme montré dans la configuration CMake recommandée, placer tout le code Protobuf généré dans une librairie statique (`proto_lib`) permet de ne le recompiler que quand les fichiers `.proto` changent, pas à chaque modification du code applicatif.

**Utiliser ccache.** Le code généré est déterministe — un même `.proto` produit toujours le même code C++. ccache (section 2.3) est très efficace dans ce contexte car le cache est systématiquement valide tant que le `.proto` ne change pas.

**Forward declarations.** Comme pour nlohmann/json (section 24.1.1), éviter d'inclure les `.pb.h` dans les headers du projet. Utiliser des forward declarations quand seul un pointeur ou une référence au type Protobuf est nécessaire dans l'interface :

```cpp
// service.h — forward declaration, pas d'include .pb.h
namespace myapp { class User; }

class UserService {  
public:  
    void process(const myapp::User& user);
};
```

```cpp
// service.cpp — include complet uniquement ici
#include "service.h"
#include "user.pb.h"

void UserService::process(const myapp::User& user) {
    // utilisation complète de l'API Protobuf
}
```

**Protobuf Lite.** Si la réflexion et la conversion JSON ne sont pas nécessaires, le runtime lite réduit à la fois la taille des headers et celle du binaire final.

## Récapitulatif des méthodes clés

| Catégorie | Méthode | Description |
|-----------|---------|-------------|
| **Sérialisation** | `SerializeToString(&str)` | Encode en binaire dans une string |
| | `SerializeToOstream(&os)` | Encode vers un flux de sortie |
| | `SerializeToArray(ptr, size)` | Encode dans un buffer |
| | `ByteSizeLong()` | Taille du message sérialisé |
| **Désérialisation** | `ParseFromString(str)` | Décode depuis une string binaire |
| | `ParseFromIstream(&is)` | Décode depuis un flux d'entrée |
| | `ParseFromArray(ptr, size)` | Décode depuis un buffer |
| **Utilitaires** | `DebugString()` | Représentation texte lisible |
| | `ShortDebugString()` | Idem, sur une seule ligne |
| | `Clear()` | Reset tous les champs |
| | `CopyFrom(other)` | Copie profonde |
| | `MergeFrom(other)` | Fusion de champs |
| | `Swap(other)` | Échange sans copie |

| | `operator==` / `!=` | Comparaison structurelle |
| **JSON** | `MessageToJsonString(msg, &str)` | Protobuf → JSON |
| | `JsonStringToMessage(str, &msg)` | JSON → Protobuf |
