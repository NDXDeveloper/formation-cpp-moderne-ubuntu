# 25.3.3 Cap'n Proto vs FlatBuffers vs Protobuf : compromis

## Introduction

Les sections précédentes de ce chapitre ont présenté Protobuf (25.1), FlatBuffers (25.2) et Cap'n Proto (25.3) individuellement. Cette section les confronte directement sur les axes qui comptent en pratique : performance de sérialisation, taille des messages, modèle d'accès aux données, évolution de schéma, écosystème et ergonomie du code. L'objectif n'est pas de désigner un « gagnant » — chaque format excelle dans des contextes différents — mais de vous donner les clés pour un choix éclairé.

---

## Rappel des philosophies

Chaque format incarne une philosophie de conception distincte qui conditionne tous ses compromis :

**Protobuf** privilégie la **compacité du format wire et l'universalité**. Le message est encodé dans un format compact (varints, length-delimited) qui minimise la bande passante, au prix d'un cycle complet encode/decode à chaque extrémité. La priorité est l'interopérabilité entre des dizaines de langages et des milliards de messages par jour chez Google.

**FlatBuffers** privilégie la **lecture zero-copy avec une construction flexible**. Le message est construit via un builder qui produit un buffer plat (flat buffer) lisible sans désérialisation. La construction nécessite un passage par le builder (pas de zero-copy en écriture), mais la lecture est un accès direct par offset. La priorité est le gaming et le mobile (Google l'a créé pour les besoins internes d'Android et de jeux).

**Cap'n Proto** privilégie le **zero-copy intégral dans les deux directions**. Le format mémoire est le format wire — pas de transformation ni à l'écriture ni à la lecture. La priorité est la performance brute des chemins I/O (IPC, proxies, bases de données) sur un nombre restreint de plateformes.

```
                    Compacité wire
                         ▲
                         │
                Protobuf ●
                         │
                         │
                         │
         FlatBuffers ●   │   ● MessagePack
                         │
              Cap'n Proto ●
                         │
         ────────────────┼──────────────────► Vitesse
              encode/    │           zero-copy
              decode     │           (pas de parsing)
```

## Comparaison des performances

### Modèle de coût théorique

| Opération | Protobuf | FlatBuffers | Cap'n Proto |
|-----------|----------|-------------|-------------|
| **Construction** | O(N) — sérialise chaque champ | O(N) — builder assemble le buffer | O(N) — écrit directement dans le buffer |
| **Envoi** | Copie du buffer sérialisé | Copie du buffer construit | Aucune transformation (le buffer est le message) |
| **Réception** | O(N) — parse tout le message | O(1) — vérification optionnelle | O(1) — vérification optionnelle |
| **Accès à 1 champ** | O(1) après parsing | O(1) — offset dans le buffer | O(1) — offset dans le buffer |
| **Accès à K champs** | O(N) parsing + O(K) accès | O(K) accès directs | O(K) accès directs |
| **Forwarding** (proxy) | O(N) parse + O(N) reserialize | O(1) copie du buffer | O(1) copie du buffer |

Le cas du **forwarding** est le plus révélateur : un proxy Protobuf qui reçoit un message, lit un champ de routage et le retransmet doit payer le coût complet de désérialisation + resérialisation. Avec FlatBuffers et Cap'n Proto, c'est une copie de buffer et un accès par offset.

### Benchmarks indicatifs

Les chiffres suivants proviennent de benchmarks publics (notamment ceux maintenus par les projets respectifs et par des tiers indépendants). Ils sont indicatifs et varient selon le message, le compilateur, le matériel et les options d'optimisation :

**Message moyen (~200 octets sérialisés, 10 champs mixtes scalaires + strings) :**

| Métrique | Protobuf | FlatBuffers | Cap'n Proto |
|----------|----------|-------------|-------------|
| Temps d'encodage | ~500 ns | ~300 ns | ~200 ns |
| Temps de décodage | ~400 ns | ~0 ns (zero-copy) | ~0 ns (zero-copy) |
| Temps total (encode + decode) | ~900 ns | ~300 ns | ~200 ns |
| Accès à 1 champ (après réception) | ~5 ns | ~5 ns | ~5 ns |

**Message volumineux (~10 Ko, listes de structs, strings longues) :**

| Métrique | Protobuf | FlatBuffers | Cap'n Proto |
|----------|----------|-------------|-------------|
| Temps d'encodage | ~8 µs | ~5 µs | ~3 µs |
| Temps de décodage | ~6 µs | ~0 µs | ~0 µs |
| Temps total | ~14 µs | ~5 µs | ~3 µs |

**Observations :**

- La différence entre FlatBuffers et Cap'n Proto en construction est modeste mais réelle : le builder de FlatBuffers effectue une passe de « finalization » pour résoudre les offsets (les données sont écrites back-to-front dans le buffer), tandis que Cap'n Proto écrit directement dans l'ordre.  
- Pour les lectures partielles (accès à 1-2 champs dans un gros message), FlatBuffers et Cap'n Proto sont équivalents et radicalement plus rapides que Protobuf (qui doit parser l'intégralité du message avant d'accéder au moindre champ).  
- Protobuf 3 avec le mode « lazy parsing » (Arena allocation) réduit l'écart pour les gros messages dont on ne lit qu'une partie, mais ne rivalise pas avec le zero-copy natif.

### Profil mémoire

| Métrique | Protobuf | FlatBuffers | Cap'n Proto |
|----------|----------|-------------|-------------|
| Allocations à la construction | Multiples (strings, sous-messages) | 1 (buffer du builder) | 1 (segment(s) du message) |
| Allocations à la lecture | Multiples (objets C++ générés) | 0 | 0 |
| Mémoire résidente (message de 1 Ko) | ~2-3 Ko (objets + overhead) | ~1 Ko (buffer seul) | ~1-1.5 Ko (buffer + padding) |

L'absence d'allocations à la lecture est un avantage majeur de FlatBuffers et Cap'n Proto dans les boucles de traitement à haute fréquence, où la pression sur l'allocateur (et donc sur le garbage collector dans les langages gérés) peut devenir un goulot d'étranglement.

## Comparaison de la taille wire

La taille des messages sur le réseau ou le disque affecte la bande passante, les coûts de stockage et le temps de transfert. C'est le domaine où Protobuf conserve un avantage structurel.

**Message exemple : `{id: 42, temperature: 23.5, location: "Paris", tags: ["indoor", "lab"]}` :**

| Format | Taille (octets) | Ratio vs Protobuf |
|--------|-----------------|-------------------|
| Protobuf | ~35 | 1.0× |
| Cap'n Proto (non-packed) | ~88 | 2.5× |
| Cap'n Proto (packed) | ~50 | 1.4× |
| FlatBuffers | ~72 | 2.1× |
| JSON (pour référence) | ~82 | 2.3× |

**Pourquoi cette différence ?**

Protobuf utilise des **varints** (entiers de taille variable) : la valeur 42 s'encode en 1 octet au lieu de 4. Les champs sont identifiés par des tags de quelques octets au lieu de pointeurs 64 bits. Le format est optimisé pour la compacité au détriment de l'accès aléatoire.

FlatBuffers et Cap'n Proto utilisent des **tailles fixes et un alignement** : un `uint32` occupe toujours 4 octets (même pour la valeur 0), et le padding d'alignement à 4 ou 8 octets ajoute des octets nuls. Ce sont ces zéros et ce padding qui permettent l'accès par offset constant — ils sont le prix structurel du zero-copy.

**Impact pratique :**

- Sur un **réseau local** (1-10 Gbps), la différence de taille est rarement significative. Le temps de sérialisation/désérialisation domine largement le temps de transfert.  
- Sur un **WAN** ou un lien mobile (latence élevée, bande passante limitée), la compacité de Protobuf peut être un avantage réel. Le format packed de Cap'n Proto réduit l'écart mais ne l'élimine pas.  
- En **stockage**, des millions d'enregistrements à 2.5× la taille s'accumulent. Une compression externe (zstd, lz4) nivelle largement les différences entre les trois formats — la taille compressée est comparable.

## Comparaison du modèle d'accès aux données

### Construction des messages

| Aspect | Protobuf | FlatBuffers | Cap'n Proto |
|--------|----------|-------------|-------------|
| Ordre de construction | Libre (objet mutable) | Contraint (bottom-up, back-to-front) | Libre (builders mutables) |
| Modification après construction | Oui (objet mutable) | Non (buffer immutable une fois fini) | Oui (via builders) |
| Strings et sous-messages | Copies dans des objets C++ | Doivent être créés avant le parent | Écrits directement dans le segment |

FlatBuffers impose la contrainte la plus forte : les objets enfants (strings, sous-tables) doivent être créés *avant* le parent dans le builder, car le buffer est construit de la fin vers le début. Cela oblige à planifier l'ordre de construction, ce qui peut compliquer le code quand les données arrivent dans un ordre arbitraire.

Cap'n Proto est le plus flexible à la construction : les builders sont mutables et permettent d'écrire les champs dans n'importe quel ordre, y compris de revenir modifier un champ déjà écrit.

### Lecture des messages

| Aspect | Protobuf | FlatBuffers | Cap'n Proto |
|--------|----------|-------------|-------------|
| Parsing initial | Obligatoire (O(N)) | Aucun (zero-copy) | Aucun (zero-copy) |
| Validation | Pendant le parsing | Optionnelle (`Verifier`) | Automatique (traversal limits) |
| Durée de vie des données | Indépendante du buffer | Liée au buffer | Liée au buffer |
| Accès aux strings | `std::string` (copié) | `const char*` dans le buffer | `capnp::Text::Reader` dans le buffer |

La durée de vie est le compromis le plus important en pratique. Avec Protobuf, une fois le message parsé, l'objet C++ est indépendant — le buffer réseau peut être libéré immédiatement. Avec FlatBuffers et Cap'n Proto, les données lues sont des vues sur le buffer reçu — il faut maintenir le buffer vivant aussi longtemps qu'on accède aux champs.

## Comparaison de l'évolution de schéma

| Opération | Protobuf | FlatBuffers | Cap'n Proto |
|-----------|----------|-------------|-------------|
| Ajouter un champ | ✅ (n'importe quel numéro libre) | ✅ (en fin de table) | ✅ (prochain ordinal) |
| Supprimer un champ | ✅ (marquer `reserved`) | ✅ (déprécier) | ✅ (remplacer par `Void`) |
| Renommer un champ | ✅ | ✅ | ✅ |
| Changer le type d'un champ | ⚠️ (limité, certains casts) | ❌ | ❌ |
| Réutiliser un numéro supprimé | ❌ (`reserved`) | ❌ | ❌ |
| Détection des erreurs | À l'exécution (parsing) | À l'exécution | À la compilation du schéma |

Cap'n Proto a l'avantage de la **détection au compile-time** : si vous tentez une modification incompatible (changer le type d'un champ, sauter un numéro ordinal), `capnp compile` refuse de générer le code. Avec Protobuf, certaines incompatibilités ne se manifestent qu'à l'exécution, quand un lecteur ancien reçoit un message d'un format nouveau.

## Comparaison de l'écosystème

### Support des langages

| Langage | Protobuf | FlatBuffers | Cap'n Proto |
|---------|----------|-------------|-------------|
| C++ | ✅ Officiel | ✅ Officiel | ✅ Officiel (référence) |
| Rust | ✅ (prost, tonic) | ✅ Officiel | ✅ (capnp-rust, mature) |
| Go | ✅ Officiel | ✅ Officiel | ✅ (go-capnp, mature) |
| Python | ✅ Officiel | ✅ Officiel | ⚠️ (pycapnp, communauté) |
| Java / Kotlin | ✅ Officiel | ✅ Officiel | ⚠️ (communauté, moins actif) |
| C# | ✅ Officiel | ✅ Officiel | ⚠️ (communauté) |
| TypeScript / JS | ✅ (protobuf.js) | ✅ Officiel | ✅ (capnp-ts, actif) |
| Swift | ✅ (swift-protobuf) | ✅ Officiel | ⚠️ (limité) |
| Dart | ✅ Officiel | ✅ Officiel | ❌ |

Protobuf domine largement par l'étendue de son support. FlatBuffers couvre les plateformes de gaming et mobiles (y compris Dart/Flutter). Cap'n Proto est solide en C++, Rust, Go et TypeScript, mais plus fragile ailleurs.

### RPC et networking

| Aspect | Protobuf + gRPC | FlatBuffers | Cap'n Proto |
|--------|-----------------|-------------|-------------|
| Système RPC | gRPC (projet séparé) | Aucun natif (gRPC compatible) | Intégré |
| Transport | HTTP/2 | Au choix | TCP/Unix socket brut |
| Streaming | Oui (server/client/bidirectionnel) | N/A | Oui (bidirectionnel) |
| Promise pipelining | Non | N/A | Oui |
| Écosystème middleware | Très riche (interceptors, load balancing, service mesh) | Limité | Minimal |

gRPC offre un écosystème incomparablement plus riche : intégration avec les service meshes (Istio, Envoy), load balancing, health checking, retry policies, observabilité native. Le RPC de Cap'n Proto est plus performant en termes de latence brute et de débit, mais il ne fournit pas cet écosystème de production — il faut construire ou intégrer ces fonctionnalités manuellement.

### Outillage et documentation

| Aspect | Protobuf | FlatBuffers | Cap'n Proto |
|--------|----------|-------------|-------------|
| Documentation | Excellente (Google) | Bonne | Correcte (site officiel) |
| Stack Overflow | ~50K questions | ~3K questions | ~500 questions |
| Tutoriels tiers | Abondants | Nombreux | Rares |
| IDE support | Plugins proto (VS Code, IntelliJ) | Plugins fbs | Limité |
| Outils CLI | `protoc`, `grpcurl`, `buf` | `flatc` | `capnp` (compile, decode, eval) |

## Matrice de décision par cas d'usage

| Cas d'usage | Recommandation | Justification |
|-------------|---------------|---------------|
| API publique (clients hétérogènes) | **Protobuf + gRPC** | Écosystème universel, documentation riche, support de tous les langages |
| Microservices internes (même stack) | **Protobuf + gRPC** ou **Cap'n Proto** | Protobuf si écosystème gRPC nécessaire ; Cap'n Proto si latence critique |
| IPC haute performance (même machine) | **Cap'n Proto** | Zero-copy, RPC sans HTTP, latence minimale |
| Gaming (client mobile/desktop) | **FlatBuffers** | Conçu pour ce cas, support mobile natif, mémoire contrôlée |
| Stockage structuré (base de données) | **Cap'n Proto** ou **FlatBuffers** | mmap-friendly, accès aléatoire sans parsing |
| Proxy / middleware | **Cap'n Proto** | Forwarding O(1), inspection sans désérialisation |
| Streaming de données (WAN) | **Protobuf** | Taille wire compacte, bande passante optimisée |
| Pipeline de données (batch) | **Protobuf** ou **Cap'n Proto** | Protobuf si interop ; Cap'n Proto si throughput CPU-bound |
| Systèmes embarqués (mémoire limitée) | **FlatBuffers** ou **Cap'n Proto** | Zéro allocation à la lecture |
| Interop Rust ↔ C++ | **Cap'n Proto** | Implémentation Rust mature, même format wire natif |

## Peut-on combiner les formats ?

Dans un système réel, rien n'interdit d'utiliser plusieurs formats à différents niveaux :

- **Cap'n Proto pour l'IPC interne** entre les services sur la même machine (latence minimale).  
- **Protobuf + gRPC pour les API externes** exposées à des clients tiers (écosystème universel).  
- **FlatBuffers pour le stockage** sur disque ou en cache (mmap-friendly, pas de dépendance runtime).

La conversion entre formats au boundary est un coût fixe acceptable si les chemins critiques (hot path) utilisent le format optimal. Un service pourrait recevoir des requêtes gRPC/Protobuf à son API, convertir en Cap'n Proto pour le traitement interne haute performance, puis stocker en FlatBuffers sur disque.

---

## Résumé en une phrase

- **Protobuf** : le choix sûr et universel — quand vous avez besoin que tout le monde puisse parler à votre service.  
- **FlatBuffers** : le choix gaming et mobile — quand la mémoire est limitée et que la lecture domine.  
- **Cap'n Proto** : le choix performance brute — quand chaque microseconde de sérialisation compte et que l'écosystème C++/Rust/Go suffit.
