# 17.4.2 — Impact sur les performances

## Mesurer concrètement l'effet de `noexcept` sur le code généré et l'exécution

---

## Introduction

La section 17.4 a posé le cadre conceptuel : `noexcept` permet au compilateur d'éliminer le code lié au stack unwinding et, surtout, conditionne la décision déplacement-vs-copie dans les conteneurs STL. La section 17.4.1 a détaillé la syntaxe. Il est maintenant temps de **mesurer** : quel est l'impact réel de `noexcept` sur la taille des binaires, sur le code assembleur généré, et sur les temps d'exécution ?

Cette section adopte une approche empirique. Chaque affirmation est accompagnée d'un protocole reproductible sur Ubuntu avec GCC 15 ou Clang 20. L'objectif n'est pas de fournir des chiffres absolus — ceux-ci dépendent du matériel et du compilateur — mais de vous donner les outils pour mesurer vous-même et comprendre *où* et *pourquoi* `noexcept` fait la différence.

---

## Impact sur le code généré : unwind tables et taille des binaires

### Les tables d'exception dans un binaire ELF

Le modèle zero-cost exceptions utilisé par GCC et Clang repose sur des **tables de déroulement** (*unwind tables*) stockées dans des sections spécifiques du binaire ELF. Ces tables décrivent, pour chaque point du programme, comment remonter la pile d'appels et quels destructeurs invoquer. Elles sont consultées uniquement lorsqu'une exception est effectivement levée, mais elles occupent de l'espace dans le binaire en permanence.

Les sections concernées sont principalement :

- `.eh_frame` — table de déroulement principale, au format DWARF. Chaque fonction qui peut participer au stack unwinding y possède une entrée (*Frame Description Entry*, FDE).
- `.eh_frame_hdr` — index permettant une recherche rapide dans `.eh_frame`.
- `.gcc_except_table` — table des zones de landing pad (blocs `catch`) spécifique à chaque fonction contenant un `try`.

### Inspection avec `readelf`

Vous pouvez visualiser ces sections sur n'importe quel binaire :

```bash
# Taille des sections liées aux exceptions
readelf -S monprogramme | grep -E '\.eh_frame|\.gcc_except_table'
```

La sortie ressemble à ceci :

```
[17] .eh_frame_hdr     PROGBITS    0000000000403000  003000  0001a4  00   A  0  0  4
[18] .eh_frame         PROGBITS    00000000004031a8  0031a8  000788  00   A  0  0  8
[19] .gcc_except_table PROGBITS    0000000000403930  003930  000120  00   A  0  0  4
```

Pour comparer l'impact de `noexcept`, compilez le même programme en deux versions :

```bash
# Version sans noexcept sur les fonctions clés
g++ -std=c++23 -O2 -o sans_noexcept programme.cpp

# Version avec noexcept
g++ -std=c++23 -O2 -o avec_noexcept programme.cpp

# Comparaison des tailles de section
echo "=== Sans noexcept ==="  
readelf -S sans_noexcept | grep '\.eh_frame'  
size sans_noexcept  

echo "=== Avec noexcept ==="  
readelf -S avec_noexcept | grep '\.eh_frame'  
size avec_noexcept  
```

La commande `size` affiche la taille totale des segments text, data et bss — un indicateur simple de l'empreinte du binaire.

### Ordre de grandeur typique

Sur un programme de taille modeste (quelques milliers de lignes), la différence de taille du binaire liée à `noexcept` est généralement **marginale** — de l'ordre de quelques centaines d'octets à quelques kilo-octets. En effet, même une fonction `noexcept` conserve une entrée dans `.eh_frame` car le déroulement de pile peut être nécessaire pour d'autres raisons (débogueurs, backtraces).

L'impact sur la taille devient significatif dans deux cas :

- **Les programmes compilés avec `-fno-exceptions`** (désactivation totale des exceptions) — les sections `.eh_frame` et `.gcc_except_table` sont alors drastiquement réduites, voire absentes. Cette option va bien au-delà de `noexcept` : elle interdit l'utilisation de `throw` et `catch` dans tout le programme.
- **Les projets à très grande échelle** (moteurs de jeu, navigateurs) où des millions de FDE s'accumulent et où chaque kilo-octet compte pour les caches d'instructions.

Pour la majorité des projets, le bénéfice de `noexcept` sur la taille du binaire est un bonus mineur, pas l'argument principal.

## Impact sur le code assembleur : optimisations du compilateur

L'effet plus intéressant de `noexcept` se situe au niveau des **optimisations** que le compilateur peut appliquer lorsqu'il sait qu'une fonction ne lèvera pas.

### Visualisation avec Compiler Explorer

Le moyen le plus direct d'observer les différences est d'utiliser Compiler Explorer (godbolt.org). Considérons une fonction simple :

```cpp
#include <vector>
#include <string>

struct Widget {
    std::string nom;
    std::vector<int> donnees;

    // Variante A : move constructor sans noexcept
    // Widget(Widget&& other) : nom(std::move(other.nom)),
    //     donnees(std::move(other.donnees)) {}

    // Variante B : move constructor noexcept
    // Widget(Widget&& other) noexcept : nom(std::move(other.nom)),
    //     donnees(std::move(other.donnees)) {}
};

void ajouter(std::vector<Widget>& v, Widget w) {
    v.push_back(std::move(w));
}
```

En comparant l'assembleur généré pour `ajouter()` dans les deux variantes avec `-O2`, vous observerez des différences notables.

Dans la **variante sans `noexcept`**, le code de `push_back` contient un chemin conditionnel : si la réallocation est nécessaire, le compilateur génère un appel à une routine de copie (ou à `__cxa_allocate_exception` et au code de nettoyage associé). Le code inclut des blocs de landing pad pour gérer le cas où la copie du n-ième élément échoue.

Dans la **variante avec `noexcept`**, le chemin de réallocation utilise directement le déplacement. Le code de nettoyage pour la copie disparaît, les landing pads sont éliminés, et la fonction résultante est plus compacte.

### Élimination du code mort

Quand le compilateur peut prouver qu'aucune exception ne traversera un bloc de code, il élimine les chemins d'erreur associés. Cela va au-delà des tables d'exception :

```cpp
void traiter(std::vector<int>& v) noexcept {
    for (auto& x : v) {
        x *= 2;
    }
}
```

Ici, le compilateur sait que ni l'itération sur un `vector<int>`, ni la multiplication d'entiers ne lèveront d'exception. Combiné avec `noexcept` sur la fonction englobante, il peut vectoriser la boucle de manière plus agressive (SIMD), car il n'a pas à préserver un état cohérent en cas d'interruption par une exception.

L'impact est souvent subtil — quelques instructions en moins, un registre libéré — mais dans des boucles critiques exécutées des millions de fois, ces micro-optimisations s'accumulent.

## L'impact majeur : `std::move_if_noexcept` dans les conteneurs

Comme expliqué en section 17.4, c'est la décision move-vs-copy dans les conteneurs STL qui produit les gains de performance les plus spectaculaires. Voyons cela en chiffres concrets.

### Protocole de mesure

Considérons un type `Ressource` qui simule un objet coûteux à copier (par exemple un buffer volumineux) mais peu coûteux à déplacer :

```cpp
#include <cstddef>
#include <cstring>
#include <utility>

class Ressource {  
public:  
    explicit Ressource(std::size_t taille = 4096)
        : taille_(taille)
        , donnees_(new char[taille])
    {
        std::memset(donnees_, 0, taille_);
    }

    ~Ressource() { delete[] donnees_; }

    // Copie : coûteuse (allocation + memcpy)
    Ressource(const Ressource& other)
        : taille_(other.taille_)
        , donnees_(new char[other.taille_])
    {
        std::memcpy(donnees_, other.donnees_, taille_);
    }

    Ressource& operator=(const Ressource& other) {
        if (this != &other) {
            auto copie = other;
            swap(*this, copie);
        }
        return *this;
    }

    // Déplacement : quasi-gratuit (échange de pointeurs)
    // ⚠️ Le noexcept ici fait toute la différence
    Ressource(Ressource&& other) noexcept
        : taille_(other.taille_)
        , donnees_(other.donnees_)
    {
        other.taille_ = 0;
        other.donnees_ = nullptr;
    }

    Ressource& operator=(Ressource&& other) noexcept {
        if (this != &other) {
            delete[] donnees_;
            taille_ = other.taille_;
            donnees_ = other.donnees_;
            other.taille_ = 0;
            other.donnees_ = nullptr;
        }
        return *this;
    }

    friend void swap(Ressource& a, Ressource& b) noexcept {
        using std::swap;
        swap(a.taille_, b.taille_);
        swap(a.donnees_, b.donnees_);
    }

private:
    std::size_t taille_;
    char*       donnees_;
};
```

### Benchmark avec Google Benchmark

```cpp
#include <benchmark/benchmark.h>
#include <vector>

static void BM_vector_push_back(benchmark::State& state) {
    for (auto _ : state) {
        std::vector<Ressource> v;
        for (int i = 0; i < state.range(0); ++i) {
            v.push_back(Ressource(4096));
        }
        benchmark::DoNotOptimize(v.data());
    }
}

BENCHMARK(BM_vector_push_back)->Arg(1000)->Arg(10000)->Arg(50000);  
BENCHMARK_MAIN();  
```

Pour comparer les deux scénarios, il suffit de retirer le `noexcept` du move constructor de `Ressource`, recompiler, et relancer le benchmark. Les résultats typiques sur un processeur récent (x86_64, Ubuntu 24.04, GCC 15, `-O2`) :

| Nombre d'éléments | Move `noexcept` | Move sans `noexcept` | Ratio |
|---|---|---|---|
| 1 000 | ~0.4 ms | ~2.1 ms | ×5.2 |
| 10 000 | ~4.0 ms | ~21 ms | ×5.3 |
| 50 000 | ~20 ms | ~108 ms | ×5.4 |

Le facteur de ralentissement dépend du coût relatif de la copie par rapport au déplacement. Pour un type dont la copie est peu coûteuse (un `std::pair<int, int>` par exemple), la différence sera négligeable. Mais pour tout type qui gère une ressource dynamique — `std::string`, `std::vector`, buffers, handles — le ratio est couramment entre 3× et 10×, parfois bien davantage.

### Vérifier quelle stratégie le `vector` utilise

En phase de diagnostic, vous pouvez vérifier à la compilation si votre type bénéficiera du déplacement :

```cpp
#include <type_traits>

static_assert(std::is_nothrow_move_constructible_v<Ressource>,
              "Le vector copiera au lieu de déplacer !");

static_assert(std::is_nothrow_move_assignable_v<Ressource>,
              "L'affectation par déplacement n'est pas noexcept !");
```

Ces `static_assert` échouent à la compilation si vos opérations de déplacement ne sont pas `noexcept`, ce qui vous alerte immédiatement plutôt que de laisser une régression de performance silencieuse s'installer.

## Observer `std::move_if_noexcept` en action

Pour comprendre le mécanisme de sélection de la STL, on peut observer directement le comportement de `std::move_if_noexcept` :

```cpp
#include <utility>
#include <type_traits>
#include <print>

struct Deplacable {
    Deplacable() = default;
    Deplacable(const Deplacable&) { std::println("  → copie"); }
    Deplacable(Deplacable&&) noexcept { std::println("  → déplacement"); }
};

struct NonSur {
    NonSur() = default;
    NonSur(const NonSur&) { std::println("  → copie"); }
    NonSur(NonSur&&) { std::println("  → déplacement"); } // pas noexcept !
};

int main() {
    Deplacable d;
    NonSur n;

    std::println("Deplacable avec move_if_noexcept :");
    auto d2 = std::move_if_noexcept(d);   // → déplacement

    std::println("NonSur avec move_if_noexcept :");
    auto n2 = std::move_if_noexcept(n);   // → copie (fallback sûr)
}
```

Sortie :

```
Deplacable avec move_if_noexcept :
  → déplacement
NonSur avec move_if_noexcept :
  → copie
```

Le comportement est limpide : `std::move_if_noexcept` retourne une rvalue reference (`T&&`) si le move constructor est `noexcept`, et une lvalue reference constante (`const T&`) dans le cas contraire, forçant la copie. C'est exactement ce que `std::vector` utilise en interne lors de la réallocation.

### Le cas particulier : move-only sans `noexcept`

Un cas intéressant se présente avec les types qui sont déplaçables mais **non copiables** (comme `std::unique_ptr`). Si le move constructor n'est pas `noexcept`, `std::move_if_noexcept` ne peut pas se rabattre sur la copie — elle n'existe pas. Dans ce cas, il utilise quand même le déplacement, car c'est la seule option disponible. Le `vector` accepte le risque plutôt que de refuser de compiler.

```cpp
struct MoveOnly {
    MoveOnly() = default;
    MoveOnly(const MoveOnly&) = delete;
    MoveOnly(MoveOnly&&) { /* pas noexcept, mais pas de copie possible */ }
};

// Compile, mais le vector ne peut pas offrir la garantie forte
// lors des réallocations
std::vector<MoveOnly> v;  
v.push_back(MoveOnly{});  // utilise le move malgré l'absence de noexcept  
```

C'est une raison supplémentaire de toujours marquer vos move constructors `noexcept` : même quand le code compile sans, vous perdez des garanties de sécurité exceptionnelle.

## Impact sur `std::variant`

`std::variant` (C++17) est un autre type standard fortement affecté par `noexcept`. Lors de l'affectation d'une nouvelle valeur à un variant, si le type cible a un move constructor `noexcept`, le variant peut effectuer l'opération en place. Dans le cas contraire, il peut se retrouver dans l'état spécial `valueless_by_exception` — un état « vide » indiquant qu'une exception a interrompu l'opération et que le variant ne contient plus de valeur valide.

```cpp
#include <variant>
#include <string>

struct Fragile {
    Fragile() = default;
    Fragile(const Fragile&) { throw std::runtime_error("copie échouée"); }
    Fragile(Fragile&&) {} // pas noexcept
};

std::variant<int, Fragile> v = 42;

try {
    v = Fragile{}; // peut provoquer valueless_by_exception
} catch (...) {
    if (v.valueless_by_exception()) {
        // Le variant est dans un état "zombie" — il ne contient rien
        // C'est le seul cas où un variant peut être valueless
    }
}
```

Un move constructor `noexcept` élimine ce risque : le variant peut toujours effectuer l'affectation de manière atomique.

## Coût des exceptions lorsqu'elles sont levées

Pour compléter le tableau, il est utile de mesurer le coût d'une exception **lorsqu'elle est effectivement levée**. Ce coût n'est pas lié à `noexcept` (puisqu'une fonction `noexcept` ne lève jamais), mais il permet de comprendre pourquoi les exceptions sont réservées aux situations rares.

Le processus de levée d'une exception implique :

- L'allocation dynamique de l'objet exception (via `__cxa_allocate_exception`).
- La recherche du handler approprié par parcours des tables `.eh_frame` et `.gcc_except_table` — une opération dont le coût est proportionnel à la profondeur de la pile d'appels.
- Le déroulement effectif de la pile avec appel des destructeurs.
- Le transfert de contrôle vers le bloc `catch`.

Sur un processeur moderne, lever et capturer une exception coûte typiquement entre **1 et 10 microsecondes**, selon la profondeur de la pile. C'est négligeable si l'exception signale une erreur rare (un fichier manquant lors du démarrage), mais prohibitif dans une boucle exécutée des millions de fois.

Ce coût asymétrique — quasi-nul sur le chemin nominal, élevé sur le chemin d'erreur — est la signature du modèle zero-cost et la raison fondamentale pour laquelle les exceptions conviennent aux erreurs rares tandis que `std::expected` (section 17.5) convient aux erreurs fréquentes.

## Synthèse des impacts

| Aspect | Impact de `noexcept` | Importance pratique |
|---|---|---|
| Taille du binaire | Réduction modeste des unwind tables | Faible (sauf projets très larges) |
| Code assembleur | Élimination de chemins d'erreur, meilleure vectorisation | Modéré |
| `std::vector` réallocation | Move au lieu de copy → gains de 3× à 10×+ | **Critique** |
| `std::variant` affectation | Évite `valueless_by_exception` | Important |
| Algorithmes STL (`rotate`, `swap`, etc.) | Sélection de l'implémentation optimale | Important |
| Inlining et optimisations globales | Le compilateur traite la fonction comme plus simple | Modéré |

Le message central est clair : le bénéfice principal de `noexcept` n'est pas un gain micro-architectural sur le code généré d'une fonction isolée. C'est la **cascade de décisions** qu'il déclenche dans la bibliothèque standard — en particulier le choix entre déplacement et copie — qui produit les gains de performance les plus significatifs et les plus mesurables.

## Outils de diagnostic récapitulatifs

Pour intégrer la vérification de `noexcept` dans votre workflow, voici les outils à disposition :

**À la compilation :**
- `static_assert(std::is_nothrow_move_constructible_v<T>)` — vérifie que votre type sera déplacé par la STL.
- `static_assert(std::is_nothrow_move_assignable_v<T>)` — même vérification pour l'affectation.
- `noexcept(expression)` — opérateur compiletime retournant `true` si l'expression est `noexcept`.

**À l'analyse du binaire :**
- `readelf -S binaire | grep eh_frame` — taille des tables d'exception.
- `size binaire` — taille globale des segments.
- `objdump -d binaire | grep cxa` — présence d'appels au runtime d'exceptions (`__cxa_throw`, `__cxa_allocate_exception`).

**Au profiling :**
- Google Benchmark pour mesurer l'impact sur les conteneurs.
- `perf stat` pour compter les instructions et les cache misses dans les scénarios move vs copy.

> 📎 *La section suivante (17.5) aborde `std::expected` (C++23) — une alternative aux exceptions qui offre un coût prévisible sur le chemin d'erreur, complémentaire de l'approche `noexcept` pour les fonctions où les erreurs sont fréquentes plutôt qu'exceptionnelles.*
