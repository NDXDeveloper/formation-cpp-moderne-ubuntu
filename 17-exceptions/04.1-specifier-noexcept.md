# 17.4.1 — Spécifier `noexcept`

## Syntaxe, formes conditionnelles et propagation dans le code générique

---

## La forme simple : `noexcept`

Dans sa forme la plus élémentaire, `noexcept` se place après la liste de paramètres d'une fonction (et après les qualificateurs `const`, `volatile`, `&`, `&&` s'il y en a). Il déclare que la fonction ne lèvera jamais d'exception, quels que soient les arguments et l'état du programme.

```cpp
int calculer_hash(int valeur) noexcept {
    return valeur ^ (valeur >> 16);
}

class Ressource {  
public:  
    // noexcept après const, avant le corps ou = default / = delete
    std::size_t taille() const noexcept { return taille_; }

    // Compatible avec les qualificateurs de référence
    std::string&& extraire() && noexcept { return std::move(nom_); }

    // Compatible avec virtual et override
    const char* what() const noexcept override { return buffer_; }

private:
    std::size_t taille_ = 0;
    std::string nom_;
    char buffer_[64]{};
};
```

Le placement de `noexcept` dans la déclaration suit un ordre strict. Pour une méthode membre complète, la séquence est :

```
type_retour nom(params) const volatile & noexcept override final;
```

En pratique, on rencontre rarement tous ces qualificateurs simultanément, mais l'ordre relatif est à connaître : `noexcept` se place **après** les cv-qualificateurs et les ref-qualificateurs, et **avant** `override` et `final`.

---

## `noexcept` et les fonctions spéciales

Les fonctions spéciales (constructeurs, destructeurs, opérateurs d'affectation) méritent une attention particulière, car le compilateur leur attribue un statut `noexcept` implicite dans certains cas.

### Destructeurs : implicitement `noexcept`

Depuis C++11, les destructeurs sont **implicitement `noexcept`** sauf si vous déclarez explicitement le contraire (ce que vous ne devriez jamais faire). Vous n'avez pas besoin d'écrire `noexcept` sur vos destructeurs — c'est déjà le comportement par défaut.

```cpp
class Widget {  
public:  
    ~Widget() { /* implicitement noexcept */ }
};

// Vérification à la compilation
static_assert(std::is_nothrow_destructible_v<Widget>);
```

### Fonctions `= default` : `noexcept` conditionnel automatique

Lorsque vous déclarez une fonction spéciale avec `= default`, le compilateur détermine automatiquement si elle est `noexcept` en se basant sur les fonctions spéciales des membres et des classes de base. Si tous les sous-objets offrent l'opération correspondante en `noexcept`, la fonction defaultée sera elle aussi `noexcept`.

```cpp
class Point {  
public:  
    double x, y;
    // Le move constructor de double est noexcept
    // → le move constructor defaulté de Point sera noexcept
    Point(Point&&) = default;
    Point& operator=(Point&&) = default;
};

static_assert(std::is_nothrow_move_constructible_v<Point>);     // ✅  
static_assert(std::is_nothrow_move_assignable_v<Point>);        // ✅  

class Document {  
public:  
    std::string titre;  // string a un move constructor noexcept
    std::vector<int> pages; // vector a un move constructor noexcept
    
    Document(Document&&) = default; // → noexcept, car tous les membres le sont
};

static_assert(std::is_nothrow_move_constructible_v<Document>);  // ✅
```

C'est pourquoi `= default` est souvent préférable à une implémentation manuelle : vous obtenez le bon statut `noexcept` gratuitement, sans risque d'erreur.

### Fonctions définies manuellement : `noexcept` explicite requis

Lorsque vous implémentez vous-même un move constructor ou un move assignment, le compilateur ne peut pas déduire automatiquement que l'opération est `noexcept`. Vous devez le spécifier explicitement, et vous portez la responsabilité de cette garantie.

```cpp
class Buffer {  
public:  
    Buffer(Buffer&& other) noexcept
        : data_(other.data_)
        , size_(other.size_)
    {
        other.data_ = nullptr;
        other.size_ = 0;
    }

    Buffer& operator=(Buffer&& other) noexcept {
        if (this != &other) {
            delete[] data_;
            data_ = other.data_;
            size_ = other.size_;
            other.data_ = nullptr;
            other.size_ = 0;
        }
        return *this;
    }

    ~Buffer() { delete[] data_; }

private:
    int*        data_ = nullptr;
    std::size_t size_ = 0;
};

static_assert(std::is_nothrow_move_constructible_v<Buffer>);  // ✅  
static_assert(std::is_nothrow_move_assignable_v<Buffer>);     // ✅  
```

Ici, les opérations internes (affectation de pointeurs, `delete[]`) ne lèvent effectivement pas d'exception, donc le `noexcept` est légitime. Si le corps de votre move constructor appelait une fonction susceptible de lever, le `noexcept` serait un mensonge dangereux — une exception déclencherait `std::terminate()`.

## La forme conditionnelle : `noexcept(expression)`

La forme simple `noexcept` est un raccourci pour `noexcept(true)`. La forme longue accepte une **expression booléenne constante** évaluée à la compilation :

```cpp
void f() noexcept(true);   // équivalent à : void f() noexcept;  
void g() noexcept(false);  // équivalent à : pas de noexcept (peut lever)  
```

L'intérêt de cette forme apparaît lorsque l'expression dépend d'un paramètre de template ou d'une propriété vérifiable à la compilation. La fonction devient `noexcept` **conditionnellement**, selon les types impliqués.

```cpp
template <typename T>  
void echanger(T& a, T& b) noexcept(std::is_nothrow_move_constructible_v<T> &&  
                                     std::is_nothrow_move_assignable_v<T>)
{
    T tmp = std::move(a);
    a = std::move(b);
    b = std::move(tmp);
}
```

Cette déclaration exprime précisément : « `echanger` est `noexcept` si et seulement si le type `T` peut être déplacé (construction et affectation) sans lever d'exception ». Pour un `T = int`, la fonction sera `noexcept`. Pour un type exotique dont le move constructor peut lever, elle ne le sera pas.

```cpp
static_assert(noexcept(echanger(std::declval<int&>(), std::declval<int&>())));

struct Bizarre {
    Bizarre(Bizarre&&) { /* peut lever */ }
    Bizarre& operator=(Bizarre&&) { return *this; }
};
static_assert(!noexcept(echanger(std::declval<Bizarre&>(), std::declval<Bizarre&>())));
```

---

## L'opérateur `noexcept(...)` : tester si une expression peut lever

`noexcept` a une seconde identité : utilisé comme **opérateur** dans une expression, il retourne un `bool` constexpr indiquant si l'expression passée en argument est `noexcept`. Cet opérateur n'évalue jamais l'expression — il se contente d'inspecter sa spécification d'exception à la compilation, exactement comme `sizeof` n'évalue pas son opérande.

```cpp
#include <vector>
#include <string>

static_assert(noexcept(42 + 1));                         // arithmétique : noexcept  
static_assert(noexcept(std::declval<int&>() = 5));       // affectation int : noexcept  

// std::string::size() est noexcept
static_assert(noexcept(std::declval<std::string>().size()));

// std::vector::push_back peut lever (réallocation → bad_alloc)
static_assert(!noexcept(std::declval<std::vector<int>>().push_back(1)));

// operator new peut lever std::bad_alloc
static_assert(!noexcept(new int));
```

L'opérateur `noexcept(...)` est l'outil de base pour construire des spécifications conditionnelles. Il est fréquemment combiné avec `std::declval` pour inspecter les opérations sur des types sans avoir besoin d'instances réelles.

---

## Le pattern `noexcept(noexcept(...))` : propagation dans le code générique

La combinaison la plus courante en code générique consiste à imbriquer l'opérateur `noexcept` (qui teste) dans le spécificateur `noexcept` (qui déclare). Le `noexcept` extérieur est le spécificateur de la fonction, le `noexcept` intérieur est l'opérateur qui évalue l'expression.

```cpp
template <typename T>  
class Wrapper {  
public:  
    // Le move constructor de Wrapper est noexcept si et seulement si
    // le move constructor de T est noexcept
    Wrapper(Wrapper&& other) noexcept(noexcept(T(std::move(other.valeur_))))
        : valeur_(std::move(other.valeur_))
    {}

    // swap est noexcept si le swap du type sous-jacent est noexcept
    void swap(Wrapper& other) noexcept(noexcept(std::swap(valeur_, other.valeur_))) {
        using std::swap;
        swap(valeur_, other.valeur_);
    }

private:
    T valeur_;
};
```

Ce pattern peut paraître verbeux à la première lecture, mais il exprime une idée simple et puissante : **la spécification d'exception de l'enveloppe suit automatiquement celle du type enveloppé**. Si `T` est `std::string` (dont le move est `noexcept`), `Wrapper<std::string>` sera `noexcept`-movable. Si `T` est un type dont le move peut lever, `Wrapper<T>` ne sera pas `noexcept`-movable — et la STL ajustera son comportement en conséquence.

### Simplification avec les type traits

Pour améliorer la lisibilité, on peut remplacer le `noexcept(noexcept(...))` par des type traits de `<type_traits>` :

```cpp
template <typename T>  
class Wrapper {  
public:  
    Wrapper(Wrapper&& other) noexcept(std::is_nothrow_move_constructible_v<T>)
        : valeur_(std::move(other.valeur_))
    {}

    Wrapper& operator=(Wrapper&& other)
        noexcept(std::is_nothrow_move_assignable_v<T>)
    {
        valeur_ = std::move(other.valeur_);
        return *this;
    }

private:
    T valeur_;
};
```

Les deux formes sont strictement équivalentes. La forme avec type traits est généralement préférée car elle est plus explicite sur l'intention et ne requiert pas de construire une expression fictive avec `std::declval`.

### Les type traits les plus utiles

Voici les traits de `<type_traits>` les plus couramment utilisés pour les spécifications `noexcept` conditionnelles :

| Trait | Teste si... |
|---|---|
| `std::is_nothrow_default_constructible_v<T>` | `T()` est `noexcept` |
| `std::is_nothrow_copy_constructible_v<T>` | `T(const T&)` est `noexcept` |
| `std::is_nothrow_move_constructible_v<T>` | `T(T&&)` est `noexcept` |
| `std::is_nothrow_copy_assignable_v<T>` | `operator=(const T&)` est `noexcept` |
| `std::is_nothrow_move_assignable_v<T>` | `operator=(T&&)` est `noexcept` |
| `std::is_nothrow_destructible_v<T>` | `~T()` est `noexcept` |
| `std::is_nothrow_swappable_v<T>` | `swap(T&, T&)` est `noexcept` |
| `std::is_nothrow_invocable_v<F, Args...>` | `F(Args...)` est `noexcept` |

## `noexcept` dans les lambdas

Les lambdas peuvent être déclarées `noexcept` de la même manière que les fonctions. Le spécificateur se place après la liste de paramètres (et après `mutable` s'il est présent) :

```cpp
auto diviser = [](double a, double b) noexcept -> double {
    return a / b; // division flottante ne lève pas
};

auto compteur = [n = 0]() mutable noexcept {
    return ++n;
};

static_assert(noexcept(diviser(1.0, 2.0)));  // ✅  
static_assert(noexcept(compteur()));          // ✅  
```

Depuis C++23, les lambdas `static` peuvent également être `noexcept` :

```cpp
auto carre = [](int x) static noexcept -> int {
    return x * x;
};
```

Dans le code générique qui accepte des callables, vous pouvez propager le `noexcept` d'une lambda ou d'un foncteur grâce à `std::is_nothrow_invocable_v` :

```cpp
template <typename Func, typename... Args>  
auto appeler(Func&& f, Args&&... args)  
    noexcept(std::is_nothrow_invocable_v<Func, Args...>)
    -> std::invoke_result_t<Func, Args...>
{
    return std::invoke(std::forward<Func>(f), std::forward<Args>(args)...);
}
```

## `noexcept` fait partie du système de types

Depuis C++17, `noexcept` fait partie intégrante du **type de la fonction**. Une fonction `noexcept` et une fonction non-`noexcept` ont des types distincts, même si leurs signatures sont par ailleurs identiques.

```cpp
void f() noexcept;  
void g();  

// f et g ont des types différents
static_assert(!std::is_same_v<decltype(&f), decltype(&g)>);
// decltype(&f) = void(*)() noexcept
// decltype(&g) = void(*)()
```

Cette distinction a des conséquences importantes pour les pointeurs de fonctions et les templates :

```cpp
// Un pointeur vers fonction noexcept
void (*ptr_ne)(int) noexcept = nullptr;

// ✅ Conversion implicite : noexcept → peut lever (spécialisation → généralisation)
void (*ptr_normal)(int) = ptr_ne;

// ❌ Conversion inverse interdite : peut lever → noexcept
// ptr_ne = ptr_normal; // erreur de compilation
```

La logique est intuitive : une fonction qui promet de ne jamais lever est utilisable partout où une fonction « ordinaire » est attendue (elle respecte un contrat plus strict). L'inverse serait dangereux — on ne peut pas traiter une fonction susceptible de lever comme si elle était `noexcept`.

### Impact sur les callbacks et les templates

Cette distinction de types peut affecter la déduction de template :

```cpp
template <typename Func>  
void executer(Func f) {  
    f();
}

void a() noexcept {}  
void b() {}  

executer(a); // Func = void(*)() noexcept  
executer(b); // Func = void(*)()  
```

Les deux appels compilent, car le template accepte n'importe quel type callable. Mais si vous avez besoin de contraindre le callback à être `noexcept`, vous pouvez utiliser un concept (C++20) :

```cpp
template <typename Func>
    requires std::is_nothrow_invocable_v<Func>
void executer_sur(Func f) noexcept {
    f(); // garanti noexcept par la contrainte
}

executer_sur(a); // ✅ compile
// executer_sur(b); // ❌ ne compile pas : b() n'est pas noexcept
```

## Vérifier vos garanties avec `static_assert`

Une bonne habitude consiste à vérifier à la compilation que vos types respectent les garanties `noexcept` attendues. Placez ces assertions dans le fichier source, juste après la définition de la classe, ou dans vos tests unitaires :

```cpp
class Session {  
public:  
    Session() = default;
    Session(Session&&) noexcept;
    Session& operator=(Session&&) noexcept;
    ~Session() = default;

    Session(const Session&) = delete;
    Session& operator=(const Session&) = delete;

    void fermer() noexcept;
    // ...
};

// Vérifications systématiques
static_assert(std::is_nothrow_move_constructible_v<Session>,
              "Session doit être nothrow-move-constructible pour la STL");
static_assert(std::is_nothrow_move_assignable_v<Session>,
              "Session doit être nothrow-move-assignable pour la STL");
static_assert(std::is_nothrow_destructible_v<Session>,
              "Le destructeur de Session ne doit jamais lever");
```

Ces `static_assert` servent de documentation exécutable : si un développeur modifie `Session` de manière à violer la garantie `noexcept` (par exemple en ajoutant un membre dont le move peut lever), l'erreur sera détectée immédiatement à la compilation, avec un message explicite.

## Erreurs courantes et pièges

### Oublier `noexcept` sur un move constructor défini manuellement

C'est l'erreur la plus fréquente et la plus insidieuse, car elle ne provoque aucune erreur de compilation. Le code fonctionne correctement — il est simplement plus lent, parfois de manière dramatique, car la STL recourt à la copie au lieu du déplacement.

```cpp
class Donnees {  
public:  
    // ⚠️ Oubli de noexcept — compilation OK, mais vector copiera au lieu de déplacer
    Donnees(Donnees&& other)
        : buffer_(other.buffer_), taille_(other.taille_)
    {
        other.buffer_ = nullptr;
        other.taille_ = 0;
    }
    // ...
};

// Ce static_assert détecte le problème à la compilation
static_assert(std::is_nothrow_move_constructible_v<Donnees>,
              "Move constructor de Donnees doit être noexcept !");
// → ÉCHEC de compilation → le développeur corrige immédiatement
```

### Marquer `noexcept` une fonction qui alloue

```cpp
// ❌ Dangereux : std::string::operator+ peut lever std::bad_alloc
std::string construire_message(int code) noexcept {
    return "Erreur #" + std::to_string(code); // allocation → peut lever !
}
```

Si `operator+` ou `std::to_string` lèvent `std::bad_alloc`, le programme appellera `std::terminate()`. La fonction ne devrait pas être `noexcept` sauf si vous acceptez explicitement ce comportement (ce qui est parfois un choix de conception légitime dans les systèmes embarqués, mais doit être documenté).

### Confondre l'opérateur et le spécificateur

```cpp
// L'opérateur noexcept() retourne un bool
constexpr bool b = noexcept(42 + 1);     // b == true

// Le spécificateur noexcept() décore une fonction
void f() noexcept(true);                  // f ne lèvera jamais

// ⚠️ noexcept sans parenthèses dans une déclaration = spécificateur noexcept(true)
void g() noexcept;                        // identique à noexcept(true)

// ⚠️ noexcept avec parenthèses dans une expression = opérateur
auto test = noexcept(g());                // test == true (opérateur)
```

La distinction est toujours résolue par le contexte syntaxique : après une liste de paramètres de fonction, c'est un spécificateur ; partout ailleurs, c'est un opérateur.

---

## Récapitulatif

| Syntaxe | Rôle | Contexte |
|---|---|---|
| `void f() noexcept` | Spécificateur — `f` ne lèvera jamais | Déclaration de fonction |
| `void f() noexcept(expr)` | Spécificateur conditionnel — `noexcept` si `expr` est `true` | Code générique (templates) |
| `noexcept(expr)` | Opérateur — retourne `true` si `expr` est `noexcept` | Expressions, `static_assert` |
| `noexcept(noexcept(expr))` | Propagation — spécifie `noexcept` selon le statut de `expr` | Wrappers génériques |

La règle pratique : utilisez `= default` dès que possible pour que le compilateur déduise le bon statut `noexcept`. Lorsque vous implémentez manuellement, annotez explicitement `noexcept` et vérifiez avec `static_assert`. Dans le code générique, propagez avec `noexcept(noexcept(...))` ou les type traits de `<type_traits>`.
