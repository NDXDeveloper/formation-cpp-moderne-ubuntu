# 28.4.1 Syntaxe déclarative et philosophie

> **Objectif** : Comprendre la syntaxe de Meson en profondeur suffisante pour lire un `meson.build` existant, en apprécier les choix de design, et le comparer mentalement aux équivalents CMake vus dans les chapitres précédents.

---

## Le fichier `meson.build`

Chaque répertoire d'un projet Meson contient un fichier `meson.build` — l'équivalent du `CMakeLists.txt`. Le fichier racine déclare le projet et inclut les sous-répertoires :

```meson
project('my_project', 'cpp',
  version : '1.2.0',
  license : 'MIT',
  meson_version : '>=1.3.0',
  default_options : [
    'cpp_std=c++23',
    'warning_level=3',
    'buildtype=release',
    'b_ndebug=if-release',
  ]
)
```

La fonction `project()` est obligatoire et doit être le premier appel du fichier racine. Comparée à CMake :

| Meson | CMake |
|-------|-------|
| `project('my_project', 'cpp', ...)` | `project(my_project LANGUAGES CXX)` |
| `version : '1.2.0'` | `VERSION 1.2.0` |
| `meson_version : '>=1.3.0'` | `cmake_minimum_required(VERSION 3.25)` |
| `default_options : ['cpp_std=c++23']` | `set(CMAKE_CXX_STANDARD 23)` |

La différence notable est que les options de compilation (`cpp_std`, `warning_level`, `buildtype`) sont déclarées directement dans `project()`, pas via des variables globales dispersées dans le fichier.

## Système de types

Meson possède un vrai système de types, contrairement à CMake où tout est une chaîne de caractères. Les types principaux sont :

### Types primitifs

```meson
# Chaînes de caractères
name = 'my_project'  
multiline = '''  
  Ceci est une chaîne
  multi-lignes.
'''

# Nombres entiers
count = 42  
result = count + 8    # 50  

# Booléens
enable_tests = true  
is_debug = get_option('buildtype') == 'debug'  

# Tableaux (listes)
sources = ['core.cpp', 'network.cpp', 'parser.cpp']  
sources += ['extra.cpp']    # Append  

# Dictionnaires
metadata = {'author': 'Dev Team', 'year': '2026'}
```

Le typage fort signifie que Meson détecte les erreurs de type à la configuration :

```meson
# ❌ Erreur Meson — on ne peut pas additionner un string et un int
result = 'hello' + 42

# ✅ Conversion explicite nécessaire
result = 'hello' + 42.to_string()
```

En CMake, cette erreur passerait silencieusement — tout est une chaîne, `42` et `"42"` sont interchangeables.

### Objets retournés par les fonctions

Les fonctions Meson retournent des **objets typés** qui encapsulent les métadonnées :

```meson
# library() retourne un objet lib (pas une chaîne)
core_lib = library('core', 'core.cpp', 'network.cpp')

# dependency() retourne un objet dep
ssl_dep = dependency('openssl')

# include_directories() retourne un objet inc
inc = include_directories('include')

# Ces objets se passent aux fonctions qui les attendent
executable('app', 'main.cpp',
  link_with : core_lib,          # Objet lib
  dependencies : ssl_dep,         # Objet dep
  include_directories : inc,      # Objet inc
)
```

Ce système d'objets est fondamentalement différent de CMake, où les cibles sont identifiées par des **noms de chaînes** qui sont résolus globalement. En Meson, les objets ont une portée lexicale — une bibliothèque déclarée dans un sous-répertoire doit être **explicitement passée** au parent via des variables, elle n'est pas magiquement visible partout.

---

## Déclaration des cibles

### Exécutables

```meson
my_app = executable('my-app',
  'main.cpp',
  dependencies : [core_dep, cli11_dep],
  install : true,
  install_dir : get_option('bindir'),
)
```

### Bibliothèques

```meson
# Statique ou dynamique selon l'option default_library
core_lib = library('my_project_core',
  'core.cpp',
  'network.cpp',
  include_directories : include_directories('../include'),
  dependencies : [ssl_dep, threads_dep],
  install : true,
)

# Forcer un type spécifique
static_lib = static_library('my_utils', 'utils.cpp')  
shared_lib = shared_library('my_plugin', 'plugin.cpp')  

# Les deux versions simultanément
both_lib = both_libraries('my_codec', 'codec.cpp')
```

La fonction `library()` (sans préfixe) produit une bibliothèque statique ou dynamique selon l'option globale `default_library`. L'utilisateur choisit au moment de la configuration :

```bash
meson setup build -Ddefault_library=static  
meson setup build -Ddefault_library=shared  
meson setup build -Ddefault_library=both  
```

C'est l'équivalent de `BUILD_SHARED_LIBS` en CMake, mais intégré au langage plutôt qu'ajouté via une variable.

### Exposer une bibliothèque comme dépendance

```meson
# Déclarer la bibliothèque
core_lib = library('my_project_core',
  'core.cpp',
  'network.cpp',
  include_directories : public_inc,
  dependencies : [ssl_dep, threads_dep],
)

# Créer un objet dependency pour les consommateurs
core_dep = declare_dependency(
  link_with : core_lib,
  include_directories : public_inc,
  dependencies : [threads_dep],    # Dépendance transitive propagée
)
```

`declare_dependency()` est l'équivalent conceptuel du système `PUBLIC`/`PRIVATE`/`INTERFACE` de CMake, mais en plus explicite : vous déclarez exactement ce qui est propagé aux consommateurs. Les dépendances listées dans `dependencies` de `declare_dependency()` sont transitives — l'équivalent de `PUBLIC` dans CMake. Les dépendances listées uniquement dans `library()` mais pas dans `declare_dependency()` sont privées — l'équivalent de `PRIVATE`.

## Dépendances externes

### `dependency()` — le point d'entrée principal

```meson
# Recherche via pkg-config (prioritaire), puis détecteurs intégrés
ssl_dep = dependency('openssl', version : '>=3.0', required : true)  
zlib_dep = dependency('zlib', required : false)  
threads_dep = dependency('threads')  
boost_dep = dependency('boost', modules : ['filesystem', 'system'])  

# Vérifier si une dépendance optionnelle a été trouvée
if zlib_dep.found()
  add_project_arguments('-DHAS_ZLIB=1', language : 'cpp')
endif
```

La méthode `.found()` retourne un booléen — pas besoin de vérifier une variable `_FOUND` comme en CMake. L'objet dépendance lui-même peut être passé aux cibles même si la bibliothèque n'est pas trouvée (Meson l'ignore alors silencieusement).

### Wraps avec fallback

```meson
# Chercher sur le système, sinon utiliser le sous-projet wrap
spdlog_dep = dependency('spdlog',
  version : '>=1.14',
  fallback : ['spdlog', 'spdlog_dep'],
  default_options : ['tests=disabled'],
)
```

Le paramètre `fallback` spécifie : « si `spdlog` n'est pas trouvé sur le système, utiliser le sous-projet `spdlog` (défini dans `subprojects/spdlog.wrap`) et en extraire la variable `spdlog_dep` ». Le paramètre `default_options` configure le sous-projet (ici, désactiver ses tests — l'équivalent du `set(SPDLOG_BUILD_TESTS OFF CACHE BOOL "" FORCE)` de CMake/Conan).

## Sous-répertoires et portée

```meson
# meson.build racine
subdir('libs/utils')     # Exécute libs/utils/meson.build  
subdir('src')            # Exécute src/meson.build  
subdir('apps')           # Exécute apps/meson.build  
```

`subdir()` est l'équivalent de `add_subdirectory()` en CMake. La différence fondamentale est la gestion de la portée :

**En CMake**, les cibles sont globales — une cible créée dans `libs/utils/CMakeLists.txt` est visible depuis `src/CMakeLists.txt` par son nom. C'est pratique mais peut créer des collisions.

**En Meson**, les variables sont **lexicalement scopées** au fichier courant et à ses sous-répertoires. Pour qu'une bibliothèque créée dans `libs/utils/meson.build` soit accessible dans `src/meson.build`, il faut la stocker dans une variable avant l'appel `subdir()` :

```meson
# libs/utils/meson.build
utils_lib = library('my_project_utils', 'string_utils.cpp',
  include_directories : include_directories('include'),
)
utils_dep = declare_dependency(
  link_with : utils_lib,
  include_directories : include_directories('include'),
)
# utils_dep est maintenant visible dans le fichier parent (qui a fait subdir())
```

```meson
# meson.build racine
subdir('libs/utils')    # utils_dep est défini ici  
subdir('src')           # src/meson.build peut utiliser utils_dep  
```

Les variables définies dans un sous-répertoire « remontent » dans la portée du parent. C'est l'inverse de CMake, où les variables d'un enfant sont locales (sauf `PARENT_SCOPE`). En Meson, les variables des sous-répertoires sont naturellement accessibles dans le parent après l'appel `subdir()`.

## Options de projet

Meson gère les options via un fichier dédié `meson.options` (ou `meson_options.txt` dans l'ancien format) :

```meson
# meson.options
option('tests', type : 'boolean', value : true,
  description : 'Compiler les tests unitaires')
option('benchmarks', type : 'boolean', value : false,
  description : 'Compiler les benchmarks')
option('with_compression', type : 'combo',
  choices : ['auto', 'enabled', 'disabled'], value : 'auto',
  description : 'Support de la compression ZLIB')
```

Les types d'options supportés : `boolean`, `combo` (choix parmi une liste), `integer`, `string`, `array`, `feature` (auto/enabled/disabled). Le type `feature` est particulièrement élégant pour les dépendances optionnelles :

```meson
# meson.options
option('ssl', type : 'feature', value : 'auto',
  description : 'Support SSL via OpenSSL')
```

```meson
# meson.build
ssl_opt = get_option('ssl')  
ssl_dep = dependency('openssl', required : ssl_opt)  
# Si ssl=auto : chercher, ne pas échouer si absent
# Si ssl=enabled : chercher, échouer si absent
# Si ssl=disabled : ne pas chercher du tout
```

L'utilisateur configure les options à l'initialisation :

```bash
meson setup build -Dtests=false -Dssl=enabled
```

En CMake, l'équivalent est `option()` pour les booléens et des `set(... CACHE STRING ...)` pour les autres types, avec une logique conditionnelle manuelle pour le pattern auto/enabled/disabled.

## Cross-compilation

Meson gère la cross-compilation via des **cross files** — l'équivalent des fichiers toolchain CMake :

```ini
# cross/aarch64-linux.ini
[binaries]
c = 'aarch64-linux-gnu-gcc'  
cpp = 'aarch64-linux-gnu-g++'  
ar = 'aarch64-linux-gnu-ar'  
strip = 'aarch64-linux-gnu-strip'  

[host_machine]
system = 'linux'  
cpu_family = 'aarch64'  
cpu = 'cortex-a72'  
endian = 'little'  
```

```bash
meson setup build-arm64 --cross-file cross/aarch64-linux.ini  
meson compile -C build-arm64  
```

La distinction host/build est gérée nativement par Meson — les outils de build (comme `protoc`) sont automatiquement compilés pour la machine de build, tandis que les bibliothèques sont compilées pour la cible. C'est l'équivalent du double profil host/build de Conan (section 27.2.3) et des fichiers toolchain CMake (section 26.6).

## Fonctionnalités de langage notables

### Conditions

```meson
if host_machine.system() == 'linux'
  add_project_arguments('-DPLATFORM_LINUX', language : 'cpp')
elif host_machine.system() == 'windows'
  add_project_arguments('-DPLATFORM_WINDOWS', language : 'cpp')
endif

# Ternaire (via select)
debug_flags = get_option('buildtype') == 'debug' ? ['-DDEBUG'] : ['-DNDEBUG']
```

### Itération

```meson
# foreach sur une liste
sources = ['core.cpp', 'network.cpp', 'parser.cpp']  
foreach src : sources  
  message('Source: ' + src)
endforeach

# foreach sur un dictionnaire
deps = {'ssl': ssl_dep, 'zlib': zlib_dep}  
foreach name, dep : deps  
  if dep.found()
    message(name + ' found')
  endif
endforeach
```

Il n'y a pas de `while` — l'itération est limitée aux `foreach` sur des collections finies. C'est la contrainte non Turing-complète en action.

### Pas de variables globales mutables

En CMake, une variable définie dans le `CMakeLists.txt` racine est visible dans tous les sous-répertoires et peut influencer leur comportement. En Meson, il n'y a **pas de variable globale mutable** que les sous-projets pourraient modifier. Les sous-projets (wraps) sont complètement isolés — ils ne voient pas les variables du projet parent et ne peuvent pas les modifier.

Cette isolation est l'une des garanties de correction de Meson : un sous-projet ne peut pas casser votre build en modifiant un flag global. En CMake, ce type de contamination est un piège courant avec `FetchContent` et `add_subdirectory` (section 26.2.4, piège de la propagation de `-Werror`).

## Commandes de configuration utiles

```bash
# Lister toutes les options et leurs valeurs
meson configure build

# Introspecter les cibles du projet
meson introspect build --targets

# Introspecter les dépendances
meson introspect build --dependencies

# Générer les commandes de compilation (comme compile_commands.json)
# → Généré automatiquement dans build/ (via Ninja)
```

La commande `meson introspect` produit du JSON structuré, exploitable par les IDE et les outils d'analyse. C'est l'équivalent de la CMake File API, mais avec une interface en ligne de commande plus directe.

## Récapitulatif : correspondance Meson ↔ CMake

| Concept | Meson | CMake |
|---------|-------|-------|
| Déclaration du projet | `project()` | `project()` + `cmake_minimum_required()` |
| Exécutable | `executable()` | `add_executable()` |
| Bibliothèque | `library()` / `static_library()` / `shared_library()` | `add_library()` |
| Dépendance externe | `dependency()` | `find_package()` |
| Téléchargement de dépendance | Wraps (`subprojects/*.wrap`) | `FetchContent` |
| Sous-répertoire | `subdir()` | `add_subdirectory()` |
| Include directories | `include_directories()` | `target_include_directories()` |
| Propagation aux consommateurs | `declare_dependency()` | `PUBLIC` / `INTERFACE` |
| Options | `meson.options` + `get_option()` | `option()` + cache variables |
| Cross-compilation | Cross files (`.ini`) | Toolchain files (`.cmake`) |
| Standard C++ | `default_options : ['cpp_std=c++23']` | `CMAKE_CXX_STANDARD` |
| Type de build | `buildtype=release` (option intégrée) | `CMAKE_BUILD_TYPE=Release` |
| Niveau de warnings | `warning_level=3` (option intégrée) | `-Wall -Wextra` (flags manuels) |
