# 28.3.1 Pourquoi Ninja est plus rapide

> **Objectif** : Analyser en détail les raisons techniques qui font de Ninja un build system plus rapide que Make — du parsing du fichier de build à l'exécution des tâches, en passant par la détection de changements et la parallélisation.

---

## Les trois phases d'un build

Pour comprendre où Ninja gagne du temps, décomposons l'exécution d'un build en trois phases distinctes :

```
Phase 1 : Chargement       Phase 2 : Planification      Phase 3 : Exécution
─────────────────────       ───────────────────────       ──────────────────
Lire le fichier de build    Vérifier quoi recompiler      Lancer les compilations  
Construire le graphe        Ordonner les tâches           Gérer le parallélisme  
en mémoire                  Détecter les changements      Afficher la progression  
```

Make et Ninja traversent ces trois phases à chaque invocation. Ninja est plus rapide dans **chacune d'elles**, mais les gains sont inégaux : la phase 1 est celle où l'écart est le plus spectaculaire.

---

## Phase 1 : chargement — l'écart majeur

### Make : un interpréteur de langage

Quand Make charge un Makefile, il ne se contente pas de le lire — il l'**interprète**. Le langage Make est Turing-complet : il supporte les variables récursives (`=`), les fonctions (`$(patsubst ...)`), les conditions (`ifeq`/`endif`), les inclusions (`include`), les règles implicites, les variables automatiques, et les règles de suffixe. Chaque ligne peut déclencher une expansion de variable qui elle-même déclenche une évaluation de fonction qui elle-même peut inclure un autre fichier.

Pour un Makefile généré par CMake, cette complexité est maximale. CMake produit des dizaines de Makefiles interconnectés (un par sous-répertoire, plus des fichiers de dépendances, de flags, de règles), chacun contenant des variables avec des expansions imbriquées. Make doit tous les charger, les parser, résoudre toutes les expansions, construire le graphe de dépendances, et fusionner les règles implicites avec les règles explicites.

Le résultat : sur un projet de 5000 fichiers sources, le simple chargement des Makefiles peut prendre **plusieurs secondes** — avant même que Make ne commence à vérifier si quelque chose doit être recompilé.

### Ninja : un parser minimaliste

Le format `build.ninja` est conçu pour être parsé par une **machine à états linéaire**. Pas de conditions, pas de fonctions, pas d'imbrication, pas d'inclusions récursives. Chaque ligne est soit une déclaration de variable, soit une définition de règle, soit un build statement. Le parsing est un passage unique du début à la fin du fichier, sans retour en arrière.

Les variables dans Ninja n'ont qu'un seul niveau d'expansion (pas de récursion). Les règles sont des templates fixes avec des placeholders. Il n'y a pas de logique à évaluer — CMake a déjà résolu toute la logique et produit un fichier `build.ninja` entièrement « aplati ».

Le résultat : sur le même projet de 5000 fichiers, Ninja charge son fichier de build en **quelques dizaines de millisecondes**. L'écart avec Make est d'un à deux ordres de grandeur.

### Fichier unique vs hiérarchie de fichiers

Make avec CMake charge une **hiérarchie** de fichiers :

```
build/Makefile  
build/CMakeFiles/Makefile2  
build/src/CMakeFiles/my_lib.dir/build.make  
build/src/CMakeFiles/my_lib.dir/depend.make  
build/src/CMakeFiles/my_lib.dir/flags.make  
build/apps/CMakeFiles/my_app.dir/build.make  
... (des dizaines de fichiers)
```

Chaque `include` est un appel système (`open`, `read`, `close`). Sur un système de fichiers réseau (NFS, CIFS — courant en entreprise), la latence de chaque opération d'I/O s'accumule.

Ninja charge un **fichier unique** (`build.ninja`, éventuellement avec un `rules.ninja` inclus). Deux ouvertures de fichier au lieu de dizaines. Sur un disque local, la différence est modeste. Sur un système de fichiers réseau, elle devient significative.

## Phase 2 : planification — détection de changements

### Make : timestamps uniquement

Make compare les timestamps des fichiers pour déterminer quoi recompiler. Son algorithme parcourt le graphe de dépendances de la cible demandée vers les feuilles, et marque comme « à reconstruire » toute cible dont un prérequis a un timestamp postérieur.

Cette approche a deux faiblesses :

**Insensibilité aux changements de commande.** Si vous modifiez un flag de compilation (par exemple `-O2` → `-O3`), les timestamps des fichiers sources ne changent pas. Make considère que tout est à jour et ne recompile rien. Vous devez faire un `make clean && make` pour forcer une recompilation complète.

**Faux positifs avec `touch`.** Un simple `touch main.cpp` déclenche une recompilation, même si le contenu du fichier n'a pas changé. Sur les systèmes avec une granularité de timestamp grossière (FAT32 : 2 secondes), des fichiers modifiés dans la même fenêtre temporelle peuvent ne pas être détectés comme changés.

### Ninja : timestamps + hash de commande

Ninja utilise une double vérification, comme introduit en section 28.3 :

1. **Timestamps des fichiers** : même logique que Make — si un fichier source est plus récent que son fichier objet, recompiler.
2. **Hash de la commande** : Ninja enregistre la commande exacte utilisée pour produire chaque fichier dans `.ninja_log`. Si la commande a changé (flags modifiés, compilateur différent, chemins d'inclusion ajoutés), le fichier est recompilé même si aucun timestamp n'a changé.

Cette double vérification élimine les cas où Make nécessite un nettoyage complet. Concrètement :

```bash
# Scénario : changer un flag de compilation

# Avec Make
cmake -B build -G "Unix Makefiles" -DCMAKE_BUILD_TYPE=Release  
cmake --build build                    # Build complet  
# ... modifier CMAKE_CXX_FLAGS dans CMakeLists.txt ...
cmake -B build                         # Reconfigure  
cmake --build build                    # Make ne recompile RIEN (timestamps inchangés)  
cmake --build build --clean-first      # Il faut un clean pour forcer  

# Avec Ninja
cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Release  
cmake --build build                    # Build complet  
# ... modifier CMAKE_CXX_FLAGS dans CMakeLists.txt ...
cmake -B build                         # Reconfigure  
cmake --build build                    # Ninja recompile les fichiers affectés (commande changée)  
```

### Le fichier `.ninja_log`

Le `.ninja_log` stocke, pour chaque cible produite, le timestamp de début de build, le timestamp de fin, le hash de la commande, et le nom du fichier de sortie. Son format est compact — une ligne par cible :

```
# ninja log v6
0	150	7a8b9c0d	src/CMakeFiles/core.dir/core.cpp.o	g++ -std=c++23 -O2 -c ../src/core.cpp ...
150	180	e1f2a3b4	src/CMakeFiles/core.dir/network.cpp.o	g++ -std=c++23 -O2 -c ../src/network.cpp ...
```

Ninja lit ce fichier au démarrage (parsing trivial, une ligne par entrée) et compare les hashs des commandes actuelles (dans `build.ninja`) avec les hashs enregistrés. Si un hash diffère, la cible est marquée pour reconstruction.

## Phase 3 : exécution — parallélisation optimale

### Make : parallélisation explicite et conservatrice

Make parallélise avec `-j N`, mais sa gestion du pool de jobs présente des inefficacités :

**Pas de détection automatique du nombre de cœurs.** Sans `-j`, Make est séquentiel. Le développeur doit explicitement spécifier `-j$(nproc)` à chaque invocation, ou le placer dans un alias shell. Oublier le `-j` divise la vitesse du build par le nombre de cœurs.

**Pas d'ajustement dynamique.** Make lance N jobs et maintient ce nombre constant, sans tenir compte de la charge système actuelle ni de la nature des tâches. Un linkage qui consomme 4 Go de RAM est traité comme une compilation qui en consomme 200 Mo.

**Communication inter-processus limitée.** En mode récursif (`$(MAKE) -C subdir`), chaque sous-Make est un processus indépendant. La coordination du nombre total de jobs entre les sous-processus repose sur un mécanisme de jobserver (pipe partagé) qui fonctionne mais ajoute de la latence.

### Ninja : parallélisation automatique et intelligente

**Détection automatique des cœurs.** Ninja détecte le nombre de cœurs CPU au démarrage et lance automatiquement le nombre optimal de jobs. Aucune option à spécifier.

**Graphe de dépendances global.** Ninja travaille toujours sur un graphe unique et complet. Il n'y a pas de sous-processus Ninja — toutes les tâches sont dans le même pool. Cela permet une parallélisation optimale : Ninja peut lancer des compilations de sous-répertoires différents en parallèle, là où Make récursif attend la fin d'un sous-répertoire avant de commencer le suivant.

**Pools de jobs.** Comme détaillé en section 28.3, les pools permettent de limiter la parallélisation de certaines tâches (linkage, génération de code lourde) sans limiter les autres. CMake utilise cette fonctionnalité quand il est pertinent.

**Démarrage des tâches au plus tôt.** Ninja utilise un ordonnancement basé sur le chemin critique : il lance en priorité les tâches qui débloquent le plus de tâches en aval. Si un fichier source doit être compilé avant que le linkage puisse commencer, Ninja le priorise. Cette heuristique réduit le temps de build total en minimisant les moments où des cœurs CPU sont inactifs en attendant qu'une dépendance soit résolue.

## Le no-op build : le scénario le plus fréquent

Le **no-op build** — un build lancé alors que rien n'a changé — est le scénario le plus fréquent pendant le développement. Vous compilez, lancez un test, relancez le build par habitude ou via un script automatique.

C'est aussi le scénario où l'écart entre Make et Ninja est le plus visible, car l'ensemble du temps est passé dans les phases 1 et 2 (chargement + planification) — la phase 3 (exécution) ne fait rien.

Les ordres de grandeur typiques, sur un projet de taille moyenne (~500 fichiers sources) :

| Phase | Make | Ninja |
|-------|------|-------|
| Chargement du fichier de build | ~300-800 ms | ~10-30 ms |
| Vérification des dépendances | ~200-500 ms | ~20-50 ms |
| **Total no-op** | **~500 ms – 1.3 s** | **~30-80 ms** |

Sur un grand projet (~5000+ fichiers), Make peut atteindre plusieurs secondes de no-op tandis que Ninja reste sous les 200 ms. Cette différence, multipliée par les dizaines de builds quotidiens, représente un gain de productivité cumulé significatif.

## Le fichier `.ninja_deps` : dépendances pré-parsées

Un avantage technique supplémentaire de Ninja est sa gestion des dépendances de headers. Make utilise des fichiers `.d` (générés par le compilateur avec `-MMD -MP`) qui sont des fragments de Makefile inclus via `-include`. Make doit parser ces fichiers texte à chaque invocation.

Ninja lit aussi les fichiers `.d` du compilateur, mais il les **convertit** immédiatement en un format binaire compact stocké dans `.ninja_deps`. Aux invocations suivantes, Ninja lit directement `.ninja_deps` au lieu de reparser les fichiers `.d` texte. Le chargement des dépendances est ainsi considérablement plus rapide.

```
Première compilation :
  compilateur → core.cpp.o.d (texte)
  Ninja → parse core.cpp.o.d → stocke dans .ninja_deps (binaire)

Builds suivants :
  Ninja → lit .ninja_deps (binaire, rapide)
  (les fichiers .d ne sont pas relus)
```

## Récapitulatif des gains par phase

| Phase | Technique Make | Technique Ninja | Gain Ninja |
|-------|---------------|-----------------|:----------:|
| **Chargement** | Interprétation d'un langage complet, hiérarchie de fichiers | Parser linéaire, fichier unique aplati | 10–50× |
| **Dépendances** | Parsing de fichiers `.d` texte à chaque build | Format binaire `.ninja_deps` pré-parsé | 3–10× |
| **Changements** | Timestamps uniquement | Timestamps + hash de commande | Évite les faux négatifs |
| **Parallélisation** | Explicite (`-j`), pas d'ajustement | Automatique, chemin critique, pools | Build total plus court |
| **No-op build** | 500 ms – plusieurs secondes | 30–200 ms | 5–20× |

Le gain total dépend de la taille du projet et de la proportion de fichiers à recompiler. Pour un build complet (phase 3 dominante), l'écart se réduit car le temps de compilation proprement dit est le même — GCC et Clang ne compilent pas plus vite sous Ninja que sous Make. L'avantage de Ninja est dans les phases 1 et 2, et dans l'efficacité de la parallélisation en phase 3.

Pour les **rebuilds incrémentaux** et les **no-op builds** — les scénarios quotidiens du développeur — l'avantage de Ninja est décisif.
