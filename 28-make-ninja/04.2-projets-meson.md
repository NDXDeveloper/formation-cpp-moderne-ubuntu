# 28.4.2 Projets notables utilisant Meson (GNOME, systemd, Mesa)

> **Objectif** : Découvrir les projets majeurs de l'écosystème Linux qui ont adopté Meson, comprendre les raisons de leur migration et les bénéfices qu'ils en ont tirés, pour apprécier concrètement les forces de Meson en contexte de production.

---

## Un build system ancré dans l'infrastructure Linux

Meson n'est pas un outil théorique — il compile une partie significative de l'infrastructure de base d'un système Linux moderne. Si vous utilisez un bureau GNOME, si votre machine a un GPU (Intel, AMD ou NVIDIA avec Mesa), ou si votre système démarre avec systemd, du code compilé par Meson tourne sur votre machine en ce moment.

Cette adoption par des projets critiques et à grande échelle est le meilleur témoignage de la maturité de l'outil. Examinons les cas les plus emblématiques.

---

## GNOME : la migration fondatrice

### Contexte

L'écosystème GNOME — qui inclut GTK, GLib, GIMP, Nautilus, GNOME Shell et des centaines de bibliothèques et d'applications — utilisait historiquement **autotools** (autoconf + automake + libtool). Autotools était le build system standard du monde Unix pendant plus de 20 ans, mais sa complexité, sa lenteur et sa difficulté de maintenance avaient atteint un point critique.

### La migration

GNOME a commencé à migrer vers Meson en 2016–2017. GTK 4, la bibliothèque graphique au cœur de GNOME, a été l'un des premiers composants majeurs à adopter Meson. GLib (la bibliothèque fondamentale de l'écosystème GNOME, utilisée bien au-delà du bureau) a suivi. En 2020, la migration était essentiellement complète : la grande majorité des modules GNOME utilisaient Meson, et les fichiers autotools avaient été supprimés.

### Pourquoi Meson et pas CMake ?

Le choix de Meson plutôt que CMake pour GNOME s'explique par plusieurs facteurs. L'écosystème GNOME repose massivement sur **pkg-config** pour la découverte de bibliothèques — c'est le mécanisme natif de Meson, alors que CMake le supporte comme citoyen de seconde classe. GNOME utilise également **l'introspection GObject** (GIR), un mécanisme de génération de bindings pour d'autres langages, et Meson fournit un support intégré pour cette fonctionnalité via son module `gnome`. Enfin, la philosophie de simplicité de Meson résonnait avec le désir de l'écosystème GNOME de réduire la barrière à la contribution.

### Bénéfices constatés

Les développeurs GNOME ont rapporté des gains significatifs après la migration. Les temps de configuration ont chuté drastiquement — GTK, qui prenait plus d'une minute à configurer avec autotools, se configure en quelques secondes avec Meson. Les fichiers de build sont devenus plus courts et plus lisibles — le `meson.build` de GLib fait une fraction de la taille des anciens `Makefile.am`. Et la cross-compilation pour les plateformes embarquées (comme les cibles ARM pour des systèmes comme le PinePhone) est devenue beaucoup plus simple grâce aux cross files.

## systemd : l'init system universel

### Contexte

systemd est le système d'init et le gestionnaire de services de la quasi-totalité des distributions Linux modernes (Ubuntu, Fedora, Debian, Arch, SUSE). C'est un projet C de grande taille (plusieurs centaines de milliers de lignes) qui compile des dizaines d'exécutables, de bibliothèques partagées et de plugins.

### La migration

systemd a migré d'autotools vers Meson en 2017, avec la version 234. La migration a été effectuée en une seule phase — les deux systèmes de build ont coexisté brièvement, puis autotools a été supprimé. Le commit de suppression des fichiers autotools est l'un des plus volumineux de l'histoire du projet (suppression de milliers de lignes de `Makefile.am` et de scripts `configure.ac`).

### Pourquoi ce choix

systemd est un projet purement Linux — pas de portabilité Windows ou macOS à gérer. Meson avec son backend Ninja exclusif était un choix naturel. Le projet avait besoin de temps de build rapides (les contributeurs compilent systemd des dizaines de fois par jour) et d'une cross-compilation fiable pour les architectures embarquées (ARM, RISC-V).

### Impact

Après la migration, le temps de configuration complet de systemd est passé de dizaines de secondes (autotools) à environ une seconde (Meson). Les contributeurs ont rapporté que les fichiers `meson.build` étaient significativement plus faciles à maintenir — ajouter un nouveau composant à systemd ne nécessite plus de comprendre les arcanes d'autotools.

## Mesa : les pilotes GPU open source

### Contexte

Mesa est l'implémentation open source d'OpenGL, Vulkan et d'autres APIs graphiques sous Linux. C'est le code qui fait tourner votre bureau, vos jeux et vos applications 3D quand vous utilisez un GPU Intel, AMD ou NVIDIA (avec le pilote Nouveau). Mesa est un projet C/C++ massif, avec des dizaines de pilotes pour différents GPUs, des bibliothèques de compilation de shaders, et un code hautement optimisé.

### La migration

Mesa a migré vers Meson progressivement entre 2017 et 2019. Le projet supportait auparavant autotools et SCons (un build system basé sur Python). Meson a remplacé les deux. La migration a été motivée par les temps de build : sur un projet de la taille de Mesa, chaque seconde de latence dans le build system est multipliée par les centaines de rebuilds quotidiens de l'équipe.

### Spécificités

Mesa illustre la capacité de Meson à gérer des projets de grande taille avec des configurations complexes. Le projet a des dizaines d'options de build (quel pilote GPU compiler, quel API graphique supporter, quelles optimisations activer), de nombreuses dépendances système optionnelles (LLVM pour la compilation de shaders, libdrm pour l'accès au noyau, wayland et X11 pour l'affichage), et doit supporter la cross-compilation pour les systèmes embarqués.

Le fichier `meson.build` racine de Mesa est substantiel mais reste lisible — le système de types et les options structurées de Meson permettent d'exprimer cette complexité de manière plus ordonnée qu'avec autotools.

---

## PipeWire : le serveur multimédia moderne

### Contexte

PipeWire est le serveur de gestion audio et vidéo qui remplace PulseAudio et JACK sur les distributions Linux modernes. Adopté par Fedora dès 2021 et progressivement par les autres distributions, il gère le routage audio, la capture vidéo (écran, webcam), et l'interopérabilité entre les applications via des APIs compatibles PulseAudio et JACK.

### Adoption de Meson

PipeWire a été construit avec Meson dès sa création — ce n'est pas une migration mais un choix initial. Lancé en 2017, PipeWire a bénéficié de l'expérience de la communauté GNOME/freedesktop avec Meson et a adopté l'outil naturellement.

Le projet illustre comment Meson gère les bibliothèques à plugins : PipeWire compile un noyau central et de nombreux modules chargés dynamiquement (sinks audio, sources vidéo, bridges de compatibilité), chacun avec ses propres dépendances optionnelles.

## Autres projets notables

Au-delà des quatre projets détaillés ci-dessus, Meson est utilisé par de nombreux composants de l'écosystème Linux et au-delà :

| Projet | Domaine | Migration depuis |
|--------|---------|-----------------|
| GStreamer | Framework multimédia | autotools |
| NetworkManager | Gestion réseau | autotools |
| Wayland / Weston | Protocole d'affichage | autotools |
| libinput | Gestion des périphériques d'entrée | autotools |
| Flatpak | Distribution d'applications sandboxées | autotools |
| QEMU | Émulation et virtualisation | Makefile custom |
| X.Org (certains composants) | Serveur d'affichage legacy | autotools |
| Enlightenment (EFL) | Environnement de bureau alternatif | autotools/CMake |
| DPDK | Data Plane Development Kit (réseau haute performance) | Makefile custom |
| Frida | Reverse engineering dynamique | Build custom |

Le point commun est frappant : la grande majorité de ces migrations partent d'**autotools**, pas de CMake. Meson s'est positionné comme le successeur naturel d'autotools dans l'écosystème Linux, tandis que CMake domine dans l'écosystème C++ au sens large (multiplateforme, bibliothèques tierces, industrie).

## Les enseignements pour un développeur C++

### Ce que ces migrations nous apprennent

**Le build system compte pour la productivité.** Les gains de temps de configuration rapportés par ces projets (de dizaines de secondes à une seconde ou moins) ne sont pas anecdotiques. Multipliés par le nombre de contributeurs et le nombre de builds par jour, ils représentent des centaines d'heures de productivité récupérées par an.

**La simplicité attire les contributeurs.** Plusieurs projets (GNOME, systemd) ont noté que la barrière à la contribution a baissé après la migration vers Meson. Des contributeurs occasionnels qui n'osaient pas toucher aux fichiers autotools sont devenus capables de modifier les fichiers `meson.build`.

**L'écosystème détermine le choix.** Ces projets sont tous ancrés dans l'écosystème Linux/freedesktop, où pkg-config est le mécanisme standard de découverte de bibliothèques, où GObject Introspection est courant, et où la portabilité Windows n'est pas une priorité. Dans cet écosystème, Meson a des avantages concrets sur CMake. Hors de cet écosystème, les avantages de CMake (Conan, vcpkg, support IDE, multiplateforme) reprennent le dessus.

### Quand vous rencontrerez Meson

En tant que développeur C++ sous Linux, vous rencontrerez Meson dans deux contextes principaux :

**Contribuer à un projet existant.** Si vous soumettez un patch à GStreamer, Mesa, systemd, ou n'importe quel projet GNOME, vous devrez lire et potentiellement modifier des fichiers `meson.build`. Les connaissances acquises en section 28.4.1 sont suffisantes pour cette tâche.

**Consommer une bibliothèque construite avec Meson.** La plupart des bibliothèques construites avec Meson installent des fichiers `pkg-config` (`.pc`). Depuis un projet CMake, vous les consommez via `find_package()` avec le module `PkgConfig` ou directement si la bibliothèque fournit aussi des fichiers CMake Config :

```cmake
# Depuis un projet CMake, consommer une bibliothèque installée par Meson
find_package(PkgConfig REQUIRED)  
pkg_check_modules(PIPEWIRE REQUIRED IMPORTED_TARGET libpipewire-0.3)  
target_link_libraries(my_app PRIVATE PkgConfig::PIPEWIRE)  
```

L'interopérabilité entre les deux écosystèmes fonctionne — elle passe par les mécanismes de distribution standard de Linux (pkg-config, headers et bibliothèques installés aux emplacements conventionnels).
