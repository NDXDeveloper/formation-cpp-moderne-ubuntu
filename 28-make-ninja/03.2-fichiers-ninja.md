# 28.3.2 Fichiers `build.ninja`

> **Objectif** : Comprendre la syntaxe du format `build.ninja` — règles, build statements, variables, dépendances — pour être capable de lire les fichiers générés par CMake et diagnostiquer les problèmes de build.

---

## Rappel : vous ne devez pas écrire ces fichiers

Le format `build.ninja` est conçu pour être **généré**, pas écrit à la main. CMake, Meson et GN produisent ces fichiers automatiquement. L'objectif de cette section n'est pas de vous apprendre à écrire du Ninja, mais de vous donner la capacité de **lire** ce que CMake produit — une compétence essentielle quand un flag semble incorrect, qu'un fichier n'est pas recompilé alors qu'il devrait l'être, ou qu'une dépendance manque.

---

## Structure d'un fichier `build.ninja`

Un fichier `build.ninja` généré par CMake se compose de trois parties :

```
┌──────────────────────────────────────┐
│  Variables globales                  │  Compilateur, flags communs
├──────────────────────────────────────┤
│  Définitions de règles (rule)        │  Templates de commandes
├──────────────────────────────────────┤
│  Build statements (build)            │  Tâches concrètes : quoi → comment
└──────────────────────────────────────┘
```

CMake sépare souvent les règles dans un fichier `rules.ninja` inclus depuis le `build.ninja` principal :

```ninja
# build.ninja (début)
ninja_required_version = 1.5

include rules.ninja

# ... suivi des build statements ...
```

---

## Variables

Les variables Ninja sont des paires clé-valeur, sans types. Elles ne supportent qu'un seul niveau d'expansion (pas de récursion) :

```ninja
# Déclaration
cxx = g++  
cxxflags = -std=c++23 -Wall -Wextra -O2  

# Utilisation — avec $
build main.o: cxx_compile main.cpp
  FLAGS = $cxxflags -DAPP_VERSION=\"1.2.0\"
```

La syntaxe d'expansion est `$variable` (pour les noms simples) ou `${variable}` (pour lever les ambiguïtés). À noter que Ninja utilise `$` et non `$()` comme Make.

Les variables peuvent être définies à trois niveaux :

| Niveau | Portée | Exemple |
|--------|--------|---------|
| Global | Tout le fichier | `cxx = g++` (en haut du fichier) |
| Règle | Toutes les utilisations de cette règle | Variables dans un bloc `rule` |
| Build statement | Une seule tâche de build | Variables indentées sous un `build` |

Les variables de build statement **surchargent** les variables de règle, qui surchargent les variables globales.

---

## Règles (`rule`)

Une **règle** est un template de commande. Elle décrit *comment* transformer des entrées en sorties, sans préciser *quels* fichiers sont concernés :

```ninja
rule cxx_compile
  depfile = $DEP_FILE
  deps = gcc
  command = g++ $FLAGS -MD -MF $DEP_FILE -c $in -o $out
  description = Compiling C++ $in
```

### Champs d'une règle

| Champ | Rôle |
|-------|------|
| `command` | La commande shell à exécuter (obligatoire) |
| `description` | Message affiché pendant le build (la barre de progression) |
| `depfile` | Chemin vers le fichier de dépendances généré par le compilateur (`.d`) |
| `deps` | Mode de gestion des dépendances (`gcc` pour les fichiers `.d` style GCC/Clang) |
| `generator` | Marque la règle comme génératrice (re-exécutée si `build.ninja` change) |
| `restat` | Après exécution, revérifier le timestamp de la sortie (utile pour les générateurs de code qui ne modifient pas toujours le fichier) |
| `rspfile` / `rspfile_content` | Fichier de réponse pour les lignes de commande trop longues |
| `pool` | Pool de parallélisation à utiliser |

### Variables spéciales dans les commandes

| Variable | Signification |
|:--------:|--------------|
| `$in` | Les fichiers d'entrée (inputs du build statement) |
| `$out` | Les fichiers de sortie (outputs du build statement) |

Ces variables sont substituées par Ninja au moment de l'exécution, pas au moment du parsing. Elles sont analogues à `$<` et `$@` de Make, mais avec des noms lisibles.

### Exemple : les règles typiques générées par CMake

```ninja
# Compilation C++
rule CXX_COMPILER__my_project_core
  depfile = $DEP_FILE
  deps = gcc
  command = /usr/bin/g++ $DEFINES $INCLUDES $FLAGS -MD -MF $DEP_FILE -o $out -c $in
  description = Building CXX object $out

# Linkage statique (archive)
rule CXX_STATIC_LIBRARY_LINKER__my_project_core
  command = /usr/bin/ar qc $out $in && /usr/bin/ranlib $out
  description = Linking CXX static library $out

# Linkage d'exécutable
rule CXX_EXECUTABLE_LINKER__my_project_app
  command = /usr/bin/g++ $FLAGS $LINK_FLAGS $in -o $out $LINK_LIBRARIES
  description = Linking CXX executable $out
```

CMake crée une règle distincte par **cible** (pas par type de compilation), ce qui permet à chaque cible d'avoir ses propres flags. Le nom de la règle inclut le nom de la cible CMake (`my_project_core`, `my_project_app`).

## Build statements (`build`)

Un **build statement** est une tâche concrète. Il associe des fichiers de sortie à une règle et à des fichiers d'entrée :

```ninja
build src/CMakeFiles/my_project_core.dir/core.cpp.o: CXX_COMPILER__my_project_core ../src/core.cpp
  DEP_FILE = src/CMakeFiles/my_project_core.dir/core.cpp.o.d
  DEFINES = -DUSE_SSL=1
  INCLUDES = -I../include -isystem /usr/include/openssl
  FLAGS = -std=c++23 -Wall -Wextra -O2
```

### Anatomie

```
build <outputs>: <rule> <inputs> | <implicit_inputs> || <order_only_inputs>
  VARIABLE = value
```

| Élément | Signification |
|---------|--------------|
| `<outputs>` | Fichier(s) produit(s) |
| `<rule>` | Nom de la règle à appliquer |
| `<inputs>` | Fichiers d'entrée explicites (apparaissent dans `$in`) |
| `<implicit_inputs>` (après `\|`) | Dépendances supplémentaires qui ne font pas partie de `$in` |
| `<order_only_inputs>` (après `\|\|`) | Doivent exister avant l'exécution, mais leur timestamp ne déclenche pas de rebuild |
| Variables indentées | Surcharges locales à ce build statement |

### Exemple complet commenté

```ninja
# Compiler core.cpp en core.cpp.o
build src/CMakeFiles/my_project_core.dir/core.cpp.o: CXX_COMPILER__my_project_core ../src/core.cpp
  DEP_FILE = src/CMakeFiles/my_project_core.dir/core.cpp.o.d
  DEFINES = -DWITH_NETWORKING=1 -DFOUNDATION_VERSION=3
  INCLUDES = -I../include -I../libs/utils/include -isystem /home/dev/.conan2/p/openss.../include
  FLAGS = -std=c++23 -Wall -Wextra -Wpedantic -O2

# Créer la bibliothèque statique à partir des .o
build libmy_project_core.a: CXX_STATIC_LIBRARY_LINKER__my_project_core \
    src/CMakeFiles/my_project_core.dir/core.cpp.o \
    src/CMakeFiles/my_project_core.dir/network.cpp.o

# Lier l'exécutable
build apps/my-app: CXX_EXECUTABLE_LINKER__my_project_app \
    apps/CMakeFiles/my_project_app.dir/main.cpp.o | \
    libmy_project_core.a \
    libmy_project_utils.a
  LINK_LIBRARIES = libmy_project_core.a libmy_project_utils.a -lssl -lcrypto -lpthread
```

La ligne de linkage de l'exécutable montre une dépendance **implicite** (après `|`) sur les bibliothèques statiques. Ces fichiers doivent être à jour avant le linkage, mais ne font pas partie de `$in` (ils sont référencés dans `$LINK_LIBRARIES` à la place).

## Dépendances implicites et order-only

### Dépendances implicites (`|`)

Les dépendances après `|` sont des prérequis dont le timestamp est vérifié, mais qui n'apparaissent pas dans `$in`. Elles sont utilisées pour les bibliothèques à lier, les headers générés, ou tout fichier qui doit être à jour mais qui n'est pas passé directement au compilateur via la commande :

```ninja
build apps/my-app: link main.o | libcore.a generated_config.h
```

Si `libcore.a` ou `generated_config.h` sont plus récents que `my-app`, Ninja relance le linkage.

### Dépendances order-only (`||`)

Les dépendances après `||` doivent **exister** avant l'exécution, mais leur timestamp n'est **pas** vérifié. C'est le mécanisme pour les répertoires de build :

```ninja
build build/obj/main.o: compile main.cpp || build/obj
```

Le répertoire `build/obj` doit exister avant la compilation, mais une modification de son timestamp (par l'ajout d'un autre fichier dans le répertoire) ne doit pas déclencher une recompilation de `main.o`. C'est l'équivalent des *order-only prerequisites* de Make (section 28.1).

---

## Gestion des dépendances de headers

Le mécanisme `depfile` + `deps = gcc` est la façon dont Ninja intègre les dépendances de headers générées par le compilateur :

```ninja
rule compile
  depfile = $DEP_FILE
  deps = gcc
  command = g++ -MD -MF $DEP_FILE -c $in -o $out
```

Le flag `-MD -MF $DEP_FILE` dit au compilateur : « en plus de compiler, écris les dépendances de headers dans `$DEP_FILE` ». Après la compilation, Ninja lit ce fichier `.d`, extrait les dépendances, les stocke dans `.ninja_deps` (format binaire), et supprime le fichier `.d`. Aux builds suivants, Ninja consulte `.ninja_deps` directement — pas besoin de reparser les fichiers texte.

## Cibles spéciales

### Cibles alias (`phony`)

Les alias permettent de donner un nom court à un ensemble de fichiers :

```ninja
build all: phony apps/my-app libmy_project_core.a

build my_project_core: phony libmy_project_core.a

build clean: CLEAN
```

`phony` est une règle intégrée qui ne fait rien — elle déclare juste une dépendance entre un nom et des fichiers. C'est l'équivalent de `.PHONY` dans Make.

CMake génère des alias phony pour chaque cible CMake, ce qui vous permet d'écrire :

```bash
ninja my_project_core    # Au lieu du chemin complet libmy_project_core.a  
ninja my_project_app     # Au lieu de apps/my-app  
```

### Reconfiguration automatique

CMake ajoute une règle spéciale qui régénère `build.ninja` si un `CMakeLists.txt` change :

```ninja
rule RERUN_CMAKE
  command = /usr/bin/cmake --regenerate-during-build -S../. -B.
  description = Re-running CMake...
  generator = 1

build build.ninja: RERUN_CMAKE ../CMakeLists.txt ../src/CMakeLists.txt ../apps/CMakeLists.txt | ...
```

Le flag `generator = 1` indique à Ninja que cette règle peut modifier le fichier `build.ninja` lui-même. Après l'exécution, Ninja **recharge** le fichier et recalcule le graphe de dépendances. C'est ce qui rend la reconfiguration transparente : vous modifiez un `CMakeLists.txt`, lancez `ninja`, et CMake est automatiquement relancé avant la compilation.

## Inclusions (`subninja` et `include`)

Ninja supporte deux formes d'inclusion :

```ninja
# include — les variables du fichier inclus sont dans la portée globale
include rules.ninja

# subninja — les variables du fichier inclus sont dans une portée locale
subninja subproject/build.ninja
```

CMake utilise `include` pour séparer les définitions de règles (`rules.ninja`) du reste du build. `subninja` est plus rare — il est utilisé quand un sous-projet a son propre espace de noms de variables.

## Diagnostic : lire le `build.ninja` généré par CMake

Quand quelque chose ne fonctionne pas dans un build, le `build.ninja` est la source de vérité. Voici les techniques de diagnostic courantes.

### Trouver les flags d'un fichier spécifique

```bash
# Chercher les flags de compilation de core.cpp
grep -A 5 "core.cpp.o:" build/build.ninja
```

La sortie montre le build statement avec toutes les variables (`FLAGS`, `INCLUDES`, `DEFINES`). Vérifiez que les flags attendus sont présents.

### Vérifier les dépendances d'une cible

```bash
# Via ninja -t deps
ninja -C build -t deps src/CMakeFiles/my_project_core.dir/core.cpp.o
```

Cette commande affiche les dépendances enregistrées dans `.ninja_deps` — la liste complète des headers dont `core.cpp.o` dépend. Si un header que vous avez modifié n'apparaît pas dans cette liste, c'est qu'il n'est pas inclus (directement ou transitivement) par `core.cpp`.

### Afficher les commandes sans les exécuter

```bash
# Dry-run : afficher les commandes qui seraient exécutées
ninja -C build -n -v
```

L'option `-n` (dry-run) combinée avec `-v` (verbose) affiche les commandes complètes sans rien exécuter. Utile pour vérifier la ligne de compilation exacte avant un rebuild.

### Visualiser le graphe de dépendances

```bash
# Graphe Graphviz pour une cible spécifique
ninja -C build -t graph my_project_app > graph.dot  
dot -Tsvg graph.dot -o graph.svg  

# Ouvrir le SVG dans un navigateur
xdg-open graph.svg
```

Le graphe montre les dépendances entre tous les fichiers impliqués dans la construction de `my_project_app` — des fichiers sources aux bibliothèques, en passant par les fichiers objets intermédiaires. C'est un outil puissant pour comprendre pourquoi un fichier est (ou n'est pas) recompilé.

### Comparer avec `compile_commands.json`

Le fichier `compile_commands.json` (généré par CMake avec `CMAKE_EXPORT_COMPILE_COMMANDS=ON`) offre une vue plus lisible des commandes de compilation, au format JSON :

```bash
# Chercher la commande pour core.cpp
python3 -c "  
import json  
with open('build/compile_commands.json') as f:  
    for entry in json.load(f):
        if 'core.cpp' in entry['file']:
            print(entry['command'])
"
```

C'est souvent plus pratique que de fouiller dans le `build.ninja` brut, surtout pour vérifier les flags d'inclusion et les définitions.

## Récapitulatif de la syntaxe

| Concept | Syntaxe Ninja | Équivalent Make |
|---------|--------------|-----------------|
| Variable | `var = value` | `VAR = value` |
| Expansion | `$var` ou `${var}` | `$(VAR)` |
| Règle (template) | `rule name` | Règle de pattern `%.o: %.cpp` |
| Tâche de build | `build out: rule in` | Règle explicite `out: in` |
| Entrées | `$in` | `$^` |
| Sortie | `$out` | `$@` |
| Dépendances implicites | `build out: rule in \| deps` | `out: in deps` (pas de distinction) |
| Order-only | `build out: rule in \|\| dirs` | `out: in \| dirs` |
| Alias | `build name: phony files` | `.PHONY: name` |
| Inclusion | `include file.ninja` | `include file.mk` |
| Inclusion scopée | `subninja file.ninja` | *(pas d'équivalent)* |
