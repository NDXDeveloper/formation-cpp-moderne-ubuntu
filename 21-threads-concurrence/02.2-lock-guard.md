# 21.2.2 std::lock_guard

## Le wrapper RAII minimal

`std::lock_guard` est le moyen le plus simple et le plus sûr de verrouiller un mutex en C++. Son fonctionnement tient en une phrase : le constructeur appelle `lock()`, le destructeur appelle `unlock()`. Rien de plus.

```cpp
#include <mutex>

std::mutex mtx;

void safe_operation() {
    std::lock_guard<std::mutex> lock(mtx);  // → mtx.lock()
    // ... section critique ...
}   // lock est détruit → mtx.unlock()
```

Ce pattern RAII (voir section 6.3) élimine toute une catégorie de bugs : qu'un `return` anticipé, une exception, ou un `break` interrompe le flux d'exécution, le destructeur de `lock_guard` est appelé, et le mutex est libéré. C'est garanti par le langage.

---

## Déduction de type (CTAD)

Depuis C++17, grâce au *Class Template Argument Deduction*, vous n'avez plus besoin de spécifier le type du mutex explicitement :

```cpp
// C++11/14 : type explicite
std::lock_guard<std::mutex> lock(mtx);

// C++17+ : déduction automatique (recommandé)
std::lock_guard lock(mtx);
```

La forme C++17 est plus concise et tout aussi lisible. Utilisez-la systématiquement.

---

## Anatomie interne

L'implémentation de `std::lock_guard` est remarquablement simple. Elle tient en quelques lignes — c'est ce qui en fait un outil fiable et sans surcoût :

```cpp
// Implémentation simplifiée (à titre pédagogique)
template <typename Mutex>  
class lock_guard {  
    Mutex& mtx_;
public:
    explicit lock_guard(Mutex& m) : mtx_(m) { mtx_.lock(); }
    ~lock_guard() { mtx_.unlock(); }

    lock_guard(const lock_guard&) = delete;
    lock_guard& operator=(const lock_guard&) = delete;
};
```

Quelques points clés ressortent de cette implémentation :

- Le mutex est stocké **par référence**. Le `lock_guard` ne possède pas le mutex, il le gère temporairement.
- Le constructeur est `explicit` pour éviter les conversions implicites accidentelles.
- La copie et l'affectation sont supprimées — un verrou ne se duplique pas.
- Le déplacement n'est pas supporté non plus. Un `lock_guard` est ancré dans sa portée.

Le compilateur inline entièrement cette classe dans la plupart des cas. Le coût d'un `lock_guard` par rapport à un appel `lock()`/`unlock()` manuel est strictement **nul** — c'est du *zero-cost abstraction* au sens propre.

## Cas d'usage typiques

### Protection d'une structure de données partagée

Le scénario le plus courant : protéger l'accès à un conteneur ou une structure partagée entre threads.

```cpp
#include <mutex>
#include <vector>
#include <thread>
#include <print>

class ThreadSafeQueue {
    mutable std::mutex mtx_;
    std::vector<int> queue_;

public:
    void push(int value) {
        std::lock_guard lock(mtx_);
        queue_.push_back(value);
    }

    std::size_t size() const {
        std::lock_guard lock(mtx_);
        return queue_.size();
    }

    void dump() const {
        std::lock_guard lock(mtx_);
        for (int v : queue_) {
            std::print("{} ", v);
        }
        std::println();
    }
};

int main() {
    ThreadSafeQueue q;

    std::thread producer([&q] {
        for (int i = 0; i < 100; ++i) {
            q.push(i);
        }
    });

    std::thread consumer([&q] {
        for (int i = 100; i < 200; ++i) {
            q.push(i);
        }
    });

    producer.join();
    consumer.join();
    std::println("Taille : {}", q.size());  // Toujours 200
}
```

> 💡 Notez que `mtx_` est déclaré `mutable` car `size()` et `dump()` sont `const`. Sans `mutable`, le compilateur interdirait d'appeler `lock()` sur un membre dans une méthode `const`. C'est une convention standard en C++ concurrent : le mutex n'est pas une donnée logique — c'est un mécanisme de protection. Le qualifier `mutable` permet aux méthodes `const` de l'acquérir sans compromettre les garanties sémantiques de `const`.

### Contrôler la portée de la section critique

En plaçant le `lock_guard` dans un bloc `{}` explicite, vous contrôlez précisément la durée du verrouillage :

```cpp
#include <mutex>
#include <print>

std::mutex mtx;  
int shared_value = 0;  

void precise_locking() {
    // Phase 1 : calcul local (pas besoin du mutex)
    int local_result = expensive_computation();

    // Phase 2 : mise à jour partagée (section critique minimale)
    {
        std::lock_guard lock(mtx);
        shared_value += local_result;
    }
    // Le mutex est déjà libéré ici

    // Phase 3 : travail qui ne nécessite pas le mutex
    log_result(local_result);
}
```

Cette technique est essentielle pour garder les sections critiques aussi courtes que possible. Le bloc `{}` rend visuellement explicite où commence et où finit la protection — c'est bien plus lisible qu'un `lock()`/`unlock()` manuel séparé par des dizaines de lignes.

### Sécurité face aux exceptions

Le scénario qui justifie à lui seul l'existence de `lock_guard` :

```cpp
#include <mutex>
#include <stdexcept>
#include <vector>

std::mutex mtx;  
std::vector<int> data;  

void exception_safe_insert(int value) {
    std::lock_guard lock(mtx);

    validate(value);          // Peut lever une exception
    data.push_back(value);    // Peut lever std::bad_alloc
    notify_observers(value);  // Peut lever une exception

    // Que validate(), push_back() ou notify_observers() lève :
    // → le stack unwinding détruit 'lock'
    // → le destructeur appelle mtx.unlock()
    // → le mutex est libéré proprement
}
```

Sans `lock_guard`, il faudrait un `try`/`catch` englobant toute la section critique, avec un `mtx.unlock()` dans le `catch` et dans le chemin normal. C'est verbeux, fragile, et facile à mal écrire. Le wrapper RAII rend le code à la fois plus court et plus correct.

## Constructeur adopt_lock

Il existe un second constructeur de `lock_guard` qui prend un tag `std::adopt_lock`. Il indique que le mutex est **déjà verrouillé** par le thread courant et que le `lock_guard` doit seulement se charger du `unlock()` à la destruction, sans appeler `lock()` dans le constructeur.

```cpp
#include <mutex>

std::mutex mtx1;  
std::mutex mtx2;  

void transfer() {
    // Verrouillage simultané de deux mutex (évite les deadlocks)
    std::lock(mtx1, mtx2);

    // Adopter les verrous déjà acquis pour gérer le unlock via RAII
    std::lock_guard lock1(mtx1, std::adopt_lock);
    std::lock_guard lock2(mtx2, std::adopt_lock);

    // ... opération nécessitant les deux mutex ...
}
// Destruction de lock2 → mtx2.unlock()
// Destruction de lock1 → mtx1.unlock()
```

`std::lock()` est une fonction libre qui verrouille plusieurs mutex simultanément avec un algorithme d'évitement de deadlock (typiquement try-and-back-off). Combiné avec `lock_guard` et `adopt_lock`, on obtient un verrouillage multi-mutex sûr.

> 💡 Ce pattern `std::lock` + `lock_guard(adopt_lock)` a été la solution standard en C++11/14. Depuis C++17, `std::scoped_lock` (section 21.2.4) fait tout cela en une seule ligne et de façon plus élégante. Si vous écrivez du code C++17 ou plus récent, préférez `std::scoped_lock`.

## Limitations

`std::lock_guard` est intentionnellement limité. C'est sa force (simplicité, prévisibilité), mais cela signifie qu'il ne convient pas à tous les scénarios :

### Pas de déverrouillage anticipé

Une fois construit, un `lock_guard` détient le verrou jusqu'à sa destruction. Vous ne pouvez pas le déverrouiller manuellement au milieu de sa portée :

```cpp
void no_early_unlock() {
    std::lock_guard lock(mtx);
    update_shared_data();
    // Impossible de relâcher le mutex ici
    // pour faire du travail local non-protégé
    long_local_computation();  // Le mutex est tenu inutilement
}
```

La solution la plus simple est le bloc `{}` montré plus haut. Si vous avez besoin d'un contrôle plus fin (verrouillage différé, déverrouillage explicite, re-verrouillage), utilisez `std::unique_lock` (section 21.2.3).

### Pas de déplacement

`std::lock_guard` n'est ni copiable, ni déplaçable. Vous ne pouvez pas le retourner depuis une fonction, le stocker dans un conteneur, ou le transférer à un autre scope :

```cpp
// ❌ Ne compile pas
std::lock_guard<std::mutex> get_lock() {
    return std::lock_guard(mtx);
}
```

Pour un verrou déplaçable, utilisez `std::unique_lock`.

### Incompatible avec std::condition_variable

`std::condition_variable::wait()` a besoin de déverrouiller et re-verrouiller le mutex en interne. `std::lock_guard` ne fournit pas cette capacité. Si vous utilisez des variables de condition (section 21.3), vous devez utiliser `std::unique_lock`.

### Un seul mutex

`std::lock_guard` ne verrouille qu'un seul mutex. Pour verrouiller plusieurs mutex simultanément, utilisez `std::scoped_lock` (section 21.2.4).

---

## lock_guard vs les alternatives : quand l'utiliser

Le choix du wrapper dépend du scénario :

| Besoin | Wrapper recommandé |
|--------|-------------------|
| Verrouillage simple, portée fixe | **`std::lock_guard`** |
| Déverrouillage anticipé ou verrouillage différé | `std::unique_lock` |
| Utilisation avec `std::condition_variable` | `std::unique_lock` |
| Verrouillage de plusieurs mutex à la fois | `std::scoped_lock` |
| Besoin d'un verrou déplaçable (retour de fonction) | `std::unique_lock` |

`std::lock_guard` est le **choix par défaut** quand aucun de ces besoins spéciaux ne s'applique. Il communique clairement l'intention au lecteur : « ce mutex est verrouillé pour toute la durée de ce scope, point final ». Sa simplicité est un signal — quand vous voyez un `lock_guard`, vous savez immédiatement qu'il n'y aura pas de subtilité de verrouillage dans le code qui suit.

> 💡 Certaines équipes adoptent la convention d'utiliser `std::scoped_lock` partout (y compris pour un seul mutex), par cohérence et parce qu'il est strictement plus général. C'est un choix de style valide — `std::scoped_lock` avec un seul mutex est fonctionnellement identique à `std::lock_guard` et n'a aucun surcoût supplémentaire.

## Résumé

| Aspect | Détail |
|--------|--------|
| Header | `<mutex>` |
| Construction | Verrouille le mutex immédiatement |
| Destruction | Déverrouille le mutex automatiquement |
| Copiable | Non |
| Déplaçable | Non |
| Déverrouillage anticipé | Non (utiliser `std::unique_lock`) |
| Multi-mutex | Non (utiliser `std::scoped_lock`) |
| Condition variables | Non compatible (utiliser `std::unique_lock`) |
| Surcoût vs lock/unlock manuel | Zéro (inliné par le compilateur) |
| CTAD (C++17) | `std::lock_guard lock(mtx);` — pas besoin de `<std::mutex>` |

> **À suivre** : la section 21.2.3 présente `std::unique_lock`, le wrapper flexible qui lève toutes les limitations de `lock_guard` — au prix d'une complexité légèrement supérieure.
