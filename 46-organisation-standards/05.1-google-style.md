# 46.5.1 — Google C++ Style Guide

> **Chapitre 46 : Organisation et Standards** · Section 46.5 : Standards de codage  
> **Niveau** : Expert · **Prérequis** : Section 46.5 (introduction aux standards de codage)

---

## Origine et contexte

Le Google C++ Style Guide est né d'un besoin concret : maintenir la cohérence d'une base de code C++ colossale — des centaines de millions de lignes — sur laquelle travaillent quotidiennement des milliers d'ingénieurs. Dans ce contexte, la lisibilité n'est pas un luxe : un ingénieur Google passe plus de temps à lire du code qu'à en écrire, souvent du code écrit par des développeurs qu'il ne connaît pas, dans des sous-systèmes qu'il découvre pour la première fois.

Le guide est public, hébergé sur `google.github.io/styleguide/cppguide.html`, et régulièrement mis à jour. Sa dernière révision majeure a modernisé plusieurs positions historiques, notamment sur les exceptions et sur l'adoption des fonctionnalités C++ récentes. Il reste l'un des guides de style les plus influents de l'industrie et sert de base à de nombreuses équipes en dehors de Google — y compris via le profil `BasedOnStyle: Google` de `clang-format`.

---

## Philosophie fondamentale

Le guide repose sur un principe directeur explicite : **optimiser pour le lecteur, pas pour l'auteur**. Un code est écrit une fois mais lu des dizaines, voire des centaines de fois. Chaque convention est évaluée à travers ce prisme : réduit-elle l'ambiguïté à la lecture ? Permet-elle de comprendre l'intention sans chercher la déclaration ? Minimise-t-elle les surprises ?

Cette philosophie se traduit par plusieurs parti-pris récurrents :

**La cohérence prime sur l'optimalité locale.** Même quand une convention alternative serait marginalement meilleure dans un cas précis, le guide préfère la règle uniforme qui s'applique partout sans exception. Un développeur qui connaît le guide sait à quoi s'attendre dans n'importe quel fichier.

**L'explicite est préféré à l'implicite.** Le guide est méfiant envers les fonctionnalités du langage qui cachent un comportement complexe derrière une syntaxe simple. C'est l'une des raisons de la prudence historique envers les exceptions et de l'encadrement strict de `auto`.

**La simplicité est une vertu.** Le guide décourage l'utilisation de fonctionnalités avancées du langage quand une approche plus simple et plus lisible existe, même si elle est techniquement moins élégante. La métaprogrammation template, par exemple, est tolérée mais encadrée.

## Conventions de nommage

Le système de nommage du Google Style est l'un de ses aspects les plus distinctifs et les plus cohérents. Chaque catégorie de symbole a un style propre, ce qui permet d'identifier la nature d'un identifiant à vue.

### Types : `PascalCase`

Toutes les entités qui définissent un type — classes, structs, enums, typedefs, alias de types, paramètres de template — utilisent le `PascalCase` :

```cpp
class ConnectionPool;  
struct ServerConfig;  
enum class LogLevel;  
using StringMap = std::unordered_map<std::string, std::string>;  
template <typename RequestHandler>  
class HttpServer;  
```

### Fonctions et méthodes : `PascalCase`

Contrairement à de nombreux autres guides, le Google Style utilise le `PascalCase` pour les fonctions et méthodes, pas le `camelCase` ni le `snake_case` :

```cpp
class Engine {  
public:  
    void StartProcessing();
    bool IsRunning() const;
    int ProcessedCount() const;

    static Engine CreateDefault();
};

// Fonctions libres
void InitializeSubsystem();  
int ComputeChecksum(std::string_view data);  
```

Les accesseurs suivent des patterns spécifiques. Pour un membre `count_`, le getter est `count()` (minuscule, sans `Get`) et le setter est `set_count()` :

```cpp
class Config {  
public:  
    int port() const { return port_; }
    void set_port(int port) { port_ = port; }

    const std::string& hostname() const { return hostname_; }
    void set_hostname(std::string hostname) { hostname_ = std::move(hostname); }

private:
    int port_ = 8080;
    std::string hostname_ = "localhost";
};
```

Cette convention distingue visuellement les accesseurs (minuscules, courts) des méthodes qui effectuent un travail réel (`PascalCase`).

### Variables : `snake_case`

Les variables locales, les paramètres de fonction et les membres de données utilisent le `snake_case` :

```cpp
void ProcessRequest(const HttpRequest& incoming_request) {
    int retry_count = 0;
    auto response_buffer = CreateBuffer();
    // ...
}
```

### Membres de données : `snake_case` avec trailing underscore

Les membres de données des classes (pas des structs) portent un underscore terminal qui les distingue des variables locales :

```cpp
class Connection {  
private:  
    std::string host_;           // Trailing underscore → membre privé
    int port_;
    bool connected_;
    std::unique_ptr<Socket> socket_;
};
```

Pour les structs (dont tous les membres sont publics par convention), le trailing underscore n'est pas utilisé :

```cpp
struct Point {
    double x;    // Pas de trailing underscore
    double y;
};
```

Cette distinction reflète une différence sémantique : les structs dans le Google Style sont des agrégats de données passifs, les classes encapsulent un comportement.

### Constantes : `k` + `PascalCase`

Les constantes à portée de fichier ou de classe sont préfixées par `k` :

```cpp
constexpr int kMaxConnections = 1024;  
constexpr double kPi = 3.14159265358979323846;  
constexpr std::string_view kDefaultConfigPath = "/etc/myapp/config.json";  

class HttpServer {
    static constexpr int kDefaultPort = 8080;
    static constexpr int kMaxHeaderSize = 8192;
};
```

Le préfixe `k` vient de la convention allemande "Konstante" et signale immédiatement qu'on lit une valeur fixe, pas une variable mutable.

### Namespaces : `snake_case`

```cpp
namespace my_project {  
namespace internal {  
namespace network_utils {  
    // ...
}
}
}
```

### Macros : `UPPER_SNAKE_CASE`

Les macros (à éviter autant que possible en C++ moderne) utilisent le tout-majuscules avec underscores, préfixé par le nom du projet :

```cpp
#define MYPROJECT_DISALLOW_COPY(ClassName) \
    ClassName(const ClassName&) = delete;  \
    ClassName& operator=(const ClassName&) = delete
```

### Enums : valeurs en `PascalCase` avec préfixe `k`

```cpp
enum class Color {
    kRed,
    kGreen,
    kBlue,
    kAlphaTransparent,
};
```

### Récapitulatif

| Catégorie | Style | Exemple |
|---|---|---|
| Classes, structs, enums, typedefs | `PascalCase` | `ConnectionPool` |
| Fonctions, méthodes | `PascalCase` | `StartProcessing()` |
| Accesseurs (getters) | `snake_case` | `port()` |
| Accesseurs (setters) | `set_` + `snake_case` | `set_port()` |
| Variables locales, paramètres | `snake_case` | `retry_count` |
| Membres de données (classe) | `snake_case_` | `host_` |
| Membres de données (struct) | `snake_case` | `host` |
| Constantes | `k` + `PascalCase` | `kMaxRetries` |
| Namespaces | `snake_case` | `my_project` |
| Macros | `UPPER_SNAKE_CASE` | `MYPROJECT_CHECK` |
| Paramètres de template | `PascalCase` | `typename ValueType` |

## Règles de formatage

Le formatage Google est entièrement encodable dans un fichier `.clang-format`. Voici les caractéristiques principales :

**Indentation : 2 espaces.** C'est l'un des choix les plus distinctifs du Google Style. La justification est la densité de code visible à l'écran, critique sur une base de code profondément imbriquée. De nombreuses équipes qui adoptent le Google Style comme base changent ce paramètre à 4 espaces.

**Longueur de ligne : 80 colonnes.** Une limite historique héritée des terminaux, mais toujours défendue par Google pour faciliter le côte-à-côte de deux fichiers et la lisibilité des diffs dans les outils de code review.

**Accolades : style K&R attaché (Attach).** L'accolade ouvrante reste sur la même ligne que la déclaration :

```cpp
if (condition) {
    DoSomething();
} else {
    DoSomethingElse();
}

class Engine {  
public:  
    void Start();
};

void Engine::Start() {
    // ...
}
```

**Espaces autour des opérateurs, pas à l'intérieur des parenthèses :**

```cpp
int result = a + b * c;  
if (x > 0) {  
    Process(x, y);
}
```

**Configuration `.clang-format` correspondante :**

```yaml
BasedOnStyle: Google
# Ajustements courants :
# IndentWidth: 4            # Si vous préférez 4 espaces
# ColumnLimit: 100          # Si 80 est trop restrictif
# AccessModifierOffset: -2  # Indentation des public:/private:
```

## Règles d'usage du langage

C'est la partie la plus substantielle du guide et celle qui suscite le plus de débat. Le Google Style ne se contente pas de fixer le formatage — il tranche sur l'utilisation des fonctionnalités du langage.

### `auto` : usage encadré

Le guide autorise `auto` mais le restreint aux cas où le type est évident dans le contexte immédiat :

```cpp
// ✅ Acceptable : le type est évident
auto it = container.begin();  
auto ptr = std::make_unique<Engine>();  
auto [key, value] = *map.begin();    // Structured bindings  

// ✅ Acceptable : le type exact est long et non informatif
auto result = std::make_shared<std::unordered_map<std::string, std::vector<int>>>();

// ❌ Déconseillé : le type n'est pas évident
auto config = GetConfig();   // Quel type retourne GetConfig() ?  
auto x = Compute(a, b);     // int ? double ? un objet complexe ?  
```

La philosophie est claire : `auto` ne doit pas forcer le lecteur à chercher la déclaration de la fonction appelée pour comprendre le type d'une variable.

### Exceptions : position évoluée

Historiquement, le Google Style interdisait les exceptions C++ — une position controversée mais justifiée par le contexte de Google (base de code massive, code legacy sans exception safety, coût de la migration). Cette restriction a été la critique la plus fréquente adressée au guide.

La position a évolué au fil des révisions. Le guide reconnaît désormais la valeur des exceptions dans le code nouveau et les projets qui les supportent dès l'origine, tout en maintenant des directives strictes : les exceptions ne doivent être utilisées que pour les erreurs véritablement exceptionnelles, jamais pour le contrôle de flux, et chaque fonction doit documenter ses garanties d'exception safety (section 17.4).

Pour les projets qui partent de zéro, le guide moderne encourage l'utilisation de `std::expected` (section 12.8) comme mécanisme principal de gestion d'erreurs, avec les exceptions réservées aux cas véritablement irrécupérables.

### Héritage : restrictions

Le guide encadre l'utilisation de l'héritage avec plusieurs règles :

- L'héritage d'implémentation (non virtuel) est limité. La composition est préférée.  
- L'héritage multiple d'implémentation est très fortement déconseillé. Seul l'héritage multiple d'interfaces pures (classes abstraites sans état) est encouragé.  
- Tout destructeur d'une classe destinée à être héritée doit être soit `virtual`, soit `protected` et non virtuel.  
- Le mot-clé `override` est obligatoire sur toute méthode qui surcharge une méthode virtuelle.

### RTTI : déconseillé

Le Run-Time Type Information (`dynamic_cast`, `typeid`) est déconseillé par le guide, au motif que le recours au RTTI signale souvent un problème de design — un switch sur le type d'un objet est habituellement mieux résolu par le polymorphisme ou le pattern Visitor.

### Fonctions et méthodes

Le guide fixe plusieurs conventions pratiques :

- Les paramètres de sortie sont passés par pointeur (pas par référence non const), pour rendre l'appel explicite côté appelant (`&result` est plus visible que `result`). Cette convention reste largement appliquée mais perd de sa pertinence avec `std::optional`, `std::expected` et les structured bindings.  
- Les paramètres d'entrée sont passés par référence const ou par valeur.  
- Les fonctions ne doivent pas dépasser une taille raisonnable — le guide ne fixe pas de limite stricte mais encourage le découpage.  
- L'ordre de déclaration dans une classe est : types et alias, constantes statiques, factory methods, constructeurs, opérateur d'affectation, destructeur, puis les autres méthodes, groupées logiquement.

### Smart pointers et propriété

Le guide est très explicite sur la sémantique de propriété :

- `std::unique_ptr` est le choix par défaut pour la propriété exclusive.  
- `std::shared_ptr` n'est utilisé que quand la propriété partagée est véritablement nécessaire — pas comme raccourci pour éviter de réfléchir au lifetime.  
- Les raw pointers (`T*`) et les références (`T&`) sont des vues non possédantes : elles n'impliquent aucune responsabilité de libération.  
- `new` et `delete` explicites sont proscrits au profit de `std::make_unique` et `std::make_shared`.

### Préprocesseur : usage minimal

Les macros sont considérées comme un dernier recours. Le guide recommande :

- `constexpr` plutôt que `#define` pour les constantes.  
- Les templates ou les fonctions `inline` plutôt que les macros fonctionnelles.  
- Si une macro est inévitable, elle doit être préfixée par le nom du projet et `#undef` dès que possible.

### Inclusion de headers

Le guide prescrit un ordre d'inclusion strict (que nous avons détaillé en section 46.2) :

1. Le header correspondant au fichier `.cpp`
2. Les headers système C
3. Les headers de la librairie standard C++
4. Les headers de bibliothèques tierces
5. Les headers du projet

Chaque groupe est séparé par une ligne vide. À l'intérieur de chaque groupe, les includes sont triés alphabétiquement. `clang-format` avec le style Google applique automatiquement cet ordre via l'option `SortIncludes`.

## Configuration `clang-tidy` pour le Google Style

Les checks `clang-tidy` préfixés par `google-` vérifient les règles spécifiques du Google Style :

```yaml
# .clang-tidy (extrait centré Google Style)
Checks: >
  google-build-explicit-make-pair,
  google-build-namespaces,
  google-build-using-namespace,
  google-default-arguments,
  google-explicit-constructor,
  google-global-names-in-headers,
  google-readability-avoid-underscore-in-googletest-name,
  google-readability-casting,
  google-readability-todo,
  google-runtime-int,
  google-runtime-operator,
  google-upgrade-googletest-case
```

Les checks les plus importants :

- **`google-explicit-constructor`** : exige le mot-clé `explicit` sur les constructeurs à un seul paramètre, évitant les conversions implicites accidentelles.  
- **`google-build-using-namespace`** : interdit `using namespace` dans les headers.  
- **`google-readability-casting`** : interdit les casts C-style (`(int)x`) au profit des casts C++ (`static_cast<int>(x)`).  
- **`google-runtime-int`** : signale l'utilisation de `short`, `long` et `long long` au profit des types de taille fixe (`int32_t`, `int64_t`).

## Forces du Google Style

**Complétude.** C'est le guide le plus exhaustif. Presque chaque question de style a une réponse, ce qui minimise les zones grises et les débats en code review.

**Écosystème d'outils.** `clang-format` et `clang-tidy` supportent le Google Style comme preset de première classe. L'adoption est quasi immédiate : un fichier `.clang-format` d'une ligne suffit.

**Battle-tested.** Le guide est utilisé sur l'une des plus grandes bases de code C++ au monde. Ses recommandations ont été éprouvées à une échelle que peu d'organisations atteignent.

**Mise à jour régulière.** Le guide évolue avec le langage. Les fonctionnalités C++17, C++20 et C++23 y sont progressivement intégrées, avec des recommandations claires sur ce qui est autorisé et ce qui est encore trop immature.

**Lisibilité immédiate.** Le système de nommage permet d'identifier la nature d'un symbole sans contexte. Un développeur qui connaît les conventions lit du code Google Style plus vite que du code sans convention.

## Limites et critiques

**Conservatisme sur certaines fonctionnalités.** Le guide a historiquement été en retard sur l'adoption de fonctionnalités modernes du C++. La restriction des exceptions, bien qu'en cours d'assouplissement, a été critiquée comme inadaptée aux projets qui ne partagent pas les contraintes de Google. La prudence envers `auto` est jugée excessive par certains développeurs qui considèrent que le type est souvent non pertinent.

**Indentation à 2 espaces.** C'est un point de friction fréquent pour les équipes habituées à 4 espaces. Le code profondément imbriqué en 2 espaces peut être difficile à lire pour des yeux non entraînés. Ce paramètre est toutefois trivial à changer dans `.clang-format`.

**Spécificité au contexte Google.** Certaines règles reflètent des contraintes propres à Google (monorepo massif, outils internes, base de code legacy) qui ne s'appliquent pas nécessairement à un projet de taille moyenne. Par exemple, la convention des paramètres de sortie par pointeur est moins pertinente dans du code moderne utilisant `std::optional` et les structured bindings.

**`PascalCase` pour les fonctions.** C'est l'une des divergences les plus visibles avec le reste de l'écosystème C++. La bibliothèque standard utilise le `snake_case` (`push_back`, `find_if`), Boost aussi, et la majorité des bibliothèques tierces également. Le code qui suit le Google Style a donc une discontinuité visuelle entre les appels à la STL (`container.push_back()`) et les méthodes du projet (`container.ProcessItems()`). Certaines équipes qui adoptent le Google Style comme base changent cette convention pour le `snake_case` afin de rester cohérentes avec la STL.

**Rigidité.** La complétude du guide est aussi sa contrainte : suivre chaque règle à la lettre peut sembler étouffant dans des contextes plus informels (prototypes, projets personnels, petites équipes). Le guide est conçu pour des équipes de grande taille où la cohérence justifie la rigidité.

## Quand adopter le Google Style

Le Google C++ Style Guide est un excellent choix de base dans les situations suivantes :

- L'équipe veut un standard complet et prêt à l'emploi, sans passer des semaines à rédiger son propre guide.  
- Le projet est de taille moyenne à grande, avec plusieurs développeurs qui doivent lire le code des autres.  
- L'outillage `clang-format` / `clang-tidy` est déjà en place ou prévu.  
- L'équipe est prête à accepter des conventions prescriptives en échange de la cohérence.

Il est moins adapté quand le projet est intimement lié à l'écosystème LLVM (où le LLVM Style est attendu), quand le domaine interdit certaines pratiques que le guide autorise (temps réel strict, embarqué avec contraintes mémoire), ou quand l'équipe a déjà un standard bien établi et fonctionnel.

Dans tous les cas, le guide est conçu pour être un point de départ adaptable, pas un dogme. La section "Background" du guide lui-même encourage les équipes à modifier les règles qui ne s'appliquent pas à leur contexte — à condition de documenter les déviations et de les appliquer uniformément.

---

## Résumé

Le Google C++ Style Guide est le standard de codage le plus complet et le plus largement adopté de l'écosystème C++. Sa philosophie — optimiser pour le lecteur — se traduit par un système de nommage qui encode la nature de chaque symbole, des règles de formatage strictes encodables dans `clang-format`, et des règles d'usage du langage qui privilégient l'explicite et la simplicité. Ses forces sont sa complétude, son écosystème d'outils et son épreuve à grande échelle. Ses limites sont un conservatisme résiduel sur certaines fonctionnalités modernes, une indentation à 2 espaces qui divise, et des conventions de nommage (`PascalCase` pour les fonctions) qui créent une discontinuité avec la bibliothèque standard. C'est un choix de base solide pour la majorité des projets professionnels, à condition d'adapter les quelques points de friction au contexte de l'équipe.
