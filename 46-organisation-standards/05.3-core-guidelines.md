# 46.5.3 — C++ Core Guidelines

> **Chapitre 46 : Organisation et Standards** · Section 46.5 : Standards de codage  
> **Niveau** : Expert · **Prérequis** : Sections 46.5.1 (Google Style), 46.5.2 (LLVM Style), section 45.6 (sécurité mémoire)

---

## Origine et ambition

Les C++ Core Guidelines sont un projet d'une ambition différente des guides de style vus précédemment. Initiées en 2015 par **Bjarne Stroustrup** (créateur du C++) et **Herb Sutter** (président du comité de standardisation ISO C++), elles ne visent pas à fixer un style de formatage ou des conventions de nommage. Leur objectif est plus profond : formuler des **règles de design, de sécurité et de correction** qui permettent d'écrire du C++ moderne, sûr et performant — en éliminant des catégories entières de bugs sans sacrifier les avantages du langage.

Le projet est hébergé sur GitHub (`isocpp/CppCoreGuidelines`), ouvert aux contributions, et activement maintenu. Il compte plus de 400 règles organisées en sections thématiques, chacune identifiée par un code mnémonique (C pour Classes, R pour Resources, F pour Functions, etc.). Le document est vivant : il évolue avec le langage et intègre progressivement les fonctionnalités de C++20, C++23 et C++26.

Ce qui rend les Core Guidelines uniques, c'est leur statut. Ce ne sont pas les conventions d'une entreprise ni d'un projet open source : ce sont les recommandations des architectes du langage eux-mêmes, fondées sur des décennies d'expérience et conçues pour être applicables à tout projet C++, indépendamment du domaine ou de la taille.

## Philosophie

### Rendre le C++ sûr par défaut

La philosophie centrale des Core Guidelines est exprimée dans leur sous-titre : *"make good C++ code the norm"*. Le C++ est un langage puissant mais permissif — il autorise des constructions dangereuses (dangling pointers, buffer overflows, data races) que d'autres langages interdisent structurellement. Les Core Guidelines ne proposent pas de changer le langage, mais de codifier un **sous-ensemble sûr** du C++ qui élimine ces dangers tout en conservant les performances et l'expressivité.

L'idée est qu'en suivant les guidelines, un développeur n'a pas besoin de maîtriser chaque piège du langage — les règles l'en protègent mécaniquement.

### Des règles vérifiables par les outils

Contrairement à de nombreux documents de bonnes pratiques, les Core Guidelines sont conçues dès l'origine pour être **vérifiables automatiquement**. Chaque règle est formulée de manière à pouvoir être détectée par un outil d'analyse statique. `clang-tidy` implémente une partie croissante de ces règles via les checks préfixés par `cppcoreguidelines-*`. Le Microsoft C++ Static Analyzer (dans Visual Studio) implémente également un sous-ensemble des guidelines.

Cette vérifiabilité est fondamentale : elle transforme les guidelines d'un document normatif en un filet de sécurité concret, appliqué à chaque compilation ou à chaque commit.

### Complémentaire, pas concurrent

Les Core Guidelines ne remplacent pas un guide de style comme le Google Style ou le LLVM Style. Elles opèrent à un niveau différent. Un guide de style répond à "comment nommer mes variables ?" et "où placer mes accolades ?". Les Core Guidelines répondent à "comment éviter les fuites mémoire ?", "comment garantir la thread-safety ?", et "comment structurer mes classes pour qu'elles soient robustes face aux erreurs ?".

En pratique, un projet professionnel combine un guide de style (Google, LLVM, ou interne) pour le formatage et le nommage, avec les Core Guidelines pour les règles de design et de sécurité.

## Structure et organisation

Les Core Guidelines sont organisées en sections thématiques, chacune identifiée par une lettre majuscule. Chaque règle porte un identifiant unique (ex : `R.1`, `C.47`, `F.16`) qui permet de la référencer dans les code reviews, les configurations `clang-tidy` et les discussions techniques.

### Sections principales

| Préfixe | Thème | Exemples de règles |
|---|---|---|
| **P** | Philosophy | Principes directeurs (type safety, resource safety) |
| **I** | Interfaces | Conception d'interfaces claires et robustes |
| **F** | Functions | Signatures, paramètres, valeurs de retour |
| **C** | Classes and class hierarchies | Constructeurs, destructeurs, héritage |
| **R** | Resource management | RAII, smart pointers, ownership |
| **ES** | Expressions and statements | Déclarations, conversions, boucles |
| **Per** | Performance | Optimisation, move semantics |
| **CP** | Concurrency and parallelism | Threads, mutex, atomics |
| **E** | Error handling | Exceptions, codes d'erreur, invariants |
| **Con** | Constants and immutability | `const`, `constexpr` |
| **T** | Templates and generic programming | Concepts, spécialisation |
| **SF** | Source files | Headers, includes, linkage |
| **NL** | Naming and layout | Conventions de nommage (section minimaliste) |
| **GSL** | Guidelines Support Library | Types et fonctions d'aide |

La section **NL** (Naming and Layout) est volontairement minimale : les Core Guidelines reconnaissent explicitement que le nommage et le formatage sont des choix d'équipe, et ne cherchent pas à imposer un style. Elles se concentrent sur les règles qui ont un impact sur la **correction** et la **sécurité** du code.

### Anatomie d'une règle

Chaque règle suit un format structuré :

- **Identifiant et titre** : `R.1: Manage resources automatically using resource handles and RAII`  
- **Reason** : justification de la règle  
- **Example** : code illustrant la bonne pratique (et parfois le mauvais pattern)  
- **Note** : précisions et cas particuliers  
- **Enforcement** : comment un outil d'analyse peut vérifier la règle  
- **See also** : liens vers d'autres règles liées

Cette structure rend chaque règle autonome et référençable. Un reviewer peut citer `R.1` dans un commentaire de code review, et l'auteur peut consulter la règle complète avec sa justification et ses exemples.

## Règles fondamentales

Parmi les 400+ règles, certaines constituent le socle sur lequel repose l'ensemble. Elles encodent les principes qui, appliqués systématiquement, éliminent la majorité des bugs C++ classiques.

### Gestion des ressources (section R)

La section R est le cœur des Core Guidelines. Elle formalise le principe RAII (section 6.3) en un ensemble de règles concrètes :

**R.1 : Gérer les ressources automatiquement via RAII.**

```cpp
// ❌ Gestion manuelle — fuite si une exception survient entre new et delete
void process() {
    auto* widget = new Widget();
    // ... code qui peut lancer une exception ...
    delete widget;
}

// ✅ RAII via smart pointer
void process() {
    auto widget = std::make_unique<Widget>();
    // ... la mémoire est libérée automatiquement, même en cas d'exception
}
```

**R.3 : Un raw pointer (`T*`) n'est jamais possédant.**

Cette règle établit une convention universelle : si vous voyez un `T*` dans du code conforme aux Core Guidelines, vous savez que ce pointeur ne possède pas la ressource. La possession est toujours exprimée via `unique_ptr`, `shared_ptr`, ou un autre handle RAII. Cette convention élimine l'ambiguïté qui est la source principale des fuites mémoire et des double-free en C++ classique.

```cpp
// Le contrat est clair par les types :
class Engine {  
public:  
    // Transfert de propriété → unique_ptr
    void setConnection(std::unique_ptr<Connection> conn);

    // Vue non possédante → raw pointer ou référence
    Connection* getConnection() const;
    Widget& getWidget();
};
```

**R.5 : Préférer les objets scoped aux allocations dynamiques.**

```cpp
// ❌ Allocation dynamique inutile
auto* config = new Config();  
config->load("settings.json");  
// ... utilisation ...
delete config;

// ✅ Objet sur la pile
Config config;  
config.load("settings.json");  
// ... destructeur appelé automatiquement à la sortie du scope
```

### Fonctions (section F)

**F.16 : Pour les paramètres d'entrée, passer les types copiables "pas chers" par valeur, les autres par `const&`.**

```cpp
// Types "pas chers" à copier : par valeur
void process(int count);  
void process(double ratio);  
void process(std::string_view name);    // string_view est une vue légère  

// Types "chers" à copier : par const référence
void process(const std::string& name);  
void process(const std::vector<int>& data);  
void process(const Config& config);  
```

**F.21 : Pour retourner des résultats multiples, préférer une struct ou un tuple nommé.**

```cpp
// ❌ Paramètre de sortie
void parse(const std::string& input, int& value, bool& success);

// ✅ Struct nommée
struct ParseResult {
    int value;
    bool success;
};
ParseResult parse(const std::string& input);

// Utilisation avec structured bindings (C++17)
auto [value, success] = parse(input);
```

**F.51 : Quand il y a le choix, préférer les arguments par défaut à la surcharge.**

```cpp
// ❌ Surcharge pour fournir une valeur par défaut
void connect(const std::string& host, int port);  
void connect(const std::string& host);  // Port par défaut  

// ✅ Argument par défaut
void connect(const std::string& host, int port = 8080);
```

### Classes (section C)

**C.2 : Utiliser `class` si la classe a un invariant ; utiliser `struct` si les données sont indépendantes.**

Cette règle clarifie une ambiguïté classique du C++ : `class` et `struct` sont techniquement identiques (seule la visibilité par défaut diffère). Les Core Guidelines leur donnent une sémantique distincte. Une `struct` est un agrégat passif de données. Une `class` maintient un invariant — une propriété que les méthodes garantissent.

```cpp
// struct : données indépendantes, pas d'invariant
struct Point {
    double x;
    double y;
};

// class : invariant maintenu (host et port doivent être cohérents)
class Connection {  
public:  
    Connection(std::string host, int port);
    void send(std::span<const std::byte> data);
private:
    std::string host_;
    int port_;
    Socket socket_;  // Invariant : socket_ est connecté à host_:port_
};
```

**C.47 : Définir et initialiser les membres dans l'ordre de déclaration.**

```cpp
class Server {
    std::string host_;      // Initialisé en premier
    int port_;              // Initialisé en second
    Logger logger_;         // Initialisé en troisième

public:
    // ✅ Liste d'initialisation dans l'ordre de déclaration
    Server(std::string host, int port)
        : host_(std::move(host))
        , port_(port)
        , logger_("server")
    {}
};
```

En C++, les membres sont toujours initialisés dans l'ordre de leur déclaration dans la classe, indépendamment de l'ordre dans la liste d'initialisation. Un ordre discordant entre déclaration et initialisation est un bug silencieux en puissance. GCC et Clang émettent un warning (`-Wreorder`) pour détecter ces incohérences.

**C.131 : Éviter les getters et setters triviaux.**

```cpp
// ❌ Getter/setter qui n'apportent rien par rapport à un membre public
class Point {
    int x_;
    int y_;
public:
    int getX() const { return x_; }
    void setX(int x) { x_ = x; }
    int getY() const { return y_; }
    void setY(int y) { y_ = y; }
};

// ✅ Si pas d'invariant à protéger, utiliser une struct avec membres publics
struct Point {
    int x;
    int y;
};
```

Un getter et un setter triviaux n'ajoutent pas d'encapsulation : ils exposent le membre de la même façon qu'un accès direct, avec plus de verbosité. Les getters/setters se justifient quand ils maintiennent un invariant (validation, notification, lazy initialization).

### Concurrence (section CP)

**CP.1 : Supposer que le code sera exécuté en contexte multi-thread.**

Cette règle encourage la conception défensive : même si le code n'est pas actuellement multi-threadé, le concevoir comme s'il pouvait l'être évite des refontes coûteuses et des bugs subtils lorsque le parallélisme est introduit plus tard.

**CP.20 : Utiliser RAII, jamais un lock/unlock explicite.**

```cpp
// ❌ Lock/unlock manuels — unlock oublié si exception
mutex_.lock();  
data_.push_back(item);  
mutex_.unlock();  

// ✅ RAII via lock_guard
{
    std::lock_guard lock(mutex_);
    data_.push_back(item);
}

// ✅ Ou scoped_lock pour le verrouillage multiple
{
    std::scoped_lock lock(mutex_a_, mutex_b_);
    // Opération atomique sur les deux ressources
}
```

**CP.44 : Nommer les `lock_guard` et `unique_lock`.**

```cpp
// ❌ Le lock est immédiatement détruit (bug silencieux)
std::lock_guard(mutex_);              // Variable temporaire !

// ✅ Variable nommée — le lock vit jusqu'à la fin du scope
std::lock_guard lock(mutex_);
```

Ce bug est l'un des plus insidieux du C++ concurrent : un `lock_guard` non nommé est un objet temporaire détruit à la fin de l'expression. Le mutex est acquis puis immédiatement libéré, sans protéger le code qui suit.

### Gestion d'erreurs (section E)

**E.6 : Utiliser RAII pour prévenir les fuites de ressources.**

C'est le pendant de R.1, appliqué à la gestion d'erreurs : si une exception est lancée entre l'acquisition et la libération d'une ressource, le RAII garantit que la libération a lieu quand même.

**E.25 : Si on ne peut pas lancer d'exceptions, simuler le RAII pour la gestion d'erreurs.**

Pour les contextes sans exceptions (embarqué, certains projets système), les guidelines proposent d'utiliser des types comme `std::expected` (section 12.8) qui portent l'erreur dans la valeur de retour tout en conservant la composabilité :

```cpp
std::expected<Config, Error> loadConfig(const std::filesystem::path& path) {
    auto content = readFile(path);
    if (!content)
        return std::unexpected(content.error());

    auto parsed = parseJson(*content);
    if (!parsed)
        return std::unexpected(parsed.error());

    return Config{*parsed};
}
```

## Les profils : sous-ensembles vérifiables

Les **profils** sont un concept clé des Core Guidelines. Un profil est un sous-ensemble cohérent de règles qui, appliqué intégralement, élimine une catégorie entière de bugs. L'idée est qu'une équipe puisse activer un profil complet plutôt que de picorer des règles individuelles — ce qui garantit une couverture systématique.

### Profil Type Safety

Élimine les bugs liés à l'utilisation incorrecte des types : casts C-style, unions non typées, utilisation de `void*`, conversions narrowing implicites.

Règles clés :
- Interdire les casts C-style (`(int)x`), utiliser les casts C++ explicites.  
- Interdire les `union` non discriminées, utiliser `std::variant`.  
- Ne pas utiliser `void*` sauf pour l'interfaçage C.  
- Préférer les enums aux entiers pour les ensembles de valeurs.

### Profil Bounds Safety

Élimine les accès hors limites (buffer overflows), l'une des catégories de bugs les plus exploitées en termes de sécurité.

Règles clés :
- Ne pas utiliser l'arithmétique de pointeurs. Utiliser `std::span` (section 13.5) pour passer des buffers.  
- Toujours vérifier les indices avant l'accès, ou utiliser `.at()` plutôt que `operator[]` dans le code non critique en performance.  
- Utiliser les algorithmes de la STL plutôt que les boucles manuelles avec des indices.

### Profil Lifetime Safety

Élimine les dangling pointers et les accès à des objets détruits — le type de bug le plus difficile à diagnostiquer en C++.

Règles clés :
- Ne jamais retourner un pointeur ou une référence vers un objet local.  
- Respecter la règle R.3 (raw pointer non possédant).  
- Utiliser `std::string_view` et `std::span` avec conscience de leur nature de vue non possédante.  
- S'assurer qu'un objet référencé survit à toutes ses références.

### Safety Profiles et le contexte réglementaire 2026

Les profils de sécurité des Core Guidelines ont pris une importance nouvelle dans le contexte réglementaire de 2025-2026 (section 45.6). Les recommandations de la NSA, de la CISA et du Cyber Resilience Act européen poussent l'industrie vers des langages "memory-safe". Les profils sont la réponse du comité C++ à cette pression : ils démontrent que le C++ peut atteindre un niveau de sécurité mémoire élevé sans migration vers un autre langage, à condition d'appliquer les règles de façon systématique et vérifiable.

L'état d'avancement de l'outillage de vérification de ces profils est détaillé en section 45.6.2 (Safety Profiles : état de maturité et adoption en 2026).

## La Guidelines Support Library (GSL)

Les Core Guidelines définissent une petite bibliothèque d'aide, la **GSL** (Guidelines Support Library), qui fournit les types nécessaires à l'application de certaines règles. L'implémentation de référence est celle de Microsoft (`microsoft/GSL` sur GitHub).

### Types principaux de la GSL

**`gsl::not_null<T>`** : un wrapper autour d'un pointeur qui garantit statiquement qu'il n'est jamais nul. Il documente et enforce l'intention dans le type lui-même :

```cpp
// Le contrat est dans le type : conn ne peut pas être null
void processConnection(gsl::not_null<Connection*> conn) {
    conn->send(data);  // Pas besoin de vérifier conn != nullptr
}

// L'appel échoue à la compilation ou au runtime si on passe nullptr
processConnection(nullptr);  // Erreur ou assertion
```

**`gsl::span<T>`** : la GSL a introduit `span` bien avant son intégration dans C++20. Pour les projets en C++20 et au-delà, `std::span` (section 13.5) remplace `gsl::span`.

**`gsl::finally`** : un scope guard qui exécute une action à la sortie du scope. Utile pour le code d'interfaçage avec des API C qui ne supportent pas le RAII :

```cpp
FILE* f = fopen("data.bin", "rb");  
if (!f) return Error("Cannot open file");  
auto cleanup = gsl::finally([&] { fclose(f); });  

// ... utilisation de f ...
// fclose est appelé automatiquement à la sortie du scope
```

**`gsl::narrow_cast<T>` et `gsl::narrow<T>`** : conversions numériques explicites. `narrow_cast` est un `static_cast` documentant une conversion narrowing intentionnelle. `narrow` vérifie à l'exécution que la conversion ne perd pas d'information :

```cpp
int64_t big_value = getSomeLargeValue();

// Conversion intentionnelle, le développeur accepte la troncation
auto small = gsl::narrow_cast<int32_t>(big_value);

// Conversion vérifiée — lance gsl::narrowing_error si perte de données
auto safe = gsl::narrow<int32_t>(big_value);
```

### GSL et C++ moderne

Avec l'évolution du langage, plusieurs types de la GSL ont été absorbés par le standard : `std::span` (C++20), `std::byte` (C++17). D'autres, comme `not_null` et `finally`, n'ont pas (encore) d'équivalent standard et conservent leur pertinence. Pour les projets en C++23/C++26, la GSL reste un complément utile mais de taille décroissante.

## Intégration avec `clang-tidy`

Les checks `clang-tidy` préfixés par `cppcoreguidelines-*` vérifient automatiquement un sous-ensemble des Core Guidelines :

```yaml
# .clang-tidy (extrait Core Guidelines)
Checks: >
  cppcoreguidelines-avoid-c-arrays,
  cppcoreguidelines-avoid-goto,
  cppcoreguidelines-avoid-magic-numbers,
  cppcoreguidelines-avoid-non-const-global-variables,
  cppcoreguidelines-c-copy-assignment-signature,
  cppcoreguidelines-init-variables,
  cppcoreguidelines-interfaces-global-init,
  cppcoreguidelines-macro-usage,
  cppcoreguidelines-narrowing-conversions,
  cppcoreguidelines-no-malloc,
  cppcoreguidelines-owning-memory,
  cppcoreguidelines-prefer-member-initializer,
  cppcoreguidelines-pro-bounds-array-to-pointer-decay,
  cppcoreguidelines-pro-bounds-constant-array-index,
  cppcoreguidelines-pro-bounds-pointer-arithmetic,
  cppcoreguidelines-pro-type-const-cast,
  cppcoreguidelines-pro-type-cstyle-cast,
  cppcoreguidelines-pro-type-reinterpret-cast,
  cppcoreguidelines-pro-type-static-cast-downcast,
  cppcoreguidelines-pro-type-union-access,
  cppcoreguidelines-pro-type-vararg,
  cppcoreguidelines-slicing,
  cppcoreguidelines-special-member-functions,
  cppcoreguidelines-virtual-class-destructor
```

Les checks les plus impactants :

- **`cppcoreguidelines-pro-type-cstyle-cast`** : interdit les casts C-style. L'un des vecteurs de bugs les plus courants en C++.  
- **`cppcoreguidelines-no-malloc`** : interdit `malloc`/`free`/`realloc` au profit des mécanismes C++.  
- **`cppcoreguidelines-owning-memory`** : vérifie que les raw pointers ne sont pas utilisés pour la possession de ressources.  
- **`cppcoreguidelines-pro-bounds-pointer-arithmetic`** : signale l'arithmétique de pointeurs, encourage `std::span`.  
- **`cppcoreguidelines-avoid-non-const-global-variables`** : signale les variables globales mutables, source de couplage et de problèmes de thread-safety.  
- **`cppcoreguidelines-special-member-functions`** : vérifie le respect de la Rule of Five (section 6.5).  
- **`cppcoreguidelines-narrowing-conversions`** : détecte les conversions numériques qui peuvent perdre des données.

### Stratégie d'activation progressive

Sur un projet existant, activer tous les checks d'un coup produit des milliers de warnings. La stratégie recommandée est d'activer les checks par profil, en commençant par les plus impactants :

1. **Phase 1** : `pro-type-cstyle-cast`, `no-malloc`, `special-member-functions` — les règles qui attrapent les bugs les plus graves avec le moins de faux positifs.
2. **Phase 2** : `owning-memory`, `pro-bounds-pointer-arithmetic`, `slicing` — règles de sécurité mémoire.
3. **Phase 3** : `avoid-magic-numbers`, `init-variables`, `avoid-non-const-global-variables` — qualité et maintenabilité.
4. **Phase 4** : les checks restants, activés au fur et à mesure que le code existant est mis en conformité.

Chaque phase est accompagnée d'un sprint de remédiation du code existant, suivi de l'activation du check en mode erreur dans le CI pour empêcher les régressions.

## Comparaison avec le Google Style et le LLVM Style

| Dimension | Google Style | LLVM Style | Core Guidelines |
|---|---|---|---|
| **Portée** | Formatage + nommage + usage | Formatage + nommage + usage | Design + sécurité + correction |
| **Prescriptivité** | Très haute | Modérée | Haute (sur le périmètre couvert) |
| **Nommage** | Oui (détaillé) | Oui (détaillé) | Minimal (section NL) |
| **Formatage** | Oui (`clang-format`) | Oui (`clang-format`) | Non (hors périmètre) |
| **Sécurité mémoire** | Quelques règles | Quelques règles | Profils dédiés |
| **Concurrence** | Quelques règles | Quelques règles | Section CP complète |
| **Vérification outil** | `clang-tidy google-*` | `clang-tidy llvm-*` | `clang-tidy cppcoreguidelines-*` |
| **Bibliothèque d'aide** | Non | Types LLVM internes | GSL |
| **Auteurs** | Ingénieurs Google | Communauté LLVM | Stroustrup + Sutter |
| **Public cible** | Équipes Google (et adoptants) | Contributeurs LLVM (et adoptants) | Tout développeur C++ |

Le tableau met en évidence la complémentarité : les Core Guidelines couvrent les dimensions que les guides de style laissent ouvertes. La combinaison la plus courante en production est un guide de style (Google ou LLVM) pour le formatage et le nommage, enrichi des Core Guidelines pour les règles de design et de sécurité.

## Forces des Core Guidelines

**Autorité.** Rédigées par les architectes du langage, elles portent un poids que peu de documents peuvent revendiquer. Citer une Core Guideline dans une code review clôt souvent un débat.

**Couverture de la sécurité.** Les profils Type Safety, Bounds Safety et Lifetime Safety constituent la réponse la plus structurée de l'écosystème C++ aux critiques sur la sécurité mémoire du langage. Aucun autre guide de style n'offre cette couverture.

**Indépendance du contexte.** Les règles sont applicables à tout projet C++ : embarqué, serveur, desktop, jeu vidéo, finance. Elles ne sont pas liées aux contraintes d'une organisation spécifique.

**Intégration outillée.** Le support `clang-tidy` est mature et en croissance continue. L'activation d'un check est une ligne dans `.clang-tidy` — la barrière à l'entrée est minimale.

**Pédagogie.** Chaque règle est justifiée avec des exemples et des contre-exemples. Lire les Core Guidelines est en soi un exercice de formation au C++ moderne. Les sections "Reason" et "Note" transmettent un savoir qui va au-delà de la règle elle-même.

## Limites et critiques

**Volume.** Plus de 400 règles représentent un corpus imposant. Un développeur ne peut pas les mémoriser intégralement. L'approche par profils atténue ce problème, mais l'assimilation complète reste un investissement.

**Granularité inégale.** Certaines règles sont très concrètes et immédiatement applicables (`R.3`, `CP.20`). D'autres sont formulées comme des principes généraux difficiles à vérifier mécaniquement (`P.1: Express ideas directly in code`). Le mélange peut donner l'impression d'un document hétérogène.

**Couverture incomplète par les outils.** `clang-tidy` implémente un sous-ensemble des Core Guidelines. Les règles les plus subtiles (notamment celles du profil Lifetime Safety) ne sont que partiellement vérifiables automatiquement en 2026. L'outillage progresse, mais le delta entre les règles écrites et les règles effectivement vérifiées reste significatif.

**Pas de guide de formatage.** C'est un choix délibéré, mais cela signifie qu'un projet ne peut pas adopter les Core Guidelines comme unique standard. Il faut les combiner avec un guide de formatage, ce qui implique la maintenance de deux documents de référence.

**GSL comme dépendance.** Certaines règles sont plus faciles à appliquer avec la GSL (`not_null`, `narrow`, `finally`). Ajouter une dépendance tierce pour quelques types utilitaires peut sembler excessif pour des projets qui minimisent leurs dépendances. L'absorption progressive de la GSL dans le standard atténue ce point.

---

## Quand et comment adopter les Core Guidelines

### Approche recommandée

L'adoption la plus efficace des Core Guidelines ne consiste pas à lire les 400 règles et à tenter de les appliquer d'un bloc. Elle repose sur une intégration progressive via `clang-tidy` :

1. **Choisir un guide de style** (Google ou LLVM) pour le formatage et le nommage. Le configurer dans `.clang-format`.
2. **Activer les checks `cppcoreguidelines-*`** dans `.clang-tidy`, en commençant par le profil Type Safety (les checks `pro-type-*`).
3. **Remédier le code existant** pour éliminer les violations détectées.
4. **Activer les checks en mode erreur** dans le CI pour empêcher les régressions.
5. **Étendre progressivement** aux profils Bounds Safety et Lifetime Safety.
6. **Former l'équipe** en utilisant les Core Guidelines comme support pédagogique — chaque règle violée est une occasion d'apprentissage.

### Pour les projets nouveaux

Un projet qui démarre en 2026 a le luxe de partir sur de bonnes bases. La configuration recommandée est :

```yaml
# .clang-tidy — projet neuf
Checks: >
  -*,
  cppcoreguidelines-*,
  modernize-*,
  readability-*,
  performance-*,
  bugprone-*,
  google-explicit-constructor,
  google-readability-casting
WarningsAsErrors: '*'
```

Tous les checks Core Guidelines sont activés dès le départ, en mode erreur. Le code est conforme dès la première ligne — il n'y a pas de dette à rattraper.

### Pour les projets existants

L'activation brutale de tous les checks sur une base de code existante est contre-productive (milliers de warnings, équipe découragée). La stratégie progressive décrite plus haut, avec des phases d'activation alignées sur les sprints de remédiation, est la seule approche viable.

---

## Résumé

Les C++ Core Guidelines sont le document de référence pour le design, la sécurité et la correction du code C++ moderne. Rédigées par les architectes du langage, elles formulent plus de 400 règles organisées en sections thématiques et en profils vérifiables. Leur contribution unique est la formalisation des profils de sécurité (Type Safety, Bounds Safety, Lifetime Safety) qui éliminent des catégories entières de bugs — une réponse directe aux critiques sur la sécurité mémoire du C++. Elles sont complémentaires des guides de style comme le Google Style ou le LLVM Style : ceux-ci fixent le formatage et le nommage, les Core Guidelines fixent les règles de design. L'intégration passe par `clang-tidy` et la GSL, avec une adoption progressive par profils. Pour tout projet C++ professionnel en 2026, les Core Guidelines ne sont plus optionnelles — elles sont le socle sur lequel repose la crédibilité du C++ en matière de sécurité et de qualité.
