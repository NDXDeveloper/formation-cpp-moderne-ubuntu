# 46.5.2 — LLVM Style

> **Chapitre 46 : Organisation et Standards** · Section 46.5 : Standards de codage  
> **Niveau** : Expert · **Prérequis** : Section 46.5.1 (Google C++ Style Guide)

---

## Origine et contexte

Les LLVM Coding Standards sont les conventions du projet LLVM — l'infrastructure de compilation qui produit Clang, LLDB, lld, libc++ et une galaxie d'outils associés. Initiées par Chris Lattner au début des années 2000, ces conventions ont façonné l'un des projets C++ open source les plus influents au monde : plus de 10 millions de lignes de code, des centaines de contributeurs actifs, et une base de code qui sert de référence en matière de qualité d'ingénierie.

Le guide est public, hébergé sur `llvm.org/docs/CodingStandards.html`. Contrairement au Google Style qui couvre exhaustivement chaque aspect du code, le LLVM Style est plus concis et plus centré sur les principes. Il assume un niveau de compétence C++ élevé chez ses contributeurs et laisse davantage de place au jugement individuel sur les aspects que le guide ne couvre pas explicitement.

L'influence du LLVM Style dépasse largement le projet LLVM lui-même. De nombreux projets liés aux compilateurs, aux outils d'analyse, et à l'infrastructure système adoptent ces conventions, soit par filiation directe (projets hébergés dans le monorepo LLVM), soit par choix (projets qui veulent s'aligner sur l'écosystème des outils C++).

---

## Philosophie

Le LLVM Style partage avec le Google Style l'objectif de lisibilité, mais sa philosophie se distingue sur plusieurs points.

**Pragmatisme et confiance envers le développeur.** Le guide est moins prescriptif que le Google Style. Là où Google tranche chaque micro-décision, LLVM formule des principes directeurs et fait confiance au jugement du contributeur pour les appliquer. Le guide dit souvent "préférez X" plutôt que "utilisez toujours X".

**Cohérence avec l'écosystème.** Le style LLVM est conçu pour être cohérent au sein d'un monorepo massif qui inclut un compilateur C++, une bibliothèque standard C++, un linker, un debugger et des dizaines d'outils. Les conventions sont choisies pour fonctionner dans ce contexte systémique, où le même développeur navigue entre du code de frontend du compilateur et du code de bibliothèque standard dans la même session.

**Adoption du C++ moderne sans dogmatisme.** Le LLVM Style est historiquement plus ouvert aux fonctionnalités modernes du C++ que le Google Style. Les exceptions sont utilisées dans certains sous-projets, `auto` est encouragé plus librement, et les nouvelles fonctionnalités du standard sont adoptées dès que le support compilateur le permet dans la politique de version minimale du projet.

**Code auto-documenté.** Le guide insiste sur le nommage expressif comme première forme de documentation, avec Doxygen en complément pour l'API publique. La philosophie est qu'un bon nom rend souvent un commentaire superflu.

## Conventions de nommage

Le système de nommage LLVM est l'une des différences les plus visibles avec le Google Style. Il utilise le `CamelCase` (variante `PascalCase`) de façon plus uniforme, avec moins de distinctions entre les catégories de symboles.

### Types : `PascalCase`

Comme le Google Style, les types utilisent le `PascalCase` :

```cpp
class BasicBlock;  
struct MachineInstr;  
enum class Linkage;  
using ValueMap = DenseMap<Value*, Value*>;  
```

### Fonctions et méthodes : `camelCase`

C'est la divergence majeure avec le Google Style. Le LLVM Style utilise le `camelCase` (première lettre minuscule) pour les fonctions et méthodes :

```cpp
class Function {  
public:  
    void addAttribute(Attribute attr);
    bool hasInternalLinkage() const;
    unsigned getNumArgs() const;
    StringRef getName() const;

    static Function *create(FunctionType *ty, Linkage linkage);
};

// Fonctions libres
bool isVirtualRegister(unsigned reg);  
void emitDiagnostic(const DiagnosticInfo &info);  
```

Ce choix rapproche le style des conventions de la bibliothèque standard Java et de nombreuses bibliothèques C++ modernes. Il crée une distinction visuelle claire entre les types (`PascalCase`) et les fonctions/variables (`camelCase` ou `CamelCase` avec minuscule initiale).

### Accesseurs : pattern `get`/`set`

Contrairement au Google Style qui utilise le nom nu pour les getters (`port()`), le LLVM Style utilise explicitement `get` et `set` :

```cpp
class Instruction {  
public:  
    BasicBlock *getParent() const;
    void setParent(BasicBlock *bb);

    unsigned getOpcode() const;
    unsigned getNumOperands() const;

    bool isTerminator() const;    // Prédicats booléens : is/has
    bool hasMetadata() const;
};
```

Les prédicats booléens sont préfixés par `is` ou `has`, ce qui rend les conditions naturellement lisibles :

```cpp
if (inst->isTerminator() && inst->hasMetadata()) {
    // ...
}
```

### Variables : `CamelCase` (initiale majuscule ou minuscule)

C'est un point où le LLVM Style est moins strict que le Google Style. Les variables locales peuvent utiliser le `CamelCase` avec initiale majuscule ou minuscule :

```cpp
void processFunction(Function &F) {
    unsigned NumBlocks = F.size();
    for (BasicBlock &BB : F) {
        unsigned NumInsts = BB.size();
        for (Instruction &I : BB) {
            // ...
        }
    }
}
```

Un pattern distinctif du code LLVM est l'utilisation de noms très courts pour les itérations sur les structures du compilateur : `F` pour une `Function`, `BB` pour un `BasicBlock`, `I` pour une `Instruction`. Cette brièveté est justifiée dans le contexte LLVM où ces types sont omniprésents et leur signification immédiatement évidente pour tout contributeur. Hors de ce contexte, des noms plus longs sont recommandés.

### Membres de données : pas de préfixe ni de suffixe systématique

Le LLVM Style n'impose pas de trailing underscore ni de préfixe `m_` pour les membres de données. Les membres portent simplement un nom en `CamelCase` :

```cpp
class Pass {
    StringRef PassName;
    unsigned PassID;
    bool IsAnalysis;
};
```

Certains sous-projets LLVM adoptent néanmoins des conventions locales (comme un préfixe) pour les membres privés. La règle est la cohérence au sein d'un fichier et d'un module.

### Namespaces : `snake_case` en minuscules

```cpp
namespace llvm {  
namespace codegen {  
namespace detail {  
    // ...
}
}
}
```

### Macros : `UPPER_SNAKE_CASE` avec préfixe

```cpp
#define LLVM_UNLIKELY(expr) __builtin_expect(!!(expr), 0)
#define LLVM_NODISCARD [[nodiscard]]
```

### Récapitulatif

| Catégorie | Style | Exemple |
|---|---|---|
| Classes, structs, enums | `PascalCase` | `BasicBlock` |
| Fonctions, méthodes | `camelCase` | `getParent()` |
| Prédicats booléens | `is`/`has` + `CamelCase` | `isTerminator()` |
| Accesseurs | `get`/`set` + `CamelCase` | `getName()`, `setName()` |
| Variables locales | `CamelCase` | `NumBlocks`, `BB` |
| Membres de données | `CamelCase` | `PassName` |
| Namespaces | `snake_case` | `llvm::codegen` |
| Macros | `UPPER_SNAKE_CASE` | `LLVM_UNLIKELY` |
| Paramètres de template | `PascalCase` | `typename NodeType` |

## Règles de formatage

Le formatage LLVM est encodable via `BasedOnStyle: LLVM` dans `clang-format`. Les caractéristiques principales :

**Indentation : 2 espaces.** Identique au Google Style sur ce point.

**Longueur de ligne : 80 colonnes.** Là aussi identique. Le projet LLVM est strict sur cette limite, notamment parce que les code reviews se font souvent dans des outils qui affichent le code en colonnes étroites.

**Accolades : style K&R attaché** pour les fonctions, les conditions et les boucles :

```cpp
if (condition) {
  doSomething();
} else {
  doSomethingElse();
}

void processBlock(BasicBlock &BB) {
  for (Instruction &I : BB) {
    // ...
  }
}
```

**Pas d'accolades pour les blocs d'une seule instruction** (contrairement au Google Style qui les exige) :

```cpp
// LLVM Style : accolades optionnelles pour une seule instruction
if (condition)
  return nullptr;

for (auto &elem : container)
  process(elem);
```

Cette convention réduit le bruit visuel mais peut introduire des bugs subtils lors de l'ajout d'une seconde instruction. Le projet LLVM l'accepte comme un compromis lisibilité/densité justifié par la rigueur des code reviews.

**Espaces dans les templates :**

```cpp
// LLVM Style
DenseMap<Value *, SmallVector<Instruction *, 4>>

// Note : espace avant * (pointeur attaché au nom, pas au type)
void process(Instruction *I);  
Value *getValue();  
```

**Configuration `.clang-format` :**

```yaml
BasedOnStyle: LLVM
# Ajustements courants :
# IndentWidth: 4
# ColumnLimit: 100
```

## Règles d'usage du langage

Le LLVM Style est moins prescriptif que le Google Style sur les règles d'usage, mais formule des principes clairs.

### `auto` : encouragé quand le type est clair ou non pertinent

Le LLVM Style est plus permissif que le Google Style sur `auto`. Il l'encourage dans les cas où le type est évident, long, ou tout simplement non pertinent pour la compréhension :

```cpp
// ✅ Le type est évident (cast explicite, constructeur, make_)
auto *BB = cast<BasicBlock>(V);  
auto Result = std::make_unique<PassManager>();  

// ✅ Le type est un itérateur ou un type complexe
for (auto I = BB->begin(), E = BB->end(); I != E; ++I) {
    auto *Inst = &*I;
    // ...
}

// ✅ Le type exact n'est pas pertinent
auto Range = llvm::make_range(Begin, End);
```

La philosophie est que le lecteur du code LLVM est un développeur C++ expérimenté qui n'a pas besoin de voir le type de chaque variable pour comprendre le code.

### Assertions et vérifications

Le projet LLVM utilise massivement les assertions pour valider les invariants internes. Le macro `assert()` est omniprésent, et le guide encourage son utilisation systématique :

```cpp
void addOperand(Value *V) {
    assert(V && "Operand must not be null");
    assert(V->getType()->isIntegerTy() && "Expected integer type");
    Operands.push_back(V);
}
```

La convention de toujours passer un message littéral à `assert` est un standard LLVM fort — l'absence de message est considérée comme un oubli.

### RTTI : désactivé dans le cœur de LLVM

Le projet LLVM compile avec `-fno-rtti` pour des raisons de performance (réduction de la taille des binaires). Le guide interdit `dynamic_cast` et `typeid`, et le projet fournit son propre système de cast (`isa<>`, `cast<>`, `dyn_cast<>`) basé sur des identifiants de classe manuels :

```cpp
if (auto *Call = dyn_cast<CallInst>(I)) {
    Function *Callee = Call->getCalledFunction();
    // ...
}

if (isa<ReturnInst>(I))
    return;
```

Cette contrainte est spécifique au projet LLVM et ne s'applique pas aux projets qui adoptent le style LLVM sans ses contraintes techniques. Mais elle illustre une philosophie : les mécanismes du langage peuvent être remplacés par des alternatives plus contrôlées quand le contexte le justifie.

### Exceptions : position variable selon les sous-projets

Le cœur de LLVM (le compilateur, le linker) n'utilise pas les exceptions. La gestion d'erreurs repose sur le type `llvm::Error` et le pattern `Expected<T>` — un précurseur de `std::expected` (section 12.8) qui a d'ailleurs influencé la conception du standard C++23.

```cpp
Expected<std::unique_ptr<Module>> parseModule(StringRef Path) {
    auto Buffer = MemoryBuffer::getFile(Path);
    if (!Buffer)
        return Buffer.takeError();

    return parseBitcodeFile(Buffer->get()->getMemBufferRef());
}

// Utilisation
auto ModuleOrErr = parseModule("input.bc");  
if (!ModuleOrErr) {  
    handleAllErrors(ModuleOrErr.takeError(), [](const ErrorInfoBase &E) {
        errs() << "Error: " << E.message() << "\n";
    });
    return 1;
}
auto &Module = *ModuleOrErr;
```

D'autres sous-projets (notamment libc++ et certains outils) utilisent les exceptions. Le guide ne les interdit pas globalement mais laisse chaque sous-projet décider.

### Smart pointers

Le LLVM Style suit les mêmes principes que le Google Style sur la propriété, avec une nuance : le projet LLVM utilise des raw pointers non possédants de façon extensive pour les références au sein du graphe de compilation (un `Instruction*` pointe vers un objet possédé par son `BasicBlock`). Cette pratique est sûre dans le contexte du compilateur grâce aux invariants de propriété bien documentés, mais ne doit pas être prise comme modèle pour du code applicatif.

Pour le code nouveau et les outils périphériques, `std::unique_ptr` est le choix par défaut pour la possession.

### Headers et includes

Le LLVM Style prescrit un ordre d'inclusion spécifique, similaire au Google Style mais avec une priorité aux headers LLVM internes :

1. Le header correspondant au fichier `.cpp` (header principal)
2. Les headers LLVM locaux (même sous-répertoire)
3. Les headers LLVM des autres sous-projets
4. Les headers de bibliothèques tierces
5. Les headers système et standard C++

Chaque groupe est séparé par une ligne vide, et les headers sont triés alphabétiquement au sein de chaque groupe. `clang-format` avec le style LLVM gère cet ordonnancement automatiquement.

### Commentaires et documentation

Le LLVM Style utilise le format `///` (triple-slash) pour les commentaires Doxygen, conformément au standard LLVM de commentaire :

```cpp
/// Represents a single basic block in the control flow graph.
///
/// A basic block is a sequence of instructions with a single entry point
/// and a single exit point (the terminator instruction).
class BasicBlock {
    /// The parent function that contains this block.
    Function *Parent = nullptr;

    /// The list of instructions in this block.
    InstListType InstList;
};
```

Le guide favorise les commentaires brefs et informatifs plutôt que les blocs de documentation verbeux. La documentation narrative est attendue dans les fichiers `.rst` du répertoire `docs/` du projet.

## Comparaison directe avec le Google Style

| Aspect | Google Style | LLVM Style |
|---|---|---|
| Fonctions/méthodes | `PascalCase` | `camelCase` |
| Accesseurs | `port()` / `set_port()` | `getPort()` / `setPort()` |
| Variables locales | `snake_case` | `CamelCase` |
| Membres de données | `snake_case_` (trailing) | `CamelCase` (pas de marqueur) |
| Constantes | `kPascalCase` | `PascalCase` ou `UPPER_CASE` |
| Indentation | 2 espaces | 2 espaces |
| Limite de ligne | 80 colonnes | 80 colonnes |
| Accolades single-statement | Obligatoires | Optionnelles |
| Commentaires Doxygen | `/** */` ou `///` | `///` (préféré) |
| Préfixe commandes Doxygen | `@` ou `\` | `\` (historique) ou `@` |
| `auto` | Encadré | Encouragé |
| Exceptions | Autorisées (récemment) | Variable selon le sous-projet |
| RTTI | Déconseillé | Interdit (cœur LLVM) |
| Prescriptivité globale | Très élevée | Modérée |

## Forces du LLVM Style

**Nommage naturel des fonctions.** Le `camelCase` pour les fonctions est plus proche des conventions de la majorité des langages (Java, C#, JavaScript, Swift) et plus naturel pour les développeurs polyglottes. La distinction visuelle entre types (`PascalCase`) et fonctions (`camelCase`) est plus marquée qu'en Google Style.

**Souplesse.** Le guide laisse une marge de jugement au développeur, ce qui convient aux équipes expérimentées qui n'ont pas besoin d'une règle pour chaque situation. Cette flexibilité réduit le sentiment d'étouffement que certains développeurs ressentent avec des guides très prescriptifs.

**Proximité avec le tooling C++.** Les développeurs qui contribuent à Clang, LLDB, ou des outils basés sur LibTooling sont déjà immergés dans ce style. L'adopter pour un projet qui interagit avec l'écosystème LLVM minimise la friction.

**Pattern `Expected<T>`.** Le modèle de gestion d'erreurs sans exceptions de LLVM, fondé sur `Expected<T>` et `Error`, a prouvé son efficacité à grande échelle et a influencé `std::expected` en C++23. C'est un pattern mature et bien documenté.

**Accesseurs explicites.** Le pattern `getX()`/`setX()` rend l'accès aux données immédiatement identifiable dans le code. Il n'y a aucune ambiguïté entre un getter et une méthode qui effectue un calcul — là où le Google Style, avec `port()` comme getter, peut parfois créer un doute.

## Limites et critiques

**`CamelCase` pour les variables locales.** Cette convention est inhabituelle dans l'écosystème C++ mainstream, où le `snake_case` domine pour les variables. Elle peut surprendre les développeurs venant d'autres projets et crée une discontinuité avec la bibliothèque standard (`container.push_back(value)` avec `Value` en `CamelCase` et `push_back` en `snake_case`).

**Absence de marqueur pour les membres.** Sans trailing underscore ni préfixe, il est parfois difficile de distinguer un membre de données d'une variable locale à la lecture. Cela repose sur le contexte et la convention de nommage des accesseurs (`getX` implique un membre `X`), ce qui est moins immédiat que le trailing underscore du Google Style.

**Accolades optionnelles.** L'absence d'accolades pour les blocs d'une seule instruction est un vecteur de bugs classiques (le "goto fail" d'Apple est l'exemple le plus célèbre). De nombreuses équipes qui adoptent le LLVM Style ajoutent une déviation locale rendant les accolades obligatoires.

**Moins complet que le Google Style.** Le guide LLVM ne couvre pas autant de cas de figure. Pour les questions non traitées, le développeur doit exercer son jugement ou chercher la réponse dans le code existant — ce qui peut mener à des incohérences entre contributeurs.

**Conventions spécifiques au compilateur.** Certaines pratiques du code LLVM (noms courts `F`, `BB`, `I` ; usage extensif de raw pointers non possédants ; système de cast custom) sont adaptées au contexte d'un compilateur mais ne doivent pas être importées telles quelles dans du code applicatif. Un projet qui adopte le LLVM Style doit distinguer les conventions généralisables des pratiques spécifiques au domaine.

## Quand adopter le LLVM Style

Le LLVM Style est un choix pertinent dans les situations suivantes :

- Le projet est lié à l'écosystème LLVM/Clang : il utilise des API LLVM, il est un plugin Clang, il étend un outil basé sur LLVM.  
- L'équipe préfère un guide moins prescriptif qui laisse de la latitude aux développeurs expérimentés.  
- Le `camelCase` pour les fonctions est jugé plus naturel que le `PascalCase` du Google Style.  
- Le projet est un outil système, un compilateur, ou un analyseur de code, où le style LLVM est la norme de facto.

Il est moins adapté quand l'équipe a besoin d'un guide exhaustif qui tranche chaque question (le Google Style est alors préférable), quand le projet est applicatif plutôt que système, ou quand l'équipe inclut des développeurs juniors qui bénéficieraient de règles plus explicites.

---

## Résumé

Le LLVM Style est un standard de codage pragmatique et modérément prescriptif, forgé par l'un des plus grands projets C++ open source. Ses conventions de nommage distinctives — `camelCase` pour les fonctions, `CamelCase` pour les variables, accesseurs `getX()`/`setX()` — le différencient nettement du Google Style. Son formatage (2 espaces, 80 colonnes, accolades optionnelles pour les single-statements) est encodable dans `clang-format`. Sa philosophie fait davantage confiance au jugement du développeur, ce qui le rend plus adapté aux équipes expérimentées travaillant sur des projets systèmes. Ses limites sont un guide moins exhaustif que le Google Style, des conventions de variables inhabituelles dans l'écosystème C++ mainstream, et des pratiques spécifiques au compilateur qui ne doivent pas être généralisées sans discernement.
