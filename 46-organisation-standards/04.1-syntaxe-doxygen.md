# 46.4.1 — Syntaxe Doxygen

> **Chapitre 46 : Organisation et Standards** · Section 46.4 : Documentation — Doxygen et commentaires  
> **Niveau** : Expert · **Prérequis** : Section 46.4 (introduction Doxygen)

---

## Formats de commentaires Doxygen

Doxygen reconnaît plusieurs styles de commentaires. Le choix du style est une question de convention d'équipe — l'important est la cohérence sur l'ensemble du projet.

### Le style Javadoc (`/** */`)

C'est le format le plus répandu, familier des développeurs venant de Java ou C# :

```cpp
/**
 * Moteur principal du système de traitement.
 *
 * Le moteur gère un pool de workers internes et distribue
 * les tâches entrantes de manière équitable.
 */
class Engine {
    // ...
};
```

Chaque ligne intermédiaire commence par ` *` (espace-étoile). Ce formatage visuel n'est pas obligatoire pour Doxygen — seul le `/**` d'ouverture est nécessaire — mais il est universellement adopté pour la lisibilité.

### Le style triple-slash (`///`)

Adopté notamment par le LLVM Style Guide, ce format utilise trois slashs par ligne :

```cpp
/// Moteur principal du système de traitement.
///
/// Le moteur gère un pool de workers internes et distribue
/// les tâches entrantes de manière équitable.
class Engine {
    // ...
};
```

Ce style a l'avantage de ne pas nécessiter de fermeture (`*/`), ce qui simplifie l'édition et évite les bugs liés à un `*/` oublié. Il est légèrement plus compact pour les commentaires courts.

### Le style Qt (`/*! */`)

Utilisé historiquement dans l'écosystème Qt :

```cpp
/*!
 * Moteur principal du système de traitement.
 */
class Engine {
    // ...
};
```

Fonctionnellement identique au style Javadoc. On le rencontre surtout dans les projets basés sur Qt.

### Commentaires sur une seule ligne

Pour les membres de données, les enums ou les brefs commentaires, Doxygen supporte des annotations courtes :

```cpp
class Config {
    int port_;          ///< Port d'écoute du serveur.
    std::string host_;  ///< Nom d'hôte ou adresse IP.
    bool tls_;          ///< Active le chiffrement TLS.
};
```

Le `///<` (avec le chevron) est un commentaire Doxygen **trailing** : il documente l'élément qui le *précède* sur la même ligne. Sans le `<`, un commentaire `///` documente l'élément qui le *suit*.

L'équivalent en style Javadoc est `/**< ... */` :

```cpp
class Config {
    int port_;          /**< Port d'écoute du serveur. */
    std::string host_;  /**< Nom d'hôte ou adresse IP. */
};
```

### Quel style choisir ?

| Style | Utilisé par | Avantage |
|---|---|---|
| `/** */` | Google, la majorité de l'industrie | Familier, bien supporté par les IDE |
| `///` | LLVM, Clang, de nombreux projets modernes | Compact, pas de balise de fermeture |
| `/*! */` | Qt, KDE | Cohérent dans l'écosystème Qt |

Choisissez un style, documentez-le dans votre guide de contribution, et appliquez-le avec `clang-format` (qui peut normaliser le style des commentaires Doxygen via l'option `CommentPragmas`).

## Préfixe des commandes : `@` vs `\`

Les commandes Doxygen peuvent être préfixées par `@` (style Javadoc) ou `\` (style historique Doxygen). Les deux sont strictement équivalents :

```cpp
/// @param config_path Chemin vers le fichier de configuration.
/// @return true si le démarrage a réussi.

/// \param config_path Chemin vers le fichier de configuration.
/// \return true si le démarrage a réussi.
```

Le style `@` est plus courant dans l'industrie (héritage Java, cohérence avec JSDoc et PHPDoc). Le style `\` est historiquement associé à Doxygen et se retrouve dans certains projets C++ anciens. Là encore, la cohérence prime sur le choix.

Dans toute la suite de cette section, nous utiliserons le préfixe `@`.

## Documenter une classe

La documentation d'une classe couvre son rôle, son modèle d'utilisation, ses garanties (thread-safety, exception safety) et ses invariants :

```cpp
/**
 * @brief Serveur TCP asynchrone basé sur epoll.
 *
 * TcpServer gère les connexions entrantes sur un port donné et dispatche
 * les requêtes vers des handlers enregistrés. Le serveur est single-threaded
 * par défaut mais supporte un mode multi-threaded via set_thread_count().
 *
 * @note Thread-safety : les méthodes start(), stop() et is_running()
 *       sont thread-safe. Les méthodes de configuration (set_port(),
 *       set_thread_count(), register_handler()) ne doivent être appelées
 *       qu'avant start().
 *
 * @see TcpConnection pour la gestion individuelle des connexions.
 * @see HttpServer pour la couche HTTP au-dessus de TcpServer.
 *
 * @since 2.0.0
 */
class TcpServer {  
public:  
    // ...
};
```

### Les commandes clés pour les classes

**`@brief`** — Description courte, affichée dans les listes et les résumés. Si vous n'utilisez pas `@brief`, Doxygen considère la première phrase (jusqu'au premier point) comme le brief. Avec l'option `JAVADOC_AUTOBRIEF = YES` dans le `Doxyfile`, ce comportement est activé automatiquement, rendant `@brief` optionnel :

```cpp
/// Serveur TCP asynchrone basé sur epoll.
///
/// Description détaillée ici...
class TcpServer { /* ... */ };
```

La première ligne devient automatiquement le brief. C'est la pratique recommandée — elle réduit le bruit syntaxique.

**`@note`** — Information importante qui ne fait pas partie de la description principale. Utilisé pour les avertissements de thread-safety, les limitations connues, les conditions spéciales.

**`@see`** — Renvoi vers un autre symbole documenté. Doxygen génère automatiquement un lien hypertexte. Précieux pour guider les utilisateurs vers des classes ou fonctions liées.

**`@since`** — Version dans laquelle la classe a été introduite. Utile pour les bibliothèques avec un cycle de release formalisé.

**`@deprecated`** — Marque un élément comme obsolète et indique l'alternative :

```cpp
/// @deprecated Depuis la version 3.0. Utiliser TcpServerV2 à la place.
class TcpServer { /* ... */ };
```

**`@tparam`** — Documente un paramètre de template (voir section templates plus bas).

## Documenter une fonction

La documentation d'une fonction est le cas le plus fréquent et le plus important. Elle doit permettre à un utilisateur d'appeler la fonction correctement sans lire son implémentation.

### Structure complète

```cpp
/**
 * @brief Envoie un message à tous les clients connectés.
 *
 * Le message est placé dans la file d'envoi de chaque connexion active.
 * L'envoi effectif est asynchrone : cette méthode retourne immédiatement.
 * Les connexions en état d'erreur sont ignorées silencieusement.
 *
 * @param message Contenu du message à envoyer. Ne doit pas être vide.
 *        Les caractères nuls intégrés sont autorisés (le message est
 *        traité comme un buffer binaire, pas comme une chaîne C).
 * @param priority Niveau de priorité (0 = basse, 9 = haute).
 *        Les messages de priorité supérieure sont envoyés en premier.
 *        Par défaut : 0.
 *
 * @return Le nombre de connexions dans lesquelles le message a été
 *         mis en file d'attente. Peut être inférieur au nombre total
 *         de connexions si certaines sont en erreur.
 *
 * @throws std::invalid_argument si @p message est vide.
 * @throws std::runtime_error si le serveur n'est pas démarré.
 *
 * @pre Le serveur doit être dans l'état démarré (is_running() == true).
 * @post Le compteur de messages envoyés est incrémenté.
 *
 * @note Thread-safe. Peut être appelée depuis n'importe quel thread.
 *
 * @code
 * TcpServer server;
 * server.start(8080);
 * auto count = server.broadcast("hello", 5);
 * std::print("Message envoyé à {} clients\n", count);
 * @endcode
 */
std::size_t broadcast(std::string_view message, int priority = 0);
```

### Les commandes essentielles pour les fonctions

**`@param`** — Documente un paramètre. La convention est de décrire ce que le paramètre représente, ses contraintes de validité, et son comportement par défaut si applicable. La référence à un paramètre dans le texte se fait via `@p` :

```cpp
/// @param timeout Durée maximale d'attente.
///        Si @p timeout est nul, la méthode retourne immédiatement.
```

Doxygen supporte aussi l'indication de direction pour les paramètres pointeurs ou références :

```cpp
/// @param[in]     input  Données d'entrée à traiter.
/// @param[out]    result Résultat du traitement.
/// @param[in,out] state  État modifié pendant le traitement.
void process(std::span<const std::byte> input,
             Result& result,
             State& state);
```

Les tags `[in]`, `[out]`, `[in,out]` sont optionnels mais précieux pour les fonctions avec des paramètres de sortie — un pattern courant en C++ système.

**`@return`** (ou `@returns`) — Décrit la valeur de retour. Pour les fonctions retournant `void`, cette commande est omise. Pour les fonctions retournant un booléen, documentez explicitement ce que `true` et `false` signifient :

```cpp
/// @return true si la connexion a été établie, false si le timeout
///         a été atteint sans réponse du serveur.
[[nodiscard]] bool connect(std::string_view host, int port);
```

**`@throws`** (ou `@throw`, ou `@exception`) — Documente chaque exception que la fonction peut lancer. C'est critique en C++ où, contrairement à Java, les exceptions ne sont pas déclarées dans la signature. Pour les fonctions `noexcept`, un `@throws` serait contradictoire — documentez plutôt la garantie via `@note` :

```cpp
/// @note Ne lance jamais d'exception (noexcept).
void shutdown() noexcept;
```

**`@pre` et `@post`** — Préconditions et postconditions. Elles formalisent le contrat de la fonction. Avec l'arrivée des contrats C++26 (section 12.14.1), ces annotations Doxygen trouveront un écho direct dans le code :

```cpp
/// @pre buffer.size() >= min_buffer_size()
/// @post bytes_written() > 0
void write(std::span<const std::byte> buffer);
```

**`@retval`** — Documente des valeurs de retour spécifiques. Utile pour les fonctions retournant des codes d'erreur ou des valeurs sentinelles :

```cpp
/// @retval 0 Succès.
/// @retval -1 Erreur d'accès au fichier.
/// @retval -2 Format de fichier invalide.
int load_config(const char* path);
```

## Documenter les templates

Les paramètres de template se documentent avec `@tparam` :

```cpp
/**
 * @brief Applique une transformation à chaque élément d'un conteneur.
 *
 * @tparam Container Type de conteneur. Doit satisfaire le concept
 *         std::ranges::input_range.
 * @tparam UnaryFunc Fonction unaire applicable aux éléments du conteneur.
 *         Doit être invocable avec Container::value_type.
 *
 * @param container Conteneur source (non modifié).
 * @param func Transformation à appliquer.
 *
 * @return Un std::vector contenant les éléments transformés.
 */
template <std::ranges::input_range Container, typename UnaryFunc>  
auto transform_all(const Container& container, UnaryFunc func);  
```

Pour les templates avec des concepts C++20 (section 16.6), la documentation `@tparam` complète les contraintes exprimées dans le code. Le concept indique *ce qui est requis* au compilateur ; la documentation explique *pourquoi* c'est requis et *ce que cela signifie* pour l'utilisateur.

## Documenter les enums

Les enums, en particulier les `enum class` de C++ moderne, se documentent au niveau du type et de chaque valeur :

```cpp
/**
 * @brief État du cycle de vie d'une connexion.
 */
enum class ConnectionState {
    Idle,          ///< En attente, aucune opération en cours.
    Connecting,    ///< Handshake TCP en cours.
    Connected,     ///< Connexion établie, prête à envoyer/recevoir.
    Draining,      ///< En cours de vidage du buffer d'envoi.
    Closing,       ///< Fermeture initiée, en attente du FIN distant.
    Closed         ///< Connexion fermée, socket libéré.
};
```

Le style `///<` (commentaire trailing) est particulièrement adapté aux enums : chaque valeur est documentée sur la même ligne, ce qui reste compact et lisible.

## Documenter les membres de données

Pour les classes avec des membres publics (structures, PODs, configurations), chaque champ mérite une annotation :

```cpp
/**
 * @brief Configuration du pool de connexions.
 */
struct PoolConfig {
    /// Nombre maximum de connexions simultanées.
    /// @note Doit être compris entre 1 et 10000.
    int max_connections = 100;

    /// Durée d'inactivité avant fermeture automatique d'une connexion.
    std::chrono::seconds idle_timeout{300};

    /// Active la réutilisation des connexions fermées.
    bool enable_recycling = true;

    /// Intervalle entre deux tentatives de health check.
    /// Mis à zéro pour désactiver les health checks.
    std::chrono::milliseconds health_check_interval{5000};
};
```

Pour les membres `private`, la documentation est optionnelle dans les headers publics — elle relève de l'implémentation interne. Si vous utilisez l'idiome Pimpl (section 46.2), les membres privés ne sont pas du tout visibles dans le header.

## Groupes et modules

Sur un projet de taille importante, la documentation plate (une longue liste de classes) devient difficile à naviguer. Doxygen permet d'organiser les symboles en **groupes** logiques via les commandes `@defgroup`, `@ingroup` et `@addtogroup`.

### Définition d'un groupe

```cpp
/**
 * @defgroup networking Networking
 * @brief Composants réseau du projet.
 *
 * Ce module contient les classes et fonctions liées à la communication
 * réseau : serveurs TCP/UDP, gestion des connexions, et protocoles
 * de niveau application.
 * @{
 */

class TcpServer { /* ... */ };  
class UdpSocket { /* ... */ };  
class Connection { /* ... */ };  

/// @}
```

Le bloc `@{` ... `@}` englobe les symboles qui font partie du groupe. Doxygen génère une page dédiée pour chaque groupe, avec la liste de ses membres et la documentation d'introduction.

### Ajout à un groupe existant

Depuis un autre fichier, on peut ajouter des symboles à un groupe déjà défini :

```cpp
/**
 * @addtogroup networking
 * @{
 */

class HttpServer { /* ... */ };  
class WebSocketHandler { /* ... */ };  

/// @}
```

### Appartenance via `@ingroup`

Pour ajouter un symbole isolé sans ouvrir un bloc :

```cpp
/**
 * @brief Résout un nom d'hôte en adresse IP.
 * @ingroup networking
 */
std::string resolve_hostname(std::string_view host);
```

Les groupes sont le principal outil de structuration de la documentation générée. Ils transforment une référence d'API monolithique en documentation modulaire navigable.

## Blocs de code et exemples

Les exemples de code sont un outil de documentation puissant. Un exemple concret vaut souvent mieux qu'un paragraphe de prose.

### `@code` / `@endcode`

```cpp
/**
 * @brief Parse un fichier JSON et retourne la configuration.
 *
 * @code
 * auto cfg = parse_config("/etc/myapp/config.json");
 * std::print("Port: {}\n", cfg.port);
 * std::print("Host: {}\n", cfg.host);
 * @endcode
 *
 * @param path Chemin vers le fichier JSON.
 * @return La configuration parsée.
 * @throws std::runtime_error si le fichier est introuvable ou malformé.
 */
Config parse_config(const std::filesystem::path& path);
```

Doxygen applique la coloration syntaxique au contenu des blocs `@code`. Le langage est détecté automatiquement, mais peut être forcé :

```cpp
/// @code{.cpp}
/// auto x = compute(42);
/// @endcode
///
/// @code{.sh}
/// $ ./mon-app --config /etc/config.json
/// @endcode
```

### `@snippet` — Inclusion depuis un fichier source

Pour éviter que les exemples de documentation ne deviennent obsolètes, `@snippet` inclut du code directement depuis un fichier source réel — typiquement un fichier de test ou un fichier d'exemple dédié :

```cpp
/**
 * @brief Crée un pool de threads.
 *
 * Exemple d'utilisation :
 * @snippet tests/core/thread_pool_test.cpp create_pool_example
 */
class ThreadPool { /* ... */ };
```

Dans le fichier source référencé, les bornes du snippet sont marquées par des commentaires :

```cpp
// tests/core/thread_pool_test.cpp

TEST(ThreadPool, BasicUsage) {
    //! [create_pool_example]
    ThreadPool pool(4);  // 4 threads
    auto future = pool.submit([] {
        return compute_heavy_task();
    });
    auto result = future.get();
    //! [create_pool_example]

    EXPECT_GT(result, 0);
}
```

L'avantage majeur : le snippet est compilé et testé dans le cadre des tests unitaires. S'il devient invalide suite à un changement d'API, le test échoue *avant* que la documentation ne devienne incorrecte.

## Pages et documentation narrative

Au-delà de la référence d'API, Doxygen peut générer des pages de documentation narrative — tutoriels, guides de démarrage, architecture — via la commande `@page` et les fichiers Markdown.

### Pages personnalisées

```cpp
/**
 * @page getting_started Guide de démarrage
 *
 * @section gs_install Installation
 *
 * Installez la bibliothèque via Conan :
 * @code{.sh}
 * conan install . --build=missing
 * @endcode
 *
 * @section gs_first_steps Premiers pas
 *
 * Créez un moteur et démarrez le traitement :
 * @code{.cpp}
 * #include <monprojet/core/engine.h>
 *
 * int main() {
 *     monprojet::core::Engine engine("config.json");
 *     engine.start();
 *     // ...
 *     engine.stop();
 * }
 * @endcode
 *
 * @see Engine pour la référence complète de l'API.
 */
```

### Fichiers Markdown comme pages

Depuis Doxygen 1.8, les fichiers `.md` placés dans le répertoire `docs/` (ou tout répertoire listé dans `INPUT` du `Doxyfile`) sont automatiquement intégrés comme pages. C'est l'approche la plus naturelle pour la documentation narrative :

```markdown
# Architecture du système {#architecture}

## Vue d'ensemble

Le système est composé de trois couches...

## Diagramme

Voir la classe @ref monprojet::core::Engine pour le composant central.
```

La syntaxe `@ref` crée un lien vers un symbole documenté. Le label `{#architecture}` définit un identifiant que d'autres pages peuvent référencer via `@ref architecture`.

### La page principale (`@mainpage`)

La page d'accueil de la documentation générée est définie par `@mainpage` :

```cpp
/**
 * @mainpage MonProjet — Documentation technique
 *
 * @section intro Introduction
 *
 * MonProjet est un framework de traitement de données en temps réel.
 *
 * @section modules Modules
 *
 * - @ref networking "Networking" — Communication réseau
 * - @ref core "Core" — Moteur de traitement
 * - @ref config "Configuration" — Gestion de la configuration
 *
 * @section quickstart Démarrage rapide
 *
 * Consultez le @ref getting_started "guide de démarrage".
 */
```

En pratique, beaucoup de projets préfèrent utiliser un fichier `docs/mainpage.md` pour cette page, plutôt qu'un commentaire dans un fichier source.

## Commandes de mise en forme

Doxygen supporte une mise en forme riche dans les commentaires :

### Texte et structure

| Commande | Effet |
|---|---|
| `@brief` | Description courte |
| `@details` | Description détaillée (implicite après le brief) |
| `@note` | Note mise en avant visuellement |
| `@warning` | Avertissement (affiché en jaune/rouge) |
| `@attention` | Similaire à `@warning`, emphase forte |
| `@todo` | Tâche à faire (Doxygen génère une page listant tous les TODO) |
| `@bug` | Bug connu (Doxygen génère une page listant tous les bugs) |

### Références croisées

| Commande | Effet |
|---|---|
| `@ref symbol` | Lien vers un symbole ou une page |
| `@see symbol` | Section "Voir aussi" |
| `@p name` | Met en forme un nom de paramètre en police code |
| `@c text` | Met un mot en police code (inline) |
| `@a text` | Met un mot en italique (emphase) |

### Formatage Markdown

Doxygen supporte la syntaxe Markdown standard dans les commentaires : `**gras**`, `*italique*`, `` `code inline` ``, listes avec `-` ou `*`, et liens `[texte](url)`. C'est souvent plus lisible que les commandes Doxygen équivalentes pour le formatage courant :

```cpp
/// Analyse le buffer d'entrée et retourne les tokens identifiés.
///
/// Le parser supporte trois modes :
/// - **strict** : rejette toute entrée non conforme
/// - **lenient** : tente de récupérer les erreurs mineures
/// - **raw** : aucune validation, retourne les tokens bruts
///
/// Pour plus de détails, voir la [spécification du format](docs/format.md).
```

## Exemple complet : un header documenté

Pour synthétiser les conventions présentées, voici un header complet documenté selon les bonnes pratiques :

```cpp
/// @file connection_pool.h
/// @brief Pool de connexions réutilisables.
/// @ingroup networking

#pragma once

#include <monprojet/network/connection.h>
#include <chrono>
#include <cstddef>
#include <functional>
#include <optional>

namespace monprojet::network {

/**
 * Pool de connexions TCP réutilisables.
 *
 * Le pool maintient un ensemble de connexions ouvertes vers un
 * serveur distant et les distribue aux appelants via acquire().
 * Les connexions rendues via release() sont recyclées pour les
 * requêtes suivantes, évitant le coût du handshake TCP.
 *
 * @note Thread-safe. Toutes les méthodes publiques peuvent être
 *       appelées depuis n'importe quel thread.
 *
 * @warning Le pool doit être détruit après tous les objets
 *          Connection qu'il a distribués.
 *
 * @see Connection
 * @since 1.4.0
 */
class ConnectionPool {  
public:  
    /// Callback invoqué quand une connexion est recyclée.
    using RecycleCallback = std::function<void(Connection&)>;

    /**
     * Construit un pool vers le serveur spécifié.
     *
     * @param host Adresse du serveur (nom ou IP).
     * @param port Port TCP du serveur (1–65535).
     * @param max_size Nombre maximum de connexions dans le pool.
     *
     * @throws std::invalid_argument si @p host est vide ou
     *         @p port est hors limites.
     */
    ConnectionPool(std::string host, int port, std::size_t max_size = 10);

    /// Destructeur. Ferme toutes les connexions ouvertes.
    ~ConnectionPool();

    /// @name Gestion des connexions
    /// @{

    /**
     * Acquiert une connexion depuis le pool.
     *
     * Si une connexion recyclée est disponible, elle est retournée
     * immédiatement. Sinon, une nouvelle connexion est créée si le
     * pool n'a pas atteint sa taille maximale.
     *
     * @param timeout Durée maximale d'attente si le pool est plein.
     *
     * @return Une connexion prête à l'emploi, ou std::nullopt si
     *         le timeout a été atteint.
     *
     * @pre Le pool n'a pas été shutdown().
     */
    [[nodiscard]]
    std::optional<Connection> acquire(
        std::chrono::milliseconds timeout = std::chrono::seconds(5));

    /**
     * Rend une connexion au pool pour réutilisation.
     *
     * @param conn Connexion à rendre. Doit provenir de ce pool.
     *        La connexion est invalidée après l'appel.
     */
    void release(Connection conn);

    /// @}

    /// @name Configuration
    /// @{

    /**
     * Enregistre un callback appelé à chaque recyclage.
     *
     * @param cb Fonction appelée avec la connexion recyclée.
     *        Passer nullptr pour supprimer le callback.
     */
    void set_recycle_callback(RecycleCallback cb);

    /// @}

    /// @name Statistiques
    /// @{

    /// Nombre de connexions actuellement dans le pool (libres).
    [[nodiscard]] std::size_t available() const;

    /// Nombre total de connexions créées depuis l'instanciation.
    [[nodiscard]] std::size_t total_created() const;

    /// @}

private:
    struct Impl;
    std::unique_ptr<Impl> impl_;
};

} // namespace monprojet::network
```

Les points à noter : le `@file` en tête identifie le fichier dans la documentation générée. Le `@ingroup` rattache tout le fichier au groupe `networking`. Les méthodes sont organisées en sections nommées via `@name` / `@{` / `@}`, ce qui produit des sous-sections dans la documentation générée. Chaque méthode publique est documentée avec ses paramètres, sa valeur de retour, ses exceptions et ses préconditions. Les getters triviaux (`available()`, `total_created()`) ont un brief sur une seule ligne — pas de sur-documentation.

## Résumé

La syntaxe Doxygen repose sur des commentaires structurés (styles `/** */`, `///` ou `/*! */`) associés à des commandes préfixées par `@` ou `\`. Les commandes essentielles sont `@brief`, `@param`, `@return`, `@throws`, `@pre`/`@post` pour les fonctions, `@tparam` pour les templates, `@note`/`@warning` pour les annotations transversales, et `@see`/`@ref` pour les liens croisés. Les groupes (`@defgroup`, `@ingroup`) structurent la documentation à l'échelle du projet. Les blocs `@code` et surtout `@snippet` intègrent des exemples vivants. Enfin, les pages (`@page`, `@mainpage`) et le support Markdown permettent de mêler documentation narrative et référence d'API dans un ensemble cohérent et navigable.
