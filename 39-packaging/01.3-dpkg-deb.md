# 39.1.3 — Construction avec dpkg-deb

## De l'arborescence au fichier `.deb` installable

---

## Introduction

Les deux sections précédentes ont posé les fondations : la structure d'un paquet DEB (39.1.1) et les scripts de contrôle qui automatisent son cycle de vie (39.1.2). Cette section met tout en œuvre. À partir d'un projet C++ compilé avec CMake, nous allons construire un paquet `.deb` fonctionnel, le vérifier, l'installer, et le désinstaller proprement.

L'objectif est de produire un workflow complet et reproductible — le genre de script que vous intégrerez tel quel dans votre pipeline CI/CD.

---

## Le projet exemple

Pour ancrer la démonstration dans le concret, nous travaillons avec un projet C++ fictif mais réaliste : un outil CLI de monitoring système appelé `syswatch`. Le projet a la structure CMake suivante :

```
syswatch/
├── CMakeLists.txt
├── src/
│   └── main.cpp
├── include/
│   └── syswatch/
│       └── collector.h
├── config/
│   └── syswatch.yaml
└── systemd/
    └── syswatch.service
```

Le `CMakeLists.txt` déclare les cibles d'installation :

```cmake
cmake_minimum_required(VERSION 3.25)  
project(syswatch VERSION 1.2.0 LANGUAGES CXX)  

set(CMAKE_CXX_STANDARD 23)

add_executable(syswatch src/main.cpp)  
target_link_libraries(syswatch PRIVATE ssl crypto)  

# Règles d'installation
install(TARGETS syswatch RUNTIME DESTINATION bin)  
install(FILES config/syswatch.yaml  
        DESTINATION /etc/syswatch
        COMPONENT config)
install(FILES systemd/syswatch.service
        DESTINATION lib/systemd/system)
```

Compilons le projet avec CMake et Ninja :

```bash
cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Release  
cmake --build build  
```

## Étape 1 : Installation staging avec DESTDIR

La première étape du packaging consiste à installer les fichiers dans un répertoire temporaire qui deviendra la zone de données du paquet. La variable `DESTDIR` redirige l'installation sans toucher au système réel :

```bash
DESTDIR="$(pwd)/pkg-root" cmake --install build --prefix /usr
```

Vérifions le résultat :

```bash
$ find pkg-root -type f
pkg-root/usr/bin/syswatch  
pkg-root/etc/syswatch/syswatch.yaml  
pkg-root/usr/lib/systemd/system/syswatch.service  
```

L'arborescence sous `pkg-root/` reflète exactement ce qui sera déposé sur le système cible. Le binaire est dans `/usr/bin/`, la configuration dans `/etc/syswatch/`, le fichier unit dans `/usr/lib/systemd/system/`.

> **Note** — Le `--prefix /usr` est important. Sans lui, CMake utilise le préfixe par défaut `/usr/local`, et votre binaire serait installé dans `/usr/local/bin/` — un emplacement inapproprié pour un paquet DEB, réservé par convention aux installations manuelles.

## Étape 2 : Création du répertoire DEBIAN/

Le répertoire `DEBIAN/` contient les métadonnées et les scripts. Créons-le et peuplons-le :

```bash
mkdir -p pkg-root/DEBIAN
```

### Le fichier control

```bash
cat > pkg-root/DEBIAN/control << 'EOF'  
Package: syswatch  
Version: 1.2.0-1  
Section: utils  
Priority: optional  
Architecture: amd64  
Maintainer: Equipe SRE <sre@exemple.com>  
Description: Outil de monitoring systeme haute performance  
 Syswatch collecte des metriques CPU, memoire et disque
 en temps reel et les expose via un endpoint Prometheus.
 .
 Concu pour les environnements de production Ubuntu.
Homepage: https://github.com/exemple/syswatch  
EOF  
```

Le champ `Depends` est manquant pour l'instant — nous allons le calculer automatiquement.

### Calcul automatique des dépendances

`dpkg-shlibdeps` analyse le binaire et détermine les paquets nécessaires à l'exécution :

```bash
$ dpkg-shlibdeps -O pkg-root/usr/bin/syswatch
shlibs:Depends=libc6 (>= 2.34), libgcc-s1 (>= 3.0), libssl3 (>= 3.0.0), libstdc++6 (>= 12)
```

Injectons ce résultat dans le fichier `control`. En production, un script automatiserait cette étape. Ici, nous ajoutons la ligne manuellement :

```bash
DEPS=$(dpkg-shlibdeps -O pkg-root/usr/bin/syswatch 2>/dev/null \
       | sed 's/^shlibs:Depends=//')

# Insérer le champ Depends après Architecture
sed -i "/^Architecture:/a Depends: ${DEPS}" pkg-root/DEBIAN/control
```

Vérifions que le fichier `control` est complet :

```bash
$ cat pkg-root/DEBIAN/control
Package: syswatch  
Version: 1.2.0-1  
Section: utils  
Priority: optional  
Architecture: amd64  
Depends: libc6 (>= 2.34), libgcc-s1 (>= 3.0), libssl3 (>= 3.0.0), libstdc++6 (>= 12)  
Maintainer: Equipe SRE <sre@exemple.com>  
Description: Outil de monitoring systeme haute performance  
 Syswatch collecte des metriques CPU, memoire et disque
 en temps reel et les expose via un endpoint Prometheus.
 .
 Concu pour les environnements de production Ubuntu.
Homepage: https://github.com/exemple/syswatch
```

### Calcul de la taille installée

Le champ `Installed-Size` indique l'espace disque en kilooctets une fois installé. Calculons-le :

```bash
SIZE=$(du -sk pkg-root --exclude=DEBIAN | awk '{print $1}')  
sed -i "/^Maintainer:/i Installed-Size: ${SIZE}" pkg-root/DEBIAN/control  
```

### Le fichier conffiles

Le fichier de configuration YAML doit être protégé lors des mises à jour :

```bash
cat > pkg-root/DEBIAN/conffiles << 'EOF'
/etc/syswatch/syswatch.yaml
EOF
```

### Les scripts de maintien

Créons un `postinst` et un `postrm` comme vus dans la section 39.1.2 :

```bash
cat > pkg-root/DEBIAN/postinst << 'SCRIPT'
#!/bin/sh
set -e

case "$1" in
    configure)
        # Créer l'utilisateur système si absent
        if ! getent passwd syswatch > /dev/null 2>&1; then
            adduser --system --group --no-create-home \
                    --home /nonexistent \
                    --shell /usr/sbin/nologin \
                    syswatch
        fi

        # Répertoires de données
        install -d -o syswatch -g syswatch -m 0750 /var/lib/syswatch
        install -d -o syswatch -g syswatch -m 0750 /var/log/syswatch

        # Activer et démarrer le service
        if [ -d /run/systemd/system ]; then
            systemctl daemon-reload
            systemctl enable syswatch.service
            systemctl start syswatch.service
        fi
        ;;
esac  
SCRIPT  

cat > pkg-root/DEBIAN/postrm << 'SCRIPT'
#!/bin/sh
set -e

case "$1" in
    remove)
        if [ -d /run/systemd/system ]; then
            systemctl daemon-reload
        fi
        ;;
    purge)
        rm -rf /var/lib/syswatch
        rm -rf /var/log/syswatch
        rm -rf /etc/syswatch

        if getent passwd syswatch > /dev/null 2>&1; then
            deluser --system syswatch 2>/dev/null || true
        fi
        if getent group syswatch > /dev/null 2>&1; then
            delgroup --system syswatch 2>/dev/null || true
        fi

        if [ -d /run/systemd/system ]; then
            systemctl daemon-reload
        fi
        ;;
esac  
SCRIPT  
```

### Permissions

Les scripts doivent être exécutables, et tous les fichiers doivent appartenir à `root:root` :

```bash
chmod 0755 pkg-root/DEBIAN/postinst pkg-root/DEBIAN/postrm
```

## Étape 3 : Stripping du binaire

En mode `Release`, le compilateur optimise le code mais conserve par défaut les symboles de débogage. Pour un paquet de production, ces symboles alourdissent le binaire inutilement. Le stripping les supprime :

```bash
$ ls -lh pkg-root/usr/bin/syswatch
-rwxr-xr-x 1 user user 2.4M ... syswatch

$ strip --strip-unneeded pkg-root/usr/bin/syswatch

$ ls -lh pkg-root/usr/bin/syswatch
-rwxr-xr-x 1 user user 420K ... syswatch
```

La réduction est souvent spectaculaire — un facteur 3 à 10 est courant. L'option `--strip-unneeded` conserve les symboles nécessaires au fonctionnement du binaire (notamment pour les librairies partagées) tout en supprimant les informations de débogage.

Si vous avez besoin des symboles pour le débogage post-mortem, la pratique Debian est de les extraire dans un paquet séparé (`syswatch-dbgsym`) avec `objcopy` :

```bash
objcopy --only-keep-debug pkg-root/usr/bin/syswatch syswatch.debug  
strip --strip-unneeded pkg-root/usr/bin/syswatch  
objcopy --add-gnu-debuglink=syswatch.debug pkg-root/usr/bin/syswatch  
```

Le fichier `syswatch.debug` est alors distribué dans un paquet `-dbgsym` séparé. GDB sait automatiquement le charger quand il est installé. Pour un paquet interne, cette séparation est optionnelle mais utile si vous analysez des core dumps en production (voir chapitre 29).

## Étape 4 : Construction du paquet

Tout est en place. Un coup d'œil final à l'arborescence complète :

```bash
$ find pkg-root | head -20
pkg-root/  
pkg-root/DEBIAN/  
pkg-root/DEBIAN/control  
pkg-root/DEBIAN/conffiles  
pkg-root/DEBIAN/postinst  
pkg-root/DEBIAN/postrm  
pkg-root/usr/  
pkg-root/usr/bin/  
pkg-root/usr/bin/syswatch  
pkg-root/usr/lib/  
pkg-root/usr/lib/systemd/  
pkg-root/usr/lib/systemd/system/  
pkg-root/usr/lib/systemd/system/syswatch.service  
pkg-root/etc/  
pkg-root/etc/syswatch/  
pkg-root/etc/syswatch/syswatch.yaml  
```

La construction elle-même tient en une commande :

```bash
fakeroot dpkg-deb --build pkg-root syswatch_1.2.0-1_amd64.deb
```

Le premier argument est le répertoire source, le second est le nom du fichier `.deb` de sortie. `fakeroot` garantit que les fichiers dans l'archive appartiendront à `root:root` même si vous construisez en tant qu'utilisateur normal.

Pour contrôler la compression, `dpkg-deb` accepte les options `-Z` (algorithme) et `-z` (niveau) :

```bash
# Compression zstd (défaut sur Ubuntu 23.04+)
fakeroot dpkg-deb -Zzstd -z9 --build pkg-root syswatch_1.2.0-1_amd64.deb

# Compression xz (meilleur ratio, plus lent — pour la distribution)
fakeroot dpkg-deb -Zxz --build pkg-root syswatch_1.2.0-1_amd64.deb
```

Le choix de la compression dépend du contexte. `zstd` offre une décompression très rapide — avantageuse pour les déploiements CI où le paquet est reconstruit et installé fréquemment. `xz` offre un meilleur ratio de compression — préférable pour les paquets distribués via un réseau lent.

## Étape 5 : Vérification

### Inspection du contenu

Avant d'installer, vérifions que le paquet contient ce qu'on attend :

```bash
# Métadonnées
$ dpkg-deb --info syswatch_1.2.0-1_amd64.deb
 new Debian package, version 2.0.
 size 184320 bytes: control archive=1024 bytes.
     452 bytes,    12 lines      control
      34 bytes,     1 lines      conffiles
     487 bytes,    19 lines   *  postinst             #!/bin/sh
     382 bytes,    22 lines   *  postrm               #!/bin/sh
 Package: syswatch
 Version: 1.2.0-1
 ...

# Liste des fichiers
$ dpkg-deb --contents syswatch_1.2.0-1_amd64.deb
drwxr-xr-x root/root       0  2026-03-18 10:00 ./usr/  
drwxr-xr-x root/root       0  2026-03-18 10:00 ./usr/bin/  
-rwxr-xr-x root/root  430080  2026-03-18 10:00 ./usr/bin/syswatch
drwxr-xr-x root/root       0  2026-03-18 10:00 ./etc/  
drwxr-xr-x root/root       0  2026-03-18 10:00 ./etc/syswatch/  
-rw-r--r-- root/root     256  2026-03-18 10:00 ./etc/syswatch/syswatch.yaml
...
```

Vérifiez trois choses dans cette sortie. Premièrement, les permissions : les répertoires doivent être en `0755`, les exécutables en `0755`, les fichiers réguliers en `0644`. Deuxièmement, le propriétaire : tout doit être `root/root`. Troisièmement, les scripts sont bien marqués avec un astérisque (`*`) dans la sortie de `--info`, confirmant qu'ils sont exécutables.

### Analyse avec lintian

```bash
$ lintian syswatch_1.2.0-1_amd64.deb
W: syswatch: no-manual-page usr/bin/syswatch  
W: syswatch: no-changelog usr/share/doc/syswatch/changelog.Debian.gz  
W: syswatch: no-copyright-file usr/share/doc/syswatch/copyright  
```

Ces trois warnings sont classiques pour un paquet interne. Le premier signale l'absence de page man pour le binaire. Le deuxième et le troisième signalent l'absence de changelog et de fichier de licence dans `/usr/share/doc/`. Pour un paquet distribué publiquement, il faudrait les corriger. Pour un déploiement interne, ces warnings sont documentés et acceptés.

Ajoutons tout de même un fichier `copyright` minimal pour supprimer le warning le plus important :

```bash
mkdir -p pkg-root/usr/share/doc/syswatch

cat > pkg-root/usr/share/doc/syswatch/copyright << 'EOF'  
Format: https://www.debian.org/doc/packaging-manuals/copyright-format/1.0/  
Upstream-Name: syswatch  
Upstream-Contact: sre@exemple.com  

Files: *  
Copyright: 2026 Exemple SAS  
License: Proprietary  
 Usage interne uniquement. Tous droits reserves.
EOF
```

## Étape 6 : Test d'installation et de désinstallation

Le test final doit se faire sur un environnement propre — idéalement un conteneur Docker ou une VM jetable, jamais sur votre machine de développement en première intention.

### Installation

```bash
$ sudo dpkg -i syswatch_1.2.0-1_amd64.deb
Selecting previously unselected package syswatch.
(Reading database ... 85432 files and directories currently installed.)
Preparing to unpack syswatch_1.2.0-1_amd64.deb ...  
Unpacking syswatch (1.2.0-1) ...  
Setting up syswatch (1.2.0-1) ...  
Adding system user 'syswatch' (UID 114) ...  
Created symlink /etc/systemd/system/multi-user.target.wants/syswatch.service → ...  
```

Si des dépendances manquent, `dpkg` le signale mais ne les résout pas (c'est le rôle d'APT). Corrigez avec :

```bash
sudo apt install -f
```

Cette commande demande à APT de résoudre les dépendances cassées en installant les paquets manquants.

### Vérification post-installation

```bash
# Le binaire est accessible
$ which syswatch
/usr/bin/syswatch

# Le service tourne
$ systemctl status syswatch.service
● syswatch.service - Syswatch Monitoring Agent
     Active: active (running) since ...

# L'utilisateur système existe
$ id syswatch
uid=114(syswatch) gid=120(syswatch) groups=120(syswatch)

# Les répertoires de données existent avec les bonnes permissions
$ ls -ld /var/lib/syswatch /var/log/syswatch
drwxr-x--- 2 syswatch syswatch 4096 ... /var/lib/syswatch  
drwxr-x--- 2 syswatch syswatch 4096 ... /var/log/syswatch  

# La configuration est en place
$ cat /etc/syswatch/syswatch.yaml
...

# dpkg connaît le paquet
$ dpkg -l syswatch
ii  syswatch  1.2.0-1  amd64  Outil de monitoring systeme haute performance
```

### Désinstallation

```bash
# remove : supprime le binaire et le service, conserve la config
$ sudo apt remove syswatch

# purge : supprime tout, y compris la config et les données
$ sudo apt purge syswatch
```

Après un `purge`, vérifiez que les artefacts créés par le `postinst` ont bien été nettoyés par le `postrm` :

```bash
# Tout doit avoir disparu
$ ls /var/lib/syswatch 2>&1
ls: cannot access '/var/lib/syswatch': No such file or directory
$ id syswatch 2>&1
id: 'syswatch': no such user
$ dpkg -l syswatch
dpkg-query: no packages found matching syswatch
```

## Script de build complet

Voici le script qui consolide toutes les étapes en un seul fichier exécutable, prêt à être intégré dans un pipeline CI :

```bash
#!/bin/sh
# build-deb.sh — Construit le paquet DEB de syswatch
set -eu

# --- Configuration ---
PKG_NAME="syswatch"  
PKG_VERSION="1.2.0"  
PKG_REVISION="1"  
PKG_ARCH="amd64"  
PKG_DIR="${PKG_NAME}_${PKG_VERSION}-${PKG_REVISION}_${PKG_ARCH}"  
DEB_FILE="${PKG_DIR}.deb"  

# --- Nettoyage ---
rm -rf build "${PKG_DIR}" "${DEB_FILE}"

# --- Compilation ---
cmake -B build -G Ninja \
    -DCMAKE_BUILD_TYPE=Release \
    -DCMAKE_CXX_STANDARD=23
cmake --build build

# --- Installation staging ---
DESTDIR="$(pwd)/${PKG_DIR}" cmake --install build --prefix /usr

# --- Stripping ---
strip --strip-unneeded "${PKG_DIR}/usr/bin/${PKG_NAME}"

# --- Métadonnées DEBIAN/ ---
mkdir -p "${PKG_DIR}/DEBIAN"

# Dépendances automatiques
DEPS=$(dpkg-shlibdeps -O "${PKG_DIR}/usr/bin/${PKG_NAME}" 2>/dev/null \
       | sed 's/^shlibs:Depends=//')

# Taille installée
SIZE=$(du -sk "${PKG_DIR}" --exclude=DEBIAN | awk '{print $1}')

cat > "${PKG_DIR}/DEBIAN/control" << EOF  
Package: ${PKG_NAME}  
Version: ${PKG_VERSION}-${PKG_REVISION}  
Section: utils  
Priority: optional  
Architecture: ${PKG_ARCH}  
Depends: ${DEPS}  
Installed-Size: ${SIZE}  
Maintainer: Equipe SRE <sre@exemple.com>  
Homepage: https://github.com/exemple/${PKG_NAME}  
Description: Outil de monitoring systeme haute performance  
 Syswatch collecte des metriques CPU, memoire et disque
 en temps reel et les expose via un endpoint Prometheus.
EOF

# conffiles
printf '/etc/syswatch/syswatch.yaml\n' > "${PKG_DIR}/DEBIAN/conffiles"

# Scripts (copiés depuis le dépôt source)
cp packaging/postinst "${PKG_DIR}/DEBIAN/postinst"  
cp packaging/postrm   "${PKG_DIR}/DEBIAN/postrm"  
chmod 0755 "${PKG_DIR}/DEBIAN/postinst" "${PKG_DIR}/DEBIAN/postrm"  

# Documentation minimale
DOC_DIR="${PKG_DIR}/usr/share/doc/${PKG_NAME}"  
mkdir -p "${DOC_DIR}"  
cp LICENSE "${DOC_DIR}/copyright"  

# --- Construction ---
fakeroot dpkg-deb --build "${PKG_DIR}" "${DEB_FILE}"

# --- Vérification ---
echo "=== Contenu du paquet ==="  
dpkg-deb --contents "${DEB_FILE}"  

echo ""  
echo "=== Analyse lintian ==="  
lintian "${DEB_FILE}" || true  

echo ""  
echo "Paquet construit : ${DEB_FILE}"  
ls -lh "${DEB_FILE}"  
```

Ce script est volontairement séquentiel et lisible. En CI, les variables `PKG_VERSION` et `PKG_REVISION` seraient typiquement extraites du tag Git courant et du numéro de build :

```bash
PKG_VERSION=$(git describe --tags --abbrev=0 | sed 's/^v//')  
PKG_REVISION=$(git rev-list --count HEAD)  
```

## Intégration CI : GitHub Actions

Pour illustrer l'intégration dans un pipeline réel, voici un extrait de workflow GitHub Actions qui construit et publie le `.deb` comme artifact de release :

```yaml
jobs:
  build-deb:
    runs-on: ubuntu-24.04
    steps:
      - uses: actions/checkout@v4

      - name: Install build dependencies
        run: |
          sudo apt-get update
          sudo apt-get install -y cmake ninja-build g++ \
               dpkg-dev fakeroot lintian libssl-dev

      - name: Build DEB package
        run: ./build-deb.sh

      - name: Upload package
        uses: actions/upload-artifact@v4
        with:
          name: syswatch-deb
          path: "*.deb"
```

Le chapitre 38 (CI/CD pour C++) couvre en détail la configuration des workflows, les matrix builds multi-compilateur, et l'accélération avec ccache. L'important ici est de constater que le packaging DEB s'intègre naturellement comme étape finale d'un pipeline de build existant.

## Alternatives à dpkg-deb pour le packaging en CI

L'approche `dpkg-deb` manuelle est didactique et offre un contrôle total, mais elle nécessite d'écrire et de maintenir le script de build. Deux alternatives méritent mention.

### nfpm

**nfpm** (Not FPM) est un outil écrit en Go qui génère des paquets DEB, RPM et APK à partir d'un fichier de configuration YAML déclaratif :

```yaml
# nfpm.yaml
name: syswatch  
version: 1.2.0  
release: 1  
arch: amd64  
maintainer: "Equipe SRE <sre@exemple.com>"  
description: "Outil de monitoring systeme haute performance"  
section: utils  
depends:  
  - libc6 (>= 2.34)
  - libssl3 (>= 3.0)
contents:
  - src: build/syswatch
    dst: /usr/bin/syswatch
  - src: config/syswatch.yaml
    dst: /etc/syswatch/syswatch.yaml
    type: config
  - src: systemd/syswatch.service
    dst: /usr/lib/systemd/system/syswatch.service
scripts:
  postinstall: packaging/postinst
  postremove: packaging/postrm
```

```bash
nfpm package --packager deb
```

L'avantage principal est que le même fichier `nfpm.yaml` peut produire un `.deb` *et* un `.rpm` sans maintenir deux pipelines séparés. C'est un choix pragmatique pour les équipes qui ciblent plusieurs distributions.

### CPack (rappel)

Comme mentionné en section 39.1, CMake intègre CPack pour la génération de paquets. Pour un projet déjà entièrement géré par CMake, c'est la voie du moindre effort — au prix d'un contrôle réduit sur les scripts de maintien et les métadonnées avancées.

## Erreurs fréquentes et diagnostic

### "dpkg-deb: error: control directory has bad permissions"

Le répertoire `DEBIAN/` doit avoir les permissions `0755`. Un `chmod 0700` ou `0777` provoque cette erreur.

```bash
chmod 0755 pkg-root/DEBIAN
```

### "dpkg-shlibdeps: error: cannot find library"

`dpkg-shlibdeps` cherche les librairies dans les chemins système standards. Si votre binaire est lié à une librairie installée dans un chemin non standard (par exemple via Conan ou vcpkg), utilisez l'option `-l` pour ajouter des chemins de recherche :

```bash
dpkg-shlibdeps -l/chemin/vers/libs -O pkg-root/usr/bin/syswatch
```

### Le paquet s'installe mais le service ne démarre pas

Vérifiez dans cet ordre : le fichier `.service` est-il au bon emplacement (`/usr/lib/systemd/system/` ou `/lib/systemd/system/`) ? Le `postinst` appelle-t-il `systemctl daemon-reload` avant `systemctl start` ? L'utilisateur système déclaré dans le fichier `.service` (`User=syswatch`) a-t-il été créé ? Les permissions sur les répertoires de données correspondent-elles à cet utilisateur ?

```bash
# Le journal systemd contient toujours la réponse
journalctl -u syswatch.service --no-pager -n 30
```

### "trying to overwrite '/usr/bin/xxx', which is also in package yyy"

Deux paquets installent un fichier au même chemin. Si votre paquet remplace un ancien paquet, ajoutez les champs `Conflicts` et `Replaces` dans le fichier `control` :

```
Conflicts: ancien-outil (<< 2.0)  
Replaces: ancien-outil (<< 2.0)  
```

## Résumé du workflow

La construction d'un paquet DEB pour une application C++ se résume à une séquence de cinq opérations. Premièrement, compiler le projet avec CMake. Deuxièmement, installer dans un répertoire staging avec `DESTDIR`. Troisièmement, ajouter le répertoire `DEBIAN/` avec le fichier `control`, le `conffiles`, et les scripts de maintien. Quatrièmement, stripper le binaire. Cinquièmement, appeler `fakeroot dpkg-deb --build` et valider avec `lintian`.

Cette séquence est entièrement scriptable, reproductible, et intégrable dans n'importe quel pipeline CI. La section 39.2 applique la même logique au format RPM pour l'écosystème Red Hat.
