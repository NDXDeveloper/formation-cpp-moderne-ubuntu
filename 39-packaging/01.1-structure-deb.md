# 39.1.1 — Structure d'un paquet DEB

## Ce que contient un `.deb`, fichier par fichier

---

## Vue d'ensemble

Un paquet DEB est construit à partir d'un simple répertoire dont l'arborescence suit deux conventions : un sous-répertoire `DEBIAN/` contenant les métadonnées et les scripts, et le reste de l'arborescence reproduisant le système de fichiers cible. C'est tout. Il n'y a pas de format binaire propriétaire à comprendre, pas d'outil graphique obligatoire — juste une structure de répertoires que `dpkg-deb` compresse en un `.deb`.

Voici à quoi ressemble cette arborescence pour une application C++ typique :

```
mon-outil_1.2.0-1_amd64/
├── DEBIAN/
│   ├── control            # Métadonnées obligatoires
│   ├── conffiles          # Liste des fichiers de configuration
│   ├── postinst           # Script post-installation (optionnel)
│   ├── prerm              # Script pré-désinstallation (optionnel)
│   ├── postrm             # Script post-désinstallation (optionnel)
│   └── preinst            # Script pré-installation (optionnel)
├── usr/
│   ├── bin/
│   │   └── mon-outil      # Binaire principal
│   ├── lib/
│   │   └── libmon-outil.so.1.2.0   # Librairie partagée
│   ├── include/
│   │   └── mon-outil/     # Headers (si paquet -dev)
│   │       └── api.h
│   └── share/
│       ├── doc/
│       │   └── mon-outil/
│       │       └── copyright       # Licence (recommandé)
│       └── man/
│           └── man1/
│               └── mon-outil.1.gz  # Page man compressée
├── etc/
│   └── mon-outil/
│       └── config.yaml    # Configuration par défaut
└── lib/
    └── systemd/
        └── system/
            └── mon-outil.service   # Unit systemd
```

Deux zones distinctes coexistent dans cette arborescence, et il est essentiel de bien comprendre leur rôle respectif.

---

## Le répertoire DEBIAN/ : le cerveau du paquet

Le répertoire `DEBIAN/` (en majuscules — c'est une convention stricte, pas une suggestion) contient tout ce que `dpkg` a besoin de savoir sur le paquet *sans* regarder les fichiers installés. Il n'est jamais copié sur le système cible : son contenu est extrait par `dpkg` au moment de l'installation et stocké dans sa base de données interne (`/var/lib/dpkg/info/`).

### Le fichier control : la carte d'identité

C'est le seul fichier véritablement obligatoire. Sans lui, `dpkg-deb` refuse de construire le paquet. Il décrit le paquet dans un format clé-valeur inspiré des en-têtes de mail RFC 822 :

```
Package: mon-outil  
Version: 1.2.0-1  
Section: utils  
Priority: optional  
Architecture: amd64  
Depends: libc6 (>= 2.35), libstdc++6 (>= 12), libssl3 (>= 3.0)  
Maintainer: Équipe DevOps <devops@exemple.com>  
Installed-Size: 2048  
Homepage: https://github.com/exemple/mon-outil  
Description: Outil de monitoring système haute performance  
 Mon-outil collecte et agrège des métriques système en temps réel.
 Il expose un endpoint Prometheus et supporte la configuration YAML.
 .
 Fonctionnalités principales :
  - Collecte CPU, mémoire, disque, réseau
  - Export Prometheus natif
  - Configuration hot-reload via SIGHUP
```

Chaque champ mérite une explication détaillée.

**`Package`** est le nom du paquet. Il doit être en minuscules, ne contenir que des caractères alphanumériques, des tirets (`-`), des points (`.`) et des plus (`+`). C'est l'identifiant que l'utilisateur passera à `apt install`.

**`Version`** suit le schéma `[epoch:]version_amont-révision_debian`. L'epoch (un entier suivi de `:`) est rarement utilisé et sert à corriger des erreurs de numérotation historiques. La version amont correspond à la version de votre logiciel. La révision Debian (`-1`) identifie la révision du packaging. Pour un logiciel interne, `1.2.0-1` est le format standard.

**`Section`** catégorise le paquet. Les valeurs courantes pour du C++ sont `utils` (utilitaires), `libs` (librairies), `net` (réseau), `devel` (développement — pour les paquets contenant des headers). Ce champ est utilisé par les outils de gestion de dépôts pour organiser les paquets.

**`Priority`** indique l'importance du paquet pour le système. `optional` est la valeur appropriée pour la quasi-totalité des logiciels tiers — les autres niveaux (`required`, `important`, `standard`) sont réservés aux paquets de base de la distribution.

**`Architecture`** spécifie l'architecture matérielle cible. Pour un binaire C++ compilé sur x86_64, c'est `amd64`. Pour un cross-compilation ARM 64 bits, c'est `arm64`. Si le paquet ne contient que des scripts ou de la documentation, utilisez `all`.

**`Depends`** est le champ le plus critique pour un paquet C++. Il liste les paquets requis à l'exécution, avec des contraintes de version optionnelles. Quand un utilisateur lance `apt install mon-outil`, APT lit ce champ et installe automatiquement les dépendances manquantes. Nous détaillerons plus bas comment déterminer cette liste.

**`Maintainer`** identifie le responsable du paquet. Pour un paquet interne, c'est typiquement l'adresse de l'équipe.

**`Installed-Size`** indique la taille approximative en kilooctets une fois installé. Ce n'est pas la taille du `.deb` compressé. Vous pouvez la calculer avec `du -s --apparent-size` sur l'arborescence de données.

**`Description`** est composée d'une ligne de résumé (courte, sans point final) suivie d'une description longue. Chaque ligne de la description longue commence par un espace. Les paragraphes sont séparés par une ligne contenant un seul point (` .`). Ce format peut sembler archaïque, mais il est strictement respecté par tous les outils Debian.

### Champs supplémentaires utiles

Selon vos besoins, d'autres champs peuvent apparaître dans le fichier `control` :

```
Pre-Depends: dpkg (>= 1.19.0)  
Recommends: prometheus-node-exporter  
Suggests: grafana  
Conflicts: ancien-outil  
Replaces: ancien-outil (<< 2.0)  
Provides: monitoring-agent  
```

**`Pre-Depends`** est une version renforcée de `Depends` : le paquet listé doit être non seulement installé mais entièrement configuré *avant* que votre paquet ne soit décompressé. Son utilisation est rare et réservée aux cas où vos scripts d'installation ont besoin d'un outil spécifique.

**`Recommends`** et **`Suggests`** déclarent des dépendances optionnelles. APT installe par défaut les `Recommends` (sauf configuration contraire) mais pas les `Suggests`. Utilisez `Recommends` pour les paquets qui enrichissent significativement votre logiciel, et `Suggests` pour les extras.

**`Conflicts`** empêche l'installation simultanée de deux paquets incompatibles. **`Replaces`** indique que votre paquet remplace des fichiers appartenant à un autre paquet — utile lors d'un renommage de projet. **`Provides`** déclare un paquet virtuel, permettant à d'autres paquets de dépendre d'une fonctionnalité plutôt que d'un nom de paquet spécifique.

### Le fichier conffiles : protéger la configuration utilisateur

Le fichier `conffiles` contient la liste des fichiers de configuration, un par ligne, avec leur chemin absolu sur le système cible :

```
/etc/mon-outil/config.yaml
```

Ce fichier a une conséquence concrète importante : lors d'une mise à jour du paquet, si l'utilisateur a modifié un fichier listé dans `conffiles`, `dpkg` détecte le conflit et demande à l'utilisateur s'il veut garder sa version, installer la nouvelle, ou examiner les différences. Sans cette déclaration, `dpkg` écraserait silencieusement la configuration personnalisée de l'utilisateur à chaque mise à jour — un comportement inacceptable en production.

Seuls les fichiers situés dans `/etc/` doivent typiquement être déclarés comme conffiles. Les fichiers de données ou les binaires n'ont pas à y figurer.

### Les scripts de maintien : preinst, postinst, prerm, postrm

Ces quatre scripts (couverts en détail dans la section 39.1.2) s'exécutent à des moments précis du cycle de vie du paquet. Ils sont optionnels. Quand ils sont présents, ils doivent être des exécutables (scripts shell, généralement en `#!/bin/sh` pour la portabilité) avec les permissions `0755`.

## L'arborescence de données : le corps du paquet

Tout ce qui se trouve en dehors de `DEBIAN/` constitue l'arborescence de données. Elle reproduit exactement la structure du système de fichiers cible, en partant de la racine `/`. Quand `dpkg` installe le paquet, il copie cette arborescence à la racine du système.

### Où placer vos fichiers

Les conventions Debian (qui suivent le Filesystem Hierarchy Standard — FHS) dictent les emplacements standards. Voici les répertoires pertinents pour une application C++ :

**`/usr/bin/`** accueille les exécutables accessibles à tous les utilisateurs. C'est l'emplacement de votre binaire principal. Évitez `/usr/local/bin/` dans un paquet — ce répertoire est réservé aux installations manuelles de l'administrateur.

**`/usr/lib/`** (ou `/usr/lib/x86_64-linux-gnu/` sur les systèmes multiarch) accueille les librairies partagées. Si votre projet produit un `.so`, c'est ici qu'il va. Le sous-répertoire multiarch est préféré sur Ubuntu pour éviter les conflits entre architectures.

**`/usr/include/`** est le répertoire des headers C/C++. Si vous distribuez une librairie avec ses headers pour les développeurs, créez un sous-répertoire à votre nom (`/usr/include/mon-outil/`). En pratique, les headers sont souvent distribués dans un paquet séparé suffixé `-dev` (par exemple `libmon-outil-dev`).

**`/etc/`** contient les fichiers de configuration. Créez un sous-répertoire dédié (`/etc/mon-outil/`) plutôt que de placer des fichiers directement dans `/etc/`. N'oubliez pas de déclarer ces fichiers dans `conffiles`.

**`/usr/share/doc/<nom-du-paquet>/`** accueille la documentation : le fichier `copyright` (fortement recommandé), un `changelog.Debian.gz` (obligatoire pour les paquets officiels, recommandé pour les paquets internes), et tout document complémentaire.

**`/usr/share/man/man1/`** accueille les pages man pour les commandes utilisateur. Les pages man doivent être compressées avec gzip (`.1.gz`).

**`/lib/systemd/system/`** accueille les fichiers unit systemd. Si votre application tourne comme service (daemon), c'est l'emplacement standard pour son fichier `.service`.

### Séparation en plusieurs paquets

Pour une librairie C++, la convention Debian est de séparer le contenu en plusieurs paquets :

- **`libmon-outil1`** — Le paquet runtime contenant uniquement le `.so` versionné. Le chiffre dans le nom correspond au SONAME (version majeure de l'ABI).  
- **`libmon-outil-dev`** — Le paquet de développement contenant les headers, les fichiers `.cmake` ou `.pc` (pkg-config), et le lien symbolique `.so` non versionné.  
- **`mon-outil`** — Le paquet contenant les binaires exécutables, qui dépend de `libmon-outil1`.

Cette séparation permet à un serveur de production d'installer uniquement le runtime sans les headers et les outils de développement, réduisant la surface d'attaque et l'espace disque consommé. Pour un outil interne simple (un seul binaire sans librairie partagée publique), un paquet unique suffit.

## Déterminer les dépendances d'un binaire C++

Le champ `Depends` du fichier `control` doit lister précisément les paquets nécessaires à l'exécution. Déclarer trop peu de dépendances produit un paquet qui échoue à l'installation sur un système minimal. En déclarer trop encombre inutilement l'installation.

### Méthode manuelle avec ldd et dpkg

La commande `ldd` liste les librairies partagées dont dépend un binaire :

```bash
$ ldd /tmp/mon-paquet/usr/bin/mon-outil
    linux-vdso.so.1 (0x00007ffce3ffe000)
    libssl.so.3 => /lib/x86_64-linux-gnu/libssl.so.3 (0x...)
    libcrypto.so.3 => /lib/x86_64-linux-gnu/libcrypto.so.3 (0x...)
    libstdc++.so.6 => /lib/x86_64-linux-gnu/libstdc++.so.6 (0x...)
    libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6 (0x...)
    ...
```

Pour chaque librairie, `dpkg -S` identifie le paquet propriétaire :

```bash
$ dpkg -S /lib/x86_64-linux-gnu/libssl.so.3
libssl3: /lib/x86_64-linux-gnu/libssl.so.3
```

Vous savez maintenant que votre paquet dépend de `libssl3`.

### Méthode automatisée avec dpkg-shlibdeps

L'outil `dpkg-shlibdeps` automatise ce processus. Il analyse les binaires et les librairies de votre paquet, résout chaque dépendance de librairie partagée vers un paquet Debian avec la contrainte de version minimale appropriée, et écrit le résultat dans un fichier `debian/substvars` :

```bash
$ dpkg-shlibdeps -O /tmp/mon-paquet/usr/bin/mon-outil
shlibs:Depends=libc6 (>= 2.34), libssl3 (>= 3.0.0), libstdc++6 (>= 12)
```

La sortie produite peut être utilisée directement comme valeur du champ `Depends` dans votre fichier `control`. C'est la méthode recommandée car elle calcule automatiquement les versions minimales à partir des symboles effectivement utilisés par votre binaire — quelque chose d'impossible à faire manuellement de manière fiable.

## Permissions et propriétaire des fichiers

Les fichiers dans l'arborescence de données doivent avoir les bonnes permissions et appartenir à `root:root`. C'est une source d'erreurs fréquente quand on construit le paquet en tant qu'utilisateur normal.

Les conventions de permissions standards sont les suivantes : les répertoires doivent être en `0755`, les exécutables en `0755`, les fichiers réguliers (configs, docs, headers) en `0644`, et les scripts dans `DEBIAN/` en `0755`.

L'outil `fakeroot` résout le problème du propriétaire sans nécessiter de droits root :

```bash
fakeroot dpkg-deb --build mon-outil_1.2.0-1_amd64
```

`fakeroot` intercepte les appels système liés aux permissions et fait croire aux outils que tous les fichiers appartiennent à `root:root`, sans modifier réellement le système de fichiers. Le `.deb` produit contiendra les bonnes métadonnées de propriété.

Si vous construisez le paquet dans un conteneur Docker tournant en tant que root (ce qui est courant en CI), `fakeroot` n'est pas nécessaire — les fichiers appartiennent déjà à root. Assurez-vous toutefois que les permissions numériques sont correctes.

## Vérification avec lintian

Une fois le `.deb` construit, passez-le systématiquement dans `lintian` :

```bash
$ lintian mon-outil_1.2.0-1_amd64.deb
W: mon-outil: no-manual-page usr/bin/mon-outil  
W: mon-outil: no-changelog usr/share/doc/mon-outil/changelog.Debian.gz  
E: mon-outil: extended-description-is-empty  
```

`lintian` distingue trois niveaux de gravité :

- **`E` (Error)** signale une violation de la Debian Policy qui rendrait le paquet inacceptable dans les dépôts officiels. Pour un paquet interne, les erreurs méritent attention mais ne sont pas toutes bloquantes.  
- **`W` (Warning)** signale un écart par rapport aux bonnes pratiques. L'absence de page man (`no-manual-page`) est un warning classique pour les outils internes — vous pouvez le corriger à terme ou le documenter comme acceptable.  
- **`I` (Info)** fournit des suggestions d'amélioration.

Même pour un paquet qui ne sera jamais soumis aux dépôts Debian, corriger les erreurs `lintian` est un bon réflexe : elles correspondent à des problèmes réels (dépendances oubliées, permissions incorrectes, fichiers mal placés) qui causeront des problèmes en production.

## Résumé de la structure

Pour récapituler, un paquet DEB est un répertoire à deux zones. La zone `DEBIAN/` contient les métadonnées — le fichier `control` (obligatoire) décrit le paquet et ses dépendances, `conffiles` protège la configuration utilisateur lors des mises à jour, et les scripts `preinst`/`postinst`/`prerm`/`postrm` automatisent le cycle de vie. La zone de données reproduit l'arborescence du système cible et respecte le FHS. `dpkg-deb` fusionne les deux en un `.deb`, et `lintian` vérifie la conformité.
