# 39.1.2 — Scripts de contrôle

## Automatiser l'installation, la configuration et le nettoyage

---

## Introduction

Les fichiers `control` et `conffiles` vus dans la section précédente décrivent le paquet de manière déclarative : qui il est, de quoi il dépend, quels fichiers de configuration protéger. Mais copier des fichiers sur le disque ne suffit pas toujours. Une application C++ déployée en production a souvent besoin d'un utilisateur système dédié pour tourner, d'un service systemd activé au démarrage, d'un répertoire de logs avec les bonnes permissions, ou d'un cache à nettoyer lors de la désinstallation.

C'est le rôle des **scripts de maintien** (maintainer scripts) : quatre scripts shell qui s'exécutent à des moments précis du cycle de vie du paquet. Ils transforment un simple dépôt de fichiers en une installation complète et autonome.

---

## Les quatre scripts et leur moment d'exécution

Le cycle de vie d'un paquet DEB comporte deux phases principales — l'installation et la désinstallation — et chaque phase dispose d'un script "avant" et d'un script "après".

### Chronologie d'une installation

Quand l'utilisateur lance `sudo apt install mon-outil` ou `sudo dpkg -i mon-outil_1.2.0-1_amd64.deb`, les étapes se déroulent dans cet ordre :

1. **`preinst`** s'exécute *avant* que les fichiers du paquet ne soient décompressés sur le système. Le système de fichiers ne contient pas encore vos binaires ni vos configs.
2. **dpkg décompresse** l'arborescence de données à la racine du système.
3. **`postinst`** s'exécute *après* la décompression. Tous vos fichiers sont en place.

### Chronologie d'une désinstallation

Quand l'utilisateur lance `sudo apt remove mon-outil` :

1. **`prerm`** s'exécute *avant* que les fichiers ne soient supprimés. Vos binaires et configs sont encore sur le disque.
2. **dpkg supprime** les fichiers du paquet (sauf les conffiles modifiés par l'utilisateur).
3. **`postrm`** s'exécute *après* la suppression.

Si l'utilisateur utilise `apt purge` au lieu de `apt remove`, les conffiles sont également supprimés et `postrm` est appelé avec l'argument `purge`.

### Chronologie d'une mise à jour

La mise à jour est le scénario le plus complexe. Quand une nouvelle version du paquet remplace l'ancienne, la séquence est la suivante :

1. **`prerm` de l'ancienne version** avec l'argument `upgrade`.
2. **`preinst` de la nouvelle version** avec l'argument `upgrade`.
3. dpkg décompresse les nouveaux fichiers (remplaçant les anciens).
4. **`postrm` de l'ancienne version** avec l'argument `upgrade`.
5. **`postinst` de la nouvelle version** avec l'argument `configure`.

Cette séquence explique pourquoi les scripts doivent être conçus de manière **idempotente** : ils peuvent être appelés plusieurs fois, dans différents contextes, et doivent produire un résultat correct à chaque fois.

## Règles d'écriture fondamentales

Avant de voir le contenu de chaque script, voici les règles que tous les scripts de maintien doivent respecter.

### Shebang et portabilité

Les scripts doivent commencer par `#!/bin/sh` et non `#!/bin/bash`. La Debian Policy exige la compatibilité avec tout shell POSIX, car `/bin/sh` pointe vers `dash` sur Debian et Ubuntu — un shell minimal plus rapide que `bash` mais qui ne supporte pas les bashismes (`[[ ]]`, `${var//pattern/replacement}`, tableaux, etc.).

```sh
#!/bin/sh
set -e
```

### set -e : sortir immédiatement en cas d'erreur

La directive `set -e` est essentielle. Sans elle, un script qui échoue à mi-parcours continue son exécution, laissant potentiellement le système dans un état incohérent. Avec `set -e`, toute commande qui retourne un code non nul interrompt le script, et `dpkg` marque le paquet comme étant dans un état d'erreur — ce qui est préférable à une installation silencieusement corrompue.

### Gestion de l'argument $1

Chaque script reçoit un argument qui indique le contexte d'appel. Ce mécanisme est ce qui permet à un même script de se comporter correctement lors d'une installation initiale, d'une mise à jour, ou d'une erreur de configuration. Les valeurs possibles sont :

Pour **`preinst`** : `install` (première installation), `upgrade` (mise à jour), `abort-upgrade` (annulation).

Pour **`postinst`** : `configure` (installation ou mise à jour réussie), `abort-upgrade`, `abort-remove`, `abort-deconfigure`.

Pour **`prerm`** : `remove` (désinstallation), `upgrade` (avant mise à jour), `deconfigure`.

Pour **`postrm`** : `remove` (après suppression des fichiers), `purge` (après suppression incluant les conffiles), `upgrade`, `disappear`.

Un script bien écrit teste cet argument pour adapter son comportement :

```sh
#!/bin/sh
set -e

case "$1" in
    configure)
        # Actions de configuration
        ;;
    abort-upgrade|abort-remove|abort-deconfigure)
        # Nettoyage en cas d'erreur
        ;;
    *)
        echo "postinst: action inconnue '$1'" >&2
        exit 1
        ;;
esac
```

### Idempotence

Un script doit produire le même résultat qu'il soit exécuté une fois ou dix fois. Si votre `postinst` crée un utilisateur système, il doit d'abord vérifier que cet utilisateur n'existe pas déjà. S'il active un service systemd, `systemctl enable` est naturellement idempotent — mais un `mkdir` sans `-p` échouera si le répertoire existe déjà.

## preinst : préparer le terrain

Le script `preinst` s'exécute avant que les fichiers du paquet ne soient décompressés. Il est le moins fréquemment utilisé des quatre, car la plupart des actions de configuration nécessitent que les fichiers soient déjà en place. Ses cas d'usage légitimes sont limités mais importants.

### Cas d'usage typiques

**Arrêter une version en cours d'exécution** — Lors d'une mise à jour, si le service tourne et qu'il verrouille des fichiers, `dpkg` ne pourra pas les remplacer. Le `preinst` peut arrêter le service avant la décompression :

```sh
#!/bin/sh
set -e

case "$1" in
    install|upgrade)
        # Arrêter le service s'il existe déjà (mise à jour)
        if [ -d /run/systemd/system ] && systemctl is-active --quiet mon-outil.service 2>/dev/null; then
            systemctl stop mon-outil.service
        fi
        ;;
esac
```

**Sauvegarder un état avant remplacement** — Si une mise à jour modifie le format d'un fichier de données, le `preinst` peut sauvegarder l'ancien fichier pour permettre une migration dans le `postinst`.

### Quand ne pas l'utiliser

Évitez de créer des utilisateurs ou des répertoires dans le `preinst`. Si le paquet échoue à s'installer après le `preinst`, ces ressources orphelines resteront sur le système sans mécanisme de nettoyage propre. Préférez le `postinst` pour ces actions.

---

## postinst : le script le plus important

Le `postinst` est le script que vous écrirez le plus souvent. Il s'exécute après que tous les fichiers du paquet sont en place, et c'est là que la configuration réelle du logiciel a lieu.

### Création d'un utilisateur système

Une application C++ déployée comme service ne doit jamais tourner en tant que `root`. La pratique standard est de créer un utilisateur système dédié, sans shell de connexion et sans répertoire home :

```sh
#!/bin/sh
set -e

case "$1" in
    configure)
        # Créer l'utilisateur système si absent
        if ! getent passwd mon-outil > /dev/null 2>&1; then
            adduser --system --group --no-create-home \
                    --home /nonexistent \
                    --shell /usr/sbin/nologin \
                    mon-outil
        fi

        # Créer et ajuster les permissions du répertoire de données
        install -d -o mon-outil -g mon-outil -m 0750 /var/lib/mon-outil
        install -d -o mon-outil -g mon-outil -m 0750 /var/log/mon-outil

        # Activer et démarrer le service
        if [ -d /run/systemd/system ]; then
            systemctl daemon-reload
            systemctl enable mon-outil.service
            systemctl start mon-outil.service
        fi
        ;;
esac
```

Plusieurs points méritent attention dans ce script.

Le test `getent passwd` est la manière portable de vérifier l'existence d'un utilisateur — ne parsez pas `/etc/passwd` manuellement. L'option `--system` crée un utilisateur avec un UID dans la plage réservée aux comptes système (typiquement en dessous de 1000), ce qui le distingue des comptes humains. L'option `--no-create-home` évite de créer un répertoire `/home/mon-outil` inutile.

La commande `install -d` crée un répertoire avec les permissions et le propriétaire spécifiés en une seule opération atomique — c'est plus propre que la séquence `mkdir` + `chown` + `chmod`.

Le test `[ -d /run/systemd/system ]` vérifie que systemd est bien le système d'init actif. Cela rend le script compatible avec les conteneurs Docker où systemd n'est souvent pas présent.

### Activation d'un service systemd

L'interaction entre les paquets DEB et systemd mérite un approfondissement. Depuis Debian 10 et Ubuntu 18.04, le helper `deb-systemd-helper` est la méthode recommandée pour gérer les services dans les scripts de maintien. Cependant, pour un paquet interne, l'appel direct à `systemctl` (protégé par le test de présence de systemd) est parfaitement acceptable et plus lisible.

L'appel à `systemctl daemon-reload` est nécessaire après chaque installation ou mise à jour d'un fichier `.service`. Sans cela, systemd utilise une version en cache du fichier unit et ignore vos modifications.

### Génération de configuration initiale

Si votre application nécessite une configuration spécifique à la machine (un identifiant unique, un token, une clé de chiffrement), le `postinst` peut la générer à la première installation :

```sh
        # Générer une clé de chiffrement locale si absente
        KEYFILE="/etc/mon-outil/encryption.key"
        if [ ! -f "$KEYFILE" ]; then
            openssl rand -base64 32 > "$KEYFILE"
            chown mon-outil:mon-outil "$KEYFILE"
            chmod 0600 "$KEYFILE"
        fi
```

Ce fichier ne doit **pas** être listé dans `conffiles` car il n'est pas fourni par le paquet — il est généré localement. Il ne sera donc pas touché par les mises à jour du paquet, ce qui est le comportement souhaité.

## prerm : préparer la désinstallation

Le `prerm` s'exécute avant que `dpkg` ne supprime les fichiers du paquet. C'est le moment d'arrêter proprement le service et de désactiver le démarrage automatique :

```sh
#!/bin/sh
set -e

case "$1" in
    remove|deconfigure)
        # Arrêter le service
        if [ -d /run/systemd/system ]; then
            systemctl stop mon-outil.service 2>/dev/null || true
            systemctl disable mon-outil.service 2>/dev/null || true
        fi
        ;;
    upgrade)
        # En cas de mise à jour, juste arrêter (le postinst redémarrera)
        if [ -d /run/systemd/system ]; then
            systemctl stop mon-outil.service 2>/dev/null || true
        fi
        ;;
esac
```

Le `|| true` après les commandes `systemctl` empêche `set -e` d'interrompre le script si le service n'est pas actif ou n'existe pas. C'est une situation normale lors de la première désinstallation d'un paquet qui n'a jamais réussi à démarrer son service.

La distinction entre `remove` et `upgrade` est importante : lors d'une mise à jour, on arrête le service mais on ne le désactive pas (le `postinst` de la nouvelle version le réactivera). Lors d'une désinstallation, on arrête et on désactive.

---

## postrm : nettoyer après la suppression

Le `postrm` s'exécute après que les fichiers du paquet ont été supprimés. Il gère deux scénarios distincts : le `remove` simple (les conffiles sont conservés) et le `purge` (tout est supprimé).

```sh
#!/bin/sh
set -e

case "$1" in
    remove)
        # Recharger systemd pour oublier le service supprimé
        if [ -d /run/systemd/system ]; then
            systemctl daemon-reload
        fi
        ;;
    purge)
        # Supprimer les données et logs créés en dehors du paquet
        rm -rf /var/lib/mon-outil
        rm -rf /var/log/mon-outil

        # Supprimer les fichiers générés (clés, caches)
        rm -rf /etc/mon-outil

        # Supprimer l'utilisateur système
        if getent passwd mon-outil > /dev/null 2>&1; then
            deluser --system mon-outil 2>/dev/null || true
        fi
        if getent group mon-outil > /dev/null 2>&1; then
            delgroup --system mon-outil 2>/dev/null || true
        fi

        if [ -d /run/systemd/system ]; then
            systemctl daemon-reload
        fi
        ;;
    upgrade|failed-upgrade|abort-install|abort-upgrade|disappear)
        ;;
esac
```

La distinction entre `remove` et `purge` est un contrat avec l'administrateur système. `apt remove` dit "désinstalle le logiciel mais garde la configuration — je pourrais le réinstaller plus tard". `apt purge` dit "supprime tout, comme si ce paquet n'avait jamais existé". Votre `postrm` doit respecter cette sémantique.

Remarquez que les répertoires `/var/lib/mon-outil` et `/var/log/mon-outil` ne font pas partie de l'arborescence de données du paquet — ils ont été créés par le `postinst`. C'est pourquoi `dpkg` ne les supprime pas automatiquement : il ne supprime que les fichiers qu'il a lui-même installés. Le `postrm` est l'endroit pour nettoyer ces artefacts créés dynamiquement.

La suppression de l'utilisateur système lors du `purge` est un sujet débattu dans la communauté Debian. Certains mainteneurs préfèrent conserver l'utilisateur pour éviter que son UID soit réassigné à un autre utilisateur, ce qui pourrait causer des problèmes de permissions sur des fichiers orphelins. Pour un paquet interne avec un cycle de vie contrôlé, la suppression est généralement acceptable.

## Un piège classique : l'échec d'installation

Que se passe-t-il si votre `postinst` échoue — par exemple, si `adduser` retourne une erreur ? Grâce à `set -e`, le script s'interrompt immédiatement. `dpkg` marque alors le paquet dans l'état `Half-Configured`. L'administrateur voit un message d'erreur et le paquet reste dans un état bancal.

Pour tenter de réparer, `dpkg` appelle `postinst abort-upgrade` ou `postinst abort-install`. C'est pour cette raison que le `case` de vos scripts doit gérer ces cas :

```sh
    abort-upgrade|abort-remove|abort-deconfigure)
        # Tenter un nettoyage minimal
        # Supprimer les ressources créées par le configure partiel
        ;;
```

En pratique, il est difficile de gérer parfaitement tous les scénarios d'échec. La meilleure stratégie est de rendre chaque étape du `postinst` aussi robuste que possible : vérifier les préconditions avant d'agir, utiliser des commandes idempotentes, et ne pas dépendre d'un ordre d'exécution fragile.

---

## Tester les scripts sans installer le paquet

Pendant le développement, vous pouvez tester vos scripts manuellement sans passer par `dpkg` :

```bash
# Simuler une installation
sudo sh -x DEBIAN/postinst configure

# Simuler une désinstallation
sudo sh -x DEBIAN/prerm remove

# Simuler un purge
sudo sh -x DEBIAN/postrm purge
```

L'option `-x` active le mode trace de `sh`, qui affiche chaque commande avant de l'exécuter — indispensable pour le débogage. Pensez à nettoyer les ressources créées (utilisateurs, répertoires) entre chaque test.

Pour vérifier la syntaxe sans exécuter :

```bash
sh -n DEBIAN/postinst
```

Cette commande parse le script et signale les erreurs de syntaxe sans rien exécuter.

## Bonnes pratiques récapitulatives

**Utilisez `#!/bin/sh` et restez POSIX.** Pas de bashismes, pas de `[[ ]]`, pas de tableaux. Testez vos scripts avec `dash` si vous avez un doute : `dash -n votre_script.sh`.

**`set -e` en première ligne après le shebang.** Toujours, sans exception.

**Testez `$1` systématiquement.** Ne présumez jamais du contexte d'appel. Un `postinst` est appelé avec `configure` lors d'une installation normale, mais aussi avec `abort-upgrade` en cas d'échec de mise à jour.

**Rendez chaque action idempotente.** `adduser` doit être précédé de `getent passwd`. `mkdir` doit être `mkdir -p` ou précédé d'un test `[ -d ]`. `systemctl enable` est déjà idempotent.

**Protégez les appels systemd.** Testez `[ -d /run/systemd/system ]` avant d'appeler `systemctl`. Votre paquet pourrait être installé dans un chroot, un conteneur Docker, ou un environnement de build qui n'a pas systemd.

**Minimisez le travail dans `preinst`.** Préférez `postinst` pour la configuration. `preinst` ne devrait intervenir que quand une action est nécessaire *avant* que les fichiers ne soient en place.

**Respectez la sémantique remove vs purge.** `remove` arrête le logiciel et retire les binaires. `purge` efface toute trace, y compris les données créées dynamiquement. Ne supprimez pas les données utilisateur lors d'un simple `remove`.

**Ne mettez pas de logique métier complexe dans les scripts.** Si votre configuration post-installation nécessite des dizaines de lignes de logique, envisagez de placer cette logique dans un binaire ou un script dédié installé par le paquet, et de l'appeler depuis le `postinst`. Cela facilite les tests et la maintenance.
