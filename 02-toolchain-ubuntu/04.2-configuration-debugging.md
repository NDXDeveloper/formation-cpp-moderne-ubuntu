# 2.4.2 — Configuration du debugging intégré

> **Chapitre 2 · Section 2.4 · Configuration de l'IDE** · Niveau Débutant

---

## Introduction

Le débogage est une activité quotidienne en développement C++. Un bug de segfault, un comportement inattendu dans une boucle, une variable qui prend une valeur aberrante — ces situations exigent de pouvoir **stopper l'exécution** à un endroit précis, **inspecter** l'état du programme (variables, mémoire, pile d'appels) et **avancer pas à pas** pour comprendre ce qui se passe réellement.

En ligne de commande, c'est le rôle de GDB (présenté en section 2.2, et approfondi au chapitre 29). Mais au quotidien, l'expérience de débogage la plus productive passe par l'**intégration dans l'IDE** : breakpoints posés d'un clic, variables affichées dans un panneau latéral, pile d'appels navigable, évaluation d'expressions en temps réel. Le debugger en ligne de commande et le debugger intégré utilisent le même moteur (GDB ou LLDB) — seule l'interface change.

Cette section détaille la mise en place du débogage intégré dans les trois éditeurs présentés à la section 2.4 : VS Code, CLion et Neovim.

---

## Prérequis communs

Quel que soit l'éditeur, le débogage C++ nécessite deux choses :

### 1. Compiler avec les symboles de débogage

Les symboles de débogage sont les informations qui permettent au debugger de faire le lien entre le code machine exécuté et votre code source : noms de variables, numéros de ligne, types des données, etc. Sans ces symboles, le debugger ne peut afficher que des adresses mémoire brutes.

Le flag `-g` active la génération de symboles de débogage :

```bash
g++ -std=c++23 -g -O0 -o mon_programme main.cpp
```

Le flag `-O0` désactive les optimisations. C'est important : les optimisations (`-O2`, `-O3`) réordonnent le code, éliminent des variables et inlinent des fonctions, ce qui rend le pas-à-pas confus (le debugger saute des lignes, des variables apparaissent comme « optimized out »). Pour le débogage, compilez toujours en `-O0` ou `-Og` (optimisation minimale compatible avec le debug).

Avec CMake, c'est le rôle du **build type** `Debug` :

```bash
cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Debug  
cmake --build build  
```

Le build type `Debug` configure automatiquement `-g -O0` sur GCC et Clang. Le type `RelWithDebInfo` active `-g -O2` — utile pour déboguer des problèmes qui n'apparaissent qu'avec les optimisations, mais l'expérience pas-à-pas sera moins confortable.

### 2. Avoir un debugger installé

```bash
# GDB (recommandé sur Linux)
sudo apt install gdb

# LLDB (alternative LLVM)
sudo apt install lldb-20
```

Vérification :

```bash
gdb --version | head -1
# GNU gdb (Ubuntu 15.x-...) 15.x
```

## Débogage dans VS Code

### Architecture

VS Code ne contient pas de debugger en soi. Le débogage repose sur l'extension **C/C++** de Microsoft (`ms-vscode.cpptools`), qui fournit un adaptateur de débogage compatible avec GDB et LLDB. L'extension communique avec GDB via le protocole **MI (Machine Interface)**, une interface texte standardisée que GDB expose pour les outils graphiques.

Le flux est le suivant :

```
VS Code UI  ←→  Extension C/C++  ←→  GDB (MI mode)  ←→  Votre programme
```

### Le fichier `launch.json`

Toute la configuration de débogage dans VS Code se fait via le fichier `.vscode/launch.json`. Ce fichier décrit une ou plusieurs **configurations de lancement** : quel programme exécuter, avec quel debugger, quels arguments, quelles variables d'environnement, etc.

Pour créer ce fichier, ouvrez la vue « Run and Debug » (`Ctrl+Shift+D`), cliquez sur « create a launch.json file » et sélectionnez « C++ (GDB/LLDB) ». VS Code crée un squelette que nous allons compléter.

### Configuration complète avec GDB

```json
// .vscode/launch.json
{
    "version": "0.2.0",
    "configurations": [
        {
            "name": "Debug (GDB)",
            "type": "cppdbg",
            "request": "launch",
            "program": "${command:cmake.launchTargetPath}",
            "args": [],
            "stopAtEntry": false,
            "cwd": "${workspaceFolder}",
            "environment": [],
            "externalConsole": false,
            "MIMode": "gdb",
            "miDebuggerPath": "/usr/bin/gdb",
            "setupCommands": [
                {
                    "description": "Pretty-printing STL",
                    "text": "-enable-pretty-printing",
                    "ignoreFailures": true
                },
                {
                    "description": "Désactiver la pagination GDB",
                    "text": "set pagination off",
                    "ignoreFailures": true
                },
                {
                    "description": "Suivre les processus enfants sur fork",
                    "text": "set follow-fork-mode child",
                    "ignoreFailures": true
                }
            ],
            "preLaunchTask": "CMake: build"
        }
    ]
}
```

Explication des champs clés :

**`type`** — `"cppdbg"` indique que cette configuration utilise l'adaptateur de débogage de l'extension C/C++ Microsoft.

**`request`** — `"launch"` démarre le programme sous le debugger. L'alternative `"attach"` connecte le debugger à un processus déjà en cours d'exécution (utile pour les services et daemons).

**`program`** — le chemin vers l'exécutable à déboguer. La variable `${command:cmake.launchTargetPath}` est fournie par CMake Tools et résout automatiquement vers l'exécutable de la cible CMake sélectionnée dans la barre de statut. Si vous n'utilisez pas CMake Tools, spécifiez le chemin directement : `"${workspaceFolder}/build/mon_programme"`.

**`args`** — un tableau de chaînes contenant les arguments de ligne de commande à passer au programme. Par exemple : `["--input", "data.txt", "--verbose"]`.

**`cwd`** — le répertoire de travail du programme. `${workspaceFolder}` est la racine du projet — c'est généralement le bon choix.

**`externalConsole`** — `false` redirige stdin/stdout vers le terminal intégré de VS Code. Mettez `true` si votre programme a besoin d'un terminal interactif complet (par exemple pour `ncurses`).

**`MIMode`** — `"gdb"` ou `"lldb"`. Spécifie le debugger à utiliser.

**`miDebuggerPath`** — chemin absolu vers le binaire du debugger. Utile si vous avez plusieurs versions de GDB installées.

**`setupCommands`** — commandes GDB exécutées avant le lancement du programme. La plus importante est `-enable-pretty-printing`, qui active l'affichage formaté des conteneurs STL (voir ci-dessous).

**`preLaunchTask`** — tâche VS Code à exécuter avant le lancement du debugger. `"CMake: build"` compile le projet automatiquement avant chaque session de debug, garantissant que vous déboguez le binaire le plus récent.

### Configuration avec LLDB

Si vous préférez LLDB (le debugger du projet LLVM) :

```json
{
    "name": "Debug (LLDB)",
    "type": "cppdbg",
    "request": "launch",
    "program": "${command:cmake.launchTargetPath}",
    "args": [],
    "stopAtEntry": false,
    "cwd": "${workspaceFolder}",
    "environment": [],
    "externalConsole": false,
    "MIMode": "lldb",
    "miDebuggerPath": "/usr/bin/lldb-mi-20",
    "preLaunchTask": "CMake: build"
}
```

> ⚠️ LLDB via l'extension C/C++ Microsoft nécessite `lldb-mi` (l'interface MI de LLDB). Sur certaines installations Ubuntu, ce binaire peut ne pas être disponible directement. GDB est le choix le plus fiable sur Linux.

### Configurations multiples

Un projet a souvent besoin de plusieurs configurations de débogage : le programme principal, un outil CLI auxiliaire, les tests unitaires. `launch.json` supporte un tableau de configurations :

```json
{
    "version": "0.2.0",
    "configurations": [
        {
            "name": "Debug — Programme principal",
            "type": "cppdbg",
            "request": "launch",
            "program": "${workspaceFolder}/build/mon_programme",
            "args": ["--config", "dev.yaml"],
            "cwd": "${workspaceFolder}",
            "MIMode": "gdb",
            "setupCommands": [
                { "text": "-enable-pretty-printing", "ignoreFailures": true }
            ],
            "preLaunchTask": "CMake: build"
        },
        {
            "name": "Debug — Tests unitaires",
            "type": "cppdbg",
            "request": "launch",
            "program": "${workspaceFolder}/build/tests/unit_tests",
            "args": ["--gtest_filter=MonTest*"],
            "cwd": "${workspaceFolder}",
            "MIMode": "gdb",
            "setupCommands": [
                { "text": "-enable-pretty-printing", "ignoreFailures": true }
            ],
            "preLaunchTask": "CMake: build"
        },
        {
            "name": "Attach — Processus en cours",
            "type": "cppdbg",
            "request": "attach",
            "program": "${workspaceFolder}/build/mon_service",
            "processId": "${command:pickProcess}",
            "MIMode": "gdb",
            "setupCommands": [
                { "text": "-enable-pretty-printing", "ignoreFailures": true }
            ]
        }
    ]
}
```

La configuration « Attach » utilise `"request": "attach"` et `${command:pickProcess}`, qui ouvre un sélecteur de processus dans VS Code. Utile pour déboguer un serveur ou un daemon déjà lancé.

Sélectionnez la configuration active dans le menu déroulant de la vue « Run and Debug » avant d'appuyer sur `F5`.

### Pretty-printing des conteneurs STL

Sans pretty-printing, un `std::vector<int>` avec les valeurs `{10, 20, 30}` apparaît dans le debugger comme une structure interne complexe avec des pointeurs `_M_impl`, `_M_start`, `_M_finish` — illisible.

Avec `-enable-pretty-printing`, le même vecteur s'affiche comme :

```
vec = std::vector of length 3, capacity 4 = {10, 20, 30}
```

Cette fonctionnalité repose sur des scripts Python fournis par libstdc++ (pour GDB) qui savent interpréter les structures internes des conteneurs STL. Elle est activée automatiquement par le `setupCommand` dans notre `launch.json`.

Les types suivants bénéficient du pretty-printing : `std::vector`, `std::map`, `std::unordered_map`, `std::set`, `std::string`, `std::optional`, `std::variant`, `std::shared_ptr`, `std::unique_ptr`, `std::pair`, `std::tuple`, et la plupart des autres conteneurs et utilitaires de la STL.

## Débogage dans CLion

### Configuration automatique

L'un des avantages majeurs de CLion est que le débogage fonctionne **sans configuration manuelle**. CLion détecte automatiquement les targets CMake, associe chaque exécutable à une configuration de debug, et utilise GDB (ou LLDB selon vos préférences) en arrière-plan.

Pour déboguer un programme :

1. Sélectionnez la cible dans le menu déroulant de la barre d'outils (en haut à droite).
2. Cliquez sur l'icône « Debug » (insecte vert) ou appuyez sur `Shift+F9`.
3. CLion compile le projet (si nécessaire), lance le programme sous GDB et affiche l'interface de débogage.

### Choisir le debugger

Par défaut, CLion utilise GDB sur Linux. Pour changer :

`File > Settings > Build, Execution, Deployment > Toolchains` — sélectionnez votre toolchain et modifiez le champ « Debugger » pour pointer vers GDB ou LLDB.

CLion intègre son propre GDB « bundled » (une version spécifiquement patchée pour une meilleure intégration). Vous pouvez utiliser cette version ou pointer vers le GDB système — les deux fonctionnent.

### Configurations de Run/Debug

CLion crée automatiquement des configurations pour chaque target CMake. Vous pouvez les personnaliser via `Run > Edit Configurations...` :

- **Program arguments** — arguments de ligne de commande.
- **Working directory** — répertoire de travail.
- **Environment variables** — variables d'environnement additionnelles.
- **Before launch** — actions à exécuter avant le lancement (compilation, scripts, etc.).

Pour les tests Google Test, CLion crée automatiquement des configurations de test avec une interface dédiée (barre verte/rouge pour chaque test, filtre par nom, réexécution des tests échoués).

### Forces du debugger CLion

CLion offre une expérience de débogage particulièrement soignée :

**Affichage des conteneurs STL** — CLion intègre ses propres renderers pour les types STL, en plus du pretty-printing GDB. Le résultat est un affichage structuré et navigable (arborescence dépliable pour les maps, indexation visuelle pour les vectors).

**Évaluation d'expressions** — pendant une pause, vous pouvez évaluer n'importe quelle expression C++ via `Alt+F8` ou directement dans la fenêtre « Evaluate ». CLion compile et exécute l'expression dans le contexte courant.

**Inline variable values** — les valeurs des variables s'affichent directement à côté du code source, en grisé, à chaque ligne. Pas besoin de regarder le panneau latéral.

**Memory view** — inspection hexadécimale de la mémoire brute, utile pour le débogage bas niveau.

## Débogage dans Neovim

### Architecture : DAP (Debug Adapter Protocol)

Neovim utilise le **Debug Adapter Protocol (DAP)**, un protocole standardisé (similaire en philosophie au LSP, mais pour le débogage) pour communiquer avec les debuggers. Les deux plugins essentiels sont :

- **nvim-dap** — le client DAP pour Neovim.
- **nvim-dap-ui** — une interface utilisateur qui affiche les variables, la pile d'appels, les breakpoints et la console dans des panneaux Neovim.

### Installation des plugins

Avec **lazy.nvim** (gestionnaire de plugins recommandé) :

```lua
-- ~/.config/nvim/lua/plugins/dap.lua
return {
    {
        "mfussenegger/nvim-dap",
        dependencies = {
            "rcarriga/nvim-dap-ui",
            "nvim-neotest/nvim-nio",
        },
        config = function()
            local dap = require("dap")
            local dapui = require("dapui")

            dapui.setup()

            -- Ouvrir/fermer l'UI automatiquement
            dap.listeners.after.event_initialized["dapui_config"] = function()
                dapui.open()
            end
            dap.listeners.before.event_terminated["dapui_config"] = function()
                dapui.close()
            end
            dap.listeners.before.event_exited["dapui_config"] = function()
                dapui.close()
            end

            -- Configuration GDB pour C/C++
            dap.adapters.cppdbg = {
                id = "cppdbg",
                type = "executable",
                command = "/path/to/cpptools/debugAdapters/bin/OpenDebugAD7",
                -- ou utiliser codelldb (voir ci-dessous)
            }

            dap.configurations.cpp = {
                {
                    name = "Debug programme",
                    type = "cppdbg",
                    request = "launch",
                    program = function()
                        return vim.fn.input(
                            "Chemin de l'exécutable: ",
                            vim.fn.getcwd() .. "/build/",
                            "file"
                        )
                    end,
                    cwd = "${workspaceFolder}",
                    stopAtEntry = false,
                    setupCommands = {
                        {
                            text = "-enable-pretty-printing",
                            description = "Pretty-printing STL",
                            ignoreFailures = true,
                        },
                    },
                },
            }
        end,
    },
}
```

### Alternative recommandée : codelldb

L'adaptateur **codelldb** (basé sur LLDB) est souvent préféré à `cppdbg` dans Neovim car il est plus simple à installer et ne dépend pas de l'extension Microsoft :

```lua
dap.adapters.codelldb = {
    type = "server",
    port = "${port}",
    executable = {
        command = "/usr/local/bin/codelldb",  -- ajuster le chemin
        args = { "--port", "${port}" },
    },
}

dap.configurations.cpp = {
    {
        name = "Debug (codelldb)",
        type = "codelldb",
        request = "launch",
        program = function()
            return vim.fn.input(
                "Exécutable: ",
                vim.fn.getcwd() .. "/build/",
                "file"
            )
        end,
        cwd = "${workspaceFolder}",
        stopOnEntry = false,
    },
}
```

codelldb est installable via le gestionnaire de paquets Mason (`MasonInstall codelldb`) si vous utilisez **mason.nvim**, ou téléchargeable depuis les [releases GitHub](https://github.com/vadimcn/codelldb/releases).

### Raccourcis clavier

La configuration des raccourcis est essentielle pour un workflow de débogage fluide dans Neovim :

```lua
-- Raccourcis DAP
vim.keymap.set("n", "<F5>",  function() require("dap").continue() end,
    { desc = "Debug: Continue / Start" })
vim.keymap.set("n", "<F10>", function() require("dap").step_over() end,
    { desc = "Debug: Step Over" })
vim.keymap.set("n", "<F11>", function() require("dap").step_into() end,
    { desc = "Debug: Step Into" })
vim.keymap.set("n", "<F12>", function() require("dap").step_out() end,
    { desc = "Debug: Step Out" })
vim.keymap.set("n", "<leader>b", function() require("dap").toggle_breakpoint() end,
    { desc = "Debug: Toggle Breakpoint" })
vim.keymap.set("n", "<leader>B", function()
    require("dap").set_breakpoint(vim.fn.input("Condition: "))
end, { desc = "Debug: Conditional Breakpoint" })  
vim.keymap.set("n", "<leader>dr", function() require("dap").repl.open() end,  
    { desc = "Debug: Open REPL" })
vim.keymap.set("n", "<leader>du", function() require("dapui").toggle() end,
    { desc = "Debug: Toggle UI" })
```

### Distributions Neovim préconfigurées

Si vous utilisez **LazyVim**, le support DAP est disponible via l'extra `dap` :

```lua
-- Dans votre configuration LazyVim
{ import = "lazyvim.plugins.extras.dap.core" },
{ import = "lazyvim.plugins.extras.dap.nlua" },
```

LazyVim configure automatiquement nvim-dap, nvim-dap-ui et les raccourcis clavier. Il ne reste plus qu'à définir l'adaptateur et la configuration C++ comme ci-dessus.

## Fonctionnalités de débogage communes

Quel que soit l'éditeur, les fonctionnalités de débogage suivantes sont disponibles via GDB/LLDB. Elles sont présentées ici dans leur forme IDE (VS Code comme référence), mais les équivalents existent dans CLion et Neovim.

### Breakpoints

**Breakpoint simple** — cliquez dans la marge gauche à côté d'un numéro de ligne (ou appuyez sur `F9`). Un point rouge apparaît. Le programme s'arrêtera à cette ligne.

**Breakpoint conditionnel** — clic droit sur un breakpoint existant → « Edit Breakpoint » → saisissez une condition C++ (par exemple `i > 100` ou `name == "error"`). Le programme ne s'arrête que si la condition est vraie. Très utile pour éviter de s'arrêter des milliers de fois dans une boucle.

**Logpoint** — un breakpoint qui ne stoppe pas l'exécution mais affiche un message dans la console. Équivalent d'un `printf` de debug sans modifier le code. Syntaxe VS Code : clic droit dans la marge → « Add Logpoint » → `"i = {i}, value = {vec[i]}"`.

**Breakpoint de fonction** — s'arrêter à l'entrée d'une fonction par son nom, sans connaître le fichier ou la ligne. Dans VS Code : vue « Breakpoints » → icône « + » → saisir le nom de la fonction.

### Contrôle de l'exécution

| Action | VS Code | CLion | Neovim (DAP) | GDB |
|---|---|---|---|---|
| Continuer | `F5` | `F9` | `<F5>` | `continue` |
| Step Over (ligne suivante) | `F10` | `F8` | `<F10>` | `next` |
| Step Into (entrer dans la fonction) | `F11` | `F7` | `<F11>` | `step` |
| Step Out (sortir de la fonction) | `Shift+F11` | `Shift+F8` | `<F12>` | `finish` |
| Arrêter | `Shift+F5` | `Ctrl+F2` | `:DapTerminate` | `quit` |
| Redémarrer | `Ctrl+Shift+F5` | `Ctrl+F5` | — | `run` |

### Inspection des variables

Pendant une pause, trois panneaux permettent d'inspecter l'état du programme :

**Variables locales** — toutes les variables de la portée courante avec leur valeur actuelle. Les conteneurs STL sont affichés sous forme lisible grâce au pretty-printing.

**Watch (surveillance)** — un panneau où vous ajoutez manuellement des expressions à surveiller. Par exemple : `vec.size()`, `ptr != nullptr`, `matrix[i][j]`. Les expressions sont réévaluées à chaque arrêt.

**Call Stack (pile d'appels)** — la liste des fonctions en cours d'exécution, de la plus récente (en haut) à `main()` (en bas). Cliquez sur un frame pour voir le code source et les variables locales de cette fonction.

### Évaluation d'expressions à la volée

Pendant une pause du debugger, vous pouvez évaluer n'importe quelle expression C++ :

- **VS Code** : panneau « Debug Console » en bas. Tapez directement une expression C++ et appuyez sur Entrée.
- **CLion** : `Alt+F8` ouvre la fenêtre « Evaluate Expression ».
- **Neovim** : `<leader>dr` ouvre le REPL DAP.

Exemples d'expressions évaluables :

```
vec.size()
*ptr
static_cast<int>(status)  
std::distance(it_begin, it_end)  
```

> ⚠️ L'évaluation d'expressions dans GDB a des limitations : les appels à des fonctions template non instanciées ou à des fonctions inlinées peuvent échouer. Les expressions simples (accès aux variables, arithmétique, déréférencement de pointeurs, appels de méthodes) fonctionnent de manière fiable.

## Débogage avec les Sanitizers

Les sanitizers (AddressSanitizer, UndefinedBehaviorSanitizer, etc.) sont des outils de détection d'erreurs mémoire qui s'intègrent parfaitement avec le débogage IDE. Quand un sanitizer détecte une erreur (buffer overflow, use-after-free, undefined behavior), il affiche un diagnostic détaillé et peut arrêter le programme au point exact de l'erreur.

Pour activer ASan (AddressSanitizer) dans votre build de debug :

```cmake
# CMakeLists.txt — Option pour activer les sanitizers en debug
if(CMAKE_BUILD_TYPE STREQUAL "Debug")
    add_compile_options(-fsanitize=address,undefined -fno-omit-frame-pointer)
    add_link_options(-fsanitize=address,undefined)
endif()
```

Quand vous déboguez un programme compilé avec ASan et qu'une erreur mémoire se produit, le programme s'arrête et le sanitizer affiche un rapport dans la console du debugger. Vous pouvez alors examiner la pile d'appels pour identifier l'origine du problème.

Pour que GDB s'arrête au moment de l'erreur détectée par ASan (plutôt que de simplement afficher le rapport et quitter), ajoutez un breakpoint sur la fonction de rapport d'erreur :

```json
// Dans setupCommands de launch.json
{
    "text": "break __asan_report_error",
    "description": "Arrêt sur erreur ASan",
    "ignoreFailures": true
}
```

Nous reviendrons en détail sur les sanitizers aux sections 5.5 et 29.4.

## Débogage d'un core dump

Quand un programme crash (segfault, abort, etc.) en dehors du debugger, le système peut générer un **core dump** — un fichier contenant l'état complet de la mémoire du processus au moment du crash. Ce fichier permet de faire un **débogage post-mortem** : analyser le crash après coup, sans avoir à le reproduire.

### Activer la génération de core dumps

```bash
# Vérifier la limite actuelle
ulimit -c
# Si "0", les core dumps sont désactivés

# Activer (pour la session courante)
ulimit -c unlimited
```

### Ouvrir un core dump dans VS Code

Ajoutez une configuration dans `launch.json` :

```json
{
    "name": "Post-mortem (core dump)",
    "type": "cppdbg",
    "request": "launch",
    "program": "${workspaceFolder}/build/mon_programme",
    "coreDumpPath": "${input:corePath}",
    "cwd": "${workspaceFolder}",
    "MIMode": "gdb",
    "setupCommands": [
        { "text": "-enable-pretty-printing", "ignoreFailures": true }
    ]
}
```

Avec une variable d'entrée pour sélectionner le fichier :

```json
"inputs": [
    {
        "id": "corePath",
        "type": "promptString",
        "description": "Chemin vers le fichier core dump",
        "default": "/tmp/core"
    }
]
```

VS Code ouvre le core dump, affiche la pile d'appels au moment du crash, et vous permet d'inspecter les variables exactement comme dans une session de débogage normale — sauf que le programme n'est pas en cours d'exécution.

### Ouvrir un core dump dans CLion

`Run > Open Core Dump...` — CLion propose un dialogue pour sélectionner l'exécutable et le core dump. L'interface de débogage s'ouvre avec la pile d'appels et les variables au moment du crash.

Le chapitre 29.3 (Core dumps et post-mortem debugging) approfondira ce sujet.

## Résolution de problèmes courants

### « No symbols loaded » ou variables affichées comme « optimized out »

**Cause :** le programme a été compilé sans `-g` ou avec des optimisations (`-O2`, `-O3`).

**Solution :** vérifiez le build type CMake (`Debug`) et recompilez :

```bash
cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Debug  
cmake --build build --clean-first  
```

### « Permission denied » au lancement du debugger

**Cause :** sur certaines configurations Ubuntu, `ptrace` (le syscall utilisé par GDB pour contrôler le programme débogué) est restreint par défaut.

**Solution :**

```bash
# Vérifier la valeur actuelle
cat /proc/sys/kernel/yama/ptrace_scope
# Si "1" ou plus, ptrace est restreint

# Autoriser temporairement (jusqu'au prochain reboot)
echo 0 | sudo tee /proc/sys/kernel/yama/ptrace_scope

# Autoriser de manière permanente
echo "kernel.yama.ptrace_scope = 0" | sudo tee /etc/sysctl.d/10-ptrace.conf  
sudo sysctl --system  
```

### Le debugger ne s'arrête pas aux breakpoints

**Causes possibles :**

- Le fichier source affiché dans l'éditeur ne correspond pas au binaire compilé (le code a été modifié mais pas recompilé). Vérifiez que `preLaunchTask` est configuré.
- Le breakpoint est dans du code qui n'est jamais exécuté (branche conditionnelle, template non instancié).
- L'optimisation a éliminé le code. Compilez en `-O0`.

### Pretty-printing ne fonctionne pas

**Cause :** les scripts Python de pretty-printing de libstdc++ ne sont pas trouvés par GDB.

**Solution :** vérifiez que `libstdc++` est installée avec ses scripts de debug :

```bash
sudo apt install libstdc++-15-dev
```

Si le problème persiste, ajoutez le chemin manuellement dans les `setupCommands` :

```json
{
    "text": "python import sys; sys.path.insert(0, '/usr/share/gcc/python')",
    "ignoreFailures": true
}
```

## Ce qu'il faut retenir

- **Compilez en `Debug`** (`CMAKE_BUILD_TYPE=Debug`, qui active `-g -O0`) pour une expérience de débogage optimale. Les optimisations rendent le pas-à-pas imprévisible.
- **VS Code** : le fichier `.vscode/launch.json` est le centre de la configuration. Utilisez `${command:cmake.launchTargetPath}` pour pointer automatiquement vers la bonne cible CMake, et `preLaunchTask` pour recompiler avant chaque session.
- **CLion** : le débogage fonctionne sans configuration. Sélectionnez la cible et appuyez sur `Shift+F9`.
- **Neovim** : nvim-dap + nvim-dap-ui fournissent une expérience complète. L'adaptateur **codelldb** est le plus simple à configurer.
- **Pretty-printing** (`-enable-pretty-printing`) est indispensable : il transforme les types STL de structures cryptiques en données lisibles.
- **Les breakpoints conditionnels** et les **logpoints** sont des outils puissants qui évitent de modifier le code source pour ajouter des traces de débogage.
- **Les sanitizers** (ASan, UBSan) se combinent avec le debugger IDE pour détecter et localiser les erreurs mémoire. Ajoutez `break __asan_report_error` dans les setupCommands pour un arrêt automatique.
- **Les core dumps** permettent le débogage post-mortem : analysez un crash après coup, sans le reproduire. Activez-les avec `ulimit -c unlimited`.
