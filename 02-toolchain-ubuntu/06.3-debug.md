# 2.6.3 Debug : `-g`, `-ggdb3`

> **Niveau** : Débutant  
> **Prérequis** : Section 2.5.2 (Inspection des binaires), Section 2.6.2 (Optimisation)  
> **Objectifs** : Comprendre ce que les informations de débogage contiennent, maîtriser les niveaux de `-g`, mesurer leur impact sur la taille du binaire, et savoir combiner débogage et optimisation selon le contexte.

---

En section 2.6.2, nous avons mentionné que `-g` ajoute des "informations de débogage" au binaire, et que cette option est indépendante des niveaux d'optimisation. Mais que contiennent exactement ces informations ? Comment un outil comme GDB parvient-il à afficher le nom de vos variables, le numéro de ligne du source, et la pile d'appels — alors qu'au niveau du processeur, il n'y a que des adresses mémoire et des registres ?

La réponse tient en un mot : **DWARF**. C'est le format standard de débogage sur Linux, et comprendre ce qu'il contient permet de faire des choix éclairés entre les différents niveaux de `-g`, de savoir pourquoi un binaire avec débogage est plus gros, et de tirer le meilleur parti du débogueur.

---

## Ce que contiennent les informations de débogage

Quand vous compilez avec `-g`, le compilateur inscrit dans le binaire ELF un ensemble de **métadonnées** qui décrivent la structure de votre programme source. Ces métadonnées n'affectent ni le code machine généré ni les performances à l'exécution — elles sont stockées dans des sections ELF séparées que le processeur n'a jamais besoin de lire. Seuls les outils de débogage (GDB, LLDB) et de profiling (`perf`, Valgrind) les consultent.

Ces métadonnées couvrent quatre catégories d'informations.

La **correspondance ligne/adresse** associe chaque adresse du code machine à la ligne du fichier source qui l'a générée. C'est ce qui permet à GDB d'afficher `Breakpoint 1 at 0x1234: file main.cpp, line 15` et de faire avancer l'exécution ligne par ligne avec `next` et `step`.

Les **descriptions de variables** indiquent pour chaque variable son nom, son type, et son emplacement à l'exécution — registre, offset sur la pile, ou expression DWARF complexe pour les cas optimisés. C'est ce qui permet à GDB de répondre à `print rayon` avec `$1 = 5.0` au lieu de vous obliger à inspecter manuellement le registre `xmm0`.

Les **descriptions de types** décrivent la structure de chaque type utilisé : les champs d'une `struct`, les membres d'une classe, les types des paramètres de chaque fonction, les hiérarchies d'héritage. C'est ce qui permet à GDB d'afficher un `std::vector<int>` comme `{1, 2, 3}` au lieu d'un bloc d'octets bruts.

Les **informations de pile d'appels** (*call frame information*, CFI) décrivent comment reconstituer la pile d'appels à n'importe quel point de l'exécution. Elles indiquent où chaque fonction a sauvegardé les registres et comment remonter au frame de l'appelant. C'est ce qui permet à GDB d'afficher un `backtrace` lisible.

---

## Le format DWARF

DWARF (*Debugging With Attributed Record Formats*) est le standard de débogage utilisé sur les systèmes Unix et Linux. Il en est à sa version 5, mais la version 4 reste la plus couramment produite par défaut. Les informations DWARF sont stockées dans des sections ELF dédiées :

```bash
g++ -g main.cpp -o main_debug  
readelf -S main_debug | grep debug  
```

```
  [28] .debug_aranges    PROGBITS   ...
  [29] .debug_info       PROGBITS   ...
  [30] .debug_abbrev     PROGBITS   ...
  [31] .debug_line       PROGBITS   ...
  [32] .debug_str        PROGBITS   ...
  [33] .debug_line_str   PROGBITS   ...
  [34] .debug_rnglists   PROGBITS   ...
```

Chaque section a un rôle spécifique :

| Section | Contenu |
|---------|---------|
| `.debug_info` | Description des types, variables, fonctions, scopes |
| `.debug_line` | Table de correspondance adresse ↔ ligne source |
| `.debug_abbrev` | Abréviations utilisées dans `.debug_info` (compression) |
| `.debug_str` | Table des chaînes (noms de variables, fichiers, types) |
| `.debug_aranges` | Index des plages d'adresses par unité de compilation |
| `.debug_rnglists` | Plages d'adresses (DWARF 5) |
| `.debug_frame` | Informations de déroulement de pile (CFI) |

Ces sections portent le flag `PROGBITS` mais **pas** le flag `ALLOC` : elles existent dans le fichier sur disque mais ne sont **pas chargées en mémoire** à l'exécution. Elles n'ont donc aucun impact sur la mémoire utilisée par votre programme en fonctionnement.

On peut inspecter le contenu DWARF brut avec `readelf` :

```bash
# Afficher les unités de compilation
readelf -wi main_debug | head -50

# Afficher la table de correspondance ligne/adresse
readelf -wl main_debug | head -30
```

La sortie est verbeuse et technique — c'est la matière première que GDB interprète pour vous. En pratique, vous n'aurez que très rarement besoin de lire le DWARF brut, mais savoir qu'il existe et ce qu'il contient éclaire le fonctionnement du débogueur.

## Les niveaux de `-g`

GCC propose plusieurs niveaux d'informations de débogage, du plus minimal au plus complet. Le nombre après `-g` contrôle la quantité de métadonnées générées.

### `-g0` : Aucune information de débogage

```bash
g++ -g0 main.cpp -o main
```

Équivalent à ne pas spécifier `-g` du tout. Aucune section `.debug_*` n'est produite. Le débogueur peut toujours afficher les adresses et désassembler le code, mais il ne peut pas faire la correspondance avec le source, afficher les noms de variables, ni produire un backtrace lisible (sauf si la table des symboles n'a pas été strippée).

### `-g1` : Informations minimales

```bash
g++ -g1 main.cpp -o main
```

Ce niveau inclut les descriptions de fonctions et la correspondance ligne/adresse, mais **pas** les descriptions de variables locales ni les informations de type détaillées. GDB peut afficher un backtrace avec les noms de fonctions et positionner un breakpoint sur une ligne, mais `print variable` échouera pour les variables locales.

C'est un compromis rarement utilisé, sauf dans les cas où la taille du binaire est critique mais où l'on souhaite quand même obtenir des backtraces lisibles dans les core dumps.

### `-g` (sans chiffre) / `-g2` : Niveau standard

```bash
g++ -g main.cpp -o main
# Équivalent à :
g++ -g2 main.cpp -o main
```

C'est le niveau par défaut et le plus couramment utilisé. Il inclut les descriptions complètes des fonctions, variables locales, types, et la correspondance ligne/adresse. GDB peut afficher les variables avec `print`, naviguer dans les structures et classes, et faire de l'évaluation d'expressions.

Ce niveau est suffisant pour la grande majorité des scénarios de débogage.

### `-g3` : Informations maximales

```bash
g++ -g3 main.cpp -o main
```

Ce niveau ajoute à `-g2` les informations sur les **macros préprocesseur**. Sans `-g3`, les macros sont invisibles dans le débogueur car elles ont été substituées au préprocesseur, bien avant la génération des métadonnées DWARF. Avec `-g3`, le compilateur enregistre chaque `#define` rencontré, ce qui permet à GDB de les évaluer :

```cpp
#define MAX_BUFFER 1024
#define VERSION_MAJOR 2
```

Avec `-g3`, dans GDB :

```
(gdb) print MAX_BUFFER
$1 = 1024
(gdb) print VERSION_MAJOR
$2 = 2
```

Avec `-g` (sans `3`), la même commande produirait :

```
(gdb) print MAX_BUFFER
No symbol "MAX_BUFFER" in current context.
```

Le surcoût en taille de `-g3` par rapport à `-g2` est notable, car la table des macros peut être volumineuse (chaque header système inclus contient des centaines de `#define`). Ce niveau est utile quand vous déboguez du code qui repose fortement sur les macros, ce qui est courant dans du code C hérité ou des librairies système.

### Comparaison de taille

```bash
g++ -g0 main.cpp -o main_g0  
g++ -g1 main.cpp -o main_g1  
g++ -g  main.cpp -o main_g2  
g++ -g3 main.cpp -o main_g3  

ls -lh main_g0 main_g1 main_g2 main_g3
```

Un résultat typique sur notre programme simple :

```
 22K main_g0
 36K main_g1
 92K main_g2
196K main_g3
```

Le binaire avec `-g3` est presque 10 fois plus gros que sans informations de débogage. Sur un projet de taille réelle avec de nombreux fichiers source, les sections `.debug_*` peuvent représenter **plusieurs centaines de Mo**, voire des Go. C'est pourquoi on sépare souvent les symboles de débogage dans un fichier dédié en production (voir plus bas).

> 💡 *Ces tailles gonflées ne concernent que le fichier sur disque. Les sections de débogage ne sont pas chargées en mémoire à l'exécution et n'affectent pas l'empreinte mémoire du processus.*

## `-ggdb` : Spécifique à GDB

```bash
g++ -ggdb main.cpp -o main  
g++ -ggdb3 main.cpp -o main  
```

L'option `-ggdb` est une variante de `-g` qui demande au compilateur de produire les informations de débogage dans le format le **plus expressif possible pour GDB**, quitte à utiliser des extensions spécifiques que d'autres débogueurs ne comprendraient pas.

En pratique, sur Linux avec GCC ou Clang récents, la différence entre `-g` et `-ggdb` est minime. Les deux produisent du DWARF, et GDB traite les deux de manière identique dans l'immense majorité des cas. La distinction était plus significative dans les versions historiques de GCC qui supportaient d'autres formats de débogage (STABS, COFF).

`-ggdb3` est simplement la combinaison de `-ggdb` (format orienté GDB) et du niveau 3 (informations maximales, incluant les macros). C'est le niveau recommandé pour le **profil de débogage approfondi** mentionné en section 2.6 :

```bash
g++ -std=c++23 -Wall -Wextra -ggdb3 -O0 -fsanitize=address,undefined main.cpp -o main
```

## `-gdwarf-4` et `-gdwarf-5` : Choisir la version DWARF

Par défaut, GCC 15 produit du DWARF version 5 et Clang 20 produit du DWARF version 4 (les défauts évoluent avec les versions des compilateurs). Vous pouvez forcer une version spécifique :

```bash
g++ -g -gdwarf-4 main.cpp -o main  
g++ -g -gdwarf-5 main.cpp -o main  
```

DWARF 5 apporte des améliorations par rapport à DWARF 4 : des sections plus compactes grâce à de meilleurs schémas de compression, un support amélioré pour le code optimisé, et de nouvelles fonctionnalités comme les index `.debug_names` qui accélèrent les recherches du débogueur.

En pratique, la version DWARF importe peu pour un usage courant de GDB. Elle peut en revanche jouer un rôle dans des scénarios avancés : compatibilité avec des outils de débogage anciens, interopérabilité entre compilateurs différents, ou réduction de la taille des informations de débogage sur des projets massifs.

```bash
# Vérifier la version DWARF produite
readelf -wi main_debug | head -5
```

La première ligne de la sortie indique `Compilation Unit @ offset 0x0: ... version = 5` (ou 4).

## Combiner `-g` et `-O` : Le tableau complet

En section 2.6.2, nous avons abordé l'interaction entre optimisation et débogage. Détaillons les combinaisons et leurs implications pratiques.

### `-O0 -g` : Le confort maximal

```bash
g++ -O0 -g main.cpp -o main
```

C'est la combinaison de **développement par défaut**. Le code machine est une traduction fidèle du source, et les informations DWARF sont précises. Dans GDB, tout fonctionne comme attendu : les breakpoints s'arrêtent sur la bonne ligne, chaque variable est inspectable, l'exécution suit le flux du source.

### `-Og -g` : Le compromis intelligent

```bash
g++ -Og -g main.cpp -o main
```

Les optimisations activées par `-Og` sont choisies pour ne pas dégrader l'expérience de débogage. Les variables restent accessibles (pas d'"optimized out"), les breakpoints se comportent normalement, et le flux d'exécution reste lisible. Le gain de performance par rapport à `-O0` (20 à 50 %) peut rendre les tests interactifs plus confortables sur du code numériquement intensif.

### `-O2 -g` : La production débogable

```bash
g++ -O2 -g main.cpp -o main
```

C'est la combinaison **recommandée pour les builds de production**. Le code est optimisé pour la performance, et les informations de débogage sont présentes pour l'analyse post-mortem. Mais le débogage interactif est dégradé par les effets de l'optimisation.

Les informations DWARF produites avec `-O2` sont intrinsèquement moins précises que celles produites avec `-O0`. Quand le compilateur élimine une variable, la fusionne avec une autre, ou la maintient dans un registre qui est ensuite réutilisé, les métadonnées DWARF doivent exprimer des situations complexes : "cette variable est dans le registre `rax` entre les adresses 0x1000 et 0x1020, puis dans `[rbp-8]` entre 0x1020 et 0x1040, puis n'existe plus". GDB fait de son mieux pour interpréter ces descriptions, mais le résultat est parfois un `<optimized out>` quand la variable n'est accessible à aucun emplacement au point d'arrêt courant.

Voyons un exemple concret. Considérons cette fonction :

```cpp
int compute(int a, int b, int c) {
    int x = a + b;
    int y = x * c;
    int z = y - a;
    return z;
}
```

En `-O0 -g`, chaque variable (`x`, `y`, `z`) est stockée sur la pile et inspectable à tout moment. En `-O2 -g`, le compilateur simplifie la chaîne de calcul : le résultat est produit directement dans un registre sans que `x`, `y` et `z` existent en tant que variables distinctes. GDB répondra `<optimized out>` pour chacune d'elles — le résultat final est correct, mais les étapes intermédiaires n'existent plus.

### `-O3 -g` et `-Ofast -g` : Possibles mais difficiles

Ces combinaisons sont valides mais rarement utiles pour le débogage interactif. Le niveau d'optimisation est si élevé que la correspondance avec le source est très dégradée. Elles restent utiles pour le profiling (obtenir des noms de fonctions dans les flamegraphs) et l'analyse de core dumps en dernier recours.

### Récapitulatif

| Combinaison | Usage | Débogage interactif | Analyse post-mortem |
|-------------|-------|--------------------|--------------------|
| `-O0 -g` | Développement | Excellent | Excellent |
| `-Og -g` | Développement (code lent en -O0) | Très bon | Très bon |
| `-O2 -g` | Production | Limité | Bon |
| `-O3 -g` | Production haute performance | Très limité | Correct |

## Séparer les symboles de débogage

Sur un projet réel, les informations de débogage peuvent augmenter la taille du binaire d'un facteur 5 à 20. Distribuer un binaire de 500 Mo en production alors que seuls 50 Mo sont du code réel n'est ni pratique ni souhaitable. La solution est de **séparer** les symboles dans un fichier dédié.

### Extraction avec `objcopy`

```bash
# Compiler avec symboles de débogage
g++ -O2 -g main.cpp mathutils.cpp -o main

# Extraire les symboles dans un fichier séparé
objcopy --only-keep-debug main main.debug

# Stripper l'exécutable
strip --strip-debug main

# Créer un lien entre les deux fichiers
objcopy --add-gnu-debuglink=main.debug main
```

Après ces opérations :

```bash
ls -lh main main.debug
```

```
 22K main
128K main.debug
```

L'exécutable `main` est léger et ne contient aucune section `.debug_*`. Le fichier `main.debug` contient uniquement les métadonnées de débogage. Le lien `gnu-debuglink` inscrit dans `main` le nom du fichier et un checksum CRC32 qui permet à GDB de retrouver et valider le fichier de symboles.

### Utilisation avec GDB

Quand vous chargez le binaire strippé dans GDB, celui-ci cherche automatiquement le fichier de symboles :

```bash
gdb ./main
```

```
Reading symbols from ./main...  
Reading symbols from /chemin/vers/main.debug...  
```

GDB cherche le fichier `.debug` dans plusieurs emplacements : le répertoire courant, un sous-répertoire `.debug/` à côté de l'exécutable, et le répertoire global `/usr/lib/debug/`. Ce dernier est le répertoire utilisé par les paquets `-dbgsym` d'Ubuntu :

```bash
# Installer les symboles de débogage de la libc
sudo apt install libc6-dbg
```

Après installation, GDB peut afficher des backtraces détaillées même dans le code de la libc, avec les noms de fonctions, les fichiers source, et les numéros de ligne.

### Build ID

Les binaires modernes incluent un **Build ID** — un hash unique inscrit dans une section `.note.gnu.build-id` :

```bash
readelf -n main | grep "Build ID"
```

```
    Build ID: a1b2c3d4e5f6...
```

GDB utilise ce Build ID comme méthode alternative pour trouver le fichier de symboles, en cherchant dans `/usr/lib/debug/.build-id/<xx>/<reste>.debug`. Ce mécanisme est plus robuste que le `gnu-debuglink` car il ne dépend pas du nom ni de l'emplacement du fichier.

## `-g` et la compilation incrémentale

Les informations de débogage interagissent avec la compilation incrémentale de manière subtile. Chaque fichier `.o` compilé avec `-g` contient ses propres sections `.debug_*`, qui incluent les descriptions de tous les types utilisés — y compris ceux provenant des headers système. Sur un projet de 100 fichiers qui incluent tous `<string>`, la description DWARF complète de `std::basic_string` est dupliquée 100 fois dans les fichiers objets.

Le linker fusionne ces descriptions au moment de la création de l'exécutable, mais cette fusion prend du temps et contribue au temps de linkage. Deux techniques atténuent ce problème.

### Compression des sections de débogage

```bash
g++ -g -gz main.cpp -o main
```

L'option `-gz` compresse les sections `.debug_*` avec zlib. La taille sur disque diminue sensiblement (souvent de 50 à 70 %), au prix d'un léger surcoût de temps de compilation et de déchargement par GDB. GDB moderne décompresse ces sections de manière transparente.

On peut aussi compresser après coup :

```bash
objcopy --compress-debug-sections main
```

### Fission de débogage (`-gsplit-dwarf`)

```bash
g++ -g -gsplit-dwarf main.cpp -o main
```

Cette option sépare les informations de débogage dans un fichier `.dwo` (*DWARF Object*) au moment de la compilation, plutôt qu'au moment du linkage. Le fichier `.o` ne contient qu'un squelette DWARF minimal, et le linker n'a pas besoin de traiter les sections de débogage complètes. Le résultat est un **temps de linkage considérablement réduit** sur les gros projets.

```bash
ls -la main.dwo
```

GDB charge les fichiers `.dwo` à la demande quand il en a besoin. L'inconvénient est que ces fichiers doivent rester accessibles au même chemin — ils ne sont pas embarqués dans l'exécutable.

> 💡 *La fission de débogage est une optimisation du **temps de build**, pas du temps d'exécution. Elle est particulièrement bénéfique sur les projets de grande taille où le linkage avec `-g` prend plusieurs minutes.*

## Vérifier l'état de débogage d'un binaire

En situation de diagnostic, il est courant de devoir vérifier si un binaire contient des informations de débogage, et à quel niveau :

```bash
# Méthode rapide : file indique "not stripped" ou "stripped"
file main

# Vérifier la présence de sections .debug_*
readelf -S main | grep -c '\.debug'

# Vérifier la présence de la table des symboles
readelf -S main | grep '\.symtab'

# Vérifier le Build ID
readelf -n main | grep "Build ID"
```

Un binaire peut être dans plusieurs états. Il peut être **"not stripped" avec debug** (compilé avec `-g`, sections `.debug_*` et `.symtab` présentes — le cas le plus complet). Il peut être **"not stripped" sans debug** (compilé sans `-g`, seule `.symtab` est présente — `nm` fonctionne, mais GDB ne peut pas faire la correspondance avec le source). Il peut être **"stripped"** (`strip` a été exécuté, plus de `.symtab` ni de `.debug_*` — seuls `.dynsym` subsiste pour les symboles dynamiques). Enfin, il peut être **"stripped" avec debuglink** (strippé mais avec un lien vers un fichier `.debug` séparé — le déploiement idéal en production).

---

## En résumé

Les informations de débogage sont un investissement à coût zéro en performance d'exécution : elles n'occupent de l'espace que sur disque, pas en mémoire. Le niveau `-g` (équivalent à `-g2`) couvre la grande majorité des besoins. `-g3` ajoute les macros, utile pour du code legacy fortement macro-isé. `-ggdb3` est le choix le plus complet pour le débogage avec GDB.

La combinaison `-O0 -g` est idéale pour le développement. La combinaison `-O2 -g` est recommandée pour la production — elle préserve les performances tout en rendant possible l'analyse post-mortem des core dumps. La séparation des symboles de débogage avec `objcopy` et `strip` permet de distribuer des binaires légers tout en conservant la capacité de diagnostic.

Sur les projets de grande taille, les options `-gz` (compression) et `-gsplit-dwarf` (fission) réduisent significativement l'impact des informations de débogage sur les temps de build et la taille des artefacts.
