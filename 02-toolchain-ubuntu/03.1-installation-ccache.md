# 2.3.1 — Installation et configuration de ccache

> **Chapitre 2 · Section 2.3 · Accélération de compilation** · Niveau Débutant

---

## Installation depuis les dépôts Ubuntu

La méthode la plus simple et celle que nous recommandons pour commencer :

```bash
sudo apt update  
sudo apt install ccache  
```

Vérification immédiate :

```bash
ccache --version
# Output (exemple sur Ubuntu 24.04) :
# ccache version 4.9.1
#
# Copyright (C) 2002-2024 Joel Rosdahl and other contributors
# ...
```

La version empaquetée dans les dépôts dépend de votre release Ubuntu. Sur Ubuntu 24.04 LTS, vous obtiendrez une version 4.9.x ; sur des releases plus récentes, une version 4.10 ou supérieure. Toutes les versions 4.x sont parfaitement utilisables pour cette formation.

### Vérifier l'emplacement de l'exécutable

```bash
which ccache
# Output :
# /usr/bin/ccache
```

L'installation crée également un répertoire de **liens symboliques** qui sera important pour l'intégration automatique (section 2.3.2) :

```bash
ls /usr/lib/ccache
# Output :
# c++  cc  g++  gcc  g++-13  gcc-13  ...  clang  clang++  ...
```

Chacun de ces fichiers est un lien symbolique vers `/usr/bin/ccache`. Quand ccache est invoqué sous le nom `g++`, il sait qu'il doit transmettre la compilation au vrai `g++` après avoir vérifié le cache.

## Installation d'une version récente (optionnel)

En mars 2026, la dernière version stable de ccache est la **4.13**. Si vous souhaitez en bénéficier — pour ses dernières optimisations ou la correction d'un bug spécifique — plusieurs méthodes sont possibles.

### Méthode 1 : Téléchargement du binaire précompilé

ccache distribue des archives binaires statiques pour Linux sur sa page de téléchargement. C'est la méthode la plus rapide :

```bash
# Télécharger la dernière version (adapter le numéro)
wget https://github.com/ccache/ccache/releases/download/v4.13/ccache-4.13-linux-x86_64.tar.xz

# Extraire
tar xf ccache-4.13-linux-x86_64.tar.xz

# Installer dans /usr/local/bin (prioritaire sur /usr/bin dans le PATH)
sudo cp ccache-4.13-linux-x86_64/ccache /usr/local/bin/ccache

# Vérification
ccache --version | head -1
```

> ⚠️ Si vous installez ccache manuellement dans `/usr/local/bin`, les liens symboliques de `/usr/lib/ccache` continueront à pointer vers la version apt dans `/usr/bin/ccache`. Pour la plupart des usages (intégration via `CMAKE_CXX_COMPILER_LAUNCHER`), cela n'a aucune importance car c'est le binaire dans le `PATH` qui est invoqué. Si vous utilisez la méthode des liens symboliques (section 2.3.2), recréez-les pour qu'ils pointent vers `/usr/local/bin/ccache`.

### Méthode 2 : Compilation depuis les sources

Pour les utilisateurs qui souhaitent la toute dernière version ou des options de compilation personnalisées :

```bash
# Dépendances de build
sudo apt install cmake ninja-build libzstd-dev libhiredis-dev

# Cloner et compiler
git clone --branch v4.13 --depth 1 https://github.com/ccache/ccache.git  
cd ccache  
cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Release  
cmake --build build  
sudo cmake --install build  
```

Pour la grande majorité des utilisateurs, la version des dépôts Ubuntu est suffisante. N'investissez du temps dans une installation manuelle que si vous avez une raison précise.

## Emplacement et structure du cache

### Répertoire par défaut

ccache stocke ses données dans un répertoire local. L'emplacement par défaut suit la convention XDG :

```
~/.cache/ccache/
```

Vous pouvez vérifier l'emplacement configuré :

```bash
ccache --get-config cache_dir
# Output :
# /home/votre_user/.cache/ccache
```

### Ce que contient le répertoire de cache

```bash
ls ~/.cache/ccache/
# Output typique :
# 0/  1/  2/  3/  ...  e/  f/  ccache.conf  stats
```

Les sous-répertoires `0/` à `f/` (hexadécimaux) contiennent les fichiers objets cachés, organisés par les premiers caractères de leur hash. Le fichier `stats` contient les compteurs de statistiques. Un éventuel fichier `ccache.conf` à cet emplacement est une configuration spécifique au cache (distincte de la configuration utilisateur).

### Personnaliser l'emplacement du cache

Vous pouvez déplacer le cache vers un autre emplacement — par exemple un SSD dédié, un volume plus grand, ou un chemin partagé entre utilisateurs :

```bash
# Via variable d'environnement (temporaire, pour une session)
export CCACHE_DIR=/mnt/fast-ssd/ccache

# Via le fichier de configuration (permanent)
# Dans ~/.config/ccache/ccache.conf :
# cache_dir = /mnt/fast-ssd/ccache
```

Si vous choisissez un emplacement personnalisé, assurez-vous que le système de fichiers supporte les opérations atomiques de renommage (c'est le cas de ext4, Btrfs, XFS, tmpfs — les systèmes de fichiers courants sur Ubuntu). NFS peut poser des problèmes dans certaines configurations.

## Le fichier de configuration

### Emplacement

ccache lit sa configuration dans cet ordre de priorité (le premier trouvé l'emporte pour chaque option) :

1. **Variables d'environnement** — par exemple `CCACHE_MAXSIZE=20G` (préfixe `CCACHE_` + nom de l'option en majuscules)
2. **Configuration spécifique au cache** — `$CCACHE_DIR/ccache.conf` (rarement utilisé)
3. **Configuration utilisateur** — `~/.config/ccache/ccache.conf`
4. **Configuration système** — `/etc/ccache.conf` (partagée entre tous les utilisateurs)
5. **Valeurs par défaut** compilées dans ccache

Pour un poste de développement personnel, le fichier **utilisateur** est l'emplacement naturel :

```bash
mkdir -p ~/.config/ccache
```

### Options de configuration essentielles

Voici un fichier `~/.config/ccache/ccache.conf` commenté avec les options les plus utiles :

```ini
# =============================================================================
# ccache — Configuration utilisateur
# Documentation complète : https://ccache.dev/manual/latest.html
# =============================================================================

# ---------------------------------------------------------------------------
# Taille du cache
# ---------------------------------------------------------------------------
# Taille maximale du cache local. Quand cette limite est atteinte, les
# entrées les plus anciennes (LRU) sont automatiquement supprimées.
# Unités : G (giga-octets), M (méga-octets), T (téra-octets).
# Défaut : 5G
max_size = 10G

# ---------------------------------------------------------------------------
# Compression
# ---------------------------------------------------------------------------
# Activer la compression Zstandard des fichiers cachés.
# Réduit la taille du cache sur disque de 30-70% selon le type de code.
# Activé par défaut depuis ccache 4.0.
compression = true

# Niveau de compression Zstandard (1-19).
# - 1 : compression rapide, bon ratio (recommandé pour les builds locaux)
# - 3 : niveau par défaut de zstd, bon compromis
# - 9+ : compression plus forte mais plus lente (utile si l'espace disque
#         est contraint ou pour un cache distant sur réseau lent)
# Défaut : 0 (= niveau par défaut de zstd, typiquement 3)
compression_level = 1

# ---------------------------------------------------------------------------
# Normalisation des chemins (base_dir)
# ---------------------------------------------------------------------------
# Si défini, ccache remplace les chemins absolus commençant par base_dir
# par des chemins relatifs AVANT de calculer le hash. Cela permet d'obtenir
# des cache hits même si le projet est compilé depuis des répertoires
# différents (par exemple /home/alice/projet vs /home/bob/projet,
# ou /workspace/run-1 vs /workspace/run-2 en CI).
#
# Restriction : base_dir doit être un chemin absolu, et le code source
# doit se trouver sous ce chemin.
#
# Défaut : (vide — désactivé)
# base_dir = /home/votre_user

# ---------------------------------------------------------------------------
# Inode cache
# ---------------------------------------------------------------------------
# L'inode cache évite de re-hasher les mêmes en-têtes à chaque fichier
# compilé au sein d'un même build. Sur un projet avec des centaines de
# fichiers incluant les mêmes headers STL, le gain peut être considérable.
# Activé par défaut depuis ccache 4.7 sur les systèmes de fichiers
# compatibles (ext4, Btrfs, XFS).
# Défaut : true (si le filesystem le supporte)
inode_cache = true

# ---------------------------------------------------------------------------
# Sloppiness (tolérance)
# ---------------------------------------------------------------------------
# Par défaut, ccache est très strict sur ce qui constitue une entrée
# identique. Certains projets bénéficient d'un relâchement ciblé :
#
# - include_file_ctime : ignorer le ctime des fichiers inclus
#   (utile si des outils modifient les métadonnées sans changer le contenu)
# - include_file_mtime : ignorer le mtime des fichiers inclus
#   (utile avec certains systèmes de fichiers réseau)
# - time_macros : ignorer __DATE__ et __TIME__
#   (utile si des bibliothèques tierces utilisent ces macros)
#
# Défaut : (vide — strict)
# sloppiness = include_file_mtime,include_file_ctime,time_macros

# ---------------------------------------------------------------------------
# Comportement divers
# ---------------------------------------------------------------------------
# Nombre maximum de fichiers dans le cache (0 = pas de limite).
# La limite par taille (max_size) est généralement suffisante.
# Défaut : 0
# max_files = 0

# Afficher un message quand un cache hit ou miss se produit.
# Utile pour le diagnostic, bruyant pour un usage quotidien.
# Défaut : false
# log_file = /tmp/ccache.log

# Stocker les résultats avec la compilation en mode debug (-g).
# Activé par défaut — les fichiers debug sont souvent plus gros mais
# le bénéfice de ccache est proportionnellement plus important.
# Défaut : true
```

### Configuration minimale recommandée pour commencer

Si vous ne voulez retenir qu'une seule chose, voici la configuration minimale qui couvre 90 % des besoins :

```ini
# ~/.config/ccache/ccache.conf — Configuration minimale
max_size = 10G  
compression = true  
compression_level = 1  
```

Trois lignes. C'est tout ce qu'il faut pour commencer à bénéficier de ccache.

## Variables d'environnement

Chaque option de configuration peut être surchargée par une variable d'environnement préfixée par `CCACHE_`. Le nom de la variable est le nom de l'option en **majuscules**, les underscores conservés :

| Option dans `ccache.conf` | Variable d'environnement |
|---|---|
| `max_size = 10G` | `CCACHE_MAXSIZE=10G` |
| `compression = true` | `CCACHE_COMPRESS=true` |
| `compression_level = 1` | `CCACHE_COMPRESSLEVEL=1` |
| `cache_dir = /path` | `CCACHE_DIR=/path` |
| `base_dir = /home/user` | `CCACHE_BASEDIR=/home/user` |
| `log_file = /tmp/ccache.log` | `CCACHE_LOGFILE=/tmp/ccache.log` |

Les variables d'environnement sont prioritaires sur le fichier de configuration. Elles sont particulièrement utiles dans les scripts CI, où vous voulez une configuration ponctuelle sans modifier de fichier :

```bash
# Exemple dans un script CI
export CCACHE_DIR=/ci/cache/ccache  
export CCACHE_MAXSIZE=5G  
export CCACHE_BASEDIR=/ci/workspace  
cmake -B build -G Ninja -DCMAKE_CXX_COMPILER_LAUNCHER=ccache  
cmake --build build  
```

## Consulter et modifier la configuration

### Afficher toute la configuration active

```bash
ccache --show-config
```

Cette commande affiche **toutes** les options avec leur valeur actuelle et leur source (défaut, fichier de configuration, variable d'environnement). C'est l'outil de diagnostic le plus utile pour comprendre pourquoi ccache se comporte d'une certaine manière :

```
(default) absolute_paths_in_stderr = false
(default) base_dir =
(/home/alice/.config/ccache/ccache.conf) compression = true
(/home/alice/.config/ccache/ccache.conf) compression_level = 1
(default) compiler =
(default) compiler_check = mtime
(default) compiler_type = auto
...
(/home/alice/.config/ccache/ccache.conf) max_size = 10.0 GB
...
```

La source entre parenthèses indique d'où provient chaque valeur. Si une option est surchargée par une variable d'environnement, cela sera indiqué.

### Lire une option spécifique

```bash
ccache --get-config max_size
# Output :
# 10.0G

ccache --get-config cache_dir
# Output :
# /home/alice/.cache/ccache
```

### Modifier une option en ligne de commande

ccache permet aussi de modifier la configuration de manière persistante depuis la ligne de commande, sans éditer le fichier manuellement :

```bash
ccache --set-config max_size=20G  
ccache --set-config compression_level=3  
```

Ces commandes écrivent dans le fichier de configuration utilisateur (`~/.config/ccache/ccache.conf`).

## Première compilation avec ccache

### Test manuel

Créons un fichier source simple et compilons-le à travers ccache pour vérifier que tout fonctionne :

```bash
# Créer un fichier source de test
cat > /tmp/test_ccache.cpp << 'EOF'
#include <iostream>
#include <vector>
#include <algorithm>
#include <string>
#include <numeric>

int main() {
    std::vector<int> numbers(100);
    std::iota(numbers.begin(), numbers.end(), 1);
    
    auto sum = std::accumulate(numbers.begin(), numbers.end(), 0);
    std::cout << "Somme de 1 à 100 = " << sum << "\n";
}
EOF

# Remettre les statistiques à zéro pour un test propre
ccache --zero-stats

# Première compilation : cache miss attendu
echo "=== Première compilation ==="  
time ccache g++ -std=c++23 -O2 -c /tmp/test_ccache.cpp -o /tmp/test_ccache.o  
echo ""  

# Supprimer le fichier objet (simuler un rebuild)
rm /tmp/test_ccache.o

# Deuxième compilation : cache hit attendu
echo "=== Deuxième compilation (cache hit attendu) ==="  
time ccache g++ -std=c++23 -O2 -c /tmp/test_ccache.cpp -o /tmp/test_ccache.o  
echo ""  

# Afficher les statistiques
echo "=== Statistiques ==="  
ccache --show-stats 2>&1 | grep -E "Hits|Misses|Cache size"  

# Nettoyage
rm -f /tmp/test_ccache.cpp /tmp/test_ccache.o
```

La sortie attendue :

- La première compilation prend un temps normal (quelques centièmes à quelques dixièmes de seconde pour un fichier simple).
- La deuxième compilation est **quasi instantanée** — le temps mesuré par `time` devrait être largement inférieur.
- Les statistiques montrent 1 miss (première compilation) et 1 hit (deuxième).

### Interpréter le résultat

Si la deuxième compilation est aussi rapide que la première, ccache ne fonctionne pas correctement. Vérifiez :

1. Que `ccache` est bien dans le `PATH` : `which ccache`
2. Que le cache n'est pas désactivé : `ccache --get-config disable`
3. Que le répertoire de cache est accessible en écriture : `ls -la ~/.cache/ccache/`
4. Activez le log pour diagnostiquer : `ccache --set-config log_file=/tmp/ccache.log`, relancez la compilation, puis examinez `/tmp/ccache.log`

## Impact mesuré sur un projet réel

Pour donner un ordre de grandeur concret, voici des mesures typiques sur un projet C++ de taille moyenne (~150 fichiers sources, headers STL et Boost inclus) :

| Scénario | Sans ccache | Avec ccache (miss) | Avec ccache (hit) |
|---|---|---|---|
| Build complet | 2 min 45 s | 2 min 50 s (+~2%) | **8 s** (-95%) |
| Rebuild après `clean` | 2 min 45 s | 2 min 50 s | **8 s** |
| Changement 1 fichier | 12 s | 12 s | 12 s (le fichier changé = miss) |
| Switch branche Git (même code) | 2 min 45 s | 2 min 50 s | **8 s** |

Les scénarios où ccache brille le plus sont les **rebuilds complets** : après un `clean`, un changement de branche Git, un retour à une version précédente, ou la reconstruction d'un projet fraîchement cloné qu'un collègue a déjà compilé avec le même cache partagé. Le léger surcoût (~2 %) lors d'un cache miss correspond au temps de calcul du hash et de stockage dans le cache — il est négligeable par rapport au temps de compilation.

## Ce qu'il faut retenir

- **L'installation** se fait en une commande : `sudo apt install ccache`. Aucune compilation manuelle n'est requise.
- Le **fichier de configuration** principal est `~/.config/ccache/ccache.conf`. Les trois options essentielles sont `max_size`, `compression` et `compression_level`.
- Chaque option peut être surchargée par une **variable d'environnement** préfixée `CCACHE_` — utile en CI/CD.
- `ccache --show-config` est votre meilleur outil de diagnostic : il affiche chaque option avec sa source.
- Le cache est stocké par défaut dans `~/.cache/ccache/`. Il peut être déplacé vers n'importe quel emplacement via l'option `cache_dir`.
- L'option **`base_dir`** est la clé pour obtenir des cache hits quand le même projet est compilé depuis des chemins différents (multi-utilisateurs, CI/CD).
- Un test rapide avec `ccache --zero-stats`, deux compilations identiques et `ccache --show-stats` suffit à vérifier que tout fonctionne.
