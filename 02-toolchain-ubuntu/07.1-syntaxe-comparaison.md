# 2.7.1 Syntaxe et comparaison avec `std::cout` et `printf`

> **Niveau** : Débutant  
> **Prérequis** : Section 2.7 (Introduction à std::print)  
> **Objectifs** : Maîtriser la syntaxe des placeholders de `std::print`, savoir reformuler n'importe quel affichage `cout` ou `printf` en `std::print`, et comprendre les pièges propres à chaque approche.

---

La section 2.7 a présenté `std::print` à travers un premier exemple. Ici, nous allons systématiser la comparaison en mettant les trois approches côte à côte sur des cas variés et représentatifs du quotidien. L'objectif est double : ancrer la syntaxe de `std::print` par la pratique, et comprendre **pourquoi** elle est supérieure, pas seulement qu'elle l'est.

Tous les exemples de cette section se compilent avec :

```bash
g++ -std=c++23 -Wall -Wextra exemple.cpp -o exemple
```

---

## Les trois philosophies en un coup d'œil

Avant de plonger dans les cas concrets, posons les principes de chaque approche :

**`printf`** utilise une chaîne de format avec des spécificateurs (`%d`, `%s`, `%f`) qui décrivent à la fois le type et le format de chaque argument. Le programmeur est responsable de la cohérence entre les spécificateurs et les arguments — le compilateur ne vérifie pas toujours.

**`std::cout`** utilise l'opérateur `<<` surchargé pour chaque type. Le type est détecté automatiquement (type-safe), mais le formatage repose sur des **manipulateurs** (`std::setw`, `std::hex`, `std::fixed`) qui modifient l'état du flux de manière persistante.

**`std::print`** utilise une chaîne de format avec des placeholders (`{}`) où le type est détecté automatiquement. Le format est spécifié **dans le placeholder** (`{:.2f}`, `{:>10}`), de manière locale et non persistante.

---

## Cas 1 : Affichage de variables simples

Le cas le plus basique — afficher un mélange de texte et de valeurs.

### `printf`

```cpp
const char* langage = "C++";  
int standard = 23;  
printf("Langage : %s, Standard : %d\n", langage, standard);  
```

Avec un `std::string`, il faut extraire le pointeur C :

```cpp
std::string langage = "C++";  
printf("Langage : %s\n", langage.c_str());  // .c_str() obligatoire  
```

Oublier `.c_str()` est un bug classique. Sur certains compilateurs, passer un `std::string` directement à `printf` provoque un comportement indéfini — le programme peut afficher des caractères aléatoires, planter, ou sembler fonctionner (le pire des cas).

### `std::cout`

```cpp
std::string langage = "C++";  
int standard = 23;  
std::cout << "Langage : " << langage << ", Standard : " << standard << "\n";  
```

Type-safe (`std::string` est accepté directement), mais la chaîne est fragmentée en sept morceaux reliés par `<<`. Sur un affichage complexe, cette fragmentation rend le code difficile à lire et le texte final difficile à visualiser mentalement.

### `std::println`

```cpp
std::string langage = "C++";  
int standard = 23;  
std::println("Langage : {}, Standard : {}", langage, standard);  
```

Le texte final est lisible d'un seul regard dans la chaîne de format. `std::string` est accepté nativement. Le type de chaque argument est vérifié à la compilation.

---

## Cas 2 : Formatage des nombres flottants

L'affichage de nombres à virgule flottante avec une précision contrôlée est l'un des cas où les différences sont les plus frappantes.

### `printf`

```cpp
double prix = 42.5;  
double pi = 3.14159265358979;  

printf("Prix : %.2f euros\n", prix);           // 42.50  
printf("Pi : %.6f\n", pi);                     // 3.141593  
printf("Scientifique : %e\n", 0.00042);        // 4.200000e-04  
printf("Automatique : %g\n", 0.00042);         // 0.00042  
```

Concis et familier. Mais le spécificateur `%f` ne fonctionne que pour `double` (et `float`, promu automatiquement). Si vous passez un `int` par erreur, le résultat est un comportement indéfini.

### `std::cout`

```cpp
double prix = 42.5;  
double pi = 3.14159265358979;  

std::cout << "Prix : " << std::fixed << std::setprecision(2) 
          << prix << " euros" << std::endl;     // 42.50
std::cout << "Pi : " << std::setprecision(6) 
          << pi << std::endl;                   // 3.141593

// ⚠️ Piège : std::fixed et std::setprecision sont PERSISTANTS
std::cout << "Autre valeur : " << 123.456 << std::endl;  // 123.46 (pas 123.456 !)
```

Le problème majeur apparaît à la troisième ligne : les manipulateurs `std::fixed` et `std::setprecision` ont modifié l'**état du flux** `cout`, et cet état persiste pour tous les affichages suivants. L'affichage de `123.456` est tronqué à deux décimales sans que le programmeur l'ait explicitement demandé. C'est une source de bugs insidieux, surtout dans du code où plusieurs fonctions partagent le même flux.

Pour restaurer le comportement par défaut, il faut sauvegarder et restaurer les flags :

```cpp
auto old_flags = std::cout.flags();  
auto old_prec = std::cout.precision();  
std::cout << std::fixed << std::setprecision(2) << prix;  
std::cout.flags(old_flags);  
std::cout.precision(old_prec);  
```

C'est du code défensif que personne n'écrit en pratique — et c'est pourquoi les bugs de formatage persistant sont si courants.

### `std::println`

```cpp
double prix = 42.5;  
double pi = 3.14159265358979;  

std::println("Prix : {:.2f} euros", prix);         // 42.50  
std::println("Pi : {:.6f}", pi);                   // 3.141593  
std::println("Scientifique : {:e}", 0.00042);      // 4.200000e-04  
std::println("Automatique : {:g}", 0.00042);       // 0.00042  
std::println("Autre valeur : {}", 123.456);        // 123.456 (aucune pollution)  
```

Chaque spécification de format est **locale** au placeholder. Le `{:.2f}` de la première ligne n'affecte en rien les lignes suivantes. Il n'y a aucun état global à gérer, aucun risque de pollution.

## Cas 3 : Affichage en colonnes alignées

L'alignement de données en colonnes (pour un tableau, un rapport, un log) est un cas d'usage fréquent et révélateur.

### `printf`

```cpp
printf("%-15s %8s %10s\n", "Produit", "Qté", "Prix");  
printf("%-15s %8d %10.2f\n", "Clavier", 12, 49.99);  
printf("%-15s %8d %10.2f\n", "Souris", 25, 29.99);  
printf("%-15s %8d %10.2f\n", "Écran 27\"", 3, 399.00);  
```

```
Produit              Qté       Prix  
Clavier               12      49.99  
Souris                25      29.99  
Écran 27"              3     399.00
```

Efficace et compact. La largeur (`15`, `8`, `10`) et l'alignement (`-` pour gauche) sont intégrés dans le spécificateur. Mais les `std::string` ne passent toujours pas directement, et la correspondance entre les `%` et les arguments reste manuelle.

### `std::cout`

```cpp
std::cout << std::left << std::setw(15) << "Produit"
          << std::right << std::setw(8) << "Qté"
          << std::setw(10) << "Prix" << "\n";
std::cout << std::left << std::setw(15) << "Clavier"
          << std::right << std::setw(8) << 12
          << std::fixed << std::setprecision(2) << std::setw(10) << 49.99 << "\n";
// ... (répéter pour chaque ligne)
```

Chaque cellule nécessite un `std::setw` (qui, lui, n'est **pas** persistant — il ne s'applique qu'à la prochaine valeur) et potentiellement un changement d'alignement. Le code est volumineux et difficile à modifier. Ajouter une colonne nécessite de toucher chaque ligne.

### `std::println`

```cpp
std::println("{:<15} {:>8} {:>10}", "Produit", "Qté", "Prix");  
std::println("{:<15} {:>8} {:>10.2f}", "Clavier", 12, 49.99);  
std::println("{:<15} {:>8} {:>10.2f}", "Souris", 25, 29.99);  
std::println("{:<15} {:>8} {:>10.2f}", "Écran 27\"", 3, 399.00);  
```

La chaîne de format est un **template visuel** du résultat. En lisant `{:<15} {:>8} {:>10.2f}`, on voit la structure du tableau : une colonne de 15 caractères alignée à gauche, une de 8 alignée à droite, une de 10 alignée à droite avec 2 décimales. Modifier le format d'une colonne se fait en un seul endroit.

## Cas 4 : Bases numériques

L'affichage de nombres en différentes bases est courant en programmation système et en débogage.

### `printf`

```cpp
int val = 255;  
printf("Décimal     : %d\n", val);       // 255  
printf("Hexadécimal : %x\n", val);       // ff  
printf("Hex (préfixe): %#x\n", val);     // 0xff  
printf("Hex (majusc.): %#X\n", val);     // 0XFF  
printf("Octal       : %#o\n", val);      // 0377  
printf("Binaire     : ???\n");           // Pas de %b standard en C  
```

`printf` ne propose **pas** de spécificateur standard pour le binaire. Il faut écrire une fonction d'affichage manuelle ou utiliser des extensions non portables.

### `std::cout`

```cpp
int val = 255;  
std::cout << "Décimal     : " << std::dec << val << "\n";    // 255  
std::cout << "Hexadécimal : " << std::hex << val << "\n";    // ff  
std::cout << "Hex (préfixe): " << std::showbase << std::hex << val << "\n"; // 0xff  
std::cout << "Octal       : " << std::oct << val << "\n";    // 0377  

// ⚠️ Piège : std::hex est PERSISTANT
std::cout << "Après hex : " << 42 << "\n";  // Affiche "2a" au lieu de "42" !
```

Le piège de la persistance est particulièrement vicieux ici. Après `std::hex`, **tous** les entiers sont affichés en hexadécimal jusqu'à ce que `std::dec` soit explicitement rétabli. Et comme pour les flottants, le binaire n'est pas supporté nativement par `std::cout`.

### `std::println`

```cpp
int val = 255;  
std::println("Décimal     : {:d}", val);      // 255  
std::println("Hexadécimal : {:x}", val);      // ff  
std::println("Hex (préfixe): {:#x}", val);    // 0xff  
std::println("Hex (majusc.): {:#X}", val);    // 0XFF  
std::println("Octal       : {:#o}", val);     // 0377  
std::println("Binaire     : {:b}", val);      // 11111111  
std::println("Bin (préfixe): {:#b}", val);    // 0b11111111  
std::println("Après : {}", 42);               // 42 (aucune pollution)  
```

`std::print` supporte nativement le **binaire** (`{:b}`) — une fonctionnalité attendue depuis longtemps et absente de `printf` et `cout`. Le préfixe `#` ajoute automatiquement `0x`, `0o` ou `0b` selon la base. Et comme toujours, aucun état persistant : la ligne suivante affiche `42` en décimal sans aucune précaution.

## Cas 5 : Caractère de remplissage

Remplir un champ avec un caractère spécifique (zéros, tirets, points) est un besoin courant pour les identifiants, les rapports, et l'affichage formaté.

### `printf`

```cpp
printf("ID : %08d\n", 42);              // ID : 00000042  
printf("Séparateur : ???\n");           // Pas de fill char arbitraire  
```

`printf` supporte le zero-padding avec `%08d`, mais ne permet pas de choisir un caractère de remplissage arbitraire. Pour remplir avec des tirets ou des points, il faut du code manuel.

### `std::cout`

```cpp
std::cout << "ID : " << std::setfill('0') << std::setw(8) << 42 << "\n";
// ⚠️ setfill est PERSISTANT, setw ne l'est PAS
std::cout << "Test : " << std::setw(8) << 99 << "\n";  // 00000099 (fill persistant)  
std::cout << "Test2 : " << 99 << "\n";                  // 99 (setw épuisé)  
```

Un nouveau piège de cohérence : `std::setfill` est persistant (le caractère `'0'` reste actif), tandis que `std::setw` ne l'est pas (il se réinitialise après chaque affichage). Ces règles de persistance incohérentes entre manipulateurs sont une source constante de confusion.

### `std::println`

```cpp
std::println("ID : {:08d}", 42);              // ID : 00000042  
std::println("Tirets : {:->20}", "FIN");      // Tirets : -----------------FIN  
std::println("Points : {:.>20}", "FIN");      // Points : .................FIN  
std::println("Centré : {:*^20}", "TITRE");    // Centré : *******TITRE********  
std::println("Test : {}", 99);                // Test : 99 (aucune pollution)  
```

La syntaxe `{:fill align width}` est unifiée et locale. Le caractère de remplissage (`-`, `.`, `*`) est spécifié directement avant l'alignement (`<`, `>`, `^`). Pas de manipulateur à gérer, pas de persistance incohérente.

---

## Cas 6 : Arguments positionnels et répétition

Parfois, on veut utiliser un même argument plusieurs fois, ou changer l'ordre d'apparition.

### `printf`

```cpp
const char* nom = "Alice";  
printf("%s a dit : « Bonjour, je suis %s »\n", nom, nom);  
// Obligé de passer 'nom' deux fois
```

`printf` ne supporte pas les arguments positionnels dans le standard C (certaines extensions POSIX proposent `%1$s`, mais ce n'est pas portable).

### `std::cout`

```cpp
std::string nom = "Alice";  
std::cout << nom << " a dit : « Bonjour, je suis " << nom << " »\n";  
```

L'argument est inséré manuellement à chaque position — la chaîne est fragmentée.

### `std::println`

```cpp
std::string nom = "Alice";  
std::println("{0} a dit : « Bonjour, je suis {0} »", nom);  
```

L'index `{0}` fait référence au premier argument. On peut le réutiliser autant de fois que nécessaire, et mélanger les indices pour changer l'ordre :

```cpp
std::println("Coordonnées : ({1}, {0})", 10, 20);  // Coordonnées : (20, 10)
```

C'est une fonctionnalité indispensable pour la localisation (traduction) : l'ordre des mots change selon les langues, et les arguments positionnels permettent de réorganiser sans modifier le code.

> ⚠️ **On ne peut pas mélanger** les placeholders automatiques (`{}`) et positionnels (`{0}`) dans la même chaîne de format. C'est l'un ou l'autre.

## Cas 7 : Booléens

Un cas simple mais illustratif des choix de design de chaque approche.

### `printf`

```cpp
bool actif = true;  
printf("Actif : %d\n", actif);          // Actif : 1  
printf("Actif : %s\n", actif ? "true" : "false");  // Actif : true  
```

`printf` ne connaît pas le type `bool`. Il l'affiche comme un entier (`0` ou `1`). Pour obtenir `true`/`false`, il faut un opérateur ternaire.

### `std::cout`

```cpp
bool actif = true;  
std::cout << "Actif : " << actif << "\n";                    // Actif : 1  
std::cout << "Actif : " << std::boolalpha << actif << "\n";  // Actif : true  
// ⚠️ std::boolalpha est persistant
```

Par défaut, `cout` affiche `0`/`1`. Le manipulateur `std::boolalpha` passe en mode textuel — mais comme d'habitude, il est persistant.

### `std::println`

```cpp
bool actif = true;  
std::println("Actif : {}", actif);    // Actif : true  
```

`std::print` affiche les booléens comme `true`/`false` **par défaut**. C'est un choix de design sensé : dans la majorité des cas, c'est ce que le programmeur souhaite. Si vous voulez l'affichage numérique, la spécification `:d` force le format entier : `std::println("{:d}", actif)` affiche `1`.

## Cas 8 : Accolades littérales

Puisque `{}` est la syntaxe des placeholders, comment afficher des accolades littérales ?

```cpp
std::println("Un placeholder : {{}}");       // Un placeholder : {}  
std::println("JSON : {{\"clé\": {}}}", 42);  // JSON : {"clé": 42}  
```

Le doublement `{{` produit une accolade littérale `{`, et `}}` produit `}`. C'est le même principe d'échappement que le `%%` de `printf` pour afficher un `%` littéral.

## Récapitulatif des spécificateurs courants

Voici une correspondance entre les trois syntaxes pour les cas les plus fréquents :

| Résultat voulu | `printf` | `std::cout` | `std::print` |
|---------------|----------|-------------|-------------|
| Entier décimal | `%d` | `<< val` | `{:d}` ou `{}` |
| Entier hexadécimal | `%x` | `<< std::hex << val` | `{:x}` |
| Avec préfixe 0x | `%#x` | `<< std::showbase << std::hex` | `{:#x}` |
| Binaire | *(non standard)* | *(non supporté)* | `{:b}` |
| Flottant défaut | `%f` | `<< val` | `{:f}` ou `{}` |
| Flottant 2 déc. | `%.2f` | `<< fixed << setprecision(2)` | `{:.2f}` |
| Scientifique | `%e` | `<< scientific` | `{:e}` |
| Largeur 10, droite | `%10d` | `<< setw(10) << val` | `{:>10}` |
| Largeur 10, gauche | `%-10d` | `<< left << setw(10)` | `{:<10}` |
| Zero-padding | `%08d` | `<< setfill('0') << setw(8)` | `{:08d}` |
| Chaîne C | `%s` | `<< str` | `{}` |
| `std::string` | `%s` + `.c_str()` | `<< str` | `{}` |
| Booléen textuel | *(ternaire)* | `<< boolalpha` | `{}` |
| Pointeur | `%p` | `<< ptr` | `{:p}` |
| Caractère | `%c` | `<< ch` | `{:c}` ou `{}` |

La colonne `std::print` est systématiquement la plus concise, et dans le cas de `std::string`, la plus sûre.

## Les pièges de `std::cout` que `std::print` élimine

Au fil des comparaisons, un motif récurrent s'est dégagé. Récapitulons les problèmes structurels de `std::cout` que `std::print` résout par conception.

### Le problème de l'état persistant

Les manipulateurs `std::hex`, `std::fixed`, `std::setprecision`, `std::setfill`, `std::boolalpha`, `std::showbase` modifient l'état du flux de manière **permanente**. Un code innocent dans une fonction peut corrompre silencieusement les affichages d'une autre fonction qui partage le même flux. `std::print` n'a aucun état : chaque appel est indépendant.

### Le problème de la fragmentation

Avec `<<`, le message est découpé en fragments alternant texte et valeurs. Plus il y a de variables, moins le message final est lisible dans le code. Avec `std::print`, la chaîne de format donne une vue d'ensemble du message final, et les arguments sont listés séparément.

### Le problème de `std::endl`

`std::endl` ajoute un retour à la ligne **et** force un flush du buffer de sortie. Ce flush est une opération d'I/O coûteuse, et dans la majorité des cas, inutile. Les débutants utilisent `std::endl` partout par habitude, ce qui peut dégrader significativement les performances d'un programme qui affiche beaucoup de texte. `std::println` ajoute un `\n` sans flush — le comportement performant par défaut.

## Quand `std::cout` reste pertinent

Malgré ses défauts, `std::cout` conserve quelques avantages dans des cas spécifiques.

Le **streaming continu** est un cas où `cout` est naturel : quand vous construisez une sortie de manière incrémentale dans une boucle complexe, écrire `<<` au fil de l'eau peut être plus lisible que de construire une chaîne de format dynamique.

La **redirection vers un flux personnalisé** (`std::ostringstream`, `std::ofstream`, ou un flux custom) est nativement intégrée dans l'écosystème IOStreams. `std::print` écrit sur `stdout` par défaut ; pour écrire sur un fichier, il faut passer un `FILE*` en premier argument (`std::print(file, "format", args...)`) ou utiliser `std::format` pour construire une chaîne puis l'écrire.

La **surcharge de l'opérateur `<<`** est le mécanisme historique pour rendre un type affichable. Des milliers de librairies C++ définissent `operator<<` pour leurs types. `std::print` utilise un mécanisme différent (les `std::formatter` spécialisations) qui est plus puissant mais qui nécessite que les librairies fournissent ce support. En 2026, la transition est en cours mais pas encore universelle.

> 📎 *La section 12.7 couvre la création de `std::formatter` personnalisés pour vos propres types.*

---

## En résumé

`std::print` combine le meilleur des deux mondes : la concision d'une chaîne de format à la `printf`, et la sécurité de type à la compilation du C++. Il élimine par conception les trois pièges majeurs de `std::cout` (état persistant, fragmentation, flush inutile) et ajoute des fonctionnalités manquantes dans les deux approches historiques (binaire, arguments positionnels, caractère de remplissage arbitraire).

La syntaxe des placeholders `{:[fill][align][sign][#][0][width][.precision][type]}` est cohérente et locale — chaque spécification n'affecte que le placeholder qui la contient. Le tableau de correspondance de cette section peut servir de référence rapide pour convertir du code `printf` ou `cout` vers `std::print`.
