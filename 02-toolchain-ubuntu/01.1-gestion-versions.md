# 2.1.1 — Gestion des versions avec `update-alternatives`

> **Chapitre 2 · Section 2.1 · Installation des compilateurs** · Niveau Débutant

---

## Le problème : plusieurs compilateurs, un seul nom

Dès que vous installez plusieurs versions de GCC ou de Clang sur la même machine, vous vous retrouvez avec une collection d'exécutables versionnés dans `/usr/bin` :

```
/usr/bin/g++-13
/usr/bin/g++-14
/usr/bin/g++-15
/usr/bin/clang++-18
/usr/bin/clang++-20
```

Or, la plupart des outils et des scripts de build s'attendent à un nom **générique** : `g++`, `gcc`, `clang++`, `clang`. Quand vous tapez `g++` dans un terminal, vers quel exécutable concret cette commande doit-elle pointer ? Et comment basculer facilement d'une version à l'autre sans renommer manuellement des fichiers ou jongler avec des liens symboliques ?

C'est exactement le rôle de **`update-alternatives`**, un mécanisme Debian/Ubuntu qui gère un système de liens symboliques de manière centralisée, propre et réversible.

---

## Principe de fonctionnement

`update-alternatives` repose sur un système à **trois niveaux** de liens symboliques :

```
/usr/bin/g++                          ← Nom générique (ce que vous tapez)
    ↓ lien symbolique
/etc/alternatives/g++                 ← Lien géré par update-alternatives
    ↓ lien symbolique
/usr/bin/g++-15                       ← Exécutable concret (l'« alternative »)
```

Le fichier intermédiaire dans `/etc/alternatives/` est la clé du mécanisme : c'est lui que `update-alternatives` modifie quand vous changez de version. Le lien `/usr/bin/g++` pointe toujours vers `/etc/alternatives/g++`, qui lui-même pointe vers l'exécutable choisi. Ainsi, aucun outil ne voit de changement dans le chemin `/usr/bin/g++` — seule la cible finale change.

Chaque alternative enregistrée possède une **priorité** (un entier). En mode automatique, le système sélectionne toujours l'alternative de plus haute priorité. En mode manuel, c'est votre choix explicite qui prime, quelle que soit la priorité.

---

## Enregistrer des alternatives

### Syntaxe de la commande

```bash
sudo update-alternatives --install <lien_générique> <nom_du_groupe> <chemin_exécutable> <priorité>
```

- **`<lien_générique>`** — le chemin du lien symbolique public (ex. `/usr/bin/g++`) ;
- **`<nom_du_groupe>`** — un identifiant logique pour le groupe d'alternatives (ex. `g++`) ;
- **`<chemin_exécutable>`** — le chemin absolu de l'exécutable concret (ex. `/usr/bin/g++-15`) ;
- **`<priorité>`** — un entier positif ; plus il est élevé, plus l'alternative est favorisée en mode automatique.

### Exemple complet pour GCC

Supposons que GCC 13, GCC 14 et GCC 15 sont installés. On enregistre chaque version avec une priorité croissante pour que la plus récente soit le défaut automatique :

```bash
# Compilateur C++ (g++)
sudo update-alternatives --install /usr/bin/g++ g++ /usr/bin/g++-13 10  
sudo update-alternatives --install /usr/bin/g++ g++ /usr/bin/g++-14 20  
sudo update-alternatives --install /usr/bin/g++ g++ /usr/bin/g++-15 30  

# Compilateur C (gcc) — même logique
sudo update-alternatives --install /usr/bin/gcc gcc /usr/bin/gcc-13 10  
sudo update-alternatives --install /usr/bin/gcc gcc /usr/bin/gcc-14 20  
sudo update-alternatives --install /usr/bin/gcc gcc /usr/bin/gcc-15 30  
```

Après ces commandes, `g++` et `gcc` pointent automatiquement vers les versions 15 (priorité la plus élevée).

### Exemple complet pour Clang

```bash
# Compilateur C++ (clang++)
sudo update-alternatives --install /usr/bin/clang++ clang++ /usr/bin/clang++-18 10  
sudo update-alternatives --install /usr/bin/clang++ clang++ /usr/bin/clang++-20 20  

# Compilateur C (clang)
sudo update-alternatives --install /usr/bin/clang clang /usr/bin/clang-18 10  
sudo update-alternatives --install /usr/bin/clang clang /usr/bin/clang-20 20  
```

## Basculer entre les versions

### Mode interactif : `--config`

La commande `--config` affiche un menu numéroté et vous laisse choisir :

```bash
sudo update-alternatives --config g++
```

Sortie typique :

```
Il existe 3 choix pour l'alternative g++ (fournissant /usr/bin/g++).

  Sélection   Chemin            Priorité  État
------------------------------------------------------------
* 0            /usr/bin/g++-15   30        mode automatique
  1            /usr/bin/g++-13   10        mode manuel
  2            /usr/bin/g++-14   20        mode manuel
  3            /usr/bin/g++-15   30        mode manuel

Appuyez sur <Entrée> pour conserver la valeur par défaut[*],  
ou tapez un numéro de sélection :  
```

L'astérisque (`*`) indique le choix actif. Le choix `0` correspond au mode automatique (priorité la plus haute). Taper `1`, `2` ou `3` bascule en **mode manuel** : le système ne changera plus automatiquement, même si vous enregistrez plus tard une alternative de priorité supérieure.

### Mode scriptable : `--set`

Pour basculer en une seule commande sans menu interactif (utile dans des scripts d'initialisation ou de CI) :

```bash
sudo update-alternatives --set g++ /usr/bin/g++-14
```

Cette commande passe en mode manuel et fixe `g++` sur la version 14.

### Revenir au mode automatique

Si vous avez basculé en mode manuel et souhaitez laisser le système reprendre le contrôle (sélection par priorité) :

```bash
sudo update-alternatives --auto g++
```

## Consulter l'état actuel

### Afficher le choix courant

```bash
update-alternatives --display g++
```

Sortie détaillée :

```
g++ - mode automatique
  le meilleur lien vers /usr/bin/g++-15
  lien actuellement dirigé vers /usr/bin/g++-15
  lien g++ est /usr/bin/g++
  esclave g++-bin est /usr/share/man/man1/g++.1.gz
/usr/bin/g++-13 - priorité 10
/usr/bin/g++-14 - priorité 20
/usr/bin/g++-15 - priorité 30
```

### Lister tous les groupes gérés

Pour voir tous les groupes d'alternatives configurés sur votre système :

```bash
update-alternatives --get-selections
```

Cette commande affiche une ligne par groupe avec le mode (auto/manuel) et le chemin actif. Pratique pour diagnostiquer rapidement la configuration globale.

### Vérification rapide de la version active

```bash
g++ --version | head -1  
gcc --version | head -1  
clang++ --version | head -1  
```

## Groupes esclaves (`--slave`) : synchroniser les outils associés

Un compilateur ne vient jamais seul. Quand vous basculez de GCC 14 à GCC 15, vous voulez probablement que les outils associés suivent : `gcc` (compilateur C), `gcov` (couverture de code), `gcc-ar`, `gcc-ranlib`, et les pages de manuel correspondantes. Retaper `update-alternatives --config` pour chaque outil serait fastidieux.

Le mécanisme **`--slave`** permet de lier un outil secondaire au choix du groupe principal, de sorte que tout bascule en une seule commande.

### Syntaxe

```bash
sudo update-alternatives --install <lien_maître> <nom_groupe> <chemin_maître> <priorité> \
    --slave <lien_esclave_1> <nom_esclave_1> <chemin_esclave_1> \
    --slave <lien_esclave_2> <nom_esclave_2> <chemin_esclave_2>
```

### Exemple : synchroniser gcc, g++, gcov et les man pages

```bash
# GCC 14 — maître : gcc, esclaves : g++, gcov
sudo update-alternatives \
    --install /usr/bin/gcc gcc /usr/bin/gcc-14 20 \
    --slave /usr/bin/g++ g++ /usr/bin/g++-14 \
    --slave /usr/bin/gcov gcov /usr/bin/gcov-14

# GCC 15 — même structure
sudo update-alternatives \
    --install /usr/bin/gcc gcc /usr/bin/gcc-15 30 \
    --slave /usr/bin/g++ g++ /usr/bin/g++-15 \
    --slave /usr/bin/gcov gcov /usr/bin/gcov-15
```

Désormais, un seul appel à `sudo update-alternatives --config gcc` bascule simultanément `gcc`, `g++` et `gcov` vers la version choisie.

### Exemple : synchroniser clang, clang++, clang-format et clang-tidy

Le même principe s'applique à la suite LLVM, où la cohérence de version entre le compilateur et les outils d'analyse est importante :

```bash
# Clang 18
sudo update-alternatives \
    --install /usr/bin/clang clang /usr/bin/clang-18 10 \
    --slave /usr/bin/clang++ clang++ /usr/bin/clang++-18 \
    --slave /usr/bin/clang-format clang-format /usr/bin/clang-format-18 \
    --slave /usr/bin/clang-tidy clang-tidy /usr/bin/clang-tidy-18 \
    --slave /usr/bin/lldb lldb /usr/bin/lldb-18

# Clang 20
sudo update-alternatives \
    --install /usr/bin/clang clang /usr/bin/clang-20 20 \
    --slave /usr/bin/clang++ clang++ /usr/bin/clang++-20 \
    --slave /usr/bin/clang-format clang-format /usr/bin/clang-format-20 \
    --slave /usr/bin/clang-tidy clang-tidy /usr/bin/clang-tidy-20 \
    --slave /usr/bin/lldb lldb /usr/bin/lldb-20
```

Un seul `sudo update-alternatives --config clang` bascule toute la suite LLVM d'un coup.

## Supprimer une alternative

Si vous désinstallez un compilateur ou si une entrée n'est plus pertinente, retirez-la du groupe :

```bash
sudo update-alternatives --remove g++ /usr/bin/g++-13
```

Le système recalcule automatiquement le choix actif parmi les alternatives restantes (en mode automatique, la plus haute priorité prend le relais).

Pour supprimer **toutes** les alternatives d'un groupe et retirer le lien générique :

```bash
sudo update-alternatives --remove-all g++
```

> ⚠️ Attention : `--remove-all` supprime le lien `/usr/bin/g++` lui-même. N'utilisez cette commande que si vous souhaitez repartir de zéro pour ce groupe.

## Quand préférer une autre approche

`update-alternatives` modifie la configuration au **niveau système** : tous les utilisateurs et tous les projets de la machine sont affectés. C'est pratique pour définir un défaut global, mais c'est trop brutal dans certains contextes.

### Variables d'environnement `CC` et `CXX`

Pour fixer le compilateur le temps d'une session shell ou d'un projet précis, la méthode la plus simple et la plus portable :

```bash
export CC=gcc-14  
export CXX=g++-14  
cmake -B build -G Ninja  
cmake --build build  
```

CMake (et la plupart des build systems) lisent `CC` et `CXX` lors de la configuration initiale. Le choix est local à votre terminal et ne modifie rien au niveau système.

### Arguments CMake explicites

Encore plus explicite, et idéal pour les scripts de CI :

```bash
cmake -B build -G Ninja \
    -DCMAKE_C_COMPILER=/usr/bin/gcc-15 \
    -DCMAKE_CXX_COMPILER=/usr/bin/g++-15
```

Cette approche est **recommandée pour les projets professionnels** car le choix du compilateur est documenté dans la commande de build (ou dans un CMake Preset — voir section 27.6), et non caché dans une configuration système que les collègues ne partagent pas forcément.

### CMake Presets

À partir de CMake 3.19, les **Presets** permettent de figer le compilateur, le générateur, les flags et les options de build dans un fichier `CMakePresets.json` versionné avec le projet. C'est l'approche la plus reproductible et la plus adaptée au travail en équipe. Nous les étudierons en détail à la section 27.6.

### Résumé des approches

| Approche | Portée | Persistance | Cas d'usage |
|---|---|---|---|
| `update-alternatives` | Système entier | Permanente | Définir un défaut global sur une machine de développement |
| `CC` / `CXX` | Session shell | Temporaire | Tester rapidement un compilateur différent |
| `-DCMAKE_CXX_COMPILER=…` | Un projet | Durée du build dir | Scripts de CI, builds ponctuels |
| CMake Presets | Un projet | Versionnée (fichier) | Projets d'équipe, reproductibilité maximale |

## Diagnostic des problèmes courants

### `g++` ne pointe vers rien après installation

Si `g++` n'existe pas alors que `g++-15` est bien installé, c'est que le lien générique n'a pas été créé. Le paquet `build-essential` crée normalement ce lien pour la version par défaut d'Ubuntu, mais si vous avez installé une version via PPA sans `build-essential`, il faut enregistrer l'alternative manuellement :

```bash
sudo update-alternatives --install /usr/bin/g++ g++ /usr/bin/g++-15 30
```

### « L'alternative g++ n'est pas encore configurée »

Ce message signifie qu'aucune alternative n'a été enregistrée pour le groupe `g++`. Enregistrez au moins une alternative avec `--install` (voir ci-dessus).

### Le mode manuel « bloque » sur une ancienne version

Si vous avez un jour sélectionné manuellement une version et que le système ne bascule plus automatiquement vers une version plus récente, c'est normal : le mode manuel est persistant. Pour revenir au mode automatique :

```bash
sudo update-alternatives --auto g++
```

### Incohérence gcc / g++

Si `gcc --version` affiche la version 15 mais `g++ --version` affiche la version 13, c'est que les deux groupes ne sont pas synchronisés. La solution est d'utiliser le mécanisme `--slave` décrit plus haut, ou de configurer chaque groupe indépendamment pour qu'ils pointent vers la même version.

---

## Ce qu'il faut retenir

- **`update-alternatives`** est l'outil Debian/Ubuntu pour gérer proprement plusieurs versions d'un même programme derrière un nom générique.
- Le mécanisme repose sur un **système de liens symboliques à deux niveaux** (via `/etc/alternatives/`) et un système de **priorités** pour le choix automatique.
- Le flag **`--slave`** permet de synchroniser un groupe d'outils (gcc + g++ + gcov, ou clang + clang++ + clang-format + clang-tidy) pour qu'ils basculent ensemble.
- Pour les projets individuels, préférez les variables **`CC`/`CXX`**, les arguments CMake ou les **CMake Presets** — plus explicites, plus portables et sans effet de bord sur le reste du système.
