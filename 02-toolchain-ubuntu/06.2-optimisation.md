# 2.6.2 Optimisation : `-O0`, `-O2`, `-O3`, `-Os`

> **Niveau** : Débutant  
> **Prérequis** : Section 2.5 (Premier programme), Section 2.5.1 (Compilation étape par étape)  
> **Objectifs** : Comprendre ce que chaque niveau d'optimisation active, mesurer son impact sur les performances et la taille du binaire, connaître les interactions avec le débogage, et savoir choisir le bon niveau selon le contexte.

---

En section 2.5.1, nous avons comparé l'assembleur généré avec `-O0` et `-O2` pour notre fonction `aire_cercle` : le code passait d'une dizaine d'instructions à trois. Ce n'était pas un cas isolé — c'est le comportement normal du compilateur quand on lui donne l'autorisation d'optimiser.

Les niveaux d'optimisation contrôlent **l'agressivité** avec laquelle le compilateur réorganise votre code pour le rendre plus rapide ou plus compact. Chaque niveau active un ensemble de **passes d'optimisation** (des transformations algorithmiques appliquées au code intermédiaire) et donne au compilateur plus ou moins de liberté pour s'écarter de la structure littérale de votre source.

Cette liberté a un coût : plus le compilateur optimise, plus le code machine s'éloigne de votre code source, et plus le débogage devient difficile. Choisir le bon niveau, c'est trouver l'équilibre entre performance, facilité de débogage, et taille du binaire.

---

## Vue d'ensemble des niveaux

GCC et Clang proposent les mêmes niveaux d'optimisation principaux. Voici une vue synthétique avant de détailler chacun :

| Niveau | Objectif | Vitesse d'exécution | Taille du binaire | Facilité de débogage | Temps de compilation |
|--------|----------|--------------------|--------------------|---------------------|---------------------|
| `-O0` | Aucune optimisation | Référence (1x) | Moyenne | Excellente | Rapide |
| `-Og` | Optimisation pour le débogage | ~1.2-1.5x | Moyenne | Très bonne | Rapide |
| `-O1` | Optimisation légère | ~1.5-2x | Réduite | Correcte | Modéré |
| `-O2` | Optimisation standard | ~2-5x | Réduite | Difficile | Modéré |
| `-O3` | Optimisation agressive | ~2-6x | Augmentée | Très difficile | Long |
| `-Os` | Optimisation pour la taille | ~2-4x | Minimale | Difficile | Modéré |
| `-Ofast` | Vitesse maximale (non conforme) | ~3-7x | Augmentée | Très difficile | Long |

Les facteurs de vitesse sont indicatifs et varient considérablement selon le type de code. Du code numérique avec des boucles serrées bénéficie énormément de l'optimisation ; du code dominé par les entrées/sorties ou les appels système en bénéficie peu.

---

## `-O0` : Aucune optimisation

```bash
g++ -O0 main.cpp -o main
```

C'est le niveau par défaut quand aucune option `-O` n'est spécifiée. Le compilateur traduit votre code de la manière la plus **littérale** possible : chaque instruction C++ correspond directement à un bloc d'instructions machine, dans le même ordre. Les variables locales sont systématiquement stockées en mémoire (sur la pile), même quand elles pourraient résider dans un registre.

Ce niveau est idéal pour le **débogage**. Dans GDB, chaque `step` avance exactement d'une ligne de code source, chaque variable est inspectable avec `print`, et l'exécution suit le chemin exact que vous lisez dans votre source. C'est pourquoi le profil de développement recommandé en section 2.6 utilise `-O0`.

L'inconvénient est que le code produit est lent — parfois **5 à 10 fois plus lent** qu'un code optimisé. Pour la plupart des programmes, ce n'est pas un problème pendant le développement. Mais si vous travaillez sur du code numérique intensif ou des tests de performance, la différence peut rendre le programme inutilisable en `-O0`.

### Ce que `-O0` ne fait *pas*

Il serait inexact de dire que `-O0` ne fait "rien". Le compilateur effectue toujours certaines transformations minimales qui font partie intégrante de la génération de code, comme l'évaluation des expressions constantes, la résolution des appels à des fonctions `constexpr`, et les conversions de types implicites. Ce qu'il ne fait pas, c'est réorganiser, fusionner, ou éliminer des instructions pour gagner en performance.

---

## `-Og` : Le compromis pour le débogage

```bash
g++ -Og main.cpp -o main
```

Introduit avec GCC 4.8, `-Og` est un niveau d'optimisation conçu spécifiquement pour le **développement avec débogueur**. Il active les optimisations qui n'interfèrent pas avec l'expérience de débogage : les variables restent inspectables, les points d'arrêt se comportent comme attendu, et le flux d'exécution reste compréhensible.

En pratique, `-Og` offre un gain de performance significatif par rapport à `-O0` (typiquement 20 à 50 %) tout en préservant une expérience de débogage presque aussi bonne. C'est un choix pertinent si votre programme est trop lent en `-O0` pour être testé confortablement, mais que vous avez besoin de le déboguer régulièrement.

Les passes activées par `-Og` incluent l'élimination des expressions redondantes, la propagation des constantes, et certaines simplifications de code — mais pas l'inlining agressif, le déroulement de boucles, ni la réorganisation du flux de contrôle, qui rendraient le débogage confus.

> 💡 *CMake ne propose pas de build type `Og` par défaut. Avec `CMAKE_BUILD_TYPE=Debug`, il utilise `-O0 -g`. Pour utiliser `-Og`, il faut configurer manuellement les flags.*

---

## `-O1` : Optimisation légère

```bash
g++ -O1 main.cpp -o main
```

`-O1` active un premier ensemble d'optimisations "évidentes" qui améliorent la performance sans investir un temps de compilation excessif. Parmi les passes activées :

- **Élimination du code mort** (*dead code elimination*) : le code qui ne peut jamais être atteint est supprimé.
- **Propagation des constantes** (*constant propagation*) : si le compilateur peut prouver qu'une variable a toujours la même valeur, il remplace les références par la constante.
- **Fusion de blocs** (*block merging*) : les blocs de code identiques ou équivalents sont fusionnés.
- **Simplification des conditions** : les branches dont le résultat est prévisible à la compilation sont éliminées.

Le débogage reste possible en `-O1`, mais certaines variables peuvent être optimisées (supprimées ou fusionnées) et certains points d'arrêt peuvent se comporter de manière inattendue.

En pratique, `-O1` est rarement utilisé directement. Les développeurs passent généralement de `-O0`/`-Og` (développement) à `-O2` (release) sans s'arrêter à `-O1`.

---

## `-O2` : L'optimisation standard

```bash
g++ -O2 main.cpp -o main
```

C'est le niveau d'optimisation **recommandé pour la production**. Il active la grande majorité des optimisations qui améliorent la performance sans comportement imprévisible. C'est le niveau utilisé par la plupart des distributions Linux pour compiler leurs paquets, et celui que CMake sélectionne avec `CMAKE_BUILD_TYPE=Release`.

### Passes d'optimisation activées par `-O2`

`-O2` inclut toutes les passes de `-O1`, plus un ensemble d'optimisations plus sophistiquées :

**L'inlining de fonctions** est l'une des optimisations les plus impactantes. Le compilateur remplace l'appel à une fonction courte par le corps de cette fonction, éliminant le coût de l'appel (sauvegarde/restauration de registres, saut, retour). Notre fonction `aire_cercle` est un candidat idéal : elle est petite, appelée peu de fois, et le coût de l'appel est comparable au coût du calcul lui-même.

**L'ordonnancement des instructions** (*instruction scheduling*) réorganise les instructions pour maximiser l'utilisation du pipeline du processeur. Les instructions indépendantes sont entrelacées pour masquer les latences mémoire et arithmétiques.

**L'élimination des sous-expressions communes** (*common subexpression elimination*, CSE) détecte les calculs effectués plusieurs fois avec les mêmes opérandes et les factorise.

**La propagation des valeurs et le repliement des constantes** (*constant folding*) évaluent les expressions constantes à la compilation plutôt qu'à l'exécution. Par exemple, `2 * PI * rayon` avec `PI` constexpr sera partiellement calculé dès la compilation.

**L'optimisation des boucles** inclut le déplacement des invariants de boucle (*loop-invariant code motion*) : une expression qui ne change pas d'une itération à l'autre est calculée une seule fois avant la boucle.

### Observer `-O2` en action

Reprenons l'observation faite en section 2.5.1 avec notre fonction `aire_cercle` :

```bash
# Sans optimisation
g++ -O0 -S -masm=intel mathutils.cpp -o mathutils_O0.s

# Avec -O2
g++ -O2 -S -masm=intel mathutils.cpp -o mathutils_O2.s
```

Version `-O0` :

```asm
aire_cercle(double):
    endbr64
    push    rbp
    mov     rbp, rsp
    movsd   QWORD PTR [rbp-8], xmm0    ; stocker rayon en mémoire
    movsd   xmm1, QWORD PTR [rbp-8]    ; recharger rayon
    movsd   xmm0, QWORD PTR .LC0[rip]  ; charger PI
    mulsd   xmm0, xmm1                 ; PI * rayon
    mulsd   xmm0, QWORD PTR [rbp-8]    ; * rayon (re-lecture mémoire)
    pop     rbp
    ret
```

Version `-O2` :

```asm
aire_cercle(double):
    mulsd   xmm0, xmm0                 ; rayon * rayon
    mulsd   xmm0, QWORD PTR .LC0[rip]  ; * PI
    ret
```

La version optimisée est radicalement plus courte. Le stack frame a été supprimé (pas besoin de sauvegarder `rbp` pour une fonction aussi simple). Le paramètre `rayon` reste dans le registre `xmm0` sans jamais être stocké en mémoire. L'ordre des multiplications a été réorganisé : `rayon * rayon * PI` au lieu de `PI * rayon * rayon`, ce qui évite une lecture mémoire intermédiaire.

### Ce que `-O2` n'active *pas*

`-O2` évite délibérément certaines optimisations jugées trop risquées ou trop coûteuses en taille de code. Parmi elles, le **déroulement de boucles** (*loop unrolling*) et la **vectorisation automatique** ne sont pas activés par `-O2` (ils le sont par `-O3`). `-O2` n'active pas non plus les optimisations qui pourraient altérer le comportement observable du programme en arithmétique flottante.

---

## `-O3` : Optimisation agressive

```bash
g++ -O3 main.cpp -o main
```

`-O3` active toutes les optimisations de `-O2` plus des transformations plus agressives :

**Le déroulement de boucles** (*loop unrolling*) duplique le corps d'une boucle plusieurs fois pour réduire le surcoût de la gestion de la boucle (test de condition, incrémentation) et exposer plus d'opportunités de parallélisme au niveau des instructions :

```cpp
// Code source
for (int i = 0; i < 100; ++i)
    tableau[i] = i * 2;

// Après déroulement (schématique)
for (int i = 0; i < 100; i += 4) {
    tableau[i]   = i * 2;
    tableau[i+1] = (i+1) * 2;
    tableau[i+2] = (i+2) * 2;
    tableau[i+3] = (i+3) * 2;
}
```

**La vectorisation automatique** transforme les boucles scalaires en instructions SIMD (SSE, AVX) qui traitent plusieurs données en parallèle. Une boucle qui additionne des tableaux élément par élément peut être transformée pour traiter 4 `float` (SSE) ou 8 `float` (AVX2) par instruction.

**L'inlining plus agressif** augmente les seuils de taille et de profondeur pour l'inlining, intégrant des fonctions plus grandes et plus profondément imbriquées.

### Quand `-O3` aide

`-O3` brille sur le code **numériquement intensif** : algèbre linéaire, traitement d'images, simulations physiques, compression de données. Sur ces workloads, le déroulement de boucles et la vectorisation peuvent apporter un gain de 20 à 50 % par rapport à `-O2`.

### Quand `-O3` n'aide pas (ou nuit)

Sur du code dominé par les accès mémoire aléatoires, les appels système, ou les branchements imprévisibles, `-O3` n'apporte généralement rien de mesurable par rapport à `-O2`. Pire, l'inlining et le déroulement de boucles **augmentent la taille** du code machine, ce qui peut provoquer des *instruction cache misses* et **dégrader** les performances.

Ce n'est pas un scénario théorique : des benchmarks de projets réels montrent régulièrement des cas où `-O3` est plus lent que `-O2`. C'est pourquoi les distributions Linux et la majorité des projets de production utilisent `-O2`, pas `-O3`. Le passage à `-O3` doit être motivé par des **mesures concrètes**, pas par l'intuition qu'un chiffre plus élevé est forcément mieux.

```bash
# Comparer les tailles de code
size main_O2 main_O3
```

Vous constaterez que la section `.text` est généralement plus grande en `-O3`, parfois significativement.

> 📎 *La section 35 (Benchmarking) et la section 31 (Profiling de performance) fournissent les outils pour mesurer objectivement l'impact des niveaux d'optimisation sur votre code spécifique.*

---

## `-Os` : Optimisation pour la taille

```bash
g++ -Os main.cpp -o main
```

`-Os` active les mêmes optimisations que `-O2`, **sauf** celles qui augmentent la taille du code. L'inlining est plus conservateur, le déroulement de boucles est désactivé, et le compilateur favorise systématiquement le choix le plus compact quand deux alternatives sont équivalentes en vitesse.

```bash
g++ -O0 main.cpp -o main_O0  
g++ -O2 main.cpp -o main_O2  
g++ -Os main.cpp -o main_Os  
g++ -O3 main.cpp -o main_O3  

size main_O0 main_O2 main_Os main_O3
```

Un résultat typique :

```
   text    data     bss     dec     hex filename
  16842     616       8   17466    443a main_O0
  14523     616       8   15147    3b2b main_O2
  13891     616       8   14515    38b3 main_Os
  15204     616       8   15828    3dd4 main_O3
```

`-Os` produit le plus petit `.text`, tandis que `-O3` est le plus gros (plus gros même que `-O0` dans certains cas, à cause de l'inlining et du déroulement). Les sections `.data` et `.bss` sont identiques car elles dépendent des variables globales, pas du niveau d'optimisation.

### Cas d'usage de `-Os`

`-Os` est pertinent dans les contextes où la taille du binaire est une contrainte : systèmes embarqués avec une mémoire flash limitée, images Docker à minimiser, outils CLI distribués largement, ou applications mobiles soumises à des limites de taille de téléchargement.

Paradoxalement, `-Os` peut aussi être **plus rapide** que `-O2` ou `-O3` sur certains workloads. Un code plus compact tient mieux dans le cache d'instructions du processeur, ce qui réduit les *instruction cache misses*. C'est un phénomène contre-intuitif mais bien documenté, qui renforce l'importance de mesurer plutôt que de supposer.

---

## `-Ofast` : Au-delà du standard

```bash
g++ -Ofast main.cpp -o main
```

`-Ofast` active toutes les optimisations de `-O3` plus des optimisations **non conformes au standard C++**. La plus notable est `-ffast-math`, qui autorise le compilateur à réorganiser les opérations en virgule flottante de manière mathématiquement équivalente mais numériquement différente.

En arithmétique flottante, l'addition n'est pas associative à cause des erreurs d'arrondi : `(a + b) + c` peut donner un résultat différent de `a + (b + c)`. Le standard C++ garantit que le compilateur respecte l'ordre écrit par le programmeur. `-ffast-math` lève cette contrainte, permettant au compilateur de réorganiser librement les opérations pour les rendre plus rapides.

Concrètement, `-ffast-math` active un ensemble de sous-options :

- `-fno-math-errno` : les fonctions mathématiques ne positionnent plus `errno`.
- `-funsafe-math-optimizations` : réorganisations algébriques non conformes.
- `-ffinite-math-only` : suppose que les valeurs ne sont jamais `NaN` ni `±Inf`.
- `-fno-trapping-math` : suppose que les opérations flottantes ne génèrent pas d'exceptions.
- `-fno-signed-zeros` : traite `+0.0` et `-0.0` comme identiques.

C'est une option à utiliser **uniquement** si vous comprenez les implications sur la précision numérique de votre programme et que vous avez vérifié par des tests que les résultats restent acceptables. Elle est appropriée pour le rendu graphique, le traitement audio, ou certains algorithmes de machine learning où la vitesse prime sur la dernière décimale de précision. Elle est dangereuse pour la finance, la simulation scientifique, ou tout domaine où la reproductibilité bit-à-bit des résultats est requise.

> ⚠️ **`-Ofast` peut casser du code correct.** Un programme qui teste `if (std::isnan(x))` peut ne plus fonctionner avec `-ffinite-math-only`, car le compilateur suppose que `x` ne peut pas être `NaN` et élimine le test.

## Visualiser les passes d'optimisation

GCC permet de voir exactement quelles passes d'optimisation sont activées à chaque niveau :

```bash
# Lister les passes activées par -O2
g++ -O2 -Q --help=optimizers main.cpp 2>&1 | grep enabled | head -30

# Comparer -O2 et -O3
diff <(g++ -O2 -Q --help=optimizers main.cpp 2>&1 | grep enabled) \
     <(g++ -O3 -Q --help=optimizers main.cpp 2>&1 | grep enabled)
```

La commande `diff` montre exactement les passes supplémentaires activées par `-O3` par rapport à `-O2`. Sur GCC 15, vous verrez des passes comme `-ftree-loop-distribution`, `-ftree-loop-vectorize`, `-funroll-loops`, et `-fpeel-loops` apparaître dans `-O3` mais pas dans `-O2`.

Vous pouvez aussi activer ou désactiver une passe individuellement. Par exemple, pour bénéficier de `-O2` avec la vectorisation automatique de `-O3` mais sans le déroulement de boucles :

```bash
g++ -O2 -ftree-vectorize main.cpp -o main
```

Ou inversement, `-O3` sans le déroulement de boucles :

```bash
g++ -O3 -fno-unroll-loops main.cpp -o main
```

Cette granularité est rarement nécessaire en pratique, mais elle est précieuse quand un profiling (section 31) identifie un hot path qui bénéficierait d'une optimisation spécifique.

## Interaction entre optimisation et débogage

La combinaison d'options `-O` et `-g` est source de confusion pour les débutants. Les deux sont **indépendantes** et peuvent être combinées librement :

```bash
# Optimisé ET avec symboles de débogage
g++ -O2 -g main.cpp -o main
```

Cette combinaison est non seulement valide mais **recommandée pour les builds de production**. Les symboles de débogage (`-g`) n'affectent ni les performances ni le code machine produit — ils ajoutent uniquement des métadonnées dans des sections ELF séparées (`.debug_*`). Ces métadonnées permettent d'analyser des core dumps en production avec GDB, ou de profiler avec `perf` et d'obtenir des noms de fonctions lisibles au lieu d'adresses brutes.

Cependant, le débogage interactif d'un binaire optimisé est **déroutant**. L'optimisation réorganise le code de manière profonde et le débogueur doit reconstruire la correspondance entre le code machine réel et le code source original. Les symptômes typiques sont les suivants.

Les **points d'arrêt sautés** : un breakpoint sur une ligne peut ne jamais être atteint si le compilateur a éliminé cette ligne (code mort) ou l'a fusionnée avec une autre.

Les **variables "optimized out"** : quand vous tentez `print variable` dans GDB, le débogueur répond `<optimized out>` parce que la variable n'existe plus en mémoire — elle a été remplacée par une constante ou maintenue uniquement dans un registre qui a été réutilisé depuis.

L'**exécution non linéaire** : un `step` peut sembler sauter en arrière ou exécuter les lignes dans un ordre différent de celui du source, car le compilateur a réordonné les instructions.

C'est pourquoi le profil de développement utilise `-O0 -g` (débogage confortable) tandis que le profil de production utilise `-O2 -g` (performance avec possibilité d'analyse post-mortem).

> 📎 *La section 2.6.3 détaille les niveaux de débogage (`-g`, `-ggdb3`) et leur interaction avec l'optimisation.*

## Optimisations au-delà de `-O` : LTO et PGO

Les niveaux `-O` contrôlent les optimisations réalisées sur chaque **unité de traduction** individuellement. Deux techniques complémentaires permettent d'aller plus loin en travaillant au niveau du projet entier.

### Link-Time Optimization (`-flto`)

```bash
g++ -O2 -flto main.cpp mathutils.cpp -o main
```

Sans LTO, le compilateur optimise chaque fichier `.cpp` indépendamment. Il ne peut pas inliner une fonction définie dans `mathutils.cpp` lorsqu'il compile `main.cpp`, car il n'en connaît que la déclaration. Avec `-flto`, le compilateur retarde l'optimisation finale jusqu'à l'étape de linkage, où il a une vue globale de tout le programme. Cela permet l'inlining inter-fichiers, l'élimination des fonctions non appelées, et d'autres optimisations impossibles à l'échelle d'un seul fichier.

Le gain typique de LTO est de 5 à 20 % sur des projets multi-fichiers. Le coût est un temps de linkage significativement plus long et une consommation mémoire accrue.

### Profile-Guided Optimization (PGO)

```bash
# Étape 1 : Compiler avec instrumentation
g++ -O2 -fprofile-generate main.cpp mathutils.cpp -o main_instrumented

# Étape 2 : Exécuter le programme avec des données représentatives
./main_instrumented

# Étape 3 : Recompiler en exploitant les profils collectés
g++ -O2 -fprofile-use main.cpp mathutils.cpp -o main_optimized
```

PGO utilise les données d'exécution réelle pour guider les décisions d'optimisation. Le compilateur sait quelles branches sont prises fréquemment, quelles fonctions sont appelées souvent, et quelles boucles itèrent le plus — et il adapte le code en conséquence. Le gain typique est de 10 à 30 %, parfois plus sur du code avec des patterns de branchement complexes.

> 📎 *LTO et PGO sont traités en profondeur dans les sections 41.5 et 41.4 respectivement.*

## Choisir le bon niveau : Guide de décision

Le choix du niveau d'optimisation n'est pas un dilemme philosophique. Voici les recommandations pratiques selon le contexte :

**Développement quotidien** : `-O0` (ou `-Og` si `-O0` est trop lent). L'objectif est le confort de débogage et la vitesse de compilation. Les performances de l'exécutable sont secondaires.

**Tests unitaires en CI** : `-O0` avec les sanitizers (`-fsanitize=address,undefined`). L'objectif est la détection maximale de bugs. Les sanitizers sont incompatibles avec les niveaux d'optimisation élevés (certains faux positifs apparaissent, certains vrais positifs disparaissent).

**Release / Production** : `-O2`. C'est le choix par défaut le plus sûr. Il offre de bonnes performances sans les risques de `-O3`. Ajoutez `-g` pour conserver la capacité de débogage post-mortem.

**Code numérique intensif après profiling** : `-O3`, éventuellement avec `-march=native` pour exploiter les instructions SIMD spécifiques à votre processeur. Le passage de `-O2` à `-O3` doit être validé par des benchmarks sur votre code.

**Binaire minimal (embarqué, Docker)** : `-Os`. Il offre un bon compromis entre taille et performance.

**Prototypage de performance maximale** : `-Ofast`, en connaissance de cause sur la précision flottante. Jamais en production sans validation approfondie.

> ⚠️ **Ne jamais optimiser à l'aveugle.** Le passage de `-O2` à `-O3` ou `-Ofast` doit toujours être motivé par un profiling qui identifie un goulot d'étranglement, et validé par un benchmark qui confirme l'amélioration.

---

## En résumé

Les niveaux d'optimisation ne sont pas une échelle linéaire où "plus haut = mieux". Chaque niveau représente un compromis différent entre la vitesse d'exécution, la taille du binaire, la facilité de débogage, et la conformité au standard. `-O0` pour le développement, `-O2` pour la production : cette règle simple couvre 90 % des situations. Les 10 % restants demandent des mesures, pas des suppositions.

Le compilateur est remarquablement bon dans ce qu'il fait — comme nous l'avons vu avec `aire_cercle` réduite à deux instructions en `-O2`. Mais il ne peut optimiser que ce que le code lui permet d'optimiser. Un algorithme en O(n²) compilé en `-O3` restera plus lent qu'un algorithme en O(n log n) compilé en `-O0`. Les options de compilation sont le dernier levier, pas le premier.
