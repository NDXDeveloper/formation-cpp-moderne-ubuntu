# 2.7.2 Formatage type-safe et performant

> **Niveau** : Débutant  
> **Prérequis** : Section 2.7 (Introduction à std::print), Section 2.7.1 (Syntaxe et comparaison)  
> **Objectifs** : Comprendre comment `std::print` garantit la sécurité de type à la compilation, connaître les erreurs détectées automatiquement, et savoir pourquoi ses performances sont supérieures à celles de `std::cout` et comparables à celles de `printf`.

---

La section précédente a montré que `std::print` est plus concis et plus lisible que `std::cout` et `printf`. Mais la concision n'est pas le seul avantage — ni même le principal. Les deux propriétés fondamentales de `std::print` sont sa **sécurité de type à la compilation** et ses **performances**. Ce sont ces propriétés qui en font un remplacement définitif, pas une simple alternative cosmétique.

---

## Type safety : Quand le compilateur vous protège

### Le problème avec `printf`

`printf` repose sur un contrat implicite entre la chaîne de format et les arguments. Le programmeur écrit `%d` et s'engage à passer un `int`. Le compilateur fait de son mieux pour vérifier ce contrat (GCC et Clang analysent les chaînes de format avec `-Wformat`), mais cette vérification a des limites structurelles.

Considérons ces exemples :

```cpp
int nombre = 42;  
double pi = 3.14;  
std::string nom = "Alice";  

// Bug 1 : Mauvais spécificateur — %d pour un double
printf("Valeur : %d\n", pi);

// Bug 2 : Mauvais spécificateur — %f pour un int  
printf("Valeur : %f\n", nombre);

// Bug 3 : std::string passé à %s sans .c_str()
printf("Nom : %s\n", nom);

// Bug 4 : Nombre d'arguments incorrect
printf("A=%d B=%d\n", nombre);

// Bug 5 : Chaîne de format construite dynamiquement — invérifiable
std::string fmt = "Valeur : %d\n";  
printf(fmt.c_str(), pi);  
```

Chacun de ces bugs compile sans erreur (certains génèrent un warning avec `-Wformat`, mais pas tous). À l'exécution, le comportement est **indéfini** : valeurs aberrantes, crash, corruption de mémoire. Le bug 3 est particulièrement traître — passer un `std::string` à `%s` transmet l'adresse de l'objet `string` lui-même, pas celle de la chaîne de caractères contenue. Le résultat peut être un affichage de caractères aléatoires ou un segfault.

Le bug 5 illustre une limite fondamentale : quand la chaîne de format est construite à l'exécution, le compilateur ne peut **rien** vérifier. C'est un vecteur classique de vulnérabilités de type *format string attack*.

### La garantie de `std::print`

`std::print` utilise un mécanisme fondamentalement différent. La chaîne de format est analysée **à la compilation** grâce aux `consteval` fonctions et aux template parameter packs de C++20. Le compilateur connaît le type exact de chaque argument et vérifie la cohérence avec la chaîne de format avant de produire le moindre code machine.

Reprenons les mêmes cas :

```cpp
int nombre = 42;  
double pi = 3.14;  
std::string nom = "Alice";  

// Cas 1 : {} accepte n'importe quel type formatable — pas de risque
std::println("Valeur : {}", pi);       // OK : affiche 3.14  
std::println("Valeur : {}", nombre);   // OK : affiche 42  

// Cas 2 : std::string est directement supporté
std::println("Nom : {}", nom);         // OK : affiche Alice

// Cas 3 : Nombre d'arguments incorrect — ERREUR DE COMPILATION
std::println("A={} B={}", nombre);     // Erreur : pas assez d'arguments  
std::println("A={}", nombre, pi);      // Erreur : trop d'arguments  

// Cas 4 : Spécificateur incompatible avec le type — ERREUR DE COMPILATION
std::println("{:d}", nom);             // Erreur : string n'est pas un entier  
std::println("{:f}", nom);             // Erreur : string n'est pas un flottant  
```

Les cas 3 et 4 ne compilent **pas**. Le compilateur rejette le code avec un message d'erreur explicite avant que le programme ne soit exécuté. Il est **impossible** de produire un binaire contenant une incohérence format/argument avec `std::print`.

### Les messages d'erreur

Quand une incompatibilité est détectée, le compilateur produit un diagnostic. Par exemple, avec trop peu d'arguments :

```cpp
std::println("x={} y={}", 42);
```

GCC 15 affiche un message qui indique que la chaîne de format attend plus d'arguments qu'il n'en a été fourni. La localisation exacte de l'erreur dans la chaîne de format rend le diagnostic immédiat.

Avec un spécificateur incompatible :

```cpp
std::println("{:d}", "hello");
```

Le compilateur indique que le type `const char*` n'est pas compatible avec le format entier `d`. C'est une erreur **à la compilation**, pas un crash à l'exécution.

### Le placeholder `{}` : Le plus sûr par défaut

Un aspect crucial du design de `std::print` est que le placeholder vide `{}` est **toujours valide** pour tout type qui possède un `std::formatter`. Le compilateur choisit automatiquement le meilleur format par défaut : décimal pour les entiers, la représentation standard pour les flottants, le texte pour les chaînes, `true`/`false` pour les booléens.

Cela signifie que l'utilisation la plus simple est aussi la plus sûre :

```cpp
std::println("{}", 42);          // int → décimal  
std::println("{}", 3.14);        // double → représentation standard  
std::println("{}", "bonjour");   // const char* → texte  
std::println("{}", nom);         // std::string → texte  
std::println("{}", true);        // bool → "true"  
```

Avec `printf`, le programmeur doit choisir le bon spécificateur (`%d`, `%f`, `%s`...) pour chaque type. Avec `std::print`, le choix par défaut est toujours correct. Les spécificateurs explicites (`{:x}`, `{:.2f}`) ne servent qu'à **modifier** la présentation, pas à **identifier** le type.

## Vérification à la compilation : Comment ça fonctionne

Pour comprendre pourquoi `std::print` peut vérifier les formats à la compilation alors que `printf` ne le peut pas, il faut effleurer le mécanisme sous-jacent. Pas besoin de maîtriser tous les détails à ce stade — l'idée générale suffit.

### `printf` : Vérification optionnelle et limitée

`printf` est une fonction C classique avec des arguments variadiques (`...`). Le compilateur ne connaît pas les types des arguments au-delà du premier (la chaîne de format). Les vérifications que GCC et Clang proposent avec `-Wformat` sont un **service supplémentaire** du compilateur, pas une garantie du langage. Elles ne fonctionnent que quand la chaîne de format est un littéral connu à la compilation, et elles ne couvrent pas tous les cas.

### `std::print` : Vérification structurelle par le langage

`std::print` est une fonction template C++ qui reçoit la chaîne de format comme un paramètre de type `std::format_string<Args...>`. Ce type est construit de sorte que la chaîne de format est **parsée à la compilation** par une fonction `consteval`. Si le parsing échoue (nombre d'arguments incorrect, spécificateur incompatible), la construction du type échoue, et la compilation s'arrête.

En pseudocode simplifié :

```cpp
// La signature (simplifiée) de std::println
template<typename... Args>  
void println(std::format_string<Args...> fmt, Args&&... args);  
```

Le type `std::format_string<Args...>` encode les types des arguments dans le système de types C++. Son constructeur vérifie — à la compilation — que la chaîne de format est compatible avec ces types. Si ce n'est pas le cas, le constructeur est `consteval` et provoque une erreur de compilation.

Ce mécanisme est invisible pour l'utilisateur : vous écrivez `std::println("x={}", 42)` et le compilateur fait le reste. Mais c'est ce qui rend la vérification **obligatoire et infaillible**, contrairement aux warnings optionnels de `printf`.

> 📎 *La section 3.5.3 couvre `consteval` en détail. La section 12.7 explore le mécanisme complet de `std::format_string` et la création de formateurs personnalisés.*

## Types formatables par défaut

`std::print` sait formater nativement un large ensemble de types de la librairie standard. Voici les plus courants :

| Catégorie | Types supportés |
|-----------|----------------|
| Entiers | `int`, `long`, `unsigned`, `int64_t`, etc. |
| Flottants | `float`, `double`, `long double` |
| Caractères | `char`, `wchar_t`, `char8_t`, etc. |
| Booléens | `bool` |
| Chaînes | `const char*`, `std::string`, `std::string_view` |
| Pointeurs | `void*`, `const void*`, `nullptr_t` |
| Chrono | `std::chrono::duration`, `std::chrono::time_point`, etc. (C++20) |

Pour les types qui ne sont pas dans cette liste — vos propres classes, par exemple — il faut fournir une spécialisation de `std::formatter`. C'est un mécanisme plus propre et plus puissant que la surcharge de `operator<<`, mais qui demande un peu plus de code. Si aucun `std::formatter` n'existe pour un type donné, l'appel à `std::println("{}", mon_objet)` échoue **à la compilation** avec un message indiquant que le type n'est pas formatable.

```cpp
struct Point { double x, y; };  
Point p{1.0, 2.0};  

std::println("{}", p);  // Erreur de compilation : pas de std::formatter<Point>
```

C'est une différence importante avec `printf`, qui accepterait silencieusement n'importe quoi (avec un comportement indéfini), et avec `std::cout`, qui donnerait une erreur similaire mais souvent plus obscure.

> 📎 *La section 12.7 montre comment écrire un `std::formatter<Point>` pour rendre vos types compatibles avec `std::print`.*

## Performance : Plus rapide que `cout`, comparable à `printf`

La sécurité de type n'est pas le seul avantage technique de `std::print`. Ses performances sont également remarquables, souvent supérieures à `std::cout` et comparables ou supérieures à `printf`.

### Pourquoi `std::cout` est lent

La lenteur relative de `std::cout` a plusieurs causes structurelles.

La **synchronisation avec `stdio`** est activée par défaut. Pour garantir que les sorties de `std::cout` et de `printf` ne s'entrelacent pas, la librairie standard synchronise les deux systèmes d'I/O à chaque opération. Cela introduit un surcoût significatif. On peut désactiver cette synchronisation avec `std::ios_base::sync_with_stdio(false)`, mais c'est une action que le programmeur doit connaître et effectuer explicitement.

Le **modèle d'appel par `<<`** génère un appel de fonction pour chaque fragment de l'affichage. `std::cout << "x=" << x << "\n"` est trois appels séparés au système d'I/O, chacun avec son propre surcoût. `std::print("x={}\n", x)` est un seul appel qui construit le texte complet en mémoire avant de l'envoyer au flux de sortie.

Le **mécanisme des manipulateurs** ajoute une couche d'indirection et de gestion d'état qui n'est pas gratuite.

Le **flush implicite de `std::endl`** (que les débutants utilisent systématiquement à la place de `\n`) force une écriture physique du buffer à chaque ligne, ce qui peut dégrader les performances de manière dramatique dans une boucle.

### Pourquoi `std::print` est rapide

`std::print` bénéficie de plusieurs optimisations structurelles.

Le **formatage en mémoire** construit d'abord la chaîne complète dans un buffer interne, puis l'écrit en une seule opération. Cela réduit le nombre d'appels système et permet au compilateur d'optimiser la construction de la chaîne.

Le **parsing de la chaîne de format à la compilation** élimine tout le travail d'interprétation à l'exécution. Quand vous écrivez `std::println("{:.2f}", pi)`, le compilateur sait dès la compilation qu'il doit formater un `double` avec deux décimales. Le code généré appelle directement la routine de formatage appropriée, sans parser la chaîne de format à chaque appel.

La **connaissance statique des types** permet au compilateur de générer du code spécialisé pour chaque combinaison de types, sans les indirections dynamiques qu'impliquent les fonctions variadiques de `printf` ou le polymorphisme d'opérateur de `cout`.

L'**absence de synchronisation stdio** par défaut (dans les implémentations de `<print>`) et l'absence de flush implicite complètent le tableau.

### Quel gain en pratique

Les benchmarks varient selon le type de données, le volume de sortie, et l'implémentation de la librairie standard. L'ordre de grandeur est le suivant :

| Scénario | `printf` | `std::cout` | `std::cout` (optimisé) | `std::print` |
|----------|----------|-------------|----------------------|-------------|
| Entiers simples | 1x (référence) | ~2-3x plus lent | ~1.2x | ~0.8-1.2x |
| Flottants formatés | 1x | ~2-4x plus lent | ~1.5x | ~0.8-1.0x |
| Chaînes multiples | 1x | ~1.5-3x plus lent | ~1.0x | ~0.8-1.0x |

La colonne "std::cout (optimisé)" correspond à `std::cout` avec `sync_with_stdio(false)` et `\n` au lieu de `std::endl`. Même optimisé, `std::cout` reste en général plus lent que `std::print` à cause du surcoût des appels multiples par `<<`.

`std::print` est souvent **légèrement plus rapide** que `printf` sur les types C++ (`std::string` notamment), car il évite la conversion en chaîne C et les indirections variadiques. Sur les types C simples (`int`, `double`, `const char*`), les performances sont comparables.

> ⚠️ *Ces ratios sont indicatifs et dépendent de l'implémentation (libstdc++, libc++), de la version du compilateur, et du système. La section 35 (Benchmarking) fournit les outils pour mesurer sur votre configuration.*

## `std::format` : Formater sans afficher

`std::print` et `std::println` formatent **et** affichent. Parfois, on veut formater une chaîne sans l'afficher immédiatement — pour la stocker, la passer à une fonction, ou la concaténer. C'est le rôle de `std::format` :

```cpp
#include <format>
#include <string>

std::string message = std::format("Erreur {} à la ligne {}", code, ligne);
// message contient la chaîne formatée, sans aucun affichage
```

`std::format` utilise exactement la même syntaxe de chaîne de format que `std::print`, avec les mêmes garanties de type safety. La différence est qu'il retourne une `std::string` au lieu d'écrire sur `stdout`.

C'est l'outil idéal pour construire des messages de log, des chaînes de diagnostic, des réponses HTTP, ou toute situation où le texte formaté doit être traité avant d'être affiché :

```cpp
#include <format>
#include <string>
#include <vector>

std::vector<std::string> lignes;  
for (int i = 0; i < 5; ++i) {  
    lignes.push_back(std::format("Ligne {:03d} : données", i));
}
// lignes = {"Ligne 000 : données", "Ligne 001 : données", ...}
```

La relation entre les deux est simple : `std::print(fmt, args...)` est conceptuellement équivalent à `fputs(std::format(fmt, args...).c_str(), stdout)`, mais avec une implémentation plus efficace qui évite l'allocation intermédiaire d'une `std::string`.

> 💡 *`std::format` est disponible dès C++20 (header `<format>`), tandis que `std::print` nécessite C++23 (header `<print>`). Si votre compilateur supporte C++20 mais pas encore C++23, vous pouvez utiliser `std::format` combiné avec `std::cout` comme solution de transition :*  
>  
> ```cpp
> std::cout << std::format("x={}, y={}\n", x, y);
> ```

## `std::print` vers un fichier

Par défaut, `std::print` écrit sur `stdout`. Pour écrire sur un autre flux, passez un `FILE*` en premier argument :

```cpp
#include <print>
#include <cstdio>

FILE* fichier = fopen("sortie.txt", "w");  
if (fichier) {  
    std::println(fichier, "Résultat : {:.4f}", 3.14159);
    std::println(fichier, "Statut : {}", "OK");
    fclose(fichier);
}
```

On peut aussi écrire sur `stderr` :

```cpp
std::println(stderr, "Erreur : fichier {} introuvable", chemin);
```

Le premier argument `FILE*` est optionnel — quand il est omis, `stdout` est utilisé. Ce design s'intègre naturellement avec les flux C standard (`stdout`, `stderr`) et les fichiers ouverts avec `fopen`. L'intégration avec les flux C++ (`std::ofstream`) passe par `std::format` :

```cpp
#include <format>
#include <fstream>

std::ofstream out("sortie.txt");  
out << std::format("Résultat : {:.4f}\n", 3.14159);  
```

## La librairie `{fmt}` : L'ancêtre et l'alternative

`std::print` et `std::format` ne sont pas nés de rien. Ils sont la standardisation de la librairie open source **`{fmt}`** (anciennement `cppformat`), créée par Victor Zverovich. La librairie `{fmt}` existe depuis 2012 et a servi de référence et de terrain d'expérimentation pour la proposition de standardisation.

En 2026, `{fmt}` reste pertinente pour deux raisons. Elle offre des fonctionnalités **au-delà** de ce que le standard propose : couleurs et styles pour le terminal, formatage des types de conteneurs (`std::vector`, `std::map`), formatage des dates et heures avec plus de flexibilité, et support du formatage localisé. Elle sert aussi de **solution de repli** pour les projets qui doivent compiler avec des compilateurs ne supportant pas encore `<print>` ou `<format>`.

L'API de `{fmt}` est quasi identique à celle de `std::format`/`std::print` — la migration de l'un vers l'autre est triviale :

```cpp
// Avec {fmt}
#include <fmt/core.h>
fmt::println("x={}", 42);

// Avec la librairie standard C++23
#include <print>
std::println("x={}", 42);
```

Si vous travaillez sur un projet qui utilise déjà `{fmt}`, la migration vers le standard est un simple changement de namespace (`fmt::` → `std::`) et de header. Si votre compilateur ne supporte pas encore `<print>`, `{fmt}` est la solution de transition recommandée.

> 📎 *La section 2.7.3 détaille l'état du support compilateur et les stratégies de repli avec `{fmt}` et `std::format`.*

## En résumé

La sécurité de type de `std::print` n'est pas un bonus — c'est un changement fondamental. Les erreurs de format qui étaient des crashs à l'exécution avec `printf` deviennent des erreurs de compilation avec `std::print`. Le compilateur vérifie la cohérence entre la chaîne de format et les arguments de manière **obligatoire et exhaustive**, grâce au mécanisme `consteval` de `std::format_string`.

Les performances sont le deuxième pilier. En éliminant le parsing de format à l'exécution, la synchronisation stdio, et les appels multiples par `<<`, `std::print` est systématiquement plus rapide que `std::cout` et comparable à `printf`.

`std::format` complète le tableau en offrant le même mécanisme de formatage type-safe pour construire des chaînes sans les afficher. Et la librairie `{fmt}` reste disponible comme alternative et extension pour les cas que le standard ne couvre pas encore.
