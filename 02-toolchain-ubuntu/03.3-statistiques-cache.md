# 2.3.3 — Statistiques et monitoring du cache

> **Chapitre 2 · Section 2.3 · Accélération de compilation** · Niveau Débutant

---

## Introduction

Installer ccache ne suffit pas : il faut **vérifier qu'il fonctionne** et **surveiller son efficacité** dans le temps. Un taux de cache hit de 90 % signifie que neuf compilations sur dix sont servies instantanément depuis le cache — un gain de productivité énorme. Un taux de 10 %, en revanche, signale un problème de configuration qui mérite d'être diagnostiqué.

ccache fournit un ensemble de commandes et de compteurs qui permettent de suivre précisément son activité. Cette section les passe en revue et montre comment les utiliser pour tirer le meilleur parti du cache.

---

## Afficher les statistiques : `ccache --show-stats`

C'est la commande que vous utiliserez le plus souvent. Elle affiche un résumé complet de l'activité de ccache depuis la dernière remise à zéro (ou depuis l'installation si les compteurs n'ont jamais été réinitialisés) :

```bash
ccache --show-stats
```

Sortie typique après quelques jours de travail sur un projet de taille moyenne :

```
Cacheable calls:    12847 / 13210 (97.25%)
  Hits:              9631 / 12847 (74.97%)
    Direct:          8954 /  9631 (92.97%)
    Preprocessed:     677 /  9631 ( 7.03%)
  Misses:            3216 / 12847 (25.03%)
Uncacheable calls:    363 / 13210 ( 2.75%)  
Local storage:  
  Cache size (GB):   3.7 /  10.0 (37.00%)
  Hits:              9631 / 12847 (74.97%)
  Misses:            3216 / 12847 (25.03%)
```

Décortiquons chaque ligne.

---

## Anatomie des compteurs

### Cacheable calls vs Uncacheable calls

La première distinction est entre les appels que ccache **peut** mettre en cache et ceux qu'il ne peut pas.

**Cacheable calls** — les compilations « normales » (transformation d'un `.cpp` en `.o`) que ccache est en mesure de traiter. C'est la grande majorité des appels. Le pourcentage (ici 97,25 %) devrait être élevé ; un chiffre inférieur à 90 % indique que beaucoup de compilations sont dans des cas non gérés.

**Uncacheable calls** — les invocations du compilateur que ccache ne peut pas cacher. Les raisons les plus fréquentes sont :

- Appels au **préprocesseur seul** (`g++ -E ...`) — pas de fichier objet produit, rien à cacher.
- Appels au **linker** (`g++ main.o utils.o -o programme`) — ccache ne cache que la compilation, pas la liaison.
- Compilations avec des options **incompatibles** : certains flags rares empêchent le caching (par exemple `-fprofile-generate` pour PGO dans certaines configurations).
- Compilation vers **stdout** plutôt que vers un fichier.
- Compilations avec **plusieurs fichiers sources** en une seule commande (`g++ a.cpp b.cpp -o prog`).

Un faible pourcentage d'uncacheable calls (< 5 %) est normal et ne doit pas inquiéter.

### Hits et Misses

Parmi les appels cacheables, ccache distingue :

**Hits** — le résultat existait déjà dans le cache et a été retourné sans invoquer le compilateur. C'est le cas idéal. La compilation est quasi instantanée (quelques millisecondes au lieu de secondes).

**Misses** — le résultat n'existait pas dans le cache. ccache a dû invoquer le compilateur réellement, puis a stocké le résultat pour une prochaine fois. Un miss se produit lors de la première compilation d'un fichier, après une modification du code source, ou après un changement d'options de compilation.

Le **taux de hit** (`Hits / Cacheable calls`) est l'indicateur principal de l'efficacité de ccache. Voici des repères pour interpréter ce chiffre :

| Taux de hit | Interprétation |
|---|---|
| > 80 % | Excellent — ccache est bien configuré et vous en profitez pleinement |
| 60 – 80 % | Bon — normal si vous travaillez activement sur du code nouveau |
| 40 – 60 % | Moyen — vérifiez la taille du cache et la configuration `base_dir` |
| < 40 % | Faible — un problème de configuration est probable |

Un taux de hit de 100 % ne se produit que si vous recompilez exactement le même code sans aucune modification — ce qui arrive typiquement lors d'un `make clean && make` ou d'un switch de branche Git suivi d'un retour à la branche originale.

### Direct hits vs Preprocessed hits

Les hits sont subdivisés en deux catégories qui reflètent le mode dans lequel ccache a trouvé la correspondance :

**Direct hits** — ccache a calculé le hash directement à partir des fichiers source et des en-têtes, **sans invoquer le préprocesseur**. C'est le mode le plus rapide. L'inode cache (activé par défaut depuis ccache 4.7) accélère encore ce mode en évitant de re-hasher les en-têtes communs (comme les headers STL) à chaque fichier compilé.

**Preprocessed hits** — le mode direct n'a pas trouvé de correspondance, alors ccache a invoqué le préprocesseur pour produire le code prétraité, calculé un hash sur cette sortie, et trouvé une correspondance dans le cache. Ce mode est plus lent que le direct (il nécessite une invocation du préprocesseur) mais reste bien plus rapide qu'une compilation complète.

Un ratio élevé de direct hits (> 90 % des hits) indique que le mode direct fonctionne correctement. Si la majorité des hits sont des « preprocessed hits », cela peut indiquer un problème avec le mode direct — par exemple, des en-têtes dont le timestamp change sans que le contenu change (ce qui invalide le hash direct mais pas le hash prétraité).

### Cache size

```
Local storage:
  Cache size (GB):   3.7 /  10.0 (37.00%)
```

Cette ligne indique l'espace disque occupé par le cache par rapport à la limite configurée (`max_size`). Quand le cache atteint sa limite, ccache effectue un **nettoyage automatique** (cleanup) en supprimant les entrées les plus anciennes selon une politique LRU (Least Recently Used).

Si le cache est régulièrement plein (proche de 100 %), cela signifie que des entrées sont expulsées pour faire de la place — et certaines de ces entrées auraient peut-être été réutilisées plus tard. Augmenter `max_size` améliore le taux de hit en conservant plus d'entrées.

## Statistiques détaillées : `--show-stats --verbose`

Pour un affichage étendu avec des compteurs supplémentaires :

```bash
ccache --show-stats --verbose
```

Cette commande ajoute des informations comme :

```
Cacheable calls:    12847 / 13210 (97.25%)
  Hits:              9631 / 12847 (74.97%)
    Direct:          8954 /  9631 (92.97%)
    Preprocessed:     677 /  9631 ( 7.03%)
  Misses:            3216 / 12847 (25.03%)
Errors:                 0  
Uncacheable calls:    363 / 13210 ( 2.75%)  
Local storage:  
  Cache size (GB):   3.7 /  10.0 (37.00%)
  Hits:              9631 / 12847 (74.97%)
  Misses:            3216 / 12847 (25.03%)
  Errors:               0
  Cleanups:            12
```

Le compteur **Cleanups** indique combien de fois ccache a dû supprimer des entrées anciennes pour respecter la limite `max_size`. Un nombre élevé de cleanups combiné à un taux de hit inférieur aux attentes est un signe clair qu'il faut augmenter la taille du cache.

Le compteur **Errors** indique les erreurs internes de ccache (problèmes de lecture/écriture du cache, permissions, etc.). Il devrait être à zéro ; une valeur non nulle nécessite une investigation.

## Statistiques en format machine

Pour intégrer les statistiques de ccache dans des scripts de monitoring, des dashboards ou des pipelines CI, le format tabulaire est disponible :

```bash
ccache --print-stats
```

Cette commande affiche les compteurs bruts sous forme de paires clé-valeur, facilement parsables :

```
cache_miss 3216  
cache_hit_direct 8954  
cache_hit_preprocessed 677  
...
```

Vous pouvez extraire une métrique spécifique :

```bash
# Extraire le taux de hit pour un rapport CI
HITS=$(ccache --print-stats | grep cache_hit_direct | awk '{print $2}')  
MISSES=$(ccache --print-stats | grep cache_miss | awk '{print $2}')  
TOTAL=$((HITS + MISSES))  
if [ "$TOTAL" -gt 0 ]; then  
    RATE=$(echo "scale=1; $HITS * 100 / $TOTAL" | bc)
    echo "ccache direct hit rate: ${RATE}%"
fi
```

## Réinitialiser les statistiques

### Remettre les compteurs à zéro

```bash
ccache --zero-stats
```

Cette commande remet tous les compteurs à zéro **sans toucher au contenu du cache**. Les fichiers objets cachés restent en place — seuls les compteurs sont réinitialisés.

C'est utile dans plusieurs scénarios :

- **Mesurer l'impact de ccache sur un build spécifique.** Remettez à zéro, lancez le build, puis consultez les stats pour voir exactement combien de hits et de misses ce build a produit.
- **Début d'une nouvelle session de travail.** Si vous voulez des statistiques « fraîches » chaque jour ou chaque semaine.
- **Après un changement de configuration.** Pour évaluer l'effet d'un changement (modification de `base_dir`, `max_size`, etc.) sur le taux de hit.

### Exemple : mesurer un rebuild

```bash
# Mesure propre d'un rebuild complet
ccache --zero-stats

cmake --build build --target clean  
cmake --build build  

echo "=== Résultat du rebuild ==="  
ccache --show-stats  
```

Si le projet n'a pas été modifié depuis le dernier build, le rebuild après clean devrait afficher un taux de hit proche de 100 % — preuve que ccache fonctionne parfaitement.

## Maintenance du cache

### Vider le cache

```bash
ccache --clear
```

Supprime **tout le contenu** du cache. Les compteurs de statistiques ne sont pas affectés. Cette opération est rarement nécessaire ; les cas typiques sont :

- **Changement majeur de compilateur** (passage de GCC 14 à GCC 15) : les entrées de l'ancien compilateur ne seront jamais réutilisées et occupent de l'espace inutilement.
- **Suspicion de corruption** : si des compilations produisent des résultats erronés et que vous suspectez le cache.
- **Libération d'espace disque** en urgence.

Pour une remise à zéro complète (cache + statistiques) :

```bash
ccache --clear  
ccache --zero-stats  
```

### Nettoyage manuel

ccache effectue des nettoyages automatiques quand la limite `max_size` est atteinte. Vous pouvez aussi déclencher un nettoyage manuellement :

```bash
ccache --cleanup
```

Cette commande parcourt le cache et supprime les entrées les plus anciennes jusqu'à ce que la taille soit en dessous de la limite configurée. En pratique, les nettoyages automatiques suffisent et le nettoyage manuel est rarement utile.

### Recompresser le cache

Si vous modifiez le niveau de compression dans votre configuration (par exemple, passage de `compression_level = 1` à `compression_level = 5` pour économiser de l'espace), vous pouvez recompresser les entrées existantes sans les invalider :

```bash
# Recompresser avec le niveau configuré
ccache --recompress

# Recompresser avec un niveau spécifique (sans modifier la config)
ccache --recompress 5

# Décompresser tout le cache (compression_level = 0)
ccache --recompress 0
```

La recompression peut prendre un certain temps sur un gros cache, mais elle se fait en arrière-plan et ne bloque pas les compilations.

## Le fichier de log : diagnostic en profondeur

Quand les statistiques globales ne suffisent pas à comprendre un problème (par exemple, un fichier spécifique qui devrait être un hit mais est systématiquement un miss), le **fichier de log** de ccache fournit un diagnostic détaillé de chaque invocation.

### Activer le logging

```bash
# Via la configuration
ccache --set-config log_file=/tmp/ccache.log

# Ou via variable d'environnement (temporaire)
export CCACHE_LOGFILE=/tmp/ccache.log
```

### Lancer un build et examiner le log

```bash
cmake --build build --target clean  
cmake --build build  

# Examiner les dernières entrées du log
tail -100 /tmp/ccache.log
```

### Anatomie d'une entrée de log

Chaque invocation du compilateur produit un bloc dans le log. Voici un exemple annoté d'un **cache hit** :

```
[2026-03-09T10:23:45.123] [pid 42567] Command line: ccache g++-15 -std=c++23 -O2 -Wall -c main.cpp -o main.o
[2026-03-09T10:23:45.124] [pid 42567] Compiler: /usr/bin/g++-15
[2026-03-09T10:23:45.124] [pid 42567] Source file: main.cpp
[2026-03-09T10:23:45.125] [pid 42567] Object file: main.o
[2026-03-09T10:23:45.126] [pid 42567] Direct mode: looking up abcdef1234567890...
[2026-03-09T10:23:45.127] [pid 42567] Direct cache hit
[2026-03-09T10:23:45.128] [pid 42567] Result: cache hit (direct)
```

Et un **cache miss** :

```
[2026-03-09T10:23:46.200] [pid 42568] Command line: ccache g++-15 -std=c++23 -O2 -Wall -c utils.cpp -o utils.o
[2026-03-09T10:23:46.201] [pid 42568] Compiler: /usr/bin/g++-15
[2026-03-09T10:23:46.201] [pid 42568] Source file: utils.cpp
[2026-03-09T10:23:46.202] [pid 42568] Object file: utils.o
[2026-03-09T10:23:46.203] [pid 42568] Direct mode: looking up 9876543210fedcba...
[2026-03-09T10:23:46.204] [pid 42568] No direct cache hit
[2026-03-09T10:23:46.205] [pid 42568] Running preprocessor...
[2026-03-09T10:23:46.350] [pid 42568] Preprocessor mode: looking up aabb112233445566...
[2026-03-09T10:23:46.351] [pid 42568] No preprocessed cache hit
[2026-03-09T10:23:46.352] [pid 42568] Running compiler...
[2026-03-09T10:24:47.890] [pid 42568] Result: cache miss
```

Le log permet de voir exactement pourquoi un fichier est un miss : le hash direct ne correspond pas (le fichier ou un en-tête a changé), le hash prétraité ne correspond pas non plus (le contenu effectif a changé), et le compilateur est invoqué.

### Chercher un fichier spécifique dans le log

```bash
grep "utils.cpp" /tmp/ccache.log | tail -10
```

### Mode debug avancé

Pour un diagnostic encore plus fin, ccache peut écrire les fichiers d'entrée hashés dans un répertoire dédié :

```bash
# Activer le mode debug
ccache --set-config debug_dir=/tmp/ccache-debug

# Compiler un fichier
ccache g++ -std=c++23 -O2 -c main.cpp -o main.o

# Examiner les fichiers de debug
ls /tmp/ccache-debug/
```

Le répertoire contiendra les fichiers source prétraités, les manifestes et les résultats utilisés pour le calcul du hash. C'est utile pour comprendre les cas subtils où deux compilations apparemment identiques produisent des hash différents.

### Désactiver le logging

Le logging ajoute un léger overhead (écriture disque à chaque compilation) et produit des fichiers volumineux. Désactivez-le après le diagnostic :

```bash
ccache --set-config log_file=
# Ou
unset CCACHE_LOGFILE
```

## Diagnostiquer un taux de hit faible

Voici un guide de diagnostic systématique quand votre taux de hit est inférieur aux attentes.

### Étape 1 : Vérifier la taille du cache

```bash
ccache --show-stats 2>&1 | grep -E "Cache size|Cleanups"
```

Si le cache est régulièrement plein et que les cleanups sont fréquents, les entrées sont expulsées avant d'être réutilisées. La solution est d'augmenter `max_size` :

```bash
ccache --set-config max_size=20G
```

### Étape 2 : Vérifier la stabilité des chemins

Si vous compilez le même projet depuis des répertoires différents (typique en CI), les chemins absolus dans les options de compilation créent des hash différents. Activez `base_dir` :

```bash
ccache --set-config base_dir=/home/votre_user
# En CI :
export CCACHE_BASEDIR=/ci/workspace
```

### Étape 3 : Vérifier la stabilité du compilateur

Chaque mise à jour du compilateur (même une version mineure, par exemple 15.1 → 15.2) invalide toutes les entrées du cache car le binaire du compilateur fait partie du hash. C'est un comportement voulu — les résultats de compilation peuvent différer entre versions.

Vérifiez si le compilateur a changé récemment :

```bash
g++ --version | head -1
```

Après une mise à jour du compilateur, un cache miss complet lors du premier build est normal. Le cache se remplira à nouveau au fil des compilations.

### Étape 4 : Vérifier les macros temporelles

Si votre code ou des bibliothèques tierces utilisent `__DATE__`, `__TIME__` ou `__TIMESTAMP__`, le résultat change à chaque compilation, invalidant le cache. Deux solutions :

```bash
# Solution 1 : Tolérer les macros temporelles
ccache --set-config sloppiness=time_macros

# Solution 2 : Fixer la date de compilation (meilleure reproductibilité)
export SOURCE_DATE_EPOCH=$(date +%s)
```

`SOURCE_DATE_EPOCH` est une convention standard (adoptée par Debian et d'autres projets) : quand cette variable est définie, ccache ne tient compte de sa valeur que si le code contient effectivement `__DATE__` ou `__TIME__`.

### Étape 5 : Examiner le log

Si les étapes précédentes ne suffisent pas, activez le log (voir ci-dessus) et examinez les raisons des misses pour les fichiers que vous attendiez comme hits.

### Arbre de décision récapitulatif

```
Taux de hit faible
  │
  ├─ Cache régulièrement plein ?
  │   └─ OUI → Augmenter max_size
  │
  ├─ Chemins de build différents ?
  │   └─ OUI → Configurer base_dir
  │
  ├─ Compilateur récemment mis à jour ?
  │   └─ OUI → Normal — le cache se reconstruira
  │
  ├─ __DATE__ / __TIME__ utilisés ?
  │   └─ OUI → sloppiness=time_macros ou SOURCE_DATE_EPOCH
  │
  └─ Aucune des causes ci-dessus ?
      └─ Activer le log et examiner les misses individuellement
```

## Automatiser le monitoring

### Script de rapport post-build

Vous pouvez intégrer un rapport ccache à la fin de vos scripts de build :

```bash
#!/bin/bash
# build-with-report.sh — Build + rapport ccache

ccache --zero-stats > /dev/null 2>&1

echo "=== Build en cours ==="  
cmake --build build -j$(nproc)  
BUILD_EXIT=$?  

echo ""  
echo "=== Rapport ccache ==="  
ccache --show-stats 2>&1 | grep -E "Cacheable|Hits|Misses|Cache size"  

exit $BUILD_EXIT
```

### Intégration CI : publier les stats comme artefact

Dans un pipeline CI (GitHub Actions, GitLab CI), vous pouvez publier les statistiques ccache dans les logs du job ou comme artefact :

```yaml
# Extrait d'un fichier .github/workflows/build.yml
- name: Build
  run: |
    ccache --zero-stats
    cmake --build build
    echo "## ccache stats" >> $GITHUB_STEP_SUMMARY
    echo '```' >> $GITHUB_STEP_SUMMARY
    ccache --show-stats >> $GITHUB_STEP_SUMMARY
    echo '```' >> $GITHUB_STEP_SUMMARY
```

Ce pattern publie les statistiques ccache directement dans le résumé du job GitHub Actions, visible dans l'interface web. L'équivalent GitLab CI utilise une section repliable dans les logs. Nous détaillerons ces intégrations à la section 38.3.1.

## Ce qu'il faut retenir

- **`ccache --show-stats`** est la commande de base : elle affiche le taux de hit, la répartition direct/preprocessed, les misses, et la taille du cache. Consultez-la régulièrement.
- Le **taux de hit** est votre indicateur principal : > 80 % est excellent, < 40 % nécessite un diagnostic.
- Les **direct hits** (> 90 % des hits) indiquent que le mode direct fonctionne correctement. Si la majorité des hits sont « preprocessed », le mode direct est perturbé.
- **`ccache --zero-stats`** remet les compteurs à zéro sans toucher au cache — utile pour mesurer l'impact de ccache sur un build spécifique.
- **`ccache --clear`** vide le cache entier — rarement nécessaire, réservé aux changements majeurs de compilateur ou aux suspicions de corruption.
- Le **fichier de log** (`log_file`) est l'outil de diagnostic ultime quand les statistiques globales ne suffisent pas : il montre la raison exacte de chaque hit et miss.
- Un taux de hit faible a quatre causes principales : **cache trop petit**, **chemins instables** (→ `base_dir`), **compilateur mis à jour** et **macros temporelles** (→ `sloppiness=time_macros`).
