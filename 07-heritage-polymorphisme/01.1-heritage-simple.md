# 7.1.1 — Héritage simple

## Section 7.1 : Héritage simple et multiple · Chapitre 7 : Héritage et Polymorphisme

---

## Introduction

L'héritage simple est la forme la plus courante et la plus intuitive de l'héritage : une classe dérivée hérite d'**une seule classe de base directe**. La hiérarchie résultante forme un arbre, chaque nœud ayant au plus un parent.

C'est le mécanisme que vous utiliserez dans la grande majorité des cas. Avant d'aborder les complications de l'héritage multiple (section 7.1.2), il est essentiel de maîtriser parfaitement la mécanique de l'héritage simple : comment les objets sont construits et détruits, comment la mémoire est disposée, et quels pièges guettent le développeur qui ne prête pas attention à des subtilités comme le *slicing*.

---

## Syntaxe de base

L'héritage simple se déclare en ajoutant `: public NomDeLaBase` après le nom de la classe dérivée :

```cpp
#include <string>
#include <print>

class Animal {  
protected:  
    std::string nom_;
    int age_;

public:
    Animal(std::string nom, int age)
        : nom_{std::move(nom)}, age_{age} {}

    void presenter() const {
        std::println("Je suis {}, {} an(s)", nom_, age_);
    }

    std::string const& nom() const { return nom_; }
    int age() const { return age_; }
};

class Chat : public Animal {
    bool interieur_;   // vit en intérieur ?

public:
    Chat(std::string nom, int age, bool interieur)
        : Animal{std::move(nom), age}   // appel au constructeur de la base
        , interieur_{interieur} {}

    void ronronner() const {
        std::println("{} ronronne...", nom_);  // accès à nom_ (protected)
    }

    bool est_interieur() const { return interieur_; }
};
```

Quelques points à noter immédiatement :

- Le mot-clé `public` devant `Animal` indique un héritage public. Sans lui, l'héritage serait **`private`** par défaut (car `Chat` est déclaré avec `class`), ce qui empêcherait la conversion implicite `Chat → Animal`.  
- Le constructeur de `Chat` **doit** initialiser la partie `Animal` en appelant explicitement un constructeur de `Animal` dans sa liste d'initialisation. Si aucun appel n'est écrit, le compilateur tente d'appeler le constructeur **par défaut** de `Animal` — qui n'existe pas ici, donc la compilation échoue.  
- Le membre `nom_` est `protected` dans `Animal`, ce qui permet à `Chat` d'y accéder directement. S'il était `private`, seules les méthodes d'`Animal` pourraient y accéder, et `Chat` devrait passer par l'accesseur public `nom()`.

L'utilisation est naturelle :

```cpp
int main() {
    Chat felix{"Félix", 3, true};
    felix.presenter();    // méthode héritée d'Animal
    felix.ronronner();    // méthode propre à Chat

    // Conversion implicite : un Chat "est-un" Animal
    Animal const& ref = felix;
    ref.presenter();      // OK — accès via référence de base
}
```

```
Je suis Félix, 3 an(s)  
Félix ronronne...  
Je suis Félix, 3 an(s)  
```

## Disposition mémoire

Un objet de type `Chat` contient en mémoire un **sous-objet** de type `Animal`, suivi des membres propres à `Chat`. Concrètement, le layout ressemble à ceci :

```
Objet Chat (felix)
┌───────────────────────────────┐
│  Sous-objet Animal            │
│  ┌─────────────────────────┐  │
│  │ nom_  (std::string)     │  │
│  │ age_  (int)             │  │
│  └─────────────────────────┘  │
│  interieur_  (bool)           │  ← membre propre à Chat
└───────────────────────────────┘
```

On peut le vérifier avec `sizeof` :

```cpp
std::println("sizeof(Animal) = {}", sizeof(Animal));  
std::println("sizeof(Chat)   = {}", sizeof(Chat));  
```

La taille de `Chat` sera **au moins** égale à `sizeof(Animal) + sizeof(bool)`, avec un éventuel padding d'alignement (voir section 3.2.3 pour `sizeof` et `alignof`).

Cette disposition est garantie par le standard : le sous-objet de la classe de base est construit en premier et se situe au début de l'objet dérivé (en l'absence de fonctions virtuelles — ce point sera nuancé en section 7.2 avec le vptr).

## Ordre de construction et de destruction

L'ordre dans lequel les sous-objets sont construits et détruits est **déterministe** et fondamental à comprendre. On peut l'observer en ajoutant des traces :

```cpp
#include <print>

class Base {  
public:  
    Base()  { std::println("Base::Base()"); }
    ~Base() { std::println("Base::~Base()"); }
};

class Derivee : public Base {  
public:  
    Derivee()  { std::println("Derivee::Derivee()"); }
    ~Derivee() { std::println("Derivee::~Derivee()"); }
};

int main() {
    Derivee d;
}
```

```
Base::Base()  
Derivee::Derivee()  
Derivee::~Derivee()  
Base::~Base()  
```

La règle est simple et invariable :

**Construction : de la base vers la dérivée.** Le sous-objet `Base` est entièrement construit avant que le corps du constructeur de `Derivee` ne s'exécute. Cela garantit que `Derivee` peut utiliser les membres hérités de `Base` dès son propre constructeur.

**Destruction : de la dérivée vers la base.** Le destructeur de `Derivee` s'exécute en premier, puis celui de `Base`. C'est l'ordre inverse de la construction — le même principe RAII que pour les membres d'une classe (section 6.3).

Avec plusieurs niveaux d'héritage, le principe se généralise. Si `C` hérite de `B` qui hérite de `A` :

```
Construction : A → B → C  
Destruction  : C → B → A  
```

## Contrôle de l'appel au constructeur de base

Le constructeur de la classe dérivée contrôle **quel** constructeur de la base est appelé via la liste d'initialisation. C'est le seul endroit où cet appel peut être fait :

```cpp
class Forme {
    std::string couleur_;
public:
    Forme() : couleur_{"noir"} {}
    explicit Forme(std::string couleur) : couleur_{std::move(couleur)} {}

    std::string const& couleur() const { return couleur_; }
};

class Cercle : public Forme {
    double rayon_;
public:
    // Appelle Forme(string) explicitement
    Cercle(std::string couleur, double rayon)
        : Forme{std::move(couleur)}, rayon_{rayon} {}

    // Appelle Forme() par défaut (implicitement)
    explicit Cercle(double rayon)
        : rayon_{rayon} {}  // Forme() est appelé automatiquement

    double rayon() const { return rayon_; }
};
```

Si la classe de base n'a **pas** de constructeur par défaut et que la classe dérivée omet l'appel explicite, la compilation échoue. C'est une erreur fréquente :

```cpp
class SansDefaut {  
public:  
    SansDefaut(int x) {}  // pas de constructeur par défaut
};

class Enfant : public SansDefaut {  
public:  
    Enfant() {}  // ❌ Erreur : pas de constructeur par défaut pour SansDefaut
};
```

## Héritage de constructeurs avec `using` (C++11)

Quand une classe dérivée ne fait qu'ajouter des membres sans modifier la logique de construction de la base, réécrire tous les constructeurs est fastidieux. Le mot-clé `using` permet d'**importer** les constructeurs de la classe de base :

```cpp
class Connexion {
    std::string hote_;
    int port_;
public:
    Connexion(std::string hote, int port)
        : hote_{std::move(hote)}, port_{port} {}

    explicit Connexion(std::string hote)
        : Connexion{std::move(hote), 443} {}  // délégation
};

class ConnexionSecurisee : public Connexion {
    bool tls_actif_ = true;  // initialiseur par défaut (C++11)

public:
    using Connexion::Connexion;  // importe TOUS les constructeurs de Connexion
};
```

On peut alors écrire :

```cpp
ConnexionSecurisee c1{"example.com", 8443};  // utilise Connexion(string, int)  
ConnexionSecurisee c2{"example.com"};         // utilise Connexion(string)  
```

Les constructeurs hérités initialisent la partie `Connexion` exactement comme le ferait la base. Les membres propres à `ConnexionSecurisee` (ici `tls_actif_`) sont initialisés via leur **initialiseur par défaut** (*default member initializer*). Si un membre propre n'a pas d'initialiseur par défaut et n'est pas initialisé autrement, il restera non initialisé — ce qui est un piège à surveiller.

> 💡 Les constructeurs de copie et de déplacement ne sont **pas** importés par `using`. Ils continuent d'être générés (ou supprimés) selon les règles habituelles de la Rule of Five (section 6.5).

## Redéfinition de méthodes (*hiding*)

Une classe dérivée peut définir une méthode portant le même nom qu'une méthode de la classe de base. En l'absence de `virtual` (qui sera traité en sections 7.2 et 7.3), ce mécanisme est appelé **name hiding** : la méthode de la dérivée **masque** celle de la base.

```cpp
class Logger {  
public:  
    void log(std::string const& message) {
        std::println("[LOG] {}", message);
    }
};

class TimestampLogger : public Logger {  
public:  
    void log(std::string const& message) {
        std::println("[2026-03-10 14:30:00] {}", message);
    }
};
```

Le comportement dépend du **type statique** de la variable, pas du type réel de l'objet :

```cpp
TimestampLogger tl;  
tl.log("test");             // appelle TimestampLogger::log  

Logger& ref = tl;  
ref.log("test");            // appelle Logger::log — PAS TimestampLogger::log !  
```

```
[2026-03-10 14:30:00] test
[LOG] test
```

C'est un piège classique. Le name hiding n'est **pas** du polymorphisme : l'appel est résolu à la **compilation** en fonction du type statique. Pour obtenir un comportement polymorphique (résolution à l'exécution en fonction du type réel), il faut déclarer la méthode `virtual` dans la classe de base — ce qui est le sujet de la section 7.2.

### Masquage de surcharges

Le name hiding peut aussi masquer des **surcharges** de la base de manière inattendue :

```cpp
class Base {  
public:  
    void traiter(int x)    { std::println("Base::traiter(int): {}", x); }
    void traiter(double x) { std::println("Base::traiter(double): {}", x); }
};

class Derivee : public Base {  
public:  
    void traiter(int x)    { std::println("Derivee::traiter(int): {}", x); }
};

int main() {
    Derivee d;
    d.traiter(42);     // OK — Derivee::traiter(int)
    d.traiter(3.14);   // ⚠️ Appelle Derivee::traiter(int) avec conversion !
                       //    Base::traiter(double) est masquée
}
```

La méthode `traiter(double)` de `Base` est complètement masquée, même si `Derivee` ne la redéfinit pas. Le `3.14` est converti en `int` pour appeler `Derivee::traiter(int)`. Pour rendre les surcharges de la base accessibles, on utilise une déclaration `using` :

```cpp
class Derivee : public Base {  
public:  
    using Base::traiter;       // rend visibles TOUTES les surcharges de Base
    void traiter(int x) { std::println("Derivee::traiter(int): {}", x); }
};
```

Avec ce `using`, `d.traiter(3.14)` appelle bien `Base::traiter(double)` tandis que `d.traiter(42)` appelle `Derivee::traiter(int)`.

## Le problème du *slicing*

Le *slicing* (ou *object slicing*) est l'une des erreurs les plus insidieuses liées à l'héritage en C++. Il se produit lorsqu'un objet dérivé est **copié par valeur** dans une variable de type base :

```cpp
class Animal {
    std::string nom_;
public:
    Animal(std::string nom) : nom_{std::move(nom)} {}
    void parler() const { std::println("{} fait du bruit", nom_); }
    std::string const& nom() const { return nom_; }
};

class Chien : public Animal {
    std::string race_;
public:
    Chien(std::string nom, std::string race)
        : Animal{std::move(nom)}, race_{std::move(race)} {}

    void parler() const { std::println("{} aboie ({})", nom(), race_); }
};
```

Le slicing en action :

```cpp
Chien rex{"Rex", "Berger Allemand"};

Animal a = rex;   // ⚠️ SLICING : seule la partie Animal est copiée  
a.parler();       // "Rex fait du bruit" — la partie Chien est perdue  
```

Ce qui se passe en mémoire :

```
rex (Chien)                    a (Animal)
┌──────────────────────┐      ┌──────────────────────┐
│ Animal::nom_ = "Rex" │ ──►  │ nom_ = "Rex"         │
│ race_ = "Berger..."  │      └──────────────────────┘
└──────────────────────┘       race_ n'existe pas dans a
```

La variable `a` est de type `Animal`, pas `Chien`. La copie n'a conservé que le sous-objet `Animal`. Le membre `race_` et la méthode `Chien::parler()` sont irrémédiablement perdus. Ce n'est pas un bug du compilateur — c'est le comportement défini par le langage quand on copie par valeur.

### Comment éviter le slicing

La règle fondamentale est : **manipulez les objets polymorphiques par pointeur ou par référence, jamais par valeur.**

```cpp
Chien rex{"Rex", "Berger Allemand"};

Animal& ref = rex;     // ✅ Référence — pas de copie, pas de slicing  
Animal* ptr = &rex;    // ✅ Pointeur — pas de copie, pas de slicing  
```

En C++ moderne, cela signifie concrètement utiliser des smart pointers (chapitre 9) :

```cpp
#include <memory>
#include <vector>

std::vector<std::unique_ptr<Animal>> animaux;  
animaux.push_back(std::make_unique<Chien>("Rex", "Berger Allemand"));  
animaux.push_back(std::make_unique<Chien>("Médor", "Labrador"));  
```

Chaque élément du vecteur est un pointeur vers un objet alloué sur le heap. Aucun slicing ne se produit. Combiné avec des fonctions virtuelles (section 7.3), chaque appel à `parler()` invoquerait la bonne version.

> 💡 Un moyen de se protéger défensivement contre le slicing est de rendre la classe de base **non copiable** si elle est destinée à être utilisée de manière polymorphique. On supprime alors le constructeur de copie et l'opérateur d'affectation :  
>
> ```cpp
> class Animal {
> public:
>     Animal(Animal const&) = delete;
>     Animal& operator=(Animal const&) = delete;
>     // ...
> };
> ```
>  
> Toute tentative de copie par valeur provoquera alors une erreur de compilation.

## Accès aux membres de la base avec l'opérateur de portée

Depuis une classe dérivée, on peut appeler explicitement une méthode de la classe de base en utilisant l'opérateur de portée `::` :

```cpp
class Fichier {  
public:  
    void ouvrir() {
        std::println("Ouverture fichier standard");
    }
};

class FichierChiffre : public Fichier {  
public:  
    void ouvrir() {
        Fichier::ouvrir();  // appel explicite à la version de base
        std::println("Initialisation du déchiffrement");
    }
};
```

Ce pattern est courant quand la classe dérivée veut **étendre** le comportement de la base plutôt que le remplacer entièrement. L'appel `Fichier::ouvrir()` est résolu à la compilation — c'est un appel direct, pas un appel virtuel, même si la méthode était `virtual`.

## Conversion entre base et dérivée

L'héritage public établit une relation de sous-typage. Les conversions suivantes sont valides :

### Dérivée → Base (upcast) : implicite et sûr

```cpp
Chat felix{"Félix", 3, true};

Animal* ptr = &felix;        // ✅ Conversion implicite  
Animal& ref = felix;          // ✅ Conversion implicite  
```

L'upcast est toujours valide car un `Chat` contient un sous-objet `Animal` complet. Le pointeur ou la référence pointe simplement vers ce sous-objet.

### Base → Dérivée (downcast) : explicite et dangereux

La conversion inverse nécessite un cast explicite et n'est sûre que si l'objet pointé est réellement du type cible :

```cpp
Animal* ptr = &felix;

// static_cast : pas de vérification à l'exécution
Chat* c1 = static_cast<Chat*>(ptr);   // OK ici, car ptr pointe bien vers un Chat

// dynamic_cast : vérification à l'exécution (nécessite virtual — section 7.2)
// Chat* c2 = dynamic_cast<Chat*>(ptr);   // nécessite au moins une fonction virtuelle
```

Le `static_cast` fait confiance au développeur. Si `ptr` ne pointe pas réellement vers un `Chat`, le comportement est **indéfini**. Le `dynamic_cast` effectue une vérification à l'exécution et retourne `nullptr` en cas d'échec (pour les pointeurs), mais il nécessite que la classe de base ait au moins une fonction virtuelle — ce qui sera couvert en section 7.2 et approfondi avec `dynamic_cast` en section 3.3.4.

## Bonnes pratiques

**Préférez l'héritage `public`.** Les héritages `protected` et `private` ont des usages légitimes mais rares. Si vous constatez que l'héritage `private` est ce dont vous avez besoin, demandez-vous si la **composition** (un membre de type `Base`) ne serait pas plus claire.

**Initialisez toujours la base dans la liste d'initialisation.** Ne comptez pas sur le constructeur par défaut implicite quand la base a des invariants à établir.

**Utilisez `override` dès que vous redéfinissez une méthode virtuelle** (section 7.3). Pour les méthodes non virtuelles, soyez conscient du name hiding et utilisez `using Base::methode` si nécessaire.

**Ne passez jamais un objet polymorphique par valeur.** Utilisez des références, des pointeurs ou des smart pointers pour éviter le slicing.

**Pensez au destructeur virtuel.** Si une classe est conçue pour servir de base dans une hiérarchie polymorphique, son destructeur doit être `virtual`. Si elle n'est pas conçue pour l'héritage, envisagez de la marquer `final` (section 7.3).

---

## Résumé

| Concept | Comportement |  
|---|---|  
| Ordre de construction | Base → Dérivée |  
| Ordre de destruction | Dérivée → Base |  
| Mode d'héritage par défaut (`class`) | `private` |  
| Mode d'héritage par défaut (`struct`) | `public` |  
| Conversion Dérivée → Base | Implicite (upcast) |  
| Conversion Base → Dérivée | Explicite (`static_cast`, `dynamic_cast`) |  
| Redéfinition sans `virtual` | Name hiding (résolution à la compilation) |  
| Copie d'un objet dérivé dans une variable de base | Slicing (perte de la partie dérivée) |  
| `using Base::Base;` | Importe les constructeurs de la base |  
| `using Base::methode;` | Rend visibles les surcharges masquées de la base |
