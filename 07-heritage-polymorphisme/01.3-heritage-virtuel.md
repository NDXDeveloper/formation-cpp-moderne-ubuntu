# 7.1.3 — Héritage virtuel

## Section 7.1 : Héritage simple et multiple · Chapitre 7 : Héritage et Polymorphisme

---

## Introduction

La section précédente a mis en évidence le problème du diamant : quand deux classes de base partagent un ancêtre commun, la classe dérivée finale se retrouve avec **deux copies** du sous-objet ancêtre, entraînant des ambiguïtés et des incohérences potentielles.

L'**héritage virtuel** est le mécanisme proposé par C++ pour résoudre ce problème. En déclarant l'héritage de l'ancêtre commun comme `virtual`, on garantit qu'**une seule instance** du sous-objet est partagée par toutes les branches de la hiérarchie, quel que soit le nombre de chemins qui y mènent.

Cette solution élégante n'est toutefois pas sans conséquences. Elle modifie la disposition mémoire des objets, change les règles de construction et introduit un surcoût à l'exécution. Comprendre ces implications est indispensable pour utiliser l'héritage virtuel à bon escient.

---

## Syntaxe

L'héritage virtuel se déclare en ajoutant le mot-clé `virtual` devant le mode d'accès (ou après, les deux ordres sont acceptés) dans la déclaration d'héritage des classes **intermédiaires** — celles qui héritent de l'ancêtre commun :

```cpp
class Peripherique {  
protected:  
    int id_;
public:
    explicit Peripherique(int id) : id_{id} {
        std::println("Peripherique::Peripherique({})", id);
    }
    int id() const { return id_; }
};

//              ↓ virtual ici
class PeripheriqueEntree : virtual public Peripherique {  
public:  
    explicit PeripheriqueEntree(int id) : Peripherique{id} {}
    void lire() { std::println("Lecture depuis périphérique {}", id_); }
};

//              ↓ et ici
class PeripheriqueSortie : virtual public Peripherique {  
public:  
    explicit PeripheriqueSortie(int id) : Peripherique{id} {}
    void ecrire() { std::println("Écriture vers périphérique {}", id_); }
};

class PeripheriqueES : public PeripheriqueEntree, public PeripheriqueSortie {  
public:  
    PeripheriqueES(int id)
        : Peripherique{id}          // ← la classe la plus dérivée initialise la base virtuelle
        , PeripheriqueEntree{id}
        , PeripheriqueSortie{id} {}
};
```

Le mot-clé `virtual` est placé dans `PeripheriqueEntree` et `PeripheriqueSortie`, pas dans `PeripheriqueES`. C'est au niveau des classes intermédiaires que la décision est prise de partager l'ancêtre commun.

---

## Le diamant résolu

Avec l'héritage virtuel, la hiérarchie en diamant est correctement aplatie :

```
Sans virtual :                    Avec virtual :

 Peripherique   Peripherique       Peripherique  (une seule copie)
      │              │                /        \
 Periph.Entree  Periph.Sortie   Periph.Entree  Periph.Sortie
      │              │                \        /
      └──────┬───────┘              PeripheriqueES
       PeripheriqueES
  (2 copies — problème)         (1 copie — résolu)
```

On peut le vérifier en observant les appels au constructeur :

```cpp
int main() {
    PeripheriqueES pes{42};
    std::println("ID = {}", pes.id());   // ✅ Plus d'ambiguïté
}
```

```
Peripherique::Peripherique(42)     ← une seule construction  
ID = 42  
```

Le constructeur de `Peripherique` n'est appelé qu'**une seule fois**. L'accès à `id()` n'est plus ambigu car il n'existe qu'un seul sous-objet `Peripherique`. La conversion `PeripheriqueES* → Peripherique*` est elle aussi non ambiguë.

---

## La règle de construction : responsabilité de la classe la plus dérivée

L'héritage virtuel introduit un changement fondamental dans les règles de construction. Normalement, chaque classe initialise ses bases directes dans sa liste d'initialisation. Mais avec une base virtuelle, la question "qui construit le sous-objet partagé ?" n'a qu'une seule bonne réponse : **la classe la plus dérivée** (*most derived class*).

### Pourquoi cette règle ?

Dans notre exemple, `PeripheriqueEntree` et `PeripheriqueSortie` passent toutes deux un argument au constructeur de `Peripherique`. Mais puisqu'il n'y a qu'un seul sous-objet `Peripherique`, un seul des deux appels peut être effectué. Lequel choisir ? C++ tranche : **c'est le constructeur de la classe effectivement instanciée** (celle sur laquelle `new` ou la déclaration de variable est appelée) qui a la responsabilité d'initialiser toutes les bases virtuelles.

```cpp
class PeripheriqueES : public PeripheriqueEntree, public PeripheriqueSortie {  
public:  
    PeripheriqueES(int id)
        : Peripherique{id}           // ✅ Obligatoire : initialise la base virtuelle
        , PeripheriqueEntree{id}     // les appels à Peripherique{id} dans ces
        , PeripheriqueSortie{id} {}  // constructeurs sont IGNORÉS
};
```

Quand `PeripheriqueES` est construit :

1. `Peripherique{id}` est appelé **une seule fois**, par `PeripheriqueES`.
2. `PeripheriqueEntree{id}` est appelé, mais son `Peripherique{id}` interne est **ignoré** (la base virtuelle est déjà construite).
3. `PeripheriqueSortie{id}` est appelé, même traitement.

### Conséquence : les classes intermédiaires ne contrôlent pas la construction de la base virtuelle

C'est un piège subtil. Si on instancie directement un `PeripheriqueEntree` (sans passer par `PeripheriqueES`), c'est `PeripheriqueEntree` qui est la classe la plus dérivée, et **son** appel à `Peripherique{id}` est alors bien exécuté :

```cpp
PeripheriqueEntree pe{10};   // PeripheriqueEntree est la "most derived class"
                              // → son Peripherique{10} est exécuté
```

Mais quand `PeripheriqueEntree` est construite en tant que sous-objet de `PeripheriqueES`, elle perd ce contrôle. Cela signifie que les classes intermédiaires doivent être conçues en sachant que leur initialisation de la base virtuelle pourrait être ignorée.

### Que se passe-t-il si la classe la plus dérivée omet l'initialisation ?

Si `PeripheriqueES` n'appelle pas explicitement le constructeur de `Peripherique`, le compilateur tente d'utiliser le constructeur **par défaut** de `Peripherique`. Si celui-ci n'existe pas, la compilation échoue :

```cpp
class PeripheriqueES : public PeripheriqueEntree, public PeripheriqueSortie {  
public:  
    PeripheriqueES(int id)
        // : Peripherique{id}        ← omis !
        : PeripheriqueEntree{id}
        , PeripheriqueSortie{id} {}
    // ❌ Erreur : pas de constructeur par défaut pour Peripherique
};
```

## Ordre de construction complet

Avec l'héritage virtuel, l'ordre de construction suit des règles précises :

1. **Toutes les bases virtuelles** sont construites en premier, dans l'ordre de parcours en profondeur de gauche à droite du graphe d'héritage.
2. **Puis les bases non virtuelles**, dans l'ordre de déclaration.
3. **Puis les membres** de la classe, dans l'ordre de déclaration.
4. **Enfin le corps** du constructeur.

Pour notre exemple :

```cpp
int main() {
    PeripheriqueES pes{42};
}
```

L'ordre est :

```
1. Peripherique(42)          ← base virtuelle (une seule fois)
2. PeripheriqueEntree(42)    ← base non virtuelle de PeripheriqueES (gauche)
3. PeripheriqueSortie(42)    ← base non virtuelle de PeripheriqueES (droite)
4. PeripheriqueES()          ← corps du constructeur
```

Et la destruction se fait dans l'ordre strictement inverse.

Un exemple plus complexe permet de bien visualiser la priorité des bases virtuelles :

```cpp
#include <print>

class V1 {  
public:  
    V1() { std::println("V1"); }
};

class V2 {  
public:  
    V2() { std::println("V2"); }
};

class A : virtual public V1, virtual public V2 {  
public:  
    A() { std::println("A"); }
};

class B : virtual public V2, virtual public V1 {  
public:  
    B() { std::println("B"); }
};

class C : public A, public B {  
public:  
    C() { std::println("C"); }
};

int main() {
    C c;
}
```

```
V1      ← bases virtuelles en premier, parcours depuis A : V1 puis V2  
V2  
A       ← bases directes de C, dans l'ordre de déclaration  
B  
C  
```

Les bases virtuelles `V1` et `V2` sont construites avant tout le reste, dans l'ordre où elles sont rencontrées lors du parcours de gauche à droite (d'abord `A`, qui déclare `V1` puis `V2`). Même si `B` déclare `V2` avant `V1`, cela ne change rien car ces bases sont déjà marquées comme construites.

## Disposition mémoire avec héritage virtuel

L'héritage virtuel modifie significativement le layout des objets en mémoire. Sans héritage virtuel, le sous-objet de base est à un offset fixe et connu à la compilation. Avec l'héritage virtuel, le sous-objet partagé peut se trouver à un offset **variable** selon le type concret de l'objet. Le compilateur doit donc utiliser un mécanisme d'**indirection** pour localiser la base virtuelle.

### Implémentation par GCC et Clang (Itanium ABI)

Sur Linux, GCC et Clang suivent l'**Itanium C++ ABI**. Chaque classe qui hérite virtuellement d'une base contient dans sa vtable (ou dans une table auxiliaire appelée *VTT — Virtual Table Table*) un **offset** vers le sous-objet de la base virtuelle. L'objet contient un pointeur (vptr) vers cette vtable, ce qui permet de retrouver la base virtuelle à l'exécution.

Voici une représentation simplifiée du layout de `PeripheriqueES` :

```
Objet PeripheriqueES
┌──────────────────────────────────────────┐
│ Sous-objet PeripheriqueEntree            │
│   ┌──────────────────────────────────┐   │
│   │ vptr → vtable PeriphEntree-in-ES │   │  ← contient l'offset vers Peripherique
│   │ (membres propres éventuels)      │   │
│   └──────────────────────────────────┘   │
│ Sous-objet PeripheriqueSortie            │
│   ┌──────────────────────────────────┐   │
│   │ vptr → vtable PeriphSortie-in-ES │   │  ← contient l'offset vers Peripherique
│   │ (membres propres éventuels)      │   │
│   └──────────────────────────────────┘   │
│ Sous-objet Peripherique (UNIQUE)         │
│   ┌──────────────────────────────────┐   │
│   │ id_ (int)                        │   │  ← partagé par les deux branches
│   └──────────────────────────────────┘   │
└──────────────────────────────────────────┘
```

Le sous-objet `Peripherique` est typiquement placé **à la fin** de l'objet (et non au début comme dans l'héritage simple). Sa position exacte dépend du type concret de l'objet complet. C'est pourquoi l'accès passe par une indirection via la vtable.

### Inspection avec les outils

On peut observer cette disposition avec les outils de GCC :

```bash
g++ -std=c++23 -fdump-lang-class -c peripherique.cpp
```

Cette commande génère un fichier `.class` décrivant la hiérarchie, les vtables et les offsets de chaque sous-objet. On y retrouvera les *virtual base offsets* qui pointent vers le sous-objet `Peripherique` partagé.

Avec `clang`, l'option équivalente est :

```bash
clang++ -std=c++23 -Xclang -fdump-record-layouts -c peripherique.cpp
```

## Coût de l'héritage virtuel

L'héritage virtuel n'est pas gratuit. Les surcoûts sont les suivants :

### Surcoût mémoire

Chaque classe qui hérite virtuellement acquiert un **vptr** (ou utilise un vptr existant étendu) pour stocker l'offset vers la base virtuelle. Sur une architecture 64 bits, cela représente 8 octets supplémentaires par classe intermédiaire dans la hiérarchie. Pour des objets volumineux ou peu nombreux, c'est négligeable. Pour des millions de petits objets, cela peut devenir significatif.

### Surcoût d'accès

Accéder à un membre de la base virtuelle nécessite une **indirection supplémentaire** : lire l'offset dans la vtable, puis l'appliquer pour calculer l'adresse du sous-objet. C'est un déréférencement de pointeur en plus par rapport à l'héritage non virtuel, où l'offset est une constante connue à la compilation.

### Surcoût de construction

La construction est plus complexe car le constructeur de la classe la plus dérivée doit initialiser la base virtuelle avant les bases directes. Le compilateur génère du code supplémentaire pour gérer cette séquence.

### En pratique

Ces surcoûts sont rarement un problème dans du code applicatif. Ils deviennent significatifs uniquement dans des contextes de haute performance avec des millions d'objets et des accès très fréquents aux membres de la base virtuelle. La section 7.5 quantifie les coûts du polymorphisme de manière plus large.

## Héritage virtuel et destructeurs

Avec l'héritage virtuel, le destructeur de la base virtuelle n'est appelé qu'**une seule fois**, par le destructeur de la classe la plus dérivée (le dernier à s'exécuter dans l'ordre de construction inversé) :

```cpp
class Peripherique {  
public:  
    explicit Peripherique(int id) { std::println("Peripherique()"); }
    ~Peripherique() { std::println("~Peripherique()"); }
};

class PeripheriqueEntree : virtual public Peripherique {  
public:  
    explicit PeripheriqueEntree(int id) : Peripherique{id} {
        std::println("PeripheriqueEntree()");
    }
    ~PeripheriqueEntree() { std::println("~PeripheriqueEntree()"); }
};

class PeripheriqueSortie : virtual public Peripherique {  
public:  
    explicit PeripheriqueSortie(int id) : Peripherique{id} {
        std::println("PeripheriqueSortie()");
    }
    ~PeripheriqueSortie() { std::println("~PeripheriqueSortie()"); }
};

class PeripheriqueES : public PeripheriqueEntree, public PeripheriqueSortie {  
public:  
    PeripheriqueES(int id)
        : Peripherique{id}, PeripheriqueEntree{id}, PeripheriqueSortie{id} {
        std::println("PeripheriqueES()");
    }
    ~PeripheriqueES() { std::println("~PeripheriqueES()"); }
};
```

```cpp
int main() {
    PeripheriqueES pes{42};
}
```

```
Peripherique()  
PeripheriqueEntree()  
PeripheriqueSortie()  
PeripheriqueES()  
~PeripheriqueES()
~PeripheriqueSortie()
~PeripheriqueEntree()
~Peripherique()          ← détruit une seule fois, en dernier
```

Le standard garantit cette unicité. Il n'y a aucun risque de double destruction de la base virtuelle.

> ⚠️ Si la hiérarchie est utilisée de manière polymorphique (suppression via un pointeur de base), le destructeur de `Peripherique` doit être **`virtual`** — comme pour tout héritage polymorphique. Voir section 7.2.

## Héritage virtuel et affectation

L'opérateur d'affectation généré par défaut dans une hiérarchie avec héritage virtuel peut provoquer une **double affectation** de la base virtuelle. Considérons :

```cpp
PeripheriqueES a{1};  
PeripheriqueES b{2};  
a = b;  
```

L'opérateur `operator=` généré pour `PeripheriqueES` appelle successivement `PeripheriqueEntree::operator=` puis `PeripheriqueSortie::operator=`. Chacun d'eux appelle `Peripherique::operator=`. Résultat : la partie `Peripherique` est affectée **deux fois**.

Dans la plupart des cas, cette double affectation est inoffensive (affecter la même valeur deux fois produit le même résultat). Mais si l'opérateur d'affectation de la base a des **effets de bord** (compteur, logging, allocation), cela peut être un bug.

La solution propre consiste à écrire manuellement l'opérateur d'affectation de la classe la plus dérivée en affectant la base virtuelle une seule fois :

```cpp
class PeripheriqueES : public PeripheriqueEntree, public PeripheriqueSortie {  
public:  
    PeripheriqueES& operator=(PeripheriqueES const& other) {
        if (this != &other) {
            Peripherique::operator=(other);           // une seule fois
            // affecter les membres propres de PeripheriqueEntree
            // affecter les membres propres de PeripheriqueSortie
            // affecter les membres propres de PeripheriqueES
        }
        return *this;
    }
};
```

C'est une des raisons pour lesquelles les hiérarchies avec héritage virtuel sont plus complexes à maintenir que les hiérarchies simples.

## Un cas concret de la bibliothèque standard : `std::iostream`

L'exemple le plus célèbre d'héritage virtuel en C++ est la hiérarchie des flux d'entrée-sortie de la bibliothèque standard :

```
            std::ios_base
                 │
             std::ios          ← base virtuelle
              /      \
       std::istream  std::ostream
              \      /
           std::iostream
```

`std::istream` et `std::ostream` héritent **virtuellement** de `std::ios` (qui hérite lui-même de `std::ios_base`). Cela garantit qu'un objet `std::iostream` (comme `std::fstream` en mode lecture-écriture) ne possède qu'un seul état de flux (flags de formatage, état d'erreur, locale, etc.).

Sans héritage virtuel, un `iostream` aurait deux copies de l'état du flux — une pour la partie lecture, une pour la partie écriture — ce qui serait incohérent.

C'est la preuve que l'héritage virtuel, malgré sa complexité, a des cas d'usage réels et importants dans du code de production à grande échelle.

## Bonnes pratiques

**Déclarez `virtual` dans les classes intermédiaires, pas dans la classe finale.** C'est au niveau des classes qui héritent directement de l'ancêtre commun que le `virtual` doit être placé. La classe la plus dérivée n'a pas besoin de déclarer quoi que ce soit de spécial — elle doit simplement initialiser la base virtuelle.

**Initialisez toujours explicitement la base virtuelle dans la classe la plus dérivée.** Ne comptez pas sur le constructeur par défaut sauf s'il existe et produit le résultat attendu. C'est une source fréquente de bugs : le développeur de la classe finale oublie qu'il est responsable de l'initialisation de la base virtuelle, parce que les classes intermédiaires semblent s'en charger.

**Fournissez un constructeur par défaut à la base virtuelle quand c'est possible.** Cela rend la hiérarchie plus facile à utiliser, car les classes les plus dérivées qui oublient l'appel explicite obtiennent au moins un objet dans un état valide plutôt qu'une erreur de compilation obscure.

**Évitez les hiérarchies profondes avec héritage virtuel.** La complexité des règles de construction et d'affectation croît rapidement. Si votre hiérarchie dépasse trois ou quatre niveaux avec des bases virtuelles, reconsidérez votre modélisation.

**Préférez les alternatives quand elles suffisent.** L'héritage virtuel est la solution au diamant, mais le diamant lui-même est souvent le signe d'une modélisation qui pourrait être simplifiée. La composition, les templates (polymorphisme statique) ou les interfaces pures sans état (section 7.4) permettent souvent d'éviter entièrement le besoin d'héritage virtuel.

---

## Résumé

| Aspect | Héritage classique | Héritage virtuel |  
|---|---|---|  
| Copies de la base commune | Une par chemin (N copies) | Une seule (partagée) |  
| Qui construit la base commune ? | Chaque classe intermédiaire | La classe la plus dérivée uniquement |  
| Offset du sous-objet base | Fixe (connu à la compilation) | Variable (indirection via vtable) |  
| Surcoût mémoire | Aucun | Un vptr/offset par branche virtuelle |  
| Surcoût d'accès aux membres | Aucun | Une indirection supplémentaire |  
| Opérateur `=` par défaut | Une affectation par copie de la base | Risque de double affectation |  
| Destructeur de la base | Appelé une fois par copie | Appelé une seule fois |  
| Cas d'usage de référence | La plupart des hiérarchies | `std::iostream`, diamants inévitables |
