# 7.1.2 — Héritage multiple et problème du diamant

## Section 7.1 : Héritage simple et multiple · Chapitre 7 : Héritage et Polymorphisme

---

## Introduction

L'héritage multiple est une fonctionnalité distinctive de C++ : une classe peut hériter simultanément de **plusieurs classes de base**. Cette capacité, absente en tant que telle de langages comme Java, C# ou Swift (qui la restreignent aux interfaces), permet de modéliser des situations où un objet possède naturellement plusieurs facettes indépendantes.

Mais cette expressivité a un coût en complexité. L'héritage multiple introduit deux catégories de problèmes que l'héritage simple ne pose jamais : les **ambiguïtés de noms** (quand plusieurs bases définissent un membre identique) et la **duplication de sous-objets** (quand plusieurs bases partagent un ancêtre commun). Ce second problème porte un nom célèbre dans la littérature C++ : le **problème du diamant**.

Cette section explore ces deux problématiques en détail, pose le diagnostic et prépare le terrain pour la section 7.1.3 qui introduit la solution : l'héritage virtuel.

---

## Syntaxe de l'héritage multiple

La syntaxe est une extension directe de l'héritage simple. Les classes de base sont séparées par des virgules, chacune précédée de son mode d'accès :

```cpp
class Imprimable {  
public:  
    void imprimer() const {
        std::println("(impression du contenu)");
    }
};

class Serialisable {  
public:  
    void serialiser() const {
        std::println("(sérialisation en binaire)");
    }
};

class Document : public Imprimable, public Serialisable {
    std::string titre_;
public:
    explicit Document(std::string titre) : titre_{std::move(titre)} {}
    std::string const& titre() const { return titre_; }
};
```

Un objet `Document` possède les méthodes des deux bases :

```cpp
Document doc{"Rapport Q1"};  
doc.imprimer();     // hérité de Imprimable  
doc.serialiser();   // hérité de Serialisable  
```

L'ordre de déclaration des bases (`Imprimable` puis `Serialisable`) détermine l'**ordre de construction** et la **disposition mémoire**.

---

## Disposition mémoire

Un objet de classe dérivée avec héritage multiple contient un sous-objet pour **chaque** classe de base, disposés dans l'ordre de déclaration, suivis des membres propres :

```
Objet Document (doc)
┌────────────────────────────────────┐
│  Sous-objet Imprimable             │
│  ┌──────────────────────────────┐  │
│  │ (pas de données membres)     │  │
│  └──────────────────────────────┘  │
│  Sous-objet Serialisable           │
│  ┌──────────────────────────────┐  │
│  │ (pas de données membres)     │  │
│  └──────────────────────────────┘  │
│  titre_  (std::string)             │  ← membre propre à Document
└────────────────────────────────────┘
```

Avec des classes de base qui portent des données, cela devient plus concret :

```cpp
class Moteur {  
protected:  
    int puissance_cv_;
public:
    explicit Moteur(int cv) : puissance_cv_{cv} {}
};

class Chassis {  
protected:  
    double poids_kg_;
public:
    explicit Chassis(double kg) : poids_kg_{kg} {}
};

class Voiture : public Moteur, public Chassis {
    std::string modele_;
public:
    Voiture(std::string modele, int cv, double kg)
        : Moteur{cv}, Chassis{kg}, modele_{std::move(modele)} {}
};
```

```
Objet Voiture
┌─────────────────────────────┐
│ Sous-objet Moteur           │
│   puissance_cv_ (int)       │    ← construit en premier
│ Sous-objet Chassis          │
│   poids_kg_ (double)        │    ← construit en second
│ modele_ (std::string)       │    ← construit en dernier
└─────────────────────────────┘
```

Un pointeur vers `Moteur` et un pointeur vers `Chassis` obtenus à partir du même objet `Voiture` ne contiennent **pas la même adresse** : chacun pointe vers le début de son sous-objet respectif. Le compilateur ajuste silencieusement les pointeurs lors des conversions (upcasts). C'est un détail d'implémentation qui a rarement un impact pratique, mais qui est important pour comprendre ce qui se passe quand on inspecte les adresses en mémoire :

```cpp
Voiture v{"Clio", 90, 1100.0};  
Moteur*  pm = &v;  
Chassis* pc = &v;  

std::println("Voiture : {}", static_cast<void*>(&v));  
std::println("Moteur  : {}", static_cast<void*>(pm));  
std::println("Chassis : {}", static_cast<void*>(pc));  
```

La sortie montrera typiquement que l'adresse de `Moteur` coïncide avec celle de `Voiture` (premier sous-objet), tandis que celle de `Chassis` est décalée de `sizeof(Moteur)` octets (avec alignement).

## Ordre de construction et de destruction

Comme pour l'héritage simple, la construction se fait **de la base vers la dérivée**, et la destruction dans l'ordre inverse. Avec l'héritage multiple, les sous-objets de base sont construits dans l'**ordre de déclaration** dans la liste d'héritage — **pas** dans l'ordre d'apparition dans la liste d'initialisation du constructeur :

```cpp
#include <print>

class A {  
public:  
    A() { std::println("A::A()"); }
    ~A() { std::println("A::~A()"); }
};

class B {  
public:  
    B() { std::println("B::B()"); }
    ~B() { std::println("B::~B()"); }
};

class C : public B, public A {   // B déclaré AVANT A  
public:  
    C() : A{}, B{} {             // ordre dans la liste d'init : A, B
        std::println("C::C()");  // mais la construction suit l'ordre de déclaration
    }
    ~C() { std::println("C::~C()"); }
};

int main() {
    C c;
}
```

```
B::B()          ← B est construit en premier (déclaré en premier)  
A::A()          ← A est construit en second  
C::C()  
C::~C()  
A::~A()         ← destruction dans l'ordre inverse  
B::~B()  
```

> ⚠️ Le compilateur émet un avertissement (`-Wall`) si l'ordre dans la liste d'initialisation ne correspond pas à l'ordre de déclaration des bases. Cet avertissement est important : s'en remettre à un ordre incorrect peut créer des bugs subtils quand l'initialisation d'une base dépend d'une autre.

## Ambiguïté de noms

Le premier problème concret de l'héritage multiple apparaît quand deux classes de base définissent un membre portant le **même nom**. Le compilateur ne peut pas deviner lequel vous voulez utiliser :

```cpp
class Ecran {  
public:  
    void allumer() { std::println("Ecran allumé"); }
};

class Clavier {  
public:  
    void allumer() { std::println("Clavier rétro-éclairé"); }
};

class Laptop : public Ecran, public Clavier {
};

int main() {
    Laptop pc;
    // pc.allumer();   // ❌ Erreur : ambiguïté — Ecran::allumer ou Clavier::allumer ?
}
```

Le message de GCC est explicite :

```
error: request for member 'allumer' is ambiguous  
note: candidates are: 'void Clavier::allumer()' and 'void Ecran::allumer()'  
```

### Résolution par qualification explicite

La solution la plus directe est de qualifier l'appel avec l'opérateur de portée :

```cpp
Laptop pc;  
pc.Ecran::allumer();    // ✅ appelle Ecran::allumer  
pc.Clavier::allumer();  // ✅ appelle Clavier::allumer  
```

### Résolution par redéfinition dans la dérivée

Une approche souvent préférable est de redéfinir la méthode dans la classe dérivée pour lever l'ambiguïté et donner une sémantique claire :

```cpp
class Laptop : public Ecran, public Clavier {  
public:  
    void allumer() {
        Ecran::allumer();    // allumer l'écran
        Clavier::allumer();  // puis le clavier
    }
};
```

Le code client appelle simplement `pc.allumer()`, sans avoir à connaître la structure d'héritage interne.

### Résolution par `using`

Si les deux méthodes ont des **signatures différentes** et que vous voulez les rendre toutes deux accessibles, `using` peut être utile :

```cpp
class Capteur {  
public:  
    void lire(int canal) { std::println("Capteur canal {}", canal); }
};

class Fichier {  
public:  
    void lire(std::string const& chemin) { std::println("Fichier {}", chemin); }
};

class SystemeAcquisition : public Capteur, public Fichier {  
public:  
    using Capteur::lire;   // rend visible lire(int)
    using Fichier::lire;   // rend visible lire(string)
};

int main() {
    SystemeAcquisition sys;
    sys.lire(3);                 // Capteur::lire(int)
    sys.lire("/tmp/data.csv");   // Fichier::lire(string)
}
```

Les deux surcharges coexistent dans `SystemeAcquisition` grâce aux déclarations `using`.

## Ambiguïté sur les données membres

L'ambiguïté ne se limite pas aux méthodes. Elle s'applique aussi aux **données membres** :

```cpp
class CapteurTemperature {  
protected:  
    double valeur_ = 0.0;
public:
    void mesurer() { valeur_ = 22.5; }
    double valeur() const { return valeur_; }
};

class CapteurHumidite {  
protected:  
    double valeur_ = 0.0;
public:
    void mesurer() { valeur_ = 65.0; }
    double valeur() const { return valeur_; }
};

class StationMeteo : public CapteurTemperature, public CapteurHumidite {  
public:  
    void rapport() {
        CapteurTemperature::mesurer();
        CapteurHumidite::mesurer();

        // valeur_ est ambigu — il faut qualifier
        std::println("Température : {}°C", CapteurTemperature::valeur());
        std::println("Humidité    : {}%",  CapteurHumidite::valeur());
    }
};
```

`StationMeteo` contient **deux** membres `valeur_` distincts, un dans chaque sous-objet. Il n'y a pas de fusion — ce sont deux emplacements mémoire différents.

## Le problème du diamant

Le problème du diamant survient quand deux classes de base partagent un **ancêtre commun**, créant un losange (diamant) dans le graphe d'héritage :

```cpp
class Peripherique {  
protected:  
    int id_;
public:
    explicit Peripherique(int id) : id_{id} {
        std::println("Peripherique::Peripherique({})", id);
    }
    int id() const { return id_; }
};

class PeripheriqueEntree : public Peripherique {  
public:  
    explicit PeripheriqueEntree(int id) : Peripherique{id} {}
    void lire() { std::println("Lecture depuis périphérique {}", id_); }
};

class PeripheriqueSortie : public Peripherique {  
public:  
    explicit PeripheriqueSortie(int id) : Peripherique{id} {}
    void ecrire() { std::println("Écriture vers périphérique {}", id_); }
};

class PeripheriqueES : public PeripheriqueEntree, public PeripheriqueSortie {  
public:  
    PeripheriqueES(int id)
        : PeripheriqueEntree{id}, PeripheriqueSortie{id} {}
};
```

La hiérarchie forme un diamant :

```
       Peripherique          Peripherique
            │                     │
   PeripheriqueEntree    PeripheriqueSortie
            │                     │
            └──────┬──────────────┘
            PeripheriqueES
```

### Le problème : duplication du sous-objet commun

L'objet `PeripheriqueES` contient **deux sous-objets `Peripherique`** distincts — un hérité via `PeripheriqueEntree`, l'autre via `PeripheriqueSortie` :

```
Objet PeripheriqueES
┌──────────────────────────────────────┐
│ Sous-objet PeripheriqueEntree        │
│   ┌────────────────────────────────┐ │
│   │ Sous-objet Peripherique  [1]   │ │
│   │   id_ = 42                     │ │
│   └────────────────────────────────┘ │
│ Sous-objet PeripheriqueSortie        │
│   ┌────────────────────────────────┐ │
│   │ Sous-objet Peripherique  [2]   │ │
│   │   id_ = 42                     │ │
│   └────────────────────────────────┘ │
└──────────────────────────────────────┘
  → Deux copies de Peripherique !
```

On peut vérifier cette duplication :

```cpp
int main() {
    PeripheriqueES pes{42};
}
```

```
Peripherique::Peripherique(42)    ← première copie (via PeripheriqueEntree)  
Peripherique::Peripherique(42)    ← seconde copie (via PeripheriqueSortie)  
```

Le constructeur de `Peripherique` est bien appelé **deux fois**.

### Conséquence 1 : ambiguïté d'accès

Tout accès à un membre de `Peripherique` depuis `PeripheriqueES` est ambigu :

```cpp
PeripheriqueES pes{42};
// pes.id();   // ❌ Erreur : ambiguïté — quel Peripherique::id() ?

// Il faut qualifier :
pes.PeripheriqueEntree::id();   // ✅ id_ de la copie "Entree"  
pes.PeripheriqueSortie::id();   // ✅ id_ de la copie "Sortie"  
```

### Conséquence 2 : conversion ambiguë vers la base commune

La conversion de `PeripheriqueES*` vers `Peripherique*` est ambiguë :

```cpp
PeripheriqueES pes{42};
// Peripherique* p = &pes;   // ❌ Erreur : conversion ambiguë

// Il faut passer par un chemin intermédiaire :
Peripherique* p1 = static_cast<PeripheriqueEntree*>(&pes);   // ✅  
Peripherique* p2 = static_cast<PeripheriqueSortie*>(&pes);   // ✅  

// p1 et p2 pointent vers des adresses DIFFÉRENTES !
```

### Conséquence 3 : incohérence potentielle des données

Puisque les deux copies de `Peripherique` sont indépendantes, rien ne garantit qu'elles restent synchronisées. Si une méthode de `PeripheriqueEntree` modifie `id_`, la copie dans `PeripheriqueSortie` n'est pas affectée. L'objet peut se retrouver dans un état **logiquement incohérent** :

```cpp
class PeripheriqueEntree : public Peripherique {  
public:  
    // ...
    void reconfigurer(int nouvel_id) { id_ = nouvel_id; }
};

// Après :
PeripheriqueES pes{42};  
pes.PeripheriqueEntree::reconfigurer(99);  

// Maintenant :
// pes.PeripheriqueEntree::id() == 99
// pes.PeripheriqueSortie::id() == 42
// → incohérence !
```

## Diagnostic : quand le diamant est-il un vrai problème ?

Le diamant n'est pas toujours problématique. Il faut distinguer deux cas :

**Le diamant est intentionnel et la duplication est souhaitée.** C'est rare, mais il existe des cas où deux sous-objets indépendants de la même classe de base sont voulus. Chaque branche de la hiérarchie possède sa propre copie avec son propre état. Dans ce cas, l'héritage non virtuel (le défaut) est correct, et les ambiguïtés se résolvent par qualification explicite.

**Le diamant est un accident de modélisation et la duplication est un bug.** C'est le cas de loin le plus fréquent. L'intention est qu'il n'existe qu'**une seule** instance de l'ancêtre commun, partagée par toutes les branches. C'est le scénario de notre exemple `Peripherique` : un périphérique d'entrée-sortie n'a qu'un seul identifiant, pas deux.

Dans ce second cas — le plus courant — la solution est l'**héritage virtuel**, qui fait l'objet de la section suivante (7.1.3).

## Cas d'usage légitimes de l'héritage multiple (sans diamant)

Le problème du diamant ne doit pas faire oublier que l'héritage multiple **sans ancêtre commun** est un outil parfaitement valide et courant en C++. Les cas d'usage les plus fréquents :

### Héritage de "mix-ins"

Un mix-in est une petite classe qui apporte une capacité orthogonale, sans données membres ou avec très peu. Les mix-ins n'ont généralement pas d'ancêtre commun, donc pas de diamant :

```cpp
class NonCopiable {  
public:  
    NonCopiable() = default;
    NonCopiable(NonCopiable const&) = delete;
    NonCopiable& operator=(NonCopiable const&) = delete;
};

class Loggable {  
public:  
    void log(std::string const& msg) const {
        std::println("[LOG] {}", msg);
    }
};

class Connexion : public NonCopiable, public Loggable {
    std::string hote_;
public:
    explicit Connexion(std::string hote) : hote_{std::move(hote)} {
        log("Connexion créée vers " + hote_);
    }
};
```

`Connexion` hérite de deux classes indépendantes qui apportent chacune une capacité distincte. Pas d'ambiguïté, pas de diamant.

### Implémentation de plusieurs interfaces

En C++, une interface se modélise par une classe abstraite contenant uniquement des fonctions virtuelles pures (section 7.4). L'héritage multiple permet d'implémenter plusieurs interfaces, exactement comme `implements` en Java :

```cpp
class ISerializable {  
public:  
    virtual void serialize(std::ostream& out) const = 0;
    virtual ~ISerializable() = default;
};

class IPrintable {  
public:  
    virtual void print() const = 0;
    virtual ~IPrintable() = default;
};

class Config : public ISerializable, public IPrintable {
    std::string data_;
public:
    void serialize(std::ostream& out) const override { out << data_; }
    void print() const override { std::println("Config: {}", data_); }
};
```

Cet usage est non seulement légitime mais **recommandé**. C'est la façon idiomatique de gérer les interfaces multiples en C++.

## Bonnes pratiques

**Limitez l'héritage multiple aux cas justifiés.** L'héritage multiple de classes concrètes (avec état) est une source de complexité. Réservez-le aux mix-ins sans état et à l'implémentation d'interfaces abstraites.

**Identifiez les diamants potentiels tôt.** Dès que deux de vos classes de base ont un ancêtre commun, vous avez un diamant. Si l'intention est de partager un seul sous-objet commun, passez à l'héritage virtuel (section 7.1.3).

**Préférez la composition quand la relation n'est pas "est-un".** Si `StationMeteo` n'est pas fondamentalement *un capteur de température* ET *un capteur d'humidité*, mais plutôt *possède* ces capteurs, la composition est plus claire :

```cpp
class StationMeteo {
    CapteurTemperature temperature_;   // composition
    CapteurHumidite humidite_;         // composition
public:
    // délègue aux membres
};
```

**Résolvez les ambiguïtés explicitement.** Si deux bases définissent un membre identique, ne laissez pas le code client découvrir l'ambiguïté. Redéfinissez la méthode dans la dérivée ou utilisez `using` pour sélectionner la version voulue.

---

## Résumé

| Situation | Symptôme | Solution |  
|---|---|---|  
| Deux bases avec une méthode de même nom | Erreur de compilation : appel ambigu | Qualification explicite (`Base::methode()`) ou redéfinition dans la dérivée |  
| Deux bases avec un membre de même nom | Erreur de compilation : accès ambigu | Qualification explicite |  
| Ancêtre commun (diamant) sans héritage virtuel | Double construction de l'ancêtre, duplication mémoire, conversions ambiguës | Héritage virtuel (section 7.1.3) |  
| Diamant intentionnel (deux copies voulues) | Pas d'erreur — comportement voulu | Qualification explicite pour lever les ambiguïtés |  
| Héritage de mix-ins indépendants | Aucun problème | Aucune action — usage idiomatique |  
| Implémentation de plusieurs interfaces abstraites | Aucun problème | Aucune action — usage recommandé |
