# 31.1.1 — perf record et perf report

## Introduction

Le workflow `perf record` → `perf report` est le cœur du profiling avec `perf`. La première commande enregistre les échantillons dans un fichier `perf.data` pendant l'exécution du programme ; la seconde ouvre ce fichier dans une interface interactive pour explorer les résultats. Ce découplage entre collecte et analyse est un avantage majeur : on peut enregistrer un profil sur une machine de production et l'analyser sur sa machine de développement, partager un fichier `perf.data` avec un collègue, ou conserver des profils comme référence pour détecter les régressions.

---

## Enregistrer un profil avec `perf record`

### Usage de base

```bash
perf record ./mon_programme --ses-arguments
```

`perf record` lance le programme, collecte les échantillons pendant toute la durée de son exécution, puis écrit le fichier `perf.data` dans le répertoire courant. À la fin, un résumé s'affiche :

```
[ perf record: Woken up 3 times to write data ]
[ perf record: Captured and wrote 1.245 MB perf.data (32541 samples) ]
```

Ce résumé indique le volume de données collectées et le nombre total de samples. Plus le programme s'exécute longtemps (ou plus la fréquence d'échantillonnage est élevée), plus le nombre de samples est grand et plus le profil est statistiquement précis.

### Avec reconstruction des piles d'appels

Sans pile d'appels, `perf report` ne montre que les fonctions *feuilles* — celles qui étaient en cours d'exécution au moment du sample. Pour comprendre *comment* on arrive dans ces fonctions, il faut activer la reconstruction des piles d'appels :

```bash
# Méthode frame pointer (requiert -fno-omit-frame-pointer à la compilation)
perf record --call-graph fp ./mon_programme

# Méthode DWARF (requiert -g à la compilation, fonctionne sans frame pointer)
perf record --call-graph dwarf ./mon_programme
```

En pratique, utilisez systématiquement `--call-graph`. Un profil sans piles d'appels est comme un rapport Valgrind sans numéros de ligne : il indique *où* se situe le problème mais pas *pourquoi*.

### Profiler un processus déjà en cours d'exécution

```bash
# Par PID, pendant 30 secondes
perf record -p $(pgrep mon_service) --call-graph fp -- sleep 30
```

L'argument `-- sleep 30` indique à `perf` de collecter pendant 30 secondes puis de s'arrêter. Sans cette limite, `perf record -p <PID>` collecte indéfiniment jusqu'à un `Ctrl+C`.

### Profiler tous les threads d'un processus

Par défaut, `perf record` profile tous les threads du processus ciblé. Pour filtrer un thread spécifique :

```bash
# Profiler un thread par son TID
perf record -t <TID> --call-graph fp -- sleep 10
```

### Options fréquentes

```bash
perf record \
    -F 4000 \                # Fréquence d'échantillonnage (Hz), défaut : 4000
    -g \                     # Raccourci pour --call-graph fp
    -e cycles \              # Événement à échantillonner (défaut : cycles)
    -o profil.data \         # Nom du fichier de sortie (défaut : perf.data)
    --call-graph dwarf,16384 \  # DWARF avec 16 Ko de pile par sample
    -- ./mon_programme
```

L'option `-e` permet de choisir un événement différent des cycles CPU. Par exemple, pour profiler les cache misses :

```bash
perf record -e cache-misses --call-graph fp ./mon_programme
```

Ce profil révèle les fonctions qui provoquent le plus d'échecs de cache, ce qui est une information différente de celles qui consomment le plus de cycles. Un programme peut passer la majorité de ses cycles dans une fonction de calcul intense, mais ses problèmes de performance réels peuvent venir d'une autre fonction qui génère des cache misses massifs.

### Taille du fichier perf.data

Le fichier `perf.data` croît avec la durée d'exécution et la fréquence d'échantillonnage. Pour un programme typique profilé 30 secondes à 4 000 Hz, attendez-vous à un fichier de 1 à 10 Mo. Avec `--call-graph dwarf`, la taille peut atteindre 50 à 200 Mo car chaque sample inclut une copie partielle de la pile.

Pour les sessions longues, limitez la taille avec `--mmap-pages` ou réduisez la fréquence. Vous pouvez aussi compresser le fichier pour l'archivage :

```bash
perf record -z --call-graph fp ./mon_programme  # Compression à la volée
```

## Analyser un profil avec `perf report`

### Lancement

```bash
perf report
```

Sans argument, `perf report` ouvre le fichier `perf.data` du répertoire courant. Pour un fichier spécifique :

```bash
perf report -i profil.data
```

L'interface est un navigateur interactif en mode texte (basé sur ncurses) qui affiche les résultats triés par pourcentage de samples.

### Vue principale : profil plat (*flat profile*)

La vue par défaut affiche les fonctions triées par overhead décroissant :

```
Samples: 32K of event 'cycles', Event count (approx.): 14235678900  
Overhead  Command        Shared Object      Symbol  
  22.45%  mon_programme  mon_programme      [.] Parser::analyser_token
  15.32%  mon_programme  mon_programme      [.] Buffer::copier_donnees
   9.87%  mon_programme  libc.so.6          [.] __memcpy_avx2_unaligned
   7.12%  mon_programme  mon_programme      [.] Reseau::lire_socket
   5.89%  mon_programme  libstdc++.so.6     [.] std::string::_M_assign
   4.56%  mon_programme  mon_programme      [.] Index::rechercher
   3.21%  mon_programme  [kernel.kallsyms]  [k] copy_user_enhanced_fast_string
   ...
```

Chaque colonne apporte une information :

**Overhead** : le pourcentage de samples dans lesquels cette fonction était au sommet de la pile d'appels — c'est-à-dire qu'elle était l'instruction en cours d'exécution. C'est le *self time*, le temps passé dans la fonction elle-même (hors fonctions appelées).

**Command** : le nom du processus.

**Shared Object** : le binaire ou la librairie contenant la fonction. `mon_programme` indique votre exécutable, `libc.so.6` la librairie C, `[kernel.kallsyms]` le noyau Linux.

**Symbol** : le nom de la fonction. Le préfixe `[.]` indique du code en espace utilisateur, `[k]` du code noyau.

Dans cet exemple, `Parser::analyser_token` consomme 22.45% du temps CPU. C'est le hotspot principal — la première cible d'optimisation.

### Navigation dans l'interface

L'interface ncurses de `perf report` offre plusieurs modes de navigation. Les touches principales :

| Touche | Action |
|---|---|
| `↑` / `↓` | Naviguer entre les fonctions |
| `Enter` | Ouvrir le menu contextuel de la fonction sélectionnée |
| `+` / `-` | Déplier/replier l'arbre d'appels |
| `a` | Annoter : afficher le code source/assembleur avec les hotspots |
| `h` | Afficher l'aide complète |
| `q` | Quitter |

En appuyant sur `Enter` sur une fonction, un menu contextuel propose plusieurs actions :

```
┌─ Annotate Parser::analyser_token
│  Zoom into Parser::analyser_token DSO
│  Browse map details
│  Run scripts for samples of symbol [Parser::analyser_token]
│  Exit
└──────────────────────────────────────
```

### Vue callers/callees : comprendre le contexte

Lorsque les piles d'appels ont été enregistrées (`--call-graph`), `perf report` peut afficher les relations appelant/appelé. En dépliant une fonction avec `+`, on voit quelles fonctions l'appellent (*callers*) et quelles fonctions elle appelle (*callees*) :

```
-  22.45%  mon_programme  [.] Parser::analyser_token
   - 18.10%  Parser::analyser_token
      - 14.20%  Parser::analyser_ligne
         - 14.20%  Parser::traiter_fichier
              main
      - 3.90%  Parser::analyser_expression
         - 3.90%  Parser::analyser_ligne
              Parser::traiter_fichier
                 main
   + 4.35%  (autres callchains)
```

Cette vue révèle que `analyser_token` est appelée principalement depuis `analyser_ligne` (elle-même appelée depuis `traiter_fichier`), mais via deux chemins distincts : un appel direct (14.20%) et un passage par `analyser_expression` (3.90%). Si l'optimisation de `analyser_token` s'avère difficile, cette information permet d'envisager des stratégies alternatives : réduire le nombre d'appels depuis `analyser_ligne`, ou court-circuiter le chemin passant par `analyser_expression`.

### Mode children vs self

Par défaut, `perf report` affiche deux colonnes : `Self` et `Children`.

```
Children      Self  Command        Symbol
  45.67%     0.12%  mon_programme  [.] main
  45.55%     2.34%  mon_programme  [.] Parser::traiter_fichier
  43.21%     5.67%  mon_programme  [.] Parser::analyser_ligne
  22.45%    22.45%  mon_programme  [.] Parser::analyser_token
```

**Self** : le pourcentage de samples où la fonction était au sommet de la pile (en cours d'exécution). C'est le temps passé *dans* la fonction.

**Children** : le pourcentage de samples où la fonction apparaît *n'importe où* dans la pile. C'est le temps passé dans la fonction *et toutes les fonctions qu'elle appelle*. C'est le *inclusive time* ou *cumulative time*.

La colonne `Children` est essentielle pour identifier les fonctions de haut niveau responsables de la majeure partie du temps CPU. Dans l'exemple, `main` a un `Self` de 0.12% (elle ne fait presque rien directement) mais un `Children` de 45.67% (elle est dans la pile pour 45% des samples, car elle appelle tout le reste).

Pour trier par `Self` uniquement (vue classique des hotspots) :

```bash
perf report --no-children
```

Ou inversement, pour se concentrer sur les fonctions de haut niveau, triez par `Children` (comportement par défaut).

## Annoter le code source avec `perf annotate`

`perf annotate` est l'une des fonctionnalités les plus puissantes de `perf`. Elle affiche le code source (ou l'assembleur) d'une fonction avec le pourcentage de samples associé à chaque ligne :

```bash
# Depuis perf report : sélectionner une fonction et appuyer sur 'a'
# Ou directement en ligne de commande :
perf annotate Parser::analyser_token
```

La sortie mélange code source et assembleur, avec les pourcentages à gauche :

```
 Percent |  Source code & Disassembly of mon_programme
---------+----------------------------------------------
         :  Token Parser::analyser_token(const char* input) {
         :      size_t pos = 0;
    0.45 :      while (pos < longueur_) {
         :          char c = input[pos];
   18.23 :  ┌──  if (is_delimiter(c)) {
    2.10 :  │       tokens_.push_back(extraire(input, debut, pos));
         :  │       debut = pos + 1;
    1.67 :  └── }
         :          pos++;
         :      }
```

Dans cet exemple, 18.23% du temps CPU de la fonction est passé sur le test `is_delimiter(c)`. C'est une information au niveau de la *ligne* — nettement plus précise que le profil par fonction. Plusieurs hypothèses émergent immédiatement : `is_delimiter` est-elle inlinée ? Le pattern d'accès à `input[pos]` génère-t-il des cache misses ? La prédiction de branche sur le `if` est-elle mauvaise ?

### Annotation en mode assembleur pur

Pour les optimisations de bas niveau, l'annotation assembleur montre les instructions machine individuelles :

```bash
perf annotate --no-source Parser::analyser_token
```

```
 Percent |      Disassembly
---------+-----------------------
    0.23 :  mov    (%rdi,%rax,1),%cl
   18.23 :  cmp    $0x2c,%cl          # comparaison avec ','
    0.89 :  je     0x401a30           # saut si délimiteur
    1.12 :  cmp    $0x3b,%cl          # comparaison avec ';'
    0.45 :  je     0x401a30
```

Cette vue révèle que la comparaison avec le premier délimiteur (`','`) concentre les samples, probablement en raison de la prédiction de branche. L'annotation assembleur est le dernier niveau de zoom du profiling — elle est rarement nécessaire, mais irremplaçable pour les optimisations critiques.

> 💡 **Rappel** : le *skid* (section 31.1) peut décaler les samples de quelques instructions par rapport à l'instruction réellement responsable. Lors de l'annotation assembleur, considérez les instructions voisines comme un ensemble plutôt que de vous focaliser sur une instruction unique.

### Prérequis pour l'annotation source

L'annotation au niveau du code source nécessite que le binaire ait été compilé avec `-g` (symboles de débogage) et que les fichiers sources soient accessibles sur la machine d'analyse au chemin enregistré lors de la compilation. Si les sources sont à un emplacement différent :

```bash
perf annotate --source --objdump=/usr/bin/objdump \
    --symfs=/chemin/vers/sources/ Parser::analyser_token
```

## Filtrage et tri des résultats

### Filtrer par DSO (librairie / binaire)

Pour se concentrer uniquement sur les fonctions de votre programme, en ignorant les librairies système :

```bash
perf report --dso=mon_programme
```

Inversement, pour examiner le temps passé dans une librairie spécifique :

```bash
perf report --dso=libc.so.6
```

### Filtrer par symbole

Pour rechercher une fonction spécifique dans le profil :

```bash
perf report --symbol-filter=Parser
```

Affiche uniquement les fonctions dont le nom contient `Parser`.

### Filtrer par thread

Pour les programmes multithreads, isoler un thread spécifique :

```bash
perf report --tid=12346
```

### Trier par différentes colonnes

```bash
# Trier par self time uniquement
perf report --no-children -s overhead

# Trier par shared object puis par symbole
perf report -s dso,sym
```

### Mode non interactif (stdio)

Pour exporter les résultats vers un fichier texte ou les traiter par script :

```bash
perf report --stdio > profil_rapport.txt
```

Le mode `--stdio` produit la même information que le mode ncurses, mais en texte brut. Il est indispensable pour l'intégration dans les scripts, les rapports et la CI :

```bash
# Extraire les 20 fonctions les plus coûteuses
perf report --stdio --no-children | head -40
```

## Comparer deux profils avec `perf diff`

`perf diff` compare deux fichiers `perf.data` et affiche les variations d'overhead par fonction. C'est l'outil de choix pour valider qu'une optimisation a bien eu l'effet escompté ou pour détecter une régression de performance entre deux versions.

```bash
# Enregistrer un profil avant modification
perf record -o avant.data --call-graph fp ./mon_programme_v1 < donnees_test.txt

# Enregistrer un profil après modification
perf record -o apres.data --call-graph fp ./mon_programme_v2 < donnees_test.txt

# Comparer
perf diff avant.data apres.data
```

La sortie montre le delta d'overhead pour chaque fonction :

```
# Event 'cycles'
#
# Baseline  Delta Abs  Shared Object    Symbol
#
    22.45%    -14.20%  mon_programme    [.] Parser::analyser_token
    15.32%     -0.45%  mon_programme    [.] Buffer::copier_donnees
     9.87%     +8.90%  mon_programme    [.] Parser::analyser_token_v2
     7.12%     +0.12%  mon_programme    [.] Reseau::lire_socket
```

Ici, `analyser_token` est passée de 22.45% à 8.25% (delta de -14.20%), tandis que la nouvelle fonction `analyser_token_v2` apparaît à 8.90%. Le bilan net est positif : la fonction optimisée consomme moins de cycles.

> ⚠️ **Attention à l'interprétation** : `perf diff` compare des *pourcentages relatifs*, pas des temps absolus. Si le programme global est 2× plus rapide après optimisation, toutes les fonctions non modifiées verront leur pourcentage relatif augmenter (elles représentent une part plus grande d'un total plus petit). Combinez `perf diff` avec une mesure du temps d'exécution total (`time`, `hyperfine`) pour une évaluation complète.

## Export des données avec `perf script`

`perf script` exporte les samples bruts sous forme textuelle, un sample par ligne. Cette sortie est le point de départ pour les outils de visualisation externes, en particulier les flamegraphs (section 31.3) :

```bash
perf script > samples_bruts.txt
```

Chaque ligne contient le processus, le PID/TID, le timestamp, l'événement, et la pile d'appels :

```
mon_programme 12345/12345 2841234.123456:     250000 cycles:
        401a2f Parser::analyser_token+0x4f (/usr/local/bin/mon_programme)
        4018c0 Parser::analyser_ligne+0x80 (/usr/local/bin/mon_programme)
        401750 Parser::traiter_fichier+0x30 (/usr/local/bin/mon_programme)
        401620 main+0xa0 (/usr/local/bin/mon_programme)
```

Cette sortie est conçue pour être consommée par des scripts. Le pipeline classique vers un flamegraph est :

```bash
perf script | stackcollapse-perf.pl | flamegraph.pl > profil.svg
```

Ce workflow est détaillé en section 31.3.

## Workflow récapitulatif

Une session typique de profiling avec `perf record` et `perf report` suit ces étapes :

```bash
# 1. Compiler avec symboles et frame pointer
g++ -std=c++23 -O2 -g -fno-omit-frame-pointer -o mon_programme mon_programme.cpp

# 2. Enregistrer le profil
perf record --call-graph fp ./mon_programme < donnees_test.txt

# 3. Vue d'ensemble : identifier les hotspots
perf report

# 4. Zoom sur une fonction suspecte : annotation source
perf annotate Parser::analyser_token

# 5. Exporter pour visualisation (flamegraph, Hotspot)
perf script > profil_script.txt

# 6. Après optimisation : comparer
perf record -o apres.data --call-graph fp ./mon_programme_v2 < donnees_test.txt  
perf diff perf.data apres.data  
```

Ce workflow — enregistrer, explorer, annoter, exporter, comparer — couvre la grande majorité des besoins de profiling CPU en développement C++. La section suivante (31.1.2) complète ce tableau avec `perf stat`, qui fournit une perspective différente : non pas *quelles fonctions* sont coûteuses, mais *pourquoi* elles sont coûteuses au niveau de la micro-architecture du processeur.
