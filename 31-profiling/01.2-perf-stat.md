# 31.1.2 — perf stat : Compteurs matériels

## Introduction

`perf record` et `perf report` répondent à la question « *où* le programme passe-t-il son temps ? ». `perf stat` répond à une question complémentaire et tout aussi importante : « *pourquoi* le programme est-il lent ? ». Là où `perf record` localise les hotspots par fonction, `perf stat` donne une radiographie globale de l'interaction entre le programme et le matériel — combien d'instructions par cycle, quel taux de cache misses, combien de mauvaises prédictions de branche.

Cette distinction est essentielle. Deux programmes peuvent avoir le même hotspot (une boucle de traitement qui consomme 60% du temps CPU), mais pour des raisons fondamentalement différentes : l'un est limité par le calcul pur (trop d'instructions), l'autre par les accès mémoire (trop de cache misses). L'optimisation sera radicalement différente dans chaque cas. `perf stat` permet de poser le diagnostic avant de choisir le remède.

---

## Usage de base

`perf stat` exécute un programme et affiche les compteurs de performance agrégés sur l'ensemble de l'exécution :

```bash
perf stat ./mon_programme < donnees_test.txt
```

La sortie typique :

```
 Performance counter stats for './mon_programme':

         3 247,52 msec  task-clock                #    0,998 CPUs utilized
               12       context-switches          #    3,694 /sec
                2       cpu-migrations            #    0,616 /sec
            4 521       page-faults               #    1,392 K/sec
   12 456 789 012       cycles                    #    3,835 GHz
    9 234 567 890       instructions              #    0,74  insn per cycle
    1 823 456 789       branches                  #  561,398 M/sec
       45 678 901       branch-misses             #    2,50% of all branches
      234 567 890       cache-references          #   72,223 M/sec
       23 456 789       cache-misses              #   10,00% of all cache refs

       3,25432 seconds time elapsed
       3,24120 seconds user
       0,01200 seconds sys
```

Chaque ligne est un compteur matériel ou logiciel. La colonne de droite fournit une métrique dérivée qui facilite l'interprétation. Détaillons les compteurs les plus importants.

---

## Compteurs fondamentaux

### task-clock et CPUs utilized

```
3 247,52 msec  task-clock    #    0,998 CPUs utilized
```

`task-clock` mesure le temps CPU réellement consommé par le programme (en millisecondes). Le ratio `CPUs utilized` divise ce temps par le temps écoulé (*wall-clock time*). Une valeur de 0,998 signifie que le programme utilise un cœur CPU à pleine capacité. Les valeurs caractéristiques :

| Valeur | Interprétation |
|---|---|
| ~1.0 | Programme monothread, CPU-bound |
| ~4.0 | Programme utilisant ~4 cœurs en parallèle |
| ~0.3 | Programme I/O-bound ou bloqué (attente réseau, disque, locks) |
| > nombre de cœurs | Valeur théoriquement impossible, indique une erreur de mesure |

Ce ratio est la première information à examiner. Un programme avec un `CPUs utilized` de 0.2 n'a pas un problème de performance CPU — il a un problème d'attente (I/O, locks, sleep). L'optimiser avec des techniques CPU (SIMD, réduction de cache misses) n'aura aucun effet.

### cycles et instructions (IPC)

```
12 456 789 012       cycles                    #    3,835 GHz
 9 234 567 890       instructions              #    0,74  insn per cycle
```

Le ratio **Instructions Per Cycle (IPC)** est le compteur le plus révélateur de l'efficacité d'un programme. Un processeur moderne peut théoriquement exécuter 4 à 6 instructions par cycle (grâce au superscalaire et à l'exécution *out-of-order*). En pratique, l'IPC dépend de la nature du code :

| IPC | Interprétation | Cause typique |
|---|---|---|
| > 2.0 | Excellent. Le pipeline est bien alimenté | Code vectorisé, accès mémoire locaux |
| 1.0 – 2.0 | Bon. Performances raisonnables | Code général bien optimisé |
| 0.5 – 1.0 | Médiocre. Le pipeline stagne fréquemment | Cache misses modérés, dépendances de données |
| < 0.5 | Mauvais. Le processeur attend la mémoire | Cache misses sévères, accès DRAM fréquents |

Un IPC de 0.74 comme dans l'exemple signale que le processeur passe une part significative de son temps à attendre — probablement la mémoire. Les compteurs de cache (ci-dessous) confirmeront ou infirmeront cette hypothèse.

L'IPC est un indicateur *relatif* : il n'existe pas de valeur « correcte » universelle. Comparez-le entre deux versions du même programme, ou avec un programme de référence réalisant un traitement similaire.

### branches et branch-misses

```
1 823 456 789       branches                  #  561,398 M/sec
   45 678 901       branch-misses             #    2,50% of all branches
```

Le processeur tente de prédire le résultat de chaque branche conditionnelle (`if`, `while`, `for`, `switch`) avant de l'évaluer, pour maintenir son pipeline rempli. Lorsque la prédiction est incorrecte (*branch miss*), le pipeline doit être vidé et les instructions spéculatives annulées — un coût typique de 10 à 20 cycles par misprediction.

| Taux de branch-misses | Interprétation |
|---|---|
| < 1% | Excellent. Les branches sont très prévisibles |
| 1% – 3% | Normal pour du code générique |
| 3% – 10% | Élevé. Pénalité significative |
| > 10% | Problématique. Les patterns de branchement sont peu prévisibles |

Un taux élevé de branch misses se rencontre typiquement dans le tri de données non ordonnées, les parsers avec de nombreuses conditions, ou les structures de données avec des accès dépendant des données (tables de hachage avec chaînage, arbres). Les stratégies d'optimisation incluent la réorganisation des conditions (cas les plus fréquents en premier), le remplacement des branches par des opérations arithmétiques conditionnelles (*branchless programming*), ou le tri préalable des données.

### cache-references et cache-misses

```
234 567 890       cache-references          #   72,223 M/sec
 23 456 789       cache-misses              #   10,00% of all cache refs
```

Ces compteurs mesurent les accès au dernier niveau de cache (LLC — *Last Level Cache*, typiquement L3) et les échecs. Un *cache miss* au LLC signifie que le processeur doit aller chercher la donnée en mémoire principale (DRAM), ce qui coûte typiquement 100 à 300 cycles — un ordre de grandeur supérieur à un accès en cache L1 (4–5 cycles) ou L2 (10–15 cycles).

| Taux de cache-misses | Interprétation |
|---|---|
| < 2% | Excellent. Les données sont bien localisées en cache |
| 2% – 10% | Acceptable. Quelques accès DRAM mais pas dominant |
| 10% – 30% | Élevé. Les accès mémoire pénalisent significativement la performance |
| > 30% | Critique. Le programme est *memory-bound* |

Un taux de cache misses élevé corrèle généralement avec un IPC faible : le processeur passe ses cycles à attendre la mémoire plutôt qu'à exécuter des instructions. Les optimisations relèvent du chapitre 41 (Optimisation CPU et Mémoire) : localité des données, *data-oriented design*, réduction de la taille des structures, utilisation de conteneurs contigus (`std::vector`, `std::flat_map`).

## Compteurs détaillés par niveau de cache

Les compteurs génériques `cache-references`/`cache-misses` donnent une vue globale. Pour un diagnostic plus fin, `perf stat` permet de mesurer les caches individuellement :

```bash
perf stat -e L1-dcache-loads,L1-dcache-load-misses,\  
L1-icache-load-misses,\  
LLC-loads,LLC-load-misses \  
./mon_programme
```

Sortie :

```
   4 567 890 123       L1-dcache-loads
     123 456 789       L1-dcache-load-misses     #    2,70% of all L1-dcache accesses
       8 901 234       L1-icache-load-misses
     234 567 890       LLC-loads
      23 456 789       LLC-load-misses            #   10,00% of all LL-cache accesses
```

| Compteur | Ce qu'il mesure | Cause d'un taux élevé |
|---|---|---|
| `L1-dcache-load-misses` | Misses en cache L1 données | Structures trop grandes, accès non séquentiels |
| `L1-icache-load-misses` | Misses en cache L1 instructions | Code trop gros (binaire volumineux, templates instanciés en masse) |
| `LLC-load-misses` | Misses au dernier niveau de cache | Jeu de données excédant la taille du LLC, patterns d'accès aléatoires |

La distinction entre cache de données (dcache) et cache d'instructions (icache) est importante. Un taux élevé de misses icache indique que le code exécuté est trop dispersé en mémoire — un problème courant dans les programmes C++ avec beaucoup de templates instanciés ou de polymorphisme dynamique (vtable dispatch). Un taux élevé de misses dcache pointe vers les structures de données.

## Événements spécialisés

### Page faults

```
4 521       page-faults               #    1,392 K/sec
```

Les *page faults* se produisent lorsqu'un programme accède à une page mémoire qui n'est pas encore mappée physiquement. Les *minor page faults* (les plus fréquents) se résolvent par un simple mappage dans la table des pages. Les *major page faults* nécessitent un accès disque (mémoire swappée ou fichier mappé). Pour distinguer les deux :

```bash
perf stat -e page-faults,minor-faults,major-faults ./mon_programme
```

Un nombre élevé de major faults indique un problème de swap ou une utilisation intensive de fichiers mappés en mémoire (`mmap`).

### Context switches et cpu-migrations

```
12       context-switches          #    3,694 /sec
 2       cpu-migrations            #    0,616 /sec
```

Les *context switches* se produisent lorsque le noyau suspend le programme pour exécuter un autre processus ou thread. Un taux élevé (milliers par seconde) indique une contention : trop de threads pour le nombre de cœurs, ou des locks très disputés.

Les *cpu-migrations* comptent le nombre de fois où un thread est déplacé d'un cœur à un autre. Chaque migration invalide le cache L1/L2 du thread, car les caches sont spécifiques à chaque cœur. Un taux élevé peut être atténué par l'affinité CPU (`taskset`, `pthread_setaffinity_np`).

## Mesurer des événements spécifiques

### Sélectionner les événements

L'option `-e` permet de choisir exactement quels compteurs mesurer :

```bash
# Uniquement IPC et cache
perf stat -e cycles,instructions,cache-references,cache-misses ./mon_programme

# Événements de branchement détaillés
perf stat -e branches,branch-misses,branch-loads,branch-load-misses ./mon_programme
```

Le nombre d'événements mesurables simultanément est limité par le nombre de compteurs matériels disponibles sur le processeur (typiquement 4 à 8 compteurs généraux). Si plus d'événements sont demandés, `perf` les multiplexe en alternant entre eux pendant l'exécution, ce qui introduit une estimation statistique signalée par un pourcentage de confiance :

```
   12 456 789 012       cycles                    (83,24%)
    9 234 567 890       instructions              (83,30%)
```

Le pourcentage entre parenthèses indique la fraction du temps pendant laquelle le compteur était actif. En dessous de 70%, la valeur est une estimation peu fiable. Pour une mesure précise, réduisez le nombre d'événements simultanés ou exécutez plusieurs passes :

```bash
# Passe 1 : CPU et instructions
perf stat -e cycles,instructions ./mon_programme < donnees_test.txt

# Passe 2 : cache
perf stat -e cache-references,cache-misses,LLC-loads,LLC-load-misses ./mon_programme < donnees_test.txt
```

### Répéter les mesures

`perf stat` peut exécuter le programme plusieurs fois et calculer des statistiques sur les résultats :

```bash
perf stat -r 5 ./mon_programme < donnees_test.txt
```

La sortie inclut la moyenne et l'écart-type pour chaque compteur :

```
3,254 +- 0,012 seconds time elapsed  ( +-  0,37% )
```

L'écart-type est un indicateur de la stabilité de la mesure. Un écart supérieur à 5% suggère une variabilité excessive — probablement causée par d'autres processus sur la machine, le scaling de fréquence CPU, ou le thermal throttling. Pour des mesures fiables, isolez le programme sur des cœurs dédiés (`taskset`) et désactivez le frequency scaling :

```bash
# Fixer la fréquence CPU au maximum
sudo cpupower frequency-set -g performance

# Isoler l'exécution sur le cœur 2
taskset -c 2 perf stat -r 5 ./mon_programme < donnees_test.txt
```

## Interpréter un rapport `perf stat` : méthodologie

Face à un rapport `perf stat`, une lecture séquentielle en quatre étapes permet de poser un diagnostic structuré.

### Étape 1 : Le programme est-il CPU-bound ?

Examinez `CPUs utilized`. Si la valeur est très inférieure à 1.0, le programme passe l'essentiel de son temps à attendre (I/O, réseau, locks). L'optimisation CPU est inutile — il faut d'abord résoudre le problème d'attente.

### Étape 2 : Le pipeline est-il efficace ?

Examinez l'IPC. Si l'IPC est supérieur à 1.5, le pipeline est raisonnablement alimenté et les gains viendront de la réduction du nombre d'instructions (algorithmes plus efficaces, élimination de travail inutile). Si l'IPC est inférieur à 1.0, le pipeline stagne — passez aux étapes suivantes pour identifier pourquoi.

### Étape 3 : Le problème est-il la mémoire ?

Examinez `cache-misses` (pourcentage des `cache-references`). Un taux élevé (> 10%) combiné avec un IPC faible confirme que le programme est *memory-bound*. Les optimisations relèvent de la localité des données : réorganisation des structures, conteneurs contigus, préchargement (prefetching), réduction de la taille des objets.

### Étape 4 : Le problème est-il les branches ?

Examinez `branch-misses` (pourcentage des `branches`). Un taux supérieur à 5% combiné avec un IPC faible indique que les mispredictions pénalisent significativement le pipeline. Les optimisations incluent la réorganisation des conditions, le *branchless programming*, ou le tri des données pour rendre les patterns de branchement plus prévisibles.

### Arbre de diagnostic

```
CPUs utilized < 1.0 ?
├─ Oui → Le programme est I/O-bound ou bloqué. Résoudre les attentes.
└─ Non → Examiner l'IPC
         IPC > 1.5 ?
         ├─ Oui → Pipeline efficace. Réduire le nombre d'instructions.
         └─ Non → Pipeline en stall. Pourquoi ?
                  cache-misses > 10% ?
                  ├─ Oui → Memory-bound. Optimiser la localité des données.
                  └─ Non → branch-misses > 5% ?
                           ├─ Oui → Branch-bound. Réduire les mispredictions.
                           └─ Non → Dépendances de données ou latence d'instructions.
```

## Cas d'usage : avant/après optimisation

`perf stat` est l'outil idéal pour mesurer l'impact d'une optimisation au niveau micro-architectural. Considérons un programme qui traite un tableau de structures :

**Avant** (structure de tableaux) :

```bash
perf stat -r 3 ./traitement_v1 < donnees.bin
```

```
   8 456 000 000       cycles
   4 230 000 000       instructions              #    0,50  insn per cycle
     312 000 000       cache-misses              #   28,36% of all cache refs
       2,204 +- 0,008 seconds time elapsed
```

IPC de 0.50 et 28% de cache misses : le programme est clairement *memory-bound*.

**Après** (réorganisation en *Structure of Arrays* pour améliorer la localité) :

```bash
perf stat -r 3 ./traitement_v2 < donnees.bin
```

```
   3 120 000 000       cycles
   3 980 000 000       instructions              #    1,28  insn per cycle
      24 000 000       cache-misses              #    3,45% of all cache refs
       0,814 +- 0,003 seconds time elapsed
```

L'IPC a plus que doublé (0.50 → 1.28), les cache misses ont chuté de 28% à 3.45%, et le temps d'exécution a été divisé par 2.7. Le nombre d'instructions a légèrement diminué (les patterns d'accès plus réguliers permettent au compilateur de mieux vectoriser), mais le gain principal vient de la suppression des stalls mémoire, confirmé par la chute spectaculaire des cache misses.

Ce type de comparaison factuelle, chiffrée au niveau matériel, est infiniment plus convaincant qu'une intuition. C'est la raison d'être de `perf stat` : transformer les hypothèses d'optimisation en mesures vérifiables.

## Intégration dans un workflow de profiling

`perf stat` et `perf record` ne sont pas des outils concurrents mais complémentaires. Un workflow d'optimisation complet les utilise en séquence :

1. **`perf stat`** : diagnostic macro. Le programme est-il CPU-bound, memory-bound, ou I/O-bound ? Quel est l'IPC ? Y a-t-il un problème de cache ou de branches ?

2. **`perf record` / `perf report`** : localisation. *Quelles fonctions* sont responsables du problème identifié par `perf stat` ? Si `perf stat` révèle des cache misses élevés, on peut enregistrer un profil sur l'événement `cache-misses` pour identifier les fonctions fautives :

    ```bash
    perf record -e cache-misses --call-graph fp ./mon_programme
    perf report
    ```

3. **`perf annotate`** : zoom au niveau de la ligne et de l'instruction.

4. **Optimisation ciblée**, puis retour à l'étape 1 pour vérifier l'impact avec `perf stat -r 5`.

Ce cycle — diagnostiquer globalement, localiser précisément, optimiser, mesurer — est le fondement d'une démarche d'optimisation disciplinée.
