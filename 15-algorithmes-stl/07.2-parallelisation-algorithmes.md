# 15.7.2 — Parallélisation de std::sort, std::transform, std::reduce

## Chapitre 15 — Algorithmes de la STL

---

## Introduction

La section 15.7.1 a posé le cadre théorique des politiques d'exécution. Cette section passe à la pratique : comment paralléliser concrètement les trois algorithmes les plus couramment concernés — `std::sort`, `std::transform` et `std::reduce` — et quel gain réel en attendre. Chaque algorithme a un profil de parallélisation différent, car la nature du travail (comparaisons, calcul indépendant, agrégation) se prête plus ou moins bien à la distribution sur plusieurs cœurs.

```cpp
#include <algorithm>
#include <numeric>
#include <execution>
#include <vector>
#include <chrono>
#include <random>
```

Pour mesurer les performances, nous utiliserons un chronomètre simple :

```cpp
class Timer {
    std::chrono::high_resolution_clock::time_point start_;
public:
    Timer() : start_(std::chrono::high_resolution_clock::now()) {}

    double elapsed_ms() const {
        auto now = std::chrono::high_resolution_clock::now();
        return std::chrono::duration<double, std::milli>(now - start_).count();
    }
};
```

> ⚠️ Les mesures de cette section sont **indicatives**. Les performances réelles dépendent du matériel (nombre de cœurs, architecture cache), de l'implémentation de la STL, de la version de TBB, de la charge système, et de nombreux autres facteurs. Utilisez toujours un framework de benchmarking (Google Benchmark, chapitre 35) pour des mesures fiables en production.

## std::sort parallèle

### Pourquoi le tri se parallélise bien

Le tri est naturellement parallélisable grâce à l'approche **diviser pour régner** : on divise la séquence en sous-séquences, on trie chaque sous-séquence indépendamment sur un thread distinct, puis on fusionne les résultats. C'est essentiellement un merge sort parallèle, même si l'implémentation peut varier.

```cpp
std::vector<int> generate_random(size_t n) {
    std::vector<int> v(n);
    std::mt19937 gen(42);
    std::uniform_int_distribution<int> dist(0, 1'000'000);
    std::generate(v.begin(), v.end(), [&]() { return dist(gen); });
    return v;
}
```

### Utilisation

```cpp
auto data = generate_random(10'000'000);

// Tri séquentiel
auto v_seq = data;  
Timer t_seq;  
std::sort(v_seq.begin(), v_seq.end());  
std::print("Séquentiel : {:.1f} ms\n", t_seq.elapsed_ms());  

// Tri parallèle
auto v_par = data;  
Timer t_par;  
std::sort(std::execution::par, v_par.begin(), v_par.end());  
std::print("Parallèle  : {:.1f} ms\n", t_par.elapsed_ms());  
```

### Ordres de grandeur typiques

Sur une machine 8 cœurs avec 10 millions d'entiers, les résultats typiques sont :

| Volume | `seq` | `par` | Speedup |
|---|---|---|---|
| 1 000 | ~0.05 ms | ~0.2 ms | **0.25x** (plus lent) |
| 10 000 | ~0.5 ms | ~0.5 ms | ~1x (neutre) |
| 100 000 | ~6 ms | ~2.5 ms | ~2.4x |
| 1 000 000 | ~65 ms | ~15 ms | ~4.3x |
| 10 000 000 | ~700 ms | ~130 ms | ~5.4x |
| 100 000 000 | ~8 000 ms | ~1 300 ms | ~6.2x |

Observations clés :

- En dessous de ~10 000 éléments, la version parallèle est **plus lente** que la séquentielle. Le coût de distribution du travail sur le pool de threads dépasse le gain.
- Le speedup ne suit pas une loi linéaire (8 cœurs ne donnent pas 8x). La fusion des sous-séquences triées est séquentielle par nature, et la bande passante mémoire devient le goulot d'étranglement sur les très grands volumes.
- Le tri est **memory-bound** pour les types simples (int, double) : les comparaisons sont rapides, c'est l'accès mémoire qui domine. Pour des types lourds avec un comparateur coûteux (par exemple, tri de chaînes longues), le speedup est généralement meilleur.

### Tri avec comparateur personnalisé

Le comparateur fonctionne exactement comme en séquentiel. Il doit respecter le strict weak ordering et être **thread-safe** (ce qui est automatique pour une lambda sans capture mutable) :

```cpp
struct Record {
    std::string key;
    double value;
};

std::vector<Record> records = /* ...millions d'entrées... */;

// Tri parallèle par valeur décroissante
std::sort(std::execution::par, records.begin(), records.end(),
    [](const Record& a, const Record& b) {
        return a.value > b.value;
    }
);
```

### std::stable_sort parallèle

`std::stable_sort` accepte aussi les politiques d'exécution. Le speedup est comparable à celui de `std::sort`, parfois légèrement meilleur grâce à la nature intrinsèquement parallélisable du merge sort sous-jacent :

```cpp
std::stable_sort(std::execution::par, records.begin(), records.end(),
    [](const Record& a, const Record& b) {
        return a.key < b.key;
    }
);
```

## std::transform parallèle

### Pourquoi transform se parallélise parfaitement

`std::transform` applique une fonction indépendamment à chaque élément. Il n'y a aucune dépendance entre les traitements — c'est le cas idéal de **parallélisme embarrassant** (embarrassingly parallel). Chaque thread peut traiter son bloc d'éléments sans aucune communication avec les autres.

### Transformation unaire

```cpp
std::vector<double> data(20'000'000);  
std::iota(data.begin(), data.end(), 1.0);  

std::vector<double> result(data.size());

// Séquentiel
Timer t_seq;  
std::transform(data.begin(), data.end(), result.begin(),  
    [](double x) { return std::sqrt(x) * std::log(x); }
);
std::print("Séquentiel : {:.1f} ms\n", t_seq.elapsed_ms());

// Parallèle
Timer t_par;  
std::transform(std::execution::par, data.begin(), data.end(), result.begin(),  
    [](double x) { return std::sqrt(x) * std::log(x); }
);
std::print("Parallèle  : {:.1f} ms\n", t_par.elapsed_ms());
```

### Transformation in-place

La source et la destination peuvent être identiques — la parallélisation fonctionne toujours car chaque thread écrit à une position différente, sans chevauchement :

```cpp
// Normaliser in-place — chaque élément est traité indépendamment
std::transform(std::execution::par,
    data.begin(), data.end(), data.begin(),
    [max_val](double x) { return x / max_val; }
);
```

### Transformation binaire

Combiner deux séquences en parallèle :

```cpp
std::vector<double> prices(5'000'000);  
std::vector<double> quantities(5'000'000);  
std::vector<double> totals(5'000'000);  
// ... initialisation ...

std::transform(std::execution::par,
    prices.begin(), prices.end(),
    quantities.begin(),
    totals.begin(),
    [](double price, double qty) { return price * qty; }
);
```

### Ordres de grandeur typiques

Le profil de performance de `transform` dépend fortement de la **complexité du callable** :

**Opération légère** (multiplication, addition) — La bande passante mémoire domine. Le speedup est modéré car les cœurs attendent la mémoire :

| Volume | `seq` | `par` | Speedup |
|---|---|---|---|
| 1 000 000 | ~2 ms | ~1.5 ms | ~1.3x |
| 10 000 000 | ~20 ms | ~7 ms | ~2.9x |

**Opération lourde** (sqrt + log, trigonométrie, parsing) — Le CPU est le goulot. Le speedup est excellent, proche du nombre de cœurs :

| Volume | `seq` | `par` | Speedup |
|---|---|---|---|
| 1 000 000 | ~25 ms | ~5 ms | ~5x |
| 10 000 000 | ~250 ms | ~40 ms | ~6.3x |

La règle : plus le callable est **coûteux en CPU**, plus la parallélisation est rentable. Pour une simple multiplication, le surcoût du thread pool peut ne pas être amorti. Pour un calcul trigonométrique complexe, le gain est quasi-linéaire avec le nombre de cœurs.

### par_unseq pour le calcul numérique pur

Quand le callable est une opération mathématique pure, `par_unseq` permet d'exploiter la vectorisation SIMD en plus du parallélisme :

```cpp
std::vector<float> data(10'000'000);
// ...

// Parallèle + vectorisé — performance maximale
std::transform(std::execution::par_unseq,
    data.begin(), data.end(), data.begin(),
    [](float x) { return x * 2.0f + 1.0f; }
);
```

Le gain supplémentaire de `par_unseq` par rapport à `par` dépend de la capacité du compilateur à vectoriser effectivement le callable. Les opérations arithmétiques simples sur `float` se vectorisent généralement bien. Les opérations avec branchements (`if/else`) ou appels de fonctions complexes se vectorisent mal.

## std::reduce parallèle

### Pourquoi reduce est conçu pour le parallélisme

`std::reduce` est le remplaçant parallélisable de `std::accumulate`. Là où `accumulate` impose un parcours séquentiel gauche-à-droite, `reduce` autorise l'implémentation à découper la séquence en blocs, réduire chaque bloc indépendamment, puis combiner les résultats partiels :

```
std::reduce(par, {a, b, c, d, e, f, g, h}, 0, +)

Thread 1: (a + b) = r1        Thread 2: (c + d) = r2  
Thread 3: (e + f) = r3        Thread 4: (g + h) = r4  

         (r1 + r2) = r12              (r3 + r4) = r34

                    (r12 + r34) = résultat final
```

C'est un arbre de réduction — la profondeur est O(log n) au lieu de O(n), ce qui permet un speedup significatif.

### Somme parallèle

```cpp
std::vector<double> data(50'000'000);  
std::iota(data.begin(), data.end(), 1.0);  

// Séquentiel — accumulate
Timer t_acc;  
double sum_acc = std::accumulate(data.begin(), data.end(), 0.0);  
std::print("accumulate : {:.1f} ms (sum = {:.0f})\n", t_acc.elapsed_ms(), sum_acc);  

// Séquentiel — reduce sans politique
Timer t_red_seq;  
double sum_red = std::reduce(data.begin(), data.end(), 0.0);  
std::print("reduce seq : {:.1f} ms (sum = {:.0f})\n", t_red_seq.elapsed_ms(), sum_red);  

// Parallèle — reduce avec par
Timer t_red_par;  
double sum_par = std::reduce(std::execution::par, data.begin(), data.end(), 0.0);  
std::print("reduce par : {:.1f} ms (sum = {:.0f})\n", t_red_par.elapsed_ms(), sum_par);  
```

### Ordres de grandeur typiques

| Volume | `accumulate` | `reduce(seq)` | `reduce(par)` | Speedup par/seq |
|---|---|---|---|---|
| 100 000 | ~0.1 ms | ~0.1 ms | ~0.3 ms | **0.3x** (overhead) |
| 1 000 000 | ~1 ms | ~1 ms | ~0.5 ms | ~2x |
| 10 000 000 | ~12 ms | ~12 ms | ~3 ms | ~4x |
| 50 000 000 | ~60 ms | ~60 ms | ~12 ms | ~5x |

La réduction est **memory-bound** pour les types simples : l'addition est quasi-instantanée, c'est la lecture des données depuis la RAM qui domine. Le speedup vient principalement de la meilleure utilisation de la bande passante mémoire via des accès parallèles depuis plusieurs cœurs.

### Réduction avec opération personnalisée

```cpp
// Trouver le maximum — parallélisable car max est associatif et commutatif
double max_val = std::reduce(std::execution::par,
    data.begin(), data.end(),
    std::numeric_limits<double>::lowest(),
    [](double a, double b) { return std::max(a, b); }
);
```

L'opération doit être **associative et commutative** pour que `reduce` parallèle produise un résultat correct. C'est le contrat fondamental de `std::reduce` — violer cette contrainte ne produit pas une erreur de compilation, mais des résultats incorrects.

## std::transform_reduce — Le couteau suisse parallèle

`std::transform_reduce` combine transformation et réduction en une seule passe, sans conteneur intermédiaire. C'est souvent l'algorithme parallèle le plus utile en pratique :

### Somme des carrés

```cpp
std::vector<double> data(10'000'000);
// ... initialisation ...

// Séquentiel : transform dans un buffer + accumulate
// → deux passes, une allocation intermédiaire

// Parallèle : une seule passe, zéro allocation
double sum_sq = std::transform_reduce(std::execution::par,
    data.begin(), data.end(),
    0.0,                              // valeur initiale
    std::plus<>{},                    // réduction
    [](double x) { return x * x; }   // transformation
);
```

### Produit scalaire parallèle

```cpp
std::vector<double> a(10'000'000), b(10'000'000);
// ... initialisation ...

// Produit scalaire — utilise + et * par défaut
double dot = std::transform_reduce(std::execution::par,
    a.begin(), a.end(),
    b.begin(),
    0.0
);
```

### Calcul de distance euclidienne

```cpp
std::vector<double> point_a(1'000'000), point_b(1'000'000);
// ... coordonnées en haute dimension ...

double distance = std::sqrt(
    std::transform_reduce(std::execution::par,
        point_a.begin(), point_a.end(),
        point_b.begin(),
        0.0,
        std::plus<>{},
        [](double a, double b) {
            double diff = a - b;
            return diff * diff;
        }
    )
);
```

### Comptage conditionnel parallèle

`std::count_if` parallèle existe, mais `transform_reduce` offre plus de flexibilité — par exemple, compter en pondérant :

```cpp
struct Measurement {
    double value;
    double weight;
    bool valid;
};

std::vector<Measurement> measurements(5'000'000);
// ...

// Somme pondérée des mesures valides
double weighted_sum = std::transform_reduce(std::execution::par,
    measurements.begin(), measurements.end(),
    0.0,
    std::plus<>{},
    [](const Measurement& m) {
        return m.valid ? m.value * m.weight : 0.0;
    }
);
```

## Autres algorithmes parallèles courants

### std::for_each parallèle

`for_each` parallèle est utile pour appliquer un effet de bord indépendant à chaque élément — typiquement une modification in-place :

```cpp
std::vector<double> data(10'000'000);
// ...

// Normalisation in-place parallèle
double max_val = std::reduce(std::execution::par,
    data.begin(), data.end(),
    std::numeric_limits<double>::lowest(),
    [](double a, double b) { return std::max(a, b); }
);

std::for_each(std::execution::par, data.begin(), data.end(),
    [max_val](double& x) { x /= max_val; }
);
```

### std::count_if et std::find_if parallèles

```cpp
std::vector<int> data(20'000'000);
// ...

// Comptage parallèle
auto n = std::count_if(std::execution::par, data.begin(), data.end(),
    [](int x) { return x > 500'000; }
);

// Recherche parallèle — renvoie le premier élément trouvé (garanti par le standard)
auto it = std::find_if(std::execution::par, data.begin(), data.end(),
    [](int x) { return x == 42; }
);
```

Note : `std::find_if` parallèle renvoie toujours un itérateur vers le **premier** élément satisfaisant le prédicat — le standard le garantit. La parallélisation concerne la recherche interne (plusieurs threads explorent des blocs simultanément), mais le résultat reste déterministe. Le surcoût de coordination entre threads fait que le gain est souvent modeste comparé à `transform` ou `reduce`.

### std::inclusive_scan et std::exclusive_scan parallèles

Les sommes préfixes se parallélisent grâce à l'algorithme de Blelloch (ou similaires). Le speedup est généralement moindre que pour `reduce`, car le scan a des dépendances séquentielles intrinsèques, mais il reste significatif sur de grands volumes :

```cpp
std::vector<double> data(10'000'000);  
std::vector<double> cumulative(data.size());  
// ...

std::inclusive_scan(std::execution::par,
    data.begin(), data.end(),
    cumulative.begin()
);
```

## Combiner plusieurs algorithmes parallèles

Un pipeline typique de traitement de données enchaîne plusieurs algorithmes parallèles :

```cpp
std::vector<double> raw_data(20'000'000);
// ... remplissage ...

// 1. Tri parallèle
std::sort(std::execution::par, raw_data.begin(), raw_data.end());

// 2. Déduplication (séquentiel — unique n'a pas de surcharge parallèle standard)
auto new_end = std::unique(raw_data.begin(), raw_data.end());  
raw_data.erase(new_end, raw_data.end());  

// 3. Transformation parallèle
std::vector<double> processed(raw_data.size());  
std::transform(std::execution::par,  
    raw_data.begin(), raw_data.end(), processed.begin(),
    [](double x) { return std::log1p(x); }
);

// 4. Statistiques parallèles
double sum = std::reduce(std::execution::par,
    processed.begin(), processed.end(), 0.0);

auto [min_it, max_it] = std::minmax_element(std::execution::par,
    processed.begin(), processed.end());

double mean = sum / static_cast<double>(processed.size());  
std::print("N={}, Mean={:.4f}, Min={:.4f}, Max={:.4f}\n",  
           processed.size(), mean, *min_it, *max_it);
```

Chaque algorithme parallèle attend la fin du précédent avant de démarrer — il n'y a pas de parallélisme *entre* les algorithmes, seulement *au sein* de chaque algorithme. Pour paralléliser des pipelines entiers, `std::execution` (Senders/Receivers, C++26) offrira des solutions plus élaborées.

## Facteurs influençant le speedup réel

Plusieurs facteurs expliquent pourquoi le speedup est rarement linéaire (N cœurs ne donnent pas Nx) :

**Bande passante mémoire** — Pour les opérations simples (addition, comparaison), le CPU attend les données venant de la RAM. Ajouter des cœurs ne résout pas ce goulot. C'est le facteur limitant principal pour `reduce` sur des `int` ou `double`.

**Loi d'Amdahl** — La fraction séquentielle incompressible (initialisation du thread pool, fusion des résultats, synchronisation) limite le speedup maximal. Même avec des cœurs infinis, si 10% du travail est séquentiel, le speedup plafonne à 10x.

**Taille des données vs nombre de cœurs** — Chaque thread a besoin d'un bloc de travail suffisant pour amortir le surcoût de synchronisation. Avec 8 cœurs et 1 000 éléments, chaque thread ne traite que ~125 éléments — le surcoût domine.

**Localité de cache** — Le partitionnement des données entre cœurs peut créer des accès mémoire non-locaux. Les implémentations essaient de partitionner en blocs contigus pour maximiser la localité.

**Complexité du callable** — C'est le facteur le plus favorable. Plus le callable est coûteux en CPU (trigonométrie, parsing, crypto…), plus la parallélisation est rentable, car le ratio calcul/overhead augmente.

**Hyper-threading** — Les cœurs logiques (hyper-threads) ne doublent pas la performance — ils partagent les unités de calcul physiques. Le gain typique de l'hyper-threading est de 15-30%, pas 100%.

## Synthèse

`std::sort`, `std::transform` et `std::reduce` représentent les trois profils fondamentaux de parallélisation : diviser-pour-régner (sort), parallélisme embarrassant (transform), et réduction arborescente (reduce). Le gain réel dépend du volume de données, de la complexité du callable, et de la bande passante mémoire. `std::transform` est celui qui se parallélise le mieux quand le callable est coûteux. `std::sort` offre un speedup robuste sur les grands volumes. `std::reduce` est limité par la bande passante mémoire pour les types simples. `std::transform_reduce` combine le meilleur des deux mondes en fusionnant transformation et réduction en une seule passe. Dans tous les cas, la mesure est souveraine : les ordres de grandeur donnés ici sont des guides, pas des garanties.
