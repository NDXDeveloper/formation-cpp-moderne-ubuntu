# 45.6.4 — Stratégie de migration progressive et interopérabilité Rust

## Section 45.6 — Sécurité mémoire : Réponses concrètes du comité C++ en 2026 🔥
## Chapitre 45 — Sécurité en C++ ⭐

---

## Introduction

Les sections précédentes ont couvert les réponses internes à l'écosystème C++ : Profiles (45.6.2), hardening en production (45.6.3). Ces réponses améliorent la sécurité du code C++ existant et futur, mais elles ne changent pas la nature fondamentale du langage : le C++ reste un langage dans lequel la sécurité mémoire dépend de la discipline du développeur et des outils, pas de garanties structurelles du compilateur.

Pour un nombre croissant de projets, la question n'est plus "comment rendre notre C++ plus sûr" mais "quand est-il pertinent d'introduire Rust pour les composants à haute criticité". Cette question n'est pas idéologique — elle est pragmatique. Google l'a démontré sur Android : écrire le code nouveau en Rust sans réécrire le code C++ existant a réduit les vulnérabilités mémoire de 76 % à 24 % en six ans. Le noyau Linux accepte du code Rust depuis 2022. Chromium intègre du Rust dans ses composants réseau. Ces décisions ont été prises par des équipes qui connaissent intimement le C++ et qui ont choisi l'interopérabilité plutôt que la réécriture.

Cette section couvre la stratégie de migration progressive — quand introduire Rust, comment structurer la coexistence avec le C++, et quels outils utiliser pour l'interopérabilité. Les détails techniques de l'interopérabilité (FFI, `cxx`, `autocxx`) sont couverts en section 43.3 ; cette section se concentre sur la vision stratégique et les critères de décision.

---

## Le principe : écrire le nouveau code safe, renforcer l'existant

La stratégie de migration progressive repose sur un constat simple : la réécriture complète d'une base de code C++ en Rust (ou dans tout autre langage) est presque toujours irréaliste et inutile. Les bases de code C++ matures contiennent des décennies de logique métier testée, des optimisations spécifiques au domaine, et une connaissance implicite des cas limites qui serait perdue dans une réécriture. L'histoire de l'informatique est jonchée de projets de réécriture totale qui ont échoué.

La stratégie viable est double :

**1. Renforcer le code C++ existant** — hardening STL, sanitizers en production, fuzzing continu, analyse statique (sections 45.6.2 et 45.6.3). Cela réduit progressivement la surface de vulnérabilités sans modification architecturale.

**2. Écrire les composants nouveaux à haute criticité en Rust** — les parsers, les codecs, les couches réseau, les opérations cryptographiques. Ces composants traitent des données non fiables provenant de l'extérieur et constituent la majorité de la surface d'attaque. Les écrire dans un langage qui garantit statiquement la sécurité mémoire élimine des classes entières de vulnérabilités dès la conception.

Les deux volets fonctionnent de concert : le renforcement du C++ produit des résultats immédiats sur la base de code existante, tandis que Rust élimine les vulnérabilités dans les composants nouveaux. Au fil du temps, la proportion de code memory-safe augmente naturellement sans nécessiter de réécriture massive.

## Quand introduire Rust : critères de décision

L'introduction de Rust dans un projet C++ n'est pas toujours pertinente. Elle a un coût réel — formation de l'équipe, complexité de build, frontière FFI à maintenir — et ce coût doit être justifié par un bénéfice de sécurité proportionné. Voici les critères pour évaluer la pertinence.

### Critères favorables à l'introduction de Rust

**Le composant traite des entrées non fiables.** Parsers de protocoles réseau, décodeurs de formats de fichiers (images, vidéo, documents), désérialiseurs — tout code qui interprète des données provenant de sources non contrôlées est un candidat naturel. Ces composants concentrent historiquement la majorité des CVE.

**Le composant est un nouveau développement.** Écrire du code nouveau en Rust est bien moins coûteux que de réécrire du code C++ existant. La frontière FFI n'existe qu'à l'interface, pas au cœur du composant.

**Le composant a une interface bien définie.** L'interopérabilité C++/Rust fonctionne mieux quand la frontière entre les deux langages est étroite et stable — quelques fonctions avec des types simples (entiers, slices, strings, structs POD). Un composant dont l'interface est un enchevêtrement de templates C++ et de hiérarchies d'héritage sera pénible à exposer via FFI.

**L'équipe a le temps d'apprendre.** Rust a une courbe d'apprentissage significative, en particulier pour le borrow checker. Un développeur C++ compétent devient productif en Rust en 2 à 4 mois, mais les 2 à 3 premières semaines sont souvent frustrantes. L'introduction de Rust sur un projet sous pression de deadline est contre-productive.

**Le contexte réglementaire l'exige.** Pour les produits soumis au Cyber Resilience Act (section 45.6.1) ou vendus à des agences gouvernementales américaines, la capacité à démontrer l'utilisation de langages memory-safe pour les composants critiques est un atout concret.

### Critères défavorables

**Le composant est du C++ existant stable et bien testé.** Réécrire un composant C++ mature qui ne produit pas de CVE n'apporte aucun bénéfice de sécurité et introduit un risque de régression. Le renforcement (hardening, fuzzing) est une meilleure approche.

**L'interface entre C++ et Rust serait trop large.** Si le composant candidat nécessite des échanges massifs d'objets C++ complexes (smart pointers, conteneurs STL, hiérarchies polymorphiques), le coût de la frontière FFI peut excéder le bénéfice de sécurité.

**L'équipe est trop petite ou ne peut pas se former.** Un développeur seul sur un projet ne peut pas maintenir deux toolchains. L'introduction de Rust nécessite au minimum deux développeurs compétents pour la revue de code et la continuité.

**Les performances au niveau de la frontière FFI sont critiques.** Chaque appel à travers la frontière C++/Rust a un coût (comparable à un appel `extern "C"`, c'est-à-dire faible mais non nul). Pour du code appelé des millions de fois par seconde dans une boucle interne, la frontière FFI peut être un point de contention. La solution est de placer la frontière à un niveau plus haut (par composant, pas par fonction), mais cela nécessite une réflexion architecturale.

## Architecture de coexistence

### Le modèle bibliothèque

Le pattern le plus courant et le plus propre : le composant Rust est compilé en bibliothèque statique (`.a`) ou dynamique (`.so`) et exposé au code C++ via une interface C. L'outil `cxx` (section 43.3.2) génère automatiquement les bindings :

```
┌──────────────────────────────────────────────────┐
│               Application C++                    │
│                                                  │
│  ┌───────────┐  ┌───────────┐  ┌──────────────┐  │
│  │  Module   │  │  Module   │  │   Module     │  │
│  │  C++ #1   │  │  C++ #2   │  │   C++ #3     │  │
│  └───────────┘  └───────────┘  └──────┬───────┘  │
│                                       │          │
│                              ┌────────▼────────┐ │
│                              │  Interface cxx  │ │
│                              │  (C ABI, types  │ │
│                              │   partagés)     │ │
│                              └────────┬────────┘ │
│                                       │          │
│                              ┌────────▼────────┐ │
│                              │  Bibliothèque   │ │
│                              │  Rust (parser,  │ │
│                              │  codec, crypto) │ │
│                              └─────────────────┘ │
└──────────────────────────────────────────────────┘
```

Le code C++ appelle les fonctions Rust à travers l'interface `cxx`. Le code Rust ne connaît pas les internals du C++ — il reçoit des données (slices, strings, structs) et retourne des résultats. La frontière est étroite, stable et testable indépendamment.

### Le modèle microservice

Pour les projets qui utilisent déjà une architecture distribuée, une alternative à l'interopérabilité in-process est l'isolation au niveau du service : le composant critique est un service Rust séparé, communiquant avec le reste du système via gRPC, HTTP ou une file de messages. Aucune frontière FFI n'est nécessaire — les deux langages communiquent par sérialisation.

Ce modèle élimine complètement les problèmes d'interopérabilité au niveau du langage, mais il introduit une latence réseau et une complexité opérationnelle. Il est adapté aux architectures qui sont déjà distribuées, pas aux applications monolithiques.

### Le modèle remplacement progressif

Dans ce modèle, les composants C++ sont progressivement remplacés par des équivalents Rust, un par un, en commençant par les plus exposés. Chaque remplacement maintient la même interface externe (C ABI) pour ne pas impacter les consommateurs :

```
Année 1 :  [C++ HTTP parser] [C++ JSON parser] [C++ crypto] [C++ business logic]  
Année 2 :  [Rust HTTP parser] [C++ JSON parser] [C++ crypto] [C++ business logic]  
Année 3 :  [Rust HTTP parser] [Rust JSON parser] [Rust crypto] [C++ business logic]  
Année 4+:  [Rust HTTP parser] [Rust JSON parser] [Rust crypto] [C++ business logic]  
           (la logique métier C++ reste — elle est stable et moins exposée)
```

La logique métier C++ — la partie du code la moins exposée aux entrées non fiables et la plus riche en logique spécifique au domaine — n'a généralement pas besoin d'être réécrite. Le ROI de la réécriture est dans les couches de parsing et de protocole, pas dans la logique métier.

## Outils d'interopérabilité : vue d'ensemble

La section 43.3 couvre les détails techniques de chaque outil. Voici un résumé orienté décision :

### `cxx` — le bridge de référence

`cxx` (section 43.3.2) est le crate Rust de référence pour l'interopérabilité C++/Rust. Il génère du code C++ et Rust à partir d'une description d'interface dans un bloc `#[cxx::bridge]`. Les types sont vérifiés à la compilation des deux côtés — une erreur de typage à la frontière est détectée avant l'exécution.

**Points forts** : sûreté de type à la frontière, support des types riches (String, Vec, UniquePtr, SharedPtr), pas de code `unsafe` exposé à l'utilisateur (le unsafe est encapsulé dans le crate `cxx`), maintenance active, utilisé en production par Chromium.

**Limitations** : nécessite une description manuelle de l'interface, ne supporte pas les templates C++ ni les hiérarchies d'héritage polymorphiques directement, courbe d'apprentissage pour la description de l'interface.

### `autocxx` — bindings automatiques

`autocxx` (section 43.3.3) génère automatiquement des bindings Rust à partir des headers C++. Il analyse les headers et produit des bindings Rust sans que le développeur ait à écrire manuellement la description de l'interface.

**Points forts** : automatisation, couverture large des constructions C++ (classes, méthodes, surcharges), réduit le travail de mise en place initiale.

**Limitations** : les bindings générés sont souvent plus complexes et moins idiomatiques que ceux écrits à la main avec `cxx`, support imparfait des patterns C++ avancés (templates, SFINAE), l'automatisation peut masquer des problèmes d'interface.

### `extern "C"` — le niveau le plus bas

Pour les interfaces très simples (quelques fonctions avec des types C), l'interopérabilité manuelle via `extern "C"` est suffisante et ne nécessite aucune dépendance externe :

```cpp
// Côté C++ — expose une fonction via C ABI
extern "C" {
    int32_t rust_parse_data(const uint8_t* data, size_t len);
}

void process(std::span<const uint8_t> input) {
    int32_t result = rust_parse_data(input.data(), input.size());
    // ...
}
```

```rust
// Côté Rust — implémente la fonction
#[no_mangle]
pub extern "C" fn rust_parse_data(data: *const u8, len: usize) -> i32 {
    let slice = unsafe { std::slice::from_raw_parts(data, len) };
    // Traitement safe à partir d'ici
    match parse(slice) {
        Ok(_) => 0,
        Err(_) => -1,
    }
}
```

Ce pattern est simple mais fragile : aucune vérification de type entre les deux côtés, gestion manuelle de la mémoire à la frontière, et code `unsafe` explicite côté Rust. `cxx` est préférable dès que l'interface dépasse quelques fonctions.

### Choix de l'outil

| Contexte | Outil recommandé |
|---|---|
| Nouveau composant Rust avec interface définie | `cxx` |
| Wrapper Rust autour d'une grande bibliothèque C++ existante | `autocxx` (initial) puis `cxx` (refinement) |
| Interface très simple (< 5 fonctions, types C) | `extern "C"` |
| Architecture de microservices | Pas d'interop langage — gRPC / Protobuf |

## Intégration dans le système de build

### CMake + Cargo

L'intégration de Rust dans un projet C++ géré par CMake est le point de friction le plus courant. Le crate `corrosion` (anciennement `cmake-cargo`) est la solution recommandée :

```cmake
# CMakeLists.txt
include(FetchContent)  
FetchContent_Declare(  
    Corrosion
    GIT_REPOSITORY https://github.com/corrosion-rs/corrosion.git
    GIT_TAG v0.5
)
FetchContent_MakeAvailable(Corrosion)

# Importer la bibliothèque Rust
corrosion_import_crate(MANIFEST_PATH rust_parser/Cargo.toml)

# Lier la bibliothèque Rust au binaire C++
add_executable(my_app main.cpp)  
target_link_libraries(my_app PRIVATE rust_parser)  
```

Corrosion invoque `cargo build` en tant qu'étape de build CMake, gère les dépendances de compilation, et expose les artefacts Rust (`.a`, `.so`) comme des cibles CMake standards. Le développeur C++ utilise `target_link_libraries` comme pour n'importe quelle autre dépendance.

### Structure de projet recommandée

```
project/
├── CMakeLists.txt              # Build principal (C++ + Rust via Corrosion)
├── src/                        # Code C++
│   ├── main.cpp
│   ├── business_logic/         # C++ — logique métier (pas de migration)
│   └── ffi/
│       └── parser_bridge.h     # Interface C++ vers le parser Rust
├── rust_parser/                # Crate Rust
│   ├── Cargo.toml
│   └── src/
│       ├── lib.rs              # Point d'entrée + bridge cxx
│       └── parser.rs           # Implémentation du parser (100% safe Rust)
├── tests/                      # Tests C++ (GTest)
├── fuzz/                       # Harnesses de fuzzing (C++ et Rust)
└── .github/workflows/
    └── ci.yml                  # CI avec Rust + C++
```

### CI/CD

Le pipeline CI doit compiler les deux langages. GitHub Actions avec la combinaison `actions/setup-rust` et un compilateur C++ installé fonctionne directement :

```yaml
- name: Install Rust
  uses: dtolnay/rust-toolchain@stable

- name: Install C++ toolchain
  run: sudo apt-get install -y clang cmake ninja-build

- name: Build
  run: |
    cmake -B build -G Ninja -DCMAKE_CXX_COMPILER=clang++
    cmake --build build

- name: Test C++
  run: ctest --test-dir build

- name: Test Rust
  run: cd rust_parser && cargo test
```

## Retours d'expérience de l'industrie

### Android (Google)

Le cas le plus documenté. Google a commencé à écrire du code nouveau en Rust pour Android en 2019. En six ans, la proportion de vulnérabilités mémoire dans les nouveaux composants est passée de 76 % à 24 %, sans réécriture du code C++ existant. Le code Rust couvre notamment les composants Bluetooth, le DNS resolver, le keystore, et des parties du framework multimédia.

L'enseignement clé : la migration n'a pas besoin d'être totale pour être efficace. Le code nouveau est une fraction du code total, mais il couvre une part disproportionnée des surfaces d'attaque (car les composants anciens ont déjà été durcis au fil des correctifs).

### Chromium (Google)

Chromium intègre du Rust depuis 2023, en commençant par les composants de parsing de formats (fontes, QR codes, images). L'interopérabilité utilise `cxx` et `autocxx`. L'équipe a documenté les défis : le système de build complexe de Chromium (GN + Ninja) a nécessité un travail d'intégration significatif, et les conventions de codage C++ de Chromium (pas d'exceptions, pas de RTTI) simplifient la frontière FFI mais limitent l'utilisation de certaines fonctionnalités de `cxx`.

### Noyau Linux

Le noyau Linux accepte du code Rust depuis la version 6.1 (décembre 2022). Les premiers sous-systèmes écrits en Rust incluent les drivers de systèmes de fichiers et les drivers de périphériques. L'interopérabilité utilise `extern "C"` direct (pas de `cxx`) car l'interface du noyau est en C pur, pas en C++.

L'enseignement pour les projets C++ : même l'écosystème le plus conservateur et le plus exigeant en performance (le noyau Linux) a jugé que l'introduction de Rust était justifiée pour les composants à haute criticité.

### Microsoft

Microsoft investit dans Rust pour les composants Windows critiques. Le projet a contribué à la toolchain Rust pour les cibles Windows et explore l'interopérabilité avec le code C++ de l'OS. En 2026, Rust est utilisé dans des composants de sécurité et de réseau.

## Pièges courants et comment les éviter

### Piège 1 : la réécriture totale

Le piège le plus coûteux est de décider de "tout réécrire en Rust". Les projets de réécriture totale échouent pour les mêmes raisons en Rust qu'en tout autre langage : perte de connaissance implicite, régression des cas limites, et coût sous-estimé par un facteur 3 à 10.

**Alternative** : migration progressive, composant par composant, en commençant par les surfaces d'attaque les plus exposées. Le code C++ existant qui ne traite pas de données non fiables n'a généralement pas besoin d'être réécrit.

### Piège 2 : la frontière FFI comme décharge de unsafe

Toute frontière FFI contient du code `unsafe` côté Rust. Le piège est de laisser cet `unsafe` proliférer au-delà du strict nécessaire. La discipline consiste à encapsuler l'`unsafe` dans une couche mince qui expose une API safe Rust :

```rust
// ❌ unsafe qui s'étale
pub fn process(raw_ptr: *const u8, len: usize) {
    unsafe {
        // 50 lignes de code unsafe
        // Tout le bénéfice de Rust est perdu
    }
}

// ✅ unsafe minimal, encapsulé
pub fn process(raw_ptr: *const u8, len: usize) -> Result<Output, Error> {
    // Conversion unsafe → safe à la frontière
    let data = unsafe {
        if raw_ptr.is_null() { return Err(Error::NullPointer); }
        std::slice::from_raw_parts(raw_ptr, len)
    };
    // À partir d'ici, tout est safe Rust
    parse_safe(data)
}
```

`cxx` résout ce problème en encapsulant l'`unsafe` dans le code généré, rendant l'API utilisateur entièrement safe. C'est une raison supplémentaire de préférer `cxx` à `extern "C"` manuel.

### Piège 3 : sous-estimer la courbe d'apprentissage

Le borrow checker de Rust est conceptuellement simple mais demande du temps à internaliser. Les développeurs C++ expérimentés avec les smart pointers et le RAII ont un avantage, mais ils butent souvent sur les lifetime parameters et les patterns d'emprunt qui n'ont pas d'équivalent direct en C++.

**Recommandation** : commencer par un composant isolé et non critique pour l'apprentissage. Un parser de fichier de configuration ou un utilitaire de conversion de format est un bon premier projet — suffisamment complexe pour être formateur, pas assez critique pour bloquer l'équipe en cas de difficulté.

### Piège 4 : ignorer le coût de maintenance de la frontière

La frontière FFI est un contrat entre deux langages. Quand l'API C++ évolue, les bindings doivent être mis à jour. Quand les types changent, les conversions à la frontière doivent être adaptées. Ce coût de maintenance est faible si l'interface est étroite et stable, mais il peut devenir significatif si l'interface évolue fréquemment.

**Recommandation** : définir l'interface FFI comme un contrat versionné, avec des types simples et stables. Placer la logique d'adaptation côté Rust (plus facile à tester et à refactorer avec le borrow checker) plutôt que côté C++.

## Construire une feuille de route de migration

Pour répondre aux exigences réglementaires (section 45.6.1), une feuille de route de migration progressive devrait inclure les éléments suivants :

### Inventaire et priorisation

1. **Lister les composants** de la base de code par catégorie : parsing/protocole, cryptographie, réseau, logique métier, UI, infrastructure.
2. **Évaluer l'exposition** de chaque composant : traite-t-il des entrées non fiables ? A-t-il un historique de CVE ? Est-il couvert par le fuzzing ?
3. **Prioriser** : les composants à haute exposition et faible couverture de test sont les candidats prioritaires.

### Planning réaliste

| Trimestre | Action |
|---|---|
| T1 | Formation de 2-3 développeurs sur Rust. Prototype sur un composant non critique. |
| T2 | Premier composant Rust en production (parser de format ou codec). Intégration CMake + Cargo. |
| T3–T4 | Deuxième et troisième composants. Stabilisation du workflow CI/CD. |
| Année 2+ | Migration progressive des surfaces d'attaque restantes. Le code C++ existant est maintenu et renforcé. |

### Métriques de suivi

- **Proportion de code memory-safe** dans le projet (lignes Rust / lignes totales).  
- **Proportion de la surface d'attaque couverte par du code memory-safe** (composants Rust / composants exposés aux entrées non fiables).  
- **Nombre de CVE mémoire** par année — la métrique ultime.  
- **Surcoût de build** introduit par le build Rust (temps de compilation additionnel).

## Résumé

La migration progressive vers Rust est une stratégie complémentaire au hardening du C++ existant, pas une alternative :

- **Ne pas réécrire l'existant.** Renforcer le C++ existant avec le hardening STL, les sanitizers et le fuzzing (sections 45.6.2–45.6.3).  
- **Écrire le code nouveau critique en Rust.** Les parsers, codecs, couches réseau et opérations cryptographiques sont les candidats prioritaires.  
- **Utiliser `cxx` comme bridge.** C'est l'outil le plus sûr et le plus ergonomique pour l'interopérabilité C++/Rust.  
- **Commencer petit.** Un composant isolé, une équipe formée, une interface étroite. Élargir après le premier succès.  
- **Mesurer.** La proportion de surface d'attaque couverte par du code memory-safe est la métrique clé.  
- **Les deux approches se renforcent mutuellement.** Le hardening C++ réduit les risques sur le code existant pendant que Rust élimine les risques sur le code nouveau. L'ensemble progresse sur le continuum de sécurité mémoire (section 45.6.1).
