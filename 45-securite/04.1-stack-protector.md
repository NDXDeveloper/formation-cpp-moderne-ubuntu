# 45.4.1 — `-fstack-protector`

## Section 45.4 — Compilation avec protections
## Chapitre 45 — Sécurité en C++ ⭐

---

## Introduction

Le stack protector — communément appelé stack canary ou stack guard — est le mécanisme de protection le plus ancien et le plus répandu contre les stack buffer overflows. Le concept a été introduit en 1998 par Crispin Cowan avec StackGuard, puis intégré dans GCC sous le nom ProPolice par IBM. Aujourd'hui, il est disponible sur tous les compilateurs majeurs (GCC, Clang, MSVC) et activé par défaut dans la plupart des distributions Linux.

Le principe est élégant dans sa simplicité : placer une valeur sentinelle (le canary) sur la pile entre les variables locales et l'adresse de retour. Un buffer overflow qui écrase l'adresse de retour doit nécessairement traverser le canary. En vérifiant l'intégrité du canary avant de retourner de la fonction, le programme détecte la corruption et se termine immédiatement — avant que l'adresse de retour falsifiée ne soit utilisée.

---

## Mécanisme détaillé

### Disposition de la pile sans protection

Sans stack protector, la pile d'une fonction contenant un tableau local ressemble à ceci :

```
Adresses hautes (début de la pile)
┌──────────────────────────┐
│    Arguments de la       │
│    fonction appelante    │
├──────────────────────────┤
│    Return Address        │  ← Cible d'un attaquant
├──────────────────────────┤
│    Saved Frame Pointer   │  ← RBP sauvegardé
│    (RBP)                 │
├──────────────────────────┤
│    Autres variables      │
│    locales (int x, ...)  │
├──────────────────────────┤
│    buffer[64]            │  ← Début du buffer overflow
│    (tableau local)       │
└──────────────────────────┘
Adresses basses (sommet de la pile)
```

Un buffer overflow sur `buffer` écrit vers les adresses hautes, écrasant d'abord les variables locales adjacentes, puis le frame pointer sauvegardé, et enfin l'adresse de retour. Lorsque la fonction exécute `ret`, le processeur saute à l'adresse écrasée.

### Disposition de la pile avec stack protector

Avec `-fstack-protector`, le compilateur insère un canary entre les variables locales et les données de contrôle de la pile :

```
Adresses hautes
┌──────────────────────────┐
│    Arguments de la       │
│    fonction appelante    │
├──────────────────────────┤
│    Return Address        │
├──────────────────────────┤
│    Saved Frame Pointer   │
├──────────────────────────┤
│  ▶ CANARY (8 octets) ◀   │  ← Valeur sentinelle
├──────────────────────────┤
│    Autres variables      │
│    locales               │
├──────────────────────────┤
│    buffer[64]            │  ← Début du buffer overflow
└──────────────────────────┘
Adresses basses
```

De plus, le compilateur réordonne les variables locales de sorte que les tableaux (les cibles les plus probables d'un overflow) soient placés aux adresses les plus basses, juste avant le canary. Ainsi, un overflow sur un tableau doit obligatoirement traverser le canary pour atteindre l'adresse de retour.

### Le code généré

Concrètement, le compilateur ajoute deux blocs de code à chaque fonction protégée :

**En prologue** — lecture de la valeur canary depuis un emplacement connu (typiquement `fs:0x28` sur x86_64 Linux, via le Thread Local Storage) et stockage sur la pile :

```asm
; Prologue — insertion du canary
mov    rax, QWORD PTR fs:0x28    ; Lire le canary depuis le TLS  
mov    QWORD PTR [rbp-0x8], rax  ; Stocker sur la pile  
xor    eax, eax                   ; Effacer rax (éviter fuite du canary)  
```

**En épilogue** — relecture du canary depuis la pile, comparaison avec la valeur de référence, et terminaison si la valeur a changé :

```asm
; Épilogue — vérification du canary
mov    rax, QWORD PTR [rbp-0x8]  ; Relire le canary depuis la pile  
xor    rax, QWORD PTR fs:0x28    ; Comparer avec la référence  
jne    .stack_chk_fail            ; Si différent → corruption détectée  
; ... retour normal ...

.stack_chk_fail:
    call   __stack_chk_fail       ; Termine le programme immédiatement
```

La fonction `__stack_chk_fail` (fournie par la libc) affiche un message d'erreur et appelle `abort()`. Le programme est terminé de manière inconditionnelle — il n'y a pas de mécanisme de récupération, car la pile est corrompue et aucune exécution fiable n'est possible.

### Caractéristiques du canary

La valeur du canary n'est pas une constante — elle est générée aléatoirement au démarrage du programme et stockée dans le TLS. Chaque thread possède sa propre copie. Cela signifie :

- Un attaquant ne peut pas prédire la valeur du canary d'une exécution à l'autre.  
- Le canary contient généralement un octet nul (`\0`) en position basse, ce qui complique les overflows via des fonctions de chaînes C (`strcpy`, `strcat`) — ces fonctions s'arrêtent au premier octet nul et ne peuvent donc pas écrire à travers le canary sans le corrompre.  
- Sur les systèmes 64 bits, le canary fait 8 octets, ce qui donne 2^56 valeurs possibles (en excluant l'octet nul fixe) — un brute-force est impraticable.

## Les variantes du flag

GCC et Clang proposent plusieurs variantes qui diffèrent par le critère de sélection des fonctions à protéger :

### `-fstack-protector` (basique)

Protège uniquement les fonctions qui contiennent un appel à `alloca()` ou qui déclarent un tableau de caractères (`char[]`) de plus de 8 octets. C'est la variante la plus conservatrice — elle ne protège qu'un sous-ensemble restreint de fonctions :

```cpp
void protected_basic() {
    char buffer[16];    // ✅ Protégée — tableau de char > 8 octets
    // ...
}

void not_protected() {
    int data[100];      // ❌ Non protégée — ce n'est pas un char[]
    char small[4];      // ❌ Non protégée — char[] ≤ 8 octets
    // ...
}
```

Cette variante est historique et trop limitée pour offrir une protection réelle. Elle rate les overflows sur les tableaux d'entiers, les structures, et les petits tableaux de caractères.

### `-fstack-protector-strong` (recommandée) ⭐

Introduite dans GCC 4.9, cette variante étend la protection à toute fonction qui :

- contient un tableau local de tout type (pas seulement `char[]`),  
- déclare une variable locale dont l'adresse est prise (`&var`),  
- utilise `alloca()`,  
- contient une variable locale de type structure ou union avec un tableau membre.

```cpp
void protected_strong_1() {
    int data[50];       // ✅ Protégée — tableau local (tout type)
}

void protected_strong_2() {
    int x = 42;
    int* p = &x;        // ✅ Protégée — adresse de variable locale prise
    process(p);
}

void protected_strong_3() {
    struct Header {
        char name[32];
        int value;
    } h;                // ✅ Protégée — structure avec tableau membre
}

void not_protected_strong() {
    int a = 1, b = 2;   // ❌ Non protégée — scalaires simples,
    return a + b;        //    pas de tableau ni de prise d'adresse
}
```

`-fstack-protector-strong` offre le meilleur compromis couverture/performance. Elle protège la grande majorité des fonctions vulnérables tout en épargnant les fonctions simples (scalaires, pas de tableau) qui ne peuvent pas faire l'objet d'un buffer overflow par construction. C'est la variante recommandée pour la production.

### `-fstack-protector-all`

Protège **toutes** les fonctions sans exception, y compris celles qui ne contiennent aucun tableau ni aucune prise d'adresse :

```cpp
// Même cette fonction triviale est protégée avec -all
int add(int a, int b) {
    return a + b;       // ✅ Protégée (inutilement)
}
```

Le surcoût est plus élevé que `-fstack-protector-strong` car chaque fonction subit le prologue et l'épilogue de vérification. Le bénéfice supplémentaire par rapport à `-strong` est marginal, car les fonctions sans tableaux ni prises d'adresse ne sont généralement pas vulnérables aux buffer overflows.

Cette variante est pertinente dans les environnements à très haute sécurité (noyaux, firmware de sécurité) où le surcoût est accepté en échange d'une couverture maximale.

### Tableau récapitulatif

| Variante | Fonctions protégées | Surcoût typique | Usage recommandé |
|---|---|---|---|
| `-fstack-protector` | `char[]` > 8 octets, `alloca` | Minimal | Historique — insuffisant |
| `-fstack-protector-strong` | Tableaux, `&var`, `alloca`, structs avec tableaux | < 1 % | **Production** (recommandé) |
| `-fstack-protector-all` | Toutes les fonctions | 1-5 % selon la charge | Sécurité maximale |

## Observer le stack protector en action

### Compilation et inspection du code généré

On peut vérifier que le canary est bien inséré en inspectant l'assembleur généré :

```bash
# Compiler avec stack protector
g++ -std=c++23 -O2 -fstack-protector-strong -S vulnerable.cpp -o vulnerable.s

# Chercher les instructions de canary dans l'assembleur
grep -n "stack_chk\|fs:0x28" vulnerable.s
```

Sortie attendue pour une fonction protégée :

```
12:     mov    rax, QWORD PTR fs:0x28
42:     xor    rax, QWORD PTR fs:0x28
43:     jne    .L5
58: .L5:
59:     call   __stack_chk_fail@PLT
```

Si ces lignes sont absentes, la fonction n'a pas été jugée éligible par le compilateur pour la variante choisie.

### Comportement à l'exécution lors d'une corruption

Lorsque le canary est corrompu, le programme se termine avec un message explicite :

```
*** stack smashing detected ***: terminated
Aborted (core dumped)
```

Le signal envoyé est `SIGABRT`, ce qui permet de générer un core dump pour l'analyse post-mortem (voir section 29.3). Le programme ne tente pas de récupérer — la pile étant corrompue, toute tentative de déroulement des exceptions ou de nettoyage serait elle-même non fiable.

## Limitations du stack protector

Le stack protector est une défense précieuse mais non absolue. Comprendre ses limites est essentiel pour ne pas en surestimer la protection.

### Seuls les stack overflows sont couverts

Le canary ne protège que la pile. Les heap buffer overflows, les use-after-free, les integer overflows et les format string vulnerabilities ne sont pas détectés par ce mécanisme. C'est pourquoi il s'inscrit dans une stratégie de défense en profondeur, combiné avec les autres protections de la section 45.4.

### Overflows non séquentiels

Le canary suppose qu'un buffer overflow écrase la mémoire de manière séquentielle (de l'adresse basse vers l'adresse haute). Un attaquant qui peut écrire à un offset arbitraire — par exemple via un write-what-where (écriture à une adresse choisie) — peut contourner le canary en écrivant directement sur l'adresse de retour sans toucher à la sentinelle :

```cpp
// Écriture indexée — le canary ne protège pas
void indexed_write(int* array, size_t index, int value) {
    array[index] = value;  // Si index est contrôlé par l'attaquant,
                           // il peut écrire n'importe où sur la pile
                           // sans traverser le canary
}
```

Ce type de vulnérabilité nécessite d'autres protections comme CFI (Control-Flow Integrity) ou les sanitizers.

### Fuite du canary (information leak)

Si un attaquant dispose d'une vulnérabilité de lecture hors bornes (information leak), il peut lire la valeur du canary depuis la pile, puis construire son payload en incluant la bonne valeur de canary à la bonne position. Le buffer overflow passe alors la vérification sans être détecté.

Ce scénario renforce l'importance de combiner le stack protector avec l'ASLR (section 45.4.3) — même si l'attaquant connaît le canary, il doit encore deviner les adresses du code cible.

### Fonctions non protégées

Selon la variante choisie, certaines fonctions ne reçoivent pas de canary. Avec `-fstack-protector-strong`, les fonctions sans tableaux ni prises d'adresse sont exclues. Un bug de corruption mémoire dans une telle fonction ne serait pas détecté par le canary — mais il est aussi structurellement moins probable.

## Configuration sur Ubuntu

Sur les versions récentes d'Ubuntu (22.04+), le stack protector est configuré comme suit par défaut :

- **GCC** : `-fstack-protector-strong` est activé par défaut via les specs de la distribution. Un programme compilé sans aucun flag de sécurité explicite bénéficie quand même de cette protection.  
- **Clang** : le comportement par défaut dépend de la version et de la configuration. Il est recommandé de toujours spécifier explicitement `-fstack-protector-strong`.

Pour vérifier le comportement par défaut de votre compilateur :

```bash
# Compiler un programme minimal sans aucun flag de sécurité
echo 'int main() { char buf[64]; buf[0]=1; }' > /tmp/test.cpp  
g++ /tmp/test.cpp -o /tmp/test -S -O2  
grep "stack_chk\|fs:0x28" /tmp/test.s  
# Si des lignes apparaissent → stack protector activé par défaut
```

Pour **désactiver** explicitement le stack protector (utile pour le benchmarking ou le code embarqué à très forte contrainte de performance) :

```bash
g++ -fno-stack-protector main.cpp -o main
```

> **Attention** : désactiver le stack protector en production doit être une décision documentée et justifiée par des mesures de performance concrètes. Le surcoût de `-fstack-protector-strong` est rarement mesurable dans des applications réelles.

## Intégration CMake

```cmake
# Recommandation : -fstack-protector-strong pour toutes les cibles
add_compile_options(-fstack-protector-strong)

# Pour des cibles spécifiques nécessitant -all (composants critiques)
target_compile_options(crypto_module PRIVATE -fstack-protector-all)

# Pour désactiver sur une cible de micro-benchmarking
target_compile_options(benchmark_hot_loop PRIVATE -fno-stack-protector)
```

Le flag `-fstack-clash-protection`, complémentaire au stack protector, protège contre une classe d'attaques différente (stack clash) où un attaquant effectue de grandes allocations sur la pile pour sauter par-dessus la page de garde. Il est recommandé de l'ajouter systématiquement :

```cmake
add_compile_options(
    -fstack-protector-strong
    -fstack-clash-protection
)
```

---

## Résumé

Le stack protector est une protection éprouvée, peu coûteuse et largement déployée. En résumé :

- **Utiliser `-fstack-protector-strong`** comme défaut de production — c'est le meilleur compromis couverture/coût.  
- **Le canary détecte les stack buffer overflows séquentiels** — pas les heap overflows, les écritures indexées, ni les use-after-free.  
- **Un canary corrompu termine immédiatement le programme** — c'est une détection, pas une prévention. Le overflow a eu lieu ; le canary empêche l'exploitation.  
- **Combiner avec les autres protections** — ASLR/PIE (section 45.4.3), `_FORTIFY_SOURCE` (section 45.4.2), NX, RELRO, et les sanitizers en phase de test.  
- **Vérifier l'activation** avec `checksec` ou en inspectant l'assembleur généré.
