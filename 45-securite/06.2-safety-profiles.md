# 45.6.2 — Safety Profiles : État de maturité et adoption en 2026 🔥

## Section 45.6 — Sécurité mémoire : Réponses concrètes du comité C++ en 2026 🔥
## Chapitre 45 — Sécurité en C++ ⭐

---

## Introduction

Les Safety Profiles sont la réponse choisie par le comité de standardisation C++ (WG21) pour améliorer la sécurité mémoire du langage. L'idée fondamentale est de définir des ensembles de règles nommés (les "profils") que le compilateur applique lorsque le développeur les active. Chaque profil restreint certaines constructions dangereuses du C++ et/ou injecte des vérifications à la compilation ou à l'exécution, offrant ainsi un sous-ensemble plus sûr du langage sans casser la compatibilité ascendante.

Le concept est porté par deux auteurs principaux — Bjarne Stroustrup (P3038, P3274, P3446, P3447) et Herb Sutter (P3081, P3436) — qui ont travaillé sur des propositions initialement distinctes avant de converger vers une direction commune. Les Profiles ont été priorisés par SG23 en novembre 2024 face à la proposition alternative Safe C++ (borrow checking à la Rust), et constituent depuis la voie officielle du comité pour la sécurité mémoire.

En mars 2026, les Profiles sont un sujet vivant, activement discuté, mais pas encore standardisé ni implémenté de façon complète dans un compilateur majeur. Cette section présente le mécanisme proposé, son état réel de maturité, les critiques qui lui sont adressées, et ce que les développeurs peuvent en attendre de façon réaliste.

---

## Le mécanisme proposé

### Qu'est-ce qu'un profil ?

Un profil est un ensemble nommé de règles que le compilateur applique lorsque le profil est activé. Chaque règle doit être décidable de façon déterministe à la compilation (même si elle peut injecter une vérification exécutée au runtime). Les règles sont portables et intégrées au compilateur, pas dans un outil externe comme un analyseur statique.

La proposition P3081 (Sutter) définit trois profils initiaux :

**Profil `bounds`** — vérifie que les accès aux conteneurs et aux tableaux restent dans les bornes. Ce profil injecte des vérifications de bornes sur les opérateurs d'indexation, les déréférencements de pointeurs avec offset, et les accès via itérateurs. C'est le profil le plus proche de ce que le hardening de la STL offre déjà (voir section 45.1) :

```cpp
// Avec le profil bounds activé :
void example(std::vector<int>& v) {
    v[10] = 42;  // Le compilateur injecte une vérification de bornes
                  // → trap si 10 >= v.size()
}
```

**Profil `type`** — renforce la sûreté de type en interdisant ou en vérifiant les conversions dangereuses. Ce profil rejette à la compilation les `reinterpret_cast`, les accès aux membres actifs de `union` non vérifiés, les conversions de pointeurs non sûres, et les arguments variadiques. Les conversions avec perte de données (narrowing conversions) sont également interdites par défaut :

```cpp
// Avec le profil type activé :
void example(void* p) {
    auto* ip = reinterpret_cast<int*>(p);
    // ❌ Rejeté par le profil type
    // Le développeur doit utiliser [[profiles::suppress(std::type)]]
    // pour opter explicitement en dehors du profil
}
```

**Profil `lifetime`** — le plus ambitieux et le plus controversé. Vise à détecter statiquement les dangling pointers, les use-after-free et les invalidations d'itérateurs. Ce profil s'appuie sur l'analyse statique de durée de vie décrite dans P1179 (Herb Sutter, 2019) et poursuivie dans P3465. Les vérifications incluent l'interdiction de la gestion manuelle de la durée de vie (`new`/`delete`/`free` directs), les vérifications de nullité des pointeurs avant déréférencement, et l'analyse des durées de vie des références retournées :

```cpp
// Avec le profil lifetime activé :
int* dangling() {
    int local = 42;
    return &local;
    // ❌ Rejeté par le profil lifetime
    // Retour d'un pointeur vers une variable locale
}

void use_after_free() {
    auto p = std::make_unique<int>(42);
    int* raw = p.get();
    p.reset();
    *raw = 10;
    // ❌ Détecté comme use-after-free par le profil lifetime
    // (dans les cas simples où le flux est analysable)
}
```

### Opt-in et opt-out

Le mécanisme est conçu autour de deux principes :

**Activation explicite (opt-in).** Les profils ne sont pas activés par défaut. Le développeur choisit de les activer par unité de traduction, via un attribut dans le source ou un switch compilateur :

```cpp
// Activation dans le source
[[profiles::enable(std::bounds, std::type)]]
// Tout le fichier est soumis aux profils bounds et type

// Ou via la ligne de compilation (syntaxe indicative, non finalisée)
// g++ -std=c++26 -fprofile=bounds,type main.cpp
```

**Suppression granulaire (opt-out).** Quand un profil est actif, le développeur peut le désactiver localement avec `[[profiles::suppress]]` :

```cpp
[[profiles::enable(std::type)]]

void safe_code() {
    // Tout le code ici est soumis au profil type
}

void interop_with_c_library() {
    [[profiles::suppress(std::type)]] {
        // Ce bloc est exempté du profil type
        // Nécessaire pour interfacer avec du code C legacy
        auto* p = reinterpret_cast<MyStruct*>(raw_buffer);
        process(p);
    }
    // Le profil type est réactivé ici
}
```

La suppression est granulaire par profil : supprimer le profil `type` ne désactive pas le profil `bounds`. Chaque `[[profiles::suppress]]` est un "trust me" explicite et auditable, comparable en intention au `unsafe` de Rust — bien que les garanties formelles soient différentes.

### Principes de conception

La proposition P3081 pose plusieurs contraintes de conception qui expliquent à la fois les forces et les limites des Profiles :

**Rétrocompatibilité.** Le code C++ existant doit continuer à compiler et à fonctionner sans changement. Les Profiles sont purement additifs — ils ne modifient pas la sémantique du code qui ne les active pas.

**Décidabilité à la compilation.** Chaque règle doit être décidable par le compilateur sans analyse inter-procédurale complexe ni impact inacceptable sur les temps de compilation. Cela exclut certaines analyses qui nécessiteraient des annotations supplémentaires dans le code source.

**Pas de modification du modèle objet.** Les Profiles ne modifient pas la représentation mémoire des objets, l'ABI, ni le modèle de durée de vie du C++. Ils opèrent comme une couche de vérification au-dessus du langage existant.

**Implémentation in-the-box.** Les règles doivent être implémentables dans le compilateur lui-même, pas dans un outil externe. Cela garantit que l'activation d'un profil produit les mêmes résultats quel que soit le compilateur conforme utilisé.

## Ce qui existe déjà comme prior art

Les Profiles ne partent pas de zéro. Plusieurs des règles proposées sont déjà implémentées sous des formes différentes :

### Profil bounds

Le hardening de la STL, déjà déployé en production, implémente l'essentiel du profil `bounds` :

| Implémentation | Mécanisme | Déploiement |
|---|---|---|
| libc++ hardened mode (Clang) | Vérifications de bornes sur `operator[]`, itérateurs, `span`, `string_view` | Production chez Google, Apple |
| libstdc++ assertions (GCC) | `_GLIBCXX_ASSERTIONS` : vérifications de préconditions | Large adoption |
| `-D_FORTIFY_SOURCE=2` | Vérifications de bornes sur fonctions C (section 45.4.2) | Standard industrie |

Le profil `bounds` standardiserait ces vérifications et les rendrait portables et activables de façon uniforme via un mécanisme de langage plutôt que via des macros spécifiques à chaque implémentation.

### Profil type

De nombreuses règles du profil `type` correspondent à des warnings ou des checks existants :

| Règle | Outil existant |
|---|---|
| Interdiction de `reinterpret_cast` | `clang-tidy`: `cppcoreguidelines-pro-type-reinterpret-cast` |
| Interdiction des unions non sûres | `clang-tidy`: `cppcoreguidelines-pro-type-union-access` |
| Interdiction des varargs | `clang-tidy`: `cppcoreguidelines-pro-type-vararg` |
| Conversions narrowing | `-Wconversion`, `-Wnarrowing` |
| Downcasts non vérifiés | `clang-tidy`: `cppcoreguidelines-pro-type-static-cast-downcast` |

La différence est que ces checks sont aujourd'hui des avertissements d'un outil externe (clang-tidy), que le développeur peut ignorer. Le profil `type` les transformerait en erreurs de compilation dans un mode standardisé.

### Profil lifetime

Le profil `lifetime` est celui qui possède le moins de prior art déployé à grande échelle. L'analyse de durée de vie P1179 (Sutter, 2019) a été partiellement implémentée dans un prototype Microsoft (intégré dans MSVC sous forme expérimentale) et dans des outils clang, mais elle n'a jamais atteint le niveau de maturité des vérifications de bornes ou de type. Les analyses de dangling existantes dans les compilateurs (`-Wdangling`, `-Wreturn-stack-address`, `-Wreturn-local-addr`) ne couvrent qu'une fraction des cas.

## État réel en mars 2026

### Ce qui est standardisé

**Rien.** Les Profiles ne font pas partie de C++26. Lors de la réunion de Kona (novembre 2025), le comité a nommé des éditeurs pour un whitepaper sur les Profiles, mais n'a pas intégré les Profiles dans le standard. Le whitepaper est attendu en 2026, et une intégration dans le standard est envisagée pour C++29 au plus tôt.

Bjarne Stroustrup a exprimé publiquement sa déception, déclarant que le comité s'est trouvé dans la confusion et n'a pas garanti l'intégration des Profiles dans C++26. Herb Sutter a proposé que C++29 se concentre sur le hardening et la réduction des comportements indéfinis, ce qui inclurait potentiellement les Profiles.

### Ce qui est implémenté

En mars 2026, aucun compilateur majeur ne propose une implémentation complète des Safety Profiles tels que décrits dans P3081 :

- **GCC 15** — pas de support des Profiles. Les vérifications de bornes reposent sur `_GLIBCXX_ASSERTIONS` et les sanitizers.  
- **Clang 20** — pas de support des Profiles en tant que tels. Le hardening libc++ (`_LIBCPP_HARDENING_MODE`) couvre le profil `bounds` de façon non standardisée. L'analyse de durée de vie (`-Wdangling`, `-Wlifetime`) existe sous forme de warnings mais ne constitue pas un profil au sens de P3081.  
- **MSVC (VS 2026)** — un prototype d'analyse de durée de vie basé sur P1179 existe depuis plusieurs années en mode expérimental, mais il n'a pas été promu en fonctionnalité de production et ne suit pas la syntaxe proposée dans P3081.

En résumé : les *composants techniques* qui alimenteraient les Profiles existent sous diverses formes (hardening STL, warnings de dangling, checks clang-tidy), mais le *mécanisme de Profiles lui-même* — avec sa syntaxe d'opt-in/opt-out, ses attributs standardisés et sa sémantique portable — n'existe dans aucun compilateur.

### Ce qui est attendu

Le calendrier réaliste, basé sur l'historique des fonctionnalités C++ comparables :

| Étape | Date estimée |
|---|---|
| Publication du whitepaper Profiles | 2026 |
| Discussion et maturation au comité | 2026–2028 |
| Intégration dans le standard (C++29 ?) | 2029 |
| Implémentation dans les compilateurs majeurs | 2030–2031 |
| Adoption significative dans l'industrie | 2032–2033 |

Ce calendrier n'est pas pessimiste — il reflète le rythme historique de l'écosystème C++. Les modules (C++20) ne sont toujours pas pleinement utilisables en production cinq ans après leur standardisation. Les concepts (C++20) ont mis trois ans à atteindre un support compilateur raisonnable. Les Profiles, qui sont plus complexes que l'un ou l'autre, suivront vraisemblablement un calendrier comparable.

## Les critiques

Les Profiles font l'objet de critiques substantielles, tant au sein du comité qu'à l'extérieur. Comprendre ces critiques est essentiel pour évaluer de façon réaliste ce que les Profiles peuvent et ne peuvent pas offrir.

### L'absence d'implémentation

La critique la plus immédiate est factuelle : les Profiles n'ont pas d'implémentation de référence complète. Les règles de P3081 sont décrites dans un document de spécification, mais elles n'ont pas été testées en tant que système cohérent sur une base de code réelle de grande taille. Les auteurs de la proposition affirment que toutes les règles individuelles ont été implémentées et que la plupart ont été utilisées en code de production, mais cela se réfère aux implémentations fragmentées existantes (clang-tidy checks, hardening STL, warnings de compilateur) — pas au système de Profiles intégré tel que proposé.

Cette absence d'implémentation rend difficile l'évaluation du taux de faux positifs, du coût en temps de compilation, de l'impact sur les bases de code existantes, et de la couverture réelle des vulnérabilités de sécurité mémoire.

### Le problème fondamental du profil lifetime

Sean Baxter, auteur de Safe C++ et du compilateur Circle, a publié une analyse détaillée ("Why Safety Profiles Failed") argumentant que les Profiles ne peuvent pas atteindre la sécurité mémoire pour le profil `lifetime`, en raison de trois informations manquantes dans les déclarations de fonctions C++ :

**L'information d'aliasing.** Le compilateur C++ ne sait pas si deux pointeurs ou références passés à une fonction peuvent pointer vers le même objet. Sans cette information, l'analyse de durée de vie produit soit des faux négatifs (bugs manqués) soit des faux positifs massifs (code correct rejeté).

**L'information de durée de vie.** Quand une fonction retourne une référence, le compilateur ne sait pas quelle référence paramètre contraint la durée de vie du retour. C'est exactement le problème que les lifetime parameters de Rust résolvent avec des annotations explicites (`'a`, `'b`). Sans annotations équivalentes, l'analyse doit appliquer des heuristiques qui sont soit trop conservatrices (faux positifs) soit trop permissives (faux négatifs).

```cpp
// Problème : quelle référence paramètre contraint le retour ?
const std::string& choose(const std::string& a, const std::string& b);
// En Rust : fn choose<'a>(a: &'a str, b: &'a str) -> &'a str
// En C++ : aucune information — le compilateur ne peut que deviner
```

**L'information de sûreté (safeness).** Sans mécanisme pour distinguer les fonctions sûres des fonctions qui ont des préconditions de validité non vérifiables, le profil `lifetime` ne peut pas propager les garanties de sécurité à travers les appels de fonctions. Le `unsafe` de Rust sert précisément à cela : il marque les frontières où les garanties statiques cessent de s'appliquer. L'attribut `[[profiles::suppress]]` des Profiles ressemble à `unsafe`, mais sans la contrepartie `safe` sur les fonctions, l'analyse ne peut pas raisonner sur la composition de fonctions.

### La critique des analystes statiques

Corentin Jabot, membre actif du comité et critique des Profiles, a souligné que les Profiles sont "un ensemble de sanitizers et de checks d'analyse statique" dont la nature exacte reste peu claire. Il note que la communauté d'analyse statique travaille depuis des décennies et a une compréhension bien établie des limites de l'analyse statique — en particulier qu'on ne peut pas garantir la sécurité mémoire par analyse statique sans annoter la plupart des références.

Cette critique touche un point fondamental : les Profiles promettent-ils une *réduction* des vulnérabilités (ce qui est atteignable) ou une *élimination* des vulnérabilités (ce qui nécessite des annotations de type borrow checker) ? La réponse influence directement l'évaluation de leur adéquation face aux exigences réglementaires.

### La défense des auteurs

Les partisans des Profiles répondent à ces critiques par plusieurs arguments :

- **L'objectif est la réduction, pas la perfection.** Les Profiles ne prétendent pas atteindre la sécurité mémoire de Rust. Leur objectif est de réduire significativement la surface de vulnérabilités, ce qui est plus réaliste que d'attendre une solution parfaite qui pourrait ne jamais arriver.

- **L'approche incrémentale est le seul chemin viable.** Avec des centaines de milliards de lignes de C++ en production, une approche qui nécessite des annotations massives ou une réécriture du code existant n'est pas réaliste. Les Profiles permettent d'améliorer la sécurité du code existant par simple recompilation dans de nombreux cas.

- **Le prior art fonctionne.** Le hardening de libc++ — qui implémente essentiellement le profil `bounds` — est déployé à grande échelle avec un impact mesurable. Le profil `type` étend des vérifications qui ont prouvé leur valeur via clang-tidy. L'approche incrémentale a déjà produit des résultats concrets.

- **Le profil `lifetime` est un début.** Les auteurs reconnaissent que le profil `lifetime` est le plus difficile et le moins mature. La direction retenue (P3465) est de poursuivre l'analyse de durée de vie comme un Technical Specification (TS) ou un whitepaper, à affiner au fur et à mesure que l'expérience d'implémentation s'accumule, avant de l'intégrer dans le standard.

## Ce que les développeurs peuvent faire aujourd'hui

En attendant la standardisation des Profiles, les développeurs C++ peuvent adopter dès aujourd'hui les pratiques qui s'alignent avec les objectifs des Profiles et qui en constitueront le socle technique :

### "Profil bounds" de facto

Activer le hardening de la STL revient à activer un profil `bounds` non standardisé mais fonctionnel :

```bash
# Clang/libc++ — mode fast (recommandé en production)
clang++ -D_LIBCPP_HARDENING_MODE=_LIBCPP_HARDENING_MODE_FAST ...

# GCC/libstdc++ — assertions
g++ -D_GLIBCXX_ASSERTIONS ...
```

Compléter avec `_FORTIFY_SOURCE=2` (section 45.4.2) pour les appels aux fonctions C.

### "Profil type" de facto

Configurer `clang-tidy` avec les checks du C++ Core Guidelines qui correspondent au profil `type` :

```yaml
# .clang-tidy — approximation du profil type
Checks: >
  cppcoreguidelines-pro-type-reinterpret-cast,
  cppcoreguidelines-pro-type-union-access,
  cppcoreguidelines-pro-type-vararg,
  cppcoreguidelines-pro-type-static-cast-downcast,
  cppcoreguidelines-pro-type-cstyle-cast,
  cppcoreguidelines-pro-type-const-cast,
  cppcoreguidelines-narrowing-conversions,
  bugprone-narrowing-conversions
WarningsAsErrors: >
  cppcoreguidelines-pro-type-reinterpret-cast,
  cppcoreguidelines-pro-type-cstyle-cast
```

Intégrer `clang-tidy` dans le pipeline CI (chapitre 32) et dans les pre-commit hooks (section 47.2) pour que ces vérifications soient appliquées systématiquement.

### "Profil lifetime" de facto (partiel)

Les outils actuels couvrent une fraction du profil `lifetime` :

```bash
# Warnings de dangling du compilateur
clang++ -Wdangling -Wreturn-stack-address -Wdangling-gsl ...  
g++ -Wreturn-local-addr -Wdangling-reference ...  

# clang-tidy checks
# bugprone-dangling-handle
# bugprone-use-after-move
# clang-analyzer-core.StackAddressEscape
```

Compléter impérativement avec ASan en phase de test (section 29.4.1) — c'est l'outil le plus efficace pour détecter les use-after-free que l'analyse statique ne peut pas couvrir.

### Résumé de la stratégie "Profiles avant les Profiles"

| Profil visé | Outil immédiat | Couverture |
|---|---|---|
| `bounds` | Hardened STL + `_FORTIFY_SOURCE=2` | Élevée — couvre la majorité des accès hors bornes |
| `type` | `clang-tidy` (Core Guidelines checks) | Bonne — couvre les conversions dangereuses |
| `lifetime` | `-Wdangling` + ASan + `clang-tidy` (dangling checks) | Partielle — les cas simples en statique, le reste en dynamique |

Cette combinaison ne fournit pas les garanties formelles que des Profiles standardisés offriraient, mais elle produit un niveau de protection mesurable et actionnable dès aujourd'hui. Et lorsque les Profiles seront disponibles dans les compilateurs, la migration depuis ces pratiques sera naturelle — les développeurs qui les appliquent déjà seront les premiers à pouvoir activer les Profiles sur leur code sans avalanche de diagnostics.

## Résumé

Les Safety Profiles représentent la direction choisie par le comité C++ pour la sécurité mémoire. Leur évaluation honnête en mars 2026 :

**Ce qui est acquis :**  
- La direction est fixée — le comité poursuit les Profiles, pas le borrow checking.  
- Les profils `bounds` et `type` s'appuient sur du prior art déployé et éprouvé.  
- Le standard library hardening (P3471), qui implémente l'essentiel du profil `bounds`, est intégré à C++26.

**Ce qui reste incertain :**  
- Les Profiles ne sont pas dans C++26 — un whitepaper est en préparation, avec un calendrier d'intégration en C++29 au plus optimiste.  
- Le profil `lifetime` pose des défis fondamentaux d'analyse statique sans annotations de durée de vie. Son niveau de couverture réel est inconnu.  
- Aucun compilateur majeur ne propose d'implémentation complète du mécanisme de Profiles.

**Ce qui est actionnable :**  
- Les outils qui sous-tendent les futurs Profiles sont disponibles dès aujourd'hui : hardening STL, clang-tidy, warnings de dangling, sanitizers.  
- Les adopter maintenant prépare la transition vers les Profiles tout en produisant un bénéfice de sécurité immédiat.

La question fondamentale — les Profiles offriront-ils une réponse suffisante à la pression réglementaire et industrielle pour la sécurité mémoire ? — reste ouverte. La section 45.6.5 propose un bilan global intégrant les Profiles, les outils existants, et l'interopérabilité Rust.
