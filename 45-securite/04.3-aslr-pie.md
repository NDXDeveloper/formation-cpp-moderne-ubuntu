# 45.4.3 — ASLR et PIE

## Section 45.4 — Compilation avec protections
## Chapitre 45 — Sécurité en C++ ⭐

---

## Introduction

Les sections précédentes ont couvert des protections qui détectent la corruption au moment où elle se produit : le stack canary (45.4.1) détecte l'écrasement de l'adresse de retour, `_FORTIFY_SOURCE` (45.4.2) détecte les copies hors bornes. Mais ces mécanismes ne couvrent pas tous les scénarios d'exploitation. Lorsqu'un attaquant parvient à rediriger le flux d'exécution — par un use-after-free, un overflow non détecté ou une corruption de pointeur de fonction — il a encore besoin de savoir *où* rediriger ce flux.

C'est précisément ce que l'ASLR et le PIE rendent difficile. Plutôt que de détecter la corruption, ils en réduisent l'exploitabilité en rendant imprévisible la disposition du programme en mémoire. Un attaquant qui peut écraser une adresse de retour doit encore deviner l'adresse du code qu'il veut atteindre — et sur un système 64 bits correctement configuré, cette adresse change à chaque exécution parmi des milliards de possibilités.

Ces deux mécanismes — l'un fourni par le système d'exploitation, l'autre par le compilateur — fonctionnent de concert. Sans PIE, l'ASLR ne protège que partiellement le programme. Sans ASLR, le PIE n'a aucun effet de sécurité. Comprendre leur interaction est essentiel pour évaluer la posture de sécurité réelle d'un binaire.

---

## ASLR : Address Space Layout Randomization

### Principe

L'ASLR est un mécanisme du noyau Linux qui randomise les adresses de base des différentes régions de l'espace d'adressage d'un processus à chaque exécution. Sur un système sans ASLR, chaque région (code, heap, stack, bibliothèques partagées) est chargée à la même adresse à chaque lancement — un attaquant peut déterminer ces adresses une fois pour toutes et les réutiliser dans tous ses exploits.

Avec l'ASLR activé, la disposition mémoire change à chaque exécution :

```
Exécution 1                          Exécution 2
┌──────────────────────┐             ┌──────────────────────┐
│                      │             │                      │
│  Stack               │ 0x7ffd3...  │  Stack               │ 0x7fff1...
│                      │             │                      │
├──────────────────────┤             ├──────────────────────┤
│                      │             │                      │
│  Libs (.so)          │ 0x7f2a8...  │  Libs (.so)          │ 0x7f4c2...
│  (libc, libstdc++)   │             │  (libc, libstdc++)   │
│                      │             │                      │
├──────────────────────┤             ├──────────────────────┤
│                      │             │                      │
│  Heap                │ 0x5612f...  │  Heap                │ 0x55d3a...
│                      │             │                      │
├──────────────────────┤             ├──────────────────────┤
│  Exécutable (.text)  │ 0x5610c...  │  Exécutable (.text)  │ 0x55d18...
│  (si PIE activé)     │             │  (si PIE activé)     │
└──────────────────────┘             └──────────────────────┘
```

Un attaquant qui cible `system()` dans la libc, par exemple, ne peut plus coder en dur l'adresse `0x7ffff7a52390` — cette adresse sera différente à chaque lancement.

### Régions randomisées par l'ASLR

| Région | Randomisée par ASLR | Condition |
|---|---|---|
| Stack | Oui | Toujours (si ASLR activé) |
| Heap (brk/mmap) | Oui | Toujours (si ASLR activé) |
| Bibliothèques partagées (.so) | Oui | Toujours — les .so sont PIC par construction |
| Exécutable principal (.text) | **Seulement si PIE** | Sans PIE, l'exécutable est à adresse fixe |
| vDSO | Oui | Toujours |

La ligne critique est celle de l'exécutable principal. Sans PIE, le segment `.text` (le code du programme) est chargé à une adresse fixe déterminée à la compilation (typiquement `0x400000` sur x86_64). Cette adresse fixe fournit à l'attaquant un point d'appui fiable pour ses attaques — les gadgets ROP (Return-Oriented Programming) dans le code principal du programme sont à des adresses prévisibles, même avec l'ASLR actif sur tout le reste.

### Entropie de l'ASLR sur x86_64

L'entropie mesure le nombre de bits aléatoires dans l'adresse de base. Plus l'entropie est élevée, plus il est difficile pour un attaquant de deviner l'adresse cible :

| Région | Entropie typique (Linux x86_64) | Positions possibles |
|---|---|---|
| Stack | 30 bits | ~1 milliard |
| Bibliothèques (.so) | 28-30 bits | ~268 millions – 1 milliard |
| Heap (mmap) | 28 bits | ~268 millions |
| Exécutable (PIE) | 28 bits | ~268 millions |

Sur un système 32 bits, l'entropie est nettement plus faible (8-16 bits selon la région), ce qui rend le brute-force réaliste. C'est une raison supplémentaire pour compiler en 64 bits chaque fois que possible.

### Vérifier l'état de l'ASLR sur Ubuntu

L'ASLR est contrôlé par un paramètre du noyau :

```bash
# Lire le niveau d'ASLR actuel
cat /proc/sys/kernel/randomize_va_space
```

| Valeur | Signification |
|---|---|
| 0 | ASLR désactivé |
| 1 | ASLR partiel (stack, libs, mmap randomisés ; heap non randomisé) |
| 2 | ASLR complet (toutes les régions randomisées) |

Ubuntu active le niveau 2 (complet) par défaut. Pour vérifier que c'est bien le cas sur votre système :

```bash
$ cat /proc/sys/kernel/randomize_va_space
2
```

On peut observer l'ASLR en action en examinant l'adresse de la stack ou d'une bibliothèque sur deux exécutions successives :

```bash
# Observer la randomisation de la stack
$ for i in 1 2 3; do
    cat /proc/self/maps | grep '\[stack\]'
  done
7ffc9a21e000-7ffc9a23f000 rw-p 00000000 00:00 0  [stack]
7ffcb8f5a000-7ffcb8f7b000 rw-p 00000000 00:00 0  [stack]
7ffd22c10000-7ffd22c31000 rw-p 00000000 00:00 0  [stack]
```

Les adresses de la stack changent à chaque invocation.

> **Important** : l'ASLR ne doit jamais être désactivé en production. Certains guides de débogage recommandent `echo 0 > /proc/sys/kernel/randomize_va_space` pour rendre les adresses reproductibles pendant le développement — c'est acceptable sur une machine de développement, mais cette modification doit être impérativement révertée avant tout déploiement. En CI/CD, ne jamais désactiver l'ASLR dans les environnements de staging ou de production.

## PIE : Position-Independent Executable

### Pourquoi PIE est indispensable

Comme vu ci-dessus, l'ASLR randomise les adresses de toutes les régions mémoire — sauf l'exécutable principal si celui-ci n'est pas compilé en PIE. Un binaire non-PIE est chargé à une adresse fixe (typiquement `0x400000` sur x86_64), ce qui laisse une faille majeure dans la randomisation :

```
Sans PIE :                           Avec PIE :
┌───────────────────────┐            ┌───────────────────────┐
│ Stack     0x7ffc...   │ Aléatoire  │ Stack     0x7ffd...   │ Aléatoire
│ Libs      0x7f2a...   │ Aléatoire  │ Libs      0x7f4c...   │ Aléatoire
│ Heap      0x1a3f...   │ Aléatoire  │ Heap      0x563b...   │ Aléatoire
│ .text     0x400000    │ ⚠️ FIXE    │ .text     0x55d1...   │ ✅ Aléatoire
└───────────────────────┘            └───────────────────────┘
```

Sans PIE, un attaquant dispose de l'intégralité du code du programme à des adresses connues. Il peut y trouver des gadgets ROP (séquences d'instructions terminées par `ret`) pour construire une chaîne d'attaque sans jamais avoir besoin de deviner une adresse. Le PIE ferme cette brèche en rendant le code du programme lui-même randomisable par l'ASLR.

### Mécanisme technique

PIE transforme l'exécutable en un objet partagé (shared object). Au lieu de générer du code avec des adresses absolues, le compilateur génère du code position-independent — toutes les références aux données globales et aux fonctions passent par des indirections relatives au compteur de programme (PC-relative addressing sur x86_64) ou via la GOT (Global Offset Table).

```bash
# Compiler avec PIE (recommandé)
g++ -fPIE -pie main.cpp -o main_pie

# Compiler sans PIE (non recommandé)
g++ -fno-pie -no-pie main.cpp -o main_no_pie
```

Le flag `-fPIE` contrôle la génération de code (compilation), et `-pie` contrôle l'édition de liens (linking). Les deux sont nécessaires.

### Vérifier si un binaire est PIE

```bash
# Méthode 1 : readelf
$ readelf -h ./main_pie | grep Type
  Type:    DYN (Position-Independent Executable)

$ readelf -h ./main_no_pie | grep Type
  Type:    EXEC (Executable file)
```

Un binaire PIE apparaît comme type `DYN` (Dynamic/Shared Object) car, du point de vue du format ELF, un exécutable position-independent est structurellement identique à une bibliothèque partagée — le noyau le charge avec le même mécanisme.

```bash
# Méthode 2 : file
$ file ./main_pie
main_pie: ELF 64-bit LSB pie executable, x86-64, ...

$ file ./main_no_pie
main_no_pie: ELF 64-bit LSB executable, x86-64, ...
```

```bash
# Méthode 3 : checksec
$ checksec --file=./main_pie
... PIE enabled ...

$ checksec --file=./main_no_pie
... No PIE ...
```

### PIE et PIC : quelle différence ?

Les deux concepts sont proches mais s'appliquent à des contextes différents :

| Aspect | PIC (`-fPIC`) | PIE (`-fPIE`) |
|---|---|---|
| Cible | Bibliothèques partagées (.so) | Exécutables |
| Obligatoire ? | Oui pour les .so | Non, mais fortement recommandé |
| Code généré | Position-independent complet | Position-independent optimisé pour les exécutables |
| Indirections | Toutes les références globales via GOT | Optimisations possibles car le linker connaît le scope |

`-fPIE` génère un code légèrement plus efficace que `-fPIC` car le linker sait que les symboles de l'exécutable ne seront pas interposés par une bibliothèque partagée — il peut donc utiliser des accès directs relatifs au PC dans certains cas, là où `-fPIC` doit systématiquement passer par la GOT.

En pratique, la différence de performance entre `-fPIE` et un code non-PIE est inférieure à 1 % sur x86_64, car l'architecture supporte nativement l'adressage relatif au PC sur 64 bits. Sur les architectures 32 bits (x86), le surcoût est légèrement plus élevé (1-3 %) car le registre d'instruction pointer n'est pas directement accessible et un registre général doit être sacrifié pour les calculs d'adresses relatives.

### État par défaut sur Ubuntu

L'état par défaut du PIE a évolué au fil des versions de GCC et des distributions :

- **GCC 6+** (Ubuntu 17.04+) : PIE est activé par défaut via les specs de la distribution. Un `g++ main.cpp -o main` produit un binaire PIE sans flag explicite.  
- **Clang** : le comportement par défaut suit la configuration de la distribution. Sur Ubuntu récent, PIE est généralement activé par défaut.

Pour vérifier le comportement par défaut de votre compilateur :

```bash
# Compiler sans aucun flag de sécurité
echo 'int main() {}' | g++ -x c++ - -o /tmp/test_pie  
readelf -h /tmp/test_pie | grep Type  
# Type: DYN → PIE activé par défaut
# Type: EXEC → PIE non activé par défaut
```

> **Même si PIE est activé par défaut sur votre distribution, spécifiez-le explicitement dans vos CMakeLists.txt.** Cela garantit la portabilité vers des environnements où le défaut est différent (conteneurs Docker avec une toolchain minimale, cross-compilation, distributions plus anciennes).

## Interaction ASLR + PIE : le tableau complet

La protection effective dépend de la combinaison des deux mécanismes :

| ASLR système | Binaire PIE | Stack | Heap | Libs | Code exécutable | Protection |
|---|---|---|---|---|---|---|
| Activé (2) | Oui | Aléatoire | Aléatoire | Aléatoire | Aléatoire | **Maximale** |
| Activé (2) | Non | Aléatoire | Aléatoire | Aléatoire | **Fixe** | Partielle — gadgets ROP disponibles dans .text |
| Désactivé (0) | Oui | Fixe | Fixe | Fixe | Fixe | **Aucune** — PIE sans ASLR est inutile |
| Désactivé (0) | Non | Fixe | Fixe | Fixe | Fixe | **Aucune** |

La seule configuration sûre est la première ligne : ASLR complet activé au niveau du système ET binaire compilé en PIE. Toute autre combinaison laisse des adresses prévisibles qu'un attaquant peut exploiter.

## Techniques de contournement et contre-mesures

L'ASLR et le PIE ne sont pas des protections absolues. Ils rendent l'exploitation plus difficile, mais des techniques de contournement existent. Les connaître permet de comprendre pourquoi la défense en profondeur est nécessaire.

### Information leaks

La principale menace contre l'ASLR est la fuite d'adresses. Si un attaquant peut lire une adresse mémoire via une vulnérabilité de lecture hors bornes, un format string (`%p`), ou un message d'erreur trop verbeux, il peut en déduire la base de la région correspondante et calculer toutes les autres adresses dans cette région :

```cpp
// ❌ Fuite d'adresse via un message d'erreur
void log_error(const void* ptr) {
    // L'adresse du pointeur révèle la disposition mémoire
    std::print(stderr, "Erreur à l'adresse : {}\n", ptr);
}
```

Une seule adresse fuitée d'une région suffit à annuler l'ASLR pour cette région entière — car tous les offsets à l'intérieur d'une région sont fixes (seule la base est randomisée).

**Contre-mesures** :

- Ne jamais afficher d'adresses mémoire dans les logs de production. Le noyau Linux masque les adresses dans `/proc/self/maps` pour les utilisateurs non privilégiés et affiche `(null)` à la place des adresses dans les messages `%p` du noyau (via `%pK`).  
- Minimiser les messages d'erreur exposés à l'utilisateur — un code d'erreur est suffisant ; les détails internes doivent rester dans les logs internes.  
- Utiliser les sanitizers et le fuzzing (section 45.5) pour trouver les vulnérabilités de lecture hors bornes qui pourraient servir de leak.

### Attaques par brute-force

Sur les systèmes 32 bits, l'entropie de l'ASLR est suffisamment faible (8-16 bits) pour qu'un brute-force soit faisable — quelques milliers à quelques dizaines de milliers de tentatives. Sur les serveurs qui relancent automatiquement le processus après un crash (comme un serveur web derrière un superviseur), un attaquant peut tenter des milliers d'adresses à la seconde.

**Contre-mesures** :

- Compiler en 64 bits — l'entropie passe à 28-30 bits, rendant le brute-force impraticable.  
- Limiter le nombre de redémarrages automatiques et alerter sur les crashes répétés.  
- Utiliser `ulimit -c 0` en production pour ne pas générer de core dumps accessibles (qui contiennent la disposition mémoire).

### Partial overwrite

Sur les architectures little-endian, un overflow d'un seul octet peut modifier l'octet de poids faible d'une adresse de retour sans toucher aux octets randomisés de poids fort. Si l'entropie des octets de poids faible est nulle (les adresses sont alignées sur des pages de 4 Ko), l'attaquant peut rediriger le flux d'exécution vers un offset différent dans la même page avec 1/16 de probabilité.

**Contre-mesure** : cette technique a un taux de succès faible et nécessite souvent d'être combinée avec un information leak. La meilleure défense reste la prévention de l'overflow initial (section 45.1) et le stack canary (section 45.4.1).

## Intégration CMake

```cmake
# Activer PIE pour toutes les cibles du projet
set(CMAKE_POSITION_INDEPENDENT_CODE ON)

# Équivalent plus explicite pour les exécutables
# (CMAKE_POSITION_INDEPENDENT_CODE s'applique aussi aux bibliothèques)
add_executable(my_app main.cpp)  
target_compile_options(my_app PRIVATE -fPIE)  
target_link_options(my_app PRIVATE -pie)  
```

`CMAKE_POSITION_INDEPENDENT_CODE ON` est la méthode idiomatique en CMake. Elle ajoute automatiquement `-fPIE` pour les exécutables et `-fPIC` pour les bibliothèques, ce qui est le comportement souhaité.

Pour vérifier après le build dans le pipeline CI/CD :

```cmake
# Post-build : vérifier que le binaire est PIE
add_custom_command(TARGET my_app POST_BUILD
    COMMAND ${CMAKE_COMMAND} -E echo "Vérification PIE..."
    COMMAND readelf -h $<TARGET_FILE:my_app> | grep -q "DYN"
            || (echo "ERREUR: binaire non-PIE" && exit 1)
    COMMENT "Vérification que le binaire est position-independent"
)
```

## Résumé

L'ASLR et le PIE sont la dernière ligne de défense — celle qui rend l'exploitation difficile même lorsque toutes les autres protections ont été contournées :

- **Activer PIE** (`-fPIE -pie` ou `CMAKE_POSITION_INDEPENDENT_CODE ON`) pour que l'exécutable bénéficie pleinement de l'ASLR. Sur Ubuntu récent, c'est le défaut, mais il faut le spécifier explicitement pour la portabilité.  
- **Vérifier que l'ASLR est au niveau 2** (`/proc/sys/kernel/randomize_va_space`) sur tous les systèmes de production. Ne jamais le désactiver en dehors du développement local.  
- **PIE sans ASLR est inutile, ASLR sans PIE est incomplet.** Les deux mécanismes sont indissociables.  
- **Le surcoût est négligeable** — inférieur à 1 % sur x86_64 pour le PIE, nul pour l'ASLR (géré par le noyau au chargement).  
- **L'ASLR n'est pas une protection absolue.** Les information leaks annulent la randomisation. C'est pourquoi l'ASLR s'inscrit dans une stratégie de défense en profondeur avec le stack canary (45.4.1), `_FORTIFY_SOURCE` (45.4.2), NX, RELRO, CFI, et les sanitizers (29.4).  
- **Compiler en 64 bits** chaque fois que possible pour maximiser l'entropie de l'ASLR (28-30 bits contre 8-16 bits en 32 bits).
