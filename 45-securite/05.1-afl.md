## Introduction

AFL++ (American Fuzzy Lop ++) est le fuzzer de référence de l'écosystème open source. Fork communautaire de l'AFL original de Michał Zalewski, il est activement maintenu et a considérablement étendu les capacités de son prédécesseur : nouveaux mutateurs, support de multiples backends d'instrumentation, stratégies de planification adaptatives, et intégration avec les sanitizers.

AFL++ est un fuzzer **externe** : il orchestre l'exécution du programme cible comme un processus séparé, lui fournit des entrées via stdin ou un fichier, et observe le résultat (crash, timeout, nouvelle couverture). Ce modèle le rend polyvalent — il peut fuzzer un outil en ligne de commande, un serveur réseau, ou même un binaire sans code source via QEMU.

Cette section couvre l'installation d'AFL++ sur Ubuntu, l'instrumentation du code C++, la configuration d'une campagne de fuzzing, le fuzzing parallèle, et l'intégration dans un workflow professionnel.

## Installation sur Ubuntu

### Depuis les paquets (rapide)

Ubuntu propose AFL++ dans ses dépôts :

```bash
sudo apt update  
sudo apt install afl++  
```

Cette méthode installe une version stable mais parfois en retard de quelques versions. Pour un projet de sécurité, la compilation depuis les sources est préférable.

### Depuis les sources (recommandé)

```bash
# Dépendances de build
sudo apt install -y build-essential python3-dev automake cmake git flex bison \
    libglib2.0-dev libpixman-1-dev python3-setuptools cargo libgtk-3-dev \
    lld llvm llvm-dev clang

# Cloner et compiler
git clone https://github.com/AFLplusplus/AFLplusplus  
cd AFLplusplus  
make distrib  # Compile AFL++ + QEMU mode + Unicorn mode + FRIDA mode  
sudo make install  
```

`make distrib` compile l'ensemble des modes d'instrumentation. Si seul le mode compile-time est nécessaire, `make all` est suffisant et plus rapide.

### Vérification de l'installation

```bash
$ afl-fuzz --version
afl-fuzz++4.x ...

$ afl-clang-fast++ --version
...
```

Les binaires clés :

| Binaire | Rôle |
|---|---|
| `afl-clang-fast++` | Compilateur instrumenté (LLVM, recommandé) |
| `afl-g++` | Compilateur instrumenté (GCC, alternative) |
| `afl-fuzz` | Moteur de fuzzing principal |
| `afl-cmin` | Minimisation du corpus (supprime les entrées redondantes) |
| `afl-tmin` | Minimisation d'un cas de test individuel |
| `afl-showmap` | Affiche la couverture d'une entrée |
| `afl-plot` | Génère des graphiques de progression |
| `afl-whatsup` | Résumé de l'état d'une campagne parallèle |

## Instrumentation du code C++

### Mode LLVM (recommandé)

Le mode LLVM (`afl-clang-fast++`) est le plus performant et le plus précis. Il insère l'instrumentation au niveau du code intermédiaire LLVM (IR), ce qui permet au compilateur d'optimiser le code instrumenté :

```bash
# Compilation instrumentée — remplacer g++/clang++ par afl-clang-fast++
afl-clang-fast++ -std=c++23 -O2 -g \
    -o fuzz_target \
    harness.cpp parser.cpp utils.cpp
```

L'instrumentation ajoute un compteur à chaque transition entre blocs de base (edge coverage). Le surcoût à l'exécution est d'environ 10-15 % — négligeable dans le contexte du fuzzing.

### Avec les sanitizers

La combinaison AFL++ + ASan + UBSan est la configuration standard pour le fuzzing de sécurité. AFL++ fournit des variables d'environnement dédiées pour activer les sanitizers :

```bash
# Méthode 1 : variables d'environnement AFL++
export AFL_USE_ASAN=1  
export AFL_USE_UBSAN=1  
afl-clang-fast++ -std=c++23 -O1 -g \  
    -o fuzz_target_asan \
    harness.cpp parser.cpp

# Méthode 2 : flags explicites (équivalent)
afl-clang-fast++ -std=c++23 -O1 -g \
    -fsanitize=address,undefined \
    -fno-omit-frame-pointer \
    -o fuzz_target_asan \
    harness.cpp parser.cpp
```

> **Attention au niveau d'optimisation** : avec ASan, `-O1` est préférable à `-O2` ou `-O3`. Les optimisations agressives peuvent éliminer du code que le sanitizer aurait instrumenté, réduisant la couverture de détection. `-O0` n'est pas recommandé car il ralentit considérablement le fuzzing.

### Mode GCC (alternative)

Si Clang n'est pas disponible, AFL++ peut instrumenter via GCC :

```bash
afl-g++ -std=c++23 -O2 -g \
    -o fuzz_target_gcc \
    harness.cpp parser.cpp
```

Le mode GCC utilise un plugin de compilateur qui insère l'instrumentation au niveau assembleur. Il est fonctionnel mais offre une couverture légèrement moins fine que le mode LLVM.

### Intégration CMake

```cmake
option(ENABLE_AFL_FUZZING "Build with AFL++ instrumentation" OFF)

if(ENABLE_AFL_FUZZING)
    # Forcer l'utilisation du compilateur AFL++
    set(CMAKE_CXX_COMPILER "afl-clang-fast++")
    set(CMAKE_C_COMPILER "afl-clang-fast")

    # Sanitizers pour la détection maximale
    add_compile_options(-fsanitize=address,undefined -fno-omit-frame-pointer -O1 -g)
    add_link_options(-fsanitize=address,undefined)

    # Cible de fuzzing
    add_executable(fuzz_json_parser
        fuzz/fuzz_json_parser.cpp
        src/json_parser.cpp
    )
endif()
```

```bash
cmake -B build_afl -DENABLE_AFL_FUZZING=ON  
cmake --build build_afl  
```

## Écrire un harness pour AFL++

AFL++ alimente le programme cible via stdin (par défaut) ou via un fichier. Le harness le plus simple lit les données depuis stdin et les passe à la fonction à tester :

### Harness basique (stdin)

```cpp
// fuzz/fuzz_json_parser.cpp
#include <iostream>
#include <vector>
#include <iterator>
#include <cstdint>

// Le code à tester
#include "json_parser.h"

int main() {
    // Lire toute l'entrée depuis stdin
    std::vector<uint8_t> input(
        std::istreambuf_iterator<char>(std::cin),
        std::istreambuf_iterator<char>()
    );

    if (input.empty()) return 0;

    // Appeler le code à tester
    try {
        auto result = json::parse(input.data(), input.size());
        // Optionnel : exercer le résultat pour couvrir plus de code
        if (result.is_object()) {
            for (const auto& [key, val] : result.items()) {
                (void)key;
                (void)val;
            }
        }
    } catch (const json::ParseError&) {
        // Les erreurs de parsing sont attendues — pas un bug
    }
    // Ne pas catcher std::exception ou ... — laisser les vrais bugs crasher

    return 0;
}
```

Points clés du harness :

- **Catcher uniquement les exceptions attendues** — les exceptions de parsing sont normales et ne sont pas des bugs. En revanche, un `std::bad_alloc`, un `std::out_of_range` inattendu, ou un signal (`SIGSEGV`, `SIGABRT`) indiquent un vrai bug et doivent provoquer un crash visible par AFL++.  
- **Ne pas faire d'I/O disque ou réseau** — le harness doit être rapide. Chaque milliseconde perdue se multiplie par des millions d'exécutions.  
- **Ne pas conserver d'état global entre les exécutions** — bien que le mode persistent (voir ci-dessous) réutilise le même processus, chaque itération doit être indépendante.

### Harness avec fichier

Si le code à tester attend un nom de fichier plutôt que des données en mémoire, AFL++ peut passer le chemin du fichier temporaire via le marqueur `@@` :

```cpp
// fuzz/fuzz_image_loader.cpp
#include "image_loader.h"
#include <cstdlib>

int main(int argc, char* argv[]) {
    if (argc < 2) return 1;

    try {
        auto image = load_image(argv[1]);  // Charge depuis un fichier
    } catch (const ImageFormatError&) {
        // Erreur de format attendue
    }

    return 0;
}
```

```bash
# @@ est remplacé par le chemin du fichier de test
afl-fuzz -i corpus/images -o findings -- ./fuzz_image_loader @@
```

### Mode persistent (performance ×10-20)

Par défaut, AFL++ lance un nouveau processus pour chaque entrée de test (via un fork server). Le mode persistent permet de traiter plusieurs entrées dans le même processus, éliminant le coût du fork à chaque itération :

```cpp
// fuzz/fuzz_parser_persistent.cpp
#include <cstdint>
#include <cstddef>
#include <unistd.h>
#include "parser.h"

// Macro AFL++ pour le mode persistent
__AFL_FUZZ_INIT();

int main() {
    // Tampon partagé entre AFL++ et le harness
    unsigned char *buf = __AFL_FUZZ_TESTCASE_BUF;

    // Boucle persistent : le processus traite N entrées avant d'être relancé
    while (__AFL_LOOP(10000)) {
        int len = __AFL_FUZZ_TESTCASE_LEN;

        if (len < 1) continue;

        // Appeler le code à tester
        try {
            parse(buf, static_cast<size_t>(len));
        } catch (const ParseError&) {
            // Attendu
        }
    }

    return 0;
}
```

Le mode persistent offre un gain de vitesse de 10× à 20× par rapport au mode fork classique. Le paramètre de `__AFL_LOOP` (ici 10000) indique le nombre d'itérations avant de relancer le processus, ce qui limite l'accumulation de fuites mémoire ou d'état résiduel.

> **Prérequis** : le mode persistent nécessite que le code à tester ne modifie pas d'état global de manière irréversible entre les appels. Si c'est le cas, réduire le nombre d'itérations ou utiliser le mode fork classique.

## Lancer une campagne de fuzzing

### Commande de base

```bash
afl-fuzz -i <corpus_dir> -o <findings_dir> -- <programme> [arguments]
```

Exemple concret :

```bash
# Créer le corpus initial
mkdir -p corpus/json  
echo '{"key": "value"}' > corpus/json/simple.json  
echo '[]' > corpus/json/empty_array.json  
echo '{"n": 42}' > corpus/json/number.json  

# Lancer le fuzzing
afl-fuzz -i corpus/json -o findings -x dictionaries/json.dict -- ./fuzz_json_parser
```

| Option | Rôle |
|---|---|
| `-i <dir>` | Répertoire du seed corpus |
| `-o <dir>` | Répertoire de sortie (crashs, corpus évolutif, stats) |
| `-x <dict>` | Dictionnaire de tokens spécifiques au format |
| `-t <ms>` | Timeout par exécution (défaut : auto-calibré) |
| `-m <Mo>` | Limite mémoire (important avec ASan — voir ci-dessous) |
| `--` | Séparateur entre les options AFL++ et la commande cible |

### Configuration mémoire avec ASan

ASan utilise beaucoup de mémoire virtuelle (adresse shadow). La limite mémoire par défaut d'AFL++ peut être insuffisante :

```bash
# Désactiver la limite mémoire quand ASan est actif
afl-fuzz -i corpus -o findings -m none -- ./fuzz_target_asan
```

Le `-m none` est nécessaire avec ASan ; sans lui, AFL++ tue le processus en croyant à un memory leak alors qu'il s'agit de l'espace d'adressage shadow d'ASan.

### Configuration système requise

AFL++ nécessite quelques réglages système pour fonctionner de manière optimale :

```bash
# Désactiver la notification de crash vers un handler externe
# (AFL++ gère lui-même la détection de crashs)
echo core | sudo tee /proc/sys/kernel/core_pattern

# Optionnel : désactiver le CPU frequency scaling pour des mesures stables
echo performance | sudo tee /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor
```

Si `core_pattern` n'est pas configuré, AFL++ affiche un avertissement au démarrage et peut manquer certains crashs gérés par le handler de la distribution (comme `apport` sur Ubuntu).

### Lire l'interface de monitoring

AFL++ affiche un tableau de bord en temps réel dans le terminal :

```
       american fuzzy lop ++4.09a {default} (./fuzz_json_parser)
┌─ process timing ────────────────────────────────────┐
│  run time : 0 days, 2 hrs, 13 min, 5 sec            │
│ last find : 0 days, 0 hrs, 0 min, 42 sec            │
│last crash : 0 days, 0 hrs, 15 min, 3 sec            │
├─ overall results ───────────────────────────────────┤
│    cycles : 12                                      │
│   corpus : 847 (max: 1.2 kB)                        │
│  crashes : 3 (3 unique)                             │
│  timeouts : 1 (1 unique)                            │
├─ cycle progress ────────────────────────────────────┤
│   now at : 423/847 (49.9%)                          │
│ map coverage : 12.45%                               │
├─ stage progress ────────────────────────────────────┤
│   now trying : havoc                                │
│  exec speed : 8543/sec                              │
└─────────────────────────────────────────────────────┘
```

Les métriques clés à surveiller :

| Métrique | Signification | Valeur saine |
|---|---|---|
| `exec speed` | Exécutions par seconde | > 1000 (idéalement > 5000) |
| `last find` | Temps depuis la dernière entrée ajoutée au corpus | Si > 1h, le fuzzer stagne |
| `crashes` | Nombre de crashs uniques trouvés | Tout crash est un bug potentiel |
| `map coverage` | Pourcentage de la bitmap de couverture utilisé | Croissance régulière attendue |
| `cycles` | Nombre de passes complètes sur le corpus | > 5 sans nouveau find → stagnation |

Si `exec speed` est inférieur à quelques centaines, le harness est trop lent — profiler et optimiser (éliminer les I/O, réduire les allocations, activer le mode persistent).

Si `last find` stagne depuis longtemps, envisager d'enrichir le corpus initial, d'ajouter un dictionnaire, ou de passer à une stratégie de mutation différente.

## Fuzzing parallèle

AFL++ supporte nativement la distribution du fuzzing sur plusieurs cœurs. L'architecture repose sur un processus primaire (qui effectue les mutations déterministes) et des processus secondaires (qui effectuent des mutations aléatoires). Tous partagent le même répertoire de sortie et synchronisent automatiquement leurs découvertes :

```bash
# Terminal 1 : instance primaire
afl-fuzz -i corpus -o findings -M primary -- ./fuzz_target

# Terminal 2 : instance secondaire 1
afl-fuzz -i corpus -o findings -S secondary01 -- ./fuzz_target

# Terminal 3 : instance secondaire 2
afl-fuzz -i corpus -o findings -S secondary02 -- ./fuzz_target
```

| Option | Rôle |
|---|---|
| `-M <name>` | Instance primaire (mutations déterministes + aléatoires) |
| `-S <name>` | Instance secondaire (mutations aléatoires uniquement) |

Règle empirique : utiliser une instance primaire et N-1 instances secondaires, où N est le nombre de cœurs disponibles.

Pour surveiller l'état de toutes les instances :

```bash
afl-whatsup findings/
```

### Script de lancement multi-cœurs

```bash
#!/bin/bash
# launch_fuzzing.sh — Lance une campagne parallèle sur tous les cœurs
CORPUS="corpus/json"  
OUTPUT="findings"  
TARGET="./fuzz_json_parser"  
DICT="dictionaries/json.dict"  

NCORES=$(nproc)  
echo "Lancement sur $NCORES cœurs"  

# Instance primaire
screen -dmS afl_primary \
    afl-fuzz -i "$CORPUS" -o "$OUTPUT" -x "$DICT" -m none -M primary -- "$TARGET"

# Instances secondaires
for i in $(seq 1 $((NCORES - 1))); do
    screen -dmS "afl_sec_$i" \
        afl-fuzz -i "$CORPUS" -o "$OUTPUT" -x "$DICT" -m none -S "secondary_$i" -- "$TARGET"
done

echo "Campagne lancée. Utilisez 'afl-whatsup $OUTPUT' pour surveiller."  
echo "Utilisez 'screen -r afl_primary' pour voir l'instance principale."  
```

## Gestion des résultats

### Structure du répertoire de sortie

```
findings/
├── primary/
│   ├── crashes/          # Entrées qui provoquent un crash
│   │   ├── id:000000,...
│   │   ├── id:000001,...
│   │   └── README.txt
│   ├── hangs/            # Entrées qui provoquent un timeout
│   │   └── id:000000,...
│   ├── queue/            # Corpus évolutif (entrées avec couverture nouvelle)
│   │   ├── id:000000,orig:simple.json
│   │   ├── id:000001,...
│   │   └── ...
│   ├── fuzzer_stats      # Statistiques de l'instance
│   └── plot_data         # Données pour afl-plot
├── secondary_1/
│   ├── crashes/
│   ├── hangs/
│   └── queue/
└── ...
```

### Trier et analyser les crashs

```bash
# Lister tous les crashs uniques
ls findings/*/crashes/

# Reproduire un crash spécifique
./fuzz_target_asan < findings/primary/crashes/id:000000,sig:06,...
```

Avec ASan activé, la reproduction affiche un diagnostic complet :

```
=================================================================
==54321==ERROR: AddressSanitizer: heap-buffer-overflow on address 0x602000000048
READ of size 1 at 0x602000000048 thread T0
    #0 0x5555555634a2 in json::Parser::parse_string() parser.cpp:142
    #1 0x555555563102 in json::Parser::parse_value() parser.cpp:87
    ...
```

### Minimiser un cas de crash

Les entrées produites par AFL++ contiennent souvent des octets superflus. `afl-tmin` réduit l'entrée au strict minimum nécessaire pour reproduire le crash :

```bash
# Minimiser un crash
afl-tmin -i findings/primary/crashes/id:000000,sig:06 \
         -o minimized_crash.bin \
         -- ./fuzz_target_asan

# Le fichier minimized_crash.bin est la plus petite entrée
# qui reproduit encore le crash
```

La minimisation est essentielle pour l'analyse : un crash reproductible avec 12 octets est bien plus facile à comprendre qu'un crash avec 4 Ko de données.

### Minimiser le corpus

Au fil du temps, le corpus de `queue/` accumule des entrées redondantes. `afl-cmin` élimine celles qui ne contribuent pas à la couverture :

```bash
# Garder uniquement les entrées qui apportent une couverture unique
afl-cmin -i findings/primary/queue -o corpus_minimized -- ./fuzz_target

# Le corpus minimisé peut servir de seed pour les campagnes futures
```

Un corpus minimisé se lance plus vite et converge plus rapidement lors de la reprise d'une campagne.

## Stratégies avancées

### Power schedules

AFL++ propose plusieurs stratégies de planification qui influencent le temps consacré à chaque entrée du corpus. La stratégie par défaut (`explore`) fonctionne bien dans la majorité des cas, mais d'autres schedules peuvent être plus efficaces selon le contexte :

```bash
# Explore (défaut) — équilibré
afl-fuzz -p explore -i corpus -o findings -- ./fuzz_target

# Fast — favorise les entrées récemment découvertes
afl-fuzz -p fast -i corpus -o findings -- ./fuzz_target

# Rare — favorise les entrées qui couvrent des edges rares
afl-fuzz -p rare -i corpus -o findings -- ./fuzz_target
```

Pour le fuzzing parallèle, il est recommandé d'utiliser des schedules différents sur les instances secondaires afin de diversifier l'exploration :

```bash
afl-fuzz -i corpus -o findings -M primary -p explore  -- ./fuzz_target  
afl-fuzz -i corpus -o findings -S sec01   -p fast     -- ./fuzz_target  
afl-fuzz -i corpus -o findings -S sec02   -p rare     -- ./fuzz_target  
afl-fuzz -i corpus -o findings -S sec03   -p explore  -- ./fuzz_target  
```

### CmpLog : résolution automatique des comparaisons

L'un des obstacles classiques du fuzzing est le franchissement des vérifications de magic bytes — un parser qui commence par `if (header == 0xDEADBEEF)` rejettera la quasi-totalité des entrées mutées. CmpLog instrumente les comparaisons du programme et extrait les valeurs attendues, permettant au fuzzer de les injecter intelligemment dans les mutations :

```bash
# Compiler un binaire spécifique pour CmpLog (sans ASan)
AFL_LLVM_CMPLOG=1 afl-clang-fast++ -std=c++23 -O2 -g \
    -o fuzz_target_cmplog \
    harness.cpp parser.cpp

# Utiliser le binaire CmpLog comme oracle de comparaisons
# Le binaire principal reste le binaire instrumenté standard (avec ASan)
afl-fuzz -i corpus -o findings \
    -c ./fuzz_target_cmplog \
    -m none \
    -- ./fuzz_target_asan
```

L'option `-c` indique à AFL++ d'utiliser le binaire CmpLog pour extraire les informations de comparaison, tout en exécutant le binaire principal (avec ASan) pour la détection des bugs. Cette séparation est nécessaire car CmpLog et ASan ne sont pas compatibles dans le même binaire.

### Fuzzing de binaires sans source (QEMU mode)

Pour les bibliothèques propriétaires ou les binaires tiers dont le code source n'est pas disponible, AFL++ peut instrumenter à l'exécution via l'émulation QEMU :

```bash
# Fuzzer un binaire non instrumenté
afl-fuzz -Q -i corpus -o findings -- ./binary_sans_source @@
```

Le mode QEMU est environ 3-5× plus lent que l'instrumentation compile-time, mais il permet de fuzzer n'importe quel binaire ELF x86/x86_64/ARM sans recompilation.

## Intégration CI/CD

### Job GitHub Actions

```yaml
# .github/workflows/fuzzing.yml
name: AFL++ Fuzzing  
on:  
  schedule:
    - cron: '0 2 * * *'  # Fuzzing nocturne quotidien
  workflow_dispatch:       # Déclenchement manuel

jobs:
  fuzz:
    runs-on: ubuntu-latest
    timeout-minutes: 360   # 6 heures de fuzzing
    steps:
      - uses: actions/checkout@v4

      - name: Installer AFL++
        run: sudo apt-get update && sudo apt-get install -y afl++

      - name: Compiler les cibles de fuzzing
        run: |
          export AFL_USE_ASAN=1
          export AFL_USE_UBSAN=1
          afl-clang-fast++ -std=c++23 -O1 -g \
              -o fuzz_json fuzz/fuzz_json_parser.cpp src/json_parser.cpp

      - name: Configurer le système
        run: echo core | sudo tee /proc/sys/kernel/core_pattern

      - name: Lancer le fuzzing
        run: |
          timeout 21000 afl-fuzz \
              -i fuzz/corpus/json \
              -o findings \
              -x fuzz/dictionaries/json.dict \
              -m none \
              -- ./fuzz_json || true
          # timeout 21000 = 5h50 (marge avant le timeout du job)
          # || true → ne pas échouer si timeout normal

      - name: Vérifier les crashs
        run: |
          CRASHES=$(find findings/default/crashes -type f ! -name README.txt | wc -l)
          echo "Crashs trouvés : $CRASHES"
          if [ "$CRASHES" -gt 0 ]; then
            echo "::error::AFL++ a trouvé $CRASHES crash(s) !"
            # Reproduire chaque crash pour le diagnostic
            for f in findings/default/crashes/id:*; do
              echo "=== Reproduction : $f ==="
              ./fuzz_json < "$f" 2>&1 || true
            done
            exit 1
          fi

      - name: Archiver le corpus évolué
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: afl-corpus-${{ github.run_id }}
          path: findings/default/queue/
          retention-days: 30

      - name: Archiver les crashs
        if: failure()
        uses: actions/upload-artifact@v4
        with:
          name: afl-crashes-${{ github.run_id }}
          path: findings/default/crashes/
```

Points clés de cette configuration :

- Le fuzzing tourne en **schedule nocturne** — 6 heures de fuzzing quotidien sans impacter les développeurs.  
- Le corpus évolué est **archivé comme artifact** — il sert de seed pour la campagne suivante, accumulant la couverture au fil des jours.  
- Les crashs font **échouer le job** — ils sont visibles dans l'interface CI et archivés pour analyse.  
- Le timeout est géré par `timeout` côté shell, avec une marge de 10 minutes avant le timeout du job pour laisser le temps à l'archivage.

## Résumé

AFL++ est le fuzzer le plus polyvalent de l'écosystème C++. Les points essentiels :

- **Instrumenter avec `afl-clang-fast++`** (mode LLVM) pour la meilleure couverture et performance.  
- **Toujours combiner avec ASan et UBSan** — le fuzzing sans sanitizer manque la majorité des bugs exploitables.  
- **Utiliser le mode persistent** (`__AFL_LOOP`) pour un gain de vitesse de 10-20×.  
- **Activer CmpLog** (`-c`) pour franchir les vérifications de magic bytes et de checksums.  
- **Fuzzer en parallèle** — une instance primaire (`-M`) et N-1 secondaires (`-S`) avec des power schedules variés.  
- **Minimiser les crashs** (`afl-tmin`) et le corpus (`afl-cmin`) régulièrement.  
- **Intégrer dans le CI/CD** — fuzzing nocturne avec archivage du corpus évolutif et alertes sur les crashs.
