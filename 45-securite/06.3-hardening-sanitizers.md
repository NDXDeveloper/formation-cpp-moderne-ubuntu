# 45.6.3 — Hardening avec les sanitizers en production

## Section 45.6 — Sécurité mémoire : Réponses concrètes du comité C++ en 2026 🔥
## Chapitre 45 — Sécurité en C++ ⭐

---

## Introduction

La section 29.4 couvre les sanitizers comme outils de test — ASan, UBSan, TSan, MSan — avec des surcoûts de 2× à 15× en temps d'exécution et 2-3× en mémoire. Ces surcoûts sont acceptables en phase de test et de fuzzing, mais pas en production. Pendant longtemps, cela signifiait que les sanitizers étaient cantonnés au développement : les bugs trouvés étaient corrigés, mais les bugs non trouvés se retrouvaient en production sans filet.

Ce modèle évolue. L'industrie déploie aujourd'hui des formes de hardening dérivées des sanitizers — mais calibrées pour la production : surcoûts inférieurs à quelques pourcents, diagnostics réduits au minimum, et comportement en cas de détection limité à un trap immédiat (terminaison du programme) plutôt qu'à un rapport détaillé. L'idée fondamentale : mieux vaut un crash contrôlé qu'une exploitation silencieuse.

Cette section couvre les techniques de hardening déployables en production sur un binaire C++, leur coût mesuré, et les stratégies d'activation progressive.

---

## La pyramide du hardening

Le hardening en production s'organise en couches de coût croissant. Chaque couche ajoute de la protection avec un surcoût supplémentaire. L'objectif est d'activer le maximum de couches dont le surcoût est acceptable pour le contexte d'exécution :

| Couche | Surcoût typique | Mécanisme | Couverture |
|---|---|---|---|
| Protections compilateur (section 45.4) | < 2 % | Stack canary, FORTIFY_SOURCE, PIE, RELRO | Exploitabilité réduite |
| Hardened STL | < 1 % | Vérifications de bornes sur les conteneurs | Buffer overflows via STL |
| UBSan trapping mode | 1-5 % | Vérifications UB avec trap immédiat | Signed overflow, null deref, shift, etc. |
| Allocateur renforcé (Scudo) | 1-3 % | Quarantaine, détection double-free | UAF, double-free, heap overflow |
| CFI (Control-Flow Integrity) | 1-3 % | Vérification des cibles d'appels indirects | Détournement de flux d'exécution |
| MTE (ARM, matériel) | 3-5 % | Tagging mémoire matériel | UAF, heap overflow |
| ASan (test uniquement) | ~100 % | Shadow memory complète | Tout (mais trop coûteux pour la production) |

Les trois premières couches (protections compilateur, hardened STL, UBSan trapping) devraient être activées dans pratiquement tous les projets. Les couches suivantes s'ajoutent en fonction du profil de risque et des contraintes de performance.

## Hardened STL en production

### libc++ (Clang/LLVM)

Le hardening de libc++ est la success story la plus importante du hardening C++ en production. Développé par Konstantin Varlamov et Louis Dionne, déployé chez Google et Apple sur des centaines de millions de lignes de code, il ajoute des vérifications de bornes aux opérations critiques de la STL avec un surcoût mesuré à une fraction de pourcent.

libc++ propose trois niveaux de hardening :

**Mode `fast`** — vérifie les préconditions dont la violation est la plus dangereuse et la plus fréquente, avec un coût minimal. C'est le mode recommandé pour la production :

```bash
clang++ -D_LIBCPP_HARDENING_MODE=_LIBCPP_HARDENING_MODE_FAST ...
```

Ce mode vérifie notamment les accès via `operator[]` sur `vector`, `string`, `string_view`, `span`, `array`, ainsi que les déréférencements d'itérateurs et les accès à `optional` vide. Chaque vérification échouée déclenche un trap (`__builtin_trap()`) qui termine immédiatement le programme — pas de message d'erreur, pas de stack trace, juste un signal `SIGILL` ou `SIGTRAP`. Ce comportement est délibéré : en production, l'objectif est d'empêcher l'exploitation, pas de diagnostiquer le bug (le diagnostic se fait en développement avec ASan).

**Mode `extensive`** — ajoute des vérifications supplémentaires dont le coût est légèrement plus élevé. Recommandé pour les environnements de staging ou les services non critiques en latence :

```bash
clang++ -D_LIBCPP_HARDENING_MODE=_LIBCPP_HARDENING_MODE_EXTENSIVE ...
```

**Mode `debug`** — vérifications maximales, incluant la vérification de la validité des itérateurs et des préconditions internes de la STL. Réservé au développement et aux tests :

```bash
clang++ -D_LIBCPP_HARDENING_MODE=_LIBCPP_HARDENING_MODE_DEBUG ...
```

### libstdc++ (GCC)

GCC/libstdc++ offre deux mécanismes :

```bash
# Assertions de préconditions — recommandé pour la production
g++ -D_GLIBCXX_ASSERTIONS ...

# Mode debug complet — développement uniquement
g++ -D_GLIBCXX_DEBUG ...
```

`_GLIBCXX_ASSERTIONS` ajoute des vérifications de bornes sur `operator[]`, des vérifications de validité des itérateurs dans les algorithmes, et des vérifications de préconditions dans les conteneurs associatifs. Le surcoût est comparable au mode `fast` de libc++.

`_GLIBCXX_DEBUG` change la représentation interne des conteneurs (ajout de métadonnées de tracking) et est incompatible au niveau ABI avec le mode normal — il ne peut pas être utilisé en production et ne doit jamais être mélangé avec du code compilé sans ce flag.

### Intégration CMake

```cmake
# Hardening STL adapté au build type
if(CMAKE_CXX_COMPILER_ID MATCHES "Clang")
    # Production : mode fast
    target_compile_definitions(my_app PRIVATE
        $<$<CONFIG:Release>:_LIBCPP_HARDENING_MODE=_LIBCPP_HARDENING_MODE_FAST>
        $<$<CONFIG:RelWithDebInfo>:_LIBCPP_HARDENING_MODE=_LIBCPP_HARDENING_MODE_EXTENSIVE>
        $<$<CONFIG:Debug>:_LIBCPP_HARDENING_MODE=_LIBCPP_HARDENING_MODE_DEBUG>
    )
elseif(CMAKE_CXX_COMPILER_ID MATCHES "GNU")
    target_compile_definitions(my_app PRIVATE
        $<$<NOT:$<CONFIG:Debug>>:_GLIBCXX_ASSERTIONS>
        $<$<CONFIG:Debug>:_GLIBCXX_DEBUG>
    )
endif()
```

## UBSan en mode trapping

### Le problème d'UBSan classique

Le UndefinedBehaviorSanitizer standard (section 29.4.2) utilise un runtime qui produit des messages d'erreur détaillés lorsqu'un comportement indéfini est détecté. Ce runtime a un coût non négligeable en taille de binaire et en performance, ce qui le rend inadapté à la production.

### La solution : le mode trapping

UBSan propose un mode dans lequel chaque détection de comportement indéfini est remplacée par un simple trap CPU — aucun runtime, aucun message, aucun surcoût de mémoire. Le programme se termine immédiatement avec un signal :

```bash
# UBSan en mode trapping — adapté à la production
clang++ -fsanitize=undefined -fsanitize-trap=all -fno-sanitize-recover=all ...

# Sélection fine des vérifications à activer
clang++ -fsanitize=signed-integer-overflow,null,alignment,bool,enum,shift \
        -fsanitize-trap=all \
        -fno-sanitize-recover=all ...
```

Le flag `-fsanitize-trap=all` remplace le runtime UBSan par des instructions trap. Le flag `-fno-sanitize-recover=all` garantit que le programme s'arrête à la première détection au lieu de continuer.

GCC propose un mécanisme similaire :

```bash
# GCC — UBSan avec trap
g++ -fsanitize=undefined -fsanitize-undefined-trap-on-error ...
```

### Quelles vérifications activer en production ?

Toutes les vérifications UBSan n'ont pas le même rapport coût/bénéfice en production. Voici les vérifications recommandées, classées par pertinence :

| Vérification | Flag | Coût | Pertinence production |
|---|---|---|---|
| Signed integer overflow | `-fsanitize=signed-integer-overflow` | Très faible | Élevée — source fréquente de bugs de sécurité |
| Null pointer dereference | `-fsanitize=null` | Faible | Élevée — empêche les crashs exploitables |
| Shift hors bornes | `-fsanitize=shift` | Très faible | Modérée |
| Alignement | `-fsanitize=alignment` | Faible | Modérée — critique sur certaines architectures |
| Bool/enum invalide | `-fsanitize=bool,enum` | Négligeable | Modérée |
| Division par zéro (entier) | `-fsanitize=integer-divide-by-zero` | Négligeable | Modérée |
| Return sans valeur | `-fsanitize=return` | Négligeable | Modérée |
| VLA bounds | `-fsanitize=vla-bound` | Négligeable | Faible (VLA rare en C++ moderne) |
| Unreachable | `-fsanitize=unreachable` | Négligeable | Faible |

La configuration recommandée pour la production combine les vérifications à fort impact et faible coût :

```bash
clang++ -fsanitize=signed-integer-overflow,null,shift,bool,enum \
        -fsanitize-trap=all \
        -fno-sanitize-recover=all \
        -O2 ...
```

Le surcoût cumulé de ces vérifications en mode trapping est typiquement de 1 à 3 %, variable selon la charge de travail.

### Intégration CMake

```cmake
# UBSan trapping pour la production
if(CMAKE_CXX_COMPILER_ID MATCHES "Clang")
    set(UBSAN_PRODUCTION_FLAGS
        -fsanitize=signed-integer-overflow,null,shift,bool,enum
        -fsanitize-trap=all
        -fno-sanitize-recover=all
    )
    target_compile_options(my_app PRIVATE
        $<$<CONFIG:Release>:${UBSAN_PRODUCTION_FLAGS}>
        $<$<CONFIG:RelWithDebInfo>:${UBSAN_PRODUCTION_FLAGS}>
    )
    target_link_options(my_app PRIVATE
        $<$<CONFIG:Release>:${UBSAN_PRODUCTION_FLAGS}>
        $<$<CONFIG:RelWithDebInfo>:${UBSAN_PRODUCTION_FLAGS}>
    )
endif()
```

## Allocateurs renforcés

### Le problème des allocateurs standard

L'allocateur par défaut de la glibc (`ptmalloc2`) est optimisé pour la performance, pas pour la sécurité. Quand un bloc mémoire est libéré, il est immédiatement disponible pour réallocation. Un use-after-free a de bonnes chances d'accéder à un bloc qui n'a pas encore été réattribué et de "fonctionner" silencieusement. Pire, un attaquant peut influencer les allocations pour placer ses données exactement à l'emplacement de l'objet libéré.

### Scudo (LLVM)

Scudo est un allocateur hardened développé par le projet LLVM. C'est l'allocateur par défaut d'Android depuis Android 11 et de Fuchsia. Il offre plusieurs protections :

- **Checksums sur les headers d'allocation** — détecte la corruption des métadonnées du heap.  
- **Quarantaine des blocs libérés** — retarde la réutilisation des adresses, réduisant l'exploitabilité des UAF.  
- **Randomisation des adresses d'allocation** — rend le heap layout moins prévisible.  
- **Détection de double-free** — vérifie que le bloc n'a pas déjà été libéré.

```bash
# Compiler avec Scudo comme allocateur
clang++ -fsanitize=scudo main.cpp -o main

# Ou précharger Scudo sur un binaire existant
LD_PRELOAD=/usr/lib/libscudo.so ./my_program
```

Le surcoût de Scudo est de 1 à 3 % en temps d'exécution et minimal en mémoire, ce qui le rend viable en production. Il ne détecte pas tous les UAF (contrairement à ASan), mais il rend leur exploitation significativement plus difficile.

### GWP-ASan (sampling ASan)

GWP-ASan (Google-Wide Performance ASan) est une approche différente : au lieu de vérifier tous les accès mémoire (comme ASan), il ne protège qu'un échantillon aléatoire des allocations. Les allocations échantillonnées sont placées dans des pages de garde dédiées, et tout accès hors bornes ou use-after-free sur ces allocations est détecté avec un diagnostic complet.

```bash
# GWP-ASan est intégré à certains allocateurs
# Sur Android : activé par défaut via Scudo
# Sur Linux avec tcmalloc :
MALLOC_CONF="gwp_asan:true" ./my_program
```

L'avantage de GWP-ASan est son surcoût quasi nul (< 1 %) : seule une fraction des allocations est protégée. L'inconvénient est probabiliste — un bug spécifique peut ne pas être détecté lors d'une exécution donnée. Sur une flotte de serveurs ou une base d'utilisateurs large, la probabilité qu'au moins une instance détecte un bug donné augmente avec le nombre d'exécutions.

Google déploie GWP-ASan en production sur Chrome et Android. Le nombre de bugs trouvés par cette méthode valide son efficacité malgré le sampling.

## Control-Flow Integrity (CFI)

CFI est une protection qui vérifie que les appels de fonctions indirects (via pointeurs de fonction ou vtables) ciblent des destinations valides. C'est une défense contre les attaques qui détournent le flux d'exécution après une corruption mémoire (ROP, JOP, vtable hijacking — voir section 45.3).

```bash
# CFI avec Clang — nécessite LTO
clang++ -flto -fsanitize=cfi -fvisibility=hidden main.cpp -o main
```

CFI est implémenté par Clang via LTO (Link-Time Optimization) : le compilateur a besoin d'une vue globale du programme pour construire les ensembles de destinations valides. Le flag `-fvisibility=hidden` est nécessaire pour limiter les destinations possibles.

Le surcoût de CFI est de 1 à 3 % en temps d'exécution. Android l'active par défaut dans les composants sensibles depuis Android 8.

GCC propose `-fcf-protection` qui utilise les instructions Intel CET (Control-flow Enforcement Technology) ou ARM BTI (Branch Target Identification) pour une protection matérielle avec un surcoût encore plus faible :

```bash
# Protection matérielle du flux de contrôle
g++ -fcf-protection=full main.cpp -o main
# Utilise ENDBR64 (Intel CET) ou BTI (ARM) selon l'architecture
```

## MTE : Memory Tagging Extension (ARM)

### Le principe

MTE est une extension matérielle des processeurs ARM v8.5+ qui associe un tag de 4 bits à chaque allocation mémoire et à chaque pointeur. Quand un pointeur est déréférencé, le processeur vérifie que le tag du pointeur correspond au tag de la mémoire accédée. Un mismatch indique un accès invalide (use-after-free, buffer overflow) et déclenche une exception.

```
Pointeur : 0x0A00_5555_6789_0100
               ^^
               Tag (4 bits dans les bits hauts du pointeur)

Mémoire  : [tag=0xA][données...][tag=0xA][données...]
```

À la libération d'un bloc, l'allocateur change le tag de la mémoire. Tout pointeur conservé (dangling pointer) porte encore l'ancien tag et sera détecté au déréférencement.

### Modes de fonctionnement

MTE propose deux modes :

**Mode synchrone** — chaque accès mémoire est vérifié immédiatement. Un mismatch de tag provoque une exception synchrone (comme un SIGSEGV). Le surcoût est de 3 à 5 % :

```bash
# Activer MTE en mode synchrone (AArch64)
clang++ -march=armv8.5-a+memtag -fsanitize=memtag-stack,memtag-heap ...
```

**Mode asynchrone** — les vérifications sont reportées et agrégées. Le surcoût est inférieur à 1 %, mais la détection est retardée et le diagnostic moins précis (pas de stack trace exacte du site d'accès).

### Disponibilité en 2026

MTE est disponible sur :

- **Serveurs ARM Graviton 3+** (AWS) — utilisable en production cloud.  
- **Processeurs mobiles récents** — Pixel 8+, certains SoC Qualcomm et MediaTek récents.  
- **Android** — support natif depuis Android 12, avec activation progressive par les fabricants.

MTE est la technologie la plus prometteuse pour le hardening en production car elle offre une détection des UAF et des heap overflows avec un surcoût compatible avec la production, et sans aucune modification du code source. Sa principale limitation est la restriction aux architectures ARM v8.5+, ce qui exclut l'intégralité du parc x86_64 existant.

## Stratégie de déploiement progressif

### Phase 1 : le socle (immédiat, coût < 2 %)

Ces protections devraient être activées sur tout projet C++ en production, sans exception :

```cmake
# Protections compilateur (section 45.4)
add_compile_options(
    -fstack-protector-strong
    -fstack-clash-protection
    -fcf-protection
)
add_compile_definitions(_FORTIFY_SOURCE=2)  
set(CMAKE_POSITION_INDEPENDENT_CODE ON)  
add_link_options(-Wl,-z,relro -Wl,-z,now -Wl,-z,noexecstack)  

# Hardened STL
if(CMAKE_CXX_COMPILER_ID MATCHES "Clang")
    add_compile_definitions(
        _LIBCPP_HARDENING_MODE=_LIBCPP_HARDENING_MODE_FAST
    )
elseif(CMAKE_CXX_COMPILER_ID MATCHES "GNU")
    add_compile_definitions(_GLIBCXX_ASSERTIONS)
endif()
```

### Phase 2 : UBSan trapping (coût additionnel 1-3 %)

Ajouter UBSan en mode trapping pour les vérifications à fort impact :

```cmake
if(CMAKE_CXX_COMPILER_ID MATCHES "Clang")
    add_compile_options(
        -fsanitize=signed-integer-overflow,null,shift,bool,enum
        -fsanitize-trap=all
        -fno-sanitize-recover=all
    )
    add_link_options(
        -fsanitize=signed-integer-overflow,null,shift,bool,enum
        -fsanitize-trap=all
    )
endif()
```

### Phase 3 : allocateur renforcé (coût additionnel 1-3 %)

Remplacer l'allocateur par défaut par Scudo ou activer GWP-ASan :

```cmake
# Option 1 : Scudo à la compilation
if(CMAKE_CXX_COMPILER_ID MATCHES "Clang")
    add_compile_options(-fsanitize=scudo)
    add_link_options(-fsanitize=scudo)
endif()

# Option 2 : GWP-ASan via préchargement (pas de recompilation)
# Configurer dans le script de démarrage du service
```

### Phase 4 : CFI et MTE (selon l'architecture)

Pour les projets à haute criticité :

```cmake
# CFI — nécessite Clang + LTO
if(CMAKE_CXX_COMPILER_ID MATCHES "Clang" AND ENABLE_CFI)
    add_compile_options(-flto -fsanitize=cfi -fvisibility=hidden)
    add_link_options(-flto -fsanitize=cfi)
endif()

# MTE — ARM v8.5+ uniquement
if(CMAKE_SYSTEM_PROCESSOR MATCHES "aarch64" AND ENABLE_MTE)
    add_compile_options(-march=armv8.5-a+memtag -fsanitize=memtag-heap)
    add_link_options(-fsanitize=memtag-heap)
endif()
```

### Synthèse des coûts cumulés

| Configuration | Surcoût estimé | Protections |
|---|---|---|
| Phase 1 seule | < 2 % | Stack canary, FORTIFY_SOURCE, PIE, RELRO, hardened STL |
| Phase 1 + 2 | 2-5 % | + UBSan trapping (signed overflow, null, shift) |
| Phase 1 + 2 + 3 | 3-7 % | + Allocateur renforcé (Scudo) |
| Phase 1 + 2 + 3 + 4 | 5-12 % | + CFI et/ou MTE |

Un surcoût de 5 à 7 % (phases 1-3) est acceptable pour la majorité des services backend, des outils CLI, et des applications desktop. Les composants à très forte contrainte de latence (trading haute fréquence, boucles de rendu temps réel) peuvent se limiter aux phases 1-2 et compenser par un fuzzing intensif et une revue de code renforcée.

## Gestion des crashes en production

Le hardening transforme les exploitations silencieuses en crashes. C'est un gain de sécurité, mais cela implique que les crashes en production sont attendus et doivent être gérés :

**Génération de core dumps.** Configurer le système pour capturer les core dumps des processus terminés par un trap UBSan, un échec de vérification de bornes STL, ou une détection Scudo. Ces core dumps permettent l'analyse post-mortem (section 29.3) :

```bash
# Configurer la capture de core dumps
ulimit -c unlimited  
echo "/var/crash/core.%e.%p.%t" | sudo tee /proc/sys/kernel/core_pattern  
```

**Monitoring des signaux.** Les crashes de hardening produisent des signaux spécifiques (SIGTRAP pour UBSan trapping, SIGILL pour les traps STL, SIGABRT pour Scudo). Configurer le monitoring pour détecter les pics de crashes de ces types et alerter l'équipe.

**Redémarrage automatique.** Les services en production doivent être configurés pour un redémarrage automatique après un crash de hardening, avec un circuit breaker pour éviter les boucles de crash. systemd, supervisord ou un orchestrateur Kubernetes assurent cette fonction.

**Conversion en bug report.** Chaque crash de hardening en production est un bug qui aurait été une vulnérabilité exploitable sans le hardening. Il doit être traité avec la même priorité qu'une vulnérabilité de sécurité — analyse du core dump, identification de la cause racine, correction, et ajout d'un test de régression.

---

## Résumé

Le hardening en production est la couche de sécurité la plus immédiatement actionnable pour un projet C++ :

- **Le hardened STL est le meilleur rapport coût/bénéfice.** Moins de 1 % de surcoût pour la couverture de la majorité des accès hors bornes via les conteneurs. L'activer est une décision qui ne se discute pas.  
- **UBSan en mode trapping est le deuxième meilleur investissement.** 1-3 % de surcoût pour la détection des signed overflows, null derefs et shifts invalides. Le mode trapping élimine le runtime UBSan et ne produit aucune dépendance supplémentaire.  
- **Les allocateurs renforcés (Scudo, GWP-ASan) ajoutent une couche heap.** 1-3 % de surcoût pour réduire l'exploitabilité des UAF et des heap overflows.  
- **CFI et MTE sont les protections les plus avancées**, adaptées aux projets à haute criticité et aux architectures qui les supportent.  
- **Le hardening transforme les exploits en crashes.** Gérer les crashes en production (core dumps, monitoring, redémarrage) est une partie intégrante de la stratégie.  
- **Le surcoût cumulé de la phase 1 à 3 est de 3 à 7 %.** C'est un prix dérisoire comparé au coût d'une exploitation de sécurité réussie.
